## 2 EAS Build 共通設定

### 2-1. `eas.json` プロファイルの確認

**MIG-X-156**: EAS Build の実行前に、`eas.json` の `production` プロファイルおよび `staging` プロファイルが以下の要件を満たすことを確認する。

```json
{
  "build": {
    "production": {
      "distribution": "store",
      "android": {
        "buildType": "app-bundle"
      },
      "ios": {
        "resourceClass": "m-medium"
      }
    },
    "staging": {
      "distribution": "internal",
      "android": {
        "buildType": "apk"
      },
      "ios": {
        "simulator": false
      }
    }
  }
}
```

`production` プロファイルは本番配布専用であり、`staging` プロファイルは社内テスト・QA 専用とする。`eas.json` が上記の構造を満たすことを確認した上で EAS Build を実行する。

### 2-2. アプリバージョンの確認

**MIG-X-157**: EAS Build の実行前に、`package.json` の `version` フィールドおよび `app.json` の `version`・`versionCode`（Android）・`buildNumber`（iOS）が以下のセマンティックバージョニング規約に従っていることを確認する。

| フィールド | 本番初回リリース値 | 確認方法 |
|---|---|---|
| `package.json` の `version` | `1.0.0` | `cat package.json` で確認 |
| `app.json` の `version` | `1.0.0` | `cat app.json` で確認 |
| `app.json` の `android.versionCode` | `1` | `cat app.json` で確認 |
| `app.json` の `ios.buildNumber` | `1.0.0` | `cat app.json` で確認 |

---

**本節で確定した方針**
- `eas.json` の `production` / `staging` プロファイルの構造確認を EAS Build 実行前の必須条件とすることを確定する。
- 本番初回リリースのバージョンを `1.0.0`（Android versionCode: 1, iOS buildNumber: 1.0.0）に対応することを確定する。
- セマンティックバージョニング規約（MAJOR.MINOR.PATCH）への準拠を確定する。

## 3 ストア審査 vs OTA 配信の判定表

### 3-1. OS 別配布方法の判定表

**MIG-X-158**: 各 OS の配布方法は以下の判定表に基づいて決定する。本番初回リリースでは社内配布を優先する。

| OS | 配布方法 | 選択基準 | 審査要否 |
|---|---|---|---|
| Android | Google Play 内部テスト配布 | Google Play Console の管理下のデバイスへの配布 | 不要（内部テストトラック） |
| Android | 社内配布（APK 直接配布） | Google Play を経由しない社内限定配布 | 不要 |
| iOS | App Store 配布 | Apple の審査を経た公開配布 | 必要（審査期間 1〜7 日） |
| iOS | TestFlight | Apple 開発者アカウント配下の限定配布 | 簡易審査（通常 24 時間以内） |
| iOS | 社内 MDM（組織管理デバイス） | MDM 管理下のデバイスへの直接配布 | 不要 |
| Windows | MSIX 社内配布 | ファイルサーバー共有または USB による配布 | 不要 |

### 3-2. 本番初回リリースにおける判定

本番初回リリースにおける推奨配布方法を以下に定める。

- **Android**: Google Play 内部テストトラックへの配布を使用する。組織外部へのデバイス管理が不要な場合は APK 直接配布を選択する。
- **iOS**: TestFlight を使用する。組織が Apple Business Manager を導入済みの場合は MDM を選択する。
- **Windows**: MSIX パッケージによる社内配布を使用する。

---

**本節で確定した方針**
- 本番初回リリースでは社内配布方式（Android: 内部テスト/APK, iOS: TestFlight/MDM, Windows: MSIX）を優先とすることを確定する。
- App Store 公開配布は審査期間（最大 7 日）を考慮したスケジュール調整に対応することを確定する。
- 配布方法の選択は本判定表に基づいて決定することを確定する。

## 5 デバイス登録と疎通確認

### 5-1. GUI ウィザード Step 4（デバイス登録）との連携

**MIG-X-160**: 全ハンディ端末のハンディ APP インストール完了後、以下の手順でデバイス登録と本番 URL への接続確認を実施する。

GUI ウィザード Step 4（デバイス登録）との連携手順を以下に定める。

| ステップ | 手順 | 担当者 | 確認方法 |
|---|---|---|---|
| 1 | マスタメンテ管理画面の「デバイス管理」メニューを開く | system_admin | ブラウザ操作 |
| 2 | 「デバイス登録」ウィザードを起動する | system_admin | ウィザード画面の表示確認 |
| 3 | 各ハンディ端末の端末 ID・OS 種別・デバイス名を登録する | system_admin | 登録完了メッセージの確認 |
| 4 | ハンディ APP の設定画面で本番サーバー URL を入力する | system_admin | URL 設定保存確認 |
| 5 | ハンディ APP から本番 API エンドポイントへの接続確認を実行する | system_admin | 接続成功メッセージの確認 |
| 6 | 全台数の接続確認完了をチェックリスト（MIG-X-165 / 170 / 175）に記録する | system_admin | チェックリスト記録 |

全ハンディ端末の本番 URL 接続確認が完了した時点で本章の手順完了と判定する。

---

**本節で確定した方針**
- GUI ウィザード Step 4（デバイス登録）の完了を全ハンディ端末への適用条件とすることを確定する。
- 全端末の本番 URL 接続確認をハンディ APP 配布の完了条件とすることを確定する。
- 接続確認結果を各 OS 章のチェックリスト（MIG-X-165/170/175）に記録することを確定する。
