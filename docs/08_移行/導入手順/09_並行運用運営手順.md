# 09 並行運用運営手順

本章の責務は、INST-A4.5（並行運用フェーズ管理）の実施版として、`移行計画/05_並行運用実施計画.md` に定めた並行運用方針を具体的な日次・週次の運営手順に落とし込み、作業完了率 95% 到達によるカットオーバー移行判断までの全プロセスを確定することである。

---

## 1 本章の責務（移行計画/05_並行運用実施計画.md の実施版）

### 1-1. 並行運用フェーズの位置づけ

本章は IPA 共通フレーム 2013 の「3.2 導入プロセス」タスク INST-A4.5（並行運用フェーズ管理）に対応する実施手順書である。並行運用フェーズ（Phase M-2）は、新システム（ハンディ APP + axum バックエンド）と旧運用（紙記録）を同時に運用し、作業員がシステムに習熟しながら移行完了条件の充足を目指す期間である。

| 文書 | 役割 |
|---|---|
| `移行計画/05_並行運用実施計画.md` | 並行運用方針・完了条件・段階移行基準の確定 |
| 本章（09_並行運用運営手順） | 日次・週次の具体的運営手順 |

---

**本節で確定した方針**
- 本章は `移行計画/05_並行運用実施計画.md` の実施版として管理することを確定する。
- 並行運用中は新システムと紙記録の両方での記録を全作業員に義務付けることを確定する。
- 作業完了率 95% 到達の確認をカットオーバー移行判断の条件とすることを確定する。

## 2 並行運用の日次フロー

### 2-1. 朝礼での周知事項

**MIG-X-186**: 並行運用期間中、毎日の朝礼において現場監督は以下の事項を全作業員に周知する。周知内容の漏れは並行運用のデータ完全性に影響するため、朝礼での周知を必須とする。

| 周知事項 | 周知内容 | 周知者 |
|---|---|---|
| 記録の二重化義務 | システムと紙の両方で記録を取ることが義務であること | 現場監督 |
| 当日の作業担当者と担当工程 | 誰がどの工程を担当するか | 現場監督 |
| 前日のエラーレート | system_admin から報告された前日エラーレート（目標: 5% 未満） | 現場監督 |
| 問合せ先 | 操作に困った場合の system_admin への連絡方法 | 現場監督 |

朝礼での周知記録は現場監督が「並行運用日誌」に記録し、quality_admin に週次で報告する。

### 2-2. 作業開始時の端末起動確認と作業完了後の記録確認

**MIG-X-187**: 各作業員は作業開始時と作業完了後に以下の手順を実施する。

**作業開始時の端末起動確認手順:**

| ステップ | 手順 | 確認項目 |
|---|---|---|
| 1 | ハンディ端末の電源を入れる | 電源が正常に入ること |
| 2 | ハンディ APP を起動する | アプリが正常に起動すること |
| 3 | ログイン画面でアカウントにログインする | ログインが成功すること（エラーが表示されないこと） |
| 4 | 本番サーバーへの接続確認メッセージを確認する | 「接続済み」表示が確認できること |
| 5 | 担当工程が一覧に表示されることを確認する | 自分の担当工程が表示されること |

端末起動確認でエラーが発生した場合は、system_admin に即時連絡し、紙記録で作業を継続する。

**作業完了後の記録確認手順:**

| ステップ | 手順 | 確認項目 |
|---|---|---|
| 1 | ハンディ APP の「作業履歴」画面で当日の作業記録を確認する | 全実施作業のレコードが存在すること |
| 2 | 各作業の完了時刻が正確に記録されていることを確認する | 実施時刻との乖離が 5 分以内であること |
| 3 | 紙記録の作業件数とシステムの記録件数が一致していることを確認する | 件数が一致すること |
| 4 | 不一致がある場合は system_admin に報告する | 報告と記録の実施 |

---

**本節で確定した方針**
- 朝礼での「システムと紙の両方で記録」周知を並行運用中の日次必須活動とすることを確定する。
- 作業開始時の端末起動確認（5 ステップ）を全作業員の義務的手順とすることを確定する。
- 作業完了後の件数照合でシステム記録と紙記録の一致を確認することを確定する。

## 3 作業完了率・エラー率の日次計測

### 3-1. system_admin によるエラーログ確認手順（MIG-T-012 対応）

**MIG-X-188**: system_admin は毎日（業務終了後）以下の手順でエラーログを確認し、結果を「並行運用日次報告書」に記録する。

エラーログ確認コマンドを以下に定める。

```bash
# バックエンドコンテナのエラーログ確認（直近 24 時間分）
docker logs terminal-api --since 24h 2>&1 | grep -E "ERROR|WARN|FATAL"
docker logs master-api --since 24h 2>&1 | grep -E "ERROR|WARN|FATAL"

# エラーレート計算（API リクエスト総数に対するエラー件数の割合）
# PostgreSQL クエリで日次集計
docker exec <postgres_container_name> psql -U <db_user> -d <db_name> -c "
  SELECT
    DATE(created_at) AS date,
    COUNT(*) AS total_requests,
    COUNT(*) FILTER (WHERE status_code >= 400) AS error_requests,
    ROUND(COUNT(*) FILTER (WHERE status_code >= 400) * 100.0 / COUNT(*), 2) AS error_rate_pct
  FROM access_logs
  WHERE created_at >= NOW() - INTERVAL '24 hours'
  GROUP BY DATE(created_at);
"
```

エラーレートの判定基準を以下に定める。

| エラーレート | 判定 | 対応 |
|---|---|---|
| 5% 未満 | 正常 | 日次報告書に記録して経過観察 |
| 5% 以上 10% 未満 | 警告 | quality_admin に翌朝までに報告・原因調査を開始 |
| 10% 以上 | 異常 | quality_admin に即時報告・縮退対応を検討 |

### 3-2. 週次集計と quality_admin への報告（MIG-T-013 対応）

週次集計では以下の指標を計算し、quality_admin に報告する。

| 集計指標 | 定義 | 目標値 |
|---|---|---|
| 作業完了率 | システムで記録された作業件数 / 紙記録の作業件数 × 100% | 95% 以上 |
| エラーレート（週次平均） | 週次 API エラー件数 / 週次 API 総リクエスト数 × 100% | 5% 未満 |
| デバイス稼働率 | 1 件以上のログインが記録されたデバイス台数 / 登録デバイス台数 × 100% | 95% 以上 |
| ユーザーログイン率 | 1 件以上のログインが記録されたアカウント数 / 有効アカウント数 × 100% | 95% 以上 |

週次報告書は毎週金曜日の業務終了後に system_admin が作成し、quality_admin に提出する。

---

**本節で確定した方針**
- エラーレートの日次確認と 5% / 10% の段階的な対応基準への準拠を確定する。
- 週次集計（作業完了率・エラーレート・デバイス稼働率・ユーザーログイン率）を毎週金曜日に実施することを確定する。
- 週次報告書の quality_admin への提出を必須とすることを確定する。

## 4 段階移行の進行管理

### 4-1. 作業完了率 95% 到達の確認方法

**MIG-X-189**: 作業完了率 95% 到達の確認は以下の手順で実施する。作業完了率は週次集計（MIG-X-188）の結果に基づいて判定する。

作業完了率の計算クエリを以下に定める。

```sql
-- 直近 1 週間の作業完了率計算
SELECT
  COUNT(*) FILTER (WHERE source = 'system') AS system_count,
  COUNT(*) FILTER (WHERE source = 'paper') AS paper_count,
  ROUND(
    COUNT(*) FILTER (WHERE source = 'system') * 100.0 /
    NULLIF(COUNT(*) FILTER (WHERE source = 'paper'), 0),
    2
  ) AS completion_rate_pct
FROM work_records_combined_view
WHERE recorded_at >= NOW() - INTERVAL '7 days';
```

作業完了率 95% 到達の判定基準を以下に定める。

| 判定条件 | 内容 |
|---|---|
| 95% 到達の連続性 | 2 週連続で週次作業完了率が 95% 以上であること |
| エラーレートの安定 | 直近 2 週間の週次エラーレート平均が 5% 未満であること |
| デバイス稼働率 | 直近 1 週間のデバイス稼働率が 95% 以上であること |

### 4-2. 並行運用終了条件の充足確認（DES-MIG-003 対応）

並行運用終了条件（DES-MIG-003）の充足確認は quality_admin が実施する。以下の 3 条件を全て充足した場合に並行運用終了の判定が可能となる。

| 終了条件 | 確認方法 | 充足基準 |
|---|---|---|
| 作業完了率 95% 到達（2 週連続） | 週次集計レポートで確認 | 2 週連続で 95% 以上 |
| 重大バグ未解決件数ゼロ | 課題管理システムで確認 | Critical/High バグが 0 件 |
| カットオーバーリハーサル完了 | リハーサル結果報告書で確認 | quality_admin 署名済み報告書が存在 |

---

**本節で確定した方針**
- 作業完了率 95% 到達は 2 週連続を確認条件とすることを確定する。
- DES-MIG-003 の 3 終了条件（完了率・重大バグゼロ・リハーサル完了）の全充足を並行運用終了の条件とすることを確定する。
- 終了条件確認は quality_admin が実施することを確定する。

## 5 並行運用終了宣言と移行計画/06 への接続

### 5-1. quality_admin による終了判定とカットオーバー実施日の決定

**MIG-X-190**: 並行運用終了条件（DES-MIG-003）の全充足確認後、quality_admin は以下の手順で並行運用終了宣言とカットオーバー日程の確定を実施する。

| ステップ | 手順 | 担当者 |
|---|---|---|
| 1 | 終了条件充足確認結果を「並行運用終了判定書」にまとめる | quality_admin |
| 2 | system_admin・現場監督と合同で終了判定会議を開催する | quality_admin |
| 3 | 終了判定書に 3 者の署名を取得する | quality_admin + system_admin + 現場監督 |
| 4 | カットオーバー実施予定日（週末または計画停止時）を決定する | quality_admin |
| 5 | カットオーバー実施日を `移行計画/06_カットオーバー実施計画.md` に追記する | quality_admin |
| 6 | `08_カットオーバー手順書.md` §2（T-72h 前通知）に移行する | quality_admin |

カットオーバー実施日の決定基準を以下に定める。

| 選択条件 | 優先度 |
|---|---|
| 週末（土曜日）の生産ライン停止時間 | 最優先 |
| 計画停止（定期メンテナンス）時間 | 次善 |
| 平日夜間（残業なし・最終作業完了後） | 最終手段 |

---

**本節で確定した方針**
- 並行運用終了宣言は quality_admin・system_admin・現場監督の 3 者署名付き終了判定書をもって確定することを確定する。
- カットオーバー実施日は週末または計画停止時間を優先して選定することを確定する。
- 終了宣言後は即時に `08_カットオーバー手順書.md` §2（T-72h 通知）へ移行することを確定する。
