## 2 ロールバック発動時の初動

### 2-1. 発動判断者と判断フロー

**MIG-REP-001**: ロールバックの発動判断は quality_admin が行う。`移行計画/07_ロールバック実施計画.md` §2 に定める発動基準に該当すると quality_admin が判断した時点で、以下の初動手順を即時実行する。

ロールバック発動の判断フローを以下に定める。

| ステップ | 判断内容 | 担当者 |
|---|---|---|
| 1 | system_admin から障害報告を受ける | quality_admin |
| 2 | 障害の状況（縮退レベル・継続時間・影響範囲）を system_admin に確認する | quality_admin |
| 3 | `移行計画/07_ロールバック実施計画.md` §2 の発動基準と照合する | quality_admin |
| 4 | 発動基準に該当する場合、quality_admin がロールバック宣言を口頭で行う | quality_admin |
| 5 | ロールバック宣言時刻を「ロールバック実施記録」に記録する | quality_admin |
| 6 | system_admin に RB-001〜005 の実施開始を指示する | quality_admin |

ロールバック宣言の記録は口頭宣言直後に電子記録に転記する。

### 2-2. 全作業員への即時連絡

**MIG-REP-002**: ロールバック宣言後、quality_admin は直ちに全作業員・現場監督に以下の内容を周知する。周知は口頭アナウンス（館内放送または朝礼）で実施し、ハンディ APP へのシステム通知は system_admin が停止する。

| 周知事項 | 内容 |
|---|---|
| 状況説明 | システムに問題が発生したため、紙運用に戻ることを告知 |
| 即時指示 | ハンディ APP の使用を直ちに停止すること |
| 代替手順 | 紙 SOP・記録用紙を使用して作業を継続すること |
| 次の連絡予定 | 復旧または再移行の予定を追って連絡することを通知 |

system_admin は周知完了後、全端末のシステム通知を停止し、バックエンドへのアクセス制限（メンテナンスモード）を有効にする。

---

**本節で確定した方針**
- ロールバック発動判断は quality_admin が唯一の権限者として実施することを確定する。
- 口頭アナウンスによる全作業員への即時周知をロールバック宣言直後の最初の行動とすることを確定する。
- system_admin によるシステム通知停止とメンテナンスモード有効化を周知完了後に実施することを確定する。

## 3 RB-001〜005 の実施手順

### 3-1. RB-001: 旧運用（紙）の即時再開宣言と紙 SOP の再配布

**MIG-REP-003**: RB-001 として、quality_admin は旧運用（紙記録）の即時再開を宣言し、現場監督とともに以下の手順で紙 SOP・記録用紙を再配布する。

| ステップ | 手順 | 担当者 |
|---|---|---|
| 1 | `08_カットオーバー手順書.md` §6-1 で施錠保管した紙書類保管棚を開錠する | quality_admin |
| 2 | 各工程の紙 SOP（最新版）を確認し、各工程の担当者に配布する | quality_admin + 現場監督 |
| 3 | 作業記録用紙を各工程の担当者に配布する | 現場監督 |
| 4 | 「システム停止・紙運用開始」の掲示を全工程エリアに掲示する | 現場監督 |
| 5 | 紙 SOP 再配布完了時刻を「ロールバック実施記録」に記録する | quality_admin |

### 3-2. RB-002: PostgreSQL バックアップからの復元実行

**MIG-REP-004**: RB-002 として、system_admin は以下のコマンドで PostgreSQL をカットオーバー直前のバックアップ（`08_カットオーバー手順書.md` §4-1 MIG-X-178 で取得）から復元する。

```bash
# バックアップファイルの確認
ls -lh /backup/cutover/
sha256sum /backup/cutover/<backup_file_name>
# 記録済みの SHA256 ハッシュ値と一致することを確認

# 復元先データベースの接続停止（アプリコンテナを停止）
docker compose stop backend

# PostgreSQL バックアップ復元（pg_restore）
docker exec <postgres_container_name> pg_restore \
  -U <db_user> \
  -d <db_name> \
  --clean \
  --if-exists \
  -F c \
  /tmp/<backup_file_name>

# バックアップファイルをコンテナ内にコピーしてから復元する場合
docker cp /backup/cutover/<backup_file_name> <postgres_container_name>:/tmp/
docker exec <postgres_container_name> pg_restore \
  -U <db_user> -d <db_name> --clean --if-exists -F c /tmp/<backup_file_name>
```

復元完了後の確認手順を以下に定める。

```bash
# 復元後のテーブル確認
docker exec <postgres_container_name> psql -U <db_user> -d <db_name> -c "\dt"

# レコード件数の確認（カットオーバー前の件数と一致することを確認）
docker exec <postgres_container_name> psql -U <db_user> -d <db_name> -c \
  "SELECT COUNT(*) FROM work_events; SELECT COUNT(*) FROM users; SELECT COUNT(*) FROM devices;"
```

### 3-3. RB-003: ハッシュチェーンの巻戻し確認

**MIG-REP-005**: RB-003 として、バックアップ復元後にハッシュチェーンの整合性バッチを実行し、復元されたデータのハッシュチェーン整合性を確認する。

```bash
# ハッシュチェーン整合性確認バッチの実行
docker exec <backend_container_name> /usr/local/bin/verify_hash_chain --full

# 整合性確認の結果を確認
# 期待される出力: "Hash chain verification completed: 0 errors found"
```

整合性バッチでエラーが検出された場合は、エラーレコードの特定と対応を quality_admin・system_admin で協議する。

### 3-4. RB-004: ハンディ端末の接続先リセット

**MIG-REP-006**: RB-004 として、system_admin は全ハンディ端末のバックエンド URL を空白（未設定）にリセットし、本番サーバーへの接続を切断する。

| 操作方法 | 手順 |
|---|---|
| MDM 管理デバイスの場合 | MDM コンソールから設定プロファイルを更新し、バックエンド URL を空白に設定する |
| 手動設定デバイスの場合 | 各端末のハンディ APP 設定画面からサーバー URL を削除する |

全端末のリセット完了後、ハンディ APP を起動して「サーバー未接続」状態であることを確認する。

### 3-5. RB-005: 全作業員への紙運用復帰アナウンス

**MIG-REP-007**: RB-005 として、quality_admin は以下の内容を含む紙運用復帰アナウンスを実施する。

| アナウンス内容 | 記載事項 |
|---|---|
| 状況説明 | システムの問題により紙運用に戻った旨 |
| 作業継続の指示 | 紙 SOP に従って作業を継続する旨 |
| 記録方法の確認 | 紙記録用紙への記録方法の再確認 |
| 今後の予定 | 原因調査と再移行計画を追って告知する旨 |

**MIG-REP-008**: アナウンス完了後、quality_admin は「ロールバック実施記録」のロールバック完了欄に完了時刻・RB-001〜005 の実施結果・担当者情報を記録する。

---

**本節で確定した方針**
- RB-001〜005 の実施順序（紙再開→DB復元→ハッシュ確認→端末リセット→アナウンス）を厳守することを確定する。
- バックアップ復元前の SHA256 ハッシュ値照合をデータ整合性確認の必須手順とすることを確定する。
- 全端末のバックエンド URL リセット完了を RB-004 の完了条件とすることを確定する。

## 4 ロールバック後処理

### 4-1. is_retroactive=true フラグの設定確認（MIG-056 対応）

**MIG-REP-009**: ロールバック完了後、カットオーバー後に入力されたシステムレコードに `is_retroactive=true` フラグが設定されていることを確認する。バックアップ復元により本フラグが存在するレコードは復元データから除外されているが、復元後に残存するレコードを以下のクエリで確認する。

```sql
-- is_retroactive=true のレコード確認
SELECT COUNT(*), is_retroactive
FROM work_events
GROUP BY is_retroactive;
```

`is_retroactive=true` のレコードが存在する場合は、quality_admin の判断でレコードの保持または削除を決定し、判断内容を「ロールバック実施記録」に記録する。

ロールバック後には以下の後処理を実施する。

| 後処理内容 | 担当者 | 期限 |
|---|---|---|
| ロールバック報告書の作成（原因・実施内容・再移行計画） | system_admin + quality_admin | ロールバック完了後 24 時間以内 |
| 課題管理システムへの障害起票 | system_admin | ロールバック完了後 24 時間以内 |
| 次回カットオーバー予定日の再設定 | quality_admin | ロールバック完了後 3 営業日以内 |
| 再移行に向けた原因対策計画の策定 | system_admin + quality_admin | ロールバック完了後 5 営業日以内 |

---

**本節で確定した方針**
- is_retroactive フラグの確認（MIG-056 対応）をロールバック後処理の最初のステップとすることを確定する。
- ロールバック報告書の作成をロールバック完了後 24 時間以内に実施することを確定する。
- 次回カットオーバー予定日の再設定をロールバック完了後 3 営業日以内に実施することを確定する。

## 5 ロールバック完了の電子署名

### 5-1. quality_admin の署名付き報告書の保存

**MIG-REP-010**: ロールバック完了後、quality_admin は「ロールバック完了報告書」を作成し、電子署名を付与して移行記録文書として保存する。

ロールバック完了報告書の必須記載事項を以下に定める。

| 記載区分 | 記載事項 |
|---|---|
| 基本情報 | ロールバック発動日時・完了日時・発動判断者（quality_admin 氏名） |
| 発動理由 | 検出した障害の種別・縮退レベル・継続時間 |
| 実施内容 | RB-001〜005 の実施時刻・実施結果（成功/失敗・特記事項） |
| バックアップ復元結果 | 復元対象バックアップファイル名・SHA256 ハッシュ値・復元確認結果 |
| ハッシュチェーン確認結果 | 整合性バッチの実行結果（エラー件数） |
| 影響範囲 | ロールバックにより失われたデータの件数・期間 |
| 再移行計画 | 原因対策の概要・次回カットオーバー予定日 |
| 電子署名 | quality_admin の電子署名・署名日時 |

ロールバック完了報告書は移行記録文書として 3 年間保存する。

---

**本節で確定した方針**
- quality_admin の電子署名付きロールバック完了報告書の作成をロールバック完了の証跡とすることを確定する。
- ロールバック完了報告書の 8 必須記載事項の全記入を完了条件とすることを確定する。
- ロールバック完了報告書を移行記録文書として 3 年間保存することを確定する。
