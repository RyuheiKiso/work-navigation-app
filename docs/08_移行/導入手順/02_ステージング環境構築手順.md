# 02 ステージング環境構築手順

本章の責務は、INST-A4-b（ステージング環境構築）に基づき、本番デプロイ前のリハーサル・最終検証を本番同等の構成で実行できるステージング環境を確立することである。ステージング環境は本番環境（01 章）と同等のハードウェア・ソフトウェア構成で構築し、本番への影響を一切発生させないデータ分離を保証する。

---

## 1 本章の責務と本番環境との差異

### 1-1. INST-A4-b との対応関係

本章は IPA 共通フレーム 2013「3.2 導入プロセス」タスク INST-A4-b に対応するステージング環境構築手順書である。ステージング環境の目的は、本番デプロイを実施する前に本番同等の環境で動作検証・リハーサルを行うことであり、ステージング環境での検証通過を本番デプロイの必要条件とする（06_実装/10_デプロイ手順.md §2 に準拠）。

**MIG-X-116**: ステージング環境は本番環境と同等のハードウェア仕様・OS バージョン・ミドルウェアバージョンで構築する。バージョン差異が存在する場合、差異を `staging_versions.lock` に明記してリスク評価を実施する。（DES-MIG-002 対応）

### 1-2. 本番環境との差異の明示

ステージング環境は本番環境と同等構成で構築することを確定するが、以下の点は意図した差異として許容する。

| 差異項目 | 本番環境 | ステージング環境 | 差異の理由 |
|---|---|---|---|
| SSL 証明書 | 認証局発行の正規証明書 | 自己署名証明書（openssl 生成） | コスト削減・リハーサル目的 |
| ドメイン名 | `wnav.example.com`（本番ドメイン） | `wnav-staging.internal`（内部名） | 本番との混同防止 |
| データ量 | 実稼働データ全件 | リハーサル用テストデータのみ | 本番データの漏洩防止 |
| DB スキーマ | `wnav_prod` | `wnav_staging` | 本番 DB への影響排除 |
| バックアップ設定 | 本番バックアップポリシー準拠 | バックアップなし（リハーサル目的） | 運用コスト削減 |
| 外部公開 | 社外からアクセス可（WiFi AP 経由） | 管理者ネットワークのみ | セキュリティリスク低減 |

---

**本節で確定した方針**
- ステージングは本番と同等のハードウェア・OS・ミドルウェア構成で構築することを確定する。
- 証明書・ドメイン名・データ量・DB スキーマの 4 点については意図した差異として許容することを確定する。
- ステージング検証通過を本番デプロイの必要条件とすることを確定する。

---

## 2 ステージング環境の前提

### 2-1. ハードウェア前提

**MIG-X-117**: ステージング環境のハードウェアは本番環境（01 章 §2-1）と同スペックの仮想マシンまたは物理マシンで構築する。（DES-MIG-002 対応）

| 構成要素 | 要件 | 準拠するドキュメント |
|---|---|---|
| CPU | 4 コア以上（本番と同スペック） | 01 章 §2-1 |
| RAM | 16 GB 以上（本番と同スペック） | 01 章 §2-1 |
| ストレージ | SSD 500 GB 以上（本番と同スペック） | 01 章 §2-1 |
| OS | Windows Server 2022（本番と同バージョン） | 01 章 §2-1 |
| NIC | 1 Gbps 以上（本番と同スペック） | 01 章 §2-1 |

仮想マシンを使用する場合（Hyper-V または VMware）は、本番サーバーとは別のホスト上に配置し、リソース競合がない状態を確保する。同一ホスト上への配置は対象外と判断する。

### 2-2. データ分離の保証

**MIG-X-116（続）**: ステージング環境のデータが本番環境のデータに影響を与えないよう、以下の分離措置を実施する。

| 分離レイヤ | 分離方法 | 確認手順 |
|---|---|---|
| ネットワーク分離 | ステージング用 VLAN を設定する（本番 VLAN とは独立） | VLAN ID が本番と異なることをスイッチ設定で確認 |
| DB 分離 | `wnav_staging` スキーマ（本番は `wnav_prod`）を使用する | `psql -c "\l"` でデータベース名が独立していることを確認 |
| 接続文字列分離 | `docker-compose.staging.yml` の `DATABASE_URL` を `wnav_staging` に向ける | 環境変数ファイルで設定値を確認 |
| 認証情報分離 | ステージング用の `DB_PASSWORD` は本番とは別の値を設定する | `docker compose config` で設定値を確認 |

リハーサル時に本番 DB への接続が発生しないことを、デプロイ前に接続文字列を目視確認する手順を手順書に含める。

---

**本節で確定した方針**
- ステージング環境は本番と同スペックの独立したマシンに構築することを確定する。
- DB スキーマは `wnav_staging` を使用し、本番 `wnav_prod` への接続は禁止することを確定する。
- ネットワーク分離は VLAN で実施し、ステージング環境からの本番環境への直接アクセスを対象外と判断する。

---

## 3 本番環境構築手順（01 章）との差分

### 3-1. 使用する Docker Compose ファイル

**MIG-X-118**: ステージング環境では `docker-compose.staging.yml` を使用する。本番向けの `docker-compose.prod.yml` との差分を以下に示す。（DES-MIG-003 対応）

`docker-compose.staging.yml` の主な差分設定を以下に示す。

バックエンドは2バイナリ構成であり、`terminal-api`（8080）と `master-api`（8081）を個別のサービスとして定義する。

```yaml
# docker-compose.staging.yml（本番との差分箇所のみ記載）
version: "3.9"

services:
  terminal-api:
    image: ghcr.io/ryuhei-kiso/wnav-terminal-api:staging
    environment:
      - WNAV_ENV=staging
      - DATABASE_URL=postgresql://wnav_staging_user:${STAGING_DB_PASSWORD}@postgres:5432/wnav_staging
      - WNAV_BASE_URL=https://wnav-staging.internal
    ports:
      - "8080:8080"

  master-api:
    image: ghcr.io/ryuhei-kiso/wnav-master-api:staging
    environment:
      - WNAV_ENV=staging
      - DATABASE_URL=postgresql://wnav_staging_user:${STAGING_DB_PASSWORD}@postgres:5432/wnav_staging
      - WNAV_BASE_URL=https://wnav-staging.internal
    ports:
      - "8081:8081"

  postgres:
    image: postgres:17-alpine
    environment:
      - POSTGRES_DB=wnav_staging
      - POSTGRES_USER=wnav_staging_user
      - POSTGRES_PASSWORD=${STAGING_DB_PASSWORD}
    volumes:
      - staging_pgdata:/var/lib/postgresql/data

volumes:
  staging_pgdata:
    name: wnav_staging_pgdata
```

起動コマンドは以下の通りとする。

```bash
# ステージング環境の起動
cd /opt/wnav
docker compose -f docker-compose.staging.yml up -d

# サービスの起動確認
docker compose -f docker-compose.staging.yml ps
```

### 3-2. 自己署名証明書の生成と使用

**MIG-X-119**: ステージング環境では OpenSSL で自己署名証明書を生成して使用する。本番環境の正規証明書とは別管理とする。（DES-MIG-003 対応）

WSL2 Ubuntu 内で以下を実行する。

```bash
# 自己署名証明書の生成（有効期限 365 日）
openssl req -x509 -nodes -days 365 -newkey rsa:2048 \
    -keyout /opt/wnav/certs/staging.key \
    -out /opt/wnav/certs/staging.crt \
    -subj "/C=JP/ST=Tokyo/O=WNAV-Staging/CN=wnav-staging.internal" \
    -addext "subjectAltName=DNS:wnav-staging.internal,IP:127.0.0.1"

# 証明書の内容を確認する
openssl x509 -text -noout -in /opt/wnav/certs/staging.crt | grep -E "Subject:|NotAfter:"
```

生成した自己署名証明書を Windows 証明書ストア（「信頼されたルート証明機関」）にインポートして、管理者 PC での HTTPS アクセス時に警告が出ないようにする。

```powershell
# 自己署名証明書を Windows 信頼済みルートに追加する
Import-Certificate `
    -FilePath "C:\wnav-install\certs\staging.crt" `
    -CertStoreLocation Cert:\LocalMachine\Root
```

IIS でのバインディング設定は 01 章 §5-2 の手順に準じて実施するが、ファイルパスを `staging.crt` / `staging.key` に変更する。

### 3-3. 本番 DB とは別スキーマの使用

**MIG-X-120**: ステージング環境では `wnav_staging` データベースを使用する。本番データベース `wnav_prod` とは完全に独立した PostgreSQL インスタンスを使用する。（DES-MIG-003 対応）

```bash
# ステージング PostgreSQL コンテナへの接続確認
docker compose -f docker-compose.staging.yml exec postgres \
    psql -U wnav_staging_user -d wnav_staging -c "SELECT current_database();"

# 期待する出力: wnav_staging
```

`docker-compose.staging.yml` で指定する環境変数ファイル（`.env.staging`）を以下のテンプレートで作成する。

```bash
# .env.staging テンプレート
# 本番環境の .env.prod とは別ファイルで管理する
STAGING_DB_PASSWORD=<ステージング専用パスワード（本番とは異なる値を設定する）>
WNAV_ENV=staging
WNAV_LOG_LEVEL=debug
WNAV_BASE_URL=https://wnav-staging.internal
```

`.env.staging` は Git にコミットしない（`.gitignore` で除外済み）。

---

**本節で確定した方針**
- ステージング環境の Docker Compose は `docker-compose.staging.yml` に独立させることを確定する。
- バックエンドは `terminal-api`（8080）と `master-api`（8081）の2サービスとして定義することを確定する。
- 自己署名証明書は OpenSSL で生成し、管理者 PC の信頼済みルートにインポートすることを確定する。
- ステージング DB 認証情報は本番とは異なる値を `.env.staging` で管理することを確定する。

## 4 ステージング環境の動作確認

### 4-1. 本番同等の E2E シナリオでの疎通確認

**MIG-X-121**: ステージング環境で以下の E2E 確認シナリオを全件実施する。全シナリオ通過を本番デプロイ許可の前提条件とする。（DES-MIG-004 対応）

| シナリオ番号 | シナリオ名 | 確認内容 | 合格基準 |
|---|---|---|---|
| STA-E2E-001 | terminal-api ヘルスチェック | `GET http://localhost:8080/healthz` | HTTP 200・`"status":"ok"` が返る |
| STA-E2E-002 | master-api ヘルスチェック | `GET http://localhost:8081/healthz` | HTTP 200・`"status":"ok"` が返る |
| STA-E2E-003 | マスタメンテ初期表示 | ブラウザで `https://wnav-staging.internal` を開く | index.html が表示・コンソールエラーなし |
| STA-E2E-004 | 管理者ログイン | quality_admin アカウントでログイン | JWT トークン取得・ダッシュボード表示 |
| STA-E2E-005 | マスタデータ参照 | 工程マスタ一覧を API で取得 | `GET /api/v1/master/processes` が HTTP 200 を返す（master-api 経由） |
| STA-E2E-006 | CSV インポート | テストデータ CSV を工程マスタへインポート | プレビュー表示・インポート成功・DB 反映確認 |
| STA-E2E-007 | ハンディ APP 接続 | ハンディ端末から terminal-api に疎通 | `GET /api/v1/auth/healthz` が HTTP 200 を返す（terminal-api 経由） |
| STA-E2E-008 | HTTPS 証明書 | ブラウザの証明書警告 | 管理者 PC では警告なし（信頼済みルート登録済み） |

確認結果を以下のテンプレートで記録する。

```
シナリオ番号: STA-E2E-NNN
実施日時:
実施者:
結果（合格 / 不合格）:
不合格の場合の事象詳細:
```

### 4-2. リハーサル用テストデータの投入確認

**MIG-X-122**: リハーサル用テストデータを本番投入手順と同一の手順（04〜05 章）で投入し、投入完了後の件数照合を実施する。（DES-MIG-004 対応）

| マスタ種別 | リハーサル用テストデータ件数 | 照合方法 |
|---|---|---|
| 工程マスタ（processes） | 5 件以上 | `SELECT COUNT(*) FROM processes;` |
| 作業マスタ（operations） | 10 件以上 | `SELECT COUNT(*) FROM operations;` |
| 製品マスタ（products） | 3 件以上 | `SELECT COUNT(*) FROM products;` |
| ユーザーマスタ（users） | 3 件以上（quality_admin 1 名含む） | `SELECT COUNT(*) FROM users;` |
| SOP マスタ（sops） | 3 件以上 | `SELECT COUNT(*) FROM sops;` |

テストデータは `docs/08_移行/付録/staging_testdata/` に格納する。本番データは一切使用しない。

```bash
# ステージング DB の件数照合コマンド
docker compose -f docker-compose.staging.yml exec postgres \
    psql -U wnav_staging_user -d wnav_staging -c "
      SELECT 'processes' AS table_name, COUNT(*) FROM processes
      UNION ALL SELECT 'operations', COUNT(*) FROM operations
      UNION ALL SELECT 'products', COUNT(*) FROM products
      UNION ALL SELECT 'users', COUNT(*) FROM users
      UNION ALL SELECT 'sops', COUNT(*) FROM sops;
    "
```

照合結果が期待件数と一致した時点でステージング環境の動作確認完了とし、本番デプロイ許可を発行する。

---

**本節で確定した方針**
- STA-E2E-001〜008 の全シナリオ通過を本番デプロイ許可の必要条件とすることを確定する。
- terminal-api（8080）と master-api（8081）の両ヘルスチェック通過を確認の前提条件とすることを確定する。
- リハーサル用テストデータは本番データを使用せず専用テストデータを準備することを確定する。
- テストデータの件数照合は SQL クエリで実施し、結果を記録に残すことを確定する。
