# 04 GUI 初期構築ウィザード操作手順

本章の責務は、INST-A4-d（GUI ウィザード起動・初期構築）に基づき、SCR-SET-001（GUI 初期構築ウィザード）を起動して初期接続設定・システムロール設定・マッピング設定・初期完了署名を実施することである。DES-MIG-021〜030 で確定したウィザードのステップ構成・バリデーション規則・ロールバック保証に従い、本番環境で 1 度のみ実行可能な初期設定処理を完了させる。

---

## 1 本章の責務と設計文書との対応

### 1-1. INST-A4-d との対応関係

本章は IPA 共通フレーム 2013「3.2 導入プロセス」タスク INST-A4-d に対応する GUI ウィザード操作手順書である。SCR-SET-001 ウィザードは 8 ステップの強制進行型であり（DES-MIG-021 参照）、本章は本番導入時の操作手順に必要な Step 1〜5 を対象とする。Step 6〜8（設備・SOP 投入）は 05 章で扱う。

| ウィザードステップ | 内容 | 本章での対応 |
|---|---|---|
| Step 1 | 接続設定（DB・バックエンド URL） | §3 Step 1 |
| Step 2 | 疎通確認（ヘルスチェック API） | §3 Step 2 |
| Step 3 | システムロールと quality_admin アカウント設定 | §3 Step 3 |
| Step 4 | マッピング設定（CSV 列 → システム列） | §3 Step 4 |
| Step 5 | 品質担当による承認と初期設定完了署名 | §3 Step 5 |

**MIG-X-129**: SCR-SET-001 ウィザードは初回セットアップ時のみ自動起動し、完了後は再表示されない。ウィザード完了状態は DB の `system_settings` テーブルの `wizard_completed` フラグで管理する。（DES-MIG-021 対応）

### 1-2. DES-MIG-021〜030 との準拠関係

| 設計 ID | 設計内容 | 本章での実施箇所 |
|---|---|---|
| DES-MIG-021 | ウィザード起動条件・ステップ構成 | §2・§3 |
| DES-MIG-022 | CSV/Excel テンプレートの仕様 | §3 Step 4 |
| DES-MIG-023 | エンティティ別テンプレート仕様 | §3 Step 4 |
| DES-MIG-024 | 投入前チェック（プレインポートバリデーション） | §3 Step 4 |
| DES-MIG-025 | プレビュー + 確認 + コミット 3 段階フロー | §3 Step 4 |
| DES-MIG-026 | トランザクション設計（ロールバック保証） | §3 Step 4〜5 |

---

**本節で確定した方針**
- SCR-SET-001 ウィザードの操作はステップ 1〜5 を本章の手順に従い順番に実施することを確定する。
- ウィザード完了後は `wizard_completed = TRUE` を DB で確認し、再実行不可の状態を記録することを確定する。
- 本章の手順は MIG-CK-073〜075 の完了により INST-A4-d の完了と判定することを確定する。

## 2 ウィザード起動前提条件

### 2-1. バックエンド（axum）起動確認

**MIG-X-130**: ウィザードを起動する前に、axum バックエンドが正常に起動していることを確認する。（DES-MIG-021 対応）

```bash
# バックエンドコンテナの起動状態を確認する
docker compose -f docker-compose.prod.yml ps backend

# ヘルスチェックエンドポイントの応答を確認する
curl -sf http://localhost:8080/healthz | python3 -m json.tool

# 期待するレスポンス
# {
#   "status": "ok",
#   "db": "connected",
#   "version": "<バージョン>"
# }
```

ヘルスチェックが `"status":"ok"` かつ `"db":"connected"` を返さない場合は、バックエンドの起動を待機するか `docker compose logs backend` でエラーを確認する。

### 2-2. マスタメンテ（React）の URL アクセス確認

**MIG-CK-073**: マスタメンテ SPA に正常にアクセスできることをウィザード起動前に確認する。

| 確認項目 | 確認手順 | 合格基準 |
|---|---|---|
| URL アクセス | 管理者 PC でブラウザを開き `https://<サーバーIP>/` を入力 | HTTP 200・index.html が表示 |
| SPA ルーティング | `https://<サーバーIP>/setup` を直接入力 | ウィザード画面が表示（404 にならない） |
| コンソールエラー | ブラウザの開発者ツールを開く | JavaScript エラーが存在しない |
| バックエンド接続 | ウィザード画面の接続テストボタンを押す | 「接続成功」のメッセージが表示 |

### 2-3. quality_admin ロールでの初回ログイン確認

ウィザードの Step 3 で quality_admin アカウントを設定するため、ウィザード起動時点では初期管理者アカウント（`system_admin`）でログインする。

```
初回ログイン情報（デフォルト値 - ウィザード内で変更必須）
ユーザー名: system_admin
パスワード: ChangeMe_OnFirstLogin!
```

初回ログイン後、ウィザードは自動的に Step 1 を表示する。デフォルトパスワードは Step 3 で必ず変更する。

---

**本節で確定した方針**
- ウィザード起動前にバックエンドのヘルスチェック `"db":"connected"` 確認を必須とすることを確定する。
- MIG-CK-073 の 4 項目全通過をウィザード起動の許可条件とすることを確定する。
- 初回ログインのデフォルトパスワードは Step 3 で変更することを確定する。

---

## 3 ウィザード Step 1〜5 の操作手順

### 3-1. Step 1: 接続設定の入力と保存

**MIG-X-131**: Step 1 では DB 接続 URL とバックエンド API の URL を入力して保存する。（DES-MIG-025 対応）

Step 1 の画面で以下の項目を入力する。

| 入力項目 | 入力値 | 備考 |
|---|---|---|
| データベース URL | `postgresql://wnav_admin:<パスワード>@localhost:5432/wnav_prod` | `.env.prod` の `DATABASE_URL` と一致させる |
| バックエンド API URL | `http://localhost:8080` | IIS リバースプロキシ経由の場合は `https://<サーバーIP>` |
| タイムゾーン | `Asia/Tokyo` | デフォルト値を確認して変更が必要な場合のみ修正する |
| システム名称 | `作業ナビゲーション・トレサビシステム` | ヘッダーに表示される名称 |

入力完了後「保存して次へ」ボタンを押す。Step 2 に自動遷移することを確認する。

### 3-2. Step 2: 疎通確認

**MIG-X-132**: Step 2 では Step 1 で入力した接続先が実際に応答することを確認する。（DES-MIG-025 対応）

Step 2 の画面で「疎通確認を実行」ボタンを押す。ウィザードは内部で以下の 3 点を確認する。

| 確認項目 | 内部処理 | 合格基準 |
|---|---|---|
| DB 疎通確認 | `SELECT 1` を PostgreSQL に送信 | クエリが正常に返る |
| バックエンド API 疎通確認 | `GET /api/healthz` を実行 | HTTP 200・`"status":"ok"` が返る |
| DB バージョン確認 | PostgreSQL バージョンを取得 | PostgreSQL 17.x 以上であることを確認 |

全 3 項目の確認が成功した場合のみ「確認完了・次へ」ボタンが有効化される。失敗した場合は Step 1 に戻って接続設定を修正する。

### 3-3. Step 3: システムロールと quality_admin アカウントの設定

**MIG-X-133**: Step 3 では初期 quality_admin アカウントを作成し、デフォルトパスワードを変更する。（DES-MIG-025 対応）

Step 3 の画面で以下を実施する。

| 操作 | 入力内容 | 制約 |
|---|---|---|
| quality_admin ユーザー名の設定 | 担当者のログイン ID（例: `qadmin01`） | 英数字のみ・最大 50 文字 |
| quality_admin 表示名の設定 | 担当者の氏名（例: `品質管理 担当者名`） | 最大 100 文字 |
| quality_admin パスワードの設定 | 新規パスワード | 12 文字以上・英大文字・小文字・数字・記号を含む |
| system_admin パスワードの変更 | 新規パスワード（デフォルトから変更） | 12 文字以上・同上 |

パスワードの設定完了後、ウィザードはシステムロール 6 種（operator / line_leader / master_admin / quality_admin / system_admin / viewer）が DB に存在することを確認する。03 章のマイグレーションが正常に完了していれば、システムロールは既に DB に存在する。

### 3-4. Step 4: マッピング設定の確認

**MIG-X-134**: Step 4 では CSV 列名とシステム内部のフィールド名のマッピング設定を確認・調整する。（DES-MIG-023 対応）

Step 4 の画面には以下のエンティティのマッピング設定が表示される。

| エンティティ | CSV 列名（デフォルト） | システムフィールド名 | 変更の必要性 |
|---|---|---|---|
| 工程マスタ | `工程番号` | `process_no` | 旧システムの列名と一致しない場合のみ変更 |
| 工程マスタ | `工程名` | `process_name` | 同上 |
| 作業マスタ | `作業番号` | `operation_no` | 同上 |
| 作業マスタ | `作業名` | `operation_name` | 同上 |
| ユーザーマスタ | `ログインID` | `username` | 同上 |
| ユーザーマスタ | `表示名` | `display_name` | 同上 |

旧システムからの CSV エクスポートで列名が異なる場合のみマッピングを変更する。デフォルト設定が旧システムの列名と一致する場合は変更なしで次へ進む。

変更後は「マッピング設定を保存」ボタンを押す。

### 3-5. Step 5: 品質担当による承認と初期設定完了署名

**MIG-X-134（続）**: Step 5 では quality_admin が初期設定内容を確認し、電子署名を付与する。（DES-MIG-026 対応）

**MIG-CK-074**: quality_admin が Step 1〜4 の設定内容を一覧で確認し、内容に誤りがないことを確認する。

Step 5 の画面で以下の確認を実施する。

| 確認項目 | 確認方法 |
|---|---|
| 接続設定（DB URL・API URL）の正確性 | 画面表示の設定値を目視で確認する |
| quality_admin アカウントの作成確認 | ログアウトして quality_admin でログインし直す |
| system_admin パスワード変更の確認 | デフォルトパスワードで system_admin がログインできないことを確認する |
| マッピング設定の正確性 | 旧システム CSV の列名と設定されたマッピングが一致していることを確認する |

**MIG-CK-075**: 確認完了後、quality_admin が電子署名を付与する。

```
電子署名入力項目:
- 署名者ユーザー名: <quality_admin のユーザー名>
- パスワード再確認: <quality_admin のパスワード>
- 署名理由: 「初期設定完了を確認し承認する」
- 署名日時: <自動入力>
```

「承認して完了」ボタンを押すと、ウィザードは以下の処理を DB に記録して完了状態に移行する。

- `system_settings` の `wizard_completed = TRUE` をセット
- `system_settings` の `wizard_completed_at = <現在日時>` をセット
- `system_settings` の `wizard_completed_by = <quality_admin ユーザー名>` をセット
- `audit_logs` にウィザード完了イベントを記録

---

**本節で確定した方針**
- Step 1〜4 の内容を Step 5 で quality_admin が目視確認してから電子署名を付与することを確定する。
- MIG-CK-074（確認）と MIG-CK-075（署名）の両方が完了した時点で INST-A4-d の Step 完了と判定することを確定する。
- quality_admin のパスワード再確認を電子署名の必要条件とすることを確定する。

---

## 4 ウィザード完了後の確認

### 4-1. ウィザード完了状態の確認

**MIG-X-135**: ウィザード完了後に、再実行不可のロック状態が DB に記録されていることを確認する。（DES-MIG-021 対応）

```bash
# wizard_completed フラグを DB で確認する
docker compose -f docker-compose.prod.yml exec postgres \
    psql -U wnav_admin -d wnav_prod -c "
      SELECT key, value
      FROM system_settings
      WHERE key IN ('wizard_completed', 'wizard_completed_at', 'wizard_completed_by');
    "
```

期待する出力を確認する。

| key | value |
|---|---|
| wizard_completed | TRUE |
| wizard_completed_at | `<完了日時>` |
| wizard_completed_by | `<quality_admin ユーザー名>` |

ブラウザで `https://<サーバーIP>/setup` に再アクセスして、「セットアップ済み」のメッセージが表示され、ウィザード画面が再表示されないことを確認する。

```

===============================================================
システム名    : 作業ナビゲーション・トレサビシステム

2. ウィザード実施結果
```

---

**本節で確定した方針**
- `wizard_completed = TRUE` の DB 確認をウィザード完了の必要条件とすることを確定する。
- ウィザード完了記録テンプレートへの quality_admin 署名をもって INST-A4-d の完了と判定することを確定する。
- `/setup` の再アクセス制限確認をウィザード完了確認の必須手順とすることを確定する。
