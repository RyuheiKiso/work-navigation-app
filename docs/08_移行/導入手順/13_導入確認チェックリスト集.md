# 13 導入確認チェックリスト集

本章の責務は、INST-A5-a（導入確認）に対応し、`移行計画/06_カットオーバー実施計画.md` および `08_カットオーバー手順書.md` で参照されるカットオーバーチェックリスト（CL-001〜010）と投入後確認チェック（POST-001〜005）を集約し、実施者・確認方法・合否基準を一箇所に確定することである。

### 1-1. INST-A5-a との対応関係

## 2 カットオーバーチェックリスト（CL-001〜010）

### 2-1. チェックリスト詳細定義

**MIG-X-196**: カットオーバーチェックリスト（CL-001〜010）の詳細定義を以下に定める。本チェックリストは `08_カットオーバー手順書.md` §5 の T-30 分 Go/No-Go 判定で使用する唯一の確認基準である。

**MIG-X-197**: 各チェック項目の確認担当者・確認方法・合否基準を以下に確定する。

**CL-001: 全マスタ投入完了確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | 全マスタデータ（ユーザー・デバイス・SOP・工程・製品）の投入が完了していること |
| 確認担当者 | quality_admin |
| 確認方法 | POST-001（件数照合）の確認結果記録を参照する |
| 合否基準 | DB 件数 = 投入ファイル件数（全マスタ区分で一致） |

**CL-002: 全デバイス登録確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | 全ハンディ端末がデバイス管理画面に登録済みであること |
| 確認担当者 | system_admin |
| 確認方法 | マスタメンテ管理画面「デバイス管理」で登録台数と実台数を照合する |
| 合否基準 | 登録台数 = 実台数（OS 別に Android・iOS・Windows で全台一致） |

**CL-003: 全ユーザーアカウント作成確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | 全作業員のユーザーアカウントが作成済みで初回ログインが完了していること |
| 確認担当者 | system_admin |
| 確認方法 | `SELECT COUNT(*) FROM users WHERE last_login_at IS NULL` を実行してゼロを確認する |
| 合否基準 | last_login_at が NULL のユーザーが 0 件であること |

**CL-004: 訓練完了確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | 全作業員の操作訓練が完了していること |
| 確認担当者 | quality_admin |
| 確認方法 | 訓練完了記録台帳（署名台帳）で全作業員の署名を確認する |
| 合否基準 | 全作業員の訓練完了署名が訓練完了記録台帳に存在すること |

**CL-005: ステージング環境でのテスト作業完了**

| 項目 | 内容 |
|---|---|
| チェック内容 | ステージング環境でのテスト作業が完了し、証跡記録が正常に生成されること |
| 確認担当者 | system_admin + quality_admin |
| 確認方法 | ステージング環境の `work_events` テーブルにテスト作業のレコードが存在することを確認する |
| 合否基準 | テスト作業に対応する `work_events` レコードが全ステップ分存在すること |

**CL-006: ハッシュチェーン初期ブロック生成確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | `audit_logs` テーブルの初期ブロック（block_number=1）が生成済みであること |
| 確認担当者 | system_admin |
| 確認方法 | `08_カットオーバー手順書.md` §4-2 MIG-X-179 の確認結果記録を参照する |
| 合否基準 | block_number=1・previous_hash=64桁ゼロのレコードが存在すること |

**CL-007: カットオーバー直前バックアップ完了確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | カットオーバー直前の PostgreSQL フルバックアップが完了していること |
| 確認担当者 | system_admin |
| 確認方法 | `08_カットオーバー手順書.md` §4-1 MIG-X-178 の確認結果記録（ハッシュ値付き）を参照する |
| 合否基準 | バックアップファイルの SHA256 ハッシュ値が記録済みであること |

**CL-008: 旧 Excel ファイルの読取専用設定確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | 旧運用で使用した Excel ファイル全件が読取専用に設定されていること |
| 確認担当者 | master_admin |
| 確認方法 | 対象 Excel ファイルのプロパティ画面で「読取専用」チェックボックスが有効であることを確認する |
| 合否基準 | 全対象 Excel ファイルが読取専用設定済みであること |

**CL-009: 3 者合意の書面取得確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | quality_admin・現場監督・system_admin の 3 者全員の合意署名が取得済みであること |
| 確認担当者 | quality_admin |
| 確認方法 | `06_本番リリース手順.md` §2 のリリース判定書の 3 者電子署名を確認する |
| 合否基準 | 3 者全員の電子署名が存在すること |

**CL-010: 外部ステークホルダーへの通知完了確認**

| 項目 | 内容 |
|---|---|
| チェック内容 | 必要な場合に外部ステークホルダーへの通知が完了していること |
| 確認担当者 | quality_admin |
| 確認方法 | 外部通知記録（メール送信ログ・通知書類の控え）を確認する。対象外の場合は「対象外」を明記する |
| 合否基準 | 通知完了記録または「対象外」の明示記録が存在すること |

---

**本節で確定した方針**
- CL-001〜010 の全合格を Go 宣言の前提条件とすることを確定する。
- 各チェック項目の確認担当者・確認方法・合否基準の変更は本章の改訂によってのみ可能とすることを確定する。
- CL-001〜010 の確認結果を `カットオーバー実施記録（MIG-X-185）` に記録することを確定する。

## 3 投入後確認チェック（POST-001〜005）

### 3-1. マスタデータ投入後の品質確認チェック定義

**MIG-X-198**: 投入後確認チェック（POST-001〜005）はマスタデータ投入完了後に実施し、結果は CL-001 の確認根拠として記録する。

**MIG-X-199**: POST-001〜005 の詳細定義を以下に確定する。

**POST-001: 全マスタ件数照合（投入ファイルの件数 vs DB の件数）**

| 項目 | 内容 |
|---|---|
| 確認内容 | 投入対象の全マスタ区分で、投入ファイルのレコード件数と DB のレコード件数が一致すること |
| 確認担当者 | system_admin |
| 確認方法 | 各マスタ区分の投入ファイル件数（CSV 行数）と DB の COUNT(*) を照合する |
| 合否基準 | 全マスタ区分で ファイル件数 = DB 件数（差異ゼロ） |
| 確認クエリ例 | `SELECT 'users' AS table_name, COUNT(*) FROM users UNION ALL SELECT 'devices', COUNT(*) FROM devices UNION ALL SELECT 'sops', COUNT(*) FROM sops;` |

**POST-002: 外部キー整合性確認（FK 違反ゼロ）**

| 項目 | 内容 |
|---|---|
| 確認内容 | 全テーブルの外部キー参照に違反するレコードが存在しないこと |
| 確認担当者 | system_admin |
| 確認方法 | PostgreSQL の外部キー制約を一時的に検証モードで実行する |
| 合否基準 | FK 違反件数が 0 件であること |
| 確認クエリ例 | `SELECT conname, conrelid::regclass, confrelid::regclass FROM pg_constraint WHERE contype = 'f' AND NOT (SELECT true FROM pg_trigger WHERE tgconstraint = pg_constraint.oid LIMIT 1);` |

**POST-003: 一意性制約確認（重複データゼロ）**

| 項目 | 内容 |
|---|---|
| 確認内容 | ユニーク制約が設定されたカラムに重複データが存在しないこと |
| 確認担当者 | system_admin |
| 確認方法 | ユニーク制約カラムで重複カウントが 1 を超えるレコードを検索する |
| 合否基準 | 重複レコード件数が 0 件であること |
| 確認クエリ例 | `SELECT employee_id, COUNT(*) FROM users GROUP BY employee_id HAVING COUNT(*) > 1;` |

**POST-004: 必須フィールド NULL 確認（NULL ゼロ）**

| 項目 | 内容 |
|---|---|
| 確認内容 | NOT NULL 制約が設定された全カラムに NULL 値が存在しないこと |
| 確認担当者 | system_admin |
| 確認方法 | 各テーブルの NOT NULL カラムに対して NULL チェッククエリを実行する |
| 合否基準 | NULL 件数が 0 件であること |
| 確認クエリ例 | `SELECT COUNT(*) FROM users WHERE name IS NULL OR email IS NULL OR role IS NULL;` |

**POST-005: 10% ランダムサンプリング目視確認**

| 項目 | 内容 |
|---|---|
| 確認内容 | 投入データの 10% をランダムサンプリングして目視確認し、データの正確性を確認すること |
| 確認担当者 | quality_admin |
| 確認方法 | `SELECT * FROM <table> ORDER BY RANDOM() LIMIT <10% 件数>` で抽出し、元データ（紙書類・Excel）と照合する |
| 合否基準 | サンプリングした全レコードで元データとの内容一致が確認されること |

---

**本節で確定した方針**
- POST-001〜005 の全合格を CL-001（全マスタ投入完了確認）の確認根拠とすることを確定する。
- POST-005（10% サンプリング）は quality_admin が元データと照合することを確定する。
- POST-001〜005 の確認結果を投入後確認チェック記録テンプレート（§4 参照）に記録することを確定する。

## 4 導入確認チェックリスト記録テンプレート

### 4-1. 記録テンプレートの構成

**MIG-X-200**: 導入確認チェックリスト記録テンプレートは以下の構成で管理し、全チェック完了後に quality_admin の電子署名を付与して移行記録文書として保存する。

| 記録区分 | 記録項目 |
|---|---|
| 基本情報 | 確認実施日時・確認場所・対象システムバージョン |
| POST 確認記録 | POST-001〜005 の各確認日時・確認者・確認方法の実施結果・合格/不合格・備考 |
| CL 確認記録 | CL-001〜010 の各確認日時・確認者・確認結果（合格/不合格）・備考 |
| 不合格項目の対処 | 不合格項目番号・不合格理由・対処内容・再確認日時・再確認結果 |
| 最終判定 | 全項目合格の判定日時・quality_admin の電子署名 |

記録テンプレートの各行の形式を以下に定める。

| 確認日時 | 確認者 | 対象 | 結果（合格/不合格） | 備考 |
|---|---|---|---|---|
| YYYY-MM-DD HH:MM | （担当者氏名） | （チェック番号・名称） | 合格/不合格 | （特記事項） |

---

**本節で確定した方針**
- 記録テンプレートへの全項目記入と quality_admin 電子署名を記録完了の必須要件とすることを確定する。
- 不合格項目の再確認記録を不合格発生時の必須記録とすることを確定する。
- 導入確認チェックリスト記録を移行記録文書として 3 年間保存することを確定する。
