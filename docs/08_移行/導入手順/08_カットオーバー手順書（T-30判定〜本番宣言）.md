## 4 T-1h 直前準備

### 4-1. PostgreSQL フルバックアップ取得（MIG-T-016 対応）

**MIG-X-178**: カットオーバー開始予定時刻の 1 時間前に、PostgreSQL の フルバックアップを以下の手順で取得する。このバックアップはロールバック時（11 章参照）の復元ポイントとなるため、確実に取得する。

```bash
# PostgreSQL フルバックアップの実行（Docker 環境）
BACKUP_FILE="backup_cutover_$(date +%Y%m%d_%H%M%S).dump"
docker exec <postgres_container_name> pg_dump \
  -U <db_user> \
  -d <db_name> \
  -F c \
  -f "/tmp/${BACKUP_FILE}"

# バックアップファイルをホストにコピー
docker cp <postgres_container_name>:/tmp/${BACKUP_FILE} /backup/cutover/${BACKUP_FILE}

# バックアップファイルのサイズ確認
ls -lh /backup/cutover/${BACKUP_FILE}

# バックアップファイルの整合性確認（リストアテスト）
pg_restore --list /backup/cutover/${BACKUP_FILE} | head -20
```

バックアップ取得完了後、以下を確認して記録する。

| 確認事項 | 確認方法 |
|---|---|
| バックアップファイルが生成されていること | `ls -lh` でファイル存在確認 |
| ファイルサイズが異常に小さくないこと | ファイルサイズで確認 |
| `pg_restore --list` が正常に実行できること | コマンド正常終了を確認 |
| バックアップファイルの SHA256 ハッシュ値 | `sha256sum /backup/cutover/${BACKUP_FILE}` で記録 |

### 4-2. ハッシュチェーン初期ブロック生成（MIG-T-017 対応）

**MIG-X-179**: カットオーバー直前のバックアップ完了後、`audit_logs` テーブルのハッシュチェーン初期ブロック生成バッチを実行し、生成結果を確認する。

```bash
# ハッシュチェーン初期ブロック生成バッチの実行
docker exec terminal-api /usr/local/bin/init_hash_chain

# 初期ブロック生成確認（PostgreSQL クエリ）
docker exec <postgres_container_name> psql -U <db_user> -d <db_name> -c \
  "SELECT id, block_number, previous_hash, current_hash, created_at FROM audit_logs ORDER BY block_number ASC LIMIT 1;"
```

確認結果の判定基準を以下に定める。

| 確認事項 | 判定基準 |
|---|---|
| `block_number = 1` のレコードが存在すること | SELECT 結果に 1 行存在 |
| `previous_hash` が 64 桁のゼロであること | `0000...0000`（64 桁）であること |
| `current_hash` が空でないこと | NULL または空文字でないこと |
| `created_at` が現在時刻の近傍であること | ±5 分以内であること |

---

**本節で確定した方針**
- PostgreSQL フルバックアップ取得（MIG-T-016）をカットオーバー開始前の必須手順とすることを確定する。
- ハッシュチェーン初期ブロック生成確認（MIG-T-017）をバックアップ完了後の必須手順とすることを確定する。
- バックアップファイルの SHA256 ハッシュ値記録を必須とすることを確定する。
