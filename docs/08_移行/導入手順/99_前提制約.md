# 99 前提制約

本章の責務は、導入手順サブ（導入手順/00〜13）の全体に適用される対象外事項（D-MIG-X-011〜015）と前提制約（CON-MIG-X-011〜015）を一箇所に集約し、各手順書が依拠する前提条件とスコープ境界を確定することである。

---

## 1 本章の責務

### 1-1. 対象外宣言と前提制約の位置づけ

本章は導入手順サブ全体のスコープ境界を明示する文書である。本章に定める対象外事項は、導入手順サブのいずれの手順書においても実施義務を持たず、本章の変更なしに個別手順書がスコープに取り込む禁止する。本章に定める前提制約は、導入手順サブ全体が依拠する前提条件であり、前提が満たされない場合は導入手順の開始を見合わせる。

---

**本節で確定した方針**
- 本章は導入手順サブのスコープ境界の唯一の権威文書として管理することを確定する。
- 対象外事項（D-MIG-X-011〜015）を個別手順書が独自に覆すことを禁止することを確定する。
- 前提制約（CON-MIG-X-011〜015）未充足の場合は導入手順の開始を中止することを確定する。

---

## 2 対象外宣言（D-MIG-X-011〜015）

### 2-1. 導入手順サブのスコープ外事項

**D-MIG-X-011**: OS バージョン差異の自動吸収機構の提供は対象外と判断する。ハンディ APP の OS 固有の事象（Android バージョン差異によるレンダリング差異・iOS の API 変更・Windows の .NET バージョン差異等）への対応は、各 OS の手順書（07a・07b・07c）が個別に記載する範囲に限定する。OS バージョン差異を自動検出して動作を切り替えるランタイム機構の実装は初回導入スコープの対象外とする。

**D-MIG-X-012**: ハードウェア故障時の現地交換手順は対象外と判断する。ハンディ端末・サーバー・UPS・スイッチング HUB 等のハードウェアが故障した場合の現地交換・保守は、工場設備管理の責任範囲であり、本システムの導入手順の対象外とする。ハードウェア故障への対応は工場の設備管理部門が定める保守手順書に従う。

**D-MIG-X-013**: セキュリティ侵害時の緊急対応手順は対象外と判断する。不正アクセス・マルウェア感染・情報漏洩等のセキュリティインシデント発生時の緊急対応は、`09_運用・保守/障害対応手順.md` が権威文書であり、本導入手順サブの対象外とする。導入手順中にセキュリティインシデントが発生した場合は即時に `09_運用・保守/障害対応手順.md` の手順に従う。

**D-MIG-X-014**: 本番環境の自動スケーリング設定は対象外と判断する。本システムは単一の Windows Server 2022 上に IIS + WSL2 + Docker Compose で配置する設計を採用しており、水平スケーリング（複数サーバーへの負荷分散）・垂直スケーリング（リソースの動的拡張）の自動設定は初回導入のスコープ外とする。将来のスケールアップ・スケールアウトは詳細設計フェーズの改訂候補として記録する。

**D-MIG-X-015**: ゼロダウンタイムデプロイ（Blue-Green デプロイ）の実装は対象外と判断する。本番初回導入では計画停止時間中のカットオーバーを採用し（DES-MIG-052 対応）、Blue-Green デプロイや Rolling Update による無停止切替の実装は初回導入のスコープ外とする。継続的デプロイでのゼロダウンタイム実現は `06_実装/10_デプロイ手順.md` の改訂として将来対応する。

---

**本節で確定した方針**
- D-MIG-X-011〜015 の 5 事項を導入手順サブのスコープ外と判断することを確定する。
- スコープ外事項への対応が必要になった場合は、本章の改訂と上位設計文書の変更を経て追加することを確定する。
- D-MIG-X-013（セキュリティ侵害）は `09_運用・保守/障害対応手順.md` を権威文書とすることを確定する。

---

## 3 前提制約（CON-MIG-X-011〜015）

### 3-1. 導入手順開始の前提条件

**CON-MIG-X-011**: 本番マシンの搬入・設置完了を移行計画開始前提とする。本番サーバー（Windows Server 2022 を実行するサーバー機）および必要な周辺機器（UPS・スイッチング HUB・ラック等）の物理的な搬入・設置・電源投入が完了していることを、導入手順 01 章（本番環境構築手順）の開始前提とする。搬入・設置が未完了の状態では導入手順を開始しない。

**CON-MIG-X-012**: 電源工事・LAN ケーブル敷設の完了を前提とする。本番サーバーおよび全ハンディ端末が必要とする電源コンセント・無停電電源装置への接続、工場内 LAN への有線および無線（WiFi）接続のための物理インフラ（電源工事・LAN ケーブル敷設・WiFi AP 設置）が完了していることを前提とする。物理インフラが未完了の状態では導入手順を開始しない。

| 物理インフラ要件 | 確認方法 | 確認担当者 |
|---|---|---|
| サーバー電源（100V/200V）の確認 | テスターによる電圧確認 | 工場設備管理担当 |
| UPS への接続確認 | UPS ステータスランプの確認 | 工場設備管理担当 |
| 工場内 LAN の疎通確認 | ping による疎通確認 | system_admin |
| WiFi AP の電波強度確認 | WiFi アナライザーで -70 dBm 以上 | system_admin |

**CON-MIG-X-013**: IT 担当者（system_admin）が本番環境への物理アクセス権を保有することを前提とする。system_admin は本番サーバーが設置されているサーバールームまたは設備室への入室権限（鍵・カードキー・入室許可）を保有していることを前提とする。物理アクセス権がない場合は工場の設備管理部門と調整して取得した上で導入手順を開始する。

**CON-MIG-X-014**: Apple 開発者アカウント（有料）が有効であることを iOS 配布の前提とする。`07b_iOS配布手順（App_Store_EAS）.md` の実施にあたり、組織の Apple Developer Program 有料メンバーシップ（年間 $99 USD）が有効であることを前提とする。メンバーシップが失効している場合は 07b 章の手順を開始しない。メンバーシップの更新は iOS 配布の 30 日前までに完了させる。

**CON-MIG-X-015**: EAS Build のクレジットが十分であることを確認することを前提とする。EAS Build（Expo Application Services）のクレジット（ビルド分数・月次クォータ）が、Android・iOS・Windows の 3 OS のプロダクションビルドを実行するために十分であることを確認することを前提とする。クレジット不足が見込まれる場合は Expo の有料プランへのアップグレードを実施してから 07 章の手順を開始する。

| EAS Build 確認方法 | コマンド |
|---|---|
| 現在のクレジット残高確認 | `eas account:view` |
| 月次ビルド使用量の確認 | Expo ダッシュボードの「Usage」画面 |

---

**本節で確定した方針**
- CON-MIG-X-011〜015 の 5 前提が全て充足されていることを導入手順開始の必須条件とすることを確定する。
- 前提未充足を発見した場合は即時に quality_admin に報告し、充足まで該当手順を保留することを確定する。
- CON-MIG-X-014（Apple 開発者アカウント）は iOS 配布の 30 日前までの有効性確認を必須とすることを確定する。
