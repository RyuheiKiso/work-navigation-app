# 04 事後レビュー報告書テンプレ

本章の責務は、IPA 共通フレーム 2013「6.5.5 文書化プロセス」および「INST-A5-d 90 日事後レビュー」に準拠した事後レビュー報告書の標準テンプレートを提供することである。本テンプレートは移行完了宣言後 90 日以内に quality_admin と system_admin が共同で実施する事後レビューに使用する。

---

## §1 本章の責務

本章の責務は、IPA 6.5.5 文書化プロセスおよび INST-A5-d 90 日事後レビューのテンプレートを提供することである。

事後レビューは、移行プロセス全体の評価・問題是正処置の完結確認・次回移行または ver2.0 向けの教訓抽出を目的とする。本テンプレートへの記入は義務であり、記入なしに移行プロセスを完結とは判断しない。

本章で提供するテンプレートは §3 に示す。§2 では実施要件・実施タイミング・保存義務を確定する。§4 では記録の保存手順を確定する。

---

**本節で確定した方針**
- IPA 6.5.5 + INST-A5-d に準拠した事後レビューテンプレートを本章で提供することを確定する。
- テンプレートへの記入なしに移行プロセスの完結を認めないことを確定する。
- 本テンプレートは quality_admin と system_admin の共同実施とすることを確定する。
---

## §2 事後レビューの実施要件

### 実施タイミング

事後レビューは移行完了宣言後 90 日以内に実施する。実施期限は「移行完了宣言日 + 90 日」とする。期限を超過した場合は PROB-MIG-NNN として問題点票を発行し、quality_admin が超過理由を記録する（MIG-068 対応）。

### 実施者

事後レビューの実施者は quality_admin と system_admin の 2 名とする。現場監督は必要に応じて参加する。外部関係者の参加が必要な場合は quality_admin が招集する。

実施者の役割分担を以下に示す。

| 役割 | 担当範囲 |
|---|---|
| quality_admin | 品質基準達成状況の評価・是正処置完結確認・教訓の品質 KPI 視点での整理 |
| system_admin | 技術的問題の評価・ロールバック手順の有効性評価・教訓の技術視点での整理 |
| 現場監督 | 作業員の訓練効果評価・運用上の問題点の報告 |

### 記録保存

事後レビュー報告書の記録保存期間は 7 年以上とする。保存は GitHub Issue（電子保存）と `evidence_files` テーブル（データベース保存）のダブル管理で実施する（§4 参照）。

保存形式は Markdown（本テンプレートに記入した .md ファイル）および PDF 変換版の双方を保存する。PDF 変換は GitHub Actions による自動生成を原則とする。

---

**本節で確定した方針**
- 実施タイミングは移行完了宣言後 90 日以内と確定する。
- 実施者は quality_admin + system_admin の 2 名と確定する。
- 記録保存期間は 7 年以上と確定する。
---

## §4 事後レビュー実施記録の保存手順

### GitHub Issue への保存

1. `work-navigation-app` リポジトリに Issue を作成する。Issue タイトルは `[事後レビュー] YYYY-MM-DD 移行完了事後レビュー報告書` とする。
2. §3 のテンプレートに記入した Markdown テキストを Issue の本文に貼り付ける。
3. quality_admin と system_admin が Issue にコメントを追記して電子署名を行う。
4. Issue に `migration-review` ラベルを付与する。
5. Issue を Close しない（7 年間 Open として保持する）。

### evidence_files テーブルへの保存

移行事後レビュー報告書を `evidence_files` テーブルに登録する。登録時は以下のカラムに値を設定する。

| カラム名 | 値 |
|---|---|
| `evidence_type` | `'migration_post_review'` |
| `file_name` | `'migration_post_review_YYYYMMDD.md'` |
| `github_issue_url` | GitHub Issue の URL |
| `created_by` | quality_admin の worker_id |
| `retention_years` | `7` |
| `created_at` | 現在のタイムスタンプ |

### 保存完了の確認

保存完了は以下の 2 点を確認することで判断する。

1. GitHub Issue が作成され、質問者・担当者の電子署名コメントが存在すること。
2. `evidence_files` テーブルに `evidence_type = 'migration_post_review'` のレコードが存在し、`github_issue_url` に有効な URL が設定されていること。

保存完了確認は MIG-CK-120 の完了条件として記録する。
