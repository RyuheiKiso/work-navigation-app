本章の責務は、`docs/08_移行/` 配下のすべての文書に適用される版数管理ルール（SemVer）・Changelog 形式・緊急変更手順・廃止規約を確定することである。本規約への準拠を宣言した時点で、すべての文書変更は本規約に従う。

本規約は版数 0.1.0 から発効する。発効日は 2026-05-18 とする。

## §2 版数規約（SemVer）

`docs/08_移行/` の各文書は Semantic Versioning 2.0.0（SemVer）に従って版数を管理する。版数の形式は `MAJOR.MINOR.PATCH` とする。

### MAJOR バージョン

MAJOR バージョンは移行プロセスの根本的変更が発生した場合に更新する。根本的変更の定義は以下の通りとする。

- 移行フェーズの追加または廃棄（例: リハーサルフェーズを 1 回から 2 回に変更）
- 移行方式の根本変更（例: 並行運用方式から一斉切替方式への変更）
- 識別子体系の再設計（例: MIG-NNN から別プレフィックスへの移行）

MAJOR バージョンが更新された場合、依存する章（MTM・カバレッジ表・ADR 索引）の改定が必須となる。

### MINOR バージョン

MINOR バージョンは後方互換性を保ったままの追加変更が発生した場合に更新する。追加変更の定義は以下の通りとする。

- 章・節の新規追加（既存章の内容変更なし）
- 識別子の新規採番（MIG-NNN・MIG-X-NNN 等の追加発番）
- 付録ファイルの新規追加

MINOR バージョンが更新された場合、採番台帳（`02_移行識別子採番台帳.md`）と MTM（`01_移行トレーサビリティマトリクス（MTM）.md`）を同時更新する。

### PATCH バージョン

PATCH バージョンは意味内容に変更を与えない修正が発生した場合に更新する。修正の定義は以下の通りとする。

- 誤記修正（漢字・送り仮名・数値の誤り）
- 表現修正（同一の意味を別表現に置換）
- リンク・パス修正（内容の変更なし）

PATCH バージョン更新の場合、採番台帳・MTM の更新は不要とする。

### 初版の版数

すべての文書の初版は `0.1.0` から開始する。`1.0.0` への昇格は移行完了宣言（MIG-060〜062 対応）の承認をもって確定する。

---

**本節で確定した方針**
- MAJOR は移行フェーズの追加・廃棄および移行方式の根本変更に対応する場合に更新することを確定する。
- MINOR は章・識別子の後方互換追加に対応する場合に更新することを確定する。
- 初版は `0.1.0` から開始し、`1.0.0` 昇格は移行完了宣言承認をもって確定することを確定する。
---

## §3 Changelog の形式

すべての文書は各ファイル末尾の「更新履歴」テーブルに変更を記録する。Conventional Commits に準拠した変更種別を「変更内容」列に記述する。

### Changelog テーブル定義

| 列名 | 型 | 制約 | 説明 |
|---|---|---|---|
| 版数 | 文字列 | NOT NULL | SemVer 形式（例: `0.1.0`） |
| 日付 | 日付 | NOT NULL, ISO 8601 | YYYY-MM-DD 形式 |
| 変更内容 | 文字列 | NOT NULL | Conventional Commits 形式 |
| 影響範囲 | 文字列 | NULL 可 | 変更の影響を受けるファイル・識別子 |
| 変更者 | 文字列 | NOT NULL | GitHub ユーザー名または役割名 |

### Conventional Commits の変更種別

本文書群で使用する Conventional Commits の種別は以下の通りとする。

| 種別 | 意味 | バージョン更新 |
|---|---|---|
| `feat:` | 新規章・識別子の追加 | MINOR |
| `fix:` | 誤記・表現の修正 | PATCH |
| `refactor:` | 意味変更を伴わない構造整理 | PATCH |
| `breaking:` | 移行プロセスの根本的変更 | MAJOR |
| `docs:` | 本規約外の参照資料の更新 | PATCH |

### Changelog 記述例

以下は Changelog テーブルの記述例である。

| 版数 | 日付 | 変更内容 | 影響範囲 | 変更者 |
|---|---|---|---|---|
| 0.1.0 | 2026-05-18 | feat: 初版作成 | — | RyuheiKiso |
| 0.1.1 | YYYY-MM-DD | fix: §2 PATCH バージョンの定義に「リンク修正」を追記 | §2 | quality_admin |
| 0.2.0 | YYYY-MM-DD | feat: MIG-075〜080 追加採番、MTM 6 行追加 | MTM §2 | system_admin |

---

**本節で確定した方針**
- Changelog テーブルに YYYY-MM-DD / 版数 / Conventional Commits 形式変更内容 / 影響範囲 / 変更者の 5 列を確定する。
- Conventional Commits の 5 種別（feat / fix / refactor / breaking / docs）を `docs/08_移行/` 全文書に対応する変更種別として確定する。
- 影響範囲列は NULL 可とし、複数ファイルに影響する変更は全ファイル名を列挙することを確定する。
---

## §4 緊急変更手順

移行実施中（移行実施フェーズ開始から移行完了宣言まで）に本番手順書の変更が必要となった場合、通常の変更手順に加えて以下の緊急変更手順を適用する。

### 緊急変更の定義

緊急変更とは、移行実施中に発生した問題対応のために本番手順書（`docs/08_移行/導入手順/` 配下のファイル）を即時修正する必要がある場合の変更手順である。

### 2 者承認要件

移行実施中の本番手順書修正は `quality_admin` と `system_admin` の 2 者承認を必須とする。1 者のみの承認による本番手順書修正は認めない。

2 者承認の実施手順は以下の通りとする。

1. `system_admin` が変更案を GitHub Pull Request として作成する。
2. `quality_admin` が変更案を Review し、承認コメントを付与する。
3. `system_admin` が承認コメントを確認し、Pull Request をマージする。
4. マージ後、変更記録を ADR として `付録/05_移行ADR索引.md` に追記する。

### 緊急変更の記録義務

緊急変更が発生した都度、変更記録を `付録/05_移行ADR索引.md` に `ADR-MIG-NNN` として記録する。ADR には変更の背景・決定内容・根拠・影響を記述する（§5 ADR テンプレート参照）。

緊急変更 ADR の `状態` 欄は「緊急変更」と明記する。緊急変更 ADR の日付は Pull Request マージ日とする。

### 緊急変更中の作業継続

2 者承認の完了前に変更内容を現場で先行適用する場合、現場監督は変更箇所を口頭で作業員に伝達し、作業員が変更前手順を実行しないよう措置を講じる。先行適用の記録は問題点票（`PROB-MIG-NNN`）に記録する。

---

**本節で確定した方針**
- 移行実施中の本番手順書修正は `quality_admin` + `system_admin` の 2 者承認を必須と確定する。
- 緊急変更記録を `付録/05_移行ADR索引.md` に `ADR-MIG-NNN` として記録することを確定する。
- ADR の状態欄に「緊急変更」と明記することを確定する。
---

## §5 廃止規約

文書内の識別子・章・節を廃止する場合、廃止対象を削除せず廃止マーカーを付与して履歴を保持する。

### 識別子の廃止

識別子を廃止する場合、当該行の行頭に以下のマーカーを付与する。行の削除は禁止する。

```
[廃止: YYYY-MM-DD 廃止理由: <廃止の根拠>]
```

廃止マーカーの記述例を以下に示す。

```markdown
| [廃止: 2026-06-01 廃止理由: MIG-015 と統合のため] MIG-014 | MIG-INI | DES-MIG-021〜025 | MIG-X-021 | MIG-CK-010 | 移行計画/02 | 廃止 |
```

廃止された識別子は MTM（`01_移行トレーサビリティマトリクス（MTM）.md`）の「状態」列を「廃止」に更新する。採番台帳（`02_移行識別子採番台帳.md`）の対応行も同様に「廃止」に更新する。

廃止された識別子の番号は再利用しない。新規識別子は次の未使用番号から採番する。

### 章・節の廃止

章または節を廃止する場合、以下の規則に従う。

- 廃止された章の見出し（H1〜H3）は保持する。
- 章のコンテンツを `> [廃止] YYYY-MM-DD 廃止。廃止理由: <根拠>` の Blockquote ノートで置換する。
- 廃止章の番号は欠番として保持し、後続の章番号を繰り上げない。

廃止規約の適用例を以下に示す。

```markdown
## §X [廃止] YYYY-MM-DD

> [廃止] 2026-06-01 廃止。廃止理由: §Y に統合のため。§Y を参照のこと。
```

### 廃止の記録

章・識別子の廃止は変更管理規約に従い、Changelog テーブルに記録する。廃止理由・代替章・影響範囲を「変更内容」列に記述する。大規模な廃止（3 件以上の識別子を同時廃止）は `付録/05_移行ADR索引.md` に ADR として追記する。

---

**本節で確定した方針**
- 識別子の廃止時は `[廃止: YYYY-MM-DD 廃止理由: ...]` を行頭付与し行削除禁止と確定する。
- 廃止章は見出しを残してコンテンツを `[廃止]` Blockquote ノートに置換することを確定する。
- 廃止された識別子の番号の再利用を禁止し次の未使用番号から採番することを確定する。
