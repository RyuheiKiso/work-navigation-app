# 05 移行 ADR 索引

本章の責務は、IPA 共通フレーム 2013「6.5.8 問題解決プロセス」に準拠し、移行段階において発生した意思決定記録（ADR-MIG-NNN）と移行問題記録（PROB-MIG-NNN）を正典的に管理することである。

---

## §1 本章の責務

本章の責務は、IPA 6.5.8 問題解決プロセスに基づき、移行計画・手順の変更・解釈の確定・トレードオフの判断を ADR-MIG-NNN として記録し、移行実施中に発生した問題を PROB-MIG-NNN として管理することである。

ADR（Architecture Decision Record）の移行適用版として ADR-MIG を定義する。ADR-MIG は移行プロセスに限定した意思決定記録であり、システムアーキテクチャ全体の ADR（`docs/04_概要設計/` 管轄）とは別管理とする。

本章が ADR-MIG と PROB-MIG の唯一の権威ソースである。他のファイルに ADR-MIG や PROB-MIG を記述する場合は、本章の当該エントリへの参照を必ず付与する。

---

**本節で確定した方針**
- ADR-MIG-NNN を本章のみで管理する唯一の権威ソースとして確定する。
- ADR-MIG は移行プロセス限定とし、システム全体の ADR とは別管理とすることを確定する。
- PROB-MIG-NNN の管理も本章 §4 で一元管理することを確定する。
---

## §2 ADR の定義と運用ルール

### ADR-MIG の採番規則

ADR-MIG は `ADR-MIG-NNN` 形式（NNN は 3 桁のゼロ埋め連番）で採番する。採番は 001 から開始し、欠番なく連続して採番する。採番台帳（`02_移行識別子採番台帳.md`）の ADR-MIG 欄を参照して次の採番番号を確認する。

採番後は台帳の「次採番」欄を更新する。ADR-MIG の採番権限は quality_admin と system_admin の双方に付与する。

### 記録トリガー

以下のいずれかに該当する場合、ADR-MIG を採番して記録する。

- 移行計画の変更（スケジュール変更・スコープ変更・方式変更）
- 手順の解釈を確定した場合（曖昧な手順について具体的な実施方法を確定した場合）
- トレードオフの判断（複数の選択肢から 1 つを選んだ場合）
- 緊急変更手順の適用（付録/00 §4 に基づく 2 者承認変更）
- 識別子の廃止・統合の決定

### 不変記録の保持

ADR-MIG は不変記録として保存する。一度確定した ADR-MIG は削除しない。ADR が廃止された場合は「状態」欄を「廃止」に変更し、`[廃止] YYYY-MM-DD 廃止理由: <根拠>` のコメントを「コンテキスト」欄に追記する。ADR 本体の記述は変更しない。

---

**本節で確定した方針**
- ADR-MIG は `ADR-MIG-NNN` 形式（3 桁ゼロ埋め）で採番し 001 から開始することを確定する。
- 記録トリガー（計画変更・解釈確定・トレードオフ判断・緊急変更・廃止決定）を確定する。
- ADR-MIG を不変記録として保存し削除禁止とすることを確定する。
---

## §3 ADR テンプレート

ADR-MIG を追加する際は以下のテンプレートを使用する。

---

```markdown
### ADR-MIG-NNN: タイトル

| 項目 | 内容 |
|---|---|
| 日付 | YYYY-MM-DD |
| 状態 | 確定済み / 廃止 / 緊急変更 |
| 採番者 | （GitHub ユーザー名または役割名）|

#### コンテキスト

判断が必要になった状況を記述する。どのような問題・疑問・制約が発生したかを事実ベースで記述する。

#### 決定内容

何を確定したかを断定的な文で記述する。「〜と確定する」の形式で終止する。

#### 根拠

なぜその決定をしたかの根拠を記述する。技術的根拠・品質上の根拠・リスク管理上の根拠を明示する。

#### 影響

この決定が与える影響を記述する。影響を受けるファイル・識別子・プロセスを明示する。

#### 代替案

検討したが採用しなかった案と、採用しなかった理由（対象外判断の根拠）を記述する。

| 代替案 | 対象外とした理由 |
|---|---|
| （代替案 1）| （対象外の根拠）|
```

---

**本節で確定した方針**
- ADR テンプレートの 6 項目（コンテキスト・決定内容・根拠・影響・代替案・日付・状態）を確定する。
- 決定内容は「〜と確定する」の断定形で終止することを確定する。
- 代替案欄に採用しなかった理由（対象外判断の根拠）を必ず記述することを確定する。
---

## §4 PROB-MIG（移行問題記録）の管理

### PROB-MIG の採番規則

移行問題記録は `PROB-MIG-NNN` 形式（NNN は 3 桁のゼロ埋め連番）で採番する。採番は 001 から開始する。PROB-MIG の採番権限は quality_admin・system_admin・現場監督の 3 者に付与する。

発見者が PROB-MIG を採番し、直ちに GitHub Issue として登録する。Issue タイトルは `[PROB-MIG-NNN] 現象の概要` とする。

### 問題点票テンプレート

| 項目 | 内容 |
|---|---|
| PROB-MIG ID | PROB-MIG-NNN |
| 発見日 | YYYY-MM-DD |
| 発見者 | （役割名または GitHub ユーザー名）|
| 発見フェーズ | リハーサル / カットオーバー / 検証 / 事後レビュー |
| 現象 | 発生した事象を事実ベースで記述する |
| 原因 | 根本原因を記述する（不明の場合は「調査中」）|
| 対策 | 是正処置の内容を記述する |
| 担当 | 是正処置の担当者（役割名）|
| 期限 | YYYY-MM-DD |
| 解消日 | YYYY-MM-DD（解消後に記入）|
| 解消確認者 | quality_admin（解消後に記入）|
| 関連 ADR | ADR-MIG-NNN（関連する ADR がある場合）|
| GitHub Issue URL | （登録した Issue の URL）|

### PROB-MIG の解消条件

PROB-MIG の解消条件は以下の通りとする。

1. 是正処置が実施されていること。
2. quality_admin が解消を確認し、GitHub Issue にコメントを追記していること。
3. `evidence_files` テーブルに解消証跡が登録されていること。

解消された PROB-MIG の GitHub Issue は Close する。Close 後も Issue は削除しない。

---

**本節で確定した方針**
- PROB-MIG は `PROB-MIG-NNN` 形式で採番し GitHub Issue として即時登録することを確定する。
- 解消条件として「是正処置実施 + quality_admin による確認 + evidence_files 登録」の 3 条件を確定する。
- 解消後の GitHub Issue を Close するが削除しないことを確定する。
---

## §5 初期 ADR 一覧

移行設計・計画段階で確定した代表的な意思決定記録を以下に示す。

---

### ADR-MIG-001: 移行方式として並行運用→カットオーバーを採用

| 項目 | 内容 |
|---|---|
| 日付 | 2026-05-18 |
| 状態 | 確定済み |
| 採番者 | RyuheiKiso |

#### コンテキスト

移行方式の選定において、「並行運用後カットオーバー方式」「一斉切替方式」「段階移行方式」の 3 案が候補に挙がった。製造現場では作業記録の断絶を許容できないため、方式選定は品質・リスク・運用負荷の観点から評価が必要であった。

#### 決定内容

移行方式として「並行運用（最大 2 週間）後にカットオーバー」する方式を採用することと確定する。

#### 根拠

並行運用期間中に旧システムと新システムの記録を照合することで、移行完全性の検証が可能となる。一斉切替方式はカットオーバー後の問題発生時のリスクが高く、製造現場での採用は ALCOA+ 観点から不適切と判断した。段階移行方式は工程分断が発生するためトレーサビリティの連続性を損なう。

#### 影響

移行計画/05〜06（並行運用・カットオーバー計画）・移行計画/07（ロールバック手順）・MIG-047〜056 の担当章に影響する。

#### 代替案

| 代替案 | 対象外とした理由 |
|---|---|
| 一斉切替方式 | カットオーバー後の縮退検出が困難であり ALCOA+ Accurate 要件を充足できないため対象外と判断する |
| 段階移行方式 | 工程ごとの部分移行でトレーサビリティの連続性が断絶するため対象外と判断する |

---

### ADR-MIG-002: 新規採番プレフィックスとして MIG-X/RH/RB/CK/REP を採用

| 項目 | 内容 |
|---|---|
| 日付 | 2026-05-18 |
| 状態 | 確定済み |
| 採番者 | RyuheiKiso |

#### コンテキスト

08 章（移行）の識別子体系を設計する際、既存の `MIG-NNN`（移行要件）・`DES-MIG-NNN`（移行方式設計）に加え、08 章実施命題・リハーサル命題・ロールバック命題・チェックリスト・報告様式の 5 種を採番する必要が生じた。

#### 決定内容

08 章実施命題を `MIG-X-NNN`・リハーサル命題を `MIG-RH-NNN`・ロールバック命題を `MIG-RB-NNN`・チェックリストを `MIG-CK-NNN`・報告様式を `MIG-REP-NNN` として採番することと確定する。

#### 根拠

プレフィックスに `MIG-` を統一することで移行文書群の識別子に統一性を持たせる。X（eXecution）・RH（ReHearsal）・RB（RollBack）・CK（ChecKlist）・REP（REPort）は略称から意味が直感的に把握できる。

#### 影響

採番台帳（`02_移行識別子採番台帳.md`）・MTM（`01_移行トレーサビリティマトリクス（MTM）.md`）・IPA カバレッジ表（`03_IPA共通フレーム2013移行プロセスカバレッジ表.md`）全体に影響する。

#### 代替案

| 代替案 | 対象外とした理由 |
|---|---|
| EXEC-NNN / RH-NNN 等（MIG- プレフィックスなし）| 移行文書群の識別子に統一性を持たせる観点から対象外と判断する |
| MIG-NNN 単一プレフィックスに分類コード付与 | 既存 MIG-NNN（移行要件）との衝突リスクがあるため対象外と判断する |

---

### ADR-MIG-003: ハンディ APP 配布をハイブリッド構成（統合 + OS 別 4 ファイル）と確定

| 項目 | 内容 |
|---|---|
| 日付 | 2026-05-18 |
| 状態 | 確定済み |
| 採番者 | RyuheiKiso |

#### コンテキスト

ハンディ APP は React Native を使用し、Android・iOS・Windows の 3 OS に対応する（CLAUDE.md 記載）。APP の配布形式として、単一ファイル統合配布・OS 別配布・ハイブリッド（統合 + OS 別）の 3 案が候補となった。

#### 決定内容

ハンディ APP の配布を「統合ファイル 1 種 + OS 別ファイル 4 種（Android・iOS・Windows Desktop・Windows Handy）」のハイブリッド構成で実施することと確定する。

#### 根拠

製造現場では iOS・Android・Windows の混在環境が発生するため、OS 別ファイルによる個別対応が必要である。一方で統合ファイルを提供することで、環境確認が困難な現場でも動作確認が可能となる。導入手順/07（ハンディ APP 配布手順）は統合ファイルを主手順とし、OS 別ファイルを補足手順として記載する。

#### 影響

導入手順/07 §1〜8・INST-A4-f・MIG-X-073〜085 に影響する。

#### 代替案

| 代替案 | 対象外とした理由 |
|---|---|
| 単一統合ファイルのみ | OS 固有の動作確認が困難になるため対象外と判断する |
| OS 別ファイルのみ（統合なし）| 混在環境での一括確認手順が存在しなくなるため対象外と判断する |

---

### ADR-MIG-004: 旧システム廃止計画（移行計画/11）を DES-MIG-003 整合しつつ残置と確定

| 項目 | 内容 |
|---|---|
| 日付 | 2026-05-18 |
| 状態 | 確定済み |
| 採番者 | RyuheiKiso |

#### コンテキスト

旧システムの廃止計画（移行計画/11）が 08 章のスコープに含まれるかについて判断が必要となった。特に DES-MIG-003（移行方式設計の前提条件）との整合において、旧システムの廃止が移行完了宣言後の後続作業として位置づけるか、移行完了条件として位置づけるかが議論となった。

#### 決定内容

旧システム廃止計画（移行計画/11）を 08 章の独立した章として残置し、DES-MIG-003 との整合を保ちながら移行完了宣言後の後続作業として位置づけることと確定する。旧システムの停止は移行完了宣言の前提条件ではなく、並行運用終了後の後続手続きとして扱う。

#### 根拠

旧システムの廃止は移行完了の証跡確認が完了した後に実施する方が、ALCOA+ Enduring（永続性）の観点から安全である。廃止完了を移行完了宣言の前提条件とすると、廃止手続きの遅延が移行完了宣言を妨げるリスクが生じる。

#### 影響

移行計画/11 §1〜8・MIG-X-015・移行計画/10 §7（移行完了宣言書）・MIG-060〜062（移行完了定義）に影響する。

#### 代替案

| 代替案 | 対象外とした理由 |
|---|---|
| 旧システム廃止を移行完了宣言の前提条件とする | 廃止手続きの遅延リスクが移行完了宣言に影響するため対象外と判断する |
| 旧システム廃止を 08 章スコープ外とする | DES-MIG-003 の整合管理が困難となるため対象外と判断する |

---

**本節で確定した方針**
- ADR-MIG-001〜004 を初期 ADR として確定し不変記録として保持することを確定する。
- 新規 ADR は §3 のテンプレートに従い採番台帳と同期して追記することを確定する。
- ADR-MIG の削除を禁止し廃止時は「状態」欄を「廃止」に変更することを確定する。
