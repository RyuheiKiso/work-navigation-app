# 99 前提制約

本章の責務は、移行計画サブグループ（移行計画/01〜11）固有の対象外宣言と前提制約を確定することである。08_移行/99_前提制約と本書が約束しないこと.md（D-MIG-X-001〜005、CON-MIG-X-001〜005）を上位宣言として継承しつつ、移行計画サブグループに特有の追加対象外事項（D-MIG-X-006〜010）と追加前提制約（CON-MIG-X-006〜010）を確定する。本章の宣言は移行計画/01〜11 の全章に適用され、各下位計画書がスコープを逸脱した設計を行わないための境界を規定する。

## 1. 移行計画サブ固有の対象外宣言（D-MIG-X-006〜010）

本節は移行計画/01〜11 が明示的に取り扱わない事項を宣言する。D-MIG-X-006〜010 は上位の D-MIG-X-001〜005 を補完するものであり、移行計画サブグループの設計・記述範囲に特化した対象外宣言である。

| D-MIG-X ID | 対象外事項 | 判断根拠 | 対処方針 |
|---|---|---|---|
| D-MIG-X-006 | 現場無線 LAN 障害時の縮退手順 | 障害発生後の縮退運用・復旧手順は移行フェーズを超えた運用・保守上の課題であり、移行計画の責務範囲外 | 無線 LAN 障害時の縮退手順は 09_運用・保守/障害対応に委任し、移行計画の対象外と判断する |
| D-MIG-X-007 | 移行リハーサルの第 3 回以降の追加実施 | リハーサルは第 1 回（Phase M-2 開始前 2 週）と、第 1 回で P1 問題が発見された場合の第 2 回を移行計画に含める。移行完了条件（CON-MIG-X-007 の SOP 総数 100 件以下・訓練完了・重大バグゼロ）を充足した場合は第 3 回以降の追加実施を不要と判断する | 第 3 回以降のリハーサルは対象外と判断する。ただし質担当が業務判断で追加リハーサルを指示する場合は、移行計画/03（リハーサル計画）の改版を経て実施する |
| D-MIG-X-008 | SOP 以外のドキュメントの移行（設計書・管理台帳等） | 移行計画は「SOP・マスタデータ・ユーザーアカウント」を移行対象スコープとして確定している。設計書（図面・品質計画書）・管理台帳（設備点検記録・教育訓練記録）等の関連ドキュメントは別管理系統に属し、本システムの保持対象ではない | SOP 以外のドキュメント移行は対象外と判断する。関連ドキュメントの管理方針は工場の文書管理規定に従い、別途実施する |
| D-MIG-X-009 | マスタデータの自動バリデーション以外の品質確認（目視以外の外部検査） | 移行計画は自動バリデーション（FK 整合性・NULL チェック・重複チェック）と quality_admin による目視サンプリング（MIG-T-006）を品質確認手段として確定する。外部の第三者検査・統計的品質管理手法（SPC）の適用は個人開発プロジェクトの規模を超える | 自動バリデーションと目視サンプリング以外の品質確認手法は対象外と判断する。追加の品質確認が必要な場合は quality_admin の判断で個別に設計する |
| D-MIG-X-010 | 移行中の本番システム以外のシステム変更 | 移行期間中（Phase M-1〜Phase M-3 完了まで）に本番システム（PostgreSQL・Rust バックエンド・React Native ハンディ APP）以外のシステム変更（基幹システム連携・新機能追加・インフラ変更）は移行の安定性を損なうリスクがある | 移行期間中の本番システム以外のシステム変更は対象外と判断する。移行完了宣言後に別途変更管理手順（09_運用・保守/変更管理）に従い実施する |

### D-MIG-X-006: 現場無線 LAN 障害時の縮退手順

移行計画が担保する WiFi 関連の事項を以下に限定する。

| 移行計画が担保する WiFi 関連事項 | 対象外（委任先） |
|---|---|
| Phase M-1 完了条件としての WiFi 疎通確認（MIG-T-008） | 移行後の WiFi 障害発生時の縮退動作設計（→ 09_運用・保守/障害対応） |
| カットオーバーチェックリスト CL-001（WiFi 電波強度 -70 dBm 以上の確認） | 障害復旧手順（接続先切り替え・4G/LTE フォールバック等） |
| Phase M-3 時の No-Go 条件（WiFi 障害発生時のカットオーバー延期） | 複数 AP 構成の冗長設計 |

移行計画の観点から WiFi に関して確定するのは「移行実施時点での接続確認基準」のみである。移行完了後の障害対応は 09_運用・保守の責務範囲となる。

### D-MIG-X-007: リハーサル第 3 回以降の追加実施

移行計画が定めるリハーサルの実施回数は以下の通りとする。

| リハーサル回数 | 実施条件 | 対応ドキュメント |

| 第 3 回以降 | 対象外と判断する | — |

CON-MIG-X-007 が規定する SOP 総数 100 件以下という前提のもとでは、第 2 回リハーサルまでに P1 問題を解消できるとの判断に基づき、第 3 回以降を対象外とする。SOP 総数が 100 件を超える場合または第 2 回でも P1 問題が残存する場合は、quality_admin の業務判断で移行計画/03 を改版し、第 3 回の実施を計画に追加する。

### D-MIG-X-009: 自動バリデーション以外の品質確認手法

移行計画が定める品質確認手段を以下に限定する。

| 移行計画が確定する品質確認手段 | 対象外の品質確認手法 |
|---|---|
| 自動バリデーション（FK 整合性・NULL・重複・型不整合の検出） | 統計的工程管理（SPC）による入力データの品質分析 |
| quality_admin による目視サンプリング（10% 以上のレコードを抜取確認） | 第三者審査員による独立検査 |
| 投入後のシステム上でのプレビュー確認（全 SOP の表示確認） | 自動比較ツールによる紙原本との OCR 読み取り結果の機械照合 |
| 電子署名付き承認（quality_admin が全 SOP を承認） | ブラインドレビュー（異なる担当者による 2 回の独立確認） |

個人開発プロジェクトの体制規模において上記の対象外手法を実施することは過剰品質要求と判断する。自動バリデーション + 目視サンプリング + 電子署名承認の 3 段階で品質を担保する。

### D-MIG-X-010: 移行中の本番システム以外のシステム変更

移行フリーズ期間（Phase M-1 開始から Phase M-3 完了宣言まで）において、以下のシステム変更は対象外と判断する。

| 対象外のシステム変更 | 理由 |
|---|---|
| 本番 PostgreSQL スキーマの変更（テーブル追加・カラム変更） | 移行中のスキーマ変更は投入済みデータとの整合性を破壊するリスクがある |
| Rust バックエンドの新機能リリース | 移行中のバグ混入リスクを最小化するため、移行フリーズ期間は修正リリースのみを対象とする |
| React Native ハンディ APP のバージョンアップ | デバイス上のアプリバージョンの不整合がカットオーバー時の動作確認を複雑化させる |
| インフラ構成の変更（Docker Compose 設定・IIS 設定） | 移行中の環境変更はカットオーバー後の問題切り分けを困難にする |

移行フリーズ期間中に緊急のセキュリティパッチ適用が必要な場合は、quality_admin と system_admin の 2 者合意の上で変更管理記録を作成し、ステージング環境での検証を経てから本番に適用する。

---

**本節で確定した方針**
- D-MIG-X-006〜010（無線 LAN 障害縮退・第 3 回以降リハーサル・SOP 以外ドキュメント移行・追加品質確認手法・移行中システム変更）を移行計画サブの対象外と判断する。
- 移行フリーズ期間（Phase M-1 開始〜Phase M-3 完了）中の本番システム変更は対象外と判断し、緊急パッチのみ 2 者合意 + 変更管理記録で対応することを確定する。
- SOP 総数 100 件以下の前提（CON-MIG-X-007）が崩れた場合は移行計画/03 の改版を経て第 3 回以降のリハーサルを実施することを確定する。

### CON-MIG-X-006: 作業員 50 名以下前提

移行計画/02（訓練計画）は作業員 50 名以下を前提として以下の設計パラメータを確定する。

| 設計パラメータ | 50 名以下前提での値 | 崩れた場合の見直し内容 |
|---|---|---|
| 訓練セッション数 | 2〜3 回（各回 15〜25 名） | 50 名超の場合は 1 セッション 20 名上限として追加回を設定する |
| 訓練所要時間（1 回あたり） | 2 時間（ハンディ操作 + Q&A） | 参加者数に応じて Q&A 時間を延長する |
| 訓練担当者数 | quality_admin 1 名（アシスタントなし） | 51 名以上の場合は現場監督をアシスタントとして訓練に追加する |
| 訓練完了確認方法 | 全作業員の初回ログイン完了確認 | 変更なし |

作業員数が Phase M-1 開始後に増加した場合（新規採用・応援人員の追加等）は、追加対象者に対して個別訓練を実施し、質担当が追加訓練完了記録を作成する。

### CON-MIG-X-007: SOP 総数 100 件以下前提

移行計画/03（リハーサル計画）および移行計画/05（マスタデータ品質計画）は SOP 総数 100 件以下を前提として以下の設計パラメータを確定する。

| 設計パラメータ | 100 件以下前提での値 | 崩れた場合の見直し内容 |
|---|---|---|
| Phase M-1 の SOP 投入所要日数 | 5〜10 営業日 | 101 件以上の場合は日数見積もりを再計算する（1 件 30 分 × 件数 ÷ 1 日の作業時間） |
| リハーサル対象 SOP 数 | 3 件（最頻使用 SOP を選定） | 101 件以上の場合は選定基準を見直し、工程カテゴリ別代表 SOP を選定する |
| 品質チェック目視サンプリング率 | 全 SOP の 100%（100 件以下の場合は全件確認が現実的） | 101 件以上の場合は 20% 以上のサンプリングに変更する |
| Phase M-1 の全 SOP 電子承認所要日数 | 1〜2 営業日（quality_admin が一元実施） | 101 件以上の場合は日数を再見積もりし、承認作業の並行実施を quality_admin が確定する |

### CON-MIG-X-008: 並行運用中の追加マスタ変更は quality_admin が一元管理

Phase M-2（並行運用期間）中に追加マスタ変更が発生する場合は、以下の手順で quality_admin が一元管理する。

| 管理手順 | 内容 |
|---|---|
| 変更申請 | 変更要求者が quality_admin に変更内容・変更理由・影響範囲を書面（電子メールまたはシステム上のコメント）で報告する |
| 影響評価 | quality_admin が変更によるカットオーバー準備状態への影響を評価する（追加投入・再バリデーションが必要かを判断する） |
| 承認と実施 | quality_admin が承認した変更のみを master_admin が実施し、変更後のデータを再バリデーションする |
| 記録 | 変更内容・承認記録・再バリデーション結果を移行記録に追記する（ALCOA+ Attributable 原則への対応） |

quality_admin の承認なしに実施されたマスタ変更が発見された場合は、変更内容を特定し、影響範囲を評価した上で再バリデーションを実施する。故意の無承認変更は移行の継続可能性評価対象とする。

### CON-MIG-X-009: カットオーバー実施日は週末または計画停止日

カットオーバー実施日の選定基準を以下の通り確定する。

| 実施日区分 | 対応 |
|---|---|
| 週末（土曜日が最優先） | 最も推奨する実施日。生産ラインが停止しており、カットオーバー所要時間 2〜4 時間に余裕がある |
| 計画停止日（設備メンテナンス日・工場休暇日） | 次点の推奨実施日。事前に生産停止が確定しているため、カットオーバー実施の予測可能性が高い |
| 平日（生産中） | 対象外と判断する（D-MIG-X-010 と合わせて移行フリーズ期間中の平日カットオーバーを禁止する） |

カットオーバー実施日は quality_admin が Phase M-2 開始時に確定し、全作業員に対して DES-MIG-051 の CO-PRE-005（24 時間前の事前通知）を実施する。

### CON-MIG-X-010: ステージング環境は本番同等構成で実施

リハーサル（移行計画/03）で使用するステージング環境は以下の本番同等要件を充足する。

| ステージング同等要件 | 確認方法 |
|---|---|
| PostgreSQL バージョンが本番と同一 | `SELECT version();` の結果を本番と照合する |
| Docker Compose 設定が本番と同一（Rust バックエンド・イメージバージョン） | docker-compose.yml のハッシュ値を本番と比較する |
| React Native APP バージョンが本番と同一 | デバイスのバージョン番号を確認する |
| データ量が本番想定規模の 80% 以上（SOP 件数・マスタレコード数） | ステージング投入後のレコード件数を確認する |
| WiFi 環境が本番工場と同等の電波強度（-70 dBm 以上） | 電波強度測定ツールで確認する |

ステージング環境が本番と異なる構成でリハーサルを実施した場合、リハーサル結果の本番適用可能性が担保されず、Phase M-3 でのカットオーバー失敗リスクが増大する。ステージング環境の構成差異が発見された場合は、リハーサル実施前に system_admin が修正する。

---

**本節で確定した方針**
- CON-MIG-X-006〜010（作業員 50 名以下・SOP 100 件以下・並行期マスタ変更の quality_admin 一元管理・週末/計画停止日カットオーバー・本番同等ステージング環境）を移行計画サブの前提制約として確定する。
- 各前提制約のパラメータ超過・崩壊時の対処方針（セッション数追加・件数再見積もり・環境修正等）を確定する。
- カットオーバー実施日は週末（土曜日）または計画停止日に限定し、平日・生産中の実施は対象外と判断する。
