# 08 移行データ品質確認計画

本章の責務は、マスタ初期投入および過去記録移行において実施するデータ品質確認の手順・実施者・合否基準・品質ゲート判定書の形式を実施計画として確定することである。概要設計（DES-MIG-066〜075）が確定したゼロトレランス原則・PRE/POST チェック・ALCOA+ コンプライアンス確認・自動チェックツール構成を受けて、各チェックの具体的な実施手順・記録方法・品質ゲート判定書テンプレートを確定する。上流要件 INST-A2-e・A4-j および DES-MIG-066〜075 に対応する。

---

## 1. 本章の責務（INST-A2-e + A4-j / DES-MIG-066〜075 対応）

本節は、移行データ品質確認計画の適用範囲・前提条件・完了定義を確定する。

### 1-1. 適用範囲

本計画は以下の品質確認活動を対象とする。

| 品質ゲート | 実施タイミング | 対象 |
|---|---|---|
| 品質ゲート 1（投入前チェック） | GUI ウィザードのプレビューフェーズ（マスタ投入前） | PRE-001〜010 の自動バリデーション |
| 品質ゲート 2（投入後チェック） | マスタ投入コミット完了後 | POST-001〜005 の自動・目視チェック |
| 品質ゲート 3（カットオーバー前チェック） | カットオーバー実施前（T-72 時間以内） | ALCOA+ コンプライアンスチェック・最終整合性確認 |

本計画は移行計画/02（マスタ初期投入実施計画）・移行計画/03（ロット履歴引継ぎ実施計画）と密接に連動する。品質確認の実施証跡は各計画の完了判定に使用する。

### 1-2. 前提条件

| 前提 ID | 内容 | 確認者 |
|---|---|---|
| PRE-QC-001 | 品質チェックツール（Rust バッチスクリプト群）が本番環境・ステージング環境の両方にデプロイ済みであること | system_admin |
| PRE-QC-002 | 品質ゲート判定書テンプレートが system_admin・quality_admin に配布済みであること | system_admin |
| PRE-QC-003 | PRE-001〜010 のチェック定義が GUI ウィザードに組み込まれていること | system_admin |
| PRE-QC-004 | POST-001〜005 のバッチスクリプトが実行可能な状態にあること | system_admin |

### 1-3. 完了定義

| 完了条件 ID | 内容 | 確認者 |
|---|---|---|
| DONE-QC-001 | 品質ゲート 1・2・3 の全 3 ゲートが合格済みであること | system_admin |
| DONE-QC-002 | 各ゲートの品質ゲート判定書に quality_admin の署名が取得済みであること | quality_admin |
| DONE-QC-003 | 整合性チェック合格証明書が発行済みであること | system_admin |
| DONE-QC-004 | ALCOA+ 確認完了報告書に quality_admin の署名が取得済みであること | quality_admin |

---

**本節で確定した方針**
- 本計画の対象を投入前・投入後・カットオーバー前の 3 品質ゲートとし、全ゲート合格をカットオーバーの必要条件として確定する。
- 品質チェックツールの本番・ステージング環境へのデプロイ完了を本計画の前提条件として確定する。
- 各ゲートの品質ゲート判定書への quality_admin 署名を完了条件として確定する。

## 2. 品質ゲートの基本方針（ゼロトレランス）（MIG-CK-001〜005）

本節は、品質ゲートの基本方針・ゼロトレランス原則・不合格時対応ルールを確定する。

### 2-1. 3 ゲートの実施タイミング

**MIG-CK-001**: 3 品質ゲートの実施タイミングと位置づけを以下の通り確定する。

| ゲート | 名称 | 実施タイミング | 合格必要条件 |
|---|---|---|---|
| ゲート 1 | 投入前チェック | GUI ウィザードのプレビューフェーズ | PRE-001〜007 のクリティカルエラー 0 件 |
| ゲート 2 | 投入後チェック | マスタ投入コミット完了後（24 時間以内） | POST-001〜005 の全件合格 |
| ゲート 3 | カットオーバー前チェック | カットオーバー実施の 72 時間以内 | ALCOA+ 確認 9 要素全件合格・ハッシュチェーン整合性エラー 0 件 |

各ゲートは前ゲートの合格を前提条件とする。ゲート 1 未合格の状態でゲート 2 に進むことを禁止する。ゲート 2 未合格の状態でゲート 3 に進むことを禁止する。

### 2-2. ゼロトレランス原則の確定

**MIG-CK-002**: クリティカルエラーに対するゼロトレランス原則を以下の通り確定する。（DES-MIG-066 対応）

以下のクリティカルエラーが 1 件でも存在する場合、カットオーバーの実施を禁止する。

| クリティカルエラー種別 | 定義 | 対象ゲート |
|---|---|---|
| 外部キー違反（FK Violation） | 参照先の値が存在しない行がインポートデータに含まれる | ゲート 1・2 |
| 必須フィールド NULL | NOT NULL 制約のあるフィールドに NULL または空文字が存在する | ゲート 1・2 |
| 重複キー | 一意制約フィールドに重複値が存在する | ゲート 1・2 |
| SOP 電子署名未付与 | 公開ステータスの SOP のうち 1 件でも電子署名が未付与 | ゲート 2・3 |
| ハッシュチェーン整合性エラー | 初期ブロック生成後の整合性チェックでエラーが検出された | ゲート 2・3 |
| ALCOA+ 必須要素の非充足 | ALCOA+ の Attributable / Contemporaneous / Original 要素のいずれかが全件で NULL | ゲート 3 |

### 2-3. 警告（許容範囲）の定義

**MIG-CK-003**: 警告として扱い、自動修正後にインポートを許容する軽微な問題を以下の通り確定する。（DES-MIG-067 対応）

| 警告種別 | 自動修正内容 | 確認義務 |
|---|---|---|
| 末尾空白文字（trailing spaces） | trim() で自動除去 | quality_admin が警告ログを確認すること |
| 全角スペース | 半角スペースに自動変換 | quality_admin が警告ログを確認すること |
| 日付形式の軽微な相違（例：2026/05/17 → 2026-05-17） | ISO 8601 形式に自動正規化 | quality_admin が警告ログを確認すること |
| 連続した改行文字 | 単一改行文字に自動正規化 | quality_admin が警告ログを確認すること |
| ファイル行数の上限接近（40,000〜50,000 行） | 自動修正なし（分割を推奨） | system_admin が分割手順を案内すること |

警告のみ存在する場合はインポート実行ボタンを有効化するが、quality_admin が警告ログを確認した上でインポートを実行する。警告ログを未確認のままインポートを実行することを禁止する。

### 2-4. 不合格時の即時停止・原因究明・再投入ルール

**MIG-CK-004**: 品質ゲート不合格時の対応ルールを以下の通り確定する。

| 手順 | 内容 | 担当者 |
|---|---|---|
| 停止 | ゲート不合格を検知した時点で移行作業を即座に停止する | system_admin |
| 報告 | 不合格内容と影響範囲を quality_admin に 30 分以内に報告する | system_admin |
| 原因究明 | 不合格の根本原因を特定し、「移行問題票（MIG-ISSUE-NNN）」に記録する | system_admin |
| 対策 | 根本原因に対する対策を実施する | system_admin・master_admin |
| 再投入 | 対策完了後に当該ゲートからやり直す（部分的な継続は禁止する） | master_admin |
| 再確認 | 再投入後に当該ゲートを再実施し、合格を確認する | system_admin・quality_admin |

原因究明・対策の所要時間は上限を設けない。品質を確保するまでカットオーバーの日程を延期する。

### 2-5. 品質ゲート通過の記録義務

**MIG-CK-005**: 各品質ゲートの通過記録の義務を以下の通り確定する。

| 記録内容 | 記録先 | 記録者 |
|---|---|---|
| ゲート 1 合格の記録 | 品質ゲート 1 判定書（GQ-001 形式） | system_admin |
| ゲート 2 合格の記録 | 品質ゲート 2 判定書（GQ-002 形式） | system_admin |
| ゲート 3 合格の記録 | 品質ゲート 3 判定書（GQ-003 形式） | system_admin |
| audit_logs への記録 | audit_logs テーブル（action='QUALITY_GATE_N_PASSED'） | system_admin |

全 3 ゲートの判定書が揃い、quality_admin の署名が取得できた時点で「整合性チェック合格証明書」を発行する。

---

**本節で確定した方針**
- 品質ゲートを投入前・投入後・カットオーバー前の 3 ゲートとして確定し、全ゲート合格をカットオーバーの必要条件として確定する。
- クリティカルエラー 1 件でも存在する場合にカットオーバーを禁止するゼロトレランス原則を確定する。
- ゲート不合格時は即時停止・原因究明・対策後に当該ゲートからやり直すことを確定する。

## 3. 投入前チェック（PRE-001〜010）の実施計画（MIG-CK-006〜020）

本節は、投入前チェック（PRE-001〜010）の各項目の具体化・実施者・実施タイミング・合否基準・ツールを確定する。

### 3-1. 各 PRE チェック項目の具体化

**MIG-CK-006**: PRE-001（ファイルエンコーディング確認）の実施詳細を以下の通り確定する。（DES-MIG-068 対応）

| 属性 | 内容 |
|---|---|
| チェック内容 | アップロードされた CSV/Excel ファイルが UTF-8 エンコーディングであることを確認する |
| 実施方式 | Rust バッチが先頭 4 バイトの BOM 検査および全バイト列の UTF-8 妥当性チェックを実施する |
| 合格基準 | UTF-8 以外のエンコーディングバイト列が 0 件であること |
| 不合格時の対応 | Excel テンプレートを UTF-8（BOM 付き可）で再保存し、再アップロードする |
| 実施者 | 自動（Rust バッチ）。結果確認は master_admin |

**MIG-CK-007**: PRE-002（必須フィールド非 NULL）の実施詳細を以下の通り確定する。

| 属性 | 内容 |
|---|---|
| チェック内容 | 各エンティティの必須列（移行計画/02 §3-3 の記入規約に定める必須フィールド）に空値（空文字・NULL・スペースのみ）が存在しないことを確認する |
| 実施方式 | Rust バッチがファイルの全行・全必須列に対して NULL チェックを実施する |
| 合格基準 | 全必須列に空値が 0 件であること |
| 不合格時の対応 | エラー明細 CSV の「行番号・列名・エラー内容」を参照し、Excel テンプレートを修正して再アップロードする |

**MIG-CK-008**: PRE-003（外部キー参照整合性）の実施詳細を以下の通り確定する。

| 属性 | 内容 |
|---|---|
| チェック内容 | FK 列（process_no / operation_no / user_id 等）の値が、依存先テーブル（投入済みデータ）に存在することを確認する |
| 実施方式 | Rust バッチが PostgreSQL に対して EXISTS サブクエリを実行し、参照先に存在しない値を検出する |

| 不合格時の対応 | 依存先テーブルのデータが未投入の場合は先行ステップを完了させてから再実施する。参照値の記入ミスの場合は Excel テンプレートを修正して再アップロードする |
| 実施者 | 自動（Rust バッチ）。修正は master_admin |

**MIG-CK-009**: PRE-004（重複キー検出・ファイル内）の実施詳細を以下の通り確定する。

| 属性 | 内容 |
|---|---|
| チェック内容 | 一意制約フィールド（process_no / operation_no / username / sop_code / equipment_code / scan_code 等）において、アップロードファイル内に重複値がないことを確認する |
| 実施方式 | Rust バッチがファイル全行の一意制約列に対してハッシュマップを構築し、重複を検出する |
| 合格基準 | ファイル内に一意制約違反の重複値が 0 件であること |
| 不合格時の対応 | エラー明細 CSV で重複行を確認し、Excel テンプレートを修正して再アップロードする |
| 実施者 | 自動（Rust バッチ）。修正は master_admin |

**MIG-CK-010**: PRE-005（重複キー検出・既存 DB との照合）の実施詳細を以下の通り確定する。

| 属性 | 内容 |
|---|---|
| チェック内容 | 一意制約フィールドの値が、既存 DB テーブルにすでに存在しないことを確認する（再投入・追加投入時の重複防止） |
| 実施方式 | Rust バッチが PostgreSQL に対して EXISTS クエリを実行し、既存レコードとの重複を検出する |
| 合格基準 | 既存 DB に重複するキー値が 0 件であること |

**MIG-CK-013**: PRE-008（文字数制限）の実施詳細を以下の通り確定する。

**MIG-CK-014**: PRE-009（末尾空白・全角スペース）の実施詳細を以下の通り確定する。

| audit_logs への記録 | audit_logs（action='ALCOA_CHECK_PASSED'） |

### 5-7. LEGACY 免除根拠記録との整合性確認

**MIG-CK-044**: LEGACY 免除根拠記録との整合性確認を以下の通り確定する。

移行計画/03（MIG-X-044）で作成した LEGACY 免除根拠記録（移行計画書付属書 B）との整合性を quality_admin が確認する。

| 確認内容 | 合格基準 |
|---|---|
| 付属書 B に記載された非適合内容と ALCOA+ チェックバッチの結果が整合していること | 付属書 B と ALCOA+ 確認レポートの非適合件数が一致すること |
| 遡及修正済みの件数が付属書 B と一致していること | 修正件数が一致すること |

### 5-8. 品質ゲート 3 の最終合格判定

**MIG-CK-045**: 品質ゲート 3（カットオーバー前チェック）の最終合格判定を以下の通り確定する。

| 合格条件 | 内容 |
|---|---|
| ALCOA+ 9 要素の全件合格（非適合管理簿への記録含む） | MIG-CK-036〜039 の全確認が完了していること |
| ハッシュチェーン整合性エラー 0 件 | POST-004 相当のチェックがエラー 0 件であること |
| 電子署名付与確認 | POST-005 相当のチェックが合格であること |
| MIG-CK-041 の追加確認項目全件合格 | 全 SOP 公開・デバイス登録最終確認・ALCOA+ 完了報告書取得が完了していること |

---

**本節で確定した方針**
- ALCOA+ 9 要素（Attributable / Legible / Contemporaneous / Original / Accurate / Complete / Consistent / Enduring / Available）の確認を Rust バッチで自動実施し、quality_admin が最終判定することを確定する。
- LEGACY_IMPORT データへの ALCOA+ 適用は部分的非適合を ALCOA-NC 管理簿で管理することを確定する。
- 品質ゲート 3 合格の必要条件として ALCOA+ 確認完了報告書への quality_admin 署名を確定する。

|---|---|---|
