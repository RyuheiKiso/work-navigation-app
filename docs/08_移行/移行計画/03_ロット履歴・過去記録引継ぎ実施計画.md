# 03 ロット履歴・過去記録引継ぎ実施計画

本章の責務は、紙・Excel による既存の過去作業記録を本システムへ引き継ぐ際の実施手順・担当者・ALCOA+ 準拠確保・ハッシュチェーンでの扱いを実施計画として確定することである。過去記録の引継ぎは顧客（quality_admin）の判断による任意実施とし、強制要求しない。概要設計（DES-MIG-041〜050）が確定した LEGACY_IMPORT 方式・ケース A/B の分類・ALCOA+ 確保方針を、実施手順・完了判定の粒度まで具体化する。上流要件 INST-A2-g および MIG-034〜043 に対応する。

---

## 1. 本章の責務（INST-A2-g / MIG-034〜043 対応）

本節は、ロット履歴・過去記録引継ぎ実施計画の適用範囲・前提条件・完了定義を確定する。

### 1-1. 適用範囲

本計画は以下の過去記録を対象とする。

| 対象記録の種別 | 具体例 | 引継ぎ可否 |
|---|---|---|
| 電子化済み作業記録（Excel ファイル・CSV エクスポート） | 旧システムからの CSV エクスポート・Excel 台帳の作業完了記録 | ケース A（任意実施） |
| 紙原本のみ存在する作業記録 | 作業完了捺印済み紙記録票・日報 | ケース B（物理保管継続） |
| 直近 1 年超前の記録 | 1 年以上前に作成された全記録種別 | 移行対象外（MIG-003 による） |

本計画の適用開始は、移行計画/02（マスタ初期投入実施計画）の完了後とする。

### 1-2. 前提条件

本計画の実施前に以下をすべて充足していること。

| 前提 ID | 内容 | 確認者 |
|---|---|---|
| PRE-HIST-001 | 移行計画書（MIG-T-001）においてケース A またはケース B の選択が quality_admin により書面決定されていること | quality_admin |
| PRE-HIST-002 | マスタ初期投入（移行計画/02）が完了済みで quality_admin の承認を取得していること | quality_admin |
| PRE-HIST-003 | work_events テーブルの LEGACY_IMPORT イベント用フィールド（is_legacy / original_date / original_author / source_document 等）がスキーマに存在していること | system_admin |
| PRE-HIST-004 | ハッシュチェーンの初期ブロックが生成済みであること | system_admin |

### 1-3. 完了定義

| 完了条件 ID | 内容 | 確認者 |
|---|---|---|
| DONE-HIST-001 | ケース A の場合: 全 LEGACY_IMPORT イベントのハッシュチェーン連結が完了し、整合性チェックがエラー 0 件であること | system_admin |
| DONE-HIST-002 | ケース A の場合: ALCOA+ 確認チェックが全件合格していること | quality_admin |
| DONE-HIST-003 | ケース B の場合: 保管台帳が作成済みで quality_admin の承認を取得していること | quality_admin |
| DONE-HIST-004 | 選択したケース（A または B）の完了根拠が移行完了報告書に記載されていること | quality_admin |

---

**本節で確定した方針**
- 本計画の適用範囲を直近 1 年以内の過去作業記録とし、1 年超前の記録は移行対象外と判断する。
- 本計画の開始条件として移行計画/02（マスタ初期投入）の完了を必要条件として確定する。
- ケース A/B の選択は quality_admin の書面決定による一回限りの選択とし、選択後の変更を禁止することを確定する。

## 2. 引継ぎ対象と任意性の確定（MIG-X-036〜037）

本節は、引継ぎ対象の 2 分類を確定し、任意実施の法的根拠と手続きを明確化する。

### 2-1. ケース A / ケース B の 2 分類

**MIG-X-036**: 電子化済み記録（ケース A）と紙原本のみ記録（ケース B）の 2 分類を以下の通り確定する。（DES-MIG-041〜042 対応）

| 分類 | 対象記録の状態 | 対応方式 | 選択基準 |
|---|---|---|---|
| ケース A（電子化済み記録の LEGACY_IMPORT） | 旧システム CSV・Excel ファイルとして存在し、デジタルデータとして引継ぎが可能な記録 | work_events テーブルへの LEGACY_IMPORT イベント投入 | 監査証跡の継続性が顧客要件として必要な場合 |
| ケース B（紙原本の物理保管継続） | 紙記録票・日報のみ存在し、デジタル化されていない記録 | 旧記録を紙原本のまま物理保管する。本システムはカットオーバー日以降の記録のみを管理する | カットオーバー日を記録の起点とする場合 |

ケース A・ケース B の混在を許容する（一部の記録をケース A、残りをケース B とすることを認める）。混在する場合は、ケース別の記録種別・対象期間を移行計画書（MIG-T-001）に明記する。

ケース A とケース B の選択基準として以下を参考情報として提供する。

| 選択基準 | ケース A 優先 | ケース B 優先 |
|---|---|---|
| 規制環境 | ISO 9001 / IATF 16949 等で証跡の連続性が監査要件となっている | 規制による遡及義務なし |
| デジタルデータの品質 | 旧 Excel データが整合性・可読性ともに良好 | 旧 Excel データが断片的・欠損が多い |
| 引継ぎ工数 | 工数が確保できる | 引継ぎ工数の確保が困難 |

### 2-2. 顧客判断による任意実施の宣言

**MIG-X-037**: 過去記録引継ぎの任意実施を以下の通り確定する。（MIG-001 対応）

本システムは、過去記録の引継ぎをシステム利用開始の必須要件としない。カットオーバー日以前に作成された全記録の管理責任は、本計画の完了後も quality_admin が継続して負う。

任意実施の宣言手続きは以下の通りとする。

| 手続き | 内容 | 記録先 |
|---|---|---|
| ケース A 選択宣言 | quality_admin が「LEGACY_IMPORT 実施宣言書」に署名し、対象期間・対象記録種別・完了期限を記載する | 移行計画書（MIG-T-001）の付属書 A |
| ケース B 選択宣言 | quality_admin が「物理保管継続宣言書」に署名し、物理保管場所・保管責任者を記載する | 移行計画書（MIG-T-001）の付属書 B |
| 混在選択宣言 | 上記両書類を作成し、記録種別ごとの選択を明記する | 移行計画書（MIG-T-001）の付属書 A・B 両方 |

---

**本節で確定した方針**
- 引継ぎ対象を電子化済み記録（ケース A）と紙原本のみ記録（ケース B）の 2 分類とし、混在を許容することを確定する。
- ケース A/B の選択は quality_admin の書面決定による一回限りの選択とし、選択後の変更を禁止することを確定する。
- 過去記録引継ぎをシステム利用開始の必須要件としないことを確定する。

## 3. ケース A（電子化済み記録の LEGACY_IMPORT）の実施手順（MIG-X-038〜042）

本節は、ケース A を選択した場合の LEGACY_IMPORT イベント生成・ハッシュチェーン連結・フィールド記入ルールを確定する。

### 3-1. LEGACY_IMPORT イベントの生成手順

**MIG-X-038**: LEGACY_IMPORT イベントを work_events テーブルへ格納する手順を以下の通り確定する。（DES-MIG-043〜047 対応）

**準備フェーズ**

| 手順 | 内容 | 担当者 |
|---|---|---|
| 準備-1 | 旧 Excel・CSV ファイルを LEGACY_IMPORT 用 CSV フォーマット（後述）に変換する | master_admin |
| 準備-2 | 変換後 CSV の PRE バリデーションを実施する（PRE-001〜010 に準じる） | system_admin |
| 準備-3 | バリデーション結果（クリティカルエラー 0 件）を quality_admin に報告する | system_admin |
| 準備-4 | quality_admin が変換 CSV の内容を原本と照合し、10% サンプリングで正確性を確認する | quality_admin |

**インポートフェーズ**

| 手順 | 内容 | 担当者 |
|---|---|---|
| インポート-1 | LEGACY_IMPORT ツールを起動し、変換 CSV をアップロードする | system_admin |
| インポート-2 | プレビュー画面でインポート予定件数を確認する | system_admin |
| インポート-3 | quality_admin が「インポート実行」を承認する | quality_admin |
| インポート-4 | インポートを実行し、work_events テーブルへの格納件数を確認する | system_admin |
| インポート-5 | 格納後に POST チェック（行数一致・ハッシュチェーン整合性）を実施する | system_admin |

LEGACY_IMPORT 用 CSV フォーマットの必須カラムを以下の通り確定する。

| カラム名 | 型 | 必須 | 説明 |
|---|---|---|---|
| original_date | TEXT（YYYY-MM-DD または YYYY-MM-DD HH:MM:SS） | 必須 | 作業が実際に行われた日時 |
| original_author_name | TEXT | 必須（original_author_id が不明な場合） | 作業実施者の氏名 |
| operation_no | TEXT | 必須 | 作業番号（operations テーブルへの参照） |
| lot_no | TEXT | 必須 | ロット番号 |
| source_document | TEXT | 必須 | 原記録の識別情報（紙記録の整理番号・スキャンファイル名） |
| result | TEXT | 必須 | 作業結果（OK / NG / UNKNOWN） |
| remarks | TEXT | 任意 | 備考 |

### 3-2. ハッシュチェーンへの連結手順

**MIG-X-039**: LEGACY_IMPORT イベントのハッシュチェーン連結手順を以下の通り確定する。（DES-MIG-046〜047 対応）

LEGACY_IMPORT イベントは通常ブロックと同一のハッシュチェーンに連結する。チェーンの分岐は禁止する。

| 属性 | 設定値 |
|---|---|
| block_type | `'LEGACY'` |
| ハッシュ計算対象 | event の全フィールド（original_date・original_author 含む） |
| 前ブロックハッシュの参照 | 通常ブロックと同様に前ブロックの SHA-256 ハッシュを参照する |
| 改ざん検知対象 | 対象（LEGACY ブロックも通常ブロックと同様に改ざん検知の対象） |

インポート完了後に以下の順序でハッシュチェーン連結の整合性を確認する。

1. system_admin がハッシュチェーン整合性チェックバッチを実行する。
2. チェック結果が「エラー 0 件」であることを確認する。
3. エラーが発生した場合は即座にインポートをロールバックし、原因を調査する。
4. 整合性確認完了を audit_logs に記録する（action='LEGACY_CHAIN_VERIFIED'）。

### 3-3. original_date / original_author / source_document の記入ルール

**MIG-X-040**: LEGACY_IMPORT イベントの主要フィールドの記入ルールを以下の通り確定する。（DES-MIG-044〜045 対応）

**original_date の記入ルール**

| 状況 | 記入ルール |
|---|---|
| 紙記録に時刻まで記録されている | YYYY-MM-DD HH:MM:SS 形式で記入する |
| 紙記録に日付のみ記録されている | YYYY-MM-DD 形式（時刻は 00:00:00 とする）で記入し、time_precision='DATE_ONLY' を設定する |
| 紙記録に日付の記録がない | UNKNOWN を記入し、quality_admin に確認を依頼する。確認ができない場合はインポートしない |

**original_author の記入ルール**

| 状況 | 記入ルール |
|---|---|
| 原記録に作業者の記名があり、本システムのユーザーとして登録済み | original_author_id に users テーブルの user_id を設定する |
| 原記録に作業者の記名があるが、本システムに未登録 | original_author_name（TEXT）に氏名を記録する |
| 原記録に作業者の記名がない | original_author_name を '不明' と記入する |

**source_document の記入ルール**

| 状況 | 記入ルール |
|---|---|
| スキャン済みの PDF/画像ファイルが存在する | ファイル名（`LEGACY_<original_date>_<lot_no>_<seq>.pdf` 形式）を記入する |
| 紙原本のみで物理保管中 | 保管場所・台帳番号・棚番号等の物理的識別情報を記入する（例：「倉庫 A 棚 B-3 記録台帳 2025-06」） |

### 3-4. インポート対象期間の確定

**MIG-X-041**: LEGACY_IMPORT の対象期間を以下の通り確定する。（DES-MIG-043 対応）

対象期間は「カットオーバー日から遡って最大 1 年以内」とする。1 年超前の記録は本計画の適用外とし、物理保管継続とする。

| 属性 | 内容 |
|---|---|
| 対象期間の起点 | カットオーバー予定日から 1 年前の同日 |
| 対象期間の終点 | カットオーバー予定日の前日 |
| 期間変更の可否 | quality_admin の書面承認により最大 ±2 週間の調整を認める |

### 3-5. インポート完了判定

**MIG-X-042**: ケース A の完了判定を以下の通り確定する。

| 完了条件 | 内容 |
|---|---|
| A-DONE-001 | インポート件数が原記録件数と一致すること（POST-001 相当） |
| A-DONE-002 | ハッシュチェーン整合性チェックがエラー 0 件であること（POST-004 相当） |
| A-DONE-003 | ALCOA+ 確認チェック（MIG-X-046〜050）が全件合格であること |
| A-DONE-004 | quality_admin が LEGACY_IMPORT 完了報告書に署名していること |

---

**本節で確定した方針**
- LEGACY_IMPORT イベントは event_type='LEGACY_IMPORT'・is_legacy=true・block_type='LEGACY' を付与し、通常チェーンに連結することを確定する。
- チェーンの分岐を禁止し、LEGACY ブロックも改ざん検知の対象として確定する。
- original_date・original_author・source_document の記入ルールを本節の表の通り対応することを確定する。

## 4. ケース B（紙原本の物理保管継続）の実施手順（MIG-X-043〜045）

本節は、ケース B を選択した場合の物理保管継続手順・保管台帳の管理・免除根拠の記録を確定する。

### 4-1. 保管台帳の作成・管理

**MIG-X-043**: ケース B 選択時の保管台帳の作成・管理方法を以下の通り確定する。（DES-MIG-048 対応）

quality_admin は紙記録の物理保管台帳を以下の仕様で作成・管理する。

| 台帳フィールド | 内容 |
|---|---|
| 台帳 ID | LEGACY-BINDER-NNN 形式（NNN は連番） |
| 記録種別 | 作業記録票・日報・検査記録等の種別 |
| 対象期間（開始） | 当台帳が管理する記録の最古日付（YYYY-MM-DD） |
| 対象期間（終了） | 当台帳が管理する記録の最新日付（YYYY-MM-DD） |
| 保管場所 | 保管棚・キャビネットの物理的な場所（建屋・フロア・棚番号） |
| 件数 | 当台帳が管理する記録件数 |
| 保管責任者 | quality_admin の氏名 |
| 廃棄予定日 | 法定保存期間に基づく廃棄予定日 |

台帳は Excel ファイルとして管理し、quality_admin が専任で管理する。年次で更新・棚卸しを実施する。

物理保管の環境要件を以下の通り確定する。

| 環境要件 | 内容 |
|---|---|
| 保管場所 | 施錠管理された専用保管室または鍵付きキャビネット |
| 湿度・温度 | 紙記録の劣化を防ぐ環境（温度 15〜25°C・湿度 40〜60%） |
| アクセス制限 | quality_admin および system_admin のみアクセス可 |
| 保存期間 | 最短 7 年（法定要件・ISO 9001 記録管理要求に基づく） |

### 4-2. LEGACY 免除根拠の記録

**MIG-X-044**: ケース B 選択時の LEGACY_IMPORT 免除根拠の記録を以下の通り確定する。

免除根拠記録は移行計画書（MIG-T-001）の付属書 B に以下を明記する。

| 記録項目 | 内容 |

| 対象記録の概要 | 対象期間・記録種別・おおよその件数 |
| 物理保管場所 | 保管台帳 ID・保管場所 |
| 監査対応方針 | 「カットオーバー日以前の記録は紙原本が正源であることを監査員に提示する」旨を明記 |
| 承認者 | quality_admin の署名・日付 |

### 4-3. ケース B 完了判定

**MIG-X-045**: ケース B の完了判定を以下の通り確定する。

| 完了条件 | 内容 |
|---|---|

| B-DONE-002 | 保管台帳の内容と物理的な紙記録の整合性が確認済みであること（台帳 ID と物理ファイルの突き合わせ） |
| B-DONE-003 | LEGACY 免除根拠記録が移行計画書の付属書 B に記載・署名済みであること |
| B-DONE-004 | 移行完了報告書にケース B 選択である旨と物理保管台帳の参照先が記載されていること |

---

**本節で確定した方針**

- 物理保管は施錠管理された専用保管室または鍵付きキャビネットで 7 年以上保存することを確定する。
- LEGACY 免除根拠を移行計画書付属書 B に quality_admin の署名付きで記録することを確定する。

## 5. 移行データの ALCOA+ 確認（MIG-X-046〜050）

本節は、ケース A で格納した LEGACY_IMPORT データに対する ALCOA+ コンプライアンス確認の手順・合否基準・非適合管理を確定する。

### 5-1. original_date / original_author / source_document の検証

**MIG-X-046**: ALCOA+ コンプライアンス確認の対象フィールドと検証手順を以下の通り確定する。（DES-MIG-049〜050 対応）

ALCOA+ 確認は LEGACY_IMPORT 投入完了後に system_admin が自動バッチを実行し、結果を quality_admin が確認する。

| ALCOA+ 要素 | 確認対象フィールド | 自動チェック内容 | 合格基準 |
|---|---|---|---|
| Attributable（帰属性） | original_author_id または original_author_name | どちらか一方が非 NULL であること | 全件で帰属者が記録されていること |
| Legible（可読性） | 全 TEXT フィールド | 文字化けパターン（?・□・〓）の非存在 | 文字化けが 0 件であること |
| Contemporaneous（同時性） | original_date | NULL でないこと・YYYY-MM-DD 形式または YYYY-MM-DD HH:MM:SS 形式であること | 全件で original_date が記録されていること |
| Original（原本性） | source_document | NULL でないこと | 全件で source_document が記録されていること |
| Accurate（正確性） | 全フィールド | 10% サンプリング目視確認（POST-003 相当） | サンプリング確認で不一致 0 件であること |
| Complete（完全性） | 全必須フィールド | NOT NULL チェック | 全件で必須フィールドが非 NULL であること |
| Consistent（一貫性） | original_date・imported_at | original_date が imported_at 以前の日時であること | 全件で original_date ≤ imported_at であること |
| Enduring（永続性） | is_legacy フラグ | is_legacy=true が全 LEGACY_IMPORT レコードに設定されていること | 全件で is_legacy=true であること |
| Available（参照可能性） | is_legacy フィルター | is_legacy フィルターによる専用ビューで全 LEGACY_IMPORT レコードが検索可能であること | 全件が検索可能であること |

**MIG-X-047**: ALCOA+ 確認のバッチ実行手順を以下の通り確定する。

1. system_admin が ALCOA+ 確認バッチスクリプトを実行する。
2. バッチが上記 9 要素の自動チェックを実行し、結果レポートを生成する。
3. system_admin が結果レポートを quality_admin に提出する。
4. quality_admin が結果レポートを確認し、合格/不合格を判定する。
5. 合格の場合は「ALCOA+ 確認完了報告書」に quality_admin が署名する。
6. 不合格の場合は MIG-X-048 に定める ALCOA+ 非適合管理手順を適用する。

### 5-2. ALCOA+ 非適合として管理する旨の記録

**MIG-X-048**: ALCOA+ チェックで不合格となった LEGACY_IMPORT データの管理方針を以下の通り確定する。

LEGACY_IMPORT データへの ALCOA+ 適用は現実的に完全な達成が困難な場合がある。この理由は、過去の紙記録が ALCOA+ 設計を念頭に作成されていないためである。

不合格項目の管理方針を以下の通り確定する。

| ALCOA+ 不合格パターン | 対応方針 |
|---|---|
| original_author が '不明' の記録が存在する | ALCOA+ 非適合として記録する。原本から確認できた場合のみ遡及修正を許容する |
| time_precision='DATE_ONLY' の記録が存在する | ALCOA+ の Contemporaneous 要素が部分的非適合であることを記録する。対応不要 |
| source_document が台帳参照のみ（物理ファイルなし）の記録が存在する | ALCOA+ の Original 要素が部分的非適合であることを記録する。紙原本の物理保管を継続することで代替する |

**MIG-X-049**: ALCOA+ 非適合記録の形式を以下の通り確定する。

ALCOA+ 非適合として管理する記録は、専用の「LEGACY ALCOA+ 非適合管理簿」に以下の内容を記録する。

| 記録フィールド | 内容 |
|---|---|
| 非適合 ID | ALCOA-NC-NNN 形式（NNN は連番） |
| 対象レコード ID | work_events テーブルの event_id |
| 非適合要素 | ALCOA+ の 9 要素のうち非適合の要素名 |
| 非適合内容 | 具体的な非適合の内容（例：「original_author_name が '不明'」） |
| 対応方針 | 「対応不要（記録のみ）」または「遡及修正を実施する」 |
| 遡及修正実施者 | 修正実施者の username |
| 遡及修正日 | YYYY-MM-DD |
| 承認者 | quality_admin の署名 |

### 5-3. 引継ぎ完了判定

**MIG-X-050**: ケース A のロット履歴・過去記録引継ぎの最終完了判定を以下の通り確定する。

| 完了条件 | 内容 |
|---|---|
| FINAL-001 | ALCOA+ 自動チェックバッチの結果レポートが生成済みであること |
| FINAL-002 | ALCOA+ 非適合管理簿が作成済みで、全非適合項目に対応方針が記載されていること |
| FINAL-003 | quality_admin が「ALCOA+ 確認完了報告書」に署名していること |
| FINAL-004 | ハッシュチェーン整合性チェックがエラー 0 件であること |
| FINAL-005 | 移行完了報告書にケース A 完了の記載と ALCOA+ 非適合管理簿の参照先が記載されていること |

---

**本節で確定した方針**
- ALCOA+ 確認の 9 要素（Attributable / Legible / Contemporaneous / Original / Accurate / Complete / Consistent / Enduring / Available）を自動バッチで確認し、結果を quality_admin が確認することを確定する。
- ALCOA+ 完全達成が困難な記録は LEGACY ALCOA+ 非適合管理簿に ALCOA-NC-NNN 形式で記録し、対応方針を quality_admin が承認することを確定する。
- ALCOA+ 確認完了報告書への quality_admin の署名を引継ぎ完了の必要条件として確定する。
