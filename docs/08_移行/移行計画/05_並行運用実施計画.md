# 05 並行運用実施計画

本章の責務は、Phase M-2（並行運用フェーズ）の実施を計画として確定することである。概要設計（DES-MIG-002〜003）が確定した並行運用方式・期間定義・終了条件を受けて、開始条件・日次フロー・データ正源の取り扱い・終了判定手続きを具体化する。本計画は Phase M-1（移行準備）の完了後に開始し、Phase M-3（カットオーバー）へ接続する。上流要件 INST-A2-c・A4.5 および MIG-047〜052 に対応する。

---

## 1. 本章の責務（INST-A2-c + A4.5 / MIG-047〜052 対応）

本節は、並行運用実施計画の適用範囲・前提条件・完了定義を確定する。

### 1-1. 適用範囲

本計画は以下の全活動を対象とする。

| 活動 | 内容 |
|---|---|
| 並行運用開始宣言 | quality_admin による Phase M-2 開始の書面宣言 |
| 日次並行運用フロー | 毎日の紙運用とシステム運用の同時実施・エラーログ確認 |
| 週次指標集計と報告 | 週次の作業完了率・エラー率の集計と quality_admin への報告 |
| 並行運用期間管理 | 終了条件の達成状況の継続監視 |
| データ正源の確定 | システム記録を正源とし、紙記録をバックアップとする宣言 |
| 終了判定 | quality_admin による Go/No-Go 判断と Phase M-3 への接続 |

### 1-2. 前提条件

| PRE-M2-002 | 移行計画/04（移行リハーサル実施計画）のリハーサル完了宣言が取得済みであること（CO-PRE-004 充足） | quality_admin |

| 完了条件 ID | 内容 | 確認者 |

**本節で確定した方針**

- Phase M-1 の完了・リハーサル完了宣言・全員訓練完了・全端末登録完了を並行運用開始の必要条件として確定する。
- 前提条件 PRE-M2-001〜006 の全充足なしに並行運用を開始することを対象外と判断する。

## 3. 並行運用期間の確定（MIG-X-054〜056）

本節は、並行運用期間の最短・最長・終了条件・延長条件・延長判断権限を確定する。

### 3-1. 最短 2 週間・最長 4 週間の根拠

**MIG-X-054**: 並行運用期間を最短 2 週間・最長 4 週間として以下の通り確定する。（DES-MIG-003 対応）

| 期間 | 根拠 |
|---|---|
| 最短 2 週間 | 週次集計を 2 回実施して終了条件の安定充足を確認するために必要な最小期間。工場の作業サイクル（週単位）に基づく |
| 最長 4 週間 | 4 週間を超える二重運用（紙とシステムの同時実施）はオペレーターの負担が許容限界を超える。最長 4 週間制限はリスク MIG-003（オペレーター混乱）への対策 |

最長 4 週間の制限は quality_admin の判断によってのみ延長できる（後述の延長条件を参照）。system_admin 単独での延長判断は禁止する。

### 3-2. 終了条件の確定

**MIG-X-055**: 並行運用の終了条件を以下の通り確定する。（DES-MIG-003 対応）

| 終了条件 ID | 条件 | 計測方法 |
|---|---|---|
| M2-END-001 | システム側の作業完了率が 95% 以上であること | 並行期間中の全作業件数のうち、work_events テーブルに記録された作業件数の割合を週次集計で計算する |
| M2-END-002 | システム操作エラー率が全作業件数の 2% 未満であること | 並行期間中の全作業件数のうち、スキップ・強制終了・異常終了のイベント件数の割合を週次集計で計算する |
| M2-END-003 | 未解決の重大バグ（Severity: Critical / High）が 0 件であること | system_admin がバグ管理ツールを確認する |

3 条件はすべてを同時に充足することが終了条件として確定する。1 条件でも未充足の場合は終了条件を充足したものとしない。

終了条件の判定タイミングを以下の通り確定する。

| 判定タイミング | 内容 |
|---|---|
| 週次定期判定 | 各週の終わり（金曜日）に週次集計を実施し、終了条件の達成状況を確認する |
| 臨時判定 | 3 条件すべてを週次集計前に充足したと system_admin が判断した場合、quality_admin に臨時判定を依頼できる |

### 3-3. 延長条件と quality_admin の判断権限

**MIG-X-056**: 並行運用の延長条件と延長判断権限を以下の通り確定する。

| 延長条件 | 内容 |
|---|---|
| M2-EXT-001 | 最長 4 週間到達時に M2-END-001〜003 のいずれかが未充足の場合 |
| M2-EXT-002 | 最長 4 週間到達時に解決中の重大バグが残存する場合 |
| M2-EXT-003 | quality_admin が業務判断上、追加の並行運用期間が必要と判断した場合 |

延長の判断権限は quality_admin に専属する。延長判断の手続きを以下の通り確定する。

| 手順 | 内容 |
|---|---|
| 手順 1 | system_admin が最長期間到達の 3 日前に、終了条件の充足状況と未充足条件の改善見通しを quality_admin に報告する |
| 手順 2 | quality_admin が延長の要否を判断する |
| 手順 3 | 延長する場合は「並行運用延長宣言書」に署名し、延長期間（1 週間単位）と再判定日を記載する |
| 手順 4 | 延長しない場合（終了条件未充足のまま）は quality_admin がリスクを受け入れた上でカットオーバー判断を行う。理由と影響評価を移行計画書に記録する |

延長は 1 週間単位とし、最大 2 回（合計 6 週間）まで延長を認める。6 週間を超える延長は system_admin・quality_admin・経営層の三者協議を必要とする。

---

**本節で確定した方針**
- 並行運用期間を最短 2 週間・最長 4 週間として確定する（DES-MIG-003 準拠）。
- 終了条件を作業完了率 95% 以上・エラー率 2% 未満・重大バグゼロの 3 条件同時充足として確定する。
- 延長判断権限を quality_admin に専属させ、延長手続きとして「並行運用延長宣言書」への署名を義務付けることを確定する。

## 4. 並行運用中の日次フロー（MIG-X-057〜062）

本節は、並行運用中の毎日の実施フロー・担当者・記録方法を確定する。

### 4-1. 朝礼での操作状況周知

**MIG-X-057**: 朝礼での操作状況周知を以下の通り確定する。

| 項目 | 内容 |
|---|---|
| 実施タイミング | 毎朝礼（生産開始前） |
| 実施者 | quality_admin または line_leader（quality_admin が不在の場合） |
| 周知内容 | 前日のシステム操作エラー件数・前日の作業完了率（速報値）・当日の注意事項 |
| 実施方式 | 口頭での周知（3 分以内）。前日のエラー数が 5 件以上の場合は原因と対処を補足説明する |

前日のエラーが 0 件の場合も朝礼での簡単な確認を実施する（「昨日のエラーはゼロ件でした」の一言報告を省略しない）。

### 4-2. 紙運用とシステム運用の同時実施

**MIG-X-058**: 並行運用中の紙運用とシステム運用の同時実施ルールを以下の通り確定する。

| ルール | 内容 |
|---|---|
| システム記録の先行 | 作業完了時はシステム（ハンディ端末）で先に記録し、その後に紙記録票に記入する |
| システム記録の失敗時 | ハンディ端末のシステム記録が失敗した場合は紙記録票に記入し、line_leader に報告する |
| 紙記録票の保管 | 並行運用中の紙記録票は 1 日分をまとめてファイルし、quality_admin が保管する |
| 不一致の報告 | システム記録と紙記録票の内容が不一致の場合は、line_leader を通じて system_admin に当日中に報告する |

オペレーターへの共通指示事項を以下の通り確定する。

| 指示事項 | 内容 |
|---|---|
| ハンディ端末優先 | 作業完了後はまずハンディ端末でシステム記録を行う |
| 操作不明時の対処 | 操作がわからない場合は放置せず、line_leader または quality_admin に即座に質問する |
| エラー発生時の報告 | システムエラー（赤いメッセージが表示された場合等）は内容をメモして line_leader に報告する |

### 4-3. 日次エラーログ確認

**MIG-X-059**: 日次エラーログ確認の手順を以下の通り確定する。

| 項目 | 内容 |
|---|---|
| 実施タイミング | 毎日終業後（16:00〜17:00 の間） |
| 実施者 | system_admin |
| 確認内容 | アプリケーションエラーログ・PostgreSQL エラーログ・作業記録の異常終了件数 |
| 記録方法 | 「日次エラーログ確認シート」（Excel）に確認日時・エラー件数・内容を記入する |

日次エラーログ確認で以下の状況が発生した場合は即座に quality_admin に報告する。

| 即時報告条件 | 内容 |
|---|---|
| Critical バグの発生 | アプリケーションクラッシュ・DB 接続断・データ損失 |
| 当日エラー件数 50 件超過 | 1 日のシステム操作エラーが 50 件を超えた場合（オペレーター混乱の兆候） |
| ハッシュチェーン整合性エラー | 日次整合性チェックバッチでエラーが検出された場合 |

### 4-4. 週次作業完了率・エラー率の集計と報告

**MIG-X-060**: 週次集計の手順を以下の通り確定する。（MIG-T-012〜013 対応）

| 項目 | 内容 |
|---|---|
| 実施タイミング | 毎週金曜日の終業後（16:30〜17:30） |
| 実施者 | system_admin（集計）・quality_admin（報告受領・判定） |
| 集計対象期間 | 当週月曜日 00:00〜金曜日 23:59 |

週次集計の計算方法を以下の通り確定する。

| 指標 | 計算方法 |
|---|---|
| 作業完了率 | （work_events テーブルの当週作業件数）/（当週の紙記録票件数 + work_events 件数）× 100 |
| エラー率 | （当週のエラーイベント件数）/（当週の全作業件数）× 100 |
| 重大バグ件数 | system_admin がバグ管理ツールを確認し、Severity Critical/High のオープン件数を集計する |

**MIG-X-061**: 週次報告書の形式を以下の通り確定する。

| フィールド | 内容 |
|---|---|
| 報告書タイトル | 「並行運用週次報告書 Week N（N は並行運用開始からの週番号）」 |
| 対象期間 | YYYY-MM-DD〜YYYY-MM-DD |
| 作業完了率（当週） | XX.X% |
| エラー率（当週） | X.X% |
| 重大バグ件数（累積） | N 件 |
| 終了条件達成状況 | M2-END-001 / M2-END-002 / M2-END-003 それぞれの充足状況（充足/未充足） |
| 主なエラー内容 | 上位 3 件のエラー内容と対処状況 |
| 次週の注意事項 | system_admin からの注意事項・改善策 |
| 作成者 | system_admin の username・日時 |

週次報告書は quality_admin に提出し、quality_admin が終了条件の達成状況を確認する。

**MIG-X-062**: 週次報告書の保管を以下の通り確定する。

週次報告書は移行計画ドキュメント管理フォルダに「M2_週次報告_Week_N.xlsx」形式で保管する。保管期間は移行完了後 3 年以上とする。

---

**本節で確定した方針**
- 朝礼での前日エラー件数・作業完了率速報値の周知を daily に実施することを確定する。
- 日次エラーログ確認を system_admin が毎日終業後に実施し、Critical バグ・50 件超過・チェーン整合性エラーの発生時は即時報告することを確定する。
- 週次集計（毎週金曜日）により作業完了率・エラー率・重大バグ件数を計測し、quality_admin への週次報告書提出を義務付けることを確定する。

## 5. 並行運用中のデータ正源の確定（MIG-X-063〜064）

本節は、並行運用期間中のデータ正源（システム記録 vs 紙記録）の取り扱いと、紙記録の保管ルールを確定する。

### 5-1. システム記録を正源とする宣言

**MIG-X-063**: 並行運用中のデータ正源を以下の通り確定する。

並行運用期間中、システム（work_events テーブル）に記録された作業記録を「正源（Single Source of Truth）」とする。紙記録票はシステム記録のバックアップとして位置づけ、システム記録との整合性確認に使用する。

| 状況 | データ正源 | 対処 |
|---|---|---|
| システム記録・紙記録の両方が存在する | システム記録が正源 | 内容の整合性を確認する |
| システム記録のみ存在する（紙記録なし） | システム記録が正源 | 記録の欠損として問題票に記録し、quality_admin に報告する |
| 紙記録のみ存在する（システム記録なし） | 紙記録を暫定正源とする | line_leader が理由を確認し、system_admin が事後的にシステムへ手動入力する。system_admin・quality_admin の共同承認を得た上で手動入力を実施する |
| システム記録と紙記録の内容が不一致 | システム記録を正源とする | 不一致の原因を system_admin が調査し、quality_admin に報告する。紙記録の誤りであれば紙記録に訂正を記入する |

システム記録を正源とする宣言書を quality_admin が並行運用開始宣言書に含める形で記録する。

### 5-2. 紙記録の保管ルール

**MIG-X-064**: 並行運用中の紙記録の保管ルールを以下の通り確定する。

| 保管ルール項目 | 内容 |
|---|---|
| 保管単位 | 1 日分の紙記録票を 1 ファイルとしてまとめる（ファイル名：紙記録_YYYY-MM-DD） |
| 保管責任者 | quality_admin |
| 保管場所 | 施錠管理された専用保管室または鍵付きキャビネット |
| 保管期間 | 並行運用終了後、カットオーバー完了確認まで（最低 4 週間）。その後は移行計画/03（ロット履歴引継ぎ実施計画）のケース B に基づく物理保管継続方針を適用する |
| 廃棄禁止期間 | カットオーバー完了宣言前の廃棄を禁止する |

---

**本節で確定した方針**
- 並行運用期間中のデータ正源をシステム記録（work_events テーブル）として確定する。
- 紙記録はシステム記録のバックアップとして位置づけることを確定する。
- 紙記録のみ存在する場合は事後的な手動入力を system_admin・quality_admin の共同承認により実施することを確定する。

---

## 6. 並行運用終了判定（MIG-X-065）

本節は、並行運用終了判定の手続き・Go/No-Go 基準・Phase M-3 への接続を確定する。

### 6-1. 3 条件充足後の Go/No-Go 判断

**MIG-X-065**: 並行運用終了の Go/No-Go 判断手続きを以下の通り確定する。（MIG-T-015 対応）

**Go 判断の手続き**

| 手順 | 内容 | 担当者 |
|---|---|---|
| 手順 1 | system_admin が週次集計で終了条件 M2-END-001〜003 の全充足を確認する | system_admin |
| 手順 2 | system_admin が「並行運用終了条件達成報告書」を作成し、quality_admin に提出する | system_admin |
| 手順 3 | quality_admin が報告書の内容を確認し、追加確認事項があれば system_admin に要求する | quality_admin |
| 手順 4 | quality_admin が「Go」を宣言し、「並行運用終了・カットオーバー移行承認書（MIG-T-015）」に署名する | quality_admin |
| 手順 5 | 署名完了後、Phase M-3（カットオーバー実施計画）の CO-PRE-002 が充足されたものとして扱う | quality_admin |
| 手順 6 | system_admin が並行運用終了を audit_logs に記録する（action='M2_END'） | system_admin |

**No-Go 判断の手続き**

| 状況 | 対応 |
|---|---|
| 終了条件が未充足の場合 | quality_admin が「No-Go」を宣言し、並行運用を継続する。延長手続き（MIG-X-056）に従う |
| 最長 4 週間到達時に未充足の場合 | MIG-X-056 の延長判断を行い、延長または終了条件未充足のままのカットオーバー移行を quality_admin が判断する |
| 新たな重大バグが発生した場合 | M2-END-003 が即座に「未充足」に戻る。修正完了後に再度全条件を確認する |

**Go/No-Go 判断の記録**

| 記録項目 | 記録先 |
|---|---|
| Go 宣言日時 | 「並行運用終了・カットオーバー移行承認書（MIG-T-015）」 |
| 終了時の作業完了率 | 同上 |
| 終了時のエラー率 | 同上 |
| 重大バグ件数 | 同上 |
| 次工程（カットオーバー）予定日 | 同上 |

### 6-2. Phase M-3（カットオーバー実施計画）への接続

並行運用終了・カットオーバー移行承認書（MIG-T-015）の署名をもって、Phase M-3（カットオーバー実施計画）への移行が開始される。カットオーバー実施の詳細は移行計画/06（カットオーバー実施計画）を参照する。

Phase M-3 開始にあたり、以下の条件が充足済みであることを改めて確認する。

| 条件 | 根拠 |
|---|---|
| CO-PRE-001 Phase M-1 の全タスク完了 | MIG-T-001〜010 の完了確認書 |
| CO-PRE-002 Phase M-2 の完了条件充足 | 並行運用終了・カットオーバー移行承認書（MIG-T-015） |
| CO-PRE-004 カットオーバーリハーサルの完了 | 移行リハーサル完了宣言書（MIG-RH-020） |

CO-PRE-003（未解決重大バグゼロ）は Phase M-3 開始時点で system_admin が最終確認する。

---

**本節で確定した方針**
- 終了条件 M2-END-001〜003 の全充足後、quality_admin が「並行運用終了・カットオーバー移行承認書（MIG-T-015）」に署名することで Go 判断が成立することを確定する。
- Go 判断をもって Phase M-3 の CO-PRE-002 が充足されたものとして扱うことを確定する。
- 新たな重大バグ発生時は M2-END-003 が即座に未充足に戻ることを確定する。
