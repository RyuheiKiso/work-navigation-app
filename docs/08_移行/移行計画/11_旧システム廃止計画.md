# 11 旧システム廃止計画

本章の責務は、IPA 共通フレーム 2013「INST-A6 旧運用の凍結・撤去・記録保管」に対応し、カットオーバー後の旧システム（紙 SOP・Excel ファイル・紙原本記録）の位置づけ・物理保管継続・アクセス制限・LEGACY_IMPORT 取込済みデータの参照保持を確定することである。廃棄の実施は D-MIG-003 の対象外宣言に従い工場側の業務判断とし、本書は廃棄手順を定めない。本章は CON-MIG-005（カットオーバー後も紙原本は物理保管を継続する）と D-MIG-003（旧システムの廃棄手順は工場管理者の業務判断に帰属する）の整合を明示する。MIG-X-099〜105 を採番する。

---

## 1. 本章の責務（INST-A6 対応）と D-MIG-003 との整合（廃棄判断は工場側）

### 1-1. INST-A6 カバレッジ

本章は IPA 共通フレーム 2013 INST-A6「旧運用の凍結・撤去・記録保管」の以下の実施命題をカバーする。

| INST-A6 要素 | 本章での対応節 | 対応状況 |
|---|---|---|
| 旧運用の凍結宣言 | Phase M-3（MIG-T-018）の参照 | 対応する（MIG-T-018 で確定済み） |
| 旧記録媒体のアクセス制限 | §2（紙 SOP フォールバック）・§3（Excel 読取専用） | 対応する |
| 旧記録の物理保管継続 | §4（紙原本の物理保管） | 対応する |
| 廃棄手順の設計 | 対象外（D-MIG-003 による） | 対象外と判断する |

### 1-2. D-MIG-003 との整合の明示

D-MIG-003「旧システム（紙・Excel）の廃棄手順は工場管理者の業務判断に帰属し、本システムの設計範囲外である」は、`04_概要設計/09_移行方式設計/99_前提制約と本書が約束しないこと.md` で確定した対象外宣言である。

本章（移行計画/11）はこの対象外宣言を継承し、以下を明示的に確認する。

| 本章が確定する範囲 | 本章が確定しない範囲（D-MIG-003 対象外） |
|---|---|
| 紙 SOP のフォールバック用途への限定 | 紙 SOP の廃棄タイミング |
| Excel ファイルの読取専用設定 | Excel ファイルの削除手順 |
| 紙原本の物理保管継続と保管台帳の整備 | 紙原本の廃棄・シュレッダー処理手順 |
| LEGACY_IMPORT データの source_document 参照保持 | 廃棄後の情報セキュリティ対応手順 |

本章でカットオーバー後の旧記録の取り扱いを確定することは、廃棄手順を設計することではなく、廃棄するまでの間の適切な管理状態を確定することである。廃棄の実施時期・方法は quality_admin が工場内部規定・法的保管義務に従って判断する。

廃棄判断の参照先は D-MIG-X-001 を参照すること。D-MIG-X-001 は本章の §4-3 で記述する。

---

**本節で確定した方針**
- INST-A6 の対応状況を確定し、旧記録の廃棄手順は D-MIG-003 の対象外宣言として本書のスコープ外であることを確定する。
- 本章が確定する範囲を「廃棄前の管理状態の確定」に限定し、廃棄の実施判断を quality_admin の業務判断に帰属させることを確定する。
- D-MIG-003 との整合を §1-2 で明示し、本章の全節が廃棄手順を含まないことを確定する。

## 2. 旧 SOP（紙）のフォールバック用途転換

### 2-1. カットオーバー後の紙 SOP の位置づけ（フォールバック専用）

**MIG-X-099**: カットオーバー完了後（MIG-T-020 完了後）の紙 SOP の位置づけを「フォールバック専用」として確定する。「フォールバック専用」とは、システム障害・電源断・WiFi 全断等の緊急事態においてシステムが使用不能となった場合のみ参照を許可する運用であることを確定する。

フォールバック発動の条件は以下のとおりとする。

| フォールバック発動条件 | 承認者 |
|---|---|
| Rust バックエンドプロセスの停止（再起動不能状態） | quality_admin または system_admin |
| PostgreSQL への接続全断（全ハンディ端末から接続不能） | quality_admin |
| WiFi 全 AP 停止による全端末のオフライン化 | quality_admin または現場監督 |
| その他 quality_admin が業務継続上必要と判断した緊急事態 | quality_admin |

フォールバック中に参照した紙 SOP に基づいて実施した作業は、システム復旧後 24 時間以内にシステムへの遡及入力を実施し、work_events テーブルに記録することを確定する。

### 2-2. 「フォールバック用途限定」ラベル付与手順

**MIG-X-100**: カットオーバー当日（MIG-T-018 の旧運用凍結宣言後）に、以下の手順で全紙 SOP に「フォールバック用途限定」ラベルを付与することを確定する。

1. quality_admin が「フォールバック用途限定」の印刷ラベル（赤文字・A6 サイズ）を準備する
2. system_admin が現場の全紙 SOP の保管場所を巡回し、各 SOP の表紙に付与する
3. 付与完了後、quality_admin が巡回確認を実施し、ラベル付与漏れがないことを確認する
4. 確認完了を移行完了記録に記録する

ラベルの記載内容は以下のとおりとする。
- 「フォールバック用途限定 — システム障害時のみ参照可」
- 「最終印刷日: YYYY-MM-DD（バージョン: X.X）」
- 「通常運用での使用禁止」

### 2-3. 紙 SOP の週次更新スケジュール（印刷日時記録）

**MIG-X-101**: フォールバック用の紙 SOP を最新の電子 SOP と同期させるため、以下の週次更新手順を確定する。紙 SOP の週次更新は、カットオーバー後 90 日間（§10-5-4 の 90 日事後レビュー期間中）を対象とし、事後レビューで紙 SOP 管理の継続要否を quality_admin が判断することを確定する。

| 週次更新ステップ | 実施者 | 実施タイミング |
|---|---|---|
| 電子 SOP の更新有無確認（PostgreSQL の SOP テーブルの更新日チェック） | system_admin | 毎週月曜 8:00 |
| 更新があった SOP の最新版印刷（A4・カラー） | master_admin | 毎週月曜 9:00〜11:00 |
| 印刷した SOP の印刷日時記録（保管台帳の「最終印刷日」欄更新） | master_admin | 印刷完了後即時 |
| 旧印刷版を「旧版」スタンプ押印のうえ廃棄対象フォルダに移動 | master_admin | 新版差替えと同時 |
| quality_admin による更新確認 | quality_admin | 毎週月曜 11:00〜12:00 |

---

**本節で確定した方針**
- カットオーバー後の紙 SOP をフォールバック専用（MIG-X-099）と定め、システム障害時以外の参照を禁止することを確定する。
- 「フォールバック用途限定」ラベルをカットオーバー当日に全紙 SOP に付与する手順（MIG-X-100）を確定する。
- 週次更新スケジュール（MIG-X-101）でフォールバック用紙 SOP を電子 SOP と同期させる義務を確定し、更新記録の保管を master_admin の義務として確定する。

## 3. 旧 Excel ファイルの読取専用設定

### 3-1. CL-008 と連動した Excel ファイルのアクセス制限

**MIG-X-102**: カットオーバー完了後 3 営業日以内に、以下の旧 Excel ファイルに読取専用設定を施すことを確定する。読取専用設定の実施者は system_admin とし、quality_admin が完了を確認することを確定する。

対象 Excel ファイルの分類:

| 分類 | 対象ファイルの例 | 読取専用設定方法 |
|---|---|---|
| 作業指示書（SOP）の Excel 版 | `SOP_工程名_v*.xlsx` 形式のファイル | ファイルサーバーのアクセス権を「読取のみ」に変更 |
| 作業証跡・記録台帳の Excel 版 | `作業記録_YYYYMM_*.xlsx` 形式のファイル | ファイルサーバーのアクセス権を「読取のみ」に変更 |
| スキル・資格管理台帳の Excel 版 | `スキル台帳_*.xlsx` 形式のファイル | ファイルサーバーのアクセス権を「読取のみ」に変更 |

読取専用設定の確認方法: system_admin が各ファイルに対して「上書き保存」操作を実施し、拒否されることを確認する。確認結果を設定完了記録に記録する。

### 3-2. ファイルサーバーでの保管継続

**MIG-X-103**: 読取専用設定を施した旧 Excel ファイルはファイルサーバーの専用フォルダ（`/archive/legacy_migration/YYYYMM_cutover/` 等の命名規約に従う）に移動し、物理削除しないことを確定する。ファイルサーバーでの保管継続期間は、CON-MIG-005 の「カットオーバー後も旧記録は物理保管を継続する」に準拠し、quality_admin が廃棄判断を行うまで継続する。

保管継続の確認は以下の頻度で実施する。

| 確認頻度 | 確認者 | 確認内容 |
|---|---|---|
| カットオーバー後 1 週間 | system_admin | アーカイブフォルダへの移動完了・読取専用設定の維持確認 |
| 90 日事後レビュー時 | quality_admin | 保管継続の要否判断（廃棄判断はスコープ外） |

---

**本節で確定した方針**

- 読取専用設定済みの旧 Excel ファイルをファイルサーバーのアーカイブフォルダに移動し、物理削除を禁止することを確定する（MIG-X-103）。
- 旧 Excel ファイルの廃棄判断を D-MIG-003 に従いスコープ外として確定し、quality_admin の業務判断に帰属させることを確定する。

## 5. LEGACY_IMPORT 取込済みデータの source_document 参照保持

### 5-1. source_document フィールドに旧記録の識別子を保持する確認手順

**MIG-X-105**: LEGACY_IMPORT 方式で取り込んだ過去記録について、work_events テーブルの source_document フィールドに旧記録の識別子（紙記録票の通し番号・Excel ファイル名・シート名等）が保持されていることをカットオーバー後 1 週間以内に確認することを確定する。

確認手順:

1. system_admin が以下の SQL クエリで source_document フィールドの設定状況を確認する

```sql
SELECT
  COUNT(*) AS total_legacy,
  COUNT(source_document) AS with_source,
  COUNT(*) - COUNT(source_document) AS without_source
FROM work_events
WHERE event_source = 'LEGACY_IMPORT';
```

2. `without_source` の件数が 0 件であることを確認する
3. 件数が 0 件の場合は確認完了として移行完了記録に記録する
4. 件数が 0 件でない場合は system_admin が原因を特定し、quality_admin に報告したうえで是正処置を実施する

source_document フィールドの保持確認が重要な理由は、CON-MIG-005 の前提「旧記録廃棄後は LEGACY_IMPORT の source_document 参照が機能しない」に対する先行確認であるためである。紙原本が物理保管されている期間中に source_document の整合性を確認し、将来の監査対応に備える。

source_document の識別子として保持する情報の推奨形式:

| 旧記録種別 | 識別子の推奨形式 |
|---|---|
| 紙記録票 | 「{工程コード}-{記録票通し番号}-{作業日}」（例: A01-00123-20250315） |
| Excel 記録 | 「{ファイル名}#{シート名}#行{行番号}」（例: 作業記録_202503.xlsx#3月分#行42） |
| 紙台帳 | 「{台帳名}-P{ページ番号}-{作業日}」（例: スキル台帳-P12-20250315） |

---

**本節で確定した方針**
- LEGACY_IMPORT 取込済みデータの source_document フィールド保持確認（MIG-X-105）をカットオーバー後 1 週間以内に実施する義務を system_admin に確定する。
- source_document フィールドの欠損（without_source > 0）を是正処置の対象として確定し、quality_admin への報告義務を確定する。
- source_document 識別子の推奨形式を定め、将来の監査対応時の参照可能性を確保することを確定する。

## 6. 旧システムの最終状態記録

### 6-1. カットオーバー時点のデータバックアップを証跡として保管

カットオーバー時点の旧システム（PostgreSQL 初期投入後・並行運用完了後）の状態を証跡として保管することは、MIG-T-016（カットオーバー前バックアップの取得）で確定済みである。

本節はそのバックアップの長期保管要件を確定する。

カットオーバー時点の PostgreSQL フルバックアップ（MIG-T-016 で取得）を以下の要件で保管することを確定する。

| 保管要件 | 内容 |
|---|---|
| 保管期間 | 最低 7 年（移行完了宣言日を起点） |
| 保管媒体 | 外付けストレージ（HDD または NAS）+ クラウドストレージ（二重化） |
| ファイル命名規約 | `pg_dump_cutover_YYYYMMDDHHMMSS.sql.gz` 形式 |
| 整合性確認 | SHA-256 ハッシュ値をバックアップ取得時に計算し、別ファイル（`pg_dump_cutover_YYYYMMDDHHMMSS.sha256`）に記録する |
| アクセス制限 | system_admin のみ読取可能（質担当は要申請） |
| 保管場所記録 | 移行完了宣言書の添付資料として保管場所・ファイル名・ハッシュ値を記録する |

SHA-256 ハッシュ値の記録は、将来の移行完了証跡の真正性確認（ALCOA+「Attributable」「Original」の観点）において必須である。保管期間中に 2 年に 1 回、ハッシュ値の再計算によりバックアップファイルの破損有無を確認することを確定する。

---

**本節で確定した方針**
- カットオーバー時点のバックアップを 7 年以上保管する義務を確定し、外付けストレージとクラウドの二重化保管を確定する。
- SHA-256 ハッシュ値を取得時に記録し、2 年に 1 回の再計算による完全性確認を義務として確定する。
- バックアップの保管場所・ファイル名・ハッシュ値を移行完了宣言書に添付することを確定する。
