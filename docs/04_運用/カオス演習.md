# カオス演習手順

> 対応 §: ロードマップ §13.4.2 §11.6 §22 §27 §29
> 対象読者: メンテナ、半期演習担当
> 改訂サイクル: §22.1 半期サイクル（演習自体の妥当性）／演習自体は半期実施

製造現場での突発障害（ネットワーク断・ストレージ満杯・電源断）は **常時発生** する想定。本書は §13.4.2 で挙げた 8 シナリオの **半期演習手順** を版管理する。Principles of Chaos Engineering（Netflix, 2014）の手法をオンプレ前提に縮約適用する。

## 1. 演習サイクル

- **頻度**: §22.1 半期サイクル（年 2 回）。
- **期間**: 各シナリオ 30 分〜2 時間。
- **記録先**: 本書の「実施記録」セクションに追加。
- **失敗時**: §22.4 是正フロー起動／§27 FMEA 行追加／§29 リスク登録簿更新。

## 2. 8 シナリオ（§13.4.2 整合）

### S-01: 端末ネットワーク断（10s〜24h）

- **注入手段**: `tc qdisc add dev <iface> root netem loss 100%` または Wi-Fi トグル。
- **検証**: §10.6 同期復旧。Lamport ts 単調性、`Inv_NoEventLoss`（INV-01）の維持。
- **手順**:
  1. 端末で 10 件の作業実績を作成。
  2. ネットワーク断を 10s／1h／24h で順次注入。
  3. 復旧後、サーバ側 G-Set に 10 件全数が到達することを確認。
- **合格基準**: §10.6.2 受入観点。

### S-02: サーバ DB 接続断

- **注入手段**: `iptables -A OUTPUT -p tcp --dport 5432 -j DROP` または `docker pause <pg-container>`。
- **検証**: §10.3.1 リトライ・デッドレター。
- **手順**:
  1. クライアント API から実績送信 100 件。
  2. DB 接続を断。
  3. 指数バックオフでリトライ／24h でデッドレター遷移を確認。
- **合格基準**: §10.3.5 受入観点。

### S-03: ストレージ満杯

- **注入手段**: `fallocate -l <残容量-1MB> /tmp/dummy` で残容量を縮減。
- **検証**: §10.4.4 警告挙動（80% で警告／90% でブロック）。
- **手順**:
  1. ストレージを段階的に消費して 80%／90% を超える状況を作る。
  2. 各しきい値で警告 UI／管理者通知の発火を確認。
- **合格基準**: §10.4.6 受入観点。

### S-04: 時刻ジャンプ

- **注入手段**: `timedatectl set-time` または NTP 強制ずれ。
- **検証**: §20.2 NTP ズレ閾値（5 秒、§21 注 4）検出。
- **手順**:
  1. 端末を ±10 秒ずらす。
  2. 同期保留通知が発火することを確認。
  3. 監査ログに端末時刻・サーバ時刻両方が記録されていることを確認。
- **合格基準**: §20.2／§11.4.1 改ざん耐性。

### S-05: 高 CPU 負荷

- **注入手段**: `stress-ng --cpu N --cpu-load 90`。
- **検証**: §5.2 性能維持（タッチ応答 ≤ 100ms、画面遷移 ≤ 200ms）。
- **手順**:
  1. 端末／サーバに CPU 負荷を注入。
  2. 主要画面の応答時間を計測。
- **合格基準**: §5.4 受入観点（CI 計測）。

### S-06: メモリ逼迫

- **注入手段**: `cgroup memory.limit_in_bytes` の縮小、または `stress-ng --vm`。
- **検証**: OOM Killer 後の自動復旧（systemd Restart=on-failure 相当）。
- **手順**:
  1. メモリ上限を縮める。
  2. アプリが OOM される。
  3. 自動再起動と直前状態への復帰を確認（§11.6 受入観点）。
- **合格基準**: §11.6 受入観点。

### S-07: アドオン暴走

- **注入手段**: テスト用無限ループアドオン（`while(true) {}`）を投入。
- **検証**: §17.5 サンドボックス／kill 検証。
- **手順**:
  1. テスト用アドオンを `wnaddon` 形式で配布・インストール。
  2. アドオンが Wasmtime の CPU／実行時間上限（既定 100ms）を超えて kill されることを確認。
- **合格基準**: §17.7 受入観点／§27 F-004。

### S-08: 電源断（端末）

- **注入手段**: バッテリ強制切断（実機）。
- **検証**: §11.6 直前状態復帰。
- **手順**:
  1. 作業中の端末を強制電源断。
  2. 再起動後、SQLite WAL からの復旧を確認。
  3. 直前のステップ進捗・撮影メディアが失われないことを確認。
- **合格基準**: §11.6 受入観点。

## 4. 失敗時のフロー

1. 失敗詳細を Issue 化（ラベル `regression`）。
2. §27 FMEA 行を追加または更新（検出度 D を経験的に更新）。
3. §29 リスク登録簿の関連行（R-007／R-008／R-015／R-016 等）を更新。
4. 修正 PR を起票し、再演習で緑になることを確認。

## 5. 受入観点（§13.4.3）

- 8 シナリオ全種が再現手順付きで本書に記載されていること。
- 半期に 1 回の実施記録が本書に存在すること。
- 検出された脆弱箇所が §22.4 是正フローおよび §27 FMEA 行追加で起票されていること。
- §29 リスク登録簿の優先度 ≥ 12 リスクと本書のシナリオが双方向参照されていること。
