# リスク登録簿

> 対応 §: ロードマップ §29 §22 §27 §19.5
> 出典: ISO 31000:2018 ／ IEC 62198:2013
> 対象読者: メンテナ、ガバナンス担当
> 改訂サイクル: §22.1 半期サイクル／優先度 ≥ 12 は必読項目

ISO 31000:2018 / IEC 62198:2013 のリスクマネジメント原則に従い、本プロジェクトのリスクを構造化された登録簿で管理する。§22 改訂サイクル／§27 FMEA と相補的（FMEA は技術故障に焦点、本書はプロジェクト・運用・市場・規制リスクを広く扱う）。

## 1. 評価軸（§29.1 再掲）

| 軸 | 値域 | 定義 |
| --- | --- | --- |
| 発生確率 | 1（稀）〜5（ほぼ確実） | 12 ヶ月以内 |
| 影響度 | 1（軽微）〜5（致命的） | プロジェクト継続・OSS 公開・規制適合に対する影響 |
| 優先度 | 確率 × 影響 | ≥ 16 を High、≥ 9 を Medium、それ未満を Low |
| 対応 | 回避／軽減／転嫁／受容 | ISO 31000 |

## 4. リスク追跡（§29.3）

- 各リスクは **オーナー**（メンテナ）と **再評価期日** を持つ（本表参照）。
- 優先度 ≥ 12 のリスクは §22.1 サイクル必読項目（R-001／R-003／R-009／R-011／R-015／R-016）。
- 新規発生は GitHub Issue（ラベル `risk-new`）→ 評価 → 登録。
- 対応完了したリスクは「クローズ済」へ移し、過去 5 期分は履歴として残す。
- §27 FMEA の AP=H 行と本表の優先度 ≥ 16 行は CI で双方向リンク整合性を検査（CI 整備後）。

## 6. 受入観点（§29.4）

- 全 17 行に対応／オーナー／再評価期日が記載されていること（達成）。
- 優先度 ≥ 16（High）のリスクが未対応のまま 1 期超持ち越されないこと（R-001 が初期 1 件、来期で再評価）。
- §22.1 半期サイクルでリスク登録簿のレビュー実施記録が GitHub Discussions に存在すること。
- §27 FMEA と本書が相互参照され、重複・矛盾しないこと（達成）。
- リスク区分 12 種の網羅性を年次で見直すこと。
