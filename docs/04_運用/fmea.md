# FMEA 完全表

> 対応 §: ロードマップ §27 §27.3 §27.5 §22.4 §29 §31
> 出典: AIAG-VDA FMEA Handbook 1st Ed. (2019) ／ IEC 60812:2018 Annex C ／ IATF 16949 §8.3
> 対象読者: メンテナ、品質レビュア、規制対応担当
> 改訂サイクル: §22.1 半期サイクル／PR で機能追加時に必須更新

製造業は IATF 16949 Section 8.3 で **FMEA（Failure Mode and Effects Analysis）** を要請しており、本アプリ自身および本アプリが介在する作業フローの両面で FMEA を扱う。本書は §27.3「代表故障モード（初期登録）」を含む **完全表** を版管理する。AP=H 行は **リリースブロッカー**。

## 1. 評価軸（§27.2 再掲）

| 構成要素 | 値域 | 規定 |
| --- | --- | --- |
| Severity（影響度 S） | 1〜10 | 10 ＝人身傷害／製造ライン停止／規制違反 |
| Occurrence（発生頻度 O） | 1〜10 | 10 ＝高頻度／予防策なし |
| Detection（検出可能性 D） | 1〜10 | 10 ＝検出不能（事後にしか分からない） |
| Action Priority | H／M／L | AIAG-VDA 2019 マトリクス |
| RPN（移行期参考） | S × O × D | 1〜1000 |

## 6. 是正措置の連鎖（§27.4）

- 各 FMEA 行は §22.4 是正フローと **双方向 ID リンク**（FMEA ID ↔ Issue ID）。
- AP=H で未対応の行はリリース前に対応必須（リリースブロッカー）。
- §13.4.2 カオス演習は FMEA 行の検出度（D）を経験的に下げるための実証手段。
- §29 リスク登録簿の R-008（暗号鍵漏洩）／R-015（端末紛失）／R-016（同期破壊）は FMEA F-008／F-002 と双方向参照。
- §31.1 SLI と本表の双方向参照: SLI-06 ↔ F-002／R-016、SLI-09 ↔ F-004。

## 7. 受入観点（§27.5）

- DFMEA／PFMEA／SFMEA 各層に最低 5 行が登録されている（本書で達成）。
- AP=H の行 100% に対応 Issue／PR が紐付く（公開前は TBD でも、リリース時には必須）。
- §22.1 サイクルで FMEA 全体の RPN／AP 分布を再評価。
- 新機能追加 PR は対応する FMEA 行追加が必須（PR テンプレ項目）。
- IATF 16949 Section 8.3 を導入企業が満たす際の参照資料として `examples/fmea-iatf16949/` に AP=H 例の汎用化版を同梱（コード初期実装後）。

## 9. 行追加・削除規約

- **追加**: PR テンプレートで「FMEA 影響あり」を Yes にした PR が本書を更新。`docs/04_運用/fmea.md` への行追加が必須。
- **削除**: ADR（§9.4）に削除理由を残す。AP=H 行の削除は §22.4 是正フローと §29 リスク登録簿の同期更新を経る。
- **更新**: S／O／D の値を変更する場合、根拠（カオス演習結果・現場報告等）を備考欄に記載する。
