## 2. 至高への姿勢と「至高」KPI

「至高」は固定された到達点ではなく、**継続改善の姿勢** である。完璧という静的状態は本質的に存在しない（評価基準は時間とともに移ろい、競合・規制・技術は変動し、ドメイン理解は深化し続ける）。

したがって本ロードマップは、

- **静的な完成図ではなく、改訂を前提とした生きた文書** として運用する。
- 改訂サイクル・是正トリガー・撤退条件は §22 に明文化する。
- 「現時点の最善」と「未来の見直し対象」を区別し、根拠の弱い決定は §21 で出典・脚注を残す。

これに反する **沈黙の妥協**（決めずに済ます／根拠を残さない／検証しない）を本プロジェクトでは禁止する。

### 2.2. 沈黙の妥協検出プロセス

§2 で禁止した「沈黙の妥協」を、レビュー時に **機械的に** 検出する。レビュアの裁量と注意力に依存しない仕組みを敷くことで、人的疲労による検出漏れを防ぐ。

1. **PR テンプレート必須項目**: 「決定に出所があるか／受入観点があるか／§24.2 に記載済か」の 3 項を必須チェック項とする。チェック未記入の PR は CI で fail。
2. **`silent-compromise` ラベルの自動付与**: チェック項のいずれかが No の PR には GitHub Action（`actions/silent-compromise.yml`）が自動付与する。
3. **CI のリンターによる強制**: 「初期決定」「採用する」等の決定語を含む新規追記行に対し、参照（§番号または出典 URL）が無い場合は CI で fail（`scripts/lint-rationale.sh`）。
4. **改訂サイクル時の集計**: §22.1 サイクルで本ラベル件数を集計し、ゼロでなければ §22.4 是正フロー起動。集計結果は §2.1 KPI に直接反映。

### 2.3. 撤退と再起の判断（§22.3 への先行宣言）

「妥協禁止」と「無謀な継続」を分離するため、§22.3 で扱う撤退条件を本章で先行宣言する。撤退は失敗ではなく、より高い至高への迂回として扱う。

- 同一軸 3 期連続の「△」未解消／規制対応コストの吸収不能／競合の OSS 化による差別化消失のいずれかで、その軸を撤退または再戦略化する。
- 撤退決定は CHANGELOG・GitHub Discussions に **公開ポストモーテム** として記録する。隠蔽を許さない。
- 撤退した軸の代替候補は §4.4 「昇格候補」または §4.9 「圧倒候補機能」から繰り上げる。

## 3. 「作業」と「ナビゲーション」の概念モデリング（中核）

メモが反復強調する最重要論点（メモ L9 / L11 / L17）。設計の前提となるため第一級項目として扱う。「作業」（§3.1〜§3.5）と「ナビゲーション」（§3.6）はそれぞれ独立した中核概念であり、本アプリの全機能はこの二つの上に組み上げられる。

### 3.1. 「作業」の正式定義と困難さの本質

#### 3.1.1. 正式定義

本書において「作業」とは、**特定の文脈（製品・設備・時間・場所・組織）下で、明示された開始条件を満たして開始し、明示された完了条件を満たして終了する、状態を持った活動の最小単位** である。作業は階層的に組成可能であり、上位作業は下位作業の有限列・並行・選択・反復で構成される。

| 構成要素 | 定義 |
| --- | --- |
| 識別 | グローバル一意な ID と人間可読な名称 |
| 文脈 | 製品・ロット・設備・場所・時間・組織が紐付く |
| 開始条件 | 形式的に検査可能な前提（先行作業完了・部材有・権限有 等） |
| 完了条件 | 形式的に検査可能な事後条件（写真・値・署名・自動判定） |
| 状態 | 有限状態機械上の状態（§3.3「状態」観点で定義） |
| 入出力 | 物・情報・記録の I/O 契約 |
| 主体 | 実行する人間または機械の identity と権限 |
| 時間属性 | 標準時間／許容時間／タイムアウト |
| 例外属性 | エラー・割込み・順序逸脱の扱い |
| 心理属性 | 認知負荷・誤操作可能性・疲労・心理的安全 |
| 階層 | 上位作業／下位作業のリンク |

この定義は §10.1 ナビゲーション、§10.2 フロー設定、§10.3 基幹連携、§10.6 同期、§17 アドオン API のすべてが共有する **ユビキタス言語の基底** である。

#### 3.1.2. 困難さの本質

「作業」概念は次の理由で困難である:

- **文脈依存**: 同一手順でも製品・設備・時間帯で意味が変わる。
- **暗黙知**: 熟練者が言語化しないコツ・予兆検知が含まれる。
- **主観性**: 完了判定が個人の判断に依存することがある。
- **多階層性**: 工程／手順／動作／微細動作 の入れ子構造を持つ。
- **例外性**: 標準フローよりも例外対応のほうが価値の差を生む。

#### 3.1.3. 困難さ × 検討観点 写像

困難さの解消責任を §3.3 検討フレームの 9 観点へ写像する。「対応観点が空白の困難さ」「困難さに対応しない観点」が無いことを §3.5 受入観点で検証する。

| 困難さ | 一次対応観点 | 補助観点 | 設計対応 |
| --- | --- | --- | --- |
| 文脈依存 | 文脈 | 主体・時間 | コンテキスト属性を作業定義の必須要素に昇格／フロー条件分岐 |
| 暗黙知 | 検証 | 心理・例外 | 完了条件の形式化／メディア記録／熟練者ヒューリスティクスのポカヨケ化 |
| 主観性 | 検証 | 主体 | 完了判定を写真／値／自動判定へ移し主観依存を最小化 |
| 多階層性 | 粒度 | 入出力・状態 | 階層モデル（工程／手順／動作）と状態機械の入れ子 |
| 例外性 | 例外 | 状態・時間 | 例外フローを第一級の状態遷移として表現／割込みキューの可視化 |

これらを正面から扱うため、後述 §3.2 のステークホルダー識別、§3.3 の検討フレーム、§3.4 の分析方法論を採る。

#### 3.1.4. 「作業」ではないものの境界（負定義）

「作業」を肯定的に定義するだけでは輪郭が曖昧になる。本アプリが扱わない（または別概念で扱う）ものを明示することで、scope creep（§26 アンチゴール集の主目的）を未然に防ぐ。

| 非作業（または別扱い） | 理由 | 別概念での扱い |
| --- | --- | --- |
| 単発の問い合わせ・連絡 | 状態を持たない／完了条件が形式化困難 | コミュニケーション機能（§17 アドオンで吸収） |
| 設備の自動運転中の機械処理 | 主体が機械のみ／人間ナビ不要 | 監視（§9.3.1 自働化）／例外発生時に作業へ昇格 |
| 行政手続・教育研修 | 製造現場の生産活動と直接連結しない | OOS（out of scope）。導入企業の別システムへ |
| 経営判断・KPI 集計 | 個別作業ではなく集約結果 | レポート（§17 アドオン） |
| 個人の私的タスク | 業務外／監査対象外 | 非対応 |
| 順序情報そのものの計画 | 基幹システムの責務（§10.3） | §10.3 で受領するのみ |
| 設備保全の計画立案 | 別ドメイン（CMMS／EAM） | §17 アドオンで連携 |
| 経理・人事・在庫管理の独立業務 | ERP のコア責務 | 連携対象だが本アプリの責務外 |

これらは §10 機能要件のスコープから外れる。スコープイン要望が来た場合は §22.2 即時トリガーで再評価し、コアに入れず §17 アドオン化が原則。アンチゴール集（§26）と双方向参照する。

#### 3.1.5. 「作業」というフレーズの言語学的・歴史的分析（メモ L17 の運用化）

メモ L17「『作業』とは？ というフレーズを **細かく分析** したうえでアプリを作成する」は、§3.1.1（形式定義）／§3.1.2（困難さ）／§3.1.4（負定義）と並ぶ第四の切り口、すなわち **語そのものの輪郭** を扱う。語の意味は文化・国・時代・業界で揺らぎ、UI 文言・コード命名・ユビキタス言語（§28）の決定に直接影響する。本節は §3.1.5.5 受入観点で §28 と双方向に同期する。

##### 3.1.5.1. 漢字構成の分解

| 字 | 訓 | 含意 | アプリでの解釈 |
| --- | --- | --- | --- |
| 作 | つく（る）／な（す） | 動作の実行・産出（外形的変化を起こす） | 状態を変える行為であること。参照・閲覧のみは作業ではない |
| 業 | わざ／ごう | 反復・専門・蓄積（修練を伴う） | 標準化・反復可能であること。一回限りのアドホック行為は §3.1.4 で除外した「単発の問い合わせ・連絡」と一致 |

「作業」は **「（標準的に反復される）行為としての産出」** を含意する。これは §3.1.1「明示された開始条件・完了条件を満たす最小単位」および §9.3.1 標準作業（大野耐一 1978）と整合する。逆に「行為」「働き」など産出を含意しない語は採用しない（§28 同義語禁止規約）。

##### 3.1.5.2. 多言語対比（§11.3.1 拡張順との整合）

| 言語 | 該当語 | ニュアンスの差 | 本アプリでの扱い |
| --- | --- | --- | --- |
| 英語 | Task | 短期・割当可能・状態を持つ | API 名（§17.3 `getCurrentTask`）と一致 |
| 英語 | Operation | 工程的・反復的・MES 文脈 | 「工程内処理」に限定（§28 で語義固定） |
| 英語 | Work | 広義・労働そのもの | 狭義の「作業」とは混同しない（§28 同義語禁止） |
| 英語 | Activity | ISA-95 で活動単位 | 規格適合時の参考のみ。UI で多用しない |
| 独語 | Arbeitsschritt | 「作業ステップ」直訳 | 動作（Step）の独語訳の参考 |
| 独語 | Vorgang | 連続した手順／プロセス | 手順（Procedure）の独語訳の参考 |

§11.3.1 言語拡張時に本表を更新し、§28 翻訳辞書（`docs/02_設計/glossary-<locale>.md`）と同期する。

##### 3.1.5.3. 「作業」と隣接概念の境界

| 隣接概念 | 「作業」との違い | アプリでの扱い |
| --- | --- | --- |
| 仕事（しごと） | 役割・職責に含まれる行為総体 | 「作業」は仕事の構成単位の一つ。本アプリは仕事の管理を目的としない |

| 行為（こうい） | 法的・倫理的な意味を含む動作 | アプリ用語として不採用 |
| 加工（かこう） | 物質変化を伴う作業 | 七つのムダ（§9.3.2）の「加工」とは概念が重なるが、本書では作業の一形態 |

##### 3.1.5.4. 業界別の「作業」語感の揺れ

メモ L7 が想定する競合（Tulip／Augmentir 等）が「作業」の語感を業界横断で曖昧に扱うことが多いため、本アプリでは業界別語感を明示し、§10.3 アダプタ／§12 規制適合表で吸収する。

| 業界 | 「作業」の典型語感 | 対応 |
| --- | --- | --- |
| 自動車 | タクトに刻まれた標準作業（IATF 16949 §8.5.1） | §3.1.1 完了条件・標準時間が必須／§5.2 タクト計測 |

#### 3.2.1. 代表ペルソナと 1 日の動線（メモ L19 の具体化）

§3.2 の抽象的なステークホルダー識別を、メモ L33-35 制約下で実在人物への聴取なしに具体化するため、公開資料（OSHA／IATF 公表資料／JILPT 製造業労働調査／日本生産性本部資料／ILO 製造業統計）の動線を合成して代表ペルソナを構築する。出典は §18.2 の制約に従う。

##### 3.2.1.1. 作業者ペルソナ「A さん」（自動車部品ライン）

- **背景**: 勤続 3 年。所属班 6 名。手袋着用必須。3 シフト制（早番／遅番／夜勤）。
- **1 日の動線**:
  - 出勤 → タイムカード → 作業着 → 朝礼（5 分） → 端末ログイン（オフライン継続中）。
  - 当日順序情報を表示 → 設備チェック表（10 ステップ） → 作業開始。
  - 例外（部材切れ・設備異常）発生 → アンドン押下 → 班長応援待ち。
  - 休憩 1 → 復帰時に「中断状態」から自動復帰（§10.6 受入観点）。
  - 終業前: 実績送信確認 → ログオフ → 引継メモ。
- **心理的ストレス上位**: 不確実性（次操作）／時間圧（タクト）／失敗コスト（不良流出）。
- **本アプリの対応**: 段階的開示（§9.2.2）／中断・再開（§10.1）／ポカヨケ（§9.3.1）。

##### 3.2.1.2. 班長ペルソナ「B さん」（医薬品包装ライン）

- **背景**: 勤続 8 年。班 12 名。GMP 教育済。
- **1 日の動線**:
  - 朝礼 → 班員配置 → 当日 KPI 共有 → 設定 UI で当日フローを確認。
  - 端末「班長監視モード」で全員の進捗・アンドン受信。
  - 不適合発生 → 是正措置記録 → 品質ホールド通知（§10.3.2）。
  - 終業: 進捗集計 → 引継書作成（テンプレート同梱）。
- **心理的ストレス上位**: 認知負荷（同時 12 人監視）／自律性喪失（強制ダイアログ）。
- **本アプリの対応**: 一覧 UI／アンドン即時通知／確認の段階化（§9.2.2）。

##### 3.2.1.3. 生産技術ペルソナ「C さん」（食品工場・HACCP）

- **背景**: 勤続 12 年。フロー設計担当。CCP 管理。
- **1 日の動線**:
  - 設定 Web UI でフロー編集 → 差分プレビュー → 試行版発行 → 当該班でパイロット運用 → 本番昇格。
  - HACCP CCP の温度・時間記録の自動化条件追加。
  - 月次: ロールバック演習・バージョン互換性確認。
- **心理的ストレス上位**: 自律性喪失（誤った設定の影響範囲）／失敗コスト（ライン停止）。
- **本アプリの対応**: 差分管理／バージョニング／影響範囲のリアルタイム計算（§3.6.2）。

##### 3.2.1.4. システム部門ペルソナ「D さん」（電子部品メーカー）

- **背景**: 勤続 5 年。社内システム保守。アドオン開発担当。
- **1 日の動線**:
  - 観測ダッシュボード（§16）で前日異常確認 → アドオン更新リハーサル → 本番反映。
  - 内製基幹システムとの連携アドオン保守（§10.3.4／§17）。
- **心理的ストレス上位**: 失敗コスト（全社影響）／認知負荷（複数アドオン同時管理）。
- **本アプリの対応**: capability 最小権限（§17.4）／アドオン独立更新／監査ログ。

##### 3.2.1.5. 受入観点

- 4 ペルソナ × 1 日動線が §13 E2E シナリオに 1 対 1 で対応する（テスト ID で双方向トレース）。
- 各ペルソナの心理的ストレス上位 3 軸が §9.2.1 5 軸に写像可能。
- ペルソナは §22.1 年次サイクルで出典文献を更新。
- 新ペルソナの追加は §22.4 是正フローを通る。

### 3.4. 分析方法論（人間インタビュー禁止下で）

メモ L33-35 により実在する人間へのインタビューは禁止される。代替として以下を採る:

1. **公開文献調査**: 産業工学（IE）、人間工学、認知科学、Lean／Six Sigma、トヨタ生産方式（大野耐一ほか）の文献を読解し、共通概念を抽出。
2. **既存 SOP の解体分析**: 公開されている作業手順書（OSHA、IATF 公表資料、教育機関の演習資料）を分解し、共通構造を導出。
3. **動画観察**: 公開されている製造現場動画（YouTube・MOOC）を作業ステップ単位に分解、状態遷移を抽出。
4. **競合製品のドメインモデル抽出**: 競合のヘルプ・API 仕様から彼らの「作業」モデルを推定し、欠落を識別（§4 と接続）。
5. **規格・標準の参照**: ISA-95（manufacturing operations management）、ISA-88（batch control）、ISO 22400（製造業 KPI）、ISO 9241（人間工学）からモデル要素を取り込む。
6. **形式化**: 状態機械・Petri net で「作業」を形式化し、矛盾を検出。性質ベーステストで網羅検証。

#### 3.4.1. 形式化 notation の確定（メモ L17 を運用化）

§3.4-6「状態機械・Petri net で『作業』を形式化」を、選択した notation で具体化する。沈黙の妥協（§2）を防ぐため、初期決定として固定し §22.1 で再評価する。

| 対象 | notation | 採用理由 | 不採用案 |
| --- | --- | --- | --- |
| 単一作業の状態 | **階層型有限状態機械（HSM, Harel statecharts 1987）** | 中断・再開・例外を入れ子で表現可能／PlantUML／sismic で図と検証を統一 | UML state machine（HSM の方が階層・並行を簡潔） |
| 並行・同期・順序情報の流れ | **着色 Petri net（CPN, Jensen & Kristensen 2009）** | 並行・資源・カラー（属性）を一体表現／CPN Tools／Access/CPN でプロパティ検証可能 | プレース変遷ネット（色情報なしで属性表現が困難） |
| データ・整合性 | **TLA+（Lamport 1999）** | 同期競合・原子性・前進性（liveness）を網羅検証／TLC／Apalache で機械検証 | Alloy（バウンデッド過ぎる） |
| 動的トポロジ（プロセス代数） | π計算／CSP は **不採用** | アプリ層に動的チャネル生成中心の処理がないため過剰 | — |

形式化成果物は `docs/03_設計/形式化/` に配置（HSM 図／CPN モデル `.cpn`／TLA+ `.tla`）。CI（`tlc` および CPN Tools のバッチ検証）で常時緑を維持する。

#### 3.4.2. 形式化の受入観点

- §3.1.1 の 11 構成要素のうち「状態」「例外」「時間」「入出力」が CPN モデル上で表現される。
- TLA+ で「作業実績は同期競合下でも消失しない」を不変式 `Inv_NoEventLoss`、「LWW は決定的」を `Inv_LWWDeterministic` として記述・検証。
- HSM の各状態が §10.1 受入観点へトレース可能（ステートマシン状態 ID とテスト ID の双方向リンク）。
- §22.1 半期サイクルでモデル化漏れを検出し、ロードマップへ反映。

### 3.5. 受入観点

- §3.1.1 の正式定義の 11 構成要素すべてが、コード上のドメインモデル型（Aggregate／Entity／Value Object）に対応する。
- §3.1.3 困難さ × 観点写像表に空白セルが無く、双方向に欠落が無い。
- §3.3 の 9 観点すべてについて用語定義と関係性を明文化したドメインモデル文書が存在する。
- §3.4 の方法論 6 種それぞれで、参照した文献・資料・規格のリストが残されている。
- ドメインモデルが UI／API／DB スキーマの 3 層に矛盾なく写像できる。
- ドメインモデルに対して単体・性質ベーステストが存在し、状態遷移網羅率を測定可能。
- §3.2 のステークホルダーごとに、対応する機能要件・E2E シナリオ・心理的ストレス評価（§9.2.1）が紐付いている。

### 3.6. 「ナビゲーション」概念のモデリング

メモ L9「関係者の作業をナビゲーションする」はプロダクト名そのものを規定する記述である。「作業」と同水準の正式定義を与えなければ、設計判断が個々人の語感に委ねられ §2 の「沈黙の妥協」を招く。よって独立サブセクションで定義する。

#### 3.6.1. 正式定義

「ナビゲーション」とは、§3.2 のステークホルダーが §3.1 の作業を実行・監督・改善するにあたり、**現在地・目的地・経路・進捗・例外・再計算結果** を、認知負荷を最小化して即時に把握できる状態を継続的に維持するための、情報提示と動的計算の総体である。地図アプリ・カーナビが「空間」を扱うのに対し、本アプリは「作業状態空間」を扱う。

#### 3.6.2. ナビゲーションの 6 基本機能

| 機能 | 説明 | 端末アプリ側の表現 | 設定 UI 側の表現 |
| --- | --- | --- | --- |
| 現在地 | 当該主体が今どの作業階層・状態に居るか | ヘッダ＋パンくず／工程ハイライト | 編集中ノードのハイライト |
| 目的地 | 直近の完了条件と最終ゴール | ステップカード／完了条件チェックリスト | フロー終端マーカ |
| 経路 | 現在地→目的地までの予定遷移列 | ステップマップ／推奨パス強調 | フローエディタの主経路強調 |
| 進捗 | 通過済／残量／所要時間 | 進捗バー／残時間／達成率 | バージョン差分の進捗表示 |
| 警告・例外 | 逸脱・期限・リスクの即時可視化 | アンドン／ハイライト／音 | 検証エラー表示 |
| 再計算 | 文脈・例外発生に応じた経路の再描画 | 例外フローへの即時遷移／代替手順提示 | 影響範囲のリアルタイム計算 |

#### 3.6.3. ナビゲーションの粒度

作業階層（工程／手順／動作）と同型に階層化する。ユーザーは任意の階層へズームイン／ズームアウト可能であり、ステークホルダーごとの **既定ズーム** を §3.2 と紐付ける（作業者: 手順／動作、班長: 工程／手順、生産技術: 工程全体、品質管理: 完了条件・例外集約）。

#### 3.6.4. アンチパターンの禁止

「ナビゲーション」を冠する以上、次のアンチパターンを設計時に排除する。

| アンチパターン | 兆候 | 設計対応 |
| --- | --- | --- |
| 過剰提示 | 1 画面に未来 5 ステップ以上を同時表示 | 段階的開示（§9.2.2） |
| 不足提示 | 「次に何をすればよいか」を画面外で問い合わせる必要 | 次アクションの単一化（§9.2.1 不確実性軸） |
| 固定経路 | 例外発生時も標準経路を表示し続ける | 例外フローを第一級遷移として §3.3「例外」観点へ紐付け |
| 不適応 | 文脈（製品・設備・時間帯）変化を反映しない | 文脈属性の変化検知 → §3.3「文脈」観点でフィルタ更新 |
| 経路ロックイン | 一度逸脱すると標準経路へ戻る術が無い | 「リカバリ経路」を必ず提示／中断・再開を §10.1 受入観点に含める |
| 過剰計算 | 軽微変更で全画面再描画 | 差分再計算と局所更新の徹底 |

#### 3.6.5. 受入観点

- 主要画面で 6 基本機能すべての要素が UI 監査チェックリスト上で識別可能。
- ステップ逸脱検知から再計算後の経路提示までを **1 秒以内**（§5.2 画面遷移目標と整合）。
- §3.6.4 のアンチパターンを §22 サイクルで監査し、検出時は §22.4 是正フローへ。
- ステークホルダー別の既定ズーム階層が設定 UI で構成可能。
- 6 機能を §10.1 ナビゲーション機能の単体・E2E テストへ写像可能（テスト ID で双方向トレース）。

### 4.4. 戦略選択（初期決定）

「全 11 軸でトップ」は妥協のリスクが高い。以下のように分離する。

- **圧倒的優位（◎を狙う）7 軸**: 非エンジニア操作性／オフライン対応／対応端末／拡張性・アドオン／セットアップ容易性／ライセンス・TCO／規制適合。設定 UI を作業 UI と同水準にする方針（メモ L45）から、非エンジニア操作性も ◎ に位置付ける。
- **同等以上を維持（○以上）残り 4 軸**: ナビ粒度／ERP・MES 連携／メディア対応／国際化。
- **「劣る」（△・×）は許容しない**。検出時は §22 サイクルで是正対象。
- **昇格候補（◎ への段階的押し上げ）**: ナビ粒度（§3 ドメインモデルの実装熟成後）→ メディア対応（騒音・低照度の現場特化機能で他社差を生成）→ 国際化（東アジア言語＋RTL の早期投入）の順で §22 サイクルにて再評価する。

### 4.5. 統合スコアカード（世界一の判定基準）

「世界一」を主観でなく集計で判定するため、軸ごとの評価を以下で集約する。

| 評価 | 配点 |
| --- | --- |
| ◎ | 3 |
| ○ | 2 |
| △ | 1 |
| × | 0 |

- 11 軸合計 33 点満点。本アプリ目標: **29 点以上**（◎ × 7 軸 = 21 ＋ ○ × 4 軸 = 8、合計 29）。本数値は `競合比較表.md` の本アプリ目標行と一致する。
- 上限余地（30〜33 点）は §22 サイクルで §4.4 の「昇格候補」順に押し上げ、年次で 1 軸以上の ◎ 化を目指す。
- 各競合との差分を四半期ごと比較し、最大点製品との差を **正の値** で維持する。

### 4.6. 採点方法論（初期決定）

「世界一」を主張するなら採点プロセスが客観的に追跡可能であることが必要。個人開発下では完全な第三者採点は得られないため、以下の現実解を採る。

#### 4.6.1. 採点者と分担

| ロール | 担当 | 制約 |
| --- | --- | --- |
| 一次採点者 | コアメンテナ | 全セルを採点。根拠を `競合比較表.md` の脚注へ必ず記載 |
| 二次レビュー | LLM（Claude 等）による独立採点 | 一次採点と差分が出たセルは差分を Issue 化 |
| 公開反論 | OSS 公開後の任意の第三者 | GitHub Issues に `competitive-score` ラベルで提出 |

#### 4.6.2. セル採点規則

- ◎: 公開された一次資料（公式ドキュメント・論文・規格）の裏付けが **2 件以上**。
- ○: 一次資料 1 件以上、または明確に公開された機能比較記事 1 件。
- △: 機能不在の一次資料、または競合の不利を示す資料 1 件。
- ×: 機能不在が公式に明示。
- `?`: 採点不能。`?` を残したままリリースしない（§4.7 受入観点）。

#### 4.6.3. 反論プロセス

1. 反論提出 → GitHub Issue（ラベル `competitive-score`）。
2. メンテナが 2 週間以内に再採点・是非判定。
3. 採点変更時は `競合比較表.md` 改訂＋ CHANGELOG 反映。
4. 不採用時は理由を Issue に明記してクローズ。

#### 4.6.4. バイアス対策

- 自社採点の高評価バイアスを抑えるため、**自軸の ◎ には他社製品 1 件以上を ◎ 並列にする** ことを許容（独占的 ◎ は禁止）。
- 競合各社の採点は **公平性監査表** を四半期更新し、採点根拠の偏りを検証。

### 4.7. 受入観点

- 競合比較表の各セルに評価値（◎○△×）が入る（`?` のセルがゼロ）。
- 各セル採点に §4.6.2 を満たす根拠が脚注として記載されている。
- 本アプリ目標スコアが **29 点以上** を維持。
- 「△」が一つでも検出された場合、§22 是正サイクルで対応起票される。
- 昇格候補軸の進捗（年次の ◎ 化計画）が CHANGELOG または GitHub Discussions で追跡可能。
- §4.6.3 の反論プロセス記録が GitHub 上で追跡可能。

### 4.8. 競合監視の自動化

メモ L7「市場にある競合を調査」を四半期手動だけに任せると、決定的瞬間（競合の重大機能リリース・OSS 化・買収）を見落とす。`scripts/competitor-watch.sh` で次を自動収集し、変化検出時に `competitor-watch` ラベル付き Issue を起票する。

| 対象 | 監視手段 | 起点 |
| --- | --- | --- |
| 競合公式ブログ・プレス | RSS／Atom フィード購読 | 各社 News／Blog／Press |
| GitHub リリース・公開 | `gh api` で release 一覧取得（OSS 化された競合） | GitHub Trending／組織検索 |
| 学術・規格動向 | arXiv（cs.HC, cs.SE, cs.SY）／ISO/IEC/IEEE 草稿 | キーワード（manufacturing, work instruction, MES） |
| 製品比較記事 | 主要メディアの定期検索（Capterra／G2／PeerSpot 等） | 最終更新日チェック |
| 特許 | USPTO／JPO／EPO 公開 | キーワード（manufacturing work instruction, MES, andon） |
| 投資・買収 | Crunchbase／PitchBook の公開 RSS | フォロー対象企業 |
| OSS 健全性 | GitHub API（commit／contributor 推移） | OSS 競合候補のリポジトリ |

四半期単位の手動レビュー（§22.1）はこの自動収集データを起点に行うことで、抜け漏れを構造的に減らす。自動収集データは `docs/01_企画/競合監視ログ/<年月>.md` に蓄積する。

### 4.9. 圧倒候補機能（killer features）

§4.4 で「◎ を狙う 7 軸」を選んだが、軸の中で **他社が機能として持ちえない／持ちにくい** 圧倒候補を明示する。「世界一」（§1, §2）の根拠を機能に落とし、§22 サイクルで追従度を計測する。

| 候補 | 関連軸 | 圧倒の論拠 |
| --- | --- | --- |
| ナビゲーションの形式化（HSM＋CPN＋TLA+） | ナビ粒度 | 競合の作業手順は「列挙」が主流。形式化で例外・並行・冪等を網羅。学術界からも引用可能 |
| 完全オフライン継続 7 日（端末監査ログ追記不変） | オフライン | 多くの競合はクラウド前提。本アプリは追記不変・端末暗号化（SQLCipher） |
| アドオン WASM サンドボックス（capability ベース） | 拡張性・アドオン | 既存 OSS 製造系は plugin 機構が貧弱。capability＋WASM は工場内ネットワーク要件と整合 |
| ワンコマンド `make doctor` ＋ QR ペアリング | セットアップ容易性 | 産業 SaaS 競合はオンプレ展開が苦手。本アプリは 5 分セットアップを公約 |
| GxP バリデーション支援テンプレ同梱（IQ／OQ／PQ） | 規制適合 | OSS 競合で IQ／OQ／PQ テンプレ同梱は事実上空白 |
| 行動心理学＋TPS 統合の文言／既定値 | 非エンジニア操作性 | UX 改善を「事後改善」でなく「設計時予測」（§9.2.1）で抑える点が他社にない |
| 完全 OSS／Apache-2.0／個人開発の透明性 | ライセンス・TCO | 競合は SaaS 課金が主流。OSS の TCO 差は構造的優位 |
| 教育素材ライブラリの循環設計（端末撮影 → 教育素材） | 非エンジニア操作性／ナビ粒度 | 現場ベストプラクティスを資産化する仕組みは競合に薄い |

各候補は「他社追従度」を §22.1 で計測し、追いつかれた段階で次世代候補へ切り替える。

### 5.1. 想定規模（3 階層）

### 5.2. 性能目標（初期決定／根拠は §21）

| 指標 | 目標値 | 規模 |
| --- | --- | --- |
| 画面遷移応答（中央値／p95） | ≤ 200ms ／ ≤ 500ms | 全 |
| タッチ操作フィードバック | ≤ 100ms | 全 |
| アプリ起動（コールド／ウォーム） | ≤ 3s ／ ≤ 1s | 全 |

| 同期スループット（大規模） | ≥ 1,000 実績／分 | 大 |

### 5.3. スループット検算

- 中規模 5,000 件／日、稼働 8 時間 → 平均 10.4 件／分。朝礼後・休憩後ピークを 10 倍と仮定し 104 件／分 → 目標 100 件／分は妥当。
- 大規模 50,000 件／日、稼働 8 時間 → 平均 104 件／分。ピーク 10 倍で 1,040 件／分 → 目標 1,000 件／分は妥当。

### 5.4. 受入観点

- 上記指標が CI で計測され、回帰時に検知される。
- 大規模負荷シナリオで構成限界が文書化されている。
- 検算が崩れた場合、§22 で目標値を更新する。

## 7. プロダクト構成

メモを精読すると UI が **二系統** であることが分かる（L23: `tauri(react) : 端末`、L25: `react : マスタメンテナンス、作業フロー設定`）。よって本アプリは三コンポーネント構成とする。

### 7.1. 端末作業ナビアプリ（Tauri＋React）

- 製造現場の作業者が利用。Android／Windows タブレット。
- オフライン一次動作。SQLite ローカルストア。
- カメラ・音声・動画・画像の入出力。
- 認知負荷を抑える段階的情報提示。

### 7.2. 設定 Web UI（React）

- マスタメンテナンス／作業フロー設定。
- 設定者（生産技術／システム部門）が利用。デスクトップ／タブレット双方で操作可能。
- ブラウザで動作するためインストール不要。
- 作業者と同等水準の UX（メモ L45）。

### 7.3. バックエンド（Rust／tokio）

- REST API。基幹システム連携・端末同期・設定 UI のサーバ。
- PostgreSQL を永続化。
- Docker で配布。

### 7.4. 受入観点

- 三コンポーネントが独立に起動・テスト可能。
- 端末アプリと設定 UI のデザインシステムが共通化されている（後述 §11.1）。
- API 契約が OpenAPI で公開されている。

## 9. 設計原則

### 9.1. アーキテクチャ

- クリーンアーキテクチャ採用。
- ドメイン駆動設計採用。「作業」概念（§3）をユビキタス言語の中心に置く。
- テスト駆動開発採用。

### 9.2. 心理学的アプローチ（独立原則）

#### 9.2.1. 心理的ストレス予測アプローチ

メモ L19「ステークホルダーの心理的ストレスを **予測** し」を独立工程として明文化する。**事後改善ではなく、設計時に予測し回避** することを原則とする。本アプローチは §3.2 ステークホルダー別に適用する。

##### 予測モデル（5 軸）

| 軸 | 観測対象 | 兆候・閾値 | 設計対応 |
| --- | --- | --- | --- |
| 認知負荷 | 同時提示する情報量・選択肢数 | 7 ± 2 を超える要素 | 段階的開示／一覧の畳み込み |
| 時間圧 | 標準時間に対する残余裕 | 残時間 < 標準の 20% | カウントダウン抑制／中断許可 |
| 不確実性 | 次操作・正解の不明瞭さ | 次アクションが一意でない | 次アクションの単一化／プレビュー |
| 自律性喪失 | 強制ダイアログ・ブロック頻度 | 1 タスクで 3 回超 | スキップ可能化／後置確認 |
| 失敗コスト | 誤操作の不可逆性 | 取消不可・データ損失 | 取消可能化／二重確認 |

##### 予測手段

1. **タスク分析**: §3.4 の方法論で得た SOP を 5 軸でヒートマップ化。
2. **NASA-TLX 推定**: NASA-TLX の 6 因子（Mental／Physical／Temporal／Performance／Effort／Frustration）を画面ごとに設計時想定値として記入し、規定値超のフローを再構成。
3. **ヒューリスティック評価**: Nielsen 10 ヒューリスティック＋本書 §9.2.2／§9.2.3／§9.2.4 をレビューチェックリスト化。
4. **LLM シミュレーション**: 主要シナリオを LLM で対話シミュレートし、フラストレーション・誤解釈の発生点を抽出（メモ L33-35 の情報源制約下での代替手段）。

##### 受入観点

- 主要画面に 5 軸ヒートマップが付随し、規定値超の項目に設計対応が紐付く。
- NASA-TLX 想定値が CHANGELOG に記録され、改訂時に増減が追跡可能。
- §3.2 ステークホルダー別に 5 軸評価が集計可能である。

#### 9.2.2. 認知科学・人間工学の原則

- **認知負荷の最小化**: 情報の段階的提示、不要要素の排除（Sweller, Cognitive Load Theory, 1988）。
- **誤操作の予防**: 取消可能性、確認の段階化、危険操作の隔離（Norman *The Design of Everyday Things*, 1988）。
- **即時かつ明瞭なフィードバック**: 成功・失敗・進行を 100ms 以内に知覚（§21 注 2）。
- **注意誘導**: 次の動作が視覚的に一意（Hick の法則で選択肢を圧縮、Fitts の法則でターゲットを最大化）。
- **心理的安全**: 責任追及的でない、回復可能な操作体系（Edmondson *Psychological Safety*, 1999）。

#### 9.2.3. 行動心理学・行動経済学の取り込み（メモ L21 対応）

メモ L21「行動心理学の観点でのアプローチも欲しい」を独立した設計原則として明文化する。各フレームは UI／文言／既定値／フロー設計のレビューチェックリストに展開し、画面ごとに紐付けを管理する。

| フレーム | 出典 | 設計への反映 |
| --- | --- | --- |
| ナッジ／選択アーキテクチャ | Thaler & Sunstein *Nudge* (2008) | 既定値は安全側・推奨側に置く。ハザード操作はオプトイン |
| デフォルト効果 | Johnson & Goldstein *Science* (2003) | フロー設定の初期値を「品質的に正しい」状態に統一。空欄起点を避ける |
| 損失回避（プロスペクト理論） | Kahneman & Tversky (1979) | 「失う」表現より「守る」表現（例: 「保存して進める」）を採用 |
| ピーク・エンドの法則 | Kahneman et al. (1993) | 作業完了画面（エンド）の体験品質に重点投資。失敗時もリカバリ画面を整える |
| ゴール・グラディエント効果 | Hull (1932) / Kivetz, Urminsky, Zheng (2006) | 進捗バーで「あと N ステップ」を可視化し、終端付近の心理的加速を活用 |
| 正の強化（操作条件付け） | Skinner (1953) | 完了時の正のフィードバック（音・色・微振動）を一貫して付与 |
| アンカリング | Tversky & Kahneman *Science* (1974) | 標準時間表示で過剰に短い／長い見積もりを抑制 |
| メンタルモデル整合 | Norman (1983) | アプリ上の「作業」表現を現場の言葉・順序と一致させる |

#### 9.2.4. ダーク・パターンの禁止

行動心理学を採用する以上、悪用排除を本プロジェクトの原則として明記する。

- 偽の希少性／罪悪感誘発／隠しコスト／確認ボタンの誤誘導など、ユーザーの不利益に働くパターンを禁止する。
- レビュー時に Brignull *Deceptive Design* 系の分類（FTC 2022 報告も参照）を用い、検出されたパターンは §22 サイクルで是正する。

#### 9.2.5. 受入観点

- 主要画面が §9.2.1 の 5 軸予測および §9.2.2 / §9.2.3 のいずれのフレームに紐付くかをマッピング表で管理。
- ダーク・パターン検出のレビューチェックリストが運用されている。

- **本体に内包しない**: 個別手法専用の UI／レポートはコア機能として組み込まず、§16 観測ダッシュボードのメトリクス／§17 アドオン／設定 UI のテンプレート（§10.2）の 3 経路に分散する。これにより手法のトレンド変化に追従可能。

- **規格適合の証跡**: ISO 22400／ISA-95／ISA-88 等の規格語彙は §28 に取り込み、§12 規制適合および §10.3.6 RACI と整合させる。

- **撤退**: 同一手法が 3 期連続で実装されない／業界トレンドから外れた場合は §22.3 撤退条件に従い本表から削除し、ADR（§9.4／§30 Type 1）で根拠を残す。

### 9.4. コーディング規律

メモ L43「メンテナンス性を意識したコード」をコードレベルの強制可能な規律へ落とす。

- **1 ファイル 500 行上限**。超過時は責務単位（クラス／モジュール／ユースケース）で分割。

### 9.5. デザインシステムと感覚モダリティ設計

メモ L41「UIUX には徹底的に拘る」を、レビュー時の主観に委ねず **構造化された設計トークン** に固定する。製造現場特有の制約（騒音・低照度・手袋・粉塵）下では、視覚以外のチャンネル（聴覚・触覚）も第一級設計対象とする。本節は §9.2 心理学的アプローチ／§11.1 UI/UX／§11.2 アクセシビリティと接続する。

#### 9.5.1. デザイントークン

`docs/02_設計/design-tokens/` で版管理する。Style Dictionary 形式（Amazon, 2018）で記述し、Tauri／React／Android／Windows のいずれにも自動変換する。

| --- | --- | --- |
| カラー | `color.semantic.success` | 12 トーン（success／warning／danger／info × 3 強度）。屋外輝度 1,000nits 下でも判別可能なコントラスト |
| タイポ | `font.size.body` | 14／16／20／24／32px。Dynamic Type 連動。CJK／ハングル／ラテンの同高保証 |

#### 9.5.2. モーション設計言語

- **意味的モーション**: 進入・遷移・状態変化・完了・失敗・警告 6 カテゴリ。各カテゴリで decel／accel／emphasized を固定。
- **減衰の禁止**: 視覚的に派手な弾性アニメは現場では認知妨害。Critically damped カーブを既定。
- **prefers-reduced-motion** 対応: OS 設定が縮減モードのとき装飾モーションを 0ms 化し、情報モーションのみ維持。
- **カウントダウンの抑制**: §9.2.1 時間圧軸の悪化要因のため進捗バーで代替（§9.2.3 ゴール・グラディエント効果）。

#### 9.5.3. サウンド設計

製造現場の典型環境音 80〜95dB を上回らずに知覚可能な音設計とする。WHO 推奨の 85dB 8 時間曝露上限を踏まえ、本アプリの通知音は **設計時に 85dB 以下** を保証する。

| 種類 | 既定波形 | 想定強度 | 用途 |
| --- | --- | --- | --- |
| 完了 | 2 音上昇（C5→E5）／200ms | 中（70dB @1m 想定） | ステップ完了 |
| 失敗 | 2 音下降（E4→C4）／300ms | 中 | 完了条件不適合 |
| 警告 | 3 音反復（A4）／500ms | 強（80dB @1m） | アンドン |
| 入力受領 | 単音（C6）／80ms | 弱（60dB @1m） | タッチ・スキャン受領 |
| 危険 | 矩形波 350Hz／断続 | 強（85dB @1m） | 危険操作の確認段階 |

- すべて視覚等価物を併設（§11.2.3）。
- 個体差・難聴対応として **可聴域シフト**（高域減衰）と振動代替を提供。
- 既定値はミュートしない（聴覚 UI 二重化と整合）。ミュート時は警告アイコンを常時表示。
- BGM・ジングルは採用しない（§9.2.4 ダーク・パターンに該当する誘導音を排除）。

#### 9.5.4. 触覚（ハプティック）設計

Android／Windows のハプティック API 差を吸収し、共通カテゴリで運用する。

| カテゴリ | 強度 | パターン | 用途 |
| --- | --- | --- | --- |
| 軽 | 弱 | 単発 20ms | タッチ受領 |
| 確認 | 中 | 単発 50ms | ステップ完了 |
| 警告 | 強 | 二発 50ms × 2（間 80ms） | 警告 |
| 危険 | 強 | 三発 80ms × 3（間 120ms） | 危険操作前 |

- 騒音下では音の代替として優先される。
- 端末がハプティック非搭載の場合、ビジュアルフラッシュへフォールバック（§11.2.3 二重化）。

#### 9.5.5. 受入観点

- §9.5.1 の 9 トークン区分が `design-tokens/` で版管理され、CI で実装と乖離していないことを検査。
- prefers-reduced-motion／prefers-color-scheme への対応が axe-core ＋ playwright で回帰検証。
- 工場環境（80dB 騒音／低照度／手袋）を想定した検証手順が `docs/04_運用/環境別検証.md` に文書化。
- §9.2 心理学アプローチ・§11.2 アクセシビリティと矛盾しない（変更時に双方向参照を更新）。
- §28 用語集の UI 表記用語と一致（同義語混入を CI 検出）。

### 10.1. 作業ナビゲーション（端末アプリ）

実現項目: §3 ドメインモデルに基づくナビ表示／順序・分岐・例外・割込みの逐次案内／メディア入出力／作業実績記録／後続工程への自動受け渡し／段階的情報提示。

#### 10.2.3. 編集中フローのストレス予測（事前検証）

§9.2.1 の 5 軸ヒートマップを **編集中のフローに対しても適用** する。設定者が「このフローを実行する作業者の予測ストレス」を試行版発行 **前に** 確認できることで、現場での事後トラブルを未然に防ぐ。

- 編集者は試行版発行前に 5 軸スコア（認知負荷／時間圧／不確実性／自律性喪失／失敗コスト）を確認できる。
- スコアが規定値超のステップは試行版発行をブロックし、再設計を促す（オプトアウト時は ADR で根拠記録）。
- 予測根拠（参照 SOP・公開資料・LLM 推定／§9.2.1）は脚注で参照可能。
- スコアの月次トレンドは §16 観測ダッシュボードに反映され、§22.1 サイクルでレビューされる。

#### 10.2.4. 受入観点

- 非エンジニアがサンプル参照のみで **5 分以内** に最初のフローを発行できる（§14.2 セットアップ容易性目標と整合）。
- 任意時点へのロールバックが操作 **3 ステップ以内**。
- 設定 UI の操作所要時間と誤操作率が作業 UI と同等水準（§4 比較軸 A1／A2 と整合）。
- §10.2.2 の 14 観点が UI 監査チェックリスト（`docs/02_設計/設定UI監査.md`）で網羅される。
- フロー編集での §3.4.1 形式検証（HSM 到達可能性・デッドロック）が CI 緑。
- 試行版 → パイロット → 本番昇格の経路が §10.6 同期戦略と整合し、E2E テスト緑。
- §10.2.3 ストレス予測の規定値超ブロックが性質ベーステストで検証される。
- 多言語化が §11.3.1 各段階で機能し、§3.1.5.2 多言語対比表に同期する。
- §9.2.4 ダーク・パターン監査で検出件数 0 を維持。

### 10.3. 基幹システム連携（バックエンド）

メモ L15／L48-50 を踏まえ、本アプリは MES／ERP の **置き換えを狙わない**。製造現場の「作業」を一次的に扱い、生産順序・品目・実績の橋渡しを REST API で双方向に行う。基幹側仕様の差異吸収はアダプタ／アドオン（§17）で行う。

#### 10.3.1. 連携契約（初期決定）

| 項目 | 規定 |
| --- | --- |
| プロトコル | HTTPS / REST / JSON（一次）。OPC UA・MQTT・Modbus 等はアドオン |
| 認証 | TLS クライアント証明書 ＋ JWT（短寿命）／API キー併用可 |
| 冪等性 | すべての書込に **Idempotency-Key** ヘッダ必須（24h 重複排除） |
| トランザクション | 結果サブミットは「順序情報単位」で原子化 |
| リトライ | 指数バックオフ（既定: 1s/2s/4s/…/最大 5 分／24h 後にデッドレター） |
| デッドレター | 監査必須／設定 UI からの再投入操作可 |
| 時刻 | 全 payload に UTC ISO 8601／§20.2 と整合 |
| スキーマ | OpenAPI 3.1／payload に `schema_version` |
| 後方互換 | MINOR ＝後方互換、MAJOR ＝最大 6 ヶ月の移行猶予（§17.3 と同方針） |
| 失敗時通知 | 設定 UI ＋ アンドン通知（§9.3.1） |

#### 10.3.2. 連携シナリオ（双方向）

| シナリオ | 方向 | 概要 |
| --- | --- | --- |
| 順序情報受領 | 基幹 → 本アプリ | 生産順序、品目、ロット、設備割付の取り込み |
| 実績フィードバック | 本アプリ → 基幹 | 完了・不良・所要時間・是正実績 |
| マスタ取り込み | 基幹 → 本アプリ | 品目、BOM、工程ルートの初期化と差分更新 |
| 品質ホールド通知 | 本アプリ → 基幹 | 規定外検出時の保留要請 |
| 例外解消通知 | 双方向 | 例外発生 → 基幹再計画 → 本アプリ反映 |

#### 10.3.3. 連携障害時のフォールバック

- 基幹接続不能時はキャッシュされた直近の順序情報で継続。
- フォールバックモードは UI に明示し、再接続時に差分再同期。
- CSV／YAML での **手動取り込み／吐き出し** を維持し、現地のオペレーション継続性を保証。

#### 10.3.4. 想定対応 ERP / MES（参考）

SAP S/4HANA Manufacturing、Oracle Manufacturing、Mcframe、IFS Cloud、Plex、GE Proficy、AVEVA Wonderware、Hitachi Lumada、Fujitsu COLMINA、内製基幹システム（製造業では最も多い）。

固有プロトコル・スキーマはアドオン（§17）で吸収する。本体は **REST 純正契約のみ** 提供する。

#### 10.3.5. 受入観点

- §10.3.1 の 10 規定がすべて OpenAPI 仕様で表現されている。
- Idempotency 重複排除、リトライ、デッドレター遷移が結合テストで確認可能。
- フォールバックモード／復旧時同期の挙動が E2E テストで緑。
- 最低 1 つの参考クライアント（CLI／サンプル統合）が `examples/` に同梱。
- 連携あり／なし両モードでの主要シナリオが §13 E2E で網羅されている。

#### 10.3.6. MES／ERP との責任分界（RACI）

メモ L48-50 を機能ごとに RACI（Responsible／Accountable／Consulted／Informed）で硬く分界する。「置き換えない」を曖昧な共有責任のまま放置すると、現場で運用事故（誰がデータの一次保管者か、誰が最終承認するか）が起きる。

凡例: R＝実行責任、A＝最終責任（ただ 1 名）、C＝諮問、I＝報告。

| 機能 | 本アプリ | 基幹（MES／ERP） | 備考 |
| --- | --- | --- | --- |
| 作業手順の記述・改訂 | R/A | C | フロー定義は本アプリの一次データ |
| 順序情報（生産計画） | R（受領・表示） | R/A | 計画は基幹が原典。本アプリは順序受領のみ |
| 当日着手判定 | R/A | C | 着手はナビ判定 |
| 作業実績（時間・人・結果） | R/A | I | 一次記録は本アプリ。基幹は集約参照 |
| 不良判定 | R/A | I | 端末で完了条件不適合 |
| 品質ホールド通知 | R | A | 通知は本アプリ、ホールド最終判断は基幹／品質保証 |
| BOM／品目マスタ | I/C | R/A | 基幹側原典。本アプリは取り込みのみ |
| 監査証跡（操作・電子署名） | R/A | I | 本アプリ追記不変ストア |
| 設備保全計画 | I | C/A（CMMS／EAM） | 別ドメイン（§3.1.4） |
| KPI 集計レポート | C | R/A | 本アプリは raw data 提供 |
| 規制バリデーション（IQ／OQ／PQ） | C | R/A（導入企業） | 本アプリはテンプレ提供（§12） |
| データ保持・削除 | R（端末／同期領域） | R/A（基幹側） | §20.3 と整合 |

- **A が空** の行が無いこと、**A が両方（共有 A）** の行が無いことを OpenAPI スキーマレビュー時に確認する。
- 行追加・修正は §22.4 是正フローを経由する。

### 10.4. メディア対応（端末アプリ）

メモ L31「カメラ、画像、動画、音声にも対応」を、現場特有の制約（騒音・低照度・手袋）を含めた仕様まで掘り下げる。

#### 10.4.1. 対応モード

| モード | 入力 | 既定品質 | 用途 |
| --- | --- | --- | --- |
| 写真 | カメラ | 1080p / JPEG q=85 | 完了証跡／不良証跡 |
| 連写 | カメラ | 5 枚/秒 × 5s | 検査ステップ |
| 動画 | カメラ | 1080p / H.265 / 30fps | 手順記録／教育素材 |
| タイムラプス | カメラ | 1fps | 長時間処理（硬化・乾燥）の証跡 |
| 音声 | マイク | 16kHz mono / Opus | 口頭メモ／PTT 通信 |
| QR ／ バーコード | カメラ | リアルタイム認識 | 工程・部材識別 |
| OCR（型番・ロット）| カメラ | 200dpi 相当 | 値の自動入力 |

#### 10.4.2. 騒音・低照度・手袋への適応

- ノイズキャンセリング（音声入力）／PTT（押している間のみ送話）。
- 自動露出補正＋暗所モード／バンドノイズ抑制。
- 手袋モード時のシャッターは大型タッチターゲット＋音声コマンド代替。

#### 10.4.3. ハンズフリー操作

- 音声コマンド: 「開始／完了／中断／戻る／メモ／撮影」を既定セットで提供。
- 端末スピーカで進行を読み上げ可能（§11.2 視覚補助／聴覚 UI 二重化と整合）。

#### 10.4.4. ストレージとライフサイクル

- 端末ローカル: 既定上限の 80% で警告／90% で新規撮影をブロックし管理者通知。
- 同期完了済みメディアの自動削除はオプション（既定: 同期 ＋ 7 日経過後）。
- メディア削除は **不可逆ログ**（§16 監査）に残す。

#### 10.4.5. 改ざん検出と監査連動

- 各メディアファイルへ生成時に **SHA-256 ハッシュ** を算出し、関連作業実績に付帯保存。
- サーバ側で再計算して一致確認。不一致は規制適合警告（§12）。
- メディア参照の削除は §10.5 監査ログへ書込み、本体は匿名化保持（§20.3 整合）。

#### 10.4.6. 受入観点

- §10.4.1 の 7 モードがすべて単体テスト・実機検証可能。
- 音声コマンド・OCR・QR の認識率が CI で計測される（最小規定値は §22 サイクルで決定）。
- 大容量メディア（500MB 動画）でも UI スレッドが固まらない（§5.2 タッチ応答 ≤ 100ms 維持）。
- メディア破損検出から再取得導線まで 3 タップ以内。
- ハッシュ不一致時の警告表示と監査ログ記録が結合テストで検証されている。

### 10.5. 認証

実現項目: ユーザー ID＋パスワード認証／オフライン時のログイン継続／バックエンド側の認証情報との整合／RBAC（後述 §17）／電子署名対応（§12）。

#### 10.5.0. 認証方式選択の根拠（メモ L54）

ID＋パスワードを既定とした根拠は次のとおり。

| 観点 | 評価 |
| --- | --- |
| 現場端末の共有運用 | パスワード覚え書きや端末管理者の引き継ぎが現実的 |
| ネット非依存 | 端末ローカル認証成立／本節のオフライン継続ログイン |
| 製造業の文化 | 多くの現場で既存の社員 ID 体系が存在（＝再利用容易） |
| 個人開発の維持コスト | 一次認証を最小化し、より高度な認証はアドオンで提供 |

拡張は次のとおりアドオン化する（§17）。本体保守範囲を最小化しつつ、導入企業要件に応じた選択を可能にする。

| 拡張認証 | capability | 想定利用シーン |
| --- | --- | --- |
| SAML / OIDC SSO | `auth.extend:saml` / `auth.extend:oidc` | 大企業の Active Directory／Azure AD 統合 |
| Passkey / WebAuthn | `auth.extend:webauthn` | 漏洩耐性を要する開発・管理者用 UI |
| スマートカード | `auth.extend:piv` | 防衛・医療等の物理トークン要件 |
| 顔認証 | `auth.extend:face` | クリーンルーム等で手袋必須環境 |

#### 10.5.1. オフライン継続ログイン仕様（初期決定）

| 項目 | 値 |
| --- | --- |
| 継続上限 | **7 日**（最終オンライン認証から起算、§21 注 9） |
| 期限到達時の挙動 | 業務操作を一律拒否し再認証要求。再認証完了まで監査ログ記録のみ許可 |
| パスワードキャッシュ | Argon2id 派生鍵のみを OS Keystore（Android）／DPAPI（Windows）に保護保存 |
| アカウント無効化伝播 | オンライン復帰時に強制ログアウト→再認証。猶予なし |
| 緊急ロック | 管理者発行のロックトークンを QR で受領し即時ログアウト可能（オフライン経路） |

受入観点:

- オフライン期間中の操作監査ログが完全で、復旧時に欠損なく送信される。
- パスワード平文または可逆形式が **どの永続層にも存在しない** ことを CI 静的解析で確認。
- 7 日経過時の業務操作拒否が単体・E2E で検証されている。
- アカウント無効化のオンライン復帰時伝播が時間計測でテスト済み。

### 10.6. オフライン対応と同期

実現項目: SQLite を一次ストアとし、ネットワーク非依存で動作／復旧時自動同期／競合解消方針の仕様化。

同期競合方針（初期決定）:

| エンティティ群 | 戦略 | 理由 |
| --- | --- | --- |
| 作業実績（イベント） | 端末優先・追記のみ | 現場発生事象を欠落させない |
| マスタ（製品・工程・設備） | サーバ優先 | 集中管理されるため |
| 作業フロー定義 | サーバ優先＋端末ローカルキャッシュ | 設定の一元性 |
| ユーザー設定 | last-write-wins＋デバイス ID＋更新時刻を監査ログ | 個人設定／不正上書き追跡 |
| 添付メディア | 追記のみ | 不可逆 |
| 解決不能ケース | 人手解決キュー（デッドレター） | 監査と説明責任 |

受入観点: 全エンティティで戦略が定義され、性質ベーステストで網羅検証されている／デッドレター運用手順が文書化されている。

#### 10.6.1. 同期の形式モデル

§10.6 の競合戦略（端末優先・サーバ優先・LWW・追記のみ）を **形式モデル** に固定する。沈黙の妥協（§2）を避けるため、採用する CRDT 種別と Lamport クロックの扱いを明示する。形式検証は §3.4.1 TLA+ と §13.2 proptest の組み合わせで担う。

| エンティティ群 | 形式モデル | 検証 |
| --- | --- | --- |
| 作業実績（イベント） | G-Set（追記のみ Set, Shapiro et al. 2011） | TLA+ 不変式 `Inv_NoEventLoss`／§3.4.1 |
| 添付メディア | LWW-Element-Set（不可逆／追記） | ハッシュ照合（§10.4.5）＋ proptest |
| マスタ（製品・工程・設備） | サーバ単一権威。CRDT 不要 | 単一書込み口 + Idempotency-Key |
| 作業フロー定義 | サーバ単一権威＋端末ローカルキャッシュ | バージョン番号でキャッシュ整合 |
| ユーザー設定 | LWW-Register（device_id, lamport_ts） | Lamport timestamp の単調性検証 |
| デッドレターキュー | サーバ単一権威 | 監査ログ追記不変 |

- **クロック**: 全イベントに Lamport timestamp を付与（壁時計とは独立）。壁時計（端末／サーバ UTC）は §20.2 に従い別途記録。
- **競合解決の決定性**: LWW 適用時は (lamport_ts, device_id) の lex 順で必ず決定。`device_id` は端末初回登録時に発行する UUID v7。
- **デッドレター遷移条件**: 24h 経過した未解決競合／形式モデル外の例外。
- **再同期境界**: 端末↔サーバの再接続時は最後に成功した Lamport ts 以降をストリーム再送する。重複は Idempotency-Key で排除。

#### 10.6.2. 受入観点（追加）

- TLA+ で `Inv_NoEventLoss`／`Inv_LWWDeterministic` の TLC／Apalache 検証が CI で緑。
- proptest で乱数順序・遅延・再送・部分障害下でも上記不変式が破れない。
- Lamport clock の単調性が CI 性質ベーステストで網羅。
- デッドレター遷移条件が結合テストで再現可能。

## 11. 非機能要件と受入観点

### 11.1. UI/UX

- 誰でも操作しやすい UI/UX。設定系も作業系と同水準（メモ L45）。
- 端末アプリと設定 Web UI で **共通デザインシステム** を採用（コンポーネント・配色・タイポ・モーション）。
- 受入観点: 同一タスクの所要時間・誤操作率・主観評価が、§4 比較軸に対して優位。

### 11.2. アクセシビリティ

メモ L29「誰でも操作しやすい UI/UX」を、現場環境への適応と多様な身体・認知特性への適応の **両軸** で扱う。

#### 11.2.1. 現場環境への適応

- 手袋・粉塵・低照度／高照度・騒音下を想定。
- タッチターゲット ≥ 9mm（§21 注 6、推奨 11mm）。
- 屋内外併用想定の輝度設計（§6.3）。

#### 11.2.2. 多様な身体・認知特性への適応

| 観点 | 設計対応 |
| --- | --- |
| 色覚多様性 | 重要情報は色のみで区別しない（形・位置・テキスト併用）。P／D／T 型でのコントラスト検証 |
| 視力 | フォントサイズ調整／ハイコントラストモード／ピンチズーム |
| 聴覚 | 音声フィードバックには視覚等価物を併設／字幕 |
| 上肢可動域 | 大型タッチ／長押し代替／スワイプの最小ストローク化 |
| 認知補助 | 易しい言葉モード（漢字ふりがな／英語平易表現）／重要操作の繰り返し読み上げ |

#### 11.2.3. 受入観点

- 主要画面が **WCAG 2.2 AA 相当** を通過（axe-core で自動回帰）。
- タッチターゲット ≥ 9mm（§21 注 6）／コントラスト比 ≥ 4.5:1（テキスト）。
- 色覚多様性 3 タイプ（P／D／T）のシミュレーションでコントラスト・識別性が成立。
- 重要操作は **音声と視覚の二重フィードバック** が成立。
- 工業記号（°C・Ω・ねじ等）と CJK フォントが代替フォント連鎖を含めて正しく表示される。

### 11.3. 国際化（i18n）

- 文字列の外部化、ロケール切替、日付・数値・単位の地域化。初期同梱は **日本語・英語**。
- 受入観点: 2 ロケールが完全動作／新規ロケール追加が翻訳ファイル追加のみで完結／RTL 言語の混入に対応可能な構造を維持／フォント不在時の代替フォント連鎖が定義されている。

#### 11.3.1. ローカライゼーション拡張計画

§11.3 で初期同梱は日本語・英語と決めたが、製造業強国を考慮した拡張順序を初期決定として固定する。順序は GDP × 製造業比率 × OSS 採用度合の合成で評価し、§22.1 で再評価する。

| 段階 | 言語 | 採用ロジック |
| --- | --- | --- |
| 初版 | 日本語、英語 | OSS 公開最大公約数（§24.2） |
| +6 ヶ月 | 中国語（簡体）、韓国語 | 東アジア製造業集積（中国＝世界最大、韓国＝半導体／自動車） |
| +12 ヶ月 | ドイツ語、スペイン語 | 欧州製造業（独）／ラテンアメリカ＋スペイン本土 |
| +18 ヶ月 | ベトナム語、タイ語、インドネシア語 | ASEAN 製造業集積／日系企業の進出先 |
| +24 ヶ月 | フランス語、ポルトガル語、中国語（繁体） | 欧州・ブラジル製造業／台湾市場 |
| RTL レーン | アラビア語、ヘブライ語 | RTL 構造の正しさを早期に検証する別レーンで先行投入可 |

- 翻訳は **公開機械翻訳ベース＋メンテナレビュー** で初版化し、コミュニティ提案で精度向上（§19.4）。機械翻訳のみで本番投入はしない（§26 アンチゴール）。
- 工業記号（°C、Ω、Ø、ねじ規格、JIS／DIN／ANSI 等）はロケール非依存で統一表現。
- 単位は SI を内部基準にし、表示時に米単位（in／lb／°F）変換を許容（§17 アドオン）。
- 数値書式（小数点・桁区切り）と日付書式（YYYY-MM-DD／DD.MM.YYYY 等）はロケール表でテンプレ管理。

#### 11.3.2. 受入観点（追加）

- 各段階の追加言語追加時に E2E ビジュアル回帰が緑。
- RTL レイアウトでの誤配置（左右反転忘れ）を CI 自動検出（`playwright` ＋ `axe-core`）。
- ロケールフォント不在時の代替フォント連鎖（CJK／ハングル／アラビア／ヘブライ）が決定論的。
- 拡張段階ごとに教育コンテンツ（§25）の言語追従が並行する。

### 11.4. セキュリティ

#### 11.4.1. 脅威モデル（STRIDE）

各構成要素（端末アプリ／バックエンド／設定 UI／アドオン／基幹連携 API／同期チャネル）に対し、STRIDE 6 種を網羅する。

| 区分 | 脅威例 | 影響 | 対策 | 残留リスク |
| --- | --- | --- | --- | --- |
| Spoofing（なりすまし） | 盗難端末でのログイン／API クライアント偽装／共有 ID 流用 | 不正実績登録・データ漏洩 | パスワード＋端末バインド／TLS クライアント証明書／JWT 短寿命＋リフレッシュ／監査ログでの ID-端末-IP 三点照合 | 共有 ID 文化が残る現場 |
| Tampering（改ざん） | 端末 SQLite 直接編集／同期パケット改ざん／ログ削除 | 監査証跡の信頼性喪失 | SQLCipher AES-256／同期メッセージへ HMAC／追記専用イベントログ／ハッシュチェーン | 端末 root 化された場合 |
| Repudiation（否認） | 「私は実行していない」主張 | 規制適合不能 | 監査ログ（端末時刻・サーバ時刻両記録）／電子署名（§10.5）／追記専用ストア | 共有 ID 文化下で低下 |
| Information Disclosure（情報漏洩） | 紛失端末からの実績・メディア漏洩／ログ過剰収集／設定 UI からの権限越境参照 | 個人情報保護法／GDPR 違反 | DB／メディア暗号化／PII 最小収集／ログ匿名化／RBAC のフィールド粒度制御 | 高度な物理攻撃には限界 |
| Denial of Service（DoS） | 同期過負荷／ストレージ枯渇／アドオン暴走／巨大メディア投稿 | 業務停止 | レート制限／アドオン CPU／メモリ／実行時間制限（WASM）／ストレージ警告閾値／メディアサイズ上限 | ネットワーク経路の DoS は外部対策に依存 |
| Elevation of Privilege（権限昇格） | アドオンによるコア API 全アクセス／RBAC 抜け／WASM 脱獄 | 任意操作・データ改ざん | capability ベース権限（§17.4）／RBAC 既定 deny／署名検証／Wasmtime 最新追従 | 0-day を含む WASM ランタイム脆弱性 |

#### 11.4.2. 端末ローカル DB の暗号化

- SQLCipher（AES-256）を採用。鍵は OS Keystore（Android）／DPAPI（Windows）に保護。
- 端末紛失時のリモートワイプは §16 観測ダッシュボードからトリガー可能（オプション）。

#### 11.4.3. サプライチェーン

- 依存関係の SBOM（CycloneDX 形式）を CI 生成。リリース成果物に同梱。
- `cargo audit`／`cargo deny`／`npm audit`／Renovate を CI 必須化。
- リリース署名は OIDC ベース（§19.3）。

#### 11.4.4. 受入観点

- §11.4.1 の STRIDE 表に空白セルが無い。
- 各脅威の対策が §10／§13／§17 のいずれかの機能要件・テストへ紐付く。
- SBOM が成果物として CI から出力される。
- 重大度高の脆弱性は §22 即時トリガー対応。
- 端末暗号化／監査ログ追記不変性が単体テストおよび侵入試験で検証されている。

### 11.5. 性能

§5.2 を参照。

### 11.6. 信頼性・耐障害性

- 端末・サーバの突発停止でデータ損失が起きない。
- 受入観点: 任意ステップでの電源断試験で再起動後に直前状態へ復帰可能（§13.4.2 カオス演習で半期検証）。

### 11.7. サステナビリティ（グリーンソフトウェア）

製造業は世界の電力消費の主要源であり、本アプリ自身も電力を消費する。Green Software Foundation の SCI（Software Carbon Intensity, ISO/IEC 21031:2024）と整合する設計を採る。本節は §6（ハードウェア）／§16（観測）と接続し、競合 SaaS との差別化軸（§4.4 ライセンス・TCO ◎）の **持続可能性証跡** を裏付ける。

#### 11.7.1. 端末バッテリー寿命

| 指標 | 目標 |
| --- | --- |
| 8 時間連続運用時のバッテリー消費 | ≤ 80%（§6.3 推奨機種） |
| アイドル時 CPU 使用率 | ≤ 1% |
| 画面点灯時間最適化 | 操作なし 60 秒で減光、180 秒でロック（設定可） |
| 撮影・通信のバッチ化 | 同期は省電力時間帯に既定でバッチ |
| カメラ・センサーアクセス | 必要時のみ（既定 deny） |

#### 11.7.2. サーバ・コンテナ効率

- Rust／tokio の選定理由の一つは「同等機能の Java／Node より RSS 30〜70% 低、CPU 20〜50% 低」（公開ベンチマーク多数）。
- アイドル時の DB 接続プールは縮退（最低 2 接続）。
- ログ・メトリクスは構造化＋サンプリング（既定 10%）。
- 不要なサイドカー・エージェントは導入しない（§14.1 ルートディレクトリ最小化と整合）。

#### 11.7.3. SCI（Software Carbon Intensity）試算

導入企業ごとの SCI 値を `docs/04_運用/sci-report-template.md` のテンプレートで算出し、可視化を支援する。

```
SCI = ((E × I) + M) / R
```

- E = 電力消費量（kWh）
- I = 電力の炭素強度（gCO2eq/kWh、地域別）
- M = ハードウェア製造の embodied carbon（按分）
- R = 機能単位（例: 完了作業 1,000 件）

#### 11.7.4. 受入観点

- バッテリー寿命の CI 計測（端末エミュレータ／実機）で目標達成。
- サーバ idle RSS／CPU の CI 計測で回帰検出。
- SCI 試算テンプレが `examples/` に同梱され、導入企業が自社環境で実行可能。
- §16 観測ダッシュボードに「電力／推定 CO2eq」セクションが配置される。
- §22.1 半期サイクルで SCI トレンドをレビューし、悪化時は §22.4 是正フロー。

#### 13.4.1. ミューテーションテスティング

| 対象 | ツール | 目標 |

#### 13.4.2. カオスエンジニアリング

製造現場での突発障害（ネットワーク断・ストレージ満杯・電源断）は **常時発生** する想定。Principles of Chaos Engineering（Netflix, 2014）の手法をオンプレ前提に縮約適用する。

| シナリオ | 注入手段 | 検証 |
| --- | --- | --- |
| 端末ネットワーク断（10s〜24h） | `tc qdisc`／Wi-Fi トグル | §10.6 同期復旧 |
| サーバ DB 接続断 | iptables ルール／Docker pause | §10.3.1 リトライ・デッドレター |
| ストレージ満杯 | `fallocate` で残容量縮減 | §10.4.4 警告挙動 |
| 時刻ジャンプ | NTP 強制ずれ | §20.2 ズレ閾値検出 |
| 高 CPU 負荷 | `stress-ng` | §5.2 性能維持 |
| メモリ逼迫 | cgroup `memory.limit` | OOM Killer 後の自動復旧 |
| アドオン暴走 | テスト用無限ループアドオン | §17.5 kill 検証 |
| 電源断（端末） | バッテリ強制切断（実機） | §11.6 直前状態復帰 |

- 半期に 1 回、`docs/04_運用/カオス演習.md` に従い実施。結果は §22 サイクル／§29 リスク登録簿へ。

- ミューテーションスコア目標が CI で月次計測。

### 14.1. 実現項目

### 14.2. セットアップ容易性目標（初期決定）

メモ L61「セットアップは簡単さを意識する」を競合比較表 A8 ◎ の根拠まで押し上げるため、目標を以下まで強化する。

| 指標 | 目標 |
| --- | --- |
| サーバ初回起動 | `docker compose up` を 1 コマンド、5 分以内に Web UI 到達 |
| 端末初回設定 | アプリ起動から作業開始まで 10 分以内 |
| 手順書 | A4 1 ページに収まる |
| 事前検査 | `make doctor` 等のコマンドで OS／Docker／空きポート／メモリ／時刻同期の自己診断を 30 秒以内に完了 |
| サンプルデータ | デモ用マスタ・作業フロー・テストユーザーを既定で同梱し、初回起動直後に動作確認可能 |
| ワンコマンド初期化 | データ初期化・管理者作成・サンプル投入・端末ペアリング用 QR 生成を 1 コマンドで完結 |

| アンインストール | `docker compose down -v` ＋ アプリアンインストールでホストに残骸が残らない |
| ネットワーク要件の事前共有 | 必要ポート・ホスト名・証明書チェーンを `make doctor` 出力に含め、現地ネットワーク担当者と即時に共有可能 |

### 14.3. 容易性を支える機構

- 設定 Web UI 初期画面に **設定ウィザード**（推奨デフォルト → 高度設定の段階開示）を配置。
- 端末アプリは **QR コード経由のサーバ接続設定** をサポートし、URL／ポート／証明書の手入力を不要化。
- マスタ・フローは **CSV／YAML インポート** とテンプレート同梱で、ゼロからの定義を強要しない。
- 失敗時の **段階的フォールバック**（HTTPS 検証エラー → 自己署名許容モードの提案 → 操作ログ提示）で、現地の試行錯誤を最小化。

### 14.4. 受入観点

- §14.2 の各目標が **手順実機検証** で達成。
- 署名済みアーティファクトの配布経路（コンテナイメージ署名／APK 署名／MSI 署名）が定義され CI で検証。
- 更新失敗時のロールバック手順が存在し、リハーサル済み。
- セットアップ手順を未経験者が辿った際の所要時間と詰まりポイントが計測されている（個人開発下では作業者ロールを再現する形で代替計測）。

## 15. データ保全・スキーマ進化

### 15.1. バックアップ・リストア・スキーマ進化

実現項目: 端末 SQLite／サーバ PostgreSQL のバックアップ・リストア／前方・後方互換のスキーマ進化／端末↔サーバ間のスキーマ整合性チェック／互換マトリクス維持。

### 15.2. RPO／RTO 目標（初期決定）

| 規模 | RPO（最大データ損失） | RTO（最大業務停止） | 戦略 |
| --- | --- | --- | --- |
| 小 | 1 時間 | 4 時間 | サーバ PITR（WAL 保管）＋日次フルダンプ／端末側は WAL 同期のみ |
| 中（基準） | 15 分 | 1 時間 | PITR ＋ 6 時間ごとフルスナップショット／週次リストア演習 |
| 大 | 5 分 | 30 分 | ストリーミングレプリカ（同期）＋PITR ＋ 1 時間ごとスナップショット |

端末側 RPO は同期遅延（§5.2 ≤ 5s）と等価。バックエンド送信前の端末側ストレージ消失は端末故障時のみ発生し、§10.6 追記専用方針で再送可能な範囲を最大化する。バックアップは復号鍵を別チャネル管理する暗号化形式で保存。

### 15.3. 受入観点

- マイグレーション失敗時のロールバックがテスト済み。
- 旧端末×新サーバ／新端末×旧サーバの互換マトリクスが定義され CI で検証。
- §15.2 の RPO／RTO が DR 演習で達成可能であることが半期に 1 回検証される。
- バックアップが暗号化され、復号鍵が別チャネルで管理されていることが運用手順で保証される。

## 17. アドオン拡張の境界

### 17.1. 拡張ポイント

| 種別 | 例 | 動作層 |
| --- | --- | --- |
| イベントフック | 作業開始／中断／完了／失敗時に処理を挿入 | 端末・サーバ両方 |
| カスタム検証 | 完了条件の追加判定（画像分析・値域・外部 API 照合） | 端末 |
| カスタム画面 | 設定 UI／端末アプリへ追加パネルを挿入 | 端末・設定 UI |
| 追加プロトコル連携 | OPC UA／MQTT／独自 PLC／Modbus | サーバ |
| レポート／ダッシュボード | KPI 集計、外部 BI 連携 | サーバ |
| 通知チャネル | Slack／Teams／メール／音声合成／物理ランプ | サーバ |
| マスタ拡張 | 独自属性の追加と検証 | サーバ |

### 17.2. SDK 言語（初期決定）

| サポート段階 | 言語 | 採否理由 |
| --- | --- | --- |
| 一次 | Rust（`wasm32-wasi`） | 本体と統一／型安全／WASM 出力サイズ最小 |
| 二次 | AssemblyScript | TypeScript ライク／生産技術部門の参入障壁低減 |
| 評価中 | Go（TinyGo `wasm32-wasi`） | エコシステム広いが WASM 出力サイズ大 |
| 不採用 | Python | 性能・配布サイズが本用途と不整合 |

公式 SDK は Rust／AssemblyScript の双方を `addon-sdk/` 配下に同梱し、サンプル・テンプレートを提供する。

### 17.3. アドオン API surface（v1 初期決定）

| 領域 | 提供 API（抜粋） | 必須 capability |
| --- | --- | --- |
| 作業情報読取 | `getCurrentTask()`／`listSteps()`／`getContext()` | `task.read` |
| 作業実績書込 | `appendRecord()`／`markStep(id, evidence)` | `task.write` |
| メディア | `attachMedia(taskId, mediaRef)`／`readMedia(ref)` | `media.write`／`media.read` |
| HTTP アウトバウンド | `fetch(url, options)` ホワイトリスト URL のみ | `net.outbound:<host>` |
| ストレージ KV | `kv.get(key)`／`kv.set(key, value)` 名前空間別 | `storage:<namespace>` |
| 通知 | `notify(channel, message)` | `notify:<channel>` |
| UI 拡張 | `registerPanel(slot, component)` | `ui.extend:<slot>` |
| ロギング | `log(level, message)` | 既定許可 |
| 設定 | `getConfig(key)` 公開設定のみ | `config.read` |
| 暗号 | `crypto.sign(payload)`／`crypto.verify(...)` | `crypto.sign` |
| 時刻 | `now()`（サーバ時刻同期済み） | 既定許可 |

破壊的変更は MAJOR バンプ＋ 6 ヶ月の移行猶予（§19.3）。SemVer 準拠。`addon-sdk/` 配下で型定義（`addon.d.ts`／`addon.rs`）を版管理。

### 17.4. 権限モデル（capability ベース）

- アドオン `manifest.toml` に必要 capability を **明示宣言**（最小権限原則）。
- 導入企業の管理者が承認した capability のみ実行時に付与。
- capability 不足の API 呼び出しはランタイムが即拒否し監査ログへ記録。
- 既定値は **deny**。
- ホスト名・URL・名前空間は **glob 不許可** で完全一致のみ。

### 17.5. サンドボックス

- 端末側: Tauri 内 Wasmtime（メモリ上限 64MB／実行時間 100ms 既定／タイムアウト超で kill）。
- サーバ側: Wasmtime ＋ OS プロセス分離（Linux: cgroups／seccomp）。
- WASI 標準 capability（fs／net）は **無効化**。本書 §17.3 の API のみ提供。
- アドオン間通信は不可（コア API 経由のみ）。

### 17.6. 配布・更新

- アドオンは `*.wnaddon`（zip 形式）として配布。マニフェスト＋WASM＋アセット。
- リリース署名（OIDC）必須。署名不正アドオンはロード拒否。
- 更新はコアと独立。設定 UI からインストール／更新／無効化／削除可能。
- 監査・公平性のため、アドオンマーケットプレイス（OSS）を将来検討（§22 サイクル）。

### 17.7. 受入観点

- §17.3 の API surface が `addon-sdk/` の型定義として公開。
- サンプルアドオン（Hello Step／Slack 通知／OPC UA 連携 の最低 3 種）が動作。
- 悪意あるアドオンがコア・他アドオン・端末データへ越境できないことが §11.4 STRIDE Elevation of Privilege テストで検証。
- API バージョン互換ポリシー（移行猶予 6 ヶ月）が明文化され、CHANGELOG で追跡可能。
- capability 不足拒否ログが監査出力で確認可能。
- アドオンの実行時資源（CPU／メモリ／実行時間）の上限超過時に kill されることを長時間連続運転テストで確認。

### 18.3. 図表とドキュメンテーション

- 図はアスキーで描かない。draw.io で作成し PNG として書き出して Markdown に挿入する（メモ準拠／§23 にソースを用意）。
- draw.io の起動・PNG 書き出しは執筆者の責任で行う（自動化を `scripts/build-diagrams.sh` で後日整備）。

### 18.4. 一次入力の整理

- メモ.txt はすべての記述が本ロードマップ系へ取り込まれた時点で削除する（メモ L69）。
- 削除前の最終チェックは §24 メモ追跡可能性マトリクスで行う。反映先が空欄の行が存在するうちは削除しない。

### 18.5. LLM 主導開発のトレーサビリティ

メモ L33-35「Claude Code に企画・設計・実装・テストまで全部任せる」は本プロジェクトの最大の前提であり、同時に最大のリスク源である。LLM 出力の **再現性・出所・誤りの是正** を仕組みで担保する。

#### 18.5.1. LLM 出力の記録規約

| 項目 | 規定 |
| --- | --- |
| 利用モデルの版 | コミットメッセージ末尾の Co-Authored-By 行に明記（例: `claude-opus-4-7`） |
| 主要決定の根拠 | ADR（§9.4）に「LLM が示した代替案」「採用理由」「却下案」を残す |
| プロンプトの再現性 | 重要セッションのプロンプト要旨を `docs/llm-sessions/<日付>.md` に保存（PII 含まず／公開許容範囲のみ） |
| 出力の検証 | 受入観点（§13）／形式検証（§3.4.1）で機械的に確認 |
| 誤出力の是正 | `llm-error` ラベルで Issue 起票し §22.4 是正フロー |
| ハルシネーション対策 | 出典 URL 必須／一次資料以外を引用しない（§18.2） |

#### 18.5.2. LLM 知識の経年劣化対策

- LLM は学習時点の情報を持つ。本ロードマップの「初期決定」は §22.1 サイクルで再検証する。
- 競合・規格・ライブラリは特に経年で陳腐化しやすいため、§4.8 自動監視と組合せる。
- 「最新と思われる」根拠は出典 URL を必ず付け、リンク切れを CI で検出（`scripts/check-links.sh`）。
- LLM の世代交代時には主要 ADR を再評価する（モデル変更時の出力ドリフト検出）。

#### 18.5.3. 受入観点

- 主要 ADR に LLM の代替案と却下理由が記載されている。
- `docs/llm-sessions/` の更新が直近 30 日で 1 件以上ある（沈黙していない証跡）。
- リンク切れ検出 CI が稼働し、失敗時に PR を fail させる。
- `llm-error` ラベル件数を §22.1 で集計し、再発防止策が記録されている。
- モデル世代交代時の再評価記録が CHANGELOG に残る。

## 20. UX・文言・時刻・データ保護のガイドライン

### 20.1. エラーメッセージ規約（初期決定）

- 人を責めない（「あなたは入力を間違えた」ではなく「入力された値が範囲外です」）。
- 回復手順を必ず含める。
- 技術用語を避け、現場の言葉で表現する。
- 一意の **エラーコード** を併記し、サポート問い合わせ時に参照可能にする。
- 致命的エラーは音と視覚の双方でフィードバック。

### 20.2. 時刻・タイムゾーン（初期決定）

- 内部保存は **UTC**、表示時のみロケール変換。
- 端末は **NTP 同期必須**。同期失敗時は警告表示。
- 端末↔サーバ間で時刻ズレ閾値（既定 5 秒、§21 注 4）を超える場合は同期を保留し管理者通知。
- 監査ログには **端末時刻とサーバ時刻の両方** を記録。

### 20.3. データ保護方針（初期決定）

- 個人データ（氏名・顔写真・音声）の保存期間は導入企業が設定可能。既定 365 日（§21 注 5）。
- 越境移転は本アプリでは行わない（企業内ネットワーク前提）。導入企業がアドオンで送出する場合は規制遵守を導入企業の責任として明記。
- 削除要求への対応: 個人特定情報の削除をサポートし、作業実績の本体は匿名化して保持。
- 受入観点: 上記が運用手順として導入ガイドに明記されている。

## 21. 数値根拠（脚注集）

「初期決定」の数値が空証文にならないよう、出典を残す。出典は §22 サイクルで再評価する。

| 注 | 値 | 根拠 |
| --- | --- | --- |
| 1 | 画面遷移 ≤ 200ms | Doherty Threshold（IBM, 1982: 400ms 以下で連続的体験）／Card・Moran・Newell *The Psychology of Human-Computer Interaction* 即応閾値 100ms／Nielsen 1993 1s 以内。本アプリは 200ms を中央値、500ms を p95 とする |
| 2 | タッチ応答 ≤ 100ms | Card et al. 1991 知覚同期閾値／Microsoft Touch Interaction Design ガイドライン |
| 3 | 同期復旧 ≤ 5s | TCP 再送タイムアウト典型値（3s〜）／ユーザー忍耐閾値（NN/g 報告では 10s で離脱開始） |
| 4 | NTP ズレ閾値 5s | NTP 同期精度はミリ秒オーダーのため 5s は十分余裕／人間の認知単位（秒）にも合致 |
| 5 | データ保持 365 日 | 業界一般慣行。GDPR は具体年限を規定しないが「目的達成に必要な期間」を要請。製造業の品質記録は規格別に 1〜10 年保持されることが多く、365 日は最小ライン。導入企業設定可 |
| 6 | タッチターゲット 9mm | ISO 9241-411 推奨値 9mm（target zone）／Apple HIG 44pt ≈ 11mm／Microsoft Touch 9mm 推奨／Material Design 48dp ≈ 12mm。手袋操作を想定すると 11mm 以上が望ましく、推奨値は 11mm、最小ライン 9mm とする |
| 7 | カバレッジ 90% | NIST: 業界水準 85% C0／Google internal target 80% C0。ドメイン層は重要度高いため 90% を採用 |
| 8 | 連続稼働 24h | 製造現場は 2〜3 シフト制が一般的。1 シフト 8h × 3 + α を吸収するため 24h |
| 9 | オフラインログイン継続 7 日 | NIST SP 800-63B のパスワードライフサイクル管理慣行／監査異常検知の現実的閾値（週次レビュー前提）／日本の連休最大 7 日（GW・夏季・年末年始）を吸収／管理者権限取消の伝播遅延を 7 日以内に抑える要件 |

## 22. 改訂・是正サイクル

### 22.1. 定期サイクル

**起点日: 2026-05-09**（本ロードマップ初期確定日）。各サイクルの実施記録は GitHub Issues／Discussions に `governance/<期>` ラベルで管理する。

### 22.2. 即時トリガー

| トリガー | 対応 |
| --- | --- |
| 規制変更 | 即時影響評価 → §12 更新 |
| 競合に追い越される（評価で △ 検出） | §22.4 是正フロー起動 |

| 重大障害 | ポストモーテム → 再発防止策を §13〜§16 に反映 |
| ユーザー意見（GitHub Issues／Discussions） | 四半期で反映可否判断 |

### 22.3. 撤退条件

「妥協禁止」と「無謀な継続」は別である。以下の場合は機能撤退または戦略軸の変更を検討する。

- 同一軸で 3 期連続「△」が解消できない。
- 規制変更で対応コストが OSS 個人開発で吸収不能。
- 競合の技術進化により本アプリの差別化軸が消失（例: OSS で対応する競合が出現）。

### 22.4. 是正フロー

1. 検出 → GitHub Issue 起票（ラベル: `regression` または `competitive-gap`）。
2. 影響評価 → 関連 §（性能なら §5、競合なら §4、規制なら §12）に注記。
3. 修正計画 → ロードマップ更新と CHANGELOG 反映。
4. 検証 → CI／実機／競合比較表で確認。
5. 完了報告 → GitHub Discussions または Release Notes。

### 22.5. 受入観点

- 上記サイクルの実施記録（Issue／PR／CHANGELOG）が GitHub 上で追跡可能。
- 直近 1 年で発生したトリガーすべてに対応済み、または未対応理由が記録されている。

## 25. 教育・オンボーディング戦略

メモ L29「誰でも操作しやすい UI/UX」と L41「UIUX に徹底的に拘る」を **教育コンテンツ** で補強する。優れた UI でも、現場文化と用語の差異は必ず発生するため、教育は UI の妥協ではなく UI と並行する第二の品質軸として扱う。本章は §3.2.1 ペルソナと §11.3.1 ロケール拡張に追従する。

### 25.1. 対象別の教育設計

| 対象 | 目的 | コンテンツ | 評価指標 |
| --- | --- | --- | --- |
| 作業者 | 端末アプリで自走できる | 5 分動画＋3 分セルフチェック（多言語） | チェックの正答率 ≥ 90% |
| 班長 | 監視モード活用・是正フロー | 10 分動画＋ロールプレイ E2E | 不適合発生時の対応時間中央値 |
| 生産技術 | フロー設計・差分管理 | 30 分ハンズオン（サンプル基幹接続あり） | サンプルフロー作成完了率 |
| システム部門 | アドオン開発・運用 | SDK チュートリアル＋サンプル 3 種（§17） | サンプル動作率＋拡張 PoC 件数 |
| 監査・規制対応 | 監査ログ・電子署名 | 監査画面ツアー＋規制マッピング表 | 監査シナリオ通過率 |
| 開発者（OSS 貢献者） | コードベース理解 | アーキテクチャ動画＋ ADR 索引＋ contributing tour | PR 採用までの中央値 |

### 25.2. コンテンツ生成の方針

- 動画は OSS 提供者として無償・字幕付き・**CC BY 4.0** で配布（ロードマップ本体の Apache-2.0 と独立）。
- スクリプトは LLM 主導生成（メモ L33-35／§18.5）＋メンテナレビュー。
- 改訂は §22.1 半期サイクルで点検し、ロケール追加（§11.3.1）に追従。

### 25.3. 教育の循環設計

- 端末アプリで撮影した動画（§10.4）は、匿名化・PII 除去後に **教育素材ライブラリ** に取り込み可能（オプトイン）。
- 班長が「良い手本」と判定した実績を教育素材候補としてフラグ付与。
- 改善提案（§9.3.1 改善）は次バージョン教育素材に反映する循環を維持。
- 撮影 → 匿名化 → ライブラリ追加の経路は §20.3 データ保護方針と整合。

### 25.4. 受入観点

- 各対象別の評価指標が §22.1 サイクルで計測可能。
- 教育コンテンツの言語数が §11.3.1 段階に追従する。
- 教育素材ライブラリのオプトインが §20.3 と整合する。
- OSS 公開後 6 ヶ月で外部貢献者向け onboarding 動画が公開済。
- 教育コンテンツが §4.9 圧倒候補機能「教育素材ライブラリの循環設計」を裏付けている。

## 26. アンチゴール（やらないこと）集

メモ L7「完璧に上回る」を達成するには「やらないことを明示する」ことが不可欠である。スコープ拡散（scope creep）は妥協（§2）と並ぶ世界一の天敵である。本章はロードマップ全節に散在する「不採用」「非対応」「対象外」を **集約** して保守する。本章を介して各章の「不採用」決定が逆引き可能になる。

## 27. FMEA・障害分析（製造業特有）

製造業は IATF 16949 Section 8.3 で **FMEA（Failure Mode and Effects Analysis）** を要請しており、本アプリ自身および本アプリが介在する作業フローの両面で FMEA を扱う必要がある。本章は §11.4 セキュリティ／§11.6 信頼性／§22 改訂サイクル／§29 リスク登録簿と接続する。

### 27.1. FMEA の対象 3 層

本アプリは「作業ナビ」を介在することで、現場の FMEA 観点が二重化する。それぞれを独立に扱う。

| 層 | 観点 | 出典 |
| --- | --- | --- |

| Software FMEA（SFMEA） | 本アプリのソフトウェア固有故障モード（メモリリーク・競合・タイムアウト等） | IEC 60812:2018 Annex C |

### 27.2. 故障モード × 影響度 × 検出度（RPN）

## 30. 意思決定の可逆性（Type 1 / Type 2）

§2「妥協禁止」を運用粒度で支えるため、意思決定を **可逆性** で 2 分類する（Bezos 1997 株主書簡 Type 1 / Type 2 概念の採用）。すべての決定を同じ重みでレビューすると、軽微な決定が肥大化し、重大な決定が拙速化する。

### 30.1. 分類

| 種別 | 性質 | レビュー粒度 | 採用例 |
| --- | --- | --- | --- |
| Type 1（不可逆） | 撤退コストが高い／戻れない | ADR 必須／メンテナ承認＋外部レビュー機会 | ライセンス選定（§19.1）／ストレージ・暗号方式（§11.4.2）／公開 API surface（§17.3）／ユビキタス言語の主要用語（§28）／形式化 notation（§3.4.1）／同期形式モデル（§10.6.1） |
| Type 2（可逆） | 試行錯誤可能／撤退容易 | 通常 PR レビュー | UI 細部／ローカライズ初版用語選択／既定値の数値（一定範囲内）／非破壊的なリファクタリング |

### 30.2. Type 1 決定のチェックリスト

PR テンプレートの「Type 1 決定」項目チェック時に必須。

- ADR が `docs/adr/` に追加されているか。
- 代替案（≥ 2 件）と却下理由が明示されているか。
- 撤退条件（§22.3）が定義されているか。
- §24.2 出所表に追記されているか。
- §29 リスク登録簿に該当行があるか／無ければ追加するか。
- §27 FMEA への影響が評価されているか。
- 既存 Type 1 決定との矛盾がないか（§28 用語集／§17 capability ／ §19.1 ライセンス）。

### 30.3. Type 2 決定の運用

- ADR 不要。コミットログと PR 説明で十分。
- 集合的に Type 1 化する兆候（同じ Type 2 が頻繁に揺れる）が現れたら Type 1 へ昇格し ADR 化。
- Type 2 決定での軽率な変更が業務影響を生んだ事例は §22.4 是正フローでレトロを実施。

### 30.4. 受入観点

- ADR 件数が Type 1 決定件数と整合（CI で件数チェック）。
- Type 1 決定の撤退条件が空欄の ADR を CI で検出。
- §24.2 行追加と Type 1 ADR 追加が同期している。
- 半期に 1 回、Type 2 決定の揺れ頻度を `git log` 解析で集計し、Type 1 化候補を検討（§22.1）。
- Type 1 決定変更時は CHANGELOG と公開ポストモーテム（§2.3）で告知。

## 31. 観測性 SLO／SLI／エラー予算

§16 観測性・運用を「世界一」（§1, §2.1）として運用するため、Google SRE Workbook (Beyer et al. 2018) のサービスレベル方法論を **個人開発 OSS かつオンプレ前提** に縮約適用する。SaaS でないため可用性は導入企業のインフラに依存するが、本アプリ起因の不良を切り分けるため SLI／SLO はアプリ側で測れる範囲に限定する。

### 31.1. SLI（サービスレベル指標）

### 31.3. エラー予算と運用

- **エラー予算**: 1 − SLO 値 を予算とし、消費が早いと検知されたら新規機能の投入を抑制し、信頼性投資へ振替（SRE Workbook の budget policy 縮約版）。
- **計測公開**: 値は端末・サーバの自走計測のみで完結し、外部 SaaS へ送信しない（§19／§37）。導入企業は `docs/04_運用/slo-dashboard.md` の Grafana テンプレで自社環境に取り込み可能。
- **逸脱対応**: 月次でエラー予算 50% を超過した SLO は §22.4 是正フロー起動。連続 2 期超過は §22.3 撤退条件の検討対象。
- **SLO の改訂**: §22.1 半期サイクルで再評価。値の変更は Type 1 決定（§30）に該当しない（撤退コストは低い）が、撤退・廃止は Type 1 として ADR 化。

### 31.4. アラート設計（バーンレート）

Google SRE Workbook 6.5 章のマルチウィンドウ・マルチバーンレート手法を採用。1 時間バーンレート 14.4× and 5 分窓 14.4× で短期アラート、6 時間バーンレート 6× and 30 分窓 6× で中期アラート、3 日バーンレート 1× で長期アラート。

- アラート経路は §17 アドオン経由（Slack／Teams／物理ランプ／メール）に切り出し、コアは Webhook を投げるのみ。
- 騒音回避: §9.2.4 ダーク・パターン禁止の延長として「アラートで人を責めない」「重要度に応じた段階提示」を維持。

### 31.5. 受入観点

- SLI／SLO が `docs/04_運用/slo-dashboard.md` の Grafana テンプレで版管理されている。
- §27 FMEA・§29 リスク登録簿との双方向リンクが CI（`scripts/observability-link-lint.sh`）で検査され、孤児な SLI／SLO は fail。

- §11.7.1 バッテリー指標と SLI-10 が同期（§9.6 月次スナップショット）。

## 32. OSS リリース・エンジニアリング

メモ L5「OSS として公開予定」と §19（ライセンス・ガバナンス）を実装可能なリリース運用へ落とす。SemVer のみでは「いつ何をどう壊すか」が決まらないため、**リリーストレイン・互換性レーン・廃止予告（deprecation）プロセス** を初期決定として固定する。

### 32.1. バージョニング規約（初期決定）

| 種別 | 規約 |
| --- | --- |
| 採用 | SemVer 2.0.0 ＋ Conventional Commits 1.0.0（§9.4 と整合） |
| 公開 API | OpenAPI 3.1（§10.3）／アドオン API surface v1（§17.3）／フロー定義スキーマ（§10.2）／監査ログスキーマ（§11.4.1）／メディア証跡フォーマット（§10.4.5）／用語集の主要用語（§28）|
| プレリリース | `MAJOR.MINOR.PATCH-rc.N`（リリース候補）／`-beta.N`／`-alpha.N` |
| ビルドメタ | `+sha.<short>`／`+date.<YYYYMMDD>` |
| LTS | §19.3 に従い X.0.0 から 18 ヶ月 |

### 32.2. リリーストレイン

個人開発でも「いつリリースが来るか分かる」状態を保つため、固定間隔のリリーストレインを採る（Spotify 2014 年公開のリリーストレイン手法を縮約）。

| トレイン | 周期 | 内容 | 起点 |
| --- | --- | --- | --- |
| パッチトレイン | 隔週（偶数週金曜） | バグ修正・セキュリティ修正のみ | 2026-05 起点 |
| マイナートレイン | 四半期（§22.1 サイクルと同期） | 後方互換な機能追加 | 同上 |
| メジャートレイン | 必要時のみ（最低 18 ヶ月間隔） | 破壊的変更を含む。§32.4 deprecation 予告必須 | LTS 期間と整合 |
| ホットフィックス | 即時 | §22.2 即時トリガー由来の重大欠陥／脆弱性 | CVSS ≥ 7.0 で即時 |

「乗り遅れたら次のトレインへ」の原則で、トレインを引き止めない（§35 §33 意思決定の遅延戦略と整合）。

### 32.3. 互換性レーン

| レーン | 対象 | 判定 |
| --- | --- | --- |
| API | OpenAPI／アドオン API surface | `openapi-diff` ／ `cargo semver-checks` で破壊検出 |
| データ | DB／SQLite スキーマ | §15 互換マトリクス／マイグレーション dry-run |
| 永続フォーマット | 監査ログ／メディア証跡／フロー定義 | フィクスチャ往復テスト（旧版 → 新版 → 旧版）|
| UI 外部依存 | URL／QR ペアリング／インストーラパス | 互換性テスト＋ドキュメント更新検査 |
| 翻訳 | §11.3.1 ロケールキー削除／改名 | 翻訳キー lint（孤児・欠落・ID 衝突）|

### 32.4. 廃止予告（deprecation）プロセス

| 段階 | 期間 | 措置 |
| --- | --- | --- |
| 予告 | 公開後最低 1 マイナー版 | CHANGELOG／ドキュメント／実行時警告（呼び出しログ） |
| 警告強化 | 次マイナー版 | UI／ログで明示／`@deprecated` 注釈／`Deprecation` ヘッダ |
| 削除 | メジャー版 | 削除。移行ガイドを `docs/04_運用/migration-<from>-to-<to>.md` に同梱 |
| 削除猶予 | アドオン API は 6 ヶ月（§17.3／§19.3） | 同上 |

### 32.5. リリースアーティファクトと検証

- アーティファクト: コンテナイメージ／APK／MSI／アドオン SDK／OpenAPI 仕様／SBOM（CycloneDX）／チェックサム／署名（OIDC）。
- 署名検証: `cosign verify` 等を CI のリリースジョブで強制（§19.3 整合）。
- リリースノート: Conventional Commits から自動生成（§9.4）。手動で「重大変更」「セキュリティ修正」「廃止予告」セクションを補足。
- ロールバック: 直前安定版へ戻す手順を `docs/04_運用/rollback.md` に常備。サーバ側 DB マイグレーションは **必ず可逆 or 二段階**（add then remove）。

### 32.6. 受入観点

- パッチ・マイナー・メジャー・ホットフィックスのトレインに **直近 1 年で漏れが無い**（§22.1 で集計）。
- すべての破壊的変更が §32.4 予告 → 警告 → 削除の 3 段階を経ている（CI で `BREAKING CHANGE` を含む PR は予告版の存在を検査）。
- リリースアーティファクトに署名・SBOM が同梱されない PR は CI fail。
- §32.3 互換性レーンの自動検査（openapi-diff／cargo semver-checks／フィクスチャ往復）が CI に登録され緑。
- §22.2 即時トリガー時のホットフィックス所要時間（検知 → 配布）が CVSS ≥ 7.0 で 72 時間以内。
- §17.3 アドオン API の deprecation が 6 ヶ月猶予を満たす（時刻計測テスト）。

## 33. 意思決定の遅延戦略（YAGNI 運用と Type 1/2 補完）

「妥協禁止」（§2／メモ L59）と「個人開発の維持コスト」（§19.5／メモ L3）の両立には、**決めなくてよい決定を意図的に遅らせる** 規律が必要である。§30 Type 1/2 が「決めるときの粒度」を扱うのに対し、本章は「決めるタイミング」を扱う。沈黙の妥協（§2.2）と区別するため、遅延は出所と再評価期日を残した状態で行う。

### 33.1. 遅延可能な決定の判定

次の 3 条件をすべて満たす決定のみ遅延可能。一つでも欠けるなら §30 Type 1 として即決する。

1. **可逆性**: §30 Type 2 に分類できる（撤退コスト低）。
2. **未確定の入力**: 決定根拠となる情報が公開資料・OSS／規格動向で 12 ヶ月以内に明確化する見込み。
3. **下流非ブロック**: 当該決定を保留しても下流タスク（§23 依存関係）が止まらない。

### 33.2. 遅延の記録規約

遅延は「未決定」ではなく **明示的な保留** として記録する。沈黙の妥協（§2.2）と区別するための手続き。

| 項目 | 規定 |
| --- | --- |
| 保留 ID | `DEFER-<YYYYMMDD>-<short>` |
| 保留対象 | 決定の主題（1 行） |
| 保留理由 | §33.1 のどの条件で遅延可能と判定したか |
| 既定の暫定値 | 保留中に採用する安全側の暫定値（§9.2.3 デフォルト効果） |
| 再評価トリガー | 期日・入力到来・上流の §22.2 トリガー等 |
| 影響評価 | 保留により発生する技術的負債・受入観点の暫定 |
| 保管場所 | `docs/adr/deferred/<DEFER-...>.md` |

### 33.3. 暫定値の安全側既定

遅延中はゼロ値・空欄を残さない。§9.2.3 デフォルト効果と整合した安全側暫定を必ず置く。

| 領域 | 安全側暫定の例 |
| --- | --- |
| しきい値（性能・容量） | §5.2 中規模目標値を採用 |
| 既定権限 | deny（§17.4） |
| 既定保持期間 | 365 日（§20.3） |
| 既定ロケール | 日本語＋英語（§11.3） |
| 既定認証 | ID＋パスワード（§10.5） |

### 33.4. 再評価の強制

- 保留 ID の再評価期日が CI（`scripts/lint-deferred.sh`）で監視され、期日超過の保留があれば fail。
- 保留期間は **最長 12 ヶ月**。延長は ADR（§9.4）に理由を残し、§22.1 で承認。
- 期日到来時に「再決定」「延長」「Type 1 への昇格」のいずれかを選び、結果を ADR 化。

### 33.5. 遅延と Type 1/2 の境界

- Type 1（§30.1） → 遅延禁止。即決。
- Type 2 + §33.1 三条件成立 → 遅延可。`DEFER-` で記録。
- Type 2 + 三条件のいずれか不成立 → 即決。記録は通常 PR レビューで足りる。
- 沈黙の妥協（§2.2） → 「決定の存在自体が記録されない」状態。本章とは厳密に区別され、CI で検出される。

### 33.6. 受入観点

- `docs/adr/deferred/` の保留 ID が CI で再評価期日を監視され、期限切れ 0 件。
- §2.2 沈黙の妥協ラベル（`silent-compromise`）と `DEFER-` の Issue 件数が独立に集計されている（区別がついている）。
- 保留期間 12 ヶ月超過の延長件数が §22.1 で公開される。
- 主要 ADR（§30.2）から「遅延候補としての検討と却下／採用」の経路がたどれる。
- 暫定値が §33.3 表のいずれかに合致しているか、または ADR で個別の安全側根拠が示されている。
