## 1. エグゼクティブサマリ

1. 本企画は、**直接作業者がライン生産現場で段取り替え時に単位作業をナビゲーションする** 場面を楔として打つ。楔の主便益は段取り替え時間（=ライン停止時間）の標準偏差低減と、改訂反映時間の即応性である。楔の射程は **中小製造業のうち MES ／ 基幹システムを保有する層** に限定する（§3.1 末尾）。
2. 成果物は、端末アプリ（Android／Windows、Tauri、SQLite ローカル、再生専用）と、メンテナンス Web（Docker 配布、SPA+API、PostgreSQL）と、共通基盤（手順 DSL／実績スキーマ／同期エンジン）の三層である。Custom 領域は手順 DSL の TWI-JI 三段一級市民化と、オフライン同期エンジンの 2 点に絞る（§9）。
3. ライセンスは **Apache-2.0** に確定（§14.1）。単一メンテナで意思決定速度を最優先する（BDFL）。バス係数 1 を §16.1 シナリオ 1 で正面から扱い、エクスポート互換（§13.4）でユーザーの離脱経路を保証する。SLO は単一メンテナ前提の現実値に書き直し、月間稼働率の数値約束は行わない（§13.1.2）。
4. 4 つの代替（紙ラミネート手順書／商用 e-SOP／Excel・kintone 自前構築／何もしない）に対し、楔の場面で本アプリが優位になる根拠を §10 で論理的に閉じる。Moat の核は TWI-JI 三段の必須属性化（認知的優位）と、ISO 9001 第7.5 項対応（スイッチングコスト）である。一次情報の不在は §3.4 の公開資料調査運用で代替し、人間ヒアリング不実施（Q38）の方針を維持する。
5. 公開後 1 年で GitHub Stars 100 未満（K1）、または楔現場で段取り替え時間の標準偏差が紙比で改善しない（K2）場合、本企画は方針転換または縮退を実施する（§15.2）。Pre-mortem の主原因は「一次情報なしで楔を打ったこと自体が間違いだった」（§16.0）を最上位に置き、シナリオ 2／3／5 をその発現形として位置づける。本人再判断対象として残るオープンクエスチョンは 3 件（§17.1）に絞った。

## 2. PR-FAQ（CLAUDE.md §1.1）

### 2.1 プレスリリース（完成して世に出た瞬間の1ページ）

**見出し**: 段取り替えの "迷い" を消す、ライン生産現場のためのオフライン作業ナビ OSS が公開。

**サブ見出し**: 紙の「止まらなさ」と、デジタルの即応改訂を、Docker 1 コマンドで両立する。ライセンスは Apache-2.0。

**本文**:
work-navigation-app は、ライン生産現場の直接作業者が、段取り替え時に手元の端末で単位作業をナビゲーションするための OSS である。タブレット端末（Android／Windows）はオフラインで動作する。手順書はメンテナンス Web で承認・改訂し、端末は差分同期で正本を取得する。手順は TWI-JI（仕事の教え方）の「主要ステップ／急所／急所の理由」を必須属性として扱う。商用 SaaS のような月額ライセンス費はなく、Docker 1 コマンドで自社オンプレに立てられる。

### 2.2 想定 FAQ（5 問、Q34 L／O／Q を含む）

**Q1. 学習コストはどの程度か。**
A. 手順 DSL は YAML / Markdown 互換で、TWI-JI 三段の 3 属性を埋めるだけで成立する。Excel テンプレートからの一括取込（§13.4、Q28-B/H）と、Markdown ／ YAML ／ PDF へのエクスポート（§13.4、Q28-L/N）を双方向で備え、ロックインを発生させない。

**Q2. 既存の基幹システムと連動できるのか。**
A. REST ／ GraphQL を一次インタフェースとし（Q10-A）、認証情報は本アプリで保管しない。連動アダプタはプラガブル設計とし、現場ごとの差分は導入先で実装する（§14.3）。基幹側に API を持たない現場は楔の対象外であり、§15.1 H8（楔の対象現場が十分に存在する）として明示する。

**Q3. 商用 SaaS との違いは何か。**
A. 商用 SaaS の優位は導入支援と SLA 付きサポートにある。本アプリは Issue テンプレート、CONTRIBUTING.md、導入チェックリスト（§14.4）でこれを代替する。代替であり同等ではないことを明記する。一方、本アプリの優位は (a) 月額ライセンス費が発生しない、(b) データの保管場所が導入先に閉じる、(c) DSL ・ スキーマがすべて公開される、の 3 点である。

**Q4. 個人開発の継続性は。**
A. §16 Pre-mortem は **シナリオ 0（一次情報なしで楔を打ったこと自体が間違いだった）** を最上位の原因として正面から書き、シナリオ 1（メンテナ離脱）を主軸として書く。エクスポート機能（§13.4）と semver 互換性ポリシー（§13.3）を実装する。Apache-2.0（§14.1）により fork 自由を保証する。これによりユーザーの離脱経路は本企画書の信頼性を担保する。

**Q5. 個人実績の労務的扱いは。**
A. 個人実績の保持期間は導入先で設定する（§13.5、Q26-D）。本人による開示機能を備える（Q26-I）。導入には労使合意・就業規則改訂を要する旨を導入チェックリストに明記する（Q26-M）。

## 3. 背景と根拠の整備状況（CLAUDE.md §7-3）

### 3.1 背景

ライン生産現場の段取り替えは、量産製造業における主要な改善対象である。トヨタ生産方式の「段取り替え時間の短縮（SMED）」が 1969 年から体系化されており、ライン停止時間の縮減はライン稼働率の主要決定因である [新郷 1985]。一方、現場で参照される手順書は、紙のラミネート手順書または社内 Excel で運用されることが多い。改訂反映には手作業の差替えを要する。

商用 e-SOP（Tulip／Operations1／カミナシ／Teachme Biz／i-Reporter 等）は手順電子化を提供する。中小製造業ではライン端末数（数十〜数百台）に対する月額ライセンス費が導入障壁となる。社内 Excel ／ kintone ／ SharePoint による自前構築は、属人化と保守不能化を招く。

本企画は、楔の場面（§6.1）でこれら 4 つの代替（紙／商用／自前／無策）に対し優位を成立させる OSS を提供する。

楔の射程はさらに **中小製造業のうち MES ／ 基幹システムを保有しライン生産計画を電子的に管理する層** に限定する。MES 未保有層に対する適用は楔の対象外とし、仮説 §15.1 H8（REST／GraphQL を持つ基幹システムを採用したライン生産現場が楔の対象として十分な数だけ存在する）の検証結果に従い、検証成立後にのみ射程拡大を検討する。

#### 3.4.1 採用しない手段

人間ヒアリング・現場観察・試作版観察・商用有償トライアル取得は本企画書の最終版まで実施しない（Q38 方針）。これらの代わりに上記公開資料調査で核心仮説の暫定支持／反証データを集める。**人間と接触する代替手段は OSS 公開後の自発的 Issue ／ PR ／ 試行導入報告のみ**とする（§15.2 K3 ／ K4 ／ K7 で観測）。

#### 3.4.2 観測値の本企画書への取込

- 容量見積（§13.1.1）の N 数 ／ 規模数値は R-10〜R-12 の中央値で v0.7.x 以降に再校正する。
- 競合比較（§10.2）の価格・機能は R-13 の四半期観測で更新し、版ごとに観測日を本文に明記する。
- Kill Criteria（§15.2 K1 ／ K8）の閾値は R-14 の類似 OSS スナップショットで再校正する候補とし、ADR-0009 で本人確定する。

#### 3.4.3 限界の自認

公開資料調査は **三現主義（現場・現物・現実）の "現場・現物" を完全には満たさない**（CLAUDE.md §8 アンチパターン）。本企画書はこの限界を Pre-mortem の主原因（§16.0）として正面から扱う。OSS 公開後に第三者から得られる Issue ／ PR ／ 試行導入報告が、最終的に三現主義の不足を埋める唯一の経路である。

## 4. 課題定義（CLAUDE.md §1.2 §7-4）

### 4.1 主軸 JTBD（楔の Job）

書式: `When _____ , I want to _____ , so I can _____ .`

> **J-A**: When 段取り替え中にライン停止が始まったとき, I want to 手元の端末で次の単位作業の手順を迷わず確認したい, so I can ライン停止時間を最小化する。 (Q35-A)

J-A を主軸とする根拠を 4 点で示す。

1. **数値化しやすさ**: 段取り替え時間はライン停止時間として直接計測される。サイクルタイム分布の標準偏差（§13.6、Q21-H）が改善指標として成立する。
2. **既存方法論との接続**: SMED [R-1] は段取り替えを「内段取り（停止中）」と「外段取り（稼働中）」に分解する。本アプリは内段取りの単位作業をナビゲーションする。
3. **本人の選択**: 本人ヒアリング Q35-A で J-A を選択済み。
4. **物理制約の収束**: ライン現場の物理制約（手袋・片手・騒音）は楔範囲で安定し、§6.3 の数値が「想定外の現場」で揺れない。

### 4.2 隣接ジョブ（楔の派生・射程内）

| ID | Job | 楔との関係 |
|---|---|---|
| J-B | 多品種混流での機種違い切替（Q35-B） | 段取り替えと連続的に発生し、同じ単位作業 DSL で扱える |
| J-C | 新人〜半年作業者の自習（Q35-C） | TWI-JI 三段（§8.4.1）の副産物として教育用途が成立する |

### 4.3 楔の射程外（§12 非スコープに移送する Job）

| ID | Job | 射程外とする理由 |
|---|---|---|
| J-D | 設備保全のトラブル対応（Q35-D） | 起動頻度が不規則／状況分岐が外部要因（設備状態）依存で予測不能／タクトが存在せず段取り替え時間に類する基準 KPI が無い。§8.4.1 の DAG ＋条件分岐は技術的に表現可能だが、楔の主便益（停止時間の標準偏差低減）と異なる経済構造を扱うことになり、単一メンテナでの楔崩しを招く |
| J-E | 班長・職長の進捗俯瞰（Q35-E） | 片手・小画面 UI と矛盾、俯瞰用画面の物理制約は楔と異なる |
| J-F | IE の標準時間見直し（Q35-F） | 楔の改善ループ（§13.6）の派生として実績は提供するが、IE 専用画面は持たない |
| J-G | 品証の証跡確認（Q35-G） | §13.5 の実績出力で代替する、専用画面は持たない |
| J-H | 手順作成者の改訂作業（Q35-H） | メンテナンス Web の改訂機能（§8.3）で代替する、現場連携専用画面は持たない |

J-D／J-E は前版で §17 OQ-2／OQ-3 として本人再判断対象だったが、本版で楔維持（§6.1）を本決定化し、両 Job は §12 NS-19／NS-20 として完全に楔の射程外として確定した（経緯は §17.2）。

### 4.4 Root cause 分析（紙運用が現役競合である理由）

> **主張**: 紙のラミネート手順書がライン生産現場でなお正本である根本原因は、手順表現の貧困さではなく、紙が持つ 3 つの利点である。

| 利点 | 内容 | 出典 |
|---|---|---|
| 止まらない | 端末・電源・Wi-Fi を要求しない | 物理的事実 |
| 一覧性 | A3 1 枚で工程全体を視覚的に把握できる | 物理的事実 |
| 監査証跡 | ISO 9001 第7.5 項の文書化情報として物理的証跡となる | R-4 |

> **So what**: 本アプリが楔の場面で紙を上回るには、機能を増やすのではなく、上記 3 利点を継承しつつ即応改訂を加える設計が必要である。これが §11 Moat の核となる。

## 5. 「作業」概念の階層定義（CLAUDE.md §2.1 必須）

### 5.1 採用する階層

| 階層 | 例 | 時間スケール | 楔における扱い | Q |
|---|---|---|---|---|
| 工程 (Process) | 塗装工程 | 時間〜日 | 文脈として保持（パンくず・割当の上位コンテキスト） | Q2-A |
| 単位作業 (Unit Work) | 右ドア塗装 | 数分〜数十分 | **楔の主たるナビ単位**（端末 1 セッション = 1 単位作業） | Q2-B |
| 要素作業 (Element Work) | マスキング／吹付／乾燥 | 数十秒〜数分 | 単位作業内の節（DAG ノード） | Q2-C |
| 動作 (Motion / Therblig) | 手を伸ばす／掴む | 秒以下 | 取扱対象外（§12 NS-1） | Q2-D 未選択 |

図 01 は単位作業を主たるナビ単位として太枠で示す（`./img/01_作業階層_v1.png`）。

### 5.2 用語の不混用ルール

- 「ステップ」= 要素作業またはその内部の最小不可分単位。
- 「ジョブ」= JTBD 文脈の Job（J-A など）。データモデルの動作セッションは「セッション」と呼ぶ。
- 「セッション」= 端末側の 1 起動 = 1 単位作業の遂行。
- 「手順」= 単位作業 1 件分の DSL 文書。
- 「タスク」= 割当の単位（§8.5）。

§18 用語集と §5 で一意に定義する。

### 5.3 階層をまたぐデータモデル（§8.4.1 と整合）

要素作業の内部は DAG（前提・並列・合流、Q19-D）と条件分岐（不良時／設備異常時、Q19-B）を許す。各要素作業は TWI-JI 三段を必須属性として持つ（Q19-E）。線形シーケンス（Q19-A）は DAG の縮退ケースである。木構造（Q19-C）は工程→単位作業→要素作業の階層に対応する。

## 6. 楔と利用文脈

### 6.1 楔の宣言

本企画は次の 1 点を楔として打つ。

> 役割: **直接作業者**（オペレータ本人、Q1-A）
> 生産形態: **ライン生産**（連続流れ・タクト管理あり・順序情報が基幹システムから供給、Q3-A）

### 6.2 楔の選定根拠

1. **改善効果の数値化**: 段取り替え時間（=ライン停止時間）はライン稼働率の主要決定因 [R-1]。サイクルタイム分布の標準偏差を §13.6 で計測する。

3. **既存資料との接続**: 標準作業3票の「標準作業組合せ票」（[R-3]）が段取り替え時間を扱う。Q28-H で公式互換テンプレートを提供する。

## 8. ソリューション概要と境界

### 8.1 全体構成（図 04）

図 04（`./img/04_アーキテクチャ概観_v1.png`）は次の 4 要素と境界を示す。

- (a) 端末アプリ（Tauri、Android／Windows、再生専用、SQLite ローカル）
- (b) メンテナンス Web（Docker、SPA+API、PostgreSQL、Keycloak／Authentik 等の認証基盤を委譲）
- (c) 基幹システム（REST／GraphQL 連動）
- (d) OSS リポジトリ（GitHub Releases、APK／MSIX／Docker イメージ配布）

端末↔Web は差分同期（プル＋プッシュ）。Web↔基幹は REST／GraphQL（プラガブル）。OSS リポジトリ↔導入先は直接配布。

### 8.2 端末アプリ（再生専用）

- プラットフォーム: Android（中位機〜産業用）、Windows（民生〜産業用）。
- 技術スタック: Tauri ＋ Web（HTML／CSS／JavaScript）。Q24-D。メンテナンス Web の UI コードを共有する。
- ローカル永続化: SQLite。Q39-H。
- 機能: 単位作業の再生／ステップ完了入力（§8.6）／写真・QR・数値の証跡／実績ローカル蓄積／差分同期／ハプティック・視覚フラッシュ通知。
- 機能境界: 再生専用。手順の作成・編集はメンテナンス Web 側のみ（Q4 回答）。
- 配布: APK 直接配布（Q23-C）／MSI ・ MSIX 直接配布（Q23-F）。ストア配布・MDM 配布は §12 NS-5。

### 8.3 メンテナンス Web

- ホスティング: 各導入先の自前ホスト。Q7-B。
- 技術スタック: SPA（React／Vue／Svelte／SolidJS のいずれか、選定は ADR-0004）＋ API。Q24-I。
- 永続化: PostgreSQL。Q39-A。メディア（画像・短い動画・音声）は同 DB のラージオブジェクトまたは隣接オブジェクトストレージ（選定は ADR-0005）。
- 機能: 手順の作成・編集・承認／改訂版管理／実績ダッシュボード（§13.6）／改善発議／権限管理。
- 配布: docker-compose.yml と DB 同梱例（Q23-J）。Helm Chart ／ ワンライナーインストーラは §12 NS-6。

### 8.4 共通基盤（Custom 領域・§9.3 の賭け）

#### 8.4.1 手順 DSL（楔 + TWI-JI 三段）

- 上位形式: YAML / Markdown 互換。独自 DSL（Q19-G）は採用しない。
- 構造: 木構造（工程→単位作業→要素作業、Q19-C）＋ DAG（要素作業内部の前提・並列・合流、Q19-D）＋ 条件分岐（Q19-B）。線形シーケンス（Q19-A）は DAG の縮退ケース。
- 必須属性（要素作業ごと）: 主要ステップ／急所／急所の理由（TWI-JI 三段、Q19-E、[R-2]）。
- 難易度属性: 簡略表示／詳細表示の切替に使用（Q27-M）。
- メディア: テキスト＋静止画＋短い動画（10 秒以内）＋音声（騒音下で補助、Q11-A／B／C）。3D ／ AR ／ SVG 動的ハイライトは §12 NS-3。

#### 8.4.2 実績スキーマ

- イベントソーシング採用（Q39-O）。実績は不変イベントとして追記する。
- 集計用ビュー: PostgreSQL の MATERIALIZED VIEW を併存（Q39-P）。
- スキーマ進化: 双方向移行を目標とする（Q39-N）。実装困難な場合は単方向移行に退避し、ADR で個別判断する（Q39-M）。
- 監査ログ: 追記専用＋ハッシュチェーンで改ざん耐性を持つ（Q30-C）。

#### 8.4.3 同期エンジン

- トリガー: プル（起動時／定期ポーリング／手動）＋ プッシュ（WebSocket／SSE）。Q20-A／B。
- 粒度: 差分同期（手順 ID＋版番号＋ハッシュ）。Q20-E。
- 実績アップロード: リアルタイム（接続あれば即時、無ければキュー蓄積）。Q20-G。
- コンフリクト方針: メンテナンス Web を絶対正本とし、端末側の手順改変は不可（Q20-J）。実績メモ・写真添付は端末→サーバの後勝ち（Q20-K）。
- 時刻整合: 端末ローカル時刻＋サーバ時刻の両方を記録（Q20-L）。NTP 同期不能端末では実績タイムスタンプに「推定」フラグを付与（Q20-M）。
- 重複送信防止: idempotency key を必須（Q32-J）。

### 8.5 割当・指示の入口（楔の現場で「朝、端末を開いた瞬間」を成立させる）

楔はライン生産であるため、基幹自動取込を主、QR 引き当てを副とする。

- 主: 基幹システム ／ MES から当日の生産計画を取り込み、ライン×時間帯で自動割当（Q31-A／F）。
- 副: 現品票・工程指示書のバーコード／QR 読取で都度引き当て（Q31-E）。
- 班長手動割当（Q31-B）は補助として備える。
- 端末側プル選択（Q31-C）／勤務シフト CSV 推定（Q31-D）は採用しない。
- 並行参照（Q31-J）／途中交代の引継ぎ（Q31-K）／中断・割込からの復帰位置記憶（Q31-L）を持つ。

### 8.6 ステップ完了の確証方法

ステップ種別に応じてシステム側が確証方式を強制する（手順設計者は選択しない）。

## 10. 4 代替に対する論理閉じ（CLAUDE.md §3 §3.2）

> 楔の場面（直接作業者×ライン生産×段取り替え×単位作業）における 4 代替に対し、本アプリが優位を成立させる根拠を示す。各代替について、ユーザー側の合理性 ／ 本アプリの上回り ／ 乗り換えコストとそれを超える便益 を順に書く。

### 10.1 代替 A: 紙のラミネート手順書

**ユーザー側の合理性（なぜなお選ばれているか）**:
端末・電源・Wi-Fi を要求しない（止まらない）。A3 1 枚で工程全体が見渡せる（一覧性）。書き込み・付箋・指差しが自由（改善メモが正本に重なる）。ISO 9001 第7.5 項の物理証跡として監査時に通る [R-4]。

**本アプリの上回り（楔における価値で）**:
- 即応改訂: 紙差替え 1〜数日 → 同期 60 秒以内（§13.1 SLO、仮説 H10）。
- 多品種混流での切替: 機種違いの差分を端末側で自動表示（J-B 隣接ジョブ）。
- 教育の組込: TWI-JI 三段で「急所の理由」を文書化し、属人化を漸進的に解消（J-C 副産物）。
- 「止まらない」「一覧性」は本アプリで継承する: オフライン完全動作（§6.4）／PDF 印刷出力（§13.4）。

**乗り換えコストと便益**:
紙からの初期投入は Excel テンプレ取込（Q28-B、§13.4）で 1 手順あたり数分。便益は段取り替え時間の標準偏差低減（H6）。便益の規模感を推定するための仮の式を以下に示す（**計算式は本企画書 v0.7.0 時点で全パラメータが未観測の仮説、検証は §15.3、観測値は v0.7.x 以降に置換**）。

```
便益（年間停止削減時間, 仮説）
  ≒ 段取り替え頻度（回／日／ライン）
    × 稼働日数（日／年）
    × 標準偏差縮減量（分／回）
```

例えば段取り替え頻度 4 回／日、稼働日数 250 日／年、標準偏差縮減量 1 分／回（標準時間 10 分・既存標準偏差 σ 約 3 分・縮減率 30% 想定）の場合、4 × 250 × 1 = 1,000 分／年／ライン ≒ 約 17 時間／年／ライン。式の各係数の出典が確定するまで、前版の「年間数百時間」表現は撤回する。

**閉じ**: 紙の 3 利点（止まらない・一覧性・監査証跡）を本アプリは継承し、即応改訂と多品種切替で勝つ。乗り換えコストは Excel テンプレ取込で吸収できる。

### 10.2 代替 B: 商用 e-SOP（Tulip ／ Operations1 ／ カミナシ ／ Teachme Biz ／ i-Reporter）

> **観測根拠**: 商用有償トライアル取得は本企画書 v0.7.0 時点で **不採用（§3.2、Q38-Q）**。本節の比較は §3.4 の運用に従い R-13（各事業者公式サイトの公開価格・機能ページ）と Internet Archive のスナップショットに基づく。観測時点・URL アーカイブは `docs/01_企画/observations/` に格納する。本人有償触察を要する論点（操作感・実運用での障害頻度等）は本節の射程外とし、§17 オープンクエスチョンに繰り込まない（Q38 方針による不採用が確定しているため）。

**ユーザー側の合理性**:
ベンダー導入支援込み（Q40-F）。法令対応の責任肩代わり（Q40-G）。SLA 付きサポート（Q40-I）。経営層への稟議が通りやすい（Q40-J）。

**本アプリの上回り**:
- 月額ライセンス費ゼロ（Apache-2.0、§14.1）。中小製造業のライン端末数（数十〜数百台、§13.1.1）で台数ライセンスが障壁となるところに効く。各製品の具体価格は R-13 の観測スナップショットに従い v0.7.x 以降の版で本表に転記する。
- データ主権: 自社オンプレ完結、SaaS 系商用が要求するクラウド送信が無い（§13.5）。
- DSL ・ スキーマの完全公開: ベンダーロックインが構造的に存在しない（Q40-N）。
- TWI-JI 三段の必須属性化: R-13 観測時点で各事業者の公開機能ページから「主要ステップ／急所／急所の理由」の **必須属性化** を確認できない（観測時点・確認方法を `docs/01_企画/observations/` に記録）。商用 e-SOP は機能追加として対応する余地があるが、設計の根に置く OSS は本観測時点で他に無い（§11.1）。

**敗ける点（認める）**:
導入支援・SLA 付きサポートは商用が圧倒的に強い。本アプリは Issue テンプレート、CONTRIBUTING.md、導入チェックリスト、SECURITY.md（§14.4）で代替する。代替であって同等ではない。

**乗り換えコストと便益**:
商用 e-SOP からの乗り換えはエクスポート互換に依存する。本アプリは Markdown ／ YAML エクスポート（Q28-L）と PDF 印刷（Q28-N）を備える。商用 e-SOP からの取込は、商用有償トライアル取得を §3.2 で **不採用** とした方針により、R-13 の公開機能ページから観測可能な範囲（事業者ごとの公開エクスポート仕様の有無、CSV ／ JSON ／ PDF 等）で扱う。多くの商用 SaaS は標準化された取込形式を持たず、本アプリ側からの一方向取込は本版時点で対応しない。

**閉じ**: 中小製造業のライン端末数規模で月額費用が壁になる現場では、本アプリが TCO で勝つ。商用 e-SOP の主要価値（導入支援・SLA）は本アプリの非スコープであり、これを必要とする現場は本アプリの楔対象ではない。

### 10.3 代替 C: 自前構築（Excel ／ kintone 改造 ／ SharePoint ／ Node-RED 流用）

**ユーザー側の合理性**:
既存資産の手内化。社内エンジニアで完結する。ライセンス費ゼロ。

**本アプリの上回り**:
- 共通基盤: 手順 DSL ／ 実績スキーマ ／ 同期エンジンが OSS として保守されている。属人化の温床にならない。
- ライン端末でのオフライン運用: Excel ／ kintone はオンライン前提が多く、Wi-Fi 不安定なライン現場で破綻する。
- 物理制約最適化: 手袋・片手・騒音・48dp タップ可能領域（§6.3）は専用設計を要する。Excel／SharePoint は破綻する。
- 監査証跡: ハッシュチェーン監査ログ（§13.9）。Excel の改訂履歴では監査要件を満たさない。

**乗り換えコストと便益**:
Excel からの取込は §13.4 のテンプレートで 1 手順あたり数分。便益は属人化リスクの解消と物理制約最適化。

**閉じ**: 自前構築の最大のリスクは保守不能化である。本アプリは OSS 共通基盤として保守される。

### 10.4 代替 D: 無策（紙＋ベテラン口伝のみ／IT 化を見送り）

**ユーザー側の合理性**:
初期投資ゼロ。ベテラン作業者がいる間は安定する。

**本アプリの上回り**:
- 段取り替え時間のばらつきが大きい現場で、SMED の "外段取り化" 改善（[R-1]）が回らない。本アプリは実績データで段取り替え時間の標準偏差を可視化（§13.6）し、改善発議を回す。
- ベテラン作業者の暗黙知が「急所の理由」として形式知化される（TWI-JI 三段、§8.4.1）。属人化リスクが漸減する。

**乗り換えコストと便益**:
本アプリ導入の最低コストは Docker 1 台＋タブレット 1 台。便益は段取り替え時間の改善ループの開始。

**閉じ**: 無策はベテラン依存である。ベテラン離脱時に正本が消える。本アプリは「急所の理由」を必須属性として残し、世代交代に耐える。

## 11. Moat（堀）の自己診断（CLAUDE.md §3.3）

### 11.1 該当判定（5 類型）

| 類型 | 判定 | 根拠 |
|---|---|---|
| ネットワーク効果 | 限定該当 | 単一メンテナ（Q15-A）の方針で初期コミュニティ形成は遅い。Q41-S 手順テンプレ寄稿は採用していない |
| データ・フライホイール | 非該当 | 端末側 LLM 推論を載せない（§12 NS-10）ため、利用ログから推論精度向上のループは形成されない |
| スイッチングコスト | 強く該当 | 標準作業3票互換（Q28-H）／TWI-JI 一級市民化（Q28-I）／ハッシュチェーン監査ログ（§13.9）。一度入ると現場用語と監査証跡で固着する |
| 規模の経済 | 該当 | Apache-2.0（§14.1、Q9-A）＋Docker／APK／MSIX 直接配布で限界費用ゼロ |
| 認知的優位 | 強く該当 | TWI-JI 三段の必須属性化を OSS で先取りする設計選択（§8.4.1）。商用 e-SOP は機能追加で対応するが、必須属性として一級市民化する設計は確認できない |

### 11.2 Moat の核（2 点）

> **(a) TWI-JI 三段の必須属性化（認知的優位）** と **(b) ISO 9001 第7.5 項の文書管理（スイッチングコスト）** を Moat の核とする。

- (a) の根拠: TWI-JI は厚労省 [R-2] の現役教材であり、現場用語として安定する。商用 e-SOP は項目数の増加で対応するが、本アプリは「必須」として 3 属性が空のままでは保存できない設計を採る（§8.4.1）。
- (b) の根拠: ISO 9001 第7.5 項 [R-4] は文書化情報の管理を要求する。紙運用は物理証跡で対応するが、デジタル化時には改訂履歴・監査ログ・出力機能が要求される。本アプリは §13.8／§13.9／§13.4 で要件を満たす。

両者は機能ではなく設計の正しさに拠る。商用 e-SOP が同等の設計を後付けするには既存機能体系を壊す必要があり、参入コストが構造的に高い。

### 11.3 Moat の脆弱性（自認）

- 単一メンテナ方針はネットワーク効果の形成を遅らせる（§16 シナリオ 1）。
- 楔範囲外の現場（J-D 設備保全・ジョブショップ等）では Moat が効かない。これは §15.1 H9（楔データモデルが隣接形態を吸収できる）の限界として明示する。

### 13.1 性能・容量・SLO

> SLO 表は **(a) 観測値・規格根拠ありの容量見積** と **(b) 設計目標（仮説、未検証）** を分けて記述する。前版（0.6.0）の同期遅延 p95 ／ 月間稼働率は単一メンテナ＋SLA なし運用で実装可能性が無いため撤回または性質を書き直した。

§8.7 のとおり。SLO は §13.1.2 の三分（要件／仕様／目標値・未検証／方針）で扱う。Q32-M（個人開発のため SLO を設けない）は採用しないが、月間稼働率の数値約束は §13.1.2 のとおり **行わない**（単一メンテナ前提との実装可能性整合）。

### 13.3 互換性・移行

- 手順 DSL は semver で版管理（Q33-P）。

- ゴールデンマスター・スナップショットテストで手順データの後方互換を担保（Q33-C）。

## 14. OSS 戦略（CLAUDE.md §4）

### 14.1 ライセンス選定

採用: **Apache License 2.0**（[R-8]）を本企画書 v0.7.0 で確定する。前版で MIT との二択を ADR-0001 に流していたが、製造業ドメインで OSS を運用する以上、特許権の取扱いを明示する必要があるため Apache-2.0 を採用する（理由は次表）。ADR-0001 は本決定の機械的記録のみを担う。

| 候補 | 採否 | 理由 |
|---|---|---|
| **Apache-2.0** | **採用** | 明示的な特許権付与（§3）と特許訴訟への報復条項（§3 後段）を備え、製造業の既存特許との衝突リスクを構造的に下げる。NOTICE ファイル要件は単一メンテナでも保守可能。著作権表示要件は MIT より厳格だが、企業導入時の法務確認を簡略化する効果がある |
| MIT | 却下 | 特許権について沈黙する。製造業向け OSS では特許トロール起因の差別圧力リスクを残すため、本企画書では却下 |
| MPL-2.0 ／ LGPL-3.0 | 却下 | 改変還元の縛りが商用組込みを遠ざける |
| GPL-3.0 | 却下 | 派生 OSS 化強制が、製造業の既存資産（クローズドな社内ツール）への組込みを阻害する |
| AGPL-3.0 | 公開時点では却下、再評価対象として保持 | SaaS 提供を行わない（NS-12）ため主たる動機が成立しない。ただし §16.1 シナリオ 1 (S) ／ §16.4 シナリオ 4 の回避策として「フォーク SaaS による Moat 侵食」が観測された場合の再評価対象として保持する |
| デュアルライセンス | 却下 | 単一メンテナ（Q15-A）の運用負荷に対し収益化余地は小さい |

依存ライセンス監査: AGPL の依存（DB／検索エンジン／LLM 関連）を採用しない（Q39-Q）。Apache-2.0 と互換性のあるライセンス（Apache-2.0 ／ MIT ／ BSD-2/3 ／ ISC ／ MPL-2.0 等）に依存範囲を限定する。

## 16. Pre-mortem（CLAUDE.md §1.5、6 シナリオ）

> 「2 年後（=2028-05）、本プロジェクトは失敗した。原因は何か」を 6 本書ききる。図 05（`./img/05_Premortem主原因マップ_v1.png`）に主原因マップを示す。本版で **シナリオ 0（一次情報なしで楔を打ったこと自体が間違いだった）** を追加し、シナリオ 2／3／5 をその発現形として位置づけ直した。

### 16.0 シナリオ 0: 一次情報なしで楔を打ったこと自体が間違いだった（最有力・上位原因）

- **概要**: §3.2 のとおり著者は楔現場の現場経験を持たず、リアル人間ヒアリング・現場観察・商用有償トライアル取得をいずれも不採用とした（§3.4.1）。§3.4 の公開資料調査と公開規格（R-1〜R-15）で代替したが、紙の "止まらなさ"／一覧性／監査証跡（§4.4）と TWI-JI 三段の必須属性化（§11.2）が **本当に楔現場で勝てるのか** を確証する手段が無い。OSS 公開後 1 年経っても核心仮説（H1／H2／H6／H8／H10／H12／H13）の暫定支持データが得られず、楔の選定そのものが机上だったと判明する。
- **本シナリオが上位原因である根拠**: シナリオ 2（UX 不一致）／シナリオ 3（H6 崩壊）／シナリオ 5（基幹連動の多様性）はいずれも、本シナリオ 0 が **観測手段を欠いたまま設計を進めたこと** の発現形である。Pre-mortem の主原因マップ（図 05）でも、シナリオ 2／3／5 の上位ノードに本シナリオを置く。
- **早期検知**: K1〜K8 のうち **2 件以上が公開後 6 ヶ月時点で同時発火**。または §3.4 の四半期観測で R-10〜R-14 の前提（中小製造業の MES 普及・商用 e-SOP 価格・OSS 競合分布）が本企画書の楔選定根拠と中央値で 2 標準偏差以上乖離する。
- **回避**: §3.4 公開資料調査の四半期実施を計画通り維持し、楔の選定根拠が崩れていないか毎回確認する／§17.1 OQ-1 ／ OQ-2 ／ OQ-3 の解消観測（公開後試行）を待つ／核心仮説（§15.1）と Kill Criteria（§15.2）の対応表を毎版見直し、観測値が出揃うまで設計の不可逆な選択（DSL の破壊的変更等）を避ける。
- **撤退**: 楔を J-A → J-B（多品種混流の機種違い切替）に切替、または OSS としての保守縮退（§16.1 (R) と一致）。本シナリオが発火した場合、シナリオ 2／3／5 の個別回避策よりも先にこの上位回避策を実行する。

### 16.1 シナリオ 1: メンテナ離脱（主原因 A）

- **概要**: 単一メンテナ（Q15-A）の興味喪失または私生活変化により、コミット頻度が低下し、外部 Issue ／ PR への応答が滞る。
- **早期検知**: K5（月次コミット数 ・ 反応速度 50% 低下）／K8（外部 PR ／ Issue 数の停滞）。
- **回避**: (O) 楔への撤退で機能を絞る／(P) 短サイクル化／(Q) 共同メンテナ招聘（§14.3 の CONTRIBUTING.md ／ Issue テンプレート ／ Good First Issue から漸進選抜）／(R) 撤退（ライセンスは Apache-2.0 で fork 自由）／(S) ライセンス変更（AGPL 化／財団寄贈、§14.1 で再評価対象として保持）。
- **本人選択**: Q15 上段で「メンテナ自身の興味喪失」を主リスクとして本人選択済み。

### 16.2 シナリオ 2: 楔現場の UX 不一致（主原因 B、シナリオ 0 の発現形）

- **概要**: 試行導入の楔現場で、片手保持＋手袋＋騒音下の UI（§6.3）が実用に至らず、紙運用に回帰する。仮説 H2 が崩れる。
- **早期検知**: K3（起動回数 ／ 作業者あたりの減少）。
- **回避**: §6.3 数値の見直し ／ 端末選定下限の引き上げ ／ §8.6 ステップ確証方式の縮退 ／ 撤退（H2 が崩れた時点で楔の前提が消滅）。
- **シナリオ 0 の発現形である理由**: §6.3 数値（48dp／18pt／7:1）の楔現場での妥当性は規格 [R-5][R-6][R-7] からの推定にとどまり、現場観察（§3.4.1 で不採用）で確証できない。これは **観測手段を欠いたまま物理制約を確定した結果** であり、根は §16.0 シナリオ 0 にある。前版（v0.6.0）は本シナリオを「最有力」と単独判定したが、上位原因を取り違えていたため本版で修正した。

### 16.3 シナリオ 3: 段取り替え時間ばらつきの主因が楔の介入対象外だった（主原因 G、楔崩壊、シナリオ 0 の発現形）

- **概要**: 仮説 H6（ばらつきの主因が段取り・手順記述）が崩れ、実際は工具不足や設備異常が主因だった。本アプリの介入が段取り替え時間の標準偏差を改善しない。
- **早期検知**: K2（標準偏差が紙比で改善しない）。
- **回避**: 楔の Job を J-A から J-B（多品種混流の機種違い切替）に切り替える ／ 改善ループ機能（§13.6）を主機能に格上げし、ばらつき可視化で価値を立て直す ／ 撤退（楔の主便益が消失）。
- **シナリオ 0 の発現形である理由**: H6 は §3.2 の一次情報欠落のもとで「公開規格と古典文献から類推した」核心仮説であり、§3.4 R-10〜R-12 で部分検証されるまで暫定的に保持される。観測手段を欠いた仮説選定の典型例。

### 16.4 シナリオ 4: 商用 SaaS の無料プラン拡充による Moat 侵食（主原因 D／E）

- **概要**: 商用 e-SOP が無料プランを大幅拡充し、本アプリの "費用ゼロ" 価値が相対的に薄れる。または OSS フォーク／クローズド派生に主導権を奪われる。
- **早期検知**: K6（四半期競合モニタリング）。
- **回避**: Moat の核を機能ではなく設計の正しさ（TWI-JI 必須属性化／ISO 9001 正面実装）に保つ ／ ライセンス変更（AGPL 化）でフォーク SaaS を抑止 ／ 撤退。

### 16.5 シナリオ 5: 楔現場の基幹連動が想定より多様だった（主原因 C、シナリオ 0 の発現形）

- **概要**: 仮説 H8（REST／GraphQL を持つライン生産現場が十分に存在）が崩れ、現場ごとの独自プロトコル（OPC UA／Modbus／独自バイナリ）対応が単一メンテナで保守不能になる。
- **早期検知**: K4（連動アダプタ追加要件 Issue の増加）。
- **回避**: アダプタ仕様の凍結 ／ 導入先実装への完全委譲（本プロジェクトはコア仕様のみ）／ 撤退。
- **シナリオ 0 の発現形である理由**: H8 は MES 普及率の公開資料調査（§3.4 R-10〜R-12）が間に合うまで暫定保持。中小製造業の MES 保有率は本企画書 v0.7.0 時点で著者の手元に観測値が無く、楔の射程（§3.1 末尾）の正当性は §3.4 の四半期観測に依存する。

### 16.6 6 本に共通する最終回避策

楔（J-A 段取り替え×単位作業×ライン生産）への即時撤退（Q44-T）。これは楔を打つ判断（§6.1）と一体である。すべてのシナリオで Moat の核（§11.2）と楔（§6.1）に資源を集中することで失敗確率を下げる。

ただし **シナリオ 0 が発火した場合**（K1〜K8 のうち 2 件以上が同時発火、または §3.4 観測値の楔前提との 2σ 乖離）、シナリオ 1〜5 の個別回避策よりも先にシナリオ 0 の上位回避策（§16.0 末尾）を実行する。具体的には楔の選定根拠そのものを §3.4 の最新観測で再評価し、楔の切替（J-A → J-B 等）または OSS としての保守縮退を即決する。
