# 企画書を至高に仕上げるための作法

本ディレクトリの成果物（特に `企画書.md`）は、製造業向け作業ナビゲーションアプリの**最初の技術成果物**である。
意思決定者がそのまま投資判断に使え、現場経験者が読んで「これは現場を分かっている」と頷き、技術者が読んで「これは実装できる」と判定できる水準で書く。
本ガイドはチェックリストではなく**思考器具**として使う。各節の **強制テスト** と **キャリブレーション例** を全件通過しない節は、書かない方がマシである。

---

## 0. 大原則（崩したら他は読まなくていい）

1. **中心問題は「作業」の形式モデル化である**（§1）。ここで負けたら他章は飾り。
2. **数字に翻訳できない主張は削除する**（§5 強制テスト）。
3. **書かない規律のほうが書く規律より重い**（§6）。
4. **文書自身がここに書いた基準を満たすこと**。満たさない CLAUDE.md は無効。

## 1. 中心問題 — 「作業」を形式モデル化せよ

メモに「作業という概念が難しい」と記された一文が、本企画の知的核心である。
ここを曖昧にしたまま機能・UI・ビジネスモデルを論じる企画書は、根のない木である。

### 1.1 なぜ難しいか（言語化）

- 「作業」は名詞ではなく **観測される行為の集合の解釈**。同じ動作でも、誰がいつ何のために行ったかで作業境界が変わる。
- 製品 / ライン / 工程 / 作業 / ステップ / 動作 の階層と境界が、工場・職場・人によって違う。
- 順序制約は単純な DAG では足りない：**やり直し・選択分岐・並行・例外復帰・暗黙ジャンプ** を含む。
- ベテランの「コツ」「カン」は標準作業書から落ちる。これを扱える／扱わないを意識的に決める。

### 1.2 最低限モデル化すべき要素

| 要素 | 内容 |
|---|---|
| Atomic Step | これ以上分割しない単位。粒度判定基準を明文化（例：1動作=単一目的・5秒以内） |
| Composition | Step → Task → Process → Line の階層と所属関係 |
| Precedence | 直列／並列／選択／繰返し／やり直し／例外復帰 を表現できる順序制約モデル |
| Exit Criteria | 完了判定。観測可能な物理量・センサ値・操作で定義（自己申告のみは不可） |
| Exception Path | 異常時の分岐先と権限（誰が止め誰が判断するか） |
| Actor Role | 作業者／班長／保全／品証 など役割と権限境界 |
| Subject | 製品ID／ロット／工番／治具 のどれに紐づくか |
| Time | 標準時間・タクト適合度・実績記録の粒度 |
| Knowledge Tier | 明示知 / 暗黙知 / 教えるべき / 教えなくてよい の4分類 |

### 1.3 強制テスト（§1 単独で）

- [ ] 上記モデルを **BNF または型定義で 1 ページに書き切れる**（疑似コード可）
- [ ] 既存の **標準作業書 1 枚** を、本モデルで構造化データ（JSON 等）に変換した実例がある
- [ ] **同一工程をベテラン 2 名で観測**したとき、両者の作業差分が本モデルで表現でき、どこが標準でどこが個人差かを区別できる
- [ ] 順序制約に「やり直し」「選択分岐」「例外復帰」が**全て**書ける
- [ ] 「暗黙知」を捨てるのか、形式化するのか、保留するのか、**意識的に選択して明記**している

通過しなければ、企画書全体が未着手とみなす。

## 2. 製造業の本物の語彙で書く

現場の意思決定者が日常使う語彙との接続が薄い文書は、それだけで二流に落ちる。
以下を**構造的に**埋め込む（飾りで散らさない）。

### 2.1 必須語彙とその使い所

| 語彙 | 文書内での使い所 |
|---|---|
| **三現主義**（現場・現物・現実） | 主要主張の脚注に「どの現場で・どの現物の・いつの観察に基づくか」を明記 |
| **QCD + SM**（品質・コスト・納期＋安全・モラール） | 効果章は必ずこの 5 軸で分類。1 軸に偏った企画は要再検討 |
| **タクトタイム／サイクルタイム** | UI 操作所要時間はタクトに対する比率（%）で論じる |
| **標準作業／標準作業組合せ票／作業要領書** | 何を電子化し何を残すかを名指しで言い切る |
| **ポカヨケ** | アラート・確認 UI の設計思想 |
| **多能工化／教育期間** | 新人立上げ短縮効果は **週**単位で示す |
| **SMED（段取り替え）** | 機種切替時の手順差替え・UI リセット |
| **MES / ERP / PLC / Andon** | 基幹連携の境界線をこの語彙で書く |
| **トレーサビリティ／4M（人・機械・材料・方法）** | 記録粒度の議論で必ず登場させる |

### 2.2 キャリブレーション（悪い文 → 良い文）

| 悪い文 | 良い文 |
|---|---|
| 現場のオペレーターが直感的に使える | 作業手袋（厚さ 1.2mm）装着・照度 300lx・粉塵環境下で、タクト 30 秒中の UI 操作所要時間を 3 秒以内に抑える |
| リアルタイムに進捗を可視化 | 班長は 1 ライン 12 工程の滞留を Andon 発報から 5 秒以内に検知（既存 Andon の上位互換） |
| DX を推進 | 作業要領書の改訂頻度を 1 回 / 日 → 10 回 / 日に上げ、設計変更からライン反映までを 3 日 → 2 時間に短縮 |
| 教育コストを削減 | 新人作業者の独り立ち期間を 8 週 → 5 週に短縮（評価指標：独立作業 50 サイクルで不良率 < 0.5%） |
| 既存システムとシームレスに連携 | MES から工番・順序情報を 5 秒間隔で取得、PLC からは Andon 信号のみを受け、書き戻しはしない |

**ルール**：右列のように書けないなら、その主張はまだ自分の中で詰まっていない。書くな。

## 3. 構成 — 14 章フラットを捨て、3 Tier に階層化

時間が足りないとき何を捨てるかが書き手の格を決める。
**Tier 1 のみで企画書は成立する**。Tier 1 が曖昧なまま Tier 2/3 を盛るのは時間の浪費。

### Tier 1（これがなければ企画書ではない）

| ID | 章 | 内容 |
|---|---|---|
| T1-A | 「作業」の形式定義 | §1 を満たす。文書冒頭に置く |
| T1-B | バリュープロポジション 140 字 | 誰の／どの作業の／何を／どれだけ良くするか。Twitter 1 ツイートに収まること |
| T1-C | 主要ユースケース | 作業者の 1 日連続 6 時間タイムライン＋班長／生産技術／情シスの代表シナリオ |
| T1-D | 効果試算（QCD+SM 5 軸 ×3 シナリオ） | 悲観／中央／楽観。前提・式・出典を脚注 |

### Tier 2（投資判断に必要）

| ID | 章 |
|---|---|
| T2-A | 競合・代替手段比較（紙手順書／既存 MES／内製ツール） |
| T2-B | システム概要（端末構成／基幹連携境界／オフライン耐性／同期戦略） |
| T2-C | コスト・収益モデル |
| T2-D | リスクと対策（**撤退基準を含む**。Pre-Mortem §4.4 由来） |
| T2-E | スケジュールと Go/No-Go ゲート（PoC → パイロット → 量産） |

### Tier 3（あれば質が上がる）

| ID | 章 |
|---|---|
| T3-A | 体制と役割 |
| T3-B | 法規・規格対応（ISO 9001 / IATF 16949 / トレーサビリティ要件 / 個人情報） |
| T3-C | 背景・一次情報サマリー（ヒアリング件数・場所・日付つき） |
| T3-D | 未決事項（誰が・いつまでに・何を決めるか） |

## 4. 思考器具（外部フレーム）— 執筆中に必ず通す

最低 1 回は、自分の企画を以下のフレームに晒すこと。通過しないなら企画が弱い。

### 4.1 Amazon Working Backwards / PR-FAQ
- 製品リリース時の **プレスリリースと FAQ を先に書く**
- 「お客様にとって魔法のような〇〇を発表しました」が書けないなら、その企画には魔法がない
- 本企画では「お客様」が複層：**工場長向け** と **作業者向け** の 2 本書く

### 4.2 Jobs To Be Done（Christensen）
- 作業者が「雇いたい仕事」（=達成したい状況）は何か
- **紙の手順書を「解雇」し、本アプリを「雇う」理由**を 1 段落で書けるか

### 4.3 Wardley Map
- 構成要素を Genesis / Custom / Product / Commodity に位置付ける
- 「作業ナビゲーション」というケーパビリティが市場のどこにあり、3〜5 年後どちらへ動くか

### 4.4 Pre-Mortem（事前検死）
- 「3 年後、本企画は失敗した。原因は？」を **10 件以上**書き出す
- これらへの対策が §T2-D リスクの本体

### 4.5 Toyota 5 Whys
- 主要 KPI 1 つに対し「なぜそれが効くのか」を 5 段掘る
- 5 段掘れない KPI は本物ではない

## 5. 強制テスト集 — 数値で殴る

執筆完了判定はこれで行う。**全件通過しないなら未完成**。

| ID | テスト | 合格基準 |
|----|--------|----------|
| F-1 | 140 字テスト | バリュープロポジションが 140 字以内に収まる |
| F-2 | 6 時間タイムラインテスト | 作業者の 1 日連続 6 時間分のタイムラインに、本アプリの介入を**秒単位**で書ける |
| F-3 | 標準作業書変換テスト | 既存の標準作業書 1 枚を §1 のモデルで JSON 化した実例がある |
| F-4 | タクト適合テスト | UI 操作 1 アクションの所要時間がタクト（例 30 秒）の 10% 以内 |
| F-5 | 撤退基準テスト | 「パイロットで KPI X が 70% 未達なら撤退」のように、条件・閾値・期日が書ける |
| F-6 | 反論先回りテスト | 想定読者の反論 TOP 3 に、本文で**先に**答えている |
| F-7 | 三現主義テスト | 主要主張 1 件以上に対し、現場・現物・現実の出典（拠点名・日付・観察者）が紐づく |
| F-8 | 5 軸テスト | 効果が QCD+SM のどの軸に効くか、KPI ごとに明示 |
| F-9 | Pre-Mortem 10 件テスト | 失敗シナリオ 10 件以上＋各対策が記載 |
| F-10 | 200 字要約テスト | 知的核心を 200 字 1 段落で書ける（読み手 = 工場長） |
| F-11 | 図独立性テスト | 各図がキャプションと本文参照のみで意味が伝わる（本文を読まないと意味不明な図は不可） |
| F-12 | バズワードゼロテスト | DX／リアルタイム／シームレス／AI 駆動／直感的 UI／クラウドネイティブが 0 回（数値に翻訳できないため） |

## 6. 書かない規律（Anti-Patterns）

書いた瞬間に企画書のクラスが落ちるもの。

- **機能羅列で価値を語る**（「①…②…③…」）
- **バズワード**（F-12）
- **形容詞で逃げる**（使いやすい／分かりやすい／効率的）
- **競合比較を「他社は高価で機能過剰」で済ませる**（典型的な自社有利バイアス）
- **リスク章を「適切に対応」で塗る**
- **効果試算を期待値のみ**で出す（3 シナリオなしは不可）
- **「現場の声を聞いた」と書きながらヒアリング件数・拠点・日付なし**
- **アスキー図**（`docs/CLAUDE.md` 違反）
- **「とりあえず PoC」**（PoC で何が証明できれば次に進めるかが書けないなら PoC ではない）
- **「将来的には〜」**（タイムライン外の機能は捨てる）
- **「他社もやっている」を理由に使う**（それは現象であって理由ではない）
- **書き手が自社の組織図に都合よく曲げた章立て**（情シスが章を持っていて作業者が持っていない、等）

## 7. 文書運用ルール

- **本体**：`企画書.md` 単一ファイル。冒頭に `v0.x / YYYY-MM-DD` を記載
- **図**：drawio で作成し PNG 出力（保管：`docs/01_企画/img/`）。アスキー図は禁止（`docs/CLAUDE.md`）。図には番号・タイトル・本文参照
- **数字の出典**：本文に脚注番号、末尾に「参考文献・出典」節（拠点・日付・観察者・式）
- **用語**：揺れたら `用語集.md` を新設、初出時リンク。本文中の同義異語は禁止
- **ヒアリング記録**：`notes/YYYY-MM-DD_対象.md`。本文に未整理メモを混ぜない
- **改訂履歴**：冒頭または末尾に表で、誰が・いつ・何を変えたか

## 8. 執筆プロセス（順序を守る）

1. **読者と決定の特定**：誰が・この文書で・何を決めるかを 1 行（10 秒で書ける、書けないなら執筆開始しない）
2. **PR-FAQ 先行執筆**（§4.1）— 工場長向け・作業者向けの 2 本
3. **「作業」の形式定義の起草**（§1）— **ここに最大時間を投じる**
4. **6 時間タイムライン作成**（F-2）
5. **Tier 1 残り（T1-B / T1-C / T1-D）**
6. **Tier 2 全章**
7. **Pre-Mortem セッション**（§4.4）→ §T2-D に反映
8. **強制テスト全件通過確認**（§5）— 1 件でも × なら該当章へ戻る
9. **エグゼクティブサマリー執筆**（最後）
10. **第三者査読**：現場経験者 1 名 + 技術者 1 名 + 経営層 1 名
