# 一流の企画書作成ガイドライン

本ディレクトリの成果物は、本プロジェクトにおける **意思決定の最上位文書** である。
ここで定めた方針が、設計・実装・OSSコミュニティ運営すべての上流制約となる。
凡庸を許容しない。テンプレを埋めただけの文書は本ガイドラインへの違反とみなす。

上位方針との関係:
- ルート `CLAUDE.md`: 「技術的に至高を目指す」
- `docs/CLAUDE.md`: アスキー作図禁止 / drawio→PNG / MVP・Phase は定めない

---

## 0. 一流の企画書の定義（このガイドラインの根本基準）

凡庸な企画書は **形式（章立て・体裁）** を満たす。一流の企画書は **論理の強度・反証耐性・意思決定速度** を満たす。
具体的には、次の3つを同時に満たすものだけを一流と呼ぶ。

1. **不可避性 (Inevitability)** — 読み終えた人間が「これは作られて当然」と感じる。代替シナリオが論理的に閉じる。
2. **反証耐性 (Falsifiability)** — どの仮説が崩れたら本企画が破綻するかが明示され、その検証手段が記述されている。
3. **行動可能性 (Actionability)** — 読み手が「次に何を確認・実装・廃案すべきか」を明確に取り出せる。

これは「分かりやすい」ことよりも厳しい要求である。やさしく書くだけの文書は一流ではない。

## 1. 思考フレームワーク（採用必須）

以下は飾りではなく**強制**。該当章を書くとき、これらの道具を使って思考した痕跡が文書に残ること。

### 1.1 Working Backwards（PR-FAQ）
企画書本体を書く前に、**完成して世に出た瞬間のプレスリリース1ページ** と **想定FAQ** を先に書く。
ここで「ユーザーが何を喜ぶか」「何を恐れるか」を一文で言えなければ、企画は未成熟。

### 1.2 Jobs-to-be-Done (JTBD)
ペルソナよりも JTBD を優先する。
書式: `When _____ , I want to _____ , so I can _____ .`
製造業向けでは「現場のオペレータが」「段取り替え時に」「迷わず手順を実行したい」「ライン停止を最小化するため」のような形で、**状況・動機・期待成果** をセットで記述する。

### 1.3 Theory of Change（変化の論理）
`Inputs → Activities → Outputs → Outcomes → Impact` の鎖で書く。
特に **Outputs と Outcomes を混同しない**（手順表示機能の存在=Output、段取り時間の短縮=Outcome、ライン稼働率の改善=Impact）。

### 1.4 Wardley Mapping（成熟度地図）
コンポーネントを `Genesis / Custom / Product / Commodity` に置き、**自分が独自に作るべき部分と、既製品に乗るべき部分** を可視化する。
本プロジェクトが OSS として価値を持つのは、Custom 領域に正しく賭けたときに限る。

### 1.5 Pre-mortem（事前解剖）
「2年後、本プロジェクトは失敗した。原因は何か？」を **書ききる**。
最低5シナリオ。各シナリオに対し回避策・早期検知シグナルを併記する。
これを書けないなら、リスク章は未完成。

### 1.6 Pyramid Principle（バーバラ・ミント）
各章は「結論先出し」「MECE」「So What? / Why So?」のテストを通す。
箇条書きを並べて満足しない。**主張→根拠→根拠→根拠** の縦の論理が通っていること。

## 2. 製造業ドメインの言語と方法論（必須語彙）

製造業の企画書で IE（Industrial Engineering）の語彙を持たないものは、それだけで一流から外れる。

### 2.1 「作業」概念の階層整理（必須）
本プロジェクト最大の難所。曖昧なまま進めることを禁ずる。最低限、以下の階層で整理する。

| 階層 | 例 | 時間スケール | 本アプリの対象か |
|---|---|---|---|
| 工程 (Process) | 塗装工程 | 時間〜日 | 文脈として保持 |
| 単位作業 (Unit Work) | 右ドア塗装 | 数分〜数十分 | **主たるナビ単位** |
| 要素作業 (Element Work) | マスキング／吹付／乾燥 | 数十秒〜数分 | ナビの内側の節 |
| 動作 (Motion / Therblig) | 手を伸ばす／掴む | 秒以下 | 取扱対象外 |

企画書では **アプリがどの階層をナビゲートするか** を明示し、それより上下の階層をどう扱う／扱わないかを書く。

### 2.2 採用すべき分析フレーム
- **ECRS** (Eliminate / Combine / Rearrange / Simplify): 既存作業の改善余地を語る根拠
- **5W1H × 三現主義 (現場・現物・現実)**: 課題を具体に降ろす
- **TPSの7つのムダ** (作りすぎ・手待ち・運搬・加工・在庫・動作・不良): どのムダを削るかを名指し
- **TWI-JI（仕事の教え方）の作業分解**: 主要ステップ・急所・急所の理由 — 本アプリの手順表現と直接対応
- **標準作業3票** (工程別能力表 / 標準作業組合せ票 / 標準作業票): 既存運用の正本がこれである現場が多い。互換性を論じる
- **動作経済の原則**: タブレットUI設計の物理的制約（手袋・片手・視線移動）の根拠
- **TOC（制約理論）**: ライン全体最適の中で本アプリが効くポイントの特定

### 2.3 現場物理制約の明示
タブレット運用は **机上のUX** ではない。次を必ず数値で記述する。
- 照度（暗所・直射日光下）／騒音／粉塵／油・水耐性／手袋装着時のタッチ感度
- 片手操作の可否／視線距離／作業姿勢
- 通信環境（Wi-Fi 不安定／オフライン要否）／電源（連続駆動時間）

「現場を考慮する」のような **抽象的言及だけでは不可**。

## 3. 競合と差別化（"完璧に上回る" の作法）

ルート方針で「市場にある競合を調査したうえで完璧に上回る価値を提供する」と宣言している以上、競合章は本企画書の核である。

### 3.1 必須の3レイヤー調査
1. **商用パッケージ** (MES / 作業手順書システム / Andon / e-SOP 系)
2. **OSS / 自前構築** (Node-RED 流用・社内 Excel/Access・kintone 改造・etc.)
3. **紙・口伝・暗黙知** (現場の実態。多くの場合これが最強の競合)

### 3.2 比較は表で終わらせない
表は導入に過ぎない。各競合について次を書く。
- **なぜそれが今なお選ばれているのか**（ユーザー側の合理性）
- **その合理性を本アプリは何によって上回るのか**（機能ではなく価値で）
- **乗り換えコストとそれを超える便益**（スイッチングコストの定量化）

### 3.3 Moat（堀）の自己診断
本プロジェクトの優位は、次のどれに該当するかを名指しする。該当なしなら計画として弱い。
- ネットワーク効果（コントリビューター数 → 手順テンプレ蓄積 → ユーザー）
- データ・フライホイール（利用ログ → ナビ精度向上）
- スイッチングコスト（標準作業3票との互換、現場用語との親和）
- 規模の経済（OSS による限界費用ゼロ配布）
- 認知的優位（手順記述 DSL / 概念モデルの先取り）

## 4. OSS 戦略（"公開予定" を本気で扱う）

OSS化は「ソースを公開する」ではない。**コミュニティを設計する**ことである。

企画書では以下を明示する。
- **ライセンス選定**: 派生物の挙動・特許条項・GPL系との互換まで吟味（候補と却下理由を並記）
- **ガバナンスモデル**: BDFL / Meritocracy / Foundation 型のいずれを志向するか
- **コントリビューション設計**: Issue/PR テンプレ・CLA/DCO・新規貢献者の最初のコミットまでの所要時間目標
- **配布チャネル**: GitHub Releases / Microsoft Store / Google Play / F-Droid / 社内オフライン配布の可否
- **持続可能性**: メンテナ離脱時の引継ぎ、バス係数、長期サポート方針
- **国際化**: 製造業ドメインを日本語前提で設計するか、英語化する場合の用語対応表
- **依存ライセンス監査**: SBOM／SPDX を企画段階から要件に入れる

## 5. スコープ・反証条件・撤退条件

KPI を並べるだけの企画書は半人前。**仮説が崩れたら何を止めるか** が書かれて初めて一流。

### 5.1 仮説の明示
本企画が依拠する **検証されていない仮説** を列挙する（最低5本）。
例: 「現場オペレータはタブレットを片手保持で操作できる」「単位作業の所要時間ばらつきは作業者要因より段取り要因が大きい」など。

### 5.2 各仮説に対する Kill Criteria
仮説ごとに「これが観測されたら本企画は方針転換または廃止」という閾値を書く。
KPI の達成目標と、**未達時の意思決定** をペアで記述する。

### 5.3 非スコープを積極的に書く
「やらないこと」を **やることと同じ分量** で書く。
やらない理由（時期尚早・領域違い・OSS思想と不整合・etc.）を併記する。

## 6. 文書としての品質基準

### 6.1 文体
- 一文一義。50字超の文は分割を検討。
- 主語省略の連鎖を禁ずる。製造業文脈では「誰が」が決定的に重要。
- 形容詞単独の主張を禁ずる（"高速な" → "p95 200ms以内")。
- 受動態・婉曲表現で意思決定を曖昧化しない（"〜と思われる" "〜が望ましい" は推敲対象）。

### 6.2 数字
- 単位 / 観測時点 / 観測対象 / N数 を必ず併記。
- 出典のない統計は「仮説」と明記する。
- 偽の精度を避ける（観測N=3で "12.4分" と書かない、"おおむね10〜15分(N=3, 2026-04)" と書く）。

### 6.3 用語
- 用語集（Glossary）を企画書末尾に置く。
- 「作業」「タスク」「工程」「手順」「ステップ」「ジョブ」を混用しない（§2.1 の階層に従う）。
- ドメイン用語は初出で1行定義。

### 6.4 図表（プロジェクト共通ルールの再掲＋強化）
- アスキーアートでの作図は **禁止**。
- drawio で作成し PNG 書き出し、`./img/` に格納、相対パス参照。
- drawio 原本（`.drawio`）も同梱（再編集可能性の保全）。
- ファイル名: `NN_<内容>_<版>.png`（例: `01_作業階層_v2.png`）。
- 図には番号・タイトル・凡例・前提条件のキャプションを付し、本文と独立して読めること。
- 「絵的に整っている」ではなく「論理が誤読されない」を優先する。

## 7. 推奨章構成（ただし形式に逃げないこと）

以下は **思考の補助線**であり、章を埋めれば一流になる保証はない。
不要章は「該当なし＋理由」を明記して省略する（沈黙の省略は禁止）。

1. エグゼクティブサマリ（5行以内・読み手の意思決定材料を凝縮）
2. PR-FAQ（§1.1）— 完成時のプレスリリース＋想定FAQ
3. 背景と一次情報（ヒアリング・観測ログへのリンク必須）
4. 課題定義（JTBD §1.2 で記述、root cause まで分解）
5. 「作業」概念の階層定義（§2.1）
6. ターゲット利用文脈（役割×現場物理制約 §2.3）
7. 提供価値と Theory of Change（§1.3）
8. ソリューション概要と境界（基幹システム連動の範囲含む）
9. Wardley Map と独自性の所在（§1.4）
10. 競合・既存手段の3レイヤー分析（§3）
11. Moat の自己診断（§3.3）
12. スコープ / 非スコープ（§5.3）
13. 非機能要件（定量・現場物理条件込み §2.3）
14. OSS 戦略（§4）
15. 仮説と Kill Criteria（§5.1, §5.2）
16. Pre-mortem（§1.5）
17. オープンクエスチョン（隠さない）
18. 用語集
19. 更新履歴（日付・変更点・変更理由）

> 本プロジェクトは個人開発のため **MVP / Phase / マイルストーン段階分けは書かない**。
> 代わりに「現在の到達点」「未着手の論点」「直近で潰す仮説」を率直に書く。

## 8. アンチパターン（一発で一流から外れる兆候）

書いた文書がこれらに該当した瞬間、書き直し対象。

- **DXワードの濫用**: "DX" "スマートファクトリー" "デジタルツイン" を定義せず使う
- **自動化シアター**: 価値を生まない自動化を機能列挙で誇る（"AI搭載" だけの記述）
- **三現主義の欠落**: 現場・現物・現実への言及がない、もしくは伝聞のみ
- **偽の精度**: 出典不明の "30%向上" "業界初" などの数字
- **網羅信仰**: 全機能列挙で読み手を疲弊させ、優先順位を語らない
- **ベンダー目線**: 「我々が提供する」が主語になり、ユーザーのジョブが見えない
- **分かりやすさ過剰**: 比喩と簡略化で論理の強度を犠牲にする
- **図のコピペ**: 既存の業界資料の図を出典なく流用、または飾り図を貼る
- **段階分けへの逃避**: 決められない判断を "Phase2 で検討" に押しつける（本プロジェクトでは禁止）

## 10. ファイル運用

- `企画書.md` — 本体。最新の意思決定を反映する単一の正。
- `ヒアリング.txt` — 一次情報。要約せず原文に近い形で保持。
- `メモ.txt` — 思考の作業メモ。本体に昇格すべき内容は速やかに移送。
- `img/` — drawio 由来 PNG ＋ `.drawio` 原本。
