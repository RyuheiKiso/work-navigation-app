# ポストモーテム 索引・命名規約

OPS-064 の実装。本ディレクトリはシステム障害・品質問題の事後分析記録（ポストモーテム）を格納する権威ディレクトリである。手順・テンプレートの権威は `../09_運用・保守/障害対応/11_ポストモーテム手順とテンプレート.md` とし、本ファイルは個別ポストモーテムファイルの索引・命名規約・配置方針を確定する。

---

## §1 Just Culture 宣言

**本ディレクトリのポストモーテムは個人非難を禁止し、学習文化の促進のみを目的とする。**

ポストモーテムは「誰が悪かったか」を問うものではなく、「システム・プロセスがなぜ失敗しやすい状態にあったか」を分析するものである。記録された内容は改善のための知識として扱い、懲戒・評価の根拠として使用しない。

本宣言は全ポストモーテム記録に適用し、宣言に反する記述が個別ファイルに含まれる場合は当該箇所を訂正する（訂正は新ファイル追加による差し替えで行い、元ファイルを削除しない）。

```
docs/ポストモーテム/
```

| 種別 | 配置先 |

| 採番台帳（PM-YYYY-NNN）| `docs/09_運用・保守/付録/99_運用保守識別子採番台帳.md` §4 |

## §3 命名規約

### 3-1. ファイル名形式

```
YYYY-MM-DD_PX_<slug>.md
```

| フィールド | 内容 | 例 |
|---|---|---|
| `YYYY-MM-DD` | 障害発生日（復旧日ではなく発生日）| `2026-06-15` |
| `PX` | 優先度（P1〜P4）| `P1`・`P2`・`P3`・`P4` |
| `<slug>` | 障害概要のスラグ（英数小文字・ハイフン区切り・40 文字以内）| `db-connection-timeout` |

```
# 命名例
2026-06-15_P1_db-connection-timeout.md
2026-08-03_P2_handy-sync-delay-spike.md
2026-09-21_P3_slo-report-generation-error.md
```

### 3-2. PM-YYYY-NNN 採番規約

| 項目 | 内容 |
|---|---|
| **形式** | `PM-YYYY-NNN`（YYYY: 発生年・NNN: 年度内連番 001 開始）|
| **欠番** | 禁止 |
| **年度またぎ** | 2026 年のインシデントは `PM-2026-NNN`。2027 年は `PM-2027-001` から再開始 |
| **権威台帳** | `../09_運用・保守/付録/99_運用保守識別子採番台帳.md` §4 |
| **採番タイミング** | 個別ファイル作成と同時に台帳に登録する（未登録識別子の使用は不正）|

```
PM-2026-001  # 2026 年度 1 件目のポストモーテム
PM-2026-002  # 2026 年度 2 件目
PM-2027-001  # 2027 年度は 001 から再開始
```

**PM-YYYY-NNN と YYYY-MM-DD_PX_<slug>.md の対応** は本章 §6 の索引テーブルで管理する。

## §4 個別ポストモーテムファイルの構成

個別ポストモーテムファイルの作成手順・テンプレートは `../09_運用・保守/障害対応/11_ポストモーテム手順とテンプレート.md` が権威である。本節では最低限の構成要件を示す。

### 4-1. 必須セクション

| セクション | 内容 |
|---|---|
| `pm_id` | `PM-YYYY-NNN`（ヘッダー YAML）|
| `related_inc` | 対応する `INC-YYYY-NNN`（ヘッダー YAML）|
| `severity` | P1〜P4（ファイル名の PX と一致させる）|
| 事象の概要 | 5W1H で記述（50〜200 字）|
| タイムライン | 発生〜復旧までの時系列（15 分刻み以下）|
| 根本原因分析 | 5-Why 分析または Fault Tree Analysis の結果 |
| 影響範囲 | ALCOA+ 影響・ユーザー影響・データ影響 |
| 改善処置 | 再発防止策の具体的アクション（PROB-NNN / REC-NNN と紐付け）|
| 学びの記録 | チームの学びとして記録する事項 |

### 4-2. ALCOA+ 記録要件

個別ポストモーテムファイルは以下の ALCOA+ 要件を満たす。

| ALCOA+ 原則 | 記録要件 |
|---|---|
| Attributable | タイムラインの全エントリに記録者を明示する |
| Contemporaneous | ポストモーテム開始日時を `pm_started_at` に記録する |
| Original | ファイルの初版コミットを原本とし、修正は §5 の訂正ルールに従う |
| Accurate | 事実と推測を明確に分け、推測部分には「（推定）」と付記する |
| Enduring | 本ディレクトリから削除しない（7 年保全・§5-2）|

## §5 ポストモーテム記録の不変性と訂正ルール

### 5-1. 訂正は新ファイル追加で行う

既存のポストモーテムファイルに重大な誤りを発見した場合、元ファイルを修正するのではなく、以下の方法で訂正する。

```bash
# 誤りのある元ファイル（削除しない）
docs/ポストモーテム/2026-06-15_P1_db-connection-timeout.md

# 訂正ファイル（元ファイルの参照を含む新ファイル）
docs/ポストモーテム/2026-06-15_P1_db-connection-timeout_v2.md
```

訂正ファイルの冒頭に以下を記載する。

```markdown
> **訂正通知**: 本ファイルは `2026-06-15_P1_db-connection-timeout.md` の訂正版です。
> 訂正理由: <誤りの内容と訂正内容>
> 訂正日時: YYYY-MM-DD HH:MM
> 訂正者: system_admin
```

軽微な誤字修正（手順・事実・数値に影響しない）は元ファイルの直接修正を許可し、git commit メッセージに修正理由を記録する。

### 5-2. 削除禁止・7 年保全

**本ディレクトリの全ポストモーテムファイルの削除を禁止する。**

ALCOA+ Enduring 原則および 21 CFR Part 11 §11.10(e) に基づき、ポストモーテム記録は 7 年間保全する義務を負う。誤って削除した場合は git reflog を使用してファイルを復元し、ADR-OPS-NNN に復元理由を記録する。

**本節で確定した方針**
- **ポストモーテムファイルの削除は禁止する。誤りは新ファイル追加（訂正ファイル）で対処する。**
- **7 年保全義務（ALCOA+ Enduring・21 CFR Part 11）に基づき、本ディレクトリは 7 年間削除しない。**
- **本ディレクトリの git history は保全証跡として扱い、`git push --force` による history 改ざんを禁止する。**

## §6 ポストモーテム索引テーブル

| PM-YYYY-NNN | ファイル名 | 発生日時 | 復旧時間 | P | 根本原因分類 | 改善処置 REC-NNN |
|---|---|---|---|---|---|---|
| — | — | — | — | — | — | — |

（障害発生時に行を追加する。初期は空）

### 6-1. 根本原因分類コード

| コード | 説明 |
|---|---|
| RC-BUG | 機能バグ（コードの欠陥）|
| RC-PERF | 性能問題（リソース枯渇・スロークエリ等）|
| RC-CONFIG | 設定ミス・環境変数誤設定 |
| RC-INFRA | インフラ障害（OS・ネットワーク・ハードウェア）|
| RC-DEP | 依存ライブラリ・外部サービスの障害 |
| RC-PROC | 手順ミス（SOP の不備・誤操作）|
| RC-UNKNOWN | 調査中または原因特定不能 |

## §7 ポストモーテム作成の SLO

| 障害優先度 | ポストモーテム作成期限 | 根拠 |
|---|---|---|
| P1 | 復旧完了から 48 時間以内 | NFR-OPS-006 |
| P2 | 復旧完了から 7 日以内 | 内部規定 |
| P3 | 任意（推奨: 翌月次作業時）| 内部規定 |
| P4 | 対象外（ポストモーテム作成は対象外と判断する）| — |

P1 ポストモーテムの 48 時間期限超過は NFR-OPS-006 違反である。超過した場合は ADR-OPS-NNN に理由を記録して quality_admin に報告する。
