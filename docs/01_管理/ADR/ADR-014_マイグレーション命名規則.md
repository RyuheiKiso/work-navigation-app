# ADR-014: マイグレーション命名規則

日付: 2026-05-19
状態: 確定

## 背景

マイグレーションファイルの命名規則として、sqlx の標準形式（`{timestamp}_{description}.sql`）と Flyway 形式（`V{version}__{description}.sql`）の両方が混在して議論されていた。

既存のマイグレーションファイル群（`src/infra/database/migrations/`）を確認すると、`V20260517120000__create_extensions.sql` のような Flyway 形式が使用されていた。
しかし sqlx はタイムスタンプ形式（`20260517120000_create_extensions.sql`）を標準としており、`V` プレフィックスと `__` 区切りは sqlx ネイティブ形式ではない。

さらに、`.up.sql` / `.down.sql` の分割形式（reversible migration）か、単一 `.sql` ファイル形式かも未確定だった。

## 決定

マイグレーション命名規則を sqlx ネイティブのタイムスタンプ形式に確定する。

```
形式: {YYYYMMDDHHmmss}_{snake_case_description}.sql
例:   20260517120000_create_extensions.sql
      20260517120058_create_app_roles.sql
```

down migration は原則作成しない（Append-only 方針との整合）。ロールバックが必要な場合は新しいマイグレーションで正方向に修正する。

## 理由

- sqlx のネイティブ命名規則に従うことで、`sqlx migrate info` / `sqlx migrate run` が追加設定なしで動作する
- Flyway 形式（`V__`）は sqlx では非標準であり、将来の sqlx バージョンアップ時に互換性問題が生じるリスクがある
- タイムスタンプ形式は複数人・複数 PR が並走してもファイル名衝突が発生しにくい
- Append-only Event Sourcing 原則（`src/CLAUDE.md`）との整合性から、down migration（削除・ロールバック）よりも正方向修正を推奨する
- 既存ファイルは `V` プレフィックスと `__` 区切りを採用しているが、sqlx が認識可能な形式への移行は次回マイグレーション追加時に統一する

## 影響

- 新規マイグレーションファイル作成時は `{YYYYMMDDHHmmss}_{description}.sql` 形式を使用する
- 既存ファイル（`V20260517XXXXXX__` 形式）は現状維持とし、sqlx の設定で認識できる範囲で運用する
- CI の `sqlx migrate info` コマンドで全ファイルが認識されることを確認する
- `scripts/apply_migrations.sh` は命名規則の変換処理を行わず、sqlx に委任する
