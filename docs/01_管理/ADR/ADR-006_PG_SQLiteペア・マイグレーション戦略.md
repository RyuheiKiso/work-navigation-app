# ADR-006: PG ↔ SQLite ペア・マイグレーション戦略

**状態**: 採択
**日付**: 2026-05-18
**担当サブ**: 05_詳細設計/01_データベース詳細設計

## 背景

バックエンド（PostgreSQL）とハンディ APP（SQLite）は論理的に同一のスキーマを参照する必要がある（トランザクション系 TBL-001/003/005/009/011/012/013/024/026/051 等のミラーテーブル群）。しかし、両者のマイグレーションツールが異なる（sqlx migrate vs TypeORM migration）ため、片方のスキーマが変更されたときに他方が追随しているかどうかを自動で検証する仕組みが存在しなかった。スキーマ乖離は実装時の発覚が最も遅く、オフライン同期の整合性破壊につながる。

## 決定事項

1. ミラー対象テーブルの DDL 変更時は、同一 PR で PG sqlx migration と TypeORM migration の両方を生成することを必須とする（ペア・マイグレーション規則）。
2. CI に `pg-sqlite-drift-check` ジョブを追加し、PG スキーマ差分が存在するブランチで TypeORM Entity AST 差分が空の場合は CI をブロックする。
3. データ型ラウンドトリップテスト（TST-intg-021〜023）を統合テストとして追加し、UUID/TIMESTAMPTZ/JSONB の PG→端末→PG 往復の一致を自動確認する。

詳細設計: `05_詳細設計/01_データベース詳細設計/07a_PG_SQLiteスキーマ同期戦略.md`

## 根拠（代替案との比較表を含む）

| 代替案 | メリット | デメリット | 採否 |
|---|---|---|---|
| ペア・マイグレーション規則 + CI チェック（採択案） | スキーマ乖離を PR レビュー段階で検知できる。両ツールの命名規則を変えることでツール混同を防止できる | CI ジョブの実装コストが発生する | **採択** |
| Liquibase/Flyway 一元管理 | ツールが 1 つで済む | SQLite の対応が限定的。TypeORM Entity との型マッピングを独自定義する必要がある | 却下 |
| 手動チェックリスト | 実装コストゼロ | レビュー漏れが必ず発生する。製造現場でのオフライン同期破壊は即停止につながるためリスクが高い | 却下 |
| schema-as-code（TypeScript で PG/SQLite を一元定義）| 型整合が自動保証される | Rust + sqlx の compile-time クエリ検証と連携できない。既存設計を大幅変更する必要がある | 却下 |

ミラー対象テーブルの数（12 件）が明確であり、ペア・マイグレーション規則で管理可能な規模と判断した。

## 結果

- PG と SQLite の両マイグレーションが同一 PR に存在することが保証される。
- CI ブロックにより、スキーマ乖離が main ブランチにマージされるリスクが排除される。
- TypeORM マイグレーション命名規則（`{ts_ms}-{desc}.ts`）と sqlx の命名規則（`V{ts}__{desc}.sql`）を意図的に変えることで、ファイル種別の誤解を防止できる。
- PG-only テーブル（TBL-002/004/006/010/014〜023/025/027〜050）は SQLite 同期対象外であることが明示されるため、将来の追加テーブルについて判断基準が明確になる。
