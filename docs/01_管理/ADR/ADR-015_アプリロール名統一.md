# ADR-015: アプリロール名統一

日付: 2026-05-19
状態: 確定

## 背景

データベースアプリケーションロールの名前が複数のドキュメントで不整合を起こしていた。

`docs/05_詳細設計/01_データベース詳細設計/` のマイグレーション SQL 群では以下のロール名が使用されていた。
- `app_event_writer`: 作業ログ INSERT 専用ロール
- `app_read_write`: マスタデータ CRUD 用ロール
- `app_admin`: スキーマ変更・GRANT 操作用ロール
- `app_event_insert`: case_locks / idempotency_keys に対する INSERT 権限ロール

一方、一部のドキュメントでは `wnav_app_writer`・`wnav_app_rw` 等の代替名が言及されていた。
また、グループロールとログインロールの境界が不明確で、アプリケーションが直接使用すべきロールが特定できなかった。

## 決定

アプリケーション用グループロール（NOLOGIN）を以下の 4 ロールに確定する。

| ロール名 | ログイン | 用途 |
|---|---|---|
| `app_event_writer` | NOLOGIN | 作業ログ（work_events）への INSERT のみ許可 |
| `app_read_write` | NOLOGIN | マスタテーブルの SELECT / INSERT / UPDATE / DELETE |
| `app_admin` | NOLOGIN | スキーマ変更・GRANT 実行（マイグレーション用） |
| `app_event_insert` | NOLOGIN | case_locks / idempotency_keys への INSERT / UPDATE（例外テーブル用）|

ログインロール（LOGIN）は別途 ADR-015 補足として `wnav_admin` / `wnav_backup` / `wnav_write` / `wnav_event_insert` / `wnav_read` を定義し、グループロールのメンバーとして付与する。

## 理由

- `src/CLAUDE.md` の「Append-only Event Sourcing」原則に従い、work_events への UPDATE / DELETE を物理禁止するため、INSERT 専用の `app_event_writer` ロールが必要
- マスタ CRUD（SOP / ユーザー / 設定等）と作業ログ記録を異なるロールで分離することで、最小権限の原則を実現する
- `app_event_insert` は case_locks（排他制御）と idempotency_keys（冪等性保証）という、通常の Append-only 例外として INSERT / UPDATE が必要な 2 テーブル専用とする
- ロール名に `app_` プレフィックスを付与することで、OS ユーザーや PostgreSQL 管理ロールとの視覚的な区別を明確にする

## 影響

- `src/infra/database/docker-entrypoint-initdb.d/02_create_app_roles.sql` でこの 4 ロールを初期化する
- `src/infra/database/migrations/` の GRANT 文はすべてこの 4 ロール名を使用する
- アプリケーション（axum バックエンド）の DB 接続ユーザーは `wnav_write`（app_read_write メンバー）または `wnav_event_insert`（app_event_writer / app_event_insert メンバー）を使用する
- ドキュメント内の代替ロール名記述は次回改訂時に本 ADR に合わせて修正する
