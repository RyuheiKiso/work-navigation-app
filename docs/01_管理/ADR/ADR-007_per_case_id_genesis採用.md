# ADR-007: per-case_id genesis 採用（グローバル genesis 不採用）

**状態**: 採択
**日付**: 2026-05-18
**担当サブ**: 05_詳細設計/07_アルゴリズム詳細設計

## 背景

ハッシュチェーンは case_id ごとに独立して管理されている（`prev_hashes: Map<UUID, [u8; 32]>` — `03_ハッシュチェーンアルゴリズム詳細.md` §5 STEP 2）。しかし、genesis ブロック（チェーンの起点）がグローバル単一か per-case_id かが明示されていなかった。これは補正レコード導入後のチェーン再構築処理の設計に直接影響する。

## 決定事項

ハッシュチェーンの genesis（`prev_hash = [0u8;32]` = `GENESIS_PREV_HASH`）は **case_id ごとに独立**して定義する。グローバルな単一 genesis（全 case_id が共通の起点を持つ構造）は採用しない。

すなわち：
- 各 case_id の **最初のイベントブロック** の `prev_hash = [0u8;32]`
- 異なる case_id のチェーンは互いに独立しており、結合を持たない
- これはすでに実装設計上の前提（`Map<UUID, [u8;32]>` のキーが case_id）と整合する

詳細設計: `05_詳細設計/07_アルゴリズム詳細設計/03_ハッシュチェーンアルゴリズム詳細.md` §1-1

## 根拠（代替案との比較表を含む）

| 代替案 | メリット | デメリット | 採否 |
|---|---|---|---|
| per-case_id genesis（採択案） | 既存実装（Map<UUID, [u8;32]>）と完全整合。case_id ごとに独立した検証が可能。補正レコードの連続計算が case_id スコープで完結する | case 間のグローバル整合性は別途証明が必要（ただし case 間の結合は要件にない） | **採択** |
| グローバル単一 genesis | 全イベントが 1 本のチェーンとなり、グローバル整合性が自明になる | 複数端末・複数 case の並列記録でシリアライズが必要になりボトルネックになる。case_id 単位の独立検証が複雑化する | 却下 |
| case_id + sequence genesis（複合キー）| 高い表現力 | 設計上の前提を超える複雑さ。製造業の ALCOA+ 監査上は case_id 単位の独立性で十分 | 却下 |

既存の `Map<UUID, [u8;32]>` 設計が per-case_id genesis を前提としており、変更コストがゼロである点が決定的。

## 結果

- `compute_chain_hash` の genesis 処理が case_id 単位で完結するため、並列書き込みの競合が最小化される。
- 補正レコード（ALG-025）は同一 case_id スコープで破断ブロックの chain_hash を継承するだけでよく、他 case_id への影響が発生しない。
- 週次検証ジョブ（BAT-001）は case_id ごとに独立したループで検証でき、並列処理が容易になる。
