# ADR-009: マルチデバイス排他方式採用（タイムスタンプ調停不採用）

**状態**: 採択
**日付**: 2026-05-18
**担当サブ**: 05_詳細設計/07_アルゴリズム詳細設計

## 背景

計画文書（`docs/02_企画/システム化計画/04_想定ユースケースとシナリオ.md` §4）には「同期競合（複数タブレットからの同一ロット記録）の解決ポリシーはタイムスタンプ優先とする」との一文があった。しかしこれは詳細設計レベルで未実装・未確定であった。

問題: ハッシュチェーン（ALG-026〜028 / ADR-007）は case_id ごとに 1 本のチェーンとして管理されるが、複数端末が同一 case_id へ並列書き込みを行った場合、`prev_hash` の決定順序が競合する（単一の `prev_hash` に対して複数の書き込みが競合するため）。これを防ぐには (A) 排他占有 か (B) 高度なシリアライズ機構（CRDT、楽観的ロック + リトライ）のいずれかが必要になる。

## 決定事項

**排他方式を採択する（1 case_id = 1 端末）**。

- POST /work-executions または POST /resume 時に TBL-051 case_locks へ原子的な INSERT を試みる（`ON CONFLICT DO NOTHING`）
- 他端末が既に占有中の場合は HTTP 409 ERR-BIZ-026 を返す
- heartbeat 未更新が 5 分を超えた場合は BAT-013（case_lock_reaper）が自動 EXPIRED 化する
- 正常なシフト交代は suspend → 解放 → 次端末で resume の順序で実現する

詳細アルゴリズム: `05_詳細設計/07_アルゴリズム詳細設計/08_Case端末占有アルゴリズム.md`（ALG-026/027/028）

計画文書 §4 の「タイムスタンプ優先」記述は本 ADR により**廃止**とする。

## 根拠（代替案との比較表を含む）

| 代替案 | メリット | デメリット | 採否 |
|---|---|---|---|
| 排他占有（case_locks）— 採択案 | ハッシュチェーンの単一書き込み者が保証され prev_hash 競合が原理的に発生しない。実装がシンプル（INSERT ... ON CONFLICT DO NOTHING）。SQLite-side の LocalCaseLock でオフライン時も既知の占有状態を保持できる | シフト交代時の解放→再占有シーケンスが必要。端末クラッシュ時の自動解放（BAT-013）まで最大 5 分のロック保持が発生する | **採択** |
| タイムスタンプ調停（計画文書の旧方針） | 複数端末の並列書き込みを許容できる | `timestamp_client` は端末時計依存で改ざん可能。`timestamp_server` 基準にしても受信順序が保証されない。ハッシュチェーンの `prev_hash` を正しく決定できず、チェーン整合性が失われる | 却下 |
| CRDT（Conflict-free Replicated Data Type）| 競合なく並列書き込みが可能 | 製造業の ALCOA+ 要件（Attributable: 一人の作業者が記録）とイベントソーシングの Append-only モデルに対応する CRDT 設計が極めて複雑 | 却下 |
| 楽観的ロック + クライアントリトライ | サーバー側のロック管理が不要 | 高頻度書き込み（step_completed イベント）でリトライ頻度が上昇する。ハッシュチェーンのアトミックな prev_hash 決定と組み合わせると実装が複雑化する | 却下 |

製造現場では 1 つの作業セッション（case_id）は原則として 1 名の作業者が担当する（ALCOA+ Attributable）。排他占有はこの物理的な現場制約を technical に反映したものである。

## 結果

- ハッシュチェーンの `prev_hash` 競合が原理的に発生しなくなる（単一書き込み者保証）。
- 端末クラッシュ等の異常ケースでも BAT-013 が 5 分以内に自動解放するため、case_id のデッドロックが発生しない。Emergency Mode 閾値（300 秒 = 5 分）と一致させることで、Emergency Mode 解除後に自動的に他端末が引き継ぎ可能になる。
- シフト交代は明示的な suspend→resume で実現されるため、work_events に `case_lock_acquired`/`case_lock_released` イベントが残り、監査証跡が完全になる。
- FR-SY-011（Case 端末排他占有）が要件として追加され、設計・実装・テストのトレーサビリティが確立される。
