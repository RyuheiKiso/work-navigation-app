# 設定 UI 監査チェックリスト

> 対応 §: ロードマップ §10.2 §10.2.2 §10.2.3 §9.2 §11.2 §28
> 対象読者: 設定 Web UI（マスタメンテナンス／作業フロー設定）の実装者・レビュア
> 改訂サイクル: §22.1 半期サイクル／PR レビュー時に都度参照
> 紐付き: §4 比較軸「非エンジニア操作性 ◎」の根拠

§10.2.2 に列挙された 14 観点を **PR レビュー時に確認可能** な形で固定する。本書は CI／PR レビューでの **沈黙の妥協（§2.2）** 検出補助として機能する。

## 1. 14 観点チェックリスト（§10.2.2）

各画面・各機能の PR で本リストを引用し、Yes／No／NA を埋めて確認する。No を残す場合は ADR（§9.4）に根拠を残す。

### 1.1. 認知負荷の最小化

- [ ] 1 画面に同時表示する設定項目が **7 ± 2 以下** に圧縮されている
- [ ] 高度設定が折り畳みまたは別タブで段階的に開示される
- [ ] 紐付き: §9.2.1 認知負荷軸

### 1.2. 自律性喪失の回避

- [ ] 強制ダイアログを禁止し、保存待ち通知バーまたは後置確認で代替している
- [ ] 操作の中断・再開が可能（離脱時下書き自動保存）
- [ ] 紐付き: §9.2.1 自律性軸／§9.2.4

### 1.3. 失敗コスト軽減

- [ ] 公開操作は 2 段階確認＋影響件数明示
- [ ] 取消可能な「予約公開」が利用可能
- [ ] ロールバック導線が常時表示されている
- [ ] 紐付き: §9.2.1 失敗コスト軸／§3.6.2 再計算

### 1.4. 不確実性の排除

- [ ] 編集中は **次推奨アクションを常時 1 つ提示**（不足提示禁止）
- [ ] 曖昧な選択肢が排除されている（複数選択肢を出すなら違いが説明されている）
- [ ] 紐付き: §3.6.4 不足提示禁止／§9.2.1 不確実性軸

### 1.5. 時間圧の回避

- [ ] **自動保存が 10 秒間隔で動作**
- [ ] オフライン下書きが保持される
- [ ] カウントダウン UI を使用していない
- [ ] 紐付き: §9.2.1 時間圧軸／§9.5.2

### 1.6. 損失回避表現（プロスペクト理論）

- [ ] 「破棄」より「下書きに残す」表記
- [ ] 「失敗」より「未到達」表記
- [ ] 紐付き: §9.2.3 プロスペクト理論

### 1.7. 安全側既定（デフォルト効果）

- [ ] 業界テンプレ＋安全側既定値が用意されている
- [ ] 空欄起点を避け、推奨値が初期化されている
- [ ] 紐付き: §9.2.3 デフォルト効果

### 1.8. ピーク・エンド設計

- [ ] **フロー公開完了画面に進捗総括＋ロールバックリンクを配置**
- [ ] 紐付き: §9.2.3 ピーク・エンドの法則

### 1.9. ゴール・グラディエント

- [ ] 編集進行のステップバー（残 N 項目）を可視化している
- [ ] 紐付き: §9.2.3 ゴール・グラディエント効果

### 1.10. メンタルモデル整合

- [ ] 業界別テンプレで現場用語に整合している（§3.1.5.4）
- [ ] [`glossary-ja.md`](./glossary-ja.md) と一致した用語のみ使用している
- [ ] 紐付き: §3.1.5／§28

### 1.11. ダーク・パターン禁止（§9.2.4）

- [ ] 公開・破棄ボタンの誤誘導が無い（位置・色・サイズで誤タップ誘導していない）
- [ ] 既定で安全側に倒れる（破壊的操作を既定にしない）
- [ ] 勧誘的バッジ・偽の希少性が無い
- [ ] 紐付き: §9.2.4

### 1.12. 即時フィードバック

- [ ] 操作 100ms 以内に視覚反応がある（§5.2 整合）
- [ ] 保存完了は **視覚＋触覚二重** でフィードバックされる
- [ ] 紐付き: §9.2.2／§9.5.4

### 1.13. 段階的開示

- [ ] 推奨 → 標準 → 高度 の 3 段で項目が漸増する
- [ ] 高度設定は折り畳み既定
- [ ] 紐付き: §3.6.4 過剰提示禁止

### 1.14. アクセシビリティ

- [ ] **キーボード単独操作で完了可能**
- [ ] WCAG 2.2 AA を axe-core が緑
- [ ] タッチターゲット 9mm 以上（推奨 11mm、§21 注 6）
- [ ] 紐付き: §11.2

## 2. ストレス予測の事前検証（§10.2.3）

設定者が試行版発行 **前に** 「このフローを実行する作業者の予測ストレス」を確認できることが §10.2.3 受入観点。

- [ ] 編集中フローに対し §9.2.1 5 軸スコア（認知負荷／時間圧／不確実性／自律性喪失／失敗コスト）が計算・表示される
- [ ] スコアが規定値超のステップは **試行版発行をブロック** する（オプトアウト時は ADR で根拠記録）
- [ ] 予測根拠（参照 SOP・公開資料・LLM 推定）が脚注で参照可能
- [ ] スコアの月次トレンドが §16 観測ダッシュボードに反映される

## 3. 性質ベーステスト要件（§10.2.4）

- [ ] §10.2.3 のブロック条件が性質ベーステストで検証されている
- [ ] HSM 到達可能性・デッドロック検出が CI 緑（§3.4.1）
- [ ] 試行版 → パイロット → 本番昇格の経路が §10.6 同期戦略と整合し E2E テスト緑

## 4. PR テンプレートとの統合

- 本書のチェックリストを `.github/PULL_REQUEST_TEMPLATE.md` から参照する。
- 「設定 UI に影響する PR」は本書全項目の Yes／NA を必須とする。
- No が残る場合は ADR（§9.4）と §24.2 出所表に追記する。

## 5. 受入観点（§10.2.4）

- 14 観点が UI 監査チェックリストとして本書で網羅されている。
- §10.2.4 受入観点「非エンジニアがサンプル参照のみで 5 分以内に最初のフローを発行できる」が手順実機検証で達成されている（コード初期実装後）。
- ロールバックが操作 3 ステップ以内（コード初期実装後）。
- §9.2.4 ダーク・パターン監査で検出件数 0 を維持。
