# 一流の設計書作成ガイドライン

本ディレクトリの成果物は、本プロジェクトにおける **実装を一意に導く文書** である。
ここで定めた方針が、コード・テスト・運用すべての直接の上流制約となる。
凡庸を許容しない。テンプレを埋めただけの設計書は本ガイドラインへの違反とみなす。

上位方針との関係:
- ルート `CLAUDE.md`: 「技術的に至高を目指す」
- `docs/CLAUDE.md`: アスキー作図禁止 / drawio→PNG / MVP・Phase は定めない
- `docs/01_企画/CLAUDE.md`: 一流の企画書の3条件（不可避性／反証耐性／行動可能性）
- `docs/01_企画/企画書.md` v0.7.1: 楔（直接作業者×ライン生産×段取り替え×単位作業）／Apache-2.0 ／ React ／ PostgreSQL ラージオブジェクト ／ Keycloak ／ Tauri ／ SQLite

---

## 0. 一流の設計書の定義（このガイドラインの根本基準）

凡庸な設計書は **形式（章立て・図の数）** を満たす。一流の設計書は **再現性・境界明示性・トレーサビリティ** を同時に満たす。

1. **再現性 (Reproducibility)** — 設計書だけを読んだ別人が、ほぼ同じ実装に到達できる。曖昧な余地を残さない。
2. **境界明示性 (Explicit Boundaries)** — モジュール間の責務・契約・変更影響範囲が明示される。何を変えると何が壊れるかが追える。
3. **トレーサビリティ (Traceability)** — 企画書の仮説・Moat・Kill Criteria・物理制約に対し、設計のどの選択がそれに応えるかが追える。

「動くものが作れる」ではなく「実装の選択を予測可能にする」が一流の設計書。

## 1. 思考フレームワーク（採用必須）

該当章を書くとき、これらの道具を使って思考した痕跡が文書に残ること。

### 1.1 C4 Model（Context／Container／Component／Code）
階層を混ぜない。最低でも C1（システム全境界）／C2（コンテナ）／C3（コンポーネント）まで書き、C4（クラス・関数）は実装に委ねる。図の凡例に「この図は C4 L?」を必ず明記する。

### 1.2 ADR（Architecture Decision Record）
すべての非自明な選択を ADR 形式で記録。MADR (Markdown ADR) を採用。**却下案の理由を必ず書く**。事後合理化（採用したものだけ書く）はアンチパターン。

### 1.3 Domain-Driven Design（戦術設計）
Aggregate ／ Entity ／ Value Object ／ Repository ／ Domain Event を明示的に切る。境界づけられたコンテキストは「楔」を中心に 1 つ。Generic／Supporting Subdomain を分離する。

### 1.4 Event Storming（同期・実績設計）
イベント ／ コマンド ／ ポリシー ／ 集約 ／ 外部システム を時系列で並べる。同期エンジン（§8.4.3）と実績スキーマ（§8.4.2、イベントソーシング）の設計は Event Storming の結果に基づく。

### 1.5 失敗モード分析（FMEA）
非機能（同期失敗 ／ 端末故障 ／ Wi-Fi 断 ／ DB 障害 ／ Keycloak 障害）について、**発生 ／ 検出 ／ 影響 ／ 対応** をマトリクス化。企画書 §8.7 のフェイルセーフは FMEA の入力。

### 1.6 セキュリティ脅威モデル（STRIDE）
Spoofing ／ Tampering ／ Repudiation ／ Information Disclosure ／ Denial of Service ／ Elevation of Privilege の 6 カテゴリで脅威を列挙し、対策を ADR に紐付け。1 カテゴリでも空欄なら未完成。

### 1.7 容量計画（数値の根拠提示）
企画書 §13.1.1 の数値（端末 10〜100 ／ 手順 100〜1,000 ／ 実績 100,000〜1,000,000 件／年 ／ メディア 10〜100MB／手順）を入力に、PostgreSQL ラージオブジェクト容量・SQLite 容量・同期帯域・バックアップサイズを式付きで試算する。

### 1.8 Pyramid Principle（バーバラ・ミント）
各章は「結論先出し」「MECE」「So What? / Why So?」のテストを通す。図と表を並べて満足しない。**主張→根拠→根拠→根拠** の縦の論理が通っていること。

## 3. 推奨章構成（ただし形式に逃げないこと）

設計書本体 `設計書.md` の章構成。不要章は「該当なし＋理由」を明記して省略する（沈黙の省略は禁止）。

1. エグゼクティブサマリ（10 行以内・実装者が次に読む章を即決できる粒度で）
2. 上流制約の再掲（§2 表へのリンク）と節ごとの応答対応表
3. システム全体（C4 L1 Context）
4. コンテナ構成（C4 L2、企画書 図 04 を本書で更新）
5. ドメインモデル（DDD 戦術: Aggregate ／ Entity ／ VO ／ Repository ／ Domain Event）
6. 手順 DSL 仕様（YAML スキーマ ／ JSON Schema ／ TWI-JI 三段の必須化方法 ／ DAG 表現 ／ 条件分岐 ／ メディア参照）
7. 実績スキーマ（イベント定義 ／ MATERIALIZED VIEW 定義 ／ 監査ハッシュチェーン ／ スキーマ移行）
8. 同期プロトコル（Event Storming 図 ／ シーケンス図 ／ idempotency key 設計 ／ 競合解決 ／ 時刻整合）
9. メンテナンス Web 設計（React コンポーネント階層 ／ ルーティング ／ 状態管理 ／ API 契約 ／ OpenAPI／GraphQL SDL）
10. 端末アプリ設計（Tauri 構成 ／ SQLite スキーマ ／ オフライン UX ／ ハプティック・視覚フラッシュ ／ 配布バンドル）
11. 認証・認可（Keycloak 連携 ／ OIDC フロー ／ ロール定義 ／ トークンライフサイクル ／ オフライン時の扱い）
12. メディア管理（pg_largeobject 設計 ／ アクセス制御 ／ バックアップ ／ 端末側キャッシュ）
13. 基幹連動アダプタ（プラガブル枠組み ／ REST／GraphQL クライアント ／ サンプル実装）
14. 容量計画・性能設計（§1.7 の試算）
15. セキュリティ設計（STRIDE ／ 監査ログ ／ TLS ／ パスワードポリシー ／ ブルートフォース対策 ／ Keycloak 連携の脅威）
16. テスト戦略（ユニット ／ ゴールデンマスター ／ プロパティ ／ ファジング ／ 性能 ／ カバレッジ目標）
17. 可観測性（ログ ／ メトリクス ／ 同期遅延・障害可視化 ／ Kill Criteria 観測導線）
18. CI／CD 設計（PR ゲート ／ 静的解析 ／ ビルドパイプライン ／ リリース署名は §14.6 と整合）
19. 失敗モード分析（FMEA、§1.5）
20. ADR 一覧（リンク集、ADR は別ファイル `adr/`）
21. オープンクエスチョン（設計内に残った未確定）
22. 用語集（企画書 §18 を継承＋設計固有語のみ追加）
23. 更新履歴（日付・変更点・変更理由）

## 4. 図表ルール（プロジェクト共通の再掲＋強化）

- アスキーアートでの作図は **禁止**。
- drawio で作成し PNG 書き出し、`./img/` に格納、相対パス参照。
- drawio 原本（`.drawio`）も同梱。
- ファイル名: `NN_<内容>_<版>.png`（例: `01_C4_Container_v1.png`）。
- 図には番号・タイトル・凡例・前提条件のキャプションを付し、本文と独立して読めること。
- シーケンス図は drawio 由来 PNG または Mermaid 由来 PNG に統一（手書きシーケンス・コードブロック内 Mermaid のみは禁止）。
- C4 図は凡例に「C4 L?」を必ず明示。

## 6. 文書としての品質基準

### 6.1 文体
- 一文一義。50 字超の文は分割を検討。
- 主語省略の連鎖を禁ずる（「設計上は〜とする」の主語を必ず書く）。
- 形容詞単独の主張を禁ずる（"高速な" → "p95 200ms 以内"）。
- 受動態・婉曲表現で意思決定を曖昧化しない（"〜と思われる" "〜が望ましい" は推敲対象）。

### 6.2 数字
- API レイテンシ・容量・スループットは「観測値・規格根拠」「設計目標（仮説、未検証）」「方針」の三分で記す（企画書 §13.1 と同様）。
- 単位 ／ 観測時点 ／ 観測対象 ／ N 数 ／ 出典を必ず併記。
- 出典のない統計は「仮説」と明記する。
- 偽の精度を避ける（観測 N=3 で "12.4ms" と書かない、"おおむね 10〜15ms (N=3, 2026-MM)" と書く）。

### 6.3 用語
- 企画書 §18 を継承。設計固有語のみ本書末尾の用語集に追加。
- 「セッション」「タスク」「ジョブ」「手順」「ステップ」を企画書定義から外して使わない（§5.2 用語不混用ルール）。
- ドメイン用語は初出で 1 行定義。

### 6.4 スキーマ・API
- DSL の JSON Schema は `schema/` に格納し、本文からリンク。本文に貼る場合は抜粋＋全文へのリンクとする。
- API 契約は OpenAPI 3.x または GraphQL SDL の正本を `schema/` に置き、本文ではエンドポイント表のみを示す。
- DB DDL は `schema/ddl/` に置き、本文には ER 図と主キー・外部キー一覧のみを示す。
- すべてのスキーマは **正本を 1 箇所** に置き、本文は参照のみ（二重管理禁止）。

## 7. アンチパターン（一発で一流から外れる兆候）

- **テクノロジー先行**: 上流制約（§2）から外れる技術選定（NoSQL 追加 ／ Redis 追加 ／ MongoDB 化 等）を、根拠なく入れる。
- **既製品の再発明**: Keycloak が担う領域（OIDC・パスワードポリシー・MFA 基盤）を自作する／PostgreSQL の機能（pub/sub・全文検索・ジョブ）を自作する。
- **網羅信仰**: 全クラス図を貼り、優先順位を語らない。設計の"中心"が読めない。
- **API シアター**: REST／GraphQL のスタイルだけ決めて、エラー応答／冪等性／リトライ／レート制限を書かない。
- **シーケンス図の飾り**: ハッピーパスのみ、失敗パス／タイムアウト／リトライ／補償が描かれていない。
- **ADR の事後合理化**: 採用案だけを書き、却下した代替を書かない。
- **テスト後付け**: 実装後の流れだけを書き、ゴールデンマスター対象 DSL ／ プロパティ対象 ／ ファジング対象を特定しない。
- **Phase への逃避**: 決められない判断を「次フェーズ」に押し付ける（本プロジェクトでは禁止）。
- **企画書の書き換え**: 設計書側で上流制約を勝手に動かす。設計が制約を動かしたい場合は **企画書を先に改訂する**。

## 9. ファイル運用

- `設計書.md` — 本体。最新の意思決定を反映する単一の正。
