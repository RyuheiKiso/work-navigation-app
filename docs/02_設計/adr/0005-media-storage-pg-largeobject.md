# ADR-0005. メディアストレージ: PostgreSQL ラージオブジェクト（pg_largeobject）

| 項目 | 内容 |
|---|---|
| Status | Accepted |
| Date | 2026-05-09 |
| Deciders | RyuheiKiso |
| 関連 | 企画書 §8.3（v0.7.1 で本決定）、上流制約 C-4 |

## Context and Problem Statement

手順 DSL のメディア（画像・短い動画 10 秒以内・音声、企画書 §8.4.1）をどこに格納するか。容量は手順 1 件あたり 10〜100MB（企画書 §13.1.1、Q37-L／M）、手順件数は 100〜1,000 件（Q37-E／F）。1 インスタンス＝1 工場（企画書 §14.10）の前提で、同一 PostgreSQL 内のラージオブジェクト（pg_largeobject）か、隣接オブジェクトストレージ（MinIO 等の S3 互換）か、ファイルシステム直接かを選定する必要がある。

## Decision Drivers

- **Docker 1 コマンドで自社オンプレに立てられる**（企画書 §2.1 プレスリリース、§14.4）。docker-compose.yml の構成要素は最小化したい。
- **単一メンテナの保守限界**（企画書 Q15-A）。コンポーネント数の増加は保守負荷を直接上げる。
- **トランザクション一貫性**: 手順 DSL の改訂とメディア追加・削除を 1 トランザクションで扱えること（企画書 §13.8 の単純な版管理に整合）。
- **バックアップ・リストアの単純化**: PostgreSQL の `pg_dump` 1 つでメディア込みのバックアップが取れること。
- **容量上限**: 楔範囲（手順 1,000 件 × メディア 100MB ＝ 約 100GB）が PostgreSQL ラージオブジェクトで実用範囲に収まること。
- **アクセス制御の単純化**: 認証・認可（Keycloak 委譲、ADR-0011）と整合する単一の制御点で済むこと。
- **オフライン同期との整合**: メディアの差分同期（手順 ID＋版番号＋ハッシュ、企画書 §8.4.3）が実装可能であること。

## Considered Options

1. **PostgreSQL ラージオブジェクト（pg_largeobject）**
2. PostgreSQL `BYTEA` カラム
3. 隣接オブジェクトストレージ（MinIO 等の S3 互換、docker-compose に同梱）
4. ファイルシステム直接（コンテナのボリュームに直接保存）
5. 外部マネージド S3（AWS S3／GCS／Azure Blob 等）

## Decision Outcome

**選定: PostgreSQL ラージオブジェクト（pg_largeobject）**

`docker-compose.yml` の構成要素を増やさずに、手順 DSL とメディアを単一 DB で管理できる。`pg_dump` 1 つでバックアップが完結し、トランザクション一貫性も担保される。容量上限（楔範囲約 100GB）は pg_largeobject の運用実績内に収まる。

### Consequences

- **Good**: docker-compose の構成要素が PostgreSQL ＋ React SPA ＋ API ＋ Keycloak の 4 つに収まる（オブジェクトストレージを足さない）。
- **Good**: バックアップ・リストアが `pg_dump` 1 系統で完結する。
- **Good**: トランザクション一貫性により、手順改訂とメディア追加・削除のアトミック性が保証される。
- **Good**: アクセス制御がアプリケーション層（Keycloak ＋ API）の単一系統で済む。
- **Bad**: ラージオブジェクトはストリーミング読出が `BYTEA` より複雑（`lo_open` ／ `lo_read` API を使う）。本決定では API 層でストリーミング・レスポンスを実装する。
- **Bad**: 容量が 100GB を超える運用は本 ADR の範囲外。楔範囲外（隣接形態・大規模工場）への適用時は別 ADR で再評価する。
- **Bad**: PostgreSQL 内部の `pg_largeobject` システムカタログが肥大化するため、定期 VACUUM・WAL 量増加の運用配慮が必要。これは設計書 §17 可観測性で監視メトリクスとして設計する。

### 採用しない選択肢の取扱い

- 隣接オブジェクトストレージ（MinIO）は **却下**。docker-compose 構成要素増加と単一メンテナの保守限界を理由とする。
- 外部マネージド S3 は **却下**（企画書 §13.5 のデータ主権要件、自社オンプレ完結の方針と矛盾）。

### 関連する後続論点

- ストリーミング配信の API 設計（範囲リクエスト対応）。
- 端末側 SQLite のメディアキャッシュ戦略（差分同期と整合）。
- メディアの MIME タイプ・サイズ上限の DSL 側バリデーション（JSON Schema）。
- VACUUM / 統計収集のスケジュール（運用設計）。

## Pros and Cons of the Options

### PostgreSQL ラージオブジェクト（採用）

- **Good**: 単一 DB、単一バックアップ、トランザクション一貫性。
- **Good**: 容量上限 4TB／オブジェクト（PostgreSQL 仕様）、楔範囲に十分。
- **Bad**: API がストリーミング向きで実装が複雑（`lo_*` 関数）。
- **Bad**: VACUUM・WAL の運用配慮が必要。

### PostgreSQL `BYTEA` カラム（却下）

- **Good**: クエリが単純（通常の SELECT で取得）。
- **Bad**: 1 行に全データを載せるためストリーミング読出が困難（toast 経由でも遅延が大きい）。
- **Bad**: 短い動画 10 秒（数 MB〜数十 MB）レベルでクライアント側のメモリピークが許容範囲を超える可能性。

### 隣接オブジェクトストレージ（MinIO 等、却下）

- **Good**: ストリーミング配信が S3 API で簡潔。
- **Good**: 容量上限が事実上ない。
- **Bad**: docker-compose 構成要素が 1 つ増え、認証・認可も二重管理になる（PostgreSQL と MinIO 双方のバックアップ・アクセス制御）。
- **Bad**: バックアップが `pg_dump` ＋ MinIO バックアップの 2 系統になり、整合性管理が単一メンテナで保守不能になりやすい。
- **Bad**: 単一メンテナ前提と矛盾（企画書 §16.1 シナリオ 1 のリスク増）。

### ファイルシステム直接（却下）

- **Good**: 最も単純、外部依存ゼロ。
- **Bad**: バックアップ整合性が DB と非同期で取れない（pg_dump 完了時刻と FS スナップショット時刻のずれが手順 DSL ↔ メディアの整合を破壊する）。
- **Bad**: トランザクション一貫性なし。改訂中の手順削除でメディアの孤児が発生する。

### 外部マネージド S3（AWS S3 等、却下）

- **Bad**: データ主権要件（企画書 §13.5、自社オンプレ完結）と矛盾。
- **Bad**: クラウド送信を要求し、企画書 §10.2 「商用 SaaS の劣後点」と本アプリの差別化を打ち消す。
