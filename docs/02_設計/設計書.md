## 1. エグゼクティブサマリ

1. 本設計書は、企画書 v0.7.1 で確定した上流制約 17 件（C-1〜C-17）を **下流の実装が一意に再現できる粒度** に展開する。設計書本体（§1〜§23）と ADR（`adr/`）と スキーマ正本（`schema/`）の三位一体で「実装の選択を予測可能にする」（02_設計/CLAUDE.md §0）。
2. システム境界は §3（C4 L1）で 1 つに閉じる。本システムは **直接作業者が端末で再生し、手順作成者・改訂者が Web で編集する単一のシステム** であり、外部は **Keycloak（認証委譲、ADR-0011）** と **基幹システム／MES（プラガブル連携、§13 で詳細化予定）** の 2 つに限定する。
3. コンテナ構成は §4（C4 L2）で 6 つに分解する: 端末アプリ（Tauri）／端末ローカル DB（SQLite）／メンテナンス Web SPA（React、ADR-0004）／API バックエンド／PostgreSQL（DSL ／ 実績 ／ ラージオブジェクトメディア ／ 監査ログ、ADR-0005）／Keycloak（外部）。基幹システムは外部参照のみ。
4. 独自性集中点（企画書 §9.2 Custom 領域 = 手順 DSL ／ オフライン同期エンジン）と Moat の核（企画書 §11.2 = TWI-JI 三段必須化 ／ ISO 9001 第 7.5 項対応）に対応する設計章は §6（DSL）／§7（実績）／§8（同期）／§13（基幹連動）／§15（セキュリティ）。これらは v0.2.0 以降で本書に追加する。
5. 本版（v0.1.0）の確定範囲は **境界とコンテナの確定まで**。次に読むべき章は実装者の役割により分岐する: ドメインモデル設計者は §5（v0.2.0）／DSL 実装者は §6（v0.2.0）／同期エンジン実装者は §8（v0.2.0）／フロント実装者は §9・§10（v0.3.0）／インフラ実装者は §11・§12・§18（v0.3.0）。

### 3.2 構成要素（5 要素）

| 種別 | 要素名 | 責務 | 出典 |
|---|---|---|---|
| Person | 直接作業者（オペレータ） | 端末アプリで単位作業を再生し、ステップ完了入力・実績入力を行う | 企画書 §6.1／§4.1 J-A |
| Person | 手順作成者・改訂者（IE／班長／品証） | メンテナンス Web で手順を作成・改訂し、実績ダッシュボードを参照する | 企画書 §4.3／§13.6 |
| System | **work-navigation-app**（本システム） | 楔の場面で単位作業をナビゲーションする OSS。端末アプリ＋メンテナンス Web＋共通基盤の三層 | 企画書 §1.2 |
| External System | 基幹システム／MES | 生産計画・順序情報を本システムに供給し、実績を任意で受領する。REST／GraphQL を一次インタフェースとする | 企画書 §8.5／Q10-A |
| External System | Keycloak | 認証・認可を OIDC で本システムに提供する。ADR-0011 で確定 | ADR-0011 |

### 3.3 主要関係（4 本）

| 関係 | 主体 → 客体 | 内容 | プロトコル |
|---|---|---|---|
| R-1 | 直接作業者 → 本システム | 単位作業の再生・実績入力 | 端末 GUI（タッチ） |
| R-2 | 手順作成者 → 本システム | 手順作成・改訂・ダッシュボード閲覧 | Web GUI（マウス・キーボード） |
| R-3 | 本システム → 基幹システム／MES | 生産計画取込／実績返却（任意） | HTTPS（REST／GraphQL、プラガブル） |
| R-4 | 本システム → Keycloak | OIDC 認証委譲（PKCE 必須） | HTTPS（OIDC） |

### 3.4 境界の根拠

- 直接作業者と手順作成者を **別 Person** に切り出した根拠は、企画書 §4.3 で楔と射程外（J-D／J-E）が分離されており、UI・物理制約・権限が役割で分かれるため。
- 基幹システムを **External** に置く根拠は、企画書 §8.4.3 ／ §13.5 で「認証情報は本アプリで保管しない」「データ主権は導入先に閉じる」が確定しているため、本システムの責務範囲外。
- Keycloak を **External** に置く根拠は、企画書 §9 Wardley Map「G 認証・認可」を Product 領域に置き、ADR-0011 で「既製品の再発明を避ける」を本決定としたため。docker-compose には同梱するが、本システムのコンテナ境界には含めない（運用上の境界 ≠ 配布の境界、§4.4）。
- システム数を **1 つに閉じる** 根拠は、企画書 §6.1 で楔が単一の場面に絞り込まれており、複数システム（例: ダッシュボード専用システムと再生専用システムの分離）に分けると C-1 の不可避性（企画書 CLAUDE.md §0）が崩れるため。

### 4.3 コンテナ間関係（9 本）

| 関係 ID | 主体 → 客体 | 内容 | プロトコル ／ 技術 |

### 4.4 配布パッケージ（C-12 配布制約への応答）

| パッケージ | 内容 | 配布チャネル |
|---|---|---|
| 端末（Android） | Co-1 ＋ Co-2 を同梱した APK | GitHub Releases（直接配布） |
| 端末（Windows） | Co-1 ＋ Co-2 を同梱した MSI／MSIX | GitHub Releases（直接配布） |
| メンテナンス Web 一式 | Co-3 ＋ Co-4 ＋ Co-5 ＋ Ex-1（Keycloak）を同梱した `docker-compose.yml` ＋ サンプル DB 初期化スクリプト | GitHub Releases（Docker イメージ） |

> Ex-1 Keycloak は **外部コンテナ**（本システムの責務外）であるが、**配布上は docker-compose に同梱**する。これは「運用上の境界」と「配布の境界」が一致しないことの明示であり、Keycloak の脆弱性対応・バージョンアップは Keycloak 上流に追従する（本プロジェクトでフォークしない）。Helm Chart ／ ワンライナーインストーラは企画書 NS-6 のため非スコープ。

### 4.5 オフライン動作の境界（C-9 物理制約・企画書 §6.4 への応答）

| 状態 | 動作 |
|---|---|
| 通常（オンライン） | R-4 の同期がプル／プッシュ双方で動作。R-7 の Refresh Token ローテーションが定期実行 |
| Wi-Fi 断（短時間） | Co-1 は Co-2 SQLite のみで完全自律動作。実績は Co-2 のキューに積み上がる。R-4 復旧時にバックフィル同期 |
| Wi-Fi 断（長時間、数日〜数週間） | Co-1 は最後に取得した Access Token の JWT 検証（公開鍵キャッシュ）で動作継続。Refresh Token の有効期限切れまでオフライン継続可能 |
| Co-4 API 停止 | Co-1 は完全自律動作（Wi-Fi 断と同等）。Co-3 SPA は読み取り不可、回復時に再接続 |
| Ex-1 Keycloak 停止 | 既ログイン端末は JWT 期限まで動作継続。新規ログインは不可。Co-3 SPA は新規ログイン不可、Ex-1 復旧待ち |
| Ex-2 基幹／MES 停止 | Co-4 は前回キャッシュで動作。実績はリトライキューに積み、Ex-2 復旧時に idempotency key で重複防止 |
