## 5. ハルシネーション対策（§18.5.1）

- 数値（タッチターゲット 9mm／カバレッジ 90%／オフラインログイン 7 日 等）はロードマップ §21 脚注からの引用に限定し、独自数値の創出はしない。
- 規格名（ISO 9241-411／NIST FIPS 197 等）はロードマップ §24.2 出所表に既記載のもののみ引用。
- LLM の内蔵知識のみで根拠付けされる主張には「（参考値）」「（推定）」等の留保を明示する。

## 6. 経年劣化対策（§18.5.2）

- 採用根拠が時刻不変でない（OSS の活動状況・規格の改訂・競合動向）箇所は §22.1 半期サイクルで再評価する。
- リンク切れ検出（`scripts/check-links.sh`、未整備）は CI 整備後に必須化する。
- モデル世代交代（Claude Opus 4.7 → 4.8 等）時には主要 ADR（特に ADR-0001／0002／0003）を再評価する。

## 7. 誤出力検出時のフロー（§18.5.1）

- 検出された誤出力は GitHub Issue（ラベル `llm-error`）で起票。
- §22.4 是正フローで対応。
- `MEMORY.md` 等のメモリ機構には本セッションの誤出力を記録しない（PII 観点）。

## 8. 受入観点（§18.5.3）

- 本書が `docs/llm-sessions/` に保存されていること（達成）。
- 主要 ADR に LLM の代替案と却下理由が記載されていること（ADR-0001 〜 0008 で達成）。
- `docs/llm-sessions/` の更新が直近 30 日で 1 件以上あること（本書で達成）。
- リンク切れ検出 CI が稼働し、失敗時に PR を fail させること（CI 整備後）。
- `llm-error` ラベル件数を §22.1 で集計し、再発防止策が記録されていること。
- モデル世代交代時の再評価記録が CHANGELOG に残ること。
