# 作業ログ分析とプロセスマイニング

## 1. プロセスマイニングとは

**プロセスマイニング (Process Mining)** は情報システムのイベントログからビジネス・製造プロセスの実態を発見・分析・改善する学際的手法である。**Wil van der Aalst** (Eindhoven 工科大学) が 2000 年代初頭に体系化し、その著書 *Process Mining: Data Science in Action* (2016) が標準的な参照書となっている。

プロセスマイニングは **データ指向の IE** と位置づけられ、従来の時間研究・動作研究が人間の観察に依拠するのとは対照的に、システムが自動生成するログデータから工程の実態を客観的に把握する。

## 2. 3 つの主要機能

van der Aalst はプロセスマイニングを 3 機能に分類した:

```
         イベントログ
              |
  ┌───────────┼───────────┐
  ↓           ↓           ↓
Discovery  Conformance  Enhancement
(発見)     (適合確認)    (拡張)
  ↓           ↓           ↓
実際の工程  逸脱の検出  パフォーマンス分析
モデルを    (標準 vs    (ボトルネック・
生成する    実際の差分)  処理時間・コスト)
```

### 2.1 Discovery (発見)

イベントログのみを入力とし、プロセスモデル (フローチャートや Petri net) を自動生成する。事前に設計図のない「as-is プロセス」を可視化するのに使用する。

### 2.2 Conformance Checking (適合確認)

既存の規範モデル (標準作業手順書) とイベントログを照合し、**逸脱 (Deviation)** を定量的に検出する。これは作業ナビゲーションアプリが生成するログと標準手順の差分分析に直結する。

### 2.3 Enhancement (拡張)

既存モデルをイベントログの情報で拡張し、処理時間・コスト・リソース使用率などのパフォーマンス情報を付加する。

## 3. イベントログのデータモデル

プロセスマイニングの入力となるイベントログは最低限以下の要素を持つ:

| 属性 | 意味 | 例 |
|---|---|---|
| **Case ID** | 1 件の作業インスタンスを一意に識別 | 製造ロット番号、作業指示 ID |
| **Activity** | 実行されたアクティビティ名 | "ステップ03_ボルト締め付け" |
| **Timestamp** | アクティビティの開始または完了時刻 | 2024-05-15 14:32:07 |
| Resource | 実行者 (オプション) | 作業者 ID、端末 ID |
| Attribute | 追加属性 (オプション) | 測定値、部品ロット番号、判定結果 |

製造ナビゲーションアプリのログ設計では、**各ステップの開始・完了・スキップ・差し戻しイベント** をCase ID = 作業指示番号に紐付けて記録することで、このデータモデルを自然に満たせる。

## 4. 代表的なアルゴリズム

### 4.1 α アルゴリズム

van der Aalst らが提案した Discovery の基礎アルゴリズム。イベントログ内のアクティビティ間の **前後関係 (Ordering Relations)** を統計的に抽出し Petri net を生成する。単純な構造には有効だが、ループ・ノイズ・並行処理に弱い限界がある。

### 4.2 Heuristic Miner (HM)

α アルゴリズムの実用的拡張版 (Weijters & van der Aalst, 2003)。頻度閾値でノイズを除去でき、製造現場のような不完全なログでも安定したモデルを生成する。最も広く使用される Discovery アルゴリズムの一つ。

### 4.3 Inductive Miner (IM)

Leemans et al. (2013) が提案。生成したモデルが **再生可能 (Sound)** であることを保証するため、工程フローの整合性が高い。複雑なプロセス構造に適する。

## 5. Conformance Checking の手法

### 5.1 Token Replay

ログの各ケースを規範モデル (Petri net) に沿ってトークンを流し、**不足 (Missing Token)** と **残余 (Remaining Token)** を計測することで適合度を定量化する。

```
標準手順モデル: A → B → C → D
実際ログ:      A → B → D (C をスキップ)
             → Token Replay で C の Missing Token = 1 を検出
```

### 5.2 Alignment-Based Conformance

各ケースと規範モデルとの最適なアライメントを計算し、**逸脱コスト** を定義する。Token Replay より高精度だが計算コストが高い。

### 5.3 適合度指標

| 指標 | 定義 | 意味 |
|---|---|---|
| Fitness | ログがモデルに従っている割合 | 標準手順遵守率 |
| Precision | モデルがログ以外の行動を許容していない程度 | 手順の厳密さ |
| Generalization | モデルが新しいケースに対応できるか | 手順の汎用性 |
| Simplicity | モデルの複雑さの逆数 | 理解しやすさ |

## 6. 製造業の適用事例

| 企業・機関 | 適用内容 | 成果 |
|---|---|---|
| Siemens AG | 製品開発プロセスの Conformance Checking | 規格逸脱の早期発見、承認フロー短縮 |
| BMW Group | 製造ライン工程の Discovery と最適化 | ボトルネック工程の特定と改善 |
| 富士フイルム | 医療機器製造の電子記録ログ分析 | FDA 規制対応の監査証跡強化 |

## 7. 統計的工程管理との融合

プロセスマイニングの時系列データは **統計的工程管理 (SPC)** と組み合わせることで強力な品質分析ツールとなる ([11_計測・工程能力と統計的品質工学](./11_計測・工程能力と統計的品質工学.md) 参照)。

具体的には:
- ステップごとの **処理時間 (Cycle Time)** の管理図を作成し、異常なばらつきを検出
- 特定のステップを頻繁にスキップ・差し戻す作業者・端末を特定 (**リソース分析**)
- 手順逸脱の頻度と不良品発生率の **相関分析** による原因特定

## 8. 作業ナビゲーションアプリへの含意

作業ナビゲーションアプリが記録するログが、そのままプロセスマイニングの入力データとなるよう以下を設計に組み込む:

1. **Case ID の設計**: 作業指示 ID / ロット番号を統一キーとして全テーブルに持たせる
2. **Activity の粒度**: 「作業 A を開始」「ステップ 3 を完了」「測定値 X を入力」を別イベントとして記録
3. **タイムスタンプの精度**: ミリ秒単位、UTC 統一で記録し端末時刻ずれを考慮
4. **スキップ・差し戻しの明示的記録**: 省略や再実行もイベントとして記録し、ログから除外しない
5. **Resource 属性**: 作業者 ID・端末 ID を必須属性として付与

## 9. 批判と限界

プロセスマイニングは記録されたイベントのみを扱い、**記録されない暗黙の判断・調整行動** は分析対象外となる。また、高品質なログがなければ精度が大きく低下するため、ログ収集インフラへの投資が前提条件となる。van der Aalst 自身も、プロセスマイニングは「what happened」を示すが「why it happened」には人間の解釈が必要と強調している。
