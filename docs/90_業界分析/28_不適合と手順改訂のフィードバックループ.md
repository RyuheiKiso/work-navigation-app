# 不適合と手順改訂のフィードバックループ

## 1. 問題の所在

作業ナビゲーションアプリは「現在の正しい手順を提示する」機能を中核とする。しかし、品質異常・ヒヤリハット・不適合が発生したとき、それを手順の改訂に反映し、次回以降の作業でより良い手順が提示されるようにする **フィードバックループ** が閉じていなければ、同一のエラーが繰り返される。

本章は「不適合の発生 → 根本原因の特定 → 手順の改訂 → 教育 → 再発防止の確認」という Closed-loop Quality の仕組みを整理する。[04_ヒューマンエラーと安全工学](./04_ヒューマンエラーと安全工学.md) がエラー発生メカニズムを扱うのに対し、本章は**発生後の組織的対応と知識の更新**を扱う。

## 2. CAPA (是正処置・予防処置)

### 2.1 CAPA の定義と規制上の位置づけ

**CAPA (Corrective Action and Preventive Action)** は品質マネジメントシステムの根幹要素:

- **是正処置 (Corrective Action, CA)**: すでに発生した不適合の原因を除去し再発を防ぐ処置
- **予防処置 (Preventive Action, PA)**: 潜在的な不適合の原因を事前に除去する処置

主要規制における CAPA 要求:

| 規制 | 条項 | 内容 |
|---|---|---|
| ISO 9001:2015 | 10.2 | 不適合・是正処置の管理 |
| ISO 13485:2016 | 8.5.2/8.5.3 | 医療機器の CAPA 要求 |
| FDA QSR (21 CFR Part 820) | Subpart J | 是正・予防処置手順の文書化 |
| IATF 16949:2016 | 10.2 | 不適合情報の特定と分析 |

### 2.2 CAPA プロセスの構造

```
CAPA プロセスのフロー:

不適合の発見 (Detection)
    ↓
封じ込め (Containment)     ← 問題が広がらないよう即時隔離
    ↓
根本原因分析 (Root Cause Analysis)
    ↓
是正処置の計画 (Action Plan)  ← 誰が・何を・いつまでに
    ↓
是正処置の実施 (Implementation)
    ↓
有効性確認 (Effectiveness Check)  ← 再発していないかを確認
    ↓
CAPA 閉鎖 (Closure)
```

有効性確認 (Effectiveness Check) が省略されると「やったことにする」だけで実際に再発防止できていない状態が続く。規制当局の査察では有効性確認の記録が重点的に確認される。

## 3. 8D レポート (Eight Disciplines Problem Solving)

### 3.1 8D の起源

**8D (Eight Disciplines)** は Ford Motor Company が 1980 年代に開発したチームベースの問題解決手法。現在は自動車業界を中心に広く普及し、**VDA** (Verband der Automobilindustrie) の品質要求にも組み込まれている。

### 3.2 8 ステップの内容

| D# | ステップ | 内容 |
|---|---|---|
| D0 | 準備 | 問題の緊急性評価、緊急封じ込め処置の要否確認 |
| D1 | チーム編成 | 問題解決チームの結成 (知識・スキルの多様性確保) |
| D2 | 問題記述 | 問題を定量的・具体的に記述 (IS/IS NOT 分析) |
| D3 | 暫定処置 | 顧客影響を即時封じ込める緊急対策 |
| D4 | 根本原因特定 | Why-Why 分析・Ishikawa 等で真の原因を特定 |
| D5 | 恒久対策の選択 | 根本原因を除去する対策の立案・選定 |
| D6 | 恒久対策の実施 | 対策の実施と実施証拠の記録 |
| D7 | 再発防止 | 有効性確認、类似工程・製品への水平展開 |
| D8 | チームへの感謝 | 活動記録のアーカイブ、ナレッジ共有 |

**D7 の水平展開 (Yokoten)** は TPS 用語でも重要視される概念であり、1 工程で発見した問題の解決策を類似工程・他製品・他拠点へ積極的に展開することを指す。

## 4. 根本原因分析手法

### 4.1 Why-Why 分析 (5 Whys)

**豊田佐吉** に由来するとされ、大野耐一が TPS に組み込んだ。事象に対して「なぜ (Why)?」を繰り返すことで表層原因から根本原因へ掘り下げる:

```
問題: ボルトの締め付けトルクが規格外だった

Why1: トルクレンチが規定値を示さなかった
Why2: トルクレンチが校正切れだった
Why3: 校正期限の管理ルールがなかった
Why4: 測定機器の台帳に校正期限欄がなかった
Why5: 台帳作成時に校正管理の要求事項が考慮されていなかった

根本原因: 測定機器の台帳設計時のレビュープロセスの不備
→ 是正処置: 台帳設計に計量管理担当者のレビューを義務付ける
```

一般に「Why を 5 回繰り返す」とされるが、問題の性質によって回数は変わる。重要なのは **回数より、根本原因に到達しているかの判断** である。

### 4.2 Ishikawa 特性要因図 (Fishbone Diagram)

**石川馨** が 1960 年代に開発。特性 (結果) に対して原因を 4M (Man, Machine, Material, Method) または 5M1E 等の大骨に整理する:

```
原材料   設備
  \       /
   \     /
    \   /
     ↘ ↙
      ← 不適合の発生 ←
     ↗ ↖
    /   \
   /     \
  /       \
作業者   方法

(Man / Machine / Material / Method + Measurement / Environment)
```

複数の潜在原因を網羅的に整理するのに適するが、各要因間の因果関係の強さは示さない。Why-Why と組み合わせて使うことが多い。

### 4.3 FTA (Fault Tree Analysis)

[10_信頼性工学と保全活動](./10_信頼性工学と保全活動.md) で詳述した FTA は、トップ事象 (不適合) から演繹的に原因を AND/OR ゲートで分解する。Ishikawa が帰納的 (原因の列挙) なのに対し、FTA は演繹的 (原因の論理的構造化) である。

## 5. 4M 変更管理

### 5.1 変更管理の重要性

製造現場で品質問題が発生する原因の大部分は **何らかの変更** に起因する:

- Man (人): 新人配置、担当者交代、残業による疲労
- Machine (設備): 修理・改造・老朽化、校正
- Material (材料): ロット変更、サプライヤー変更、仕様変更
- Method (方法): 手順変更、工具変更、作業条件変更

**4M 変更管理** は、これらの変更を事前に計画・承認し、変更前後での品質確認を行う管理活動である。

### 5.2 変更点監査と手順書連携

4M の変更が承認された場合、対応する手順書の改訂が必要になるケースが多い。変更管理と手順書の版管理を連携させる仕組み:

```
4M 変更申請
    ↓
品質/技術レビュー
    ↓
手順書改訂要否の判定
    ↓ (改訂要の場合)
手順書改訂 (ドラフト → レビュー → 承認)
    ↓
関連作業者への教育・認定
    ↓
変更後品質確認期間 (初回 N ロット検査強化)
    ↓
変更承認クローズ
```

ナビゲーションアプリでは「特定のロット以降は新版手順を適用」というロット連動の版管理が理想的である。

## 6. 組織学習理論

### 6.1 Argyris & Schön の Single-Loop / Double-Loop 学習

**Chris Argyris & Donald Schön** は *Organizational Learning* (1978) で、組織が問題からどのように学ぶかを 2 層モデルで説明した:

| ループ | 内容 | 例 |
|---|---|---|
| Single-Loop 学習 | 行動を変えて問題を修正。目標・価値観は変えない | トルクレンチが原因 → 締め付け手順を変更 |
| Double-Loop 学習 | 目標・前提・価値観そのものを問い直す | 「校正管理の文化がない」という根本認識の変革 |
| Deutero-Learning | 「学ぶ方法を学ぶ」メタ学習 | CAPA プロセス自体の改善 |

製造現場での CAPA の多くは Single-Loop にとどまる。同じ種類の不適合が繰り返される場合は、Double-Loop 学習が必要なサインである。

### 6.2 Senge の学習する組織

**Peter Senge** は *The Fifth Discipline* (1990) で **学習する組織 (Learning Organization)** を提唱し、5 つのディシプリン (自己マスタリー・メンタルモデル・共有ビジョン・チーム学習・**システム思考**) を組み合わせて組織の学習能力を高めることを論じた。

システム思考の観点では、不適合と手順改訂のフィードバックループは **遅延** (Delay) が問題になりやすい:

```
不適合発生 → CAPA 開始 → 根本原因特定 → 手順改訂 → 教育 → 再発防止確認

このループの遅延が大きいほど、同一原因の不適合が間にもう一度発生するリスクが高まる
```

## 7. ナレッジ更新ループとシステム設計

### 7.1 SECI モデルとの接続

[05_暗黙知と技能伝承](./05_暗黙知と技能伝承.md) で述べた Nonaka & Takeuchi の **SECI モデル** は、不適合フィードバックループにも適用できる:

| フェーズ | SECI における位置 | 内容 |
|---|---|---|
| 不適合発生時の熟練者の感知 | 共同化 (Socialization) | 熟練者の「何かおかしい」という暗黙知 |
| Why-Why 分析・根本原因の言語化 | 表出化 (Externalization) | 暗黙知を形式知へ変換 |
| 8D レポート・手順書への反映 | 連結化 (Combination) | 形式知の体系化 |
| 新手順での実際の作業と体得 | 内面化 (Internalization) | 作業者が新手順を習慣化 |

### 7.2 Lesson Learned データベース

不適合の根本原因・対策・有効性確認の結果は **Lesson Learned (教訓) データベース** として蓄積し、類似問題への横断的参照を可能にする。有効な管理要素:

- 問題カテゴリのタグ付け (工程・材料・設備・手順書 etc.)
- 根本原因タイプの分類 (ヒューマンエラー・設備・材料 etc.)
- 対策効果の評価記録 (有効性確認の結果)
- 検索可能なキーワードと全文検索

### 7.3 手順書改訂のトリガー管理

システム設計上、手順書改訂が必要な条件を自動的に検出・通知できると運用コストを下げられる:

```
自動トリガーの候補:
- 特定工程での不適合発生率が閾値を超えた
- 同一ステップでの作業者の「引っかかり」記録が多い
- 4M 変更承認が完了した
- 手順書の定期レビュー期限が到来した
```

## 8. 批判と限界

### 8.1 Double-Loop 学習の実装困難性

Argyris の理論は組織変革論として強力だが、実際に Double-Loop 学習を定着させることは極めて難しい。防衛的ルーティン (Defensive Routines) — 「問題の指摘は非難と受け取られる」「根本原因が経営層の意思決定にある場合は誰も指摘できない」 — が組織内に根付くと、CAPA は表面的な是正にとどまる。

### 8.2 改訂頻度過剰による手順書疲労

CAPA ループが機能しすぎると、手順書が頻繁に改訂され、作業者が「また変わった」と手順書を信用しなくなる現象が生じる。[19_電子チェックリストと手順遵守の科学](./19_電子チェックリストと手順遵守の科学.md) で述べた **Procedural Drift** の逆説として、過剰改訂も手順遵守を阻害する。

改訂は「小さな改善を随時」より「十分な検討を経た重要な変更のみ」にまとめ、改訂サイクルを過度に短くしないことが実務的に有効である。

### 8.3 CAPA 記録の負荷

規制業界で完全な CAPA 記録を維持するコストは高く、特に中小規模の製造現場では「記録のための記録」が目的化するリスクがある。Lean 観点からは、記録そのものの附加価値を問うべきであり、**記録が監査対応のためだけでなく実際の改善活動に活用されているか** を定期的に評価する必要がある。
