# オフライン同期とデータ整合性

## 1. 問題の所在

作業ナビゲーションアプリはタブレット端末で現場作業者が利用し、完了ステップや測定値をリアルタイムで記録する。本アプリは社内 LAN を前提とするが、それでも以下の状況が発生する:

- 現場の電波死角 (棚の裏、金属製設備の影) で一時的に接続が切れる
- 端末の再起動・バッテリー切れ中に作業が進む
- サーバ側の再起動・メンテナンス中に作業を続けたい
- 複数端末から同一ロットに同時アクセスする場合

これらの状況でトレサビ記録がロスすると **ALCOA+ の Contemporaneous (同時性) 原則に違反** し、品質・規制対応上の問題となる ([22_規制別トレーサビリティ要件詳論](./22_規制別トレーサビリティ要件詳論.md) 参照)。本章は分散システム理論を基礎に、オフライン動作と整合性確保の設計論を整理する。

## 2. CAP 定理と一貫性モデル

### 2.1 CAP 定理

**Eric Brewer** は 2000 年の基調講演で **CAP 定理** を提唱し、後に **Gilbert & Lynch (2002)** が形式的に証明した:

```
分散システムは以下の 3 つの性質を同時に満たすことはできない:

C — Consistency   (強一貫性): 全ノードが同じ最新データを返す
A — Availability  (可用性)  : すべてのリクエストが応答を返す
P — Partition tolerance (分断耐性): ネットワーク分断があっても動作を続ける
```

社内 LAN でも `P` (ネットワーク分断) は実際に発生するため、設計者は `C` と `A` のどちらを優先するかを選択しなければならない:

- **CP 系** (例: HBase, Zookeeper): 分断時はサービス停止、結果の正確性を優先
- **AP 系** (例: Cassandra, Dynamo): 分断時も継続動作、整合性は後から収束

作業ナビゲーションアプリは**現場での作業継続が最優先**であるため、**AP 志向 (オフライン優先 + 後からの同期)** が適切な設計方針となる。

### 2.2 一貫性モデルの種類

| モデル | 定義 | 用途例 |
|---|---|---|
| Strong Consistency | 書き込み後の全読み取りが最新値を返す | 銀行残高・在庫数 |
| Sequential Consistency | 全ノードが同じ順序で操作を観測 | ゲームの操作ログ |
| Causal Consistency | 因果関係のある操作の順序のみを保証 | メッセージアプリ |
| Eventual Consistency | 一定時間後に全ノードが収束 | DNS, Shopping Cart |
| Read-Your-Writes | 自分が書いた内容は次の読み取りで見える | SNS 投稿 |

作業ログの記録においては「**自分が記録したステップは次の確認で必ず表示される (Read-Your-Writes)**」と「**最終的にサーバに同期される (Eventual)**」の組み合わせが現実的な目標となる。

## 3. Vogels の Eventual Consistency

**Werner Vogels (Amazon CTO)** は 2008 年の論文で **Eventual Consistency** を詳説した。Amazon Dynamo の設計哲学に基づき、「更新が伝播されれば最終的に全レプリカが一致する (収束する)」モデルを定義した。

重要な付加概念:

- **Monotonic Read Consistency**: 一度読んだデータより古いデータを二度と返さない
- **Write Follows Read**: 読み取り後の書き込みは、読んだ版以降の版に対して行われる
- **Bounded Staleness**: 古さの上限時間を保証 (例: 「最大 30 秒遅れ」)

作業ナビゲーションにおけるローカルキャッシュは少なくとも **Monotonic Read** を保証すべきであり、「作業ステップが突然消える」現象を防ぐ。

## 4. CRDT (Conflict-free Replicated Data Type)

### 4.1 CRDT とは

**Shapiro et al. (2011)** が提唱した **CRDT (Conflict-free Replicated Data Type)** は、複数ノードが独立して更新しても、後から合成すると矛盾なく正しい結果が得られるデータ型の設計理論である。

```
CRDT の条件:
  1. 操作が可換 (commutative): 操作の順序が変わっても結果が同じ
  2. 操作が結合的 (associative): 順序に依存しない
  3. 操作が冪等 (idempotent): 同じ操作を何度行っても同じ結果

→ ネットワーク分断後に再接続して操作を交換・再適用しても整合性が保たれる
```

### 4.2 主な CRDT 型

| CRDT 型 | 意味 | 製造現場での用途例 |
|---|---|---|
| G-Counter | 増加専用カウンタ | 完了ステップ数の集計 |
| PN-Counter | 増減可能カウンタ | 在庫数 (要注意: 製造への直接応用は慎重に) |
| G-Set | 要素追加専用集合 | 完了済みステップ ID の集合 |
| 2P-Set | 追加・削除可能集合 | 作業中フラグの管理 |
| OR-Set | 削除競合を解決する集合 | 複数端末からの登録 |
| LWW-Register | Last-Write-Wins 単一値 | 測定値・ステップ結果 (上書き型) |
| RGA / Logoot | 順序付き文字列 | 自由記述フィールド |

**LWW-Register** は実装が単純だが「最後に書いた側が勝つ」という性質上、同時書き込みで情報がロスするリスクがある。作業ログのような **追記のみのデータ** には **G-Set (追記専用)** が最も適合する。

### 4.3 作業ログへの CRDT 適用

```
作業完了ログの G-Set モデル (概念例):

ローカル端末 A での操作:
  add("step-001:completed:2026-05-16T09:10:00")
  add("step-002:completed:2026-05-16T09:12:30")

オフライン中にローカル端末 A で実施した操作:
  add("step-003:completed:2026-05-16T09:15:00")

再接続後のマージ (サーバ側に追加するだけ):
  merge = union(server_set, local_set)
  → 矛盾なく全ステップが揃う
```

追記専用 (Append-only) のログは CRDT として自然に実装可能であり、作業ナビゲーションのトレサビ記録に適している。

## 5. Offline-First アーキテクチャ

### 5.1 Local-First Software

**Kleppmann et al. (2019)** の論文 "Local-First Software" は、クラウド依存のアーキテクチャに対し、**データの一次ソースをローカルデバイス** に置くことを提唱した。主な性質:

```
Local-First の 7 つの理想:
1. 高速: ローカルデータへのアクセスは即座
2. 複数デバイス対応: 端末間で同期
3. オフライン動作: ネットワーク不要で完全動作
4. コラボレーション: リアルタイム共同編集
5. 永続性: ベンダー撤退でもデータが残る
6. セキュリティ/プライバシー: 暗号化されたローカルデータ
7. ユーザーコントロール: データのエクスポート・移行が可能
```

製造現場の本アプリは 1〜3 が特に重要であり、5 (永続性) は社内オンプレで担保される。

### 5.2 React Native でのオフライン実装

| 技術 | 役割 | 例 |
|---|---|---|
| AsyncStorage / SQLite (expo-sqlite) | 端末ローカルのデータ永続化 | 未送信ログのキュー |
| Network State API | オンライン/オフライン状態の検出 | `@react-native-community/netinfo` |
| Background Sync | バックグラウンドでのサーバ同期 | オフライン→復帰時の一括送信 |
| Optimistic UI | ローカル即時反映、後からサーバ確定 | ステップ完了ボタンの即時グレーアウト |

**Optimistic UI** は「ユーザーには即時反映→バックグラウンドでサーバ送信→失敗時はロールバックまたは再試行」のパターンで、LAN 遅延下の体感速度を向上させる。

## 6. Conflict Resolution の業務的解釈

### 6.1 LWW (Last-Write-Wins) の問題

LWW は実装が単純だが、**同時更新で一方の操作が無音でロスする**。製造トレサビにおいて「測定値の上書き」は特に問題となる:

- 検査者 A が「合格」と記録
- 検査者 B が同時に「再確認中」と記録
- サーバで後着の LWW → 一方の記録が消える

### 6.2 Append-only Event Sourcing

競合を根本的に避けるアプローチは **Append-only のイベントログ**:

```
すべての状態変化を「イベント」として追記する。
上書きは存在しない。新しい状態は新しいイベントとして追記。

イベント列 (例):
  1. step-003 COMPLETED by user-42 at 09:15:00
  2. step-003 FLAGGED_FOR_REVIEW by user-71 at 09:16:30
  3. step-003 CONFIRMED_OK by user-09 at 09:18:00

現在状態は全イベントのリプレイで導出する。
```

これはトレサビ要件 (ALCOA+ の Attributable + Original) と相性が良い。誰が何時に何をしたかのイベントは消せないため、改ざん困難性も担保される。

### 6.3 Idempotency Key (冪等性キー)

同一操作が複数回送信される (重複送信) を防ぐパターン。端末でユニークな操作 ID (UUID) を生成し、サーバは同一 ID の操作を 1 回のみ処理する:

```
POST /steps/complete
{
  "idempotency_key": "uuid-a1b2c3d4-...",
  "step_id": "step-003",
  "worker_id": "user-42",
  "timestamp": "2026-05-16T09:15:00Z",
  "result": "ok"
}

→ サーバが同じ idempotency_key を受け取った場合は 200 OK を返すが再処理しない
```

ネットワーク不安定でリトライが発生しても、記録が重複しない。

## 7. 本アプリへの実装上の選択肢

| 設計要素 | 推奨アプローチ | 理由 |
|---|---|---|
| ローカル永続化 | SQLite (expo-sqlite) | Expo 標準で安定、offline-first 向け |
| 同期タイミング | 操作ごとに即送信 + オフライン時はキュー | 最小遅延でサーバに反映 |
| 操作の型 | Append-only (イベントソーシング) | 上書き競合を回避、トレサビに適合 |
| 重複排除 | Idempotency Key (UUID) | リトライ安全性を保証 |
| 現在状態の導出 | イベントのリプレイ or サーバサイドのマテリアライズドビュー | 端末は raw events、表示は server-side aggregation |
| 競合検出 | Vector Clock (Lamport Clock) or タイムスタンプ比較 | サーバが複数端末からの操作順序を判定 |

## 8. 批判と限界

### 8.1 CRDT の表現力限界

CRDT はデータ型ごとに意味的な制約がある。例えば OR-Set での削除は「追加に対してキャンセル」するが、「まだ追加されていないものを削除する」操作は表現できない。製造手順の「ステップの追加・削除・順序変更」を CRDT で表現しようとすると、モデルが急速に複雑化する。

### 8.2 Eventual Consistency とトレサビの緊張

ALCOA+ の **Contemporaneous (同時性)** 原則は「作業と同時に記録すること」を求める。Eventual Consistency 下では「ローカルには記録されているがサーバにはまだ届いていない」状態が生じる。この状態での監査・照会はサーバの状態を見るため、「記録されていない」と誤判断されるリスクがある。

対策: ローカルキューに残っている操作も「記録済み (未同期)」として監査画面に表示し、同期完了後に確定表示に変える UI 設計が必要。

### 8.3 LAN 特有の楽観

社内 LAN は一般的に信頼性が高く、「オフライン設計は不要では」という判断が生じやすい。しかし工場内の金属構造物は電波遮蔽が強く、Wi-Fi デッドゾーンが意外に多い。「LAN があるから常時接続前提」という設計判断は現場での実態確認後に下すべきである。
