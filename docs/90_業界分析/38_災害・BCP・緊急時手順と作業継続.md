# 災害・BCP・緊急時手順と作業継続

## 1. 事業継続管理 (BCM) の基盤

**事業継続管理 (BCM: Business Continuity Management)** は、組織が重大な事業中断リスクを識別・評価し、中断が発生した際に許容される時間・品質水準で事業を継続または迅速に回復できるようにする総合的な管理活動である。BCM の国際標準として **ISO 22301:2019 (Societal security – Business continuity management systems)** が確立しており、PDCA サイクル (計画 Plan → 実施 Do → 評価 Check → 改善 Act) に基づくマネジメントシステムとして構造化されている。

BCM の中核成果物は **BCP (Business Continuity Plan: 事業継続計画)** であり、緊急事態発生時に「誰が・何を・どの順序で」実行するかを事前に規定した行動計画書である。BCP の策定に先立ち実施される **BIA (Business Impact Analysis: 事業影響分析)** は、中断が組織に及ぼす財務的・業務的・評判的影響を時間軸に沿って定量化し、保護すべき重要プロセスの優先順位付けを行う。BIA の主要アウトプットは **RTO (Recovery Time Objective: 目標復旧時間)** と **RPO (Recovery Point Objective: 目標復旧時点)** であり、これらは IT システム復旧要件の仕様策定において特に重要な指標となる。

製造業への ISO 22301 適用においては、単なる IT システムの BCP に留まらず、生産設備・原材料在庫・作業人員・電力・工業用水等の物理的リソースを含む包括的な継続性確保が求められる点が他業種と大きく異なる。生産ライン停止 1 日あたりの損失額が億円規模に達する自動車組立工場のような環境では、RTO を数時間単位で設定することも現実的な要件となる (Herbane, 2010)。

| BCM 主要概念 | 定義 | 製造業での具体例 |
|---|---|---|
| RTO (目標復旧時間) | 中断発生からシステム・機能を回復するまでの最大許容時間 | 生産管理システム: 4 時間以内 / 作業ナビアプリ: 1 時間以内 |
| RPO (目標復旧時点) | 障害発生時に許容されるデータ損失の最大範囲 (時間単位) | 作業記録: 15 分前までのデータは保証 |
| MTPD (最大許容中断期間) | 事業への致命的影響を回避できる最大中断時間 | 重要工程: 24 時間 |
| BIA (事業影響分析) | 中断の影響を財務・業務・評判別に定量化する分析 | 生産停止による売上損失・顧客ペナルティの試算 |
| Tier 1 / Tier 2 サプライヤー | 重要度に応じた供給者の優先分類 | 単一ソース品の代替供給先特定 |

## 2. 製造業特有の災害リスク

製造業が直面する災害リスクは多岐にわたるが、日本の製造業固有の重要リスクとして**地震**は筆頭に位置する。**2011 年東日本大震災**は製造業のサプライチェーンへの壊滅的影響という観点で歴史的転換点となった。自動車・電子機器産業において、東北地域の一次サプライヤーの被災が世界的なサプライチェーン途絶を引き起こし、Japanese Single-Source 問題が国際的に認知された (Sheffi & Rice, 2005 の理論枠組みに照らして多くの研究が事後分析を行った)。製造工場自体への地震直撃リスクに加え、停電・交通遮断・通信途絶という二次的影響への備えが不可欠である (17 章と接続)。

**火災・爆発リスク**は可燃性材料・溶剤・圧縮ガスを扱う製造現場では常在する危険であり、防爆設備 (35 章と接続) および自動消火システムとの連動が重要である。火災発生時の緊急停止手順・避難誘導と、その後の設備損傷評価・業務継続の流れを事前に計画しておく必要がある。

**水害・台風・浸水**については、近年の気候変動による激甚化が顕著であり、2019 年の台風 19 号では複数の製造拠点が浸水被害を受けた。内閣府「令和元年台風第 19 号に係る被害状況等について」は製造業への深刻な影響を記録している。一階建てプレハブ工場・地下電気室・屋外変電設備等の浸水脆弱性は、BIA において重点的に評価すべき項目である。

**OT (Operational Technology) セキュリティへのサイバー攻撃**は近年急増しており、製造業の ICS (Industrial Control System) を標的とした攻撃事例が ICS-CERT (米国 CISA の前身) により多数報告されている。2017 年の NotPetya マルウェアは製造業大手の生産ラインを停止させ、数十億ドル規模の損失をもたらした (Mondelez International 対 Zurich Insurance 訴訟が保険補償範囲をめぐる争点となった)。作業ナビゲーションアプリを含む生産現場の IT システムも潜在的な攻撃経路となりうる。

**パンデミック**については、COVID-19 (2020–2022 年) が製造業に多面的な影響をもたらした。工場停止・稼働制限・要員不足・サプライヤー途絶という複合的課題が顕在化し、「製造業のパンデミック BCP」が喫緊の課題として認識された。感染症対策 BCP では空間的分散・勤務形態変更 (在宅勤務の製造現場への限界) ・要員の相互乗り入れ訓練 (多能工化, 26 章と接続) が主要な対策として浮上した。

## 3. 緊急停止手順と安全確保

製造設備の緊急停止に係る安全設計は機能安全規格によって体系化されている。**IEC 62061:2021 (機械の安全性 — 安全関連制御システムの機能安全)** は機械の安全関連電気制御システムの設計・実装・検証の要求事項を定め、SIL (Safety Integrity Level) による信頼性評価を規定する。**ISO 13849:2015 (機械の安全性 — 制御システムの安全関連部分)** はカテゴリー・PL (Performance Level) 概念によって制御システムの安全性を評価するものであり、両規格は補完的に適用される。

**E-stop (緊急停止スイッチ)** の設計要件 (ISO 13850:2015) は、機械的な力 (押す・引く) で確実に動作すること、明確な色表示 (赤/黄背景)、セルフロック機能、意図しない解除防止機構を要求する。作業員が緊急時に迅速に操作できる位置への設置と、操作後の安全確認手順が重要である。作業ナビゲーションアプリは E-stop 操作とは独立した (ソフトウェア的な緊急停止ボタンが物理的 E-stop の代替となってはならない) が、緊急事態発生時の記録・報告・手順案内の起点となりうる。

避難・点呼の運用については **JIS Q 45001:2018 (ISO 45001:2018 の国内版)** の 8.2 節「緊急時への準備及び対応」が要求事項を定める。具体的には緊急事態の計画立案・手順の文書化・定期的な演習実施・避難誘導担当の指定・傷者救護手順・火災・化学物質漏洩等のシナリオ別対応計画が含まれる。点呼については全員安否確認が第一優先であり、人員確認が完了するまで誰も危険区域に立ち入らないルール (Accountability System) の徹底が重要である。

| 緊急停止・避難の主要規格 | 規格番号 | 要求の要点 |
|---|---|---|
| 機能安全 (電気制御) | IEC 62061:2021 | SIL 1–3 評価、診断カバレッジ要求 |
| 機能安全 (制御システム) | ISO 13849-1:2015 | PLa–PLe 評価、カテゴリー B–4 |
| 緊急停止 | ISO 13850:2015 | E-stop の設計・色・操作要件 |
| 安全マネジメント | JIS Q 45001:2018 | 緊急事態対応計画の文書化・演習 |
| リスクアセスメント | ISO 12100:2010 | 残留リスク低減の 3 段階 |

## 4. Shared Mental Model と緊急時コミュニケーション

緊急時の組織対応において **Shared Mental Model (SMM: 共有メンタルモデル)** の構築は決定的に重要である。Mathieu et al. (2000) は SMM を「チームメンバーが保有する課題・設備・状況についての共有された知識構造」と定義し、SMM の質がチームパフォーマンスと正の相関を持つことを実証した。通常時から緊急時対応手順・役割分担・緊急連絡経路を全員が共有しておく「事前訓練によるメンタルモデル共有」が緊急時の円滑な対応を支える。

**ICS (Incident Command System: 事故指揮システム)** は 1970 年代の米国で山火事対応組織として発展し、FEMA・DHS の標準指揮体制として確立したものである。単一指揮官 (Incident Commander)・機能部署 (Operations / Planning / Logistics / Finance)・スパン・オブ・コントロール (1 人の管理者が直接管理する人数: 3–7 人が原則) という構造的特徴を持ち、複数機関・部門が協働する大規模緊急事態への適用に優れる。製造工場での応用においては、工場長を IC として、生産・設備・安全・総務・外部連絡の各機能を対応させる形が採られることが多い。

コミュニケーション確認技術として、医療現場由来の **SBAR (Situation-Background-Assessment-Recommendation)** が製造緊急対応にも有効とされる (Leonard et al., 2004)。S (状況): 「現在何が起きているか」、B (背景): 「経緯・文脈」、A (評価): 「原因の判断」、R (推奨): 「必要なアクション」の順に情報を整理することで、緊急時の情報伝達を構造化し誤解を防ぐ。無線・インカムによる通信では、NATO フォネティックアルファベット (Alpha, Bravo, Charlie…) の使用と「確認復唱 (Read-back)」が誤伝達防止に有効である。

デジタル通信への移行については、製造現場では従来のアナログ業務用無線 (DECT 規格等) からデジタル IP 無線への移行が進んでいる。IP 無線はネットワークへの依存を生むため、基幹ネットワーク停止時の通信途絶リスクを BCP で考慮する必要がある。バックアップとしてのアナログ無線・衛星電話・固定電話の維持が推奨される。

## 5. 引き継ぎ・バトンタッチの情報保全

**Handoff (引き継ぎ)** は情報転送と責任移管が同時に行われる重要な業務境界点であり、Patterson et al. (2004) は医療環境での Handoff 研究において「情報の喪失・歪曲・未伝達」が患者有害事象の主要な先行要因であることを示した。製造現場のシフト交代時の引き継ぎは、進行中の作業状態・異常の有無・設備コンディション・重要な判断保留事項等の情報を確実に次番担当者へ伝達する機能を持つが、時間的プレッシャー・疲労・「口頭のみ」の慣行により情報喪失が生じやすい。

**I-PASS プロトコル** (Starmer et al., 2014) は患者引き継ぎの精度向上のために開発されたものであるが、その構造は製造現場の引き継ぎにも転用可能である: I (Illness Severity / 重要度): 当該作業・設備の重要度・緊急度 → P (Patient Summary / 作業状態要約): 進行中作業の現状・達成度 → A (Action List / アクションリスト): 次番で対応すべき未完了事項 → S (Situation Awareness / 状況認識): 予見されるリスク・注意事項 → S (Synthesis by Receiver / 受領者の確認): 受領者が復唱・確認。

作業ナビゲーションアプリにおける**引き継ぎ状態の保存**設計は、上記プロトコルをデジタル化する形で実現できる。特定の作業インスタンスに「引き継ぎメモ」フィールドを設け、シフト交代時に自動的に次番担当者のダッシュボードに引き継ぎ待ちの作業が通知される仕組みが有効である。これにより口頭引き継ぎへの依存を低減し、情報漏れを防ぐ。

| 引き継ぎプロトコル | 起源 | 製造現場への転用 |
|---|---|---|
| SBAR | 医療 (Kaiser Permanente) | 緊急時・異常発生時の状況報告構造化 |
| I-PASS | 小児科研修 (Harvard/Starmer) | シフト交代時の作業状態引き継ぎ |
| ISOBAR | 看護 (オーストラリア) | 設備異常・進行中修理の引き継ぎ |
| T-SBAR | 外傷外科 | 多発設備異常の優先順位付き引き継ぎ |

## 6. 災害時のオフライン運用

作業ナビゲーションアプリのサーバ停止時における**オフライン動作**は BCP の観点から設計時に必ず検討すべき要件である。27 章 (オフライン同期とデータ整合性) が詳述するオフライン同期アーキテクチャは、BCP の文脈ではさらに強化された要件を持つ。すなわち「同期が遅れるだけ」ではなく「サーバが完全停止しても作業を継続できること」が求められる。

**Emergency Mode (緊急動作モード)** として、サーバ接続を失った際に自動的に切り替わる最小機能セットを定義することが有効である。緊急モードでは: (1) 直近でキャッシュされた作業手順の参照、(2) 作業記録のローカルストレージへの保存 (復旧後に自動同期)、(3) 緊急連絡先・避難手順への即時アクセス、(4) 最終同期時刻の常時表示。逆に、緊急モードでは不必要な機能 (管理者ダッシュボード・設定変更・マスタ更新) を非表示として UI を簡素化することで作業員が迷わず必要機能を使用できる状態を維持する。

電源喪失シナリオについては、Android タブレット・Windows タブレットのバッテリー容量 (一般的に 4–8 時間) を考慮した設計が必要である。工場内停電発生時に全端末が同時に充電不能となる事態を考慮し、UPS (無停電電源装置) による充電ステーション保護、または交代充電による端末稼働時間延伸を BCP に組み込むことが推奨される。

紙バックアップ vs. オフライン端末動作の選択については、2021 年以前は多くの工場で「電子化のバックアップは紙」という慣行が主流であった。しかし紙帳票のメンテナンスコスト・紙版と電子版の乖離リスク・緊急時の帳票配布の煩雑さを考慮すると、オフライン端末動作を主とし、紙は「端末も使用不能な場合の最終手段」と位置づける設計が現実的である。

## 7. 復旧手順と電子記録

設備復旧手順書の電子化は、紙媒体に比べて最新版管理・参照容易性・多言語展開において優れるが、緊急時の電子記録の信頼性確保には特有の課題がある。**ALCOA+ 原則 (Attributable・Legible・Contemporaneous・Original・Accurate + Complete・Consistent・Enduring・Available)** は FDA が定める GxP 文書の品質基準であり (22 章と接続)、「Contemporaneous (同時性: 記録は事実発生と同時またはそれに近い時点で作成されること)」は緊急事態下では達成が困難になる (27 章と接続)。電源喪失・通信途絶・要員不足の状況下では「後からまとめて記録する」という実態が生じやすく、記録の信憑性が低下する。この問題に対処するため、緊急時でも手順ステップごとのワンタップ確認 (入力ハードルを最低限に下げる) を維持するオフライン設計が重要である。

被害状況の記録については、損傷設備・被害範囲・被害量を緊急時に素早く記録できる機能 (写真撮影・音声メモ・定型チェックリスト) をアプリが提供することで、その後の損害保険請求・再調達計画・行政への被害報告に活用できる。これは通常業務の作業記録機能の延長として設計可能である。

**CAPA サイクル (28 章と接続)** への教訓反映は BCM サイクルの「改善 (Act)」フェーズに相当する。緊急事態の対応後には Debrief (振り返り) を実施し、BCP の改善事項・訓練課題・設備投資計画として文書化・反映する。作業ナビアプリのインシデント記録機能が、このデブリーフィングの情報源として機能するよう設計すると、CAPA との統合運用が可能となる。

## 8. Tabletop Exercise (机上演習)

**Tabletop Exercise (TTX: 机上演習)** は、実際の緊急事態を模擬したシナリオを参加者が「卓上で議論・検討する」形式の演習であり、実動訓練と比べてコスト・時間・安全リスクが低い反面、実際の行動・判断を伴わないため習熟度向上の限界もある。FEMA の **HSEEP (Homeland Security Exercise and Evaluation Program)** フレームワークは TTX の設計・実施・評価の国際的標準として広く参照されており、シナリオ設計・ファシリテーション手法・アフターアクションレポート (AAR) 作成の方法論を提供する。

製造業向け TTX のシナリオ例として: (1) **地震シナリオ**: 震度 6 強発生・停電・通信途絶・複数設備損傷・複数名の負傷者発生という複合事態への組織対応の確認、(2) **停電・長時間停電シナリオ**: 生産計画の緊急見直し・冷凍・真空設備の保全・在庫への影響・顧客への連絡対応、(3) **サイバー攻撃シナリオ**: 製造ラインの制御システムへのランサムウェア感染・作業ナビアプリサーバの停止・オフラインモードへの切替対応・CSIRT 連携の確認が挙げられる。

TTX の運営においては、参加者がシナリオに「没入」できる進行 (Facilitator によるインジェクト投入で状況を時系列に展開) と、意思決定プロセスの記録 (誰が・何を・いつ・なぜ判断したか) が重要である。TTX 後の **ディブリーフィング** では「うまくできた点 (Hot Wash)」「改善すべき点 (Improvement Points)」「計画の修正要否」を整理して AAR に記録し、BCP・手順書の更新につなげる。

日本の中小製造業における BCP の整備状況は、2022 年の中小企業白書によれば BCP を「策定済み」と回答した中小企業は全体の 17% に過ぎず (製造業でも同様の傾向)、TTX の実施率はさらに低い。大企業・上場企業では取引先からの BCP 整備要求や ISO 22301 認証取得のプレッシャーがあるが、中小企業においては人的・財務的リソースの制約が BCM 推進の最大の障壁となっている。

| TTX シナリオ種別 | 主なテスト対象 | 参加推奨者 |
|---|---|---|
| 地震・複合災害 | 避難・点呼・設備停止・顧客連絡 | 工場長・生産・設備・安全・総務 |
| 停電・長時間停電 | オフライン運用・冷凍設備・バッテリー | IT・設備・品質・生産管理 |
| サイバー攻撃 | OT/IT 分離・CSIRT 連携・オフライン切替 | IT・情報セキュリティ・生産管理 |
| パンデミック | 要員不足・多能工展開・在宅化限界 | HR・生産・多能工担当 |
| サプライヤー途絶 | 代替調達・在庫管理・生産計画変更 | 調達・生産管理・営業 |

## 9. 本アプリへの実装含意

災害・BCP の観点から、作業ナビゲーション & トレサビ記録アプリに以下の設計上の含意が導かれる。

**Emergency Mode の実装**: サーバ接続が失われた時点で自動的にオフライン緊急モードに移行し、「最終同期時刻: XX 分前」をすべての画面に常時表示する。緊急モードへの移行時は作業員の画面に目立つバナーを表示し、オフライン状態であることと「記録はローカル保存中」であることを明示する。緊急連絡先・避難手順マップをトップ画面の固定ボタンからワンタップでアクセスできるようにする。

**引き継ぎ確認フロー**: 緊急事態後の「復旧確認引き継ぎ」を専用のワークフローとして実装する。設備復旧確認チェックリスト (作業ナビの一種として定義)・損傷部位の写真添付・担当者サイン機能を持つ復旧記録テンプレートを BCP フェーズとしてアプリ内に組み込む。このレコードはトレサビデータとして通常の作業記録と同一のストレージに保存し、監査対応・保険請求・CAPA 分析に活用する。

**サーバの地理冗長**: バックエンド (Rust + Axum + PostgreSQL) は同一 LAN 内での二重化 (Active-Standby 構成) を最低限の冗長化として実装し、主サーバ障害時に待機系が自動引き継ぐ構成とする。製造工場内の場合、LAN 内二重化であっても建屋の異なる場所に設置することで、火災・浸水への耐性を高める。ただし本アプリのスコープがオンプレ・社内 LAN であることを踏まえ、クラウドへのオフサイトバックアップはスコープ外として、あくまで LAN 内二重化を主な回答とする。

**RPO/RTO の設計目標**: 作業ナビゲーションアプリとして現実的な RPO・RTO 目標を設定する。RPO: 最大 15 分 (同期失敗の最大許容範囲)、RTO: 1 時間以内 (待機系切替または再起動)。これらの目標値は Docker Compose による迅速な再デプロイ・PostgreSQL の WAL (Write-Ahead Logging) による Point-in-Time Recovery を組み合わせることで技術的に実現可能である。

## 10. 批判と限界

BCM の実践における最大の問題の一つは「**書類整備に終わる形式主義**」リスクである。ISO 22301 認証取得を目的として BCP を作成・更新するが、実際の緊急時には誰も BCP を参照せず、属人的な即興対応に依存するという状況が多くの組織で観察される (Herbane et al., 2004)。特に大規模な BCP 文書は緊急時に「読めない・探せない」という根本的な問題を持っており、「1 枚に収まる緊急時アクションカード」等への要約が実践上重要である。作業ナビアプリが緊急時手順の参照媒体となることで、デジタルでの BCP 参照を容易にする効果は期待できるが、アプリ自体がオフラインになった際の逆説的問題が残る。

**RTO/RPO 目標と実際の復旧能力のギャップ**も深刻な問題である。BIA で設定した RTO が「4 時間」であっても、実際のインシデント対応では被害確認・ベンダ手配・設備調達・手順確認・スタッフ召集に要する時間が想定を大幅に超えることが多い。Herbane (2010) が指摘するように、特に中小企業では RTO 目標が「希望値」に過ぎず、実現のためのリソース・訓練が裏付けられていないケースが多い。RTO/RPO は定期的な演習によって実際の復旧能力と照合し、必要に応じて現実的な目標値に修正する「生きた BCP 管理」が重要である。

TTX の**演技化**問題も見落とせない。管理職が参加する TTX では「日常業務の延長としての当たり障りのない議論」に終わりがちで、実際の緊急時に生じる情報混乱・感情的反応・リソース争奪等のリアリティが再現されない。より実効的な訓練のためには、参加者に「インジェクト (追加情報・状況変化の突発的投入)」によって意思決定ストレスをかけるファシリテーション技術が不可欠であるが、製造業でこのレベルの TTX を設計・運営できる人材は希少である。

中小製造業の**リソース制約**は BCM 推進の構造的障壁である。BCP 策定・維持・演習・更新に必要な人的・時間的・財務的投資は中小企業には重い負担であり、「BCP の専任担当者がいない」「TTX を実施する余裕がない」という現場の声は現実的である。また、製造現場のナビゲーションアプリの BCP を「IT システムの DR (Disaster Recovery)」として IT 部門のみで検討する縦割り問題も、製造プロセスの視点を欠いた不完全な BCP の原因となる。
