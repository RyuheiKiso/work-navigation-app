# 非テキスト記録と証拠品質管理

## 1. 非テキスト記録の類型と製造現場における役割

製造現場における品質記録は、長らく紙帳票や数値データが中心であったが、デジタル化の進展とともに静止画・動画・音声・センサストリームといった非テキスト記録が実務に浸透しつつある。これら非テキスト記録は、テキストや数値では捉えきれない「現象の直接証拠」としての価値を持ち、規制準拠・品質保証・工程改善の各局面で活用されている (Ziegler et al., 2019)。

**静止画 (写真記録)** は、外観検査における合否判定根拠、不良現品の現象記録、受入検査における梱包状態の確認など、最も広く活用される非テキスト記録である。スマートフォン・タブレットの普及によりカメラの高解像度化と低コスト化が進んだことで、現場での即時撮影が現実的となった。不良品写真は「何が」「どのような状態で」発生したかを後続工程や品質担当者に伝える最短経路であり、人的解釈の仲介を最小化する (Hutchins, 1995)。

**動画記録** は、作業手順の動的な流れ・工具の動き・作業者の姿勢を時系列で保存できる点で静止画を補完する。作業標準化の場面では「正しい手順の映像見本」として訓練教材となり (Fletcher & Tobias, 2006)、不良再現試験では現象の再現性を証明する証拠となる。医療機器・航空宇宙・自動車などの高規制産業では、クリティカルステップの動画記録が規制当局の監査対象となるケースも増加している。

**音声記録** は、両手が塞がっている状況や汚染環境での入力手段として有効である。口述メモは検査員が視線を維持したままデータを記録できる利点を持ち (Malkewitz & Rosenbaum, 2021)、設備異音の記録は振動センサと組み合わせて故障診断の補助根拠となる。ただし、工場環境の騒音レベルは 80〜100 dB に達することも多く (ISO 9612, 2009)、音声認識精度の確保には特別な配慮が必要となる。

**センサストリーム** は、温度・圧力・振動・電流・湿度などを連続的に取得した時系列データである。工程条件のリアルタイム監視と事後検証の双方に利用でき、製品への物理的作用を定量的に証明する最も客観性の高い記録形式である。とりわけ HACCP (Hazard Analysis Critical Control Points) を要件とする食品産業や、IEC 60601 準拠の医療機器製造では、センサログが規制記録の中核を担う (Mortimore & Wallace, 2013)。

| 記録類型 | 主な用途 | 代表フォーマット | 容量オーダー |
|---------|---------|----------------|------------|
| 静止画 | 外観検査・不良現品記録 | JPEG, PNG, HEIF | 1〜5 MB/枚 |
| 動画 | 手順確認・不良再現 | H.264/MP4, HEVC | 50〜500 MB/分 |
| 音声 | 口述メモ・異音記録 | AAC, FLAC, WAV | 0.5〜10 MB/分 |
| センサストリーム | 工程条件記録・故障診断 | CSV, Parquet, HDF5 | 1 KB〜1 MB/時 |

## 2. 規制要件と電子記録の完全性

電子記録は規制当局から「テキスト記録と同等の証拠能力」を要求されており、各産業規制はその要件を詳細に規定している。非テキスト記録もこれらの規制対象であり、「完全性 (Integrity)」の確保が共通の核心概念となっている。

**FDA 21 CFR Part 11** (米国食品医薬品局, 2003) は、電子記録・電子署名の要件を定める連邦規則であり、医薬品・医療機器・食品の GxP 対象製造に適用される。同規則は、電子記録が「ヒューマンリーダブルかつシステム独立的に再現可能」であることを求めており、監査証跡 (Audit Trail) の自動生成・変更履歴保持・アクセス制御を義務付けている (FDA, 2003)。写真ファイルに対しても、「誰が・いつ・なぜ」作成・変更したかの監査証跡が必要とされる。

**EU GMP Annex 11** (EMA, 2011) はコンピュータ化システムに関する欧州 GMP 附則であり、データインテグリティ確保のためのシステム要件を規定する。特に「オリジナルデータ (Raw Data) はシステムから直接取得されなければならず、後から再構成されたものは認められない」という原則は、センサストリームや機器測定値の取得方式に直接影響する。後処理加工前の原始データを保存する義務が生じる。

**IATF 16949** (IATF, 2016) は自動車産業のサプライヤー品質マネジメントシステム規格であり、測定システム分析 (MSA) の記録保持を要求する。外観検査の合否判定基準 (視覚的限度見本・写真見本) も文書化された記録として管理対象となる。非テキスト記録としての限度見本写真は、ISO/IEC 17025 (試験所認定規格) の要件とも交差する。

**ALCOA+ 原則** は電子記録の完全性を評価する業界横断的フレームワークであり、もともと GMP 文脈で発展し、現在は医薬品・医療機器のみならず製造業全般に適用されている (GAMP5, 2008)。ALCOA の頭文字は Attributable (帰属性)・Legible (判読性)・Contemporaneous (同時性)・Original (原本性)・Accurate (正確性) を表し、これに Complete・Consistent・Enduring・Available の 4 原則を加えた ALCOA+ として運用されることが多い。

| 原則 | 意味 | 非テキスト記録における含意 |
|------|------|--------------------------|
| Attributable | 誰が記録したか特定可能 | 撮影者 ID・端末 ID のメタデータへの埋め込み |
| Legible | 判読・解釈可能 | 解像度確保・フォーカス品質・ホワイトバランス |
| Contemporaneous | 事象発生と同時に記録 | 撮影タイムスタンプと工程ログの突合 |
| Original | 原本性の保証 | 暗号ハッシュによる改ざん検知 |
| Accurate | 事実を正確に反映 | センサ校正記録・カメラキャリブレーション |
| Complete | 必要情報が揃っている | 必須メタデータフィールドの入力確認 |
| Consistent | 矛盾なく一貫している | 時刻同期 (NTP) による整合性確保 |
| Enduring | 保存期間を通じて再現可能 | フォーマット選択・マイグレーション計画 |
| Available | 必要時に参照可能 | ストレージ冗長化・検索インデックス設計 |

## 3. メタデータ標準と記録の自己記述性

非テキスト記録が証拠としての価値を持つためには、ファイル本体のみならずそれを取り巻くメタデータが適切に管理されていなければならない。メタデータは「データについてのデータ」であり、記録の文脈 (コンテキスト) を保存する役割を担う (Gilliland, 2016)。

**Exif (Exchangeable image file format)** は JEITA が策定した JPEG・TIFF 向けメタデータ規格であり (CIPA DC-008-2019)、カメラモデル・シャッタースピード・ISO 感度・GPS 座標・撮影日時などを画像ファイルのヘッダに埋め込む。製造現場での応用では、GPS 座標は工場内位置情報として利用でき、撮影日時は工程タイムラインとの突合に用いられる。ただし Exif は後から書き換え可能であるため、改ざん検知手段と組み合わせる必要がある。

**XMP (Extensible Metadata Platform)** は Adobe Systems が開発し W3C の RDF (Resource Description Framework) をベースとしたメタデータ記述フォーマットである (Adobe, 2010)。JPEG・PDF・AI などのファイルに埋め込み可能なほか、サイドカーファイルとして独立保存もできる。カスタムスキーマを定義できるため、製造固有のメタデータ (ロット番号・工程 ID・検査員 ID など) を標準的なメタデータ枠組みで記述することが可能である。

**ISO 23081 (記録管理メタデータ)** は記録管理の国際標準 ISO 15489 の補完規格として策定されており (ISO 23081-1:2006, 23081-2:2009)、記録のビジネス的・技術的コンテキストを記述するメタデータフレームワークを定める。非テキスト記録を公式な「記録 (Record)」として管理する組織では、このフレームワークに準拠したメタデータスキーマを設計することが望ましい。

**W3C PROV** (W3C Provenance Working Group, 2013) はデータの来歴 (Provenance) を記述するオントロジーであり、Entity (エンティティ)・Activity (活動)・Agent (主体) の三元モデルでデータの生成・変換・利用の連鎖を表現する。センサデータや画像データが複数の処理ステップを経て品質記録として確立されるプロセスを PROV-O で記述することで、監査証跡の機械可読性を高めることができる。

製造現場における実装では、アプリケーションレイヤでメタデータを付与するアプローチが現実的である。タブレットアプリが写真撮影後にロット番号・工程 ID・作業員 ID・端末 ID・撮影目的 (外観検査 / 不良記録 / 作業確認 など) をアプリケーションレベルのメタデータとしてサーバに送信し、サーバ側でメタデータをデータベースに保存する設計が、Exif の改ざんリスクを回避しつつ必要なコンテキストを確保する方法として推奨される (Ziegler et al., 2019)。

## 4. 改ざん検知と証跡保全

電子記録の証拠能力は、改ざんが行われていないことを証明できることに依存する。暗号技術を活用した完全性保証は、現代の電子記録管理において不可欠な要素となっている。

**暗号ハッシュ関数** は入力データから固定長のダイジェスト値を生成する一方向関数であり、元データの 1 ビットでも変化すれば出力値が全く異なる値となる性質 (雪崩効果) を持つ。SHA-256 (NIST FIPS PUB 180-4, 2015) は現在最も広く使用されるハッシュ関数であり、256 ビット (32 バイト) のダイジェストを生成する。写真ファイル・動画ファイルを受信した直後にサーバがハッシュを計算してデータベースに保存し、以後任意のタイミングで再計算して比較することで、改ざんの有無を検知できる。

**電子署名** は非対称暗号技術を応用して、署名者の秘密鍵でハッシュを暗号化し、対応する公開鍵で検証する仕組みである。CMS (Cryptographic Message Syntax, RFC 5652) / PKCS#7 は電子署名のコンテナフォーマットであり、X.509 証明書による署名者の身元確認と組み合わせることで「誰が署名したか」も証明できる。医薬品製造での 21 CFR Part 11 対応では、電子署名が紙への手書き署名と同等の法的効力を持つとされる (FDA, 2003)。

**RFC 3161 タイムスタンプ** (Adams et al., 2001) は信頼できる第三者機関 (TSA: Time Stamp Authority) がある時刻にデータが存在したことを証明するプロトコルである。TSA はデータのハッシュを含むタイムスタンプトークンを発行し、暗号署名を付与する。RFC 3161 タイムスタンプを記録作成時に付与することで、「いつ作られたか」の事後証明が可能となり、Contemporaneous 要件の客観的証拠となる。社内システムでは公的な TSA への接続が困難な場合もあるため、社内 NTP サーバとデータベースのトランザクションタイムスタンプによる代替設計も実務的解である。

**ブロックチェーン活用** は非改ざん性の保証手段として近年注目されてきたが、製造記録への実用的適用には慎重な評価が必要である。パブリックブロックチェーン (Ethereum, Bitcoin) は社内オンプレ環境では使用できず、プライベートブロックチェーン (Hyperledger Fabric 等) の導入コストと維持管理の複雑性は、暗号ハッシュ+TSA の組み合わせで達成できる保証水準に対して過剰となる場合が多い (Yli-Huumo et al., 2016)。ブロックチェーンが真に価値を発揮するのは、複数の独立した組織間で記録の改ざん検知を共有する必要がある場面であり、単一の製造拠点内での記録管理には通常不要である。

## 5. ストレージ設計と容量管理

非テキスト記録、特に画像・動画は大容量であり、長期保存を前提とするとストレージ設計が全体システムの実現可能性を左右する。容量見積もりは設計初期段階における重要な検討事項である。

### 5.1 容量見積もり方法論

容量見積もりの基本式は以下の通りである:

```
総容量 = ファイルサイズ × 1日撮影枚数 × 稼働日数/年 × 保存年数 × 冗長化係数
```

標準的な 2 MP JPEG (品質 85%) は概ね 1〜2 MB/枚である。製造ラインで 1 工程あたり 3 枚の写真を記録し、10 工程/日・稼働 250 日/年・7 年保存・冗長化係数 2 を仮定すると:

```
2 MB × 30 枚/日 × 250 日 × 7 年 × 2 = 210 GB
```

全ライン 5 本で 1 TB 程度が必要となる。動画を含む場合は 10〜100 倍以上のオーダーとなるため、動画記録の対象工程と記録頻度を業務要件から慎重に絞り込む必要がある。

### 5.2 フォーマット選択

| フォーマット | 圧縮方式 | 用途適性 | 長期保存性 |
|------------|---------|---------|----------|
| JPEG | 非可逆 (DCT) | 外観写真・一般記録 | 高 (普及率高) |
| PNG | 可逆 (LZ77) | スクリーンショット・図面 | 高 |
| HEIF/HEIC | 非可逆 (HEVC) | 高解像度写真 | 中 (特許問題あり) |
| WebP | 可逆/非可逆 | Web 配信 | 中 (新興) |
| TIFF | 可逆/無圧縮 | アーカイブ原本 | 高 (巨大) |

長期保存記録においては、特許制約のない JPEG または TIFF を原本フォーマットとして採用し、配信用に WebP への変換を行う二層構成が安全である (Library of Congress, Digital Preservation, 2021)。

### 5.3 オンプレ配信設計

社内オンプレ環境では CDN が利用できないため、NGINX の静的ファイル配信機能を活用した設計が標準的となる。画像ファイルは UUID ベースのパスで保存し、データベースには UUID のみを記録することで、ファイルパスとビジネスエンティティの疎結合を実現する。アクセス制御は NGINX の内部リダイレクト (X-Accel-Redirect ヘッダ) とアプリケーションサーバの認証・認可を組み合わせて実装する。

アーカイブ層として LTO (Linear Tape-Open) テープストレージは、GB あたりの保存コストがハードディスクの 1/10 以下であり (Fujifilm, 2022)、非稼働状態でのデータ保持期間が 30 年超とされる高信頼・低コスト記録媒体である。ホットストレージ (PostgreSQL + ファイルシステム)・ウォームストレージ (NAS / S3互換 MinIO) ・コールドストレージ (LTO テープ) の三層構造は、医療・金融分野のアーカイブ設計における実証済みパターンである (SNIA, 2020)。

## 6. 注釈・マスキングと個人情報保護

製造現場での写真記録には、作業員の顔・個人情報が含まれる場合と、競合他社に開示できない機密部品の形状が含まれる場合がある。これらに対するマスキング処理は、法的義務 (GDPR・個人情報保護法) と業務機密保護の両観点から必要となる。

**個人情報保護法対応** では、作業員の顔が写り込んだ写真を記録システムに保存する際に、顔の識別が可能な場合は個人情報として扱われる可能性がある (個人情報保護委員会, 2016)。GDPR (EU 一般データ保護規則, 2018) では顔画像は生体データ (Biometric Data) として特別カテゴリ個人データに該当し、明示的同意または正当な利益の立証が必要となる。実務的対応として、撮影範囲を製品・工程のみに限定するカメラフレーミングガイドラインの策定、またはアプリケーションレベルでの顔検出 + ぼかし処理の自動適用が挙げられる。

**SVG/JSON アノテーション** は画像への注釈 (領域ハイライト・矢印・テキストコメント) を、元画像を破壊せずにオーバーレイとして記述する方式である。W3C Web Annotation Data Model (Sanderson et al., 2017) はこの目的のための標準データモデルを提供しており、JSON-LD フォーマットで注釈を記述することで、注釈の検索・バージョン管理・共有が可能となる。注釈操作の監査証跡として「誰が・いつ・どの領域を・どのような目的でマスキングしたか」をデータベースに記録することは、マスキング行為自体が証拠記録に影響を与えることの透明性確保として重要である。

## 7. 音声入力と工場騒音対策

音声記録・音声入力は、手指が汚染または拘束されている場面でのデータ入力手段として有望であるが、工場環境の騒音は深刻な制約となる。

### 7.1 工場騒音環境の特性

ISO 9612:2009 は職業騒音暴露の測定方法を規定しており、製造現場では 80〜100 dB(A) の騒音レベルが一般的である (ISO 9612, 2009)。特にプレス・鍛造・金属切削工程では 100 dB を超える瞬間騒音が発生し、通常の音声認識モデルが学習したクリーン音声との乖離が著しくなる。Signal-to-Noise Ratio (SNR) が 10 dB を下回ると、多くの音声認識モデルの認識精度は急速に低下する (Seltzer et al., 2013)。

### 7.2 対策技術

**ノイズキャンセリングマイク** は、参照マイクが収音した環境音の逆位相波を生成して主マイク信号から減算するアクティブノイズキャンセリング (ANC) と、指向性マイクが特定方向の音だけを収音する指向性マイクの二種類がある。骨伝導マイクはインフラサウンドと超音波の中間帯域の振動を皮膚から検出するため、外部騒音の影響を大幅に低減できる (Ono et al., 2021)。

**ウェイクワード検出** はオフラインで動作する軽量モデルによるキーワード検出であり、Picovoice の Porcupine は Raspberry Pi 級のエッジデバイスで動作し、カスタムウェイクワードの登録も可能である。Snowboy (現在は非推奨) の後継として、openWakeWord も OSS 実装として注目されている。

**オフライン STT (Speech-to-Text)** として、OpenAI Whisper の量子化版である Whisper.cpp は ARM64 CPU 上でリアルタイム動作が可能であり (Radford et al., 2022)、工場の閉鎖ネットワーク環境でのオフライン音声認識に適する。Vosk は Apache 2.0 ライセンスのオフライン STT エンジンであり、日本語モデルも提供されているため、社内オンプレ環境での採用実績がある。

## 8. センサデータ統合と記録収集設計

PLCとの双方向制御は本アプリのスコープ外であるが、センサデータを「記録収集」する側面での設計は品質トレーサビリティに直接寄与する。

### 8.1 産業プロトコルの概観

**Modbus TCP/RTU** は 1979 年に Modicon が発表した産業通信プロトコルであり (Modicon, 1979)、シンプルなマスター/スレーブ構造と実装の容易さから現在も最も広く使用される産業プロトコルである。温度・圧力・電流センサのほとんどが Modbus インタフェースをサポートしており、Python の pymodbus ライブラリや Rust の tokio-modbus クレートを用いた読み取り実装が容易である。

**OPC-UA (IEC 62541)** は OPC Foundation が策定したプラットフォーム独立型の産業通信標準であり (IEC 62541, 2020)、セキュリティ (TLS 1.2 以上必須)・情報モデル定義・歴史データアクセスを統合した包括的な仕様を持つ。Industry 4.0 の文脈では OPC-UA が標準通信基盤として位置づけられており、センサデータの意味論的記述 (どのセンサが・どの物理量を・どの単位で測定するか) を機械可読な情報モデルとして公開できる。

**MQTT** は軽量なパブリッシュ・サブスクライブ型プロトコルであり (OASIS MQTT v5.0, 2019)、IoT エッジデバイスからのセンサデータ収集に広く使用される。Mosquitto は軽量なオープンソース MQTT ブローカーであり、EMQ X (現 EMQX) はより高スループット向けのオープンソース実装である。社内 LAN 限定での構成では、MQTT over TLS を社内 CA 証明書で実装することで、追加コストなしにトランスポート層のセキュリティを確保できる。

### 8.2 サンプリング設計とナイキスト定理

Shannon のサンプリング定理 (Shannon, 1949) は、正確な信号再現には対象周波数成分の 2 倍以上のサンプリングレートが必要であることを示す。製造工程では対象現象に応じたサンプリング設計が必要であり、例えば振動診断では機械の固有振動数 (数 Hz〜数 kHz) の 2 倍以上でサンプリングし、温度制御では熱時定数 (秒〜分オーダー) を考慮した低周波数でのサンプリングが適切である。

| センサ種別 | 典型的サンプリング間隔 | データ型 | 1年間データ量 (推定) |
|-----------|---------------------|---------|-------------------|
| 温度 | 1〜10 秒 | Float32 | 13〜130 MB |
| 圧力 | 0.1〜1 秒 | Float32 | 130 MB〜1.3 GB |
| 振動 (加速度) | 1〜10 ms | Float32 | 13〜130 GB |
| 電流 | 0.1〜1 秒 | Float32 | 130 MB〜1.3 GB |

時系列データの格納には TimescaleDB (PostgreSQL 拡張) または Apache Parquet 形式が適しており、特に TimescaleDB はチャンク管理・圧縮・連続集計クエリを提供し、PostgreSQL との統合が本アプリのバックエンド選択と親和性が高い (Freedman et al., 2018)。

## 9. 本アプリへの含意

### 9.1 タブレットカメラによる写真記録フロー

本アプリのタブレット側では、作業ステップの完了トリガとして写真撮影を位置づけることが、非テキスト記録の品質を確保しつつ作業フローを自然に組み込む設計として推奨される。

```
撮影 → クライアント側ハッシュ計算 (SHA-256) 
     → メタデータ付与 (作業員ID・工程ID・ロット番号・端末ID・撮影日時) 
     → マルチパートアップロード 
     → サーバ側ハッシュ検証 (転送完全性確認) 
     → ファイルシステム保存 (UUID パス) 
     → データベース記録 (UUID・ハッシュ・メタデータ・タイムスタンプ)
```

React Native (Expo) では `expo-camera` と `expo-file-system` を組み合わせてネイティブカメラと画像取得が可能であり、`expo-crypto` または `react-native-crypto` で SHA-256 ハッシュを計算できる。アップロードの信頼性確保には、オフライン時のキューイング (27 章参照) と再送処理の実装が必要となる。

### 9.2 証拠記録の UI パターン

写真添付をオプショナルではなく、特定工程では必須とする UI パターンが ALCOA+ の Contemporaneous・Complete 要件への対応として有効である。ステップ完了ボタンを「写真撮影後のみ有効化」する設計、または撮影した写真のサムネイルをステップ完了確認画面に表示してユーザが意図的に確認できる UI は、証拠記録の品質を高める。

写真の品質チェック (ブレ・ピンぼけ・暗さの検出) をアプリ側で実施することも検討に値する。ML Kit の Image Labeling API または OpenCV の Laplacian 分散を用いたブレ検出が比較的容易に実装できる。

### 9.3 音声のフォールバック位置づけ

工場騒音環境での音声認識の信頼性問題を踏まえ、本アプリでは音声入力を「プライマリ入力手段」ではなく「補助的フォールバック手段」として設計することが現実的である。具体的には、QR スキャン (32 章参照) とタップ操作をプライマリとし、音声は付記コメントや異常報告の際の口述メモとして活用する。Whisper.cpp のオフラインモデルをアプリに同梱することで、ネットワーク非依存の音声メモ機能を提供できる。

## 10. 批判と限界

### 10.1 「百聞は一見に如かず」信仰の誤謬

写真記録が普及するにつれ、「写真があれば客観的証拠になる」という無批判な信念が広がっているが、これは重大な認識論的誤謬である。写真の解釈は本質的に主観的であり、同一の不良品写真を見た複数の検査員が異なる原因を推定する事例は品質管理文献に繰り返し登場する (Juran & Godfrey, 1998)。写真は「現象の断片」を記録するに過ぎず、因果関係や手順の全体像は別の証拠と組み合わせて初めて意味を持つ。「写真を撮った」という行為が完全な記録義務の履行であるという誤解は、ALCOA+ の Accurate 要件への適合を表面的なものに留める危険がある。

写真の構図・アングル・照明条件は撮影者によって意図的に選択されており、これは無意識の偏向を生む。不良品を「軽微に見える角度」から撮影することで、後の評価を誘導する可能性がある (Messaris, 1994)。規制監査の文脈では、写真証拠の構図選択が証拠の隠蔽と解釈されるリスクを常にはらんでいる。

### 10.2 動画の証拠能力と編集リスク

動画記録はその連続性から写真より高い証拠能力を持つように見えるが、編集技術の普及により「動画だから信頼できる」という前提は崩壊している。フレーム削除・速度変更・クリップ結合は、市販のビデオ編集ソフトで誰でも実施できる。GAN (Generative Adversarial Network) を用いたディープフェイク技術の進展は、動画の改ざん検知をより困難にしつつある (Chesney & Citron, 2019)。

電子署名とハッシュによる完全性保証は「ファイルが署名後に改ざんされていない」ことを証明できるが、「署名前のファイル内容が真正である」ことは保証しない。つまり、事前に作成した虚偽の動画に正当な電子署名を付与することは技術的に可能であり、この問題は現在のいかなる暗号技術でも解決できない。

### 10.3 ストレージコスト爆発と経済合理性

非テキスト記録の大容量化は、長期保存要件 (多くの産業で 10〜30 年が要求される) と組み合わさると、ストレージインフラの維持コストが事業継続を圧迫する可能性がある。動画を全工程で全時間記録するアプローチは、数年以内にペタバイト規模のストレージを要求することになり、中小製造業では非現実的である。

撮影頻度・解像度・保存期間の組み合わせを業務上の必要性 (Business Justification) に基づいて最小化する「データ最小化原則」(GDPR Article 5(1)(c)) は、法的要求であるとともに経済合理性の観点からも重要である。しかし、何が「業務上必要な記録」かの判断は往々にして主観的であり、規制当局の事後的な要求範囲と企業の事前認識のギャップが問題となることがある (Kohn et al., 2000)。

### 10.4 ALCOA+ の Contemporaneous と非テキスト記録の緊張

Contemporaneous 原則は「事象発生と同時に記録すること」を求めるが、非テキスト記録の性質上、これが厳密には達成困難なケースが存在する。センサストリームは連続記録であるため問題ないが、写真記録では「撮影」「アップロード」「サーバ保存完了」の三つの時刻が異なり、オフライン環境ではさらに長時間の遅延が発生する。工程完了後に撮影した写真が本当に工程中の状態を反映しているかどうかは、撮影タイムスタンプからは証明できない。

また、写真記録が「記録すべき事象を証拠として残す」のか「記録行為が作業員の行動を変える」のか、という観察者効果の問題がある (Hawthorne Effect; Mayo, 1933)。写真を撮られることを意識した作業員が通常とは異なる作業手順を取る場合、記録された写真は真の作業実態を反映せず、証拠としての価値が損なわれる。この問題はテキスト記録でも存在するが、写真記録の「視覚的具体性」がより強い行動変容をもたらす可能性がある。
