# 多言語化・外国人労働者と読み書き能力差

## 1. 日本の外国人労働者と製造業の現状

日本の外国人労働者数は2023年10月末時点で約204万人に達し、初めて200万人を突破した（厚生労働省, 2024）。製造業はその最大の受け入れ業種のひとつであり、約45万人（全外国人労働者の約22%）が従事している。主要国籍はベトナム（約51万人、全体最多）、中国（約39万人）、フィリピン（約22万人）、ブラジル（約13万人）、インドネシア（約12万人）の順であり、これらの国籍が製造業の現場に広く分布している。ブラジル・ペルーなどの日系人は1990年代の入管法改正以降「定住者」資格での長期就労が認められており、製造業の基幹労働力として定着している地域も多い（移住連, 2022）。

就労資格の枠組みとして、技能実習制度（1993年創設）は建前上「技術移転」を目的とするが、実態は労働力確保の手段として機能してきたという批判が国内外から根強くあった（ILO, 2020; 法務省・厚生労働省共同調査, 2021）。失踪者数の多さ、低賃金、移動の自由の制限などの問題が国際的に指摘された結果、2024年の入管法改正により「育成就労制度」が創設され、2027年をめどに技能実習制度から移行する方向性が示された（法務省, 2024）。育成就労制度は労働者の転籍要件を緩和し、より労働者保護に重点を置いた設計となっており、製造業における外国人労働者の雇用環境・技能水準・日本語習得期間に大きな変化をもたらすと予想される。

特定技能制度（2019年創設）は、介護・建設・製造業など14分野で人材不足に対応するために創設された在留資格であり、特定技能1号（技能と日本語能力の試験合格が必要、最大5年）と特定技能2号（熟練技能者、期限なし・家族帯同可）の2段階からなる（出入国在留管理庁, 2023）。製造業分野（素形材・産業機械・電気電子情報関連）では特定技能1号の受け入れが進んでいるが、日本語能力要件（日本語能力試験N4相当）を満たす人材確保が課題となっている。特定技能と育成就労の併存は制度の複雑化を招いており、現場の人事・管理担当者の理解を求める文書・研修の整備が急務である。

製造業における外国人労働者の集中は地域的偏在を伴っており、愛知・静岡・群馬・三重・埼玉などの自動車・電機関連製造業集積地に特に高い密度で集中している（厚生労働省, 2024）。こうした地域では工場内の事実上の作業言語が「やさしい日本語」＋複数の外国語の混合となっているケースも珍しくなく、作業手順書・安全指示・品質チェック項目の多言語提供は実務的緊急課題となっている。作業ナビゲーションアプリが将来的に多言語対応を実現するうえで、こうした地域産業の実態把握は不可欠な前提知識である。

## 2. 識字能力差と認知負荷：リテラシーのスペクトラム

識字能力（Literacy）は「読み書き能力の有無」という二分法ではなく、連続的なスペクトラムとして捉えるべきである。OECDの国際成人能力調査（PIAAC: Programme for the International Assessment of Adult Competencies, 2012）は読解力をレベル1〜5に分類しており、レベル2以下（短い文の理解・単純な語彙の照合は可能だが、複数の情報統合や推論が困難）の成人が日本でも約30%存在することを示している（OECD, 2013）。これは日本語母語話者においても機能的非識字（Functional Illiteracy）の問題が存在することを意味し、外国人労働者に限定した問題ではない。

日本語能力試験（JLPT）は国際交流基金・日本国際教育支援協会が実施する最も広く認知された日本語能力測定試験であり、N1（最高）〜N5（最低）の5段階で能力を認定する。製造現場での日本語業務遂行には一般にN3〜N4水準が必要とされるが、技能実習生の日本語学習は来日前の短期研修（3〜6か月）に依存しており、来日直後のN5水準（ひらがな・カタカナの基礎のみ）で現場配属されるケースも多い（公益財団法人国際研修協力機構, 2021）。日本語の表記体系の複雑さ（ひらがな46字+ひらがな濁音/半濁音、カタカナ46字、漢字常用2136字+専門漢字）は、漢字圏（中国・ベトナム）と非漢字圏（インドネシア・フィリピン・ブラジル）の労働者で習得の難易度が大きく異なり、非漢字圏の労働者には特別な視覚的・音声的サポートが必要となる。

認知負荷理論（Sweller, 1988; Sweller et al., 2019）の観点からは、作業員が外国語（日本語）の文字を解読する認知的努力は「外在的認知負荷（Extraneous Load）」として作業パフォーマンスに悪影響を与える。作業指示の理解に認知資源を多く消費する外国人労働者は、作業そのものの「関連認知負荷（Germane Load）」に割ける資源が減少し、結果としてエラー率の上昇・安全上のリスク・習熟の遅れが生じる（van Merriënboer & Sweller, 2005）。この観点から、テキスト情報を最小化しピクトグラムと動画を活用した作業指示は、認知負荷の観点から理論的に正当化される。

| JLPT レベル | 目安の能力 | 製造現場での業務対応 |
|------------|-----------|-------------------|
| N1 | 幅広い場面の日本語理解 | 全作業文書の独立した理解が可能 |
| N2 | 日常的・一般的な日本語理解 | ほぼ全作業指示を理解可能 |
| N3 | 日常的な日本語をある程度理解 | ルーティン作業の指示は概ね理解 |
| N4 | 基本的な日本語を理解 | 簡単な指示・ひらがな主体なら理解 |
| N5 | 非常に基本的な日本語を理解 | ピクトグラム・数字・サポートが必要 |
| 未受験・初学者 | ほぼゼロ | 視覚的指示・通訳必須 |

## 3. ピクトグラム・視覚的指示の科学

ピクトグラム（Pictogram）は特定の概念を図象によって表現する視覚的コミュニケーション手段であり、言語の壁を超えた情報伝達の手段として工業安全の分野で古くから活用されてきた。ISO 7010「安全標識（Safety Signs）」はハザード警告・禁止・必須行動・緊急時設備に関する国際標準ピクトグラムを定義し、製造現場での安全表示の国際統一に寄与している（ISO, 2019）。JIS Z 8210「公共案内記号」はISO 7001に対応する日本規格であり、公共空間での情報表示に関して定める（日本規格協会, 2017）。

アイコン設計理論においてDreyfus（1984）は、アイコンの具象性（Concreteness）が理解の容易さに相関するとし、高い具象性（実物の写実的表現）から低い具象性（抽象記号）までの連続体として分類した。Horton（1994）は文化横断的なアイコン設計の5原則として「具象性」「類似性」「関連性」「恣意性最小」「文化的中立性」を提唱し、ピクトグラムが異文化間で誤解されるリスクを設計段階で低減する方法論を示した。しかし研究の蓄積は、いわゆる「普遍的ピクトグラム」が実際には普遍的でないことを示しており、Hofstede（1991）の文化次元理論—特に「不確実性の回避」「権力格差」—がピクトグラム解釈の文化差に影響することが報告されている（Tijus et al., 2007）。

絵手順書（Pictorial Work Instructions; PWI）は、テキストを最小限に留め、実際の作業動作を連続的な画像（写真・イラスト・アニメーション）で示した作業指示の形式である。Tulis ら（2015）の実験研究は、テキスト主体の指示と比較してPWIが組み立て作業の誤り率を26%低減したことを示した。製造業でのPWI導入事例では、シーメンスや GEなどのグローバルメーカーが組立ラインでPWIを標準採用し、外国人・高齢者・識字能力の低い作業員にもわたって理解度向上の効果が確認されている（MacLeod, 2016）。

| 視覚的指示の種類 | 特徴 | 製造現場での適用例 |
|--------------|------|-----------------|
| ISO 7010ピクトグラム | 国際標準・安全特化 | 危険物・PPE着用指示 |
| 写真手順書 | 高い具象性 | 部品組み付け・配線 |
| イラスト手順書 | 抽象化で汎用性向上 | 手順フロー・分解組立 |
| 動画・GIF | 動作の時系列表現 | 複雑な操作・検査手順 |
| AR/ホログラム | 実空間への重ね表示 | 精密組み付け・保全 |
| QRコード連動 | テキスト補助情報 | 詳細仕様・部品番号 |

## 4. やさしい日本語（Plain Japanese）

「やさしい日本語」は1995年阪神淡路大震災後の外国人被災者支援の経験を踏まえ、弘前大学の佐藤和之らが体系化した日本語の平易化アプローチである（佐藤, 1998）。基本原則は「短い文で書く」「一文に一情報」「難しい言葉を避ける・言い換える」「漢字にはふりがなを付ける」「受動態より能動態を使う」「否定より肯定で表現する」「重要情報を先に書く」「抽象語を使わず具体的に書く」の8原則として整理されている。これらは言語学における「Fog Index（霧指数）」や「Plain Language」運動（アメリカのPlain Writing Act, 2010）と思想的に共通する。

JIS X 0001（情報処理 — 用語）の一部改訂では「理解が困難な人々のための平易な言語」の参照が加えられており、行政文書・公共サービスにおけるやさしい日本語の活用が推進されている。総務省のガイドラインは地方自治体に対し、多文化共生の観点から防災情報・行政サービス案内のやさしい日本語化を求めている（総務省, 2020）。製造業においては、厚生労働省の外国人労働者雇用管理ガイドラインが「作業標準等の母国語又は分かりやすい表現への翻訳・掲示」を求めており（厚生労働省, 2018）、やさしい日本語版作業指示書の作成が推奨されている。

ふりがな（ルビ）の実装はデジタルUI設計においても可能であり、HTMLの `<ruby>` タグ、Androidの `RubySpan`、Webの CSS Ruby Layout が標準的な実装手段である。タブレットアプリにおいてユーザー設定として「ふりがな表示ON/OFF」を実装することで、N3以上の日本語能力を持つ作業員には通常表示を、N5相当や習熟前の外国人には全ルビ表示を提供できる。さらに「常用漢字のみ」「JLPTレベル別漢字」に応じた難易度調整機能は、段階的な日本語習熟を支援する学習機能として位置付けることができる。

## 5. ソフトウェア国際化と地域化の設計原則

ソフトウェアの国際化（Internationalization: I18n）は「ソフトウェアを特定の言語・文化に依存しない形で設計・実装すること」であり、地域化（Localization: L10n）は「国際化されたソフトウェアを特定の言語・文化向けにカスタマイズすること」と定義される（Esselink, 2000）。I18nはL10nの前提条件であり、後からI18nを追加するのは多大なリファクタリングコストを伴う。文字列のハードコーディング回避、日付・時刻・数値・通貨の国際化ライブラリ（Intl API）の活用、フォントの多言語対応（Noto Fontsなど）、テキスト拡張への対応（英語 → ドイツ語で約30%増）はI18nの基本要素である。

Unicode/UTF-8はテキスト表現の国際標準であり（Unicode Consortium, 2023）、Unicodeのコードポイントによってほぼすべての言語の文字が表現可能である。PostgreSQLはUTF-8エンコーディングをデフォルトでサポートし、CJK（中日韓）統合漢字、アラビア文字、デーバナーガリー文字など多様な文字集合を単一データベースで扱える。CLDR（Common Locale Data Repository）はUnicode Consortiumが管理する国際化データ標準であり、各ロケールの日付フォーマット・数値区切り・通貨記号・複数形規則などを定義する。react-intlやi18nextなどのI18nライブラリはCLDRデータを活用し、開発者がロケール固有のフォーマットを個別実装する手間を削減する。

書字方向（Directionality）は見落とされがちなI18n要素であり、アラビア語・ヘブライ語・ペルシャ語はRTL（Right-to-Left）で記述される。CSSのlogical propertiesやFlexboxの `direction: rtl` 設定、ReactのBidi（Bidirectional）サポートによって対応可能だが、日本製製造業アプリでアラビア語・ヘブライ語対応が急務となることは現時点では稀である。一方、日本語縦書き（CSS `writing-mode: vertical-rl`）は伝統的な文書形式との整合性で求められる場合があり、製造業の正式文書テンプレートへの対応として検討価値がある。東南アジア言語（ベトナム語：ラテン文字ベース・声調記号付き、インドネシア語：ラテン文字）は書字方向・文字集合の点でLTR（Left-to-Right）であり、UTF-8対応のみで基本的に問題ない。

| 言語 | 文字体系 | 書字方向 | 製造業での優先度 |
|------|---------|---------|----------------|
| 日本語 | ひらがな・カタカナ・漢字 | LTR（横）/縦書き | 必須（ベース） |
| ベトナム語 | ラテン文字＋声調記号 | LTR | 最優先（在留外国人最多） |
| 中国語（簡体） | 漢字（簡体） | LTR | 高優先 |
| ポルトガル語（BR） | ラテン文字 | LTR | 中優先（日系ブラジル人） |
| インドネシア語 | ラテン文字 | LTR | 中優先 |
| フィリピン語/英語 | ラテン文字 | LTR | 中優先 |
| タイ語 | タイ文字 | LTR | 低優先（現時点） |

## 6. 多言語UI設計の実践：ライブラリと翻訳管理

React Native（Expo）における多言語対応の主流ライブラリは `i18next` + `react-i18next` + `expo-localization` の組み合わせである（i18next contributors, 2023）。`expo-localization` はデバイスのロケール設定を取得し、`i18next` が対応言語リソース（JSONまたはPoファイル）を選択する。管理Web（React + Vite）でも同一の `i18next` + `react-i18next` を使用でき、フロントエンド全体で統一されたI18nアーキテクチャを維持できる。名前空間（namespace）機能によって「作業手順」「機器名称」「エラーメッセージ」「行政・法務用語」などカテゴリ別にリソースを分割管理することが可能であり、翻訳作業の効率化と影響範囲の限定に寄与する。

翻訳管理プラットフォーム（TMS: Translation Management System）はローカライゼーションワークフローを自動化するツールであり、Crowdin・POEditor・Lokalise・Phrase（旧Transifex）などが製造業DXプロジェクトでも採用されている。これらはGitリポジトリとの連携（Pull Request自動作成・翻訳ファイルの自動マージ）、機械翻訳API（DeepL・Google Translate）との統合、翻訳メモリ（TM）による一貫性確保、グロッサリ（専門用語集）管理機能を提供する。製造業の作業指示翻訳では、「トルク」「治具」「仮締め」「絞り値」といった日本語固有の製造用語・社内用語が機械翻訳で誤訳されるリスクが高く、グロッサリ管理と人間レビューのワークフロー設計が品質確保の鍵となる（Moorkens et al., 2018）。

Tulip社のような製造業向けプラットフォームは絵手順書を中心とした「ビジュアルファースト」の作業指示設計を採用しており、テキスト翻訳の必要性自体を低減するアプローチを取る（Tulip Interfaces, 2023）。この方向性はローカライゼーションコストの削減という実用的観点から示唆を与える。テキストより動画・画像・音声を優先した多言語対応では、ナレーション音声の多言語録音コストと字幕生成コストのトレードオフを考慮する必要があるが、音声合成（TTS）技術の向上により追加コストを抑えた多言語音声ガイダンスの実現可能性が高まっている。

## 7. 音声・音声読み上げの役割

テキスト音声合成（TTS: Text-to-Speech）技術の多言語対応は急速に向上しており、Google Cloud Text-to-Speech API・Amazon Polly・Apple AVSpeechSynthesizer（オフライン対応）がベトナム語・インドネシア語・ポルトガル語・タガログ語を含む主要言語の音声出力をサポートしている。製造現場での作業ナビゲーションにおいて、ハンズフリー操作が求められる場面（クリーンルーム・防爆環境・両手使用作業）では音声読み上げ機能が特に価値を持つ。TTSによる多言語読み上げは、識字能力の低い作業員や言語習得初期段階の外国人労働者に対して、テキスト表示と音声案内を組み合わせたマルチモーダルな情報提示が可能となる（Mayer, 2009）。

音声認識（ASR: Automatic Speech Recognition）の非母語話者への精度問題は、現場での音声入力を設計する際に考慮すべき重要な制約である。主要ASRエンジン（Google Speech-to-Text・Microsoft Azure Speech）の認証精度は母語話者で95%以上に達する一方、非母語話者（外国語アクセント・方言）では精度が80%以下に低下するという報告があり（Markl, 2022）、製造現場の騒音（85dB以上の環境）と非母語話者の組み合わせでは信頼できる音声入力は現時点では困難である。このため音声入力は補助手段として位置付け、主たる入力インタフェースはタッチ・スキャン・物理ボタンとし、ASRが補完的役割を担う設計が現実的である。騒音環境での音声活用については31章（非テキスト記録と証拠品質管理）との連携が参考になる。

多言語TTS実装においては、Expo/React Native上では `expo-speech`（AVSpeechSynthesizerラッパー）またはReact Native Text-to-Speech（react-native-tts）が利用可能であり、Android/Windows両プラットフォームをターゲットとする本アプリの場合、Android標準のTTSエンジン（Google TTS）を活用することで追加ライセンスコストなしに多言語音声出力を実現できる。ただし各言語の音声品質はエンジンによって差があり、特にベトナム語の6声調の正確な発音には高品質TTSエンジン（Google WaveNet・Microsoft Neural TTS）の使用が推奨される。

## 8. バリアフリー・ユニバーサルデザインとの交差

ユニバーサルデザイン（UD）の7原則（Mace, 1997）は、外国人・高齢者・障害者を含むすべての利用者を包摂する設計思想を提供する。製造業のデジタル作業指示においても、読字障害（ディスレクシア）・視覚障害・認知障害への配慮は、外国人労働者への多言語対応と多くの場合方向性が一致する。すなわち、ピクトグラム優先・短文・音声サポート・高コントラスト表示・フォントサイズ調整はいずれも複数のユーザー群に同時に恩恵をもたらすインクルーシブデザインの要素である。

WCAG 2.1（Web Content Accessibility Guidelines; W3C, 2018）は「知覚可能（Perceivable）」「操作可能（Operable）」「理解可能（Understandable）」「堅牢（Robust）」の4原則に基づくWebアクセシビリティの国際標準であり、日本ではJIS X 8341-3:2016として対応規格が制定されている（日本規格協会, 2016）。WCAG 2.1の「理解可能」の原則には「読みやすさ（成功基準3.1.3〜3.1.6）」が含まれ、「通常より高い読解レベルを要する内容には平易なバージョンを提供する」こと（成功基準3.1.5）が推奨されている。これはやさしい日本語や多言語翻訳の提供と技術的・概念的に整合している。

Mayer（2009）のマルチメディア学習理論は、人間の情報処理系統として「言語チャネル（テキスト・音声）」と「映像チャネル（画像・動画）」が独立して機能するとする「二重符号化理論（Dual Coding Theory; Paivio, 1986）」に基づき、テキストと関連画像の同時提示が学習効率を高めることを示す。特に言語能力が限られた学習者にとって、画像とテキストの対応付けによる「マルチメディア原則」は認知負荷の低減に有効であり（Sweller et al., 2019）、この理論的背景は外国人労働者向けの作業指示設計を科学的に正当化する。

## 9. 本アプリへの含意：多言語対応ロードマップと設計示唆

**フェーズ1（当面）：I18n基盤の整備とやさしい日本語対応**
まず全テキスト文字列を `i18next` リソースファイル（ja.json）に外部化し、将来の多言語追加に備えた構造を設計段階から組み込む。現時点では日本語のみで運用するが、I18nの構造が整備されていれば言語追加コストが大幅に低減される。並行して、SOPテキストにふりがな表示オプション（`<ruby>` タグ相当のコンポーネント）を実装し、漢字レベル設定（N3/N4/N5相当）を作業員プロファイルに追加する。作業指示のテキストをやさしい日本語の8原則に沿って執筆するスタイルガイドをSOP作成者向けに整備する。

**フェーズ2（中期）：ベトナム語・英語の優先追加**
在留外国人最多のベトナム語と国際共通語としての英語を最初の対応言語として追加する。翻訳品質管理のため、機械翻訳（DeepL API）による素案生成 + 現場ベトナム語話者・専門翻訳者によるレビューのワークフローを確立する。製造用語グロッサリ（ja-vi, ja-en対訳表）を整備し、TMSで管理する。ベトナム語TTSによる作業手順読み上げ機能をオプション実装する。

**フェーズ3（長期）：ピクトグラム優先UI・中国語・ポルトガル語追加**
作業手順書のSOP設計ツールにピクトグラム・写真・動画の挿入機能を強化し、テキストへの依存度を構造的に低減する。ISO 7010準拠のピクトグラムライブラリを内部管理し、安全指示への自動挿入を支援する。中国語（簡体）・ポルトガル語（ブラジル）を追加し、育成就労制度移行後の主要国籍カバレッジを高める。

**アーキテクチャ上の注意点：** バックエンド（Rust/Axum）では多言語テキストをデータベース（PostgreSQL）のJSONBカラムに `{"ja": "トルク締め付け", "vi": "Siết mô-men xoắn", "en": "Torque tightening"}` 形式で保存し、APIレスポンスで要求言語のテキストを返すロケール対応のエンドポイント設計が管理コストを最小化する。Accept-Language ヘッダーまたは user_locale プロファイル設定に基づく言語選択ロジックをミドルウェアレベルで実装することで、各エンドポイントへの言語切り替えロジックの散在を防ぐ。

## 10. 批判と限界

### 10.1 機械翻訳の製造専門用語・社内用語ミス

DeepLやGoogle翻訳は汎用テキストの翻訳精度は高いが、製造業固有の専門用語や各社固有の社内用語の翻訳品質は著しく低下する。「仮締め」「なじみ出し」「カジリ」「ビビリ」といった日本語の製造用語は英語でも対応する標準用語が必ずしも確立しておらず、機械翻訳が意味不明な直訳を生成するリスクがある（Moorkens et al., 2018）。特に安全上重要な指示（「締め付けトルク 25 N·m を超えないこと」など）の誤訳は、製品品質の問題を超えて人身事故に直結する可能性があり、機械翻訳のみによる運用は安全上の観点から推奨できない。翻訳メモリとグロッサリの整備は有効な対策だが、それ自体が継続的なメンテナンスコストを生み、中小製造業では専任の翻訳管理者を置けないケースも多い。翻訳の品質保証体制なしに多言語対応を展開することは、むしろ安全リスクを高める逆説をはらんでいる。

### 10.2 ピクトグラムの文化差解釈リスク

ピクトグラムが言語の壁を超えた普遍的コミュニケーション手段であるという前提は、実験研究によって繰り返し疑問を付されている。Tijus ら（2007）は20か国以上の参加者によるISO安全標識の解釈実験において、同一ピクトグラムに対する正解率が文化・教育水準によって大きく異なることを示した。例えば「立入禁止」の赤い斜線付き円は比較的普遍的に理解されるが、特定の工具の使用禁止・操作手順の順序を示すピクトグラムは、設計者の意図と受け手の解釈が乖離するリスクが高い。Hofstede（1991）が示した「集団主義vs.個人主義」「権力格差」などの文化次元は、指示への従属姿勢（不明点を上司に確認するか、自己判断するか）にも影響し、ピクトグラムを「確認なしに実行する」指示として解釈するか「例示」として解釈するかの差が生じる。製造現場でのピクトグラム採用に際し、対象となる外国籍労働者の文化的背景を踏まえた理解度テスト（Comprehension Test）の実施が強く推奨される。

### 10.3 やさしい日本語がもたらす情報劣化と安全リスク

やさしい日本語化のプロセスは、平易化によって技術的精度が失われるというトレードオフを内包している。「高圧部に触れるな」をやさしい日本語に変換すると「ここに手を触れないでください（高い電気が流れています）」となるが、「高圧」の概念（具体的な電圧値・感電のメカニズム）が省略されることで、危険の程度の認識が「やや危険」にとどまり、即死リスクの緊迫感が伝わらない可能性がある（大北, 2010）。また、やさしい日本語は「日本語の平易化」であるため、日本語知識がほぼゼロの労働者には根本的な解決にならない点も批判される。「やさしい日本語」の普及が多言語対応への真剣な投資を先送りにする言い訳として使われるリスクも指摘されており（野山, 2015）、外国人労働者の言語権（Language Rights）の観点から、平易化による一方的な「分かりやすさ」の押し付けではなく、労働者の言語で情報提供することへの責任を企業が負うべきとする議論がある。

### 10.4 多言語対応コストの中小製造業における壁

多言語対応の実装・維持コストは規模の大きい製造業には吸収可能でも、従業員数十〜数百名規模の中小製造業にとって大きな負担となる。翻訳費用（専門翻訳1ページ5,000〜15,000円）、翻訳管理ツールのライセンス料（年間数万〜数十万円）、翻訳の継続的更新コストが積み上がり、手順書の改訂頻度が高い現場では多言語版の更新が追いつかず、日本語版と多言語版の乖離が生じるリスクがある（中小企業庁, 2022）。外国語への翻訳品質を保証できる社内人材がいない場合、同国籍の先輩労働者が「口頭説明」で補完するという非公式なコミュニケーション体制が実態として機能しており、こうした暗黙知的伝達は記録に残らず、担当者の退職とともに失われるという脆弱性を持つ。アプリが多言語コンテンツの生成・管理を支援するAI機能を組み込むことで、中小製造業のコスト障壁を低減する可能性はあるが、現時点ではその品質保証体制の確立が課題である。

### 10.5 外国人労働者のプライバシーと行動監視の倫理

多言語対応アプリが外国人労働者の言語設定・作業速度・エラー頻度等のデータを記録・分析する場合、24章（作業者プライバシー・データ倫理と労務監視）が論じる労働監視の倫理的問題と交差する。外国人労働者は言語的に不利な立場に置かれており、アプリの利用規約・プライバシーポリシーを十分に理解できないまま同意する「インフォームドコンセントの欠如」が生じやすい（FRA, 2020）。また、言語テスト的要素（作業速度・正解率）のデータが人事評価に流用されることへの懸念は、労使間の信頼関係を損ない、外国人労働者がアプリへの正直な入力を避けるバイアスを生む可能性がある。多言語対応の実装においては、プライバシーポリシーの多言語化（対象言語での説明）とデータ使用目的の透明性確保が倫理的要件として求められる。
