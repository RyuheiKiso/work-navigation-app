# 99 前提制約と本書が約束しないこと

本章は 06_実装 全体の前提・制約・適用範囲外の事項を明記する。本書を利用する際に誤解を避けるための境界を定義し、本書が責任を負わない領域を宣言する。本章は AI/ML に関する方針を言及できる唯一の節でもある。

---

## 1. 本書の適用範囲外

以下の事項は本書（06_実装）の対象外である。

| 対象外の事項 | 権威ドキュメント | 理由 |
|---|---|---|
| プロダクトコード本体（`src/`）の実際の実装 | `src/CLAUDE.md` / `src/backend/CLAUDE.md` | 本書はデプロイ・CI/CD・テスト方針・管理プロセスを定める文書であり、コード実装は src/ 配下の CLAUDE.md が規定する |
| テスト仕様書・テスト実施結果・カバレッジレポート本体 | `docs/07_テスト/` | テスト仕様と結果は 07_テスト が権威である。本書は方針とガイドのみ定める（13 章参照） |
| 移行作業（データ移行・既存システムからの移行） | `docs/08_移行/` | 移行は別フェーズで管理する |
| 運用保守作業（監視・インシデント対応・定期メンテナンス） | `docs/09_運用・保守/` | 運用は別フェーズで管理する |
| システム化構想・計画（なぜ作るか） | `docs/02_企画/` | 企画フェーズで確定済み |
| 要件定義（何を作るか） | `docs/03_要件定義/` | 要件定義フェーズで確定済み |
| 概要設計・詳細設計（どう設計するか） | `docs/04_概要設計/` / `docs/05_詳細設計/` | 設計フェーズで確定済み |

本書は「確定した設計をどのように構築・管理するか」を定める文書である。「何を作るか」「どう設計するか」は上流ドキュメントが権威であり、本書はそれを変更しない。

**本節で確定した方針**
- **本書の適用範囲を構築フェーズのプロセス管理に限定する**: コード実装・テスト仕様は本書の対象外とする
- **上流ドキュメントとの境界を明確にする**: 権威の所在を明示して混乱を防止する
- **本書で定めた方針は src/ の実装を規律する**: 本書は命令文書ではなくプロセス文書である

## 2. 個人開発スコープでの取捨選択

本システムは個人開発であり、チーム開発を前提としたプロセスのうち一部を対象外と判断する。

| 手法・プロセス | 採用状況 | 代替手段 |
|---|---|---|
| ペアプログラミング | **対象外と判断する** | セルフレビュー時間の増加（1 PR あたり 60 分以上） |
| モブプログラミング | **対象外と判断する** | AI アシスタントのレビュー補助活用（任意） |
| 複数名のコードレビュー | **対象外と判断する** | セルフレビュー強化 + 外部レビューの任意活用 |
| セルフホスト CI Runner | **対象外と判断する** | GitHub-hosted runner（ubuntu-latest）で代替 |
| スプリントレトロスペクティブ | **対象外と判断する** | 月次問題集計（17 章 § 7）で代替 |
| スクラムセレモニー | **対象外と判断する** | ADR・問題ログ・トレーサビリティ表で進捗管理 |
| デイリースタンドアップ | **対象外と判断する** | GitHub の PR / Issues でのセルフトラッキング |
| ブルーグリーン / カナリアデプロイ | **対象外と判断する** | 停止時間 5 分以内のローリング更新で代替 |

個人開発スコープでの取捨選択は、品質を下げることを意図しない。チーム開発向けのプロセスを省いた分は、セルフレビュー・テスト・ADR・問題ログの充実で品質を代替的に確保する。

**本節で確定した方針**
- **対象外とした手法は「対象外と判断する」と明記する**: 省略なのか意図的判断なのかを明確にする
- **対象外とした各手法に代替手段を明示する**: 品質低下を黙認しない
- **個人開発スコープは将来の変更を妨げない**: チームが増えた場合は対象外判断を ADR で再評価する

## 3. AI/ML に関する方針

本節はこのプロジェクトにおいて AI/ML を言及できる唯一の節である。他の章での AI/ML 言及は禁止する（本章 § 5「本書が約束しないこと」にも反映）。

### 3.1 ハッシュチェーン異常検知

ハッシュチェーンの破断検知には **ML を採用しない**。理由は以下の通りである。

| 観点 | 説明 |
|---|---|
| 検知ロジック | SHA-256 の比較は決定論的処理であり、ML による確率的判断は不要 |
| ALCOA+ 原則 | Attributable（帰属可能）原則により、異常の根拠が明確である必要がある |
| 監査対応 | 将来の監査で「なぜ異常とみなしたか」を説明できる決定論的アルゴリズムを採用する |

### 3.2 JSON Logic による Step 条件評価

Step の分岐条件評価には **ML を採用しない**。ルールベース（JSON Logic）による宣言的評価とする。

```json
// JSON Logic による Step 条件評価の例（ML ではなくルールベース）
{
  "if": [
    { ">": [{ "var": "measured_value" }, 100] },
    "step_a",
    "step_b"
  ]
}
```

`eval()` / `new Function()` を使った動的評価は `src/CLAUDE.md` で禁止されており、JSON Logic による宣言的評価はこれと整合する。

### 3.3 本システムへの AI/ML 機能追加

本システムへの AI/ML 機能追加は **現時点では対象外と判断する**。理由と将来の検討条件を以下に示す。

| 将来検討が考えられる AI/ML 機能 | 現時点での判断 | 将来の検討条件 |
|---|---|---|
| 作業手順の最適化提案 | 対象外と判断する | ver 2.0 以降で ADR を経て検討する |
| 異常値の機械学習的検出 | 対象外と判断する | トレーサビリティ要件との整合を ADR で評価してから検討する |
| 自然言語による手順検索 | 対象外と判断する | ver 2.0 以降でユーザーニーズが確認されてから ADR で検討する |
| 作業時間の予測 | 対象外と判断する | ALCOA+ の Attributable 原則との整合を ADR で評価してから検討する |

AI/ML 機能の追加は ver 1.0.0 リリース後に ADR-IMPL-NNN を経て正式に検討する。現在の設計・実装はこれらの機能を前提としない。

**本節で確定した方針**
- **ハッシュチェーン異常検知と Step 条件評価に ML を採用しない**: 決定論的処理で監査対応性を確保する
- **AI/ML 機能の追加は ver 1.0.0 リリース後の ADR で検討する**: 現フェーズでは対象外と判断する
- **AI/ML 言及は本節（99 章）のみとする**: 他の章での言及を技術的規約として禁止する

## 4. 規制との関係

本システムにおける規制・基準への対応関係を「**準拠する** / **対応する** / **対象外と判断する**」の 3 段階で宣言する。

### 準拠する規制・基準

| 規制・基準 | 準拠内容 |
|---|---|
| ISO/IEC 25010 品質特性 | 8 特性 + 倫理品質（第 9 の特性）を品質モデルとして採用する。`docs/02_企画/システム化計画/08_品質特性と非機能要件方針.md` が権威 |
| IPA 共通フレーム 2013 6.3.6 | 実装プロセスの構造化に準拠する。本書の各章は CF 2013 のタスクに対応づけて設計している |

### 対応する規制・基準

| 規制・基準 | 対応内容 |
|---|---|
| ALCOA+ データ完全性原則 | Attributable / Legible / Contemporaneous / Original / Accurate + Complete / Consistent / Enduring / Available の各原則に対応する。特に Complete 原則（スキップステップを記録から消さない）はデータモデルに実装する |
| WCAG 2.1 AA アクセシビリティ | カラーコントラスト比 4.5:1 以上・スクリーンリーダー対応・フォーカス可視化に対応する。ただし完全準拠の監査は実施しない |

### 対象外と判断する規制

| 規制 | 判断理由 |
|---|---|
| 薬機法・GMP（医薬品製造） | 製造業の業種固有規制であり、本システムを採用する顧客が対応すべき範囲である。本システムは業種非依存の汎用プラットフォームとして設計する |
| ISO 13485（医療機器） | 同上 |
| ISO 9001 認証取得 | 個人開発・ver 1.0.0 フェーズでは認証取得を対象外と判断する。将来の組織化時に再検討する |
| SOC 2 Type II | 同上 |

対象外と判断した規制への準拠は、顧客がシステムを採用する際に顧客側で実施する。本システムのドキュメントとデータモデルは顧客の規制対応を支援する設計であるが、規制への準拠そのものは保証しない。

**本節で確定した方針**
- **規制表記は「準拠する」「対応する」「対象外と判断する」の 3 種のみ使用する**: 「考慮する」「参考にする」などの曖昧表現は禁止する
- **薬機法・GMP は対象外と判断する**: 業種固有規制への対応は顧客の責任範囲とする
- **ALCOA+ と WCAG 2.1 AA には対応する**: データ完全性とアクセシビリティは設計原則に取り込む

## 5. 本書が約束しないこと

以下の事項は本書（06_実装）が約束しない。

- **実際のソースコード（`src/`）の自動生成**: 本書はデプロイ・CI/CD・テスト方針を定めるものであり、コードを生成しない。コードは `src/` 配下の CLAUDE.md に従って実装者が書く
- **本番環境での自動デプロイ**: 10 章 § 10 に定めるとおり、prod への最終デプロイには手動確認ステップを必ず残す。自動デプロイは staging まで
- **SLA / SLO の確約**: 可用性・パフォーマンスの SLA/SLO は `docs/07_テスト/` でのテスト完了後に確定する。本書の Glanceable 要件は目標値であり、保証値ではない
- **コードレビューの第三者による実施**: 14 章に定めるとおり、個人開発のためピアレビューは任意である。セルフレビューの品質向上で代替する
- **CI/CD のセルフホスト Runner 提供**: 11 章に定めるとおり、セルフホスト Runner は対象外と判断する
- **障害発生時の 24 時間サポート**: 個人開発であり、インシデント対応は `docs/09_運用・保守/` で別途定める
- **データの永続性の無条件保証**: PostgreSQL のバックアップ戦略は `docs/06_実装/10_デプロイ手順.md` に定めるが、バックアップの完全性は 07_テストフェーズで検証する

**本節で確定した方針**
- **本書が約束しないことを明示する**: 利用者の誤解と過度な期待を防止する
- **约束しない理由を各項目に対応するドキュメントへの参照で示す**: 問題が発生した際の正しい参照先を案内する
- **SLA/SLO は 07_テスト 完了後に確定する**: テスト前の数値コミットを禁止する
