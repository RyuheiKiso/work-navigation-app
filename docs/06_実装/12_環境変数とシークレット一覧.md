# 12 設定ファイル・環境変数・シークレット一覧

本章は本システムの **設定管理全体**（YAML ファイル / 環境変数 / シークレット）を一元管理する。  
ADR-IMPL-001 に基づき、非機密設定は `src/infra/config/config.{profile}.yml` で管理し、  
機密は `secret_ref:` 経由で env / DPAPI / Docker secrets から取得する。  
開発者が環境を構築する際の参照資料であり、本番環境のシークレット管理方針を定める。

---

## 1. 命名規約

環境変数名は以下の形式で統一する。

```
WNAV_<SCOPE>_<KEY>
```

### スコープ一覧

| SCOPE | 対象 | 説明 |
|---|---|---|
| `TERMINAL` | Backend terminal-api | wnav_terminal_api（ポート 8080）固有の設定 |
| `MASTER` | Backend master-api | wnav_master_api（ポート 8081）固有の設定 |
| `DB` | Database | PostgreSQL 接続・認証情報 |
| `FE_HA` | Frontend Handy | React Native ハンディ APP 固有の設定 |
| `FE_MA` | Frontend Master | React マスタメンテナンス SPA 固有の設定 |
| `INFRA` | Infrastructure | nginx / TLS / インフラ共通設定 |

両バイナリで共通の設定（JWT 鍵・ログレベルなど）は `WNAV_TERMINAL_*` / `WNAV_MASTER_*` にそれぞれ同じ値を設定する。

### 命名ルール

- スコープと KEY はすべて大文字のスネークケースとする
- KEY は 1 語または複数語をアンダースコアで連結する（例: `JWT_SECRET` / `CORS_ORIGINS`）
- 略語は慣習的に許容されるもののみ使用する（例: `URL` / `TTL` / `TLS`）

### YAML 階層との対応

`WNAV_<SCOPE>_<KEY>` 形式の環境変数名と、YAML `src/infra/config/config.{profile}.yml` のキーパスの対応:

| 旧環境変数名（廃止 / env 残置） | YAML パス | 保管場所 |
|---|---|---|
| `WNAV_BE_PORT` | `server.terminal_api.port` / `server.master_api.port` | YAML |
| `WNAV_BE_CORS_ORIGINS` | `cors.allow_origins` | YAML |
| `WNAV_BE_JWT_TTL_SEC` | `jwt.ttl_sec` | YAML |
| `WNAV_BE_IDEMPOTENCY_TTL_SEC` | `idempotency.ttl_sec` | YAML |
| `WNAV_BE_OUTBOX_INTERVAL_MS` | `outbox.interval_ms` | YAML |
| `WNAV_BE_HASH_CHAIN_VERIFY_CRON` | `hash_chain_verify.cron` | YAML |
| `WNAV_BE_RATE_LIMIT_RPS` | `rate_limit.rps` | YAML |
| `WNAV_BE_LOG_LEVEL` | `observability.log_level` | YAML |
| `WNAV_DB_HOST` / `_PORT` / `_NAME` | `database.host` / `database.port` / `database.name` | YAML |
| `WNAV_DB_USER_WRITE` | `database.write.user` | YAML |
| `WNAV_DB_USER_EVENT_INSERT` | `database.event_insert.user` | YAML |
| `WNAV_DB_USER_READ` | `database.read.user` | YAML |
| `WNAV_DB_MAX_CONNECTIONS` | `database.max_connections` | YAML |
| `WNAV_DB_SSL_MODE` | `database.ssl_mode` | YAML |
| `WNAV_INFRA_NGINX_UPSTREAM` | `infra.nginx.upstream_terminal` / `upstream_master` | YAML |
| `WNAV_INFRA_BACKUP_DIR` | `infra.backup.dir` | YAML |
| `WNAV_INFRA_ALLOWED_IPS` | `infra.network.allowed_ips` | YAML |
| `WNAV_FE_HA_API_BASE_URL` | `frontend.handy.default_api_base_url` | YAML |
| `WNAV_FE_HA_OFFLINE_TIMEOUT_MIN` | `frontend.handy.offline_timeout_min` | YAML |
| `WNAV_FE_HA_SYNC_RETRY_MAX` | `frontend.handy.sync_retry_max` | YAML |
| `WNAV_FE_MA_API_BASE_URL` | `frontend.master.api_base_url` | YAML |
| `WNAV_FE_MA_OPENAPI_URL` | `frontend.master.openapi_url` | YAML |
| `WNAV_FE_MA_SESSION_TIMEOUT_MIN` | `frontend.master.session_timeout_min` | YAML |
| （新規）`WNAV_PROFILE` | YAML 選択用メタ変数 | env のみ（必須） |
| （新規）`WNAV_CONFIG_DIR` | YAML 配置ディレクトリ | env のみ（任意） |
| `WNAV_BE_DATABASE_URL` | **廃止**（YAML 構成要素から組み立て） | cli 用 `DATABASE_URL` のみ維持 |

**本節で確定した方針**
- **`WNAV_<SCOPE>_<KEY>` 形式を全環境変数に適用する**: 非機密設定は YAML で管理し、検索性と管理性を確保する（ADR-IMPL-001）
- **SCOPE は 6 種（TERMINAL / MASTER / DB / FE_HA / FE_MA / INFRA）に限定する**: 新スコープ追加は ADR-IMPL-NNN に記録する
- **命名に略語を使用する場合は慣習的なもののみとする**: 独自略語は禁止する

| `WNAV_BE_WEBHOOK_SECRET` | BE | Webhook 配信の HMAC-SHA256 署名キー | - | ○ | ○ | ○ | 高 | env / DPAPI / docker_secret |
| `WNAV_BE_BACKUP_NOTIFICATION_URL` | BE | バックアップ完了通知の Webhook URL | - | - | ○ | ○ | 中 | env / DPAPI / docker_secret |

機微度の定義:
- **最高**: 漏洩した場合にシステム全体の侵害につながる（JWT 秘密鍵・TLS 秘密鍵）
- **高**: 漏洩した場合に DB への不正アクセスを可能にする（DB パスワード）
- **中**: 漏洩した場合に情報収集に利用される可能性がある（ユーザー名・内部 URL）
- **低**: 漏洩しても直接的な被害がない（ポート番号・タイムアウト値・ログレベル）

**本節で確定した方針**
- **非機密設定は YAML で管理し、機密のみを env / DPAPI / Docker secrets に残す（ADR-IMPL-001）**
- **terminal-api と master-api の環境変数を明確に分割する**: DB ロールの違い（app_event_insert / app_write）と CORS 設定の違いを環境変数レベルで表現する
- **CORS オリジンをバイナリごとに分ける**: terminal-api はハンディ APP（React Native / タブレット）向け、master-api は FE-MA / FE-MC（Vite dev server、デフォルト `http://localhost:5173`）向けに設定する
- **JWT 鍵は両バイナリで同一値を使用する**: monorepo 構成のため `WNAV_TERMINAL_JWT_SECRET` / `WNAV_MASTER_JWT_SECRET` には同じキーペアを設定する
- **機微度を 4 段階で定義する**: 機微度に応じて保管場所と取得元を使い分ける
- **YAML にコミットできない値のみ `.env` に残す**: 開発者が迷わない分類を維持する
- **local/dev 環境は `.env` を使用する**: `.env` は `.gitignore` に必ず追加する

### 3.1 ローカル環境

```bash
# .env ファイルで管理する（必ず .gitignore に追加する）
cp .env.example .env
# .env を編集して実際の値を設定する（<PLACEHOLDER> を置換する）
```

`.env` ファイルには機微度「高」「最高」の変数も含まれるが、**絶対に Git にコミットしない**。`.gitignore` に以下を追加する。

```
# シークレットファイルをバージョン管理から除外する
.env
.env.local
.env.*.local
```

### 3.2 CI 環境

GitHub Actions Secrets を使用する。`Settings > Secrets and variables > Actions` で登録する。

```yaml
# ワークフロー内での参照例（2 バイナリ対応）
env:
  WNAV_TERMINAL_DATABASE_URL: ${{ secrets.WNAV_TERMINAL_DATABASE_URL }}
  WNAV_MASTER_DATABASE_URL: ${{ secrets.WNAV_MASTER_DATABASE_URL }}
  WNAV_TERMINAL_JWT_SECRET: ${{ secrets.WNAV_JWT_SECRET }}
  WNAV_MASTER_JWT_SECRET: ${{ secrets.WNAV_JWT_SECRET }}
```

### 3.3 本番環境（Windows Server 2022）

機微度「高」「最高」の変数は Windows DPAPI（Data Protection API）で暗号化して保存する。

```powershell
# Windows DPAPI でシークレットを暗号化して環境変数に設定する
$secret = Read-Host "Enter JWT Secret" -AsSecureString
$encrypted = ConvertFrom-SecureString $secret
[Environment]::SetEnvironmentVariable("WNAV_TERMINAL_JWT_SECRET_ENCRYPTED", $encrypted, "Machine")
[Environment]::SetEnvironmentVariable("WNAV_MASTER_JWT_SECRET_ENCRYPTED", $encrypted, "Machine")
```

Docker Compose 環境では Docker secrets を使用する。

```yaml
# docker-compose.yml でのシークレット参照例（2 バイナリ対応）
secrets:
  jwt_secret:
    external: true
  terminal_db_url:
    external: true
  master_db_url:
    external: true

services:
  terminal-api:
    secrets:
      - jwt_secret
      - terminal_db_url
    environment:
      WNAV_TERMINAL_JWT_SECRET_FILE: /run/secrets/jwt_secret
      WNAV_TERMINAL_DATABASE_URL_FILE: /run/secrets/terminal_db_url

  master-api:
    secrets:
      - jwt_secret
      - master_db_url
    environment:
      WNAV_MASTER_JWT_SECRET_FILE: /run/secrets/jwt_secret
      WNAV_MASTER_DATABASE_URL_FILE: /run/secrets/master_db_url
```

**本節で確定した方針**
- **環境別に保管場所を分ける**: local は `.env` / CI は GitHub Secrets / 本番は DPAPI または Docker secrets
- **`.env` の Git コミットを `.gitignore` で技術的に防止する**: ヒューマンエラーに依存しない
- **本番の最高機微度変数は DPAPI または Docker secrets で保護する**: 平文でのファイル保存を禁止する

## 4. ローテーション手順

### 4.1 JWT 秘密鍵ローテーション

```bash
# 新しい RS256 キーペアを生成する
openssl genrsa -out wnav_jwt_new.pem 4096
openssl rsa -in wnav_jwt_new.pem -pubout -out wnav_jwt_new_pub.pem

# 新旧キーを並行稼働させてトークンの失効を待つ（TTL 8h 分の猶予を設ける）
# 1. GitHub Secrets に新しいキーを登録する
# 2. backend を再起動して新しいキーで署名を開始する
# 3. 8h 後に旧キーの検証を無効化する
```

### 4.2 PostgreSQL パスワードローテーション

```sql
-- 新しいパスワードを生成する
ALTER ROLE app_write WITH PASSWORD 'new_secure_password_here';
ALTER ROLE app_event_insert WITH PASSWORD 'new_secure_password_here';
ALTER ROLE app_read WITH PASSWORD 'new_secure_password_here';
```

```bash
# 環境変数を更新して両バイナリを再起動する
# GitHub Secrets の WNAV_DB_PASSWORD_* を新しい値に更新する
docker compose restart terminal-api master-api
```

### 4.3 TLS 証明書ローテーション

TLS 証明書は有効期限 30 日前に更新する（10_デプロイ手順.md § 6.3 参照）。

### 4.4 API キーローテーション

外部 Webhook の HMAC 署名キー（`WNAV_TERMINAL_WEBHOOK_SECRET`）は 90 日ごとにローテーションする。

**本節で確定した方針**
- **JWT 秘密鍵ローテーションは旧キーの TTL 分の猶予を設ける**: 既存トークンの突然の無効化を防止する
- **DB パスワードローテーション後は即座に backend を再起動する**: 旧パスワードの接続プールを確実にクリアする
- **TLS 証明書は有効期限 30 日前に更新する**: 自動更新の仕組みがない場合はカレンダーリマインダーを設定する

## 5. YAML 配置と secret_ref 文法

ADR-IMPL-001 の採用に伴い、非機密設定は YAML ファイルで管理し、機密のみを env / DPAPI / Docker secrets で管理する。

### YAML ファイル配置

```
src/infra/config/
  config.base.yml               # 全環境共通の既定値（コミット対象）
  config.local.yml.example      # local プロファイルのテンプレート（コミット対象）
  config.local.yml              # 各開発者の実値（.gitignore 対象）
  config.dev.yml.example        # dev プロファイルのテンプレート（コミット対象）
  config.dev.yml                # dev 実値（.gitignore 対象）
  config.staging.yml            # staging 実値・機密は secret_ref（コミット対象）
  config.prod.yml               # prod 実値・機密は secret_ref（コミット対象）
  schema/config.schema.json     # JSON Schema（バリデーション・TS 型生成）
```

YAML の起動時読込: `config.base.yml` + `config.{WNAV_PROFILE}.yml` をマージ後、`SecretRefProvider` で全 secret_ref を解決する。

### secret_ref 文法

```yaml
# 文法: secret_ref: "<scheme>:<identifier>"
some_credential:
  secret_ref: "env:WNAV_DB_PASSWORD_WRITE"          # 環境変数から取得
other_cred:
  secret_ref: "dpapi:WNAV_DB_PASSWORD_WRITE_ENC"    # Windows DPAPI 復号
docker_cred:
  secret_ref: "docker_secret:db_write_password"     # /run/secrets/<name>
file_cred:
  secret_ref: "file:/etc/wnav/jwt_private.pem"      # ファイル全体を読む
```

| scheme | 解決方法 | 利用環境 |
|---|---|---|
| `env:` | `std::env::var()` | local / dev / CI |
| `dpapi:` | Windows DPAPI `Unprotect` | prod (Windows Server 2022) |
| `docker_secret:` | `/run/secrets/<name>` を読む | Docker (dev / staging / prod) |
| `file:` | 指定パスのファイル内容 | TLS 証明書 PEM など |

secret_ref の実体を含む YAML は **Git にコミットしない**（`config.local.yml` / `config.dev.yml` は .gitignore 対象）。

## 8. 緊急失効手順

シークレットが漏洩した可能性がある場合は以下の手順を実行する。

```
1. GitHub Secrets の該当シークレットを即座に削除または無効化する
2. 新しいシークレットを生成してローテーションする
   - JWT: 新しいキーペアを生成（openssl genrsa）
   - DB: ALTER ROLE でパスワードを変更
   - TLS: 証明書を再発行
3. backend を再起動してセッションをすべて無効化する
4. GitHub の Audit Log で不正アクセスがないかを確認する
5. 影響範囲を分析する（いつから・何が・どの程度漏洩したか）
6. ADR-IMPL-NNN として失効対応の記録を残す
7. 将来の再発防止策を ADR に記録する
```

**本節で確定した方針**
- **漏洩疑惑が発生した場合は即時失効を優先する**: 影響範囲の分析より先に失効処理を実施する
- **対応記録は ADR-IMPL-NNN として残す**: 失効対応のトレーサビリティを確保する
- **再発防止策を ADR に記録する**: 同じ漏洩パターンを繰り返さない

## 9. 監査ログ

| 監査対象 | 監査手段 | 確認方法 |
|---|---|---|
| GitHub Secrets のアクセス | GitHub Audit Log | `Settings > Audit Log` でフィルタ |
| DB への接続 | PostgreSQL ログ（`log_connections = on`） | `pg_log/` のログを定期確認 |
| backend のアクセス | tracing ログ（JSON 形式・相関 ID 付き） | Docker logs から確認 |
| CI パイプライン実行 | GitHub Actions の実行履歴 | `Actions` タブ |

定期的な監査として月 1 回以上 GitHub Audit Log を確認し、不審なアクセスがないことを確認する。

**本節で確定した方針**
- **GitHub Audit Log を月 1 回確認する**: シークレットへの不審なアクセスを早期に発見する
- **DB 接続ログを有効化する**: `log_connections = on` を PostgreSQL 設定に追加する
- **相関 ID を全ログに付与する**: 不審なリクエストを特定可能にする
