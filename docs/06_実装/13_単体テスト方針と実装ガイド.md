# 13 単体テスト方針と実装ガイド

本章は作業ナビゲーション＆トレサビ記録アプリの単体テストに関する方針・実装ガイド・カバレッジ目標を定める。テスト仕様書・実施結果・カバレッジレポート本体は `docs/07_テスト/` が権威となる。

---

## 1. テストピラミッド

図: fig_unit_test_pyramid（img/ 配下）を参照

テストピラミッドは以下の 3 層構成とする。

| 層 | 種別 | 割合 | ツール | 実行速度 |
|---|---|---|---|---|
| 底辺 | 単体テスト | 70% | cargo nextest / Jest / Vitest | 高速（ミリ秒単位） |
| 中段 | 統合テスト | 20% | testcontainers-rs / msw | 中速（秒単位） |
| 頂点 | E2E テスト | 10% | Detox (handy) / Playwright (master) | 低速（分単位） |

単体テストは「一つの関数・コンポーネント・ドメインオブジェクトを独立して検証する」テストであり、外部依存（DB・ネットワーク・ファイルシステム）はモックする。統合テストは「複数のコンポーネントが連携する動作を実際のインフラ（PostgreSQL コンテナ）で検証する」テストである。

**本節で確定した方針**
- **単体 70% / 統合 20% / E2E 10% の割合を目標とする**: 単体テストを中心に品質を確保する
- **単体テストは外部依存をモックする**: DB・ネットワークへの実接続は統合テストで行う
- **E2E テストはクリティカルパスのみカバーする**: コストパフォーマンスを考慮して選択的に実装する

## 4. Rust 単体テスト

### 4.1 テストフレームワーク

`cargo nextest` を使用する。`cargo test` との比較で並列実行・詳細なレポート・JUnit XML 出力の面で優れている。

```bash
# 単体テストを実行する（並列実行）
cargo nextest run --all-features

# 特定のテストのみ実行する
cargo nextest run --test-threads 4 domain::hash_chain

# JUnit XML レポートを出力する（CI での解析用）
cargo nextest run --all-features --junit-output-path test-results.xml
```

### 4.2 テストモジュール構造

```rust
// src/domain/hash_chain/mod.rs

#![forbid(unsafe_code)]

// ハッシュチェーン生成・検証ロジックを実装するモジュール
pub mod chain;
pub mod verifier;

#[cfg(test)]
mod tests {
    use super::*;
    use crate::domain::hash_chain::chain::HashChain;

    // ハッシュチェーンの初期ブロックが正しく生成されることを検証する
    #[test]
    fn should_create_genesis_block_with_zero_hash() {
        let genesis = HashChain::genesis();
        assert_eq!(genesis.prev_hash, [0u8; 32]);
        assert!(!genesis.hash.is_empty());
    }

    // 連続するイベントのハッシュチェーンが正しく連結されることを検証する
    #[test]
    fn should_chain_consecutive_events_correctly() {
        let event_a = HashChain::genesis();
        let event_b = HashChain::next(&event_a, b"payload_b");
        assert_eq!(event_b.prev_hash, event_a.hash);
    }
}
```

### 4.4 プロパティベーステスト

```rust
use proptest::prelude::*;

// プロパティベーステスト: ハッシュチェーンの連続性を任意の入力で検証する
proptest! {
    #[test]
    fn hash_chain_is_always_contiguous(
        payloads in proptest::collection::vec(any::<Vec<u8>>(), 2..=100)
    ) {
        let mut chain = vec![HashChain::genesis()];
        for payload in &payloads {
            let next = HashChain::next(chain.last().unwrap(), payload);
            chain.push(next);
        }
        // 連続性の検証: 各ブロックの prev_hash が前ブロックの hash と一致することを確認する
        for window in chain.windows(2) {
            prop_assert_eq!(window[1].prev_hash, window[0].hash);
        }
    }

    // Idempotent API: 同一 Idempotency-Key での重複送信が同じレスポンスを返すことを検証する
    #[test]
    fn idempotent_api_returns_same_response_for_duplicate_key(
        key in "[a-f0-9]{32}"
    ) {
        let cache = IdempotencyCache::new();
        let response_a = cache.get_or_insert(&key, || create_response());
        let response_b = cache.get_or_insert(&key, || create_different_response());
        prop_assert_eq!(response_a, response_b);
    }
}
```

**本節で確定した方針**
- **cargo nextest を採用する**: `cargo test` の代替として並列実行とレポート機能を活用する
- **テーブル駆動テストで境界値を網羅する**: 単一テスト関数で複数ケースを検証する
- **proptest でプロパティベーステストを実施する**: ハッシュチェーンと冪等性の検証に必須とする

## 7. モック方針

| 対象 | モック方針 | 理由 |
|---|---|---|
| 外部 HTTP（バックエンドへのリクエスト） | `msw` でモックする | ネットワーク層の分離・テスト安定性 |
| デバイス API（カメラ・位置情報） | Jest/Expo モックを使用する | 実デバイス不要でテスト可能にする |
| システムクロック（`Date.now()` / `Utc::now()`） | 注入可能な Clock インターフェースでモックする | 時刻依存テストの安定性確保 |
| DB（単体テスト） | モックしない（統合テストで実 DB 使用） | SQL の正確性を実 DB で保証する |
| ドメインロジック | モックしない | ビジネスロジックは直接テストする |
| ファイルシステム | 必要な場合のみ一時ディレクトリを使用 | 実環境に近いテストを優先する |

```typescript
// msw での API モック設定例（handy / master 共通）
import { rest } from 'msw';
import { setupServer } from 'msw/node';

// ネットワーク層のみをモックする（ドメインロジックはモックしない）
const server = setupServer(
  rest.post('/api/v1/work-events', (req, res, ctx) => {
    return res(ctx.status(202), ctx.json({ id: 'mock-event-id' }));
  }),
  rest.get('/api/v1/work-orders/:id', (req, res, ctx) => {
    return res(ctx.status(200), ctx.json(buildWorkOrderFixture()));
  }),
);

beforeAll(() => server.listen());
afterEach(() => server.resetHandlers());
afterAll(() => server.close());
```

**本節で確定した方針**
- **外部 HTTP のみを msw でモックする**: DB とドメインロジックはモックしない
- **Clock を注入可能な設計とする**: 時刻依存コードのテスタビリティを確保する
- **デバイス API は公式モックを使用する**: 実デバイスなしでテストを完結させる

## 8. testability 設計

テストしやすいコードを書くための設計原則を定める。

### 8.1 依存性注入

```rust
// 外部依存を トレイトで抽象化して注入可能にする
pub trait Clock: Send + Sync {
    fn now_ms(&self) -> i64;
}

pub struct SystemClock;
impl Clock for SystemClock {
    fn now_ms(&self) -> i64 {
        Utc::now().timestamp_millis()
    }
}

// テスト用のモック Clock
pub struct MockClock { pub fixed_ms: i64 }
impl Clock for MockClock {
    fn now_ms(&self) -> i64 { self.fixed_ms }
}

// EventService は Clock を注入可能な構造とする
pub struct EventService<C: Clock> {
    clock: Arc<C>,
    pool: PgPool,
}
```

### 8.2 ポート/アダプタ構造

ビジネスロジックは外部依存（DB・HTTP・時計）を抽象化したトレイトに依存し、具体的な実装はアダプタが提供する。テスト時はアダプタをモックに差し替える。

```
Domain Layer (純粋なビジネスロジック)
    ↑ depends on
Port (trait/interface)
    ↑ implements
Adapter (実装: PostgreSQL / HTTP / SystemClock)
```

**本節で確定した方針**
- **Clock を注入可能なトレイトで定義する**: 時刻依存テストの安定性を保証する
- **ポート/アダプタ構造でドメインを外部依存から分離する**: テストとプロダクションで差し替えを容易にする
- **純粋関数を積極的に使用する**: 副作用を境界に集約してロジックをテストしやすくする

## 9. オフラインシナリオテスト

オフラインシナリオのテストは **必須** とする。ネットワーク状態の 4 段階（高品質接続 / 低品質接続 / 切断 / Emergency Mode）それぞれについてテストケースを実装する。

```typescript
// ネットワーク状態 4 段階のモックと Emergency Mode 遷移テスト
describe('OfflineFirst', () => {
  // 切断時にローカル操作が継続できることを確認する
  it('should continue local operation when disconnected', async () => {
    server.use(
      rest.post('/api/v1/work-events', (req, res) =>
        res.networkError('Network disconnected')
      ),
    );

    const engine = createStepEngine();
    await engine.completeStep(buildStep());

    // ローカル SQLite に記録されていることを確認する
    const localEvents = await localDb.getOutboxPending();
    expect(localEvents).toHaveLength(1);
  });

  // オフライン継続時間が設定値を超えた場合に Emergency Mode に遷移することを確認する
  it('should transition to emergency mode after offline timeout', async () => {
    const clock = new MockClock(Date.now());
    const networkMonitor = new NetworkMonitor({ clock, offlineTimeoutMin: 5 });

    networkMonitor.setDisconnected();
    clock.advance(5 * 60 * 1000 + 1); // 5 分 1ms 経過させる

    const state = networkMonitor.getState();
    expect(state).toBe(NetworkState.EmergencyMode);
  });
});
```

**本節で確定した方針**
- **オフラインシナリオテストを必須とする**: ネットワーク障害は製造現場で頻発するリスクがある
- **4 段階のネットワーク状態を全てテストする**: 各状態の動作を明示的に検証する
- **Emergency Mode への遷移を自動テストで保証する**: 手動確認に依存しない

## 10. ハッシュチェーン・冪等性・Append-only テストパターン

### 10.1 ハッシュチェーン検証テスト

```rust
// SHA-256 ハッシュチェーンの連続性・破断検出テスト
#[cfg(test)]
mod hash_chain_tests {
    use super::*;

    // 1000 件のイベントでハッシュチェーンの連続性が保たれることを検証する
    #[test]
    fn should_maintain_chain_integrity_for_large_event_sequence() {
        let mut prev = HashChain::genesis();
        for i in 0..1000 {
            let payload = format!("event_{}", i).into_bytes();
            let current = HashChain::next(&prev, &payload);
            assert_eq!(current.prev_hash, prev.hash,
                "Chain broken at event {}", i);
            prev = current;
        }
    }

    // ハッシュチェーンの破断を検出できることを確認する
    #[test]
    fn should_detect_tampered_event_in_chain() {
        let mut chain = vec![HashChain::genesis()];
        chain.push(HashChain::next(&chain[0], b"payload_a"));
        chain.push(HashChain::next(&chain[1], b"payload_b"));

        // チェーンの中間イベントを改竄する
        chain[1].hash = [0xFF; 32];

        let result = verify_chain_integrity(&chain);
        assert!(result.is_err(), "Tampered chain must be detected");
    }
}
```

### 10.2 冪等性テスト

```rust
// 同一 Idempotency-Key での重複送信が同じ結果を返すことを検証する
#[tokio::test]
async fn should_return_same_response_for_duplicate_idempotency_key() {
    let key = Uuid::new_v4().to_string();
    let payload = create_work_event_payload();

    let response_a = post_work_event(&key, &payload).await;
    let response_b = post_work_event(&key, &payload).await; // 重複送信

    assert_eq!(response_a.status(), response_b.status());
    assert_eq!(response_a.body(), response_b.body());

    // DB にイベントが 1 件しか存在しないことを確認する
    let event_count = count_events_by_idempotency_key(&key).await;
    assert_eq!(event_count, 1, "Duplicate event must not be recorded");
}
```

**本節で確定した方針**
- **ハッシュチェーンの連続性テストを 1000 件以上のシーケンスで実施する**: 大規模データでの正確性を保証する
- **改竄検出テストを必須とする**: セキュリティ要件の自動検証
- **冪等性テストで DB の重複排除を確認する**: アプリ層の保証だけでなく DB レベルの確認も実施する

```typescript
// 形式: it('should <expected>', ...) または it('should <expected> when <condition>', ...)
it('should display emergency banner when offline for 5 minutes', () => { ... });
it('should queue event to outbox when network is disconnected', () => { ... });
it('should reject invalid idempotency key format', () => { ... });
```

命名規約のルール:
- テスト名はテストの意図を日本語ではなく英語で記述する（CI レポートとの整合性のため）
- `should` で始め、期待される動作を動詞句で表現する
- 条件がある場合は `when` 以降に条件を追加する
- `and` を含む場合はテストを分割することを検討する

**本節で確定した方針**
- **Rust は `should_<expected>_when_<condition>` 形式を採用する**: CI レポートで読みやすい名前にする
- **TypeScript は `it('should ...')` 形式を採用する**: BDD スタイルの読みやすさを確保する
- **1 テスト関数 = 1 アサーション（主要アサーション）を原則とする**: テスト失敗時の原因特定を容易にする

```rust
// Rust でのテストデータビルダ
#[cfg(test)]
pub mod builders {
    use super::*;

    // テストデータを構築するビルダ
    pub struct WorkEventBuilder {
        case_id: Option<Uuid>,
        activity: Option<String>,
    }

    impl WorkEventBuilder {
        pub fn new() -> Self {
            Self { case_id: None, activity: None }
        }

        pub fn case_id(mut self, id: Uuid) -> Self {
            self.case_id = Some(id);
            self
        }

        pub fn activity(mut self, activity: &str) -> Self {
            self.activity = Some(activity.to_string());
            self
        }

        pub fn build(self) -> WorkEvent {
            WorkEvent {
                case_id: self.case_id.unwrap_or_else(Uuid::new_v4),
                activity: self.activity.unwrap_or_else(|| "step_completed".to_string()),
                timestamp: Utc::now().timestamp_millis(),
                ..Default::default()
            }
        }
    }
}
```

Fixture ファイルは `tests/fixtures/` ディレクトリに JSON 形式で管理し、大きなテストデータセット（マスタデータ・シナリオデータ）はファイルから読み込む。

**本節で確定した方針**
- **Builder パターンで差分最小のテストデータを生成する**: 変更に強いテストを作成する
- **Fixture ファイルは `tests/fixtures/` で管理する**: テストコードとデータを分離する
- **デフォルト値を持つビルダをデフォルト引数で呼び出し可能にする**: テストコードの簡潔性を確保する
