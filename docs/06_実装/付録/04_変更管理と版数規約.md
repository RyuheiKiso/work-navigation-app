# 04 変更管理と版数規約

本章は `06_実装` フェーズの版数体系・変更種別・変更管理プロセス・文書版数管理・緊急変更手順を確定する。`04_概要設計/付録/06_変更管理と版数規約.md` の規約を継承し、実装フェーズ固有の事項を追加定義する。

---

## 1. 版数体系

### 1-1. セマンティックバージョニング（SemVer）

`06_実装` フェーズのドキュメントおよびソフトウェアリリースは `メジャー.マイナー.パッチ` で管理する。

| 変更種別 | バージョン操作 | 例 |
|---|---|---|
| 破壊的変更（API の後方非互換変更・DB スキーマの非互換変更・識別子体系の変更）| メジャー上げ | 0.x.y → 1.0.0 |
| 機能追加（新 IMPL-NNN 実装完了・新 API エンドポイント・新テーブル追加）| マイナー上げ | 0.1.y → 0.2.0 |
| バグ修正・軽微修正（ロジック修正・誤字修正・パフォーマンス改善）| パッチ上げ | 0.1.0 → 0.1.1 |

**初期版数**: `0.1.0`（実装着手段階。`1.0.0` は全 IMPL-NNN 確定済み・全 TST-NNN 通過・本番デプロイ完了時に付与）

### 1-2. ライフサイクル

| 状態 | 定義 | 移行条件 |
|---|---|---|
| **Draft** | 実装着手前・実装中。IMPL-NNN が「予約済み」または「実装中」の状態が存在する | 全 IMPL-NNN が確定済み状態 |
| **RC (Release Candidate)** | 全実装完了・品質ゲート通過前。TST-NNN の全件実施完了待ち | 全 TST-NNN 通過・セキュリティ審査完了 |
| **Released** | Ver 1.0.0 リリース済み。変更には本規約 §3 のプロセスが必要 | Git タグ `v1.0.0` 付与 |

## 2. Changelog 規約

### 2-1. Conventional Commits ベースの変更分類

Git コミットメッセージおよび `CHANGELOG.md` の記録は Conventional Commits 形式を使用する。

| Type | 用途 | バージョン影響 |
|---|---|---|
| `feat` | 新機能実装（新 IMPL-NNN の確定）| マイナー上げ |
| `fix` | バグ修正（PROB-NNN 起票の問題解決）| パッチ上げ |
| `refactor` | リファクタリング（機能変化なし）| パッチ上げ |
| `test` | テスト追加・修正 | パッチ上げ |
| `docs` | ドキュメントのみの変更 | パッチ上げ |
| `chore` | ビルド・CI 設定変更 | パッチ上げ |
| `perf` | パフォーマンス改善 | パッチ上げ |
| `security` | セキュリティ修正（CR-NNN・QA-NNN 参照）| パッチ上げ（場合によりメジャー上げ）|
| `BREAKING CHANGE` | 破壊的変更（コミットフッタに記載）| メジャー上げ |

### 2-2. コミットメッセージ形式

```
<type>(<scope>): <description>

[optional body]

[optional footer(s)]
IMPL-NNN: <実装識別子>
ADR-IMPL-NNN: <判断記録参照>（変更判断を伴う場合）
PROB-NNN: <問題管理参照>（バグ修正の場合）
```

## 3. 変更管理プロセス

### 3-1. 通常変更プロセス（機能追加・バグ修正）

1. **変更起票**: `docs/01_管理/change_log.md` に 変更種別・対象 IMPL-NNN・変更内容・理由を記録する。
2. **影響分析**: 本付録/01_ITM.md で影響する FR-NNN・TST-NNN を列挙し、05_詳細設計との逸脱有無を確認する。
3. **ADR 起票判断**: 以下の 3 条件を全て満たす場合のみ `ADR-IMPL-NNN` を起票する。
   - 技術的に不可逆な決定（例: ライブラリ選定・DB スキーマ変更）
   - 有力な代替案が存在した
   - 選択理由が自明でない
4. **ドキュメント先行更新**: 本 `06_実装/` 内の関連 md ファイルを更新してからコードを変更する（ドキュメント先行原則）。
5. **採番台帳更新**: 付録/99 の対応 IMPL-NNN・PROB-NNN・ADR-IMPL-NNN 等のステータスを更新する。
6. **Git コミット**: §2-2 の形式でコミットメッセージを記述する。

### 3-2. 設計変更プロセス（詳細設計への遡及影響を伴う変更）

詳細設計（05_詳細設計）の設計成果物（DDL・OpenAPI・アルゴリズム仕様）の変更を伴う場合は以下を追加する。

1. 05_詳細設計の当該章を更新する（ドキュメント先行）。
2. 本付録/03_05詳細設計との対応一覧.md の該当行を「部分実装（ADR-IMPL-NNN 参照）」に更新する。
3. ADR-IMPL-NNN を必ず起票する（詳細設計逸脱は ADR 必須）。

### 3-3. 承認フロー

個人開発のため外部承認者を設定しない。全変更はセルフレビュー（CR-NNN 記録）を実施してコミットする。

## 4. 文書版数管理

### 4-1. 版数表の書き方

`docs/06_実装/` 内の各 md ファイルの末尾に以下の形式で版数履歴を設ける。

```markdown
## 版数履歴

| 版 | 日付 | 変更者 | 変更内容 |
|---|---|---|---|
| 0.1.0 | YYYY-MM-DD | RyuheiKiso | 初版 |
```

### 4-2. 版数と識別子の対応

- ドキュメント版数とソフトウェア版数は同期させる（ドキュメント先行のため、ドキュメントが先に上がる）。
- `06_実装/` 全体の版数は `ビルド・デプロイ/README.md` の `09_ビルド手順.md` 参照先で管理する。

## 5. 緊急変更手順

### 5-1. 緊急変更（hotfix）の定義

セキュリティ脆弱性・本番データ破損・SLO 違反（可用性 99.5% 未達等）を引き起こす障害の修正を緊急変更と定義する。

### 5-2. hotfix ブランチ手順

1. `main`（または `v{N.N.x}` ブランチ）から `hotfix/{PROB-NNN}-{short-description}` ブランチを切る。
2. 最小限の修正をコミットする。コミットメッセージに `fix(hotfix): ... PROB-NNN: {問題管理 ID}` を含める。
3. セルフレビュー（CR-NNN）を実施してから `main` にマージする。
4. パッチバージョン（0.x.y → 0.x.(y+1)）を上げて Git タグを付与する。

### 5-3. 事後 ADR 記録

緊急変更が技術的な設計判断を伴う場合（ライブラリバージョンの緊急ピン止め・回避策の実装等）は、**マージ後 2 営業日以内** に ADR-IMPL-NNN を起票して事後記録する。

## 6. 版数履歴テンプレ

各 md ファイルに設ける版数履歴節の標準形式を示す。

```markdown
| 版 | 日付 | 変更者 | 変更内容 |
|---|---|---|---|
| 0.1.0 | 2026-05-17 | RyuheiKiso | 初版 |
| 0.2.0 | YYYY-MM-DD | RyuheiKiso | IMPL-001〜005 確定・TST-unit-BE-001〜005 通過 |
| 0.2.1 | YYYY-MM-DD | RyuheiKiso | PROB-001 修正（ハッシュチェーン計算バグ）|
| 1.0.0 | YYYY-MM-DD | RyuheiKiso | 全 31 IMPL 確定・全 116 TST 通過・初版リリース |
```

---

**本節で確定した方針**

- **06_実装 フェーズは SemVer 0.x.x → 1.0.0 で管理し、1.0.0 は全 IMPL-NNN 確定済み・全 TST-NNN 通過・本番デプロイ完了時に付与する。**
- **変更管理は Conventional Commits ベースのコミットメッセージと change_log.md の二重管理とする。**
- **詳細設計逸脱は ADR-IMPL-NNN 必須とし、本付録/03 の継承種別を更新する。**
- **緊急変更（hotfix）は事後 2 営業日以内に ADR-IMPL-NNN を起票して記録する。**
