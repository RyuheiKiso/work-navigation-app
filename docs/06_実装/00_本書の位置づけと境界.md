# 00 本書の位置づけと境界

## 1. IPA 共通フレーム 2013 における本書の位置

IPA 共通フレーム 2013（SLCP-JCF 2013）の開発プロセスにおいて、本書は **6.3.6 ソフトウェア構築プロセス**（Software Construction Process）の定義文書として位置づける。

### 6.3.6 ソフトウェア構築プロセスのタスク対応一覧

| タスク ID | タスク名 | 本書の対応章 |
|---|---|---|
| 6.3.6.1 | ソフトウェアコンポーネントの実装 | 01〜05 コーディング規約 |
| 6.3.6.2 | ソフトウェアユニットの検証 | 09 ビルド手順・07 Git 運用 |
| 6.3.6.3 | ソフトウェアのコード化 | 01〜05 コーディング規約 |
| 6.3.6.4 | 開発環境の構築 | 06 開発環境構築手順 |
| 6.3.6.5 | ソフトウェアのビルドとインテグレーション | 09 ビルド手順 |

本書はコーディングが開始される前に確定し、全開発活動の根拠となる。実装者（現時点では RyuheiKiso 個人）は本書の規約に従い、逸脱する場合は ADR（Architecture Decision Record）として記録する。

**本節で確定した方針**
- **本書は IPA 共通フレーム 2013「6.3.6 ソフトウェア構築プロセス」の実施根拠文書である。**
- **コーディング規約・開発環境・Git 運用・ビルド手順の全章を本書が一元管理する。**
- **逸脱は ADR-IMPL-NNN として記録することを義務とし、サイレントな逸脱を禁止する。**

### アーキテクチャ不変原則（変更禁止）

以下の 5 原則は `src/CLAUDE.md` に記載された不変原則であり、本書のいかなる規約もこれらに違反できない。

1. **Offline-First**: タブレット端末は 4 段階ネットワーク管理、ローカルが一次記録メモリ
2. **Append-only Event Sourcing**: `work_events` テーブルへの UPDATE/DELETE は DB ロール権限で物理禁止
3. **Idempotent API**: `Idempotency-Key`（UUID v4）で全作業ログ受信エンドポイントを保護
4. **SHA-256 ハッシュチェーン**: `event_N.hash = SHA256(prev_hash || canonical_payload)`
5. **Local-First**: `server_received_at` はサーバー側で付与、クライアント上書き禁止

**本節で確定した方針**
- **05_詳細設計の 31MOD・35TBL・39API・86FR・35 画面は本書で変更しない。**
- **採用スタックおよびアーキテクチャ 5 原則は本書の全章を通じて不変とする。**
- **上流固定値への変更要求が発生した場合は CR（Change Request）を起票し上流文書を改訂する。**

## 3. 下流文書との分担

本書（06_実装）は実装規約とビルド手順を確定する。下流文書は本書が確定した内容を入力として受け取り、各文書固有の範囲のみを扱う。

| 文書 | 担当範囲 | 本書からのハンドオフ内容 |
|---|---|---|
| `07_テスト` | テスト仕様書・テスト結果・品質メトリクス | コーディング規約（テスト要件）、ビルド手順（テスト実行コマンド）、Git 運用（PR マージ条件としての CI グリーン要件） |
| `08_移行` | データ移行手順・環境移行手順 | 開発環境構築手順（§ 6 PostgreSQL セットアップ）、ビルド手順（成果物一覧と検証方法） |
| `09_運用・保守` | 運用手順書・障害対応手順・監視設定 | 開発環境の環境変数一覧（`§ 10 .env テンプレ`）、ログ規約（構造化 JSON・相関 ID・PII マスキング）、ビルド再現性の確保方法 |

### ハンドオフ境界の明確化

本書が**確定する**もの：
- コーディング規約（01〜05 章）の内容
- 開発環境の構成手順（06 章）
- Git ブランチ規約とリリース管理方式（07 章）
- IDE 設定の標準（08 章）
- ビルドコマンドと成果物の検証方法（09 章）

本書が**確定しない**もの（下流文書が扱う）：
- テストケースの詳細（07_テスト が担当）
- 本番環境への適用手順（08_移行 が担当）
- 障害発生時の対応フロー（09_運用・保守 が担当）
- セキュリティ診断の実施（07_テスト または 09_運用・保守 が担当）

**本節で確定した方針**
- **コーディング規約・開発環境・Git 運用・ビルド手順が本書の担当範囲であり、テスト仕様・移行手順・運用手順は下流文書が担当する。**
- **下流文書が本書の章を参照する場合は章番号とセクション番号で引用する（例: `06_実装/04_コーディング規約_SQL.md §9`）。**
- **本書が確定した内容の変更は本書を改訂することで行い、下流文書を先に変更することを禁止する。**

## 4. src/ との関係

`src/` 配下の各 CLAUDE.md は「暫定的なコーディング規約の権威」として機能していたが、本書の整備によってその役割は本書（01〜05 章のコーディング規約）に移管される。

### 権威の移管マップ

| src/ の CLAUDE.md | 移管先（本書の章） | 移管後の src/ CLAUDE.md の扱い |
|---|---|---|
| `src/CLAUDE.md`（横断共通規約） | `01_コーディング規約_共通.md` が権威化 | 本書へのリンクに差し替え（暫定フラグを除去） |
| `src/backend/CLAUDE.md`（Rust 規約） | `02_コーディング規約_Rust.md` が権威化 | 本書へのリンクに差し替え |
| `src/frontend/CLAUDE.md`（フロントエンド共通） | `03_コーディング規約_TypeScript.md` が権威化 | 本書へのリンクに差し替え |
| `src/frontend/handy/CLAUDE.md`（RN 規約） | `03_コーディング規約_TypeScript.md §8` が権威化 | 本書へのリンクに差し替え |
| `src/frontend/master/CLAUDE.md`（React 規約） | `03_コーディング規約_TypeScript.md §7 §9` が権威化 | 本書へのリンクに差し替え |

### 精緻化の方針

`src/CLAUDE.md` が定めた不変原則・コメント規約・多言語対応などの方針は、本書の各章でより詳細な実装レベルの規約として精緻化される。src 側の記述は「方針レベル」であり、本書が「実装レベルの権威」となる。

例えば `src/CLAUDE.md` に記載の「日本語コメント・Why 重視」方針は、本書 `01_コーディング規約_共通.md §3` にて rustdoc・JSDoc のフォーマット例・自明コメント禁止リスト・ブロックコメントの必須要件まで精緻化される。

**本節で確定した方針**
- **本書整備後、`src/` 配下の CLAUDE.md はコーディング規約を直接記載せず、本書の対応章へのリンクを持つ参照文書となる。**
- **src/ の暫定規約と本書に矛盾が生じた場合は本書が優先される。**
- **矛盾の解消は本書の改訂によって行い、src/ CLAUDE.md の更新は本書改訂後に実施する。**

## 5. 本章の前提制約

本書全体に適用される前提制約を以下に列挙する。すべてのコーディング判断はこれらの制約を満足したうえで行われなければならない。

### 倫理的制約

- **個人別労務監視禁止**: 個人別生産性ランキング・休憩回数計測・移動距離追跡等の機能を実装してはならない。この制約は RBAC 6 ロールおよびデータモデルに技術的に埋め込まれ、設定変更では解除できない構造とする。
- **ALCOA+ 原則の遵守**: ALCOA+（Attributable, Legible, Contemporaneous, Original, Accurate, Complete, Consistent, Enduring, Available）の全属性を技術的に保証する実装を行う。

### セキュリティ制約

- **最小権限原則**: DB ロール 3 分離（`app_write` / `app_event_insert` / `app_read`）を実装の全期間で維持する。
- **シークレットハードコード禁止**: JWT 秘密鍵・DB パスワード等を `src/` 配下のいかなるファイルにも記載しない。
- **OWASP Top 10 準拠**: 全入出力に OWASP Top 10 の対策を実施する。

### 品質制約

- **テストなきコードは未完成**: ユニット・統合・E2E の 3 層テストが存在しないコードは完成とみなさない。
- **型安全性の強制**: TypeScript の `any` 型使用禁止、Rust の `unsafe` 禁止を全コードで徹底する。
- **CI グリーン必須**: `cargo fmt --check`・`cargo clippy --deny warnings`・`pnpm eslint`・テスト全通過が PR マージの必須条件である。

### プロジェクトポリシー

ルート `CLAUDE.md` に定められた「**妥協は禁止する。完璧かつ至高を目指す。**」は、本書の全章に適用される。この方針は品質への妥協を構造的に防止するための宣言であり、全実装判断の基準となる。

**本節で確定した方針**
- **倫理的制約（個人別労務監視禁止）は RBAC とデータモデルで技術的に実施し、設定変更では解除できない構造とする。**
- **CI グリーン（fmt + clippy + eslint + テスト全通過）を PR マージの必須条件とする。**
- **「妥協は禁止する」の方針は品質・セキュリティ・倫理の全側面に適用される。**
