# 18 実装判断ADR索引

本章は実装フェーズにおけるアーキテクチャ決定記録（ADR）の索引・テンプレート・プロセスを定める。詳細設計からの逸脱を伴う実装判断と、詳細設計に未記載の補足判断を記録し、将来の参照可能性を確保する。

---

## 1. ADR の位置づけ

本章で管理する ADR（`ADR-IMPL-NNN`）は以下の 2 種類の判断を対象とする。

| 種類 | 定義 | 例 |
|---|---|---|
| 設計逸脱判断 | `docs/05_詳細設計/` に記載された設計から逸脱した実装判断 | 詳細設計では `tower-governor` でレートリミットを実装するとされていたが、パフォーマンス評価の結果 `axum-limit` に変更した |
| 補足判断 | 詳細設計に記載がなく、実装時に新たに発生した設計判断 | SQLite のジャーナルモードを WAL に設定することを決定した（詳細設計に未記載） |

本章の ADR と上流 ADR（`docs/04_概要設計/付録/05_設計判断ADR索引.md`）の関係は以下の通りである。

```
上流 ADR（04_概要設計）: アーキテクチャ・技術スタックの大きな決定
  ↓ 継承
実装 ADR（06_実装 / 本章）: 実装時の細かい判断・設計逸脱の記録
```

実装 ADR は上流 ADR を覆さない。上流 ADR を覆す判断が必要な場合は上流ドキュメントの改訂を行い、上流 ADR を更新する。

**本節で確定した方針**
- **実装 ADR のスコープを設計逸脱・補足判断に限定する**: 上流 ADR の内容を本章で再定義しない
- **実装 ADR は上流 ADR を継承するものとする**: 上流 ADR と矛盾する場合は上流 ADR の改訂を行う
- **詳細設計に記載がない重要な実装判断は全て ADR-IMPL-NNN として記録する**: 判断の根拠を後から追跡可能にする

---

## 2. ADR テンプレ

`src/CLAUDE.md` の簡易フォーマットをベースに、実装フェーズの特性を反映したテンプレートを定義する。ADR は `docs/06_実装/ADR/ADR-IMPL-NNN_タイトル.md` に記録する。

```markdown
# ADR-IMPL-NNN: タイトル（判断の内容を端的に表す）

日付: YYYY-MM-DD
状態: 提案 / 確定 / 廃止
提案者: RyuheiKiso

## 背景

なぜこの判断が必要になったか。
- 詳細設計の記述: （05_詳細設計/ の該当箇所を引用する）
- 実装で発生した問題・制約: （具体的な状況を記述する）

## 決定

何を決めたか（一文で明確に記述する）。

## 理由

なぜこの選択をしたか。

### 採用した選択肢の根拠

（採用した選択肢のメリットを列挙する）

### 却下した代替案

| 代替案 | 却下理由 |
|---|---|
| （代替案 A） | （却下理由） |
| （代替案 B） | （却下理由） |

## 影響

この判断がもたらす制約・副作用・リスク。

- 影響範囲: IMPL-NNN（影響する実装モジュール）
- 追加のテスト: TST-NNN（追加が必要なテスト）
- ドキュメント更新: （更新が必要なドキュメント）

## 参照

- 関連 ADR: ADR-IMPL-NNN（または 04_概要設計の ADR）
- 関連 PROB-NNN: PROB-NNN（この判断に至った問題）
- 関連 FR-NNN: FR-NNN（この判断が影響する機能要件）
```

**本節で確定した方針**
- **ADR テンプレートを固定してフォーマットを統一する**: 参照性を確保する
- **却下した代替案を必ず記録する**: 同じ代替案を繰り返し検討しないようにする
- **影響範囲（IMPL-NNN / TST-NNN）を明示する**: ADR の変更が何に影響するかを追跡可能にする

## 3. ADR の状態遷移

```
提案
  ↓ 7 日間のクーリング期間
確定（または廃止）
```

| 状態 | 意味 | 変更条件 |
|---|---|---|
| 提案 | ADR を起票したが 7 日間のクーリング期間中 | — |
| 確定 | クーリング期間後に判断を確定した | クーリング期間（7 日）の満了 + 自己承認 |
| 廃止 | より良い判断に置き換えられた、または不要になった | 後継の ADR-IMPL-NNN に置き換えられた場合 |

廃止された ADR は削除しない。ファイルを保持し、状態を「廃止」に変更して廃止理由と後継 ADR-IMPL-NNN への参照を追記する。

**本節で確定した方針**
- **状態遷移を「提案 → 確定」または「提案 → 廃止」に限定する**: 中間状態を設けない
- **廃止 ADR を削除しない**: 過去の判断の履歴を保持する
- **「確定」状態にのみ実装を依存させる**: 「提案」状態の ADR に基づく実装は仮実装として扱う

---

## 4. 承認プロセス

個人開発のため、ADR の承認は自己承認とする。ただし、判断の熟慮を保証するために 7 日間のクーリング期間を設ける。

| ステップ | 内容 | 期間 |
|---|---|---|
| 1. ADR 起票 | `状態: 提案` でファイルを作成する | 即時 |
| 2. クーリング期間 | 7 日間置いて判断を熟慮する。この間は「提案」状態を変更しない | 7 日間 |
| 3. 自己レビュー | 7 日後に判断が依然として正しいかを再評価する | — |
| 4. 確定 | `状態: 確定` に変更して実装に正式採用する | 即時 |

### クーリング期間を設ける理由

個人開発では「書いた直後は最善に見えるが、翌週には問題が見える」判断が発生しやすい。7 日間の間隔を設けることで、実装の熱量が冷めた状態での再評価を強制する。これは「妥協は禁止する。完璧かつ至高を目指す」ポリシーを判断プロセスに実装したものである。

### 緊急判断の取り扱い

Critical の PROB-NNN への対処など、7 日間待てない場合は以下のプロセスで即時確定できる。

```markdown
## 緊急確定の理由（通常は使用しない）
緊急確定日: YYYY-MM-DD
緊急確定理由: PROB-NNN（Critical）への対処が必要なため即時確定
事後レビュー予定日: YYYY-MM-DD（7 日後）
```

緊急確定後の 7 日以内に事後レビューを実施し、判断を維持するか廃止するかを決定する。

**本節で確定した方針**
- **自己承認に 7 日間のクーリング期間を義務付ける**: 衝動的な設計判断を防止する
- **緊急確定は Critical PROB-NNN への対処時のみ許可する**: 例外を厳しく限定する
- **緊急確定後は 7 日以内に事後レビューを実施する**: 緊急確定を正式な手続きの省略として使わない

## 5. 上流 ADR との関係

上流 ADR（`docs/04_概要設計/付録/05_設計判断ADR索引.md`）と本章（06_実装）の差分管理方針を定める。

| ケース | 対処 |
|---|---|
| 上流 ADR の決定を実装で踏襲する | ADR-IMPL-NNN は不要（上流 ADR を参照するコメントをコードに追記する） |
| 上流 ADR の決定を実装で補足する | ADR-IMPL-NNN を起票（`参照` フィールドに上流 ADR を記載） |
| 上流 ADR の決定と異なる実装が必要になった | 上流 ADR の改訂を先に行い、改訂 ADR に基づいて ADR-IMPL-NNN を起票する |
| 上流 ADR に記載がない新たな判断が必要になった | ADR-IMPL-NNN を起票する |

```rust
// 上流 ADR を踏襲したコードへのコメント例
// docs/04_概要設計/付録/05_設計判断ADR索引.md ADR-ARCH-003 に準拠して
// Idempotency-Key のキャッシュ TTL を 24h に設定する
const IDEMPOTENCY_TTL_SECS: u64 = 86_400;
```

**本節で確定した方針**
- **上流 ADR を踏襲する場合は ADR-IMPL-NNN を起票しない**: 不要な ADR の増殖を防止する
- **上流 ADR との関係を `参照` フィールドで明示する**: ADR 間の依存関係を追跡可能にする
- **上流 ADR と異なる判断は上流 ADR の改訂から始める**: 実装 ADR が上流 ADR を無効化することを禁止する

## 6. ADR 索引表

実装フェーズで起票した ADR-IMPL-NNN を以下の索引で管理する。現時点は空（実装着手前）。

| ADR-IMPL-NNN | 日付 | タイトル | 状態 | 関連 IMPL | 関連 PROB |
|---|---|---|---|---|---|
| （実装着手後に記録する） | — | — | — | — | — |

### 索引の更新タイミング

| トリガー | 更新内容 |
|---|---|
| ADR-IMPL-NNN 起票時 | 索引表に行を追加する（状態: 提案） |
| クーリング期間満了・確定時 | 状態を「確定」に更新する |
| ADR-IMPL-NNN 廃止時 | 状態を「廃止」に更新して廃止理由を括弧書きで追記する |
| 実装フェーズ完了時 | 索引全体を棚卸しして漏れがないことを確認する |

```bash
# ADR-IMPL-NNN のファイル一覧を確認する
ls docs/06_実装/ADR/ 2>/dev/null || echo "ADR ディレクトリが未作成です"

# 状態が「提案」の ADR を確認する（クーリング期間の管理）
grep -r "状態: 提案" docs/06_実装/ADR/ 2>/dev/null
```

**本節で確定した方針**
- **索引表を唯一の ADR 一覧とする**: ADR ファイルを直接探す必要がない構造にする
- **索引表は ADR 起票と同時に更新する**: ファイル作成と索引更新を同一コミットで行う
- **実装フェーズ完了時に索引を棚卸しする**: 索引漏れ・状態の不整合を解消する
