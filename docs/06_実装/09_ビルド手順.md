# 09 ビルド手順

## 1. ビルド全体マップ

図: fig_build_pipeline_overall（img/ 配下）を参照

本プロジェクトは 4 つのサブシステムで構成され、それぞれが独立してビルド可能である。

### 4 サブシステムの並列ビルド

| サブシステム | ビルドツール | 成果物 | 所要時間（目安） |
|---|---|---|---|
| backend（Rust） | `cargo build --release` | `wnav-api` バイナリ + Docker イメージ | 初回 5〜10 分・増分 1〜2 分 |
| handy（React Native） | `eas build` | Android APK / iOS IPA / Windows MSIX | 初回 30〜60 分（EAS クラウド） |
| master（React SPA） | `pnpm build`（Vite） | `dist/` 静的ファイル | 初回 2〜3 分・増分 30 秒 |
| infra（Docker/nginx/IIS） | `docker build` / `nginx -t` / PowerShell | Docker イメージ・設定ファイル | 5〜10 分 |

### ビルド依存関係

```
backend ビルド
  └─ .sqlx/ キャッシュ（sqlx prepare で生成）
  └─ openapi.json（utoipa から生成） ──→ master ビルドの API クライアント生成

handy ビルド（独立）

master ビルド
  └─ openapi.json（backend から生成）

infra ビルド
  └─ backend Docker イメージ
```

**本節で確定した方針**
- **4 サブシステムは原則として並列ビルド可能とし、依存がある場合のみ直列実行する。**
- **`openapi.json` の生成は backend ビルドに含め、master がビルド前に backend の OpenAPI スキーマを参照できるようにする。**
- **CI では backend → master の順序でビルドし、型安全な API クライアントを保証する。**

## 2. backend ビルド

### リリースビルド

```bash
# WSL2 上で実行する
cd /home/ryuhei_kiso/github/work-navigation-app/src/backend

# SQLX_OFFLINE=true を設定してビルドする（CI では DB 接続不要）
SQLX_OFFLINE=true cargo build --release --target x86_64-unknown-linux-gnu

# ビルド確認
ls -lh target/release/wnav-api
# -rwxr-xr-x 1 user user X.XM ... wnav-api
```

### SQLX_OFFLINE モードの準備

CI でのビルドに使用する `.sqlx/` キャッシュを事前に生成する。

```bash
# DB が起動していることを確認する
docker compose up -d postgres

# sqlx prepare でクエリキャッシュを生成する
DATABASE_URL="postgres://wnav_write:dev_password_write@localhost:5432/wnav_dev" \
    sqlx prepare --workspace

# .sqlx/ ディレクトリをバージョン管理に含める
git add .sqlx/
git commit -m "chore: sqlx クエリキャッシュを更新する"
```

### Docker イメージのビルド

```bash
# マルチステージ Dockerfile でビルドする
docker build \
    --tag wnav-api:latest \
    --tag wnav-api:$(git describe --tags --always) \
    --file src/backend/Dockerfile \
    .

# イメージの確認
docker image inspect wnav-api:latest | jq '.[0].Config.User'
# "65532:65532"（nonroot ユーザーであること）

# ローカルでのヘルスチェック確認
docker run --rm -e WNAV_BE_LISTEN_ADDR=0.0.0.0:8080 -p 8080:8080 wnav-api:latest &
curl -s http://localhost:8080/healthz | jq .
```

### SBOM 生成

```bash
# cargo-sbom で SBOM（Software Bill of Materials）を生成する
cargo sbom > sbom-backend.json

# SBOM の内容確認（依存クレートの一覧）
jq '.components | length' sbom-backend.json
```

### 成果物の検証

```bash
# SHA-256 ハッシュの生成（成果物の改竄検出用）
sha256sum target/release/wnav-api > target/release/wnav-api.sha256
cat target/release/wnav-api.sha256

# Docker イメージのダイジェスト確認
docker image inspect wnav-api:latest | jq '.[0].Id'
```

**本節で確定した方針**
- **CI では `SQLX_OFFLINE=true` を設定し、`.sqlx/` キャッシュから SQL を検証する。**
- **Docker イメージは nonroot ユーザー（UID 65532）で実行し、root 実行を禁止する。**
- **SBOM を `sbom-backend.json` として生成し、リリース成果物に含める。**

## 3. handy ビルド

図: fig_build_pipeline_handy（img/ 配下）を参照

### prebuild（ネイティブコードの生成）

```bash
cd /home/ryuhei_kiso/github/work-navigation-app/src/frontend/handy

# expo prebuild でネイティブプロジェクトを生成する（--clean で以前の生成を削除）
expo prebuild --clean

# 生成確認
ls android/  # Android プロジェクト
ls ios/      # iOS プロジェクト（macOS のみ）
ls windows/  # Windows プロジェクト（react-native-windows）
```

### EAS Build（Android）

```bash
# プロダクションビルド（EAS クラウドでビルドする）
eas build \
    --platform android \
    --profile production \
    --non-interactive

# ビルドステータスの確認
eas build:list --platform android --status finished --limit 5

# ビルド成果物のダウンロード
eas build:download --id <build-id>
```

### EAS Build（iOS）

```bash
# iOS ビルドは macOS 環境または EAS クラウドが必要
eas build \
    --platform ios \
    --profile production \
    --non-interactive

# ビルドの確認
eas build:list --platform ios --status finished --limit 5
```

### EAS Build（Windows - react-native-windows）

```bash
# Windows タブレット向けビルド
eas build \
    --platform windows \
    --profile production \
    --non-interactive
```

### SBOM 生成（handy）

```bash
# pnpm で SBOM を生成する
pnpm sbom

# または npm audit の出力を使用する
pnpm audit --json > sbom-handy.json
```

**本節で確定した方針**
- **`expo prebuild --clean` を使用してネイティブプロジェクトを毎回クリーン生成し、ビルドの再現性を保証する。**
- **EAS Build のプロダクションビルドは `--non-interactive` フラグで CI から実行できるようにする。**
- **3 プラットフォーム（Android/iOS/Windows）の全ビルドが成功することをリリース条件とする。**

## 4. master ビルド

図: fig_build_pipeline_master（img/ 配下）を参照

### Vite ビルド

```bash
cd /home/ryuhei_kiso/github/work-navigation-app/src/frontend/master

# プロダクションビルド
pnpm build

# ビルド確認
ls -lh dist/
# dist/index.html
# dist/assets/

# バンドルサイズの確認（500KB を目安とする）
du -sh dist/
find dist -name "*.js" -exec du -sh {} + | sort -h
```

### OpenAPI クライアント再生成

```bash
# backend のビルド後に OpenAPI スキーマを使用してクライアントを再生成する
# backend を起動して openapi.json を取得する
cargo run &
sleep 3
curl -s http://localhost:8080/api/openapi.json > /tmp/openapi.json

# openapi-generator-cli でクライアントを生成する
openapi-generator-cli generate \
    -i /tmp/openapi.json \
    -g typescript-fetch \
    -o src/api/generated/ \
    --additional-properties=supportsES6=true,typescriptThreePlus=true

# 生成されたファイルをバージョン管理に含める
git add src/api/generated/
```

### ビルド確認

```bash
# プレビューサーバーで静的ファイルを確認する
pnpm preview
# http://localhost:4173 でアクセスして動作確認する
```

**本節で確定した方針**
- **`openapi-generator-cli` で TypeScript クライアントを自動生成し、手動実装を禁止する。**
- **生成された `src/api/generated/` をバージョン管理に含め、ビルドのたびに差分を確認する。**
- **`dist/` のバンドルサイズを定期的に確認し、500KB を超えた場合はコード分割を検討する。**

## 5. infra ビルド

### PostgreSQL Docker イメージ

```bash
# カスタム PostgreSQL イメージのビルド（初期化スクリプト含む）
docker build \
    --tag wnav-postgres:latest \
    --tag wnav-postgres:17 \
    --file infra/database/Dockerfile \
    .

# イメージの確認
docker image inspect wnav-postgres:latest

# ローカルでの起動確認
docker run --rm \
    -e POSTGRES_DB=wnav_test \
    -e POSTGRES_PASSWORD=test_password \
    -p 5432:5432 \
    wnav-postgres:latest
```

### nginx 設定検証

```bash
# nginx の設定ファイルを検証する（構文チェック）
docker run --rm \
    -v "$(pwd)/infra/nginx:/etc/nginx/conf.d:ro" \
    nginx:latest \
    nginx -t -c /etc/nginx/nginx.conf

# 成功時の出力:
# nginx: the configuration file /etc/nginx/nginx.conf syntax is ok
# nginx: configuration file /etc/nginx/nginx.conf test is successful
```

### IIS 設定スクリプト検証

```powershell
# PowerShell で DryRun モードを使用して検証する
# CI 環境（IIS がないため DryRun で構文と冪等性を確認する）
pwsh -NonInteractive -File .\infra\iis\setup.ps1 -DryRun

# 出力例:
# DryRun モード: 実際の変更は行わない
# Set-WNavAppPool: WNavMasterPool を作成する予定
# Set-WNavSite: WNavMaster を作成する予定
```

**本節で確定した方針**
- **PostgreSQL のカスタム Docker イメージにロール作成・拡張有効化の初期化スクリプトを含める。**
- **nginx の設定は `nginx -t` で構文チェックを CI に組み込む。**
- **IIS 設定スクリプトは `-DryRun` スイッチで CI 環境での構文・冪等性検証を可能にする。**

## 6. 成果物の検証

### SHA-256 ハッシュ検証

```bash
# 成果物のハッシュを生成する
sha256sum target/release/wnav-api > checksums.txt
sha256sum dist/index.html >> checksums.txt

# ハッシュの検証
sha256sum --check checksums.txt
```

### cosign による署名検証

```bash
# cosign で成果物に署名する（リリース時のみ）
cosign sign-blob \
    --key cosign.key \
    --output-signature wnav-api.sig \
    target/release/wnav-api

# 署名の検証
cosign verify-blob \
    --key cosign.pub \
    --signature wnav-api.sig \
    target/release/wnav-api

echo "署名検証: OK"
```

### Docker イメージのスキャン

```bash
# Trivy で Docker イメージの脆弱性スキャンを実行する
trivy image \
    --exit-code 1 \
    --severity HIGH,CRITICAL \
    wnav-api:latest

# 結果の確認（HIGH/CRITICAL が 0 件であること）
echo "脆弱性スキャン: $?"
```

**本節で確定した方針**
- **全成果物の SHA-256 ハッシュを `checksums.txt` に記録し、配布前に完全性を検証する。**
- **リリースビルドの Docker イメージを Trivy でスキャンし、HIGH/CRITICAL の脆弱性が 0 件であることを必須とする。**
- **`cosign` による署名をリリース成果物の標準として採用し、未署名の成果物を本番配置禁止とする。**

## 7. ビルド再現性

### Cargo.lock の管理

```bash
# Cargo.lock を必ずバージョン管理に含める（ライブラリの場合でも必須）
git add Cargo.lock
git commit -m "chore: Cargo.lock を更新する"

# ロックファイルが存在することを CI で確認する
if [ ! -f "src/backend/Cargo.lock" ]; then
    echo "ERROR: Cargo.lock が存在しない" >&2
    exit 1
fi
```

### pnpm-lock.yaml の管理

```bash
# pnpm-lock.yaml をバージョン管理に含める
git add pnpm-lock.yaml
git commit -m "chore: pnpm-lock.yaml を更新する"

# ロックファイルが存在することを CI で確認する
if [ ! -f "pnpm-lock.yaml" ]; then
    echo "ERROR: pnpm-lock.yaml が存在しない" >&2
    exit 1
fi
```

### Docker BuildKit によるビルドプロベナンス

```bash
# --provenance=true でビルドプロベナンスを生成する（ビルドの入力・出力を記録）
docker buildx build \
    --provenance=true \
    --sbom=true \
    --tag wnav-api:latest \
    --push \
    src/backend/

# プロベナンスの確認
docker buildx imagetools inspect wnav-api:latest
```

**本節で確定した方針**
- **`Cargo.lock` / `pnpm-lock.yaml` を必ずバージョン管理に含め、再現可能なビルドを保証する。**
- **`docker buildx build --provenance=true` でビルドプロベナンスを生成し、サプライチェーン攻撃を検出可能にする。**
- **CI でロックファイルの存在を確認し、ロックファイルなしのビルドを禁止する。**

## 8. ビルド失敗時の対処

### よくあるエラーと対処法

#### DATABASE_URL 未設定エラー

```
error: `DATABASE_URL` must be set to use the `sqlx::query!` macro
```

```bash
# 対処: .env を確認し、DATABASE_URL を設定する
cat .env | grep DATABASE_URL

# または SQLX_OFFLINE=true を設定してオフラインモードを使用する
SQLX_OFFLINE=true cargo build
```

#### sqlx offline mode エラー（`.sqlx/` キャッシュが古い）

```
error: failed to find data for query
  = note: this can happen if the compile-time type-checking cache (`.sqlx`) is out of date
```

```bash
# 対処: sqlx prepare を再実行して .sqlx/ を更新する
DATABASE_URL="postgres://wnav_write:...@localhost:5432/wnav_dev" \
    sqlx prepare --workspace

# .sqlx/ をコミットする
git add .sqlx/
git commit -m "chore: sqlx クエリキャッシュを更新する"
```

#### EAS 認証エラー

```
Error: Not logged in
```

```bash
# 対処: EAS CLI でログインする
eas login

# CI 環境の場合: EAS_TOKEN 環境変数を設定する
export EAS_TOKEN="your-eas-access-token"
eas build --platform android --profile production --non-interactive
```

#### Rust コンパイルエラー（OOM）

```
error: could not compile `wnav-api`
  = note: process killed by SIGKILL
```

```bash
# 対処 1: .wslconfig でメモリ上限を増やす（WSL2 の場合）
# memory=12GB

# 対処 2: 並列ジョブ数を減らす
cargo build --release -j 4  # デフォルトは CPU コア数

# 対処 3: リンカの変更（lld の使用）
# .cargo/config.toml に追加:
# [target.x86_64-unknown-linux-gnu]
# linker = "clang"
# rustflags = ["-C", "link-arg=-fuse-ld=lld"]
```

#### pnpm install エラー（Node.js バージョン不一致）

```
ERR_INVALID_URL: Invalid URL
```

```bash
# 対処: Volta で正しい Node.js バージョンを使用しているか確認する
volta list
node --version  # v20.x.x であること

# プロジェクトの package.json に volta フィールドを確認する
cat package.json | jq '.volta'
```

**本節で確定した方針**
- **ビルドエラーのトラブルシューティング手順をこの章に集約し、エラー解決の初動コストを削減する。**
- **`SQLX_OFFLINE=true` と `.sqlx/` キャッシュの更新手順を明示し、CI での SQL エラーを防止する。**
- **EAS の CI ビルドには `EAS_TOKEN` 環境変数を使用し、対話的なログインを排除する。**
