# 17 実装問題管理ログ

本章は実装フェーズで発生した問題を IPA 共通フレーム 2013「支援プロセス 6.7 問題解決プロセス」に準拠して管理するための方針・プロセス・テンプレートを定める。

---

## 1. 問題管理プロセス

問題解決プロセスは IPA 共通フレーム 2013 の 6.7 節（問題解決プロセス）の以下のタスクに準拠する。

| プロセスタスク | 本プロジェクトでの実施方法 |
|---|---|
| 6.7.1 問題の識別 | 実装中・レビュー中・テスト中に発見した問題を PROB-NNN として登録する |
| 6.7.2 問題の記録 | 本章のテンプレートに従って記録する |
| 6.7.3 問題の分析 | 現象・再現手順・原因を調査して記録する |
| 6.7.4 問題の解決 | 対処を実施してテストで確認する |
| 6.7.5 問題クローズ | クローズ条件（§ 6）を満たした時点でクローズする |
| 6.7.6 問題の傾向分析 | 月次集計（§ 7）でパターンと傾向を把握する |

問題管理の流れ:

```
問題発見
  ↓ 24 時間以内
PROB-NNN 登録（種別・Severity・現象・再現手順を記載）
  ↓
原因調査（5 Why 法・コードレビュー記録 CR-NNN との連携）
  ↓
対処実施（コード修正・設計変更・ドキュメント更新）
  ↓
テスト通過（TST-NNN に対応する単体・統合テストが通過すること）
  ↓
CR-NNN クローズ（コードレビュー記録で対処確認）
  ↓
PROB-NNN クローズ（クローズ日を記録する）
```

**本節で確定した方針**
- **問題管理は IPA CF 2013 問題解決プロセスに準拠する**: 個人開発でも形式的な問題管理プロセスを維持する
- **問題発見から 24 時間以内に PROB-NNN を登録する**: 発見した問題を放置しない
- **クローズ条件（§ 6）を満たすまで PROB-NNN をクローズしない**: 対処確認を省略しない

---

## 2. PROB-NNN 採番規約

| 規約項目 | ルール |
|---|---|
| 形式 | `PROB-` + 3 桁ゼロ埋め（例: `PROB-001`, `PROB-042`） |
| 採番台帳 | `docs/06_実装/付録/99_採番台帳.md` が権威（欠番禁止） |
| 連番 | PROB-001 から始めて連続番号とする。廃止しても番号は再利用しない |
| 欠番 | 欠番は禁止する。登録した PROB-NNN は必ず管理し続ける |
| スコープ | 実装フェーズで発見した問題に限定する。要件フェーズの問題は別管理 |

```bash
# 次に使用可能な PROB-NNN を確認する
grep -h "^# PROB-" docs/06_実装/問題ログ/*.md | sort | tail -1
```

**本節で確定した方針**
- **欠番禁止を厳守する**: 採番台帳を権威として全 PROB-NNN を追跡する
- **廃止された PROB-NNN も番号を保持する**: 採番台帳に廃止理由を記録する
- **実装フェーズ外の問題は別管理とする**: 問題ログのスコープを明確にする

## 3. 問題分類

| 種別コード | 名称 | 説明 | 例 |
|---|---|---|---|
| `BUG` | コードのバグ | 実装に起因する動作の誤り | ハッシュチェーンの prev_hash が NULL になる |
| `DESIGN` | 設計矛盾または設計不備 | 詳細設計と実装の間に矛盾が発生した | API 仕様と Rust の型定義が不一致 |
| `DOC` | ドキュメント不整合 | 実装とドキュメントの記述が食い違う | 本章 § 2 の IMPL-NNN が実際のファイルパスと異なる |
| `TOOL` | ツール・環境起因 | 開発ツール・CI・依存ライブラリに起因する問題 | sqlx が特定バージョンのコンパイルでパニックする |
| `SPEC` | 仕様曖昧・矛盾 | 上流要件または設計仕様が曖昧で実装判断に迷いが生じた | FR-045 の「スキップ記録」の定義が不明確 |

種別コードは問題ログの `種別:` フィールドに記載する。1 つの問題が複数種別にまたがる場合は主要な種別を記載し、副次的な種別をコメントに記す。

**本節で確定した方針**
- **5 種別（BUG / DESIGN / DOC / TOOL / SPEC）で問題を分類する**: 傾向分析（§ 7）で種別別の改善策を検討する
- **種別の追加は ADR-IMPL-NNN で記録する**: 種別体系の恣意的な変更を防止する
- **複数種別にまたがる問題は主要種別を選択する**: 分類のあいまいさを最小化する

---

## 4. Severity 分類

| Severity | 定義 | 対処期限 | 構築完了への影響 |
|---|---|---|---|
| Critical | 本番データ損失またはセキュリティ侵害リスクがある | 発見当日中に対処開始 | 解消まで構築完了を宣言しない |
| High | 主要機能（Offline-First / Append-only / Idempotent / ハッシュチェーン）が動作不能 | 3 日以内に対処 | 解消まで構築完了を宣言しない |
| Medium | 機能に制約はあるが回避策がある | 1 週間以内に対処 | 回避策と解消予定を記載すれば構築完了可 |
| Low | 軽微な不具合（表示崩れ・ログメッセージの誤字等） | 次回スプリント（またはタスク）で対処 | 件数と内容を 16 章に記録すれば構築完了可 |

Severity 判定のガイドライン:

- **Append-only 制約の違反 → Critical**: データ整合性の基本原則侵害
- **JWT 秘密鍵のログへの出力 → Critical**: シークレット漏洩リスク
- **Emergency Mode への遷移失敗 → High**: オフライン動作保証の不全
- **ページ遷移 600ms（目標 500ms） → Medium**: Glanceable 要件の軽微な超過
- **ログメッセージのタイポ → Low**: 機能への影響なし

**本節で確定した方針**
- **Critical / High は構築完了のブロッカーとする**: これらが未解消の状態でのリリースを禁止する
- **Severity の判定基準を事前に定義する**: 判定の恣意性を排除する
- **対処期限を Severity で決定する**: 後回しにしないための期限管理

## 5. 問題ログ構造（テンプレ）

問題は `docs/06_実装/問題ログ/PROB-NNN_タイトル.md` に記録する。

```markdown
# PROB-NNN: タイトル（問題を端的に表す）

日付: YYYY-MM-DD
種別: BUG / DESIGN / DOC / TOOL / SPEC
Severity: Critical / High / Medium / Low
検出フェーズ: 実装中 / コードレビュー（CR-NNN） / 単体テスト / 統合テスト
報告者: RyuheiKiso
状態: 調査中 / 対処中 / 検証中 / クローズ

## 現象

（問題の発現方法を具体的に記述する。スクリーンショット・ログ・エラーメッセージを含める）

## 再現手順

1. （前提条件: OSS など環境情報を含める）
2. （操作ステップ 1）
3. （操作ステップ 2）
4. （実際の結果）
5. （期待する結果）

## 原因

（調査の結果判明した根本原因を記述する。コードの該当箇所を引用する）

```rust
// 問題の根本原因となるコードの抜粋（実際のコードを引用する）
```

## 対処

（実施した修正内容を記述する。コミットハッシュまたは PR 番号を含める）

- 対処コミット: abc123def456
- 対処 PR: #NNN

## ADR 連携

（設計から逸脱した判断が生じた場合、ADR-IMPL-NNN を記録する）

- 逸脱なし（または ADR-IMPL-NNN を参照）

## クローズ条件

- [ ] 対処コードが実装されている（commit: ）
- [ ] 単体テストまたは統合テストで対処を確認した（TST-NNN）
- [ ] ドキュメントが最新の実装を反映している
- [ ] コードレビュー（CR-NNN）で対処を確認した

## クローズ日

YYYY-MM-DD（クローズ条件を全件達成した日付）
```

**本節で確定した方針**
- **再現手順を必ず記録する**: 再現できない問題の修正は確認できない
- **ADR 連携フィールドを設ける**: 設計逸脱を伴う問題対処を ADR-IMPL-NNN として記録する
- **クローズ条件をチェックリスト形式で管理する**: 条件を全件完了するまでクローズしない

---

## 6. クローズ条件

PROB-NNN をクローズするには以下の全条件を満たす必要がある。

```
✓ 対処（コード修正・設計変更・ドキュメント更新）が実施された
✓ 単体テストまたは統合テストで対処が検証された（TST-NNN で確認）
✓ 影響するドキュメントが最新の実装を反映している（15 章 ITM 更新含む）
✓ コードレビュー記録（CR-NNN）で対処を確認した（`[must]` コメントが解消された）
✓ 関連する ADR-IMPL-NNN が必要な場合は記録済みである
```

Critical / High の問題については上記に加えて「同種の問題が再発しない仕組みを実装した（テストの追加など）」を条件とする。

**本節で確定した方針**
- **クローズは全条件の達成を確認してから行う**: 対処実施だけでクローズしない
- **Critical / High には再発防止策を必須とする**: 同じ問題を繰り返さない
- **テスト通過をクローズ条件に含める**: 対処の効果を自動テストで保証する

---

## 7. 統計集計テンプレ

月次で問題の傾向を集計して改善活動に活かす。

```markdown
## YYYY-MM 月次問題集計

集計期間: YYYY-MM-01 〜 YYYY-MM-DD
集計者: RyuheiKiso

### Severity 別集計

| Severity | 新規 | クローズ | 繰越（月末時点残） |
|---|---|---|---|
| Critical | 0 | 0 | 0 |
| High | 0 | 0 | 0 |
| Medium | 0 | 0 | 0 |
| Low | 0 | 0 | 0 |
| **合計** | **0** | **0** | **0** |

### 種別別集計

| 種別 | 新規 | クローズ | 残 |
|---|---|---|---|
| BUG | 0 | 0 | 0 |
| DESIGN | 0 | 0 | 0 |
| DOC | 0 | 0 | 0 |
| TOOL | 0 | 0 | 0 |
| SPEC | 0 | 0 | 0 |

### 傾向分析と改善アクション

- 最も多い種別: （種別名）
- 繰越が多い Severity: （Severity 名）
- 改善アクション: （来月実施する改善策）

### 注目 PROB-NNN

- PROB-NNN: （月次レビューで特に注意すべき問題の要約）
```

**本節で確定した方針**
- **月次集計を実施して傾向を把握する**: 問題の放置を防ぎ、改善サイクルを維持する
- **種別別集計で根本的な問題領域を特定する**: BUG が多い場合はテストを強化、DESIGN が多い場合は設計見直しを検討する
- **繰越件数をトレンドで管理する**: 繰越が増加傾向の場合は対処リソースを増加させる
