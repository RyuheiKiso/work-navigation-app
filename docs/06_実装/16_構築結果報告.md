# 16 構築結果報告

本章は実装フェーズ（06_実装）の完了を報告するためのテンプレートである。現時点は骨子のみを定義し、実装完了後に各節を記入する。

---

## 1. 構築完了判定基準

実装フェーズの完了は以下の全条件を満たした場合に宣言する。いずれか 1 条件でも未達の場合、完了と判定しない。

| 判定項目 | 判定基準 | 確認方法 |
|---|---|---|
| IMPL-NNN 完了 | 全 31 IMPL-NNN のステータスが「完了」 | 15 章 § 2 の表を確認する |
| CI グリーン | unit-test / integration-test / E2E の全ジョブが通過 | GitHub Actions のステータスを確認する |
| コードレビュー | 全 PR の CR-NNN がクローズ（`[must]` 指摘が全件解消） | 14 章の CR 記録を確認する |
| 図品質 | drawio-lint / svg-postcheck ERROR 0（docs 変更がある場合） | CI ログを確認する |
| カバレッジ | backend ≥ 80% / handy ≥ 75% / master ≥ 75% | CI カバレッジレポートを確認する |
| ドキュメント整合 | 本書の全章が最新コードと整合している | セルフレビューで確認する |
| TST-NNN 対応 | 全 IMPL-NNN に TST-NNN ≥ 1 件が存在する | 15 章 § 3 の表を確認する |

```bash
# 構築完了判定の自動確認コマンド
# 1. CI ステータスを確認する
gh run list --limit 5

# 2. カバレッジレポートをダウンロードして確認する
gh run download --name coverage-reports

# 3. 未クローズの CR 記録を確認する
ls docs/06_実装/レビュー記録/ | grep -v "closed"
```

**本節で確定した方針**
- **全 7 条件を満たすまで構築完了を宣言しない**: 部分完了の宣言は行わない
- **判定は客観的な証跡（CI ログ・カバレッジレポート）に基づく**: 主観的判断による完了宣言を禁止する
- **判定結果は本章に記録して 07_テストへのハンドオフ資料とする**: 次フェーズの担当者が状況を把握できるようにする

## 3. 未対応項目と理由

実装フェーズ完了時点で未対応の IMPL-NNN がある場合は以下の表に記録する。

| IMPL-ID | モジュール名 | 未対応の理由 | 解消予定フェーズ | 対処方針 |
|---|---|---|---|---|
| （記入例）IMPL-NNN | （例）webhook 配信 | 外部 Webhook 受信先が未確定 | 07_テスト | 設計確定後に ADR-IMPL-NNN を記録して実装 |

未対応項目がない場合は「なし」と記載する。

未対応理由として許容されるもの:
- 外部依存（外部サービス・ハードウェア）が未確定
- 設計上の再検討が必要（ADR-IMPL-NNN を記録済み）

未対応理由として許容されないもの:
- 時間不足（スケジュール圧力による手抜きは禁止する）
- 優先度が低い（必須 IMPL-NNN の優先度は常に最高である）

**本節で確定した方針**
- **未対応の IMPL-NNN は必ず理由と解消予定を記録する**: 漠然とした未対応は禁止する
- **時間不足を未対応理由として許容しない**: 妥協は禁止するポリシーを実装フェーズでも遵守する
- **未対応項目は ADR-IMPL-NNN と連携して管理する**: 設計変更が生じた場合の根拠を残す

## 4. 既知の不具合一覧

実装フェーズ完了時点で判明している不具合を記録する。

| PROB-ID | Severity | 現象 | 解消予定 | 回避策 |
|---|---|---|---|---|
| （記入例）PROB-NNN | High | オフライン 30 分超でハッシュチェーン検証が誤検出する | IMPL-028 修正後 | 検証間隔を 24h に設定して回避 |

不具合一覧が空の場合は「現時点で既知の不具合なし」と記載する。

Severity の定義（詳細は 17 章 § 4 参照）:
- **Critical**: 本番データ損失またはセキュリティ侵害リスク
- **High**: 主要機能の動作不能
- **Medium**: 機能制約あり回避策あり
- **Low**: 軽微な不具合

Critical の不具合が存在する場合、構築完了判定を行わない。

**本節で確定した方針**
- **Critical 不具合が存在する場合は構築完了としない**: データ損失・セキュリティリスクのある状態でのリリースを禁止する
- **既知不具合は回避策を必ず記載する**: テストフェーズで回避策を前提としたテストを設計できるようにする
- **全不具合を PROB-NNN で 17 章と連携する**: 問題管理を一元化する

## 5. パフォーマンス計測結果

Glanceable 要件（現在ステップ表示 200ms / ページ遷移 500ms / QR 認識 1 秒）の達成状況を記録するテンプレート。

| Glanceable 要件 | 目標値 | 計測値 | 計測環境 | 判定 |
|---|---|---|---|---|
| 現在ステップ表示 | ≤ 200ms | — ms | — | — |
| ページ遷移 | ≤ 500ms | — ms | — | — |
| QR コード認識 | ≤ 1,000ms | — ms | — | — |
| API レスポンス（P50） | ≤ 100ms | — ms | — | — |
| API レスポンス（P99） | ≤ 500ms | — ms | — | — |

### 計測方法

```bash
# backend レイテンシの計測（criterion ベンチマーク）
cargo bench --bench api_latency
```

```typescript
// handy のページ遷移時間を計測する（React Native Performance API）
import { PerformanceObserver } from 'react-native-performance';
```

計測結果が目標を達成していない場合は PROB-NNN として問題ログに記録し、最適化を実施してから再計測する。

**本節で確定した方針**
- **Glanceable 要件の達成を数値で証明する**: 「体感的に速い」という主観的評価は不可
- **計測環境を記録する**: 環境差異による計測値のばらつきを追跡可能にする
- **目標未達の場合は最適化を実施してから次フェーズに移行する**: 性能要件を妥協しない

## 6. セキュリティスキャン結果

実装フェーズ完了時点のセキュリティスキャン結果を記録するテンプレート。

| スキャン種別 | ツール | 実施日 | Critical 件数 | High 件数 | Medium 件数 | 判定 |
|---|---|---|---|---|---|---|
| Rust 依存関係監査 | cargo audit | YYYY-MM-DD | — | — | — | — |
| JS/TS 依存関係監査 | pnpm audit | YYYY-MM-DD | — | — | — | — |
| SAST（静的解析） | Clippy / ESLint | YYYY-MM-DD | — | — | — | — |

### cargo audit の実行と結果記録

```bash
# Rust 依存関係の脆弱性スキャンを実行する
cargo audit

# 結果をファイルに保存する
cargo audit --json > security-scan/cargo-audit-$(date +%Y%m%d).json
```

### pnpm audit の実行と結果記録

```bash
# JS/TS 依存関係の脆弱性スキャンを実行する
cd src/frontend/handy && pnpm audit --json > ../../security-scan/pnpm-handy-$(date +%Y%m%d).json
cd src/frontend/master && pnpm audit --json > ../../security-scan/pnpm-master-$(date +%Y%m%d).json
```

Critical の脆弱性が 1 件でも存在する場合、構築完了判定を行わない。High 以下は PROB-NNN として記録し、対処方針を決定する。

**本節で確定した方針**
- **Critical 脆弱性が 0 件であることを構築完了の必要条件とする**: セキュリティリスクのある状態でのリリースを禁止する
- **スキャン結果を JSON 形式で保存する**: 証跡として残し、次回スキャン時の比較を可能にする
- **High 脆弱性は PROB-NNN として対処方針を記録する**: 放置しない管理体制を維持する

## 8. 07_テストへのハンドオフ事項

実装フェーズからテストフェーズへのハンドオフに必要な情報を記録するテンプレート。

| 引き継ぎ項目 | 内容 | 参照先 |
|---|---|---|
| 未解消の PROB-NNN | Severity Medium 以上の未解消問題一覧 | 17 章 |
| 既知の弱いカバレッジ領域 | テストが薄い IMPL-NNN とその理由 | 15 章 § 3 |
| テスト環境の前提 | testcontainers-rs の PostgreSQL 設定・msw の設定 | 13 章 § 6 |
| 特別なセットアップが必要なテスト | EAS ビルドが必要な E2E テスト・実デバイスが必要なテスト | — |
| 実装上の判断（ADR-IMPL-NNN） | テストに影響するアーキテクチャ判断 | 18 章 |

### ハンドオフチェックリスト

```
[ ] 15 章の構築トレーサビリティ表が最新状態に更新されている
[ ] 全 PROB-NNN の対処状況が 17 章に記録されている
[ ] 全 ADR-IMPL-NNN が 18 章に記録されている
[ ] CI が全ジョブグリーンであることを確認している
[ ] セキュリティスキャン結果（§ 6）が記録されている
[ ] パフォーマンス計測結果（§ 5）が記録されている
[ ] 本章の全節が記入されている
```

**本節で確定した方針**
- **ハンドオフチェックリストを全件完了してからテストフェーズを開始する**: 未完了のままテストを開始しない
- **既知の弱いカバレッジ領域を明示的に申し送る**: テストフェーズで重点的にカバーしてもらう
- **ADR-IMPL-NNN をハンドオフ資料に含める**: テスト設計に影響する実装判断を伝達する
