# 99 前提制約と本書が約束しないこと

本章の責務は、04_概要設計が引き継ぐ前提制約と、本書が明示的に「対象外と判断する」設計事項を一元的に宣言することである。各サブの `99_前提制約と本書が約束しないこと.md` はこの章を一次マスタとして参照し、当該サブ固有の禁止事項を付加的に宣言する。

## 2. 概要設計が「対象外と判断する」事項

### 2-1. OT（運用技術）との直接接続（永続的非提供）

**OT への制御信号送信は永続的に対象外と判断する。**

- 非提供: PLC（シーケンサ）への制御信号送信
- 非提供: SCADA システムとの双方向通信
- 非提供: OPC-UA / MQTT によるリアルタイム制御
- 理由: 本アプリは「IT 領域のみ」を設計範囲とする（構想 04 章 倫理スタンス・計画 05 章 P4 原則）
- 担保: `05_外部インターフェース設計/` の IF 一覧（IF-001〜007）に OT 系 IF を含めない。`01_システム方式設計/99` でネットワークゾーニング上の境界を宣言する

### 2-2. eval / 動的コード生成（永続的禁止）

**eval / Function constructor / 動的コード生成は永続的に対象外と判断する。**

- 禁止: JavaScript `eval()`・`new Function()`
- 禁止: Rust の `proc_macro` 外での動的コード生成
- 許可: JSON Logic（Apache 2.0）による条件式評価（演算子ホワイトリスト方式）
- 理由: セキュリティリスク（計画 05 章 P7 原則）および ALCOA+ Consistent 原則の担保
- 担保: `02_ソフトウェア方式設計/04_拡張Stepエンジン設計.md` にこの制約を組み込む

### 2-3. データロックイン（永続的禁止）

**データロックインを招く設計は永続的に対象外と判断する。**

- 禁止: 独自バイナリ形式での一次データ格納（PostgreSQL 標準型のみ使用）
- 禁止: エクスポート鍵を開発者のみが保持する設計
- 禁止: クラウド固有サービス（AWS S3・Azure Blob 等）への直接依存
- 必須: SQL 直接エクスポート / CSV エクスポート / OpenAPI JSON エクスポートの 3 手段を全段階で維持
- 理由: データオーナーシップ宣言（構想 13 章）
- 担保: `04_データ設計/99` で物理スキーマの OSS 依存性のみを使用することを明示する

### 2-4. 個人別生産性ランキング（永続的非提供）

**個人識別可能な形での生産性・速度・品質ランキングは永続的に対象外と判断する。**

- 禁止: 作業員別の工程完了速度ランキング
- 禁止: 個人別不適合発生率の一覧表示
- 禁止: `GROUP BY worker_id` を含む集計ビューの作成
- 許可: 匿名化集計（工程別・シフト別・製品別の集計は可）・品質改善のための工程別統計
- 理由: Just Culture 原則・NFR-ETH・BR-BUS-029
- 担保: `07_セキュリティ方式設計/08_倫理ガード設計.md` で技術的に拒否するアーキテクチャを確定する

### 2-5. マルチテナント横断クエリ（ver1.0.0 対象外）

**factory_id / site_id をまたぐ横断クエリは ver1.0.0 の対象外と判断する。**

- 非提供: 複数工場のデータを単一 PostgreSQL インスタンスで横断集計する機能
- 理由: ver1.0.0 のスコープは単一工場。マルチサイトは Enterprise 版（ver1.0.0 後の別計画）
- 担保: `04_データ設計/02_物理テーブル一覧.md` に factory_id 列を予約フィールドとして配置する（現時点では定数値）

### 2-6. eBR/MES への自動代替昇格（永続的非提供）

**本アプリが eBR（電子バッチ記録）または MES（製造実行システム）の完全代替となる機能を追加することは対象外と判断する。**

- 理由: 導入先組織の ERP/MES との「子機モード」での共存が基本設計前提。代替化は親機との責務を混乱させる
- 担保: `05_外部インターフェース設計/99` に明示する

### 2-7. クラウド冗長・地理冗長（対象外）

**クラウドを活用した地理冗長・マルチリージョン冗長は対象外と判断する。**

- 理由: NFR-AVL-010 で明示的に除外済み。中小製造業のオンプレ前提・IT 担当 1 名制約
- 代替: 単一建屋内 Active-Standby + 週次オフサイトバックアップ（NFR-AVL-011/017）

### 2-8. 多要素生体認証・顔認識（永続的非提供）

**多要素生体認証（指紋・静脈・顔認識）のシステム統合は対象外と判断する。**

- 例外許可: 端末 OS 標準生体認証（指紋/顔）を PIN の代替として OS ネイティブ機能で提供することは許可（本アプリがバイオメトリクスデータを保持しない前提）
- 理由: NFR-ETH（データ最小化）・リスクに比した実装コスト
- 担保: `07_セキュリティ方式設計/99` に明示する

### 2-9. AMQP・外部 MQ（ver1.0.0 対象外）

**AMQP（RabbitMQ/ActiveMQ/Kafka 等）を使用した外部メッセージキューシステムは ver1.0.0 の対象外と判断する。**

- 代替: PostgreSQL + Outbox テーブルによる at-least-once 配信（計画 12 章確定）
- 理由: IT 担当 1 名の運用負荷・中小製造業での MQ 運用難易度
- 担保: `02_ソフトウェア方式設計/05_イベント駆動と内部メッセージング.md` に明示する

### 2-10. 24/7 有償サポート・SLA（ver1.0.0 対象外）

**24/7 オンコール体制・SLA 保証を前提とするアーキテクチャ設計は対象外と判断する。**

- 提供: GitHub Issues 受付・ドキュメント全量 OSS 公開
- 非提供: SLA 保証付きサポート契約・オンコールエンジニア体制
- 理由: 個人開発・ver1.0.0 スコープ（計画 09 章）

**本節で確定した方針**
- **P-01〜P-09（企画由来）・R-01〜R-07（要件定義由来）の 16 前提制約を継承し、概要設計・詳細設計・実装を通じた不変拘束として確定する。**
- **OT 直結・eval・データロックイン・個人別ランキング・マルチテナント横断・eBR 代替昇格・地理冗長・生体 MFA・外部 MQ・有償 SLA の 10 事項を「対象外と判断する」として権威的に宣言し、技術的担保先サブを明示する。**
- **詳細設計以降に委任する事項を列挙し、概要設計の境界を明確にする。**
