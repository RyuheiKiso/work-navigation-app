# 03 概要設計の識別子体系と採番規約

本章の責務は、概要設計フェーズで新設する 16 種類の設計識別子のプレフィックス・採番形式・採番ルール・廃止規約を権威的に確定することである。本章は `03_要件定義/付録/00_要件IDルールと採番規約.md` の採番規約を完全に継承し、設計識別子の追加定義を行う。

---

## 1. 基本原則（要件定義書からの継承）

以下の採番基本規則は要件定義書の採番規約と同一であり、全設計識別子に適用する。

| 規則 | 内容 |
|---|---|
| 識別子形式 | `[プレフィックス]-[連番 3 桁]` または `[プレフィックス]-[サブカテゴリ]-[連番 3 桁]` の 2 パターン |
| 連番 | 各プレフィックス（またはサブカテゴリ）内で `001` から開始し、単調増加 |
| ゼロ埋め | 3 桁固定（001, 002, ..., 099, 100, ...） |
| 欠番禁止 | 連番をスキップしない |
| 再使用禁止 | 廃止した識別子の番号は再使用しない |
| 廃止規約 | 廃止時は `[廃止: YYYY-MM-DD 廃止理由: ...]` メタデータを先頭行に付与し、行を削除しない |
| 上限 | 1 プレフィックス（またはサブカテゴリ）あたり 999 件。超過時は規約改訂で対応 |
| 命名言語 | 物理名（テーブル名・API リソース名・モジュール名）は英小文字スネークケースのみ |

## 4. 各識別子の採番規約詳細

### 4-1. TBL（物理テーブル）

- **形式**: `TBL-NNN`
- **採番粒度**: プロジェクト全域（マスタ・トランザクション・制御テーブル全て通し番号）
- **物理テーブル命名**: 英小文字スネークケース複数形（例: `work_events`, `master_versions`）
- **必須メタデータ**: 物理テーブル名・対応論理エンティティ（EN-NNN）・対応関係（1:1/1:N/N:M）・種別（Append-only/更新可/マスタ）・推定行数・保存期間
- **上位リンク**: `EN-NNN`（論理エンティティとの対応）。中間テーブルは N:M として複数の EN-NNN を列挙
- **採番開始**: TBL-001

### 4-2. IDX（インデックス）

- **形式**: `IDX-NNN`
- **採番粒度**: プロジェクト全域
- **物理インデックス命名**: `idx_{table}_{cols}[_{type}]` 例: `idx_work_events_case_id_ts`
- **必須メタデータ**: 物理名・対応テーブル（TBL-NNN）・インデックス種別（B-Tree/Hash/BRIN/GIN）・対象列・Partial 条件（あれば）・性能根拠（NFR-PRF-NNN）
- **採番開始**: IDX-001

### 4-3. VW（ビュー・実体化ビュー）

- **形式**: `VW-NNN`
- **採番粒度**: プロジェクト全域
- **物理名命名**: 通常ビュー `vw_{purpose}`、実体化ビュー `mv_{purpose}`
- **必須メタデータ**: 物理名・種別（VIEW/MATERIALIZED VIEW）・構成テーブル（TBL-NNN 群）・消費側（API-NNN または RP-NNN）
- **採番開始**: VW-001

### 4-4. API（内部 REST エンドポイント）

- **形式**: `API-{resource}-NNN`（resource は英小文字ハイフン区切り複数形）
- **採番粒度**: resource 別（例: `API-work-execs-001`, `API-master-001`）
- **URL 規約**: `/api/v1/{resource}[/{id}][/{sub-resource}]`
- **operationId**: OpenAPI `operationId` は `{verbCamel}{ResourcePascal}` 形式（例: `postWorkEvent`）で別管理
- **HTTP メソッド規約**: GET=参照、POST=作成/操作、PATCH=部分更新。DELETE は Append-only 原則により論理廃止（POST `/{id}/deactivate`）に変換
- **必須メタデータ**: HTTP メソッド・URL パス・operationId・関連 FR-NNN・関連 BR-BUS-NNN・関連 TBL-NNN・エラーコード（ERR-NNN）・**担当バイナリ**（terminal-api / master-api / 両方）
- **廃止**: 破壊的変更時は新 resource/NNN を採番し、旧を廃止メタデータ付きで保持
- **採番開始**: 各 resource で 001 から
- **バイナリ割り当て原則**: ハンディ端末操作・作業実行・同期・IQC 受入測定は `terminal-api`。SOP 管理・承認・監査・ユーザー管理・レポートは `master-api`。認証系（/api/v1/auth/*）は両バイナリに配備する

主要 resource 一覧（暫定、04_データ設計・05_外部IF設計で確定）:

```
auth, work-orders, work-execs, step-events, evidences, electronic-signs,
master, master-versions, andon, capa, kaizen, sync, trace, reports, ops, system
```

### 4-5. MOD（モジュール）

- **形式**: `MOD-{target}-NNN`
- **target サブカテゴリ**:
  - `BE`（Backend Rust crate）
  - `FE-HA`（Frontend Handy React Native）
  - `FE-MA`（Frontend Master React SPA）
  - `FE-MC`（Frontend Console React SPA）
  - `SH`（Shared 共通ライブラリ）
  - `IN`（Infra 設定・DB ロール等）
- **採番粒度**: target 別 001 から
- **物理名命名**: Rust: `wnav_{name}`、React: PascalCase ディレクトリ名
- **必須メタデータ**: 物理名・ファイルパス（src/ 相対）・責務・関連 FR-NNN・関連 TBL-NNN（BE の場合）・関連 SCR-NNN（FE の場合）
- **採番開始**: 各 target で 001 から

**MOD-BE 確定済みエントリ（BE バイナリ分割対応）**:

| ID | 物理名 | crate パス | 責務 | 備考 |
|---|---|---|---|---|
| MOD-BE-001 | wnav_terminal_api | `crates/wnav_terminal_api/` | ハンディ端末向け axum ルータ・ミドルウェア（作業実行・証拠記録・同期・IQC 受入） | port 8080 / terminal-api |
| MOD-BE-010 | wnav_master_api | `crates/wnav_master_api/` | マスタメンテ・管理コンソール向け axum ルータ・ミドルウェア（SOP 編集・承認・監査・ユーザー管理・HashChainVerifier） | port 8081 / master-api |

MOD-BE は MOD-BE-001・MOD-BE-010 を確定済み（MOD-BE-002〜009 は共有ライブラリ crate 群として 02_ソフトウェア方式設計/11_MODカタログ.md 完成後に採番する）。次採番は **MOD-BE-011**。

### 4-6. CMP（画面コンポーネント）

- **形式**: `CMP-{app}-NNN`（app = HA/MA/MC）
- **採番粒度**: アプリ別 001 から
- **命名**: PascalCase（例: `StepCard`, `ElectronicSignPad`）
- **必須メタデータ**: 物理名・ファイルパス・配置画面（SCR-NNN）・所属モジュール（MOD-FE-*-NNN）
- **採番開始**: 各 app で 001 から

### 4-7. TRN（画面遷移）

- **形式**: `TRN-NNN`（全域単一連番）
- **採番粒度**: プロジェクト全域
- **必須メタデータ**: 発生元画面（SCR-NNN）・遷移先画面（SCR-NNN）・トリガーアクション・ガード条件（ロール/状態）・副作用（API-NNN、あれば）・関連 UC-NNN
- **注意**: ID 自体には画面名を含めない。画面名変更時に ID が変わる形式（`TRN-{from}-{to}-NNN`）は禁止
- **採番開始**: TRN-001

### 4-8. BAT（バッチ/ジョブ）

- **形式**: `BAT-NNN`
- **採番粒度**: プロジェクト全域
- **必須メタデータ**: 名称・実行契機（cron 式/イベント駆動/手動）・冪等性方式・失敗時動作（再試行/DLQ/アラート）・関連 FR-NNN または NFR-NNN・推定実行時間
- **採番開始**: BAT-001

### 4-9. MSG（メッセージ/トピック）

- **形式**: `MSG-NNN`
- **採番粒度**: プロジェクト全域
- **命名規約**: `{domain}.{entity}` のドット区切り（例: `outbox.work_event`）
- **必須メタデータ**: トピック名・配信先・配信保証（at-least-once/exactly-once）・スキーマバージョン・関連 IF-NNN
- **採番開始**: MSG-001

### 4-10. ERR（エラーコード）

- **形式**: `ERR-{layer}-NNN`
- **layer サブカテゴリ**:
  - `AUTH`（認証・認可エラー）
  - `VAL`（入力バリデーション違反）
  - `BIZ`（業務ルール違反）
  - `DB`（データ整合性エラー）
  - `EXT`（外部 IF エラー）
  - `SYS`（システム内部エラー）
- **採番粒度**: layer 別 001 から
- **必須メタデータ**: エラー名（英小文字スネークケース）・HTTP ステータスコード・RFC 9457 type URI・関連 BR-BUS-NNN（業務ルール違反の場合）・関連 API-NNN
- **採番開始**: 各 layer で 001 から

### 4-11. LOG（ログイベント）

- **形式**: `LOG-NNN`
- **採番粒度**: プロジェクト全域
- **命名規約**: `{domain}.{action}.{result}` のドット区切り（例: `auth.login.success`）
- **必須メタデータ**: イベント名・ログレベル（DEBUG/INFO/WARN/ERROR/SECURITY）・PII マスク要否・保管期間・関連 NFR-NNN
- **採番開始**: LOG-001

### 4-12. KEY（暗号鍵・秘密）

- **形式**: `KEY-NNN`
- **採番粒度**: プロジェクト全域
- **注意**: KEY-NNN は論理識別子のみ。値はリポジトリ外（環境変数/Secret Manager）で管理する
- **必須メタデータ**: 用途・アルゴリズム・鍵長・ローテーション周期・保管先・関連 FR-AU-NNN または NFR-SEC-NNN
- **採番開始**: KEY-001

### 4-13. CFG（設定パラメータ）

- **形式**: `CFG-NNN`
- **採番粒度**: プロジェクト全域
- **命名規約（キー名）**: ドット区切り階層（例: `network.emergency_mode.timeout_sec`）
- **必須メタデータ**: キー名・デフォルト値・型・許容範囲・変更権限（ロール）・必須/任意・関連 FR-NNN
- **採番開始**: CFG-001

### 4-14. SEQ（シーケンス図）

- **形式**: `SEQ-NNN`
- **採番粒度**: プロジェクト全域
- **注意**: 図ファイル名は `fig_des_seq_{name}` で対応。1 SEQ = 1 図ファイルを必須とする
- **必須メタデータ**: 名称・関連 UC-NNN・関与 API-NNN 群・関与 TBL-NNN 群・関与 MOD-NNN 群・図ファイル名
- **採番開始**: SEQ-001

### 4-15. MET（監視メトリクス）

- **形式**: `MET-NNN`
- **採番粒度**: プロジェクト全域
- **必須メタデータ**: 名称・カテゴリ（Latency/Throughput/Error/Saturation/Availability）・Prometheus メトリクス名・SLO 値・SLO 算定窓・関連 API-NNN・関連 NFR-NNN
- **採番開始**: MET-001

### 4-16. PRM（権限マトリクスセル）

- **形式**: `PRM-NNN`
- **採番粒度**: プロジェクト全域
- **必須メタデータ**: ロール・操作対象（SCR-NNN または API-NNN または TBL-NNN の列レベル）・許可種別（CRUD + 特殊操作）・判定（許可/拒否/マスク）・根拠（BR-BUS-NNN）
- **採番開始**: PRM-001

## 5. 識別子の廃止手順

1. 採番台帳（`付録/99_設計識別子採番台帳.md`）の当該行の先頭に `[廃止: YYYY-MM-DD 廃止理由: ...]` を付与
2. 当該識別子を参照している DTM・付録文書のセルに `[廃止]` を付記
3. 行・セルを削除しない（削除すると採番連続性の検証が不可能になるため）
4. 廃止した識別子の番号は再使用しない

## 6. 双方向リンクの実装方針

設計識別子のメタデータには上位リンク（要件 ID）を必ず記載する。逆方向（要件 ID → 設計 ID）のリンクは DTM（`付録/01_設計トレーサビリティマトリクス（DTM）.md`）の 6 マトリクスで管理する。

```
[上流] FR-EV-001 ─────────────────────────────────────────────┐
                                                               │
[設計] TBL-001（↑ EN-012）  API-step-events-001（↑ FR-EV-001）│
       MOD-BE-003（↑ FR-EV-001, TBL-001）                     │
       SEQ-001（↑ UC-002, API-step-events-001, TBL-001）       │
                                                               │
[DTM M1] FR-EV-001 → API-step-events-001  ◄────────────────────┘
[DTM M2] FR-EV-001 → TBL-001 (C)
[DTM M4] UC-002 → SEQ-001
```

## 7. 採番台帳の管理場所

| 対象 | 台帳ファイル |
|---|---|
| 要件識別子（BR/FR/NFR/UC/EN/IF/RP/SCR 等） | `03_要件定義/付録/99_採番台帳.md`（変更不可） |
| 設計識別子（TBL/IDX/VW/API/MOD/CMP/TRN/BAT/MSG/ERR/LOG/KEY/CFG/SEQ/MET/PRM 等） | `04_概要設計/付録/99_設計識別子採番台帳.md` |
| 全識別子プレフィックス索引 | `01_管理/採番台帳/INDEX.md`（リンク表のみ） |

---

**本節で確定した方針**
- **概要設計フェーズで 16 種類の設計識別子プレフィックス（TBL/IDX/VW/API/MOD/CMP/TRN/BAT/MSG/ERR/LOG/KEY/CFG/SEQ/MET/PRM）を新設し、既存 16 種の要件識別子プレフィックスとの衝突がないことを確認した。**
- **全設計識別子は要件定義書の採番規約（3桁ゼロ埋め・欠番禁止・再使用禁止・廃止メタデータ保持）を完全継承し、管理台帳は `04_概要設計/付録/99_設計識別子採番台帳.md` で一元管理する。**
- **バックエンドは `wnav_terminal_api`（MOD-BE-001 / port 8080）と `wnav_master_api`（MOD-BE-010 / port 8081）の 2 バイナリ構成とする。MOD-BE は MOD-BE-001〜010 まで確定。API-NNN には担当バイナリ列を付与し、BAT-NNN には動作主体バイナリ列を付与する。**
- **双方向リンク（上流 → 設計 ID は設計 ID メタデータ、設計 ID → 上流は DTM 6 マトリクス）で完全トレーサビリティを担保する。**
