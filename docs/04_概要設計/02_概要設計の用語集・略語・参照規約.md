## 2. 概要設計フェーズで新規定義する設計語

以下の用語は概要設計フェーズで初めて使用される設計語として定義する。

### 2-1. アーキテクチャ・コンポーネント語

| 用語 | 定義 |
|---|---|
| **レイヤードアーキテクチャ** | Presentation / Application / Domain / Infrastructure の 4 層でコンポーネントを分離する設計スタイル。各層は下位層にのみ依存する |
| **境界づけられたコンテキスト（Bounded Context）** | ドメインモデルの一貫性を保つ設計上の境界。本プロジェクトでは Authoring / Execution / Evidence / Quality / Integration の 5 コンテキストを定義する |
| **モジュール** | 本書では `MOD-NNN` 識別子で管理するソフトウェア構成単位。Rust の crate、React Native/React の feature ディレクトリ、共通ライブラリを指す |
| **コンポーネント（CMP）** | 本書では `CMP-NNN` 識別子で管理する UI 画面部品。React / React Native の 1 ファイル = 1 コンポーネントに対応する |
| **パッケージ** | 本書では `PKG-NNN` 識別子で管理する Rust モジュールまたは npm パッケージ |
| **サブシステム（SUB）** | システム全体を構成する独立した機能単位。本プロジェクトではハンディAPP / マスタメンテAPP / 管理コンソール / バックエンド / PostgreSQL DB の 5 サブシステムを定義する |
| **ノード（NODE）** | 物理的または仮想的なコンピュータ実行環境。Docker コンテナ・Windows Server ホスト・タブレット端末など |
| **wnav_terminal_api** | ハンディ端末向け axum バイナリ（MOD-BE-001）。port 8080 で待機し、作業実行・証拠記録・同期・IQC 受入等のハンディ操作 API を提供する。Docker Compose サービス名は `terminal-api`。OpenAPI 仕様は `openapi-terminal.yaml`（utoipa 生成） |
| **wnav_master_api** | マスタメンテ・管理コンソール向け axum バイナリ（MOD-BE-010）。port 8081 で待機し、SOP 編集・承認・監査・ユーザー管理・ハッシュチェーン検証等の管理操作 API を提供する。Docker Compose サービス名は `master-api`。OpenAPI 仕様は `openapi-master.yaml`（utoipa 生成） |
| **terminal-api** | Docker Compose におけるサービス名。`wnav_terminal_api` バイナリのコンテナを指す |
| **master-api** | Docker Compose におけるサービス名。`wnav_master_api` バイナリのコンテナを指す |
| **openapi-terminal.yaml** | `terminal-api`（wnav_terminal_api）の OpenAPI 3.1 仕様ファイル。utoipa により自動生成する |
| **openapi-master.yaml** | `master-api`（wnav_master_api）の OpenAPI 3.1 仕様ファイル。utoipa により自動生成する |
| **wnav_api** | [廃止: 2026-05-18 廃止理由: `wnav_terminal_api`（MOD-BE-001）と `wnav_master_api`（MOD-BE-010）への2バイナリ分割により単一バイナリ構成を廃止] |

### 2-2. データ設計語

| 用語 | 定義 |
|---|---|
| **物理テーブル（TBL）** | PostgreSQL に実体として存在するテーブル。論理エンティティ（EN）を物理化したもの。論理と物理は 1:1 または 1:N の関係 |
| **インデックス（IDX）** | 検索性能最適化のための B-Tree・Hash・BRIN・GIN インデックス。NFR-PRF 要件に基づいて定義する |
| **実体化ビュー（MV）** | PostgreSQL の `MATERIALIZED VIEW`。読み取り専用の事前集計結果を物理化したもの |
| **パーティション（PART）** | 大量データテーブルを時間範囲・エンティティ種別等で分割した物理テーブル断片。work_events の月次分割が主要ユースケース |
| **Append-only テーブル** | INSERT のみを許可し UPDATE/DELETE を DB ロールで禁止したテーブル。work_events・outbox_events・hash_chain_blocks 等 |
| **時点参照（Point-in-Time Reference）** | トランザクションレコードが記録時点のマスタ版数（sop_version_id 等）を固定し、マスタ変更の遡及を防ぐ設計手法 |
| **ハッシュチェーン** | 各 WorkEvent に直前レコードの SHA-256 ハッシュ（prev_hash）を連鎖させ、後から記録の改竄を検知可能にする構造 |

### 2-3. API・インタフェース語

| 用語 | 定義 |
|---|---|
| **エンドポイント（EP）** | 特定 HTTP メソッドと URL パスの組み合わせ。`POST /api/v1/work-executions/{id}/events` の形式で表現する |
| **Idempotency Key** | 冪等性を保証するためにリクエストヘッダに付与する UUID v7 識別子。`Idempotency-Key` ヘッダに設定する |
| **Webhook** | サーバーからクライアントへのイベント通知。HMAC-SHA256 署名付きの HTTP POST で配信する |
| **OpenAPI 3.1** | REST API の仕様記述標準（OAS 3.1.0）。本プロジェクトの全内部 API はこれで仕様化する |
| **RFC 9457 Problem Details** | API エラーレスポンスの標準 JSON 構造。`type` / `title` / `status` / `detail` / `instance` を含む |
| **Contract Test** | API のスキーマ契約（OpenAPI 仕様）に対する自動テスト。内部モックや Pact を使用して検証する |

### 2-4. 運用・セキュリティ語

| 用語 | 定義 |
|---|---|
| **ジョブ（BAT/JOB）** | 定期的またはイベント駆動で実行されるバックグラウンド処理。Outbox Consumer・バックアップ・ハッシュチェーン検証など |
| **DLQ（Dead Letter Queue）** | 一定回数のリトライ後も配信に失敗したメッセージを格納するキュー。手動介入を要する |
| **STRIDE** | Spoofing / Tampering / Repudiation / Information Disclosure / Denial of Service / Elevation of Privilege の 6 種脅威分類 |
| **mTLS** | 相互 TLS（Mutual TLS）。クライアントとサーバーが互いに証明書を提示して認証する方式 |
| **RBAC マトリクス** | ロール × 操作対象（画面 / API / テーブル）の権限許可/拒否を一覧化した表。`PRM-NNN` 識別子で管理する |
| **SQLCipher** | SQLite データベースを AES-256 で透過的に暗号化する OSS ライブラリ。端末 SQLite の暗号化に使用する |
| **倫理ガード** | 個人別生産性ランキング・監視機能の実装を技術的に拒否するアーキテクチャ上の制約。RBAC と API 設計の両面から実現する |

### 2-5. テスト語

| 用語 | 定義 |
|---|---|
| **テストピラミッド** | 単体テスト（多）→ 統合テスト（中）→ E2E テスト（少）の比率で自動テストを構成する設計哲学 |
| **テストレベル（TLVL）** | テストの抽象度・範囲を表す区分（単体・結合・システム・受入）。本書では `TLVL-NNN` で識別する |
| **テスト環境（TENV）** | テスト実行のための環境構成（DEV / STG / PRD）。本書では `TENV-NNN` で識別する |
| **品質ゲート** | CI/CD パイプラインにおいて、カバレッジ・lint・型チェックが基準を満たさない場合にマージを阻止するチェック |

## 3. 参照規約

### 3-1. 上流文書への参照形式

```
[`02_企画/システム化計画/XX_*.md`](../../02_企画/システム化計画/XX_*.md)
[`03_要件定義/機能要件/XX_*.md`](../../03_要件定義/機能要件/XX_*.md)
```

### 3-2. 本書内章間参照

```
[`04_概要設計/04_データ設計/01_論理ER.md`](../04_データ設計/01_論理ER（要件EN-001〜027継承）.md)
```

### 3-3. 識別子参照

要件 ID または設計 ID を参照する場合は、対応ファイルパスを必ず併記する。

```
`TBL-001`（`04_データ設計/02_物理テーブル一覧（TBLカタログ）.md` 参照）
`FR-EV-001`（`03_要件定義/機能要件/01_機能一覧.md` §3 参照）
```

### 3-4. 業界分析への参照形式

```
[`90_業界分析/XX_*.md`](../90_業界分析/XX_*.md)
```

参照区分: 「必須」（設計命題に直接根拠を持つ業界分析）と「関連」（補助的な参照）の 2 区分。「関連」は最大 4 件まで。

## 4. 禁止表現一覧（継承・全章適用）

以下の表現は `03_要件定義/00_本書の位置づけと要件定義の境界.md` §4 から継承し、概要設計全章で禁止する。

| 禁止表現 | 代替表現 |
|---|---|
| 考慮する | 「〜とする」「〜を採用する」「〜を対象外と判断する」 |
| 検討する | 設計命題を確定した形で記述する |
| 参考にする | 「〜に準拠する」「〜に対応する」 |
| 可能性がある | 確定した設計命題として記述する |
| 場合がある | 分岐条件を明示した形で記述する |
| なるべく準拠する | 「準拠する」または「対応する」に変更 |
| 準拠を目指す | 「準拠する」または「対象外と判断する」に変更 |
| 適宜対応する | 具体的な条件・方法を記述する |
| 〜したい | 設計命題は意思ではなく確定命題として記述する |
| 〜と思われる | 調査・検証を行い確定命題として記述する |

---

**本節で確定した方針**
- **要件定義書の全用語・略語を継承し、本章では概要設計フェーズ固有の設計語（レイヤード・Bounded Context・物理テーブル・ハッシュチェーン等）のみを追加定義する。**
- **バックエンドは `wnav_terminal_api`（port 8080 / terminal-api）と `wnav_master_api`（port 8081 / master-api）の 2 バイナリ構成とする。旧称 `wnav_api`（単一バイナリ）は廃止し、本章以降では上記 2 用語を使用する。**
- **参照規約と禁止表現リストは本書全章に適用される権威規約であり、下位章での再解釈を禁止する。**
- **「する論（設計命題）」で記述することを全章に義務づけ、不確定・任意・希望を示す表現の混入を禁止する。**
