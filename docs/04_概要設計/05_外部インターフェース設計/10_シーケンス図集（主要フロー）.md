# 10 シーケンス図集（主要フロー）

本章の責務は、SEQ-001〜008 の全シーケンスを文章で記述し、drawio 図（`img/` 配下）の作成指示を確定することである。

## SEQ-001: 作業開始〜Step 実行〜完了（UC-001/002）

図: fig_des_seq_step_execution（img/ 配下）を参照。

```
参加者: Operator（端末）、StepEngine（FE-HA）、LocalDB（SQLite）、Backend（Rust）、PostgreSQL

1. Operator: QR スキャンまたはリスト選択
2. FE-HA → Backend: GET /api/v1/work-orders?sop_code={code}
3. Backend → FE-HA: SOP 情報・Step 一覧（マスタキャッシュが新しければ LocalDB から取得）
4. FE-HA → Backend: POST /api/v1/work-executions（Idempotency-Key 付き）
5. Backend → PostgreSQL: work_executions INSERT
6. Backend → FE-HA: 201 Created（work_execution_id）

[Step 実行ループ]
7. Operator: Step 内容を確認・入力（boolean_check / numeric_input 等）
8. StepEngine: ロックステップ検証（BR-BUS-001/003）
9. StepEngine → LocalDB: work_events INSERT（SQLite・Offline-First）
10. StepEngine → LocalDB: outbox_events INSERT（PENDING）
11. OutboxWorker（バックグラウンド）→ Backend: POST /api/v1/work-executions/{id}/events
12. Backend: Idempotency Key 確認 → HashChainService で prev_hash 取得・content_hash 計算
13. Backend → PostgreSQL: work_events INSERT（Append-only）+ outbox_events INSERT（同一 TX）
14. Backend → LocalDB（OutboxWorker 経由）: status = SENT
15. FE-HA: 次 Step へ進行 or 完了

[作業完了]
16. FE-HA → Backend: POST /api/v1/work-executions/{id}/complete
17. Backend → PostgreSQL: work_executions.status = COMPLETED
```

## SEQ-002: 証拠記録と電子サイン（UC-006/009）

図: fig_des_seq_evidence_sign（img/ 配下）を参照。

```
1. Step で evidence_required = TRUE のタイプに到達
2. StepEngine: カメラ起動（FR-EV-002）
3. Operator: 写真撮影
4. FE-HA: Exif 削除 → JPEG 圧縮 → SHA-256 計算
5. FE-HA → Backend: POST /api/v1/evidences（multipart）
6. Backend: SHA-256 突合 → NAS ファイル書き込み → TBL-009 INSERT
7. Backend → FE-HA: evidence_id 返却
8. FE-HA: evidence_id を Step 完了 payload に含める

[電子サイン（sign_required = TRUE のStep 等）]
9. FE-HA: 電子サインパッド表示（CMP-HA-007）
10. Operator: PIN 入力（本人認証再入力 BR-BUS-011）
11. FE-HA → Backend: POST /api/v1/electronic-signs（signer_id + timestamp + meaning + content_hash）
12. Backend → PostgreSQL: electronic_signs INSERT（Append-only）
13. Backend → FE-HA: sign_id 返却
```

## SEQ-003: Outbox 同期と Idempotency（UC-020/021）

図: fig_des_seq_outbox_sync（img/ 配下）を参照。

```
[Outbox Consumer ループ（BAT-002）]
1. BAT-002: TBL-003 から status = PENDING の行を N 件取得
2. BAT-002: status → SENDING（楽観ロック付き UPDATE）
3. BAT-002 → Backend: POST /api/v1/sync/outbox/inbound（親機）
   または: POST /api/v1/work-executions/{id}/events（内部 API）
4. 送信成功: status → SENT、sent_at = NOW()
5. 送信失敗（4xx/5xx）: retry_count++、next_retry_at = NOW() + backoff
   → retry_count >= 5: status → DLQ、MET-005 increment

[Idempotency 確認（受信側）]
6. 受信 API: TBL-035（idempotency_keys）に Idempotency-Key を確認
7. 存在する場合: 前回の 200 OK レスポンスをそのまま返す
8. 存在しない場合: 通常処理 → TBL-035 に INSERT（24h TTL）
```

## SEQ-004: 中断・再開（UC-010/011）

図: fig_des_seq_suspend_resume（img/ 配下）を参照。

```
[中断]
1. Operator: 中断ボタン押下（SCR-HA-011）
2. FE-HA: 中断理由カテゴリ選択（FR-ST-001）
3. FE-HA → Backend: POST /api/v1/work-executions/{id}/suspend
4. Backend → PostgreSQL: suspensions INSERT（Append-only）+ work_executions.status = SUSPENDED
5. FE-HA: 電子サイン要求（FR-ST-003）→ POST /api/v1/electronic-signs

[再開]
6. 別の Operator: SCR-HA-012 で再開可能作業を選択
7. FE-HA → Backend: POST /api/v1/work-executions/{id}/resume
8. Backend: 再開可否判定（FR-ST-006）・本人認証確認
9. Backend → PostgreSQL: work_executions.status = IN_PROGRESS、current_step_index 復元
10. FE-HA: チェックポイント表示（FR-ST-005）→ Step 実行再開
```

## SEQ-005: マスタ Publish 承認フロー（UC-014/016）

図: fig_des_seq_master_publish（img/ 配下）を参照。

```
1. MasterAdmin: SOP ドラフト作成（POST /api/v1/master-versions/draft）
2. MasterAdmin: レビュー依頼（POST /api/v1/master-versions/{id}/submit）
3. QualityAdmin: 差分確認（GET /api/v1/master-versions/{id}）
4. QualityAdmin: dry-run で影響確認（POST /api/v1/master-versions/{id}/dry-run）
5. QualityAdmin: 電子サインで承認（POST /api/v1/electronic-signs + POST /api/v1/master-versions/{id}/approve）
6. Backend: sign_id 確認 → master_versions.status = PUBLISHED
7. Backend → MSG-004（internal.master_published）: tokio channel で配信
8. BAT-003（master_sync_puller）: 次回同期で端末に差分配信
```

## SEQ-006: Webhook 配信失敗 → DLQ → 再送（UC-020）

図: fig_des_seq_webhook_dlq（img/ 配下）を参照。

```
1. BAT-002: 親機 API に POST 送信
2. 親機 API: 503 Service Unavailable を返す
3. BAT-002: retry_count++、次 retry_at を計算
4. [3 回リトライ後も失敗]
5. BAT-002: status → DLQ、LOG-008 記録、MET-005 increment
6. SCR-MC-007（OutboxMonitor）: DLQ カウントをダッシュボードに表示
7. IT 担当: API-ops-002（POST /ops/outbox/{id}/requeue）で手動再投入
8. BAT-002: 再投入された PENDING 行を次ループで処理
```

## SEQ-007: ハッシュチェーン検証ジョブ（BAT-001）

図: fig_des_seq_hash_chain_verify（img/ 配下）を参照。

```
1. BAT-001（週次 cron）起動
2. PostgreSQL: 最新 HashChainBlock（TBL-031）の last_content_hash を取得
3. 対象期間の work_events を timestamp_server 昇順で全件取得
4. 各 work_event に対して: 
   prev_hash が直前 work_event の content_hash と一致するか検証
5. 一致しない行を発見 → ERR-DB-003 + LOG-007（SECURITY）記録 + MET-006 increment
6. 全件一致 → 新 HashChainBlock を TBL-031 に INSERT
7. MET-006（hash_chain.break_count）: 0 のまま
```

## SEQ-008: 親機マスタ同期（UC-019）

図: fig_des_seq_master_pull（img/ 配下）を参照。

```
1. BAT-003（60 分 cron）またはユーザー手動トリガ
2. Backend → 親機: GET {親機}/api/v1/sync/master?as_of={last_sync_ts}
3. 親機: 差分データ（SOP・Step 等）を JSON で返す
4. Backend: 差分を PostgreSQL（マスタテーブル）に UPSERT
5. Backend: device_sync_states を更新
6. 次回ハンディ APP の同期時（API-sync-001）に端末へ差分配信
```
