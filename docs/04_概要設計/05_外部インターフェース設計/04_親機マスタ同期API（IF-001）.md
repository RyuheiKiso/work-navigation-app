# 04 親機マスタ同期 API（IF-001）

本章の責務は、子機バックエンドが親機から SOP マスタ等を受信する Pull 型差分同期 API（IF-001）の設計を確定することである。対応する API 識別子: `API-sync-001`。

---

## 1. 同期の基本設計

### 1-1. Pull 型・差分同期

子機が親機に GET リクエストを送り、差分のみを取得する。親機から子機への Push は設計しない（計画 12 章確定）。

```
[子機バックエンド]
  BAT-003（master_sync_puller） → GET {親機}/api/v1/sync/master?as_of={last_sync_ts}
  [親機]
    → `as_of` 以降に更新された SOP/Step/Process/Operation/Product 等のデータを返す
  [子機バックエンド]
    → 端末 SQLite キャッシュに差分を適用
    → TBL-034（device_sync_states）を更新
```

### 1-2. API 仕様概要

```http
GET /api/v1/sync/master?as_of=2026-05-01T00:00:00Z&master_types=sops,steps,processes
Authorization: Bearer {JWT}
Accept: application/json
```

```json
// レスポンス例（差分のみ）
{
  "as_of": "2026-05-17T10:00:00Z",
  "sops": [{ "sop_id": "...", "sop_code": "ASS-001-003-SOP-001", ... }],
  "steps": [...],
  "processes": [...],
  "operations": [...],
  "products": [...],
  "users": [...],
  "roles": [...],
  "skills": [...]
}
```

### 1-3. 初回同期（FR-SY-002）

```http
GET /api/v1/sync/master?as_of=1970-01-01T00:00:00Z
```

`as_of` = epoch 0 で全量取得。30 分以内の完了を NFR-MIG が要件とする。

---

## 2. 認証・セキュリティ

| 項目 | 内容 |
|---|---|
| 認証 | JWT RS256（子機バックエンドが自身の JWT で認証）|
| TLS | 1.3 必須 |
| レート制限 | 60 req/min（`sync` カテゴリ）|
| 圧縮 | gzip 圧縮（大容量差分の転送効率化）|

---

## 3. 外部キーバインディングとの連動

親機との外部キーマッピングは TBL-027（external_key_bindings）で管理する。マスタ同期時に `work_patterns` テーブルも差分同期対象に含める（IF-001 の拡張部分）。

---

**本節で確定した方針**
- **IF-001 は子機 Pull 型・差分同期・JWT 認証で実現し、`as_of` パラメータで増分取得する設計を確定した。**
- **初回同期は `as_of = epoch 0` で全量取得し、30 分以内を NFR-MIG の品質目標とする。**
- **圧縮（gzip）と Partial（master_types 絞り込み）で大容量マスタの転送効率を確保する。**
