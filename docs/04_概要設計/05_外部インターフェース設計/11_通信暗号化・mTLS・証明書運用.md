# 11 通信暗号化・mTLS・証明書運用

本章の責務は、TLS 1.3 必須化・社内 CA による証明書管理・mTLS（相互認証）の設計を確定することである。

---

## 1. TLS 1.3 必須化

### 1-1. 通信経路と TLS の適用

| 経路 | TLS | 証明書 |
|---|---|---|
| クライアント → IIS（リバプロ）| TLS 1.3 必須 | KEY-007（公開 CA または社内 CA）|
| IIS → Backend（コンテナ内部）| HTTP 可（同一ホスト内）| — |
| Backend → 親機（IF-001/002）| TLS 1.3 + mTLS | KEY-007/008 |
| Backend → LDAP（IF-003）| LDAPS（TLS 636）| — |
| Backend → プリンタ（IF-004）| HTTP 可（社内 LAN 内）| — |

HTTP → HTTPS リダイレクトは IIS の URL Rewrite で強制（`Strict-Transport-Security` ヘッダ付き）。

### 1-2. IIS での TLS 設定

```
TLS バージョン: 1.3 必須（1.0/1.1/1.2 を無効化）
暗号スイート: TLS_AES_128_GCM_SHA256 / TLS_AES_256_GCM_SHA384
証明書: Let's Encrypt（ACME）または Windows Server 証明書マネージャー
```

---

## 2. mTLS（相互 TLS）の設計

### 2-1. 子機 → 親機の相互認証（IF-002）

OAuth 2.1 Client Credentials が使用不可の環境では mTLS で認証する。

```
[子機バックエンド]
  クライアント証明書（KEY-008）を提示
  → [親機] サーバー証明書を提示
  → 双方の証明書チェーンを検証
  → 認証成功 → API アクセス許可
```

### 2-2. 証明書のローテーション

| 証明書 | 期限 | ローテーション | 担当バッチ |
|---|---|---|---|
| KEY-007（API サーバー TLS）| 13 ヶ月 | ACME 自動更新 | BAT-009（残 30 日で通知）|
| KEY-008（mTLS クライアント）| 1 年 | 手動（IT 担当）| BAT-009（同上）|

---

**本節で確定した方針**
- **全外部通信（クライアント↔IIS・Backend↔親機）に TLS 1.3 を必須とし、HTTP 平文通信を IIS の URL Rewrite で禁止する。**
- **親機との mTLS（KEY-008）はカスタムアダプタで OAuth 2.1 Client Credentials と切替え可能とし、親機の認証方式に依存しない設計を確定した。**
- **証明書ローテーションは BAT-009 で残 30 日になると IT 担当に通知する設計を確定した。**
