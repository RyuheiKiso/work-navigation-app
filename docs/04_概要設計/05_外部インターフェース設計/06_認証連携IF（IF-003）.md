# 06 認証連携 IF（IF-003）

本章の責務は、LDAP/AD との認証連携（IF-003）の設計を確定することである。

---

## 1. LDAP 連携の設計方針

### 1-1. LDAP Bind 認証フロー

```
1. ユーザー: POST /api/v1/auth/login（API-auth-001）
2. バックエンド（MOD-BE-005）:
   → ldap3 crate（Rust）で LDAP サーバーに LDAPS（636 ポート）接続
   → `uid={username},ou=users,dc=factory,dc=example,dc=com` に Bind
   → Bind 成功 → TBL-016（users）に存在確認
   → TBL-016 に存在しない → 自動プロビジョニング（role = operator デフォルト）
   → JWT 発行
3. LDAP 接続失敗 → ローカル認証フォールバック（§2）
```

### 1-2. 接続設定（CFG-NNN で管理）

| 設定項目 | CFG キー | 例 |
|---|---|---|
| LDAP サーバー | `ldap.host` | `ldap.factory.example` |
| LDAP ポート | `ldap.port` | `636`（LDAPS）|
| Base DN | `ldap.base_dn` | `dc=factory,dc=example,dc=com` |
| ユーザー検索フィルタ | `ldap.user_filter` | `(uid={username})` |
| 接続タイムアウト | `ldap.timeout_ms` | `3000` |

---

## 2. ローカル認証フォールバック

LDAP が利用不可の場合（ネットワーク障害・AD サーバー停止）、ローカルパスワードでの認証にフォールバックする。

- TBL-016（users）に `password_hash` 列（bcrypt, cost 12）を保持
- LDAP 成功時はローカルパスワードを更新（LDAP パスワードをキャッシュ）
- `it_admin` ロールがローカルパスワードを発行・リセット可能（FR-MA-015）

---

## 3. 未実装事項（ver1.0.0 対象外）

- SAML 2.0 による SSO
- OpenID Connect（OIDC）連携
- Multi-Factor Authentication（MFA）

（これらは計画 15 章で明示的に「ver1.0.0 対象外」として確定済み）

---

**本節で確定した方針**
- **LDAP/AD（IF-003）への LDAPS 接続で BIND 認証し、LDAP 不可時はローカル bcrypt パスワードにフォールバックする二重認証フローを確定した。**
- **LDAP 成功時にローカルパスワードを同期することで、フォールバック時も最新パスワードで認証できる設計を確定した。**
