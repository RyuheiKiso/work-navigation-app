# 縮退運用・BCP方式設計

本節では、障害発生時の縮退運用（Degraded Operation）方針および事業継続計画（BCP: Business Continuity Plan）を定義する。製造現場においてシステム停止は生産ラインの停止に直結しうるため、段階的なトリガーレベルを設け、各レベルに応じた縮退対応・業務継続手段を明確化する。単一管理者・個人運用を前提とし、24時間有人監視は設けないが、アラート通知と明文化された手順書によって LEVEL-3（サーバー停止）でも 15分以内の復旧（RTO=15分）を目指す。

---

## 1. トリガーレベル定義

| レベル | 名称 | 発動条件 | 影響範囲 |
|--------|------|----------|----------|
| LEVEL-1 | API 縮退 | API エラー率 ≥5%（5分間移動平均）、または P95 レイテンシ ≥2秒 | API 応答遅延・一部機能制限 |
| LEVEL-2 | 緊急オフラインモード | WiFi 切断（ハンディ機器〜AP間の疎通断）、またはサーバー到達不可 60秒以上 | マスタ同期不可・新規ログイン不可 |
| LEVEL-3 | サーバー停止 | Docker ホスト停止・PostgreSQL 停止・IIS 停止 | 全オンライン機能停止 |

---

## 2. LEVEL-1: API 縮退モード

### 2.1 検知条件

- API エラー率（4xx + 5xx / 全リクエスト）≥5%（MET-002 参照）
- API P95 レイテンシ ≥2秒（MET-001 参照）
- ヘルスチェック `/health` が連続3回失敗

### 2.2 縮退時の動作

| 機能 | 縮退時の動作 |
|------|-------------|
| 作業ナビ表示 | ハンディのローカルキャッシュ（SQLite）から表示。最終同期時点の SOP を使用。 |
| 作業実績記録 | Outbox に記録し、接続回復後に BAT-002 が同期。 |
| マスタ参照 | キャッシュから参照（更新不可）。 |
| 帳票生成 | 接続回復後に遅延生成（BAT-006〜009 が Outbox 処理で補完）。 |
| ログイン | 既存セッション（キャッシュ JWT）は継続。新規ログインは接続回復後。 |

### 2.3 復旧手順

system_admin が SCR-MC-001（運用ダッシュボード）でアラート確認後、API ログを確認して原因特定・対処。Rust API の再起動は `docker-compose restart api`。

---

## 3. LEVEL-2: 緊急オフラインモード（Emergency Mode）

### 3.1 検知条件

ハンディ端末がサーバーへの接続を 60秒連続で喪失した場合、端末側アプリが自動的に Emergency Mode へ移行する。

### 3.2 Emergency Mode の動作仕様

| 機能 | Emergency Mode での動作 | 制限事項 |
|------|------------------------|----------|
| 作業ナビ表示 | SQLite ローカルキャッシュから表示 | 最終オンライン同期時点の SOP のみ使用可能 |
| 作業実績記録 | SQLite の Outbox テーブルに書き込み | サーバー送信は接続回復後に BAT-002 が実施 |
| 証跡撮影 | ローカル保存（後で rsync） | 即時サーバー送信は不可 |
| マスタ参照 | 最終キャッシュを使用 | マスタ更新は不可 |
| 新規ログイン | 不可（キャッシュ JWT のみ有効） | JWT 有効期限: **8時間**（工場の標準シフト時間） |
| 不適合登録 | Outbox 経由でローカル記録 | 接続回復後に同期 |

- JWT 8時間ウィンドウの根拠: 製造現場の標準シフト（8時間勤務）において、シフト開始時にオンラインで認証できれば、シフト内は継続動作可能とする設計。
- マスタ同期は不可。ただし「マスタが古い可能性がある」旨の警告バナーをハンディ UI に表示する。

### 3.3 Emergency Mode 表示仕様

ハンディ端末のヘッダー部に以下のバナーを表示する（画面設計 SCR-HH-001 参照）。

```
[オフラインモード] サーバーに接続できません。最終同期: HH:MM
作業記録はローカルに保存され、接続回復後に自動送信されます。
```

### 3.4 接続回復後の処理

```
接続回復検知（ハートビート 30秒毎）
  → BAT-002（Outbox リトライ）が Outbox をフラッシュ
  → BAT-003（ハッシュチェーン検証）をオプション実行
  → ハンディ端末: マスタ差分同期
  → Emergency Mode バナーを解除
```

## 4. LEVEL-3: サーバー停止 BCP

### 4.1 BCP の基本方針

LEVEL-3（サーバー完全停止）では、デジタルシステムへの依存を断ち切り、**ペーパーバックアップシート（紙ベース作業指示書）**による手動運用へ移行する。これは製造業における「紙を撲滅する」思想ではなく、「紙はシステム停止時の最後の砦」という現実的な BCP 設計である。

### 4.2 ペーパーバックアップ対象 SOP

| 対象 | 内容 | 保管場所 |
|------|------|----------|
| クリティカル SOP 一覧 | 生産停止に直結するワークフロー（組立・検査・出荷の重要工程） | 各ライン横の BCP バインダー |
| 不適合記録用紙 | 手書き記録フォーム（RP-003 相当） | 品質管理室・各ライン |
| 検査記録用紙 | 手書き検査チェックシート（RP-005 相当） | 各検査ポイント |

ペーパーバックアップシートはシステムリリース時に最新版を印刷・配布し、以後 MAJOR バージョン変更時に更新する。

### 4.3 LEVEL-3 復旧手順（スタンバイサーバー使用）

```
1. system_admin が障害検知（アラートまたは直接通報）
2. スタンバイサーバーにアクセス
3. 04_バックアップ・リストア方式 の Step 1〜5 を実施
4. DNS または IIS のリバースプロキシ設定をスタンバイサーバーへ切り替え
5. ヘルスチェック確認
6. ハンディ端末が自動的に接続回復を検知 → BAT-002 でデータ同期
7. ペーパー記録をシステムへ手動入力（最優先: 不適合記録・検査記録）
```

目標: **RTO=15分**（04_バックアップ・リストア方式の設計目標と整合）

### 4.4 スタンバイサーバー要件

| 項目 | 仕様 |
|------|------|
| OS | Windows Server 2022 + WSL2（プライマリと同一構成） |
| Docker Compose | プライマリと同一 `docker-compose.yml` |
| データ | BAT-001 rsync により最新バックアップが保管済み |
| 切り替え操作 | IIS リバースプロキシの upstream 変更（手動） |

---

## 5. 接続回復後の整合性確認

LEVEL-2/3 からの復旧後、以下の確認を実施する。

```
1. BAT-002（Outbox リトライ）: 蓄積されたオフラインイベントをサーバーへ送信
   → outbox_events テーブルの pending 件数が 0 になることを確認

2. BAT-003（ハッシュチェーン検証）: LEVEL-2/3 中に記録されたイベントのハッシュチェーン整合性を確認
   → 正常: "hash chain integrity: OK"
   → 異常: ERR-SEC-002 → system_admin が SCR-MC-008 で承認後に再開

3. マスタ差分同期: ハンディ端末がサーバーから最新マスタを取得
   → Emergency Mode バナーが解除されることで確認
```

---

## 6. BCP 訓練

| 項目 | 内容 |
|------|------|
| 実施頻度 | **6ヶ月に1回** |
| 参加者 | system_admin + quality_admin |
| 訓練内容 | LEVEL-3 シナリオによる実機復旧演習（スタンバイサーバーへの切り替え + RTO 計測） |
| 訓練記録 | `maintenance_log` テーブルに記録（log_type='bcp_drill'） |
| 合否基準 | RTO=15分 以内での完全復旧確認 + BAT-002/BAT-003 正常完了 |

**本節で確定した方針**

- 障害を LEVEL-1（API 縮退）/ LEVEL-2（緊急オフラインモード）/ LEVEL-3（サーバー停止）の3段階に分類し、各レベルで業務継続手段（SQLite キャッシュ・Outbox・ペーパーバックアップ）を明確化する。
- LEVEL-2 の Emergency Mode では、JWT 有効期限を工場標準シフト（8時間）に合わせ、シフト内の継続動作を保証する。マスタ同期と新規ログインのみ制限とし、作業記録は Outbox で確実に保全する。
- BCP 訓練を 6ヶ月ごとに system_admin + quality_admin で実施し、RTO=15分 以内の復旧を `maintenance_log` に記録することで、復旧手順の形骸化を防ぐ。
