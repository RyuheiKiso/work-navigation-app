# GUI初期構築ウィザード方式

本節では、システム初回セットアップを単一管理者（system_admin）が迷いなく完了できるようにするための GUI 初期構築ウィザード方式を定義する。作業ナビゲーションシステムは PostgreSQL 初期化・管理者ユーザー作成・マスタデータインポート・デバイス登録といった複数の初期設定ステップを必要とするが、CLI 操作に不慣れな管理者でも Web ブラウザ上で完結できる設計とする。ウィザードはべき等性を持ち、途中中断・再開が可能であり、全ステップ完了後は自動的に通常運用モードへ移行する。

---

## 1. ウィザードの位置づけと目的

| 項目 | 内容 |
|------|------|
| 目的 | 1人運用の初期セットアップ複雑度を削減し、設定漏れ・手順ミスを防止する |
| 対象者 | system_admin（IT 専門家でない製造現場管理者を想定） |
| 実行タイミング | システム初回起動時（`config_snapshots` テーブルの `app_initialized` フラグが false の場合） |
| 再実行可能性 | **べき等的**: 完了済みステップはスキップし、未完了ステップから再開できる |
| アクセス方法 | http://[サーバー IP]:8090/wizard（初期化完了前のみアクセス可能） |

---

## 2. ウィザードポータルの独立構成

初期構築ウィザードはメインアプリ（ポート 8080）とは独立した React Web アプリとして稼働する。

```
ポート 8090: ウィザードポータル（初期化前専用）
  ↓ 初期化完了後
ポート 8080: 通常のマスタメンテナンス UI（9090 は各環境に依存）
```

| 項目 | 仕様 |
|------|------|
| フレームワーク | React（マスタメンテナンス UI と同一ライブラリ、コンポーネント共通化） |
| ポート | 8090 |
| アクセス制限 | `app_initialized = true` になると HTTP 302 で 8080 ポートへリダイレクトし、ウィザードへのアクセスを終了する |
| 認証 | 初期化完了前は認証なし（ネットワーク的に閉域工場内のみアクセス可能を前提） |
| HTTPS | 工場内クローズドネットワークのため HTTP 許容。ただし外部公開時は HTTPS 必須（詳細設計で判断） |

## 3. ウィザード起動条件

```rust
// Rust API 起動時のチェック（axum ミドルウェア）
async fn check_initialization(State(db): State<DbPool>) -> impl IntoResponse {
    let initialized = sqlx::query_scalar!(
        "SELECT value FROM config_snapshots WHERE key = 'app_initialized'"
    )
    .fetch_optional(&db)
    .await
    .unwrap_or(None);

    match initialized.as_deref() {
        Some("true") => { /* 通常モードで起動 */ }
        _ => {
            // ウィザードモードで起動
            // ポート 8080 の API は /wizard エンドポイントのみ有効
            // ポート 8090 のウィザード UI を有効化
        }
    }
}
```

`config_snapshots` テーブルの `app_initialized` キーが `"true"` 以外（未存在・false・null）の場合、ウィザードモードで起動する。

## 4. ウィザードステップ定義（5ステップ）

### Step 1: PostgreSQL 初期化（sqlx migrate）

| 項目 | 内容 |
|------|------|
| 目的 | データベーススキーマを最新マイグレーションまで適用する |
| 実行内容 | `sqlx migrate run` をバックエンドで実行 |
| 冪等性 | 既に適用済みのマイグレーションはスキップ（sqlx の標準動作） |
| 完了判定 | すべてのマイグレーションが Applied 状態 |
| 失敗時 | エラー詳細を画面表示し、再実行ボタンを提供 |
| スキップ条件 | `job_executions` テーブルが存在 かつ マイグレーションバージョンが最新 |

```
画面表示:
[Step 1 of 5] データベース初期化
  ✓ マイグレーション適用中...
  ✓ 20260101_create_users.sql - Applied
  ✓ 20260102_create_trace_events.sql - Applied
  ...
  [次へ →]
```

### Step 2: system_admin ユーザー作成

| 項目 | 内容 |
|------|------|
| 目的 | 最初の管理者ユーザー（system_admin ロール）を作成する |
| 入力項目 | ユーザー名（表示名）/ メールアドレス / パスワード（確認入力含む） |
| バリデーション | パスワード最小 12文字・大小英数記号混在（SEC-001 準拠）|
| 冪等性 | system_admin ロールを持つユーザーが既に 1名以上存在する場合はスキップ |
| 完了判定 | `users` テーブルに `role='system_admin'` のレコードが 1件以上存在 |
| 失敗時 | バリデーションエラーをインライン表示 |

```
画面表示:
[Step 2 of 5] 管理者ユーザー作成
  表示名 [              ]
  メールアドレス [              ]
  パスワード [              ]
  パスワード（確認） [              ]
  [作成する →]
```

### Step 3: マスタデータインポート（CSV/Excel）

| 項目 | 内容 |
|------|------|
| 目的 | 品目マスタ・工程マスタ・部署マスタ等を一括インポートする |
| 対応形式 | CSV（UTF-8）/ Excel（.xlsx）|
| インポート対象 | 品目マスタ / 工程マスタ / 部署マスタ / SOP テンプレート（ZIP 形式）|
| 冪等性 | 同一 PRIMARY KEY のレコードは UPSERT（ON CONFLICT DO UPDATE）で更新 |
| 完了判定 | 少なくとも 1件の工程マスタが存在すること（任意完了：スキップ可能）|
| スキップ | 「後でインポートする」ボタンで Step 4 へ進める。マスタなしでもシステム起動可能。 |
| エラー時 | 行番号・エラー内容をテーブル表示し、修正ファイルの再アップロードを促す |

```
画面表示:
[Step 3 of 5] マスタデータインポート（省略可）
  [品目マスタ CSV をドロップ]  [工程マスタ CSV をドロップ]
  [部署マスタ CSV をドロップ]  [SOP ZIP をドロップ]
  [インポート実行] または [後でインポートする →]
```

### Step 4: ハンディデバイス登録

| 項目 | 内容 |
|------|------|
| 目的 | システムで使用するハンディ端末を事前登録する |
| 入力項目 | デバイス名称 / デバイス識別子（IMEI または任意の文字列）/ ライン割り当て（任意）|
| 冪等性 | 同一デバイス識別子が存在する場合は更新（UPSERT）|
| 完了判定 | 1台以上のデバイスが登録されていること（任意完了：スキップ可能）|
| スキップ | 「後で登録する」ボタンで Step 5 へ進める |
| 一括登録 | CSV による一括登録にも対応（デバイス識別子・名称列を持つ CSV）|

```
画面表示:
[Step 4 of 5] デバイス登録（省略可）
  デバイス名称 [              ]
  デバイス識別子 [              ]
  ライン [▼ 選択]
  [+ 追加]
  ---
  | デバイス名 | 識別子 | ライン |
  | HH-001 | IMEI:012... | ライン1 |
  [次へ →] または [後で登録する →]
```

### Step 5: システムヘルスチェック

| 項目 | 内容 |
|------|------|
| 目的 | セットアップ完了後のシステム全体の動作確認を行う |
| チェック内容 | PostgreSQL 接続確認 / API ヘルスチェック / マイグレーション適用確認 / ディスク空き容量確認（>10GB 推奨）|
| 完了判定 | 全チェック項目が OK になること（WARNING は許可、ERROR は再確認を促す）|
| 完了後の処理 | `config_snapshots` テーブルに `app_initialized = 'true'` を INSERT → ウィザードポータル（8090）を無効化 → 8080 ポートへリダイレクト |

```
画面表示:
[Step 5 of 5] システムヘルスチェック

  ✓ PostgreSQL 接続         OK
  ✓ マイグレーション適用     OK（最新: 20260501_v1）
  ✓ API ヘルスエンドポイント OK（200 / version: 1.0.0）
  ✓ ディスク空き容量         OK（234 GB 空き）
  ⚠ デバイス未登録          WARNING（後で登録可能）

  [セットアップ完了 → 運用開始]
```

完了ボタン押下後:

```sql
INSERT INTO config_snapshots (key, value, updated_at, updated_by)
VALUES ('app_initialized', 'true', NOW(), 'wizard')
ON CONFLICT (key) DO UPDATE SET value = 'true', updated_at = NOW();
```

## 5. ウィザードの再実行（べき等性）

ウィザードを再実行した場合（誤操作・設定変更が必要になった場合）、各ステップの完了状態を確認し、完了済みステップは「完了済み（スキップ可）」として表示する。ユーザーは完了済みステップを明示的に再実行することも可能。

```
[Step 1 of 5] データベース初期化  ✓ 完了済み [再実行]
[Step 2 of 5] 管理者ユーザー作成  ✓ 完了済み [再実行]
[Step 3 of 5] マスタデータインポート → 未完了 [実行する]
[Step 4 of 5] デバイス登録        ✓ 完了済み [再実行]
[Step 5 of 5] ヘルスチェック      → 未実行  [実行する]
```

`app_initialized = 'true'` の状態でウィザードを再実行するには、管理者設定画面（SCR-MC-020）から「セットアップウィザードを再度実行」を選択する（誤操作防止のため確認ダイアログ付き）。

---

## 6. ウィザード進行状態の永続化

ウィザードの各ステップ完了状態は `config_snapshots` テーブルに記録する。ブラウザを閉じても再開可能。

```sql
-- ウィザード進行状態の記録例
-- キー命名規則: wizard.step_{n}.completed
INSERT INTO config_snapshots (key, value, updated_at)
VALUES ('wizard.step_1.completed', 'true', NOW())
ON CONFLICT (key) DO UPDATE SET value = 'true', updated_at = NOW();
```

| キー | 意味 |
|------|------|
| `wizard.step_1.completed` | Step 1（DB 初期化）完了フラグ |
| `wizard.step_2.completed` | Step 2（管理者ユーザー作成）完了フラグ |
| `wizard.step_3.completed` | Step 3（マスタインポート）完了フラグ |
| `wizard.step_4.completed` | Step 4（デバイス登録）完了フラグ |
| `wizard.step_5.completed` | Step 5（ヘルスチェック）完了フラグ |
| `app_initialized` | 全ステップ完了・通常運用モード移行フラグ |

**本節で確定した方針**

- GUI 初期構築ウィザードはメインアプリとは独立したポート 8090 の React Web アプリとして稼働し、`config_snapshots.app_initialized` フラグが `true` になると自動的に無効化・リダイレクトする。初期化前のみアクセス可能とすることで、本番環境への誤アクセスを構造的に防ぐ。
- ウィザードの全ステップはべき等的に設計されており、途中中断・再実行・部分スキップが可能である。ステップ完了状態は `config_snapshots` テーブルに永続化されるため、ブラウザ再起動・サーバー再起動後も正確な再開地点から継続できる。
- Step 3（マスタインポート）および Step 4（デバイス登録）はシステム起動の必須条件ではなくスキップ可能とし、「まずシステムを起動してみる」という単一管理者の試行錯誤を支援する。
