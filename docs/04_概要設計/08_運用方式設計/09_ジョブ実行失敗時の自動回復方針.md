# ジョブ実行失敗時の自動回復方針

本節では、BAT-001〜010 の各バッチジョブが失敗した際の自動回復ロジック・リトライポリシー・DLQ 移行条件・アラート通知仕様を定義する。単一管理者（system_admin）運用を前提とし、ジョブ失敗をすべて手動対応とすることは現実的でないため、可能な限り自動回復を試みた上で、人手介入が不可避な場合にのみアラートを発報する設計とする。すべてのジョブは `job_executions` テーブルへの実行記録（べき等性管理）を前提とし、重複実行による二重処理を防止する。

---

## 1. 自動回復方針の共通原則

1. **まず自動リトライ**: ネットワーク瞬断・一時的な DB 負荷など、一過性の障害はリトライで解消する
2. **リトライ上限を設ける**: 無限リトライは避け、上限到達後は DLQ / アラートへ移行する
3. **人手介入が必要なエラーは即時アラート**: データ整合性破壊（ハッシュチェーン破断等）は即時 system_admin へ通知する
4. **べき等性の保証**: リトライ時も `job_executions` テーブルと idempotency_key により二重処理を防止する
5. **優先度（P1〜P3）によるアラートトリアージ**: 全障害を同じ重要度で通知せず、影響度に応じた分類を行う

---

## 2. アラート優先度定義

| 優先度 | 定義 | 対応時間目安 | 通知先 |
|--------|------|------------|--------|
| P1 | データ整合性・業務継続に即時影響（ハッシュチェーン破断、証跡喪失リスク等） | 即時（1時間以内） | system_admin + アラートダッシュボード |
| P2 | 業務機能に影響（バックアップ失敗、Outbox 蓄積等） | 当日中 | system_admin + アラートダッシュボード |
| P3 | 非クリティカル（アーカイブ遅延等、データは安全） | 翌営業日中 | アラートダッシュボードのみ |

## 3. BAT-002 Outbox リトライ失敗回復

### 3.1 リトライポリシー（指数バックオフ）

| 試行回数 | 待機時間 | 備考 |
|---------|---------|------|
| 1回目 | 1秒 | 即時再送 |
| 2回目 | 2秒 | |
| 3回目 | 4秒 | |
| 4回目 | 8秒 | |
| 5回目（最終） | 16秒 | この試行が失敗でDLQ移行 |

最大待機合計: 31秒（1+2+4+8+16）

### 3.2 DLQ 移行条件と後処理

```sql
-- 5回失敗後: outbox_dlq テーブルへ INSERT
INSERT INTO outbox_dlq (
    original_event_id, event_type, payload,
    failed_at, failure_reason, retry_count
)
SELECT id, event_type, payload, NOW(), last_error, 5
FROM outbox_events
WHERE id = :event_id;

-- outbox_events の status を 'dlq' に更新
UPDATE outbox_events SET status = 'dlq' WHERE id = :event_id;
```

DLQ 移行後:
- ERR-SYS-005 アラート（P1）を system_admin へ通知
- SCR-MC-007（DLQ 管理画面）で確認・手動再処理または破棄が可能
- MET-005（DLQ 件数）が SCR-MC-001 ダッシュボードに反映

### 3.3 DLQ からの手動回復

```
1. SCR-MC-007 で DLQ イベントの内容を確認
2. 再処理可能な場合: 「再送」ボタン → outbox_events へ差し戻し → BAT-002 次回実行で再送
3. 処理不要な場合: 「破棄」ボタン → outbox_dlq.status = 'discarded'
4. 再送時は idempotency_key をそのまま使用（サーバー側で重複を吸収）
```

## 4. BAT-006〜009 帳票生成失敗回復

### 4.1 共通リトライポリシー

| 試行回数 | 待機時間 | 備考 |
|---------|---------|------|
| 1回目 | 30秒 | |
| 2回目 | 60秒 | |
| 3回目（最終） | 120秒 | この試行が失敗で DLQ 移行 |

### 4.2 DLQ 移行条件

```sql
-- 3回失敗後: report_dlq テーブルへ INSERT
INSERT INTO report_dlq (
    report_type, trigger_event_id, trigger_event_type,
    failed_at, failure_reason, retry_count
)
VALUES (:report_type, :event_id, :event_type, NOW(), :error, 3);
```

- BAT-006（RP-001）/ BAT-007（RP-003）/ BAT-009（RP-005）: **P1 アラート**（品質証跡に直結）
- BAT-008（RP-004）: **P2 アラート**（改善記録、即時性は低い）

### 4.3 手動回復

`report_dlq` テーブルの内容を system_admin が SCR-MC-019（帳票 DLQ 管理画面）で確認し、トリガーイベントを再発火（再処理ボタン）することで帳票生成をリトライする。

## 5. BAT-003 ハッシュチェーン破断検出時の回復

ハッシュチェーン破断は品質証跡の完全性を脅かす最重大障害であり、自動回復は行わない。

### 5.1 検出後の自動処置（即時）

```
1. 以降の trace_events への新規イベント受付を停止
   （API エンドポイント /api/v1/trace_events を 503 で応答）
2. ERR-SEC-002 アラートを P1 で system_admin へ即時通知
3. SCR-MC-001 ダッシュボードに「ハッシュチェーン異常 - 管理者対応が必要」バナーを表示
4. 破断箇所・破断前後のレコードを audit_log へ記録
```

### 5.2 system_admin による承認後の復旧

```
1. system_admin が SCR-MC-008（ハッシュチェーン管理画面）にアクセス
2. 破断箇所と影響範囲を確認
3. 原因調査（改ざん / バグ / 移行時の不整合 を判別）
4. 対処完了後、「承認して再開」ボタンをクリック
   → hash_chain_breaks テーブルに resolved_at・resolved_by を記録
   → イベント受付を再開
5. BAT-003 を手動実行して再検証
```

**自動再開は禁止**: ハッシュチェーン破断は人手確認なしに回復させない。

## 6. BAT-001 バックアップ失敗時の回復

### 6.1 リトライポリシー

| 試行 | 待機時間 | トリガー |
|------|---------|---------|
| 1回目 | 1時間後 | 02:00 失敗 → 03:00 再試行 |
| 2回目 | 1時間後 | 03:00 失敗 → 04:00 再試行 |
| 3回目 | 1時間後 | 04:00 失敗 → 05:00 再試行（最終） |

### 6.2 アラート

3回連続失敗後:
- **P2 アラート**を system_admin へ通知
- MET-007（バックアップ最終成功経過時間）が警告/危険閾値を超えるため SCR-MC-001 でも可視化される

### 6.3 失敗時の暫定対応

system_admin がアラート受信後、手動で実行する。

```bash
/opt/work-nav/scripts/bat_001_backup.sh --force
```

## 7. BAT-004 アーカイブ失敗時の回復

### 7.1 リトライポリシー

```
当月1日 04:00 失敗 → 翌日 04:00 に自動リトライ
翌日も失敗 → P3 アラート（非クリティカル）
```

### 7.2 非クリティカルとする根拠

アーカイブ失敗時、対象データは引き続きホット層（メインテーブル）に保管される。データの喪失や整合性破壊は発生しない。アーカイブ遅延によってディスク使用量が増加する可能性があるため、MET-009（ディスク使用率）で監視し、85% を超えた場合に P2 アラートへ格上げする。

## 8. BAT-005 PII 匿名化失敗時の回復

### 8.1 リトライポリシー

四半期バッチのため、失敗後の即時リトライは行わない。

```
四半期バッチ失敗 → P2 アラートを system_admin へ通知
→ 翌四半期の実行まで保留
```

### 8.2 即時手動対応が必要な場合

個人情報保護の観点から、system_admin が意図的に手動実行する必要がある場合は `--force` オプションで実行する。

```bash
/opt/work-nav/scripts/bat_005_pii_anon.sh --force
```

## 9. BAT-010 集計レポート失敗時の回復

| 粒度 | リトライポリシー | 最終失敗時の対応 |
|------|---------------|----------------|
| 日次（06:00） | 07:00 に1回リトライ → 失敗で report_dlq | P2 アラート |
| 週次（月曜 07:00） | 月曜 08:00 に1回リトライ → 失敗で report_dlq | P2 アラート |
| 月次（1日 07:00） | 1日 09:00 に1回リトライ → 失敗で report_dlq | P2 アラート |

report_dlq に移行後は SCR-MC-019 から手動再処理が可能。

## 10. job_executions テーブルを用いたべき等性保証

```sql
-- ジョブ開始時: running レコードを INSERT（UNIQUE 制約でトランザクション保護）
INSERT INTO job_executions (job_id, execution_date, started_at, status)
VALUES (:job_id, CURRENT_DATE, NOW(), 'running')
ON CONFLICT (job_id, execution_date, status) DO NOTHING;
-- レコードが挿入できなければスキップ（既実行済みを意味する）

-- 成功時: running → success に更新
UPDATE job_executions
SET status = 'success', completed_at = NOW()
WHERE job_id = :job_id AND execution_date = CURRENT_DATE AND status = 'running';

-- 失敗時: running → failed に更新（retry_count インクリメント）
UPDATE job_executions
SET status = 'failed', completed_at = NOW(), retry_count = retry_count + 1, error_detail = :error
WHERE job_id = :job_id AND execution_date = CURRENT_DATE AND status = 'running';
```

## 11. アラート通知方式（概要）

通知経路: `alerts` テーブル INSERT → SCR-MC-001 ダッシュボードに反映（30秒リフレッシュ）

将来的な通知拡張（メール・Slack 等）は D-OPS-001（閉域ネットワーク制約）により現時点では実装しない。system_admin は定期的にダッシュボードを確認することを運用規約とする。

```sql
CREATE TABLE alerts (
    id          BIGSERIAL PRIMARY KEY,
    alert_code  TEXT NOT NULL,        -- 'ERR-SYS-005' 等
    priority    TEXT NOT NULL,        -- 'P1' | 'P2' | 'P3'
    source_job  TEXT,                 -- 'BAT-002' 等
    message     TEXT NOT NULL,
    status      TEXT NOT NULL DEFAULT 'active',  -- 'active' | 'acknowledged' | 'resolved'
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    resolved_at TIMESTAMPTZ,
    resolved_by TEXT
);
```

---

**本節で確定した方針**

- BAT-002（Outbox リトライ）は指数バックオフ（1s/2s/4s/8s/16s）で5回リトライ後、`outbox_dlq` テーブルへ移行し ERR-SYS-005（P1）アラートを発報する。SCR-MC-007 での手動再処理を回復手段とする。
- BAT-003（ハッシュチェーン破断）は自動回復を禁止とし、即時イベント受付停止・ERR-SEC-002（P1）発報後、system_admin が SCR-MC-008 で原因確認・承認を完了するまで縮退状態を維持する。
- すべての BAT ジョブは `job_executions` テーブルへの `running/success/failed` ステータス記録によってべき等性を保証し、意図しない二重実行を防止する。
