# バックアップ・リストア方式

本節では、製造業向け作業ナビゲーションシステムにおけるデータ保護の根幹をなすバックアップ・リストア方式を定義する。単一管理者運用（system_admin 1名）を前提としつつ、工場内の証跡データ（ハッシュチェーン）および業務マスタの喪失が品質保証上の重大リスクとなるため、自動化・暗号化・冗長転送を組み合わせた多層防御を採用する。Recovery Time Objective（RTO）15分・Recovery Point Objective（RPO）1時間を設計目標とし、月次リストアテストによる実効性検証を義務付ける。

---

## 1. バックアップ識別子と基本方針

| 識別子 | 種別 | 対象 | スケジュール | 保管先 |
|--------|------|------|--------------|--------|
| BAT-001 | フルバックアップ | PostgreSQL 全DB | 毎日 02:00 | ローカル + スタンバイ |
| BAT-002 | WALアーカイブ | PostgreSQL WALセグメント | 継続（archive_command） | ローカルアーカイブ領域 |
| BAT-001-SUB | 証跡ファイル同期 | 証跡（エビデンス）ファイル群 | 毎日 03:00（BAT-001 サブタスク） | スタンバイ rsync |

バックアップ戦略は「フルバックアップ＋WALアーカイブ」の二本柱とし、任意時点へのポイントインタイムリカバリ（PITR）を可能にする。

---

## 2. BAT-001: 日次フルバックアップ

### 2.1 実行手順

```
02:00  pg_dump --format=custom --compress=9 \
               --dbname=postgres://... \
               --file=/backup/db/work_nav_$(date +%Y%m%d).dump

02:05  gzip -k /backup/db/work_nav_$(date +%Y%m%d).dump
       # --format=custom は既に圧縮を内包するが、外層 gzip で
       # 転送サイズをさらに削減する。

02:10  openssl enc -aes-256-gcm \
               -K $KEY_004_HEX -iv $KEY_004_IV \
               -in  /backup/db/work_nav_$(date +%Y%m%d).dump.gz \
               -out /backup/db/work_nav_$(date +%Y%m%d).dump.gz.enc
       # 暗号化キー KEY-004 はセキュリティ方式設計 KM-004 参照

02:15  rsync -avz --checksum \
               /backup/db/work_nav_$(date +%Y%m%d).dump.gz.enc \
               standby:/backup/db/
```

- 暗号化アルゴリズム: AES-256-GCM（認証付き暗号、改ざん検知付き）
- 暗号化キー管理: KEY-004（キー管理台帳 KM-004 にて別途定義、Docker Secret または環境変数で注入）
- バックアップファイル命名規則: `work_nav_YYYYMMDD.dump.gz.enc`
- 保管期間: ローカル 30日、スタンバイ 90日（自動パージスクリプト併用）

### 2.2 idempotency 設計

`job_executions` テーブルに当日付けの BAT-001 成功レコードが存在する場合はスキップする。手動再実行時は `--force` フラグで強制実行可能。

### 2.3 失敗時ハンドリング

次節（BAT-001 失敗）および File 6（09_ジョブ実行失敗時の自動回復方針）を参照。

## 3. BAT-002: WALアーカイブ（継続的バックアップ）

### 3.1 postgresql.conf 設定

```ini
# postgresql.conf （抜粋）
wal_level = replica
archive_mode = on
archive_command = 'cp %p /wal_archive/%f && gzip /wal_archive/%f'
archive_timeout = 300          # 5分以内に強制アーカイブ（RPO補助）
```

- WAL保管期間: **14日間**（それ以前は自動削除、`find /wal_archive -mtime +14 -delete` を cron 実行）
- WALアーカイブ先: ローカル `/wal_archive/` + BAT-001 の rsync で翌 03:00 までにスタンバイへ転送

### 3.2 PITR（ポイントインタイムリカバリ）適用範囲

WAL + フルバックアップの組み合わせにより、バックアップ取得時点から最大で直前5分以内（archive_timeout=300秒）の時点まで復元可能。これが RPO=1時間の実現根拠となる（実際は約5分だが、運用マージンとして1時間をコミット値とする）。

---

## 4. BAT-001-SUB: 証跡ファイル同期

エビデンスファイル（撮影画像・PDF等）は PostgreSQL 外部に保管されるため、独立した rsync ジョブで同期する。

```
03:00  rsync -avz --checksum --delete \
               /data/evidence/ \
               standby:/data/evidence/
       # --checksum により差分のみ転送、転送後に MD5 照合
```

- 実行タイミング: BAT-001 完了後（03:00）
- 失敗時: BAT-001 と同じ P2 アラートポリシーを適用

---

## 5. リストア手順（5ステップ）

リストアは system_admin が手動で実施する。以下の手順は RTO=15分 を前提に設計されている。

### Step 1: Docker サービス停止

```bash
cd /opt/work-navigation-app
docker-compose stop api master-ui
# PostgreSQL コンテナはリストア中も停止する
docker-compose stop postgres
```

所要時間目安: 1分

### Step 2: pg_dump リストア（フルリカバリ）

```bash
# 暗号化解除
openssl enc -d -aes-256-gcm \
        -K $KEY_004_HEX -iv $KEY_004_IV \
        -in  /backup/db/work_nav_YYYYMMDD.dump.gz.enc \
        -out /backup/db/work_nav_YYYYMMDD.dump.gz

gunzip /backup/db/work_nav_YYYYMMDD.dump.gz

# リストア先コンテナ起動（PostgreSQL のみ）
docker-compose start postgres

# pg_restore 実行
docker exec -i postgres_container pg_restore \
        --dbname=work_navigation \
        --clean --if-exists \
        /backup/db/work_nav_YYYYMMDD.dump
```

所要時間目安: 5分

### Step 3: WAL 適用（PITR）

障害直前時点へのポイントインタイムリカバリが必要な場合のみ実施する。

```bash
# recovery.conf（PostgreSQL 12+: postgresql.conf + standby.signal）
cat >> /var/lib/postgresql/data/postgresql.conf <<EOF
restore_command = 'cp /wal_archive/%f %p'
recovery_target_time = '2026-MM-DD HH:MM:SS'
recovery_target_action = 'promote'
EOF

touch /var/lib/postgresql/data/standby.signal
docker-compose restart postgres
# PostgreSQL がリカバリモードで起動し、WALを順次適用する
```

所要時間目安: 3分（WAL量による）

### Step 4: 証跡ファイルリストア

```bash
rsync -avz standby:/data/evidence/ /data/evidence/
```

所要時間目安: 2分（ファイルサイズによる）

### Step 5: ハッシュチェーン整合性検証

```bash
# API 起動前に整合性検証を実行（BAT-003 相当の単体実行）
docker-compose run --rm api ./hash-chain-verify --full
# 正常: 0 exit, "hash chain integrity: OK"
# 異常: exit 1, ERR-SEC-002 を標準エラーに出力 → 管理者確認必須
docker-compose up -d api master-ui
```

所要時間目安: 2分

**合計目安: 13分（RTO=15分 の範囲内）**

---

## 6. RTO / RPO 定義

| 指標 | 目標値 | 根拠 |
|------|--------|------|
| RTO（Recovery Time Objective） | **15分（手動）** | 上記5ステップの所要時間合計 |
| RPO（Recovery Point Objective） | **1時間** | WAL archive_timeout=5分、フルバックアップ間隔24時間、運用マージン考慮 |

- 自動フェイルオーバーは対象外（D-OPS-004 参照、99_前提制約参照）
- スタンバイサーバーへの切り替えも手動操作とする

---

## 7. 月次リストアテスト

| 項目 | 内容 |
|------|------|
| 実施頻度 | **毎月1回**（第1土曜日推奨） |
| 実施者 | system_admin |
| テスト環境 | スタンバイサーバーまたは検証用 Docker 環境 |
| テスト内容 | 上記5ステップを完全実施し、ハッシュチェーン整合性検証 OK を確認 |
| 結果記録 | 運用ログ（`maintenance_log` テーブル）に日時・実施者・結果を記録 |
| 合否基準 | Step 5 が正常終了し、RTO 15分以内に完了すること |

月次テストにより「バックアップは取得できているがリストアできない」という形骸化リスクを排除する。

---

## 8. バックアップ保管ディレクトリ構成

```
/backup/
  db/
    work_nav_YYYYMMDD.dump.gz.enc   # 日次フルバックアップ（暗号化済み）
    work_nav_YYYYMMDD.dump.gz.enc.sha256  # 整合性検証用チェックサム
  wal/
    /wal_archive/                   # WAL アーカイブ
/data/
  evidence/                         # 証跡ファイル（rsync 対象）
```

---

> **手順具体化先**: `docs/09_運用・保守/運用手順/02_バックアップ運用と完了確認（OPS-PROC-002）.md`（確認手順）および `docs/09_運用・保守/運用手順/03_月次リストア検証手順（OPS-PROC-003）.md`（リストア実証手順）に物理手順を確定した。

**本節で確定した方針**

- BAT-001（日次フルバックアップ）は `pg_dump --format=custom` + gzip + AES-256-GCM 暗号化（KEY-004）の三段構成とし、ローカルおよびスタンバイサーバーへ二重保管する。
- WALアーカイブ（BAT-002）を併用することで RPO=1時間、PITR による任意時点リストアを実現する（WAL保管期間 14日）。
- 月次リストアテストを system_admin が手動実施し、RTO=15分 以内の完全復旧を `maintenance_log` に記録することを義務付ける。
