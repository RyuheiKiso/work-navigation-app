# デプロイ・リリース方式

本節では、作業ナビゲーションシステムのリリース・デプロイに関する方針と手順を定義する。個人開発・単一管理者運用を前提とするため、CI/CD パイプラインは設けず、system_admin が手動で直接デプロイを実施する。ただし手順の明文化・バージョン管理・ロールバック手段の確保によって、デプロイ品質と再現性を担保する。稼働中サービスへの無停止デプロイを基本とし、スキーマ変更を伴う場合のみ計画メンテナンス時間（低負荷帯）を設定する。

---

## 1. CI/CD パイプライン不採用の根拠

本システムは個人開発・単一管理者・閉域工場ネットワークという運用制約を持つ。GitHub Actions / GitLab CI 等のクラウド CI/CD サービスは、閉域ネットワークからアクセス不可であり、かつ 1人のデプロイ頻度に対してオーバーエンジニアリングとなる。よって CI/CD パイプラインは設けず、以下の手動デプロイ手順と事前チェックリストによって品質を担保する。

---

## 2. バージョン管理方針（SemVer）

セマンティックバージョニング（MAJOR.MINOR.PATCH）を採用する。

| バージョン番号 | 変更種別 | 例 |
|----------------|----------|----|
| MAJOR | 破壊的スキーマ変更（マイグレーションロールバック不可を含む） | 1.0.0 → 2.0.0 |
| MINOR | 後方互換機能追加（新 API エンドポイント・新マスタ項目等） | 1.0.0 → 1.1.0 |
| PATCH | バグ修正・セキュリティパッチ（スキーマ変更なし） | 1.0.0 → 1.0.1 |

- バージョンは `Cargo.toml`（Rust API）および `package.json`（React マスタメンテナンス UI）に記載する
- Docker イメージタグはバージョン番号をそのまま使用する（例: `work-nav-api:1.2.3`）
- `latest` タグのみへの依存を禁止する（ロールバック不能になるため）

---

## 3. リリースノート管理

- `CHANGELOG.md` を keep-a-changelog フォーマット（https://keepachangelog.com/）で管理する
- リリース前に CHANGELOG.md の `[Unreleased]` セクションをバージョン番号付きセクションへ昇格させる
- 記載カテゴリ: Added / Changed / Deprecated / Removed / Fixed / Security

---

## 4. リリース前チェックリスト

デプロイ実施前に以下の全項目が完了していることを確認する。

```
[ ] 1. BAT-001（日次バックアップ）が当日分で正常完了していること
[ ] 2. ローカルでのテスト（cargo test / npm test）が全件 PASS していること
[ ] 3. スキーマ変更がある場合: sqlx migrate --dry-run が正常終了していること
[ ] 4. CHANGELOG.md の [Unreleased] → バージョン番号への昇格が完了していること
[ ] 5. Docker イメージのビルドが成功していること（docker build のエラーなし）
[ ] 6. 本番環境と同じ .env / Docker Compose 設定でローカル動作確認済みであること
```

## 5. リリース手順（5ステップ）

### Step 1: バックアップ確認と直前バックアップ

```bash
# BAT-001 当日分の正常完了を確認
psql -c "SELECT * FROM job_executions WHERE job_id='BAT-001' AND execution_date=CURRENT_DATE AND status='success';"

# 未実行の場合は手動で即時バックアップ実行
/opt/work-nav/scripts/bat_001_backup.sh --force
```

### Step 2: Docker イメージ取得

```bash
# 最新イメージをプル（または git pull + ビルド）
docker pull ghcr.io/ryuhei_kiso/work-nav-api:1.2.3
docker pull ghcr.io/ryuhei_kiso/work-nav-master-ui:1.2.3

# ローカルビルドの場合（閉域ネットワーク）
cd /opt/work-navigation-app
git pull origin main
docker-compose build --no-cache
```

### Step 3: ローリングリスタート（ゼロダウンタイム）

```bash
# Rust API のローリングリスタート（1インスタンス構成）
docker-compose up -d --no-deps api
# 旧コンテナは 30秒のグレースピリオド後に停止
# （in-flight リクエストを処理完了させてから切断）
```

- グレースピリオド: 30秒（`docker-compose.yml` の `stop_grace_period: 30s`）
- React マスタメンテナンス UI は静的ファイル配信のため即時切り替え可能

### Step 4: データベースマイグレーション

```bash
# スキーマ変更がある場合のみ実施
docker-compose run --rm api sqlx migrate run

# マイグレーション確認
docker-compose run --rm api sqlx migrate info
# すべてのマイグレーションが Applied 状態であることを確認
```

スキーマ変更がない場合（PATCH バージョン等）は本 Step をスキップする。

### Step 5: ヘルスチェック

```bash
# API ヘルスチェック
curl -f http://localhost:8080/health
# 期待レスポンス: 200 OK {"status":"healthy","version":"1.2.3"}

# PostgreSQL 接続確認
docker-compose run --rm api sqlx database check

# 主要エンドポイント疎通確認
curl -f http://localhost:8080/api/v1/works?limit=1
curl -f http://localhost:9090/health   # マスタ UI

echo "リリース完了: $(date)"
```

## 6. ゼロダウンタイム戦略

| コンポーネント | 戦略 | 根拠 |
|----------------|------|------|
| Rust API | ローリングリスタート（30秒グレースピリオド） | 1インスタンス構成、in-flight リクエスト保護 |
| React マスタ UI | 即時切り替え（静的ファイル置き換え） | 状態を持たないため切り替えコストなし |
| PostgreSQL | 無停止（マイグレーション online） | sqlx migrate は接続を維持したまま実行 |
| ハンディアプリ | オフラインキャッシュで継続動作 | SQLite + Outbox により API 短断を吸収 |

- MAJOR バージョン（破壊的スキーマ変更）の場合は計画メンテナンス（低負荷帯: 深夜02:00〜04:00）を設定する

---

## 7. ロールバック手順

### 7.1 スキーマ変更なし（PATCH/MINOR の一部）

```bash
# 旧イメージタグを指定して再デプロイ
docker-compose up -d --no-deps api
# docker-compose.yml の image タグを旧バージョンに戻す
```

### 7.2 スキーマ変更あり（sqlx migrate が実施済み）

```bash
# マイグレーションのリバート（down マイグレーションが存在する場合）
docker-compose run --rm api sqlx migrate revert

# down マイグレーションが存在しない場合（MAJOR バージョン変更）
# → BAT-001 で取得したバックアップからリストア（04_バックアップ・リストア方式 参照）
```

- `revert` は直近1マイグレーションのみ取り消す。複数バージョン戻す場合は繰り返す。
- 破壊的スキーマ変更（データ削除を伴うもの）は `down` マイグレーションで完全復元できない場合があるため、**必ず事前バックアップを完了させること**（Step 1 で担保）。

---

## 8. デプロイ記録

デプロイ完了後、`maintenance_log` テーブルへ記録する。

```sql
INSERT INTO maintenance_log (log_type, version, performed_by, performed_at, detail)
VALUES ('deploy', '1.2.3', 'system_admin', NOW(), 'ローリングリスタート完了、スキーマ変更なし');
```

---

**本節で確定した方針**

- CI/CD パイプラインは採用せず、system_admin が 5ステップの手順書に従い手動デプロイを実施する。デプロイ前チェックリスト（バックアップ確認・テスト PASS・マイグレーション dry-run）を必須とする。
- バージョン管理は SemVer を採用し、MAJOR = 破壊的スキーマ変更 / MINOR = 機能追加 / PATCH = バグ修正とする。Docker イメージには必ずバージョンタグを付与し、`latest` タグのみへの依存を禁止する。
- ロールバックは「旧イメージタグへの切り替え」を基本とし、スキーマ変更を伴う場合のみ `sqlx migrate revert` または BAT-001 バックアップからのリストアを実施する。
