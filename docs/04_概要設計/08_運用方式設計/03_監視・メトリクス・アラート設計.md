# 03 監視・メトリクス・アラート設計

本章の責務は、社内 LAN 完結・外部監視 SaaS 禁止という制約下での監視・メトリクス・アラート設計命題を確定することである。MET-001〜010 の全メトリクスカタログ・Prometheus スクレイプ構成・Grafana ダッシュボード（SCR-MC-001）の表示設計・アラートルール 4 件の閾値と通知先を確定する。上流の NFR-OPS-043/044・NFR-AVL-001 および 01 章の「個人開発・個人運用」ポリシーを受けて確定する。

---

## 1. 監視基盤の構成確定

### 1-1. 監視コンポーネント

本システムで採用する監視基盤を以下に確定する。外部監視 SaaS（Datadog / New Relic / PagerDuty 等）は社内 LAN 完結・インターネット接続なしの制約から対象外と判断する。

| コンポーネント | 製品 | バージョン | 配置 | ポート |
|---|---|---|---|---|
| メトリクス収集 | Prometheus | 最新 LTS | Docker Compose（コンテナ） | 9090（内部） |
| ダッシュボード | Grafana | 最新 LTS | Docker Compose（コンテナ） | 3000（内部）|
| メトリクスエンドポイント | Rust axum `/metrics` | — | Rust API コンテナ | 8080（内部） |
| アラート通知 | Grafana Alerting | — | Grafana コンテナ | — |

Grafana（port 3000）は管理コンソール（SCR-MC-001）から社内ブラウザ経由でアクセスする。外部からのアクセスは禁止し、IIS リバースプロキシで管理コンソール認証通過後のみ転送する（07_セキュリティ方式設計参照）。

### 1-2. Prometheus スクレイプ設定方針

| 設定項目 | 値 | 根拠 |
|---|---|---|
| スクレイプ間隔 | 15 秒 | 個人運用・軽量構成として標準値を採用。5 秒未満は不要 |
| 評価間隔 | 15 秒 | スクレイプ間隔と同一（Prometheus デフォルト）|
| データ保持期間 | 15 日 | ローカルディスク容量制約。30 日以上はディスク消費超過となるため採用しない |
| ターゲット | `api:8080/metrics`（Docker 内部 DNS）| Docker Compose サービス名で解決 |

## 2. MET-001〜010 メトリクスカタログ

### 2-1. 全メトリクス定義

| MET-ID | メトリクス名称 | Prometheus メトリクス名 | カテゴリ | SLO 値 | SLO 算定窓 | アラート閾値 | 関連 NFR |
|---|---|---|---|---|---|---|---|
| MET-001 | API ステップイベント POST P99 レイテンシ | `api_step_event_post_duration_p99_ms` | Latency | ≤200ms | 30 日 | >500ms | NFR-PRF-001 |
| MET-002 | API マスタ同期 P95 レイテンシ | `api_master_sync_duration_p95_ms` | Latency | ≤1000ms | 30 日 | >5000ms | NFR-PRF-003 |
| MET-003 | API 可用性（月次）| `api_availability_ratio` | Availability | ≥99.5% | 月次 | <99.0% 3 ヶ月連続 | NFR-AVL-001 |
| MET-004 | Outbox キュー遅延秒数 | `outbox_queue_lag_seconds` | Lag | ≤60s p95 | 30 日 | >300s | NFR-AVL-002 |
| MET-005 | Outbox DLQ 件数 | `outbox_dlq_total` | Error | =0 | — | >CFG-011 設定値（デフォルト 100）| FR-SY-008 |
| MET-006 | ハッシュチェーン破損件数 | `hash_chain_break_count_total` | Error | =0 | — | >0（即時アラート）| NFR-SEC-040 |
| MET-007 | PostgreSQL アクティブ接続数比率 | `pg_connections_used_ratio` | Saturation | ≤80% | — | >80% | NFR-AVL-001 |
| MET-008 | 画面描画 P95 レイテンシ | `screen_render_duration_p95_ms` | Latency | ≤500ms | 30 日 | >2000ms | NFR-PRF-003 |
| MET-009 | バックアップ経過時間（最終バックアップからの時間）| `backup_age_hours` | Freshness | ≤24h | — | >25h（アラート発火）| NFR-OPS-033, NFR-OPS-046 |
| MET-010 | ディスク使用率 | `disk_usage_percent{mount="/"}` | Saturation | ≤80% | — | >85%（アラート発火）| NFR-OPS-002 |

### 2-2. Prometheus メトリクス実装方針

Rust axum の `/metrics` エンドポイントは `prometheus` crate（または `axum-prometheus` ラッパー）を使用する。エンドポイントには認証なしでアクセス可能とするが、外部ネットワークには公開しない（Docker 内部 DNS のみ）。

| メトリクス種別 | Prometheus 型 | 採用メトリクス |
|---|---|---|
| レイテンシ分布 | Histogram | MET-001, MET-002, MET-008 |
| 可用性比率 | Gauge | MET-003 |
| キュー遅延 | Gauge | MET-004 |
| カウンター（単調増加）| Counter | MET-005, MET-006 |
| リソース使用率 | Gauge | MET-007, MET-010 |
| バックアップ経過時間 | Gauge | MET-009 |

### 3-2. ダッシュボードアクセス制御

Grafana は内部監視ツールとして、以下のアクセス制御を適用する。

| 設定 | 内容 |
|---|---|
| 外部公開 | 禁止（IIS リバースプロキシで system_admin 認証通過後のみ転送） |
| Grafana ローカル認証 | system_admin アカウントのみ（Grafana Admin 権限）|
| 匿名アクセス | 禁止（grafana.ini `auth.anonymous.enabled = false`）|

## 4. アラートルール設計

### 4-1. アラートルール 4 件の確定

本システムのアラートルールを以下の 4 件に確定する。過剰なアラートは system_admin の対応疲弊を引き起こすため、「対応必須の事象のみアラート発火」を原則とする。

| アラート ID | アラート名 | 発火条件 | 重大度 | 通知先 | 関連 MET |
|---|---|---|---|---|---|
| ALERT-001 | DLQ 超過 | `outbox_dlq_total > CFG-011`（デフォルト 100 件）| CRITICAL | system_admin | MET-005 |
| ALERT-002 | ディスク使用率超過 | `disk_usage_percent > 85%` | HIGH | system_admin | MET-010 |
| ALERT-003 | API P95 レイテンシ超過 | `api_master_sync_duration_p95_ms > 5000`（5 秒）| WARN | system_admin | MET-002 |
| ALERT-004 | バックアップ齢超過 | `backup_age_hours > 25` | HIGH | system_admin | MET-009 |

### 4-2. アラート通知方式

アラートの通知先は system_admin 単独とする。外部メール・Slack・SMS は対象外と判断する（インターネット接続なし・個人開発制約）。

| 通知手段 | 実装方式 | 備考 |
|---|---|---|
| Grafana アラートパネル | Grafana ダッシュボード（SCR-MC-001）の色変化・バナー表示 | system_admin が SCR-MC-001 を確認した際に視覚的に把握する |
| ローカル smtp（任意）| Windows Server 上の IIS SMTP を使用した社内メール通知 | LAN 内に SMTP サーバーがある場合のみ設定可。NFR-OPS-044 の方針と整合する範囲で採用 |

ALERT-001（DLQ 超過）は業務データの損失リスクがあるため最優先で対応する。ALERT-002/004 は当日中に対応する。ALERT-003 はユーザー体験への影響を確認し、P3 インシデントとして週次ヘルスチェックでフォローする。

### 4-3. アラート閾値の設定変更

アラート閾値は以下の CFG 識別子で変更可能とする。変更は system_admin のみが実施できる。

| アラート ID | 閾値変更用 CFG | デフォルト値 | 変更可否 |
|---|---|---|---|
| ALERT-001 | CFG-011（DLQ しきい値）| 100 件 | system_admin のみ変更可 |
| ALERT-002 | — | 85%（固定）| 変更不可 |
| ALERT-003 | — | 5000ms（固定）| 変更不可 |
| ALERT-004 | — | 25h（固定）| 変更不可 |

## 5. 外部監視サービス非採用の根拠

本システムは以下の理由から外部監視 SaaS を採用しない。この判断は NFR-OPS-025 を継承する。

| 非採用理由 | 説明 |
|---|---|
| インターネット接続なし | 工場 LAN 内にインターネット接続を持たない閉域ネットワーク（C-OPS-003）|
| 個人開発コスト制約 | 外部 SaaS の月額費用が個人開発・単一工場の便益に見合わない |
| データ外部送信禁止 | 作業実績・監査証跡データを外部サービスに送信することは BR-BUS-028（用途三限定）に抵触する可能性がある |
| 単一担当者運用 | IT 担当 1 名の兼務体制では外部 SaaS のオンコール対応が設定できない |

---

**本節で確定した方針**
- **MET-001〜010 の全 10 件のメトリクスカタログ（SLO 値・Prometheus メトリクス名・アラート閾値）を確定し、Prometheus（15 秒スクレイプ・15 日保持）+ Grafana（SCR-MC-001 常時表示）を監視基盤として確定した。**
- **アラートルールを DLQ 超過（CFG-011 設定値超過）・ディスク使用率 85% 超・API P95 5000ms 超・バックアップ齢 25h 超の 4 件に絞り、過剰アラートによる system_admin の対応疲弊を防ぐ設計を確定した。**
- **外部監視 SaaS・自動オンコール通知を社内 LAN 完結・個人開発・BR-BUS-028 制約から対象外と判断し、Grafana ダッシュボードの視覚的表示と任意の社内 SMTP 通知のみを採用した。**
