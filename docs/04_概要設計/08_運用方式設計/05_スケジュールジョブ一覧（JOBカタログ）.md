# スケジュールジョブ一覧（JOBカタログ）

本節では、作業ナビゲーションシステムが定期・イベント駆動で実行するすべてのバッチジョブ（BAT-001〜010）を一覧化し、スケジュール・トリガー条件・べき等性設計・失敗時ハンドリングを確定する。個人運用・単一管理者を前提とするため、ジョブ実行基盤は Docker Compose + cron（または Windows タスクスケジューラ）を使用し、外部ジョブスケジューラ（Quartz / Temporal 等）は採用しない。すべてのジョブは `job_executions` テーブルへの実行記録を必須とし、べき等性を保証する。

## 2. BAT-001: 日次フルバックアップ

| 項目 | 内容 |
|------|------|
| 識別子 | BAT-001 |
| ジョブ名 | 日次フルバックアップ |
| スケジュール | 毎日 02:00（cron: `0 2 * * *`） |
| トリガー種別 | 時刻 |
| 処理概要 | `pg_dump --format=custom` → gzip → AES-256-GCM 暗号化（KEY-004） → ローカル保存 → スタンバイ rsync |
| 対象テーブル / ファイル | PostgreSQL 全 DB + 証跡ファイル（BAT-001-SUB） |
| 期待完了時間 | 15分以内 |
| べき等性設計 | `job_executions` テーブルに当日 BAT-001 成功レコードが存在する場合はスキップ（`--force` で強制実行） |
| 失敗時ハンドリング | 1時間後にリトライ × 3回 → 3回連続失敗で P2 アラートを system_admin へ通知 |
| 依存関係 | なし（ジョブチェーンの先頭） |

## 3. BAT-002: Outbox リトライ

| 項目 | 内容 |
|------|------|
| 識別子 | BAT-002 |
| ジョブ名 | Outbox リトライ処理 |
| スケジュール | 5分毎（cron: `*/5 * * * *`） |
| トリガー種別 | 時刻（準イベント駆動） |
| 処理概要 | `outbox_events` テーブルの未送信レコードを取得し、エンドポイントへ再送。ネットワーク復旧後のオフライン蓄積データをバルク同期する。 |
| 対象テーブル | `outbox_events`（status = 'pending' または 'failed', retry_count < 5） |
| べき等性設計 | イベント ID のべき等キー（idempotency_key）をサーバー側で検証。重複送信は 409 Conflict で受信側が吸収。 |
| リトライポリシー | 指数バックオフ: 1s → 2s → 4s → 8s → 16s（計5回）|
| DLQ 移行条件 | retry_count = 5 かつ最終失敗後 16秒経過で `outbox_dlq` テーブルへ INSERT |
| 失敗時ハンドリング | DLQ 移行後、ERR-SYS-005 アラートを system_admin へ通知。DLQ は SCR-MC-007 で確認可能。 |
| 依存関係 | ネットワーク接続（オフライン時は空振り終了、エラーとしない） |

## 4. BAT-003: 週次ハッシュチェーン検証

| 項目 | 内容 |
|------|------|
| 識別子 | BAT-003 |
| ジョブ名 | 週次ハッシュチェーン整合性検証 |
| スケジュール | 毎週月曜 03:00（cron: `0 3 * * 1`） |
| トリガー種別 | 時刻 |
| 処理概要 | `trace_events` テーブルの全レコードについて前レコードとのハッシュチェーン連続性を検証。破断箇所を特定し報告。 |
| 対象テーブル | `trace_events`（全レコード） |
| べき等性設計 | 検証のみ（書き込みなし）。複数回実行しても副作用なし。 |
| 正常終了条件 | ハッシュチェーン全件連続、exit 0 |
| 失敗時ハンドリング | ハッシュチェーン破断を検出した場合: (1) 以降のイベント受付を停止、(2) ERR-SEC-002 アラートを system_admin へ通知、(3) system_admin が原因確認・承認を行うまでシステムを縮退モードへ移行 |
| 承認後の復旧 | system_admin が管理画面（SCR-MC-008）で承認操作 → イベント受付再開 |
| 依存関係 | BAT-001 完了後（ハッシュチェーン検証はバックアップ後が望ましい） |

## 5. BAT-004: データアーカイブ（ティア移行）

| 項目 | 内容 |
|------|------|
| 識別子 | BAT-004 |
| ジョブ名 | データアーカイブ（ホット→ウォーム→コールド） |
| スケジュール | 毎月1日 04:00（cron: `0 4 1 * *`） |
| トリガー種別 | 時刻 |
| 処理概要 | `trace_events` の作成日時が 90日以上前のレコードをウォームストレージ（圧縮テーブル）へ移行。180日以上をコールドストレージ（別スキーマ / 外部ストレージ）へ移行。 |
| ティア定義 | ホット: 0〜90日（メインテーブル）/ ウォーム: 91〜180日（圧縮パーティション）/ コールド: 181日〜（外部ストレージ or 別 DB） |
| べき等性設計 | 移行フラグ（`archived_at`）チェックにより重複移行を防止。再実行時は未移行レコードのみ対象。 |
| 失敗時ハンドリング | 翌日 04:00 に自動リトライ。連続失敗後 P3 アラート（非クリティカル、データはホット層に残留）。 |
| 依存関係 | BAT-001 完了後（アーカイブ前にバックアップ取得が完了していること） |

## 6. BAT-005: PII 匿名化

| 項目 | 内容 |
|------|------|
| 識別子 | BAT-005 |
| ジョブ名 | 個人情報（PII）匿名化 |
| スケジュール | 四半期（1月・4月・7月・10月の1日 05:00、cron: `0 5 1 1,4,7,10 *`） |
| トリガー種別 | 時刻 |
| 処理概要 | 保管期限を超過したユーザー名等の PII を SHA-256 ハッシュ値に置換。元のテキストは復元不可とする。 |
| 対象カラム | `users.display_name`（退職フラグ立脚後 1年経過分）、`trace_events.operator_name`（作業完了後 2年経過分） |
| べき等性設計 | `anonymized_at` カラムが NULL のレコードのみ処理。既匿名化レコードはスキップ。 |
| ハッシュ方式 | `SHA-256(original_value || salt)` （salt = インスタンス固有シークレット KEY-005） |
| 失敗時ハンドリング | P2 アラート + 翌四半期まで保留（データ喪失リスクが低いため）。失敗詳細を `job_executions` に記録。 |
| 依存関係 | BAT-001 完了後（匿名化前バックアップ取得を保証） |

## 12. job_executions テーブル定義（概要）

すべての BAT ジョブは実行開始・完了・失敗を `job_executions` テーブルへ記録する。

```sql
CREATE TABLE job_executions (
    id              BIGSERIAL PRIMARY KEY,
    job_id          TEXT NOT NULL,           -- 'BAT-001' 等
    execution_date  DATE NOT NULL,           -- 実行日
    started_at      TIMESTAMPTZ NOT NULL,
    completed_at    TIMESTAMPTZ,
    status          TEXT NOT NULL            -- 'running' | 'success' | 'failed' | 'skipped',
    retry_count     INT NOT NULL DEFAULT 0,
    error_detail    TEXT,
    UNIQUE (job_id, execution_date, status)  -- べき等性保証
);
```

## 13. cron スケジュール一覧（まとめ）

```cron
# BAT-001: 日次フルバックアップ
0  2  *  *  *   /opt/work-nav/scripts/bat_001_backup.sh

# BAT-002: Outbox リトライ
*/5  *  *  *  *   /opt/work-nav/scripts/bat_002_outbox_retry.sh

# BAT-003: 週次ハッシュチェーン検証
0  3  *  *  1   /opt/work-nav/scripts/bat_003_hash_verify.sh

# BAT-004: 月次データアーカイブ
0  4  1  *  *   /opt/work-nav/scripts/bat_004_archive.sh

# BAT-005: 四半期 PII 匿名化
0  5  1  1,4,7,10  *   /opt/work-nav/scripts/bat_005_pii_anon.sh

# BAT-010: RP-006 日次集計レポート
0  6  *  *  *   /opt/work-nav/scripts/bat_010_report.sh daily

# BAT-010: RP-006 週次集計レポート
0  7  *  *  1   /opt/work-nav/scripts/bat_010_report.sh weekly

# BAT-010: RP-006 月次集計レポート
0  7  1  *  *   /opt/work-nav/scripts/bat_010_report.sh monthly
```

BAT-006〜009 はイベント駆動のため cron 登録なし。Axum の内部イベントハンドラから非同期タスクとして起動される。

---

**本節で確定した方針**

- すべての BAT ジョブは `job_executions` テーブルへの実行記録を通じてべき等性を保証し、再実行時は既完了ステップをスキップすることで二重処理を防止する。
- イベント駆動ジョブ（BAT-006〜009）は Outbox パターンを経由して発火し、3回リトライ後に `report_dlq` へ移行、P1/P2 アラートで system_admin へ通知する。
- BAT-002（Outbox リトライ）は指数バックオフ（最大16秒）後に DLQ 移行し、ERR-SYS-005 で通知することで、オフライン蓄積データの確実な同期と詰まりの可視化を両立する。
