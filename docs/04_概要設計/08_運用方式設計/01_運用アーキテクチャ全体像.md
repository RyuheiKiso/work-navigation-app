# 01 運用アーキテクチャ全体像

本章の責務は、IT 担当 1 名（system_admin）という不変の制約下で本システムの安定稼働を実現する運用アーキテクチャを確定することである。「個人開発・個人運用」を設計の第一前提とし、自動化可能なタスクはすべて BAT ジョブに移譲し、system_admin が手動で介入するタスクを必要最小限に絞った設計命題を確定する。上流の NFR-OPS-001〜003・NFR-AVL-005・OPS-001〜008 を受けて、本章で運用責任分担・定期タスク体系・緊急エスカレーションパスを確定する。

---

## 1. 個人開発・個人運用ポリシーの宣言

### 1-1. 設計の第一前提

本システムの運用は以下の 3 制約を必須前提として設計する。これらの制約は他サブとの分界に関わらず、本サブの全設計命題に優先する。

| 制約 ID | 制約内容 | 緩和条件 |
|---|---|---|
| C-OPS-001 | 運用担当者は system_admin 1 名（IT 担当兼務）のみとする | IT 担当増員を前提とした設計変更を禁止する |
| C-OPS-002 | 通常運用タスクの週次合計を 2 時間以内に収める | 2 時間超のタスクが恒常化する設計を禁止する |
| C-OPS-003 | インターネット接続を必要とする外部監視 SaaS を使用しない | 社内 LAN 内で完結する監視基盤のみ採用する |

「個人開発・個人運用」の宣言は、過剰なエンタープライズ監視基盤（PagerDuty・Datadog・New Relic 等）の導入禁止を意味する。本システムは Windows Server 2022 + WSL2 + Docker Compose という既存インフラ上で動作し、Prometheus + Grafana（Docker Compose 内）を唯一の監視基盤として採用する。

## 4. 定期運用タスク体系

### 4-1. 日次タスク（自動実行・手動確認不要）

| タスク | 実行時刻（JST） | 担当 BAT | 失敗時アクション |
|---|---|---|---|
| PostgreSQL フルダンプ | 01:00 | BAT-005 | DLQ 転送・翌朝ヘルスチェックで検知 |
| PII 匿名化 | 03:00 | BAT-004 | DLQ 転送・MET-005 アラート発火 |
| 証明書有効期限確認 | 09:00 | BAT-009 | system_admin へ通知（有効期限 30 日前から）|

### 4-2. 週次タスク（自動 + system_admin 手動確認）

| タスク | 実行タイミング | 担当 | 所要時間 |
|---|---|---|---|
| ハッシュチェーン整合性検証 | 日曜 02:00（BAT-001） | 自動 | — |
| 週次ヘルスチェックリスト（OPS-PROC-001） | 毎週月曜朝 | system_admin | 30 分以内 |
| バックアップ完了確認（OPS-PROC-002） | ヘルスチェック時 | system_admin | 15 分以内 |
| アプリケーションログの ERROR/WARN 確認 | 毎週月曜朝 | system_admin | 30 分以内 |
| cargo audit 実行・脆弱性確認 | 毎週（任意曜日）| system_admin | 30 分以内 |

### 4-3. 月次タスク（system_admin 手動実行）

|---|---|---|---|---|

| pg_stat_statements スロークエリ確認 | 月次 | system_admin | 30 分以内 | — |
| OS・Docker イメージ月次更新 | 月次・夜間計画停止時 | system_admin | 120 分以内 | — |

### 4-4. 四半期タスク（system_admin 手動実行）

| タスク | 実行タイミング | 担当 | 所要時間 | 関連手順 |

## 5. 緊急エスカレーションパス

### 5-1. 障害検知と通知フロー

本システムはクローズド LAN・個人運用のため、外部自動アラートシステムを採用しない。緊急時の通知は以下のフローで実施する。

| フェーズ | アクション | 担当 | 所要時間目標 |
|---|---|---|---|
| 1. 障害検知 | 端末異常・作業員からの口頭/内線報告 または Grafana ダッシュボード（SCR-MC-001）の確認 | 現場担当・system_admin | — |
| 2. IT 担当通知 | 社内電話・内線・対面のいずれかで system_admin へ連絡 | 現場担当 → system_admin | 15 分以内（NFR-AVL-005）|
| 3. 初動対応 | system_admin がシステムへの接触を開始（ステータス確認・原因特定） | system_admin | 通知から 15 分以内 |
| 4. quality_admin への通知 | P1 障害（生産停止影響）の場合、system_admin が quality_admin（品質担当）に状況を報告 | system_admin → quality_admin | 初動対応開始から 30 分以内 |
| 5. 復旧完了 | 障害復旧・作業再開確認 | system_admin | 障害発生から 1 時間以内（NFR-AVL-002 RTO） |

### 5-2. 障害分類と対応方針

| 障害分類 | 定義 | 対応方針 | 目標復旧時間 |
|---|---|---|---|
| P1（生産停止） | 全端末がオフライン縮退モードに移行し、作業記録の同期が停止している | 最優先で対応・quality_admin 通知必須 | RTO 1 時間以内 |
| P2（機能縮退） | 一部機能の利用不可（例：帳票生成不可・マスタ同期遅延） | 当日中に対応・OPS-PROC-010 または 09 で対処 | 当日営業時間内 |
| P3（性能劣化） | P95 応答時間 SLO 超過・DLQ 件数増加 | 翌営業日以内に原因特定・次週ヘルスチェックでフォロー | 翌営業日 |
| P4（監視アラート） | ディスク使用率 85% 超・バックアップ齢 25h 超 | 週次ヘルスチェック時に確認・当週内に対処 | 当週内 |

## 6. 運用インフラ構成の固定

本サブで採用する運用インフラを以下に確定する。変更には ADR の作成を必須とする。

| 構成要素 | 採用製品・バージョン | 配置場所 | 用途 |
|---|---|---|---|
| OS | Windows Server 2022 | 物理/VM | ホスト OS（IIS 含む） |
| コンテナランタイム | WSL2 + Docker Compose | Windows Server 上 | Rust API・PostgreSQL・Prometheus・Grafana |
| DB | PostgreSQL 16 | Docker コンテナ | プライマリデータストア |
| メトリクス収集 | Prometheus（port 9090） | Docker コンテナ | /metrics スクレイプ（15s 間隔） |
| ダッシュボード | Grafana（port 3000） | Docker コンテナ | SCR-MC-001 表示先 |
| ログ転送 | nxlog（Windows 版）| Windows Server | Docker ログ → Windows Event Log |
| バックアップ先 | NAS（LAN 内共有フォルダ）| 同一建屋内 NAS | WAL アーカイブ・pg_dump 格納 |
| オフサイト媒体 | 外付け HDD（AES-256 暗号化）| 別棟または建屋外 | 週次オフサイトバックアップ |

---

**本節で確定した方針**
- **IT 担当 1 名・週次 2 時間以内の「個人開発・個人運用」ポリシーを不変の設計前提として確定し、自動化タスク（BAT-001〜010）と system_admin 手動タスク（OPS-PROC-001〜010）の責任分担を明示した。**
- **緊急エスカレーションパスを「現場担当 → system_admin（15 分以内）→ quality_admin（30 分以内）」の 5 フェーズで確定し、外部自動アラートシステムを個人開発・クローズド LAN 制約から対象外と判断した。**
- **日次・週次・月次・四半期の定期タスク体系を確定し、運用インフラを Windows Server 2022 + WSL2 + Docker Compose + Prometheus + Grafana + nxlog の固定構成として確定した。**
