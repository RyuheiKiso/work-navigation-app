# 前提制約と本書が約束しないこと

本節では、運用方式設計（08_運用方式設計）において意図的に対象外とした事項、および本書の適用範囲の外縁を明示する。「約束しないこと」を明文化することで、将来のシステム拡張・他者によるメンテナンス時に「なぜその機能がないのか」の根拠を提供し、設計の意図を保護する。なお、本節に列挙した事項は「品質が低いため省いた」のではなく、「現在の運用制約・規模・コストに照らして合理的に対象外と判断した」ものである。システム要件が変化した際には本節を見直し、設計に反映することを推奨する。

---

| 識別子 | 対象外事項 | 分類 |
|--------|-----------|------|
| D-OPS-001 | 外部監視サービスとの連携 | ネットワーク制約 |
| D-OPS-002 | Kubernetes 等コンテナオーケストレーション | 規模・運用制約 |

| D-OPS-004 | 自動フェイルオーバー | 規模・コスト制約 |
| D-OPS-005 | 詳細チューニングパラメータの確定 | フェーズ制約 |

## 2. D-OPS-001: 外部監視サービス（DataDog/Mackerel 等）との連携

**対象外と判断する。**

### 根拠

本システムは閉域工場ネットワーク内に設置される。DataDog・Mackerel・New Relic・Prometheus Remote Write 等の SaaS 型監視サービスはすべて外部インターネット接続を必要とするが、工場内ネットワークはセキュリティポリシーにより外部 SaaS への送信が禁止されている。

### 代替手段

- 運用ダッシュボード（SCR-MC-001）による自己監視
- `alerts` テーブルへのアラート記録 + 30秒リフレッシュで実現するオンプレ通知

### 将来の再考条件

工場ネットワークポリシーが変更され、特定エンドポイントへの HTTPS アウトバウンドが許可された場合は、Prometheus（オンプレ）+ Grafana の導入を優先的に検討する（SaaS 依存なしの監視基盤）。

---

## 3. D-OPS-002: Kubernetes / EKS / GKE によるコンテナオーケストレーション

**対象外（個人運用・単一建屋）。**

### 根拠

本システムは以下の規模・運用特性を持つ。

- 単一建屋内の閉域展開
- 個人（system_admin 1名）による運用
- 同時接続ハンディ台数: 最大 30台（想定）
- 高可用性のためのマルチノード構成が費用・管理コスト面で不釣り合い

Kubernetes は複数ノードにまたがる水平スケールアウト・自動フェイルオーバー・ローリングアップデート管理を提供するが、本システムの規模ではその恩恵よりも学習コスト・管理複雑度の増大が上回る。

### 代替手段

Docker Compose による単一ホスト運用。必要十分な HA は「スタンバイサーバーへの手動切り替え（BCP 方式）」で担保する。

### 将来の再考条件

建屋数が複数に拡大し、ハンディ台数が 100台を超えた場合、または system_admin が複数人体制になった場合に Kubernetes 移行を検討する。

---

## 4. D-OPS-003: 24時間365日有人監視

**対象外。system_admin による日次確認とアラート通知で対応。**

### 根拠

個人運用のため、夜間・休日の有人監視は現実的でない。製造現場の稼働時間は原則として日中シフト（6:00〜22:00 等）であり、深夜の重大障害は翌朝の確認時に対処する運用で許容可能である。

### 代替手段

- BAT-001（深夜 02:00 バックアップ）による自動データ保護
- Emergency Mode（LEVEL-2）によるハンディ端末の自律継続動作
- `alerts` テーブル + SCR-MC-001 ダッシュボード（翌朝確認）

### 前提となる運用規約

system_admin は業務日の始業時（推奨: 始業 30分以内）に SCR-MC-001 ダッシュボードを確認し、未解決の P1 アラートを優先的に対処することを運用規約とする。

### 将来の再考条件

24時間製造ライン（夜間稼働）が追加された場合、またはトレサビ要件として「リアルタイム人手監視」が品質規格（ISO9001 等）で求められた場合は本方針を見直す。

---

## 5. D-OPS-004: 自動フェイルオーバー

**対象外（個人運用、手動対応 RTO で十分）。**

### 根拠

自動フェイルオーバー（Active-Passive の自動切り替え）は、Keepalived / Pacemaker / PostgreSQL Patroni 等の追加コンポーネントを必要とする。これらは設定の複雑性が高く、誤設定による「スプリットブレイン」（両ノードがプライマリと認識する状態）がデータ破壊を招くリスクがある。個人運用においてこのリスクを管理する体制がないため、対象外とする。

### 代替手段

- RTO=15分の手動切り替え手順（04_バックアップ・リストア方式 / 07_縮退運用・BCP方式設計）
- 6ヶ月ごとの BCP 訓練による手順習熟

### 手動で十分な根拠

本システムの工場稼働は日中シフト主体であり、深夜のサーバー停止がそのまま業務停止に直結するシナリオは限定的である。LEVEL-2 Emergency Mode によるハンディ端末の自律動作が 8時間のバッファを提供するため、RTO=15分（手動）でも業務への実害は軽微とみなす。

### 将来の再考条件

24時間連続製造ラインへの拡張、または RPO=0 のゼロデータロスト要件が加わった場合は PostgreSQL の同期レプリケーション + 自動フェイルオーバー（Patroni 等）を検討する。

## 6. D-OPS-005: 詳細なチューニングパラメータの確定

**詳細設計フェーズで確定。**

### 根拠

以下のパラメータは実際の負荷測定・プロファイリングなしに適切な値を決定できない。概要設計では目標値（SLO）のみを定義し、具体的な設定値は詳細設計フェーズで確定する。

| パラメータ | 概要設計での扱い | 詳細設計での決定内容 |
|-----------|---------------|------------------|
| PostgreSQL `shared_buffers` | 未確定 | 実測に基づく最適値 |
| PostgreSQL `max_connections` | 警告閾値 80 を目安 | 接続プールサイズと連動して決定 |
| Rust Tokio ワーカースレッド数 | 未確定 | CPU コア数と実負荷に基づく設定 |
| Docker メモリ制限 | 未確定 | コンテナごとの実消費量計測後に設定 |
| `archive_timeout`（WAL） | 300秒（RPO 補助） | 実運用後の WAL 生成量に応じて調整 |
| バックアップ保管期間（ローカル/スタンバイ） | ローカル 30日・スタンバイ 90日 | ディスク容量の実測値で調整 |

### 確定のトリガー

ver1.0.0 リリース前の負荷テスト（テスト方式設計参照）の結果を受けて、本パラメータ群を詳細設計ドキュメントに記載する。

## 7. 本書が約束しないことの整理

以下は本運用方式設計ドキュメントがカバーしない領域である。

| 約束しないこと | 理由 |
|--------------|------|
| 運用コスト（月額・年額）の見積もり | 個人開発のため変動要素が大きく、見積もり根拠が薄い |
| SLA の顧客向け保証 | 外部顧客との契約が発生しないため不要 |
| 監査ログの第三者認証 | 現時点では内部利用のみ |
| ディザスタリカバリ（DR）サイトの設置 | 個人運用・単一建屋の規模を超える |
| マルチテナント対応 | 単一工場・単一組織向けの設計 |

---

**本節で確定した方針**

- D-OPS-001（外部監視 SaaS）は閉域ネットワーク制約により対象外とし、オンプレ運用ダッシュボード（SCR-MC-001）と `alerts` テーブルによる自己監視で代替する。
- D-OPS-002（Kubernetes）は個人運用・単一建屋の規模に対してオーバーエンジニアリングであるため対象外とし、Docker Compose + 手動 BCP 切り替えで代替する。
- D-OPS-004（自動フェイルオーバー）はスプリットブレインリスクと管理複雑度を考慮して対象外とし、RTO=15分の手動切り替え手順（07_縮退運用・BCP方式設計）と6ヶ月 BCP 訓練で代替する。
