# 00 設計 ID ルールと採番規約

本章は `04_概要設計/03_概要設計の識別子体系と採番規約.md` で確定した設計識別子 16 種の詳細な採番規約を完全形で格納する権威定義文書である。設計識別子に関して矛盾が生じた場合は本章を最終権威とする。

---

## 1. 基本採番規則（全識別子共通）

要件定義書 `03_要件定義/付録/00_要件IDルールと採番規約.md` §2 を完全継承する。

| 規則 | 内容 |
|---|---|
| 形式 | `[プレフィックス]-[連番 3 桁]` または `[プレフィックス]-[サブカテゴリ]-[連番 3 桁]` |
| 連番開始 | 各プレフィックス（またはサブカテゴリ）で `001` から開始 |
| ゼロ埋め | 3 桁固定（001〜999） |
| 欠番禁止 | スキップ不可 |
| 再使用禁止 | 廃止した番号は永久に欠番扱い |
| 廃止メタ | `[廃止: YYYY-MM-DD 廃止理由: ...]` を行の先頭に付与し、行を削除しない |
| 上限 | 1 プレフィックス（サブカテゴリ）あたり 999 件。超過時は規約改訂 |
| 命名言語 | 物理名は英小文字スネークケース |
| 管理台帳 | `付録/99_設計識別子採番台帳.md` で一元管理 |

## 3. 各識別子の必須メタデータ定義

### 3-1. TBL（物理テーブル）の必須メタデータ列

```
TBL-ID | 物理テーブル名 | 対応 EN-NNN | 対応関係 | 種別 | 推定行/年 | 保存期間 | 担当サブ | 状態
```

- **対応関係**: `1:1`（EN と完全対応）/ `1:N`（EN の派生拡張）/ `N:M`（中間テーブル、複数 EN-NNN を列記）
- **種別**: `Append-only`（INSERT 専用）/ `更新可`（CRUD 全許可）/ `マスタ`（ほぼ不変の参照データ）/ `制御`（システム制御用）
- **状態**: `Draft` → `In Review` → `Agreed` → `Frozen`

### 3-2. IDX（インデックス）の必須メタデータ列

```
IDX-ID | 物理名 | 対応 TBL-NNN | 種別 | 対象列 | Partial 条件 | 根拠 NFR-NNN | 状態
```

### 3-3. VW（ビュー）の必須メタデータ列

```
VW-ID | 物理名 | 種別 | 構成 TBL-NNN 群 | 消費側（API-NNN or RP-NNN） | 更新頻度 | 状態
```

### 3-4. API の必須メタデータ列

```
API-ID | HTTP メソッド | URL パス | operationId | 関連 FR-NNN | 関連 BR-BUS-NNN | 関連 TBL-NNN | エラー（ERR-NNN） | 認証要否 | ロール制限 | 状態
```

### 3-5. MOD の必須メタデータ列

```
MOD-ID | 物理名 | ファイルパス（src/ 相対） | 責務（1行） | 依存 MOD-NNN 群 | 関連 FR-NNN | 関連 TBL-NNN（BE のみ）| 関連 SCR-NNN（FE のみ）| 状態
```

### 3-6. CMP の必須メタデータ列

```
CMP-ID | 物理名 | ファイルパス（src/ 相対） | 配置 SCR-NNN | 所属 MOD-FE-*-NNN | 状態
```

### 3-7. TRN の必須メタデータ列

```
TRN-ID | 発生元 SCR-NNN | 遷移先 SCR-NNN | トリガーアクション | ガード条件 | 副作用 API-NNN | 関連 UC-NNN | 状態
```

### 3-8. BAT の必須メタデータ列

```
BAT-ID | 名称 | 実行契機 | スケジュール（cron 式） | 冪等性方式 | 失敗時動作 | 関連 FR/NFR-NNN | 状態
```

### 3-9. MSG の必須メタデータ列

```
MSG-ID | トピック名 | 配信先 | 配信保証 | スキーマバージョン | 関連 IF-NNN | 状態
```

### 3-10. ERR の必須メタデータ列

```
ERR-ID | エラー名 | HTTP ステータス | RFC 9457 type URI | 関連 BR-BUS-NNN | 関連 API-NNN（発生エンドポイント） | 状態
```

### 3-11. LOG の必須メタデータ列

```
LOG-ID | イベント名 | ログレベル | PII マスク要否 | 保管期間 | 関連 NFR-NNN | 状態
```

### 3-12. KEY の必須メタデータ列

```
KEY-ID | 用途 | アルゴリズム | 鍵長 | ローテーション周期 | 保管先 | 関連 NFR-SEC-NNN | 状態
```

（KEY の値はリポジトリ外管理。本台帳は識別子・用途・保管場所のみ記録する）

### 3-13. CFG の必須メタデータ列

```
CFG-ID | キー名（ドット区切り） | デフォルト値 | 型 | 許容範囲 | 変更権限 | 必須/任意 | 関連 FR/NFR-NNN | 状態
```

### 3-14. SEQ の必須メタデータ列

```
SEQ-ID | 名称 | 関連 UC-NNN | 関与 API-NNN 群 | 関与 TBL-NNN 群 | 関与 MOD-NNN 群 | 図ファイル名 | 状態
```

### 3-15. MET の必須メタデータ列

```
MET-ID | 名称 | カテゴリ | Prometheus メトリクス名 | SLO 値 | SLO 算定窓 | 関連 API-NNN | 関連 NFR-NNN | 状態
```

### 3-16. PRM の必須メタデータ列

```
PRM-ID | ロール | 操作対象（SCR/API/TBL 列 単位） | 許可種別 | 判定 | 根拠 BR-BUS-NNN | 状態
```

## 4. 廃止手順（全識別子共通）

1. `付録/99_設計識別子採番台帳.md` の対象行先頭に `[廃止: YYYY-MM-DD 廃止理由: ...]` を付与
2. DTM・RTM 拡張版・各サブ章でその識別子を参照しているセルに `[廃止]` を付記
3. 行・セルを削除しない（採番連続性の検証不能になるため）
4. `付録/06_変更管理と版数規約.md` の変更履歴に廃止理由を記録

## 5. 識別子管理台帳のファイル分担

| 管理対象 | 台帳ファイル |
|---|---|
| 要件識別子（BR/FR/NFR/UC/EN/IF/RP/SCR 等） | `03_要件定義/付録/99_採番台帳.md`（変更禁止） |
| 設計識別子（TBL〜PRM 等 16 種） | `04_概要設計/付録/99_設計識別子採番台帳.md` |
| 全識別子プレフィックス索引 | `01_管理/採番台帳/INDEX.md`（リンク表のみ・初期版） |

---

**本節で確定した方針**
- **設計識別子 16 種（TBL〜PRM）と補助識別子 6 種（SUB/NODE/ENV-ITEM/RPT-LAYOUT/TLVL/TENV）の採番規約・必須メタデータを確定し、管理台帳を `付録/99_設計識別子採番台帳.md` に一元化する。**
- **廃止識別子は行を削除せず廃止メタデータを付与して保持し、採番連続性の後方検証を保証する。**
- **要件識別子台帳（03_要件定義/付録/99）との責務分離を維持し、概要設計フェーズで要件側識別子を再採番することを禁止する。**
