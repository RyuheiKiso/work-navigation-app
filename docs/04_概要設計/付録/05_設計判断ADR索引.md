# 05 設計判断 ADR 索引

本章の責務は、概要設計フェーズで起票する ADR（アーキテクチャ決定記録）の索引を管理することである。ADR 本体は `docs/01_管理/ADR/ADR-NNN.md` に格納する。

---

## 1. ADR 起票基準

以下の 3 条件を全て満たす場合のみ ADR を起票する。

1. **技術的不可逆性**: 後から変更するコストが非常に高い決定
2. **有力な代替案の存在**: 複数の実装可能な代替案が存在した
3. **選択理由の非自明性**: なぜその選択をしたかが将来の読者に明らかでない

単純な実装選択（「Rust で書く」「PostgreSQL を使う」等、計画書で既確定の事項）には ADR を起票しない。

---

## 2. ADR テンプレート（`docs/01_管理/ADR/ADR-NNN.md` 形式）

```markdown
# ADR-{NNN}: {技術選定のタイトル}

**状態**: 提案中 / 採択 / 却下 / 廃止（廃止日: YYYY-MM-DD）
**日付**: YYYY-MM-DD
**担当サブ**: 04_概要設計/{サブ名}

## 背景

なぜこの決定が必要になったか。

## 決定事項

何を選択したか。

## 根拠（代替案との比較表を含む）

| 代替案 | メリット | デメリット | 採否 |
|---|---|---|---|
| 選択案 | ... | ... | 採択 |
| 代替案 A | ... | ... | 却下 |
| 代替案 B | ... | ... | 却下 |

なぜ選択案を選んだか。

## 結果

この決定がもたらす影響（正負両面）。

## 更新履歴

| 日付 | 変更 | 変更者 |
|---|---|---|
| YYYY-MM-DD | 初版 | RyuheiKiso |
```

## 3. 概要設計フェーズの ADR 一覧

| ADR-ID | タイトル | 状態 | 担当サブ | 本体ファイル |
|---|---|---|---|---|
| ADR-001 | 設計識別子採番体系の確立 | 採択 | 04_概要設計/付録 | `docs/01_管理/ADR/ADR-001.md`（要作成） |
| ADR-002 | RTM ハイブリッド拡張（本体 11 列 + DTM 6 マトリクス） | 採択 | 04_概要設計/付録 | `docs/01_管理/ADR/ADR-002.md`（要作成） |
| ADR-003 | Outbox Pattern（外部 MQ 非採用） | 採択 | 02_ソフトウェア方式設計 | `docs/01_管理/ADR/ADR-003.md`（要作成） |
| ADR-004 | Append-only EventStore（DB ロール分離による物理保証） | 採択 | 04_データ設計 | `docs/01_管理/ADR/ADR-004.md`（要作成） |
| ADR-005 | JSON Logic（eval 非採用）による DSL 評価 | 採択 | 02_ソフトウェア方式設計 | `docs/01_管理/ADR/ADR-005.md`（要作成） |

（概要設計の各サブ章完成時に ADR を逐次追加する）

---

**本節で確定した方針**
- **ADR は「技術的不可逆性・代替案の存在・選択理由の非自明性」の 3 条件を全て満たす場合のみ起票し、単純な実装選択には起票しない。**
- **ADR 本体は `docs/01_管理/ADR/ADR-NNN.md` に格納し、本章はその索引として機能する。**
- **本章の ADR 一覧は概要設計フェーズを通じて逐次更新する。**
