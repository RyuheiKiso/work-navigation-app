# 06 変更管理と版数規約

本章の責務は、概要設計書（04_概要設計）の版数管理・変更種別・変更プロセス・Frozen ライフサイクルを確定することである。要件定義書 `03_要件定義/付録/04_変更管理と版数規約.md` の規約を継承し、概要設計フェーズ固有の事項を追加定義する。

---

## 1. 版数管理

### 1-1. セマンティックバージョニング

概要設計書はセマンティックバージョニング `メジャー.マイナー.パッチ` で管理する。

| 変更種別 | バージョン操作 | 例 |
|---|---|---|
| 設計識別子体系の変更（プレフィックス追加・削除・形式変更） | メジャー上げ | 0.x.y → 1.0.0 |
| 新 TBL/API/MOD/SEQ 等の採番・章構成の変更 | マイナー上げ | 0.1.y → 0.2.0 |
| 誤字修正・表現明確化・図の軽微な修正 | パッチ上げ | 0.1.0 → 0.1.1 |

初期版数: `0.1.0`（作成中 Draft 状態）

### 1-2. ライフサイクル

| 状態 | 定義 | 次状態への移行条件 |
|---|---|---|
| Draft | 初稿作成中。設計命題が確定していない章が存在する | 全章の「本節で確定した方針」節が記入済み |
| In Review | 章間整合性・DTM 完全性のセルフレビュー中 | V-01〜V-10 の完全性検証全件 OK |
| Agreed | 全設計命題が確定。詳細設計着手可能 | Git タグ `docs-des-agreed-v{版数}` を付与 |
| Frozen | 実装フェーズ着手後。変更には PRJ-037 設計変更プロセスが必要 | Git タグ `docs-des-frozen-v{版数}` を付与 |

### 1-3. Frozen 移行のタイミング

詳細設計（05_詳細設計）着手と同時に概要設計書を Frozen に移行する。Frozen 後の変更は全て設計変更プロセス（§3）を経由する。

---

## 2. 変更種別

### 2-1. 設計変更（PRJ-037）

設計命題（API エンドポイント・テーブル構造・コンポーネント分割等）の変更。以下の 3 条件を全て満たす場合のみ ADR を起票する。

- 技術的に不可逆な決定（例: テーブルの物理スキーマ変更）
- 有力な代替案が存在した
- 選択理由が自明でない

### 2-2. 識別子採番変更

新識別子の追加・廃止。採番台帳（付録/99）の更新を必須とする。

### 2-3. 軽微修正

誤字・表現明確化・図の視覚的改善（設計命題に影響しない範囲）。ADR 不要・版数パッチ上げ。

---

## 3. 設計変更プロセス（PRJ-037）

1. **変更起票**: `docs/01_管理/change_log.md` に 設計 ID・変更内容・理由を記録
2. **影響分析**: DTM 6 マトリクスで影響する識別子を列挙
3. **ADR 起票判断**: §2-1 の 3 条件を確認。必要なら `docs/01_管理/ADR/ADR-NNN.md` を作成
4. **ドキュメント先行更新**: ドキュメントを更新してから実装を変更（ドキュメント先行原則）
5. **採番台帳更新**: 付録/99 で廃止・新規 ID を更新
6. **Git コミット**: メッセージに `change: 設計変更 PRJ-037 ID={設計ID}` を含める

---

## 4. 変更履歴記録形式

各 md 章末（または章内）に以下の変更履歴節を設ける。

| フィールド | 内容 |
|---|---|
| 版数 | X.Y.Z |
| 変更日 | YYYY-MM-DD |
| 変更内容 | 「TBL-015 のスキーマを変更（保存期間を 5 年→7 年）」等 |
| 変更者 | RyuheiKiso |
| レビュー者 | セルフレビュー |

---

## 5. Git タグ規約

| タグ名 | 付与タイミング |
|---|---|
| `docs-des-v{版数}` | 概要設計の版数区切り |
| `docs-des-agreed-v{版数}` | Agreed 移行時 |
| `docs-des-frozen-v{版数}` | Frozen 移行時 |

過去版の参照は Git タグを使用する。Main への `git rebase -i` / `force push` は禁止。

---

**本節で確定した方針**
- **概要設計書は SemVer で管理し、Draft→In Review→Agreed→Frozen の 4 状態ライフサイクルを経て Frozen 後は設計変更プロセス（PRJ-037）なしの変更を禁止する。**
- **設計変更は「ドキュメント先行原則」を適用し、ADR 起票 3 条件（不可逆・代替案あり・理由非自明）を満たす場合のみ ADR を起票する。**
- **変更履歴は全 md 章末の変更履歴節と `docs/01_管理/change_log.md` に二重管理する。**
