# 10 帳票保管・廃棄・出力監査

本章の責務は、6 件の帳票（RP-001〜006）の保管方式・廃棄方針・出力監査ログを確定することである。NFR-ALC-006（完全性）・NFR-ALC-008（永続性）・NFR-ALC-009（入手可能性）・NFR-SEC-009（保管暗号化）を本章で担保する。

---

## 1. 帳票保管方式

### 1-1. 保管場所と構成

帳票ファイルは以下の 2 段階で保管する。

| 段階 | 保管場所 | 形式 | 暗号化 | 保管期間 |
|---|---|---|---|---|
| プライマリ | Windows Server 2022 ローカルファイルシステム | PDF/A-3 + XLSX/XES | AES-256-GCM（ボリューム暗号化: BitLocker）| 7 年以上 |
| バックアップ | pg_dump + WAL アーカイブ（`08_運用方式設計` 参照）| 同上 | AES-256-GCM（バックアップ暗号化）| 7 年以上 |

### 1-2. ファイルシステム階層構造

```
/data/reports/
├── {year}/
│   ├── {month}/
│   │   ├── RP-001/
│   │   │   └── {work_execution_id}.pdf
│   │   ├── RP-002/
│   │   │   └── {export_id}.zip       # XES + CSV
│   │   ├── RP-003/
│   │   │   └── {nonconformity_id}.pdf
│   │   ├── RP-004/
│   │   │   └── {kaizen_id}.pdf
│   │   ├── RP-005/
│   │   │   └── {inspection_id}.pdf
│   │   └── RP-006/
│   │       └── {period}_{type}.pdf   # 例: 2026-01_monthly.pdf
```

各 `pdf` ファイルに対応する `.sha256` ファイルを同階層に併置する。

### 1-3. report_metadata テーブル（TBL-033）

帳票ファイルの索引として `report_metadata` テーブルを使用する（TBL-001 の WorkEvent とは独立したアクセスパス）。

| カラム名 | 型 | 説明 |
|---|---|---|
| `report_id` | UUID v7 PK | 帳票固有 ID |
| `report_type` | TEXT | `RP-001` 〜 `RP-006` |
| `ref_id` | UUID | 参照対象 ID（work_execution_id 等）|
| `document_hash` | TEXT | SHA-256（hex 64 文字）|
| `file_path` | TEXT | プライマリ保管パス（絶対パス）|
| `template_version` | TEXT | `TPL-NNN:M.N.P` 形式 |
| `generated_at` | TIMESTAMPTZ | 生成日時（UTC）|
| `generated_by_role` | TEXT | 生成ジョブのサービスアカウント名 |
| `status` | TEXT | `ACTIVE` / `CORRECTED` / `ARCHIVED` |
| `corrected_by` | UUID NULLABLE | 訂正版帳票の `report_id`（訂正時のみ）|

## 2. 保管期間と廃棄方針

### 2-1. 保管期間

| 帳票種別 | 法令根拠 | 最低保管期間 | 本システムの方針 |
|---|---|---|---|
| RP-001: SOP 実行記録 | ISO 13485 §4.2.5（記録管理）| 3 年 / 製品寿命 + 1 年 | 7 年以上 |
| RP-002: 監査ログ（XES/CSV）| GxP ガイダンス | 5 年 | 7 年以上 |
| RP-003: 不適合・CAPA 記録 | ISO 9001 §10.2.2 | 3 年 | 7 年以上 |
| RP-004: Kaizen 記録 | 社内規程 | 3 年 | 7 年以上 |
| RP-005: 設備点検記録 | 労働安全衛生法 第 45 条 | 3 年 | 7 年以上 |
| RP-006: 集計レポート | 社内規程 | 1 年 | 3 年 |

7 年統一方針の根拠: ALCOA+ の「永続性（Enduring）」原則を満たすため、最も厳しい法令要件（GxP 5 年）を超える 7 年を全帳票の基準値とする。RP-006 のみ集計データとして 3 年に短縮する。

### 2-2. 廃棄方式

廃棄はシステムからの自動削除でなく、以下の手続きを踏む。

| ステップ | 実施者 | 方法 |
|---|---|---|
| 廃棄対象の抽出 | BAT（年次実行）| `report_metadata.generated_at` から保管期間を計算 |
| 廃棄承認 | quality_admin + system_admin の二段階承認 | SCR-MC-004 の監査ログから確認可能 |
| ファイル削除 | system_admin | `shred -u -z` コマンド（セキュア削除）|
| 廃棄記録 | BAT | `report_disposal_log` テーブルに INSERT（廃棄日・承認者・対象 report_id・document_hash）|

廃棄後も `report_disposal_log` と `report_metadata`（`status = 'DISPOSED'` に更新）は永続保管する。

## 3. 出力監査ログ

### 3-1. 帳票出力イベントの記録

帳票の生成・ダウンロード・印刷の全操作を `audit_logs` テーブル（TBL-032）に記録する（LOG-010）。

| イベント種別 | トリガー | 記録内容 |
|---|---|---|
| `REPORT_GENERATED` | BAT-006〜010 完了時 | report_id, report_type, ref_id, generated_at, template_version |
| `REPORT_DOWNLOADED` | SCR-MC-009 のダウンロードボタン押下 | report_id, downloader_id（ハッシュ）, ip_address, downloaded_at |
| `REPORT_PRINTED` | 印刷 API 呼び出し時（IF-004）| report_id, printer_id, printed_at |
| `REPORT_CORRECTED` | 訂正版生成時 | old_report_id, new_report_id, corrector_id（ハッシュ）, reason |
| `REPORT_DISPOSED` | 廃棄実行時 | report_id, document_hash, disposed_at, approver_ids |

### 3-2. 出力監査の閲覧

出力監査ログは SCR-MC-004（監査ログ閲覧）で `event_type = 'REPORT_*'` フィルタを使用して閲覧できる。XES エクスポート（RP-002）には帳票出力イベントも含める。

## 4. 3 つのエクスポート手段（NFR-ALC-009 入手可能性）

データ所有者が帳票データをいつでも取り出せることを保証するため、以下の 3 手段を提供する。

| 手段 | 操作方法 | 取得形式 |
|---|---|---|
| 1. Web ダウンロード | SCR-MC-009 で期間・種別を選択 | PDF/A-3 個別 / ZIP 一括 |
| 2. ファイルシステム直接アクセス | Windows Server の共有フォルダ（CIFS）| PDF/A-3 生ファイル |
| 3. API エクスポート | `GET /api/reports/{report_id}/download`（Bearer Token 認証）| PDF/A-3 バイナリ |

3 手段のいずれかが利用不能になった場合も、残る 2 手段でデータ取得が可能な設計とする。

---

**本節で確定した方針**
- **帳票を 7 年以上（RP-006 は 3 年）プライマリ + バックアップの 2 段階保管し、AES-256-GCM 暗号化・`report_metadata` テーブル索引・セキュア廃棄手続きを確定した。**
- **帳票の生成・ダウンロード・印刷・訂正・廃棄の 5 イベントを `audit_logs` テーブルに記録し、SCR-MC-004 で閲覧可能とする出力監査方式を確定した。**
- **Web ダウンロード・ファイルシステム直接アクセス・API エクスポートの 3 手段を保証し、ALCOA+ の「入手可能性（Available）」原則を技術的に担保した。**
