# 02 帳票生成方式とパイプライン

本章の責務は、RP-001〜006 の全帳票に共通する生成パイプライン（データ抽出→テンプレートレンダリング→署名付帯→ハッシュ登録→保管・通知）の方式を確定し、各ステップの技術実装・エラーハンドリング・バッチジョブ定義を概要設計レベルで固定することである。個別帳票固有のレイアウト・データ項目は各詳細設計章（03〜08）で定義する。

---

## 1. パイプライン概要（5 ステップ）

図: fig_des_report_pipeline（img/ 配下）を参照。

全帳票は以下の 5 ステップからなる共通パイプラインを経て生成される。各ステップは Rust の非同期タスク（tokio::task）として実装し、ステップ間のデータは構造体（ReportContext）で受け渡す。

| ステップ | 処理名 | 実装モジュール（案）| 説明 |
|---|---|---|---|
| Step 1 | データ抽出 | `report::extractor` | PostgreSQL から対象テーブルをクエリし、ReportContext を構成する |
| Step 2 | テンプレートレンダリング | `report::renderer` | Jinja2（テキスト系）/ openpyxl（XLSX）/ WeasyPrint（PDF）で帳票ファイルを生成する |
| Step 3 | 電子サイン付帯 | `report::signer` | PKCS#7 形式で PDF にデジタル署名を付与し、sign_id を紐付ける |
| Step 4 | SHA-256 ハッシュ計算・登録 | `report::hasher` | 帳票ファイル全体の SHA-256 を算出し、hash_chain_blocks テーブルへ登録する |
| Step 5 | 保管・通知 | `report::storage` + `report::notifier` | ローカルファイルシステムへ書き込み、バックアップを取得後、quality_admin へ webhook 通知する |

### Step 1: データ抽出

- 対象テーブル: `work_events`・`nonconformities`・`capas`・`kaizens`・`equipment_inspections`・`electronic_signs`・`hash_chain_blocks` 等。
- クエリはトリガーとなったエンティティ ID（`work_instance_id`・`nc_id`・`kaizen_id` 等）を起点とし、関連レコードを JOIN で取得する。
- クエリ結果はスナップショットとして ReportContext 構造体に格納する。生成後のデータ変更は帳票内容に影響しない（不変性担保）。
- タイムゾーンは全フィールドを JST（Asia/Tokyo）に変換して格納する。

### Step 2: テンプレートレンダリング

| 出力形式 | 使用ライブラリ | 実装言語 | 備考 |
|---|---|---|---|
| PDF/A-3 | WeasyPrint（Python サブプロセス経由）| Python 3.11 以上 | Rust から `std::process::Command` で呼び出す |
| XLSX | openpyxl（Python サブプロセス経由）| Python 3.11 以上 | テンプレートファイル（.xlsx）を開き、データを書き込む |
| XES XML | Jinja2（Python サブプロセス経由）| Python 3.11 以上 | IEEE 1849 準拠の XML を生成する |
| CSV | Jinja2 または csv モジュール | Python 3.11 以上 | UTF-8 BOM 付きで出力（Excel 互換）|

テンプレートファイルは TPL-001〜006（`2. テンプレート管理` 参照）を使用する。レンダリング前にテンプレートの SHA-256 と格納済みマスター版のハッシュを照合し、改ざんがないことを確認する。

### Step 3: 電子サイン付帯

- PDF/A-3 ファイルに対して PKCS#7（CMS SignedData）形式のデジタル署名を付与する。
- 署名に使用する証明書はシステム内部 CA が発行した証明書（X.509 v3）を使用する。
- 署名完了後、`electronic_signs` テーブルの `sign_id` を ReportContext に記録し、生成された帳票メタデータテーブル（`report_records`）へ外部キーとして保存する。
- XLSX・CSV・XES XML 形式の帳票に対しては PDF デジタル署名の代わりに、Step 4 の SHA-256 ハッシュを `report_records` テーブルで sign_id と紐付けることで帰属可能性を担保する。

### Step 4: SHA-256 ハッシュ計算とハッシュチェーン登録

- Step 3 完了後の帳票ファイルバイト列全体に対して SHA-256 を計算する（hex エンコード 64 文字）。
- `hash_chain_blocks` テーブルへ以下を INSERT する:
  - `block_id`: UUID v7（新規）
  - `content_type`: `"report"`
  - `content_id`: 帳票レコード ID
  - `content_hash`: 算出した SHA-256
  - `prev_hash`: 直前ブロックの `content_hash`
  - `created_at`: サーバー時刻（TIMESTAMPTZ）
- PDF/A-3 形式の場合は XMP メタデータ（`dc:description` 拡張フィールド）にも同ハッシュを埋め込む。

### Step 5: 保管・通知

**保管（ローカル + バックアップ）:**
- 生成ファイルは `{storage_root}/{rp_id}/{YYYY}/{MM}/{report_id}.{ext}` のパスで保存する。
- `storage_root` は設定ファイル（CFG キー: `report.storage_root`）で管理し、デプロイ環境ごとに切り替え可能とする。
- 保存完了後、バックアップ先（第 2 ストレージ）へ同期コピーを行い、`backup_logs` テーブルに記録する。

**通知（webhook）:**
- 保管完了後、quality_admin ロールの通知エンドポイントへ POST リクエストを送信する。
- ペイロードには `report_id`・`rp_id`・`generated_at`・`file_path`・`sha256` を含む。
- webhook URL は設定ファイル（CFG キー: `report.notification.webhook_url`）で管理する。
- 通知失敗は ERR-SYS-003 として記録するが、帳票生成自体は成功扱いとする（通知は非必須処理）。

## 2. テンプレート管理

帳票テンプレート（TPL-001〜006）は以下の方針で管理する。

| TPL-ID | 対応 RP | テンプレートファイルパス（リポジトリ相対）| マスター版ハッシュ検証 |
|---|---|---|---|
| TPL-001 | RP-001 | `src/report/templates/tpl_sop_execution_record.jinja2` 他 | 要（起動時・生成時）|
| TPL-002 | RP-002 | `src/report/templates/tpl_audit_log_xes.jinja2` 他 | 要（起動時・生成時）|
| TPL-003 | RP-003 | `src/report/templates/tpl_nonconformity_capa.jinja2` | 要（起動時・生成時）|
| TPL-004 | RP-004 | `src/report/templates/tpl_kaizen_record.jinja2` 他 | 要（起動時・生成時）|
| TPL-005 | RP-005 | `src/report/templates/tpl_equipment_inspection.jinja2` | 要（起動時・生成時）|
| TPL-006 | RP-006 | `src/report/templates/tpl_aggregate_report.jinja2` 他 | 要（起動時・生成時）|

次採番値: **TPL-007**

**版管理方針:**
- テンプレートファイルは git リポジトリで版管理する。
- 各テンプレートのマスター SHA-256 ハッシュはデータベースの `config_snapshots` テーブルに保持し、サービス起動時および帳票生成直前に照合する。
- テンプレートの改ざん検知時は ERR-SYS-003 を発生させ、帳票生成を中断する。
- テンプレートの変更（バグ修正・レイアウト変更）は `config_snapshots` への新規バージョン登録と、git タグ付与を必須とする。

## 3. バッチジョブ一覧

帳票生成に関連するバッチジョブを以下に確定する。詳細実装は `08_運用方式設計/` で定義する。

| BAT-ID | 対応 RP | トリガー種別 | トリガー条件 | 実行内容 | CFG 参照 |
|---|---|---|---|---|---|
| BAT-006 | RP-001 | イベント駆動 | 作業完了（`work_events.activity = 'step_completed'` かつ全ステップ完了） | SOP 実行記録の自動生成（PDF/A-3 + XLSX）| — |
| BAT-007 | RP-003 | イベント駆動 | 不適合登録（`nonconformities` テーブルへの INSERT）| 不適合・CAPA 記録の自動生成（PDF/A-3）| — |
| BAT-008 | RP-004 | イベント駆動 | Kaizen 承認完了（`kaizens.status = 'approved'` への UPDATE）| Kaizen 記録の自動生成（PDF/A-3 + CSV）| — |
| BAT-009 | RP-005 | イベント駆動 | 設備点検完了（`equipment_inspections.status = 'completed'` への UPDATE）| 設備点検記録の自動生成（PDF/A-3）| — |
| BAT-010 | RP-006 | スケジュール | 日次・週次・月次（設定ファイルで選択可能）| 集計レポートの定期生成（PDF + XLSX）。個人別集計は含まない | `report.batch.schedule`（CFG）|

| CFG キー | 型 | デフォルト値 | 説明 |
|---|---|---|---|
| `report.batch.schedule.daily.enabled` | bool | `true` | 日次集計の有効化 |
| `report.batch.schedule.daily.cron` | string | `"0 6 * * *"` | 日次実行時刻（cron 式・JST）|
| `report.batch.schedule.weekly.enabled` | bool | `true` | 週次集計の有効化 |
| `report.batch.schedule.weekly.cron` | string | `"0 7 * * 1"` | 週次実行時刻（月曜 7:00 JST）|
| `report.batch.schedule.monthly.enabled` | bool | `true` | 月次集計の有効化 |
| `report.batch.schedule.monthly.cron` | string | `"0 8 1 * *"` | 月次実行時刻（1 日 8:00 JST）|

## 4. エラーハンドリング

### 4-1. 生成失敗時の処理フロー

パイプラインのいずれかのステップで致命的エラーが発生した場合、以下の順序で処理する。

1. エラーコード `ERR-SYS-003` を発生させ、`system_logs` テーブルへ記録する。
2. 生成途中の一時ファイルを削除し、不完全な帳票がストレージへ保存されないことを保証する。
3. `report_records` テーブルの当該レコードのステータスを `"failed"` に更新する。
4. リトライキューへエンキューする（後述）。

### 4-2. リトライポリシー

| 項目 | 値 | 備考 |
|---|---|---|
| 最大リトライ回数 | 4 回 | 初回失敗 + 最大 4 回 = 計 5 回試行 |
| リトライ間隔 | 指数バックオフ（初回 30 秒・2 倍ずつ・上限 8 分）| — |
| 対象エラー | ネットワークエラー・DB 一時障害・ファイル I/O エラー | 設定不正・テンプレート改ざん等の論理エラーはリトライしない |
| リトライ非対象 | ERR-SYS-003（テンプレート改ざん検知）・電子サイン未取得 | 即時 DLQ 送り |

### 4-3. デッドレターキュー（DLQ）

- 5 回連続して失敗した場合（初回 + 4 回リトライ後も失敗）、当該ジョブを DLQ（Dead Letter Queue）へ移動する。
- DLQ は PostgreSQL の `report_dlq` テーブルで実装する（Outbox パターンと同一方式）。
- DLQ への移動時は `system_admin` ロールへ webhook 通知を送信する（quality_admin 通知とは別途）。
- DLQ 内のジョブは system_admin による手動確認・再実行・破棄のいずれかを選択できる（画面設計: `03_画面設計/` 参照）。

### 4-4. 部分成功の取り扱い

- Step 5 の通知（webhook）失敗は帳票生成の成否に影響しない（非必須処理）。
- バックアップ失敗はログへ記録し、system_admin へ通知するが、帳票生成は成功扱いとする。ただし次回バックアップジョブで再試行する。
- ファイル保管（Step 5 一次保管）の失敗は致命的エラーとして扱い、ERR-SYS-003 を発生させる。

---

**本節で確定した方針**

- **全帳票共通の 5 ステップパイプライン（データ抽出→テンプレートレンダリング→電子サイン付帯→SHA-256 ハッシュチェーン登録→保管・通知）を確定した。各ステップは Rust 非同期タスクとして実装し、Python サブプロセス（WeasyPrint / openpyxl / Jinja2）を呼び出す方式とする。**
- **BAT-006〜010 の 5 バッチジョブを確定した。BAT-006〜009 はイベント駆動、BAT-010 はスケジュール駆動（日次・週次・月次を CFG で設定可能）とする。**
- **エラーハンドリングは ERR-SYS-003 発生→最大 4 回リトライ（指数バックオフ）→5 回失敗で DLQ の 3 段階とし、テンプレート改ざん・論理エラーはリトライせず即時 DLQ 送りとする。**
