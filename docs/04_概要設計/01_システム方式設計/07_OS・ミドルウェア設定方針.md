# 07 OS・ミドルウェア設定方針

本章は、Windows Server 2022 + WSL2 + Docker Composeのオンプレミス構成において、OSおよび各ミドルウェアが本番稼働に必要な設定方針を確定する。設定方針は「What（何を設定するか）」と「Why（なぜその値か）」を確定するにとどまり、パラメータの詳細チューニング（例：PostgreSQLのcheckpoint_completion_target・random_page_cost等の個別値）は詳細設計・構築フェーズに委任する。本章で確定した設定はCFG識別子として採番台帳に登録し、セキュリティ方式設計（07サブ）の入力となる。

---

## 1. Windows Server 2022 設定方針

### 1-1. IIS 10（Application Request Routing リバースプロキシ）

IIS 10はリバースプロキシとしてのみ機能し、静的ファイルサービス・マスタメンテナンスWebアプリのサーブ、およびhttps終端を担う。バックエンドRust APIサーバーへのプロキシはARR（Application Request Routing）モジュールを使用する。

| 設定項目 | 確定値 | 根拠 |
|---|---|---|
| TLSバージョン | TLS 1.3のみ（TLS 1.0 / 1.1 / 1.2を無効化） | PCI DSS / NIST SP 800-52 Rev.2の要求。TLS 1.2は既知の脆弱性（BEAST/POODLE等）のリスクを持つ |
| HSTS（HTTP Strict-Transport-Security）ヘッダー | `max-age=31536000; includeSubDomains` | ダウングレード攻撃防止。1年間のキャッシュで内部ブラウザのHTTPS強制 |
| URLリライト（SPAルーティング） | すべての非APIパスを`index.html`にリライト | React SPAのクライアントサイドルーティング対応 |
| ARRプロキシルール（terminal-api）| `/api/v1/events/*`・`/api/v1/evidence/*`・`/api/v1/sync/*`・`/api/v1/iqc/inspect/*`・`/api/v1/rework/execute/*` → `http://localhost:8080/api/v1/...` | ハンディ端末向け Rust axum バックエンドへの転送 |
| ARRプロキシルール（master-api）| `/api/v1/sop/*`・`/api/v1/master/*`・`/api/v1/approval/*`・`/api/v1/users/*`・`/api/v1/audit/*`・`/api/v1/reports/*`・`/api/v1/ops/*`・`/api/v1/trace/*`・`/api/v1/alerts/*`・`/api/v1/capas/*`・`/api/v1/kaizen-proposals/*`・`/api/v1/iqc/judge/*`・`/api/v1/iqc/concession/*`・`/api/v1/rework/disposition/*`・`/api/v1/scrap/*`・`/api/v1/return/*` → `http://localhost:8081/api/v1/...` | 管理・マスタメンテナンス向け Rust axum バックエンドへの転送 |
| ARRプロキシルール（認証）| `/api/v1/auth/*` → デフォルト `http://localhost:8080/api/v1/auth/...`（terminal-api）。フロントエンドが JWT の `aud` クレームで接続先を決定してもよい | 認証エンドポイントの転送先（詳細は 05_外部インターフェース設計 §03 で確定）|
| X-Frame-Options | `DENY` | クリックジャッキング防止 |
| X-Content-Type-Options | `nosniff` | MIMEスニッフィング防止 |
| Content-Security-Policy | `default-src 'self'`（詳細は07_セキュリティ方式設計に委譲） | XSS攻撃の基本防御 |

### 1-2. WSL2設定

| 設定項目 | 確定値 | 根拠 |
|---|---|---|
| systemd有効化 | `systemd=true`（/etc/wsl.conf） | Docker Daemonのsystemd管理・サービス自動起動 |
| DNS解決 | `/etc/resolv.conf`でWindows側DNSサーバー（NODE-001のWindows DNSロール）を参照 | 内部FQDN（wnav-server.factory.local）の解決 |
| メモリ上限 | `.wslconfig`でWSL2に割り当てるメモリ上限を設定（推奨: 24 GB、残8 GBをWindows OSへ） | PostgreSQLのshared_buffers 4 GB + Docker等の合算 |
| スワップ | `.wslconfig` swap = 4 GB | PostgreSQL work_mem最悪ケースの一部をスワップで吸収 |

### 1-3. ストレージ・セキュリティ

| 設定項目 | 確定値 | 根拠 |
|---|---|---|
| BitLocker（データボリューム） | 有効化（データ格納ドライブに適用） | 物理メディア盗難時のデータ保護（NFR-SEC-035相当） |
| Windows Defender | 有効化（除外リスト: PostgreSQLデータディレクトリ・Dockerボリューム） | リアルタイムスキャンによるDB性能劣化を防ぐため除外設定を適用 |
| Windows Firewall | 有効化（インバウンド許可: HTTPS 443, NTP 123/UDP, DNS 53/UDP のみ外部向け。WSL2内部は全開放） | ネットワークゾーニング設計と整合 |
| 自動更新 | Windows Update を手動承認モード（メンテナンスウィンドウ 02:00-04:00 に限定） | 計画外再起動によるサービス停止防止 |

## 2. PostgreSQL 16 設定方針

### 2-1. 接続・パフォーマンス設定

| CFG識別子 | パラメータ名 | 確定値 | 根拠 |
|---|---|---|---|
| CFG-001 | max_connections | 100 | バックエンド接続プール（PgBouncer使用の場合は25〜50に縮小可。詳細設計に委譲）|
| — | shared_buffers | 4 GB | ホットデータ（直近パーティション）のメモリ収容（§06参照） |
| — | effective_cache_size | 16 GB | クエリプランナーがインデックスを優先選択するためのヒント値 |
| — | work_mem | 256 MB | ソート・ハッシュ操作のメモリ上限（最悪ケース: 100接続 × 256 MB） |
| — | maintenance_work_mem | 1 GB | VACUUM・ANALYZE・REINDEX の処理速度向上 |
| — | max_parallel_workers_per_gather | 2 | バッチ集計・帳票生成の並列クエリ |

### 2-2. WAL・アーカイブ設定

| CFG識別子 | パラメータ名 | 確定値 | 根拠 |
|---|---|---|---|
| — | wal_level | replica | WALアーカイブおよびPITR（Point-In-Time Recovery）を有効化するための最低レベル |
| — | archive_mode | on | WALセグメントを自動的にアーカイブディレクトリにコピー |
| — | archive_command | `cp %p /var/lib/postgresql/wal_archive/%f`（パスは構築時確定） | バックアップ方式設計（§08）と整合 |
| — | wal_keep_size | 1 GB | アーカイブ失敗時のWAL自動削除を防ぐバッファ |
| — | checkpoint_timeout | 10min | WAL書き込み頻度のトレードオフ（デフォルト5minより緩和） |

### 2-3. ログ設定

| パラメータ名 | 確定値 | 根拠 |
|---|---|---|
| log_min_duration_statement | 1000 ms | 1秒を超えるSQLをスロークエリログに記録。性能劣化の早期検出 |
| log_checkpoints | on | チェックポイント発生のログ記録（I/Oスパイク分析） |
| log_connections / log_disconnections | on | 接続履歴の監査証跡（NFR-AUD-001相当） |
| log_lock_waits | on | デッドロック前のロック待ちを記録 |
| log_line_prefix | `'%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h '` | 接続単位での追跡可能性確保 |

### 2-4. セキュリティ設定

| 設定項目 | 確定値 | 根拠 |
|---|---|---|
| pg_hba.conf接続制限 | localsocket + Docker内部ネットワーク（127.0.0.1/32）のみ。外部ネットワークからの直接接続禁止 | バックエンドのみがDBに接触する設計（NFR-SEC-030相当） |
| パスワード認証方式 | scram-sha-256 | md5は廃止扱いのため非採用 |
| SSL | on（バックエンド↔DB間はDockerネットワーク内部のため任意、外部露出なし） | 詳細設計で方針を確定 |

## 3. Docker Compose 設定方針

### 3-1. サービス構成とリソース制限

| サービス名 | コンテナイメージ | CPU制限 | メモリ制限 | 備考 |
|---|---|---|---|---|
| `terminal-api` | `wnav-terminal-api`（自作イメージ：Rust axum） | 1.5 CPU | 1.5 GB | ハンディ端末向けAPIバックエンド（ポート 8080）|
| `master-api` | `wnav-master-api`（自作イメージ：Rust axum） | 1.0 CPU | 1.0 GB | 管理・マスタメンテナンス向けAPIバックエンド（ポート 8081）|
| `postgres` | postgres:16-alpine | 2.0 CPU | 6 GB（shared_buffers 4 GB + OS用） | DB（WSL2内Docker） |
| `wnav-frontend-dev` | node:xx-alpine（開発時のみ） | 1.0 CPU | 512 MB | マスタメンテナンス（本番はIIS静的サーブ） |

Dockerコンテナのリソース制限（`deploy.resources.limits`）を必ず設定し、単一コンテナの暴走が他サービスに影響しないよう隔離する。

### 3-2. 再起動ポリシー・ヘルスチェック

| 設定項目 | 確定値 | 根拠 |
|---|---|---|
| restart ポリシー | `unless-stopped` | 手動停止（メンテナンス）以外は常時自動再起動 |
| `terminal-api` ヘルスチェック | `GET http://localhost:8080/api/health` → HTTP 200（30秒間隔・3回失敗でunhealthy） | axumに `/api/health` エンドポイントを実装必須 |
| `master-api` ヘルスチェック | `GET http://localhost:8081/api/health` → HTTP 200（30秒間隔・3回失敗でunhealthy） | axumに `/api/health` エンドポイントを実装必須 |
| `postgres` ヘルスチェック | `pg_isready -U wnav -d wnavdb`（30秒間隔） | DBが応答不能な場合に各APIサービスを依存停止させる |
| 依存関係 | `terminal-api` / `master-api` ともに `depends_on: postgres: condition: service_healthy` | DB起動前のAPI起動を防止 |

### 3-3. ボリューム設定

| ボリューム名 | マウント先（コンテナ内） | ホスト側パス（WSL2） | 備考 |
|---|---|---|---|
| `wnav_postgres_data` | `/var/lib/postgresql/data` | WSL2内の専用ディレクトリ | PostgreSQLデータディレクトリ |
| `wnav_wal_archive` | `/var/lib/postgresql/wal_archive` | WSL2内（Windowsドライブにシンボリックリンク可） | WALアーカイブ |
| `wnav_evidence_files` | `/app/evidence` | Windowsホスト側ドライブ（D:\evidence等） | エビデンスファイル（Windowsバックアップ対象） |

## 4. CFGパラメータ一覧（本章確定分）

| 識別子 | パラメータ名 | 確定値 | 設定場所 |
|---|---|---|---|
| CFG-001 | db_max_connections | 100 | `postgresql.conf` |
| CFG-002 | jwt_expiry | 8時間（詳細は07_セキュリティ方式設計で確定） | `src/infra/config/config.{profile}.yml`（`jwt.ttl_sec`） |
| CFG-003 | offline_emergency_timeout | 5分（WiFi切断からEmergency Mode移行まで） | `src/infra/config/config.{profile}.yml`（`frontend.handy.offline_timeout_min`） |
| CFG-013 | tls_min_version | TLS 1.3 | IIS SSL設定 |
| CFG-014 | hsts_max_age | 31536000秒（1年） | IIS カスタムヘッダー |
| CFG-015 | log_slow_query_threshold | 1000ms | `postgresql.conf` |

---

**本節で確定した方針**

- **TLSはTLS 1.3のみを許可（TLS 1.0/1.1/1.2を無効化）し、HSTSヘッダー（max-age=31536000）をIIS 10で設定することで、内部ネットワーク通信を含む全HTTPS接続のセキュリティ基準を確定する。**
- **PostgreSQL 16はwal_level=replica・archive_mode=onでWALアーカイブを有効化し、log_min_duration_statement=1000msでスロークエリ検出を有効化する（詳細チューニングは構築フェーズに委任）。**
- **Docker Composeはrestart=unless-stopped・ヘルスチェック付きで全サービスを定義し、`terminal-api`（ポート 8080）および `master-api`（ポート 8081）がともに `postgres` の service_healthy 状態を確認してから起動するサービス依存関係を確定する。IIS ARR のパスルールにより、ハンディ端末向けと管理向けトラフィックを各バイナリへ振り分ける設計を確定する。**
