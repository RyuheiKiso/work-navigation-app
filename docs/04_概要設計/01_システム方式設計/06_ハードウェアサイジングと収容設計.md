# 06 ハードウェアサイジングと収容設計

本章は、単一工場・最大50同時ユーザーの運用条件下で、7年間の稼働を前提としたハードウェアサイジング計算を確定する。算定対象はストレージ（PostgreSQLデータ領域・エビデンスファイル・バックアップ）、メモリ（PostgreSQL共有バッファ・接続ごとのwork_mem・バックエンドプロセス・OS）、およびCPU（APIスループット）の3領域とする。本章の数値は要件定義の非機能要件（NFR-PRF-001 P95レイテンシ≤2000ms、NFR-CAP-001〜003）を充足する最小推奨仕様として確定し、詳細設計・構築フェーズでの調達仕様書の入力とする。

---

## 1. 前提条件

| 項目 | 値 | 根拠 |
|---|---|---|
| 最大同時ユーザー数 | 50名 | オペレーター × 工場フロア数の上限見積（NFR-CAP-001） |
| 運用年数 | 7年（v1.0.0リリースから） | データ保持要件（NFR-CAP-002） |
| work_eventsレコード生成レート | 100イベント/ユーザー/日 | 作業ステップ実行・アンドン・品質記録等の合算上限 |
| エビデンス写真撮影レート | 5枚/ユーザー/日 | 各作業ステップの証跡写真上限 |
| 写真平均ファイルサイズ | 200 KB | JPEG圧縮後（ハンディカメラ解像度相当） |
| PostgreSQL接続数上限 | 100接続（CFG-001） | バックエンド接続プール上限 |
| APIスループット目標 | P95レイテンシ ≤ 2000ms（NFR-PRF-001） | 要件定義確定値 |

---

## 2. ストレージサイジング

### 2-1. work_eventsテーブル

work_eventsは本システムの中核データであり、7年間の全記録を保持する。

| 計算要素 | 計算式 | 値 |
|---|---|---|
| 7年間の総レコード数 | 100 イベント × 50ユーザー × 365日 × 7年 | 127,750,000行（≈ 128M行） |
| 1レコードあたりの平均サイズ | JSONB（work_data）+ タイムスタンプ4本 + UUID + 整数4本 + 文字列等 | ≈ 500 バイト |
| データ領域（概算） | 128M行 × 500B | ≈ 64 GB |
| インデックス（主要3〜5インデックス） | データの≈ 30% | ≈ 20 GB |
| **work_events 小計** | | **≈ 84 GB** |

設計上の安全係数（パーティション管理オーバーヘッド・TOAST領域・MVCC Bloat）として1.3倍を適用する。work_eventsは月次パーティション（PARTITION BY RANGE on occurred_at）を採用し、古いパーティションはpg_partman経由でアーカイブ可能にする（詳細は`04_データ設計`に委譲）。

### 2-2. エビデンスファイル（写真）

| 計算要素 | 計算式 | 値 |
|---|---|---|
| 7年間の総ファイル数 | 5枚 × 50ユーザー × 365日 × 7年 | 638,750ファイル（≈ 640K件） |
| 総容量 | 640,000ファイル × 200 KB | ≈ 128 GB（≈ 130 GB） |

エビデンスファイルはWindows Server 2022のファイルシステム（D:ドライブ等、ハードコードしない）に格納する。ファイルパスはevidence_filesテーブルに記録する（詳細は`04_データ設計`に委譲）。

### 2-3. その他PostgreSQLデータ

| テーブル群 | 推定サイズ | 備考 |
|---|---|---|
| work_instances / work_templates / steps | ≈ 2 GB | マスタデータ・テンプレートは低更新頻度 |
| users / roles / audit_logs / outbox_events | ≈ 4 GB | audit_logsは7年累積でも限定的 |
| その他（capa / andon_alerts / notifications等） | ≈ 2 GB | |

### 2-4. PostgreSQL全体ストレージ

| 領域 | サイズ |
|---|---|
| work_events（データ + インデックス × 1.3安全係数） | ≈ 110 GB |
| エビデンス外その他テーブル | ≈ 8 GB |
| WALバッファ（pg_wal領域、archive前のローカル保持） | ≈ 20 GB |
| **PostgreSQL合計** | **≈ 138 GB → 余裕を見て 160 GB** |

### 2-5. バックアップ（ローカル保持）

| バックアップ種別 | 保持期間 | 推定サイズ |
|---|---|---|
| pg_dumpフルバックアップ（日次、BAT-001） | 7世代 | ≈ 30 GB（圧縮後） |
| WALアーカイブ（14日保持） | 14日 | ≈ 15 GB |
| エビデンスファイルrsync差分 | スタンバイ側に1世代 | 本サーバー側追加コストなし |
| **バックアップ合計** | | **≈ 45 GB → 余裕を見て 50 GB** |

### 2-6. ストレージ総計と推奨構成

| 領域 | サイズ |
|---|---|
| PostgreSQL（データ + WAL） | 160 GB |
| エビデンスファイル | 130 GB |
| ローカルバックアップ | 50 GB |
| OS + ミドルウェア + Dockerイメージ + ログ | 60 GB |
| **合計** | **400 GB** |

推奨ストレージ: **2TB RAID-1（ミラーリング）**

提供容量に対する7年後の使用率：400 GB / 2000 GB = 20%。実際の使用量に対して**5倍の余裕**を確保し、想定外の急増にも対応できる。RAID-1により単一ドライブ障害時もデータ欠損なく継続稼働する。

## 3. メモリサイジング

### 3-1. PostgreSQL共有バッファ

PostgreSQLの推奨設定は物理RAMの25〜40%をshared_buffersに割り当てることである。32 GBの場合、shared_buffers = 8 GBが上限の目安であるが、本システムでは作業系テーブルのホットデータが集中するためwork_eventsの直近パーティション（≈ 4 GB）が常にメモリ上に乗ることを目標とする。

| パラメータ | 確定値（CFG識別子は§CFG一覧参照） | 根拠 |
|---|---|---|
| shared_buffers | 4 GB | ホットパーティション＋インデックス収容の実用値 |
| effective_cache_size | 16 GB | OSページキャッシュとshared_buffersの合算目安（クエリプランナーヒント） |

### 3-2. 接続ごとのwork_mem

| 計算要素 | 値 |
|---|---|
| work_mem設定値 | 256 MB |
| 最大同時接続（max_connections=100） | 100接続 |
| 最大消費（全接続がソート同時実行の最悪ケース） | 256 MB × 100 = 25.6 GB |

実運用では全接続が同時に大規模ソートを行う確率は低いため、実効消費は25 GBを大幅に下回る。ただしメモリサイジングは最悪ケースを考慮して行う。

### 3-3. メモリ総計と推奨

| 消費源 | 推定サイズ |
|---|---|
| PostgreSQL shared_buffers | 4 GB |
| PostgreSQL work_mem × 100接続（最悪ケース） | 25.6 GB |
| Rust axum バックエンドプロセス | 2 GB |
| React / マスタメンテナンス（Nginxサーブ） | 0.5 GB |
| Docker Compose オーバーヘッド | 1 GB |
| Windows Server 2022 OS カーネル・WSL2 | 4 GB |
| **合計（最悪ケース）** | **≈ 37 GB** |

推奨RAM: **32 GB（通常運用の余裕を十分に確保）**

通常運用（平均同時ソート接続 ≈ 10〜20）では実効メモリ消費は15 GB前後であり、32 GBで十分な余裕を持つ。将来的な拡張（バッチ処理・レポート生成の並行実行等）にも対応できる。

## 4. CPU・スループットサイジング

### 4-1. APIスループット目標

NFR-PRF-001が要求するP95レイテンシ ≤ 2000msを、同時50ユーザーのピーク時でも維持することを確認する。

| 指標 | 計算根拠 | 値 |
|---|---|---|
| ピークリクエストレート（推定） | 50ユーザー × 2リクエスト/秒（作業ステップ送信ピーク） | ≈ 100 req/s |
| Rust + axum の処理能力（4コア構成） | axumはtokioの非同期I/Oを活用し、4コアで1,000 req/s以上のスループットを達成（DBボトルネックなしの場合） | 1,000+ req/s |
| DBボトルネック下でのP95レイテンシ | 単純なINSERT（work_events）+ SELECTのP95は適切なインデックス設計下で ≤ 50ms | ≤ 50ms（DB） |
| エンドツーエンドP95（API処理 + DB + WiFi往復） | 50ms（DB） + 20ms（Rust処理） + WiFi遅延（LAN内≤ 10ms） | ≈ 80ms << 2000ms |

想定ピーク（100 req/s）に対して処理能力（1,000+ req/s）は10倍以上の余裕を持つ。NFR-PRF-001への適合を確定する。

### 4-2. 推奨CPU仕様

| 項目 | 推奨値 | 根拠 |
|---|---|---|
| コア数 | 4コア / 8スレッド以上 | tokio非同期ランタイムはコア数に比例してスループット向上 |
| クロック周波数 | 2.5 GHz以上 | PostgreSQLのシングルスレッド処理性能 |
| 推奨例 | Intel Core i7-12xxx相当以上 または Xeon E-2300シリーズ相当 | 構築時に市場調達品で確定 |

図: fig_des_sys_hw_sizing_summary（img/ 配下）を参照。

| コンポーネント | 最低仕様 | 推奨仕様 | 根拠 |
|---|---|---|---|

---

**本節で確定した方針**

- **7年間・128M行のwork_eventsおよび640Kファイルのエビデンスに対してPostgreSQL 160 GB + ファイル 130 GB + バックアップ 50 GB + OS/MW 60 GBの合計400 GBを算定し、2TB RAID-1（5倍余裕）を推奨ストレージとして確定する。**
- **メモリはPostgreSQL shared_buffers 4 GB + work_mem 256MB × 100接続（最悪ケース25.6 GB）+ バックエンド2 GB + OS 4 GBの合算から32 GB RAMを推奨として確定する。**
- **Rust + axumは4コアで1,000 req/s以上のスループットを達成し、想定ピーク100 req/sに対して10倍以上の余裕を持つことを確認し、NFR-PRF-001（P95 ≤ 2000ms）への適合を確定する。**
