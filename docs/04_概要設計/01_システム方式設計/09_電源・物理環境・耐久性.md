# 09 電源・物理環境・耐久性

本章は、工場オンプレミス環境に設置するサーバー・ネットワーク機器の電源保護設計、物理設置環境の要件、サーバー突然停止時のデータ保護方式、およびハンディ端末の物理耐久性要件を確定する。工場環境特有の電力品質（瞬時停電・電圧変動）・温湿度・振動・粉塵に対する対策を設計命題として確定し、ハードウェア調達仕様書の入力とする。本章が確定する要件はソフトウェア設計に影響する範囲（SQLite WAL・PostgreSQLクラッシュリカバリ・Graceful shutdown手順）を含む。

---

## 1. UPS（無停電電源装置）設計

### 1-1. UPS適用範囲と要件

| 保護対象機器 | UPS必要性 | 根拠 |
|---|---|---|
| Windows Server 2022（NODE-001） | 必須 | PostgreSQLのデータ破損防止・WAL書き込み中断防止 |
| ネットワークスイッチ（工場LANスイッチ） | 必須 | UPS停電中もハンディ端末がサーバーに接続できる状態を維持 |
| WiFiアクセスポイント | 必須 | ハンディ端末のEmergency Mode移行タイミングを制御するため |
| ハンディ端末（Android / iOS / Windows） | 不要（内蔵バッテリーで対応） | バッテリー駆動のため別途UPS不要 |

### 1-2. UPSランタイム要件

| 要件 | 確定値 | 根拠 |
|---|---|---|
| 最低ランタイム | 30分以上（サーバー + スイッチ + APの合算負荷で） | Graceful shutdown完了（PostgreSQLチェックポイント + ファイルシステムsync）に最大10分。残余の20分は復電待機または外部給電待ち |
| 推奨ランタイム | 60分以上 | 電力会社の一般的な停電対応時間（〜30分）をカバーし、長時間停電時の計画的シャットダウンに十分な余裕 |
| 自動シャットダウントリガー | バッテリー残量50%（15〜30分相当）以下でGraceful shutdownを自動開始 | UPS管理ソフトウェア（NUT / APC PowerChute等）とOS連携 |

### 1-3. UPS監視・アラート

| 監視項目 | アラート条件 | 通知先 |
|---|---|---|
| 停電検出 | AC電源喪失 → 即時 | system_adminへのローカルアラート（メール/Windows通知） |
| バッテリー残量低下 | 50%以下 | system_adminへの緊急通知 + 自動Graceful shutdown開始 |
| バッテリー劣化 | 自己診断テスト失敗または推定ランタイムが閾値（20分）を下回る | system_adminへの保守アラート（月次点検対象） |

## 2. Graceful Shutdown手順

停電検出時のGraceful shutdown手順を以下に確定する。手順はsystemdのpre-stopフックとWindowsのシャットダウンスクリプトで自動実行する。

| ステップ | 処理内容 | 実行方式 | 最大所要時間 |
|---|---|---|---|
| 1. 新規APIリクエスト受付停止 | axumサーバーへのLISTEN停止（SIGTERM送信） | systemd service pre-stop hook | 30秒 |
| 2. 処理中リクエストの完了待機 | axum graceful shutdown（デフォルトタイムアウト: 30秒） | axum組み込み機能 | 30秒 |
| 3. PostgreSQLチェックポイント実行 | `CHECKPOINT` SQL発行 → WAL書き込み完了保証 | pg_ctlまたはDocker exec | 60秒 |
| 4. PostgreSQL停止 | `pg_ctl stop -m fast` | Docker stop | 30秒 |
| 5. ファイルシステムsync | `sync` コマンド実行 | シェルスクリプト | 10秒 |
| 6. Docker Compose停止 | `docker compose down` | シェルスクリプト | 30秒 |
| 7. WSL2シャットダウン | `wsl --shutdown` | Windows PowerShell | 30秒 |
| 8. Windowsシャットダウン | `Stop-Computer` | Windows PowerShell | 60秒 |
| **合計** | | | **≤ 5分** |

Graceful shutdown完了後、PostgreSQLの次回起動時に追加のクラッシュリカバリは不要となる。

## 3. PostgreSQLクラッシュリカバリ

Graceful shutdownが完了する前に電源が遮断された場合（UPS電池切れ・UPS故障）のリカバリ手順を確定する。

### 3-1. 自動クラッシュリカバリ

PostgreSQL 16はWrite-Ahead Logging（WAL）に基づく自動クラッシュリカバリを提供する。

| 状況 | PostgreSQLの動作 | system_admin操作 |
|---|---|---|
| Graceful shutdown完了済み | 次回起動時にクラッシュリカバリ不要 | 通常起動 |
| 突然停止（WAL書き込み中断） | 次回起動時にWALを自動的にreplayしてデータ整合性を復元（`pg_ctl start`） | 起動コマンド発行のみ |
| データファイル破損（ディスク障害） | 自動リカバリ不可 | バックアップ（BAT-001）からpg_restoreを実施 + WAL PITRで最新時点まで復旧 |

### 3-2. 起動時クラッシュリカバリの確認手順

`docker compose up`後にPostgreSQLログ（`docker logs wnav-postgres`）に以下のメッセージが含まれる場合はクラッシュリカバリが実行された証拠であり、完了後は正常運用となる。

- `database system was shut down unexpectedly` → クラッシュリカバリ実行中
- `database system is ready to accept connections` → リカバリ完了・正常起動

リカバリに失敗した場合はERR-DB-CRASH-001としてsystem_adminがバックアップ復旧手順（§08 BAT-001参照）を実施する。

## 4. サーバー室の物理環境要件

| 環境項目 | 要件値 | 根拠 |
|---|---|---|
| 動作温度 | 10〜35°C | サーバー製造元仕様の一般的範囲（ASHRAE A1クラス） |
| 動作湿度 | 20〜80%（結露なきこと） | 同上。結露によるショートサーキット防止 |
| 塵埃 | 防塵対策（フィルター付きラックまたはサーバー室への粉塵流入防止） | 工場環境の粉塵がサーバー冷却ファンに侵入するリスクへの対処 |
| 冷却 | 空調によるサーバー室温度管理（10〜35°C範囲を維持） | 夏季の高温による熱暴走防止 |
| 施錠 | サーバー室に物理鍵による施錠（system_adminのみアクセス） | NFR-SEC-050相当。物理的不正アクセス防止 |
| 耐震対策 | ラックのアンカーボルト固定（地震対策） | 日本の地震リスク。ラック転倒によるHDD物理損傷防止 |

## 5. ハンディ端末の物理耐久性要件

ハンディ端末の耐久性要件はソフトウェア設計で強制できないため、**調達仕様書（推奨）** として確定する。本システムが端末の物理スペックを実行時に検証する機能は実装しない。

| 要件 | 推奨仕様 | 識別子 | 備考 |
|---|---|---|---|
| 防塵・防水性能 | IP54以上（推奨: IP65） | CFG-DEV-001（デバイス調達推奨） | IP54: 粉塵の侵入を制限 + 全方向からの水の飛沫に耐える。工場フロアの洗浄水・切削油のスプレーに対応 |
| 耐衝撃性 | MIL-STD-810H準拠（推奨） | CFG-DEV-002 | 落下（1.5m）・振動・温度衝撃への耐性。工場フロアでの落下リスクに対応 |
| 振動耐性 | IEC 60068-2-6準拠またはMIL-STD-810H Method 514準拠 | CFG-DEV-003 | 機械振動の多い工場環境でのデータ欠損防止 |
| バッテリー稼働時間 | 8時間以上（1シフト稼働） | CFG-DEV-004 | 充電の機会が限られるフロア作業への対応 |
| 画面視認性 | 屋内照明下での視認性確保（輝度 400 nit以上推奨） | CFG-DEV-005 | 工場照明（蛍光灯・ハロゲン等）の反射下での操作性 |

### 5-1. データ永続性とソフトウェアの役割

ハンディ端末の突然の電源断（バッテリー切れ・電源ボタン誤操作）に対して、ソフトウェアレイヤーで以下の保護を確定する。

| 保護機構 | 実装方式 | 保証内容 |
|---|---|---|
| SQLite WALモード | `PRAGMA journal_mode=WAL` | コミット済みトランザクションは電源断後も永続化される。WALファイルが残存しても次回起動時に自動的にメインDBにマージされる |
| SQLCipher（AES-256-CBC） | SQLiteの全ページを暗号化 | 端末紛失・盗難時のデータ保護。電源断でも暗号化状態が維持される |
| Outbox書き込みのアトミック性 | work_eventsとoutbox_eventsを同一SQLiteトランザクション内で書き込み | 両方が書き込まれるか、両方が書き込まれないかのいずれかを保証 |

---

**本節で確定した方針**

- **UPSはサーバー・スイッチ・APをカバーする最低30分（推奨60分）ランタイムを要件とし、バッテリー残量50%でGraceful shutdownを自動開始するUPS管理ソフトウェア連携を確定する。**
- **Graceful shutdown（systemd pre-stopフック → PostgreSQLチェックポイント → filesystem sync → 電源断）により突然停止からのクラッシュリカバリをWAL自動replayで解決し、手動操作なしでデータ整合性を復元する。**
- **ハンディ端末の耐久性はIP54以上・MIL-STD-810H準拠を調達推奨として確定し、ソフトウェアレイヤーではSQLite WAL + SQLCipherでコミット済みデータを電源断後も保護する。**
