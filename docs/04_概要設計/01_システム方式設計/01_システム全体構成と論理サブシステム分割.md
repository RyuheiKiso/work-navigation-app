# 01 システム全体構成と論理サブシステム分割

本章は、IPA 共通フレーム 2013「2.3.2 システム方式設計の実施」に対応し、単一工場・単一建屋という前提のもとで作業ナビゲーションシステムを 3 つの論理サブシステムに分割し、各サブシステムの責務・技術選択・相互連携を確定する。論理サブシステム分割は、設計識別子 SUB-001〜003 に対応する。

---

## 1. 配置前提の確定

### 1-1. 単一工場・単一建屋の前提

本システムは以下の前提（企画フェーズで確定した制約）に従い設計する。この前提を変更する場合は設計全体の再評価を必要とする。

| 前提 ID | 内容 | 根拠 |
|---|---|---|
| P01 | 単一工場・単一建屋への配置 | `02_企画/システム化計画/11_前提・制約・リスク.md` |
| P02 | 地理的冗長（マルチサイト）は対象外と判断する | 個人開発・IT 担当 1 名の制約 |
| P03 | 工場内クローズドネットワーク（インターネット接続不要）| `02_企画/システム化計画/05_アーキテクチャ原則.md` |
| P04 | OT（PLC/SCADA）との直接ネットワーク接続は永続禁止 | 製造 OT 安全原則・ADR-SYS-004 |
| P05 | 最大作業員数 500 名（NFR-PRF-001）| `03_要件定義/非機能要件/02_性能・拡張性要件.md` |

単一建屋内冗長（Active-Standby）は採用するが、地理的分散配置は採用しない。この決定は ADR-SYS-002 に記録する。

### 1-2. ネットワーク前提

| 項目 | 内容 |
|---|---|
| 通信路 | 工場内 LAN（有線 Gigabit Ethernet + Wi-Fi） |
| 外部接続 | 親機（外部システム）との IF-001/002/003 のみ（TLS 1.3 + ファイアウォール許可リスト） |
| インターネット | 不要（クローズドネットワーク前提） |
| OT 接続 | 禁止（P4 原則・永続禁止） |

## 2. 論理サブシステム分割

本システムを以下の 3 論理サブシステムに分割する。分割の基準は「データの発生場所・処理の主体・ユーザーの種別」の 3 軸である。

> 原本: [`img/fig_des_deploy_system_overview.drawio`](img/fig_des_deploy_system_overview.drawio)

### 2-1. SUB-001: フィールド層（Handy APP）

| 項目 | 内容 |
|---|---|
| 識別子 | SUB-001 |
| 名称 | フィールド層（Handy APP） |
| 主責務 | 作業員が現場で作業ナビゲーションを受け、作業実績・証拠を記録する |
| 動作端末 | Android タブレット・iOS タブレット・Windows タブレット（ハンディ端末） |
| フレームワーク | React Native 0.74+ |
| ローカルストレージ | SQLite（SQLCipher 暗号化）+ TypeORM |
| 外部接続 | BLE（スキャナ・計測器：IF-005/006）、カメラ（IF-007）、Wi-Fi 経由で SUB-002 |
| 設計方針 | Offline-First（ADR-SYS-001）: サーバー非接続時も作業記録を継続し、Outbox キューに保存する |

**FR 割付**

| 機能要件カテゴリ | 件数 | 割付根拠 |
|---|---|---|
| FR-NV（ナビゲーション）| 13 件 | 作業員がタブレットで Step を進める主フロー |
| FR-EV（証拠記録）| 12 件 | 写真・バーコード・計測値の現場取得 |
| FR-ST（ステップ制御）| 12 件中 8 件 | タブレット側のローカル Step 制御 |
| FR-UI（UI 共通）| 11 件中 6 件 | タブレット UI 規約（グランサブル・アクセシビリティ） |

**Offline-First の動作定義**

| 状態 | 動作 |
|---|---|
| オンライン | SQLite ローカル書込み後、即時 Outbox キューへ push して SUB-002 と同期する |
| オフライン（Wi-Fi 切断）| SQLite ローカル書込みのみ。Outbox キューに未送信として保持する |
| 復帰後 | Outbox Consumer が差分を SUB-002 API へ自動送信する |

NFR-AVL-004（Offline-First による縮退稼働）・NFR-AVL-010（Outbox バッファ自動同期）を本設計で充足する。

---

### 2-2. SUB-002: サーバー層（Backend + DB）

| 項目 | 内容 |
|---|---|
| 識別子 | SUB-002 |
| 名称 | サーバー層（Backend + DB） |
| 主責務 | 作業実績の永続化・マスタデータ管理・外部システム連携 API の提供 |
| 動作ホスト | Windows Server 2022（NODE-001: Active）+ NODE-002（Standby） |
| 仮想化 | WSL2 上の Docker Compose（ENV-ITEM-002/003/004） |
| API サーバー | Rust Edition 2024 + tokio + axum（ENV-ITEM-007/008/009）。2 バイナリ構成：`terminal-api`（ポート 8080）+ `master-api`（ポート 8081） |
| DB | PostgreSQL 16+（ENV-ITEM-006） |
| リバースプロキシ | IIS 10（ENV-ITEM-005）: Windows Server 2022 上に直接配置し、コンテナへの HTTPS → HTTP リバースプロキシを担う |
| 外部連携 | IF-001（親機マスタ同期）・IF-002（Outbox 実績送信）・IF-003（LDAP/AD 認証） |

**FR 割付**

| 機能要件カテゴリ | 件数 | 割付根拠 |
|---|---|---|
| FR-SY（同期・認証）| 9 件 | Outbox Consumer・JWT 発行・LDAP 連携 |
| FR-AU（監査証跡）| 6 件 | 改ざん防止 Append-only テーブル・電子サイン |
| FR-ST（ステップ制御）| 12 件中 4 件 | サーバー側 Step 状態検証・不正操作防止 |
| FR-MA（マスタ管理）| 15 件中 7 件（READ 系）| SOP/作業手順マスタの参照 API |

**コンテナサービス構成（概要）**

バックエンドは 2 バイナリ構成とし、ハンディ端末向けと管理者向けのプロセスを分離する。

| サービス名 | 主責務 | 詳細 |
|---|---|---|
| `terminal-api`（ポート 8080）| ハンディ端末向け Rust axum API サーバー | 作業実績記録・ナビゲーション・Outbox 同期・IQC 検査 REST エンドポイント。バッチジョブ（BAT-001/002）を tokio task として内包する |
| `master-api`（ポート 8081）| マスタメンテ・管理コンソール向け Rust axum API サーバー | SOP/マスタ管理・承認・ユーザー管理・監査・帳票・CAPA REST エンドポイント。バッチジョブを tokio task として内包する |
| `wnav_iqc` | IQC 専用バウンデッドコンテキスト | 入荷ロット管理・AQL 判定・特採/選別 API を担当する |
| `postgres` | PostgreSQL 16 | ACID 永続化・RLS・Append-only 強制 |

`wnav_scheduler` は廃止。BAT-001（フルバックアップ）・BAT-002（Outbox Consumer）は各 API の tokio task に内包する。

詳細なコンテナ配置表は §03（配置設計）で確定する。

**IQC サブシステム（wnav_iqc）の概要**

| 項目 | 内容 |
|---|---|
| サービス名 | `wnav_iqc` |
| 担当 BC | IQC Bounded Context（MOD-BE-iqc-001）|
| 主責務 | 入荷ロットの QC 状態管理・AQL 判定・サンプリング計画適用・特採承認・選別サブロット発行 |
| 主エンティティ | TBL-036〜042（materials / suppliers / incoming_inspections / sampling_plans 等）|
| 提供 API | API-iqc-001〜006（/api/v1/iqc/* エンドポイント群）|
| 既存 BC との分離方針 | 既存の NonConformity BC（Quality Context）とは ID 参照のみで連携し、モジュール境界を越えた直接依存を禁止する |

## 3. サブシステム間インタラクション

### 3-1. 主要インタラクション一覧

| インタラクション ID | 送信元 | 受信先 | プロトコル | 方向 | 主要データ |
|---|---|---|---|---|---|
| INT-001 | SUB-001 | SUB-002 | HTTPS REST（TLS 1.3）| 双方向 | Step 実績・Outbox イベント・マスタ差分 |
| INT-002 | SUB-003 | SUB-002 | HTTPS REST（TLS 1.3）| 双方向 | SOP CRUD・承認操作・監査ログ照会 |
| INT-003 | SUB-002 | 親機（外部）| HTTPS REST（TLS 1.3 + mTLS）| 双方向 | IF-001〜003（外部システム連携） |
| INT-004 | SUB-001 | スキャナ・計測器 | BLE GATT / USB HID | 双方向 | IF-005/006（自動認識） |
| INT-005 | SUB-001 | カメラ | OS Camera API | 単方向 | IF-007（写真取込） |

### 3-2. 認証フロー概要

すべての SUB-001 / SUB-003 → SUB-002 通信は JWT（RS256）による認証を必須とする。詳細は 05_外部インターフェース設計 §03 で確定する。

| 接続元 | 認証方式 | フォールバック |
|---|---|---|
| SUB-001（Handy APP）| JWT RS256（LDAP/AD 連携 IF-003 で取得）| ローカル認証（工場 LAN 断絶時） |
| SUB-003（Web APP）| JWT RS256（同上）| なし（Web APP はオンライン専用） |
| Outbox Consumer | サービスアカウント JWT | なし |

## 4. データ所有と永続化責任の分担

| データ種別 | 一次ストア | 二次ストア |
|---|---|---|
| 作業実績（work_events）| SUB-002（PostgreSQL、Append-only）| SUB-001（SQLite ローカルキャッシュ） |
| マスタ（SOP・手順）| SUB-002（PostgreSQL）| SUB-001（SQLite リードオンリーキャッシュ） |
| 証拠ファイル（写真・PDF）| SUB-002（ファイルストレージ + PostgreSQL メタデータ）| SUB-001（一時保存後削除） |
| 監査ログ | SUB-002（PostgreSQL、Append-only）| 帳票出力時に CSV/PDF へ書き出す |
| ローカル作業コンテキスト | SUB-001（SQLite のみ）| なし |

---

**本節で確定した方針**
- システムを SUB-001（フィールド層）・SUB-002（サーバー層）・SUB-003（クライアント層）の 3 論理サブシステムに分割し、各サブシステムの FR 割付・技術選択・データ所有責任を確定する。
- 単一工場・単一建屋への集中配置と地理的冗長の非採用を前提 P01〜P05 として確定し、ADR-SYS-001〜005 に設計判断を記録する。
- SUB-001 における Offline-First（ADR-SYS-001）を NFR-AVL-004/010 の充足手段として確定し、Outbox キュー方式による自動同期を設計基盤とする。
- バックエンドを `terminal-api`（ポート 8080：ハンディ端末向け）と `master-api`（ポート 8081：マスタメンテ・管理コンソール向け）の 2 バイナリ構成に分割し、関心の分離と障害ドメイン分離を実現する。`wnav_scheduler` は廃止し、BAT-001/002 の各ジョブを対応する API の tokio task に内包する。
- IQC 専用バウンデッドコンテキスト（`wnav_iqc`）を SUB-002 内の独立コンテナサービスとして追加し、入荷ロット管理・AQL 判定・特採/選別の責務を既存の API サーバーから分離する。既存 NonConformity BC とはロット ID による参照のみで連携し、直接依存を禁止する。
