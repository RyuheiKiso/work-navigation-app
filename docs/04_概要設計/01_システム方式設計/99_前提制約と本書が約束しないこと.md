# 99 前提制約と本書が約束しないこと

本章は、01_システム方式設計サブが対象外と判断した設計領域を権威的に列挙する。IPA共通フレーム2013の2.3システム方式設計プロセスはスコープ境界を明示することを要求しており、本章はその義務を履行する。各除外事項はD-SYS識別子で採番し、除外の根拠を必ず付記する。本章に列挙した項目はいかなる下位章においても再定義・逸脱してはならない。除外事項に起因する実装の試みが発生した場合は、architectによるスコープレビューを必須とする。

本章の記述様式は「対象外と判断する（根拠: P4原則・個人運用・単一建屋等）」に統一する（言語規約: 00_本書の位置づけ §4参照）。

---

## 1. システム構成に関する除外事項

### D-SYS-001: OT直結（PLC / SCADA / OPC-UA / MQTT永続接続）

**対象外と判断する。**

本システムのIFリスト（IF-001〜007）にOTプロトコルを使用した外部インタフェースは存在しない。PLC（Programmable Logic Controller）・SCADA（Supervisory Control and Data Acquisition）・OPC-UA・MQTT等のOT（Operational Technology）層との永続的な通信インターフェースは、本システム設計の外部境界に含まれない。

| 除外の根拠 | 内容 |
|---|---|
| P4原則（データオーナーシップ・オフラインファースト） | OT直結は外部システム依存を生じさせ、オフライン継続稼働の設計原則（企画/システム化計画/05_アーキテクチャ原則）と抵触する可能性がある |
| スコープ境界 | 本システムは「作業者がハンディ端末でナビゲーションを受けながら記録する」ユースケースに特化する。PLCからのセンサーデータ自動取込みは別システムの責務と判断する |
| 実装複雑性 | OPC-UAクライアントライブラリの選定・セキュリティ（OPC-UA SecureChannel）・リアルタイム性の要求はシステム方式設計の範囲を超える |

将来的にOT連携が必要になった場合は、新規IFリスト追加・スコープ変更管理手順を経て設計変更を行う。

---

### D-SYS-002: 複数建屋・複数工場への展開

**対象外と判断する。**

本システムは**単一建屋・単一工場**の構成を前提とする。地理的に離れた複数拠点（建屋・工場）への同一システムの横展開、拠点間のデータレプリケーション、マルチテナント対応、および地理的冗長性（Geo-redundancy）は本設計に含まれない。

| 除外の根拠 | 内容 |
|---|---|
| 構成前提 | 単一建屋のオンプレミス構成（Windows Server 2022 + WSL2 + Docker Compose）が確定前提であり、地理的分散設計はこの前提と非整合 |
| 個人運用の制約 | 1名での複数拠点管理は運用コストが非現実的 |
| ネットワーク設計の簡素化 | 単一工場LAN内のゾーニング設計（04_ネットワーク構成）が複数拠点では根本的に再設計を要する |

---

### D-SYS-003: クラウドインフラへのデプロイ

**対象外と判断する。**

本システムのデプロイ先は**Windows Server 2022 + WSL2 + Docker Composeのオンプレミス構成に限定する**。AWS・Azure・Google Cloud・OCI等のパブリッククラウド、ハイブリッドクラウド、プライベートクラウド（VMware等）へのデプロイは設計スコープに含まれない。

| 除外の根拠 | 内容 |
|---|---|
| インターネット非接続要件 | 工場セキュリティポリシーとして外部インターネット接続なしの運用を前提とする。クラウドデプロイはインターネット経由のAPIエンドポイントを必要とし、この前提と非整合 |
| データオーナーシップ原則 | 工場データをクラウドプロバイダのインフラに格納することは、データ主権・規制上の懸念を生じさせる可能性がある |
| ライセンス方針 | AGPL系ライセンスの禁止理由（§10参照）と整合。クラウドデプロイを設計スコープに含めた場合、AGPL依存のリスク評価が複雑化する |

---

### D-SYS-004: Kubernetes / コンテナオーケストレーション

**対象外と判断する。**

コンテナオーケストレーションはDocker Composeに限定する。Kubernetes（k8s）・OpenShift・Nomad・Docker Swarm等のオーケストレーションプラットフォームは採用しない。

| 除外の根拠 | 内容 |
|---|---|
| 運用複雑性 | Kubernetesのetcd管理・ノードプール管理・Helmチャート管理・ネットワークポリシー等の運用タスクは1名での管理が非現実的 |
| スケール要件 | 最大50同時ユーザー・単一サーバーの構成においてKubernetesの水平スケール能力は不要。Docker Composeで十分な信頼性と運用性を提供できる |
| 習得コスト | Kubernetesの習得・維持コストは個人開発プロジェクトの制約と非整合 |

---

### D-SYS-005: 自動フェイルオーバー

**対象外と判断する。**

PostgreSQLのHigh Availability構成（Patroni・Repmgr・Pacemaker）、Rustバックエンドの自動ヘルスチェック・自動切り替え、およびネットワーク機器の自動冗長切替（VRRP等）は採用しない。代わりに手動フェイルオーバー（RTO 15分）を採用する。

| 除外の根拠 | 内容 |
|---|---|
| 管理の複雑さが利益を超える | Patroniによる自動フェイルオーバーはsplit-brain問題・フェンシング設定・Etcd/Consul管理等の複雑性を伴い、1名での安全な運用は困難 |
| RTO要件との整合 | NFR-AVL-002が要求するRTO 15分は手動フェイルオーバーで充足可能。自動フェイルオーバーの複雑さ導入によるリスクがRTO短縮の利益を超える |
| 単一建屋のリスクプロファイル | 地理的分散がないため、自動フェイルオーバーの主要メリット（遠隔地間の自動切替）が本構成では効果を発揮しない |

---

### D-SYS-006: 詳細なチューニング仕様の確定

**対象外と判断する。**

本書（01_システム方式設計）は、PostgreSQL・OS・ミドルウェアのサイジング方針と設定方針を確定するにとどまる。以下の詳細パラメータの確定値は詳細設計・構築フェーズに委任する。

| 委任先 | 項目例 |
|---|---|
| 詳細設計・構築手順書 | PostgreSQLの`checkpoint_completion_target`・`random_page_cost`・`autovacuum_vacuum_cost_delay`等の個別チューニング値 |
| 詳細設計・構築手順書 | IISのconnection timeout・request queue length・ARRのTimeout設定値 |
| 詳細設計・構築手順書 | Docker Composeのswapiness・ulimit・カーネルパラメータ（`sysctl`）の詳細値 |
| 詳細設計・構築手順書 | WSL2の`.wslconfig`の詳細なメモリ・CPUコア数割り当て値（ハードウェア確定後） |

本書が確定するのは設計方針（What・Why）であり、実装値（具体的な数値）は実環境でのベンチマーク・負荷試験（`10_テスト方式設計`参照）に基づいて構築フェーズで確定する。

## 2. 機能・非機能の除外事項

| 識別子 | 除外事項 | 除外理由（要約） |
|---|---|---|
| D-SYS-007 | MFA（多要素認証） | 社内LAN完結・中小製造業1名運用のため費用対効果が低い（詳細は07_セキュリティ方式設計に委譲） |
| D-SYS-008 | リアルタイムダッシュボード（WebSocket / SSE） | v1.0.0スコープ外。ポーリングベースのダッシュボードで要件充足。WebSocketの複雑性は現フェーズで不要 |
| D-SYS-009 | 機械学習・AI推論の組み込み | v1.0.0スコープ外。`90_業界分析/36_AI・ML支援と知的作業ナビゲーション.md`は将来検討項目であり現設計に含まれない |
| D-SYS-010 | ERP / MES との双方向API統合 | IFリストにERP/MES統合インターフェースは存在しない。スコープ外。単方向エクスポート（IF-EXPORT-001〜003）のみ提供 |

## 3. 本章の適用ルール

| ルール | 内容 |
|---|---|
| 再定義禁止 | 本章に列挙した除外事項は後続章・サブで再定義・逸脱してはならない |
| スコープ変更手順 | 除外事項を取り込む必要が生じた場合は、architectが本章を改訂し版数を上げる |
| 実装試みの制限 | 除外事項に相当する実装を試みる前に、architectによるスコープレビューを必須とする |
| 除外事項の完全性 | 本章に記載されていない除外事項が発見された場合は、本章への追記を優先する |

---

**本節で確定した方針**

- **OT直結・複数建屋展開・クラウドデプロイ・Kubernetes・自動フェイルオーバーの5領域をD-SYS-001〜005として対象外と確定し、単一建屋・オンプレミス・Docker Compose・手動フェイルオーバーの構成境界を権威的に定める。**
- **PostgreSQL / OS / ミドルウェアの詳細チューニング値（具体的数値）はD-SYS-006として構築フェーズに委任し、本書はサイジング方針と設定方針（Why）の確定にとどまることを明示する。**
- **除外事項はD-SYS識別子で採番し、後続章での再定義を禁止することで設計スコープの安定性と一貫性を保証する。**
