# 04 ネットワーク構成とゾーニング

本章は、IPA 共通フレーム 2013「2.3.5 ネットワーク方式設計タスク」に対応し、単一工場・単一建屋内のネットワークを 4 ゾーンに分割し、ゾーン間の通信許可ルール・Docker 内部ネットワーク分離・Wi-Fi 構成を確定する。NFR-OPS-036〜045 を充足する設計を具体化する。OT ゾーンとの接続禁止（P4 原則・ADR-SYS-004）は本章における最優先制約である。

---

## 1. ゾーニング設計方針

### 1-1. ゾーン分割の基本方針

本システムのネットワークを以下の原則で分割する。

| 原則 | 内容 |
|---|---|
| OT 完全分離 | OT ゾーン（PLC/SCADA/OPC-UA）とのネットワーク接続は永続禁止とする（ADR-SYS-004 / P4 原則）。物理的・論理的に分離する |
| 最小権限 | 各ゾーン間の通信は必要最小限のポート・プロトコルのみ許可する |
| TLS 必須 | すべての HTTP 通信は TLS 1.3 以上を必須とする（社内 LAN 内であっても例外なし）|
| Docker 内部分離 | PostgreSQL（5432）・コンテナ間通信は Docker 内部 bridge ネットワークに限定し、ホスト OS の外部インタフェースには公開しない |
| 外部接続最小化 | インターネット接続は不要。親機（外部システム）との接続のみをファイアウォール許可リスト（IP + ポート）で制御する |

## 2. ネットワークゾーン定義

### 2-1. ゾーン一覧

| ゾーン ID | ゾーン名 | 対象機器・システム | 接続方式 |
|---|---|---|---|
| ZONE-OT | OT ゾーン（永続禁止）| PLC・SCADA・HMI・OPC-UA サーバー | 接続禁止（物理的分離） |
| ZONE-FL | 工場 LAN ゾーン | ハンディ端末（Android/iOS/Windows タブレット）・Wi-Fi AP | Wi-Fi（2.4GHz/5GHz）|
| ZONE-MG | 管理 LAN ゾーン | PC クライアント（マスタメンテナンス APP・管理コンソール）・NODE-001/002/003 | 有線 Gigabit Ethernet・VLAN 分離 |
| ZONE-EX | 外部接続ゾーン | NODE-001 ↔ 親機（外部システム）| TLS 1.3 + ファイアウォール許可リスト |

### 2-2. OT ゾーン（ZONE-OT）— 永続禁止

| 項目 | 内容 |
|---|---|
| 対象 | PLC（Programmable Logic Controller）・SCADA システム・HMI（Human Machine Interface）・OPC-UA サーバー |
| 接続方針 | **永続禁止（ADR-SYS-004 / P4 原則）** |
| 禁止根拠 | OT 機器へのサイバー攻撃・誤操作は生産設備の物理的損傷・人身事故につながる。作業ナビゲーションシステムが OT ゾーンとネットワーク接続することで、本システムが OT への攻撃経路になるリスクを永続的に排除する |
| 物理的担保 | OT ネットワークは本システムと物理的に異なる LAN セグメントに接続する。ルーターおよびファイアウォールで双方向の通信を全ポートでドロップする |
| 将来変更可否 | 本制約は設計変更不可（Permanent / P4 原則）。将来的に OT データを参照する場合は、DMZ 経由のデータダイオード（一方向通信デバイス）を別途設計する |

### 2-3. 工場 LAN ゾーン（ZONE-FL）

| 項目 | 内容 |
|---|---|
| 対象機器 | Android/iOS/Windows タブレット（ハンディ端末）・Wi-Fi AP（NODE-004 × 複数台）|
| 接続方式 | Wi-Fi 802.11ac（5GHz）/ 802.11n（2.4GHz）|
| 認証方式 | WPA3-Enterprise（RADIUS サーバー）を推奨する。WPA2-PSK は工場レベルの意思決定に委ねる（本設計では WPA3-Enterprise を選定根拠として記録し、WPA2-PSK を代替として許容する）|
| VLAN | ZONE-MG と別 VLAN に分離する（NFR-OPS-042）。VLAN タグは L2/L3 スイッチ（NODE-005）で設定する |
| AP 配置 | NFR-OPS-041 に従い複数台の AP を工場フロアに配置する。AP 台数・配置座標は RF サイトサーベイの結果を受けて現地決定する（本設計では範囲外とする）|
| 通信先 | ZONE-MG の NODE-001（IIS: HTTPS 443）のみ許可する。ZONE-OT・ZONE-EX への直接通信は禁止する |

### 2-4. 管理 LAN ゾーン（ZONE-MG）

| 項目 | 内容 |
|---|---|
| 対象機器 | PC クライアント・NODE-001（Active サーバー）・NODE-002（Standby サーバー）・NODE-003（NAS）・NODE-005（L2/L3 スイッチ）|
| 接続方式 | 有線 Gigabit Ethernet（IEEE 802.3ab）|
| VLAN | ZONE-FL とは別 VLAN。OT ゾーン（ZONE-OT）とは物理的に分離 |
| PC ↔ サーバー | HTTPS 443（IIS リバースプロキシ経由）。PC ブラウザから管理コンソール・マスタメンテナンス APP にアクセスする |
| サーバー間 | NODE-001 ↔ NODE-002 の PostgreSQL ストリーミングレプリケーション（TCP 5433: レプリケーション専用ポート）。NODE-001/002 ↔ NODE-003 の WAL アーカイブ転送（SMB/NFS）|
| IT 担当保守 | NODE-001/002 への RDP（Windows Remote Desktop）はこのゾーン内のみ許可する。ポート 3389 は ZONE-FL からは禁止する |

### 2-5. 外部接続ゾーン（ZONE-EX）

| 項目 | 内容 |
|---|---|
| 対象 | NODE-001 ↔ 親機（外部システム）の IF-001/002/003 通信 |
| プロトコル | HTTPS（TLS 1.3）+ mTLS（相互認証：IF-001/002）|
| ファイアウォール制御 | 送信元 IP（親機固定 IP）+ 宛先ポート（HTTPS 443）による許可リスト。それ以外の外部通信は全てドロップ |
| インターネット | 不要（クローズドネットワーク。親機自体が工場内または専用回線上に存在することを前提とする）|
| 外部 IF | IF-001（親機マスタ同期）・IF-002（Outbox 実績送信）・IF-003（LDAP/AD 認証連携）の 3 件のみ |

## 3. Docker 内部ネットワーク設計

### 3-1. Docker ネットワーク分離

Docker Compose 内部のコンテナ間通信は、ホスト OS の外部インタフェースに露出させない。

| Docker ネットワーク名 | 種別 | 参加コンテナ | 目的 |
|---|---|---|---|
| `wnav_backend` | bridge（Docker 内部）| `wnav_api`・`postgres`・`wnav_scheduler` | API → DB・スケジューラ → DB の内部通信 |

**外部露出ルール**:
- `wnav_api` の 8080 ポートは `127.0.0.1:8080:8080` にバインドする（IIS リバースプロキシからのみアクセス可）。
- `postgres` の 5432 ポートは `127.0.0.1:5432:5432` にバインドする（保守目的のみ。本番運用時はコメントアウトを推奨する）。
- `0.0.0.0` バインドは全サービスで禁止する（ADR-SYS-005）。

## 4. ファイアウォールルール表

本表は境界ファイアウォール（ルーター・L3 スイッチ）への設定仕様である。以下に記載されていない通信はデフォルト DROP とする。

### 4-1. インバウンドルール

| ルール ID | 送信元ゾーン | 宛先ゾーン | プロトコル | 宛先ポート | アクション | 根拠 |
|---|---|---|---|---|---|---|
| FW-IN-001 | ZONE-FL | ZONE-MG（NODE-001）| TCP | 443（HTTPS）| ALLOW | ハンディ端末 → IIS API |
| FW-IN-002 | ZONE-MG（PC）| ZONE-MG（NODE-001）| TCP | 443（HTTPS）| ALLOW | PC ブラウザ → IIS Web APP |
| FW-IN-003 | ZONE-MG（NODE-002）| ZONE-MG（NODE-001）| TCP | 5433 | ALLOW | Standby → Primary レプリケーション受信 |
| FW-IN-004 | ZONE-MG（NODE-001/002）| ZONE-MG（NODE-003）| TCP | 445（SMB）または 2049（NFS）| ALLOW | WAL アーカイブ → NAS |
| FW-IN-005 | ZONE-EX（親機固定 IP）| ZONE-MG（NODE-001）| TCP | 443（HTTPS）| ALLOW | 親機 IF-001/002/003 |
| FW-IN-006 | ZONE-MG（IT 担当 PC）| ZONE-MG（NODE-001/002）| TCP | 3389（RDP）| ALLOW | IT 担当 保守用 RDP |
| FW-IN-999 | ANY | ANY | ANY | ANY | DROP | デフォルト拒否 |

### 4-2. アウトバウンドルール

| ルール ID | 送信元ゾーン | 宛先ゾーン | プロトコル | 宛先ポート | アクション | 根拠 |
|---|---|---|---|---|---|---|
| FW-OUT-001 | ZONE-MG（NODE-001）| ZONE-EX（親機固定 IP）| TCP | 443（HTTPS）| ALLOW | IF-001/002/003 の送信側 |
| FW-OUT-002 | ZONE-MG（NODE-001）| ZONE-MG（NODE-003）| TCP | 445 または 2049 | ALLOW | WAL アーカイブ書込み |
| FW-OUT-003 | ZONE-MG（NODE-001）| ZONE-MG（NODE-002）| TCP | 5433 | ALLOW | Primary → Standby WAL 送信 |
| FW-OUT-999 | ANY | ZONE-OT | ANY | ANY | DROP | OT ゾーン接続永続禁止（ADR-SYS-004）|
| FW-OUT-998 | ANY | インターネット | ANY | ANY | DROP | インターネット接続不要 |
| FW-OUT-997 | ANY | ANY | ANY | ANY | DROP | デフォルト拒否 |

### 4-3. ゾーン間通信マトリクス

| 送信元 ↓ / 宛先 → | ZONE-OT | ZONE-FL | ZONE-MG | ZONE-EX | インターネット |
|---|---|---|---|---|---|
| ZONE-OT | — | 禁止 | 禁止 | 禁止 | 禁止 |
| ZONE-FL | 禁止 | AP 間 | HTTPS 443 のみ | 禁止 | 禁止 |
| ZONE-MG | 禁止 | HTTPS 443 のみ | 限定許可（FW-IN/OUT）| HTTPS 443 のみ（親機 IP）| 禁止 |
| ZONE-EX（親機）| 禁止 | 禁止 | HTTPS 443 のみ（NODE-001）| — | 禁止 |

## 5. Wi-Fi 設計概要

### 5-1. Wi-Fi 認証方式

| 認証方式 | 採用可否 | 説明 |
|---|---|---|
| WPA3-Enterprise（RADIUS）| 推奨（第一選択）| 802.1X 認証。端末証明書またはユーザー認証でアクセス制御。RADIUS サーバーの構築が必要 |
| WPA2-PSK | 許容（代替）| RADIUS サーバーを構築しない場合の代替。PSK の定期変更（3 ヶ月以内）を運用ルールとして設定する。工場レベルの意思決定に委ねる |

WPA3-Enterprise を採用する場合、RADIUS サーバーは NODE-001（Windows Server 2022 の NPS: Network Policy Server）で提供する。

### 5-2. Wi-Fi 帯域

| 帯域 | 用途 | 備考 |
|---|---|---|
| 5GHz（802.11ac/ax）| ハンディ端末のデータ通信（主）| 干渉が少なく高スループット。工場内の金属構造物による反射に注意する |
| 2.4GHz（802.11n）| 5GHz カバー外のバックアップ | 到達距離が長いが干渉が多い。BLE スキャナとの干渉管理が必要 |

---

**本節で確定した方針**
- OT ゾーン（ZONE-OT）との物理的・論理的完全分離を ADR-SYS-004（P4 原則）として永続確定し、ファイアウォールでデフォルト DROP としてすべての OT 向け通信を禁止する。
- 4 ゾーン（OT/工場 LAN/管理 LAN/外部接続）と Docker 内部 bridge ネットワーク（`wnav_backend`）の分離を確定し、ファイアウォールルール表（FW-IN-001〜006・FW-OUT-001〜003）を設計として確定する。
- すべての HTTP 通信に TLS 1.3 を必須とし、Docker サービスの `0.0.0.0` バインドを禁止（ADR-SYS-005）・IIS リバースプロキシ経由の外部公開のみを許可する設計を確定する。
