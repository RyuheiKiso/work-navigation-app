# 04 設計トレーサビリティ枠組み

本章の責務は、要件 ID（BR/FR/NFR/UC/EN/IF/RP/SCR）と設計 ID（TBL/API/MOD/SEQ 等）の双方向トレーサビリティの確立方法と、設計トレーサビリティマトリクス（DTM）の構造を確定することである。

---

## 1. トレーサビリティの全体構造

### 1-1. 6 段トレーサビリティチェーン（要件定義フェーズ確定部分）

`03_要件定義/04_要件分類体系とトレーサビリティ枠組み.md` が確定した要件側のトレースチェーンを継承する。

```
Why（経営課題・規制・業界課題）
  → What（BR-BUS / BR-SC）
    → How（FR-XX / NFR-XX）
      → 制約（NFR-AVL/SEC/DQ 等）
        → 受入基準（AC-{要件ID}）
          → 検証（TST-NNN）
```

### 1-2. 概要設計フェーズの拡張（8 段トレーサビリティ）

本書は上記 6 段に設計 ID と追跡 ID を加え、8 段に拡張する。

```
Why → What（BR） → How（FR/NFR）
  → 設計命題（SCR / API / TBL / MOD / SEQ / JOB / SEC）   ← 本書が確定
    → 実装（ファイルパス）                                  ← 05_詳細設計が確定
      → 検証（TST / AC）
```

実装時の双方向性:
- **前方追跡（Forward Tracing）**: BR → FR → API → MOD → ファイル
- **後方追跡（Backward Tracing）**: ファイル → MOD → API → FR → BR
- **影響範囲分析**: BR 変更 → 変更影響 FR/API/TBL/MOD の特定

---

## 2. RTM 本体の拡張（既存 8 列 → 11 列）

### 2-1. 拡張後の列定義

`03_要件定義/付録/01_要件トレーサビリティマトリクス（RTM）.md` を以下のとおり 11 列に拡張する。追加する 3 列（「主 SCR」「主 API」「主 TBL」）は当該要件を最も代表する設計 ID の要約列（1〜3 件）であり、網羅性は後述の DTM-DES が担保する。

| 列 | 内容 | 変更 |
|---|---|---|
| BR-ID | 業務要件 ID | 既存 |
| BR タイトル | 業務要件要約（20 字以内） | 既存 |
| FR-ID | 対応機能要件 ID（複数可、カンマ区切り） | 既存 |
| NFR-ID | 対応非機能要件 ID（複数可） | 既存 |
| TST-ID | 対応テスト要件 ID | 既存 |
| 上流リンク | 構想章・計画章 | 既存 |
| MoSCoW | M / S / C / W | 既存 |
| 状態 | Agreed / Frozen / 廃止 | 既存 |
| **主 SCR** | 主要実現画面（SCR-NNN、最大 3 件） | **追加** |
| **主 API** | 主要実現 API（API-NNN、最大 3 件） | **追加** |
| **主 TBL** | 主要データテーブル（TBL-NNN、最大 3 件） | **追加** |

### 2-2. RTM 本体の更新タイミング

RTM 本体（要件定義/付録/01）の「主 SCR/API/TBL」列は概要設計の各サブが完成した段階で逐次追記する。最終的に付録/02 の DTM と整合させる。

## 3. DTM（設計トレーサビリティマトリクス）の 6 マトリクス構造

`付録/01_設計トレーサビリティマトリクス（DTM）.md` に格納する 6 マトリクスの定義。

### M1: FR × API（多対多）

目的: 各機能要件を実現する REST API エンドポイントの対応を確定する

| 列 | 内容 |
|---|---|
| FR-ID | 機能要件 ID |
| API-ID | 対応エンドポイント ID（複数行で多対多を表現） |
| 関与種別 | 主（そのAPIが当該FRを主に実現）/ 副（関連するが従属） |

完全性条件: Won't 以外の全 86 FR が ≥1 行に出現する

### M2: FR × TBL（多対多 + CRUD）

目的: 各機能要件がどのテーブルを何の操作（CRUD）で使用するかを確定する

| 列 | 内容 |
|---|---|
| FR-ID | 機能要件 ID |
| TBL-ID | 対応テーブル ID |
| CRUD | C（Insert）/ R（Select）/ U（Update: 非Append-onlyテーブルのみ）/ D（物理削除: 原則不可） |

完全性条件: データ操作を含む全 FR が ≥1 行に出現する

### M3: UC × SCR（多対多）

目的: 各ユースケースを実現する画面の組み合わせを確定する

| 列 | 内容 |
|---|---|
| UC-ID | ユースケース ID |
| SCR-ID | 関与画面 ID |
| 関与種別 | 必須経由（ユースケース遂行時に必ず通る）/ 条件経由（分岐で通る場合がある） |

完全性条件: 全 22 UC がいずれかの SCR に紐づく

### M4: UC × SEQ（多対多）

目的: 各ユースケースに対応するシーケンス図（SEQ-NNN）の対応を確定する

| 列 | 内容 |
|---|---|
| UC-ID | ユースケース ID |
| SEQ-ID | 対応シーケンス図 ID |
| 図ファイル名 | `fig_des_seq_{name}.drawio` |

完全性条件: 業務根幹 UC（UC-001/002/009/012/014/016/019/020）は ≥1 SEQ を持つ

### M5: NFR × 設計章（多対多）

目的: 各非機能要件をどの概要設計章が実現するかを確定する

| 列 | 内容 |
|---|---|
| NFR-ID | 非機能要件 ID |
| 設計章 | `04_概要設計/{サブ}/{章番号}` 形式 |
| 実現方法要約 | 設計命題の 1 行要約 |

完全性条件: 全 NFR がいずれかの設計章に出現するか「下流委任」「対象外」が明示される

### M6: BR-BUS × API/ERR（多対多）

目的: 各業務ルールを実装するバリデーション層とエラーコードを確定する

| 列 | 内容 |
|---|---|
| BR-BUS-ID | 業務ルール ID |
| API-ID | ルールを適用する API エンドポイント ID |
| バリデーション層 | アプリ層（Rust サービス層）/ DB 制約（CHECK/TRIGGER）/ RBAC ミドルウェア |
| ERR-ID | 違反時に発生するエラーコード ID |
| 違反時処理 | 拒否（HTTP 4xx）/ 警告ログ / ブロック |

完全性条件: 全 45 BR-BUS が ≥1 行に出現するか「[非実装: 禁止 BR-BUS-029 等]」が明示される

## 4. DTM 完全性の自動検証方針

以下の検証を概要設計フェーズ完了時に実施する。

| 検証 ID | 検証内容 | 判定基準 |
|---|---|---|
| V-01 | M1 の FR 列に全 86 FR が出現する（Won't 除く） | ゼロ件の FR なし |
| V-02 | M2 の FR 列にデータ操作 FR が出現する | 空白 FR なし |
| V-03 | M3 の UC 列に全 22 UC が出現する | ゼロ件の UC なし |
| V-04 | M4 の UC 列に業務根幹 8 UC が出現する | 8 件全て ≥1 SEQ |
| V-05 | M5 の NFR 列に全 NFR が出現する | 「下流委任」または設計章が明示 |
| V-06 | M6 の BR-BUS 列に全 45 BR-BUS が出現する | 「[非実装]」も可 |
| V-07 | 全設計 ID が採番台帳に登録済み | 孤立 ID なし |
| V-08 | RTM 本体の「主 SCR/API/TBL」が DTM のいずれかに出現する | 齟齬なし |
| V-09 | 廃止 ID が DTM に未廃止として出現しない | 廃止 ID には `[廃止]` 付記 |
| V-10 | 全 SEQ-NNN の図ファイルが img/ に存在する | `.drawio` と `.svg` ペアが存在 |

## 5. FR 実装方針カード（86 件）

各 FR に 1 枚の「実装方針カード」を 02_ソフトウェア方式設計/13_実装方針カード集.md に格納する。カードの標準フォーマット：

```markdown
### {FR-ID}（{FR名称}）— 実装方針カード

| 観点 | 内容 |
|---|---|
| 主 API   | API-{resource}-NNN （{HTTP} {URL}） |
| 副 API   | （あれば）|
| 主 TBL   | TBL-NNN {table_name} （{CRUD}） |
| 主 MOD   | MOD-{target}-NNN {module_name} |
| 主 SEQ   | SEQ-NNN |
| 関連 SCR | SCR-{app}-NNN （CMP-{app}-NNN） |
| 業務制約 | BR-BUS-NNN（複数可） |
| エラー   | ERR-{layer}-NNN |
| 設定     | CFG-NNN（あれば） |
| 計測     | MET-NNN（あれば） |
| 受入基準 | AC-{FR-ID} |
```

全 86 件のカードが存在し、主 API が空でないことが概要設計完成条件の一つである（Won't 要件は「[Won't: 理由]」として明示）。

## 6. 禁止実装要件の DTM 管理

BR-BUS-029「個人別ランキング非実装」等の禁止実装要件（Must だが実装してはならない）は、DTM M6 において以下の形式で記録し、削除しない。

```
| BR-BUS-029 | [非実装: 個人別ランキングは Just Culture 原則（構想 04 章）により永続非提供。
               実装を技術的に拒否するアーキテクチャ保護を 07_セキュリティ §08 で確定。] |
```

---

**本節で確定した方針**
- **要件 RTM を 8 列→11 列（主 SCR・主 API・主 TBL 追加）に拡張し、詳細は DTM 6 マトリクス（M1: FR×API、M2: FR×TBL、M3: UC×SCR、M4: UC×SEQ、M5: NFR×設計章、M6: BR-BUS×API/ERR）で管理するハイブリッド方式を採用する。**
- **10 件の完全性検証（V-01〜V-10）を概要設計フェーズ完了の必須条件とし、禁止実装要件も「[非実装]」として DTM に保持し削除しない。**
- **全 86 FR に 1 枚の「実装方針カード」を発行し、設計 ID との双方向リンクを確立する。**
