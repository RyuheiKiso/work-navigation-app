# 99 前提制約と本書が約束しないこと

本章の責務は、テスト方式設計サブ（10）が概要設計として確定した内容の範囲を明示し、本書が意図的に扱わない事項・詳細設計フェーズに委任した事項・対象外と判断した事項を宣言的に記録することである。本章の記述は設計の不完全さを示すものではなく、フェーズ境界と設計判断の根拠を透明に示すものである。

---

## 1. 詳細設計フェーズへの委任事項

本書は概要設計（テストレベル・方式・品質ゲートの骨格）を確定する文書である。以下の事項は概要設計の境界を超えるため、詳細設計フェーズで確定する。

### D-TST-001：詳細テストケース設計

| 属性 | 内容 |
|---|---|
| 委任 ID | D-TST-001 |
| 委任先 | 詳細設計フェーズ（`05_詳細設計/`）|
| 委任内容 | TC-NNN 識別子の採番・各テストケースの入力値・期待値・前提条件・後処理手順の詳細設計 |
| 根拠 | 詳細テストケースは実装コードが確定した後でなければ精確に設計できない |
| 影響範囲 | TST-NNN は概要設計で定義済み。TC-NNN（バリアント）は詳細設計フェーズで採番する |
| ブロッカー | 本委任は ver1.0.0 リリース前の詳細設計フェーズ着手時に解決する |

### D-TST-002：負荷試験の詳細シナリオ

| 属性 | 内容 |
|---|---|
| 委任 ID | D-TST-002 |
| 委任先 | 詳細設計フェーズ（`05_詳細設計/`）|
| 委任内容 | k6 負荷テストの VU ランプアップ曲線・混合シナリオの API 操作重み付け・チェックポイント関数・Soak テストの監視ダッシュボード設定の詳細設計 |
| 根拠 | 負荷テストシナリオは API 実装（エンドポイント数・レスポンスサイズ）が確定した後でなければ現実的な負荷分布を設計できない |
| 影響範囲 | 07 章で確定した合格基準（P95 ≤ 2000ms・50 VU・5xx ゼロ）は変更しない |
| ブロッカー | バックエンド API 実装の概ね完了後に詳細設計を着手する |

## 2. 対象外と判断した事項

以下の事項は、本プロジェクトの規模・体制・リソースを考慮し、対象外と判断する。対象外判断は妥協ではなく、適切なリスク管理の範囲内での設計判断である。

### D-TST-003：第三者機関によるセキュリティ監査

| 属性 | 内容 |
|---|---|
| 対象外 ID | D-TST-003 |
| 対象外の内容 | ペネトレーションテスト・ISMS 認証・外部セキュリティ監査機関によるセキュリティレビュー |
| 対象外の根拠 | 個人開発・単一工場のクローズド OT 環境であり、インターネット非接続（NFR-SEC-031）のため外部攻撃の攻撃面が最小化されている。自己検査（OWASP ZAP・cargo audit・npm audit・RBAC 検証・JWT 偽造テスト）で対応する |
| リスク受容 | 第三者監査を実施しないことのリスクを ADR に記録の上、自己検査ゲートで代替することを受容する |
| 将来の再検討 | 工場外へのデプロイ・クラウド接続・規制要件変更が発生した場合は対象外判断を再評価する |

### D-TST-004：テスト自動化の実装詳細

| 属性 | 内容 |
|---|---|
| 対象外 ID | D-TST-004 |
| 対象外の内容 | CI/CD パイプラインの YAML 詳細実装・Detox の設定ファイル詳細・Playwright の設定ファイル詳細・k6 スクリプトの完全実装 |
| 対象外の根拠 | 本書はテスト方式設計の「概要設計」である。実装コードに相当するテスト自動化スクリプトの詳細は実装フェーズ（`06_実装/`）で確定する |
| 影響範囲 | 02 章で確定したコマンド体系・ツール選定・品質ゲート条件は変更しない |

## 3. 本書が確定した設計命題の有効範囲

本書（10_テスト方式設計）が確定した設計命題の有効範囲を以下に示す。範囲外の事項を本書に基づいて判断することを禁止する。

| 設計命題 | 有効範囲 | 範囲外 |
|---|---|---|
| テストピラミッド 4 レベル（01 章） | レベル定義・ツール選定・カバレッジ目標数値 | 各 TC の入力値・テストコード実装 |
| テスト自動化基盤（02 章） | CI パイプライン構成・テストランナーのコマンド体系 | Dockerfile・YAML 詳細・環境変数設定 |
| テストデータ管理（03 章） | ファクトリパターン方針・シードデータ種別 | ファクトリ関数の実装コード |
| 品質ゲート（04 章） | 数値目標・判定基準・違反時対応の方針 | ゲート判定スクリプトの実装 |
| UAT 方式（05 章） | 参加者・環境・スコープ・合否基準の骨格 | UAT チェックリスト詳細・シナリオ手順書 |
| ALCOA+ 検証（06 章） | 9 原則の検証観点・TST-060〜068 の骨格 | 各テストの実装コード |
| 非機能テスト（07 章） | シナリオ概要・合格基準数値 | k6 スクリプト詳細・シナリオ重み付け |
| RTM/DTM 反映（08 章） | ステータスライフサイクル・docstring 規約 | RTM/DTM の自動更新ツール実装 |

## 4. 本書の暗黙的前提（変更時は ADR に記録）

本書が依拠する暗黙的前提のうち、変更が生じた場合に本書の見直しを要する事項を以下に示す。

| 前提 | 内容 | 変更時の影響 |
|---|---|---|
| PostgreSQL 16 | テスト DB として PostgreSQL 16 を使用 | バージョン変更時は sqlx テストの互換性確認が必要 |
| React Native（Expo） | Detox は Expo + RN 構成を前提 | ネイティブモジュール追加時は Detox の設定見直しが必要 |
| Rust Edition 2024 | cargo test・Clippy は Rust Edition 2024 を前提 | Edition 変更時は Clippy 設定の見直しが必要 |
| GitHub Actions（オンライン） | CI は GitHub Actions を前提としつつローカル実行も可能 | CI 基盤変更時は `scripts/test-all.sh` との同期が必要 |
| 単一工場・社内 LAN 完結 | OWASP ZAP はクローズド環境での実施を前提 | クラウド接続時は外部 ZAP スキャンの検討が必要 |
| 個人開発（Bus Factor = 1） | テスト設計・実行・結果判断は開発者 1 名が担当 | チーム開発に移行時は UAT 参加者・レビュー体制の再設計が必要 |

---

**本節で確定した方針**
- D-TST-001（詳細テストケース設計）および D-TST-002（負荷試験詳細シナリオ）を詳細設計フェーズへ委任することを確定した。委任は品質の妥協ではなく、フェーズ境界に基づく適切な設計判断である。
- D-TST-003（第三者セキュリティ監査）を個人開発・単一工場・インターネット非接続環境という根拠に基づき対象外と判断し、OWASP ZAP・cargo audit・npm audit による自己検査で代替することを確定した。
- 本書が確定した設計命題の有効範囲を 8 章分にわたって明示し、実装詳細との混同を防ぐ境界を設定した。
