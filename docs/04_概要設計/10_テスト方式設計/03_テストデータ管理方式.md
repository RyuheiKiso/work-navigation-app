# 03 テストデータ管理方式

本章の責務は、テストで使用するデータの生成・管理・廃棄・バージョン管理の方式を確定することである。本番データのテスト環境流用は BR-BUS-029（個人別ランキング生成禁止）および NFR-SEC-020（PII 5 分類取扱方針）との整合から一切禁止する。すべてのテストデータは合成データ（ファクトリパターン）で生成し、実在する作業員の個人情報が一切含まれないことを確保する。

---

## 1. テストデータの基本原則

### 1-1. 本番データ不使用原則

| 原則 | 内容 | 根拠 |
|---|---|---|
| 本番データ流用禁止 | テスト環境に本番 DB のデータを直接コピーしない | BR-BUS-029・NFR-SEC-020 |
| PII 合成必須 | 氏名・社員番号・スキル資格情報はすべて架空データを使用 | NFR-SEC-020（PII 5 分類） |
| 実在工場名禁止 | テストデータに実在する工場・企業・製品名を使用しない | 個人情報保護方針 |
| テスト DB 完全分離 | テスト DB（TENV-002）は本番 DB ネットワークから分離する | NFR-SEC-031（インターネット非接続） |

### 1-2. ファクトリパターンの採用

テストデータはファクトリパターンで生成する。フィクスチャ（固定データファイル）は更新コストが高くなりやすいため採用しない。各テストが必要なデータを自己完結で生成することを基本とする。

---

## 2. Rust バックエンドのテストデータファクトリ

### 2-1. `fake` クレートによる合成データ生成

Rust の統合テスト・ユニットテストでは `fake` クレートを使用して合成テストデータを生成する。

```rust
use fake::{Fake, Faker};
use fake::faker::name::ja_jp::*;
use fake::faker::company::en::*;

// テストデータファクトリ例
fn create_test_worker() -> WorkerEntity {
    WorkerEntity {
        worker_id: uuid::Uuid::new_v4(),
        // 架空の氏名（日本語ローカライズ）
        display_name: Name().fake::<String>(),
        // 架空の社員番号（実在番号と衝突しないプレフィックス TST- を付与）
        employee_code: format!("TST-{}", Faker.fake::<u32>()),
        role: WorkerRole::Operator,
        skill_level: 1,
        created_at: chrono::Utc::now(),
    }
}
```

| ファクトリ対象エンティティ | 生成方法 | 架空化の方針 |
|---|---|---|
| Worker（作業員） | `fake::faker::name::ja_jp` | 氏名は日本語架空名・社員番号は `TST-` プレフィックス |
| WorkOrder（作業指示） | Faker struct | 製品番号は `TEST-PROD-NNNN` 形式 |
| WorkEvent（作業イベント） | Faker struct + 固定 UUID | worker_id は ファクトリ生成 Worker の UUID を使用 |
| EvidenceFile（証拠ファイル） | 1x1 PNG バイナリ（固定テスト画像） | 実際の製品写真を使用しない |
| InstrumentCalibration（校正情報） | Faker struct | 校正機関名は架空（`TestLab-001`） |

## 3. TypeScript（React/RN）のテストデータファクトリ

### 3-1. `@faker-js/faker` による合成データ生成

React および React Native のテストでは `@faker-js/faker` を使用して合成テストデータを生成する。

```typescript
import { faker } from '@faker-js/faker/locale/ja';

// テストデータファクトリ例
export function createTestWorker(): Worker {
  return {
    workerId: faker.string.uuid(),
    // 架空の氏名（日本語ロケール）
    displayName: faker.person.fullName(),
    // 架空の社員番号
    employeeCode: `TST-${faker.number.int({ min: 1000, max: 9999 })}`,
    role: 'operator',
    skillLevel: faker.number.int({ min: 1, max: 5 }),
  };
}
```

| ファクトリ対象 | ライブラリ | 架空化の方針 |
|---|---|---|
| ユーザー情報 | `faker.person` | 日本語ロケールの架空氏名 |
| 作業データ | `faker.commerce` / `faker.string` | 架空製品名・架空 QR コード |
| タイムスタンプ | `faker.date` | 2026-01 〜 2026-12 の範囲 |
| バイナリ（証拠ファイル） | `faker.image` またはテスト固定 PNG | 実製品画像を使用しない |

## 4. データベースのテスト分離方式

### 4-1. sqlx テストトランザクション

統合テストは `#[sqlx::test]` マクロを使用したトランザクションベースの分離を採用する。各テストは独立したトランザクション内で実行され、テスト終了後に自動ロールバックされる。これによりテスト間のデータ汚染をゼロにする。

| 分離手法 | 適用レベル | 効果 |
|---|---|---|
| `#[sqlx::test]` トランザクション | L2 統合テスト（Rust） | 各テストがクリーンな DB 状態から開始 |
| Docker Compose `test` プロファイル | L2/L3 | 本番 DB と完全に分離した専用インスタンス |
| `tmpfs` ボリューム | テスト DB | テスト実行後にデータが自動消去 |

### 4-2. テスト実行前後のフック

```bash
# テスト実行前
docker compose --profile test up -d postgres_test
sqlx migrate run --database-url "$TEST_DATABASE_URL"
sqlx database seed   # ロール・権限・基準マスタのシードのみ

# テスト実行後
docker compose --profile test down -v  # ボリューム含めて破棄
```

## 5. リファレンスデータ（シードデータ）の管理

### 5-1. シードデータの種別

テスト実行に必要なリファレンスデータ（マスタデータ）は `db/seeds/test/` 配下の SQL ファイルとして管理する。シードデータは個人情報を含まない参照データのみとする。

| シードデータ種別 | ファイル | 内容 |
|---|---|---|
| ロール定義 | `seeds/test/01_roles.sql` | 6 ロール定義（RBAC テスト用） |
| 標準 SOP テンプレート | `seeds/test/02_sop_templates.sql` | テスト用 SOP 3 種類（架空工程） |
| 計測器校正データ | `seeds/test/03_instruments.sql` | 架空計測器 5 件・校正有効期限あり・期限切れ各 1 件 |
| 設定パラメータ | `seeds/test/04_config.sql` | CFG 識別子に対応するテスト用設定値 |

### 5-2. シードデータのバージョン管理

シードデータは本番マイグレーションファイル（`migrations/`）と同一の git リポジトリで管理する。スキーマ変更時はシードデータも同一コミットで更新する。

| 規則 | 内容 |
|---|---|
| git 管理 | `db/seeds/test/` 配下の全 SQL ファイルを git で追跡する |
| スキーマ連動更新 | マイグレーション追加時は影響するシードデータを同一 PR で更新する |
| 個人情報混入禁止 | CI でシードデータ内の PII パターン（氏名・社員番号）を検出するスクリプトを実行する |

## 6. PII テストデータの合成基準

### 6-1. 合成 PII の識別ルール

テストデータが実在する PII に誤って混入することを防ぐため、以下の識別ルールを適用する。

| PII 種別 | 合成ルール | 例 |
|---|---|---|
| 氏名 | `faker.person.fullName()` で生成（日本語ロケール） | 「田中 テスト太郎」等の架空名 |
| 社員番号 | `TST-` プレフィックス必須 | `TST-00001` |
| UUID（worker_id） | `uuid::Uuid::new_v4()` または `faker.string.uuid()` で毎回生成 | テスト実行ごとに異なる UUID |
| 証拠ファイル | テスト専用 1x1 PNG バイナリ固定使用 | 実製品画像を使用しない |
| タイムスタンプ | テスト用固定日時または Faker 生成（2026 年範囲内） | `2026-01-01T00:00:00Z` 等 |

---

**本節で確定した方針**
- テストデータへの本番データ流用を一切禁止し、Rust は `fake` クレート、TypeScript は `@faker-js/faker` によるファクトリパターンで全テストデータを合成生成することを確定した。
- `#[sqlx::test]` マクロと Docker Compose `test` プロファイルを組み合わせてテスト間のデータ汚染をゼロにする DB 分離方式を確定した。
- シードデータ（ロール・SOP テンプレート・校正データ・設定パラメータ）を `db/seeds/test/` に配置し、スキーマ変更と同一コミットで更新する git 管理方針を確定した。
