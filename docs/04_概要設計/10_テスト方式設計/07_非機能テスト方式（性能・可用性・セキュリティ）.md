# 07 非機能テスト方式（性能・可用性・セキュリティ）

本章の責務は、性能・可用性・セキュリティの非機能テスト方式を概要設計レベルで確定することである。機能テスト（L1〜L4）が「正しく動くこと」を検証するのに対し、本章のテストは「十分な性能で」「障害が起きても継続し」「攻撃に耐えて」動くことを検証する。NFR-PRF（性能）・NFR-AVL（可用性）・NFR-SEC（セキュリティ）の非機能要件が実際に充足されていることを実証する。

---

## 1. 性能テスト方式（NFR-PRF 対応）

### 1-1. 負荷テストツールと実行環境

| 項目 | 内容 |
|---|---|
| ツール | k6（Grafana k6）|
| 実行環境 | TENV-006（性能テスト環境・本番相当スペック）|
| テスト DB | テスト DB（TENV-006）に 1 万件の WorkEvent を事前シードした状態 |
| 実行タイミング | 品質ゲートの一環として ver1.0.0 リリース前に 1 回以上実施 |

### 1-2. 負荷テストシナリオ（概要）

詳細シナリオは詳細設計フェーズで確定する（D-TST-002）。概要設計では以下のシナリオ骨格を確定する。

| シナリオ | 仮想ユーザー数 | 継続時間 | 目標 |
|---|---|---|---|
| 混合読み書き（スパイク） | 50 VU（ランプアップ 5 分） | 10 分 | P95 ≤ 2000ms・5xx ゼロ |
| ソークテスト（長時間） | 10 VU（定常） | 24 時間 | P95 劣化 ≤ 10%・メモリリークなし |

**混合読み書きシナリオの内訳（概要）:**

| API 操作 | 比率 | 対象 API |
|---|---|---|
| 作業実績取得（GET） | 40% | `GET /api/v1/work-execs/{id}` |
| WorkEvent 作成（POST） | 30% | `POST /api/v1/step-events` |
| SOP 一覧取得（GET） | 20% | `GET /api/v1/master/sops` |
| 監査ログ取得（GET） | 10% | `GET /api/v1/reports/audit-log` |

### 1-3. 性能テストの合格基準

| 指標 | 合格基準 | 根拠 |
|---|---|---|
| P95 レイテンシ | ≤ 2000ms | NFR-PRF |
| P99 レイテンシ | ≤ 5000ms | NFR-PRF |
| HTTP 5xx エラーレート | 0%（ゼロ） | NFR-QUA-020 |
| スループット | ≥ 50 req/s（50 VU 定常時） | NFR-PRF |
| ソーク 24h 後の P95 劣化 | ≤ 10% 増加 | NFR-QUA-020 |

## 2. 可用性テスト方式（NFR-AVL 対応）

### 2-1. カオステスト（Manual）

可用性テストは手動カオステストとして実施する。自動化は概要設計フェーズでは確定しない（詳細設計フェーズで検討）。

**カオステスト手順（概要）:**

```
1. TENV-003（E2E 環境）で作業実行 UC-003 をアクティブに開始する
2. Docker コンテナを停止: docker compose stop api-server
3. ハンディ APP が Emergency Mode（オフラインモード）に切り替わることを確認する
4. オフライン状態で 3 ステップ分の WorkEvent を入力する（Outbox に蓄積）
5. サーバーを再起動: docker compose start api-server
6. Outbox イベントが自動同期され、WorkEvent が DB に格納されることを確認する
7. 欠損・重複ゼロを DB クエリで確認する
```

| 検証観点 | 期待結果 | 対応 NFR |
|---|---|---|
| Emergency Mode 切替 | API 不到達 30 秒以内に Emergency Mode が起動 | NFR-QUA-021 |
| Outbox 累積 | オフライン中の WorkEvent が TBL-003（outbox）に蓄積 | FR-SY-005 |
| 自動同期 | サーバー復旧後の自動同期で欠損・重複ゼロ | NFR-QUA-022 |
| RTO 確認 | サーバー再起動から同期完了までの時間を計測 | NFR-AVL-002（RTO 1 時間）|

### 2-2. ネットワーク切断テスト（WiFi 物理切断）

実機テストとして、実際のハンディ端末の WiFi を切断した状態で作業を継続し、再接続後に同期されることを確認する。

| テスト条件 | 内容 |
|---|---|
| 端末 | 本番使用予定の Android / iOS ハンディ端末 |
| 切断方法 | WiFi を物理的にオフにする |
| 確認項目 | 切断中の作業継続・再接続後の同期完了 |
| 実施タイミング | UAT 前・ステージング環境（TENV-008）で実施 |

### 2-3. バックアップ・リカバリテスト

| テスト項目 | 内容 | 合格基準 |
|---|---|---|
| pg_dump リストア | BAT-005 が生成したバックアップファイルから pg_restore を実施 | データ完全性確認・スキーマ一致 |
| WAL PITR | WAL アーカイブから任意時点へのリカバリ | 指定時点のデータが復元される |
| RTO 実測 | リストア開始から業務再開可能になるまでの時間計測 | ≤ 1 時間（NFR-AVL-002）|

## 3. セキュリティテスト方式（NFR-SEC 対応）

### 3-1. OWASP ZAP ベーススキャン

OWASP ZAP ベーススキャンは TENV-007（セキュリティテスト環境）の API サーバーに対して実施する。

```bash
# OWASP ZAP ベーススキャン実行コマンド（概要）
docker run -t ghcr.io/zaproxy/zaproxy:stable zap-baseline.py \
    -t http://api-server:8080 \
    -r zap-report.html \
    -I
```

| スキャン対象 | 内容 |
|---|---|
| Web アプリ（マスタメンテナンス） | 全ページへのクローリング + アクティブスキャン |
| API サーバー（axum） | 全 API エンドポイント |
| 対象外 | ハンディ APP（React Native ネイティブアプリ）|

**合格基準:**

| アラートレベル | 合格基準 |
|---|---|
| HIGH | 0 件 |
| MEDIUM | ≤ 2 件（既知誤検知を除く） |
| LOW / INFORMATIONAL | 件数制限なし |

### 3-2. 依存ライブラリ脆弱性スキャン

| ツール | 対象 | 実行タイミング | 合格基準 |
|---|---|---|---|
| `cargo audit` | Rust 依存ライブラリ | CI 毎回 + 週次 | CVSS HIGH/CRITICAL ゼロ |
| `npm audit` | TypeScript 依存ライブラリ | CI 毎回 + 週次 | CVSS HIGH/CRITICAL ゼロ |

### 3-3. RBAC 検証テスト（手動 + 自動）

各ロールが許可されたリソースのみにアクセスできることを検証する。

| テスト項目 | 検証観点 | 実施方法 |
|---|---|---|
| operator のマスタ編集禁止 | operator ロールで SOP 編集 API を呼び出す | L2 統合テスト |
| it_admin の作業ログ閲覧禁止 | it_admin ロールで WorkEvent 一覧取得を試みる | L2 統合テスト |
| 監督者以外のアンドン応答禁止 | operator が `/api/v1/andon/{id}/resolve` を呼び出す | L2 統合テスト |
| 画面ガード確認 | 各ロールでログイン後、許可外画面への直接 URL アクセス | Playwright E2E |
| BR-BUS-040（スキル不足ブロック） | スキルレベル不足の作業員で上位スキル要件の SOP を開始 | L2 統合テスト |
| BR-BUS-043（IT 担当の作業ログ直接参照禁止） | it_admin でログ API を呼び出す | L2 統合テスト |

### 3-4. JWT 偽造テスト

| テスト項目 | 検証内容 | 期待結果 |
|---|---|---|
| アルゴリズム混乱攻撃（RS256→HS256） | `alg: HS256` に書き換えた JWT で API を呼び出す | HTTP 401 |
| JWT 署名改ざん | 有効な JWT のペイロードを書き換えた後に署名をそのまま使用 | HTTP 401 |
| 期限切れ JWT | `exp` が過去の JWT で API を呼び出す | HTTP 401 |
| ロール昇格 | `role: operator` を `role: quality_admin` に書き換えた JWT | HTTP 401 |

### 3-5. SQL インジェクションテスト

| テスト項目 | 検証内容 | 期待結果 |
|---|---|---|
| API パラメータ注入 | `' OR 1=1 --` をクエリパラメータに注入 | HTTP 422（sqlx コンパイル時クエリにより理論的に安全だが実証テスト）|
| JSON ボディ注入 | JSON フィールドに SQL 文字列を注入 | HTTP 422 |

sqlx の Compile-time Query（`query!` マクロ）はコンパイル時に SQL をバインドするため、理論上 SQL インジェクションは不可能である。ただし「理論上安全」に依存せず、実証テストを実施することを確定する。

## 4. オフライン耐性テスト

### 4-1. テストシナリオ

| ステップ | 操作 | 確認内容 |
|---|---|---|
| 1 | ハンディ端末で UC-003 を開始（ステップ実行開始） | 正常に作業開始 |
| 2 | WiFi を切断（または API サーバーを停止）する | Emergency Mode が 30 秒以内に起動 |
| 3 | オフライン状態で 3 ステップを実行する | WorkEvent が SQLite ローカルキューに蓄積 |
| 4 | WiFi を再接続する（または API サーバーを起動する） | Outbox Consumer が自動同期を開始 |
| 5 | サーバー DB の WorkEvent を確認する | ステップ 3 で実行した 3 件が欠損・重複なく格納 |

### 4-2. 合格基準

| 指標 | 合格基準 | 対応 NFR |
|---|---|---|
| Emergency Mode 起動時間 | ≤ 30 秒（API 不到達検出から） | NFR-QUA-021 |
| オフライン継続時間 | ≥ 30 分の業務継続（ローカルキューへの記録継続） | NFR-QUA-021 |
| 同期後の欠損件数 | 0 件 | NFR-QUA-022 |
| 同期後の重複件数 | 0 件（Idempotency Key による排除） | FR-SY-006 |

---

**本節で確定した方針**
- 性能テストを k6 で実施し、50 同時ユーザー・P95 ≤ 2000ms・5xx ゼロを合格基準として確定した。24 時間ソークテストも実施しメモリリークを検証する。
- 可用性テストを手動カオステスト（Docker コンテナ停止）と WiFi 物理切断テストの組み合わせで実施し、Emergency Mode 起動・Outbox 累積・自動同期の三点を検証することを確定した。
- セキュリティテストを OWASP ZAP（HIGH ゼロ）・cargo audit/npm audit（CVSS HIGH/CRITICAL ゼロ）・RBAC 検証・JWT 偽造・SQL インジェクション実証テストの五層構造で実施することを確定した。
