# 01 テストレベルとピラミッド戦略

本章の責務は、本プロジェクトのテストピラミッド戦略を 4 レベルで確定し、各レベルのスコープ・ツール・カバレッジ目標・対象 FR/UC を設計命題として定めることである。上流の TST-020〜022（テスト・受入要件）および NFR-QUA-001/002/034 を継承し、概要設計レベルの具体的方式を確立する。個人開発・単一工場という制約下で実現可能かつ妥協ゼロのテスト方式を確定する。

---

## 1. テストピラミッドの方針

本プロジェクトのテスト戦略はテストピラミッドを採用する。ピラミッドの実行比率はユニットテスト（多数・高速） > 統合テスト > E2E テスト > 契約テストの順とし、E2E テストの過剰実装による個人開発のボトルネック化を防ぐ。マニュアルテストは UAT（受入テスト）のみとし、そのほかの検証はすべて自動化する。

図: fig_des_tst_pyramid（img/ 配下）を参照

| レベル | テスト種別 | 識別子 | 主ツール（Rust Backend） | 主ツール（React/RN） |
|---|---|---|---|---|
| L1 | ユニットテスト | TLVL-001 | `cargo test`（`#[cfg(test)]`） | Jest + React Testing Library |
| L2 | 統合テスト | TLVL-002 | sqlx test トランザクション + テスト DB | Jest + React Testing Library（API モック） |
| L3 | E2E テスト | TLVL-003 | — | Playwright（Web）/ Detox（RN） |
| L4 | 契約テスト | TLVL-004 | Schemathesis（OpenAPI 3.1 準拠） | — |

---

## 2. L1 ユニットテスト（TLVL-001）

### 2-1. スコープと対象

ユニットテストはドメインロジック・ユーティリティ・エラーハンドリングを対象とする。外部依存（DB・ネットワーク・ファイルシステム）は `#[cfg(test)]` 配下のモックトレイト実装で置き換える。

| テスト観点 | 対象モジュール | 関連 FR/NFR |
|---|---|---|
| ドメインロジック検証 | ドメインサービス全件 | FR-NV/EV/ST/MA/SY/KZ/AU 全件 |
| ERR ハンドリング | ERR-BIZ-001〜006 / ERR-VAL-001〜010 | BR-BUS-001〜045 |
| UCUM 単位換算 | MOD-BE-010（UcumConverter） | NFR-QUA-013 |
| JSON Logic 評価 | MOD-BE-011（JsonLogicEvaluator） | FR-NV-005 |
| SHA-256 ハッシュ計算 | MOD-BE-007（HashChainService） | NFR-DQ-007 |
| RBAC 権限判定ロジック | MOD-BE-003（RbacMiddleware） | FR-AU-002 |

### 2-2. カバレッジ目標

| 指標 | 目標値 | 根拠 |
|---|---|---|
| 行カバレッジ（ライン） | ≥ 80% | NFR-QUA-034・TST-020 |
| ドメイン層カバレッジ | ≥ 90% | ドメインロジックの誤りが最大リスク |
| ERR ハンドリング網羅 | 100%（全 ERR-{layer} コードのトリガ確認） | M6 リリース Priority 1 欠陥ゼロ条件 |

### 2-3. 実行条件

- 実行タイミング: コード変更のたびに `cargo test --all` を実行する
- ブロック条件: テスト失敗時はマージをブロックする（02 章・品質ゲート参照）
- React/RN: Jest でコンポーネント単体テストを実施し、`@testing-library/react-native` でハンディ APP コンポーネントを検証する

---

## 3. L2 統合テスト（TLVL-002）

### 3-1. スコープと対象

統合テストは API エンドポイント・DB 操作・Outbox パターンの結合を確認する。テスト用 PostgreSQL（Docker Compose テストプロファイル）に対し、sqlx の `#[sqlx::test]` マクロを用いたトランザクションベースのテストを実施する。各テストはトランザクションロールバックで独立性を保証する。

| テスト観点 | 対象 | 関連 FR/NFR/BR-BUS |
|---|---|---|
| API エンドポイント全件 | API-{resource}-NNN 全エンドポイント | FR 全 86 件 |
| Append-only 制約 | WorkEvent・EvidenceFile への UPDATE/DELETE 拒否 | NFR-DQ-004・BR-BUS-015 |
| ハッシュチェーン連結 | prev_hash が直前レコードの hash_current と一致 | NFR-DQ-007・FR-AU-006 |
| Outbox パターン | Outbox レコード生成 → 配送 → 冪等性キー検証 | FR-SY-005/006 |
| RBAC エンドポイント制御 | 各ロールが許可/禁止リソースに正しくアクセスされること | NFR-SEC-010・BR-BUS-040〜045 |
| BR-BUS 全 45 件 | 各業務ルール違反時の ERR コード返却 | BR-BUS-001〜045 |

### 3-2. カバレッジ目標

| 指標 | 目標値 | 根拠 |
|---|---|---|
| API エンドポイント統合テスト率 | ≥ 60%（全 API-{resource}-NNN エンドポイント件数の 60% 以上） | 04 章品質ゲート |
| BR-BUS 統合テスト率 | 100%（全 45 件が ≥1 テストでカバー） | DTM M6 完全性条件 V-06 |
| Append-only 制約テスト | 100%（WorkEvent/EvidenceFile の UPDATE/DELETE が 403 を返すことを確認） | NFR-DQ-004 |

## 5. L4 契約テスト（TLVL-004）

### 5-1. スコープと対象

契約テストは REST API が OpenAPI 3.1 仕様書と完全に整合することを Schemathesis による自動生成テストケースで検証する。API の全エンドポイントに対してランダム入力を生成し、スキーマ違反・未定義エラーコードの漏れを検出する。

| テスト観点 | 対象 | 関連 NFR |
|---|---|---|
| OpenAPI スキーマ準拠 | 全 API-{resource}-NNN エンドポイント | NFR-QUA-032（OpenAPI 3.1 必須） |
| レスポンスボディ型検証 | 全エンドポイントのレスポンス JSON スキーマ | NFR-QUA-032 |
| エラーレスポンス形式 | ERR-{layer}-NNN 形式の統一 | 02_ソフトウェア方式設計/07 章 |

### 5-2. カバレッジ目標

| 指標 | 目標値 | 根拠 |
|---|---|---|
| OpenAPI スキーマ準拠率 | 100%（全エンドポイント） | NFR-QUA-032 |

---

**本節で確定した方針**
- テストピラミッドを L1（ユニット）/L2（統合）/L3（E2E）/L4（契約）の 4 レベルで構成し、それぞれのスコープ・ツール・カバレッジ目標を確定した。
- UC 22 件全件に対して E2E ゴールデンパステストを最低 22 件実施することを確定し、Playwright（Web）と Detox（RN）の二枚使い方針を確定した。
- ユニットテスト行カバレッジ ≥ 80%、BR-BUS 統合テスト 100% 網羅、OpenAPI 準拠率 100% を本フェーズの設計命題として確定した。
