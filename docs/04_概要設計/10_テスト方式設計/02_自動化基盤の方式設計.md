# 02 自動化基盤の方式設計

本章の責務は、テスト自動化を支える CI パイプライン・実行環境・テストランナー・静的解析・品質ゲート通過条件の設計命題を確定することである。個人開発かつオフライン開発が発生しうる環境（社内 LAN・インターネット非接続）を前提とし、GitHub Actions（オンライン時）とローカル実行スクリプト（オフライン時）の両方で同一品質ゲートを通過できる方式を確定する。

---

## 1. CI/CD パイプラインの方式

### 1-1. 実行環境

| 環境 | 実行基盤 | 主な用途 |
|---|---|---|
| オンライン開発時 | GitHub Actions（ubuntu-latest） | プッシュ・PR 時の自動テスト |
| オフライン開発時 | ローカル実行スクリプト（`scripts/test-all.sh`） | 社内 LAN 完結・インターネット不要 |

両環境で同一のテストスイートが実行できることを設計要件とする。GitHub Actions と `scripts/test-all.sh` は同一コマンドシーケンスで実装する。

### 1-2. テスト環境識別子（TENV）

本サブで採番するテスト環境識別子を以下に確定する。

| TENV-ID | 環境名 | 説明 | Docker Compose プロファイル |
|---|---|---|---|
| TENV-001 | ユニットテスト環境 | 外部依存なし・モック使用 | 不要 |
| TENV-002 | 統合テスト環境 | PostgreSQL テスト DB + Docker Compose | `test` |
| TENV-003 | E2E テスト環境（Web） | ブラウザ + API サーバー + PostgreSQL | `e2e` |
| TENV-004 | E2E テスト環境（RN） | Detox + Android Emulator / iOS Simulator | `e2e` + エミュレータ |
| TENV-005 | 契約テスト環境 | API サーバー + PostgreSQL | `e2e` |
| TENV-006 | 性能テスト環境 | k6 + API サーバー + PostgreSQL（本番相当スペック） | `perf` |
| TENV-007 | セキュリティテスト環境 | OWASP ZAP + API サーバー | `security` |
| TENV-008 | ステージング環境（UAT） | 本番同等スペック・IIS/Windows Server 2022 または Docker | 本番同等構成 |

## 2. Rust バックエンドのテスト自動化

### 2-1. テストコマンド体系

```bash
# L1 ユニットテスト + L2 統合テスト
cargo test --all

# L1 ユニットテストのみ（統合テストを除外）
cargo test --all --lib

# L4 カバレッジ計測（llvm-cov 使用）
cargo llvm-cov --all --lcov --output-path lcov.info

# コードリント（Clippy）
cargo clippy --all-targets --all-features -- -D warnings

# セキュリティ監査
cargo audit
```

### 2-2. 統合テスト用 DB 分離方式

統合テストは `#[sqlx::test]` マクロを使用する。このマクロはテストごとに独立したデータベーストランザクションを作成し、テスト終了時に自動ロールバックする。各テスト実行は完全に独立したクリーンな状態で開始される。

```rust
// 統合テスト例
#[sqlx::test(migrations = "./migrations")]
async fn test_work_event_append_only(pool: PgPool) -> sqlx::Result<()> {
    // テスト用データをセットアップ（全体はトランザクション内）
    // テスト終了後に自動ロールバック
    Ok(())
}
```

テスト用 PostgreSQL インスタンスは Docker Compose テストプロファイル（TENV-002）で起動する。本番 DB とは完全に分離した専用インスタンスを使用し、テスト実行後にコンテナを破棄する。

### 2-3. Clippy 設定（ゼロ警告ポリシー）

Rust コードのリントは `cargo clippy --all-targets --all-features -- -D warnings` で実施し、警告をエラーとして扱う。Clippy の警告ゼロを品質ゲートの通過条件とする（04 章参照）。

## 3. React Native（ハンディ APP）のテスト自動化

### 3-1. テストコマンド体系

```bash
# L1 ユニットテスト（Jest）
npx jest --coverage

# L3 E2E テスト（Detox）
npx detox test --configuration android.emu.debug
npx detox test --configuration ios.sim.debug

# リント（ESLint）
npx eslint src/ --max-warnings 0

# 型チェック
npx tsc --noEmit
```

### 3-2. Detox 設定方針

Detox は React Native アプリのグレーボックス E2E テストフレームワークである。実機相当のエミュレータ/シミュレータ上で UC ゴールデンパスを実行し、アプリの動作を検証する。

| 設定項目 | 方針 |
|---|---|
| Android | Android Emulator（API 33 相当）を使用 |
| iOS | iOS Simulator（iOS 17 相当）を使用 |
| Windows | react-native-windows の E2E テストは詳細設計フェーズで確定（D-TST-001） |
| テストデータ | 各テスト実行前に API モックまたはテスト DB をリセットする |

## 4. React（Web マスタメンテナンス）のテスト自動化

### 4-1. テストコマンド体系

```bash
# L1 ユニットテスト（Jest）
npx jest --coverage

# L3 E2E テスト（Playwright）
npx playwright test

# リント（ESLint）
npx eslint src/ --max-warnings 0

# 型チェック
npx tsc --noEmit
```

### 4-2. Playwright 設定方針

Playwright は Web アプリ（マスタメンテナンス React）の E2E テストに使用する。Chromium をデフォルトブラウザとし、Firefox も合わせて実行する。IE11 対応は要件外であるため対象外と判断する。

---

## 5. データベーステスト環境の分離

### 5-1. Docker Compose テストプロファイル

統合テスト・E2E テストで使用するデータベースは Docker Compose の `test` / `e2e` プロファイルで起動する専用インスタンスを使用する。

```yaml
# docker-compose.test.yml（抜粋）
services:
  postgres_test:
    image: postgres:16-alpine
    environment:
      POSTGRES_DB: wnavdb_test
      POSTGRES_USER: wnav_test
      POSTGRES_PASSWORD: test_secret
    ports:
      - "5433:5432"
    tmpfs:
      - /var/lib/postgresql/data   # テスト後に破棄（永続化なし）
```

- テスト実行ごとに `sqlx migrate run` でマイグレーションを適用する
- テスト実行後に `docker compose --profile test down -v` でコンテナとボリュームを破棄する

### 5-2. テスト DB のマイグレーション管理

テスト DB のスキーマは本番と同一のマイグレーションファイル（`migrations/` 配下）を使用する。テスト DB 専用の追加マイグレーションは禁止する。

## 6. GitHub Actions パイプライン設計

### 6-1. ジョブ構成

```yaml
# .github/workflows/ci.yml（概要）
jobs:
  rust-unit:       # cargo test --all --lib + cargo clippy + cargo audit
  rust-integration: # docker compose up postgres_test + cargo test --all
  react-unit:      # jest --coverage（Web + RN）
  react-e2e-web:   # playwright test
  react-e2e-rn:    # detox test
  contract:        # schemathesis run
  coverage-gate:   # lcov カバレッジチェック（≥80%）
```

### 6-2. 品質ゲート通過条件（概要）

品質ゲートの詳細は 04 章で確定する。本章では CI パイプラインが以下のすべてをパスした場合にのみマージを許可することを確定する。

| ゲート | 条件 |
|---|---|
| cargo test | 全テスト PASS |
| cargo clippy | 警告ゼロ（-D warnings） |
| cargo audit | HIGH/CRITICAL CVSS 脆弱性ゼロ |
| ESLint | エラーゼロ（--max-warnings 0） |
| tsc | 型エラーゼロ |
| jest（ユニット） | 全テスト PASS + 行カバレッジ ≥ 80% |
| Playwright / Detox（E2E） | 全 UC ゴールデンパス PASS |
| Schemathesis（契約） | OpenAPI 準拠率 100% |

## 7. オフライン固有テスト

### 7-1. Emergency Mode（WiFi 切断）テスト

社内 LAN が不到達になった場合に Emergency Mode が正しく起動することを検証する統合テストを実施する。

| テストシナリオ | 検証内容 | 関連 FR/NFR |
|---|---|---|
| WiFi 切断シミュレーション | Outbox への累積記録が継続される | FR-SY-008/009・NFR-QUA-021 |
| サーバー復旧シミュレーション | Outbox イベントが自動同期され欠損・重複ゼロ | NFR-QUA-022 |
| クロックスキュー注入 | 5 秒超過のタイムスタンプを持つイベントが ERR-VAL-010 で拒否される | NFR-DQ-003・ERR-VAL-010 |

オフライン固有テストは TENV-002（統合テスト環境）内でネットワーク疎通をモックで遮断することで再現する。実機を用いた WiFi 物理切断テストは 07 章（可用性テスト）で実施する。

---

**本節で確定した方針**
- GitHub Actions（オンライン時）とローカル実行スクリプト（オフライン時）の両方で同一品質ゲートを通過できるテスト自動化基盤方式を確定した。
- 統合テスト DB 分離方式として `#[sqlx::test]` マクロによるトランザクションロールバックと Docker Compose テストプロファイルを確定し、テスト間の状態汚染をゼロにする。
- Rust（cargo test/clippy/audit）、React（Jest/ESLint/Playwright）、React Native（Jest/Detox）の各テストランナーと実行コマンドを設計命題として確定した。
