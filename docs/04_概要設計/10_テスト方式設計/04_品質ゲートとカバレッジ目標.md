# 04 品質ゲートとカバレッジ目標

本章の責務は、ver1.0.0 リリース前に通過しなければならない品質ゲートの条件・カバレッジ目標の数値・測定方法・違反時の対応を設計命題として確定することである。NFR-QUA-020（成熟性・Priority 1 既知欠陥ゼロ）および NFR-QUA-001（機能完全性・E2E 全 UC 通過）を受入基準の核とし、測定可能な数値目標を設定する。妥協は禁止する。

---

## 1. 品質ゲートの全体構成

品質ゲートは「コード品質」「テストカバレッジ」「セキュリティ」「API 準拠」「性能」「DTM 完全性」の 6 領域で構成する。いずれか 1 つでも未達の場合はリリースをブロックする。

図: fig_des_tst_quality_gate（img/ 配下）を参照

| ゲート領域 | 判定基準 | 測定ツール | 担当章 |
|---|---|---|---|
| コード品質（Lint） | Rust: Clippy 警告ゼロ / TypeScript: ESLint エラーゼロ | cargo clippy / ESLint | 本章 2 節 |
| テストカバレッジ | ユニット ≥ 80% / 統合 ≥ 60% API / E2E ≥ 95% UC | llvm-cov / jest-coverage | 本章 3 節 |
| セキュリティ | CVSS HIGH/CRITICAL ゼロ / OWASP ZAP HIGH ゼロ | cargo audit / npm audit / OWASP ZAP | 本章 4 節 |
| API 準拠 | OpenAPI スキーマ準拠率 100% | Schemathesis | 本章 5 節 |
| 性能 | P95 ≤ 2000ms（50 同時ユーザー）・5xx ゼロ | k6 | 本章 6 節 |
| DTM 完全性 | M5: 全 NFR が検証章に対応 / M6: 全 BR-BUS が統合テストに対応 | 手動確認スクリプト | 本章 7 節 |

## 4. セキュリティゲート

### 4-1. 依存ライブラリの脆弱性

| 指標 | 目標値 | ツール | 違反時対応 |
|---|---|---|---|
| CVSS HIGH（7.0〜8.9）件数 | 0 件 | cargo audit / npm audit | CI ブロック・72 時間以内にパッチ適用または除外申請 |
| CVSS CRITICAL（9.0〜10.0）件数 | 0 件 | cargo audit / npm audit | CI ブロック・即時パッチ適用必須 |
| CVSS MEDIUM（4.0〜6.9）件数 | 累積 ≤ 3 件（30 日以内に対処） | cargo audit / npm audit | 警告のみ（CI ブロックなし）・ADR に記録 |

### 4-2. OWASP ZAP ベーススキャン

| 指標 | 目標値 | 根拠 |
|---|---|---|
| HIGH アラート数 | 0 件 | NFR-SEC-052（OWASP Top 10 対策） |
| MEDIUM アラート数 | ≤ 2 件（既知誤検知を除く） | 自動スキャンの誤検知を考慮 |

OWASP ZAP ベーススキャンは TENV-007（セキュリティテスト環境）で実施し、API サーバーの全エンドポイントを対象とする。スキャン結果は `reports/zap-baseline-YYYYMMDD.html` に保存し、git で管理する。

## 5. API 準拠ゲート

| 指標 | 目標値 | ツール | 根拠 |
|---|---|---|---|
| OpenAPI スキーマ準拠率 | 100%（全エンドポイント） | Schemathesis | NFR-QUA-032 |
| レスポンスボディスキーマ適合率 | 100% | Schemathesis | NFR-QUA-032 |
| 未定義ステータスコード発生率 | 0 件 | Schemathesis | ERR 体系の完全性 |

Schemathesis は OpenAPI 3.1 仕様書から自動的にテストケースを生成し、API サーバーに対して実行する。仕様書と実装の乖離を継続的に検出する。

## 6. 性能ゲート

### 6-1. k6 負荷テスト目標

| 指標 | 目標値 | テスト条件 | 根拠 |
|---|---|---|---|
| P95 レイテンシ（読み取り系） | ≤ 2000ms | 50 同時ユーザー・混合読み書き | NFR-PRF（性能要件） |
| P99 レイテンシ（読み取り系） | ≤ 5000ms | 同上 | NFR-PRF |
| 5xx エラーレート | 0% | 50 同時ユーザー | NFR-QUA-020 |
| 4xx エラーレート（不正入力除く） | ≤ 0.1% | 50 同時ユーザー | NFR-QUA-020 |

k6 負荷テストの詳細シナリオ（VU ランプアップ曲線・混合シナリオの重み付け・チェックポイント設定）は詳細設計フェーズで確定する（D-TST-002）。

### 6-2. ソークテスト目標

| 指標 | 目標値 | テスト条件 |
|---|---|---|
| P95 レイテンシの劣化 | テスト開始から 24 時間後に ≤ 10% 増加 | 10 同時ユーザー・24 時間継続 |
| メモリ使用量の増加 | 24 時間で ≤ 10% 増加（メモリリークなし） | 10 同時ユーザー・24 時間継続 |

## 7. DTM 完全性ゲート

### 7-1. DTM M5 完全性（NFR × 設計章）

全 NFR が DTM M5 においていずれかの設計章に対応づけられていること。「下流委任」または「対象外と判断する」が明示されていれば可とする。

| 検証条件 | 目標値 | 検証方法 |
|---|---|---|
| M5 未対応 NFR 件数 | 0 件 | `scripts/check-dtm-m5.sh`（手動実行） |

本サブ（10_テスト方式設計）は NFR-QUA・NFR-DQ・NFR-PRF・NFR-SEC（一部）・NFR-AVL（一部）を検証担当章として DTM M5 に登録する義務を持つ。

### 7-2. DTM M6 完全性（BR-BUS × API/ERR）

全 45 BR-BUS が DTM M6 において少なくとも 1 件の統合テストでカバーされること。

| 検証条件 | 目標値 | 検証方法 |
|---|---|---|
| 統合テストゼロの BR-BUS 件数 | 0 件 | `scripts/check-dtm-m6.sh`（手動実行） |

---

**本節で確定した方針**
- コード品質（Clippy 警告ゼロ / ESLint エラーゼロ）・テストカバレッジ（ユニット ≥ 80% / 統合 BR-BUS 100% / E2E ≥ 95% UC）・セキュリティ（CVSS HIGH/CRITICAL ゼロ）・API 準拠（Schemathesis 100%）・性能（P95 ≤ 2000ms）・DTM 完全性（M5/M6 未対応ゼロ）の 6 領域ゲートすべての通過を ver1.0.0 リリース条件として確定した。
- BR-BUS 全 45 件の統合テストカバレッジ 100% を確定し、DTM M6 完全性条件（V-06）との整合を明示した。
- k6 負荷テストの詳細シナリオは D-TST-002 として詳細設計フェーズに委任することを確定した。
