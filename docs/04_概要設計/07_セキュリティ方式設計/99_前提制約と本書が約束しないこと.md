# 99 前提制約と本書が約束しないこと

本章の責務は、07_セキュリティ方式設計サブが設計レベルで確定した事項と、意図的に対象外とした事項を明確に区別し、読者が本ドキュメントのスコープを誤解しないよう宣言することである。「対象外」は設計の漏れではなく、根拠に基づいた意思決定の結果である。

---

## 2. 対象外と判断した事項

### D-SEC-001: 外部セキュリティ監査（ペネトレーションテスト）の実施

**対象外とする。詳細設計フェーズ以降で計画する。**

- 根拠: 概要設計段階での実施は実装前であり、ペネトレーションテストの有効性が限定される。実装完了後（詳細設計〜実装フェーズ）に実施することが技術的に適切である。
- 将来対応: ver1.0.0 リリース後、外部セキュリティコンサルタントによるペネトレーションテストの実施を計画する。範囲は Web アプリ（OWASP Top 10 対応）・REST API・認証フローを含む。
- 本書との関係: 本書の脅威モデル（01 章）・RBAC（02 章）・認証方式（03 章）はペネトレーションテストの scope 定義の基礎資料となる。

---

### D-SEC-002: FIPS 140-2/3 認定モジュールの使用

**対象外と判断する。本システムは単一工場の内部システムであり、FIPS 認定要件は適用されない。**

- 根拠: FIPS 140-2/3 は米国政府機関の調達要件であり、国内製造業向けの社内システムには法的な適用義務がない。採用した暗号アルゴリズム（AES-256-GCM / TLS 1.3 / RSA-4096 / SHA-256）はいずれも NIST 推奨アルゴリズムであり、認定モジュール非使用であっても同等の暗号強度を持つ。
- 将来対応: 顧客が防衛関連・政府調達に関与する場合、FIPS 認定モジュール（OpenSSL FIPS Provider 等）への切替を別途計画する。

---

### D-SEC-003: 生体認証・顔認証

**対象外と判断する（BR-BUS-028 に基づく）。**

- 根拠: BR-BUS-028（用途三限定）の下位規定として、本システムは生体情報（指紋・顔・虹彩）の収集・保存・処理を行わないことを設計上の原則とする。生体認証は高い本人確認強度を持つが、生体データの漏洩は不可逆であり、製造現場の作業者に対して収集することへの同意取得・GDPR/個人情報保護法への対応が必要となる。これらはシステムの設計複雑性と目的の均衡を大きく超える。
- 代替手段: PIN 認証（BR-BUS-036: 6 桁以上）+ 電子サイン（FR-ES-001）の組み合わせで十分な本人確認強度を確保する。

---

### D-SEC-004: IDS/IPS・SIEM 外部連携

**対象外と判断する。単一建屋内のクローズドネットワークを前提とし、外部 SOC との連携は要件に含まれない。**

- 根拠: 09_インシデント対応の技術的フック §5（外部 SIEM・SOC 非連携の設計根拠）に詳述した通り、本システムの脅威モデルは工場 LAN 内部の限られたアクターを対象とし、外部からの高度持続的脅威（APT）は工場のファイアウォール・物理セキュリティが主防衛線となる。
- 将来対応: マルチサイト展開・ISO/IEC 27001 認証取得を検討する段階で IDS/IPS・SIEM 連携を評価する。
- 代替手段: TBL-032 Append-only 監査ログ + SHA-256 ハッシュチェーン + BAT-003 週次整合性検証バッチにより、内部完結での改ざん検知機能を提供する。

---

### D-SEC-005: OT 直結（PLC/SCADA/OPC-UA）

**対象外と判断する（P4 原則）。**

- 根拠: 本システムのアーキテクチャ原則 P4（OT 直結禁止）に基づき、PLC・SCADA・OPC-UA との直接インターフェースは実装しない。作業ナビゲーションは「人間の作業者への SOP 表示」であり、機器への自動制御コマンド送信は用途外かつ安全性リスクが高い。
- IF リスト: IF-001〜007 はすべて人間のオペレーターを介したデータ交換であり、PLC/SCADA への直接 TCP/IP 接続インターフェースは含まれない。
- 将来対応: スマートファクトリー拡張（ver2.0.0 以降）において OPC-UA ゲートウェイを介した読み取り専用データ取得を別途評価する。

## 3. 本書が前提とする制約

本セキュリティ方式設計全体を通じて、以下の前提条件が成立することを要件とする。これらの前提が崩れた場合、本設計の見直しが必要となる。

| 前提 | 内容 | 崩れた場合のリスク |
|---|---|---|
| 単一工場・単一建屋 | すべての端末・サーバーが同一 LAN セグメントに存在する | マルチサイト対応が必要となり、VPN / mTLS 設計の全面見直しが必要 |
| IT 担当 1 名 | 鍵管理・インシデント対応・パッチ適用を 1 名が実施 | 担当者不在時の代替手順が必要（詳細設計フェーズで対応計画を策定）|
| インターネット非接続 | 工場 LAN はインターネットに直接接続されない | クラウド連携・外部サービス利用時に本設計の通信暗号化方針の再評価が必要 |
| 作業者端末は MDM 管理下 | Android / iOS 端末は MDM で管理され、ルート化・脱獄は防止されている | 端末の SQLCipher 保護が MDM 前提であるため、MDM 非管理端末は接続禁止とする |
| PostgreSQL 16 以降 | データベースは PostgreSQL 16 以降を使用する | 15 以前では TBL 設計・パーティショニング方式の再確認が必要 |

## 4. 詳細設計フェーズへの委任事項

本概要設計段階では設計レベルの方針のみを確定し、以下の事項は詳細設計フェーズで具体化する。

| 委任事項 | 委任先フェーズ |
|---|---|
| axum ミドルウェアの具体的な Rust 型定義・エラー処理実装 | 05_詳細設計 |
| SQLCipher パスフレーズ導出の Rust / TypeScript 実装コード | 05_詳細設計 |
| ETHICS-001〜004 カスタム clippy / ESLint ルールの実装 | 05_詳細設計 |
| BAT-003 / BAT-004 / BAT-005 / BAT-006 の具体的な実装 | 05_詳細設計 |
| キーボルトのディレクトリ構造・マスターキー管理手順書 | 06_実装・運用手順書 |
| ペネトレーションテスト計画・実施 | リリース後（ver1.0.0 以降）|
| WORM NAS / LTO テープによる 7 年アーカイブ選定 | 05_詳細設計（インフラ調達含む）|
| P1 インシデント時の Blue/Green デプロイ検討 | ver1.1.0 以降 |

---

**本節で確定した方針**
- **外部ペネトレーションテスト（D-SEC-001）は実装完了後に計画し、FIPS 140-2/3（D-SEC-002）および生体認証（D-SEC-003）は本システムの規模・用途・BR-BUS-028 の観点から意図的に対象外とする。この判断は設計の漏れではなく根拠に基づく意思決定である。**
- **IDS/IPS・SIEM 外部連携（D-SEC-004）および OT 直結（D-SEC-005）は OT 隔離原則と P4 原則に基づき対象外とし、代替手段として TBL-032 Append-only 監査ログ + ハッシュチェーン + 週次整合性バッチの内部完結型セキュリティ監視を採用する。**
- **本設計の前提条件（単一工場・IT 担当 1 名・インターネット非接続・MDM 管理端末）が変化した場合は設計の全面見直しが必要であることを明示し、将来の拡張局面での設計継続性を担保する。**
