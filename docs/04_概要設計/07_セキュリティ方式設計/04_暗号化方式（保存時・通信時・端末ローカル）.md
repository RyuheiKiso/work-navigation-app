# 04 暗号化方式（保存時・通信時・端末ローカル）

本章の責務は、本システムが取り扱うデータを「通信経路上」「サーバーサイドストレージ上」「端末ローカルストレージ上」の三文脈で暗号化する方式を確定することである。NFR-SEC-003（通信暗号化）・NFR-SEC-004（保存時暗号化）・NFR-SEC-005（端末暗号化）を受けて、アルゴリズム選定・鍵管理方針（KEY 識別子との紐付け）・実装上の判断基準を設計レベルで固定する。

---

## 1. 暗号化三文脈の適用範囲

| 文脈 | 対象 | 主要規格 | KEY 参照 |
|---|---|---|---|
| 通信時暗号化 | 全外部通信経路（ハンディ APP ↔ バックエンド / マスタメンテナンス ↔ バックエンド / 親システム IF-001〜003） | TLS 1.3 | KEY-005 |
| 保存時暗号化 | PostgreSQL ボリューム / バックアップファイル / エビデンス写真・PDF | AES-256-GCM / BitLocker / dm-crypt | KEY-004、KEY-006 |
| 端末ローカル暗号化 | SQLCipher 管理の SQLite DB（ハンディ APP） | AES-256-CBC（SQLCipher v4） | KEY-003 |

図の配置: `img/fig_des_sec_encryption_contexts.drawio` / `img/fig_des_sec_encryption_contexts.svg` を参照。

## 2. 通信時暗号化

### 2-1. TLS 1.3 強制方針

すべての外部通信経路において TLS 1.3 を必須とする。TLS 1.2 以下は接続拒否とする。

| 通信経路 | プロトコル | エンドポイント例 | 備考 |
|---|---|---|---|
| ハンディ APP → バックエンド | HTTPS (TLS 1.3) | `https://wnav.factory.internal/api/v1/` | 工場 LAN 内 |
| マスタメンテナンス → バックエンド | HTTPS (TLS 1.3) | 同上 | 工場 LAN 内 |
| 管理コンソール → バックエンド | HTTPS (TLS 1.3) | 同上 | 工場 LAN 内 |
| バックエンド → 親システム（IF-001） | HTTPS (TLS 1.3) | 親システム指定エンドポイント | mTLS 切替可能 |
| バックエンド → 親システム（IF-002） | HTTPS (TLS 1.3) | 同上 | mTLS 切替可能 |
| バックエンド → 親システム（IF-003） | HTTPS (TLS 1.3) | 同上 | OAuth 2.1 との組み合わせ |

### 2-2. 許可する暗号スイート

TLS 1.3 では暗号スイートの指定が RFC 8446 によってサーバー側で制限される。本システムは以下の 2 スイートのみを許可し、その他を明示的に無効化する。

| 優先順 | 暗号スイート識別子 | 鍵交換 | 認証 | 暗号 | ハッシュ |
|---|---|---|---|---|---|
| 1 | `TLS_AES_256_GCM_SHA384` | ECDHE (x25519 / P-384) | — | AES-256-GCM | SHA-384 |
| 2 | `TLS_CHACHA20_POLY1305_SHA256` | ECDHE (x25519) | — | ChaCha20-Poly1305 | SHA-256 |

TLS 1.3 における後方互換スイート（`TLS_AES_128_GCM_SHA256` 等）は有効化しない。

### 2-3. 証明書管理

本システムは単一工場クローズドネットワーク運用であるため、パブリック CA ではなく内部自己署名 CA を用いる。

| 項目 | 内容 |
|---|---|
| CA 識別子 | KEY-005（内部 CA 証明書） |
| CA 鍵アルゴリズム | RSA-4096 |
| サーバー証明書有効期間 | 398 日（Apple 上限に準拠） |
| CA 証明書有効期間 | 10 年 |
| ローテーション周期 | 年 1 回（CA 証明書は 10 年サイクル） |
| 配布方法 | ハンディ端末 MDM 経由でルート CA をプロビジョニング |
| OCSP/CRL | 工場内 OCSP レスポンダー（利用不可時はハードフェイル禁止・ソフトフェイル許可） |

### 2-4. 親システム連携（IF-001/002/003）の mTLS 切替

親システムが mTLS を要求する場合、以下の設定で切替可能とする。

| 設定識別子 | 内容 | デフォルト値 |
|---|---|---|
| CFG-TLS-001 | 親システム mTLS 有効フラグ | `false` |
| CFG-TLS-002 | クライアント証明書パス（KEY-005 配下） | `certs/client.pem` |
| CFG-TLS-003 | OAuth 2.1 との組み合わせモード | `oauth_with_tls` |

OAuth 2.1 を使用する場合でも TLS 1.3 は必須であり、TLS を廃止して OAuth トークンのみで認証することは禁止する。

## 3. 保存時暗号化

### 3-1. PostgreSQL ボリューム暗号化

PostgreSQL データ本体を格納するボリュームは、ホスト OS レベルの暗号化によって保護する。

| デプロイ環境 | 暗号化方式 | 備考 |
|---|---|---|
| Windows Server 2022 (IIS + ネイティブ) | BitLocker（AES-256-XTS） | TPM 2.0 を使用したフルボリューム暗号化 |
| WSL2 + Docker | dm-crypt / LUKS（AES-256-XTS） | Docker ボリュームを LUKS パーティション上に配置 |

PostgreSQL レベル（TDE: Transparent Data Encryption）は PostgreSQL 16 の公式機能に含まれないため、ボリューム暗号化をもってデータ保護の主手段とする。ページレベル暗号化拡張（pgcrypto）は特定カラムの追加保護に限り使用し、主暗号化基盤とはしない。

### 3-2. バックアップファイル暗号化

| 項目 | 内容 |
|---|---|
| 暗号化アルゴリズム | AES-256-GCM |
| 鍵識別子 | KEY-004（バックアップ暗号化鍵）、KEY-006（サーバーサイドストレージ暗号化鍵） |
| 鍵保管場所 | 暗号化済みキーボルト（後述 `06_キー・秘密情報管理.md` 参照） |
| バックアップ対象 | pg_basebackup 出力・WAL アーカイブ・エビデンス写真・署名済み PDF |
| 暗号化タイミング | バックアップ取得直後・ファイルシステム書き込み前に暗号化 |
| IV/Nonce | 128-bit ランダム生成（バックアップファイルごとに一意） |
| 認証タグ長 | 128-bit |

暗号化後のファイルフォーマット:

```
[IV: 16 bytes][GCM TAG: 16 bytes][暗号文: 可変長]
```

### 3-3. エビデンス写真・PDF のファイルシステム暗号化

品質確認時に取得するエビデンス写真（JPEG/PNG）および生成された品質報告 PDF は、ファイルシステム上で AES-256-GCM により暗号化して保存する。

| 項目 | 内容 |
|---|---|
| 暗号化アルゴリズム | AES-256-GCM |
| 鍵識別子 | KEY-006 |
| ファイルパス | `storage/evidence/{work_execution_id}/{ファイル名}.enc` |
| メタデータ管理 | TBL-028（evidence_files）に暗号文パス・IV・タグを記録 |
| アクセス制御 | RBAC（`06_ネットワーク・ゾーニング設計.md` 参照）で quality_admin / system_admin のみ復号可能 |

## 4. 端末ローカル暗号化（ハンディ APP）

### 4-1. SQLCipher AES-256-CBC

ハンディ APP の SQLite データベースは SQLCipher バージョン 4 を使用して暗号化する。

| 項目 | 内容 |
|---|---|
| 暗号化アルゴリズム | AES-256-CBC（SQLCipher v4 デフォルト） |
| 鍵導出関数 | PBKDF2-HMAC-SHA512 |
| PBKDF2 反復回数 | 256,000 回（SQLCipher v4 デフォルト） |
| ページサイズ | 4096 バイト（SQLCipher v4 デフォルト） |
| HMAC アルゴリズム | HMAC-SHA512（ページ完全性検証） |

### 4-2. パスフレーズ導出とデバイス結合鍵（KEY-003）

SQLCipher のパスフレーズは、デバイス固有の KEY-003 とユーザー識別子ハッシュを組み合わせて導出する。これにより、端末紛失時に別端末での DB 解読を不可能にする。

| 要素 | 内容 | 保管場所 |
|---|---|---|
| KEY-003（デバイス結合鍵） | AES-256、デバイスごとに一意 | Android Keystore / iOS Secure Enclave |
| user_id ハッシュ | `SHA-256(user_id + KEY-003)` | DB ファイル内には非保存 |
| パスフレーズ | `HKDF(KEY-003, user_id_hash, "sqlcipher-v4")` | 揮発性メモリのみ（起動時導出・終了時消去） |

パスフレーズ導出フロー:

```
アプリ起動
  → Android Keystore / iOS Secure Enclave から KEY-003 を取得
  → ログイン済み user_id を取得
  → HKDF(KEY-003, SHA-256(user_id), info="sqlcipher-v4") でパスフレーズ導出
  → PRAGMA key = '${パスフレーズ}'; を実行
  → SQLCipher が PBKDF2-HMAC-SHA512（256000回）で実際の暗号化鍵を生成
  → HMAC ヘッダ検証 → 成功 → 通常起動
  → 検証失敗 → ERR-SEC-003 を記録し DB 復旧フロー（09_キャッシュとオフラインストレージ方式 §4 参照）
```

### 4-3. デバイス間可搬性の禁止

KEY-003 は Android Keystore / iOS Secure Enclave から外部エクスポートできないように設定する（`setUserAuthenticationRequired(false)` + `KeyProperties.PURPOSE_ENCRYPT | PURPOSE_DECRYPT`）。端末交換時はサーバーから SOP マスタおよび個人設定を再取得し、ローカルの作業記録は送信済みのものに限りサーバー側 PostgreSQL を正本とする。

### 4-4. SQLCipher バージョン固定方針

SQLCipher の API バージョンおよびストレージフォーマットは v4 に固定し、メジャーバージョンアップを行う場合は既存 DB の移行手順（`ATTACH DATABASE`・`sqlcipher_export`）を詳細設計フェーズで策定する。バージョン固定はモバイルプロジェクトの `package.json` および `Cargo.toml` 双方で exact バージョン指定する。

## 6. 禁止事項

| 禁止事項 | 根拠 | 代替 |
|---|---|---|
| TLS 1.2 以下の使用 | NFR-SEC-003 | TLS 1.3 のみ |
| `TLS_AES_128_GCM_SHA256` の有効化 | 本章 §2-2 方針 | TLS_AES_256_GCM_SHA384 / TLS_CHACHA20_POLY1305_SHA256 |
| ECB モードの使用 | 既知の脆弱性（ブロックパターン露出） | GCM / CBC with HMAC |
| ソースコードへのハードコード（KEY-001〜009） | BR-BUS / NFR-SEC-006 | Docker Compose secrets / Keystore |
| SQLCipher パスフレーズのファイル保存 | デバイス紛失時の漏洩リスク | 揮発性メモリでの導出・使用・消去 |

---

**本節で確定した方針**
- **通信時暗号化は TLS 1.3 のみ（TLS_AES_256_GCM_SHA384 / TLS_CHACHA20_POLY1305_SHA256 の 2 スイート限定）を必須とし、内部自己署名 CA（KEY-005）で証明書を管理する。親システム IF-001/002/003 への mTLS は設定フラグ CFG-TLS-001 で切替可能とする。**
- **保存時暗号化は PostgreSQL ボリュームを BitLocker（Windows）/ dm-crypt（WSL2）で保護し、バックアップファイルとエビデンス写真・PDF は AES-256-GCM（KEY-004/KEY-006）で個別に暗号化してファイルシステムに格納する。**
- **端末ローカル暗号化は SQLCipher v4（PBKDF2-HMAC-SHA512、256,000 回反復）を使用し、パスフレーズは Android Keystore / iOS Secure Enclave の KEY-003 とログイン user_id から HKDF で揮発的に導出して使用後即消去する。これによりデバイス間の DB 持ち出しを技術的に不可能にする。**
