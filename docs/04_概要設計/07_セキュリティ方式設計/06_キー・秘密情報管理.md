# 06 キー・秘密情報管理

本章の責務は、本システムが使用する暗号鍵・証明書・秘密情報（KEY-001〜009）のライフサイクル全体（生成・配布・使用・ローテーション・失効・破棄）を設計し、鍵管理の方針・運用手順・禁止事項を確定することである。NFR-SEC-006（鍵管理）・BR-BUS-040〜045（セキュリティ統制）を受けて、クラウド KMS を使用しない自己完結型の鍵管理方式を固定する。

---

## 1. 設計前提と KMS 選定方針

本システムは単一工場の OT（Operational Technology）隔離環境に設置される。インターネット接続を前提とするクラウド KMS（AWS KMS・Google Cloud KMS 等）の採用は、OT 境界原則（P4: OT 直結禁止）および可用性要件（工場 LAN のみで稼働継続）に抵触するため採用しない。

| 判断事項 | 選択肢 | 採用判断 | 理由 |
|---|---|---|---|
| クラウド KMS | AWS KMS / GCP KMS | 不採用 | OT 隔離・インターネット依存 |
| ハードウェア HSM | nCipher / Thales Luna | ver1.0.0 対象外 | 単一工場規模でコスト非合理 |
| 自己ホスト型キーボルト | Docker Compose secrets + 暗号化ファイル | 採用 | 閉域網完結・IT 担当 1 名運用可能 |
| 端末側 | Android Keystore / iOS Secure Enclave | 採用 | OS 提供のハードウェア保護 |

自己ホスト型キーボルトの設計制約: キーボルト自体の暗号化鍵（マスターキー）はパスワードベースの KDF（Argon2id）で導出し、マスターパスワードは IT 担当者が管理する。このマスターパスワードは本ドキュメントに記載しない。

## 2. KEY 識別子カタログとライフサイクル

### 2-1. 全 KEY の一覧

| KEY 識別子 | 用途 | アルゴリズム | 生成方式 | ローテーション周期 |
|---|---|---|---|---|
| KEY-001 | JWT 署名秘密鍵 | RSA-4096 | OpenSSL `genrsa` | 90 日（CFG-007） |
| KEY-002 | JWT 検証公開鍵 | RSA-4096（KEY-001 の公開鍵） | KEY-001 から自動導出 | KEY-001 に連動 |
| KEY-003 | 端末ローカル暗号化鍵（デバイス結合） | AES-256 | OS Keystore 内で端末初期化時に生成 | 端末初期化時（再生成不可） |
| KEY-004 | バックアップ暗号化鍵 | AES-256-GCM | `openssl rand -base64 32` | 1 年（CFG-009） |
| KEY-005 | 内部 CA 証明書 | RSA-4096 | OpenSSL `req -x509` | 10 年（CA）/ 1 年（サーバー証明書） |
| KEY-006 | サーバーサイドストレージ暗号化鍵 | AES-256-GCM | `openssl rand -base64 32` | 1 年（CFG-009 に準拠） |
| KEY-007 | PDF デジタル署名鍵 | RSA-4096 | OpenSSL `genrsa` | 2 年（CFG-008） |
| KEY-008 | HMAC-SHA256 Outbox 署名鍵 | HMAC-SHA256（256 bit） | `openssl rand -hex 32` | 90 日（CFG-007 に準拠） |
| KEY-009 | ハッシュチェーン salt（日次） | AES-256 相当のランダムバイト列 | BAT-004 が毎日 00:00 に生成 | 1 日 |

### 2-2. ローテーション設定値（CFG-NNN）

| CFG 識別子 | 対象 KEY | デフォルト値 | 変更可否 |
|---|---|---|---|
| CFG-007 | KEY-001 / KEY-002 / KEY-008 | 90 日 | system_admin のみ変更可 |
| CFG-008 | KEY-007 | 2 年（730 日） | system_admin のみ変更可 |
| CFG-009 | KEY-004 / KEY-006 | 365 日 | system_admin のみ変更可 |

## 3. 鍵生成手順

### 3-1. 非対称鍵（KEY-001/007）の生成

```bash
# KEY-001: JWT 署名鍵（RSA-4096）
openssl genrsa -aes256 -out key_001_private.pem 4096
openssl rsa -in key_001_private.pem -pubout -out key_002_public.pem

# KEY-007: PDF 署名鍵（RSA-4096）
openssl genrsa -aes256 -out key_007_private.pem 4096
```

生成したファイルはキーボルト（`/secrets/vault/` ディレクトリ、権限 `0600`）に格納し、平文でソースコードリポジトリに追加することを厳禁する。

### 3-2. 対称鍵（KEY-004/006/008）の生成

```bash
# KEY-004: バックアップ暗号化鍵（AES-256）
openssl rand -base64 32 > key_004.secret

# KEY-006: ストレージ暗号化鍵（AES-256）
openssl rand -base64 32 > key_006.secret

# KEY-008: HMAC-SHA256 Outbox 署名鍵
openssl rand -hex 32 > key_008.secret
```

### 3-3. KEY-009（日次 salt）の自動生成

BAT-004（`cron: 0 0 * * *`）が毎日 00:00 に実行し、前日の KEY-009 を保全した後に新しい salt を生成してキーボルトに格納する。前日 salt はハッシュチェーン検証（BAT-003）のために 7 年間保全する。

## 4. 鍵配布と格納方式

### 4-1. Docker Compose Secrets による注入

バックエンドコンテナへの鍵配布は Docker Compose の `secrets` 機能を使用する。環境変数への直接埋め込み（`ENV KEY_001=...`）は禁止する。

```yaml
# docker-compose.yml（抜粋）
secrets:
  key_001_private:
    file: ./secrets/vault/key_001_private.pem
  key_006:
    file: ./secrets/vault/key_006.secret

services:
  backend:
    secrets:
      - key_001_private
      - key_006
```

コンテナ内では `/run/secrets/{secret_name}` としてファイルマウントされる。バックエンド（Rust）はアプリ起動時にのみこのファイルを読み込み、メモリ上に保持する。ファイルをコピー・ログ出力する処理は実装禁止とする。

### 4-2. 端末側（KEY-003）の格納

KEY-003 は端末初期化時に OS が生成し、ハードウェアセキュリティモジュールに格納する。アプリはこの鍵を「使用」するのみであり、エクスポート・バックアップは行わない。

| OS | 格納場所 | 鍵属性 |
|---|---|---|
| Android | Android Keystore（StrongBox 対応端末では StrongBox 使用） | `PURPOSE_ENCRYPT | PURPOSE_DECRYPT`、`setUserAuthenticationRequired(false)` |
| iOS | Secure Enclave | `kSecAttrTokenIDSecureEnclave`、`kSecAttrAccessibleWhenUnlocked` |

## 5. ダウンタイムなしのローテーション（Dual-Key 方式）

JWT（KEY-001/002）および Outbox 署名鍵（KEY-008）は稼働中にローテーションが必要なため、以下の Dual-Key 方式を採用する。

### 5-1. JWT ローテーション手順

| フェーズ | 期間 | 動作 |
|---|---|---|
| 通常稼働 | — | KEY-001（現行）のみ有効 |
| ローテーション準備 | 移行窓 0 日目 | KEY-001（現行）+ KEY-001-NEW（新規）の両方で署名を検証可能にする |
| 新鍵へ切替 | 移行窓 1 日目 | 新規発行 JWT を KEY-001-NEW で署名。KEY-001（現行）は検証のみ継続 |
| 旧鍵失効 | 移行窓 3 日目（全 JWT の TTL 超過後） | KEY-001（現行）の検証を無効化。KEY-001-NEW を KEY-001 に昇格 |
| 完了 | — | KEY-001-NEW のみ有効（新たな通常稼働） |

移行窓はデフォルト 3 日とし、CFG-007 で調整可能とする。ローテーション操作は system_admin が管理コンソール（SCR-MC-009）から実行し、LOG-010 に記録する。

### 5-2. ストレージ鍵（KEY-004/006）のローテーション手順

ストレージ暗号化鍵のローテーションは、既存暗号文の再暗号化が必要なため計画メンテナンス窓内で実施する。

1. メンテナンス窓を宣言（オペレーター・監督者にシステムアナウンス）
2. バックアップを新旧両鍵で復号可能な状態を確認
3. 新鍵（KEY-004-NEW / KEY-006-NEW）で既存ファイルを再暗号化
4. 再暗号化完了後、旧鍵を失効・廃棄
5. LOG-010 に記録

## 6. 鍵バックアップ

| 項目 | 内容 |
|---|---|
| バックアップ形式 | GPG アーマー形式（`gpg --armor --symmetric`）で暗号化 |
| 暗号化パスワード | IT 担当者が管理する別パスワード（キーボルトマスターパスワードとは異なる） |
| バックアップ媒体 | 書き込み済み後ロックした USB メモリ |
| コピー数 | 2 部 |
| 保管場所 | 物理的に分離した 2 か所（例: サーバー室 + 管理事務所金庫） |
| 更新タイミング | 鍵ローテーション都度 |
| 確認 | 年 1 回、バックアップからの復元テストを実施し LOG-008 に記録 |

---

## 7. 鍵失効・廃棄手順

| 状況 | 失効手順 |
|---|---|
| 定期ローテーション | §5 の Dual-Key 方式でグレースフルに失効 |
| 鍵漏洩が疑われる場合（P3 インシデント） | 即時失効: `09_インシデント対応の技術的フック.md` §3 の P3 フローを起動。新鍵に即時切替後、旧鍵を上書き 3 回（DoD 方式）でファイル消去 |
| 端末紛失（KEY-003） | MDM から端末のリモートワイプを実行。サーバー側で当該 terminal_id のセッションを失効 |
| 廃棄確認 | 廃棄を LOG-010（SYSTEM_KEY_ROTATE イベント、action=DESTROY）として記録 |

## 8. 禁止事項とCIによる強制

| 禁止事項 | エラー識別子 | CI 検出方法 |
|---|---|---|
| ソースコードへの鍵・シークレットのハードコード | ERR-SEC-001 | `gitleaks` / `truffleHog` による CI スキャン（pre-commit hook + PR チェック） |
| `.env` ファイルのリポジトリへのコミット | ERR-SEC-001 | `.gitignore` + gitleaks ルール |
| Docker イメージへの鍵の COPY | ERR-SEC-001 | Dockerfile lint（hadolint `DL3010` 相当カスタムルール） |
| 鍵ファイルのログ出力 | ERR-SEC-001 | カスタム clippy/eslint ルール |
| KEY-003 の端末外エクスポート | — | Android Keystore / Secure Enclave の非エクスポート属性で技術的に強制 |

---

**本節で確定した方針**
- **クラウド KMS は OT 隔離原則に抵触するため採用せず、Docker Compose secrets による自己ホスト型キーボルトを採用する。KEY-001〜009 の全鍵について生成・配布・ローテーション・失効・廃棄の手順を明示し、IT 担当 1 名でも運用可能な手順とする。**
- **JWT（KEY-001/002）および Outbox 署名鍵（KEY-008）のローテーションは 3 日間の Dual-Key 移行窓方式でダウンタイムなしに実施し、ストレージ鍵（KEY-004/006）は計画メンテナンス窓内で既存ファイルの再暗号化を伴うローテーションを実施する。**
- **鍵バックアップは GPG アーマー形式で 2 部の USB メモリに格納し、物理的に分離した 2 か所に保管する。ソースコードへのハードコードは ERR-SEC-001 として CI（gitleaks + hadolint + カスタム clippy/eslint）で自動検出・ブロックする。**
