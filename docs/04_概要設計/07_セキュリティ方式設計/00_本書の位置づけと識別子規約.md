## 4. 識別子規約

### 4-1. KEY 識別子（暗号鍵・秘密）

KEY 識別子は `KEY-NNN`（3 桁ゼロ埋め）形式を採用する。KEY-NNN は論理識別子のみであり、値はリポジトリ外（環境変数・Docker Secret）で管理する。

本サブで採番する KEY 識別子の一覧を以下に確定する。

| KEY-ID | 用途 | アルゴリズム | 鍵長 | ローテーション周期 | 保管先 | 関連 NFR |
|---|---|---|---|---|---|---|
| KEY-001 | JWT 署名秘密鍵（RS256 Signing Key） | RSA | 4096 bit | 90 日 | Docker Secret | NFR-SEC-003, NFR-SEC-004 |
| KEY-002 | JWT 検証公開鍵（RS256 Verification Key） | RSA | 4096 bit | KEY-001 連動 | Docker Secret | NFR-SEC-003, NFR-SEC-004 |
| KEY-003 | JWT 署名秘密鍵 - 次世代（Rotation Next） | RSA | 4096 bit | 90 日 | Docker Secret | NFR-SEC-003 |
| KEY-004 | JWT 検証公開鍵 - 次世代（Rotation Next） | RSA | 4096 bit | KEY-003 連動 | Docker Secret | NFR-SEC-003 |
| KEY-005 | SQLCipher 暗号化鍵（ハンディ APP オフラインキャッシュ） | AES | 256 bit | 180 日 | iOS Keychain / Android Keystore | NFR-SEC-004 |
| KEY-006 | PostgreSQL 列暗号化鍵（PII カラム・pgcrypto） | AES | 256 bit | 365 日 | Docker Secret | NFR-SEC-020, NFR-SEC-004 |
| KEY-007 | バックアップ暗号化鍵（pg_dump 出力 AES-256 暗号化） | AES | 256 bit | 365 日 | Docker Secret | NFR-SEC-004 |
| KEY-008 | Webhook HMAC-SHA256 署名鍵（親機連携 IF-003） | HMAC | 256 bit | 365 日 | Docker Secret | NFR-SEC-004 |
| KEY-009 | ElectronicSign サーバー署名鍵（HMAC-SHA256） | HMAC | 256 bit | 180 日 | Docker Secret | NFR-SEC-045 |

KEY-NNN の値は本ファイルに記載しない。採番台帳は `04_概要設計/付録/99_設計識別子採番台帳.md` で管理する。

### 4-2. PRM 識別子（権限マトリクスセル）

PRM 識別子は `PRM-NNN`（3 桁ゼロ埋め）形式を採用する。1 PRM = 1 権限マトリクスセル（ロール × リソース × 許可種別の組み合わせ）に対応する。

採番規約：
- PRM-001 から順に採番する（欠番・再使用禁止）
- 必須メタデータ: ロール・操作対象（SCR-NNN または API-NNN または TBL-NNN の列レベル）・許可種別（CRUD + 特殊操作）・判定（許可 / 拒否 / マスク）・根拠（BR-BUS-NNN）
- 採番台帳: `04_概要設計/付録/99_設計識別子採番台帳.md`
- 本サブ（07）での採番開始: PRM-001

PRM の詳細一覧は 02 章（RBAC 設計）で確定する。

## 5. ALCOA+ 対応マトリクス（本サブの設計命題が充足する原則）

ALCOA+ 9 原則のうち、本サブのセキュリティ設計が技術的に担保する原則を以下に示す。

| ALCOA+ 原則 | 英語 | セキュリティ設計での充足手段 | 担当章 |
|---|---|---|---|
| 帰属可能 | Attributable | RBAC + signer_id 必須 + 共有アカウント禁止（FR-AU-001） | 02・03 章 |
| 読取可能 | Legible | TLS 1.3 転送 + AES-256 保管による改ざんなし転送 | 06 章 |
| 同時記録 | Contemporaneous | timestamp_device + timestamp_server 二重保持 | 03 章（電子サイン） |
| 元データ | Original | Append-only + ハッシュチェーン + UPDATE/DELETE 禁止 | 05 章 |
| 正確 | Accurate | JWT 検証必須 + なりすまし防止 | 03 章 |
| 完全 | Complete | 4 種のイベント完全記録 | 05 章 |
| 一貫 | Consistent | RBAC による権限一貫性の技術的強制 | 02 章 |
| 耐久 | Enduring | ハッシュチェーン + Append-only 設計 | 05 章 |
| 利用可能 | Available | 監査ログ検索・エクスポート機能 | 05 章 |

---

**本節で確定した方針**
- 本サブが担当する上流要件の範囲（NFR-SEC 全 21 件・FR-AU 全 6 件・BR-BUS 9 件）を確定し、他サブとの設計責任分界を明示した。
- KEY-001〜009 の 9 種の暗号鍵論理識別子と採番規約を確定し、PRM-001 からの権限マトリクスセル採番を本サブで開始することを決定した。
- ALCOA+ 9 原則の各原則をセキュリティ設計の 5 担当章に対応させ、未対応原則をゼロとすることを確定した。
