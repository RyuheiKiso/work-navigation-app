# 07 脆弱性管理・依存バージョン管理

本章の責務は、本システムを構成するすべてのソフトウェアコンポーネント（Rust クレート・Node.js パッケージ・PostgreSQL・OS）に対して、CVE 追跡・依存バージョン固定・SBOM 生成・OSS ライセンス統制の方針を確定することである。NFR-SEC-010（脆弱性管理）・NFR-SEC-011（依存管理）を受けて、CVSS スコアリングに基づく対応期限と CI/CD への組み込み方法を設計レベルで固定する。

---

## 1. 脆弱性管理の全体方針

本システムは IT 担当 1 名体制での運用を前提とするため、可能な限り CI/CD の自動化によって脆弱性検出と対応管理を支援する。手動確認が必要な場合も「いつまでに何をするか」を CVSS スコアに基づいて明確に定義する。

| CVSS スコア範囲 | 重大度分類 | 対応期限 | 対応方針 |
|---|---|---|---|
| 9.0 〜 10.0 | Critical | 3 営業日以内 | 即時対応・必要に応じてサービス停止 |
| 7.0 〜 8.9 | High | 7 営業日以内 | 優先対応・ホットフィックスリリース |
| 4.0 〜 6.9 | Medium | 30 日以内 | 次回定期リリースまでに対応 |
| 0.1 〜 3.9 | Low | 四半期定期リリース | 定期的なバージョンアップで対応 |
| 0.0 | None | 対応不要 | リスク受容を記録し管理台帳に記載 |

リスク受容（Medium 以下を意図的に許容する場合）は system_admin の承認を得て、`docs/99_リスク管理台帳.md`（詳細設計フェーズで整備）に根拠とともに記録する。

## 2. Rust / Cargo の脆弱性管理

### 2-1. cargo audit の運用

| 項目 | 内容 |
|---|---|
| 実行タイミング | CI: 全 PR / main ブランチへの push（必須）。定期: 毎週月曜 06:00（BAT-005）|
| 使用ツール | `cargo-audit`（RustSec Advisory Database 参照） |
| CI での扱い | 1 件でも HIGH / CRITICAL が検出された場合はビルド失敗（`--deny warnings` 相当） |
| 定期実行結果 | LOG-008 イベント（SYSTEM_CONFIG_CHANGE 型）として記録し、system_admin に通知 |
| 例外処理 | 特定の Advisory を一時的に無視する場合は `audit.toml` に記載し system_admin が承認 |

```toml
# .cargo/audit.toml（例）
[advisories]
# RUSTSEC-YYYY-NNNN: 理由と期限を記載（例: 代替手段なし、CVSS 3.1 Low、2026-08-31 期限）
ignore = ["RUSTSEC-2026-0001"]
```

### 2-2. Dependabot による自動 PR

GitHub リポジトリで Dependabot を有効化し、Cargo.lock の脆弱なクレートに対する自動更新 PR を生成する。PR はレビュー承認後にマージし、CI を通過したもののみ取り込む。

| 設定 | 内容 |
|---|---|
| スキャン対象 | `Cargo.toml` / `Cargo.lock` |
| スケジュール | 週 1 回（月曜） |
| PR 自動マージ | 不可（human review 必須） |
| ブランチ保護ルール | main ブランチへの直接 push 禁止・PR + CI 通過必須 |

### 2-3. Cargo.lock の exact バージョン固定

`Cargo.toml` では `^` による範囲指定を許容するが、`Cargo.lock` は必ずコミットする。CI は `cargo build --locked` オプションを使用し、`Cargo.lock` と一致しないバージョンでのビルドを失敗とする。

```toml
# Cargo.toml（抜粋）
[dependencies]
axum = "0.7.5"          # SemVer 範囲は許容
tokio = { version = "1.38.0", features = ["full"] }
sqlx = { version = "0.8.0", features = ["postgres", "runtime-tokio-native-tls"] }
```

```bash
# CI でのビルドコマンド（Cargo.lock 厳守）
cargo build --release --locked
cargo test --locked
```

## 3. Node.js / React の脆弱性管理

### 3-1. npm audit の運用

| 項目 | 内容 |
|---|---|
| 実行タイミング | CI: 全 PR / main への push。定期: 毎週月曜 06:00（BAT-005 と同バッチ）|
| 使用ツール | `npm audit`（高: `--audit-level=high`） |
| CI での扱い | severity `high` / `critical` が 1 件でもあればビルド失敗 |
| moderate 以下 | ビルドは続行するが WARNING を CI ログに記録 |

### 3-2. npm outdated と更新方針

```bash
# CI での確認コマンド
npm outdated --json | tee outdated.json
```

CI は `outdated.json` をアーティファクトとして保存し、定期レビュー（四半期）のインプットとする。 `npm outdated` の結果によるビルド失敗は設けない（Dependabot PR で対応）。

### 3-3. package-lock.json の exact バージョン固定

`package.json` での `^`・`~` の範囲指定は許容するが、`package-lock.json` は必ずコミットする。CI は `npm ci`（`package-lock.json` 厳守）のみ使用し、`npm install` は CI 環境では使用しない。

```bash
# CI でのインストールコマンド
npm ci --prefer-offline
```

## 4. PostgreSQL の脆弱性管理

| 項目 | 内容 |
|---|---|
| 追跡対象 | PostgreSQL 16.x の CVE（NVD: `cpe:2.3:a:postgresql:postgresql:16.*`）|
| 追跡方法 | PostgreSQL Security Advisories（https://www.postgresql.org/support/security/）を週次で確認（BAT-005 で通知）|
| 適用方針 | マイナーバージョンアップ（16.x → 16.y）は 7 営業日以内（High CVSS 場合）または定期メンテナンス窓で適用 |
| メジャーバージョンアップ | ver1.0.0 スコープでは PostgreSQL 16 に固定。ver1.1.0 以降で計画的に移行 |
| pgcrypto 等の拡張 | PostgreSQL 本体と同一バージョンで管理。拡張固有の CVE も追跡対象 |

## 5. OS / 実行環境の脆弱性管理

### 5-1. Windows Server 2022

| 項目 | 内容 |
|---|---|
| パッチ管理 | WSUS（Windows Server Update Services）を使用し、Critical / Important パッチは自動承認 |
| 適用スケジュール | 毎月第 2 火曜日（Microsoft Patch Tuesday）翌 7 営業日以内 |
| 再起動 | 計画メンテナンス窓（深夜 02:00〜04:00）に実施。オペレーターへ事前通知必須 |
| ログ | Windows Event Log（System / Security）を LOG-008 相当の記録として保全 |

### 5-2. WSL2 + Docker

| 項目 | 内容 |
|---|---|
| WSL2 カーネル | Windows Update 経由で自動更新（`wsl --update`）|
| Docker Engine | `docker version`・`docker scout cves` で週次確認（BAT-005）|
| ベースイメージ | `rust:1.79-slim-bookworm`・`node:20-slim`（詳細は Dockerfile 固定）|
| イメージスキャン | `docker scout cves` / `trivy image` を CI に組み込み、HIGH/CRITICAL で失敗 |

## 6. SBOM（Software Bill of Materials）

各リリース時点でのソフトウェア部品表を自動生成し、変更管理の証跡とする。

| 項目 | 内容 |
|---|---|
| Rust SBOM | `cargo cyclonedx` で `bom.xml`（CycloneDX 形式）を生成 |
| Node.js SBOM | `cyclonedx-npm` で `bom.json`（CycloneDX 形式）を生成 |
| 出力タイミング | リリースビルド（main ブランチへの tag push）時に自動生成 |
| 格納場所 | `docs/releases/vX.Y.Z/sbom-backend.xml`・`docs/releases/vX.Y.Z/sbom-frontend.json` |
| フォーマット | CycloneDX 1.5 |
| 用途 | 脆弱性ライセンス追跡・監査対応・リスク管理台帳との紐付け |

## 7. OSS ライセンス管理

### 7-1. 許可ライセンス

本システムに組み込む OSS ライブラリは以下のライセンスのみを許可する。

| 許可ライセンス | 根拠 |
|---|---|
| MIT | 商用利用可能・コピーレフトなし |
| Apache-2.0 | 特許条項あり・商用利用可能 |
| BSD-2-Clause | 商用利用可能・コピーレフトなし |
| BSD-3-Clause | 商用利用可能・コピーレフトなし |

### 7-2. 禁止ライセンス

| 禁止ライセンス | 禁止理由 |
|---|---|
| GPL-2.0 / GPL-3.0 | コピーレフト（本システムのソースコード開示義務が生じる） |
| LGPL-2.0 / LGPL-2.1 / LGPL-3.0 | 動的リンクの解釈が曖昧・法務リスク |
| AGPL-3.0 | ネットワーク経由提供でもソース開示義務 |
| 商用ライセンス（SaaS 利用含む） | ライセンスコスト・依存リスク |
| Proprietary / Unknown | ライセンス不明確 |

### 7-3. CI によるライセンスチェック

```bash
# Rust: cargo-deny でライセンスチェック
cargo deny check licenses

# Node.js: license-checker
npx license-checker --onlyAllow 'MIT;Apache-2.0;BSD-2-Clause;BSD-3-Clause' --excludePrivatePackages
```

CI でライセンスチェックが失敗した場合はビルドを失敗とする。代替ライブラリが存在しない場合は system_admin の承認を得て `deny.toml` / `license-checker.json` に例外を記載する。

## 8. 実装禁止事項

| 禁止事項 | 根拠 | CI 検出方法 |
|---|---|---|
| `eval()` の使用（JavaScript / TypeScript） | OWASP A03・P7 原則 | ESLint `no-eval` ルール（エラー） |
| `new Function()` の使用 | 同上 | ESLint `no-new-func` ルール（エラー） |
| 動的コード生成（Rust proc_macro 外） | P7 原則 | カスタム clippy lint |
| ユーザー制御文字列の SQL への直接埋め込み | OWASP A03・SQL インジェクション | sqlx コンパイル時クエリ検証（`query!` マクロ必須） |
| `cargo build --no-lock` オプション | 再現性破壊 | CI Makefile でオプション禁止 |
| `npm install` を CI で使用 | `package-lock.json` 無視のリスク | CI スクリプトで `npm ci` のみ許可 |

sqlx の使用においては、すべての SQL クエリを `sqlx::query!` / `sqlx::query_as!` マクロを用いてコンパイル時に検証する。実行時文字列結合による動的クエリ（`format!("SELECT ... WHERE id = {}", id)`）は禁止し、カスタム clippy ルールで検出する。

---

**本節で確定した方針**
- **CVSS ≥ 7.0 の脆弱性は 7 営業日以内、CVSS ≥ 9.0 は 3 営業日以内に対応し、CI（cargo audit / npm audit / trivy image）で HIGH/CRITICAL を検出した場合はビルドを自動失敗とする。Cargo.lock / package-lock.json は exact バージョンとしてコミットし、CI は `--locked` / `npm ci` で再現性を担保する。**
- **SBOM は各リリース時に CycloneDX 形式で自動生成し、リリース資材として保管する。OSS ライセンスは MIT / Apache-2.0 / BSD-2/BSD-3 のみ許可し、CI（cargo deny + license-checker）で自動検証する。GPL / LGPL / AGPL / 商用ライセンスは禁止とする。**
- **`eval()` / 動的コード生成 / 文字列結合による SQL は CI の静的解析（ESLint + カスタム clippy）で自動ブロックし、sqlx の全クエリをコンパイル時検証（`query!` マクロ）で SQL インジェクションを設計レベルで排除する。**
