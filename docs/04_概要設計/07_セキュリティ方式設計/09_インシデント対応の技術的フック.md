# 09 インシデント対応の技術的フック

本章の責務は、本システムが自律的に検出・記録・通知できるセキュリティおよび運用インシデントの技術的検出フック・エスカレーション経路・対応優先度を設計レベルで確定することである。NFR-SEC-013（インシデント対応）・FR-KZ-001（アンドン）・FR-AU-006（ハッシュチェーン整合性）・FR-SY-007（DLQ 管理）を受けて、各フックの発動条件・記録内容・通知先・対応の初動手順を固定する。

---

## 1. インシデント重大度分類

本システムが扱うインシデントを 4 段階の重大度（Priority）で分類する。

| 重大度 | 記号 | 定義 | 例 |
|---|---|---|---|
| P1 | システムダウン | ハンディ APP が工場全体で使用不能。作業ナビゲーション機能の完全停止 | バックエンドプロセスのクラッシュ・PostgreSQL 停止 |
| P2 | データ整合性リスク | 作業記録の完全性・正確性・信頼性に影響するリスクが検出された状態 | ハッシュチェーン破断・Outbox 永続未送信・DB 整合性エラー |
| P3 | セキュリティアラート | 不正アクセス試行・鍵漏洩疑い・認証異常が検出された状態 | JWT 検証連続失敗・アカウント自動ロック多発・ERR-SEC-001 検出 |
| P4 | 運用上の警告 | サービス継続は可能だが対応が必要な警告状態 | DLQ 蓄積・マスタ同期遅延・ディスク使用率閾値超過 |

## 2. 技術的フック一覧

### 2-1. アンドン発報（FR-KZ-001）

| 項目 | 内容 |
|---|---|
| 発動条件 | オペレーターが SCR-HA-006（アンドン発報画面）から発報操作を実施したとき |
| 重大度 | P4（運用上の警告） |
| 技術的処理 | バックエンドが WebSocket プッシュを監督者端末（SCR-SV-001）に送信 |
| 監査記録 | LOG-005（WORK_SUSPEND イベント）を TBL-032 に INSERT |
| 通知先 | WebSocket 接続中の supervisor ロール全端末（SCR-SV-001）|
| タイムアウト | 監督者が 5 分以内に応答しない場合、LOG-008 に未応答記録を追記 |
| 重複発報対策 | 同一 `work_execution_id` からの発報は 60 秒以内の重複を無効化（CFG-013）|

### 2-2. アカウントロック（BR-BUS-035）

| 項目 | 内容 |
|---|---|
| 発動条件 | 同一 `user_id` で PIN 認証が連続 3 回失敗した場合 |
| 重大度 | P3（セキュリティアラート） |
| 技術的処理 | バックエンドが `users.status` を `'LOCKED'` に UPDATE |
| 監査記録 | LOG-004（AUTH_LOCK イベント）を TBL-032 に INSERT |
| 通知先 | supervisor ロールおよび system_admin へプッシュ通知 |
| ロック解除 | master_admin / system_admin のみ `users.status` を `'ACTIVE'` に戻せる |
| 追加制御 | ロック後の追加 PIN 試行は ERR-AUTH-002 を返却し試行を記録のみ（総当たり観測のため）|

### 2-3. ハッシュチェーン破断（FR-AU-006）

| 項目 | 内容 |
|---|---|
| 発動条件 | BAT-003（週次整合性検証バッチ、毎週月曜 02:00）が `prev_hash` 再計算値の不一致を検出した場合 |
| 重大度 | P2（データ整合性リスク） |
| エラー識別子 | ERR-SEC-002 |
| 技術的処理 | 破断箇所の `work_execution_id`・`event_id`・`detected_at` を `hash_chain_anomalies` テーブルに記録 |
| 監査記録 | LOG-010 相当（SYSTEM_HASH_BREAK イベント）を TBL-032 に INSERT |
| 通知先 | SCR-MC-008（セキュリティアラート画面）に表示 + system_admin へプッシュ通知 |
| エスカレーション | 24 時間以内に system_admin が対応しない場合、quality_admin にもプッシュ通知（エスカレーション） |
| 初動対応 | system_admin は対象期間のバックアップから元データと比較し、改ざんか DB 障害かを判別する |

### 2-4. DLQ オーバーフロー（FR-SY-007）

| 項目 | 内容 |
|---|---|
| 発動条件 | Dead Letter Queue のイベント数が CFG-011（デフォルト 100 件）を超過した場合 |
| 重大度 | P4（運用上の警告）。72 時間以上解消されない場合 P3 へ昇格 |
| エラー識別子 | ERR-SYS-005 |
| 技術的処理 | DLQ カウント監視バッチ（BAT-006、5 分間隔）が閾値超過を検出 |
| 監査記録 | LOG-009（SYSTEM_DLQ_ACTION イベント）を TBL-032 に INSERT |
| 通知先 | SCR-MC-007（DLQ 管理画面）にアラート表示 |
| 対応 UI | SCR-MC-007 から DLQ イベントの再処理・破棄・詳細確認が可能 |

### 2-5. 不正 JWT 検証失敗（NFR-SEC）

| 項目 | 内容 |
|---|---|
| 発動条件 | 同一 IP アドレスから 1 分間に 5 回以上の JWT 検証失敗（`invalid_token` エラー）が発生した場合 |
| 重大度 | P3（セキュリティアラート） |
| 技術的処理 | レートリミットミドルウェア（axum-tower-governor 相当）が当該 IP を一時的にレート制限 |
| 監査記録 | LOG-002 相当（AUTH_LOGIN_FAIL イベント）を TBL-032 に INSERT |
| ブロック期間 | CFG-014（デフォルト 15 分）。工場 LAN 内の IP 範囲（CFG-015）はレート制限から除外可能 |
| 通知先 | system_admin へのプッシュ通知（P3 ルート） |
| 注意 | 工場 LAN 内端末の NAT によって同一 IP からの正常リクエストが集中する可能性があるため、CFG-015 で除外 IP 範囲を適切に設定すること（詳細設計フェーズで確定）|

## 3. 通知ルーティング方針

### 3-1. 重大度別の通知先マトリクス

| 重大度 | system_admin | quality_admin | supervisor | ops ダッシュボード |
|---|---|---|---|---|
| P1 | プッシュ通知（即時） | プッシュ通知（即時） | — | SCR-MC-001 に赤バナー |
| P2 | プッシュ通知（即時） | プッシュ通知（即時） | — | SCR-MC-008 アラート |
| P3 | プッシュ通知（即時） | — | — | SCR-MC-008 アラート |
| P4 | — | — | — | SCR-MC-007 / SCR-MC-001 警告バナー |

### 3-2. プッシュ通知の実装方式

| 項目 | 内容 |
|---|---|
| 配信方式 | バックエンドから管理コンソール（React Web）への WebSocket プッシュ |
| 外部 SIEM 連携 | 対象外（単一工場クローズドネットワーク。SIEM への外部送信は行わない）|
| Webhook | 管理コンソール内部への Webhook（自己完結型）のみ。外部 URL への Webhook は禁止 |
| 通知の永続化 | 通知は TBL-032 の監査ログに記録されるため、WebSocket 切断中のイベントも管理コンソール再接続時にリプレイ可能 |

## 4. インシデント対応の初動フロー

### 4-1. P1（システムダウン）

```
検出: バックエンドのヘルスチェック（/health）が 30 秒以上 503 を返す
  → system_admin + quality_admin にプッシュ通知
  → SCR-MC-001 に全画面赤バナー「システム停止中」表示
  → system_admin がサーバーログを確認（/var/log/wnav/backend.log）
  → 原因が特定できた場合: バックエンド再起動 or Docker コンテナ再起動
  → 原因特定不能の場合: 最新バックアップからの復元フローを起動
  → 復旧後、LOG-008（SYSTEM_CONFIG_CHANGE, action=INCIDENT_RESOLVE_P1）を記録
```

### 4-2. P2（データ整合性リスク）

```
検出: BAT-003 または BAT-006 による検出
  → ERR-SEC-002 または ERR-SYS-005 を発行
  → system_admin + quality_admin にプッシュ通知
  → 対象の work_execution_id / event_id を特定
  → 前回バックアップのデータと照合し改ざんか障害かを判別
  → 改ざんが確認された場合: 証拠保全 → 関係者報告 → 詳細設計フェーズで定める対応手順を実施
  → LOG-010 に対応経緯を全件記録
```

### 4-3. P3（セキュリティアラート）

```
検出: 不正 JWT 集中・アカウントロック多発・ERR-SEC-001 CI 検出
  → system_admin にプッシュ通知
  → SCR-MC-008 にアラート表示
  → 当該 IP / user_id のアクセスログを確認
  → アカウントロックの場合: ユーザーへの確認後、master_admin がロック解除
  → 鍵漏洩が疑われる場合: §3-2 の Dual-Key 方式で即時ローテーション（06_キー・秘密情報管理.md §7 参照）
  → LOG-002 / LOG-004 を根拠に対応記録を LOG-008 に追記
```

### 4-4. P4（運用上の警告）

```
検出: DLQ 閾値超過・マスタ同期遅延・ディスク警告
  → SCR-MC-007 / SCR-MC-001 に警告バナー
  → system_admin が次の定期作業時間帯（翌営業日）に対応
  → DLQ の場合: SCR-MC-007 から原因イベントを確認し再処理 or 破棄
  → LOG-009 に対応記録
```

## 5. 外部 SIEM・SOC 非連携の設計根拠

本システムは単一建屋内のクローズドネットワーク運用であり、外部 SOC・SIEM との連携は要件に含まれない。この判断の根拠を以下に示す。

| 判断理由 | 内容 |
|---|---|
| ネットワーク境界 | 工場 LAN は外部インターネットから隔離されており、外部 SIEM への送信経路が存在しない |
| 規模適合性 | 単一工場・IT 担当 1 名体制において外部 SOC 契約は過剰投資 |
| データ主権 | 作業記録・証跡データを外部クラウドサービスに送信することはデータ主権の観点で不適切 |
| 代替手段 | TBL-032 の Append-only 監査ログ + ハッシュチェーン + 週次整合性検証バッチ（BAT-003）が同等の改ざん検知機能を内部完結で提供する |

将来的に工場がマルチサイト展開または ISO/IEC 27001 認証を取得する場合、SIEM 連携は詳細設計フェーズ以降で検討する。

---

**本節で確定した方針**
- **インシデントを P1〜P4 の 4 段階で分類し、P1/P2 は system_admin + quality_admin への即時プッシュ通知、P3 は system_admin のみ、P4 は ops ダッシュボード警告バナーに通知をルーティングする。すべてのインシデント発動は TBL-032（audit_logs）に対応する LOG 識別子で記録する。**
- **アカウントロック（BR-BUS-035）は PIN 3 回連続失敗で即時発動し `users.status='LOCKED'` + LOG-004 + supervisor/system_admin 通知の 3 アクションを原子的に実行する。ロック解除は master_admin / system_admin のみが行える。**
- **外部 SIEM / SOC との連携は OT 隔離・規模適合性・データ主権の観点から意図的に採用せず、TBL-032 Append-only 監査ログ + ハッシュチェーン + BAT-003 週次検証の三重構造で改ざん検知を自己完結させる。**
