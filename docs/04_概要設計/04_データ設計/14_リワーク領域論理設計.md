# 14 リワーク領域論理設計

本章はリワーク（修正作業）機能領域の論理設計を確定する。EN-032〜039 の 8 エンティティを中心に、ALCOA+ Original 不変性・Two-Person Integrity・リワーク状態機械の実装方針を設計命題として記述する。

---

## 1. エンティティ関係概要

EN-033 Disposition（ディスポジション判定）が EN-032 Rework の起点となる。Disposition 確定後に EN-032 Rework が生成され、リワーク種別（REWORK_FULL/TOUCH_UP/SORTING/SCRAP/RETURN）に応じて後続エンティティが分岐する。

- REWORK 系: `TBL-005 work_executions`（`execution_type='REWORK'`、新 `rework_case_id`）→ EN-034 ReworkVerification（再検査）→ EN-036 ReworkedLotLabel（QR ラベル発行）
- SCRAP 系: EN-038 ScrapRecord → TBL-049 scrap_records
- RETURN 系: EN-039 ReturnToVendorRecord → TBL-050 return_to_vendor_records

EN-037 ReworkCostRecord は BAT-011 が日次で集計して更新する（唯一の上書き可能エンティティ）。

---

## 2. 設計確定事項

### 2-1. ALCOA+ Original 不変性（NFR-DQ-010）

リワーク実施時に親 WorkExecution（`reworks.parent_case_id`）の `work_events` テーブルは一切変更しない。リワーク作業は `execution_type='REWORK'` を持つ新 WorkExecution（`reworks.rework_case_id`）として独立して記録する。両者は双方向 FK で関連付ける。

### 2-2. Two-Person Integrity（NFR-SEC-048）

EN-033 Disposition は以下の 2 つの独立した電子サインを必須とする:

| 署名者ロール | カラム名 | 検証方式 |
|---|---|---|
| 品質担当（quality_admin）| quality_admin_sign_id FK | DB トリガ `check_disposition_distinct_signers` |
| 現場監督（supervisor）| supervisor_sign_id FK | 上記と同じトリガで signer_id 異一性を検証 |

同一 worker_id による二重署名は INSERT 時に DB トリガが拒否し `ERR-BIZ-021` を返す。

### 2-3. リワーク無限ループ防止

同一 `parent_lot_id` に対するリワーク件数が `CFG-026`（デフォルト 3）を超えた場合、システムは SCRAP を強制提案する。`TBL-024 lots.rework_history_count` で件数を追跡する。

### 2-4. 修正品 QR ラベル（GS1 AI 8003 + AI 91）

再検査合格後に `TBL-047 reworked_lot_labels` へ Append-only で発行する。GIAI（Global Individual Asset Identifier）フォーマット: `CFG-027 prefix + rework_id + parent_lot_id の下 12 桁`。AI 91（ベンダー固有）に `rework_sop_version_id` を格納する。

---

## 3. 再検査 Two-Person Integrity

EN-034 ReworkVerification の `verifier_id` は リワーク実施者（`work_executions.worker_id` where execution_id = rework_case_id）と一致してはならない（`ERR-BIZ-023`）。検証は API 層で実施し、CFG-025（デフォルト true）で強制・緩和を制御可能とする。

---

## 4. wnav_rework BC 境界

- **IN**: `nonconformities` からの `disposition_id` FK が Rework BC の起点
- **OUT**: `reworks.status=CLOSED_OK_RELEASE` 時に `lot_qc_states` を更新し後工程を解放
- **CAPA 連携**: `reworks.related_capa_id` で疎結合（FK のみ）。CAPA BC は Rework BC の状態を直接変更しない
- **イベント**: `DispositionDecided`、`ReworkCompleted`、`ReworkScrapped` の 3 ドメインイベントを Outbox Pattern で発行
