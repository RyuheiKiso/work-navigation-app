# 12 暗号化・マスキング設計

本章の責務は、データの保存時・通信時・端末ローカルにおける暗号化方式と、PII のマスキング・倫理ガード（禁止クエリ）を確定することである。

---

## 1. 保存時暗号化

### 1-1. PostgreSQL（サーバーサイド）

| 対象 | 暗号化手段 | 根拠 NFR |
|---|---|---|
| PostgreSQL データファイル全体 | OS 暗号化（BitLocker または dm-crypt）| NFR-SEC |
| バックアップファイル（pg_dump） | AES-256-GCM（KEY-006 で鍵管理）| NFR-AVL-012 |
| WAL アーカイブ | OS 暗号化（BitLocker/dm-crypt と同一）| NFR-AVL-015 |
| pgcrypto（カラム暗号）| `pgcrypto` 拡張を利用可能にするが、ver1.0.0 では未適用（OS 暗号化で十分と判断）| — |

### 1-2. 端末 SQLite（クライアントサイド）

| 対象 | 暗号化手段 | 鍵管理 |
|---|---|---|
| 端末 SQLite（Android）| SQLCipher（AES-256）| Android Keystore（KEY-003）|
| 端末 SQLite（iOS）| SQLCipher（AES-256）| iOS Keychain（KEY-004）|
| 端末 SQLite（Windows）| SQLCipher（AES-256）| Windows DPAPI（KEY-005）|

SQLCipher の鍵は端末初期化時に OS のセキュアストレージに生成し、アプリが起動するたびに取り出して SQLite の暗号化鍵として使用する。アプリリセット時に鍵を削除し、データを実質的に消去する（KEY-003〜005 が消去される）。

### 1-3. 証拠ファイル（NAS）

| 対象 | 暗号化手段 |
|---|---|
| NAS ファイルシステム | BitLocker（Windows Server の NAS ドライブ）|
| バックアップ転送 | SFTP with TLS 1.3 |

## 2. 通信時暗号化

全通信は `05_外部インターフェース設計/11_通信暗号化・mTLS・証明書運用.md` で確定する。本章ではデータ設計の観点から以下を宣言する。

- API 通信: TLS 1.3 必須（KEY-007）
- 親機→子機 mTLS: KEY-008 クライアント証明書
- HTTP 禁止（`Strict-Transport-Security` ヘッダ + IIS での HTTP → HTTPS リダイレクト）

## 3. PII マスキング設計

### 3-1. PII 分類と取扱方針

| PII 種別 | 格納テーブル・列 | 取扱方針 |
|---|---|---|
| 氏名 | TBL-016 users.display_name | 退職 60 日後に匿名化（§09 §2 参照）|
| ログイン ID | TBL-016 users.login_id | 退職 60 日後に匿名化 |
| 認証ログ IP アドレス | TBL-032 auth_logs.ip_address | IPv4 は /24 マスク（最終 1 オクテット削除）して保存 |
| 顔写真 | TBL-009（NULLABLE, 本人同意時のみ）| 同意取消時: NAS ファイル削除 + TBL-009 file_path を `[DELETED]` に UPDATE |
| 位置情報 | **非収集**（対象外と判断する）| — |

### 3-2. ログの PII マスク要否（LOG 識別子との連動）

| LOG-ID | イベント | PII マスク |
|---|---|---|
| LOG-003 | auth.login.success | user_id のみ記録（display_name を記録しない）|
| LOG-004 | auth.login.failure | 試みたログイン ID は /32 ハッシュに変換後記録 |
| LOG-005 | rbac.denied | user_id + denied_resource のみ |
| LOG-010 | report.generated | user_id + report_type のみ |

## 4. 禁止クエリ設計（倫理ガード）

BR-BUS-029（個人別ランキング禁止）・NFR-ETH（用途三限定）を技術的に担保する。

### 4-1. 禁止クエリパターン

以下のクエリパターンはアプリケーション層（MOD-BE 系）に存在させない。

```sql
-- 禁止: 作業員別の速度ランキング
SELECT resource, AVG(completion_time)
FROM work_events
GROUP BY resource
ORDER BY completion_time;

-- 禁止: 個人別の不適合発生率
SELECT resource, COUNT(*) as nc_count
FROM work_events we JOIN nonconformities nc ON ...
GROUP BY resource;

-- 禁止: 個人別の品質スコアリング
```

### 4-2. 禁止の技術的担保

| 担保手段 | 内容 |
|---|---|
| コードレビュー規則 | `GROUP BY resource`（または `worker_id`）を含む SQL のプルリクエストは自動 lint で警告を発生させる |
| ビュー設計の禁止 | `GROUP BY users.user_id` を含む VIEW・MV を作成しない（§03 VW カタログで確認）|
| 読み取り権限の制限 | `app_readonly` ロールは `users` テーブルへのフルアクセスを禁止し、匿名化済み view のみ公開 |
| `07_セキュリティ方式設計/08_倫理ガード設計.md` で API エンドポイントとして確認 | 実装フェーズのチェックリストに組み込み |

---

**本節で確定した方針**
- **保存時暗号化は PostgreSQL → OS 暗号化（BitLocker/dm-crypt）・バックアップ → AES-256-GCM・端末 SQLite → SQLCipher（KEY-003〜005 を OS セキュアストレージで管理）の 3 層で確定した。**
- **PII（氏名・ログイン ID）は退職 60 日後に不可逆ハッシュで匿名化し、`user_id` UUID を維持してトレーサビリティの継続性を保つ。**
- **個人別集計クエリ（`GROUP BY resource`）をアプリ層・DB ビュー・API 設計の三層で技術的に排除し、Just Culture・NFR-ETH の担保を設計命題として確定した。**
