# 11 DB スキーマ移行と版数管理

本章の責務は、PostgreSQL スキーマの管理方式・マイグレーションツール・バージョニング規約・前方互換ルールを確定することである。NFR-QUA（PostgreSQL 移行 15 分以内）の品質目標を担保する設計を確定する。

---

## 1. マイグレーションツールと管理方式

### 1-1. sqlx migrate の採用

Rust バックエンドが依存する `sqlx` のマイグレーション機能（`sqlx::migrate!` マクロ）を採用する。

| 比較項目 | sqlx migrate | Flyway | 採否 |
|---|---|---|---|
| Rust との統合 | ネイティブ対応 | 別プロセス必要 | sqlx 採用 |
| バージョン管理 | SQL ファイル + タイムスタンプ | SQL ファイル + バージョン番号 | — |
| ロールバック | 手動（Down マイグレーション記述）| 有料版のみ | — |
| CI 統合 | cargo test で自動実行 | 別 CI ステップ | sqlx 有利 |

ADR-NNN「sqlx migrate 採用」（付録/05 ADR 索引参照）。

### 1-2. マイグレーションファイルの命名規約

```
src/backend/migrations/
  {UNIX_TIMESTAMP}_{description}.sql
  例:
    1716000000_create_work_events.sql
    1716000100_add_idx_work_events_case_id.sql
    1716000200_create_outbox_events.sql
```

- タイムスタンプは `date +%s` で生成
- `description` は英小文字スネークケース・動詞始まり（`create_` / `add_` / `alter_` / `drop_index_`）
- Down マイグレーション（ロールバック）は別ファイル `{UNIX_TIMESTAMP}_{description}.down.sql` として作成

### 1-3. マイグレーションの適用順序

sqlx は UNIX_TIMESTAMP 昇順でマイグレーションを適用する。適用済みマイグレーションは `_sqlx_migrations` テーブルで管理（sqlx 標準）。

## 2. 前方互換ルール（ブルー/グリーンデプロイ準備）

ver1.0.0 は Blue/Green デプロイを採用しないが、将来の無停止アップグレードに備えて以下の前方互換ルールを適用する。

| ルール | 内容 |
|---|---|
| **禁止: 列削除** | 列削除は行わない。不要列は `deprecated` プレフィックスを付けた後、次回以降で削除検討 |
| **禁止: 列の型変更（縮小）**| VARCHAR(256) → VARCHAR(64) 等の縮小は禁止。拡大のみ許可 |
| **禁止: NOT NULL 制約の追加**（既存データあり）| 既存行があるテーブルへの NOT NULL 列追加は DEFAULT 値と同時に行う |
| **推奨: 拡張時は新列 ADD** | 既存列の変更ではなく新列の ADD を選ぶ |
| **必須: 削除前に UNUSED 期間**| 列や制約を削除する前に UNUSED（アプリが参照しない状態）で 1 バージョン以上を経過させる |

## 3. マイグレーション品質目標（NFR-QUA）

NFR-QUA では「PostgreSQL 移行 15 分以内」を品質目標としている。

| チェック項目 | 基準 |
|---|---|
| テスト DB 上の実行時間 | 15 分以内（100 万行データ相当）|
| ロックの種類 | ACCESS EXCLUSIVE ロックを最小化（`ADD COLUMN` は AccessShare ロックのみ）|
| バックグラウンドインデックス | `CREATE INDEX CONCURRENTLY` を使用してテーブルロックを回避 |

### 3-1. 長時間マイグレーションの分割戦略

1 回のマイグレーションで 15 分を超えると予測される場合:
1. スキーマ変更（列追加・型変更）と データ変換を別ファイルに分割
2. スキーマ変更を先にデプロイ（古いアプリが新列を無視できるため後方互換）
3. バックグラウンドデータ変換バッチ（BAT-NNN）で段階的に実行
4. 全完了後にアプリを新版にデプロイ

## 4. スキーマのバージョン管理と公開コミット

計画 13 章（データ移行とマスタ初期投入戦略）が確定した「PostgreSQL スキーマの永続的バージョン管理・公開コミット」を物理設計に落とす。

- マイグレーションファイルは Git で管理し、Main ブランチへのマージをもって「適用済み」とする
- MIT ライセンスに基づき、全マイグレーション SQL を OSS として公開する
- スキーマの Breaking Change は SemVer メジャーバージョン上げと合わせて tag を付与する

---

**本節で確定した方針**
- **Rust バックエンドと統合済みの `sqlx migrate` を採用し、`src/backend/migrations/{UNIX_TIMESTAMP}_{description}.sql` の形式でスキーマを管理する。**
- **前方互換ルール（列削除禁止・型縮小禁止・NOT NULL 追加時 DEFAULT 必須）を適用し、将来の無停止アップグレードを可能にする設計基盤を確保する。**
- **マイグレーション実行は 15 分以内を品質目標とし、長時間マイグレーションはスキーマ変更とデータ変換を別 BAT に分割する戦略を確定した。**
