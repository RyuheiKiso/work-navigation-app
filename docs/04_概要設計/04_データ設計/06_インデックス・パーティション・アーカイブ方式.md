# 06 インデックス・パーティション・アーカイブ方式

本章の責務は、NFR-PRF（性能要件）を実現するインデックス（IDX）・パーティション（PART）・アーカイブポリシーを確定することである。

---

## 1. インデックス設計（IDX カタログ）

### 1-1. 設計原則

- 全 Append-only テーブルは `created_at` 降順インデックスを必須とする（新しい行を効率的に取得するため）
- NFR-PRF-001（Step 完了応答 P95 200ms）を達成するため `work_events` への検索インデックスを最優先で設計する
- Partial インデックスで不要行を除外し、インデックスサイズと更新コストを低減する

## 2. パーティション設計

### 2-1. work_events の月次レンジパーティション

NFR-PRF-015（5 年 1.5TB 以下）と 7 年保存（NFR-OPS-035）を両立するため、work_events を月次レンジパーティションで管理する。

```sql
-- パーティション親テーブル（概念）
CREATE TABLE work_events (
    event_id UUID,
    case_id UUID NOT NULL,
    ...
    timestamp_server TIMESTAMPTZ NOT NULL
) PARTITION BY RANGE (timestamp_server);

-- 月次パーティション（例: 2026年5月分）
CREATE TABLE work_events_2026_05
    PARTITION OF work_events
    FOR VALUES FROM ('2026-05-01') TO ('2026-06-01');
```

### 2-2. パーティション命名規約

`work_events_{YYYY}_{MM}` 形式。例: `work_events_2026_05`。

### 2-3. パーティション管理

| 作業 | 担当 | タイミング |
|---|---|---|
| 新月パーティション作成 | BAT（自動・`pg_partman` 活用）| 毎月 1 日 00:00（前月末に事前作成）|
| 7 年超パーティションのアーカイブ | BAT-005（バックアップ後 detach）| 年次レビュー時 |
| アーカイブ済みテーブルの読み取り専用保存 | DBA 手動 | 必要に応じて |

### 2-4. パーティション識別子（PART）

| PART-ID | パーティション名（例）| 親テーブル | 期間 |
|---|---|---|---|
| PART-001 | work_events_2026_01 | TBL-001 | 2026-01 |
| PART-002 | work_events_2026_02 | TBL-001 | 2026-02 |
| ...（以降は自動生成）| — | — | — |

## 3. アーカイブ方式

### 3-1. 段階的保存ポリシー

| フェーズ | データ状態 | ストレージ | 検索可否 |
|---|---|---|---|
| 0〜2 年（ホット）| 通常パーティション | SSD（PostgreSQL）| 即時検索可 |
| 2〜5 年（ウォーム）| detach 後に `ALTER TABLE SET TABLESPACE warm_ts` | HDD（別 tablespace）| 検索可（パーティション attach で）|
| 5〜7 年（コールド）| pg_dump + gzip → NAS 保存 | NAS | SQL 実行で参照可 |
| 7 年超 | オフサイト AES-256 暗号化バックアップ | テープ/クラウドストレージ | 要手動リストア |

### 3-2. アーカイブバッチ（BAT-005 連携）

BAT-005（pg_dump 日次）は NAS の `/backups/postgresql/` 以下に `work_events_{YYYY}_{MM}.dump.gz` として月次単位でダンプを出力する。

---

**本節で確定した方針**
- **IDX-001（work_events の case_id + timestamp_server B-Tree）を最優先インデックスとして確定し、NFR-PRF-001（P95 200ms）の主要実現手段とする。**
- **work_events を月次レンジパーティション（`work_events_{YYYY}_{MM}` 命名）で管理し、0〜2 年ホット・2〜5 年ウォーム・5〜7 年コールドの 3 段保存を採用する。**
- **Partial インデックス（status 条件付き）でインデックスサイズを削減し、大量イベント記録時の INSERT パフォーマンスへの影響を最小化する。**
