# 09 データライフサイクル・保持・廃棄

本章の責務は、各テーブルのデータ保持期間・廃棄方法・PII 匿名化・エクスポート手段を確定することである。ALCOA+ Enduring 原則（データが必要な期間中ずっと利用可能であること）とデータオーナーシップ宣言（構想 13 章）を設計レベルで担保する。

---

## 2. PII 匿名化設計

### 2-1. 対象フィールド

TBL-016（users）の以下の列が PII 対象：
- `login_id`（LDAP DN または社員番号）
- `display_name`（氏名）

### 2-2. 匿名化フロー（BAT-004）

```
退職（is_active = FALSE に設定）から CFG-010 日数（デフォルト 60 日）経過後:
  [BAT-004] pii_anonymizer が対象 user_id を取得
  → users.login_id ← SHA-256(user_id + salt)[:32] の hex 文字列
  → users.display_name ← '匿名ユーザー-' + SHA-256(user_id + salt)[:8]
  → users.anonymized_at ← NOW()
  ※ user_id（UUID）は変更しない（WorkEvent の resource FK が壊れないため）
  ※ LOG-009（pii.anonymized）を Append-only ログ（TBL-032 相当）に記録
```

### 2-3. 匿名化後の記録の扱い

- 匿名化後も WorkEvent の `resource` 列には元の `user_id` が残る（UUID は個人を特定できない識別子に変化）
- トレーサビリティ（誰が何をしたか）の観点では `display_name` の復元は不可能になるが、UUID 基のレコード連鎖は維持される
- GDPR 観点での「完全消去権」と「作業記録の整合性維持」の両立を実現する

## 3. エクスポート手段（データオーナーシップ保証）

計画 13 章が確定した「エクスポート 3 手段を全段階で維持」を物理設計に落とす。

| 手段 | 対象 | 実現方法 |
|---|---|---|
| **手段 1: SQL 直接（最低レベル）** | 全テーブル | PostgreSQL の read-only ユーザー（`app_readonly`）を提供。接続情報を IT 担当に開示 |
| **手段 2: CSV エクスポート（中間）** | 主要テーブル | API-reports-002（XES）・SCR-MC-009（帳票出力）から CSV 形式でダウンロード |
| **手段 3: OpenAPI JSON（上位）** | 全 API レスポンス | `/api/v1/openapi.json`（API-system-003）で OpenAPI 3.1 仕様を提供。クライアントが自動化 |

追加: `pg_dump` による完全バックアップを IT 担当が取得可能（BAT-005）。

## 4. 長期保存時の整合性検証

7 年保存データの整合性を維持するため、以下を設計上必須とする。

| 検証 | 担当バッチ | 頻度 |
|---|---|---|
| ハッシュチェーン検証（work_events）| BAT-001 | 週次 |
| 年次復旧テスト（バックアップ復元）| BAT-005 + 手動 | 年次（NFR-AVL-019）|
| 証拠ファイル SHA-256 検証（サンプリング）| 手動（月次レビュー時）| 月次 |

---

**本節で確定した方針**
- **全 Append-only テーブルの保持期間（7 年以上を基準）と廃棄方法（パーティション detach + アーカイブ）を確定し、ALCOA+ Enduring 原則を物理設計レベルで担保した。**
- **PII 匿名化は物理削除ではなく `login_id` と `display_name` の不可逆ハッシュ置換で実現し、`user_id`（UUID）を維持してトレーサビリティの連続性を保つ。**
- **エクスポート 3 手段（SQL 直接・CSV・OpenAPI JSON）を全段階で維持し、データオーナーシップ宣言（構想 13 章）のインフラ設計レベルでの担保を確定した。**
