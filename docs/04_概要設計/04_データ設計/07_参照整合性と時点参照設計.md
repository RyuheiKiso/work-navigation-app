# 07 参照整合性と時点参照設計

本章の責務は、テーブル間の参照整合性（外部キー制約・論理削除との調停）と、ALCOA+ Original 原則の核心である「時点参照」（マスタ変更が過去記録に遡及しない設計）を確定することである。

---

## 1. 時点参照設計（ALCOA+ Original の技術的担保）

### 1-1. 設計原則

作業記録（WorkEvent・WorkExecution 等）は記録時点のマスタ版数（`sop_version_id`）を FK として固定し、その後のマスタ改訂が遡及して記録を変更しない。

```
SOP 版 v1.0 (sop_version_id = V1)  → 2026-04-01 公開
  WorkEvent E1: sop_version_id = V1 ← 永久に V1 を参照する
  WorkEvent E2: sop_version_id = V1
SOP 版 v1.1 (sop_version_id = V2)  → 2026-05-01 公開
  WorkEvent E3: sop_version_id = V2 ← V1.1 以降の記録は V2 を参照
```

E1・E2 は SOP を v1.1 に改訂しても依然として v1.0 の内容で参照される。

### 1-2. 時点参照を実現する設計要素

| テーブル | 時点参照列 | 参照先 | 用途 |
|---|---|---|---|
| TBL-001 work_events | sop_version_id | TBL-004 master_versions | 実施時の SOP 版数を固定 |
| TBL-005 work_executions | sop_version_id | TBL-004 master_versions | 作業セッションの版数を固定 |
| TBL-002 electronic_signs | master_version_id | TBL-004 master_versions | 承認時の対象版数を固定 |
| TBL-008 steps | （版は TBL-004 が管理）| TBL-004 master_versions | 改訂版は新 step_id で分岐 |

### 1-3. PUBLISHED 版の不変性保証

`master_versions.status` が `PUBLISHED` になった後:
- `sops`・`steps`・`step_type_definitions`・`step_flow_rules` の本文列の UPDATE を DB トリガで禁止
- 改訂は必ず新バージョン（新レコード INSERT + 旧版 ARCHIVED）として作成する

---

## 2. 外部キー制約設計

### 2-1. 制約設計の基本方針

| 参照方向 | 方針 | 理由 |
|---|---|---|
| Append-only テーブル → マスタ | `ON DELETE RESTRICT`（FK 制約を設定し、マスタの物理削除を阻止）| Append-only テーブルが参照するマスタを削除させない |
| 更新可テーブル → マスタ | `ON DELETE RESTRICT` | 同上 |
| マスタ → マスタ（階層）| `ON DELETE RESTRICT` | Process → Operation → SOP の階層を保護 |
| CASCADE DELETE | **全テーブルで禁止** | Append-only 原則および ALCOA+ Enduring との矛盾 |

### 2-2. 主要外部キー一覧

| テーブル | 列 | 参照先 | 制約 |
|---|---|---|---|
| work_events | case_id | work_executions | RESTRICT |
| work_events | resource | users | RESTRICT |
| work_events | sop_version_id | master_versions | RESTRICT |
| work_events | terminal_id | devices | RESTRICT |
| work_events | step_id | steps | RESTRICT |
| electronic_signs | signer_id | users | RESTRICT |
| electronic_signs | event_id | work_events | RESTRICT |
| work_executions | sop_id | sops | RESTRICT |
| work_executions | sop_version_id | master_versions | RESTRICT |
| work_executions | primary_worker_id | users | RESTRICT |
| sops | operation_id | operations | RESTRICT |
| operations | process_id | processes | RESTRICT |
| outbox_events | event_id | work_events | RESTRICT |

### 2-3. 論理削除との調停

全マスタは `is_active = FALSE` による論理削除を採用する。`is_active = FALSE` になったマスタへの新規 FK 参照を禁止するため、アプリ層でバリデーションを行う。

```
新規 WorkExecution 作成時:
  → sop.status = 'PUBLISHED' AND is_active = TRUE を確認 (BR-BUS-004)
  → user.is_active = TRUE を確認 (BR-BUS-044)
```

このバリデーションは DB 制約ではなく、アプリ層（MOD-BE 系サービス）で担当する。

## 3. 参照整合性 dry-run（マスタ廃止の安全確認）

計画 19 章（マスタ編集者体験設計）で確定した「参照整合性 dry-run」を物理設計レベルで確定する。

### 3-1. dry-run の動作（API-master-007）

1. 廃止対象マスタの ID を受け取る
2. 下位テーブルへの参照件数を集計するクエリを READ-ONLY トランザクション内で実行
3. 参照件数と参照元テーブル名を JSON で返す（実際の廃止 UPDATE は行わない）
4. フロントエンドが干渉件数を表示し、マスタ管理者が確認後に廃止を確定

```sql
-- dry-run クエリ例（SOP 廃止時）
SELECT
  'work_executions' AS table_name,
  COUNT(*) AS references
FROM work_executions
WHERE sop_id = $1

UNION ALL

SELECT
  'work_events' AS table_name,
  COUNT(*) AS references
FROM work_events we
JOIN work_executions wx ON we.case_id = wx.work_execution_id
WHERE wx.sop_id = $1;
```

### 3-2. 廃止フロー（dry-run 確認後）

1. dry-run で影響件数を確認
2. マスタ管理者が確認・電子サイン（FR-MA-009）
3. `master_versions.status` を ARCHIVED に UPDATE
4. `sops.is_active` を FALSE に UPDATE（新規参照を防ぐ）
5. 既存の Append-only 記録は変更しない（時点参照により過去記録は依然として廃止版を参照可能）

---

## 4. デッドロック回避設計

Append-only テーブルへの INSERT は行ロックを取らないため、通常のデッドロックは発生しない。更新可テーブル（work_executions 等）については以下の方針を採る。

- 更新 SQL は必ず `FOR UPDATE SKIP LOCKED` を使用せず、楽観ロック（`updated_at` の比較）で競合を検知する
- 複数テーブルを同一トランザクション内で更新する場合は、常に TBL-ID の昇順でロックを取得する

---

**本節で確定した方針**
- **時点参照設計（`sop_version_id` FK 固定）によって、マスタ改訂が過去の作業記録に遡及しないことを設計命題として確定した。PUBLISHED 後のマスタ本文 UPDATE を DB トリガで禁止する。**
- **CASCADE DELETE を全テーブルで禁止し、全参照関係を `ON DELETE RESTRICT` に統一することで、意図しない連鎖削除から Append-only データを保護する。**
- **参照整合性 dry-run（API-master-007）によって、廃止操作の影響範囲を事前に確認してからマスタを廃止する安全フローを確定した。**
