# 05 イベントストア設計とハッシュチェーン

本章の責務は、Append-only イベントストア（work_events テーブル）の物理設計と、SHA-256 ハッシュチェーンによる改ざん検知機能の技術的設計を確定することである。これは IPA 7 アーキテクチャ原則 P2（Append-only Event Sourcing）・P5（改竄不可性 3 層保証）・ALCOA+ Consistent/Original/Enduring の実装的担保である。

---

## 1. イベントストアの 3 層保証設計

| 層 | 保証手段 | 設計箇所 |
|---|---|---|
| **第 1 層: DB 権限** | `app_event_writer` ロールに INSERT・SELECT のみ付与。UPDATE/DELETE は権限なし | 本章 §2 |
| **第 2 層: 訂正イベント方式** | 誤記録の修正は `step_rejected` + 再 `step_completed` のイベント追記のみ | 本章 §3 |
| **第 3 層: ハッシュチェーン** | 各 WorkEvent の `prev_hash` 列に直前ブロックの SHA-256 を連鎖。BAT-001（週次検証）が破断を検知 | 本章 §4 |

---

## 2. PostgreSQL ロール設計（DB 権限・第 1 層）

本アプリは PostgreSQL 上で以下の 3 ロールを定義する（MOD-IN-001 が DDL を管理）。

| ロール | 権限 | 用途 |
|---|---|---|
| `app_event_writer` | `INSERT, SELECT` on work_events / evidence_files / measurements / suspensions / electronic_signs / auth_logs / hash_chain_blocks | バックエンドの作業記録処理 |
| `app_read_write` | `INSERT, SELECT, UPDATE` on 更新可テーブル群（work_executions / andon_alerts 等）| バックエンドの通常 CRUD |
| `app_admin` | 全権限（`SUPERUSER` を付与せず `CREATEROLE` のみ）| マイグレーション・ユーザー管理 |

ロール名はアプリ設定ファイル（`CFG-NNN`）で注入し、バックエンド接続文字列に直接埋め込まない。

```sql
-- work_events への UPDATE/DELETE 禁止の確認例（権限チェック DDL）
REVOKE UPDATE, DELETE ON work_events FROM PUBLIC;
REVOKE UPDATE, DELETE ON work_events FROM app_event_writer;
REVOKE UPDATE, DELETE ON work_events FROM app_read_write;
-- app_admin のみが DDL 変更可能（スキーマ移行用）
```

---

## 3. 訂正イベント方式（第 2 層）

作業ログの修正は以下のフローに統一する。`UPDATE` は一切使用しない。

```
1. 誤記録発見（監督または品質担当）
2. step_rejected イベントを追記（event_type = 'step_rejected'）
   payload.rejection_reason = "誤入力: 数値 XXX → 正しくは YYY"
   payload.original_event_id = {修正対象の event_id}
3. 正しい値で step_completed イベントを再追記
4. 電子サイン（ElectronicSign）を step_rejected と再 step_completed に付与
5. CAPA 起票（FR-KZ-005）は任意（誤記録の重大度に応じて）
```

この手順により、誰が・いつ・何を修正したかがイベント列から完全に追跡可能になる。

## 4. SHA-256 ハッシュチェーン設計（第 3 層）

### 4-0. per-case_id genesis 採用方針（ADR-007）

**per-case_id genesis 採用（ADR-007）**: ハッシュチェーンは case_id ごとに独立したチェーンとして管理する。
グローバルな単一 genesis は採用しない。各 case_id の最初のイベントが genesis（prev_hash = [0u8;32]）となる。

- 各 case_id（作業セッション）のチェーンは相互に独立しており、あるケースの破断が他のケースに影響しない。
- 週次検証バッチ（BAT-001）は case_id ごとに prev_hashes マップを保持してチェーンを独立追跡する。
- グローバルな genesis ブロックは存在しない。factory_id を起点とする単一チェーンは採用しない（ADR-007）。

### 4-1. チェーン構造

各 WorkEvent は直前の WorkEvent のハッシュを `prev_hash` 列に保持する。

```
event_id = E1
prev_hash = "genesis" (チェーン開始点)
content_hash = SHA-256(E1 全列の正規化 JSON)

event_id = E2
prev_hash = content_hash of E1  ← E1 のハッシュを保持
content_hash = SHA-256(E2 全列 + E1.content_hash)

event_id = E3
prev_hash = content_hash of E2
...
```

### 4-2. ハッシュ計算対象列

ハッシュ計算対象: `event_id` + `case_id` + `activity` + `timestamp_client` + `timestamp_server` + `resource` + `sop_version_id` + `payload`（JSONB の canonical 形式）+ `terminal_id` + `prev_hash`

正規化形式: JSON keys をアルファベット昇順ソートした UTF-8 バイト列の SHA-256。

### 4-3. HashChainBlock（TBL-031）の設計

週次でハッシュチェーンの最末尾を HashChainBlock テーブルに記録する（BAT-001）。

| 属性名 | 型 | 説明 |
|---|---|---|
| block_id | UUID v7 | PK |
| block_period | DATE | 集計週（月曜 00:00 〜 日曜 23:59:59）|
| event_count | INTEGER | 期間内の WorkEvent 件数 |
| last_event_id | UUID | 期間内の最終 event_id |
| last_content_hash | CHAR(64) | 期間内の最終 content_hash（チェーン継続点）|
| created_at | TIMESTAMPTZ | ブロック作成時刻 |
| block_hash | CHAR(64) | このブロック自体のハッシュ（block_period + last_content_hash）|

### 4-4. 検証バッチ（BAT-001）の動作

```
1. 前回 HashChainBlock の last_content_hash を読み取る
2. 対象期間の work_events を timestamp_server 順に全件取得
3. 各 WorkEvent の prev_hash が直前 WorkEvent の content_hash と一致するか検証
4. 不一致発見 → ERR-DB-003（hash_chain_broken）を発生させ LOG-007（SECURITY）を記録
5. 問題なければ新 HashChainBlock を TBL-031 に INSERT し SEQ-007 のシーケンスを完了
```

## 5. XES 互換性の確認

work_events テーブルは IEEE 1849（XES）の 5 必須属性（+ 本プロジェクト追加 3 属性）を持つ。

| XES 属性 | work_events 列 | マッピング |
|---|---|---|
| Case ID | case_id | WorkInstance を表す UUID |
| Activity | activity | step_completed 等 9 種の列挙 |
| Timestamp | timestamp_server | UTC ISO 8601 |
| Resource | resource（user_id）| 実施者 |
| （拡張）| payload | activity 固有データ（JSONB）|
| （追加: ハッシュ）| prev_hash + content_hash | 改ざん検知 |
| （追加: オフライン）| is_offline + sync_lag_ms | ALCOA+ Contemporaneous 補足 |
| （追加: 端末）| terminal_id | Attributable 担保 |

XES エクスポート（FR-AU-005・RP-002）時には work_events を XES 1.0 XML 形式に変換する。変換仕様は `05_外部インターフェース設計/04_Outbox実績送信API.md` および `06_帳票設計/04_RP-002` で確定する。

---

**本節で確定した方針**
- **Append-only 物理保証（DB ロール分離）・訂正イベント方式・SHA-256 ハッシュチェーンの 3 層で work_events の改ざん不可性を確定する。UPDATE/DELETE 権限を持つロールを app_event_writer から完全に排除する。**
- **ハッシュ計算対象列を確定し、週次 HashChainBlock（TBL-031）で long-term な整合性証拠を生成することを BAT-001 の責務として確定する。**
- **XES（IEEE 1849）互換 8 属性を work_events テーブルの物理列に 1:1 でマッピングし、RP-002（XES エクスポート）の直接変換を可能にする。**
