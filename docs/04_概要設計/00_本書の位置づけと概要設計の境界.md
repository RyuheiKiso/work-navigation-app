# 00 本書の位置づけと概要設計の境界

本章の責務は、概要設計書（04_概要設計）全体のIPA共通フレーム2013における位置を確定し、上流（要件定義）と下流（詳細設計）との境界、および本書全章に適用される言語規約・禁止事項を権威的に定めることである。本章に記載した規約は本書の全章・全サブに適用される権威規約であり、下位章での再解釈を禁止する。

## 1. IPA 共通フレーム 2013 上の位置

### 1-1. 開発プロセス全体の中での概要設計

IPA 共通フレーム 2013（SLCP-JCF2013）は、システム開発をプロセス群として定義する。本書は以下の 2 プロセスに対応する。

| プロセス | 対応タスク | 担当サブ |
|---|---|---|
| **2.3 システム方式設計プロセス** | サブシステム分割・HW/SW配分・配置設計・ネットワーク設計・可用性設計・セキュリティ方式設計・運用方式設計・移行方式設計・評価 | 01・07・08・09 |
| **2.4 ソフトウェア方式設計プロセス** | ソフトウェアアーキテクチャ設計・コンポーネント定義・データ構造設計・ヒューマンインタフェース設計・インタフェース設計・出力設計・評価 | 02・03・04・05・06・10 |

本書（04_概要設計）は開発プロセス全体において「第四層」に位置する。

| 層 | プロセス | 言語 | 主な問い | 文書 |
|---|---|---|---|---|
| 第一層 | 企画プロセス（構想） | べき論（規範的命題） | Why — なぜ作るか・解決すべき課題・倫理スタンス | `02_企画/システム化構想/` |
| 第二層 | 企画プロセス（計画） | する論（確定命題） | What/How-Frame — 対象範囲・機能スコープ・アーキテクチャ原則・技術スタック | `02_企画/システム化計画/` |
| 第三層 | 要件定義プロセス | する論（検証可能命題） | What 詳細 — 受入基準付き機能要件・非機能要件・ユースケース | `03_要件定義/` |
| **第四層** | **システム方式設計プロセス / ソフトウェア方式設計プロセス** | **する論（設計命題）** | **How 骨格 — 画面構成・API設計・DB物理スキーマ骨格・コンポーネント分割** | **本書（04_概要設計/）** |

## 3. 下流（詳細設計以降）への引き渡し境界

### 3-1. 本書が確定し下流に引き渡す事項

本書（概要設計）が確定し、`05_詳細設計/` 以降へ引き渡す事項：

- **FR-ID × API-ID 対応**: 各機能要件を実現する API エンドポイントの責任割当
- **FR-ID × TBL-ID 対応**: 各機能要件のデータ操作とテーブルの対応（CRUD 単位）
- **UC-ID × SEQ-ID 対応**: 主要ユースケースのシーケンス図（API 呼び出し順・トランザクション境界）
- **UC-ID × SCR-ID × TRN-ID 対応**: ユースケースを実現する画面遷移
- **コンポーネント構成（MOD-NNN 一覧・依存関係）**
- **テーブル物理スキーマ骨格（列名・型・制約・インデックス方針）**
- **認証認可方式（JWT 構造・RBAC 判定アルゴリズム骨格）**
- **エラーハンドリング統一方式（ERR-NNN × HTTP レスポンス対応）**

### 3-2. 本書が確定しない事項（詳細設計以降に委任）

以下は本書の設計命題に含まれない。詳細設計（`05_詳細設計/`）に委任する。

- Rust 関数/メソッドの引数・戻り値・型定義の詳細
- SQL DDL の詳細（インデックス列の指定・JSONB スキーマの詳細・デフォルト式の詳細）
- React Component の Props 型定義・状態管理実装（useState/Context の使い方）
- JSON Logic 条件式の具体的な実装例
- 単体テストのケース設計（テスト方式のみ本書で確定）
- OpenAPI スキーマの全フィールドと制約（API エンドポイント責任割当のみ本書で確定）

## 4. 本書全章の言語規約

### 4-1. 使用する言語様式

本書は**する論（設計命題）**で記述する。設計命題は「受入可能な実装パターンが具体的に想定できる言明」である。

| 許可 | 例 |
|---|---|
| 設計命題 | 「バックエンドは Rust の `wnav_api` crate が axum を用いてルーティングを担当する」 |
| 設計命題（禁止の確定） | 「CASCADE DELETE は対象外と判断する。論理削除（`is_active = false`）に統一する」 |
| 設計命題（方式の選択） | 「認証方式は JWT RS256 に準拠する。有効期限は 8 時間とする」 |

| 禁止 | 理由 |
|---|---|
| 「可能性がある」「場合がある」 | 設計命題は一意に確定する必要がある |
| 「考慮する」「検討する」 | 検討の結論が命題に含まれていない |
| 「なるべく準拠する」「準拠を目指す」 | 部分準拠の範囲が未確定 |
| 「適宜対応する」「参考にする」 | 実装者の恣意に委ねる不完全な設計 |
| 「べき」「〜したい」「望ましい」 | 規範的言明は構想書の言語であり本書には不適切 |

### 4-2. 規制・標準への言及の三段階

| 表現 | 意味 | 例 |
|---|---|---|
| 「準拠する」 | 当該標準の要件を設計に完全に組み込む | 「XES（IEEE 1849）に準拠する」 |
| 「対応する」 | 当該標準の主要要素を設計に取り込む（全要件ではない） | 「WCAG 2.1 AA に対応する」 |
| 「対象外と判断する」 | 当該標準またはその一部を本プロジェクトの設計範囲外とする（根拠を必ず付記） | 「MFA（多要素認証）は対象外と判断する（社内 LAN 完結・中小製造業 IT 担当 1 名運用のため）」 |

## 5. 概要設計完成の定義（妥協禁止ポリシーの操作的定義）

プロジェクトポリシー「妥協は禁止する。完璧かつ至高を目指す。」に基づき、本書の完成条件を以下に定める。

| 完成条件 | 検証方法 |
|---|---|
| 全 md ファイル（試算約 130 件）が章末「本節で確定した方針」「参照業界分析」節を含む | `grep -L "本節で確定した方針"` でゼロ件 |
| 全図ファイル（試算約 50〜70 ペア）が `img/` 配下に `.drawio` と `.svg` の両方で存在 | `find img/ -name "*.drawio"` と `find img/ -name "*.svg"` の件数が一致 |
| 全 `img/` の `.drawio` が `drawio-lint` を ERROR ゼロで通過 | lint スクリプト実行 |
| 全 `img/` の `.svg` が `svg-postcheck` を ERROR ゼロで通過 | postcheck スクリプト実行 |
| 全 86 FR が DTM M1（FR×API）に出現（Won't を除く） | DTM M1 の FR 列を要件 RTM と突合 |
| 全 27 EN が物理テーブル TBL カタログにマッピング済 | TBL カタログの EN-NNN 列でゼロ件の空白なし |
| 全 35 SCR が DTM M3（UC×SCR）に出現 | DTM M3 の SCR 列を機能要件 09 章と突合 |
