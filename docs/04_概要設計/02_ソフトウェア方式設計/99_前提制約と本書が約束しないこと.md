# 99 前提制約と本書が約束しないこと

本章の責務は、ソフトウェア方式設計サブが「対象外と判断する」事項を宣言し、ルート 99 章の一次宣言を本サブの設計観点から補強することである。

---

## 1. eval / 動的コード実行（永続的禁止）

**JavaScript の `eval()` / `new Function()` / Rust の proc_macro 外での動的コード生成は対象外と判断する。**

- 理由: P7 原則・セキュリティリスク（OWASP A01）
- 代替: JSON Logic（Apache 2.0）による条件評価（§04）
- 担保: コードレビューでの静的解析・ESLint の `no-eval` ルール必須

---

## 2. マイクロサービス分割（ver1.0.0 対象外）

**各機能を独立したマイクロサービスに分割することは対象外と判断する。**

- 理由: IT 担当 1 名運用前提・cargo workspace モノリポが中小規模に最適
- 代替: Rust cargo workspace 内の crate 分割でモジュール境界を維持

---

## 3. DLL / DSO 動的読み込み（対象外）

**バイナリ DLL / .so の実行時動的読み込みによるプラグインは対象外と判断する。**

- 理由: セキュリティ・ビルド複雑性・プラットフォーム互換性
- 代替: JSON スキーマによる Step タイプ定義・cargo feature flags による静的コンパイル

---

## 4. WASM ランタイム（ver1.0.0 対象外）

**WebAssembly（WASM）をプラグインランタイムとして使用することは対象外と判断する。**

- 理由: 開発・デバッグの複雑性がスコープを超える
- 将来: Enterprise 版でのカスタム Step 拡張時に検討余地あり

---

## 5. GraphQL / gRPC（対象外）

**GraphQL または gRPC による API 提供は対象外と判断する。**

- 理由: REST + OpenAPI 3.1（既確定）で全要件を満たす。クライアント多様性（RN / React / 将来の外部連携）に対して REST が最も相互運用性が高い
- 担保: `05_外部インターフェース設計/02_REST_API設計方針.md` に明示

---

## 6. ホットリロード / ライブパッチ（対象外）

**実行中プロセスへのコードパッチ（ホットリロード・ライブパッチ）は対象外と判断する。**

- 理由: ALCOA+ Original / Consistent との矛盾・監査証跡の完全性毀損リスク
- 代替: Blue/Green デプロイ（将来検討）または定期メンテナンス停止（現 ver1.0.0 方針）

---

## 7. 詳細設計への委任事項

| 委任事項 | 委任先 |
|---|---|
| Rust 関数の引数・戻り値・型定義 | 05_詳細設計 |
| React Component の Props 型・hooks 実装 | 05_詳細設計 |
| SQLite TypeORM マイグレーション詳細 | 05_詳細設計 |
| JSON Logic バリデータの具体的な実装例 | 05_詳細設計 |
| Docker Compose のネットワーキング詳細 | 06_実装 |

---

**本節で確定した方針**
- **eval・DLL 動的読み込み・WASM ランタイム・マイクロサービス分割・GraphQL/gRPC・ホットリロードを対象外として宣言し、ver1.0.0 スコープをソフトウェア設計観点から明確化した。**
- **全禁止事項について代替手段を示し、設計意図の理解を容易にした。**
