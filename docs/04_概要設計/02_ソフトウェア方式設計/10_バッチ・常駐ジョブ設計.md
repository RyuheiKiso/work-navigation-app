# 10 バッチ・常駐ジョブ設計

本章の責務は、BAT 識別子（BAT-001〜011）で管理するバッチ・常駐ジョブの実行基盤・スケジューリング・冪等性設計を確定することである。

---

## 1. バッチ実行基盤

### 1-1. Rust バックエンドの常駐 tokio task

長時間稼働するジョブ（Outbox Consumer・WAL アーカイブ等）は Rust の tokio 非同期タスクとして実装する。

```rust
// wnav_terminal_api/src/main.rs
// BAT-002（outbox_dispatcher）・BAT-003（master_sync）・BAT-008（webhook_retry）を内包
#[tokio::main]
async fn main() {
    // 端末 API サーバー（ポート 8080 / DBロール: app_event_insert）
    let api_handle = tokio::spawn(serve_terminal_api(app_state.clone()));

    // Outbox Dispatcher（BAT-002 / 常駐 60s ループ）
    let outbox_handle = tokio::spawn(
        outbox_dispatcher::run(app_state.clone())
    );

    // Master Sync Puller（BAT-003 / 毎 60 分 cron）
    let master_sync_handle = tokio::spawn(
        master_sync_puller::run(app_state.clone())
    );

    // Webhook Retry Scheduler（BAT-008 / 毎 1 分 cron）
    let webhook_retry_handle = tokio::spawn(
        webhook_retry_scheduler::run(app_state.clone())
    );

    tokio::select! {
        _ = api_handle => {},
        _ = outbox_handle => {},
        _ = master_sync_handle => {},
        _ = webhook_retry_handle => {},
    }
}
```

```rust
// wnav_master_api/src/main.rs
// BAT-001（hash_chain_verifier）および管理系 BAT を内包
#[tokio::main]
async fn main() {
    // 管理 API サーバー（ポート 8081 / DBロール: app_write）
    let api_handle = tokio::spawn(serve_master_api(app_state.clone()));

    // HashChain 検証（BAT-001 / 週次 cron 月曜 02:00）
    let hash_verify_handle = tokio::spawn(
        hash_chain_verifier::run(app_state.clone())
    );

    // PII 匿名化（BAT-004 / 日次 03:00）
    let pii_handle = tokio::spawn(pii_anonymizer::run(app_state.clone()));

    // バックアップ（BAT-005/006）・帳票ハッシュ（BAT-007）・
    // 証明書通知（BAT-009）・JWT キーローテーション（BAT-010）・
    // リワークコスト集計（BAT-011）は同様に tokio::spawn で起動

    tokio::select! {
        _ = api_handle => {},
        _ = hash_verify_handle => {},
        _ = pii_handle => {},
        // 他バッチハンドル...
    }
}
```

### 1-2. cron スケジューリング

cron 式で実行するジョブは `tokio-cron-scheduler` を使用し、各 API バイナリの起動時に `tokio::spawn` で常駐タスクとして内包する。`wnav-scheduler` という別コンテナは廃止した。スケジューラ責務の内包先は BAT-ID ごとに以下の通り確定する。

| 内包バイナリ | BAT-ID | 理由 |
|---|---|---|
| `wnav_terminal_api` | BAT-002, BAT-003, BAT-008 | 端末イベント送信・マスタ同期・Webhook リトライはすべて端末 API プロセスが所有するデータを扱う |
| `wnav_master_api` | BAT-001, BAT-004, BAT-005, BAT-006, BAT-007, BAT-009, BAT-010, BAT-011, BAT-013 | HashChain 検証・PII 匿名化・バックアップ・帳票ハッシュ・証明書通知・JWTキーローテーション・リワークコスト集計・Case ロック解放（reaper）は管理側プロセスで集中管理する |

## 2b. BAT-013 詳細（case_lock_reaper）

| 項目 | 内容 |
|---|---|
| 担当バイナリ | `wnav_master_api` 内 tokio task |
| 実行契機 | cron（60 秒間隔） |
| 処理概要 | case_locks.heartbeat_at が現在時刻から 5 分（300s）以上古いかつ lock_status='ACTIVE' のレコードを lock_status='EXPIRED' に UPDATE する。監査記録として work_events に case_lock_expired イベントを INSERT する |
| 関連 FR | FR-SY-011 |
| ステータス | Draft |

## 3. 冪等性設計の統一パターン

全バッチの冪等性は以下のいずれかで担保する。

| パターン | 適用バッチ | 実装例 |
|---|---|---|
| 実行記録確認 | BAT-001/005/007/009 | `IF NOT EXISTS (SELECT 1 FROM batch_logs WHERE batch_id = 'BAT-001' AND period = 'week-2026-20')` |
| タイムスタンプ比較 | BAT-003/004 | `WHERE last_sync_at < NOW() - INTERVAL '60 minutes'` |
| Idempotency Key | BAT-002/008 | `outbox_events.idempotency_key` の重複確認 |
| WAL LSN | BAT-006 | `pg_current_wal_lsn()` との比較 |

## 4. ジョブ実行失敗時の自動回復方針

### 4-1. 再試行ポリシー（BAT 種別ごと）

| BAT-ID | 自動再試行 | 最大試行 | 人手介入条件 |
|---|---|---|---|
| BAT-001 | なし（週次再実行）| 1 | 3 週連続失敗 |
| BAT-002 | 指数バックオフ | CFG-002（5）| DLQ 移行後 |
| BAT-003 | 次回 cron 実行 | — | 24h 以上失敗継続 |
| BAT-004 | 翌日 cron 実行 | — | — |
| BAT-005 | 6h 後リトライ | 3 | 復旧不可時 |
| BAT-006 | 自動（WAL daemon）| — | WAL 容量超過時 |
| BAT-007 | 手動（API-ops-002）| — | IT 担当判断 |
| BAT-008 | 指数バックオフ | 5 | DLQ 移行後 |

---

**本節で確定した方針**
- **BAT-001〜011・BAT-013 の全バッチを確定し、実行契機・冪等性担保方式・失敗時処理・担当バイナリを設計命題として確定した。**
- **`wnav-scheduler` 別コンテナを廃止し、全バッチを各 API バイナリ内の tokio バックグラウンドタスクとして内包する。BAT-002/003/008 は `wnav_terminal_api` に、BAT-001 および集計・管理系 BAT（BAT-013 含む）は `wnav_master_api` に内包する。**
- **長時間稼働ジョブ（BAT-002 Outbox Dispatcher・BAT-006 WAL）は Rust tokio task として API サーバーと同一プロセスで起動し、`tokio::select!` でライフサイクルを管理する。**
- **全バッチは冪等性を担保し、二重実行が発生しても安全に動作する設計を確定した。**
- **BAT-013（case_lock_reaper）を新規追加し、heartbeat_at が 5 分超過した ACTIVE ロックを EXPIRED 化する自動解放機構を確定した（FR-SY-011 / ADR-009）。**
