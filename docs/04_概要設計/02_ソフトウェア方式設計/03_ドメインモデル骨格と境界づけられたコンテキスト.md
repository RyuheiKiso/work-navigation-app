# 03 ドメインモデル骨格と境界づけられたコンテキスト

本章の責務は、27 論理エンティティ（EN-001〜027）を 5 つの境界づけられたコンテキスト（Bounded Context）に配置し、各コンテキストの責務と相互関係を確定することである。また、5 種の状態機械（WorkInstance・MasterVersion・OutboxEvent・CAPA・AndonAlert）のソフトウェア設計レベルでの表現を確定する。

図: fig_des_arch_bounded_context（img/ 配下）を参照。

---

## 1. 5 つの境界づけられたコンテキスト

| コンテキスト | 英名 | 担当エンティティ | 主要責務 |
|---|---|---|---|
| **Authoring** | Authoring Context | EN-005/006/007/008/009/010/029/030（SOP 階層・版数・フロー定義）| SOP マスタの作成・編集・承認・公開 |
| **Execution** | Execution Context | EN-011/012/016/023/024（作業実行・イベント・中断・端末）| Step 実行・進捗管理・Offline-First |
| **Evidence** | Evidence Context | EN-013/014/015（証拠・測定・電子サイン）| 証拠取得・SHA-256 ハッシュ・電子サイン |
| **Quality** | Quality Context | EN-017/018/019/020/025（アンドン・不適合・CAPA・改善・HashChain）| 品質異常対応・Kaizen・監査証跡 |
| **Integration** | Integration Context | EN-021/022/027/028（Outbox・AuthLog・外部キー・WorkPattern）| 同期・認証ログ・親機連携 |

ユーザー（EN-001〜004）はすべてのコンテキストで参照される横断エンティティ（Shared Kernel）として定義する。

---

## 2. コンテキスト間の関係と通信方式

| From → To | 関係 | 通信方式 |
|---|---|---|
| Authoring → Execution | Supplier-Customer | Execution が Authoring の SOP 版数を ID 参照（時点参照）|
| Execution → Evidence | Supplier-Customer | Evidence が Execution の WorkEvent に紐づく |
| Execution → Integration | Supplier-Customer | Integration（Outbox）が Execution の WorkEvent を受け取り同期 |
| Evidence → Quality | Supplier-Customer | Quality（CAPA 等）が Evidence を参照 |
| Execution → Quality | Supplier-Customer | AndonAlert が WorkExecution を参照 |
| Authoring ← Integration | — | Integration（外部キーマッピング）が Authoring の SOP を解決 |

コンテキスト間の通信は必ず **ID（UUID）による参照**で行い、オブジェクトのメモリ直接参照は禁止する。

---

## 3. 状態機械のソフトウェア表現

状態機械は Rust では `enum` + `impl From<State>` パターンで、React Native/React では `XState v5` または `useReducer` + `switch` で実装する。

### 3-1. WorkInstance 状態機械

```rust
// Rust での表現（Domain Layer）
#[derive(Debug, Clone, PartialEq)]
pub enum WorkExecutionStatus {
    NotStarted,
    InProgress,
    Suspended,
    Completed,
    Cancelled,
}

impl WorkExecutionStatus {
    pub fn can_transition_to(&self, next: &Self) -> bool {
        matches!(
            (self, next),
            (Self::NotStarted, Self::InProgress)
            | (Self::InProgress, Self::Suspended)
            | (Self::InProgress, Self::Completed)
            | (Self::InProgress, Self::Cancelled)
            | (Self::Suspended, Self::InProgress)
        )
    }
}
```

### 3-2. MasterVersion 状態機械

```rust
pub enum MasterVersionStatus {
    Draft,
    UnderReview,
    Published,
    Archived,
}
```

- `Published` への遷移は `ElectronicSign` + `quality_admin` ロール必須（BR-BUS-012）
- `Published` から `Archived` への遷移は改訂版公開時のみ
- `Draft` / `UnderReview` への逆遷移は不可

### 3-3. OutboxEvent 状態機械

```rust
pub enum OutboxEventStatus {
    Pending,
    Sending,
    Sent,
    Failed { retry_count: u8 },
    Dlq { reason: String },
}
```

### 3-4. CAPA 状態機械・AndonAlert 状態機械

CAPA: `Open → Investigating → CorrectiveAction → Verification → Closed`
AndonAlert: `Alerting → Acknowledged → Resolved`

## 4. ドメインイベント（内部通知）

コンテキスト間の疎結合を実現するため、以下のドメインイベントを定義する。

| ドメインイベント | 発生元コンテキスト | 消費コンテキスト | 用途 |
|---|---|---|---|
| `WorkStepCompleted` | Execution | Integration（Outbox 追加）| 完了イベントの Outbox 登録 |
| `WorkInterrupted` | Execution | Quality（アンドン連動）| 中断時の異常検知 |
| `MasterVersionPublished` | Authoring | Integration（同期キュー更新）| 新版マスタの端末配信トリガー |
| `EvidenceRecorded` | Evidence | Integration（Outbox 追加）| 証拠の Outbox 登録 |

ドメインイベントは同一プロセス内のインメモリチャネル（tokio channel）で配信する。外部配信は MSG-004/005（internal.*）を使用する。

---

**本節で確定した方針**
- **5 コンテキスト（Authoring/Execution/Evidence/Quality/Integration）への EN 配置を確定し、コンテキスト間通信は UUID 参照のみ（オブジェクト参照禁止）とする。**
- **5 種の状態機械を Rust の `enum` で表現し、`can_transition_to` メソッドで状態遷移の合法性をドメイン層でバリデーションする。**
- **ドメインイベント 4 種（WorkStepCompleted 等）を定義し、コンテキスト間の疎結合をインメモリチャネルで実現する。**
- **IQC BC および Rework BC を新設し、既存 Quality BC（NonConformity 集約）から分離する。各 BC は UUID 参照のみで連携し、直接依存を禁止する。**

---

## 5. 新規バウンデッドコンテキスト（IQC/Rework）

IQC および Rework の 2 つの新しい BC を追加する。これらは既存 Quality BC の NonConformity 集約とは明確に分離され、入荷品質管理とリワーク処理の専門ロジックをカプセル化する。

### 5-1. IQC BC（wnav_iqc）

| 項目 | 内容 |
|---|---|
| コンテキスト名 | IQC Context |
| 英名 | IQC Bounded Context |
| モジュール識別子 | MOD-BE-iqc-001 |
| ソースパス | src/bounded_contexts/iqc/ |
| 担当エンティティ | EN-028（Material）・EN-029（Supplier）・EN-030（IncomingInspection）・EN-031（SamplingPlan）|
| サポートエンティティ | TBL-036〜042（materials / suppliers / incoming_inspections / sampling_plans / incoming_inspection_measurements / concession_approvals / lot_qc_states）|
| 主要責務 | 入荷ロット管理・AQL 判定・検査厳しさ自動切替（普通/強化/緩和）・特採承認フロー・選別サブロット発行 |
| 分離の根拠 | 入荷検査は後工程の Execution BC への「ロットゲート」として機能する。NonConformity（製造中不適合）とは発生起点・承認者・法規制根拠が異なるため、同一 BC に混在させると境界が曖昧になる |

**主要集約（Aggregate）**

| 集約ルート | エンティティ | 責務 |
|---|---|---|
| `IncomingInspection` | TBL-038 + TBL-040 + TBL-041 + TBL-042 | ロット受入から AQL 判定までの一連の状態を管理する |
| `SamplingPlan` | TBL-039 | AQL 値・検査水準・JSONB スナップショットを版管理する |
| `Material` | TBL-036 | 材料コード・材料種別を版管理する |
| `Supplier` | TBL-037 | 仕入先コード・連絡先・取引品質実績リンクを版管理する |

**状態機械: IncomingInspection**

```rust
pub enum IqcStatus {
    Pending,           // 受入登録済・未検査
    InspectionInProgress, // サンプル測定中
    Passed,            // AQL 判定: 合格
    Failed,            // AQL 判定: 不合格
    ConditionalPass,   // 特採承認済
    Screening,         // 選別中（サブロット発行済）
}

impl IqcStatus {
    pub fn can_transition_to(&self, next: &Self) -> bool {
        matches!(
            (self, next),
            (Self::Pending, Self::InspectionInProgress)
            | (Self::InspectionInProgress, Self::Passed)
            | (Self::InspectionInProgress, Self::Failed)
            | (Self::Failed, Self::ConditionalPass)
            | (Self::Failed, Self::Screening)
        )
    }
}
```

**コンテキスト間の関係**

| From → To | 関係 | 通信方式 |
|---|---|---|
| IQC → Execution | ロットゲート | Execution は IncomingInspection の `qc_status = Passed or ConditionalPass` を ID 参照で確認する。直接オブジェクト参照禁止 |
| IQC → Quality（NonConformity）| 参照のみ | 不合格ロットを NonConformity として登録する際、`incoming_inspection_id` を外部キーとして渡す（逆参照不可）|
| IQC → Integration（Outbox）| Supplier-Customer | `IqcJudged` ドメインイベントを Outbox に登録し、外部親機へ実績送信する |

**ドメインイベント**

| ドメインイベント | 発生元 | 消費先 | 用途 |
|---|---|---|---|
| `IqcJudged` | IQC Context | Integration（Outbox 追加）| 合否判定完了の外部通知 |
| `LotQcStateChanged` | IQC Context | Execution Context | 後工程ロットゲート更新 |
| `ConcessionApproved` | IQC Context | Quality Context | 特採承認結果の NonConformity 紐づけ |

### 5-2. Rework BC（wnav_rework）

| 項目 | 内容 |
|---|---|
| コンテキスト名 | Rework Context |
| 英名 | Rework Bounded Context |
| モジュール識別子 | MOD-FE-HA-002（フロントエンド）・MOD-FE-MC-003/004（管理コンソール）|
| ソースパス（ハンディ）| src/features/rework-flow/ |
| 担当エンティティ | EN-032（Rework）・EN-033（Disposition）・EN-034（ReworkVerification）・EN-035（ReworkSopMapping）・EN-036（ReworkedLotLabel）・EN-037（ReworkCostRecord）・EN-038（ScrapRecord）・EN-039（ReturnToVendorRecord）|
| サポートエンティティ | TBL-043〜050（reworks / dispositions / rework_verifications / rework_sop_mapping / reworked_lot_labels / rework_cost_records / scrap_records / return_to_vendor_records）|
| 主要責務 | ディスポジション判定（二者電子サイン）・リワーク作業実施・修正品 QR 発行・再検査・廃却処理（Append-only）・仕入先返却 |
| 分離の根拠 | リワークは NonConformity（不適合記録）の後続工程だが、独自の SOP マッピング・コスト記録・廃却・返却という専用の集約ライフサイクルを持つため、NonConformity BC 内に混在させると集約の凝集度が低下する |

**主要集約（Aggregate）**

| 集約ルート | エンティティ | 責務 |
|---|---|---|
| `Rework` | TBL-043 + TBL-045 + TBL-047 | リワーク案件（case_id）の状態と修正品 QR 発行を管理する |
| `Disposition` | TBL-044 | 二者電子サインによる廃却/修正/返却の意思決定を Append-only で記録する |
| `ScrapRecord` | TBL-049 | 廃却処理と立会者電子サインを Append-only で記録する |
| `ReturnToVendorRecord` | TBL-050 | 仕入先返却と追跡番号を Append-only で記録する |
| `ReworkCostRecord` | TBL-048 | 日次バッチ（BAT-011）によるコスト集計値を管理する |

**状態機械: Rework**

```rust
pub enum ReworkStatus {
    Pending,       // ディスポジション決定待ち
    InProgress,    // リワーク作業中
    Verifying,     // 再検査中
    Passed,        // 再検査合格
    Scrapped,      // 廃却済
    Returned,      // 仕入先返却済
}

impl ReworkStatus {
    pub fn can_transition_to(&self, next: &Self) -> bool {
        matches!(
            (self, next),
            (Self::Pending, Self::InProgress)
            | (Self::InProgress, Self::Verifying)
            | (Self::Verifying, Self::Passed)
            | (Self::Verifying, Self::InProgress)  // 再リワーク（CFG-026 上限あり）
            | (Self::Pending, Self::Scrapped)
            | (Self::Pending, Self::Returned)
        )
    }
}
```

**コンテキスト間の関係**

| From → To | 関係 | 通信方式 |
|---|---|---|
| Quality（NonConformity）→ Rework | Supplier-Customer | NonConformity の `nonconformity_id` を Rework の外部キーとして渡す。Rework 側から NonConformity への逆参照は禁止 |
| Rework → Evidence | Supplier-Customer | 前後写真・修正品 QR は Evidence Context に登録し、`rework_id` で紐づける |
| Rework → Integration（Outbox）| Supplier-Customer | `ReworkCompleted` / `ReworkScrapped` ドメインイベントを Outbox に登録し、外部親機へ実績送信する |
| Rework → IQC | 参照のみ | 修正品の再検査結果を IQC の `incoming_inspection_id` で参照する（将来拡張）|

**ドメインイベント**

| ドメインイベント | 発生元 | 消費先 | 用途 |
|---|---|---|---|
| `ReworkCompleted` | Rework Context | Integration（Outbox 追加）| 再検査合格の外部通知 |
| `ReworkScrapped` | Rework Context | Integration（Outbox 追加）| 廃却処理の外部通知 |
| `DispositionDecided` | Rework Context | Quality Context | ディスポジション決定結果の NonConformity 紐づけ |

**セキュリティ制約**

| 制約 | 内容 | 関連設定 |
|---|---|---|
| 二者サイン強制 | ディスポジション判定は quality_admin と supervisor の異なる 2 名の電子サインを必須とする | NFR-SEC-048, ERR-BIZ-021 |
| 再検査者分離 | リワーク作業者と再検査者は同一 worker_id を禁止する | CFG-025, ERR-BIZ-023 |
| 廃却立会者分離 | 廃却処理者と立会者は同一 worker_id を禁止する | ERR-BIZ-024 |
| リワーク上限 | 同一ロットのリワーク回数は CFG-026 で設定した上限（デフォルト 3 回）を超えられない | CFG-026, ERR-BIZ-022 |
