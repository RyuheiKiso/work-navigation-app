## 1. バックエンド（Rust）のレイヤー構成

### 1-1. 4 層アーキテクチャ

バックエンドは2つの独立したバイナリクレートに分割する。`wnav_terminal_api`（ハンディ端末 API）と `wnav_master_api`（管理 API）はそれぞれ独立してビルド・デプロイされ、異なる PostgreSQL ロールで動作する。

**wnav_terminal_api レイヤー構成（ハンディ端末 API / ポート 8080 / DBロール: app_event_insert）**

| レイヤー | クレート | 主な責務 |
|---|---|---|
| Presentation | `wnav_terminal_api` | ハンディ端末向け Handler / Router / Middleware（Idempotency: MOD-BE-001）/ OpenAPI docs |
| Application | `wnav_domain::service` | Use Case / Command / Query / Transaction 管理 |
| Domain | `wnav_domain::model`, `wnav_domain::repo` | Entity / Value Object / Repository Trait / Error |
| Infrastructure | `wnav_db` | sqlx クエリ / Repository 実装 / DB Connection Pool（ロール: app_event_insert）|

**wnav_master_api レイヤー構成（管理 API / ポート 8081 / DBロール: app_write）**

| レイヤー | クレート | 主な責務 |
|---|---|---|
| Presentation | `wnav_master_api` | 管理コンソール向け Handler / Router / Middleware / OpenAPI docs |
| Application | `wnav_domain::service` | Use Case / Command / Query / Transaction 管理 |
| Domain | `wnav_domain::model`, `wnav_domain::repo` | Entity / Value Object / Repository Trait / Error |
| Infrastructure | `wnav_db` | sqlx クエリ / Repository 実装 / DB Connection Pool（ロール: app_write）|

> **DBロール物理保証**: `app_event_insert` ロールは `wnav_terminal_api` プロセスのみが保持し、WorkEvent 系テーブルへの INSERT 権限を持つ。`app_write` ロールは `wnav_master_api` プロセスのみが保持し、マスタテーブルへの読み書き権限を持つ。ロールを跨いだ接続は接続文字列レベルで物理的に不可能にする。

### 1-2. 依存方向のルール

- **Presentation → Application → Domain ← Infrastructure**（依存逆転）
- Domain 層は外部クレート（sqlx・axum 等）に依存しない
- Infrastructure 層が Domain の Repository Trait を実装する（依存逆転の原則）
- 横断関心事（wnav_auth・wnav_hash_chain・wnav_outbox）は Application 層から利用
- バイナリ別の横断関心事の担当は「5. 横断関心事の設計」を参照

### 1-3. ディレクトリ構造（略）

```
crates/
  wnav_terminal_api/src/      # ハンディ端末 API バイナリ（MOD-BE-001 / ポート 8080）
    main.rs                   tokio::main + tokio::spawn（BAT-002/003/008）
    router.rs                 ハンディ端末向けルーティング
    middleware/
      idempotency.rs          Idempotency ミドルウェア（MOD-BE-001）
    handlers/                 ハンディ端末向けハンドラ群

  wnav_master_api/src/        # 管理 API バイナリ（MOD-BE-010 / ポート 8081）
    main.rs                   tokio::main + tokio::spawn（BAT-001 他）
    router.rs                 管理コンソール向けルーティング
    handlers/                 管理コンソール向けハンドラ群

  wnav_domain/src/
    model/
      work_event.rs           WorkEvent 構造体・Value Objects
      master_version.rs       MasterVersion 構造体
      ...（EN-001〜027 に対応する 27 モデル）
    service/
      event_service.rs        WorkEvent 記録サービス
      master_service.rs       マスタ管理サービス
      outbox_service.rs       Outbox 発行サービス
    repository/
      work_event_repo.rs      WorkEvent リポジトリ Trait
      master_repo.rs          マスタリポジトリ Trait
      ...
    error.rs                  ドメインエラー定義

  wnav_db/src/                # Infrastructure 共有クレート（両 API が依存）
    pool.rs                   DB 接続プール（ロール別に設定）
    repo/                     Repository Trait 実装群
```

## 2. ハンディ APP (React Native) のレイヤー構成

### 2-1. Feature スライスアーキテクチャ

```
src/
  app/          アプリルート・ナビゲーション設定・Store 初期化
  features/
    navigation/  Step ナビゲーション（FR-NV 全件）
    evidence/    証拠記録（FR-EV 全件）
    suspension/  中断・再開（FR-ST 全件）
    kaizen/      アンドン・不適合・CAPA（FR-KZ 全件）
    auth/        認証（FR-AU 全件）
  shared/
    ui/          共通 UI コンポーネント（CMP-HA）
    i18n/        多言語対応（FR-UI-001〜003）
    network/     Outbox Worker・Network State 検知
    db/          SQLite + TypeORM 接続・マスタキャッシュ
  assets/        翻訳ファイル・アイコン
```

### 2-2. 状態管理方針

- Zustand を採用（Redux より軽量・RN との親和性）
- 端末永続化が必要な状態（ユーザー設定・マスタキャッシュ）は SQLite（TypeORM）で管理
- API 通信は React Query（TanStack Query）でキャッシュ・再取得管理

## 5. 横断関心事の設計

| 関心事 | 担当モジュール | 技術 |
|---|---|---|
| 認証・RBAC | MOD-BE-005（wnav_auth）| JWT RS256 / axum ミドルウェア（`aud` クレームで handy / web 判別）|
| Idempotency | MOD-BE-001（`wnav_terminal_api` middleware）のみ | TBL-035 照合（端末 API 専用）|
| 構造化ログ | MOD-SH（共通設定）| Rust: `tracing` / React: `pino` |
| UUID v7 生成 | MOD-SH-002（IdGenerator）| `uuid` crate（Rust）・`uuidv7` npm |
| 時刻取得（Clock）| MOD-SH-003（ClockService）| テスト替え可能な抽象（時刻の固定化をテストで活用）|
| i18n | MOD-SH-001（LocaleResolver）| RN: `i18next` + `react-i18next` |

---

**本節で確定した方針**
- **バックエンドは Presentation/Application/Domain/Infrastructure の 4 層アーキテクチャを採用し、Domain 層が外部クレートに依存しない依存逆転の原則を適用する。**
- **バックエンドを `wnav_terminal_api`（ハンディ端末 API / ポート 8080 / DBロール: app_event_insert）と `wnav_master_api`（管理 API / ポート 8081 / DBロール: app_write）の 2 バイナリに分割し、独立可用性・セキュリティ境界・DBロール物理保証を実現する。**
- **Idempotency ミドルウェア（MOD-BE-001）は `wnav_terminal_api` 専用とし、管理 API には適用しない。JWT の `aud` クレームで端末用（`handy`）と管理用（`web`）を判別する。**
- **フロントエンド 3 アプリはすべて Feature スライスアーキテクチャを採用し、Feature ごとに独立性を持たせてスパゲティ化を防ぐ。**
- **横断関心事（認証・Idempotency・ログ・UUID・時刻・i18n）は共通モジュール（MOD-SH / MOD-BE-001/005）に集約し、各 Feature からの共有を単方向にする。**
