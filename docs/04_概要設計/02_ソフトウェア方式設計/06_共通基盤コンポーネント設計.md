# 06 共通基盤コンポーネント設計

本章の責務は、全 MOD が依存する横断的な共通基盤（i18n・ログ・ID 生成・時刻・ハッシュチェーン・例外整形）の設計を確定することである。共通基盤は MOD-SH および一部 MOD-BE として実装する。

---

## 1. i18n（多言語対応）—— MOD-SH-001 LocaleResolver

### 1-1. 技術選定

| アプリ | 使用ライブラリ | ファイル形式 |
|---|---|---|
| FE-HA（React Native）| `i18next` + `react-i18next` | JSON（`assets/locales/{lang}.json`）|
| FE-MA / FE-MC（React）| `i18next` + `react-i18next` | JSON |
| BE（Rust・エラーメッセージ）| `rust-i18n` crate | TOML |

### 1-2. 翻訳キー命名規約

```
{feature}.{component}.{key}
例:
  navigation.step.instruction_text
  auth.login.submit_button
  error.biz.lock_step_violation
```

### 1-3. 必須言語（FR-UI-001）

- `ja`（日本語・必須・100% 翻訳率）
- `en`（英語・必須・100% 翻訳率）
- `ja-simple`（やさしい日本語・JLPT N4 相当・FR-UI-002）

## 2. 構造化ログ—— LOG-NNN カタログとの連動

### 2-1. バックエンド（Rust tracing）

```rust
// 構造化ログ出力形式（JSON・NFR-OPS-038）
tracing::info!(
    log_id = "LOG-001",
    event_name = "api.request.received",
    request_id = %request_id,
    method = %method,
    path = %path,
);
```

ログレベル: `ERROR` / `WARN` / `INFO` / `DEBUG` / `TRACE`。本番環境は `INFO` 以上のみ出力。

### 2-2. フロントエンド（pino）

```typescript
const log = pino({
  level: 'info',
  formatters: { level: (label) => ({ level: label }) },
});

log.info({ logId: 'LOG-001', eventName: 'api.request.received', ... });
```

### 2-3. 日次ローテーション（NFR-OPS-038）

`logrotate` で日次ローテーション・30 日保全。

## 3. ID 生成—— MOD-SH-002 IdGenerator

### 3-1. UUID v7 生成

```rust
// Rust（wnav_domain）
use uuid::Uuid;
pub fn new_id() -> Uuid {
    Uuid::now_v7()
}
```

```typescript
// TypeScript（共通）
import { uuidv7 } from 'uuidv7';
const newId = () => uuidv7();
```

### 3-2. Idempotency Key 生成

API の書き込みリクエストすべてに `Idempotency-Key: {UUID v7}` ヘッダを必須付与する。Outbox の `idempotency_key` 列と `event_id` は同一値を使用する。

## 4. 時刻取得—— MOD-SH-003 ClockService

テスト替え可能な時刻抽象を提供し、テストでの時刻固定を可能にする。

```rust
pub trait Clock: Send + Sync {
    fn now_utc(&self) -> chrono::DateTime<chrono::Utc>;
}

pub struct SystemClock;
impl Clock for SystemClock {
    fn now_utc(&self) -> chrono::DateTime<chrono::Utc> {
        chrono::Utc::now()
    }
}

// テスト用: FakeClock（固定時刻）
```

## 5. ハッシュチェーン計算—— MOD-BE-003 HashChainService

```rust
use sha2::{Digest, Sha256};

pub fn compute_content_hash(
    event_id: &Uuid,
    case_id: &Uuid,
    activity: &str,
    timestamp_client: &DateTime<Utc>,
    timestamp_server: &DateTime<Utc>,
    resource: &Uuid,
    sop_version_id: &Uuid,
    payload_canonical: &str,
    terminal_id: &Uuid,
    prev_hash: &str,
) -> String {
    let canonical = serde_json::to_string(&serde_json::json!({
        "event_id": event_id.to_string(),
        "case_id": case_id.to_string(),
        "activity": activity,
        "timestamp_client": timestamp_client.to_rfc3339(),
        "timestamp_server": timestamp_server.to_rfc3339(),
        "resource": resource.to_string(),
        "sop_version_id": sop_version_id.to_string(),
        "payload": payload_canonical,
        "terminal_id": terminal_id.to_string(),
        "prev_hash": prev_hash,
    })).expect("JSON serialization");

    let mut hasher = Sha256::new();
    hasher.update(canonical.as_bytes());
    format!("{:x}", hasher.finalize())
}
```

ハッシュ計算対象列の正規化: JSON キーをアルファベット昇順ソートした canonical JSON 文字列。

## 6. CFG（設定パラメータ）読み込み設計

`CFG-001`〜`CFG-014` の設定値は `src/infra/config/config.{profile}.yml` から読み込む（ADR-IMPL-001）。  
Rust では `wnav_config` クレートが `figment` crate を使って YAML + 環境変数オーバーレイを合成し、型安全にデシリアライズする。

```rust
// wnav_terminal_api/src/main.rs — 起動時の設定ロード
let config: TerminalApiConfig = wnav_config::load_terminal_api()?;
// wnav_master_api/src/main.rs
let config: MasterApiConfig = wnav_config::load_master_api()?;
```

```rust
// wnav_config/src/schema.rs — 構造体例
#[derive(Debug, serde::Deserialize, Clone)]
pub struct SharedConfig {
    pub schema_version: u32,
    pub observability: ObservabilityConfig,
    pub cors: CorsConfig,
    // ...
}
```

プロファイルは `WNAV_PROFILE={local,dev,staging,prod}` 環境変数で選択する。  
機密（DB パスワード・JWT 鍵等）は YAML に書かず `secret_ref: "<scheme>:<id>"` で間接参照し、  
`env:` / `dpapi:` / `docker_secret:` / `file:` スキームで実行時に解決する。  
詳細は `src/infra/config/README.md` および `docs/06_実装/ADR/ADR-IMPL-001_設定管理を_YAML_figment_secret_refで一元化.md` を参照。

---

**本節で確定した方針**
- **i18n は `i18next` で統一し、`ja` / `en` / `ja-simple` の 3 言語を必須とし、翻訳キーを `{feature}.{component}.{key}` 形式で一元管理する。**
- **構造化ログ（JSON 形式・`LOG-NNN` 連動・日次ローテーション 30 日保全）をバックエンド（tracing）とフロントエンド（pino）で統一する。**
- **UUID v7・ClockService・HashChainService を抽象として定義し、テスト時に差し替えやすい設計を確定した。**
