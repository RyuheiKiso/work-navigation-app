# 09 キャッシュとオフラインストレージ方式

本章の責務は、端末 SQLite のオフラインストレージ方式・マスタキャッシュの TTL・WAL 設定・破損時自己修復の設計を確定することである。P1（Offline-First）原則の中核設計である。

---

## 1. 端末 SQLite のキャッシュ設計

### 1-1. 格納データとキャッシュ戦略

| データ種別 | SQLite テーブル名 | TTL | 更新方式 |
|---|---|---|---|
| 作業実行レコード | `work_executions` | 永続（完了後もローカル保持）| INSERT / UPDATE（ローカル）|
| WorkEvent | `work_events` | 永続（Outbox 送信後も保持）| INSERT のみ |
| OutboxEvent | `outbox_events` | SENT: 7 日後削除 | INSERT + status UPDATE |
| SOP マスタ | `sops`, `steps` | CFG-007（60 分）で差分更新 | バックグラウンド同期 |
| プロセス・オペレーション | `processes`, `operations` | 同上 | 同上 |
| ユーザー・ロール | `users`, `roles` | 同上 | 同上 |
| 端末設定 | `device_config` | 永続 | 初回セットアップ時 INSERT |

### 1-2. マスタキャッシュの有効性判定

```typescript
// マスタ差分同期（FR-SY-003）のロジック
async function syncMasterIfNeeded(deviceSyncState: DeviceSyncState): Promise<void> {
  const now = new Date();
  const lastSync = new Date(deviceSyncState.last_sync_at);
  const intervalMs = Number(getConfig('master_sync.interval_minutes')) * 60 * 1000;

  if (now.getTime() - lastSync.getTime() < intervalMs) {
    return; // 60 分以内なら同期不要
  }

  // GET /api/v1/sync/master?as_of={last_sync_at}
  const diff = await api.getMasterDiff(deviceSyncState.last_sync_at);
  await applyMasterDiff(diff);
  await updateDeviceSyncState(now);
}
```

## 2. TypeORM エンティティ定義（概要）

SQLite の TypeORM エンティティは PostgreSQL の TBL カタログと 1:1 に対応するが、以下の差分がある。

| 差分 | 内容 |
|---|---|
| 主キー型 | PostgreSQL: `UUID`、SQLite: `TEXT`（UUID を文字列として格納）|
| JSONB | SQLite: `TEXT`（JSON 文字列）|
| TIMESTAMPTZ | SQLite: `TEXT`（ISO 8601 文字列）|
| BOOLEAN | SQLite: `INTEGER`（0/1）|

```typescript
@Entity('work_events')
export class WorkEventEntity {
  @PrimaryColumn('text') event_id: string;
  @Column('text') case_id: string;
  @Column('text') activity: string;
  @Column('text') timestamp_client: string;
  @Column('text') resource: string;
  @Column('text') sop_version_id: string;
  @Column('text') payload: string;  // JSON 文字列
  @Column('text') prev_hash: string;
  @Column('text') content_hash: string;
  @Column('text') terminal_id: string;
  @Column('integer') is_offline: number;
}
```

## 3. SQLCipher 初期化方式

### 3-1. 起動時フロー

```
アプリ起動
  → [OS Keystore/Keychain] KEY-003〜005 を取得
  → [SQLCipher] PRAGMA key = '{取得した鍵}'; を実行
  → 既存 DB の HMAC ヘッダを検証
  → 検証成功 → 通常起動
  → 検証失敗（鍵不一致）→ §4 破損時自己修復フローへ
```

### 3-2. WAL モードの設定

```typescript
// 起動時に必ず実行する PRAGMA
PRAGMA journal_mode = WAL;
PRAGMA wal_autocheckpoint = 1000;  // 1000 ページごとに checkpoint
```

## 4. 破損時自己修復

端末 DB が破損した場合（HMAC 検証失敗・ヘッダ不整合）、以下のフローで復旧する。

```
1. DB 破損を検出
2. バックアップ DB（`wnav.db.bak`）が存在する場合:
   → バックアップから復元（未送信 OutboxEvent は消失する可能性あり）
3. バックアップなし:
   → ローカルデータを全て消去（`DROP ALL`）
   → 空 DB で再起動
   → サーバーからマスタを再取得（FR-SY-002 初回同期）
   → UI で「ローカルデータが消失しました。未送信の記録はサポートに連絡してください」を表示
4. LOG-NNN（SECURITY レベル）を記録
```

## 5. ネットワーク状態（4 段階）の管理

P1（Offline-First）原則で確定した 4 段階の検知と UI 反映。

| 状態 | 条件 | UI 表現 | Outbox 動作 |
|---|---|---|---|
| 高品質接続 | 応答時間 < 1s | 通常表示 | 即時送信 |
| 低品質接続 | 応答時間 1〜5s | 遅延インジケーター | バッチ送信 |
| 切断 | 疎通失敗 | OFFLINE バナー | Outbox 蓄積 |
| Emergency Mode | 切断 5 分超過 | フルスクリーン警告 + 最終同期時刻表示 | 蓄積継続 |

```typescript
// NetworkProvider（MOD-FE-HA）でネットワーク状態を管理
type NetworkState = 'GOOD' | 'DEGRADED' | 'DISCONNECTED' | 'EMERGENCY';

const EMERGENCY_THRESHOLD_MS = 5 * 60 * 1000; // CFG-001（デフォルト 300s）
```

---

**本節で確定した方針**
- **端末 SQLite は SQLCipher で暗号化し、WAL モードで I/O 効率を上げる。TypeORM エンティティは PostgreSQL TBL と 1:1 対応させ、型変換（UUID→TEXT 等）をエンティティ層で吸収する。**
- **マスタキャッシュは CFG-007（60 分間隔）で差分同期し、オフライン時は古いキャッシュを使用してナビゲーションを継続する。**
- **ネットワーク 4 段階（高品質/低品質/切断/Emergency）を MOD-FE-HA（NetworkProvider）で管理し、Emergency Mode 時のフルスクリーン警告でオペレーターが状況を把握できるようにする。**
