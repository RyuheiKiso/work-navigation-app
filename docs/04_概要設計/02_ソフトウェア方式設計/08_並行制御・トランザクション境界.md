# 08 並行制御・トランザクション境界

本章の責務は、楽観ロック・Outbox トランザクション境界・SQLite 書き込み直列化の設計を確定することである。

---

## 1. 楽観ロック（更新可テーブル）

Append-only テーブル以外の更新可テーブル（work_executions 等）では楽観ロックを採用する。

### 1-1. updated_at による楽観ロック

```rust
// Application 層のサービス
pub async fn suspend_execution(
    &self,
    id: Uuid,
    expected_updated_at: DateTime<Utc>,
    suspension: Suspension,
) -> Result<(), AppError> {
    let rows_affected = self.repo
        .update_status_if_unchanged(id, WorkExecutionStatus::Suspended, expected_updated_at)
        .await?;

    if rows_affected == 0 {
        return Err(AppError::Db(DomainError::new(
            "ERR-DB-002",
            "optimistic_lock_conflict: 別の操作が先に更新しました"
        )));
    }
    Ok(())
}
```

```sql
-- Repository 実装（sqlx）
UPDATE work_executions
SET status = $1, updated_at = NOW()
WHERE work_execution_id = $2
  AND updated_at = $3;  -- ← 楽観ロック条件
```

### 1-2. 楽観ロック競合時の処理

- HTTP 409 を返す（ERR-DB-002）
- フロントエンドは最新状態を再取得してから再試行する
- 自動再試行は行わない（業務上の判断が必要なため）

## 2. Outbox トランザクション境界（§05 と連動）

WorkEvent + OutboxEvent の INSERT は必ず同一 PostgreSQL トランザクション内に含む。

### 2-1. 境界の確定

| 処理 | トランザクション要否 | 理由 |
|---|---|---|
| WorkEvent INSERT | **必須（単一 TX）** | Outbox と同期して一貫性を保つ |
| WorkEvent + OutboxEvent | **同一 TX に束ねる** | 片方だけ成功する状態を防ぐ |
| WorkExecution status UPDATE | 楽観ロック付きの単独 TX | WorkEvent とは別のライフサイクル |
| MasterVersion status UPDATE | 単独 TX（電子サイン AFTER の連鎖）| 承認フローの順序制御 |

### 2-2. トランザクションの実装パターン（Rust）

```rust
pub async fn record_step_completed(
    &self,
    event: WorkEvent,
    outbox: OutboxEvent,
) -> Result<(), AppError> {
    let mut tx = self.db.begin().await?;

    // 1. Idempotency Key 確認
    self.idempotency_repo.ensure_new(&mut tx, &event.event_id).await?;

    // 2. prev_hash 取得（直前の WorkEvent）
    let prev_hash = self.event_repo.latest_hash(&mut tx, &event.case_id).await?;

    // 3. content_hash 計算
    let content_hash = hash_chain::compute(&event, &prev_hash);

    // 4. WorkEvent INSERT
    self.event_repo.insert(&mut tx, event.with_hashes(prev_hash, content_hash)).await?;

    // 5. OutboxEvent INSERT
    self.outbox_repo.insert(&mut tx, outbox).await?;

    tx.commit().await?;
    Ok(())
}
```

## 3. SQLite 書き込み直列化（ハンディ APP）

端末 SQLite は WAL モードを使用するが、書き込みは SQLite のロック機構により直列化される。

### 3-1. WAL モードの設定

```typescript
// TypeORM DataSource 設定
new DataSource({
  type: 'expo-sqlite',
  database: 'wnav.db',
  synchronize: false,
  migrations: ['migrations/*.ts'],
  extra: {
    journalMode: 'WAL',  // Write-Ahead Logging
  },
});
```

### 3-2. 並行書き込みの排除

ハンディ APP では、1 つの作業セッション（WorkExecution）に対して同時に書き込む処理は 1 つだけになるよう設計する。

- Step 完了・証拠記録・中断は全て直列の UI 操作でトリガされる（並行 UI 操作がない）
- Outbox Consumer（バックグラウンド）はメインの WorkEvent 書き込みと競合しない（別テーブルの status 更新）

---

**本節で確定した方針**
- **楽観ロック（`updated_at` 比較）をすべての更新可テーブルに適用し、競合時は ERR-DB-002（409）を返して再試行をフロントエンドに委ねる。**
- **WorkEvent + OutboxEvent INSERT は同一 PostgreSQL トランザクションに束ね、片方だけ成功する状態を排除する設計を確定した。**
- **端末 SQLite は WAL モードで直列化し、単一作業セッションに対して並行書き込みが発生しない UI 設計で競合を事前に防ぐ。**
