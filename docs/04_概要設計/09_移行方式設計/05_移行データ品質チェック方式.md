# 05 移行データ品質チェック方式

本章の責務は、マスタ初期投入および過去記録移行において実施するデータ品質チェックの方式を概要設計レベルで確定することである。上流要件（MIG-064〜074）を受けて、投入前チェック・投入後チェック・ALCOA+ コンプライアンス確認・自動チェックツール構成・品質ゲート基準を確定する。品質ゲートを通過しない限りカットオーバーを進めない。

---

## 1. 品質ゲートの基本方針

### 1-1. ゼロトレランス原則

**DES-MIG-066**: 以下のクリティカルエラーが 1 件でも存在する場合、カットオーバーの実施を禁止する。

| クリティカルエラー種別 | 定義 |
|---|---|
| 外部キー違反（FK Violation） | 参照先の値が存在しない行がインポートデータに含まれる |
| 必須フィールド NULL | NOT NULL 制約のあるフィールドに NULL または空文字が存在する |
| 重複キー | 一意制約フィールドに重複値が存在する |
| SOP 電子署名未付与 | 公開ステータスの SOP のうち 1 件でも電子署名が未付与 |
| ハッシュチェーン整合性エラー | 初期ブロック生成後の整合性チェックでエラーが検出された |

### 1-2. 警告（許容範囲）の定義

**DES-MIG-067**: 以下の軽微な問題は警告として扱い、インポートツールによる自動修正後に警告ログを残すことで許容する。インポートは実行可能とするが、警告内容を quality_admin が確認した上で実行する。

| 警告種別 | 自動修正内容 |
|---|---|
| 末尾空白文字（trailing spaces） | trim() で自動除去 |
| 全角スペース | 半角スペースに自動変換 |
| 日付形式の軽微な相違（例：2026/05/17 → 2026-05-17） | ISO 8601 形式に自動正規化 |
| 連続した改行文字（\n\n） | 単一 \n に自動正規化 |

## 2. 投入前チェック（プレインポートバリデーション）

### 2-1. チェック項目の確定

**DES-MIG-068**: インポート実行前（プレビューフェーズ）に以下のチェックを自動実行する。

| チェック ID | チェック項目 | 対象データ | 判定 |
|---|---|---|---|
| PRE-001 | ファイルエンコーディング確認（UTF-8） | CSV/Excel 全ファイル | クリティカル |
| PRE-002 | 必須フィールド非 NULL | 全エンティティの必須列 | クリティカル |
| PRE-003 | 外部キー参照整合性 | FK 列すべて | クリティカル |
| PRE-004 | 重複キー検出（ファイル内） | 一意制約列すべて | クリティカル |
| PRE-005 | 重複キー検出（既存 DB との照合） | 一意制約列すべて | クリティカル |
| PRE-006 | 日付形式の妥当性 | DATE 型フィールド | クリティカル |
| PRE-007 | ロール値の妥当性 | users.role | クリティカル |
| PRE-008 | 文字数制限 | TEXT 型フィールド | 警告（自動修正可） |
| PRE-009 | 末尾空白・全角スペース | TEXT 型フィールド | 警告（自動修正可） |
| PRE-010 | ファイル行数の上限確認（50,000 行以下） | CSV 全ファイル | 警告（分割を推奨） |

### 2-2. バリデーション結果レポートの仕様

**DES-MIG-069**: プレビューフェーズで以下の情報をレポートとして表示する。

| レポート項目 | 内容 |
|---|---|
| 総行数 | インポートファイルの総行数 |
| クリティカルエラー件数 | PRE-001〜007 のエラー件数 |
| 警告件数 | PRE-008〜010 の警告件数（自動修正適用数を含む） |
| 正常行数 | エラーなし・警告自動修正後の正常行数 |
| エラー明細 | 行番号・列名・エラー内容・期待値の一覧 |

## 3. 投入後チェック（ポストインポートバリデーション）

### 3-1. チェック項目の確定

**DES-MIG-070**: インポートコミット完了後に以下のチェックを自動実行する。

| チェック ID | チェック項目 | 確認方式 | 判定 |
|---|---|---|---|
| POST-001 | 行数の一致確認 | ファイルの有効行数と DB の INSERT 件数が一致すること | クリティカル |
| POST-002 | SOP ステップ数の一致確認 | Excel テンプレートのステップ定義行数と sop_steps の件数が SOP ごとに一致すること | クリティカル |
| POST-003 | 10% ランダムサンプリング目視確認 | 投入後のデータから 10% をランダム抽出し、quality_admin が原本と突き合わせる | 必須確認（自動化不可） |
| POST-004 | ハッシュチェーン整合性 | 初期ブロックを含む全チェーンエントリの整合性バッチが「エラー 0 件」であること | クリティカル |
| POST-005 | 電子署名付与確認 | published ステータスの全 SOP に電子署名が存在すること | クリティカル |

### 3-2. POST-003 サンプリング目視確認の手順

**DES-MIG-071**: POST-003 の目視確認は以下の手順で実施する。

1. インポートツールがランダムサンプリング対象リスト（全件の 10%、最低 5 件・最大 50 件）を自動生成し、PDF でダウンロード可能にする。
2. quality_admin がサンプリングリストに記載された SOP・マスタデータを画面で確認し、紙原本・Excel との突き合わせを実施する。
3. 突き合わせ結果（確認済み件数・不一致件数）を確認記録フォームに入力する。
4. 不一致が 1 件でも発見された場合はクリティカルエラーとして扱い、全件再確認を実施する。

## 4. ALCOA+ コンプライアンスチェック

**DES-MIG-072**: すべての投入済み SOP に対して以下の ALCOA+ コンプライアンスチェックを自動実行する。

| ALCOA+ 要素 | チェック内容 | 確認クエリの対象テーブル |
|---|---|---|
| Attributable（帰属性） | author_username が NULL でないこと | sops.author_id IS NOT NULL |
| Contemporaneous（同時性） | created_at が NULL でないこと | sops.created_at IS NOT NULL |
| Original（原本性） | version が '1.0.0' 以上であること | sops.version IS NOT NULL |
| Accurate（正確性） | POST-003 の目視確認が完了していること | 別途確認記録フォームで管理 |
| Complete（完全性） | 必須フィールドすべてが非 NULL であること | NOT NULL 制約で DB レベルで保証 |

## 5. 自動チェックツールの構成

**DES-MIG-073**: 移行データ品質チェックに使用するツールの構成を確定する。

| ツール | 役割 | 実装方式 |
|---|---|---|
| プレインポートバリデーター | PRE-001〜010 の自動チェック | Rust（axum ハンドラ内で sqlx クエリを使用） |
| ポストインポートバリデーター | POST-001・002・004・005 の自動チェック | Rust バッチプロセス（tokio タスク） |
| 行数照合スクリプト | ファイル行数と DB 行数の比較 | SQL（COUNT クエリ）+ Rust バッチ |
| サンプリングリスト生成ツール | POST-003 用の 10% サンプルリスト生成 | Rust バッチ → PDF 生成（React でダウンロード提供） |
| エラーレポート生成ツール | エラー明細 CSV の自動生成 | Rust（CSV シリアライズ） |
| ハッシュチェーン整合性チェック | 全チェーンエントリの整合性確認 | Rust バッチ（07_セキュリティ方式設計と共有） |

---

**本節で確定した方針**
- 品質ゲートを「クリティカルエラー（FK 違反・NULL・重複・署名未付与・チェーン整合性エラー）0 件」として確定し、1 件でも存在する場合はカットオーバーを禁止することを確定する。
- 投入後の 10% ランダムサンプリング目視確認（POST-003）を必須工程として確定し、自動ツールのみによる品質確認を禁止することを確定する。
- チェックツールは Rust（axum + sqlx + tokio）で実装し、システム本体と同一リポジトリで管理することを確定する。
