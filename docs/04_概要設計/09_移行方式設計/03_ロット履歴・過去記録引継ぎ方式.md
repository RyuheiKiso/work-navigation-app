# 03 ロット履歴・過去記録引継ぎ方式

本章の責務は、紙・Excel による既存の過去作業記録を本システムへ引き継ぐ方式を概要設計レベルで確定することである。過去記録の移行は顧客（quality_admin）の判断による任意移行とし、強制的に要求しない。移行する場合の技術方式・ALCOA+ 準拠確保・ハッシュチェーンでの扱いを確定する。上流要件 MIG-034〜043 に対応する。

---

## 1. 移行の基本方針（任意移行）

### 1-1. 方針の確定

**DES-MIG-041**: 過去作業記録の移行はオプションとして設計し、以下の 2 ケースを明示的にサポートする。

| ケース | 説明 | 対応方式 |
|---|---|---|
| ケース A（移行する） | 監査証跡の継続性が顧客要件として必要な場合 | LEGACY_IMPORT 方式で work_events テーブルに取込む |
| ケース B（移行しない） | カットオーバー日を記録の起点とする場合 | 旧記録は紙原本で物理保管を継続する。本システムはカットオーバー日以降の記録のみを管理する |

**DES-MIG-042**: ケース A・ケース B の選択は quality_admin が書面で決定し、移行計画書（MIG-T-001）に記録する。選択後の変更は禁止する。

### 1-2. 移行対象期間

**DES-MIG-043**: ケース A を選択した場合、移行対象期間は直近 1 年以内とする。1 年超前の記録は移行しない（要件 MIG-003 による）。

---

## 2. LEGACY_IMPORT イベントの設計（ケース A）

### 2-1. work_events テーブルへの格納方式

**DES-MIG-044**: ケース A で移行する過去作業記録は、work_events テーブルに以下の特別属性を付与して格納する。

| フィールド | 値 | 意味 |
|---|---|---|
| event_type | `'LEGACY_IMPORT'` | 移行由来のイベントであることを識別する |
| is_legacy | `true` | 通常の作業記録と区別するフラグ |
| original_date | 原記録の日付（紙の記録日） | 作業が実際に行われた日時（ALCOA+ の Attributable 要件） |
| imported_at | インポート実行時のサーバー UTC タイムスタンプ | 電子記録としての作成日時 |
| imported_by | インポート実行者のユーザー ID | ALCOA+ の Attributable 要件 |
| source_document | 原記録の識別情報（紙記録の整理番号・スキャンファイル名） | 原本参照の担保 |

### 2-2. データ格納の方針

**DES-MIG-045**: LEGACY_IMPORT イベントの格納にあたり、以下の方針を確定する。

| 方針 | 内容 |
|---|---|
| 原著者の帰属 | 原記録に作業者の記名がある場合は imported_by ではなく original_author_id を設定する。システムにユーザーが存在しない場合は original_author_name（TEXT）に氏名を記録する |
| 未知の作業者 | 原記録に作業者の記名がない場合は original_author_name を '不明'（UNKNOWN）と明示する |
| タイムスタンプの精度 | 原記録に時刻の記録がない場合は original_date を日付のみ（時刻 00:00:00）とし、time_precision='DATE_ONLY' フラグを設定する |

---

## 3. ハッシュチェーンでの LEGACY ブロック管理

### 3-1. LEGACY ブロックの設計

**DES-MIG-046**: ケース A で移行した LEGACY_IMPORT イベントは、ハッシュチェーン（`07_セキュリティ方式設計/05_監査証跡（AuditTrail）方式` で設計）に対してチェーンエントリを生成するが、通常ブロックと区別するために `block_type='LEGACY'` を付与する。

| チェーンエントリ属性 | 通常ブロック | LEGACY ブロック |
|---|---|---|
| block_type | `'NORMAL'` | `'LEGACY'` |
| ハッシュ計算対象 | event の全フィールド | event の全フィールド（original_date 含む） |
| チェーン連結 | 前ブロックのハッシュを参照 | 前ブロックのハッシュを参照（連結を切断しない） |
| 改ざん検知対象 | 対象 | 対象（LEGACY ブロックも改ざん検知対象） |

**DES-MIG-047**: LEGACY ブロックは通常ブロックと同一のハッシュチェーンに連結する。LEGACY ブロックのためにチェーンを分岐させることを禁止する。

### 3-2. ALCOA+ 準拠の確保

**DES-MIG-048**: 移行データの ALCOA+ 準拠を以下の通り確保する。

| ALCOA+ 要素 | 通常記録での確保 | 移行記録（LEGACY）での確保 |
|---|---|---|
| Attributable（帰属性） | JWT ユーザー ID で保証 | original_author_id または original_author_name で保証 |
| Legible（可読性） | システム自動生成で保証 | インポート時の文字化けチェックで保証（05 章参照） |
| Contemporaneous（同時性） | サーバー UTC タイムスタンプで保証 | original_date で原記録日時・imported_at でインポート時刻を分離記録 |
| Original（原本性） | DB の Append-only 設計で保証 | 紙原本の物理保管継続・source_document でリンクを保持 |
| Accurate（正確性） | システム制御入力で保証 | インポート前の目視サンプリング（10%）で保証（05 章参照） |
| Complete（完全性） | 必須フィールド制約で保証 | 投入前バリデーション（NOT NULL チェック）で保証 |
| Consistent（一貫性） | サーバー側タイムスタンプで保証 | time_precision フラグで精度の明示 |
| Enduring（永続性） | 7 年保全方針で保証 | LEGACY イベントも同一保全方針を適用 |
| Available（参照可能性） | 通常の検索機能で保証 | is_legacy フィルターによる専用ビュー提供 |

---

## 4. 証拠ファイルのスキャン移行

### 4-1. スキャン済み紙記録の取込方式

**DES-MIG-049**: 既存の紙記録をスキャンした PDF/画像ファイルを、本システムの証拠ファイルとして取込む場合の方式を確定する。

| 属性 | 内容 |
|---|---|
| 格納先テーブル | evidence_files テーブル |
| evidence_type | `'SCANNED_LEGACY'`（通常の `'PHOTO'` や `'VIDEO'` と区別） |
| 関連付け | 対応する LEGACY_IMPORT イベントの work_event_id に紐付け |
| ファイル形式 | PDF・JPEG・PNG |
| ファイル名規則 | `LEGACY_<original_date>_<lot_no>_<seq>.pdf` 形式 |

**DES-MIG-050**: スキャン移行は必須ではない。紙原本が物理保管されている場合、スキャンなしで LEGACY_IMPORT イベントのみを格納することを許可する。

---

## 5. 移行しない場合の取り扱い（ケース B）

**DES-MIG-050**: ケース B（移行しない）を選択した場合の取り扱いを確定する。

| 項目 | 内容 |
|---|---|
| 旧記録の保管責任 | quality_admin が管理する既存の紙・Excel 保管体制を継続する |
| 本システムの起点 | カットオーバー日の初回作業から記録を開始する |
| 旧記録への参照 | 本システムは旧記録への参照リンクを持たない。物理保管場所を別途台帳で管理する |
| 監査対応 | カットオーバー日以前の記録は紙原本・Excel が正源であることを移行完了報告書に明記する |

---

**本節で確定した方針**
- 過去作業記録の移行はオプションとし、quality_admin の書面決定によって移行する（ケース A）か移行しない（ケース B）かを確定することを設計として定める。
- ケース A で移行する場合は event_type='LEGACY_IMPORT'・is_legacy=true を付与し、ハッシュチェーンには block_type='LEGACY' として通常チェーンに連結することを確定する。
- 移行記録の ALCOA+ 準拠は original_date・original_author・source_document・time_precision フラグの組み合わせで確保することを確定する。
