# 06 移行ツール構成（CSV/Excel 双方向）

本章の責務は、マスタデータの CSV/Excel 双方向インポート・エクスポートを実現する移行ツールのアーキテクチャ構成を概要設計レベルで確定することである。上流要件（MIG-010〜013・FR-MA-006）を受けて、インポートツール・エクスポートツール・テンプレートファイル仕様・大量データ処理方式・エラーレポート生成方式を確定する。

---

## 1. 移行ツールの全体構成

### 1-1. ツール構成の概要

**DES-MIG-076**: 移行ツールは以下の 4 コンポーネントで構成する。

| コンポーネント | 役割 | 技術スタック |
|---|---|---|
| インポート UI（React） | CSV/Excel ファイルのアップロード・プレビュー・確認・エラー表示 | React（SCR-MA-005） |
| インポート API（Rust） | ファイル受信・バリデーション・DB 投入・エラーレポート生成 | Rust + axum（API-master-006） |
| エクスポート UI（React） | エクスポート対象の選択・ダウンロード操作 | React（SCR-MC-009） |
| エクスポート API（Rust） | DB からのデータ取得・CSV/Excel 生成・ダウンロード提供 | Rust + axum（GET /api/export/*） |

### 1-2. ツールの実装原則

**DES-MIG-077**: 移行ツールは以下の原則に従って実装する。

| 原則 | 内容 |
|---|---|
| システム本体との一体化 | 移行専用の使い捨てスクリプトとして実装することを禁止する。システム本体のリポジトリに組み込み、移行完了後も保守対象として維持する（MIG-013 対応） |
| テンプレートスキーマの凍結 | テンプレートの列定義は ver1.0.0 リリース時点で凍結する。凍結後の変更は後方互換性を保証した上でマイナーバージョンアップとして管理する（MIG-011 対応） |
| エラーの明示 | インポート失敗時は拒否行・エラー理由を記載したエラーレポート CSV を自動生成する |

---

## 2. インポートツール（CSV/Excel → DB）

### 2-1. インポートフロー

**DES-MIG-078**: インポートの処理フローを以下の通り確定する。

| フェーズ | 処理内容 | 担当コンポーネント |
|---|---|---|
| 1. ファイル受信 | multipart/form-data でファイルを受信し、一時ストレージに保存する | Rust API（axum handler） |
| 2. エンコーディング検査 | UTF-8（BOM 付き可）であることを確認する | Rust API |
| 3. Excel 解析 | .xlsx の場合は calamine クレートでパースし、CSV 形式に正規化する | Rust API |
| 4. バリデーション | PRE-001〜010 のチェックを実行し、結果を JSON で返す | Rust API（sqlx クエリ） |
| 5. プレビュー表示 | バリデーション結果をテーブル形式で表示する | React UI |
| 6. 確認・実行 | ユーザーが「インポート実行」を押下する | React UI |
| 7. チャンク分割 | 1000 行を超える場合はチャンク分割する | Rust API |
| 8. トランザクション INSERT | sqlx のトランザクション内で一括 INSERT する | Rust API（sqlx） |
| 9. コミット or ロールバック | エラーなしでコミット。エラー発生時はロールバック | Rust API |
| 10. 結果通知 | 成功件数・エラー件数・エラーレポート CSV のダウンロードリンクを返す | React UI |

### 2-2. 大量データのチャンク処理

**DES-MIG-079**: 1000 行を超えるインポートデータは以下の方式でチャンク処理する。

| 設計項目 | 内容 |
|---|---|
| チャンクサイズ | 1000 行/チャンク |
| チャンク単位のトランザクション | 各チャンクは独立したトランザクション（1 チャンクの失敗が他チャンクをロールバックしない） |
| エラー発生チャンクの扱い | 失敗チャンクをロールバックし、エラーレポートに当該行範囲を記録する |
| タイムアウト防止 | 1000 行/チャンクで DB タイムアウト（30 秒）以内に収めることを設計目標とする |
| プログレス表示 | React UI でチャンク処理の進捗率（%）を表示する |

### 2-3. 対応ファイル形式

**DES-MIG-080**: インポートで受け入れるファイル形式を確定する。

| 形式 | 対応状況 | 備考 |
|---|---|---|
| CSV（.csv） | 対応 | UTF-8（BOM 有無両方可）・カンマ区切り |
| Excel（.xlsx） | 対応 | Office 2016 以降の形式（.xls は非対応） |
| TSV（.tsv） | 非対応 | 本バージョンのスコープ外 |
| JSON（.json） | 非対応 | API 経由の直接投入を検討する場合は詳細設計で判断 |

## 3. エクスポートツール（DB → CSV/Excel）

### 3-1. エクスポート方式の確定

**DES-MIG-081**: エクスポートは以下の 3 種類の方式を提供する。

| エクスポート方式 | 対象 | API エンドポイント | 用途 |
|---|---|---|---|
| エンティティ単位エクスポート | 各マスタエンティティ（工程・作業・ユーザーなど） | `GET /api/export/masters/{entity_type}` | データポータビリティ・バックアップ確認 |
| SOP 全件エクスポート | SOP + ステップ の全件 | `GET /api/export/sops` | SOP 管理・外部レビュー用 |
| 監査証跡エクスポート | work_events（期間指定） | `GET /api/export/work-events?from=&to=` | 監査対応・帳票出力 |

### 3-2. エクスポートのファイル形式

**DES-MIG-082**: エクスポートファイルは以下の仕様とする。

| 仕様 | 内容 |
|---|---|
| CSV 形式 | UTF-8 BOM 付き（Excel で直接開いた際に文字化けしない） |
| Excel 形式 | .xlsx（umya-spreadsheet または rust_xlsxwriter クレートを使用） |
| ヘッダー行 | 1 行目に日本語ラベル・2 行目にシステムフィールド名を出力する |
| 日付形式 | ISO 8601（YYYY-MM-DD HH:MM:SS UTC） |
| NULL 値 | 空文字として出力する |

## 4. テンプレートファイルの仕様

### 4-1. テンプレート提供方針

**DES-MIG-083**: システムと一体で提供する Excel テンプレートファイルの仕様を確定する。

| 仕様 | 内容 |
|---|---|
| ファイル形式 | .xlsx（Excel 2016 以降互換） |
| エンコーディング | UTF-8 BOM 付き |
| シート構成 | 「入力シート」（データ入力用）・「記入例」（サンプルデータ）・「注意事項」（制約説明）の 3 シート |
| 列ヘッダー（1 行目） | 日本語ラベル（太字・グレー背景） |
| フィールド名行（2 行目） | システムフィールド名（細字・薄グレー背景）・セルにコメントで説明を付与 |
| Excel セルの入力規則 | ドロップダウン（ロール選択等）・文字数制限・日付形式のバリデーションを設定 |
| 行数上限 | 入力シートのデータ行は最大 10,000 行を想定してフォーマットする |

### 4-2. テンプレートファイル一覧

**DES-MIG-084**: 提供するテンプレートファイルの一覧を確定する。

| テンプレートファイル名 | 対象エンティティ |
|---|---|
| `template_processes.xlsx` | 工程マスタ（processes） |
| `template_operations.xlsx` | 作業マスタ（operations） |
| `template_products.xlsx` | 製品マスタ（products） |
| `template_users.xlsx` | ユーザーマスタ（users） |
| `template_user_skills.xlsx` | ロール・スキルマスタ（user_skills） |
| `template_sops.xlsx` | SOP マスタ（sops + sop_steps）（複数シート構成） |
| `template_lots_legacy.xlsx` | ロットマスタ（過去 1 年分・is_legacy=true 用） |

## 5. エラーレポートの仕様

**DES-MIG-085**: インポート失敗時に自動生成するエラーレポート CSV の仕様を確定する。

| 列名 | 内容 |
|---|---|
| row_number | 入力ファイル内の行番号（ヘッダー行を除く） |
| entity_type | エンティティ種別（例：processes・users） |
| field_name | エラーが発生したフィールド名 |
| input_value | 入力値（実際に入力された値） |
| error_code | エラーコード（例：ERR_FK_VIOLATION・ERR_NULL・ERR_DUPLICATE） |
| error_message | エラーの説明（日本語） |
| suggestion | 修正方法の提案（可能な場合） |

---

**本節で確定した方針**
- 移行ツールは React UI（SCR-MA-005・SCR-MC-009）+ Rust API（API-master-006・GET /api/export/*）の構成で実装し、システム本体と同一リポジトリで保守対象として管理することを確定する。
- 大量データは 1000 行/チャンクで分割処理し、チャンク単位のトランザクションでタイムアウトを防止することを確定する。
- エクスポートファイルは UTF-8 BOM 付き CSV または .xlsx 形式で出力し、Excel での文字化けを防止することを確定する。
