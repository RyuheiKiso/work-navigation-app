# 99 前提制約と本書が約束しないこと

本章の責務は、移行方式設計サブ（09）の対象外事項を明示的に宣言し、将来の設計拡張・外部要求に対してスコープを権威的に確定することである。記載する対象外宣言（D-MIG-NNN）はシステム本体の設計方針・個人開発という開発体制・単一工場という運用環境に基づく合理的な判断であり、妥協ではない。

---

## 1. 対象外宣言（Disclaimer）一覧

| D-MIG ID | 対象外事項 | 判断根拠 | 対処方針 |
|---|---|---|---|
| D-MIG-001 | 基幹システム（ERP/MES）からの自動マイグレーション | 本システムは単一工場向けスタンドアロン運用であり、ERP/MES との接続は本バージョンのスコープ外 | 手動 CSV/Excel 経由のインポートのみをサポートする |
| D-MIG-002 | 移行コンサルティング・支援体制の設計 | 個人開発プロジェクトであり、外部移行コンサルへの依存を設計として持ち込まない | GUI ウィザードと本書（移行方式設計）でセルフサービスを完結させる |
| D-MIG-003 | 旧システム（紙・Excel）の廃棄手順 | 廃棄の実施タイミング・方法は工場管理者の業務判断に帰属し、本システムの設計範囲外 | 旧システムの廃棄は quality_admin の判断で別途実施する。本書はカットオーバー後の物理保管継続を前提とする |
| D-MIG-004 | データクレンジングの詳細手順 | 詳細なクレンジングルール（表記統一・異常値の扱い）は実際の顧客データを見なければ確定できないため、概要設計では確定しない | 詳細設計フェーズ（`05_詳細設計/`）でデータクレンジング仕様書を別途作成する |

---

## 2. 各対象外宣言の詳細

### D-MIG-001: 基幹システム（ERP/MES）からの自動マイグレーション

本システムは単一工場向けのスタンドアロン作業ナビゲーション・証跡記録システムとして設計する。ERP や MES との連携インターフェース（IF）は本バージョン（ver1.0.0）のスコープ外である。

| 項目 | 内容 |
|---|---|
| 対象外の内容 | ERP/MES の REST API・DB 直接接続・ETL パイプラインによる自動マイグレーション |
| 代替手段 | CSV/Excel エクスポート（ERP/MES 側）→ 本システムの CSV/Excel インポート（06 章参照）を手動で実施する |
| 将来対応の可能性 | ver2.0 以降の機能拡張として ERP 連携 IF を検討する場合は、`05_外部インターフェース設計` にて別途設計する |

### D-MIG-002: 移行コンサルティング・支援体制の設計

本プロジェクトは個人開発であり、外部の移行コンサルタント・支援ベンダーの関与を前提とした設計を行わない。移行の実施主体は quality_admin・master_admin・system_admin（同一人物または少人数）であることを前提とする。

| 項目 | 内容 |
|---|---|
| 対象外の内容 | 移行コンサルタントの役割定義・ベンダー契約の設計・外部支援 SLA の設定 |
| 代替手段 | 本書（移行方式設計）の全章・README の読み順に従い、quality_admin が自主的に移行を進める |
| 想定利用者のスキル前提 | quality_admin が Excel 操作・基本的な IT リテラシーを保有していることを前提とする |

### D-MIG-003: 旧システム（紙・Excel）の廃棄手順

カットオーバー後の紙記録票・Excel ファイルの廃棄は、工場の内部規定・情報セキュリティ方針・法的保管義務（記録類の保管年数）に従って quality_admin が判断する。本システムはこれを統制しない。

| 項目 | 内容 |
|---|---|
| 対象外の内容 | 紙記録の廃棄スケジュール・シュレッダー処理手順・Excel ファイルの削除タイミング |
| 本書の前提 | カットオーバー後も旧記録は物理保管を継続する（03 章 ケース B の前提） |
| リスクの帰属 | 旧記録の不適切廃棄によるリスクは quality_admin の業務判断に帰属する |

### D-MIG-004: データクレンジングの詳細手順

概要設計段階では顧客の実際のデータ（工程名・SOP 内容・ユーザー一覧）が確定していないため、詳細なクレンジングルールを確定することができない。

| 項目 | 内容 |
|---|---|
| 対象外の内容 | 表記統一ルール（全角/半角・略称/正式名称）・異常値の扱い・重複データの名寄せルール |
| 本書での対応 | 05 章でクリティカルエラーの定義（FK 違反・NULL・重複）を確定し、これ以外の詳細クレンジングルールは詳細設計フェーズで確定する |
| 詳細設計での成果物 | `05_詳細設計/` にデータクレンジング仕様書（詳細設計フェーズ）を別途作成する |

## 3. 本書全体の前提制約

本書（移行方式設計サブ）を適用する際の前提制約を確定する。前提が崩れた場合は設計の見直しが必要になる。

| 制約 ID | 前提制約 | 崩れた場合の影響 |
|---|---|---|
| CON-MIG-001 | 対象は単一工場である | 多拠点展開の場合は移行フェーズの再設計が必要 |
| CON-MIG-002 | 移行実施者（quality_admin）が Excel 操作を行える | GUI ウィザードの代替操作手順（CLI）を設計する必要がある |
| CON-MIG-003 | WiFi インフラは移行計画開始前に調達済みである | Phase M-1 のタスク順序（デバイス調達・WiFi 設置）を前倒しする必要がある |
| CON-MIG-004 | 並行運用期間中も生産ラインは停止しない | 並行運用方式の前提が崩れるため、方式の再検討が必要 |
| CON-MIG-005 | カットオーバー後も紙原本は物理保管される | 旧記録廃棄後は LEGACY_IMPORT 方式の source_document 参照が機能しない |

---

**本節で確定した方針**
- D-MIG-001〜004（ERP 自動移行・移行コンサル・旧システム廃棄・クレンジング詳細手順）を対象外として確定し、本バージョン（ver1.0.0）のスコープに含めないことを宣言する。
- データクレンジングの詳細手順は詳細設計フェーズ（`05_詳細設計/`）で確定し、概要設計では確定しないことを宣言する。
- 本書の前提制約（CON-MIG-001〜005）が崩れた場合は移行方式設計全体の見直しが必要であることを明示する。
