# 04 変更管理と版数規約

本章の責務は、本書（要件定義書）の版数管理規約・変更履歴の記録形式・変更影響分析の手順・過去版の保存要件を確定することである。本章はプロジェクト要件 03 章（PRJ-035〜051）の変更管理プロセスと連動し、文書レベルの具体的な規約を確定する。

---

## 1. 版数管理規約

### 1-1. セマンティックバージョニングの採用

本書の版数はセマンティックバージョニング（`メジャー.マイナー.パッチ`）を採用する。

| バージョン要素 | 形式 | 採番ルール |
|---|---|---|
| メジャー版数 | `X.0.0`（X は 1 以上の整数）| 要件の基本的な性質・スコープ・分類体系が変わる大規模改訂の場合に上げる |
| マイナー版数 | `1.Y.0`（Y は 1 以上の整数）| 既存要件の変更・新規要件の追加・既存要件の廃止の場合に上げる |
| パッチ版数 | `1.0.Z`（Z は 1 以上の整数）| 誤字修正・表現の明確化・参照リンクの修正等の軽微な修正の場合に上げる |

### 1-2. 版数アップの条件と例

| 変更の種類 | 版数アップの区分 | 例 |
|---|---|---|
| 新規要件（TST・PRJ）の追加 | マイナー版数を上げる | TST-001〜007 を追加: v0.1.0 → v0.2.0 |
| 既存要件の記述変更（受入基準・MoSCoW 変更） | マイナー版数を上げる | TST-003 の SLO 数値目標の更新: v1.0.0 → v1.1.0 |
| 既存要件の廃止 | マイナー版数を上げる | TST-005 の廃止（代替要件に統合）: v1.1.0 → v1.2.0 |
| 要件分類体系・採番規約の変更 | メジャー版数を上げる | サブカテゴリコードの追加: v1.x.x → v2.0.0 |
| 誤字・表現の明確化 | パッチ版数を上げる | 「受入基準」→「受入判定基準」の用語統一: v1.0.0 → v1.0.1 |
| 参照リンクの修正 | パッチ版数を上げる | 業界分析参照パスの修正: v1.0.1 → v1.0.2 |

### 1-3. 初期版数の扱い

本書の作成開始時点の版数は `0.1.0` とする。`0.x.y` バージョンは「要件定義書の作成中・未確定」を意味する。`1.0.0` は「要件定義書の初版確定（M1 完了・要件定義書 Agreed 状態への移行）」を意味する。

Frozen 状態への移行（M2 実装開始時点）は版数変更を伴わない。ただし Git タグ `req-baseline-{版数}` を付与し、ベースラインとして記録する。

---

**本節で確定した方針**
- セマンティックバージョニング（メジャー.マイナー.パッチ）を本書の版数管理規約として確定する。
- 新規要件追加・廃止はマイナー版数アップ・要件分類体系変更はメジャー版数アップとすることを確定する。
- `1.0.0` を M1 完了・Agreed 状態確立の版数として確定する。

---

## 2. 変更履歴の記録形式

### 2-1. 変更履歴の必須 5 フィールド

本書の変更履歴は以下の 5 フィールドを必須として記録する。変更履歴は本書 README または各サブの先頭に「変更履歴」セクションとして配置する。

| フィールド | 内容 | 形式 |
|---|---|---|
| 版数 | セマンティックバージョニング形式の版数 | `X.Y.Z` |
| 変更日 | 変更を確定した日付 | `YYYY-MM-DD` |
| 変更内容 | 変更した要件 ID・変更の概要 | 「TST-003 SLO 数値目標を更新（計測ベースライン取得後の確定値反映）」等 |
| 変更者 | 変更を実施した担当者（個人開発のため開発者本人）| 開発者名 |
| レビュー者 | 変更をレビューした担当者（セルフレビューの場合は「セルフレビュー」と記録）| 開発者名またはレビュー者名 |

### 2-2. 変更履歴の記録例

```markdown
## 変更履歴

| 版数 | 変更日 | 変更内容 | 変更者 | レビュー者 |
|---|---|---|---|---|
| 0.1.0 | 2026-05-17 | 初版作成（テスト・受入要件・プロジェクト要件・付録の全章）| （開発者名）| セルフレビュー |
| 0.2.0 | YYYY-MM-DD | TST-008 新規追加（負荷テスト合格基準の明確化）| （開発者名）| セルフレビュー |
| 1.0.0 | YYYY-MM-DD | 要件定義書初版確定（全サブ Agreed 状態、M1 完了）| （開発者名）| セルフレビュー |
```

### 2-3. 変更履歴の保存場所

変更履歴は以下の 2 か所で管理する。

- **各サブの変更履歴セクション**: 各章ファイルの末尾に「変更履歴」セクションを設ける。当該章に影響する変更のみを記録する
- **`docs/01_管理/change_log.md`（要件変更の場合）**: PRJ-036 変更管理プロセスの一部として、要件変更の詳細（影響分析・承認記録）を記録する。変更履歴の補完記録として機能する

---

**本節で確定した方針**
- 変更履歴の必須 5 フィールド（版数・変更日・変更内容・変更者・レビュー者）を確定する。
- 変更履歴を各サブの末尾セクション（章別記録）と `change_log.md`（詳細記録）の 2 か所で管理することを確定する。
- セルフレビューの場合はレビュー者欄に「セルフレビュー」と記録することを確定する。

## 3. 変更影響分析の手順

### 3-1. 変更影響分析の手順

要件が変更された場合、以下の手順で変更の影響範囲を分析する。

**ステップ 1: 変更された要件の特定**
- 変更する要件の ID（例: TST-003）と変更内容（例: SLO 数値目標の変更）を特定する

**ステップ 2: 下流要件への影響確認**
- RTM（付録 01）から変更対象の BR-ID・FR-ID・NFR-ID・TST-ID のリンクを確認する
- 変更が下流の要件（変更した要件を参照している要件）に影響するかを確認する

影響パターン別の確認:

| 変更の種類 | 影響範囲の確認先 |
|---|---|
| TST 受入基準の変更 | 対応する FR/NFR の受入基準・テスト記録形式 |
| FR（機能要件）の変更 | 対応する TST の受入基準・受入シナリオのステップ |
| NFR（非機能要件）の数値目標変更 | 対応する TST の受入基準 |
| PRJ（プロジェクト要件）の変更 | 他の PRJ 要件・TST 要件との整合性 |

**ステップ 3: 関連図への影響確認**
- 変更が `img/` 配下の図（`fig_acceptance_gate`・`fig_test_phase_flow`・`fig_release_decision_tree`・`fig_rtm_overview` 等）に影響するかを確認する
- 図への影響がある場合は、図の更新を変更作業に含める（drawio-lint + svg-postcheck の再実行）

**ステップ 4: 影響分析結果の記録**
- 影響範囲・影響する要件 ID・影響する図のリストを `change_log.md` に記録する
- 「影響なし」の判断も明示的に記録する（「影響なし」の理由を記録する）

**ステップ 5: 変更の実施**
- 影響分析に基づき、変更対象の要件・下流要件・RTM・図を順番に更新する
- ドキュメント更新完了後に実装を更新する（計画 10 章 §3 ドキュメント先行原則）

---

**本節で確定した方針**
- 変更影響分析を 5 ステップ（特定→下流確認→図確認→記録→実施）として確定する。
- 「影響なし」の場合も理由を明示的に記録することを確定する。
- 図への影響がある場合は drawio-lint + svg-postcheck の再実行を必須とすることを確定する。

## 4. 過去版の保存要件

### 4-1. Git での管理

本書の全バージョンは Git リポジトリで管理する。Git の履歴が版数の変遷を記録する主要手段である。

- **コミット履歴**: すべての変更はコミットとして記録される。特定の版数への復元は `git checkout {commit-hash}` で行える
- **Git タグ**: セマンティックバージョニングの版数に対応する Git タグ（`docs-req-v1.0.0` 等）を付与する。タグは付与後に削除・移動しない

Git 履歴の保全:
- Main ブランチへのマージ後に `git rebase -i` または `git push --force` で履歴を書き換えない（PRJ-042 変更履歴保存要件）
- リポジトリの削除・クローン先のリセットは開発者本人が意図的に判断する場合に限る

### 4-2. PDF スナップショットの保存条件

以下の条件が発生した場合に、本書の PDF スナップショットを `docs/archive/` に保存する。

| 保存条件 | ファイル名形式 |
|---|---|
| M1 完了時（要件定義書初版確定 v1.0.0）| `req_def_v1.0.0_YYYYMMDD.pdf` |
| 各マイルストーン完了時（M2〜M5）に要件変更があった場合 | `req_def_v{版数}_YYYYMMDD.pdf` |
| Frozen 状態への移行時 | `req_def_frozen_YYYYMMDD.pdf` |
| 大規模変更（マイナー版数 2 以上の上昇）の前 | `req_def_v{版数}_pre_YYYYMMDD.pdf` |

PDF スナップショットは `docs/archive/` 配下に格納し、Git リポジトリで管理する。Git リポジトリ外（USB・共有ドライブ）へのバックアップも取得する（TST-071 受入完了記録のバックアップ要件と同様の方針）。

### 4-3. 過去版参照の手順

過去版の文書を参照する手順は以下のとおりとする。

- **Git 履歴から参照する**: `git log --oneline docs/03_要件定義/` で変更履歴を確認し、`git show {commit-hash}:{ファイルパス}` で過去版を参照する
- **PDF スナップショットから参照する**: `docs/archive/` 配下の PDF を直接参照する
- **廃止要件の参照**: 廃止された要件は削除されておらず、現行ファイルの `[廃止]` メタデータ付き箇所を参照する

---

**本節で確定した方針**
- Git タグによる版数管理と PDF スナップショットの保存を過去版管理の 2 手段として確定する。
- M1 完了時・Frozen 移行時を必須のスナップショット保存タイミングとして確定する。
- Git 履歴の書き換え禁止を確定し、過去版への参照可能性を永続的に維持することを確定する。
