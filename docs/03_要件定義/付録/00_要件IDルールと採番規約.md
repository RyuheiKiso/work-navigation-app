# 00 要件 ID ルールと採番規約

本章の責務は、本書（要件定義書）で使用する全要件 ID の識別子プレフィックス・採番ルール・管理方法を確定することである。本章は要件定義書 04 章「要件分類体系とトレーサビリティ枠組み」の採番規約を補完し、全識別子の完全なリファレンスとして機能する。

---

### 1-6. 付録・索引のみの識別子（番号なし）

以下の識別子は番号を持たない参照・索引識別子として使用する。

| 識別子 | 用途 |
|---|---|
| `AC-{要件 ID}` | 受入基準（Acceptance Criteria）。各要件の受入基準を `AC-{FR-NV-001}` 等の形式で参照する |
| `ADR-NNN` | アーキテクチャ決定記録。`docs/01_管理/ADR/` に格納する |

---

**本節で確定した方針**
- 識別子プレフィックスを BR/FR/NFR/MIG/OPS/MNT/EDU/TST/PRJ/UC/IF/EN/RP の 13 カテゴリで確定する。
- FR と NFR はサブカテゴリを持ち、BR も BUS/SC の 2 サブカテゴリを持つことを確定する。
- 付録・索引のみの識別子（AC・ADR）は番号なし参照識別子として確定する。

## 2. 採番ルール

### 2-1. 採番の基本規則

採番は各サブカテゴリ内で `001` から開始し、3 桁ゼロ埋めで表記する。連番は追番であり、欠番を作らない。

| ルール | 内容 |
|---|---|
| 開始番号 | `001` から開始する。`000` は使用しない |
| 桁数 | 3 桁ゼロ埋め（`001`〜`999`）。999 件を超える場合は本章の改訂で対応する |
| 連続性 | 新規要件は既存の最大番号 + 1 を採番する。番号をスキップしない |
| 欠番禁止 | 採番した番号は欠番にしない。廃止した要件は削除しない（次節参照） |
| 再使用禁止 | 廃止した要件の番号を新しい要件に再使用しない |

採番例:
- `TST-001`〜`TST-042` は本書テスト・受入要件サブで採番済みの番号
- `PRJ-001`〜`PRJ-073` は本書プロジェクト要件サブで採番済みの番号
- 次に TST サブに要件を追加する場合は `TST-043` から採番する

### 2-2. 廃止要件の扱い

廃止した要件は削除しない。廃止した要件は以下の形式でメタデータを付与して保持する。

```markdown
### [廃止: 2026-XX-XX 廃止理由: {理由}] TST-NNN: 要件タイトル

[廃止: YYYY-MM-DD 廃止理由: {理由}。代替要件: {代替要件 ID または「なし」}]

（元の要件記述をそのまま保持する）
```

廃止要件の採番台帳への記録: 廃止した要件の番号を採番台帳（次節参照）に廃止日・廃止理由とともに記録する。

---

**本節で確定した方針**
- 採番は `001` から開始し、欠番・再使用を禁止することを確定する。
- 廃止要件は削除せず `[廃止]` メタデータを付与して保持することを確定する。
- 廃止要件の番号は再使用しないことを確定する。

## 3. 識別子の管理方法

### 3-1. RTM との連携

識別子は要件トレーサビリティマトリクス（RTM）（付録 01）と連携して管理する。新規要件を採番した場合は RTM に対応するエントリを追加する。廃止要件の RTM エントリは「廃止」として保持する（削除しない）。

RTM の更新タイミング:
- 要件の新規追加時: 追加と同時に RTM エントリを作成する
- 要件の変更時: 変更内容を RTM に反映する
- 要件の廃止時: RTM エントリを「廃止」にする。廃止要件を参照する下流要件の RTM エントリも確認し、必要に応じて更新する

### 3-2. 採番台帳の管理

採番台帳は `docs/03_要件定義/付録/99_採番台帳.md`（別途作成）に管理する。採番台帳には各識別子の全採番済み番号・状態（Agreed/Frozen/廃止）・担当ファイルを記録する。

採番台帳は RTM（付録 01）とペアで管理する。RTM がトレーサビリティを管理し、採番台帳が識別子の状態を管理する役割分担とする。

採番管理の責任者: 本プロジェクトは個人開発のため、開発者本人が採番台帳を管理する。新規要件の起票時に採番台帳を更新することを要件変更プロセス（PRJ-036）の一部とする。

---

**本節で確定した方針**
- 識別子を RTM と採番台帳のペアで管理し、新規追加・変更・廃止時の両方の更新を義務とすることを確定する。
- 採番台帳を `docs/03_要件定義/付録/99_採番台帳.md` に管理し、開発者本人が責任者として管理することを確定する。
- RTM はトレーサビリティを管理し、採番台帳は状態管理を担う役割分担を確定する。
