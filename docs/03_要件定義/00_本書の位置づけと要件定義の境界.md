# 00 本書の位置づけと要件定義の境界

## 1. IPA 共通フレーム 2013 上の位置

### 三層アーキテクチャにおける要件定義プロセスの位置

IPA 共通フレーム 2013 (SLCP-JCF2013) は、システム開発のライフサイクルを「企画プロセス」「要件定義プロセス」「システム開発プロセス」等に分類する。本書は要件定義プロセスに相当する。

企画プロセスは「システム化構想」と「システム化計画」の 2 段階に分かれ、いずれも `docs/02_企画/` 配下に既に完成している。
本書（要件定義書）は企画プロセスの成果物を受けて、「何をシステムが行うか」を検証可能命題として確定する段階に位置する。

三層の責務と言語の対比は以下のとおりである。

| 層 | プロセス | 言語 | 主な問い | 文書 |
|---|---|---|---|---|
| 第一層 | 企画プロセス（構想） | べき論（規範的命題） | Why — なぜ作るか、解決すべき課題、倫理スタンス | `02_企画/システム化構想/` |
| 第二層 | 企画プロセス（計画） | する論（確定命題） | What/How-Frame — 対象範囲・機能スコープ・アーキテクチャ原則・技術スタック | `02_企画/システム化計画/` |
| 第三層 | **要件定義プロセス** | **する論（検証可能命題）** | **What 詳細 — 受入基準付き機能要件・非機能要件・ユースケース記述** | **本書** |
| 第四層（未作成） | システム開発プロセス（概要設計） | する論（実装命題） | How 骨格 — 画面構成・API 設計・DB スキーマ骨格 | `04_概要設計/`（未作成） |

図: fig_req_ipa_position（img/ 配下）を参照。

## 2. 上流（構想・計画）からの継承事項の概観

### システム化構想からの継承

システム化構想（`02_企画/システム化構想/`）が確定した 6 つの Why は、本書の要件として具体化される。

| 構想での Why | 本書での具体化先 |
|---|---|
| Why ①: 解決すべき課題（二重管理・WAI/WAD 乖離・暗黙知喪失・改ざん・デジタルテイラリズム化リスク） | `業務要件/` サブ全体・`機能要件/` サブ全体 |
| Why ②: 価値命題（8 種ステークホルダー × 価値提供） | `03_ステークホルダーと要件責任マトリクス.md` |
| Why ③: 倫理スタンス（支援/監視の意図的分離・行動データ用途三限定・Just Culture） | `非機能要件/06_倫理品質要件.md`・本書全章 |
| Why ④: プロダクトビジョン（Intrinsic EPSS × ALCOA+ × Just Culture） | `機能要件/01・02`・`非機能要件/06` |
| Why ⑤: 時代認識・経営環境（中小製造業・SaaS否定環境・IT担当1名） | `99_前提制約と本書が約束しないこと.md` |
| Why ⑥: 推進方針（個人開発・phase なし・妥協禁止） | `プロジェクト要件/README.md` |

### システム化計画からの継承

システム化計画（`02_企画/システム化計画/`）が確定した What/How-Frame は、本書の前提条件および上流リンクとして機能する。

| 計画で確定した What/How-Frame | 本書での継承先 |
|---|---|
| 対象範囲（中小製造業・単一工場・社内LAN・オンプレ） | `99_前提制約と本書が約束しないこと.md` の P01〜P07 |
| 機能スコープ（ver1.0.0 実装機能と明示的非スコープ） | `機能要件/` サブ全体 |
| アーキテクチャ原則（Offline-First・Append-only・OT-IT境界） | `機能要件/03_同期・オフライン機能要件.md` |
| 技術スタック（React Native・Rust/axum・PostgreSQL・IIS） | `99_前提制約と本書が約束しないこと.md` |
| データモデル中核設計（ALCOA+ 3 階層 × 4M+1E × Event Sourcing） | `機能要件/02_トレーサビリティ機能要件.md` |
| 品質特性と非機能要件方針 | `非機能要件/` サブ全体 |
| 前提・制約・リスク（P01〜P07・T01〜T05・リスクレジスタ） | `99_前提制約と本書が約束しないこと.md` |

---

**本節で確定した方針**
- 構想の 6 つの Why を本書の対応サブへの継承先として確定し、Why から要件への変換ルートを明示する。
- 計画の What/How-Frame を本書の上流リンクとして継承し、要件 ID に上流リンクを付与する。
- 本書は計画の前提条件・技術スタック選定を所与とし、これらの再検討は行わない。

## 3. 下流（概要設計以降）への引き渡し境界

### 要件定義が扱うもの

本書が確定する内容は以下に限定する。

- **業務要件（BR）**: 業務フロー・ユースケース・業務ドメイン用語
- **機能要件（FR）**: 「何をするか」の検証可能命題・受入基準・優先度・上流リンク
- **非機能要件（NFR）**: 品質特性ごとの数値目標・グレード採否
- **移行要件（MIG）**: データ移行の受入基準・マスタ初期投入の品質基準
- **運用要件（OPS）**: バックアップ・監視・縮退運転の運用基準
- **保守要件（MNT）**: 技術的負債管理・ADR・依存バージョン管理の基準
- **教育要件（EDU）**: OJT 設計・多言語教材の要件
- **テスト・受入要件（TST）**: 受入基準の詳細化・テスト種別定義
- **プロジェクト要件（PRJ）**: 個人開発制約・変更管理プロセス

### 要件定義が扱わないもの

以下は概要設計以降の責務であり、本書では記述しない。

| 非対象 | 担当文書 |
|---|---|
| 画面レイアウト・UI 実装詳細 | `04_概要設計/`（画面設計） |
| API エンドポイント定義・リクエスト/レスポンス仕様 | `04_概要設計/`（API 設計） |
| 物理 DB スキーマ（テーブル定義・インデックス・制約） | `04_概要設計/`（DB 設計） |
| テストケースの実装・テストコードの記述 | 詳細設計書・実装 |
| 具体的なアルゴリズム・ロジックの実装方式 | 詳細設計書 |
| インフラ構成の詳細（Docker Compose 定義・IIS 設定値） | インフラ設計書 |

本書に上記が混入した場合、それは概要設計書または詳細設計書への移管対象である。

---

**本節で確定した方針**
- 本書が扱う範囲を 9 カテゴリ（BR/FR/NFR/MIG/OPS/MNT/EDU/TST/PRJ）に確定する。
- 画面実装詳細・物理DBスキーマ・API仕様・テストコードは本書の非対象として確定し、これらの記述は概要設計書以降に委ねる。
- 本書への混入を発見した場合は即座に下流文書へ移管する。

## 4. 「する論」継承宣言と検証可能命題の文体規約

### 「する論」の定義

計画書が確立した「する論」の言語を本書は完全に継承する。
「する論」とは「〜する」という具体的確定命題の言語であり、「〜すべきである」「〜したい」「〜であると思われる」という規範的命題・希望的命題は本書の言語ではない。

本書に「べき論」が混入した場合、それは構想書への移管対象である。
本書に曖昧語（考慮する・検討する・可能性がある・場合がある・〜したい・思われる）が混入した場合、それは削除対象である。

### 要件の記述形式

本書のすべての要件は以下の 5 要素を備えた検証可能命題として記述する。

| 要素 | 定義 | 例 |
|---|---|---|
| 要件 ID | 採番規約に従う一意の識別子 | `FR-NV-001` |
| 検証可能命題 | 「〜する」形式の機能の記述。テストで Pass/Fail を判定できる粒度 | 「作業員がステップを完了したとき、システムは完了タイムスタンプをサーバー側で付与した上で PostgreSQL に書き込む」 |
| 受入基準（AC） | 要件が満たされたと判断する具体的な検証条件 | 「完了タイムスタンプとクライアント送信時刻の差が 500ms 以内である」 |
| 優先度（MoSCoW） | Must/Should/Could/Won't のいずれか | `Must` |
| 上流リンク | 対応する構想章・計画章の参照 | `計画 06章`・`構想 Why ③` |

要件は表形式または箇条書き形式で記述し、フリーテキストのみの要件記述は禁止する。

### 禁止表現リスト

以下の表現は本書全章で使用を禁止する。

| 禁止表現 | 代替表現 |
|---|---|
| 考慮する | （具体的な対応を要件として記述する） |
| 検討する | （要件として確定するか、非対象として明示する） |
| 可能性がある | （YESかNOかを確定する） |
| 場合がある | （条件を明示して要件化する） |
| 〜したい | （「〜する」として確定する） |
| 思われる / 考えられる | （根拠とともに断定する） |
| 基本的に | （例外なく適用するか、例外条件を明示する） |
| できる限り | （数値目標を確定する） |
| 適切に | （適切の基準を受入基準として記述する） |

---

**本節で確定した方針**
- 「する論」を完全継承し、全要件を「ID・検証可能命題・受入基準・優先度・上流リンク」の 5 要素で記述することを確定する。
- 禁止表現リストの 9 項目を全章適用とし、自己監査チェックリストの確認対象とする。
- フリーテキストのみの要件記述を禁止し、表形式または箇条書き形式を義務とする。

## 5. 本書の有効期限

### 要件定義書の権威期限

本書の権威は ver1.0.0 リリースをもって概要設計書に移行する。
リリース後、実装の正とするのは概要設計書・詳細設計書・実装コードであり、本書はそれらへの入力文書として歴史的参照の位置づけに降格する。

ただし、「本書が約束しないこと」宣言（本 README §9 および `99_前提制約と本書が約束しないこと.md` に記載）は、ver1.0.0 リリース後も変更履歴として永続的に保持する。

### 要件の凍結と変更管理

本書の要件は以下のライフサイクルで管理する。

| フェーズ | 状態 | 変更可否 |
|---|---|---|
| 起票 | Draft | 自由に変更可 |
| レビュー | In Review | 指摘反映後に確定 |
| 合意 | Agreed | 変更管理プロセスを経なければ変更不可 |
| 凍結 | Frozen | 概要設計書の作成開始以降。変更は全章影響確認必須 |

概要設計開始後に要件変更が発生した場合は、`プロジェクト要件/README.md` の変更管理プロセスに従う。

### ver1.0.0 以降の扱い

ver1.0.0 リリース以降、本書への機能追加要求は受け付けない。
追加要求は次バージョン（ver1.x.0 等）の要件定義として別途起票する。
これはシステム化計画書の ver1.0.0 スコープ確定（計画 `02章`・`03章`）と整合する。

---

**本節で確定した方針**
- 本書の権威は ver1.0.0 リリースをもって概要設計書に移行することを確定する。
- 要件は「起票→レビュー→合意→凍結」の 4 フェーズで管理し、概要設計開始後の変更は変更管理プロセスを必須とする。
- ver1.0.0 リリース後の機能追加要求は次バージョン要件定義として別途起票することを確定する。
