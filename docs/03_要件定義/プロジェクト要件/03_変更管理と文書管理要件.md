# 03 変更管理と文書管理要件

本章の責務は、要件変更・設計変更・スコープ変更の管理プロセスと、ドキュメントの版数管理・廃止手順を確定することである。個人開発前提であるが、変更が管理されない状態で蓄積されることを防ぎ、「ドキュメントが権威である」原則（計画 10 章 §3）を維持する。

---

## 1. 変更管理プロセス

### 1-1. PRJ-035: 変更の種別と影響分析

| 要件 ID | PRJ-035 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 10 章 §3 ドキュメント先行原則 / 付録 04 変更管理と版数規約 |

変更を以下の 3 種別に分類し、各種別に応じたプロセスを適用する。

| 変更種別 | 定義 | プロセス |
|---|---|---|
| 要件変更 | 要件定義書（本書）の要件記述・受入基準・MoSCoW の変更 | PRJ-036 要件変更プロセスを適用する |
| 設計変更 | 概要設計・ADR の変更（要件への影響の有無にかかわらず） | PRJ-037 設計変更プロセスを適用する |
| スコープ変更 | Could/Should 要件の追加・削除・Won't への移行 | PRJ-036 要件変更プロセスを適用する（スコープ縮小も含む） |

Must 要件の削除を伴うスコープ変更は認めない。Must 要件を削除する場合は、削除理由と代替要件（または代替手段）を ADR に記録し、RTM を更新する。

### 1-2. PRJ-036: 要件変更プロセス

| 要件 ID | PRJ-036 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-035 / 付録 04 変更管理と版数規約 |

要件変更は以下の手順で実施する。

1. **変更の起票**: 変更が必要な要件 ID・変更内容・変更理由を `docs/01_管理/change_log.md` に記録する
2. **影響分析**: 変更対象の要件に紐づく下流要件（FR → NFR → TST）・RTM エントリ・関連図を特定し、影響範囲を `change_log.md` に記録する
3. **承認**: 個人開発のため、開発者本人が影響分析の妥当性を確認し「承認」の判断をする。Git コミットメッセージに `change: 要件変更 PRJ-036 承認 要件ID={xxx}` を含める
4. **ドキュメントの更新**: 対象要件の記述・受入基準・MoSCoW を更新する。要件が Frozen 状態の場合は版数を上げる（付録 04 §1 の版数管理規約に従う）
5. **RTM の更新**: 変更された要件に関連する RTM エントリを更新する（付録 01 §3 RTM 更新規約に従う）
6. **実装の更新**: ドキュメント更新完了後に実装を変更する。ドキュメントと実装が不一致の状態でコミットしない

### 1-3. PRJ-037: 設計変更プロセス

| 要件 ID | PRJ-037 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 10 章 §3・§4 / PRJ-005（ADR 要件）|

設計変更は以下の手順で実施する。

1. **ADR の判断**: 変更が ADR 作成条件（技術的不可逆性・有力な代替案・選択理由の非自明性）を満たす場合は ADR を先行作成する。条件を満たさない場合は ADR なしで進める
2. **ドキュメントの更新**: 概要設計書・関連する要件定義書の章を更新する
3. **テストへの影響確認**: 変更が既存のテストケース（L1〜L6）に影響する場合は、テストを更新する
4. **回帰テストの実施**: PRJ-035 で定義した回帰テスト（TST-030）を変更種別に応じて実施する

---

**本節で確定した方針**
- 変更を要件変更・設計変更・スコープ変更の 3 種別に分類し、各種別に対応するプロセス（PRJ-036・PRJ-037）を確定する。
- Must 要件の削除は ADR + RTM 更新を必須とし、無条件の削除を禁止することを確定する。
- 変更はドキュメント更新先行・実装後追いの順序を必ず守ることを確定する。

## 2. 文書管理要件

### 2-1. PRJ-040: ドキュメント先行原則の管理

| 要件 ID | PRJ-040 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 10 章 §3 ドキュメント先行原則 |

本プロジェクトのすべての文書は Git リポジトリで管理する。文書管理の要件は以下のとおりとする。

- **保存場所**: 本書（要件定義書）は `docs/03_要件定義/` 配下に格納する。概要設計書は `docs/04_概要設計/` 配下（未作成）に格納する
- **コミット粒度**: 章単位・節単位での変更が判別できるコミット粒度とする。複数章にまたがる変更は章ごとに分割してコミットすることを推奨する（必須ではない）
- **コミットメッセージ**: 変更内容が分かるコミットメッセージを記述する。`docs: {ファイルパス}: {変更内容の要約}` の形式を推奨する

### 2-2. PRJ-041: 変更履歴の記録

| 要件 ID | PRJ-041 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-040 / 付録 04 §2 変更履歴記録形式 |

要件定義書の変更履歴は以下の 2 か所に記録する。

- **Git コミット履歴**: すべての変更は Git でコミットされるため、コミット履歴が変更履歴として機能する
- **`docs/01_管理/change_log.md`（要件変更の場合）**: PRJ-036 の手順に従い、要件変更を起票・影響分析・承認のプロセスで記録する。Git コミット履歴だけでは要件変更の意図・影響範囲が不明確になるため、本ファイルに補完記録する

変更履歴の保存要件: Git リポジトリの履歴を削除・書き換えしない。`git rebase -i` での履歴整理は Main ブランチへのマージ後は実施しない。

### 2-3. PRJ-042: 古い文書の廃止手順

| 要件 ID | PRJ-042 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-041 / 付録 04 §4 過去版の保存要件 |

要件が廃止・統合・分割された場合の文書廃止手順を以下のとおり確定する。

- **要件の廃止**: 廃止した要件は削除しない。要件 ID はそのまま保持し、要件の先頭に `[廃止: YYYY-MM-DD 廃止理由: {理由}]` のメタデータを付与する（付録 00 §2 採番ルールを参照する）
- **章の廃止**: 章全体が不要になった場合は、ファイルの先頭に `[廃止: YYYY-MM-DD この章は {移行先ファイル} に統合された]` のメタデータを付与し、ファイルを削除しない
- **旧版文書のアーカイブ**: Frozen 状態のドキュメントに大規模な改訂が生じる場合は、改訂前の PDF スナップショットを `docs/archive/` に保存してから改訂する

---

**本節で確定した方針**
- 全文書を Git リポジトリで管理し、Main ブランチへのマージ後の履歴書き換えを禁止することを確定する。
- 要件変更は `change_log.md` への補完記録を必須とし、Git 履歴のみに依存しないことを確定する。
- 廃止要件・廃止章は削除せず `[廃止]` メタデータを付与して保持することを確定する。

## 3. ADR 起票要件

### 3-1. PRJ-045: ADR の起票・更新・廃止要件

| 要件 ID | PRJ-045 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 10 章 §4 ADR 運用方針 / PRJ-005 |

ADR の起票・更新・廃止の要件を以下のとおり確定する。

**起票の条件（計画 10 章 §4 を継承）:**
- 技術的不可逆性・有力な代替案の存在・選択理由の非自明性の 3 条件をすべて満たす場合に起票する

**更新の条件:**
- 既存 ADR の決定事項が部分的に変更された場合は、ADR の「状態」を「更新」に変更し、変更内容と変更理由を追記する
- ADR の本質的な決定事項が変更された場合は、既存 ADR を「廃止」にして新しい ADR を作成する

**廃止の条件:**
- 決定事項が完全に覆された場合は「廃止」に変更し、廃止理由と代替 ADR（ある場合）への参照を追記する

### 3-2. PRJ-046: ADR テンプレートの定義

| 要件 ID | PRJ-046 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-045 / 計画 10 章 §4 |

ADR は以下のテンプレートに従って記述する。

```markdown
# ADR-{NNN}: {技術選定のタイトル}

**状態**: 提案中 / 採択 / 却下 / 廃止（廃止日: YYYY-MM-DD）

## 背景

この決定が必要になった技術的・業務的文脈を記述する。

## 決定事項

何を選択したかを記述する。

## 根拠

なぜこの選択をしたかを記述する（代替案との比較を含む）。

| 代替案 | 棄却理由 |
|---|---|
| ... | ... |

## 結果

既知のトレードオフ・将来の再評価条件を記述する。

## 更新履歴

| 日付 | 変更内容 |
|---|---|
| YYYY-MM-DD | 初版作成 |
```

---

**本節で確定した方針**
- ADR の起票・更新・廃止の条件と手順を確定する。
- ADR のテンプレートを 6 項目（タイトル・状態・背景・決定事項・根拠・結果）＋更新履歴で確定する。
- 既存 ADR の部分変更は「更新」・本質変更は「廃止 + 新規作成」とすることを確定する。

## 4. 要件ベースライン

### 4-1. PRJ-050: 凍結された要件の変更管理

| 要件 ID | PRJ-050 |
|---|---|
| 優先度 | Must |
| 上流リンク | 要件定義書 04 章 §5 要件ライフサイクル（Frozen 状態）|

要件が Frozen 状態（概要設計開始後）に遷移した後の変更は、以下の変更管理プロセスを必ず経ること。

- Frozen 要件の変更は PRJ-036（要件変更プロセス）を適用する
- Frozen 要件の変更は文書の版数を上げる（付録 04 §1 の版数管理規約に従う）
- Frozen 要件の変更後は影響する下流文書（概要設計書・詳細設計書）の整合性を確認し、不一致があれば下流文書も更新する

Frozen 状態への移行タイミング: 概要設計書の作成開始時点（M2 実装開始時点）で、要件定義書の全要件を一括して Frozen 状態に移行する。Frozen 移行を Git タグ（`req-baseline-v1.0.0`）として記録する。

### 4-2. PRJ-051: ベースライン管理の方法

| 要件 ID | PRJ-051 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-050 |

要件ベースラインの管理は以下のとおり実施する。

- **Git タグによるベースライン管理**: 要件定義書が Agreed 状態に揃った時点で Git タグ `req-baseline-YYYYMMDD` を付与する。タグを付与した後の変更は PRJ-036 変更管理プロセスを経て実施する
- **版数の管理**: 付録 04 §1 の版数管理規約（メジャー.マイナー.パッチ）に従い、変更の大きさに応じて版数を上げる
- **RTM との整合**: ベースライン変更後は RTM（付録 01）を必ず更新する。RTM の未更新はベースライン変更完了とみなさない

---

**本節で確定した方針**
- Frozen 状態の要件変更は PRJ-036 変更管理プロセスを必ず経ることを確定する。
- 要件ベースラインを Git タグで管理し、タグ付与後の変更はプロセスを経ることを確定する。
- ベースライン変更後の RTM 更新を必須とし、未更新はベースライン変更完了とみなさないことを確定する。
