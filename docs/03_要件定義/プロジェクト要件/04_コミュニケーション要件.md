# 04 コミュニケーション要件

本章の責務は、個人開発前提での進捗の可視化方法・課題管理の手順を確定することである。チーム開発における「コミュニケーション」は「将来の自分・将来の引継ぎ者に対して現状を明確に伝えること」として再定義する。

---

## 1. 進捗報告要件

### 1-1. PRJ-055: 進捗の可視化方法

| 要件 ID | PRJ-055 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-008（ナレッジダンプ）/ 計画 10 章 §6 自己引継ぎ設計 |

個人開発の進捗は以下の 3 つの手段で可視化する。

**Git コミット履歴による進捗可視化**:
- 各コミットのメッセージは変更内容が分かる形式で記述する
- コミットのプレフィックスは以下の慣例に従う: `feat:` 新機能実装・`fix:` バグ修正・`docs:` ドキュメント変更・`test:` テスト追加・`refactor:` リファクタリング・`chore:` ビルド・設定変更
- コミット粒度は「1 つの論理的な変更単位」とする。複数の無関係な変更を 1 コミットに混在させない

**マイルストーン達成記録（milestone_log.md）**:
- 各マイルストーン完了時に完了日・完了確認した項目のサマリー・次マイルストーンへの引き渡し事項を記録する
- `docs/01_管理/milestone_log.md` に追記形式（Append-only）で記録する

**NEXT.md による中断時の状態記録（セッション管理）**:
- 各作業セッション終了時に `NEXT.md` を更新する
- `NEXT.md` は `.gitignore` に含め、コミット対象としない（個人作業メモとして管理する）
- 記載内容は「どこまで完成したか」「次に何をするか」「未解決の問題は何か」の 3 項目を必須とする

### 1-2. PRJ-056: Git コミットの粒度と頻度

| 要件 ID | PRJ-056 |
|---|---|
| 優先度 | Should |
| 上流リンク | PRJ-055 |

コミットの頻度と粒度について以下の方針を確定する。

- **粒度**: 1 コミット = 1 論理変更（1 機能の実装・1 バグの修正・1 文書の更新等）
- **頻度**: 1 日以上コミットが途絶えないことを目安とする。ただし疾病・休暇・繁忙期での途絶えは許容する
- **WIP（Work in Progress）コミット**: 作業の途中でも `WIP:` プレフィックスを付けてコミットし、中断可能な状態にする。WIP コミットはマイルストーン完了前に通常コミットに整理する（Main ブランチへの WIP コミットのマージは認めない）

---

**本節で確定した方針**
- 進捗の可視化を Git コミット履歴・milestone_log.md・NEXT.md の 3 手段で行うことを確定する。
- Git コミットのプレフィックス慣例（feat/fix/docs/test/refactor/chore）を確定する。
- WIP コミットは Main ブランチマージ前に整理することを確定する。

## 2. 課題管理要件

### 2-1. PRJ-060: バグ・課題の記録形式

| 要件 ID | PRJ-060 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-055 / TST-040（テスト記録要件）|

バグ・課題の記録は以下のいずれかの手段で管理する。個人開発のオーバーヘッドを最小化するため、複雑な課題管理ツールは使用しない。

- **GitHub Issues**: リポジトリの Issues 機能を使用する。各 Issue には「タイトル・再現手順・期待動作・実際の動作・優先度ラベル・対応状況」を記録する
- **`docs/01_管理/issue_log.md`（GitHub が使用できない場合）**: Markdown 形式で Issue を記録する。各エントリには「Issue-ID・登録日・タイトル・詳細・優先度・ステータス・解決日」の 7 フィールドを記録する

### 2-2. PRJ-061: 優先度付けの定義

| 要件 ID | PRJ-061 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-060 / TST-012（不合格時の重大度）|

バグ・課題の優先度は以下の 4 段階で定義する。

| 優先度 | 定義 | 対応方針 |
|---|---|---|
| P1（Critical） | システムが動作しない・データ破損・ALCOA+ 違反・セキュリティ脆弱性（High 以上） | 発見後直ちに対応する。対応完了まで他の作業を停止する |
| P2（High） | 主要機能が動作しない・受入テストが Fail する・オフライン動作が失敗する | 現在のマイルストーン内で対応する |
| P3（Medium） | 機能の一部が意図しない動作をする・UI の表示ズレ・エラーメッセージが不適切 | 次のマイルストーンまでに対応する |
| P4（Low） | 改善要望・文書の誤字・コメントの整備 | 都合のよい時に対応する（対応しない場合はその理由を記録する） |

### 2-3. PRJ-062: 解決確認手順

| 要件 ID | PRJ-062 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-060・PRJ-061 |

バグ修正の完了確認は以下の手順で実施する。

1. バグの再現手順を実施し、バグが発生しないことを確認する
2. バグ修正に関連する回帰テスト（TST-030）を実施し、Pass であることを確認する
3. Issue を「解決済み」に更新し、修正コミットの Git ハッシュを記録する
4. P1（Critical）バグの場合は、解決後に「根本原因分析」と「再発防止策」を Issue に記録する

---

**本節で確定した方針**
- バグ・課題の記録を GitHub Issues または `issue_log.md` で管理し、7 フィールドの記録形式を確定する。
- 優先度を P1〜P4 の 4 段階で定義し、P1 は直ちに対応・他作業停止を確定する。
- P1 バグの解決後には根本原因分析と再発防止策の記録を必須とすることを確定する。
