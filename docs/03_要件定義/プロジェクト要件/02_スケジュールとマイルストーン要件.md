# 02 スケジュールとマイルストーン要件

本章の責務は、ver1.0.0 リリースまでのマイルストーン定義・スケジュール制約・各工程の完了定義を確定することである。計画 10 章のマイルストーン定義を受けて、要件定義レベルでの詳細条件を確定する。

---

## 1. マイルストーン定義

### 1-1. PRJ-020: マイルストーン M1〜M6 の完了条件

| 要件 ID | PRJ-020 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 10 章 §2 マイルストーン定義 |

マイルストーンは「ドキュメントの完成度」と「機能の動作確認」の両方で判定する。どちらか一方のみを満たしてもマイルストーン完了とは見なさない（計画 10 章の方針を継承する）。

| マイルストーン | 完了条件（ドキュメント） | 完了条件（機能） | 品質ゲート |
|---|---|---|---|
| **M1: 設計完成** | 要件定義書（本書）全サブ完成・概要設計書完成 | 実装なし（ドキュメント完成のみ） | 本書チェックリスト 10 項目全 Pass |
| **M2: データ層完成** | DB スキーマ定義書・マイグレーション手順書・Rust エンティティ定義完成 | PostgreSQL + sqlx で全エンティティの CRUD + Append-only + SHA-256 が `cargo test` Pass | L1 単体テスト・L2 統合テスト（DB）全 Pass |
| **M3: バックエンド API 完成** | OpenAPI 3.1 仕様書（全エンドポイント）完成 | 全 API エンドポイントが `cargo test`（ユニット + 統合）Pass | L1・L2 全 Pass・行カバレッジ 80% 以上 |
| **M4: タブレット APP 完成** | 画面設計書（全画面ワイヤーフレーム + インタラクション仕様）完成 | React Native で全 UC（A-J）が Detox E2E Pass | L1・L2・L3（Detox）全 Pass |
| **M5: 管理 Web 完成** | 管理 Web UI 設計書完成 | React 管理 Web 全管理機能動作確認済み | L1・L2・L3（Playwright）全 Pass |
| **M6: ver1.0.0 リリース** | M5 完了 + ALCOA+ 自己監査全 Pass + 倫理品質自己監査全 Pass | ステージングで 1 週間以上の安定稼働 + 30 日パイロット運用完了 | TST-001〜007 全 Pass・L4〜L6 全 Pass |

### 1-2. PRJ-021: マイルストーン判定の手順

| 要件 ID | PRJ-021 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-020 |

マイルストーンの完了を宣言する手順は以下のとおりとする。

1. マイルストーンの完了条件（ドキュメント・機能・品質ゲート）を各チェックリストで確認する
2. チェックリストの全項目が Pass であることを確認する
3. マイルストーン完了を Git コミットメッセージに `feat: M{N} 完了: {完了内容の要約}` の形式で記録する
4. `docs/01_管理/milestone_log.md` にマイルストーン完了日・確認した内容のサマリーを追記する

---

**本節で確定した方針**
- M1〜M6 のマイルストーン完了条件をドキュメント・機能・品質ゲートの 3 観点で確定する。
- マイルストーン完了の宣言は Git コミットメッセージと milestone_log.md の双方に記録することを確定する。
- どちらか一方（ドキュメントのみ・機能のみ）の達成ではマイルストーン完了と見なさないことを確定する。

## 2. スケジュール制約

### 2-1. PRJ-025: 個人開発のスケジュール制約

| 要件 ID | PRJ-025 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 P07（IT 担当 1 名兼務）/ 計画 R-O01 |

個人開発前提のスケジュール制約を以下のとおり確定する。

- **並行作業の制限**: M2 以降の実装作業は先行マイルストーンの完了を待つ。ただし、確定済みの範囲でのドキュメント先行作業（設計書・API 仕様書の作業）は実装と並行して実施できる
- **外部依存による遅延リスク**: Rust Edition 2024・axum・sqlx の互換性破壊変更（計画 R-T01）・React Native for Windows のメンテナンス停止（計画 R-T02）等の外部依存変更によるスケジュール滑りを許容する。外部依存変更が発生した場合はADR に記録し、スケジュールへの影響を評価する
- **疾病・休暇等による中断の許容**: 開発者の疾病・休暇・繁忙期等による中断を許容する。中断期間中の進捗は停止し、再開後に NEXT.md を参照して継続する

### 2-2. PRJ-026: 計画変更の条件

| 要件 ID | PRJ-026 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-025 / プロジェクト要件 PRJ-035（変更管理）|

スケジュール計画の変更は以下の条件が発生した場合に許容する。

| 変更の種類 | 条件 | 手順 |
|---|---|---|
| マイルストーン延期 | 品質ゲートの未達・外部依存変更・重大な設計見直しが発生した場合 | 延期理由を `milestone_log.md` に記録し、変更後の目標を設定する |
| スコープの調整 | Could 要件の削除・Should 要件の次バージョン移行が必要な場合 | 変更管理プロセス（PRJ-035）を経て、RTM を更新する |
| アーキテクチャの変更 | 設計の根本的見直しが必要な場合 | ADR を作成し、影響する要件・設計文書・テストを洗い出してから変更する |

Must 要件を削除する形のスコープ変更は認めない。

---

**本節で確定した方針**
- 個人開発のスケジュール制約（外部依存変更・疾病・休暇による中断の許容）を確定する。
- マイルストーン延期・スコープ調整・アーキテクチャ変更の条件と手順を確定する。
- Must 要件を削除する形のスコープ変更は認めないことを確定する。

## 3. 工程要件

### 3-1. PRJ-030: 各工程の完了定義と品質ゲート

| 要件 ID | PRJ-030 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-020 / TST-026〜028（カバレッジ要件）|

各開発工程の完了定義と品質ゲートを以下のとおり確定する。

**要件定義工程の完了定義（M1 前提）:**
- 全サブ（業務要件・機能要件・非機能要件・移行要件・運用要件・保守要件・教育要件・テスト受入要件・プロジェクト要件）が本書 README §10 の自己監査チェックリスト 10 項目を全 Pass していること
- 全要件 ID に 5 要素（ID・検証可能命題・受入基準・MoSCoW・上流リンク）が揃っていること
- RTM（付録 01）が最新状態に更新されていること

**実装工程の完了定義（各 M2〜M5）:**
- 対象マイルストーンの完了条件（PRJ-020）が達成されていること
- 行カバレッジが 80% 以上（TST-026）であること
- CI/CD パイプラインが Pass であること（L1 単体テスト全 Pass・カバレッジ 80% 以上）

**リリース工程の完了定義（M6）:**
- 受入テスト全 Pass（TST-001〜007）であること
- ALCOA+ 自己監査チェックリスト全 Pass であること
- 倫理品質自己監査チェックリスト（計画 08 章 7 節）全 Pass であること
- 全操作マニュアル（PRJ-007）の実機確認済みであること
- ステージング環境での 1 週間以上の安定稼働が確認されていること

### 3-2. PRJ-031: 品質ゲートの遵守

| 要件 ID | PRJ-031 |
|---|---|
| 優先度 | Must |
| 上流リンク | PRJ-030 |

品質ゲートを通過していない工程の成果物を次工程の入力として使用しない。以下の場合は直前工程の品質ゲートを再確認してから次工程に進む。

- 要件定義書の自己監査チェックリストが未完了の状態で概要設計を開始しない
- M2（データ層）が未完了の状態で M3（バックエンド API）の実装を開始しない
- 受入テスト（TST-001〜007）が未 Pass の状態でリリース作業を開始しない

---

**本節で確定した方針**
- 各工程の完了定義（要件定義・実装・リリース）を品質ゲート要件として確定する。
- 品質ゲート未通過の工程成果物を次工程に持ち込まないことを確定する。
- リリース工程の品質ゲートに受入テスト全 Pass・ALCOA+ 自己監査・倫理品質自己監査を含めることを確定する。
