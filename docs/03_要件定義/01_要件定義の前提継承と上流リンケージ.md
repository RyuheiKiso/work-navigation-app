# 01 要件定義の前提継承と上流リンケージ

## 1. システム化構想からの倫理制約継承

### 継承する倫理制約の一覧

システム化構想 `04_倫理スタンスと支援・監視の意図的分離.md` が確定した倫理制約は、本書の全要件に対する上位制約として機能する。これらの倫理制約は要件の個別承認によって覆ることはなく、設計制約として全要件の受入基準に組み込む。

| 倫理制約 ID | 制約内容 | 構想参照章 | 本書での具体化先 |
|---|---|---|---|
| EC-01 | 「支援」と「監視」の意図的分離：作業員の作業速度・行動パターンを管理側に提示する機能を実装しない | 構想 04章 H2-1 | `非機能要件/06_倫理品質要件.md` NFR-ETH-001 |
| EC-02 | 個人別生産性スコアリング禁止：個人別速度・達成率・ランキングの表示機能を実装しない | 構想 04章 H2-2 | `非機能要件/06_倫理品質要件.md` NFR-ETH-002 |
| EC-03 | 行動データ用途三限定：品質保証・工程改善・教育設計の三用途のみ許容し、人事評価への転用を技術的・制度的双方で禁止する | 構想 04章 H2-3 | `非機能要件/06_倫理品質要件.md` NFR-ETH-003 |
| EC-04 | Work-as-Done の記録を「違反」として解釈する自動プロセスを実装しない | 構想 04章 H2-4 | `機能要件/02_トレーサビリティ機能要件.md` |
| EC-05 | AI/ML による自動判断・異常検知・自然言語指示生成を永続的に提供しない | 構想 08章・11章 | `99_前提制約と本書が約束しないこと.md` |
| EC-06 | データオーナーシップは工場（利用組織）に帰属し、ベンダーロックイン・クラウド移送・データ消失を永続禁止とする | 構想 13章 | `非機能要件/06_倫理品質要件.md` NFR-ETH-006 |

### Just Culture の組織原則継承

システム化構想 `04_倫理スタンスと支援・監視の意図的分離.md` H2-5 が採用した James Reason の Just Culture モデルは、本書の以下の要件に直接反映する。

- スリップ・ラプス（意図なき行為の失敗）を自動的に問責プロセスに結びつける機能は実装しない（FR の明示的非対象）
- ヒヤリハット・Kaizen 起票は匿名投稿オプションを提供し、提案疲労を防ぐ応答義務を運用要件として確定する
- SOP との乖離記録を「適応の証拠」として収集し、SOP 改訂ループに活用する機能を機能要件として確定する

---

**本節で確定した方針**
- 構想 04章・08章・11章・13章の倫理制約 EC-01〜EC-06 を全要件に対する上位制約として継承することを確定する。
- Just Culture の 3 点（問責プロセス非自動連携・匿名投稿オプション・SOP 改訂ループ）を要件として具体化することを確定する。
- 倫理制約は個別の要件合意によって覆ることはなく、変更には構想書の改訂を要することを確定する。

## 2. システム化計画からの対象範囲・技術スタック・データモデル・前提制約継承

### 対象範囲の継承

システム化計画 `02_対象範囲とビジネス目標.md` が確定した対象範囲を本書の前提条件として継承する。

| 計画での確定事項 | 本書での前提 ID | 本書への影響 |
|---|---|---|
| 単一工場・単一サイト | P01 | 権限スコープは工場単位。マルチサイト制御ロジックは全要件の非対象 |
| 社内 LAN のみ（インターネット接続なし） | P02 | 外部認証（OAuth2/OIDC）・クラウド連携は全要件の非対象 |
| Android/Windows タブレット主要接点 | P03 | React Native の対応プラットフォームを Android/Windows/iOS とする |
| Wi-Fi カバレッジ不完全（デッドゾーン存在） | P04 | Offline-First 設計を機能要件の Must 要件として確定する |
| 外国人技能実習生・特定技能労働者が 1 名以上在籍 | P05 | 多言語対応（最低限：日本語・英語）を機能要件の Must 要件として確定する |
| 規制当局への電子記録提出義務は現在発生していない | P06 | GMP/FDA Part 11 準拠は非機能要件の対象外。ALCOA+ は設計品質基準として採用 |
| IT 担当は 1 名でサーバー・ネットワークを兼務する | P07 | 運用自動化・Docker GUI ウィザード・自己引継ぎ設計を運用要件・保守要件に反映 |

### 技術スタックの継承

システム化計画 `07_技術スタック選定根拠.md` が確定した技術スタックを本書の制約として継承する。本書はこれらの選定根拠の再評価を行わない。

| 技術要素 | 本書での制約 |
|---|---|
| フロントエンド: React Native（Expo） + SQLite + TypeORM | ローカルストレージ要件はSQLite前提。Windows/Android/iOS タブレット対応を機能要件に反映 |
| バックエンド: Rust Edition 2024 + tokio + axum | API 仕様は Rust/axum の実装可能範囲内で確定する |
| データベース: PostgreSQL | Append-only Event Sourcing・ALCOA+ 実装はPostgreSQL前提 |
| デプロイ: IIS + Windows Server 2022 + WSL2 + Docker | 可用性要件・RTO/RPO はこのインフラ前提で確定する |
| マスタメンテナンス: React（Web） | 管理 Web の機能要件は React Web フロントエンド前提 |

## 4. 差分管理規約（上流文書と要件定義が齟齬を生じた場合の優先順位）

### 優先順位の定義

本書の要件記述と上流文書（構想・計画）の間に齟齬が生じた場合、以下の優先順位で解決する。

| 優先順位 | 文書 | 理由 |
|---|---|---|
| 1位（最優先） | システム化構想の倫理制約 EC-01〜EC-06 | 倫理制約は下流文書の合意によって覆らない設計制約として確定されている |
| 2位 | システム化計画（最新版） | 計画書は What/How-Frame の確定権威であり、要件定義は計画を継承する |
| 3位 | 本書（要件定義書、最新版） | 計画書の詳細化が本書の責務であり、計画書を覆す記述は移管対象 |

### 齟齬発生時の対処手順

上流文書と本書に齟齬が生じた場合は以下の手順で対処する。

1. **齟齬の分類**: 計画書の記述と本書の要件記述を比較し、齟齬の種類（誤謬・詳細化の過剰・新規発見事項）を分類する
2. **誤謬の場合**: 本書の要件記述を計画書に合わせて修正する
3. **詳細化の過剰の場合**: 本書から概要設計書へ移管し、本書の要件粒度に戻す
4. **新規発見事項の場合**: 計画書の変更提案として記録し、計画書改訂後に本書の要件を確定する

### 計画書の変更が本書に与える影響

計画書が改訂された場合、本書の以下の項目を必ず再確認する。

- `99_前提制約と本書が約束しないこと.md` の前提条件 P01〜P10
- `03_ステークホルダーと要件責任マトリクス.md` のステークホルダー定義
- 影響を受ける機能要件・非機能要件の受入基準
- 本章の継承マップ（テーマの追加・削除・変更がないか）

---

**本節で確定した方針**
- 上流文書との優先順位を「構想倫理制約 > 計画書 > 本書」の順で確定する。
- 齟齬発生時の 4 ステップ対処手順（分類・誤謬修正・移管・計画書改訂提案）を確定する。
- 計画書改訂時は本章継承マップを起点として影響範囲を確認することを確定する。
