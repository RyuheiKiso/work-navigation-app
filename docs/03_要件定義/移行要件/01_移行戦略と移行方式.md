# 01 移行戦略と移行方式

本書の責務は、既存の紙・Excel SOP および過去トレサビ記録を本システムへ移行するための戦略と方式を要件として確定することである。上流文書である「システム化計画 13 章（データ移行とマスタ初期投入戦略）」の計画方針を受けて、要件定義レベルで実施可能な具体的要件に落とし込む。移行の失敗は導入初日から業務を停止させる最大リスクであり、本書はそのリスクを構造的に排除するための要件集として機能する。

---

## 1. 移行の位置づけと IPA 共通フレームでの対応

### 1-1. IPA 共通フレーム 2013 における位置づけ

本書は IPA 共通フレーム 2013（SLCP-JCF2013）の「移行要件定義」タスクに対応する。共通フレームが定める移行要件定義タスクは「移行計画の作成」「移行対象データの特定」「移行方式の確定」「移行検証要件の明確化」の 4 要素を含む。本書はこれら 4 要素すべてを要件として記述する。

### 1-2. 本書と上流文書の分担

| 文書 | 扱う粒度 | 内容 |
|---|---|---|
| システム化計画 13 章 | 計画レベル | 移行方針の確定・取込手順の方針・カットオーバ 3 段階の方針 |
| 本書（移行要件） | 要件レベル | 移行要件 ID（MIG-NNN）の確定・品質基準の数値化・ツール提供要件の確定 |
| 概要設計（後続） | 設計レベル | 移行ツールのアーキテクチャ設計・テーブル定義・移行スクリプト設計 |

### 1-3. 移行要件の全体像

本書が定める移行要件の ID 範囲は MIG-001〜MIG-074 とする。ID は 6 つの副領域に分割する。

| ID 範囲 | 副領域 |
|---|---|
| MIG-001〜MIG-009 | 移行戦略・全体方針（データ対象・移行方式・フェーズ定義） |
| MIG-010〜MIG-013 | 移行ツール提供要件 |
| MIG-014〜MIG-033 | マスタ初期投入 |
| MIG-034〜MIG-043 | ロット履歴・過去記録引継ぎ |
| MIG-044〜MIG-063 | リハーサル・並行運用・カットオーバ |
| MIG-064〜MIG-074 | 移行判定基準・移行完了定義 |

**本節で確定した方針**
- 本書を IPA 共通フレーム 2013「移行要件定義」タスクの成果物として位置づけ、MIG-001〜MIG-074 の要件 ID 体系で管理する。
- 計画レベルの方針確定は 13 章に完了しており、本書は要件レベルの確定に専念する。
- 上流文書との分担境界は「計画：方針確定 / 要件：数値化と実施条件確定 / 設計：実装方式確定」の 3 層で管理する。

## 2. 移行対象と非対象

### 2-1. 移行対象データの確定

MIG-001: 以下の 5 種類のデータを移行対象として確定する。

| 移行対象 | 現在の保管形式 | 移行先 | 備考 |
|---|---|---|---|
| SOP マスタ（工程・作業・Step・判定基準） | 紙・Excel | PostgreSQL の SOP テーブル群 | 品質担当の電子署名を移行時に付与する |
| ユーザーマスタ（氏名・ロール・スキル情報） | 紙台帳・Excel | PostgreSQL の users テーブル | パスワードは初期値を設定し初回ログイン時変更を必須とする |
| 設備マスタ（設備名・設備番号・担当工程） | 紙台帳 | PostgreSQL の equipment テーブル | 保全記録は移行非対象 |
| 計測器マスタ（器名・校正証明書番号・有効期限） | 紙台帳 | PostgreSQL の instruments テーブル | 有効期限切れ器は移行時に警告フラグを付与する |
| 直近 1 年以内のロットマスタ（ロット番号・製品コード） | Excel・紙 | PostgreSQL の lots テーブル（is_legacy=true） | ALCOA+ 非準拠として管理する |

MIG-002: 移行対象データの確定は品質担当の書面承認を経て確定する。書面承認なしの移行作業開始を禁止する。

### 2-2. 移行非対象の確定

MIG-003: 以下のデータは移行対象外と判断する。

| 移行非対象 | 判断根拠 |
|---|---|
| 外部 ERP / MES のトランザクションデータ | 外部システムに保持されており、本システムの責務範囲外であるため |
| 1 年超前のロット完了記録 | 参照頻度が低く、移行コストに見合わないため。必要時は紙原本を参照する |
| 設備保全の詳細作業記録（過去分） | 設備保全システムが別途存在する場合はそちらに保持するため |
| 個人別の勤怠・生産性評価データ | 倫理境界宣言（計画 01 章）により本システムの対象外と判断する |
| OCR 変換前の生スキャン PDF（移行完了後） | 移行後に本システムへ取込済みと確認された SOP の生スキャンは外部ストレージで管理する |

MIG-004: 移行非対象と判断したデータの保管責任は移行完了後も既存システム・紙原本の管理者に帰属する。本システムはこれらのデータの参照を保証しない。

**本節で確定した方針**
- SOP マスタ・ユーザーマスタ・設備マスタ・計測器マスタ・直近 1 年のロットマスタの 5 種を移行対象として確定する。
- 外部システムデータ・1 年超前の記録・個人評価データは移行対象外と判断し、既存管理体制を維持する。
- 移行対象の確定は品質担当書面承認を必須条件とし、承認なしの移行開始を禁止する。

## 3. 移行方式の確定

### 3-1. 3 方式の定義

MIG-005: 移行方式として以下の 3 方式を確定する。方式は対象 SOP の特性に応じて選択し、1 つの SOP に複数方式を混在させることを禁止する。

**方式 1 — 手入力方式**

品質担当が本システムの Web 管理画面のマスタ登録フォームを使用して直接手入力する。工程・作業・Step の 3 階層に沿って 1 Step ずつ入力する。

- 適用基準: Step 数が 15 未満、または規制上の管理特性が指定されたクリティカル工程
- 品質保証要件: 入力完了後に現場監督が全 Step を照合確認し、品質担当が電子署名で承認する

**方式 2 — Excel テンプレート取込方式**

本システムが提供する Excel テンプレート（標準フォーマット）に既存 SOP を転記し、バルク取込 CLI ツールで一括投入する。

- 適用基準: Step 数が 15 以上、かつ Excel または構造化されたデータが存在する工程
- 品質保証要件: 取込前にバリデーション全通過・取込後にプレビュー確認（Web 管理画面）・品質担当の電子署名

**方式 3 — OCR 変換方式**

紙 SOP をスキャンして PDF 化し、OCR ソフトウェアでテキストに変換後、Excel テンプレートに手動マッピングして方式 2 で取込む。

- 適用基準: 電子データが存在しない、かつ Step 数が多く手入力コストが高い工程
- 品質保証要件: OCR 変換誤変換の目視照合を必須とし、誤変換ゼロ確認後に取込む

### 3-2. 方式選択基準

MIG-006: 方式選択は品質担当が以下のマトリクスに従って決定する。

| 電子データ有無 | Step 数 | クリティカル工程か | 採用方式 |
|---|---|---|---|
| 問わない | 問わない | はい | 方式 1（手入力）必須 |
| あり | 15 未満 | いいえ | 方式 1 または方式 2 |
| あり | 15 以上 | いいえ | 方式 2 推奨 |
| なし | 15 未満 | いいえ | 方式 1 必須 |
| なし | 15 以上 | いいえ | 方式 3 → 方式 2 |

### 3-3. 方式共通の品質保証要件

MIG-007: 方式によらず以下の品質保証要件を必須とする。

- 全移行 SOP に対して品質担当の電子署名を付与すること（ユーザー ID + タイムスタンプ + サーバー署名）
- 移行後プレビューで全 Step の表示確認を実施すること
- 移行ツールが出力するバリデーションレポートで「エラー 0 件」を確認してから本番環境に投入すること
- 取込完了後 24 時間以内に現場監督が代表 Step のナビゲーション動作確認を実施すること

**本節で確定した方針**
- 手入力・Excel テンプレート取込・OCR 変換の 3 方式を確定し、工程特性と Step 数に応じた選択基準を確定する。
- クリティカル工程への方式 1（手入力）必須適用と電子署名必須を品質保証の最低基準として確定する。
- 移行ツールのバリデーションレポートで「エラー 0 件」を確認してからの本番投入を必須とする。

## 4. 移行スケジュールとフェーズ

図: fig_migration_roadmap（img/ 配下）を参照

### 4-1. 4 フェーズの確定

MIG-008: 移行作業を以下の 4 フェーズに分割して実施する。

| フェーズ | 名称 | 期間目安 | 主要成果物 |
|---|---|---|---|
| フェーズ 1 | 移行準備 | 移行開始 2〜4 週間前 | 移行対象 SOP リスト・方式選択結果・Excel テンプレート準備完了 |
| フェーズ 2 | リハーサル | 本番移行 1〜2 週間前 | ステージング環境での移行リハーサル完了・不具合ゼロ確認 |
| フェーズ 3 | 本番移行 | カットオーバ日 | 全対象 SOP の本番環境投入・整合性チェック全通過・品質担当承認完了 |
| フェーズ 4 | 検証完了 | カットオーバ後 2 週間 | 並行運用完了・全移行検証チェック通過・移行完了報告書承認 |

### 4-2. フェーズ間の移行条件

MIG-009: 各フェーズの次フェーズへの移行は、前フェーズの完了条件を品質担当が書面確認した上で実施する。品質担当の確認なしに次フェーズに進むことを禁止する。

| フェーズ移行 | 完了条件 |
|---|---|
| フェーズ 1 → フェーズ 2 | 移行対象 SOP リスト確定・方式選択完了・ステージング環境構築完了 |
| フェーズ 2 → フェーズ 3 | リハーサルで不具合ゼロ・品質担当のリハーサル承認完了 |
| フェーズ 3 → フェーズ 4 | 本番投入件数確認・整合性チェック全通過・品質担当署名完了 |
| フェーズ 4 → 移行完了 | 並行運用期間終了・移行完了判定基準の全条件充足（MIG-060 参照） |

**本節で確定した方針**
- 移行準備・リハーサル・本番移行・検証完了の 4 フェーズと各フェーズの完了条件を確定する。
- フェーズ移行の承認権限は品質担当に集約し、承認なしのフェーズ先行を禁止する。
- ステージング環境での事前リハーサルを本番移行前の必須フェーズとして確定する。

## 5. 移行ツール提供要件

### 5-1. 移行 CLI ツールの提供要件

MIG-010: バルク取込 CLI ツールを ver1.0.0 の成果物として提供する。ツールは以下の機能を必須として実装する。

| 機能 | 仕様 |
|---|---|
| Excel / CSV バルク取込 | 標準フォーマットの xlsx または CSV ファイルを受け付け、PostgreSQL に一括投入する |
| バリデーション実行 | 必須フィールド完全性・データ型整合・参照整合性の 3 種を投入前に検証する |
| バリデーションレポート出力 | エラー・警告を CSV 形式で出力し、担当者が修正後に再投入できる形式とする |
| ドライラン（試し投入）モード | `--dry-run` オプションで実際のデータ変更なしにバリデーションのみ実行できる |
| 移行ログ出力 | 投入件数・スキップ件数・エラー件数を構造化ログ（JSON）で記録する |

MIG-011: 取込テンプレートのスキーマを ver1.0.0 リリース時点で凍結する。凍結後の列定義変更は後方互換性を保証した上でマイナーバージョンアップとして管理する。既存テンプレートの無効化は禁止する。

### 5-2. Web 管理画面での手入力支援要件

MIG-012: Web 管理画面のマスタ登録フォームは以下の手入力支援機能を提供する。

| 機能 | 仕様 |
|---|---|
| Step 自動採番 | Step 追加時に連番を自動付与し、空番防止を保証する |
| コピー機能 | 既存 SOP の工程・作業・Step 構造をコピーして新規 SOP の雛形として使用できる |
| 入力途中保存（Draft） | 入力完了前に Draft 状態で保存し、セッション切断時のデータ消失を防止する |
| プレビュー表示 | 取込後のナビゲーション画面を取込確認時に表示し、作業員目線での確認を可能にする |

MIG-013: 移行ツールのソースコードはシステム本体のリポジトリと同一の Git リポジトリで管理し、移行完了後も保守対象として維持する。移行専用の使い捨てスクリプトとして実装することを禁止する。

**本節で確定した方針**
- バルク取込 CLI ツールを ver1.0.0 成果物として提供し、ドライランモード・バリデーションレポート・移行ログを必須機能として確定する。
- 取込テンプレートスキーマを ver1.0.0 リリース時点で凍結し、後方互換性を保証した上でのみ変更を認める。
- 移行ツールをシステム本体リポジトリで保守対象として管理し、使い捨てスクリプト化を禁止する。
