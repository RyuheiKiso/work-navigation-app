# 01 可用性要件

本章は IPA 非機能要求グレード「可用性」カテゴリに対応し、製造現場の 2〜3 シフト稼働を前提とした稼働率目標・RTO/RPO・計画停止条件・DR 範囲・データ復旧手続を確定する。計画 08 章 3 節で方針として確定した RTO 1 時間・RPO 15 分を SLO 数値として具体化する。

---

## 1. 稼働率目標と計測方法

### 1-1. 稼働率 SLO の確定

| 要件 ID | NFR-AVL-001 |
|---|---|
| 要件名 | 稼働率目標 |
| 目標値 | 営業時間（6:00-22:00）ベースで年間稼働率 99.5% 以上 |
| IPA グレード | レベル 2〜3 中間 |
| 計測方法 | 月次に以下を記録する。計画停止時間および障害停止時間を「6:00-22:00 の総時間」で除した値を月次稼働率とする |
| 計算式 | 稼働率 = (計測対象時間 - 障害停止時間) / 計測対象時間 × 100 |
| 受入基準 | 月次稼働率が 3 ヶ月連続で 99.0% を下回る場合はインフラ構成の見直しを必須とする |

稼働率 99.5% は年間（248 営業日×16 時間 = 3,968 時間）において、障害による停止時間が 19.8 時間以内であることを意味する。計画停止は別途管理する（1-3 節参照）。

### 1-2. IPA グレード選定根拠

IPA グレードのレベル 2（一般業務システム水準）とレベル 3（ミッション重要度高水準）の中間を選定する。根拠は以下のとおりである。

- 製造ラインの作業ナビゲーションは、停止が直接的な生産停滞を引き起こすため、レベル 2（99% 程度）では不十分である。
- 一方、医療・金融・インフラ系ミッションクリティカル（レベル 4〜5: 99.99%+）の要求水準は、単一工場オンプレ・IT 担当 1 名という制約から達成不可能である。
- 99.5% は Offline-First 設計（サーバー障害時も端末が継続稼働）によって実質的な業務影響を最小化しつつ、インフラコストが現実的な水準である。

### 1-3. SLI の定義と計測基盤

| SLI 名 | 定義 |
|---|---|
| server_availability | API サーバーが HTTP 2xx を返せる状態の割合（6:00-22:00 基準）|
| db_availability | PostgreSQL への接続が可能な状態の割合 |
| sync_availability | Outbox キューの Consumer が稼働している割合 |

計測は IT 担当が週次手動確認（ヘルスチェックエンドポイントへの疎通確認・DB 接続テスト）で実施する。リアルタイム監視 SaaS の導入は対象外と判断する（個人開発・IT 担当 1 名の制約）。

**本節で確定した方針**
- 稼働率目標を営業時間ベース 99.5% 以上として確定し、IPA グレードのレベル 2〜3 中間選定の根拠を明示する。
- server_availability / db_availability / sync_availability の 3 SLI を定義し、週次手動確認を計測方法として確定する。
- リアルタイム監視 SaaS の導入は IT 担当 1 名・個人開発の制約から対象外と判断する。

## 2. RTO / RPO 定義

### 2-1. SLO 数値の確定

| 要件 ID | NFR-AVL-002 |
|---|---|
| 要件名 | RTO / RPO |
| RTO | 1 時間以内（Recovery Time Objective: 障害発生から業務再開まで）|
| RPO | 15 分以内（Recovery Point Objective: 障害発生前にどこまでデータを復旧できるか）|
| 計画 08 章との整合 | 計画 08 章 3 節で「RTO 1 時間・RPO 15 分」を方針として確定済み。本節でその値を SLO として確定する |

### 2-2. RTO 達成手段

| 手段 | 内容 |
|---|---|
| Active-Standby 構成 | LAN 内に予備サーバーを配置し、プライマリ障害時に切替操作（手動）で 1 時間以内に復旧する |
| Docker Compose 再起動 | コンテナの異常終了時は自動再起動ポリシー（restart: always）で自動復旧する |
| ドキュメント駆動の復旧手順書 | 復旧手順を IT 担当 1 名が一人で実行できる手順書として整備する（GUI 操作ベース）|

RTO 1 時間はオートフェールオーバー（30 秒以内）を意味しない。手動切替での 1 時間以内の復旧を達成目標とする。Offline-First 設計により、サーバー停止中もタブレット端末での作業記録は継続できるため、業務への実質的影響を最小化できる。

### 2-3. RPO 達成手段

| 手段 | 内容 |
|---|---|
| PostgreSQL WAL ストリーミング | WAL を 5 分間隔で NAS に転送し、PITR（Point-in-Time Recovery）で 15 分以内の時点に復旧する |
| Offline-First バッファ | タブレット端末のローカルキューに保持された未同期イベントは、サーバー復旧後に自動同期されるため、RPO の実質的影響が軽減される |

**本節で確定した方針**
- RTO 1 時間・RPO 15 分を SLO 値として確定し、計画 08 章方針からの継承を明示する。
- RTO 達成を手動切替＋Docker 自動再起動で実現し、オートフェールオーバーは達成目標に含めないことを確定する。
- RPO 達成を WAL PITR（5 分間隔転送）と Offline-First バッファの組み合わせで実現することを確定する。

## 3. 計画停止・障害復旧時間

### 3-1. 計画停止条件

| 要件 ID | NFR-AVL-003 |
|---|---|
| 要件名 | 計画停止上限 |
| 条件 | 計画停止は夜間（22:00 以降）または休日のみ実施する |
| 時間上限 | 1 回あたり最大 2 時間 |
| 頻度上限 | 月 1 回まで |
| 事前通知 | 計画停止は実施 3 日前までに全作業員・現場監督に通知する |

| 要件 ID | NFR-AVL-004 |
|---|---|
| 要件名 | 緊急パッチの停止条件 |
| 条件 | 重大脆弱性（CVSS スコア 9.0 以上）の緊急パッチは営業時間内の短停止（最大 15 分）を許容する |
| 通知方法 | 実施 30 分前に現場監督・品質担当に通知する |

### 3-2. 障害復旧目標

| 要件 ID | NFR-AVL-005 |
|---|---|
| 要件名 | 障害発生から初動対応までの時間 |
| 目標値 | IT 担当への通知から 15 分以内に初動対応（ステータス確認・原因特定開始）を開始する |
| 前提 | 自動通知はタブレット端末からの手動報告またはスケジュール確認で検知する（自動アラートシステムは対象外と判断する）|

**本節で確定した方針**
- 計画停止を夜間・月 1 回・最大 2 時間・3 日前通知の 4 条件で確定する。
- 重大脆弱性（CVSS 9.0+）の緊急パッチは最大 15 分の短停止を許容することを確定する。
- 障害発生から 15 分以内の初動対応を NFR-AVL-005 として確定するが、自動通知システムは対象外と判断する。

## 4. DR 範囲とオンプレ単一工場の限界

### 4-1. DR 方針の確定

| 要件 ID | NFR-AVL-010 |
|---|---|
| 要件名 | 地理冗長の範囲 |
| 方針 | 地理冗長（異なる建屋・異なるデータセンターへの複製）は対象外と判断する |
| 根拠 | 前提 P01（単一工場）・P02（社内 LAN 完結）・個人開発・IT 担当 1 名の 4 制約。単一工場前提では地理冗長の技術コスト・運用コストが便益を上回る |
| 許容リスク | 建屋全体に影響する火災・大規模停電等の広域障害は、本システムのリカバリ対象外とする |

| 要件 ID | NFR-AVL-011 |
|---|---|
| 要件名 | 同一建屋内冗長構成 |
| 方針 | 同一建屋内 Active-Standby 構成を採用する。プライマリサーバーと予備サーバーを同一 LAN 内に配置する |
| 手順 | 切替は手動操作（DNS 切替または IP 切替）で実施する。切替手順書の整備を必須とする |

| 要件 ID | NFR-AVL-012 |
|---|---|
| 要件名 | オフサイトバックアップ |
| 方針 | 週次で可搬媒体（外付け HDD）にバックアップを取得し、別棟（または建屋外）に保管する |
| 頻度 | 週 1 回（営業日外に実施）|
| 暗号化 | AES-256 暗号化を適用する（鍵は別管理）|

**本節で確定した方針**
- 地理冗長 DR を P01・P02 の前提制約から対象外と判断し、許容リスクを明示する。
- 同一建屋内 Active-Standby（手動切替）+ 週次オフサイト媒体保管を DR 構成として確定する。
- オフサイトバックアップは AES-256 暗号化・鍵別管理を必須とすることを確定する。

## 5. データ復旧手続の要件

### 5-1. PostgreSQL PITR 復旧手順要件

| 要件 ID | NFR-AVL-015 |
|---|---|
| 要件名 | WAL PITR 復旧手順 |
| 要件内容 | IT 担当 1 名が手順書のみで PostgreSQL WAL PITR を実行し、任意の時点（15 分以内の精度）にデータを復旧できること |
| 手順書要件 | GUI またはコマンドラインの手順を段階的に記述する。専門知識なしで実行可能であること |

| 要件 ID | NFR-AVL-016 |
|---|---|
| 要件名 | 日次 pg_dump 復旧手順 |
| 要件内容 | WAL PITR が利用不可の場合の代替として、日次 pg_dump から復旧できること。復旧所要時間は 1 時間以内（RTO と整合）|
| バックアップ保存先 | NAS の専用ディレクトリ。世代管理: 日次 7 世代・週次 13 世代・月次 12 世代 |

### 5-2. 復旧テスト要件

| 要件 ID | NFR-AVL-020 |
|---|---|
| 要件名 | 復旧テストの定期実施 |
| 頻度 | 年次 1 回（任意の営業日外に実施）|
| 内容 | ステージング環境でバックアップからの完全復旧テストを実施する。RTO 1 時間・RPO 15 分の達成を確認する |
| 記録 | 復旧テスト結果を記録し、IT 担当・管理者が確認する |

**本節で確定した方針**
- WAL PITR 復旧手順を IT 担当 1 名・手順書のみで実行可能な形式で整備することを確定する。
- pg_dump バックアップは日次 7 世代・週次 13 世代・月次 12 世代の世代管理を確定する。
- 年次 1 回の復旧テストを必須とし、RTO 1 時間・RPO 15 分の達成確認を受入基準とすることを確定する。
