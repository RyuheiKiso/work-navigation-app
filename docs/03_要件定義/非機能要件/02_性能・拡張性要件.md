# 02 性能・拡張性要件

本章は、同時接続数・応答時間 SLI・Outbox スループット・5 年データ量見積もり・スケーラビリティ前提を確定する。計画 08 章 2 節で方針として確定した「Glanceable 1 秒以内の状態認識」を計測可能な SLO 値として具体化する。

---

## 1. 同時接続数

### 1-1. 想定規模の確定

| 要件 ID | NFR-PER-001 |
|---|---|
| 要件名 | 最大同時接続数 |
| 目標値 | 500 名同時接続 |
| IPA グレード | レベル 2 |
| 根拠 | 単一工場の作業員 500 名を上限として設定する。中小製造業の現実的な最大規模として設定し、超過した場合は前提 P01（単一工場）の変更として扱う |

### 1-2. 接続種別の内訳

| 接続種別 | 想定接続数 | 備考 |
|---|---|---|
| ハンディ APP（タブレット端末）| 最大 400 接続 | 作業員が 2〜3 シフトで使用 |
| マスタメンテナンス APP | 最大 10 接続 | 品質担当・マスタ編集者・現場監督 |
| 管理コンソール | 最大 10 接続 | IT 担当・品質担当・管理者 |
| Outbox Consumer（サービス）| 最大 5 プロセス | バックグラウンドサービス |
| 合計 | 最大 425 接続（安全係数 1.2 倍: 510 接続）| 500 接続を目標値として設定 |

**本節で確定した方針**
- 最大同時接続数を 500 名として確定し、IPA グレードのレベル 2 を選定する。
- 接続種別を HA / MA / MC / Outbox Consumer の 4 種に分類し、実態に即した割り当てを確定する。
- 500 名超過時は前提 P01 変更として設計再評価を必須とすることを確定する。

## 2. 応答時間 SLI 目標

### 2-1. SLO 値の確定

計画 08 章 2 節で確定した方針値を SLO として以下に確定する。

| 要件 ID | NFR-PER-002 |
|---|---|
| 要件名 | Step 完了記録の応答時間 |
| 目標値（P95）| 200ms 以内 |
| 前提 | ローカル SQLite クエリのみ。サーバー通信が完了するまでブロックしない |

| 要件 ID | NFR-PER-003 |
|---|---|
| 要件名 | 画面遷移の応答時間 |
| 目標値（P95）| 500ms 以内 |
| 前提 | ローカル SQLite クエリ。次 Step の手順テキスト・写真 URL のローカルキャッシュ参照 |

| 要件 ID | NFR-PER-004 |
|---|---|
| 要件名 | 管理 Web クエリの応答時間 |
| 目標値（P95）| 3 秒以内 |
| 前提 | LAN 内 PostgreSQL クエリ。トレサビ照会・集計クエリを含む |

| 要件 ID | NFR-PER-005 |
|---|---|
| 要件名 | バーコード / QR スキャンの処理時間 |
| 目標値 | 1 秒以内（スキャン開始からデコード結果表示まで）|
| 前提 | ML Kit によるオフライン処理。カメラフレームレート 30fps 以上 |

| 要件 ID | NFR-PER-006 |
|---|---|
| 要件名 | マスタ検索の応答時間（マスタメンテナンス APP）|
| 目標値（P95）| 200ms 以内 |
| 前提 | 計画 08 章 2 節「管理 Web 編集 UX 非機能要件」からの継承 |

### 2-2. Glanceable UI との関係

計画 08 章 2 節で確定した「Glanceable 1 秒以内の状態認識（Endsley SA レベル 1）」は、現在ステップ表示の NFR-PER-002（200ms 以内）によって達成される。画面遷移後 200ms 以内に現在ステップのテキストと次アクションボタンが表示されることで、1 秒以内の状態認識が実現される。

**本節で確定した方針**
- Step 完了記録 P95=200ms / 画面遷移 P95=500ms / 管理 Web クエリ P95=3 秒の 3 SLO を確定する。
- 全 SLO はローカル SQLite またはサーバー不到達時も達成可能であることを設計制約として確定する。
- バーコードスキャン 1 秒以内を NFR-PER-005 として確定し、ML Kit オフライン処理を前提とする。

## 3. Outbox 同期スループット

### 3-1. スループット目標

| 要件 ID | NFR-PER-010 |
|---|---|
| 要件名 | Outbox 送信スループット |
| 目標値 | 150,000 イベント/日を処理できること |
| 計算根拠 | 500 名 × 100 Step/日 × 3 シフト = 150,000 イベント/日（最大値）|
| ピーク対応 | シフト交代時（6:00・14:00・22:00 前後）に同期が集中する。ピーク時に 10,000 イベント/時間を処理できること |

### 3-2. Outbox Consumer の設計要件

| 要件 ID | NFR-PER-011 |
|---|---|
| 要件名 | Consumer ポーリング間隔 |
| 目標値 | PENDING レコードの検知から送信開始まで 30 秒以内 |
| 設計方針 | Consumer は 30 秒間隔でポーリングし、バッチサイズ 100 件で送信する |

| 要件 ID | NFR-PER-012 |
|---|---|
| 要件名 | Outbox Consumer の冗長性 |
| 方針 | Consumer プロセスを Docker Compose の restart: always で管理し、クラッシュ時の自動復旧を保証する |

**本節で確定した方針**
- Outbox スループット目標を 150,000 イベント/日・ピーク 10,000 イベント/時間として確定し、計算根拠を明示する。
- Consumer ポーリング 30 秒間隔・バッチ 100 件を設計要件として確定する。
- Consumer プロセスの Docker restart: always による自動復旧を確定する。

## 4. データ量 5 年見積もり

### 4-1. 5 年後データ量の計算

| 要件 ID | NFR-PER-015 |
|---|---|
| 要件名 | 5 年後データ量上限 |
| 目標値 | PostgreSQL の総データ量が 1.5TB 以下 |
| 計算根拠 | 以下のとおり |

**WorkEvent テーブルの見積もり**
- 1 イベントあたりのレコードサイズ（概算）: 1KB（payload JSONB 含む）
- 1 日のイベント数: 150,000 件
- 1 年のイベント数: 150,000 × 250 営業日 = 37,500,000 件
- 5 年のイベント数: 187,500,000 件
- WorkEvent テーブルサイズ: 187,500,000 × 1KB = 約 178GB

**写真ファイル（NAS）の見積もり**
- 1 日の写真枚数: 500 名 × 10 枚 = 5,000 枚
- 1 枚あたりのサイズ（圧縮後）: 約 300KB
- 1 年の写真データ: 5,000 × 300KB × 250 日 = 約 375GB
- 5 年の写真データ: 約 1,875GB（NAS 管理。PostgreSQL 非格納）

**PostgreSQL その他テーブル**
- MasterVersion・User・Equipment・CAPA・Kaizen 等: 合計 50GB 以内（見積もり）

**PostgreSQL 合計（5 年後）**
- WorkEvent: 178GB + インデックス 50% = 約 267GB
- その他テーブル: 50GB
- WAL・バッファ: 50GB
- **合計: 約 367GB**（目標の 1.5TB 以内で余裕あり）

### 4-2. ディスク要件

| 要件 ID | NFR-PER-016 |
|---|---|
| 要件名 | サーバーディスク容量 |
| 目標値 | SSD 1TB 以上（OS・ミドルウェア含む）|
| NAS 容量 | 4TB 以上（写真 1,875GB + バックアップ 2 世代分 750GB + 余裕）|

**本節で確定した方針**
- 5 年後 PostgreSQL データ量を約 367GB（目標 1.5TB 以内）と見積もり、計算根拠を明示する。
- 写真ファイルは NAS 管理（PostgreSQL 非格納）とし、NAS 容量目標を 4TB 以上として確定する。
- サーバー SSD 1TB 以上の要件を確定する。

## 5. スケーラビリティ前提と対象外

### 5-1. ver1.0.0 のスケーラビリティ方針

| 要件 ID | NFR-PER-020 |
|---|---|
| 要件名 | 水平スケールアウトの対象外判断 |
| 方針 | 水平スケールアウト（複数 API サーバーへの負荷分散）は対象外と判断する |
| 根拠 | 前提 P01（単一工場）・500 名の想定接続数は単一 API サーバーで対応可能。複数サーバー構成は Outbox Consumer の競合制御・セッション管理に追加の複雑性を生む |

| 要件 ID | NFR-PER-021 |
|---|---|
| 要件名 | 垂直スケールアップの方針 |
| 方針 | パフォーマンス不足が発生した場合は垂直スケールアップ（CPU・メモリ・SSD の増強）で対応する |
| 手順 | スケールアップが必要になった場合は ADR を作成し、インフラ変更を記録する |

| 要件 ID | NFR-PER-022 |
|---|---|
| 要件名 | 将来の水平スケーラビリティ余地 |
| 方針 | API サーバーをステートレス設計（セッション状態を DB に持つ）とし、将来的な水平スケールへの設計余地を確保する。ただし ver1.0.0 では単一サーバー構成で動作することを確認する |

**本節で確定した方針**
- 水平スケールアウトを P01 前提・500 名規模から対象外と判断し、垂直スケールアップで対応することを確定する。
- API サーバーをステートレス設計とし、将来の水平スケーラビリティへの設計余地を確保することを確定する。
- スケールアップ時は ADR を作成しインフラ変更を記録することを確定する。
