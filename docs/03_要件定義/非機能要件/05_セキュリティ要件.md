# 05 セキュリティ要件

本章は、計画 08 章 4 節・計画 15 章（セキュリティ深堀り）で確定したセキュリティ方針を NFR-SEC 識別子付きの具体的要件として確定する。認証・認可・データ機密性・ネットワーク・改ざん検知・電子サイン・ALCOA+ 9 原則マトリクス・脆弱性管理の 8 節で体系化する。

---

## 1. 認証要件

### 1-1. JWT 認証の確定

| 要件 ID | NFR-SEC-001 |
|---|---|
| 要件名 | JWT 署名アルゴリズム |
| 要件内容 | すべての API 認証に JWT（RS256 署名）を使用する。HS256（対称鍵）は使用禁止とする |

| 要件 ID | NFR-SEC-002 |
|---|---|
| 要件名 | JWT 有効期限 |
| 要件内容 | JWT のアクセストークン有効期限を 8 時間とする（シフト 1 本相当）。リフレッシュトークンは 24 時間とする |

| 要件 ID | NFR-SEC-003 |
|---|---|
| 要件名 | JWT 署名鍵ローテーション |
| 要件内容 | JWT RS256 署名鍵を 90 日周期でローテーションする。ローテーション手順: 新鍵生成 → grace period 24 時間の並行稼働 → 旧鍵廃止 |

| 要件 ID | NFR-SEC-004 |
|---|---|
| 要件名 | 鍵の保管 |
| 要件内容 | JWT 署名鍵（秘密鍵・公開鍵ペア）は環境変数または Docker Secret で管理する。ソースコード・リポジトリへの平文ハードコーディングを禁止する。鍵の形式は PEM 形式の RSA 4096bit |

| 要件 ID | NFR-SEC-005 |
|---|---|
| 要件名 | MFA の非適用 |
| 方針 | MFA（多要素認証）は対象外と判断する |
| 根拠 | 社内 LAN 完結・単一工場・製造現場での手袋操作という 3 条件のもと、MFA は操作性を著しく低下させる。JWT RS256 + RBAC 6 ロール + TLS 1.3 の組み合わせで必要十分なセキュリティレベルを実現する |

**本節で確定した方針**
- JWT RS256・8 時間有効期限・90 日鍵ローテーション・RSA 4096bit・Docker Secret 保管を認証要件として確定する。
- HS256 を禁止し、鍵の平文ハードコーディングを設計レベルで禁止することを確定する。
- MFA を社内 LAN 完結・製造現場操作性の制約から対象外と判断する。

**本節で確定した方針**
- RBAC 6 ロール × 15 機能の権限マトリクスを確定し、実装の基準とする。
- IT 担当の worker_id 実名参照禁止を RBAC 制約として確定し、計画 08 章の「IT 担当は作業ログの直接参照不可」方針を要件として具体化する。
- 電子サイン（完了）を作業員・現場監督・管理者に限定し、品質担当への誤付与を防止する。

## 3. データ機密性要件

### 3-1. PII 5 分類と取扱方針

| 要件 ID | NFR-SEC-020 |
|---|---|
| 要件名 | PII 分類と取扱方針 |
| 要件内容 | 本システムが取り扱う PII を以下の 5 分類で管理する |

| PII 種別 | 分類 | 取扱方針 | 保存期間 | 廃棄方針 |
|---|---|---|---|---|
| 氏名 | 直接識別子 | 表示は権限ある RBAC ロールのみ。検索・一覧はマスク表示 | 在籍中 + 退職後 60 日 | 退職後 60 日以内に匿名化（UUID 置換）|
| 電子サイン（worker_id 紐付き）| 間接 PII | 品質担当・管理者のみ参照可。ElectronicSign テーブルに格納 | 7 年以上 | 退職後 60 日以内に worker_id を内部 UUID に置換 |
| 作業記録（誰が・いつ・何を）| 間接 PII | 用途三限定（品質保証・工程改善・教育設計）のみ。RBAC でアクセス制御 | 7 年以上 | 退職後 60 日以内に worker_id を UUID 置換（記録は保全）|
| 計測値（worker_id 紐付き）| 間接 PII | WorkEvent の payload に含まれる。RBAC で制御 | 7 年以上 | worker_id の UUID 置換で対応 |
| CAPA・不適合記録の報告者 ID | 間接 PII | 品質担当・管理者のみ実名参照可 | 7 年以上 | 退職後 60 日以内に UUID 置換 |

**本節で確定した方針**
- PII を 5 分類（氏名・電子サイン・作業記録・計測値・報告者 ID）で確定し、各分類の取扱方針・保存期間・廃棄方針を明示する。
- 退職者の PII を 60 日以内に UUID 置換する匿名化処理を必須とし、証拠の完全性と削除権を両立させることを確定する。
- 用途三限定（品質保証・工程改善・教育設計）を RBAC とデータモデルで技術的に強制することを確定する。

## 4. ネットワーク要件

### 4-1. 通信セキュリティ

| 要件 ID | NFR-SEC-030 |
|---|---|
| 要件名 | TLS 1.3 必須 |
| 要件内容 | すべての通信（タブレット↔API サーバー / 管理 Web↔API サーバー / 親機↔子機）に TLS 1.3 以上を使用する。HTTP（平文）による通信を禁止する。社内 LAN 内の通信も例外としない |

| 要件 ID | NFR-SEC-031 |
|---|---|
| 要件名 | インターネット経由アクセスの禁止 |
| 要件内容 | 本システムへのアクセスはすべて社内 LAN 経由とする。インターネットからのアクセス経路を設けない（前提 P02: 社内 LAN 完結）|

| 要件 ID | NFR-SEC-032 |
|---|---|
| 要件名 | ゼロトラスト原則の適用 |
| 要件内容 | 社内 LAN 内の通信であっても「信頼できる」とは見なさない。すべての API リクエストで JWT 検証を実施する。検証失敗時はデフォルトで 403 Forbidden を返す |

| 要件 ID | NFR-SEC-033 |
|---|---|
| 要件名 | API サーバーの最小ポート公開 |
| 要件内容 | Docker コンテナ外部への公開ポートは API サーバーの HTTPS ポート（443 または 8443）のみとする。DB ポート（5432）をコンテナ外部に公開することを禁止する |

図: fig_network_zoning（img/ 配下）を参照

**本節で確定した方針**
- TLS 1.3 を社内 LAN 内通信を含む全通信に必須とし、HTTP 平文を禁止することを確定する。
- ゼロトラスト原則を適用し、全 API リクエストで JWT 検証を必須とすることを確定する。
- API サーバーの HTTPS ポートのみ外部公開し、DB ポートの外部公開を禁止することを確定する。

## 5. 不正追跡・改ざん検知

### 5-1. 改ざん検知要件

| 要件 ID | NFR-SEC-040 |
|---|---|
| 要件名 | SHA-256 ハッシュチェーン |
| 要件内容 | WorkEvent テーブルの各レコードに前レコードの SHA-256 ハッシュ（prev_hash）を格納する。定期バッチ（週次）でチェーン全体の整合性を検証する |

| 要件 ID | NFR-SEC-041 |
|---|---|
| 要件名 | 改ざん検知時の対応 |
| 要件内容 | ハッシュチェーン不整合が検出された場合、管理者に即時通知する（管理コンソールの警告バナー + IT 担当への社内メール）。不整合の範囲（最初の不整合 event_id から末尾まで）を特定し、ADR に記録する |

| 要件 ID | NFR-SEC-042 |
|---|---|
| 要件名 | 完全な操作ログ |
| 要件内容 | 作業実績の全 CRUD・認証イベント・SOP 改訂・子機連携イベントを監査ログの記録対象とする（計画 15 章 5 節参照）|

**本節で確定した方針**
- SHA-256 ハッシュチェーンを WorkEvent テーブルに実装し、週次で整合性検証を実施することを確定する。
- 改ざん検知時の即時通知・ADR 記録・不整合範囲特定を対応手順として確定する。
- 4 種の監査対象イベント（CRUD / 認証 / SOP 改訂 / 連携）の完全記録を確定する。

## 6. 電子サイン要件

### 6-1. 電子サインの実装要件

| 要件 ID | NFR-SEC-045 |
|---|---|
| 要件名 | 電子サインの構成要素 |
| 要件内容 | ElectronicSign レコードには以下を必須とする。signer_id（ALCOA+ Attributable）/ timestamp_device（端末時刻・ALCOA+ Contemporaneous）/ timestamp_server（サーバー受信時刻・ALCOA+ Contemporaneous）/ server_signature（サーバー側 HMAC 署名・ALCOA+ Original）|

| 要件 ID | NFR-SEC-046 |
|---|---|
| 要件名 | 電子サインの認証再確認 |
| 要件内容 | 電子サイン操作は必ずサインアップ済みユーザーが現在のセッションで PIN またはパスワードを再入力することで実施する。セッション継続状態での「ワンクリック署名」は認めない |

| 要件 ID | NFR-SEC-047 |
|---|---|
| 要件名 | 電子サインの遡及変更禁止 |
| 要件内容 | 記録済みの ElectronicSign レコードへの変更・削除を禁止する。PostgreSQL のロール権限で UPDATE / DELETE を付与しない |

### 6-2. Two-Person Integrity（二者独立性）

| 要件 ID | NFR-SEC-048 |
|---|---|
| 要件名 | リワーク・ディスポジションの二者電子サイン分離 |
| 要件内容 | リワーク作業のディスポジション（処遇判定）は、品質担当（quality_admin ロール）と現場監督（supervisor ロール）の 2 名が**独立して**電子署名することを必須とする。同一 worker_id による二重署名を DB トリガ（check_disposition_distinct_signers）で禁止する。|
| 根拠 | Just Culture 原則（作業員を決定責任から保護）+ ISO 9001:2015 §8.7.1 不適合処置の責任分離 |
| 実装詳細 | dispositions テーブルに quality_admin_sign_id / supervisor_sign_id の 2 FK を設ける。API 受信時に signer_id の不一致を検証（ERR-BIZ-021）。DB トリガで ElectronicSign の worker_id が一致する場合は INSERT を拒否する |
| 同一規則の適用範囲 | 廃却処理（scrap_records.witness_id ≠ rework 実施者）・再検査（rework_verifications.verifier_id ≠ リワーク実施者、ERR-BIZ-023）にも同じ Two-Person Integrity を適用する |

**本節で確定した方針**
- ElectronicSign の必須構成要素（signer_id / timestamp_device / timestamp_server / server_signature）を確定し、ALCOA+ 対応原則を明示する。
- 署名操作に PIN/パスワード再入力を必須とし、セッション継続状態での自動署名を禁止することを確定する。
- ElectronicSign への物理削除禁止を DB 権限レベルで実装することを確定する。
- **リワーク・ディスポジションの二者電子サイン分離（NFR-SEC-048）を確定する**。同一 worker_id による二重署名を DB トリガで排除し、ディスポジション決定における独立二者の承認を技術的に保証する。

## 7. ALCOA+ 9 原則対応マトリクス

| ALCOA+ 原則 | 対応する NFR-SEC 要件 ID | 実装概要 |
|---|---|---|
| Attributable（帰属可能）| NFR-SEC-010, NFR-SEC-045, NFR-SEC-048 | RBAC 6 ロール・電子サインに signer_id 必須・二者独立署名で決定責任を明確化 |
| Legible（読取可能）| NFR-SEC-030 | TLS 1.3・HTTPS 強制で改竄なしの転送を保証 |
| Contemporaneous（同時記録）| NFR-SEC-045, NFR-SEC-046 | timestamp_device + timestamp_server 双方保持 |
| Original（元データ）| NFR-SEC-040, NFR-SEC-047 | Append-only + ハッシュチェーン + 削除禁止 |
| Accurate（正確）| NFR-SEC-032 | JWT 検証必須・なりすまし防止 |
| Complete（完全）| NFR-SEC-042 | 4 種の監査対象イベント完全記録 |
| Consistent（一貫）| NFR-SEC-010, NFR-SEC-020, NFR-SEC-048 | RBAC で権限の一貫性を技術的に保証・DB トリガで二者署名の一貫性を強制 |
| Enduring（耐久）| NFR-SEC-040, NFR-SEC-047 | ハッシュチェーン + 削除禁止で永続性を保証 |
| Available（利用可能）| NFR-SEC-042 | 監査ログの検索・エクスポートを品質担当に提供 |

**本節で確定した方針**
- ALCOA+ 9 原則の各原則と NFR-SEC 要件 ID の対応マトリクスを確定し、証拠品質の実装基盤を明示する。
- 全 9 原則に対応する NFR-SEC 要件が存在することを確認し、未対応原則をゼロとすることを確定する。

## 8. 脆弱性管理

### 8-1. 依存ライブラリの脆弱性管理

| 要件 ID | NFR-SEC-050 |
|---|---|
| 要件名 | cargo audit 週次実行 |
| 要件内容 | Rust 依存ライブラリの脆弱性スキャンを cargo audit で週次実行する。脆弱性が検出された場合は NFR-OPS-010/041 のパッチ適用 SLO に従い対処する |

| 要件 ID | NFR-SEC-051 |
|---|---|
| 要件名 | npm audit 週次実行 |
| 要件内容 | React Native / React の依存ライブラリを npm audit で週次スキャンする。High 以上の脆弱性は 30 日以内に対処する |

| 要件 ID | NFR-SEC-052 |
|---|---|
| 要件名 | OWASP Top 10 対策 |
| 要件内容 | OWASP Top 10（2021 年版）に対応する。特に以下の 3 項目を優先する。A01 Broken Access Control: RBAC 6 ロールで対応する / A02 Cryptographic Failures: TLS 1.3 + AES-256 + RSA 4096bit で対応する / A03 Injection: sqlx のパラメータバインディング・JSON Logic サンドボックスで対応する |

**本節で確定した方針**
- cargo audit / npm audit の週次実行を脆弱性管理の基本手順として確定する。
- OWASP Top 10（2021）への対応を確定し、A01 / A02 / A03 の対策実装を優先とする。
- 重大脆弱性（CVSS 9.0+）の 7 日以内パッチ適用を NFR-OPS-010 と整合した SLO として確定する。
