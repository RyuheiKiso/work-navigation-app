# 非機能要件サブ

## 1. 本サブの責務

非機能要件サブは IPA 共通フレーム 2013 要件定義プロセスの「システム要件の定義（非機能要件）」アクティビティに相当する。

本サブが確定する内容は以下の 2 点に限定する。

- **非機能要件（NFR）**: 品質特性ごとの数値目標・グレード採否。ISO/IEC 25010 の品質特性モデルを基盤とし、本システムに適用する 9 種の品質特性ファイル（01〜09）で管理する NFR を確定する
- **グレード採否一覧**: 各品質特性の適用グレードを「採用する」「対象外と判断する」のいずれかで確定する

非機能要件サブは機能要件サブの要件に対して品質目標を設定する。機能要件の受入基準（何をするか）ではなく、品質の水準（どの程度の品質でするか）を確定する。

### ISO/IEC 25010 との対応

| ISO/IEC 25010 品質特性 | 本サブの対応ファイル |
|---|---|
| 機能適合性（Functional Suitability） | 機能要件サブ（本サブの対象外） |
| 信頼性（Reliability）※可用性含む | `01_可用性要件.md` |
| 性能効率性（Performance Efficiency） | `02_性能・拡張性要件.md` |
| 保守性（Maintainability）※運用含む | `03_運用・保守性要件.md` |
| 移植性（Portability）※移行性含む | `04_移行性要件.md` |
| セキュリティ（Security） | `05_セキュリティ要件.md` |
| システム環境・エコロジー | `06_システム環境・エコロジー要件.md` |
| 品質特性（ISO/IEC 25010 補助） | `07_品質特性要件（ISO_IEC_25010補助）.md` |
| データ品質（ALCOA+整合） | `08_データ品質要件（ALCOA+整合）.md` |
| 使用性（Usability）・アクセシビリティ | `09_ユーザビリティ・アクセシビリティ要件.md` |
| 倫理品質（Ethics Quality）※本プロジェクト追加 | `05_セキュリティ要件.md` NFR-ETH-xxx（EC-01〜EC-06の実装要件を含む） |

---

**本節で確定した方針**
非機能要件サブは ISO/IEC 25010 を基盤とし、9 種の品質特性ファイル（01〜09）に分類した NFR を確定する。
各 NFR は数値目標（グレード）を受入基準として持つ。「適切に」「できる限り」等の曖昧目標は禁止する。
倫理品質（NFR-ETH: EC-01〜EC-06 の技術的実装要件）は 05_セキュリティ要件.md に統合する。

## 3. IPA 6 大項目 34 中項目とグレード採否一覧

IPA 非機能要求グレード（NFUG）に基づく品質特性の採否を以下のとおり確定する。

### 可用性（Reliability / Availability）

| 中項目 | 採否 | 設定グレード / 根拠 |
|---|---|---|
| 継続性（稼働率） | 採用 | 月次稼働率 99.0% 以上（社内 LAN 単独動作前提） |
| 耐障害性（RTO） | 採用 | RTO: 1 時間以内（手動切替 + Docker 自動再起動 + Offline-First 設計による業務影響最小化）|
| 回復性（RPO） | 採用 | RPO: 15 分以内（WAL PITR 5 分間隔転送 + Offline-First バッファ）|
| 縮退運転 | 採用 | Offline-First 設計による作業継続保証を確定する |

### 性能効率性（Performance Efficiency）

| 中項目 | 採否 | 設定グレード / 根拠 |
|---|---|---|
| 時間効率性（API レスポンスタイム） | 採用 | p95 500ms 以内（社内 LAN、同時接続 50 端末以下） |
| 時間効率性（UI 描画） | 採用 | Step 遷移: 200ms 以内 |
| リソース効率性 | 採用 | タブレット RAM 2GB 前提での動作保証 |
| スケーラビリティ | 採用（限定） | 単一工場 500 名・同時接続 50 端末を上限として設計する |

### セキュリティ（Security）

| 中項目 | 採否 | 設定グレード / 根拠 |
|---|---|---|
| 機密性（認証・認可） | 採用 | RBAC 必須。役割（Role）ごとのアクセス権限を確定する |
| 完全性（Audit Trail） | 採用 | 全変更操作の Audit Trail 必須。電子記録の Append-only 設計 |
| 否認防止性 | 採用 | 電子サイン付きステップ完了（否認不可設計） |
| 責任追跡性 | 採用 | 全 API 操作のログ記録（操作者・日時・操作内容） |
| 真正性（暗号化） | 採用 | 通信は TLS 1.2 以上を確定する |
| 可用性（セキュリティ観点） | 採用 | DoS 対策（Rate Limiting）を社内 LAN 前提で設定する |

### 保守性（Maintainability）

| 中項目 | 採否 | 設定グレード / 根拠 |
|---|---|---|
| モジュール性 | 採用 | コンポーネント分割・関心分離の設計原則を採用する |
| 再利用性 | 採用 | 将来の SIer 導入・チーム開発移管に備えた設計を確定する |
| 解析性 | 採用 | 構造化ログ・ADR 全記録・自己診断コマンドを確定する |
| 修正性 | 採用 | テストカバレッジ: ユニット 80% 以上・E2E: コアフロー 100% |
| 試験性 | 採用 | CI/CD での自動テスト実行を確定する |

### 移植性（Portability）

| 中項目 | 採否 | 設定グレード / 根拠 |
|---|---|---|
| 適応性 | 採用（限定） | Android/iOS/Windows タブレット対応を確定する |
| 設置性 | 採用 | Docker Compose によるワンコマンド起動を確定する |
| 置換性 | 採用（将来余地） | Linux サーバー環境への移行余地を設計で確保する |
| SaaS 化対応 | 対象外と判断する | P02（社内 LAN 前提）による。永続非対象 |

### 倫理品質（Ethics Quality）※本プロジェクト追加カテゴリ

| 中項目 | 採否 | 設定グレード / 根拠 |
|---|---|---|
| データ最小化 | 採用 | GDPR 第 5 条第 1 項 (c) に対応する。業務目的外データの収集を RBAC で禁止する |
| 用途制限（三用途） | 採用 | 品質保証・工程改善・教育設計の三用途のみ許容する技術的制約を実装する |
| 個人スコアリング禁止 | 採用（禁止実装要件） | 個人別速度・ランキング・達成率の表示 API を実装しない |
| Just Culture 設計 | 採用 | スリップ・ラプスの自動問責プロセス連携を実装しない |
| データオーナーシップ | 採用 | 工場（利用組織）帰属のデータ移動権を実装する |

| ステークホルダー | 推奨読み順 | 読む理由 |

| 作業員 | `05_セキュリティ要件.md` | システムが自分を監視ツールとして機能しないことの技術的根拠確認 |

| 品質担当 | `05_セキュリティ要件.md` → `01_可用性要件.md` | Audit Trail・電子記録改ざん防止・RTO/RPO の確認 |
| 保全担当 | `01_可用性要件.md` | 設備停止時の業務継続保証の確認 |
| IT 担当 | `01_可用性要件.md` → `02_性能・拡張性要件.md` → `05_セキュリティ要件.md` → `03_運用・保守性要件.md` → `04_移行性要件.md` | 全 NFR の実装可能性・運用負荷の確認 |
| 経営層 | `01_可用性要件.md`（RTO/RPO）→ `05_セキュリティ要件.md`（RBAC 方針） | 法的責任・投資判断に直結する品質目標の確認 |
| 規制当局想定 | `05_セキュリティ要件.md`（Audit Trail 完全性）→ `01_可用性要件.md`（記録保全期間）→ `05_セキュリティ要件.md`（EC-01〜EC-06） | 監査証拠としての電子記録品質基準の確認 |
| 労働者代表 | `05_セキュリティ要件.md` | 行動データ用途三限定・個人スコアリング禁止の技術的実装確認 |
