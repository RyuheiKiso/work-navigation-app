# 03 運用・保守性要件

本章は、IT 担当 1 名兼務という制約下での通常運用・保守運用・障害時運用・監視・バックアップ・ログ管理・ライフサイクル要件を確定する。計画 09 章（運用・組織・教育方針）で確定した運用体制をシステム要件として具体化する。

---

## 1. 通常運用要件

### 1-1. 日常運用の最低基準

| 要件 ID | NFR-OPS-001 |
|---|---|
| 要件名 | IT 担当 1 名による継続運用 |
| 要件内容 | 本システムのすべての通常運用タスクが IT 担当 1 名で実施できること。専門的な DB 管理者・インフラエンジニアを必要とする運用手順を設計しない |

| 要件 ID | NFR-OPS-002 |
|---|---|
| 要件名 | ヘルスチェック頻度 |
| 要件内容 | IT 担当が以下を週次で確認する。確認所要時間は 30 分以内とする（チェックリスト化）|
| チェック項目 | API サーバー稼働確認 / DB 接続確認 / バックアップ完了確認 / Outbox キュー残留件数確認 / ディスク使用量確認 |

| 要件 ID | NFR-OPS-003 |
|---|---|
| 要件名 | GUI ベースの運用操作 |
| 要件内容 | バックアップ確認・ユーザー管理・ログ確認の日常運用操作は管理コンソールの GUI から実施できること。コマンドライン操作のみで実施できる運用タスクを設けない |

**本節で確定した方針**
- 全通常運用タスクを IT 担当 1 名・週次 30 分以内で完結可能な設計とすることを確定する。
- ヘルスチェック 5 項目（サーバー / DB / バックアップ / Outbox / ディスク）をチェックリスト化し、週次確認の標準とすることを確定する。
- GUI ベースの運用操作を必須設計要件として確定し、CLI 専用タスクを排除する。

---

## 2. 保守運用要件

### 2-1. パッチ適用要件

| 要件 ID | NFR-OPS-010 |
|---|---|
| 要件名 | セキュリティパッチ適用 SLO |
| 要件内容 | CVSS スコア 9.0 以上の重大脆弱性が報告された場合、3 日以内にパッチを適用する（保守要件 MNT-017 との整合） |

| 要件 ID | NFR-OPS-011 |
|---|---|
| 要件名 | 依存ライブラリの定期更新 |
| 要件内容 | `cargo audit` を週次実行し、脆弱性が検出された場合は NFR-OPS-010 に従い対処する。OS および Docker イメージの月次更新を実施する |

### 2-2. マスタ改訂運用要件

| 要件 ID | NFR-OPS-012 |
|---|---|
| 要件名 | SOP 改訂の運用手順 |
| 要件内容 | SOP の改訂は計画 09 章 2 節「SOP 改訂フロー」6 段階（申請受付→ドラフト→確認承認→有効化日設定→廃止処理→教育完了確認）に従い実施すること。フロー外の変更手順を設けない |

| 要件 ID | NFR-OPS-013 |
|---|---|
| 要件名 | マスタ改訂の適用確認 |
| 要件内容 | 新版 SOP の有効化後、品質担当は当日中に管理コンソールで「対象作業員の SOP 受信完了件数」を確認する。未受信端末がある場合は手動同期を指示する |

**本節で確定した方針**
- CVSS 9.0+ 脆弱性の 3 日以内パッチ適用（NFR-OPS-010、保守要件 MNT-017 との整合）と cargo audit 週次実行（NFR-OPS-011）を確定する。
- SOP 改訂フロー 6 段階を運用要件として確定し、フロー外変更を禁止する。
- 新版 SOP 有効化後の受信確認を品質担当の当日確認事項として確定する。

## 3. 障害時運用要件

### 3-1. 障害対応 SLO

| 要件 ID | NFR-OPS-020 |
|---|---|
| 要件名 | 障害発生から通知までの時間 |
| 要件内容 | 障害を現場担当者（作業員または現場監督）が検知した場合、IT 担当に 15 分以内に連絡が届くこと |
| 通知手段 | 社内電話・内線・対面のいずれか。自動アラートシステムは対象外と判断する（個人開発・コスト制約）|

| 要件 ID | NFR-OPS-021 |
|---|---|
| 要件名 | 初動対応から復旧完了までの時間 |
| 要件内容 | 初動対応開始（IT 担当がシステムに触れた時点）から障害復旧完了まで 1 時間以内（NFR-AVL-002 RTO 1 時間と整合）|

| 要件 ID | NFR-OPS-022 |
|---|---|
| 要件名 | Offline-First 縮退時の通知 |
| 要件内容 | サーバー障害でタブレット端末がオフライン縮退モードに切り替わった場合、端末上に「オフライン記録中（未同期 x 件）」を表示し、作業員が通常どおり作業を継続できることを明示する |

**本節で確定した方針**
- 障害通知を IT 担当 15 分以内・復旧完了 1 時間以内として確定し、NFR-AVL-002 と整合させる。
- 自動アラートシステムを個人開発・コスト制約から対象外と判断し、人手通知を前提とする。
- Offline-First 縮退時の端末上の「オフライン記録中」表示を UI 要件として確定する。

## 4. 監視要件

### 4-1. 監視方式の確定

| 要件 ID | NFR-OPS-025 |
|---|---|
| 要件名 | 監視方式 |
| 方針 | Pull 型（手動確認）監視を採用する。プッシュ型の自動アラートシステムは対象外と判断する |
| 根拠 | 個人開発・IT 担当 1 名・社内 LAN 完結の制約。Prometheus / Grafana 等の監視 SaaS はソフトウェア依存を増加させる |

| 要件 ID | NFR-OPS-026 |
|---|---|
| 要件名 | 週次ヘルスレポート |
| 要件内容 | IT 担当が週次に以下の項目を管理コンソールで確認し、レポートをテキストファイルで記録する |
| レポート項目 | API サーバー稼働時間・DB 接続可否・バックアップ完了世代数・Outbox DLQ 件数・ディスク使用率・認証失敗件数（週次）|

| 要件 ID | NFR-OPS-027 |
|---|---|
| 要件名 | PostgreSQL スロークエリ確認 |
| 要件内容 | 月次で `pg_stat_statements` を確認し、P95 応答時間が SLO（3 秒）を超えるクエリを特定して対処する |

**本節で確定した方針**
- Pull 型手動確認監視を採用し、プッシュ型自動アラートシステムを対象外と判断する。
- 週次ヘルスレポート 6 項目をチェックリストとして確定し、テキストファイルでの記録を要件とする。
- 月次 pg_stat_statements 確認を保守業務として確定する。

## 5. バックアップ要件

### 5-1. バックアップポリシー

| 要件 ID | NFR-OPS-030 |
|---|---|
| 要件名 | WAL アーカイブ設定 |
| 要件内容 | PostgreSQL の WAL を 5 分間隔で NAS に転送する。PITR で RPO 15 分以内の復旧を可能にする |

| 要件 ID | NFR-OPS-031 |
|---|---|
| 要件名 | 日次 pg_dump |
| 要件内容 | 毎日 01:00（夜間）に pg_dump をフルダンプする。NAS の専用ディレクトリに保存する。世代管理: 日次 7 世代 |

| 要件 ID | NFR-OPS-032 |
|---|---|
| 要件名 | 週次オフサイトバックアップ |
| 要件内容 | 毎週日曜日（または直近の休日）に週次スナップショットを取得し、AES-256 暗号化した上で可搬媒体（外付け HDD）に保存する。別棟または建屋外に保管する |

| 要件 ID | NFR-OPS-033 |
|---|---|
| 要件名 | バックアップ完了確認 |
| 要件内容 | バックアップ実行後、完了ステータス（成功/失敗）をログに記録する。IT 担当は週次ヘルスチェック時にバックアップ完了ログを確認する |

**本節で確定した方針**
- WAL 5 分間隔転送・日次 pg_dump（7 世代）・週次オフサイト AES-256 暗号化の 3 層バックアップを確定する。
- バックアップ完了ログの週次確認を IT 担当の必須チェック項目として確定する。
- オフサイト保管媒体の鍵（AES-256）を PostgreSQL 暗号化鍵と別管理することを確定する。

## 6. ログ管理要件

### 6-1. ログ種別と保全期間

| 要件 ID | NFR-OPS-035 |
|---|---|
| 要件名 | 作業実績紐付きログの保全 |
| 要件内容 | WorkEvent テーブルに記録された全作業イベントログを 7 年以上保全する。設定値で業種に応じて延長可能とする |

| 要件 ID | NFR-OPS-036 |
|---|---|
| 要件名 | 認証ログの保全 |
| 要件内容 | ログイン成功・失敗・JWT 失効・ロールアウトの認証イベントログを 90 日間保全する。設定値で変更可能とする |

| 要件 ID | NFR-OPS-037 |
|---|---|
| 要件名 | ログの改ざん防止 |
| 要件内容 | WorkEvent テーブルは Append-only（INSERT のみ）設計とし、PostgreSQL のロール権限で UPDATE / DELETE を禁止する。SHA-256 ハッシュチェーンで改竄を検知する |

| 要件 ID | NFR-OPS-038 |
|---|---|
| 要件名 | アプリケーションログの管理 |
| 要件内容 | API サーバー（Rust/axum）のアプリケーションログを JSON 形式でファイルに出力する。ローテーションは日次・保全期間は 30 日とする |

**本節で確定した方針**
- 作業実績ログ 7 年以上・認証ログ 90 日の保全期間を確定し、設定値による変更を許容する。
- WorkEvent の Append-only + SHA-256 ハッシュチェーンによるログ改ざん防止を確定する（NFR-OPS-037）。
- アプリケーションログを JSON 形式・日次ローテーション・30 日保全として確定する。

---

## 7. ライフサイクル要件

### 7-1. OSS 依存の EOL 追跡

| 要件 ID | NFR-OPS-040 |
|---|---|
| 要件名 | OSS ライブラリの EOL 追跡 |
| 要件内容 | 主要依存ライブラリ（axum / tokio / sqlx / React Native / PostgreSQL）の EOL（End of Life）日程を年次確認する。EOL まで 12 ヶ月以内になった場合はバージョンアップ計画を ADR に記録する |

| 要件 ID | NFR-OPS-041 |
|---|---|
| 要件名 | セキュリティパッチ適用の SLO |
| 要件内容 | CVSS スコア別の適用 SLO は以下のとおりとする。CVSS 9.0〜10.0（Critical）: 3 日以内（保守要件 MNT-017 との整合）。CVSS 7.0〜8.9（High）: 30 日以内。CVSS 4.0〜6.9（Medium）: 次回月次更新時 |

| 要件 ID | NFR-OPS-042 |
|---|---|
| 要件名 | 年次自己監査 |
| 要件内容 | 年次 1 回、本書の自己監査チェックリスト全項目を実施し、不合格項目の対処計画を ADR に記録する |

**本節で確定した方針**
- 主要 OSS の EOL 年次確認・12 ヶ月前の ADR 記録義務を確定する。
- CVSS スコア別パッチ適用 SLO（Critical: 3 日（保守要件 MNT-017 との整合） / High: 30 日 / Medium: 月次）を確定する。
- 年次自己監査を保守運用タスクとして義務化し、不合格項目の ADR 記録を確定する。
