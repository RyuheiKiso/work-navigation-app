# 04 要件分類体系とトレーサビリティ枠組み

## 1. 要件分類体系

### 9 カテゴリの定義

本書の全要件は以下の 9 カテゴリに分類する。カテゴリコードは要件 ID の先頭に付与する。

| カテゴリコード | カテゴリ名 | 定義 | 担当サブ |
|---|---|---|---|
| BR | 業務要件（Business Requirements） | 業務フロー・ユースケース・業務ドメイン上の目的。「なぜその機能が必要か」の業務的根拠 | `業務要件/` サブ |
| FR | 機能要件（Functional Requirements） | システムが実行する機能の検証可能命題。「何をするか」の具体的記述 | `機能要件/` サブ |
| NFR | 非機能要件（Non-Functional Requirements） | 品質特性（可用性・性能・セキュリティ・保守性・移植性・倫理品質）の数値目標 | `非機能要件/` サブ |
| MIG | 移行要件（Migration Requirements） | 既存データ・マスタの移行・カットオーバー・ロールバックに関する要件 | `移行要件/` サブ |
| OPS | 運用要件（Operations Requirements） | バックアップ・監視・縮退運転・SOP 改訂ループ等の運用上の要件 | `運用要件/` サブ |
| MNT | 保守要件（Maintenance Requirements） | 技術的負債管理・ADR・依存バージョン管理・テストカバレッジの保守基準 | `保守要件/` サブ |
| EDU | 教育要件（Education Requirements） | OJT 設計・多言語教材・習熟度別訓練計画の要件 | `教育要件/` サブ |
| TST | テスト・受入要件（Test & Acceptance Requirements） | 受入基準の詳細化・テスト種別定義・ALCOA+ 検証計画 | `テスト・受入要件/` サブ |
| PRJ | プロジェクト要件（Project Requirements） | 個人開発制約・変更管理プロセス・意思決定権限の要件 | `プロジェクト要件/` サブ |

---

**本節で確定した方針**
- 要件分類を 9 カテゴリ（BR/FR/NFR/MIG/OPS/MNT/EDU/TST/PRJ）で確定し、各カテゴリを専用サブに割り当てる。
- 各要件は必ずいずれか 1 つのカテゴリコードを持つ。複数カテゴリにまたがる要件は、主カテゴリで管理し、従カテゴリへの参照を上流リンクとして記録する。
- カテゴリの追加・変更は本章の改訂として管理する。

---

## 2. 要件 ID 採番規約

### 採番形式

要件 ID は以下の形式で採番する。

```
[カテゴリコード]-[サブカテゴリコード]-[連番 3 桁]
```

例: `FR-NV-001`（機能要件 / ナビゲーション / 001 番）

### 採番ルール

- 連番は各サブカテゴリ内で `001` から始め、3 桁ゼロ埋めで記述する
- 欠番は作らない。廃止した要件は `[廃止]` タグを付与してそのまま残す
- 同一サブカテゴリ内で最大 999 件まで採番できる。超過する場合は本章の改訂で対処する

図: fig_req_id_taxonomy（img/ 配下）を参照。

---

**本節で確定した方針**
- 要件 ID の採番形式を `[カテゴリ]-[サブカテゴリ]-[連番 3 桁]` に確定する。
- 30 サブカテゴリコードを確定し、各コードを担当ファイルに紐付ける。
- 廃止要件は削除せず `[廃止]` タグ付きで保持することを確定する。

## 3. MoSCoW 優先度の定義と判断基準

### MoSCoW の定義

| 優先度 | 定義 | 判断基準 |
|---|---|---|
| Must | ver1.0.0 リリース時に必ず実装する要件。不在の場合、システムとして機能しないか、安全・倫理・法的な問題が生じる | 「この要件なしに本番運用できるか？」の答えが No である要件 |
| Should | ver1.0.0 リリース時に実装することが強く推奨される要件。不在でも代替手段があるが、著しく価値が低下する | 「代替手段（紙・口頭）で 3 ヶ月以上乗り切れるか？」の答えが No である要件 |
| Could | ver1.0.0 リリース時に余裕があれば実装する要件。スケジュール・品質制約を超過しない範囲でのみ実装する | 「最初の 10 名のユーザーが困るか？」の答えが No である要件 |
| Won't | ver1.0.0 では実装しないことを明示した要件。将来バージョンへの候補として記録するか、永続非提供として記録する | `99_前提制約と本書が約束しないこと.md` に記載した非対象項目 |

### MoSCoW の判断フロー

図: fig_req_moscow_criteria（img/ 配下）を参照。

MoSCoW の判断は以下の順序で行う。

1. **Won't の先行確定**: `99_前提制約と本書が約束しないこと.md` の非対象項目に該当する要件は即座に Won't とする
2. **倫理制約の確認**: `01_要件定義の前提継承と上流リンケージ.md` の EC-01〜EC-06 に抵触する要件は Must として禁止実装要件（「〜を実装しない」形式の要件）に変換するか、Won't として永続非提供に確定する
3. **業務的必須性の確認**: 「この要件なしに本番運用できるか？」を問い、No なら Must とする
4. **代替手段の確認**: 「代替手段で 3 ヶ月以上乗り切れるか？」を問い、No なら Should とする
5. **価値・スケジュール評価**: 残りを Could として個人開発スケジュールに対して評価する

### MoSCoW 別の要件数管理

個人開発の現実として、Must 要件が多すぎると ver1.0.0 リリースが不可能になる。
以下のガイドラインで Must 要件数を管理する。

- Must 要件は 1 サブあたり上限 20 件を目安とする（超過時は Should への格下げを実施する）
- Should 要件は Must 要件の完了後に着手する
- Could 要件は実装するかどうかの最終判断をリリース直前の段階で行う

---

**本節で確定した方針**
- MoSCoW の 4 段階を定義し、判断フロー（Won't先行確定→倫理制約→業務必須→代替手段→価値評価）を確定する。
- 倫理制約 EC-01〜EC-06 に抵触する要件は Must 禁止実装要件または Won't 永続非提供に確定することを義務とする。
- Must 要件数の上限目安を 1 サブあたり 20 件とし、超過時は Should への格下げを実施することを確定する。

## 4. トレーサビリティ枠組み（6 段トレーサビリティ）

### 6 段トレーサビリティの定義

本書の全要件は以下の 6 段トレーサビリティチェーンで管理する。

```
Why（構想）→ What（計画）→ 業務要件 BR → 機能要件 FR → 非機能要件 NFR → 受入基準 AC
```

図: fig_req_traceability_chain（img/ 配下）を参照。

| 段 | 文書 | 内容 | 例 |
|---|---|---|---|
| 段 1: Why | 構想書章 | なぜその機能が必要か（課題・価値命題・倫理スタンス） | 構想 04章（倫理スタンス）|
| 段 2: What | 計画書章 | 何をスコープとするか（機能スコープ・非スコープ・技術スタック） | 計画 06章（データモデル）|
| 段 3: BR | `業務要件/` サブ | 業務上の目的・業務フロー・ユースケース | `BR-BUS-001`: ステップ完了記録の業務要件 |
| 段 4: FR | `機能要件/` サブ | 検証可能な機能の記述 | `FR-TR-001`: ALCOA+ Contemporaneous 実装要件 |
| 段 5: NFR | `非機能要件/` サブ | 品質特性の数値目標 | `NFR-AVL-002`: RTO 1 時間 |
| 段 6: AC | 各要件の受入基準欄 | テストで Pass/Fail を判定できる具体的な検証条件 | 「完了タイムスタンプとクライアント送信時刻の差が 500ms 以内である」 |

### トレーサビリティマトリクス（RTM）の構造

各要件の記述には以下の上流リンクを必ず付与する。

| 要件カテゴリ | 必須上流リンク | 任意上流リンク |
|---|---|---|
| BR | 構想書章（Why）、計画書章（What） | — |
| FR | BR 要件 ID（段 3）、計画書章（What） | 構想書章（Why） |
| NFR | 計画書章（What: 品質特性方針）、構想書章（Why: 倫理スタンス等） | BR 要件 ID |
| MIG/OPS/MNT/EDU/TST/PRJ | 計画書章（What: 対応する計画章） | BR/FR/NFR 要件 ID |

### RTM の維持義務

要件が変更・廃止された場合は、以下の RTM 更新を義務とする。

1. 廃止された要件を参照している下流要件（FR・NFR・AC）を確認する
2. 下流要件の上流リンクを更新するか、下流要件も廃止する
3. 変更を `プロジェクト要件/README.md` の変更管理ログに記録する

---

**本節で確定した方針**
- 6 段トレーサビリティ（Why→What→BR→FR→NFR→AC）を全要件の管理枠組みとして確定する。
- 各要件に必須上流リンクを付与することを義務とし、リンクのない要件は起票完了とみなさない。
- 要件変更・廃止時の RTM 更新を義務とし、変更管理ログへの記録を確定する。

## 5. 要件ライフサイクル

### 状態遷移の定義

要件は以下の 5 状態で管理する。

| 状態 | 定義 | 遷移条件 |
|---|---|---|
| Draft | 起票済みだが未確定の要件。5 要素（ID・検証可能命題・受入基準・MoSCoW・上流リンク）が揃っていない要件もこの状態 | 起票時に設定 |
| In Review | C（協議先）ステークホルダーへのレビュー中 | ドラフト完成後、C に送付時に設定 |
| Agreed | A（承認者）が承認した要件。5 要素が揃っており、MoSCoW が確定している | A の承認を Git コミットまたは ADR で記録後に設定 |
| Frozen | 概要設計書の作成開始以降。変更管理プロセスなしに変更できない | 概要設計開始時に一括設定 |
| 廃止 | 不要になった要件。削除せず `[廃止]` タグを付けて保持 | 変更管理プロセスの承認後に設定 |

### 状態遷移のルール

- **Draft → In Review**: 5 要素がすべて記入されたことを確認してから遷移する
- **In Review → Agreed**: C のレビューコメントへの対応完了後、A が承認する
- **Agreed → Frozen**: 概要設計開始時に一括して Frozen に遷移する
- **Frozen 状態の変更**: 変更管理プロセス（`プロジェクト要件/README.md` 参照）を経て、A の再承認後に変更する
- **廃止への遷移**: 廃止理由・廃止日・代替要件 ID（ある場合）を記録してから遷移する

---

**本節で確定した方針**
- 要件ライフサイクルを Draft/In Review/Agreed/Frozen/廃止 の 5 状態で確定する。
- Frozen 状態の変更は変更管理プロセスと A の再承認を必須とすることを確定する。
- 廃止要件は削除せず廃止理由・廃止日を記録して保持することを確定する。
