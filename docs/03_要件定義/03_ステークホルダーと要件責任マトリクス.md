# 03 ステークホルダーと要件責任マトリクス

## 1. 8 種ステークホルダーの定義

本節は本書（要件定義書）における 8 種ステークホルダーの定義を確定する。
定義は計画書 `01_ステークホルダーと価値仮説.md` の定義を継承し、要件定義プロセスにおける役割・権限・受入責任を追加する。

ステークホルダー関係の全体像については、図: fig_req_stakeholder_raci（img/ 配下）を参照。

| ステークホルダー | 役割・定義（計画書からの継承） | 要件定義での役割 | 主な接点 |
|---|---|---|---|
| 作業員 | タブレットで手順に従って作業し、記録を入力する。習熟度・年齢・母語が多様であり、IT リテラシーに大きな差がある | 業務要件・機能要件（ナビゲーション系）の一次受益者。ユースケースシナリオの主要アクター | タブレット APP |
| 現場監督 | ラインの生産進捗・品質・安全を管理し、異常時に判断する。シフトをまたぐ引継ぎと進捗可視化が主な責務 | 業務要件・機能要件（アラート・改善系）の一次受益者。運用要件の一次レビュー者 | タブレット APP / 管理 Web |
| 品質担当 | 手順マスタの承認・不適合処理・トレサビ照会・内部監査を担う。ALCOA+ に準拠した記録品質の維持に責任を持つ | 機能要件（トレーサビリティ系）・非機能要件（ALCOA+・Audit Trail）の要件合意者 | 管理 Web |
| 保全担当 | 設備の点検・保全作業を実施し、設備履歴・校正記録を管理する | 機能要件（マスタ管理・保全記録系）の要件合意者 | タブレット APP / 管理 Web |
| IT 担当 | システムの導入・運用・バックアップ・障害対応を担う。本システムでは 1 名前提であり、専任 IT 部門なしの場合を想定する | 非機能要件全般・移行要件・運用要件・保守要件の主要レビュー者・受入担当者 | バックエンド / インフラ |
| 経営層 | 投資判断・導入可否・規制責任・労働政策の意思決定者 | プロジェクト要件の最終意思決定者。本書全体の最終承認者 | 管理 Web（レポート） |
| 規制当局想定 | ISO 9001 内部監査・第三者品質認証・顧客監査（サプライヤー監査）における審査者として定義する（実在する外部機関ではなく想定審査者） | 機能要件（Audit Trail・電子サイン）・非機能要件（記録保全・完全性）の品質基準の引き上げ根拠。受入基準の外部参照基準 | 記録・証拠・監査証跡 |
| 労働者代表 | 作業員の利害を代表する。デジタル監視ツール化・人事評価への転用・外国人労働者への配慮についての懸念を代表する | 非機能要件（倫理品質）・教育要件の必須レビュー者。導入合意プロセスの参加者 | 導入合意プロセス |

---

**本節で確定した方針**
- 8 種ステークホルダーの定義を計画書 01章から継承し、要件定義での役割（要件合意者・受入担当者・レビュー者）を追加して確定する。
- 規制当局想定を実在する外部機関ではなく「ISO 9001 内部監査・第三者認証・顧客監査の審査者」として定義することを継承する。
- 労働者代表を倫理品質要件・教育要件の必須レビュー者として位置づけることを確定する。

## 2. 要件定義書全体における受入責任マトリクス（RACI）

### RACI の定義

| 記号 | 意味 |
|---|---|
| R | Responsible（実行責任者：要件の記述・詳細化を実施する） |
| A | Accountable（説明責任者：最終承認者。1 サブに 1 名のみ） |
| C | Consulted（協議先：要件確定前に意見を求める） |
| I | Informed（通知先：要件確定後に結果を通知する） |

**本節で確定した方針**
- RACI マトリクスを 25 サブ × 8 ステークホルダーで確定する。
- 説明責任者（A）は各サブに必ず 1 名を設定し、経営層またはIT担当または品質担当が担うことを確定する。
- 倫理品質要件・教育要件については労働者代表を C（協議先）として必須参加とすることを確定する。

## 3. 受入基準合意プロセス（個人開発前提での合意方法）

### 合意プロセスの設計思想

本システムは個人開発であるため、複数人のチームレビューは存在しない。
しかし RACI マトリクスで C（協議先）として定義したステークホルダーとの合意なしに要件を凍結することは、ver1.0.0 リリース後の受入失敗リスクを生む。
したがって、個人開発前提での現実的な合意プロセスを以下のとおり確定する。

### 合意の三段階

| 段階 | 内容 | 担当者 | 成果物 |
|---|---|---|---|
| 段階 1: ドラフト作成 | 要件を `Draft` 状態で記述する。上流リンク・受入基準・MoSCoW を全て記入する | 開発者（個人） | 各サブの要件ドラフト |
| 段階 2: C（協議先）レビュー | RACI で C に指定された各ステークホルダーとのレビューを実施する。個人開発の現実として、導入予定先工場の担当者または対象業務の有識者を C として設定する。担当者が実在しない場合は業界分析文書・計画書を代替参照として記録する | C ステークホルダー（または業界分析文書） | レビューコメント・フィードバック記録 |
| 段階 3: A（承認者）承認 | RACI で A に指定された承認者が要件を `Agreed` 状態に変更する。個人開発の現実として、開発者自身が A を兼務する場合、レビューコメントへの対応完了後に A として承認する | A ステークホルダー（または開発者） | 承認記録（Git コミットメッセージまたは ADR） |

### 業界分析文書を代替参照として使用する場合のルール

実在するステークホルダーが C の役割を担えない場合（個人開発の初期段階など）、以下のルールで業界分析文書を代替参照として記録する。

- 代替参照した業界分析文書のパスを要件の上流リンク欄に記載する
- 代替参照である旨を注記として明示する（例: 「実在ステークホルダー未確定のため、業界分析/06 を代替参照として使用」）
- 実在ステークホルダーが確定した時点で追認レビューを実施し、追認記録を ADR に記載する

---

**本節で確定した方針**
- 個人開発前提での合意プロセスを「ドラフト作成→C レビュー→A 承認」の 3 段階で確定する。
- C ステークホルダーが実在しない場合、業界分析文書を代替参照として使用するルールを確定する。
- 代替参照使用時は要件の上流リンクに記録し、実在ステークホルダー確定後の追認を義務とする。

## 4. 労働者代表・規制当局想定の取扱い

### 労働者代表の要件定義での取扱い

労働者代表は計画書 `01_ステークホルダーと価値仮説.md` の倫理境界宣言・透明性公約（計画書 01章 H2-4・H2-6）で保護されるべき存在として確定している。
本書では以下の取扱いを確定する。

**必須協議（C）の対象サブ**:
- `非機能要件/06_倫理品質要件.md`（行動データ用途三限定の実装要件）
- `教育要件/README.md`（多言語教材・外国人労働者対応の要件）
- `機能要件/01_ナビゲーション機能要件.md`（多言語インターフェース要件）

**労働者代表への通知（I）の対象サブ**:
- `99_前提制約と本書が約束しないこと.md`（P05 外国人労働者・P09 倫理制約の確定）
- 上記以外の全サブ（確定後の通知として記録）

**実在しない場合の取扱い**:
本システムが個人開発の初期段階にあり、実在する労働者代表が存在しない場合、以下を代替参照として使用する。
- [`90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md`](../../90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md)
- [`90_業界分析/09_職務設計とモチベーション論.md`](../../90_業界分析/09_職務設計とモチベーション論.md)

### 規制当局想定の要件定義での取扱い

規制当局想定は計画書 `01_ステークホルダーと価値仮説.md` H2-5 で「ISO 9001 内部監査・第三者認証・顧客監査の審査者」として定義された。
本書では以下の取扱いを確定する。

**規制当局想定が要件品質基準を引き上げる対象サブ**:

| サブ | 引き上げの根拠 |
|---|---|
| `機能要件/02_トレーサビリティ機能要件.md` | 監査時に証拠として提出できる記録構造の設計。Audit Trail の完全性が審査対象となる |
| `非機能要件/03_セキュリティ要件.md` | 電子記録の改ざん不可能性・アクセスログの完全性が審査対象となる |
| `テスト・受入要件/README.md` | 記録品質の検証計画が「内部監査に耐えうる水準」で設計されているかが審査される |

規制当局想定は本書の C（協議先）として機能し、上記 3 サブの受入基準に「規制当局想定の審査視点」を明示する。

---

**本節で確定した方針**
- 労働者代表を倫理品質要件・教育要件・多言語機能の必須協議先（C）として確定する。
- 実在しない場合は業界分析/24・業界分析/09 を代替参照として使用することを確定する。
- 規制当局想定をトレーサビリティ機能・セキュリティ・テスト受入の品質基準引き上げ根拠として確定する。
