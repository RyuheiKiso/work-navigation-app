# 06 UC 記述_マスタ管理

本章の責務は、マスタ管理に関わる UC-014〜UC-018 の 5 つのユースケースを固定フォーマットで記述することである。各 UC は計画 19 章（マスタ編集者体験設計）の「失わない・壊さない・繰り返さない・覚えない」4 原則と ALCOA+ 整合を踏まえて記述する。

---

## 1. ユースケース記述

### UC-014: SOP を Draft 編集する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当（マスタ編集者）が公開中の SOP を Draft 状態にコピーして安全に編集する。Draft First 原則により「保存 = 即時本番反映」の恐怖を解消し、Auto-Save・Undo/Redo・Diff 常時表示の 5 不変条件のもとで編集を行う |
| 主アクター | 品質担当（マスタ編集者） |
| 副アクター | システム（Draft 作成・Auto-Save 30 秒・楽観的ロック・master_edit_event 記録） |
| 事前条件 | 品質担当ロールのユーザーが JWT 認証済みである（FR-AU-002）。編集対象の SOP が approved または published 状態で存在している |
| 事後条件 | 対象 SOP が draft 状態にコピーされている。master_edit_event に draft_saved イベントが Append-only で記録されている。Auto-Save が有効化されており最大 30 秒前の状態に復元できる |

**主シナリオ**
1. 品質担当は管理 Web でマスタ管理画面から編集対象の SOP を検索・選択する（マスタ検索応答 P95 200ms 以内）
2. 品質担当は「Draft 編集を開始」ボタンをクリックする。システムは対象 SOP の draft 状態コピーを作成する（FR-MA-001）
3. システムは Draft 編集画面を表示する。右ペインに現在の公開版との差分（追加:緑・削除:赤・変更:黄）をリアルタイム表示する（FR-MA-009）
4. 品質担当は Step の instruction_text・input_type・judgment_condition・evidence_required 等を編集する
5. システムは 30 秒ごとに Draft を自動保存する（FR-MA-002）。画面右上のステータスインジケータに「最終保存: X 秒前」を常時表示する
6. 品質担当は直近 50 操作まで Undo（Ctrl+Z）・Redo（Ctrl+Y）でやり直しができる（FR-MA-003）
7. 編集完了後、品質担当は「レビュー依頼」ボタンをクリックする。システムは SOP を under_review 状態に遷移させ master_edit_event に submitted_for_review を記録する（FR-MA-015）

**代替シナリオ**
- A1: 同一 SOP を別の品質担当が同時に編集しようとした場合、システムは楽観的ロック（lock_version 比較）で競合を検知し「別のユーザーが編集中です。差分ビューで確認してください」を表示する（FR-MA-013）
- A2: CSV/Excel からの一括インポートを行う場合、システムはプレビュー表示と不整合レコードの赤強調一覧を表示し、品質担当の確認後にのみインポートを実行する（FR-MA-012）
- A3: 既存 SOP のクローンから新規 SOP を作成する場合、システムは derived_from フィールドにクローン元文書番号を自動記録し Draft 状態で作成する（FR-MA-011）
- A4: SOP テンプレートライブラリから作成する場合、システムはテンプレートの全 Step・メタデータを複製し独立した Draft SOP として作成する

**例外シナリオ**
- E1: ブラウザがクラッシュした場合、最後の Auto-Save（最大 30 秒前）の状態から復元できる。品質担当は次回ログイン時に「未完了の Draft が存在します。復元しますか？」プロンプトで復元を選択できる（FR-MA-002）
- E2: ネットワーク断絶で Draft 保存に失敗した場合、システムはブラウザのローカルストレージに一時保存し接続回復後に同期する
- E3: Draft の保持期間（最終編集から 30 日）を超過した場合、システムは品質担当に「Draft が期限切れになります」警告メールを 7 日前から送信する

**関連要件 ID**
- FR-MA-001, FR-MA-002, FR-MA-003, FR-MA-009, FR-MA-011, FR-MA-012, FR-MA-013, FR-MA-015, FR-AU-002

### UC-015: 条件分岐 DSL を編集する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当（マスタ編集者）がビジュアル DSL エディタを使用して step_flow_rule の分岐ルール（JSON Logic 条件式）を設定する。コードの知識がなくても正確な JSON Logic を組み立てられる設計とする |
| 主アクター | 品質担当（マスタ編集者） |
| 副アクター | システム（ビジュアル DSL ビルダ・DAG ビジュアライザ・循環参照検証・プレビュー実行） |
| 事前条件 | UC-014 が開始されており Draft 状態の SOP が存在している。品質担当ロールのユーザーが JWT 認証済みである（FR-AU-002） |
| 事後条件 | step_flow_rule が SOP の draft に追加または更新されている。DAG 検証が合格している（循環参照なし）。master_edit_event に draft_saved イベントが記録されている |

**主シナリオ**
1. 品質担当は Draft 編集画面で条件分岐を設定したい source_step（前 Step）を選択し「分岐ルールを追加」をクリックする（FR-MA-007）
2. システムはビジュアル DSL エディタを表示する。DAG ビジュアライザで SOP 全体の Step フローをノード・エッジ形式で表示する
3. 品質担当は演算子パレット（`==`・`!=`・`>=`・`<=`・`>`・`<`・`and`・`or`・`!`）から演算子をクリックまたはドラッグ＆ドロップで条件ツリーに追加する
4. 品質担当は変数ピッカーから `{"var": "value"}`（source_step の記録値）を選択して挿入する。JSON のキー名を手書きしない
5. 品質担当はアクション種別（skip/goto/insert）と target_step_id を設定する
6. 品質担当は「プレビュー実行」ボタンをクリックする。サンプル値を入力し、条件評価結果（true/false）が 200ms 以内で表示される（クライアント完結）
7. 品質担当は「保存」をクリックする。システムは API サーバー側で DAG 検証（循環参照チェック）を実行する。合格の場合のみ step_flow_rule を保存する（FR-MA-007）
8. システムは master_edit_event に draft_saved を記録する（FR-MA-015）

**代替シナリオ**
- A1: 上級者が直接 JSON を編集する場合（RAW JSON モード）、システムは JSON Logic の構文エラーを行・列単位でインライン表示する。ビジュアルモードと RAW JSON モードは随時切り替え可能で両モードの表示は同期する
- A2: 複数の分岐ルールが同一 source_step に存在する場合、システムは priority 順のルール一覧を表示し優先度の変更（ドラッグ並び替え）を提供する
- A3: フロー全体のシミュレーションを行う場合、システムは Step 1 から順番にサンプル値を入力してどのパスを通るかを可視化する

**例外シナリオ**
- E1: DAG 検証で循環参照が検出された場合、システムは「循環参照が検出されました: Step-X → Step-Y → Step-X」をエラー表示し保存を拒否する。DAG ビジュアライザで循環経路を赤強調表示する
- E2: eval/Function constructor が含まれた JSON が入力された場合（RAW JSON モードで）、システムはサーバー側でサンドボックス違反として検知し保存を拒否する（FR-MA-007 のセキュリティ制約）
- E3: target_step_id が存在しない Step を参照している場合、システムは「参照先 Step が存在しません」バリデーションエラーを表示し保存をブロックする

**関連要件 ID**
- FR-MA-007, FR-MA-015, FR-AU-002

### UC-016: マスタを Publish する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当（承認者）が under_review 状態の SOP を審査し電子署名で承認・published 状態に昇格させる。published 状態になった SOP のみタブレット APP に配信される |
| 主アクター | 品質担当（承認者） |
| 副アクター | システム（状態遷移制御・電子署名生成・effective_date 適用・タブレット配信） |
| 事前条件 | 対象 SOP が under_review 状態である。品質担当ロールのユーザーが JWT 認証済みであり電子署名権限を持つ（FR-AU-002） |
| 事後条件 | SOP が published 状態に遷移している。master_edit_event に approved・published イベントが Append-only で記録されている。effective_date 到達後にタブレット APP への配信が開始されている |

**主シナリオ**
1. 品質担当（承認者）は管理 Web でレビュー依頼一覧から対象 SOP を選択する
2. システムは Draft と現在の公開版の差分ビュー（追加:緑・削除:赤・変更:黄）を表示する（FR-MA-009）
3. 品質担当は差分内容・Step 定義・判定条件・クリティカルフラグを審査する
4. 承認する場合、品質担当は「承認」ボタンをクリックする。システムは電子署名画面を表示する（FR-EV-005 を includes として実行）
5. 品質担当はサインを確認し「電子署名で承認」をクリックする。システムは電子署名（承認者 ID + timestamp_device + サーバー署名）を生成する（FR-MA-005）
6. システムは SOP を approved 状態に遷移させ master_edit_event に approved を記録する（FR-MA-015）
7. 品質担当は effective_date（有効開始日）とロット番号紐付けを設定する（FR-MA-014）
8. 「公開」ボタンをクリックする。システムは SOP を published 状態に遷移させ master_edit_event に published を記録する（FR-MA-015）
9. システムは effective_date 到達時刻にタブレット APP へのマスタ配信を自動実行する（FR-SY-001 が子機モードの場合）

**代替シナリオ**
- A1: 差し戻す場合、品質担当は「差し戻し」ボタンをクリックし差し戻し理由を入力する。SOP は draft 状態に戻り master_edit_event に rejected が記録される
- A2: カスタム Step タイプを公開する場合も同一フローを適用する。step_type_definition の公開には品質担当の電子署名を必須とする（FR-MA-010）
- A3: step_flow_rule を公開する場合も同一フローを適用する（FR-MA-007）

**例外シナリオ**
- E1: 電子署名生成のサーバー呼び出しが失敗した場合、システムは「電子署名の生成に失敗しました」を表示し再試行を促す。電子署名なしでの approved 状態遷移を禁止する（FR-MA-005）
- E2: effective_date が過去の日付に設定された場合、システムは「有効開始日が過去です。確認してください」確認プロンプトを表示する（強制ブロックは行わない）
- E3: 公開対象 SOP が参照する設備マスタまたは計測器マスタが廃止済みの場合、システムは参照整合性警告を表示し「廃止済みマスタを参照しています。続けますか？」確認を求める（FR-MA-008）

**関連要件 ID**
- FR-MA-005, FR-MA-014, FR-MA-015, FR-EV-005, FR-AU-002, FR-MA-009

### UC-017: マスタをロールバックする

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当（承認者）が published 版の SOP に重大な誤りを発見し旧バージョンに緊急ロールバックする。ロールバック後、旧版を新たに published 状態に設定し現行版を obsolete 状態に設定する |
| 主アクター | 品質担当（承認者） |
| 副アクター | システム（参照整合性確認・状態遷移・master_edit_event 記録） |
| 事前条件 | ロールバック対象 SOP が published 状態であり、ロールバック先の旧バージョンが存在している（obsolete 状態の旧版）。品質担当ロールのユーザーが JWT 認証済みである（FR-AU-002） |
| 事後条件 | 旧バージョン SOP が published 状態に遷移している。現行版 SOP が obsolete 状態に設定されている（物理削除しない）。master_edit_event に rollback イベントが記録されている。進行中の作業指示は現行版（ロールバック前）で完了させる |

**主シナリオ**
1. 品質担当（承認者）は管理 Web でバージョン履歴一覧を表示する。ロールバック対象バージョンとロールバック先バージョンを確認する
2. 「ロールバック」ボタンをクリックする。システムはロールバック確認ダイアログを表示する
3. システムは参照整合性確認（UC-018 を extends として実行）を行い、影響を受けるレコードを一覧表示する（FR-MA-008）
4. 品質担当はロールバック理由をテキスト入力する（必須）
5. 品質担当は電子署名画面でサインを行う（FR-EV-005 を includes として実行）
6. 「ロールバック実行」ボタンをクリックする。システムは以下を実行する（FR-MA-006）:
   - 現行版 SOP を obsolete 状態に設定する（物理削除しない）
   - 旧版 SOP を published 状態に昇格させる
   - master_edit_event に rollback イベントを Append-only で記録する（FR-MA-015）
7. システムは進行中の作業指示に「SOP がロールバックされました」通知を送信する。進行中作業は旧版（現在の published 版）で完了させる設計とする（ALCOA+ Consistent 原則）
8. タブレット APP への旧版 SOP の配信を開始する（FR-SY-001）

**代替シナリオ**
- A1: ロールバック先バージョンが obsolete ではなく under_review の場合、システムは「対象バージョンがレビュー中です」警告を表示し承認完了後にロールバックを促す
- A2: ロールバックが緊急を要する場合（安全上の重大な誤り）、システムは管理者ロールによる緊急ロールバック（電子署名の即時生成・全端末への強制配信）フローを提供する

**例外シナリオ**
- E1: 旧版 SOP が存在しない場合（初版のロールバックは不可）、システムは「ロールバック先のバージョンが存在しません」エラーを表示する
- E2: ロールバック中にネットワークエラーが発生した場合、システムはトランザクションをロールバックし SOP 状態を変更前の状態に戻す。部分的な状態遷移を禁止する
- E3: 旧版 SOP が参照するカスタム Step タイプが廃止済みの場合、システムは「旧版が参照する Step タイプが廃止されています」警告を表示しロールバック可否の判断を品質担当に委ねる

**関連要件 ID**
- FR-MA-006, FR-MA-015, FR-MA-008, FR-EV-005, FR-AU-002

### UC-018: 参照整合性をチェックする

| 項目 | 内容 |
|---|---|
| 目的 | システムがマスタ削除・廃止・改訂時に、削除対象を参照している SOP・作業パターン・カスタム Step タイプ等を dry-run で検出し影響範囲を操作者に提示する。「壊さない」体験を技術的に保証する |
| 主アクター | システム（内部プロセス） |
| 副アクター | 品質担当・保全担当・IT 担当（操作者として dry-run 結果を確認する） |
| 事前条件 | 操作者がマスタの削除・廃止・ロールバックを要求している。操作者が適切な RBAC ロールを持つ（FR-AU-002） |
| 事後条件 | dry-run の影響件数と参照元レコード一覧が操作者に提示されている。影響件数がゼロでない場合は削除がブロックされている |

**主シナリオ**
1. 操作者がマスタ（設備マスタ・ユーザーマスタ・カスタム Step タイプ・step_flow_rule 等）の削除操作を開始する（FR-MA-008）
2. システムは削除操作前に dry-run を自動実行する。実際のデータを変更せず参照整合性のシミュレーションのみを行う
3. システムは影響範囲ビューを表示する。参照元を以下の 4 区分で表示する: SOP（文書番号・版番号・ステータス）、作業パターン（名称・対象ライン）、カスタム Step タイプ（タイプ名・バージョン）、条件分岐ルール（rule_id・対象 SOP）（FR-MA-008）
4. 影響件数がゼロの場合、システムは削除の即時実行を許可する
5. 影響が存在する場合、システムは「N 件のレコードが影響を受けます。参照を解除してから削除してください」を表示する。削除ボタンをグレーアウトする
6. 操作者は影響範囲の各レコードリンクをクリックして詳細を確認し、参照の解除または対象レコードの削除を実施する
7. 参照件数がゼロになったことを確認して削除を実行する

**代替シナリオ**
- A1: SOP 改訂の場合（削除ではなく版変更）、システムは「この改訂を適用した場合、過去 N 件の実績が旧版を参照した状態になる」件数を表示する（FR-MA-008）。旧版の実績参照はそのまま維持される（ALCOA+ Enduring 原則）
- A2: 計測器マスタの廃止の場合、廃止後も過去の作業ログが計測器 ID を参照し続けることを許容する（廃止マスタへの参照維持）。廃止後は新規記録からの参照のみをブロックする

**例外シナリオ**
- E1: dry-run の実行中にデータベースエラーが発生した場合、システムはエラーを管理者に通知し削除操作を一時停止する。データ整合性が確認されるまで削除を禁止する
- E2: 参照件数がゼロでない状態での API 経由の直接削除要求（クライアント側チェックをバイパスした試み）が来た場合、システムは API レベルでブロックしエラーレスポンスを返す（FR-MA-008 の API レベルブロック）

**関連要件 ID**
- FR-MA-008, FR-MA-004, FR-AU-002
