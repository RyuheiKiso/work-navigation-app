# 00 本書の位置づけと識別子規約

本章の責務は、機能要件サブ全体の構造的位置づけを確立し、本書を通じて使用する識別子体系・優先度区分・要件導出の方向性を確定することである。本章に記載した規約は機能要件サブ全章に適用される権威規約であり、下位章での再解釈を禁止する。

---

## 1. 機能要件サブの責務

### 1-1. IPA 共通フレームにおける位置づけ

本書（機能要件サブ）は IPA 共通フレーム 2013（SLCP-JCF2013）の「ソフトウェア要件定義プロセス」における以下の 2 タスクに対応する。

| IPA 共通フレームのタスク | 本書の対応 |
|---|---|
| 機能要件の定義 | FR-ID で識別される機能要件の検証可能命題としての記述（01 章）|
| ユースケース記述 | UC-ID で識別されるユースケースの固定フォーマットでの記述（02〜07 章）|

上位文書として「02_企画/システム化計画」が存在し、本書はその成果物を受けて作成する。下流文書として「04_概要設計」「05_詳細設計」が存在し、本書の確定事項を引き継ぐ。

### 1-2. 上位文書からの引き継ぎ

本書の記述は以下の上位文書を権威として参照する。上位文書の確定事項を本書で覆すことは禁止する。

| 上位文書 | 引き継ぐ内容 |
|---|---|
| `02_企画/システム化計画/03_機能スコープと非スコープ.md` | 機能の「コア / 対応 / 対象外」三区分の権威判定 |
| `02_企画/システム化計画/06_データモデル中核設計.md` | 3 階層手順モデル・ALCOA+ 展開・イベントストア構造 |
| `02_企画/システム化計画/12_外部システム連携アーキテクチャ（子機モード）.md` | 連携モード 3 形態・外部一意キー設計・Outbox Pattern |
| `02_企画/システム化計画/18_拡張可能Stepエンジン（アドオン機構）.md` | カスタム Step タイプ定義スキーマ・条件分岐フロー DSL |
| `02_企画/システム化計画/19_マスタ編集者体験設計（オーサリング UX とガバナンス）.md` | 5 不変条件・参照整合性ガード・RBAC マトリクス |

### 1-3. 下流文書への引き渡し

本書で確定した機能要件と UC 記述は以下の下流文書の入力として機能する。

| 下流文書 | 引き渡す内容 |

**本節で確定した方針**
本書は SLCP-JCF2013 の「機能要件定義」「ソフトウェア要件定義」タスクに対応し、FR-ID と UC-ID の二軸で要件を体系化する。
上位文書の「コア / 対応 / 対象外」三区分は本書での再判定を禁止し、上位文書の権威を維持する。
本書の確定事項は概要設計・詳細設計・テスト計画への入力文書として権威を持つ。

## 2. 識別子体系

### 2-1. 識別子の全体構成

本書で使用する識別子を以下のとおり確定する。各識別子の形式・採番規則・用途を本節で一元管理する。

| 識別子種別 | 形式 | 採番単位 | 用途 |
|---|---|---|---|
| 機能要件 ID | `FR-XX-NNN` | 機能カテゴリ別 001 から単調増加 | 機能一覧での各機能要件の永続識別子 |
| ユースケース ID | `UC-NNN` | 001 から単調増加（全 UC 横断一意） | UC 記述・対応表での UC の永続識別子 |
| 業務要件 ID | `BR-BUS-NNN` | 001 から単調増加 | 業務要件文書が管理する業務上の制約・ルール |
| インタフェース要件 ID | `IF-NNN` | 001 から単調増加 | 外部システム連携・API の要件識別 |
| エンティティ ID | `EN-NNN` | 001 から単調増加 | データモデルの主要エンティティ識別 |
| レポート要件 ID | `RP-NNN` | 001 から単調増加 | 帳票・ダッシュボード出力の要件識別 |

### 2-2. FR 機能カテゴリコード

機能要件 ID に使用するカテゴリコード `XX` を以下のとおり確定する。

| コード | 意味 | 代表機能 |
|---|---|---|
| `NV` | Navigation: SOP ナビゲーション | ロックステップ進行・ステップ詳細表示・習熟度別表示 |
| `EV` | Evidence: 証拠記録・電子サイン | 写真証拠・測定値・QR スキャン・電子サイン |
| `ST` | Status: 状態管理・中断再開・アンドン | 中断保存・再開・プレースキーパー・アンドン発報 |
| `MA` | Master: マスタ管理 | SOP 版管理・Draft 編集・Publish・ロールバック |
| `SY` | Sync: 同期・外部連携 | Outbox 送信・READ-ONLY 同期・外部キー解決 |
| `KZ` | Kaizen/CAPA | 不適合起票・CAPA 進捗・改善提案 |
| `AU` | Auth: 認証認可 | JWT 認証・RBAC 6 ロール・セッション管理 |
| `UI` | UI/UX | 多言語・ダークモード・グローブ操作・オフラインインジケーター |

### 2-3. 採番規則

採番は以下の規則に従う。

- NNN は 3 桁の 0 埋め整数（001〜999）とする。1000 以上の識別子が必要になった場合は設計を見直す
- 採番は昇順単調増加とし、欠番は許容しない。ただし廃止された要件に割り当てた ID は再利用しない
- 廃止された要件は `[廃止]` プレフィックスを付けて表に残す。物理削除は禁止する
- UC-ID は全ユースケース横断で一意とし、カテゴリコードを含まない
- BR-BUS-NNN・IF-NNN・EN-NNN・RP-NNN は各々独立した通し番号で採番する

**本節で確定した方針**
識別子は FR-XX-NNN・UC-NNN・BR-BUS-NNN・IF-NNN・EN-NNN・RP-NNN の 6 種類に確定し、各形式を本節で一元管理する。
廃止要件は物理削除を禁止し、`[廃止]` プレフィックスで表に保持することを確定する。
NNN は 3 桁 0 埋め整数とし、1000 件超過時は設計見直しのトリガーとすることを確定する。

## 3. 優先度・MoSCoW 区分の定義

### 3-1. MoSCoW 区分の定義

機能要件の優先度は MoSCoW 手法に基づき 4 区分で表現する。各区分の定義と判定基準を以下のとおり確定する。

| 区分 | 意味 | 判定基準 | ver1.0.0 での扱い |
|---|---|---|---|
| **M: Must Have** | 必須。未実装は ver1.0.0 未達と同義 | システムの二大価値（Navigation / Traceability）に直結、または ALCOA+ / ISO 9001 の充足に必要 | すべて実装する |
| **S: Should Have** | 重要。品質・安全・倫理観点で省略できない | コア機能を支える周辺機能。代替手段が存在するが代替では品質が低下する | 原則実装する。工数超過時はスコープ外に退避し PR 管理する |
| **C: Could Have** | あると良い。ユーザー体験向上に寄与するが代替手段がある | 利便性・操作性の向上が主目的。未実装でもシステム目的は達成できる | 工数に余裕があれば実装する |
| **W: Won't Have** | ver1.0.0 では実装しない | 将来版への持ち越し、または永続非提供と明示したもの | ver1.0.0 スコープ外とする |

### 3-2. MoSCoW 判定における制約

- 計画 03 章で「コア機能」と判定された機能は、すべて Must Have とする。再判定を禁止する
- 計画 03 章で「対象外と判断する」とされた機能は、すべて Won't Have とする。再判定を禁止する
- 「対応機能」として判定されたものは、設計上の必要性に応じて Must Have または Should Have を付与する
- MoSCoW 区分の変更（降格）は変更管理ログに記録し、上位文書との整合性を確認した上で行う

### 3-3. 優先度の記述形式

機能一覧（FR リスト）における優先度の記述は `M` `S` `C` `W` の 1 文字で表す。複数 UC に跨る機能で UC ごとに優先度が異なる場合は、最高優先度を当該機能の優先度とする。

**本節で確定した方針**
MoSCoW 4 区分を Must Have / Should Have / Could Have / Won't Have と定義し、各区分の判定基準を本節で確定する。
計画 03 章のコア機能は Must Have、対象外機能は Won't Have に強制マッピングすることを確定する。
優先度の変更には変更管理ログの記録を必須とすることを確定する。

## 4. 機能要件と業務要件の関係

### 4-1. 要件導出の方向性

機能要件（FR）は業務要件（BR）から導出する。逆方向の導出（機能要件から業務要件を遡及して定義する）は禁止する。この原則を「導出の一方向性」と呼び、本書全体を通じた設計制約とする。

| 文書層 | 識別子 | 責務 | 記述の視点 |
|---|---|---|---|
| 業務要件 | BR-BUS-NNN | 「なぜその業務が必要か」「業務ルールは何か」を確定する | 業務の立場から |
| 機能要件 | FR-XX-NNN | 「システムが何をするか」を検証可能命題として記述する | システムの立場から |
| ユースケース | UC-NNN | 「誰が・何のために・どのように使うか」を具体化する | ユーザーの立場から |

### 4-2. 検証可能命題としての記述

機能要件は「する論」で記述し、検証可能命題として書く。曖昧語（「適切に処理する」「できる限り対応する」等）は使用を禁止する。

許容される記述形式の例:
- 「システムは作業者が Step 完了操作を行ったとき、step_completed イベントを Append-only でイベントストアに記録する」
- 「システムは証拠ファイルに SHA-256 ハッシュを計算し、photo_hash フィールドに保存する」
- 「システムは校正期限が当日以前である計測器による数値入力操作をブロックする」

禁止される記述形式の例:
- 「システムは適切な方法で証拠を記録する」（曖昧：何が「適切」かが検証不能）
- 「システムはできるだけ早く同期する」（曖昧：数値目標がない）

### 4-3. 機能要件と UC の対応関係

1 つの FR は複数の UC から参照される場合がある。1 つの UC は複数の FR を使用する場合がある。この多対多の関係は、02 章「ユースケース総覧と関係図」の「UC ↔ FR 対応表」で管理する。

**本節で確定した方針**
機能要件は業務要件から導出し、逆流（機能要件からの業務要件遡及定義）を禁止することを確定する。
機能要件は「する論」で記述し、検証可能命題の形式を必須とすることを確定する。
FR と UC の多対多対応関係は 02 章の対応表で一元管理することを確定する。
