# 04 UC 記述_証拠記録と電子サイン

本章の責務は、証拠記録・電子サインに関わる UC-006〜UC-009 の 4 つのユースケースを固定フォーマットで記述することである。各 UC は ALCOA+ 9 原則（特に Attributable・Contemporaneous・Original・Accurate）との整合を明示した上で主シナリオ・代替シナリオ・例外シナリオ・関連 FR を記述する。

---

## 1. ユースケース記述

### UC-006: 写真証拠を取得する

| 項目 | 内容 |
|---|---|
| 目的 | クリティカルステップまたは evidence_required が true の Step において、作業員が写真を撮影・添付することで証拠を取得する。ALCOA+ の Attributable（帰属）・Contemporaneous（同時性）・Original（元データ）・Accurate（正確性）原則を技術的に充足する |
| 主アクター | 作業員 |
| 副アクター | システム（カメラ制御・SHA-256 ハッシュ計算・タイムスタンプ記録・イベント格納） |
| 事前条件 | UC-002 の Step 完了フローが進行中であり、現在の Step の input_type が photo_capture である、または evidence_required が true である。タブレットのカメラが使用可能状態にある |
| 事後条件 | step_completed イベントの payload に photo_hash（SHA-256）・photo_path（UUID）・timestamp_device が設定されている。証拠ファイルがローカルストレージに保存されている（オフライン時）またはサーバーにアップロード済みである |

**主シナリオ**
1. システムは現在の Step の instruction_text と共にカメラ起動ボタンを表示する。クリティカルフラグが true の場合は「証拠撮影が必須です」メッセージを明示する（FR-NV-005）
2. 作業員はカメラ起動ボタンをタップする。システムは ML Kit Barcode Scanning に対応したカメラ API を通じてカメラを起動する
3. 作業員は対象部品・作業状態・測定器等を撮影する
4. システムは撮影された画像ファイルに対して SHA-256 ハッシュを計算し photo_hash フィールドに保存する（FR-EV-002）。これにより ALCOA+ の Original（元データ改竄検知）原則を充足する
5. システムは撮影時点の端末タイムスタンプ（timestamp_device）を記録する。ALCOA+ の Contemporaneous（同時性）原則を充足するため、タイムスタンプは撮影操作と同一トランザクションで記録する（FR-EV-007）
6. 作業員は撮影結果プレビューを確認し「採用」または「再撮影」を選択する
7. 採用を選択した場合、システムは photo_path（UUID ベースのパス）に画像を保存し step_completed イベントへの添付を確定する
8. 作業員は「完了」ボタンをタップする。システムは step_completed イベントを Append-only で記録する（FR-EV-001）

**代替シナリオ**
- A1: 既存の画像ファイルを選択する場合（ライブラリから選択）、システムはファイル作成タイムスタンプを timestamp_device として記録し「ライブラリ選択」フラグを payload に付与する
- A2: 照明不足で撮影品質が低下している場合、システムは「照明が不足している可能性があります。確認してください」警告を表示するが、強制再撮影は求めない
- A3: 顔写り込みが検出された場合（プライバシー保護設定が有効の場合）、システムは「個人情報が含まれる可能性があります」を表示し確認操作を必須とする

**例外シナリオ**
- E1: カメラ API の起動に失敗した場合（デバイス権限拒否等）、システムは「カメラを起動できません。設定を確認してください」を表示する。クリティカルステップかつ evidence_required の場合は「証拠なしでの完了は許可されていません」を表示し操作をブロックする（FR-NV-005）
- E2: SHA-256 ハッシュ計算中にエラーが発生した場合、システムは撮影を再試行するよう促す。ハッシュなしでの記録を禁止する（ALCOA+ Original 原則）
- E3: ストレージ容量不足で画像保存に失敗した場合、システムは「ストレージが不足しています」を表示しデバイスのストレージ解放を促す

**関連要件 ID**
- FR-EV-002, FR-EV-007, FR-EV-001, FR-NV-005, FR-UI-004

### UC-007: 測定値を取得する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が計測器から測定値を入力し、USL/LSL バリデーション・校正期限プレフライトチェック・単位整合性検証・不確かさ付き記録を完了する。ALCOA+ の Accurate・Contemporaneous 原則を充足する |
| 主アクター | 作業員 |
| 副アクター | システム（校正期限プレフライト・USL/LSL 判定・単位変換・Cp/Cpk 計算・不確かさ記録） |
| 事前条件 | 現在の Step の input_type が numeric_input である。当該 Step に使用する計測器マスタ（instrument_id）が設定されている。計測器マスタの next_calibration_date が設定されている |
| 事後条件 | step_completed イベントの payload に measured_value・unit_code・uncertainty_u・uncertainty_k・instrument_id・calibration_ref が設定されている。USL/LSL 判定結果が記録されている |

**主シナリオ**
1. システムは当該 Step の表示時に使用する計測器の校正期限プレフライトチェックを実行する（FR-EV-003）。next_calibration_date が当日以前の場合はハードブロックまたはソフト警告を発する（業種設定による）
2. システムは測定値入力フィールドと共に USL・LSL の範囲をグラフィカルに表示する（FR-NV-003）。expected_unit（UCUM コード）を入力フィールドのラベルに表示する
3. 作業員は計測器で測定し数値を入力する
4. システムは入力値の unit_code と SOP の expected_unit を比較する（FR-EV-011）。換算不可能な単位不一致の場合は入力をブロックし単位修正を要求する
5. システムは入力値を measured_value（SI 基本単位）と display_value（表示単位）に分離して保持する。機器マスタの分解能で有効桁数を自動制限し False Precision を防止する
6. システムは USL・LSL との照合結果（合格/不合格）を即時表示する（FR-EV-003）
7. システムは Cp/Cpk を即時算出して画面に表示する（FR-EV-012）
8. 作業員は不確かさ情報（uncertainty_u・uncertainty_k）を確認または入力する（FR-EV-006）
9. 作業員は「完了」ボタンをタップする。システムは step_completed イベントを Append-only で記録する（FR-EV-001）

**代替シナリオ**
- A1: 測定値が USL/LSL 範囲外（不合格）の場合、システムは「規格外です。再測定するか不適合起票を行ってください」を表示し、再測定または UC-013（不適合起票）への分岐を提示する
- A2: 換算可能な異なる単位を入力した場合（例: cm → mm）、システムは換算値を自動計算して確認プロンプトを表示する
- A3: 計測器マスタに uncertainty_u が事前設定されている場合（登録値）、システムはデフォルト値を自動入力し作業員の確認操作のみを求める

**例外シナリオ**
- E1: 校正期限切れ計測器でハードブロックが設定されている場合、システムは「校正期限が超過しています。この計測器は使用できません」を表示し入力をブロックする。管理者・保全担当に警告を送信する（FR-EV-003, FR-MA-004）
- E2: instrument_id が設定されていない Step で numeric_input が要求された場合、システムは「計測器が設定されていません」警告を表示し、校正参照なしで記録される旨を作業員に通知する（ALCOA+ Accurate 原則の注意喚起）
- E3: 測定値として数値以外の文字列が入力された場合、システムはバリデーションエラーを表示し数値入力を促す

**関連要件 ID**
- FR-EV-003, FR-EV-006, FR-EV-011, FR-EV-012, FR-EV-001, FR-EV-007, FR-MA-004, FR-NV-003

### UC-008: QR スキャンで照合する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が GS1 体系（EAN/UPC/ITF-14/GS1-128）またはQR コード・Data Matrix を ML Kit Barcode Scanning で読み取り、SOP の判定条件（正誤・ロット照合）と照合して合否を判定する |
| 主アクター | 作業員 |
| 副アクター | システム（ML Kit スキャン制御・判定条件照合・scan_value 記録） |
| 事前条件 | 現在の Step の input_type が qr_scan である。SOP の judgment_condition に照合対象の正解値または照合ルールが設定されている。タブレットのカメラが使用可能状態にある |
| 事後条件 | step_completed イベントの payload に scan_value（スキャン値）と照合結果（合否）が記録されている |

**主シナリオ**
1. システムは現在の Step の instruction_text と共にスキャン対象（部品番号・ロット番号等）の説明を表示する（FR-NV-003）
2. 作業員はスキャンボタンをタップする。システムは ML Kit Barcode Scanning を通じてカメラをスキャンモードで起動する（FR-EV-004）
3. 作業員は対象のバーコード/QR コード/Data Matrix にカメラをかざす
4. システムは GS1 体系（EAN/UPC/ITF-14/GS1-128）・QR コード・Data Matrix を自動認識しデコードする（FR-EV-004）
5. システムはデコードされた scan_value と SOP の judgment_condition を照合する。合否を即時判定して画面に表示する
6. 照合が合格の場合、作業員は「完了」ボタンをタップする。システムは step_completed イベントを Append-only で記録する（FR-EV-001）

**代替シナリオ**
- A1: 照合が不合格（スキャン値と期待値が不一致）の場合、システムは「照合不一致: 期待値 [XX]、スキャン値 [YY]」を表示し再スキャンまたは不適合起票（UC-013）への誘導を行う
- A2: 複数の判定条件（許容されるロット範囲等）が設定されている場合、システムは全条件に対して照合を実行しいずれか 1 件が合格の場合に合格と判定する

**例外シナリオ**
- E1: スキャンに 5 回連続で失敗した場合（コード汚損・照明不足等）、システムは「手動入力に切り替えますか？」を提示する。手動入力の場合は手動入力フラグ（is_manual_input: true）を payload に付与する
- E2: GS1 体系以外のフォーマット（独自バーコード等）が読み取られた場合、システムは「対応していないバーコード形式です」を表示し再スキャンを促す
- E3: カメラ使用不可の場合は FR-EV-002 例外シナリオ E1 に準じる

**関連要件 ID**
- FR-EV-004, FR-EV-001, FR-NV-003

### UC-009: 電子サインを付与する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が Step 完了サインを付与する、または現場監督が作業完了の確認サインを付与する。ALCOA+ の Attributable 原則を充足し「誰がサインしたか」を不変証拠として記録する |
| 主アクター | 作業員（Step 完了サイン）・現場監督（作業完了確認サイン） |
| 副アクター | システム（サイン生成・ユーザー ID + タイムスタンプ + サーバー署名の組み合わせ記録） |
| 事前条件 | 作業員または現場監督が JWT 認証済みである（FR-AU-001）。current Step の input_type が signature である、または作業完了確認フローが開始されている |
| 事後条件 | step_completed イベントの payload に electronic_sign（ユーザー ID + タイムスタンプ + サーバー署名のハッシュ）が記録されている。ALCOA+ の Attributable 原則が技術的に充足されている |

**主シナリオ**
1. システムは電子サイン画面を表示する。サインの目的（Step 完了サイン / 作業完了確認サイン）と対象 Step を明示する（FR-EV-005）
2. サイン操作者（作業員または現場監督）はタブレット画面上で署名または確認ボタンをタップする
3. システムは現在ログイン中のユーザー ID・JWT クレームの役割・端末タイムスタンプ（timestamp_device）を取得する（FR-AU-001）
4. システムはサーバー側でユーザー ID + timestamp_device + case_id + step_id の組み合わせに対して HMAC-SHA256 サーバー署名を生成する
5. システムはサイン情報（worker_id または supervisor_id・timestamp_device・server_signature）を step_completed イベントの payload に記録する（FR-EV-001）
6. 現場監督が確認サインを行う場合、システムは監督者 ID・タイムスタンプ・サーバー署名をイベントに追記する
7. サイン完了をシステムが確認し操作者に完了フィードバックを表示する

**代替シナリオ**
- A1: 別の作業員が誤って他の作業員のサイン画面を操作した場合、システムは「ログイン中のユーザーでサインします」確認プロンプトを表示する。ログインユーザーと操作ユーザーが一致することを確認する
- A2: 現場監督が同一デバイスで確認サインをする場合、システムは監督者ロールを持つユーザーへの再認証を要求する（JWT の再発行）
- A3: 作業完了確認サインで監督者が内容に問題を発見した場合、システムは「差し戻し」操作を提供する。差し戻し理由の入力を必須とし step_rejected イベントを Append-only で記録する

**例外シナリオ**
- E1: サーバー署名生成のサーバー呼び出しが失敗した場合（オフライン時）、システムはクライアント側の暫定署名でローカル記録し is_offline: true を設定する。サーバー同期後に署名を更新する（FR-EV-007）
- E2: JWT の有効期限が切れている場合、システムは再認証を要求する。再認証成功後にサイン操作を継続する（FR-AU-005）
- E3: 役割が不適切なユーザー（作業員が監督者確認サインを行おうとした等）の場合、システムは「この操作の権限がありません」を表示しブロックする（FR-AU-002）

**関連要件 ID**
- FR-EV-005, FR-EV-001, FR-EV-007, FR-AU-001, FR-AU-002, FR-AU-005
