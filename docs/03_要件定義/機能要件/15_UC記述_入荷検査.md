# 15 UC 記述_入荷検査（IQC）

本章の責務は、入荷検査（IQC）に関わる UC-027〜UC-034 の 8 つのユースケースを固定フォーマットで記述することである。各 UC は JIS Z 9015-1（AQL 抜取検査）・ALCOA+ 9 原則・後工程ハードゲートとの整合を明示した上で主シナリオ・代替シナリオ・例外シナリオ・関連 FR を記述する。

### UC-027: 入荷材料を受入登録する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当（IQC 検査員）が入荷した材料・部品のロット情報をシステムに登録し、サンプリング計画の解決を起動する |
| 主アクター | 品質担当（IQC 検査員ロール） |
| 副アクター | システム（参照整合性検証・サンプリング計画解決・ダッシュボード更新） |
| 事前条件 | 材料マスタ（TBL-036）と仕入先マスタ（TBL-037）に当該材料・仕入先が登録済みである |
| 事後条件 | `incoming_inspections` レコードが生成され、`lots.qc_status = PENDING` が設定されている |

**主シナリオ**
1. 品質担当がハンディ APP の受入登録画面（SCR-HA-016）を起動する
2. 品質担当が入荷伝票の QR コードまたはバーコードをスキャンするか、material_code・supplier_code・lot_no・数量・到着日を手動入力する（FR-IQ-001）
3. システムは material_id・supplier_id をマスタで照合し参照整合性を検証する
4. システムはサンプリング計画（TBL-039）を material_id × supplier_id で解決し、n・Ac・Re・サンプル文字・`aql_table_snapshot` を `incoming_inspections` に時点固定で記録する（FR-IQ-002）
5. システムは `lots.qc_status = PENDING` を設定し、IQC ダッシュボード（FR-IQ-019）の「受入待ち」一覧に追加する
6. 品質担当は登録完了を確認し、サンプリング指示画面（UC-028）に遷移する

**代替シナリオ**
- A1: EDI/CSV による受入予定が事前登録済みの場合（FR-IQ-018）、システムは入荷伝票スキャンで既存レコードを自動照合し、残りの必須項目のみ入力を要求する
- A2: 緊急発注品で IQC 未完了のまま投入が必要な場合、品質担当は「緊急特採申請」オプションを選択し UC-030 の特採承認フローを先行起動する

**例外シナリオ**
- E1: material_code が材料マスタに存在しない場合、システムは「この材料コードは登録されていません」を表示し、品質担当に材料マスタの登録（UC-034）を促す
- E2: サンプリング計画が材料 × 仕入先で解決できない場合、システムはデフォルト CFG（CFG-018 AQL, CFG-019 検査水準）を適用し「デフォルトサンプリング計画を使用します」を通知する（FR-IQ-002）

**関連要件 ID**
- FR-IQ-001, FR-IQ-002, FR-IQ-018, FR-IQ-019

### UC-028: サンプリング指示を取得する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当が AQL サンプリング計画に基づく採取個数・採取方法・検査項目リストを確認し、サンプル採取準備を完了する |
| 主アクター | 品質担当（IQC 検査員ロール） |
| 副アクター | システム（サンプリング指示表示・厳しさ状態確認） |
| 事前条件 | UC-027 が完了し `incoming_inspections` レコードが PENDING 状態で存在する |
| 事後条件 | 品質担当がサンプル採取を開始し、`incoming_inspections.status = INSPECTING` になっている |

**主シナリオ**
1. システムはハンディ APP の IQC 入荷検査画面（SCR-HA-017）に以下を表示する（FR-IQ-003）:
   - サンプルサイズ n・合格判定数 Ac・不合格判定数 Re
   - 検査の厳しさ（なみ/きつい/ゆるい）と現在の切替状態（直近 X ロットの合格率）
   - 採取方法（ランダム採取 / 先頭末尾中間 / 指定個所）
   - 検査項目リスト（寸法・外観・重量等）
2. 品質担当はサンプリング指示の内容を確認し「サンプル採取開始」をタップする
3. システムは `incoming_inspections.status = INSPECTING`・`started_at` を記録する（FR-IQ-004 への遷移）

**例外シナリオ**
- E1: 計測器の校正期限が超過している場合、システムは「使用計測器の校正期限が超過しています」を表示し、保全担当への通知を自動送信する（FR-MA-004 と連携）

**関連要件 ID**
- FR-IQ-002, FR-IQ-003, FR-IQ-004, FR-IQ-008

### UC-029: 検査測定値を入力し AQL 自動判定を受ける

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当がサンプル各個体の測定値・写真・スキャン証拠を入力し、AQL 自動合否判定を受けて合否 4 区分の一次判定を完了する |
| 主アクター | 品質担当（IQC 検査員ロール） |
| 副アクター | システム（AQL 自動判定・`lots.qc_status` 更新・エスカレーション通知） |
| 事前条件 | UC-028 が完了し `incoming_inspections.status = INSPECTING` である |
| 事後条件 | `incoming_inspections.qc_status` が PASSED または REJECTED に更新されている。PASSED の場合 `lots.qc_status = PASSED` となり後工程解放される |

**主シナリオ**
1. 品質担当は各サンプル個体について測定値（FR-IQ-004）を入力する:
   - 数値測定: USL/LSL 範囲表示付きで測定値・計測器 ID・校正証明書 ID を入力
   - 写真証拠: SHA-256 ハッシュ付きで撮影・記録（FR-EV-002 基盤を再利用）
   - QR スキャン: GS1 体系で部品ロット番号を照合
2. 各サンプルの入力後にシステムが不良判定フラグを表示する
3. 全 n 個のサンプル入力完了後、システムが AQL 自動合否判定を実行する（FR-IQ-005）:
   - 不良数 ≤ Ac → ACCEPT（合格）: `lots.qc_status = PASSED`、後工程解放
   - 不良数 ≥ Re → REJECT（不合格）: 品質担当へ 4 区分判定要求を通知（UC-030 へ）
4. 判定結果・不良数・Ac/Re・検査者 worker_id・二重タイムスタンプ（device + server）が `incoming_inspections` に記録される
5. 検査の厳しさ自動切替ロジック（FR-IQ-008）が更新を評価する

**代替シナリオ**
- A1: 選別（SCREENING）判定確定後は全数選別が必要。品質担当は全個体を個別検査し合否を振り分ける。合格個体の子ロットを発行（FR-IQ-011）して `qc_status = PASSED` とする

**例外シナリオ**
- E1: 途中で計測器が校正期限超過と検出された場合、システムはその時点で入力をブロックし「校正済み代替計測器を使用してください」を表示する

**関連要件 ID**
- FR-IQ-004, FR-IQ-005, FR-IQ-006, FR-IQ-008, FR-IQ-011, FR-EV-002

### UC-030: 特採承認・返品処理を行う

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当が AQL 不合格ロットに対して特採（CONCESSION）・選別（SCREENING）・返品（REJECT）のいずれかの判定を電子サイン付きで確定する |
| 主アクター | 品質担当（SH-03） |
| 副アクター | システム（特採記録・警告設定・仕入先通知） |
| 事前条件 | UC-029 で `incoming_inspections.qc_status = REJECTED` が設定されている |
| 事後条件 | `lots.qc_status` が CONDITIONAL_PASS / SCREENING_REQUIRED / REJECTED のいずれかに更新されている。特採の場合は `concession_approvals`（TBL-041）に Append-only 記録がある |

**主シナリオ（特採の場合）**
1. 品質担当が管理 Web の特採承認コンソール（SCR-MC-010）で対象ロットを確認する
2. 品質担当が「特採（CONCESSION）」を選択し以下を入力する（FR-IQ-007）:
   - 特採理由（必須フリーテキスト）
   - 有効範囲（対象ロット番号・使用可能数量・使用可能工程の JSONB 入力）
   - 有効期限（CFG-023 デフォルト 90 日）
3. 品質担当が電子サイン（PIN 認証）で特採承認を確定する
4. システムは `concession_approvals`（TBL-041）に Append-only で記録し `lots.qc_status = CONDITIONAL_PASS` を設定する
5. 後工程で当該ロットがスキャンされた際に黄色警告バナーが強制表示される（FR-IQ-010）

**代替シナリオ（返品の場合）**
1. 品質担当が「返品（REJECT）」を選択する
2. 品質担当が返品書 PDF をアップロードし仕入先 ID・運送業者・追跡番号を入力する（FR-IQ-012）
3. システムが `return_to_vendor_records`（TBL-050）を記録し仕入先別品質実績を更新する

**例外シナリオ**
- E1: 特採有効期限が CFG-023 の最大値を超える場合、システムはエラーを表示し短縮を要求する

**関連要件 ID**
- FR-IQ-006, FR-IQ-007, FR-IQ-010, FR-IQ-012, FR-EV-005

### UC-031: 検査厳しさ切替を確認・記録する

| 項目 | 内容 |
|---|---|
| 目的 | JIS Z 9015-1 §10 の切替ルールによりシステムが検査の厳しさ（なみ/きつい/ゆるい）を自動切替した際に、品質担当がその変更を確認・記録する |
| 主アクター | システム（自動切替トリガー）・品質担当（確認） |
| 副アクター | なし |
| 事前条件 | 同一材料 × 仕入先の組み合わせで複数ロットの検査実績がある |
| 事後条件 | `incoming_inspections.severity_state` が更新され、変更理由・変更前後の状態・変更日時が記録されている |

**主シナリオ（なみ→きつい切替の場合）**
1. システムが 5 ロット中 2 ロット不合格の条件を検出し切替ルールを適用する（FR-IQ-008）
2. システムが品質担当に「検査厳しさが『なみ』→『きつい』に切り替わりました」通知を送信する
3. 品質担当が IQC ダッシュボードで切替内容を確認し「承認」を記録する
4. 次の検査ロットから「きつい」の n/Ac/Re が適用される

**例外シナリオ**
- E1: 「きつい」状態で 5 ロット連続合格できない場合、システムは「検査停止」を提案し品質担当と仕入先との協議記録を要求する

**関連要件 ID**
- FR-IQ-008

### UC-032: 後工程で材料ロットをスキャンし合格状態を確認する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が SOP 実行中に材料ロットの QR コードをスキャンした際に、IQC の合否状態を検証し未合格ロットの使用を API ハードゲートでブロックする |
| 主アクター | 作業員（SH-01） |
| 副アクター | システム（IQC バリデーション・警告バナー表示） |
| 事前条件 | UC-002 の SOP 実行フローが進行中であり、現在の Step が qr_scan タイプで target: 'material' である |
| 事後条件 | 合格ロットは通過。特採ロットは警告確認後に通過。未合格ロットはブロックされ step_completed が記録されない |

**主シナリオ（合格ロットの場合）**
1. 作業員が材料ロットの QR コードをスキャンする（FR-EV-004 基盤）
2. システムが `lots.qc_status` を参照し PASSED を確認する（FR-IQ-009）
3. システムが「照合成功」を表示し step_completed への遷移を許可する

**代替シナリオ（特採 CONDITIONAL_PASS ロットの場合）**
1. システムが CONDITIONAL_PASS を検出し黄色警告バナーを強制表示する（FR-IQ-010）
2. バナーには特採理由・有効期限・有効範囲（使用可能工程）が表示される
3. 作業員が「確認してこの材料を使用する」をタップする（dismissable 不可）
4. 確認操作が worker_id・時刻付きで記録される
5. step_completed への遷移が許可される

**例外シナリオ（未合格ロットの場合）**
- E1: `lots.qc_status` が PENDING / INSPECTING / SCREENING_REQUIRED / REJECTED の場合、システムは ERR-BIZ-016 を返し「この材料は検査未完了または不合格のため使用できません」を表示する。step_completed を受け付けない
- E2: `lots.qc_status` が NULL（IQC 未実施）の場合、システムは ERR-BIZ-017 を返し「入荷検査が未実施のため使用できません」を表示する
- E3: 特採の有効期限が超過している場合、システムはブロックし「特採期限切れです」を表示する

**関連要件 ID**
- FR-IQ-009, FR-IQ-010, FR-EV-004, FR-EV-013

### UC-033: 仕入先別品質実績を照会する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当・経営層が仕入先別の品質実績（合格率・特採率・返品率・検査の厳しさ状態）を照会し、品質改善判断を行う |
| 主アクター | 品質担当（SH-03）・経営層（SH-06） |
| 副アクター | システム（VW-010 マテリアライズドビュー・RP-009 生成） |
| 事前条件 | `incoming_inspections` に一定数のデータが蓄積されている。BAT-012 による日次集計が完了している |
| 事後条件 | 品質担当・経営層が仕入先別の品質傾向を確認できている |

**主シナリオ**
1. 品質担当が管理 Web のダッシュボード（SCR-MC-012）にアクセスする（FR-IQ-013）
2. システムは仕入先 × 材料 × 月単位の以下を表示する:
   - 受入ロット数・合格数・特採数・選別数・返品数・不合格率のトレンドグラフ
   - 現在の検査厳しさ状態（なみ/きつい/ゆるい）
   - 受入検査履歴一覧（FR-IQ-014）
3. 品質担当が特定仕入先の品質実績レポート（RP-009）を PDF で出力する
4. 個人（検査員 worker_id）単位の集計・ランキングは画面に存在しない（BR-BUS-029）

**例外シナリオ**
- E1: worker ロール（一般作業員）がダッシュボードにアクセスした場合、RBAC により ERR-AUTH-004 でアクセス拒否される

**関連要件 ID**
- FR-IQ-013, FR-IQ-014, FR-IQ-019, FR-AU-002, FR-KZ-007

### UC-034: IQC マスタを Draft-First 編集する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当が材料マスタ（TBL-036）・仕入先マスタ（TBL-037）・サンプリング計画マスタ（TBL-039）を Draft-First 方式（承認フロー付き）で登録・編集・廃止する |
| 主アクター | 品質担当（SH-03） |
| 副アクター | システム（楽観的ロック・変更履歴記録・承認通知） |
| 事前条件 | マスタ編集者ロールを持つユーザーが認証済みである |
| 事後条件 | 承認後に新版マスタが有効化されている。版管理履歴が記録されている |

**主シナリオ（サンプリング計画の新規登録の場合）**
1. 品質担当がマスタメンテ APP の SCR-MA-014（サンプリング計画マスタ）にアクセスする
2. 品質担当が材料・仕入先・AQL 値・検査水準を選択する（FR-IQ-017）
3. システムが JIS Z 9015-1 のサンプル文字テーブルから n/Ac/Re を自動計算し `aql_table_snapshot`（JSONB）を生成する
4. 品質担当が検査項目リスト（寸法・外観・重量等）を入力する
5. システムが Draft として保存し、承認者に承認要求を送信する（FR-MA-005）
6. 承認者が承認電子サインを付与し、サンプリング計画が有効化される

**関連要件 ID**
- FR-IQ-015, FR-IQ-016, FR-IQ-017, FR-MA-005, FR-MA-013
