# 12 データ要件（論理）

本章は、論理エンティティの一覧・主要属性・CRUD マトリクス・ライフサイクルと保存期間・参照整合性と時点参照・Append-only / Event Sourcing の論理境界を確定する。物理テーブル定義・カラム型・インデックス設計は概要設計書に委ねる。本章は計画 06 章（データモデル中核設計）の要件定義版権威章として機能する。

---

## 1. 論理エンティティ一覧

EN-NNN 形式の識別子で全エンティティを体系化する。マスタエンティティ（M）とトランザクションエンティティ（T）を区別する。

| ID | エンティティ名 | 区分 | 説明 |
|---|---|---|---|
| EN-001 | Tenant | M | テナント（将来のマルチサイト対応用予約。ver1.0.0 では単一テナント固定）|
| EN-002 | Site | M | 工場サイト（将来のマルチサイト用予約。ver1.0.0 では単一サイト固定）|
| EN-003 | Process（L1） | M | 工程。製造ラインの大分類（溶接・検査・組立等）|
| EN-004 | Operation（L2） | M | 作業。工程内の中分類（溶接準備・本溶接・後処理等）|
| EN-005 | Step（L3） | M | ステップ。作業内の最小実行単位かつトレサビ最小記録単位 |
| EN-006 | MasterVersion | M | SOP 版管理レコード。DRAFT→UNDER_REVIEW→PUBLISHED→ARCHIVED |
| EN-007 | WorkOrder | T | 作業指示。発行時点の sop_version_id を時点固定 |
| EN-008 | WorkExecution | T | 作業実行。WorkOrder に紐づく一実行インスタンス |
| EN-009 | StepRecord | T | Step 実行記録の集約ビュー（Append-only イベントから導出）|
| EN-010 | Evidence | T | 証拠記録（写真・数値・スキャン値）。SHA-256 ハッシュ付き |
| EN-011 | ElectronicSign | T | 電子署名レコード。WorkExecution・Step に紐づく |
| EN-012 | WorkEvent | T | 全作業イベントの Append-only ストア（XES 互換 8 属性必須）|
| EN-013 | Defect | T | 不適合レコード。4M 原因軸・写真・ステータスを持つ |
| EN-014 | CAPA | T | 是正処置・予防処置レコード。Defect または KaizenProposal に紐づく |
| EN-015 | KaizenProposal | T | Kaizen 提案レコード。起票者 ID・カテゴリ・採否を持つ |
| EN-016 | Alert | T | アンドン発報レコード。IDLE/ALERTING/ACKNOWLEDGED/RESOLVED |
| EN-017 | User | M | ユーザーマスタ。ロール・スキル・有効期限を持つ |
| EN-018 | Role | M | ロールマスタ。6 ロール定義 |
| EN-019 | Equipment | M | 設備マスタ。点検周期・is_active フラグ |
| EN-020 | Material | M | 材料マスタ。品番・品目コード・単位 |
| EN-021 | Lot | M | ロットマスタ。材料・製品の個体識別 |
| EN-022 | ExternalKey | M | 外部一意キーマッピング（external_key_binding）。親機との橋渡し |
| EN-023 | OutboxEvent | T | Outbox パターン送信キュー。PENDING→SENT/DLQ |
| EN-024 | Instrument | M | 計測器マスタ。校正証明書・次回校正期限・分解能 |
| EN-025 | WorkPattern | M | 作業パターン。外部一意キーと SOP の中間エンティティ |
| EN-026 | StepTypeDefinition | M | カスタム Step タイプ定義（JSON DSL）|
| EN-027 | StepFlowRule | M | 条件分岐フロー定義（JSON Logic）|

図: fig_logical_er（img/ 配下）を参照

**本節で確定した方針**
- 論理エンティティを EN-001〜EN-027 の 27 件で確定し、マスタ（M）とトランザクション（T）を区別する。
- Tenant / Site / WorkPattern / ExternalKey は将来拡張のための予約エンティティとして含め、ver1.0.0 では単一固定値で運用する。
- WorkEvent（EN-012）を Append-only イベントストアの中核エンティティとして確定し、XES 互換 8 必須属性を持つことを確定する。

## 2. 主要エンティティの属性・キー・関連

### 2-1. WorkEvent（EN-012）— 最重要エンティティ

| 属性 | 型・制約 | 役割 |
|---|---|---|
| event_id | UUID・NULL 禁止・PRIMARY KEY | Idempotency Key を兼用 |
| case_id | UUID・NULL 禁止・FK → WorkExecution | XES Case ID。プロセスマイニング用 |
| activity | VARCHAR・NULL 禁止 | step_id + イベント種別の結合表現（XES Activity）|
| timestamp_device | TIMESTAMPTZ・NULL 禁止 | 端末タイムスタンプ（申告時刻）|
| timestamp_server | TIMESTAMPTZ・NULL 禁止 | サーバー受信タイムスタンプ（証拠時刻）|
| worker_id | UUID・NULL 禁止・FK → User | 実施者 ID（XES Resource）|
| terminal_id | UUID・NULL 禁止 | 端末 ID（XES Resource 補足）|
| sop_version_id | UUID・NULL 禁止・FK → MasterVersion | 時点参照。発行時に固定 |
| event_type | ENUM・NULL 禁止 | step_started / step_completed / step_skipped / step_rejected / work_interrupted / work_resumed / andon_triggered / nonconformance_filed / kaizen_filed / step_skipped_by_rule / custom_step_completed |
| payload | JSONB | イベント種別固有データ（numeric_value / photo_hash / scan_value 等）|
| prev_hash | CHAR(64) | 前レコードの SHA-256 ハッシュ。ハッシュチェーン用 |
| is_offline | BOOLEAN・NULL 禁止 | オフライン記録フラグ |
| sync_lag_ms | BIGINT | 同期遅延時間（監査用）|
| is_retroactive | BOOLEAN・DEFAULT false | 後入力フラグ |

### 2-2. WorkExecution（EN-008）

| 属性 | 型・制約 | 役割 |
|---|---|---|
| execution_id | UUID・PRIMARY KEY | 実行インスタンス識別子 |
| work_order_id | UUID・FK → WorkOrder | 作業指示への参照 |
| sop_version_id | UUID・FK → MasterVersion | 時点固定 SOP 版 |
| worker_id | UUID・FK → User | 主担当作業員 |
| status | ENUM | NOT_STARTED / IN_PROGRESS / SUSPENDED / COMPLETED / CANCELLED |
| started_at | TIMESTAMPTZ | 作業開始日時 |
| completed_at | TIMESTAMPTZ | 作業完了日時 |
| skill_level_snapshot | JSONB | 作業開始時の作業者スキルレベルのスナップショット |

### 2-3. ElectronicSign（EN-011）

| 属性 | 型・制約 | 役割 |
|---|---|---|
| sign_id | UUID・PRIMARY KEY | 署名識別子 |
| execution_id | UUID・FK → WorkExecution | 対象作業実行 |
| step_id | UUID・FK → Step | 対象 Step |
| signer_id | UUID・FK → User | 署名者 ID |
| sign_type | ENUM | COMPLETION / APPROVAL / WITNESS | 
| timestamp_device | TIMESTAMPTZ・NULL 禁止 | 端末時刻（ALCOA+ Contemporaneous）|
| timestamp_server | TIMESTAMPTZ・NULL 禁止 | サーバー時刻 |
| server_signature | BYTEA | サーバー側の HMAC 署名（改竄検知用）|

### 2-4. MasterVersion（EN-006）

| 属性 | 型・制約 | 役割 |
|---|---|---|
| version_id | UUID・PRIMARY KEY | 版識別子 |
| sop_id | UUID・FK → Process/Operation 階層 | 対象 SOP |
| version_number | INTEGER・NULL 禁止 | 単調増加の版番号 |
| revision_type | ENUM | MAJOR / MINOR |
| revision_reason | TEXT・NULL 禁止・NOT EMPTY | 改訂理由 |
| effective_date | DATE・NULL 禁止 | 有効開始日 |
| approved_by | UUID・FK → User | 承認者 ID |
| approved_at | TIMESTAMPTZ | 承認日時 |
| invalidation_date | DATE | 廃止日（null = 現行版）|
| status | ENUM | DRAFT / UNDER_REVIEW / PUBLISHED / ARCHIVED |

**本節で確定した方針**
- WorkEvent に XES 互換 8 必須属性（event_id / case_id / activity / timestamp_device / timestamp_server / worker_id / terminal_id / sop_version_id）を確定し、全属性の NULL 禁止を設計制約とする。
- ElectronicSign に timestamp_device・timestamp_server の双方と server_signature を確定し、ALCOA+ Contemporaneous・Original を実装する。
- MasterVersion の revision_reason・effective_date・approved_by を NULL 禁止フィールドとして確定し、承認なしの PUBLISHED 遷移を DB 制約で防止する。

## 3. CRUD マトリクス

主要機能 × 主要エンティティの Create / Read / Update / Delete 対応表。Delete は論理削除（is_active=false）のみを意味する。

| 機能 | WorkEvent | WorkExecution | MasterVersion | ElectronicSign | Defect / CAPA | KaizenProposal | User | Equipment |
|---|---|---|---|---|---|---|---|---|
| SOP ナビゲーション | C | C/R | R | - | - | - | R | R |
| 電子サイン入力 | C | R | R | C | - | - | R | - |
| 証拠記録（写真）| C | R | R | - | - | - | - | - |
| 作業中断・再開 | C | U | R | - | - | - | - | - |
| アンドン発報 | C | R | R | - | - | - | - | - |
| Kaizen 起票 | C | R | R | - | - | C | - | - |
| CAPA 管理 | - | R | R | C | C/U | R/U | - | - |
| 監査ログ閲覧 | R | R | R | R | R | R | R | - |
| SOP 編集（Draft）| - | - | C/U | - | - | - | - | - |
| SOP 承認 | - | - | U | C | - | - | - | - |
| ユーザー管理 | - | - | - | - | - | - | C/U | - |
| 設備管理 | - | - | - | - | - | - | - | C/U |
| Outbox 送信 | R | R | - | - | - | - | - | - |
| トレサビ照会 | R | R | R | R | R | R | R | R |

注記: WorkEvent への Update / Delete は全機能で禁止（Append-only 原則）。MasterVersion への Delete は全機能で禁止。

**本節で確定した方針**
- WorkEvent への Update / Delete を CRUD マトリクスから排除し、Append-only 原則をエンティティ定義レベルで確定する。
- MasterVersion への物理 Delete を禁止し、廃止は status=ARCHIVED による論理廃止のみとすることを確定する。
- トレサビ照会機能は全主要エンティティを Read することを確定し、横断クエリ能力を保証する。

## 4. ライフサイクルと保存期間

### 4-1. 保存期間ポリシー

| エンティティ / データ種別 | 保存期間 | 根拠 |
|---|---|---|
| WorkEvent（作業実績全イベント）| 7 年以上（設定値で延長可能）| 製品保証期間＋法定保存期間。業種設定から参照 |
| ElectronicSign | WorkEvent と同期（7 年以上）| 署名の証拠価値は実績と同期 |
| MasterVersion | 永続保存（廃止後も削除不可）| ALCOA+ Enduring。廃止版参照の保証 |
| Defect / CAPA | 7 年以上 | 品質保証の根拠として同期保存 |
| KaizenProposal | 5 年以上 | 改善活動の証拠 |
| OutboxEvent（SENT / DLQ）| 90 日（SENT）/ 1 年（DLQ）| 再送対応・障害調査用 |
| 認証ログ（ログイン・JWT 失効等）| 90 日（設定値で変更可能）| セキュリティ監査用 |
| バックアップファイル | 日次 7 世代・週次 13 世代・月次 12 世代 | PITR との組み合わせで RPO 15 分を保証 |

### 4-2. データ廃棄ポリシー

退職者の PII（氏名・worker_id 実名）は退職後 60 日以内に匿名化処理（UUID 置換）を実施する。作業実績レコードの worker_id は内部 UUID へ置換し、証拠の完全性は維持する。匿名化操作は監査ログに記録する。

**本節で確定した方針**
- 作業実績 WorkEvent・ElectronicSign・Defect/CAPA を 7 年以上保存し、保存期間を設定値で業種に応じて変更可能とすることを確定する。
- MasterVersion の永続保存を確定し、廃止後も廃止版参照が機能することを保証する。
- 退職者 PII を 60 日以内に匿名化（UUID 置換）し、証拠完全性と削除権を両立させる方針を確定する。

## 5. 参照整合性と時点参照

### 5-1. 時点参照の実装原則

WorkExecution が参照するマスタ版 ID（sop_version_id）は、作業指示発行時点で固定される。固定後は外部からの更新を許容しない。この原則は計画 06 章の「原則 C: マスタの版は時点参照で固定する」を要件定義レベルで確定するものである。

時点固定される参照キーの一覧:

| 参照元フィールド | 参照先エンティティ | 固定タイミング |
|---|---|---|
| WorkExecution.sop_version_id | MasterVersion | 作業指示発行時 |
| WorkEvent.sop_version_id | MasterVersion | WorkExecution から継承 |
| WorkEvent.calibration_ref | Instrument（校正証明書）| 計測値入力時 |
| WorkExecution.skill_level_snapshot | User（スキルレベル）| 作業開始時（JSONB コピー）|
| StepRecord.instrument_id | Instrument | 計測値入力時 |

### 5-2. 外部キー整合性の方針

マスタエンティティの物理削除を禁止するため、外部キーの参照先が存在しなくなるケースを防止する。廃止（is_active=false）後も外部キーとして参照可能な状態を維持する。

WorkEvent の sop_version_id が参照する MasterVersion は廃止後も存在し続けるため、過去の WorkEvent は常に有効な参照先を持つ。この設計が ALCOA+ Available（過去記録が常に照会可能）の実装基盤となる。

**本節で確定した方針**
- 時点固定される参照キーを sop_version_id / calibration_ref / skill_level_snapshot の 3 種類として確定し、固定タイミングを明示する。
- マスタの物理削除禁止と廃止後の外部キー参照継続を確定し、ALCOA+ Available 原則の実装基盤とする。
- WorkExecution.sop_version_id の作業指示発行後更新禁止を確定する。

## 6. Append-only / Event Sourcing の論理境界

### 6-1. Append-only エンティティ

以下のエンティティはすべて Append-only（INSERT のみ許可、UPDATE / DELETE 禁止）とする。

| エンティティ | 理由 |
|---|---|
| WorkEvent（EN-012）| イベントストアの中核。ALCOA+ Original 原則 |
| ElectronicSign（EN-011）| 署名の遡及変更を禁止 |
| OutboxEvent（EN-023）| Outbox の Exactly-once を保証 |

Append-only エンティティへの「修正」は、訂正イベント（event_type: correction）を新規追記する差分レコード方式で行う。訂正イベントには訂正理由・訂正者 ID・承認者 ID・訂正対象 event_id を必須とする。

### 6-2. Mutable エンティティ

以下のエンティティは更新（UPDATE）を許容するが、変更履歴を変更イベントとして WorkEvent に記録する。

| エンティティ | 更新を許容する操作 | 変更イベント記録 |
|---|---|---|
| WorkExecution | status の遷移更新 | work_interrupted / work_resumed / work_completed イベントで代替 |
| Defect / CAPA | status の遷移更新・担当者変更 | CAPA 進捗イベントを WorkEvent に付随記録 |
| KaizenProposal | 採否判定・CAPA 昇格 | kaizen_decision イベントを WorkEvent に付随記録 |
| MasterVersion | status 遷移（DRAFT→PUBLISHED 等）| SOP 承認イベントとして別途記録 |
| User | スキルレベル更新・有効期限更新 | 変更前後の値を skill_change イベントで記録 |

### 6-3. 物理削除が一切禁止されるエンティティ

計画 06 章「原則 A: 書いたことは消えない」に従い、以下のエンティティは物理削除を禁止する。PostgreSQL のロール権限として DELETE 権限を持つロールを作成しない。

- WorkEvent（EN-012）
- ElectronicSign（EN-011）
- MasterVersion（EN-006）
- Defect / CAPA（EN-013 / EN-014）
- KaizenProposal（EN-015）

**本節で確定した方針**
- Append-only エンティティ（WorkEvent / ElectronicSign / OutboxEvent）への UPDATE / DELETE を禁止し、修正は訂正イベント追記のみを認めることを確定する。
- Mutable エンティティの更新は変更イベントとして WorkEvent に記録し、Append-only の精神を間接的に実装することを確定する。
- 物理削除禁止エンティティ 5 種を確定し、PostgreSQL ロール権限レベルで削除不可を実装することを確定する。
