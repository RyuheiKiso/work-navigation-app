# 07 UC 記述_子機モード同期

本章の責務は、子機モード同期に関わる UC-019〜UC-022 の 4 つのユースケースを固定フォーマットで記述することである。各 UC は計画 12 章（外部システム連携アーキテクチャ）の「本アプリは常に子機である」原則と Offline-First 設計の不変条件を踏まえて記述する。

### UC-019: 親機からマスタを取得する

| 項目 | 内容 |
|---|---|
| 目的 | READ-ONLY で親機（ERP/MES）からマスタデータ（SOP・ユーザー・ロット等）を REST API 経由で受信し、ローカル SQLite に同期する。親機マスタへの書き込みは行わない。「マスタの正は親機にある」原則を維持する |
| 主アクター | システム（内部プロセス） |
| 副アクター | IT 担当（連携設定・同期状態監視）・親機（ERP/MES）（マスタデータ提供） |
| 事前条件 | 連携モードが「子機モード READ-ONLY」または「子機モード双方向」に設定されている（FR-SY-006）。親機との認証設定（OAuth 2.1 Client Credentials または mTLS）が完了している（FR-AU-003）。OpenAPI 3.1 に準拠した親機 API が利用可能である |
| 事後条件 | 親機から受信したマスタデータがローカル SQLite に同期されている。sync_status が synced に更新されている。最終同期時刻がステータスバーに表示されている（FR-SY-009）。sync_failure_log にエラーがない（正常終了の場合） |

**主シナリオ**
1. システムは定期スケジュール（または Webhook の master.updated 通知受信時）でマスタ同期処理を起動する（FR-SY-001）
2. システムは親機 API に OAuth 2.1 Client Credentials フローでアクセストークンを取得する（FR-AU-003）。マスタ読み取り専用スコープを要求する
3. システムは親機 REST API（`GET /api/v1/masters?since={last_sync_timestamp}` 相当）に差分同期リクエストを送信する（URL-based バージョニング /api/v1/）
4. システムは受信した外部一意キー（external_key_source・external_key_value）を external_key_binding テーブルに格納する（FR-SY-003）
5. システムは受信したマスタデータをローカル SQLite に保存する（Offline-First 維持）（FR-SY-001）
6. システムは sync_status を synced に更新し最終同期時刻を記録する（FR-SY-009）
7. システムはステータスバーの最終同期時刻表示を更新する

**代替シナリオ**
- A1: Webhook で master.updated 通知を受信した場合、システムは HMAC-SHA256 署名を検証した上で差分同期を即時実行する（FR-SY-007）。定期スケジュールとの二重実行は Idempotency-Key ヘッダーで防止する
- A2: 親機のマスタに external_key_binding テーブルに存在しない新規外部一意キーが含まれる場合、システムは未解決キーとして pending 状態で保存しIT担当に通知する（UC-021 で解決）
- A3: マスタの受信量が大量の場合（初回同期等）、システムはページングで複数回に分けて受信し各ページ受信後に SQLite に保存する

**例外シナリオ**
- E1: 親機 API への接続が失敗した場合（ネットワーク断・親機停止等）、システムは sync_failure_log にエラーを記録する（FR-SY-004）。最終同期成功時刻のマスタで縮退稼働を継続する。「マスタが古い可能性あり」バナーを表示する（FR-SY-009）
- E2: 認証トークンの取得に失敗した場合（鍵更新・権限変更等）、システムはエラーを sync_failure_log に記録し IT 担当に通知する。作業は最終同期済みマスタで継続する
- E3: 受信マスタのコンフリクト（親機・子機の同一レコードに差異）を検出した場合、システムは親機の値を権威として解決し差異を sync_conflict_log に記録する（FR-SY-008）

**関連要件 ID**
- FR-SY-001, FR-SY-006, FR-SY-007, FR-SY-008, FR-SY-009, FR-AU-003

### UC-020: Outbox 経由で実績を送信する

| 項目 | 内容 |
|---|---|
| 目的 | 作業完了記録（step_completed イベント等）を SQLite の outbox テーブルに蓄積し、通信回復後に端末タイムスタンプ順を維持して親機 API へ非同期送信する。event_id を Idempotency Key として重複送信を防止する |
| 主アクター | システム（内部プロセス） |
| 副アクター | 親機（ERP/MES）（実績データ受信）・IT 担当（Outbox 状態監視） |
| 事前条件 | 連携モードが「子機モード双方向」に設定されている（FR-SY-006）。step_completed 等のイベントが SQLite のイベントストアに記録されている。親機 API の実績送信エンドポイント（URL・認証方式）が設定されている |
| 事後条件 | outbox テーブルの送信済みイベントが sent 状態に更新されている。親機 API から 2xx レスポンスを受信している。同期完了通知が作業者に表示されている（FR-SY-005） |

**主シナリオ**
1. システムは step_completed・work_interrupted・andon_triggered 等のイベントを SQLite の outbox テーブルに蓄積する（FR-SY-002）。各レコードに event_id（Idempotency Key）・timestamp_device・送信状態（pending）を記録する
2. システムはネットワーク接続を定期的に確認する。接続が利用可能な場合に Outbox 送信処理を起動する
3. システムは outbox テーブルから pending 状態のイベントを timestamp_device 昇順で取得する（端末タイムスタンプ順維持）（FR-SY-002）
4. システムは親機 API の認証アダプタ（API キー・OAuth 2.0・SAML 等、親機仕様に応じてプラガブル設計）でリクエストを認証する
5. システムは各イベントを Idempotency-Key ヘッダー付き（値は event_id）で親機 REST API に送信する
6. 親機 API から 2xx レスポンスを受信した場合、システムは outbox のレコードを sent 状態に更新する
7. システムは全件送信完了後に「同期完了」通知を作業者に表示する（FR-SY-005）

**代替シナリオ**
- A1: 送信が一時失敗した場合（5xx エラー・タイムアウト等）、システムは指数バックオフ（1 秒・4 秒・16 秒）でリトライする。3 回リトライ後も失敗した場合は IT 担当に通知し pending 状態を維持する
- A2: 親機 API が「重複受信」を返した場合（409 等、Idempotency Key の一致を検知した場合）、システムは重複として扱い outbox のレコードを sent 状態に更新する（設計上の重複無害化）
- A3: 送信バッチが大量の場合（長期オフライン後の一括送信等）、システムは 100 件ずつバッチ送信し各バッチ完了後に進捗を管理 UI に表示する

**例外シナリオ**
- E1: 親機 API が 4xx エラー（リクエスト不正等）を返した場合、システムはイベントを error 状態に設定し IT 担当に通知する。自動リトライは行わない。手動確認フローへ誘導する
- E2: outbox テーブルのレコード件数が上限（設定値）を超えた場合、システムは「未送信記録が蓄積しています。ネットワーク接続を確認してください」警告を IT 担当・管理者に通知する
- E3: 送信中のイベントの event_id が outbox テーブルに存在しない場合（データ不整合）、システムはエラーログを記録し管理者に通知する

**関連要件 ID**
- FR-SY-002, FR-SY-005, FR-SY-006

### UC-021: 外部一意キーを解決する

| 項目 | 内容 |
|---|---|
| 目的 | 親機のロット ID・製品コード（型式・機番・品目コード等の組み合わせ）を external_key_binding テーブルを通じて本システムの内部 ID（work_pattern_id）に変換する。「外部一意キーの正は親機にある」原則を維持する |
| 主アクター | システム（内部プロセス） |
| 副アクター | IT 担当（マッピング設定・未解決キーの解消）・親機（外部一意キーの提供元） |
| 事前条件 | 連携モードが「子機モード READ-ONLY」または「子機モード双方向」に設定されている。親機から受信した外部一意キー（external_key_source・external_key_value）が external_key_binding テーブルに存在している、または新規受信した外部一意キーの解決が要求されている |
| 事後条件 | 外部一意キーが内部の work_pattern_id に解決されている。external_key_binding テーブルの sync_status が synced に更新されている。解決不能の場合は pending 状態で IT 担当への通知が発行されている |

**主シナリオ**
1. システムは親機から受信した外部一意キー（external_key_source + external_key_value）を external_key_binding テーブルで検索する（FR-SY-003）
2. 有効期間（valid_from ≦ 現在時刻 ≦ valid_to）の条件を満たすマッピングレコードが存在する場合、対応する work_pattern_id を返す
3. work_pattern_id から SOP の 3 階層（工程→作業→Step）を取得し、ナビゲーションに使用する
4. マッピング解決後、システムは external_key_binding.sync_status を synced に更新する

**代替シナリオ**
- A1: 同一の外部一意キーに対して複数のマッピングレコードが存在する場合（バージョン切替期間中の並立）、システムは valid_from / valid_to の条件を評価して有効期間内の 1 件を選択する
- A2: IT 担当が管理 Web で外部一意キーのマッピングを手動設定する場合、システムは external_key_value（JSON 形式）・work_pattern_id・valid_from・valid_to の入力フォームを提供する。設定後に sync_status を pending から synced に更新する
- A3: 外部一意キーの構成が変更された場合（親機マスタ改訂）、IT 担当は旧レコードの valid_to を設定し新レコードを追加する（Append-only 方式）。UPDATE による上書きを禁止する

**例外シナリオ**
- E1: 有効な外部一意キーマッピングが存在しない場合（pending 状態または未登録）、システムは「作業パターンが解決できません。IT 担当に連絡してください」を表示する。作業を開始させない（安全側への倒し込み）
- E2: external_key_value の JSON パースに失敗した場合（不正なフォーマット）、システムはエラーを sync_failure_log に記録し IT 担当に通知する
- E3: work_pattern_id が存在しない SOP を参照している場合（マスタ不整合）、システムはエラーを記録し IT 担当に通知する。作業は開始させない

**関連要件 ID**
- FR-SY-003, FR-SY-001

### UC-022: 通信断・縮退時に動作する

| 項目 | 内容 |
|---|---|
| 目的 | Wi-Fi 断線・サーバー障害等の通信断時にローカル動作を継続し、通信回復後に全記録をサーバーへ同期する。「通信断を正常系の一部として設計する」Offline-First 原則を体現する |
| 主アクター | システム（内部プロセス） |
| 副アクター | 作業員（通信断状態での作業継続）・IT 担当（通信断の検知・復旧対応） |
| 事前条件 | タブレットが直前の同期でマスタデータをローカル SQLite に取得済みである。作業員が JWT 認証済みである（ローカルキャッシュ認証）。作業が進行中であるまたは開始しようとしている |
| 事後条件 | 通信断中に記録されたすべてのイベントが、通信回復後に Outbox Pattern でサーバーに同期されている。is_offline: true フラグ付きのイベントが正常に受け入れられている。sync_lag_ms（同期遅延時間）が監査ログに記録されている |

**主シナリオ**
1. システムはネットワーク状態を常時監視する。通信断を検知した場合、ステータスバーに「オフライン」インジケーターを即時表示する（FR-UI-005）
2. 通信断絶が 5 分を超えた場合、システムは Emergency Mode を自動起動する（FR-ST-012）。フルスクリーンバナーで「サーバー障害発生中」「現在の記録はローカルに保存されています」を表示する
3. Emergency Mode では緊急連絡先（IT 担当・現場監督）へのワンタップ通話と紙フォールバック PDF へのワンタップアクセスを画面上に配置する（FR-ST-012）
4. 作業員はオフライン状態でも全作業ステップの参照・実行・記録を継続する（Offline-First 設計）
5. 記録されたイベントは is_offline: true・timestamp_device を設定して SQLite ローカルに保存する（FR-EV-007）
6. 記録済みステップに「記録済（未同期）」バッジを表示し ALCOA+ の Contemporaneous 原則との整合を UI 上で明示する（FR-UI-007）
7. 通信回復を検知した場合、システムは Emergency Mode バナーを解除し同期処理を自動起動する（FR-SY-002）
8. Outbox Pattern により端末タイムスタンプ順でイベントをサーバーに送信する。event_id で重複を防止する（FR-SY-002）
9. 全件同期完了後に「同期完了」通知を作業者に表示する（FR-SY-005）

**代替シナリオ**
- A1: サーバー側の BCP 対策で Active-Standby への切り替えが発生した場合、RTO 1 時間以内に Standby サーバーが起動し作業者の記録が正常に同期される設計とする（RPO 15 分以内のデータ損失で抑える）
- A2: 子機モードでの通信断の場合、マスタ同期も停止する。「マスタが古い可能性あり」バナーを表示し最終同期時刻を明示する（FR-SY-004）
- A3: 通信断が長期間（24 時間超）にわたる場合、システムは「長期間オフラインです。IT 担当に連絡してください」警告を 6 時間ごとに表示する

**例外シナリオ**
- E1: 通信回復後の同期で親機側に重複イベントが検出された場合、サーバーは event_id の一致を検知し 2 回目以降の挿入を無視する（Idempotency Key 機構）（FR-SY-002）
- E2: ローカル SQLite のストレージが満杯になった場合、システムは「ローカルストレージが満杯です。同期を行ってください」警告を表示する。ストレージ不足でのイベント記録失敗を防ぐため、ストレージ使用量を定期監視する
- E3: 通信回復後の同期中にネットワークが再度断絶した場合、システムは送信済みイベントのみを sent 状態に更新し残りを pending 状態で保持する。次回回復時に中断位置から再開する

**関連要件 ID**
- FR-SY-002, FR-SY-004, FR-SY-005, FR-SY-009, FR-ST-012, FR-UI-005, FR-UI-007, FR-EV-007
