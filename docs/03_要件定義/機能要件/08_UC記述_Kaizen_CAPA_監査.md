# 08 UC 記述 — Kaizen / CAPA / 監査

本章は Kaizen 起票・CAPA 進捗管理・監査ログ閲覧の 3 ユースケース（UC-023〜UC-025）を詳述する。計画 09 章（不適合・CAPA・Kaizen Teian フィードバックループ）で確定した 4 段階ループをユースケース構造として具現化し、ALCOA+ の Attributable・Contemporaneous・Enduring 原則が各 UC でどう実装されるかを明示する。

---

## 1. 本章の位置づけと前提

### 1-1. UC 番号の連続性

本章の UC-023〜UC-025 は、機能要件 04 章（UC-001〜009: SOP ナビゲーション系）・05 章（UC-010〜015: マスタ管理系）・06 章（UC-016〜020: アンドン・割込み系）・07 章（UC-021〜022: 子機モード系）に続く連番として付番する。

### 1-2. Just Culture との関係

UC-023 の Kaizen 起票は、計画 09 章 6 節「Just Culture の組織原則」の技術的実装である。報告行為が人事査定と切り離されることをシステム設計で保証する。起票者 ID は ALCOA+ Attributable のために必須保持するが、RBAC により品質担当・管理者ロール以外には起票者氏名を表示しない。

**本節で確定した方針**
- UC-023〜UC-025 を Kaizen / CAPA / 監査の 3 ユースケースとして確定し、計画 09 章の 4 段階ループを UC 構造で具現化する。
- 起票者 ID は ALCOA+ 要件として保持し、RBAC で氏名の可視範囲を品質担当・管理者のみに制限する。
- 本章の全 UC は Just Culture 原則に基づく心理的安全設計の一部である。

## 2. UC-023: Kaizen を起票する

### UC-023: Kaizen を起票する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員または現場にいる誰もが、気づいた改善点・手順上の疑問・ヒヤリハットを即座に起票し、ALCOA+ に適合したトレサビ記録として残す |
| 主アクター | 作業員 |
| 副アクター | 現場監督（通知受信）、品質担当（採否判断・CAPA 昇格）|
| 事前条件 | 作業員が端末にログイン済みである。カテゴリ選択肢（手順・設備・材料・環境・その他）がマスタに登録されている |
| 事後条件 | Kaizen Teian レコードが Append-only で作成され、品質担当・現場監督に通知が配信される。起票者に「受付番号」が表示される |

**主シナリオ**
1. 作業員が現在実行中の作業画面または専用メニューから「Kaizen 起票」ボタンをタップする。
2. システムが起票フォームを表示する（カテゴリ・タイトル・詳細テキスト・写真添付（任意）・関連 Step ID（任意））。
3. 作業員がカテゴリを選択し（手順 / 設備 / 材料 / 環境 / その他）、タイトル（50 文字以内）と詳細（500 文字以内）を入力する。
4. 作業員が必要に応じてカメラで写真を撮影して添付する（最大 3 枚）。写真には SHA-256 ハッシュが自動付与される。
5. 作業員が「送信」ボタンをタップする。
6. システムがバリデーションを実行する（タイトル必須・カテゴリ必須・詳細テキスト必須）。
7. システムが Kaizen Teian レコードを生成する（kaizen_id・起票者 ID・端末 ID・timestamp_device・timestamp_server・sop_version_id を含む）。
8. オフライン状態の場合は Outbox キューに追加し、同期可能になり次第サーバーに送信する。
9. システムが起票者に受付番号（kaizen_id の先頭 8 桁）を表示する。
10. システムが品質担当・現場監督に通知を配信する（アプリ内通知）。

**代替シナリオ**
- A1: 作業員が現在作業中でない（待機状態・ログイン直後）場合: ホーム画面の「Kaizen 起票」メニューから同一フローで起票できる。
- A2: 写真添付時に容量超過（1 枚 5MB 超）が発生した場合: システムが自動でリサイズ（短辺 1200px）してから添付する。

**例外シナリオ**
- E1: オフライン同期後にサーバー側で kaizen_id の重複が検出された場合: サーバーは Idempotency Key（event_id）で重複を無視し、起票者には成功応答を返す。
- E2: タイトルが規定文字数を超過した場合: システムが即時エラーを表示し、送信をブロックする。

**関連要件 ID**
- FR-KAI-001, FR-KAI-002, FR-KAI-003, FR-KAI-010

**本節で確定した方針**
- UC-023 を作業員が起票主体として、オフライン状態でも完全に機能するユースケースとして確定する。
- 写真 SHA-256 ハッシュ・kaizen_id・timestamp の 3 属性を ALCOA+ Attributable/Contemporaneous/Original の実装として確定する。
- 起票者氏名の表示範囲を RBAC で品質担当・管理者に限定し、Just Culture の心理的安全を技術的に担保する。

## 3. UC-024: CAPA 進捗を管理する

### UC-024: CAPA 進捗を管理する

| 項目 | 内容 |
|---|---|
| 目的 | 品質担当が不適合・Kaizen Teian に基づき CAPA（是正処置・予防処置）を起票し、根本原因分析から効果確認クローズまでの進捗を管理する |
| 主アクター | 品質担当 |
| 副アクター | 現場監督（対策実施担当）、保全担当（設備関連対策）、作業員（起票者への通知受信）、管理者（閲覧のみ） |
| 事前条件 | 不適合レコードまたは Kaizen Teian レコードが存在する。品質担当ロールでログイン済みである |
| 事後条件 | CAPA レコードが Append-only で更新され、ステータスが確定する。クローズ時は品質担当の電子署名が付与される。起票者に採否通知が配信される |

**主シナリオ**
1. 品質担当が管理 Web または管理コンソールの CAPA 一覧画面を開く。
2. 対象の不適合レコードまたは Kaizen Teian レコードを選択し、「CAPA 起票」ボタンを押す。
3. システムが CAPA 起票フォームを表示する（不適合概要・発生日時・関連 SOP・影響範囲・根本原因分析手法の選択）。
4. 品質担当が根本原因分析手法（Why-Why 分析 / Ishikawa 4M 分析）を選択し、分析内容を入力する。4M 分析では Man / Machine / Material / Method / Environment の 5 軸を等価提示し、複数選択を許容する。
5. 品質担当が是正処置・予防処置の内容・担当者・期限を設定する。
6. システムが CAPA レコードを OPEN ステータスで生成する（capa_id・作成者 ID・タイムスタンプ・関連 kaizen_id または nonconformance_id を含む）。
7. 品質担当が CAPA 進捗を更新する。ステータスを OPEN → INVESTIGATING → CORRECTIVE_ACTION → VERIFICATION の順に遷移させる。各遷移には変更者 ID・タイムスタンプ・コメントを付与する。
8. 効果確認が完了したら品質担当が「クローズ」操作を行う。
9. システムが電子署名フォームを表示する（品質担当の PIN またはパスワード入力）。
10. 品質担当が電子署名を入力する。システムがサーバー側で署名を検証し、CAPA ステータスを CLOSED に更新する。
11. システムが起票者（作業員）に「採用完了通知」を配信する。

**代替シナリオ**
- A1: 対策が完了できず期限超過した場合: システムが品質担当・管理者に遅延アラートを送信する（アプリ内通知）。品質担当は期限を延長するか、CAPA を再オープンして担当者を変更する。
- A2: 根本原因が設備起因と判断した場合: 品質担当が保全担当を対策担当者に指定し、設備保全記録レコードへの参照リンクを CAPA に設定する。

**例外シナリオ**
- E1: 電子署名が 3 回連続で失敗した場合: システムがアカウントを 15 分ロックし、IT 担当に通知する。
- E2: クローズ操作時に関連する全是正処置が未完了の場合: システムが警告を表示し、クローズを強制ブロックする。未完了項目の一覧を提示する。

**関連要件 ID**
- FR-CAP-001, FR-CAP-002, FR-CAP-003, FR-CAP-004, FR-CAP-010, FR-CAP-011

**本節で確定した方針**
- UC-024 において CAPA ステータス遷移（OPEN→INVESTIGATING→CORRECTIVE_ACTION→VERIFICATION→CLOSED）を確定し、各遷移に変更者 ID・タイムスタンプを必須とする。
- 4M 分析で Man 軸単独選択を設けず 5 軸等価提示とし、根本原因分析の偏りをシステム設計で抑制する。
- クローズ操作に品質担当の電子署名を必須とし、ALCOA+ Original・Attributable 原則を電子署名で実装する。

## 4. UC-025: 監査ログを閲覧・エクスポートする

### UC-025: 監査ログを閲覧・エクスポートする

| 項目 | 内容 |
|---|---|
| 目的 | IT 担当または品質担当が、作業実行記録・認証イベント・マスタ変更履歴からなる監査証跡を閲覧し、CSV または XES 形式でエクスポートする |
| 主アクター | IT 担当、品質担当 |
| 副アクター | 管理者（エクスポート承認）|
| 事前条件 | IT 担当または品質担当ロールでログインしている。監査ログが Append-only イベントストアに格納されている |
| 事後条件 | 閲覧または指定した期間・条件のエクスポートファイルが生成される。エクスポート操作自体が監査ログに記録される（ALCOA+ Available）|

**主シナリオ**
1. IT 担当または品質担当が管理コンソールの「監査ログ」メニューを開く。
2. システムが検索フィルターを表示する（期間・イベント種別・作業員 ID・SOP ID・工程 ID）。
3. 担当者が検索条件を入力し、「検索」ボタンを押す。
4. システムがイベントログ一覧を表示する（event_id・case_id・activity・timestamp_server・worker_id（マスク表示）・terminal_id・sop_version_id の各フィールドを含む）。
5. 担当者が特定レコードをクリックして詳細を確認する（payload 含む全フィールドを展開）。
6. 担当者が「エクスポート」ボタンを押し、出力形式を選択する（CSV / XES）。
7. システムがエクスポートファイルを生成し、ダウンロードリンクを表示する（5,000 件以上の場合はバックグラウンド処理、完了時に通知）。
8. 担当者がファイルをダウンロードする。
9. システムがエクスポート操作を監査ログに記録する（操作者 ID・操作種別: export・対象期間・件数・タイムスタンプ）。

**代替シナリオ**
- A1: XES 形式を選択した場合: システムが XES 国際標準（IEEE 1849）互換フォーマットで出力する。必須属性は event_id（concept:name）・case_id（case:concept:name）・timestamp_server（time:timestamp）・worker_id（org:resource）・activity（concept:name に相当）の 5 属性とする。
- A2: IT 担当がエクスポートを操作する場合: worker_id フィールドは匿名化（内部 UUID のみ表示）する。品質担当の場合は RBAC に従い起票者氏名を表示できる。

**例外シナリオ**
- E1: ハッシュチェーン検証でチェーン不整合が検出された場合: システムが警告バナーを表示し、不整合の範囲（最初の不整合レコード ID から末尾まで）を明示する。管理者に緊急通知を送信する。
- E2: 検索結果が 100,000 件を超える場合: システムが「件数が上限を超えています。期間を絞り込んでください」と表示し、エクスポートを保留する。

**関連要件 ID**
- FR-AUD-001, FR-AUD-002, FR-AUD-003, FR-AUD-004, FR-AUD-010

**本節で確定した方針**
- UC-025 において監査ログ閲覧・CSV/XES エクスポートを IT 担当・品質担当の操作として確定する。
- XES エクスポートは IEEE 1849 互換フォーマットとし、event_id・case_id・timestamp_server・worker_id・activity の 5 必須属性を保証する。
- エクスポート操作自体を監査ログに記録し、ALCOA+ Available・Attributable 原則を二重に担保する。
