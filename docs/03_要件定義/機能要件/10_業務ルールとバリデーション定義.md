# 10 業務ルールとバリデーション定義

本章は、システムが技術的に強制する業務ルールを BR-BUS-NNN 形式の識別子で体系化し、ロックステップ進行条件・電子サイン条件・条件分岐 DSL の評価規則・バリデーション規則・権限ゲートを確定する。実装担当者はこれらのルールをアプリケーション層の実装規約として使用する。

---

## 1. ルール識別子 BR-BUS-NNN の採番規約

### 1-1. 識別子の構造

業務ルール識別子は `BR-BUS-NNN` の形式を採用する。BR は Business Rule の略称、BUS は本システムにおける業務ルール（Business）の区分、NNN は 3 桁の連番である。

| 範囲 | 区分 |
|---|---|
| BR-BUS-001〜009 | ロックステップ進行条件 |
| BR-BUS-010〜019 | 電子サイン条件 |
| BR-BUS-020〜029 | 条件分岐 DSL の評価規則 |
| BR-BUS-030〜039 | バリデーション規則 |
| BR-BUS-040〜049 | 権限・スキル等級ゲート |

### 1-2. ルールの記述形式

各ルールは以下の形式で記述する。

| 項目 | 内容 |
|---|---|
| ID | BR-BUS-NNN |
| ルール名 | 短い識別名 |
| 前提条件 | ルールが適用される前提 |
| ルール内容 | システムが強制する制約の正確な記述 |
| 違反時の処理 | ブロック / 警告 / ログ記録 のいずれか |
| 関連 FR | 対応する機能要件 ID |

**本節で確定した方針**
- BR-BUS-NNN 識別子を確定し、5 つの区分（001-009 / 010-019 / 020-029 / 030-039 / 040-049）で体系化する。
- 各ルールの前提・内容・違反時処理・関連 FR を必須記述項目として確定する。
- 本章のルールはアプリケーション層の実装規約として機能し、違反をシステムが技術的に強制する。

## 2. ロックステップ進行条件（BR-BUS-001〜009）

### 2-1. ルール一覧

**BR-BUS-001: 前 Step 完了前の次 Step 開始禁止**
- 前提条件: 作業員が現在 Step の次の Step を開始しようとしている
- ルール内容: 現在 Step の step_completed または step_skipped イベントが Append-only ストアに記録されていない限り、次 Step の step_started イベントを受け付けない
- 違反時の処理: ブロック（次 Step ボタンを無効化。タップしても反応しない）
- 関連 FR: FR-NAV-001, FR-NAV-002

**BR-BUS-002: クリティカル Step の証拠取得完了要件**
- 前提条件: Step の evidence_required フラグが true に設定されている
- ルール内容: 写真 / バーコードスキャン値 / 計測値のいずれか（Step の input_type に従う）が payload に含まれていない step_completed イベントを受け付けない
- 違反時の処理: ブロック（「証拠を記録してください」エラーを表示）
- 関連 FR: FR-NAV-003, FR-EVI-001

**BR-BUS-003: FMEA 高 RPN Step の追加確認要件**
- 前提条件: Step の fmea_rpn_flag が true に設定されている
- ルール内容: step_completed 前に確認チェックボックス（「この Step の記録を確認しました」）への同意操作を必須とする
- 違反時の処理: ブロック（確認チェックなしの完了を受け付けない）
- 関連 FR: FR-NAV-004

**BR-BUS-004: 電子サイン必須 Step の署名完了要件**
- 前提条件: SOP マスタで電子サイン必須フラグが当該 Step に設定されている
- ルール内容: ElectronicSign レコードが当該 Step・当該 WorkExecution に紐づいて存在しない場合、step_completed イベントを受け付けない
- 違反時の処理: ブロック（「電子サインが必要です」エラーを表示）
- 関連 FR: FR-SIG-001

**BR-BUS-005: 作業完了条件**
- 前提条件: 作業員が「作業完了」操作を行おうとしている
- ルール内容: SOP の全 Step（スキップ可と設定されたものを除く）について step_completed または step_skipped イベントが存在しない場合、作業完了イベントを受け付けない
- 違反時の処理: ブロック（未完了 Step 一覧を表示）
- 関連 FR: FR-NAV-005

**BR-BUS-006: 差し戻し後の再実行要件**
- 前提条件: 現場監督が Step を差し戻した（step_rejected イベントが記録された）
- ルール内容: step_rejected が記録された Step については、作業員が改めて step_started → step_completed の手順を実行する必要がある。差し戻し前の step_completed は有効な完了として扱わない
- 違反時の処理: ブロック（差し戻された Step を再度実行するよう促す）
- 関連 FR: FR-NAV-006

**BR-BUS-007: 校正期限切れ計測器の入力ブロック（ハードモード）**
- 前提条件: 計測器マスタの設定がハードブロックモードであり、使用計測器の next_calibration_date が当日以前である
- ルール内容: 当該計測器を使用した数値入力の step_completed イベントを受け付けない
- 違反時の処理: ブロック（「計測器の校正期限が切れています。管理者に連絡してください」を表示）
- 関連 FR: FR-EVI-005

**BR-BUS-008: 作業指示発行時の SOP 版固定**
- 前提条件: 作業指示が発行される
- ルール内容: 作業指示発行時点で有効な SOP 版 ID を sop_version_id として WorkExecution レコードに固定する。以後、その作業指示に紐づく全イベントは同じ sop_version_id を参照する
- 違反時の処理: ログ記録（sop_version_id が変更された場合は不整合エラーとして記録し、差異を管理者に通知）
- 関連 FR: FR-NAV-007

**BR-BUS-009: 進行中作業への SOP 版切替禁止**
- 前提条件: 作業指示のステータスが IN_PROGRESS である
- ルール内容: 進行中の作業指示に紐づく sop_version_id を変更できない。新版 SOP は新規作業指示から適用する
- 違反時の処理: ブロック（API レベルで更新操作を拒否）
- 関連 FR: FR-NAV-007, FR-MAS-003

**本節で確定した方針**
- BR-BUS-001〜009 のロックステップ進行条件を確定し、前 Step 完了・証拠取得・電子サイン・全 Step 完了の 4 条件を作業完了の必須ゲートとして設定する。
- 校正期限切れ計測器のハードブロック（BR-BUS-007）を確定し、ALCOA+ Accurate 原則の実装とする。
- 作業指示発行時の SOP 版固定（BR-BUS-008）と進行中の版切替禁止（BR-BUS-009）を確定し、時点参照の不変性を保証する。

## 3. 電子サイン条件（BR-BUS-010〜019）

### 3-1. ルール一覧

**BR-BUS-010: 電子サイン実施者の認証要件**
- 前提条件: 電子サイン操作が行われる
- ルール内容: 電子サインは必ずシステムにログイン済みのユーザーが、現在のセッションで認証を再確認（PIN または パスワード再入力）した上で実施する。他者への委任は認めない
- 違反時の処理: ブロック（未認証状態での署名操作を禁止）
- 関連 FR: FR-SIG-001, FR-SIG-002

**BR-BUS-011: スキル等級による電子サイン制約**
- 前提条件: Step の skill_level_required が L3 以上に設定されている
- ルール内容: 対象 Step の電子サインは、当該作業員の工程スキルレベルが Step の skill_level_required 以上である場合のみ有効とする。スキル不足の作業員が署名した場合、署名は記録されるが「スキル不足署名」フラグが付与され、現場監督への確認通知が送信される
- 違反時の処理: 警告（署名は受け付けるが、フラグ付与と通知を実施）
- 関連 FR: FR-SIG-003

**BR-BUS-012: 二重サイン禁止**
- 前提条件: 同一 Step への 2 回目の署名操作が行われる
- ルール内容: 同一 WorkExecution・同一 Step に対して同一ユーザーが複数回署名することを禁止する。ただし「確認署名」（別ロールによる二重確認）は別ユーザーによる署名として許容する
- 違反時の処理: ブロック（「すでに署名済みです」エラーを表示）
- 関連 FR: FR-SIG-002

**BR-BUS-013: 承認操作への電子サイン必須**
- 前提条件: SOP 版の承認・CAPA クローズ・校正証明書登録のいずれかが行われる
- ルール内容: 上記 3 操作には必ず品質担当または管理者ロールによる電子サイン（PIN または パスワード再入力）を必須とする
- 違反時の処理: ブロック（署名なしの承認操作を API レベルで拒否）
- 関連 FR: FR-SIG-004, FR-MAS-002

**BR-BUS-014: 電子サインのタイムスタンプ保証**
- 前提条件: 電子サインが実施される
- ルール内容: ElectronicSign レコードには timestamp_device（端末時刻）と timestamp_server（サーバー受信時刻）の双方を必須とする。いずれか一方でも欠ける場合は署名を受け付けない
- 違反時の処理: ブロック（ALCOA+ Contemporaneous 原則の実装）
- 関連 FR: FR-SIG-001

**本節で確定した方針**
- BR-BUS-010〜014 の電子サイン条件を確定し、認証再確認・スキル等級チェック・二重サイン禁止・タイムスタンプ双方保証を電子サインの必須要件とする。
- 承認・CAPA クローズ・校正証明書登録への電子サイン必須（BR-BUS-013）を確定し、ALCOA+ Original・Attributable の実装とする。
- スキル不足署名は禁止せず警告フラグ付与で記録することで、ALCOA+ Complete 原則（完全な記録）を優先する。

## 4. 条件分岐 DSL の評価規則（BR-BUS-020〜029）

### 4-1. ルール一覧

**BR-BUS-020: step_flow_rule の評価タイミング**
- 前提条件: source_step に紐づく step_completed イベントが記録された直後
- ルール内容: source_step_id が一致する全 step_flow_rule を、priority の昇順（小さい値が優先）で評価する。最初に条件が true になったルールのアクションを実行し、残りのルールの評価を停止する
- 違反時の処理: ログ記録（評価結果をトランザクションに記録）
- 関連 FR: FR-FLW-001

**BR-BUS-021: 競合条件の解決**
- 前提条件: 同一 source_step に同一 priority の step_flow_rule が複数存在する
- ルール内容: 同一 priority の複数ルールが存在する場合はエラーとして扱い、SOP 保存時に登録を拒否する。priority の重複を許容しない
- 違反時の処理: ブロック（SOP 保存時にバリデーションエラーを返す）
- 関連 FR: FR-FLW-002

**BR-BUS-022: デフォルト分岐の保証**
- 前提条件: step_flow_rule の全条件が false になった場合
- ルール内容: 全ルールの条件が false の場合は、通常の次 Step 進行（SOP 上の次番号 Step）に進む。デフォルト分岐を step_flow_rule に明示的に記述する必要はない
- 違反時の処理: 通常遷移を実行
- 関連 FR: FR-FLW-001

**BR-BUS-023: skip アクションの記録要件**
- 前提条件: step_flow_rule の action が skip である
- ルール内容: skip が発動した場合、`step_skipped_by_rule` イベントを Append-only ストアに記録する。rule_id・condition_evaluated（条件式と評価時入力値のスナップショット）を必須フィールドとする
- 違反時の処理: ブロック（記録なしの skip を禁止）
- 関連 FR: FR-FLW-003

**BR-BUS-024: insert アクションの挿入位置制約**
- 前提条件: step_flow_rule の action が insert である
- ルール内容: insert_position は `before_target` または `after_source` のいずれかのみ有効とする。挿入される Step は SOP の通常フローには含まれず、条件分岐によってのみ動的に追加される
- 違反時の処理: ブロック（insert_position が許可リスト外の場合は SOP 保存時にエラー）
- 関連 FR: FR-FLW-004

**BR-BUS-025: 循環参照の禁止**
- 前提条件: step_flow_rule の登録時
- ルール内容: DAG（有向非巡回グラフ）検証を API サーバー側で実行し、goto または insert によって SOP の実行順序に循環が生じないことを確認する。循環が検出された場合は登録を拒否する
- 違反時の処理: ブロック（DAG 検証エラーを返す）
- 関連 FR: FR-FLW-005

**本節で確定した方針**
- BR-BUS-020〜025 の条件分岐 DSL 評価規則を確定し、priority 昇順評価・デフォルト通常遷移・skip 時の記録義務・循環禁止を確定する。
- 同一 priority の重複禁止（BR-BUS-021）を SOP 保存時バリデーションとして確定し、曖昧な評価順序を排除する。
- skip 発動の完全記録（BR-BUS-023）を確定し、ALCOA+ Complete 原則の step_flow_rule 版実装とする。

## 5. バリデーション規則（BR-BUS-030〜039）

### 5-1. ルール一覧

**BR-BUS-030: 数値入力の上下限チェック**
- 前提条件: Step の input_type が numeric_input であり、Step に USL/LSL が設定されている
- ルール内容: 入力された数値が LSL 以上 USL 以下の場合のみ「合格」と判定する。LSL 未満または USL 超過の場合は「不合格」フラグを付与してイベントに記録する。入力自体はブロックしない（ALCOA+ Contemporaneous 優先）
- 違反時の処理: 警告（不合格フラグ付与・不適合報告への誘導バナーを表示）
- 関連 FR: FR-EVI-002

**BR-BUS-031: 必須入力項目のブランク禁止**
- 前提条件: Step の input_type が numeric_input / text_input / photo_capture / qr_scan のいずれかである
- ルール内容: 入力値が null または空文字の場合は step_completed イベントを受け付けない。各 input_type の必須入力定義は以下のとおりとする。numeric_input: 数値が入力されていること。text_input: 1 文字以上入力されていること。photo_capture: 写真が 1 枚以上添付されていること。qr_scan: スキャン値が取得されていること
- 違反時の処理: ブロック（入力フィールドのエラーハイライトを表示）
- 関連 FR: FR-EVI-001

**BR-BUS-032: 文字数制約**
- 前提条件: text_input フィールドに文字数上限が設定されている
- ルール内容: 入力文字数が Step 設定の max_length を超える場合は入力をブロックする。Kaizen 起票のタイトルは 50 文字以内、詳細は 500 文字以内を上限とする。不適合報告の概要は 200 文字以内を上限とする
- 違反時の処理: ブロック（リアルタイム文字数カウンタで超過を即時表示）
- 関連 FR: FR-KAI-001, FR-CAP-001

**BR-BUS-033: UCUM 単位の整合チェック**
- 前提条件: Step の expected_unit（UCUM コード）が設定されており、numeric_input が行われる
- ルール内容: 入力値の unit_code と Step の expected_unit を比較する。同一単位または換算可能な単位の場合は受け付ける。換算不可能な単位の場合は入力をブロックし、単位の修正を要求する
- 違反時の処理: ブロック（換算不可の場合）または警告（換算可能な場合）
- 関連 FR: FR-EVI-003

**BR-BUS-034: 計測器分解能による有効桁数制限**
- 前提条件: instrument_id が指定されており、計測器マスタに resolution（分解能）が設定されている
- ルール内容: 入力された numeric_value の小数点以下桁数が計測器の分解能を超える場合、自動で分解能に丸めてからイベントに保存する。False Precision の防止が目的である
- 違反時の処理: 自動丸め（警告バナーで「有効桁数に丸めました」を表示）
- 関連 FR: FR-EVI-004

**BR-BUS-035: スキップ理由の必須入力**
- 前提条件: 作業員が Step をスキップしようとしている
- ルール内容: step_skipped イベントの reason フィールドが null または空文字の場合は受け付けない。スキップ理由は自由テキストで必須入力とする
- 違反時の処理: ブロック（「スキップ理由を入力してください」エラーを表示）
- 関連 FR: FR-NAV-008

**BR-BUS-036: 後入力フラグの付与と承認要件**
- 前提条件: is_retroactive フラグが true のイベントが送信される
- ルール内容: 後入力イベント（紙フォールバック後の再入力等）には必ず「後入力理由・後入力日時・承認者 ID」を付与する。承認者は現場監督以上のロールを持つユーザーとする
- 違反時の処理: ブロック（承認者 ID が欠ける後入力イベントを拒否）
- 関連 FR: FR-NAV-009

**本節で確定した方針**
- BR-BUS-030〜036 のバリデーション規則を確定し、数値上下限・必須入力・文字数・単位整合・桁数・スキップ理由・後入力承認の 7 規則を確定する。
- 数値の上下限違反は記録をブロックせず不合格フラグで記録することで、ALCOA+ Contemporaneous（同時記録）を優先する設計を確定する。
- 単位換算不可能な場合のブロック（BR-BUS-033）と計測器分解能による自動丸め（BR-BUS-034）を確定し、ALCOA+ Accurate 原則を実装する。

## 6. 権限・スキル等級ゲート（BR-BUS-040〜049）

### 6-1. ルール一覧

**BR-BUS-040: スキル等級不足による単独実行ブロック**
- 前提条件: Step の skill_level_required が設定されており、実行しようとしている作業員の当該工程スキルレベルが要求を満たさない
- ルール内容: 作業員のスキルレベル（L1〜L4）が Step の skill_level_required 未満の場合、step_started イベントを受け付けない
- 違反時の処理: ブロック（「このステップの実行にはスキルレベル Lx 以上が必要です」を表示し、現場監督への確認を促す）
- 関連 FR: FR-SKL-001

**BR-BUS-041: スキル有効期限切れのブロック**
- 前提条件: 作業員のスキル有効期限が以下の 3 条件のいずれかを満たす場合（有効期限が当日以前 / 対象 SOP が X 版以上更新 / 最終実施日から N 日経過）
- ルール内容: スキル有効期限切れの作業員による当該工程の Step 実行をブロックする。ブロック後、現場監督に再認定を促す通知を送信する
- 違反時の処理: ブロック（「スキル有効期限が切れています。現場監督に再認定を申請してください」を表示）
- 関連 FR: FR-SKL-002

**BR-BUS-042: SOP 承認操作の権限制限**
- 前提条件: SOP の状態を UNDER_REVIEW から PUBLISHED に変更する操作が行われる
- ルール内容: SOP 承認操作は品質担当ロールまたは管理者ロールのみが実行できる。マスタ編集者・現場監督は承認操作を行えない
- 違反時の処理: ブロック（API レベルで権限チェックを実施し、403 Forbidden を返す）
- 関連 FR: FR-MAS-002

**BR-BUS-043: 監査ログ直接参照の IT 担当制限**
- 前提条件: IT 担当ロールが監査ログの worker_id フィールドを含むクエリを実行する
- ルール内容: IT 担当ロールには監査ログの worker_id を実名で参照する権限を与えない。worker_id は内部 UUID のみで表示する。品質担当・管理者ロールは実名参照を許可する
- 違反時の処理: 自動マスク（RBAC フィルタを API レスポンス層に実装）
- 関連 FR: FR-AUD-005

**BR-BUS-044: 集計画面での個人別表示禁止**
- 前提条件: 作業実績集計クエリが実行される
- ルール内容: 管理コンソールの集計 API は worker_id 単位でのグループ集計結果を返さない。集計単位は工程・シフト・日付のみとする。個人別生産性ランキングの算出クエリを API として実装しない
- 違反時の処理: ブロック（個人別集計クエリを受け付けない API 設計で実装）
- 関連 FR: FR-AGG-001

**BR-BUS-045: 不適合報告者 ID の権限制限**
- 前提条件: 不適合レコードまたは Kaizen Teian レコードの reporter_id フィールドへのアクセスが行われる
- ルール内容: reporter_id（起票者 ID）の実名参照は品質担当・管理者ロールのみに許可する。現場監督・IT 担当・作業員は reporter_id を UUID のみで参照する
- 違反時の処理: 自動マスク（RBAC フィルタで実装）
- 関連 FR: FR-KAI-003, FR-CAP-005

**本節で確定した方針**
- BR-BUS-040〜045 の権限・スキル等級ゲートを確定し、スキル不足ブロック・SOP 承認権限制限・IT 担当の個人情報制限・個人別集計禁止を確定する。
- 個人別集計クエリの API 非実装（BR-BUS-044）を確定し、RBAC の技術的制約として倫理品質「目的限定性」を実装する。
- 不適合報告者 ID の実名参照制限（BR-BUS-045）を確定し、Just Culture の心理的安全をデータアクセス層で保証する。
