# 14 帳票・出力要件

本章は、本システムが生成する帳票・出力物を RP-NNN 形式の識別子で体系化し、各帳票の必須項目・出力形式・改ざん防止技術要件・保存要件を確定する。ALCOA+「Original（元データ）」の原則に基づき、全帳票の原本性確保方法を明示する。

図: fig_report_catalog（img/ 配下）を参照

---

## 1. 帳票一覧

| ID | 帳票名 | 対象データ | 出力形式 | 主な利用者 |
|---|---|---|---|---|
| RP-001 | SOP 実行記録 | WorkExecution・WorkEvent・ElectronicSign | PDF / CSV | 品質担当・監査者・現場監督 |
| RP-002 | 監査ログ出力 | WorkEvent（全イベント）| CSV / XES | IT 担当・品質担当・監査者 |
| RP-003 | 不適合・CAPA 記録 | Defect・CAPA | PDF / CSV | 品質担当・管理者 |
| RP-004 | Kaizen 記録 | KaizenProposal | PDF / CSV | 品質担当・管理者・経営層 |
| RP-005 | 設備点検記録 | WorkExecution（保全作業）・Equipment | PDF | 保全担当・品質担当 |
| RP-006 | 出力レポート（集計）| WorkExecution・WorkEvent（集計）| CSV | 管理者・経営層 |

**本節で確定した方針**
- 帳票を RP-001〜RP-006 の 6 種で確定し、各帳票の対象データ・出力形式・主利用者を確定する。
- 全帳票の生成操作を監査ログに記録し（ALCOA+ Attributable）、誰がいつ帳票を出力したかを追跡可能とする。
- RP-006（集計レポート）は工程・シフト単位の集計のみとし、個人別データを含めないことを確定する。

## 2. RP-001: SOP 実行記録

### 2-1. 必須項目

| 区分 | 必須項目 |
|---|---|
| ヘッダー | 帳票タイトル・工場名・生成日時・生成者 ID・文書番号（WorkExecution ID の先頭 8 桁）|
| 作業情報 | SOP 番号・SOP 版番号（sop_version_id）・改訂理由・有効開始日・作業員氏名（または ID）・作業開始日時・作業完了日時・使用端末 ID |
| Step 実行記録 | 全 Step の番号・指示テキスト・入力値（数値 / テキスト / スキャン値）・合否判定結果・証拠写真のサムネイルと SHA-256 ハッシュ・実施日時 |
| 電子サイン欄 | 署名者氏名・署名者ロール・署名日時・デジタル印影・server_signature 値 |
| フッター | ハッシュチェーン値（直前のイベントの prev_hash）・「本帳票は参照用複製です。原本はシステム内のイベントレコードです」注記 |

### 2-2. 出力形式と生成方式

PDF 出力は ISO 32000 準拠フォーマットで生成する。管理 Web のバックエンド（Rust）でサーバーサイド PDF 生成を行う。フォントは Noto Sans JP（日本語）・Noto Sans（英語）を埋め込み、文字化けを防止する。CSV 出力は UTF-8 BOM 付きで生成し、Excel での直接開封を保証する。

### 2-3. 改ざん防止技術要件

- PDF には WorkExecution ID・生成日時・生成者 ID・ハッシュ値を埋め込みメタデータとして記載する。
- 生成された PDF の SHA-256 ハッシュを DocumentHash テーブルに記録する。事後的な改竄を検知可能にする。
- フッターに「本帳票は参照用複製です。原本はシステム内のイベントレコードです」を固定文として表示する。

### 2-4. 保存要件

生成された PDF は生成操作のログとともに保存する。保存期間は WorkEvent の保存期間に準じる（7 年以上）。PDF ファイルは NAS の専用ディレクトリに UUID ファイル名で格納し、DB に格納パスと SHA-256 ハッシュを記録する。

**本節で確定した方針**
- RP-001 の必須項目として全 Step 記録・証拠写真 SHA-256 ハッシュ・ElectronicSign の server_signature・ハッシュチェーン値を確定する。
- PDF は ISO 32000 準拠・Noto フォント埋め込み・サーバーサイド生成・DocumentHash テーブルへのハッシュ記録を確定する。
- 「本帳票は参照用複製です」注記を全 PDF の固定フッターとして確定し、ALCOA+ Original 原則（原本はイベントレコード）を明示する。

## 3. RP-002: 監査ログ出力

### 3-1. XES 互換フォーマットの必須 8 属性

XES（eXtensible Event Stream、IEEE 1849）国際標準に準拠した出力フォーマットとする。必須 8 属性は以下のとおりである。

| XES 属性名 | 対応フィールド | データ型 |
|---|---|---|
| concept:name | activity（step_id:event_type）| string |
| case:concept:name | case_id（WorkExecution ID）| string |
| time:timestamp | timestamp_server | date |
| org:resource | worker_id | string |
| lifecycle:transition | event_type（start / complete 等）| string |
| concept:instance | event_id | string |
| org:group | terminal_id | string |
| sop:version | sop_version_id | string（拡張属性）|

### 3-2. 出力契機と対象範囲

| 出力契機 | 対象範囲 |
|---|---|
| 手動出力（管理コンソールからの操作）| 指定期間・指定工程・指定作業員（RBAC によるフィルタリング）|
| 内部監査時 | 品質担当が指定した WorkExecution 群 |
| 外部監査対応時 | 品質担当・管理者が指定した証拠範囲 |

### 3-3. CSV 出力フォーマット

CSV 出力は以下のカラム構成とする: event_id, case_id, activity, timestamp_device, timestamp_server, worker_id（マスク処理の有無は RBAC による）, terminal_id, sop_version_id, event_type, is_offline, sync_lag_ms, payload（JSON 文字列）。

**本節で確定した方針**
- RP-002 の XES 出力を IEEE 1849 準拠・必須 8 属性（concept:name / case:concept:name / time:timestamp / org:resource / lifecycle:transition / concept:instance / org:group / sop:version）として確定する。
- CSV 出力は UTF-8 BOM 付き・worker_id の RBAC マスク処理を確定する。
- 出力操作を監査ログに記録し（ALCOA+ Attributable）、帳票出力の証跡を残すことを確定する。

## 4. RP-003: 不適合・CAPA 記録

### 4-1. 記録内容

| 区分 | 項目 |
|---|---|
| 不適合情報 | 不適合 ID・発生日時・発見者（起票者 ID）・発生工程・不適合種別・4M 原因軸（複数選択可）・詳細テキスト・証拠写真（サムネイル + SHA-256 ハッシュ）|
| CAPA 情報 | CAPA ID・起票日・担当者・根本原因分析手法・分析内容・是正処置・予防処置・期限・完了日・効果確認結果 |
| 承認・クローズ情報 | クローズ者（品質担当）・クローズ日時・電子署名値 |

### 4-2. 出力形式と保存期間

PDF 出力（ISO 32000 準拠）を主形式とし、CSV 出力もサポートする。保存期間は 7 年以上（WorkEvent と同期）。RP-003 の生成操作も監査ログに記録する。

**本節で確定した方針**
- RP-003 に不適合の 4M 原因軸（5 軸）・CAPA の是正・予防処置・電子署名値を必須項目として確定する。
- 不適合 ID と CAPA ID の相互参照を帳票に含め、トレサビ一元照会を可能にすることを確定する。
- 保存期間 7 年以上を確定し、WorkEvent との整合を保つ。

## 5. 出力形式と改ざん防止

### 5-1. ALCOA+「Original（元データ）」を確保する技術要件

全帳票に共通して以下の改ざん防止技術を適用する。

| 技術要件 | 内容 |
|---|---|
| 原本の宣言 | 全 PDF の固定フッターに「原本はシステム内のイベントレコードです。本 PDF は参照用複製です」を表示する |
| 生成時ハッシュ記録 | 全 PDF 生成時に SHA-256 ハッシュを DocumentHash テーブルに記録する。PDF の事後改竄を検知可能にする |
| 生成操作の監査ログ | 全帳票の生成操作を WorkEvent テーブルに `report_generated` イベントとして記録する（操作者 ID・対象期間・件数・生成ファイルハッシュを含む）|
| タイムスタンプの開示 | 全帳票に timestamp_server（サーバー時刻）を表示する。timestamp_device（端末時刻）との差分（sync_lag_ms）も記載する |

### 5-2. 業務要件

| 業務要件 | 内容 |
|---|---|
| 帳票の訂正手順 | 帳票内容に誤りが判明した場合、元のイベントレコードに訂正イベントを追記し、新たな帳票を再生成する。旧帳票に「更新版があります」注記を付与する |
| 帳票への電子封印 | RP-001 の電子サイン付き帳票については、서버 署名（server_signature）をフッターに表示し、真正性を証明する |
| 長期保存フォーマット | PDF/A-3（ISO 19005-3）サブセットを採用し、50 年後のレンダリング可能性を担保する |

**本節で確定した方針**
- 全帳票に「原本はイベントレコード」の固定フッター・生成時 SHA-256 ハッシュ記録・`report_generated` イベント記録の 3 点セットで改ざん防止を実装することを確定する。
- 帳票の訂正は訂正イベント追記＋再生成で行い、旧帳票への「更新版あり」注記を確定する。
- 長期保存フォーマットを PDF/A-3 として確定し、ALCOA+ Enduring（耐久性）の帳票版実装とする。
