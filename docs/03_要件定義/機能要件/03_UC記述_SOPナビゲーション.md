# 03 UC 記述_SOP ナビゲーション

本章の責務は、SOP ナビゲーションに関わる UC-001〜UC-005 の 5 つのユースケースを固定フォーマットで記述することである。各 UC は主シナリオ・代替シナリオ・例外シナリオ・関連 FR を必ず記述する。

---

## 1. ユースケース記述

図: fig_sequence_step_execution（img/ 配下）を参照

### UC-001: 作業を開始する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員がタブレット端末で SOP を選択し、工程→作業→Step の選択フローを経て作業セッション（case_id）を開始する |
| 主アクター | 作業員 |
| 副アクター | システム（case_id 生成・習熟度プロファイル取得・プレースキーパー確認） |
| 事前条件 | 作業員が JWT 認証済みである（FR-AU-001 充足）。タブレットが最新の SOP マスタをローカルに保持している（前回同期完了済みまたはオンライン状態） |
| 事後条件 | work_started イベントが Append-only イベントストアに記録される。case_id が生成されロックステップが最初の未完 Step を指している。ナビゲーション進捗表示が画面に表示されている |

**主シナリオ**
1. 作業員はトップ画面で「作業開始」ボタンをタップする。システムは当該作業員が資格を持つ工程一覧を表示する（FR-NV-001）
2. 作業員は対象工程を選択する。システムは選択工程に属する作業一覧を表示する
3. 作業員は対象作業を選択する。システムは習熟度プロファイル（Novice/Competent/Proficient/Expert）を取得し、表示モード（Linear/Reference）を決定する（FR-NV-004）
4. システムはプレースキーパーに中断済み作業が存在するか確認する。存在しない場合は新規 case_id を生成し work_started イベントを記録する
5. システムは最初の Step を表示する。instruction_text（多言語対応）・media_refs・判定条件（USL/LSL）・tips_refs を提示する（FR-NV-003）
6. システムはナビゲーション進捗バー（Step N/M・完了割合）を表示する（FR-NV-013）
7. 作業員は表示された Step の内容を確認し、作業を開始する

**代替シナリオ**
- A1: 資格期限切れ工程を選択した場合、システムは「資格の有効期限が切れています」警告を表示し選択をブロックする（FR-AU-004）
- A2: スキルレベルが不足する作業を選択した場合、システムは「必要スキルレベルが不足しています」エラーを表示し現場監督への連絡を促す
- A3: 中断済み作業が存在する場合、システムは「中断中の作業があります。再開しますか？」確認ダイアログを表示する（UC-011 へ分岐）
- A4: Novice プロファイルの作業員の場合、システムは手順テキストにふりがなを付与し動画・ピクトグラムをフル表示する（FR-UI-011）
- A5: 改訂直後（改訂後 N 回未満）または最終実施 30 日超の場合、システムは Expert であっても Linear モードへ強制切り替えし切り替え理由を表示する（FR-NV-004）

**例外シナリオ**
- E1: マスタ同期が 24 時間以上未実施の場合、システムは「マスタが古い可能性があります」バナーを表示する。作業継続は可能とし強制停止しない（FR-UI-005）
- E2: 工程一覧取得時にネットワークエラーが発生した場合、システムは最終同期済みのローカルキャッシュから工程一覧を表示しオフラインインジケーターを表示する（FR-SY-004）
- E3: case_id 生成のサーバー呼び出しが失敗した場合、システムはローカルで暫定 case_id を生成し、同期回復後にサーバー側 ID に統合する

**関連要件 ID**
- FR-NV-001, FR-NV-003, FR-NV-004, FR-NV-013, FR-AU-001, FR-AU-004, FR-UI-003, FR-UI-004, FR-UI-005, FR-UI-011

### UC-002: Step を完了する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員がひとつの Step を完了する。ロックステップ制約・次 Step 解放条件・証拠入力強制・クリティカルステップ二重確認を制御する |
| 主アクター | 作業員 |
| 副アクター | システム（ロックステップ制御・イベント記録・次 Step 解放） |
| 事前条件 | UC-001 が完了し case_id が存在している。現在の Step のロックステップが当該 Step を指している |
| 事後条件 | step_completed イベントが Append-only イベントストアに記録される。worker_id・timestamp_device・timestamp_server・sop_version_id の必須 8 属性が設定されている。次 Step が解放される（または作業が完了する） |

**主シナリオ**
1. 作業員は現在の Step の instruction_text・media_refs・判定条件を確認する（FR-NV-003）
2. 作業員は Step の input_type に応じた入力操作を行う（チェック・数値入力・写真撮影・スキャン・サイン）。evidence_required が true の場合は証拠入力を完了しなければ進行できない（FR-NV-005）
3. 作業員は「完了」ボタンをタップする。システムはロックステップ制御を実行し、入力値・タイムスタンプ・作業者 ID の検証を行う（FR-NV-002）
4. システムは step_completed イベントを生成し Append-only でイベントストアに記録する。timestamp_device と timestamp_server の二重タイムスタンプを設定する（FR-EV-001, FR-EV-007）
5. システムはナビゲーション進捗を更新し次の Step を表示する（FR-NV-013）
6. 作業員は次の Step の内容を確認し手順を続行する

**代替シナリオ**
- A1: 深夜帯連続夜勤 3 日目以降でクリティカルフラグ付き Step の場合、システムは 2 秒長押し確認プロンプトを表示する（FR-NV-011）
- A2: Step 完了所要時間が通常の 50% 以下の場合、システムは「操作が速すぎます。確認しましたか？」アラートを表示する（FR-NV-012）
- A3: Reference モードの熟練作業員の場合、詳細手順は折り畳み表示とし、コツコメントを閲覧可能とする（FR-NV-009）
- A4: 最後の Step を完了した場合、システムは「作業完了」ダイアログを表示し work_completed イベントを記録する
- A5: バリデーション警告が発生した場合（数値が USL/LSL 範囲外等）、システムは警告内容を提示し意図的な承認操作（ダブルタップ等）を必須とする（FR-EV-003）

**例外シナリオ**
- E1: step_completed イベントのサーバー記録が失敗した場合、システムはイベントを SQLite ローカルに保存し is_offline: true を設定する。「記録済（未同期）」バッジを表示する（FR-UI-007）
- E2: 既存の completed イベントへの上書き要求（誤操作等）が発生した場合、システムは UPDATE を拒否し「訂正申請フロー」へ誘導する（FR-EV-008）
- E3: ALCOA+ Complete チェックで step_count と完了件数の不一致を検知した場合、システムは作業完了ボタンを無効化し警告を表示する（FR-EV-009）

**関連要件 ID**
- FR-NV-002, FR-NV-003, FR-NV-005, FR-NV-009, FR-NV-011, FR-NV-012, FR-NV-013, FR-EV-001, FR-EV-007, FR-EV-008, FR-EV-009, FR-UI-002, FR-UI-007

### UC-003: 条件分岐 Step を通過する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が条件分岐 Step において step_flow_rule の JSON Logic 条件式を評価し、skip/goto/insert のいずれかのアクションで次の Step を決定して進む |
| 主アクター | 作業員 |
| 副アクター | システム（step_flow_rule 評価・DAG 経路決定・スキップ記録） |
| 事前条件 | 現在の Step に紐づく step_flow_rule が 1 件以上存在している。前 Step の記録値（payload）がイベントストアに存在している |
| 事後条件 | 条件評価結果に基づいて次 Step が決定されている。アクションが skip の場合は step_skipped_by_rule イベントが記録されている。アクションが insert の場合は追加 Step がフローに挿入されている |

**主シナリオ**
1. システムは前 Step の step_completed イベントから payload（記録値）を取得する
2. システムは当該 Step に紐づく step_flow_rule を priority 昇順で取得し、条件式（JSON Logic）を順番に評価する（FR-NV-006）
3. 最初に条件が成立した rule のアクションを実行する
4. アクションが skip の場合、システムは step_skipped_by_rule イベントを Append-only で記録する。記録には rule_id・condition_evaluated（条件式と評価時入力値のスナップショット）・skipped_at を含める
5. アクションが insert の場合、システムは指定 Step をフローに動的に挿入し、挿入済み Step として作業員に提示する
6. アクションが goto の場合、システムはナビゲーションカーソルを指定 Step に移動する
7. 作業員は決定された次の Step の内容を確認し作業を続行する

**代替シナリオ**
- A1: 全ての rule の条件が不成立の場合、システムは通常の順番（次 Step）に進む
- A2: 同一 source_step に複数の rule が存在し複数の条件が成立した場合、システムは priority が最小の rule のみを適用し残りを無視する
- A3: 条件分岐の結果として作業員への確認プロンプトが必要な場合（insert アクション）、システムは「追加手順が発生しました」通知を表示して作業員の確認を待つ

**例外シナリオ**
- E1: step_flow_rule の condition に循環参照が検出された場合（実際には登録時の DAG 検証で防止されるが）、システムは rule を無効化してエラーログを記録し通常フローで進む
- E2: 参照する前 Step の記録値が null の場合（例外的な記録状態）、システムはデフォルトで通常フローを採用し管理者にアラートを送信する

**関連要件 ID**
- FR-NV-006, FR-MA-007

### UC-004: カスタム Step を実行する

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が step_type_definition に登録されたカスタム Step タイプ（color_chart・slider_range・composite_check 等）の入力フォームを実行し、JSON Logic バリデーションを通過して完了する |
| 主アクター | 作業員 |
| 副アクター | システム（step_type_definition 取得・フォームレンダリング・JSON Logic バリデーション実行・イベント記録） |
| 事前条件 | 当該 Step の step_type が step_type_definition に登録された有効なカスタムタイプを参照している。step_type_definition のキャッシュが SQLite ローカルに存在している（Offline-First 保証） |
| 事後条件 | custom_step_completed イベントが Append-only でイベントストアに記録される。step_type_name・value（JSONB）・validation_result・performed_by・performed_at が設定されている |

**主シナリオ**
1. システムは current Step の step_type から step_type_definition（JSON スキーマ）を SQLite キャッシュから取得する（FR-NV-007）
2. システムは input_field の type/min/max/ui_hint に基づいて入力フォームをレンダリングする（number_picker・slider・select 等）
3. システムは label（多言語）をカスタムタイプのタイトルとして表示する
4. 作業員は表示されたフォームに値を入力する
5. システムは入力値に対して validators（JSON Logic）を評価する。全バリデーションが pass の場合は完了操作を受け付ける
6. 作業員は「完了」ボタンをタップする。システムは custom_step_completed イベントを生成し Append-only で記録する
7. ロックステップが次の Step に進む（FR-NV-002 に戻る）

**代替シナリオ**
- A1: バリデーション警告（pass ではなく warn レベル）の場合、システムは警告内容を提示し作業員の意図的な承認操作（ダブルタップ等）を必須として「warn_overridden」で記録する
- A2: ui_hint が許可リスト外の値の場合、システムは当該フィールドを text 入力として縮退表示する（セキュリティ制約の順守）
- A3: 多言語ラベルで表示言語のキーが存在しない場合、システムは日本語（ja）フォールバックで表示する

**例外シナリオ**
- E1: step_type_definition のキャッシュがローカルに存在しない場合（初回同期前等）、システムは text_input タイプへ縮退動作し「カスタムタイプが利用できません」を表示する
- E2: step_type_definition の is_active が false の場合（論理削除済み）、システムは text_input へ縮退動作し管理者にアラートを送信する

**関連要件 ID**
- FR-NV-007, FR-MA-010, FR-NV-002, FR-EV-001

### UC-005: 多言語を切り替える

| 項目 | 内容 |
|---|---|
| 目的 | 作業員が表示言語を変更し、承認済み多言語 SOP の表示に切り替える。承認済みでない言語版 SOP は表示しない |
| 主アクター | 作業員 |
| 副アクター | システム（承認済み言語一覧取得・instruction_text 多言語 JSONB 切り替え・ピクトグラム表示制御） |
| 事前条件 | 作業員が JWT 認証済みである。作業員プロファイルに言語設定が存在する（または設定変更操作を行う） |
| 事後条件 | 選択言語の instruction_text（多言語 JSONB の対応キー）が全ての Step 表示に適用されている。クリティカルフラグ付き Step にピクトグラムと日本語原文が表示されている |

**主シナリオ**
1. 作業員はプロファイル設定画面または SOP 表示中のメニューから「言語切り替え」を選択する
2. システムは当該 SOP に対して承認済み（approved/published 状態）の言語版一覧を表示する（FR-NV-008）
3. 作業員は表示したい言語を選択する
4. システムは instruction_text の多言語 JSONB から選択言語のキー値を取得し全 Step の表示に即座に適用する（FR-UI-001）
5. システムはクリティカルフラグ付き Step に ISO 7010 安全標識ピクトグラムを表示する（FR-UI-008）
6. システムはクリティカルフラグ付き Step に日本語原文（ja キーの値）を併記表示する（FR-UI-010）
7. システムはクリティカルフラグ付き Step の画面に「現場監督を呼ぶ」ボタンを常設する（FR-UI-009）
8. 作業員は選択言語で表示された SOP の手順を確認し作業を継続する

**代替シナリオ**
- A1: 選択言語の SOP が承認済みでない場合、システムは「この言語のSOP は承認済みではありません」を表示し選択をブロックする
- A2: 選択言語のキーが instruction_text JSONB に存在しない Step がある場合、システムは日本語（ja）フォールバックで当該 Step を表示し警告アイコンを付与する
- A3: やさしい日本語（日本語 ja モード）を選択した場合、システムはふりがな付き・平易な表現・短文に変換された手順テキストを表示する（JLPT N5 レベルに対応）

**例外シナリオ**
- E1: 多言語 JSONB のパース処理でエラーが発生した場合、システムは当該 Step を日本語でフォールバック表示しエラーログを記録する
- E2: 動画の字幕データが存在しない言語を選択した場合、システムは字幕なしで動画を再生し「字幕が利用できません」を表示する

**関連要件 ID**
- FR-NV-008, FR-UI-001, FR-UI-008, FR-UI-009, FR-UI-010
