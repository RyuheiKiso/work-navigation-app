# 05 SOP実行業務フロー

本章の責務は、作業員が出社してから終業するまでの標準的な業務フロー（BF-01）を確定することである。ロックステップ進行の業務的根拠・クリティカルステップの証拠取得手続・電子サインの業務手続・オフライン中の業務継続・例外処理の各業務ルールを定義する。

図: fig_sop_execution_flow（img/ 配下）を参照

---

## 1. 代表的業務フロー（着工〜終業）

### 1-1. 通常業務の七段階

作業員の標準的な 1 日の業務フローを以下の七段階で確定する。

**第一段階: ログイン・シフト開始**

作業員はタブレット APP にユーザー ID と PIN で認証する。シフト開始時刻が記録される。前シフトからの引継ぎ票が存在する場合、引継ぎ票の確認完了が「作業開始可能」状態への移行条件となる。引継ぎ票未確認の場合、作業開始画面への遷移がブロックされる。

認証後、スキル有効期限・資格有効期限・長期ブランク（最終実施日から 30 日超）・SOP 改訂直後フラグのチェックが自動実行される。いずれかの条件に該当する場合は警告画面が表示され、作業員は内容を確認してから進む。

**第二段階: 作業指示の受付・段取確認**

現場監督または生産管理から作業指示（品番・ロット番号・工程名・SOP 版・台数）が割り当てられる。作業員はタブレットで自分に割り当てられた作業指示を確認する。

作業指示に紐付く SOP が表示される。SOP の版番号・有効開始日・改訂理由をタブレットで確認できる。作業に必要な設備・計測器・材料の準備状況をステップとして確認する（段取 Step）。

**第三段階: SOP 選択・ロックステップ進行開始**

作業員が対象作業を開始する。タブレット APP は習熟度プロファイル（Novice / Competent / Proficient-Expert / Skill Decay）に応じた表示モード（Linear / Reference）を適用してステップを提示する。

ロックステップ進行が開始する。第一 Step から順に完了操作を行わない限り第二 Step へ進めない。

**第四段階: 各 Step の実行と記録**

作業員は表示された手順に従って作業を実施し、Step の完了を記録する。

- `boolean_check` 型 Step: チェックボックスのタップで完了記録
- `numeric_input` 型 Step: 測定値・単位を入力し判定条件（USL/LSL）と照合
- `photo_capture` 型 Step: カメラで証拠写真を撮影し SHA-256 ハッシュ付きで記録
- `qr_scan` 型 Step: バーコード・QR コードをスキャンしロット・部品と照合
- `signature` 型 Step: 電子サインを押下し作業員 ID・タイムスタンプを付与
- カスタム Step: 工場固有の定義に従い入力・判定

各 Step の完了時に `step_completed` イベントが生成され、worker_id・timestamp_device・timestamp_server・sop_version_id が自動付与される。

**第五段階: 全ステップ完了・作業完了確認**

全ステップの `step_completed` イベントが確認された後、「作業完了」ボタンが有効化される。作業完了時に作業員が電子サインを押下する。電子サインには完了時刻・作業員 ID・完了した SOP 版 ID が記録される。

作業完了の電子サインは Contemporaneous 原則の技術的充足として機能し、後付け入力ではないことをタイムスタンプで証明する。

**第六段階: 作業完了報告・ライン進捗更新**

作業完了が記録されると、現場監督の進捗画面にリアルタイム更新が反映される。ロット番号・完了台数・完了時刻が更新される。

**第七段階: 終業・シフト終了**

終業時刻にシフト終了操作を行う。未完了の作業指示が存在する場合は引継ぎ票（I-PASS 構造）の作成が促される。引継ぎ票作成完了後にシフト終了が確定する。

### 1-2. フローの前提条件

| 前提 | 内容 |
|---|---|
| SOP の公開状態 | 当該作業の SOP が承認・公開済みであること |
| スキル要件の充足 | 作業員のスキル等級が当該 SOP の最低要件（skill_level_required）以上であること |
| 計測器の校正有効期限 | 使用計測器の次回校正日が未来であること（期限超過の場合は計測 Step がブロックされる） |
| タブレットの状態 | 最終同期から 24 時間以内にマスタ同期が完了していること（ステータスバーに表示） |

**本節で確定した方針**
SOP 実行業務を七段階（ログイン・段取確認・SOP 選択・Step 実行記録・作業完了確認・報告・終業）で確定する。
各 Step の完了時に worker_id・timestamp・sop_version_id の三属性が自動付与されることを業務手続として確定する。
引継ぎ票未確認時の作業開始ブロックを業務ルールとして確定する。

## 2. ロックステップ進行条件

### 2-1. ロックステップの業務根拠

ロックステップ（前ステップ完了なしに次ステップへ進めない強制進行）は以下の三つの業務根拠に基づく。

**根拠 A: Post-Completion Error の防止**

認知科学の知見（Memory for Goals モデル）によれば、主目的を達成した後の付随ステップを忘却するエラー（Post-Completion Error）は人間の認知特性として避けられない。ロックステップは完了すべきステップの見逃しを構造的に防止する。

**根拠 B: クリティカルステップの証拠確保**

FMEA-RPN 高値のステップは、証拠（写真・測定値・スキャン）の添付を完了条件とする。ロックステップにより証拠なしの通過が業務上不可能となり、ALCOA+ の Complete 原則を充足する。

**根拠 C: トレサビ記録の連続性保証**

全ステップが順序通りに記録されることで、どの順序で何が実施されたかの証明が可能になる。逆トレサビ照会時に「ステップ B が実施された時点でステップ A は完了済みか」を確認できる。

### 2-2. クリティカルステップの定義

`fmea_rpn_flag = true` の Step をクリティカルステップとして扱う。クリティカルステップは以下の追加条件が課せられる。

- `evidence_required = true` が強制される（写真・測定値・スキャンのいずれかが必須）
- EWI（早期警告指示）が Step 開始前に表示される
- 連続夜勤 3 日目以降・深夜帯（0:00〜5:00）では二重確認プロンプトが追加される
- 30 分超の中断後の再開時に「目視再確認」プロンプトが表示される

### 2-3. 飛び越え禁止の例外条件

以下の条件を充足する場合のみ、ステップのスキップが業務上許容される。

| 例外条件 | 必要な承認 | 記録方法 |
|---|---|---|
| 工程上不適用（対象材料の仕様変更等） | 現場監督の電子サイン | `step_skipped` イベント＋理由テキスト（NULL 禁止）＋承認者 ID |
| 緊急停止・安全優先措置 | 現場監督の事後確認 | `step_skipped` イベント＋緊急理由フラグ＋現場監督の確認電子サイン |
| 品質担当承認が必要なスキップ | 品質担当の電子サイン | `step_skipped` イベント＋品質担当承認 ID |

スキップ理由が空文字の `step_skipped` イベントはシステムが受け付けない。理由の入力は業務手続として必須とする。

**本節で確定した方針**
ロックステップの業務根拠を Post-Completion Error 防止・証拠確保・トレサビ連続性の三点で確定する。
クリティカルステップは evidence_required 強制・EWI 表示・二重確認（夜勤条件）の三機能が必須適用される。
飛び越えは三条件（工程不適用・緊急停止・品質担当承認）でのみ許容し、全スキップは理由記録必須とする。

## 3. クリティカルステップの証拠取得業務

### 3-1. 証拠取得の業務手続

クリティカルステップ（`evidence_required = true`）の証拠取得業務を以下のとおり確定する。

**写真撮影証拠**

1. タブレット APP のカメラ機能を起動する
2. 対象物（製品・治具・測定値画面等）を撮影する
3. 撮影と同時に SHA-256 ハッシュ・RFC 3161 タイムスタンプ・作業員 ID・ステップ ID が自動付与される
4. 撮影完了後、写真サムネイルが表示され確認を求めるダイアログが表示される
5. 「確認」押下で `step_completed` イベントの証拠フィールドに photo_hash・photo_path が設定される

**測定値入力証拠**

1. 計測器を使用して測定を実施する
2. 測定値・単位（UCUM コード）を入力する
3. 入力前に計測器の校正有効期限プレフライトチェックが実行される（期限超過の場合は入力ブロック）
4. USL/LSL との比較判定が即時実行され、範囲内は緑・範囲外は赤で表示される
5. 範囲外の場合は「この値で確定しますか？」の確認ダイアログが表示され、確認後に `step_completed` イベントが生成される
6. `step_completed` イベントの payload に numeric_value・unit_code・uncertainty_u・calibration_ref が設定される

**QR/バーコードスキャン証拠**

1. タブレット APP のスキャン機能を起動する
2. 対象のバーコード・QR コード（GS1 体系・QR・Data Matrix）を読み取る
3. スキャン値と SOP に設定された期待値を照合する
4. 一致の場合は OK・不一致の場合は警告が表示される
5. スキャン値・照合結果が `step_completed` イベントの payload に scan_value として設定される

**本節で確定した方針**
証拠取得手続を写真・測定値・QR スキャンの三種で確定する。
写真には SHA-256 ハッシュ・RFC 3161 タイムスタンプが自動付与される。測定値には校正証明書 ID が自動紐付けられる。
証拠添付なしの `step_completed` イベントをシステムが受け付けないことを業務手続として確定する。

## 4. 電子サインの業務手続

### 4-1. 電子サインが必要な業務イベント

| 業務イベント | 対象者 | 電子サインの業務的意味 |
|---|---|---|
| Step 完了（signature 型 Step） | 作業員 | 当該ステップを自分が実施したことの証明 |
| 作業完了 | 作業員 | 全ステップを完了したことの業務上の宣言 |
| 引継ぎ票の作成（引渡し側） | 作業員（前シフト） | 引継ぎ内容が正確であることの業務確認 |
| 引継ぎ票の受領確認（受取側） | 作業員（次シフト） | 引継ぎ内容を理解し作業を引き継ぐことの業務承諾 |
| アンドン発報の確認応答 | 現場監督 | 発報内容を認識し対応開始することの業務記録 |
| ステップスキップの承認 | 現場監督または品質担当 | スキップを業務上許可したことの記録 |

### 4-2. 電子サインの業務手順

1. 電子サインが必要な操作の前に「本人確認」画面が表示される
2. 作業員は認証情報（PIN または生体認証代替としての再ログイン）を入力する
3. 認証成功後、電子サインイベントが生成される
4. 電子サインには worker_id・timestamp_device・timestamp_server・対象イベント ID が自動付与される
5. 電子サインの失敗（認証エラー）は記録され、3 回失敗した場合は現場監督への通知が発生する

### 4-3. 電子サインの業務上の位置づけ

本システムの電子サインは電子署名法上の認定電子署名ではない。ALCOA+ の Attributable 原則（誰がこの記録を作成したかの帰属）を技術的に充足する業務上の承認記録として機能する。電子署名法上の法的効力が求められる業務には適用しない。

**本節で確定した方針**
電子サインが必要な六種の業務イベントとその業務的意味を確定する。
電子サインは ALCOA+ の Attributable 原則の技術的充足手段として位置づけ、電子署名法上の認定電子署名ではないことを明示する。
認証失敗 3 回での現場監督通知を業務手続として確定する。

## 5. オフライン中の業務継続

### 5-1. オフライン状態の定義

作業員のタブレットがバックエンドサーバーと通信できない状態をオフラインと定義する。原因は Wi-Fi デッドゾーン・サーバー障害・ネットワーク障害のいずれであっても、作業員の業務手続は同一である。

### 5-2. オフライン中の業務手続

オフライン状態においても作業員は以下の業務を通常通り継続する。

- SOP の参照（前回同期時に取得済みのマスタを参照）
- 全ステップの実行・記録（ローカル SQLite に `is_offline = true` フラグ付きで記録）
- 電子サインの押下（ローカル記録に worker_id・timestamp_device を付与）
- アンドン発報（ローカルキューに格納し通信回復後に送信）
- ヒヤリハット・Kaizen 起票（ローカルキューに格納）

オフライン中は以下の業務が制限される。

- 新版 SOP の反映（通信回復後に最新マスタを再同期するまで旧版で継続）
- 他のタブレットからの引継ぎ票の受領（通信回復後に確認が必要）

### 5-3. ステータスバーの表示業務ルール

タブレット APP は全画面共通のステータスバーに「最終同期時刻」を常時表示する。作業員はこの表示で現在の同期状態を常に把握できる。

| 表示内容 | 業務上の意味 |
|---|---|
| 最終同期: HH:MM | X 時 Y 分に同期完了。以後の記録は未同期 |
| オフライン（記録中） | 通信断中。作業は継続可能。記録はローカルに保存中 |
| 同期中... | 通信回復後の同期処理中 |

### 5-4. 再接続後の業務手続

通信回復後、以下の手順で業務を継続する。

1. Outbox Pattern によりローカルキューのイベントが自動的にサーバーへ送信される
2. 「同期完了」通知が表示される
3. 作業員は同期完了を確認して通常業務を継続する
4. 同期完了後、オフライン中に取得できなかった最新マスタが自動更新される

**本節で確定した方針**
オフライン中も全 Step の実行・記録・電子サインが継続可能であることを業務手続として確定する。
オフライン中の記録は `is_offline = true` フラグで識別し、ALCOA+ の Contemporaneous 原則との整合を記録上証明する。
再接続後の Outbox 同期は自動実行とし、作業員の追加操作を不要とする。

## 6. 例外処理

### 6-1. 飛び越え要求の業務手続

現場の判断でステップを飛び越える必要が生じた場合の業務手続を確定する。

1. 作業員が「飛び越え申請」ボタンを押下する
2. 飛び越えの理由をテキストで入力する（必須・空文字禁止）
3. 現場監督に飛び越え申請通知が送信される
4. 現場監督が申請内容を確認し「承認」または「却下」を電子サインで記録する
5. 承認された場合: `step_skipped` イベントが承認者 ID・理由付きで記録される
6. 却下された場合: 作業員にメッセージが送信され、対象ステップを実施するよう促される

### 6-2. 上位承認の業務手続

品質担当の承認が必要なケース（製品安全に関わるクリティカルステップのスキップ・規格外測定値の採用等）の業務手続を確定する。

1. 現場監督が品質担当に承認依頼を行う（管理 Web 通知）
2. 品質担当が内容を確認し「承認」または「却下」を電子サインで記録する
3. 承認された場合: 品質担当承認者 ID・理由が `step_skipped` イベントに記録される
4. 品質担当が不在の場合: 作業は一時停止し、品質担当の代理権限者（経営層等）が対応する

### 6-3. 緊急停止の業務手続

作業員が安全上の問題を発見した場合の緊急停止業務手続を確定する。

1. 作業員がアンドン発報ボタン（安全種別）をワンタップで押下する
2. 発報時刻・作業員 ID・対象ステップ・発報種別（安全）が自動記録される
3. 現場監督にプッシュ通知が送信される
4. 現場監督がライン停止判断を行い、電子サインで確認を記録する
5. 安全確認完了後、現場監督が「再起動許可」の電子サインを行う
6. 作業員が作業を再開する（プレースキーパーにより中断ステップから再開）

**本節で確定した方針**
例外処理として飛び越え要求・上位承認・緊急停止の三種の業務手続を確定する。
いずれの例外も承認者の電子サインと理由テキストを業務上の必須記録として確定する。
緊急停止後の再開はプレースキーパーによる中断ステップからの再開とし、ステップの飛び越えは発生しない。
