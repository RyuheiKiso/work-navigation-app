# 03 業務ドメイン用語集

本章の責務は、本システムで使用するドメイン用語を正規定義し、表記揺れを排除することである。本用語集は業務要件サブ全ファイル・機能要件サブ・概要設計書・詳細設計書・API 仕様書・テストケースにわたって共通参照される語彙基盤である。用語の追加・変更・廃止は本章の改訂として管理する。

---

## 1. 用語集の管理規約

### 1-1. 管理原則

本用語集は以下の原則に従い管理する。

| 原則 | 内容 |
|---|---|
| 一語一義 | 同一の概念に複数の語を充てることを禁止する |
| 一義一語 | 同一の語を複数の意味で使うことを禁止する |
| 表記統一 | 本用語集に登録された正式表記のみを全文書で使用する |
| 和英対照 | 主要用語には英語表記を付記する |
| 追記管理 | 廃止用語は削除せず廃止日・代替語を付記して残す |

### 1-2. 用語追加の手順

用語の追加・変更の手順は以下のとおりである。

1. 品質担当が「用語追加申請」を起票し、正式表記・定義・使用例・廃止する表記揺れを記載する
2. IT 担当・現場監督が確認し、業務上の妥当性を評価する
3. 品質担当が承認し、本用語集に追記する
4. 追記完了後、全下位文書の表記揺れを修正する

**本節で確定した方針**
本用語集は全文書共通の語彙基盤として確定し、一語一義・一義一語を不変条件とする。
用語の追加・変更は品質担当が承認し、追記後に全文書の統一を実施する手順を確定する。

## 2. SOP 系用語

### 2-1. SOP（Standard Operating Procedure）

**日本語正式表記**: 標準作業手順書
**定義**: 特定の作業を一定の品質で再現するために文書化された手順の集合体。本システムでは工程（L1）・作業（L2）・Step（L3）の三階層で構成される電子文書として管理する。SOP は版管理・承認フロー・Effective Date を持つ。

**類義語（禁止）**: 作業標準・作業指示書・作業マニュアル（業務文書としては存在するが本システムのデータモデル上の用語は「SOP」に統一する）

### 2-2. EWI（Early Warning Indicator）

**日本語正式表記**: 早期警告指示
**定義**: 作業員がクリティカルステップを実施する直前に表示される注意喚起メッセージ。FMEA-RPN 値が高いステップ・過去に不適合が発生したステップ・スキル Decay 検知時に表示される。EWI は作業の妨害ではなく支援として機能するよう設計される。

### 2-3. 三階層（工程 L1 / 作業 L2 / Step L3）

**工程（Process / L1）**: SOP の最上位階層。独立した品質管理ポイントを持つ作業の塊。例：溶接工程・検査工程・組立工程。

**作業（Operation / L2）**: 工程の下位階層。一人の作業員が担当する作業単位。開始・終了が明確に定義できる。例：溶接準備・本溶接・後処理。

**Step（L3）**: 作業の下位階層。Navigation の最小実行単位かつ Traceability の最小記録単位。1 Step 1 操作の粒度。合否判定・数値入力・写真撮影等が紐付く。

**省略形態**: 小規模現場では L2（作業）を省略し L1（工程）の直下に Step を配置する 2 階層構成を許容する。

### 2-4. ロックステップ

**定義**: 前のステップを完了しない限り次のステップへ進めない強制進行制御。DO-LIST 型の手順実行方式。スキップ操作は特定条件（上位承認・飛び越え申請）を除いて禁止される。

**類義語（禁止）**: 順次実行・シーケンシャル実行（ロックステップが正式表記）

### 2-5. プレースキーパー（Placekeeping）

**定義**: 作業中断時に、どのステップで中断したかを自動的に記録・保持する機能。再開時に Welcome 画面を表示し、中断状態を次の操作（または次の作業員）に引き渡す機構。中断ログには時刻・ステップ番号・作業員 ID・中断理由種別を含む。

### 2-6. カスタム Step

**定義**: 標準の Step タイプ（boolean_check / numeric_input / photo_capture / text_input / qr_scan / signature）に属さない、工場固有の入力フォーム・バリデーションロジックを持つ Step。JSON スキーマで定義し、ビジュアル DSL エディタで作成する。カスタム Step の定義は `step_type_definition` テーブルに格納される。

### 2-7. 条件分岐 DSL（step_flow_rule）

**定義**: SOP 内で前 Step の記録結果を条件に後続 Step の skip・goto・insert を制御するルール定義。JSON Logic ベースの宣言的記法で定義し、`step_flow_rule` テーブルに格納される。eval による任意コード実行は禁止される。

### 2-8. 電子サイン（電子署名）

**定義**: Step 完了・SOP 承認・CAPA クローズ等の業務操作に対して、作業員 ID・タイムスタンプ・サーバー署名を付与する技術的手続。本システムでの電子サインは電子署名法上の認定電子署名ではなく、ALCOA+ の Attributable 原則を技術的に充足する業務上の承認記録として機能する。

**本節で確定した方針**
SOP 系の主要用語を SOP・EWI・三階層・ロックステップ・プレースキーパー・カスタム Step・条件分岐 DSL・電子サインの 8 語として正規定義する。
類義語（禁止表記）を明示し、全文書での表記統一を強制する。

## 3. トレサビ系用語

### 3-1. 4M+1E

**定義**: 製造品質に影響する要因の分類フレームワーク。Man（人）・Machine（設備）・Material（材料）・Method（方法）の 4M に Environment（環境）と Measurement（計測）を加えた 6 軸として本システムでは運用する。全トレサビイベントはこの 6 軸を参照キーとして持つ。

**用途限定**: 不適合起票時の原因分類に使用する。「Man」を単独で選択させる UI は設けず、6 軸を等価に提示する。

### 3-2. ALCOA+（9 原則）

**定義**: 電子記録の品質を担保するための 9 原則。Attributable（帰属可能）・Legible（読取可能）・Contemporaneous（同時記録）・Original（元データ）・Accurate（正確）・Complete（完全）・Consistent（一貫）・Enduring（耐久）・Available（利用可能）。本システムはこれら 9 原則をデータモデル・UI・運用の三層で実装する。

**準拠範囲**: 本システムは ALCOA+ に準拠する。FDA 21 CFR Part 11 への適合は対象外と判断する（機能スコープと非スコープ 4 節参照）。

### 3-3. XES（eXtensible Event Stream）

**定義**: プロセスマイニングの国際標準フォーマット。Case ID・Activity・Timestamp・Resource の 4 属性を必須とする。本システムのイベントストアは XES 互換の属性構造を持ち、XES 形式でのエクスポートを提供する。

### 3-4. イベントソーシング（Event Sourcing）

**定義**: 全状態変化を「不変イベント」として記録し、現在状態をイベントの累積から導出するアーキテクチャパターン。本システムは Append-only Event Sourcing を採用し、作業ログテーブルへの UPDATE・DELETE を禁止する。

### 3-5. Append-only

**定義**: データストアに対して INSERT のみを許可し、UPDATE・DELETE を禁止するデータ整合性保証原則。本システムでは PostgreSQL のロール権限として強制する。Append-only は ALCOA+ の Original 原則の技術的実装手段である。

### 3-6. マスタバージョン時点参照（時点固定）

**定義**: 作業ログが参照したマスタ（SOP 版・校正証明書・スキルスナップショット）の版を、後からマスタが変更されても変わらない形で記録する仕組み。作業ログは `sop_version_id`・`calibration_ref`・`skill_level_snapshot` を記録時点の値で固定する。マスタ改訂が過去の実績記録に遡及することを構造的に防止する。

**本節で確定した方針**
トレサビ系の主要用語を 4M+1E・ALCOA+・XES・イベントソーシング・Append-only・マスタバージョン時点参照の 6 語として正規定義する。
ALCOA+ は準拠対象として確定し、FDA 21 CFR Part 11 は対象外と明示する。

## 4. 品質系用語

### 4-1. 不適合（Nonconformance）

**定義**: 製品・工程・手順が定められた要求事項を満たさない状態の事象。本システムでは作業員・現場監督・品質担当がタブレット APP または管理 Web から起票する。起票後は CAPA への昇格可否を品質担当が判断する。

**類義語（禁止）**: 不良・欠陥・品質問題（不適合が正式表記）

### 4-2. CAPA（Corrective and Preventive Action）

**定義**: 不適合の根本原因を分析し、再発防止（是正処置）および予防措置を実施する品質管理手続。ISO 9001:2015 §10.2 に準拠する。本システムでは起票→原因分析→対策→効果確認→クローズの 5 フェーズで管理する。

### 4-3. FMEA-RPN（Failure Mode and Effects Analysis - Risk Priority Number）

**定義**: 潜在的な故障モードの重大性（Severity）・発生確率（Occurrence）・検出可能性（Detection）を数値化した指標（RPN = S × O × D）。本システムでは FMEA の結果（RPN 値）をマスタとして取り込み、RPN が設定閾値を超えるステップを `fmea_rpn_flag = true` とし、強化チェック（証拠付き完了の強制・EWI 表示）を発動する。

### 4-4. ヒヤリハット

**定義**: 実際には事故・不適合に至らなかったが、状況によっては至りえた未遂事象・危険発見。本システムでは作業員が起票する安全改善インプットとして機能する。Just Culture の原則に基づき、ヒヤリハット件数の増加を「安全文化の改善指標」として評価する。

### 4-5. Just Culture

**定義**: James Reason が提唱する組織安全文化の概念。スリップ・ラプス（意図なき行為失敗）・ミステイク（誤判断）は非難せずシステム改善問題として扱い、故意の規則違反・悪意ある行為のみを問責する文化。本システムの組織運用原則として採用する。

### 4-6. アンドン（Andon）

**定義**: 生産ラインで品質・設備・安全上の問題が発生した際に作業員がワンタップで発報する異常通知機能。発報時刻・作業員 ID・発報種別・対象ステップを自動記録する。発報後、現場監督・品質担当・保全担当へのエスカレーション経路が確定する。

**語源**: トヨタ生産方式（TPS）の自働化概念に由来するが、本システムでは作業員による手動発報のみを実装し、IoT センサとの自動連動は対象外と判断する。

### 4-7. Kaizen（改善提案 / Kaizen Teian）

**定義**: 作業員が業務改善アイデアを起票し、現場監督・品質担当が評価・採否を決定・フォローする継続的改善活動。本システムでは起票・スクリーニング・評価・採否決定・SOP 改訂連動のフローを管理する。採用された提案には「採用通知」を起票者に返信することを業務手続として定める。

### 4-8. KY 活動（危険予知活動）

**定義**: 作業前に潜在する危険要因を洗い出し対策を合意するリスクアセスメント活動。本システムのヒヤリハット機能と連動するが、KY シート自体の電子化は ver1.0.0 の対象外と判断する。

**本節で確定した方針**
品質系の主要用語を不適合・CAPA・FMEA-RPN・ヒヤリハット・Just Culture・アンドン・Kaizen・KY 活動の 8 語として正規定義する。
不適合の類義語（不良・欠陥）は全文書で使用禁止とし、「不適合」に統一する。

## 5. 連携系用語

### 5-1. Outbox Pattern

**定義**: オフライン記録をローカルキュー（Outbox）に保存し、通信回復後にサーバーへ順次送信するアーキテクチャパターン。タブレット APP の SQLite に記録した作業イベントを通信回復後にバックエンドへ同期する仕組みとして本システムで採用する。Idempotency Key（event_id）により重複送信時のデータ二重登録を防止する。

### 5-2. Idempotency（冪等性）

**定義**: 同一のリクエストを複数回実行しても結果が変わらない性質。本システムでは `event_id`（UUID）を Idempotency Key として使用し、オフライン同期時の重複送信が発生しても二重登録が起きないことを保証する。

### 5-3. GS1

**定義**: 国際的なバーコード・データキャリア標準規格体系。EAN-13・UPC-A・ITF-14・GS1-128・GS1 DataMatrix 等を含む。本システムはこれらの GS1 標準バーコードを ML Kit Barcode Scanning で読み取り、ロット・部品・設備との照合に使用する。

### 5-4. 外部一意キー（external_key_binding）

**定義**: 親機（基幹システム: ERP / MES / 生産管理パッケージ）が使用する型式・機番・品目コード等の一意識別属性の組み合わせ。本システムの子機モードでは `external_key_binding` テーブルを通じて親機の外部一意キーと内部 SOP・作業パターンを紐付ける。

### 5-5. 子機モード

**定義**: 本システムが基幹システム（親機）と論理連携して動作するモード。READ-ONLY 同期（親機→子機マスタ受信）と実績送信（子機→親機）の二方向を持つ。単独動作モードとの切替は設定ファイルで制御する。IoT センサ・PLC への直接接続は対象外と判断する。

### 5-6. 親機

**定義**: 本システムの子機モードにおける連携対象の基幹システム。ERP・MES・生産管理パッケージ等を指す。親機の存在は子機モードの前提であり、単独動作モードでは親機は不要である。

**本節で確定した方針**
連携系の主要用語を Outbox Pattern・Idempotency・GS1・外部一意キー・子機モード・親機の 6 語として正規定義する。
子機モードは親機との論理連携として確定し、IoT センサ・PLC への直接接続は対象外と明示する。

## 6. 別名・同義語管理

### 6-1. 表記揺れ禁止リスト

以下に列挙する表記揺れは全文書で使用を禁止する。正式表記のみを使用する。

| 正式表記 | 禁止する表記揺れ |
|---|---|
| SOP | 作業標準書・作業指示書・作業マニュアル・手順書 |
| 不適合 | 不良・欠陥・品質問題・NG |
| アンドン | 異常通知・警告ボタン・助け呼び |
| プレースキーパー | 中断保存・再開機能・ブックマーク |
| Kaizen | 改善提案・提案制度 |
| CAPA | 是正予防処置・改善処置・品質是正 |
| Step | ステップ・工程・操作（本用語集の定義に従い使い分ける） |
| 電子サイン | 電子署名（法的な電子署名との混同を防ぐため、本文中では電子サインを正式表記とする） |
| 作業員 | 作業者・オペレーター・従業員 |
| 現場監督 | 班長・ラインリーダー・ライン監督 |
| 品質担当 | 品質管理者・QC 担当・品証担当 |
| IT 担当 | 情報システム担当・SE・システム管理者 |
| タイムスタンプ | 時刻・記録時刻・入力時刻 |
| 工程 | プロセス（英語表記は Process として許容） |

### 6-2. 英語正式表記

API 仕様書・コメント・変数名では以下の英語表記を正式とする。

| 日本語正式表記 | 英語正式表記 |
|---|---|
| 作業員 | worker |
| 現場監督 | supervisor |
| 品質担当 | quality_manager |
| 保全担当 | maintenance_staff |
| IT 担当 | it_admin |
| 工程 | process |
| 作業 | operation |
| ステップ（Step） | step |
| 不適合 | nonconformance |
| 是正予防処置 | capa |
| アンドン | andon |
| 改善提案 | kaizen_teian |
| ヒヤリハット | near_miss |
| 電子サイン | electronic_sign |
| 引継ぎ票 | handover_note |

**本節で確定した方針**
表記揺れ禁止リストを 14 項目確定し、全文書での統一を強制する。
英語正式表記リストを API 仕様書・コード・変数名での統一基盤として確定する。
新たな表記揺れを発見した場合は本節に追記する管理ルールを確定する。
