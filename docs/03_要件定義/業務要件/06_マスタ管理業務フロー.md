# 06 マスタ管理業務フロー

本章の責務は、SOP・ユーザー・設備・計測器マスタの作成・改訂・承認・公開・廃止にかかるマスタ管理業務フロー（BF-02）を確定することである。Draft First 原則の業務的意義・改訂提案から公開までの六段階フロー・参照整合性ガードの業務手続・バージョン凍結の業務的意味・緊急改訂手順を定義する。

## 1. Draft First 原則の業務的意味

### 1-1. 原則の定義

Draft First 原則とは、全マスタの編集操作をドラフト状態として保存し、明示的な「公開」操作（品質担当の電子サインによる承認フロー）を経て初めてタブレット APP に反映する業務原則である。「保存 = 即時本番反映」の運用を業務として禁止する。

### 1-2. 業務リスクの回避

Draft First 原則が排除する業務上のリスクを以下に確定する。

**リスク A: 編集中 SOP の誤配信**

品質担当が SOP を編集途中で意図せず保存した場合、次のシフトからその未完成 SOP が作業員に配信される。これにより誤った手順での作業が発生し、不適合・安全事故の原因となる。Draft First により、意図的な「公開」操作なしには編集中 SOP が配信されない。

**リスク B: 承認なしの手順変更**

手順の変更には品質担当の電子サインによる承認が業務上必須である。Draft First により、承認フローを経ない手順変更をシステムが技術的に禁止する。ISO 9001:2015 §7.5 の文書化情報管理要件に対応する。

**リスク C: 進行中作業への遡及影響**

公開中の SOP を即時変更すると、現在進行中の作業指示に即時反映される。これにより「作業員が参照していた手順と実際の記録が食い違う」状態が発生する。Draft First により、進行中の作業指示は旧版で完了し、新版は次の作業指示から適用される。

### 1-3. Draft 状態の業務上の取り扱い

| SOP の状態 | タブレット APP への表示 | 業務上の意味 |
|---|---|---|
| `draft` | 表示しない | 編集中。品質担当のみ管理 Web で参照可 |
| `under_review` | 表示しない | レビュー依頼済み。差分確認可 |
| `approved` | 表示しない | 承認済み。Effective Date 到達待ち |
| `published` | 表示する（最新版） | 有効化済み。作業員が参照可 |
| `obsolete` | 表示しない（過去記録からは参照可） | 廃止済み。Append-only で保持 |

**本節で確定した方針**
Draft First 原則を業務的必須原則として確定し、「保存 = 即時本番反映」を業務として禁止する。
誤配信・承認なし変更・進行中作業への遡及の三リスクを Draft First が回避することを確定する。
SOP の五状態（draft / under_review / approved / published / obsolete）の業務的意味を確定する。

## 2. 改訂提案〜Publish 承認の業務フロー

### 2-1. 六段階フローの定義

SOP 改訂の業務フローを以下の六段階で確定する。この六段階は ISO 9001:2015 §7.5 に対応し、すべての段階がシステムの電子記録として追跡可能である。

**第一段階: 改訂提案（申請受付）**

以下のいずれかの起点からSOP改訂提案が発生する。

- 作業員によるKaizen Teian（改善提案）がSOP改訂を提案する
- CAPA レコードの対策として「SOP 改訂」が指定される
- 4M 変更（Man・Machine・Material・Method の変更）申請が起票される
- 現場監督・品質担当が直接改訂を起案する

改訂提案には提案者 ID・提案理由・影響する工程・緊急度が記録される。品質担当に自動通知が送信される。

**第二段階: 影響分析**

品質担当は改訂提案の影響範囲を分析する。

- 改訂対象の SOP を参照する作業指示の件数
- 改訂に伴い再教育が必要な作業員の範囲
- 改訂後の SOP と連動するカスタム Step・条件分岐ルールへの影響
- 参照整合性ガードで影響を受けるマスタの一覧

影響分析の結果が改訂提案レコードに記録される。

**第三段階: ドラフト編集**

品質担当が管理 Web の SOP エディタで改訂内容を編集する。Draft First により編集内容はドラフト状態で保存され、タブレット APP には反映されない。

- Auto-Save（30 秒間隔）により編集内容の喪失を防止する
- Diff Always Visible により公開版との差分がリアルタイム表示される
- Undo/Redo（50 ステップ）により誤操作を即時復元できる

編集完了後、品質担当が「レビュー依頼」操作を行い、状態が `under_review` に遷移する。

**第四段階: レビュー**

レビュー担当者（現場監督または指定レビュアー）がドラフト内容を確認する。

- 現場監督は改訂内容が現場での実施可能性・安全性を充足するかを確認する
- レビュー完了後、承認依頼コメントを付与して「承認依頼」操作を行う
- 修正が必要な場合は「差し戻し」操作でドラフト状態に戻す

**第五段階: 承認（品質担当の電子サイン）**

品質担当が最終確認を行い、電子サインで承認する。

- 承認時に Effective Date（有効開始日）・適用ロット番号（任意）を設定する
- 電子サインには承認者 ID・承認時刻・承認理由のテキストが記録される
- 承認操作により SOP の状態が `approved` に遷移する

**第六段階: 公開（Publish）**

Effective Date 当日の 00:00 に自動で `published` 状態に遷移する（または品質担当が手動で「公開」操作を実行する）。

- 公開と同時に旧版 SOP の状態が `obsolete` に遷移する（旧版は物理削除しない）
- 影響する作業員グループへの「SOP 改訂通知」が送信される
- 作業員は新版 SOP の最初の実施時に、改訂理由・変更点のサマリを確認する義務が課せられる

### 2-2. 教育完了を公開条件にする場合の手続

外国人労働者を含む作業員への多言語 SOP が存在する場合、多言語版の母語話者レビュー完了を公開の追加条件として設定できる。品質担当が「多言語レビュー必須」フラグを付与した場合、各言語版の承認者電子サインが揃わない限り公開操作が完了しない。

**本節で確定した方針**
SOP 改訂フローを六段階（改訂提案・影響分析・ドラフト編集・レビュー・承認・公開）で確定する。
六段階すべての業務操作が電子記録として追跡可能であることを業務要件として確定する。
多言語版の母語話者レビュー完了を公開の追加条件として設定できる業務手続を確定する。

## 3. 参照整合性ガードの業務手続

### 3-1. 参照整合性ガードの業務的意義

マスタを削除・廃止する際に、そのマスタを参照している SOP・作業パターン・カスタム Step・条件分岐ルールが存在する場合、無条件の削除は業務上の重大なリスクを生む。

- 参照中の設備マスタを削除すると、保全作業記録の参照先が消え監査証跡が破損する
- 参照中のカスタム Step タイプを削除すると、そのタイプを使用する SOP が正常に機能しなくなる
- 参照中のユーザーマスタを削除すると、そのユーザーの電子サインが付いた過去記録の帰属が不明になる

### 3-2. 業務手続の確定

参照中のマスタを削除・廃止しようとした場合の業務手続を以下のとおり確定する。

1. 品質担当・IT 担当が削除・廃止操作を開始する
2. システムが「削除前 dry-run」を自動実行する（実際のデータは変更しない）
3. dry-run 結果として「影響を受けるレコード数: N 件」が表示される
4. **影響ゼロの場合**: 削除・廃止を即時実行できる
5. **影響が存在する場合**: 影響範囲一覧（SOP 文書番号・作業パターン名・カスタム Step 名等）が表示される
6. 品質担当は影響を受けるレコードを個別に確認し、参照先の変更または削除を実施する
7. すべての参照が解除されたことを確認した後、削除・廃止を実行する
8. 削除・廃止操作と理由が `master_edit_event` テーブルに記録される

### 3-3. 廃止マスタの取り扱い業務規則

廃止されたマスタは物理削除しない。以下の業務規則を確定する。

- 廃止マスタは `is_active = false` フラグで管理し、新規参照は禁止する
- 廃止マスタを参照する過去の作業ログ・不適合レコードは廃止後も正常に存在し続ける
- 廃止マスタへの参照を持つトランザクションを照会する場合、「参照先が廃止されています」の注記が表示される
- 廃止マスタの物理削除は永続禁止とし、ストレージコストの理由による削除も許容しない

**本節で確定した方針**
参照中マスタの削除前 dry-run を業務手続の必須ステップとして確定する。
影響が存在する場合は参照解除を先行させる二段階手続を確定する。
廃止マスタの物理削除を永続禁止とし、廃止後も過去トランザクションからの参照を保証する。

## 4. バージョン凍結と時点参照の業務的意味

### 4-1. バージョン凍結の定義

承認・公開された SOP 版は「凍結（Frozen）」状態となり、内容を変更できない。凍結の業務的意味は以下のとおりである。

**過去記録の再現性保証**

ロット A の工程 X で発生した不適合を 1 年後に調査する場合、当時の作業員が参照した SOP の内容を正確に再現できなければならない。SOP が頻繁に改訂される環境でも、各作業ログが `sop_version_id` で記録時点の版を時点固定しているため、「その時点の手順」を正確に再現できる。

**規制監査への対応**

顧客監査・内部監査において「この製品ロットはいつ・どの手順で製造されたか」を証明できることが求められる。凍結された SOP 版と時点参照により、製造時の手順を証拠として提示できる。

### 4-2. 時点参照の業務的運用

作業ログが参照する `sop_version_id` は作業指示が発行された時点の有効版 ID に固定される。以下の業務ルールを確定する。

- 新版 SOP が公開されても、進行中の作業指示は旧版で完了させる
- 新版は次の作業指示から適用される（Effective Date またはロット切替設定による）
- 過去の作業ログが参照する旧版 SOP を管理 Web から照会できる
- 旧版 SOP は `obsolete` 状態であっても照会・参照は可能（表示のみ。編集・再公開は禁止）

### 4-3. バージョン番号管理の業務規則

SOP の版番号は `{文書番号}_{メジャー}.{マイナー}_{Effective Date}` 形式で管理する。

| 改訂種別 | バージョン変更 | 使用条件 |
|---|---|---|
| メジャー改訂 | {N+1}.0 | 手順の構造変更・作業順序の変更・クリティカルステップの追加・削除 |
| マイナー改訂 | {M}.{N+1} | 誤字脱字の修正・文言の明確化・写真の差し替え（手順の実質変更なし） |

メジャー改訂後の初回実施は、熟練者（Expert プロファイル）であっても Linear モードへの強制切り替えが適用される。

**本節で確定した方針**
承認・公開された SOP 版の変更を業務として禁止し、改訂は新版作成のみで行うことを確定する。
時点参照により過去の作業記録が常に当時の SOP 版を参照できることを業務要件として確定する。
メジャー改訂後の熟練者への Linear モード強制切り替えを業務ルールとして確定する。

## 5. 緊急改訂・ロールバック

### 5-1. 緊急改訂の業務手順

通常の六段階改訂フローを待てない緊急事態（安全上の重大な誤りの発見・顧客からの緊急変更要求等）が発生した場合の業務手順を確定する。

1. 品質担当が「緊急改訂」フラグを付与して改訂起案する
2. 通常のレビュー段階を省略し、品質担当が単独で電子サインによる承認を行う（単独承認は記録上明示される）
3. Effective Date をその日の即時または翌日朝に設定し、公開する
4. 緊急改訂の理由・省略したプロセス・承認者の記録が `master_edit_event` に保存される
5. 緊急改訂から 5 営業日以内に、通常の改訂フローによる正式版改訂を実施する
6. 緊急改訂版は正式版が公開されるまでの暫定版として位置付け、暫定である旨をタブレット APP に表示する

### 5-2. ロールバックの業務手順

新版 SOP に重大な誤りが発見された場合、直前の有効版（旧版）へのロールバックを実施する業務手順を確定する。

1. 品質担当が「ロールバック」操作を実行する
2. ロールバックの理由をテキストで入力する（必須）
3. 旧版 SOP の状態が `obsolete` から `published` に復帰する
4. 現在の問題版 SOP の状態が `obsolete` に遷移する
5. 作業員に「SOP が変更されました」の通知が送信される
6. ロールバック操作と理由が `master_edit_event` に記録される
7. ロールバック後 5 営業日以内に、問題を修正した新版を正式なフローで公開する

**本節で確定した方針**
緊急改訂は単独承認・暫定フラグ付きで実施し、5 営業日以内の正式改訂を義務付けることを確定する。
ロールバックは旧版の published 復帰と現在版の obsolete 遷移により実施することを確定する。
いずれの例外操作も master_edit_event への記録を必須とし、全変更の追跡可能性を担保する。
