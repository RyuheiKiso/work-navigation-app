# 04 To-Be業務フロー総覧

本章の責務は、本システム導入後の To-Be 業務フローを六区分に分類し、各フローの相互依存関係と記法規約を確定することである。各フローの詳細は 05〜09 の個別ファイルに委ねる。本章は詳細フローを参照する際の全体地図として機能する。

---

## 1. 業務フローの分類体系

### 1-1. 六区分の定義

本システムが関与する業務フローを以下の六区分に確定する。

| フロー ID | 業務フロー名 | 概要 | 詳細ファイル |
|---|---|---|---|
| BF-01 | SOP 実行業務フロー | 出社から終業までの通常作業の流れ。ロックステップ進行・証拠記録・電子サインを含む | 05_SOP実行業務フロー.md |
| BF-02 | マスタ管理業務フロー | SOP・ユーザー・設備・計測器マスタの作成・改訂・承認・公開・廃止の流れ | 06_マスタ管理業務フロー.md |
| BF-03 | アンドン・不適合対応業務フロー | 異常発報から停止・原因究明・CAPA 起動・再発防止・水平展開の流れ | 07_アンドン・不適合対応業務フロー.md |
| BF-04 | Kaizen・ヒヤリハット起票業務フロー | 改善提案・危険発見の起票から採否決定・SOP 改訂連動までの流れ | 08_Kaizen・ヒヤリハット起票業務フロー.md |
| BF-05 | 夜勤交代業務フロー | シフト交代時の引継ぎ業務・プレースキーパーによる中断引継ぎの流れ | 09_夜勤交代・実習生教育業務フロー.md |
| BF-06 | 実習生教育業務フロー | 多言語・読み書き支援・スキル等級管理による実習生教育・習熟度評価の流れ | 09_夜勤交代・実習生教育業務フロー.md |
| BF-07 | 入荷検査（IQC）業務フロー | 材料・部品の入荷から AQL 抜取検査・合否判定・後工程解放・仕入先実績集計までの流れ | 12_入荷検査業務フロー.md |
| BF-08 | リワーク・修正作業業務フロー | 不適合現品の発見からディスポジション判定・リワーク実施・再検査・出荷可否確定までの流れ | 13_リワーク・修正作業業務フロー.md |

### 1-2. 各フローのトリガーとゴール

| フロー ID | トリガー（開始条件） | ゴール（終了条件） |
|---|---|---|
| BF-01 | 作業員がタブレットにログインし、SOP を選択する | 全ステップの電子記録が完了し、電子サインで終業報告が確定する |
| BF-02 | SOP 改訂申請・4M 変更申請・CAPA による SOP 見直し要求が起票される | 新版 SOP が承認・公開され、対象作業員への通知が完了する |
| BF-03 | 作業員がアンドン発報を行うか、不適合が発見される | CAPA がクローズされ、水平展開が完了し、SOP 改訂または対策実施が確認される |
| BF-04 | 作業員が Kaizen 提案・ヒヤリハットを起票する | 採否決定通知が起票者に送達され、採用時は SOP 改訂または対策実施が確認される |
| BF-05 | 作業中のシフト交代時刻が到来する | 次シフト作業員が引継ぎ票を確認・電子サインし、作業を継続できる状態になる |
| BF-06 | 実習生が新たにラインに配属される | スキル等級認定が記録され、対象工程への作業開始が承認される |
| BF-07 | 入荷材料・部品の受入ロットが登録される | 合否判定が記録され、合格ロットが後工程での使用解放状態になる |
| BF-08 | 不適合現品が確認され、ディスポジション判定が要求される | リワーク完了・再検査合格（または廃却・返却）が記録され、出荷可否が確定する |

**本節で確定した方針**
業務フローを BF-01〜BF-08 の八区分として確定し、各フローのトリガーとゴールを明示する（2026-05-18: BF-07 入荷検査・BF-08 リワーク・修正作業を追加し六区分→八区分へ拡張）。
八区分の境界は「業務イベント」単位で設定する。同一のシステム機能を使う場合でも業務の文脈が異なれば別フローとして扱う。

## 2. 六業務フローの相互依存関係

### 2-1. フロー間の依存関係

各業務フローは独立して進行するが、以下の依存関係を持つ。

**BF-01（SOP 実行）← BF-02（マスタ管理）**

BF-01 は BF-02 が公開した最新版 SOP を参照して進行する。BF-02 で新版 SOP が公開された場合、BF-01 の次の作業指示から新版が適用される。進行中の作業指示は旧版で完了させる。

**BF-01（SOP 実行）→ BF-03（アンドン・不適合対応）**

BF-01 の実行中に異常を検知した場合、BF-03 が起動する。BF-01 は BF-03 の CAPA クローズを待たずに継続可能だが、クリティカルな不適合が確認された場合は BF-01 の新規開始をブロックできる（品質担当の権限による）。

**BF-01（SOP 実行）→ BF-04（Kaizen・ヒヤリハット起票）**

BF-01 の実行中に任意のタイミングで BF-04 を並行起動できる。BF-04 の完了は BF-01 の完了を妨げない。

**BF-03（不適合対応）→ BF-02（マスタ管理）**

BF-03 の CAPA 対策として SOP 改訂が必要な場合、BF-02 が起動される。BF-03 の CAPA レコードに BF-02 で作成された SOP 改訂版 ID が紐付けられる。

**BF-04（Kaizen）→ BF-02（マスタ管理）**

BF-04 の Kaizen 提案が採用された場合、SOP 改訂が必要な場合は BF-02 が起動される。

**BF-05（夜勤交代）← BF-01（SOP 実行）**

BF-05 は BF-01 が進行中の状態でトリガーされる。プレースキーパーが BF-01 の中断状態を BF-05 の引継ぎ票に引き渡す。

**BF-06（実習生教育）← BF-02（マスタ管理）**

BF-06 は BF-02 が管理するスキルマスタ・資格マスタを参照してスキル等級認定を行う。

**BF-07（入荷検査）→ BF-01（SOP 実行）**

BF-07 が合格判定したロット（`lots.qc_status = PASSED`）のみ BF-01 の材料ロット QR スキャン照合が通過できる。未合格ロットは API ハードゲート（ERR-BIZ-016）でブロックされる。特採（CONCESSION）ロットは通過を許可するが黄色警告バナーを強制表示する。

**BF-07（入荷検査）← BF-02（マスタ管理）**

BF-07 は BF-02 が管理する材料マスタ・仕入先マスタ・サンプリング計画マスタを参照して検査指示を生成する。

**BF-08（リワーク）← BF-03（アンドン・不適合対応）**

BF-08 は BF-03 の不適合起票（nonconformities）をトリガーとして開始する。ディスポジション判定後、rework_type が REWORK / TOUCH_UP / SORTING の場合に BF-08 が起動する。

**BF-08（リワーク）→ BF-03（アンドン・不適合対応）**

BF-08 で廃却・修正不可と判定された場合は BF-03 の CAPA が継続される。リワーク件数・廃却率は CAPA 記録（TBL-014 `capas.rework_summary`）に集計され、水平展開判断の材料となる。

**BF-08（リワーク）← BF-02（マスタ管理）**

BF-08 はリワーク SOP（sop_type = REWORK / INSPECTION / SCRAP_RECORD）を BF-02 で管理するマスタから参照する。

### 2-2. フロー全体概念図

図: fig_business_flow_overview（img/ 配下）を参照

**本節で確定した方針**
六フロー間の依存関係を本節の記述で確定する。フロー間の接続点は「マスタ参照」「イベント起動」「引継ぎデータ連携」の三種に分類する。
BF-01 が中心フローであり、他の五フローは BF-01 の実行品質を向上させるための補完フローとして設計する。

## 3. フロー記法規約

### 3-1. アクターの表現規則

業務フロー図（05〜09 各ファイル）では以下のアクターを使用する。アクターは泳ぎレーン（Swim Lane）として表現する。

| アクター記号 | 表す存在 |
|---|---|
| 作業員 | SH-01（作業員） |
| 現場監督 | SH-02（現場監督） |
| 品質担当 | SH-03（品質担当） |
| 保全担当 | SH-04（保全担当） |
| IT 担当 | SH-05（IT 担当） |
| タブレット APP | システムコンポーネント（タブレット側） |
| 管理 Web | システムコンポーネント（Web 側） |
| バックエンド | システムコンポーネント（サーバー側） |

### 3-2. データストアの表現規則

| データストア記号 | 意味 |
|---|---|
| ローカル DB（SQLite） | タブレット内のオフラインストレージ |
| サーバー DB（PostgreSQL） | バックエンドの永続ストレージ |
| Outbox キュー | 未同期レコードの一時キュー |
| SOP マスタ | SOP・版管理・Step の参照データ |

### 3-3. 判断・イベントの表現規則

| 記号種 | 意味 |
|---|---|
| 菱形（判断） | 業務上の分岐条件（Yes/No・条件成立/不成立） |
| 円（開始イベント） | フローの開始トリガー |
| 二重円（終了イベント） | フローのゴール完了 |
| 太字矢印（必須遷移） | 省略・スキップ不可の遷移 |
| 点線矢印（任意遷移） | 条件付きまたは任意で発生する遷移 |

### 3-4. フロー図ファイル管理

各業務フロー図は `img/` ディレクトリ配下に以下の形式で格納する。draw.io ファイルと SVG ファイルをペアで管理する。

| フロー ID | ファイル名（拡張子なし） |
|---|---|
| BF-00 全体概念図 | fig_business_flow_overview |
| BF-01 | fig_sop_execution_flow |
| BF-02 | fig_master_governance_flow |
| BF-03 | fig_andon_escalation_flow |
| BF-04 | fig_kaizen_intake_flow |
| BF-05/06 | fig_shift_handover_flow |
| BF-07 | fig_iqc_business_flow |
| BF-08 | fig_rework_business_flow |

**本節で確定した方針**
業務フロー図の記法規約をアクター・データストア・判断/イベントの三種で確定する。
全フロー図は `img/` ディレクトリに draw.io + SVG のペアで格納し、アスキー図・PNG 単独配置を禁止する。
BF-07（IQC）の図ファイルは `fig_iqc_business_flow.{drawio,svg}`、BF-08（リワーク）は `fig_rework_business_flow.{drawio,svg}` として管理する。

## 4. 例外パスの取扱い規則

### 4-1. 例外パスの定義

例外パスとは、通常フロー（ハッピーパス）から外れる経路を指す。業務フロー図では以下の二種類の例外パスを扱う。

**種類 A: 業務上の例外（Exceptional Business Path）**

業務として想定された非標準の経路。例：アンドン発報によるライン停止・飛び越え申請による上位承認・緊急改訂による通常承認フローのバイパス。これらはフロー図に点線矢印で明示する。

**種類 B: システム障害時の縮退パス（Fallback Path）**

Wi-Fi 断・サーバー障害・電源断等のシステム要因による縮退動作経路。タブレット APP のオフライン動作・Emergency Mode・紙フォールバックの三段階を含む。

### 4-2. 例外パスの処理原則

| 原則 | 内容 |
|---|---|
| 例外も記録する | 例外パスを通過した場合も、通過理由・承認者・時刻を記録する |
| 例外は例外として識別できる | 例外パス経由の記録には is_exception フラグまたは理由テキストを必須付与する |
| 例外が常態化しない | 例外パスの使用頻度を KPI として計測し、頻度が高い場合は正常フロー側の改善を行う |
| 縮退でも記録は継続する | システム障害時も作業記録は継続し、復旧後に同期する |

### 4-3. 例外パスの除外対象

以下は業務要件サブでは扱わない（詳細は機能要件・非機能要件サブに委ねる）。

- サーバー側のエラーハンドリング詳細
- API タイムアウトの再試行ロジック
- データベース整合性エラーの復旧処理
- ネットワーク障害の自動検知仕様

**本節で確定した方針**
例外パスを業務上の例外と縮退パスの二種に分類し、取扱い規則を四原則で確定する。
例外パスも記録対象であり、例外記録は通常記録と区別して追跡できる状態を維持する。
例外パスの詳細実装は機能要件サブに委ね、業務要件サブでは経路の業務的意味のみを確定する。
