# 10 業務シナリオ集

本章の責務は、業務要件の達成を検証するための代表的業務シナリオを確定することである。各シナリオは受入テストの前段階定義として機能し、業務フロー（05〜09）・ステークホルダー定義書（02）・ビジネス KPI（01）との整合を確認するための業務視点の記述である。

---

## 1. シナリオ記述形式

### 1-1. 記述フォーマット

本章の各シナリオは以下のフォーマットで記述する。

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-NNN（連番） |
| シナリオ名 | 業務の概要を表す名称 |
| 目的 | このシナリオが検証する業務要件 |
| 主アクター | 主に操作を行うステークホルダー |
| 副アクター | 関与するステークホルダー |
| 前提条件 | シナリオ開始前に成立している必要がある状態 |
| 業務手順（時系列） | 開始から完了までの業務操作を時系列で記述 |
| 確認事項 | シナリオ完了後に確認すべき業務上の成果物・記録 |

### 1-2. シナリオ一覧

本章が扱うシナリオを以下のとおり確定する。

| シナリオ ID | シナリオ名 | 主アクター | カバーする業務フロー |

| SCN-002 | 品質担当の照合作業 | 品質担当（SH-03） | BF-01（記録照会）・BF-03（不適合確認） |

| SCN-004 | 規制監査受入 | 品質担当（SH-03）・規制当局想定（SH-07） | BF-01（記録照会）・BF-03（CAPA 照会） |

**本節で確定した方針**
シナリオは SCN-001〜SCN-010 の十件を確定し、各シナリオの記述フォーマットを統一する（2026-05-18: SCN-006〜010 の IQC/リワークシナリオ 5 件を追加し五件→十件へ拡張）。

## 2. SCN-001: 作業員の標準業務

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-001 |
| シナリオ名 | 作業員の標準業務（朝礼〜着工〜作業〜終業） |
| 目的 | BF-01 の七段階フローが業務として正常に完了し、ALCOA+ 準拠の記録が自動生成されることを確認する |
| 主アクター | 作業員（SH-01） |
| 副アクター | 現場監督（SH-02） |

**前提条件**

- 当該作業の SOP が承認・公開済み（`published` 状態）である
- 作業員のスキル等級が当該 SOP の `skill_level_required` を充足している
- 使用する計測器の校正有効期限が未来日付である
- タブレットのマスタ同期が 24 時間以内に完了している

**業務手順（時系列）**

1. 作業員が出社し、タブレット APP に PIN でログインする
2. 朝礼での作業指示確認後、タブレット APP で自分の作業指示を確認する（品番・ロット番号・SOP 版・台数）
3. 段取ステップ（設備・治具・材料の準備確認）をロックステップで実施する
4. 第一のステップが表示される。作業員は習熟度（Competent プロファイル）に従い Linear モードで手順を確認する
5. 各ステップを順に実施し、入力種別に応じた記録を行う（チェック・数値入力・写真撮影・スキャン）
6. クリティカルステップ（fmea_rpn_flag = true）で EWI が表示される。作業員が確認し、証拠（写真）を添付する
7. 全ステップ完了後、「作業完了」ボタンが有効化される
8. 作業員が電子サインで作業完了を確認する
9. 現場監督の進捗画面にリアルタイムで完了が反映される
10. 終業時刻に次のシフトへの引継ぎ票を作成し、電子サインで確定する

**確認事項**

| 確認項目 | 期待される記録 |
|---|---|
| イベントレコードの完全性 | 全ステップに `step_completed` イベントが存在し、worker_id・timestamp_device・timestamp_server・sop_version_id の四属性がすべて非 NULL である |
| クリティカルステップの証拠 | evidence_required = true の全ステップに photo_hash が存在する |
| 電子サインの記録 | 作業完了の電子サインレコードが worker_id・承認時刻付きで存在する |
| 引継ぎ票の記録 | I-PASS 五項目が入力された引継ぎ票レコードが電子サイン付きで存在する |

**本節で確定した方針**
SCN-001 は BF-01 の七段階フロー全体を通じた業務手順と確認事項を確定する。
イベントレコードの四属性（worker_id・timestamp_device・timestamp_server・sop_version_id）の非 NULL を受入確認の必須条件とする。

## 3. SCN-002: 品質担当の照合作業

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-002 |
| シナリオ名 | 品質担当の照合作業（SOP 実行記録の照合・不適合の確認） |
| 目的 | 品質担当が管理 Web で SOP 実行記録を照合し、不適合記録の ALCOA+ 準拠を確認できることを確認する |
| 主アクター | 品質担当（SH-03） |
| 副アクター | 作業員（SH-01）・現場監督（SH-02） |

**前提条件**

- 対象のロット番号の作業が完了し、記録がサーバーに同期済みである
- 当該ロットに関連する不適合レコードが 1 件以上起票されている

**業務手順（時系列）**

1. 品質担当が管理 Web にログインする
2. トレサビ照会画面で対象のロット番号を検索する
3. 逆方向トレサビ（製品→工程→作業員・材料・設備）クエリで以下を確認する
   - 使用した材料ロット・部品ロットの一覧
   - 作業を実施した作業員（worker_id）と実施時刻
   - 使用した設備 ID と校正証明書の有効状態
   - 適用した SOP の版番号
4. 全ステップの完了記録を確認する（step_count と completed イベント数が一致しているか）
5. クリティカルステップの証拠（写真・測定値）を確認する
6. 当該ロットに関連する不適合レコードを確認する
7. 不適合の根本原因・CAPA 状態（オープン・進行中・クローズ）を確認する
8. 問題がない場合: 照合記録として確認完了電子サインを記録する
9. 追加調査が必要な場合: 追加 CAPA を起票する

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| 逆トレサビクエリの完全性 | 製品から材料ロット・作業員・設備まで途切れなく追跡できる |
| 記録完全性率 | 全ステップに complete イベントが存在し、抜けがない |
| 監査証跡の確認 | 照合完了電子サインが品質担当 ID・時刻付きで記録される |

**本節で確定した方針**
SCN-002 は品質担当による逆方向トレサビ照会と照合完了電子サインの業務手順を確定する。
監査照合は 30 秒以内のクエリ完了と照合完了電子サインの記録を受入条件とする。

## 4. SCN-003: IT 担当のマスタ改訂

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-003 |
| シナリオ名 | IT 担当のマスタ改訂（改訂提案受領から Publish まで） |
| 目的 | BF-02 の六段階フローが IT 担当・品質担当の協働で業務として完了し、Draft First 原則が遵守されることを確認する |
| 主アクター | IT 担当（SH-05）・品質担当（SH-03） |
| 副アクター | 現場監督（SH-02） |

**前提条件**

- Kaizen Teian（採用決定済み）が SOP 改訂を必要としている
- 改訂対象の SOP が現在 `published` 状態である

**業務手順（時系列）**

1. 品質担当が管理 Web で改訂提案を受領する（Kaizen Teian レコードから自動リンク）
2. 品質担当が影響範囲を確認する（参照整合性ガードで影響する SOP・作業パターンの件数を確認）
3. 品質担当が「新規ドラフト作成」操作を行い、改訂内容をドラフトエディタで編集する
4. Auto-Save（30 秒間隔）で編集が自動保存される（ブラウザクラッシュ対策）
5. Diff Always Visible で公開版との差分を確認しながら編集する
6. 編集完了後、品質担当が「レビュー依頼」操作を行う（状態: `under_review`）
7. 現場監督が差分内容を確認し「承認依頼」操作を行う
8. 品質担当が最終確認を行い、Effective Date を設定して電子サインで承認する（状態: `approved`）
9. Effective Date 当日に自動公開（または手動公開）で `published` 状態に遷移する
10. 旧版が `obsolete` に遷移し、影響する作業員に「SOP 改訂通知」が配信される

**確認事項**

| 確認項目 | 期待される記録 |
|---|---|
| ドラフト中のタブレット非表示 | `draft` 状態の SOP がタブレット APP に表示されない |
| 承認電子サインの記録 | `master_edit_event` に承認操作・承認者 ID・時刻が記録されている |
| 旧版の保持 | 旧版 SOP が `obsolete` 状態で物理削除されずに存在する |
| 改訂通知の配信 | 影響する作業員グループへの通知が送信済みである |
| Kaizen Teian との連動 | Kaizen Teian レコードに改訂版 SOP ID が参照リンクで設定されている |

**本節で確定した方針**
SCN-003 は BF-02 の六段階フローにおける Draft First 原則の業務的実現を確定する。
ドラフト中のタブレット非表示・承認電子サインの master_edit_event 記録・旧版の物理削除禁止を受入条件とする。

## 5. SCN-004: 規制監査受入

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-004 |
| シナリオ名 | 規制監査受入（監査員への作業記録提示・ALCOA+ 対応） |
| 目的 | 内部監査・顧客監査（SH-07: 規制当局想定）において、ALCOA+ 準拠の記録を証拠として提示できることを確認する |
| 主アクター | 品質担当（SH-03） |
| 副アクター | 規制当局想定（SH-07）・IT 担当（SH-05） |

**前提条件**

- 監査対象の製造ロットの全作業記録がサーバーに存在する
- 対象期間内の CAPA レコード・不適合レコードが存在する
- 管理 Web のトレサビ照会機能が正常動作している

**業務手順（時系列）**

1. 品質担当が監査員（SH-07）に対し、管理 Web のトレサビ照会機能を使って記録を提示する
2. 監査員が「製品ロット XX はいつ・誰が・どの手順で製造したか」を問い合わせる
3. 品質担当が逆方向トレサビクエリを実行し、製品→工程→作業員・材料ロット・設備・SOP 版を提示する
4. 監査員が ALCOA+ の各原則への対応を確認する
   - Attributable: worker_id がすべての記録に存在する
   - Contemporaneous: timestamp_device と timestamp_server の二重記録が存在する
   - Original: Append-only による UPDATE/DELETE 禁止の設計説明（IT 担当が DB ロール権限を説明）
   - Enduring: バックアップポリシー・保管期間設定の確認（IT 担当が提示）
5. 監査員が特定のステップの証拠（写真・測定値）の閲覧を要求する
6. 品質担当が対象 Step の photo_hash・測定値・校正証明書参照を提示する
7. 監査員が Audit Trail（`master_edit_event` テーブルの SOP 変更履歴）の確認を要求する
8. 品質担当が SOP の改訂履歴・承認者・Effective Date の一覧を提示する
9. 監査員が監査所見を記録し、監査完了を宣言する
10. 品質担当が監査対応記録を作成し、指摘事項を CAPA として起票する

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| 逆トレサビクエリの速度 | 対象ロットの全記録照会が 30 秒以内に完了する |
| ALCOA+ 九原則の説明 | 各原則に対応する記録・設計を品質担当・IT 担当が説明できる |
| 証拠の提示 | 写真・測定値・校正証明書参照を監査員が確認できる |
| Audit Trail の提示 | SOP 変更履歴・承認者・Effective Date が一覧で提示できる |

**本節で確定した方針**
SCN-004 は規制当局想定（SH-07）への ALCOA+ 九原則対応説明と Audit Trail 提示の業務手順を確定する。
逆トレサビ 30 秒以内・証拠提示・Audit Trail 一覧提示を受入条件とする。

## 6. SCN-005: 夜勤縮退運用

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-005 |
| シナリオ名 | 夜勤縮退運用（Wi-Fi 全断時の紙フォールバック業務） |
| 目的 | BF-01 の縮退フローが夜勤帯の Wi-Fi 全断時に正常に機能し、作業記録が継続されることを確認する |
| 主アクター | 作業員（SH-01）・現場監督（SH-02） |
| 副アクター | IT 担当（SH-05） |

**前提条件**

- 夜勤帯（20:00 以降）に工場内 Wi-Fi が全断する
- タブレット APP のオフライン用マスタが前回同期時（24 時間以内）のデータで存在する
- 各工程の紙 SOP（最新版 PDF の印刷物）が紙保管庫に存在する

**業務手順（時系列）**

1. タブレット APP がオフライン状態を検知し、ステータスバーに「オフライン（記録中）」を表示する
2. 作業員は通常と同一の操作で SOP を参照・実行・記録を継続する（Offline-First 動作）
3. 現場監督がタブレットのオフライン状態を確認し、IT 担当に連絡する
4. IT 担当がサーバー・ネットワーク障害の診断を開始する
5. 障害が長期化（1 時間超）する兆候がある場合、現場監督が「紙フォールバックへの切替」を判断する
6. 作業員は紙保管庫から最新版 SOP 印刷物を取り出し、紙チェックリストで記録を継続する
7. タブレット内のオフラインキュー記録は継続して保持される
8. Wi-Fi 復旧後、タブレット APP が自動的に Outbox Pattern でサーバーへ同期を実行する
9. 現場監督が「紙フォールバック中の記録」をシステムに後入力する（is_retroactive = true フラグ付き）
10. 後入力者 ID・入力時刻・紙原本の保管場所が記録される
11. 紙原本はシステムへの同期完了後も定められた保管期間（ISO 9001 §7.5 に従う）保管する

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| オフライン継続動作 | タブレット APP がオフライン中も全 Step の実行・記録を継続できる |
| ローカルキューの同期 | 通信回復後に Outbox Pattern で全ローカルレコードがサーバーへ同期される |
| 後入力レコードの識別 | 後入力レコードに is_retroactive = true・後入力者 ID・時刻が付与されている |
| 紙記録との突合 | 紙原本の保管場所が記録に紐付いており、後からの確認が可能 |
| ALCOA+ Contemporaneous の整合 | is_offline = true レコードは「通信断中の実際の記録時刻」を timestamp_device で保持している |

**本節で確定した方針**
SCN-005 は Wi-Fi 全断時の Offline-First 継続動作・Outbox 同期・紙フォールバック後の後入力手続を確定する。
is_offline・is_retroactive フラグの正確な付与を受入条件とし、ALCOA+ Contemporaneous 原則との整合を担保する。

## 7. SCN-006: 入荷検査から後工程解放

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-006 |
| シナリオ名 | 入荷材料の AQL 抜取検査から後工程ロット解放まで |
| 目的 | BF-07 の受入登録〜サンプリング〜検査〜合否判定〜後工程解放フローが正常に機能することを確認する |
| 主アクター | 品質担当（IQC 検査員ロール）（SH-03） |
| 副アクター | 作業員（SH-01）、IT 担当（SH-05） |

**前提条件**
- 材料マスタ（TBL-036）に当該材料が登録済みである
- 仕入先マスタ（TBL-037）に当該仕入先が登録済みである
- サンプリング計画マスタ（TBL-039）に当該材料 × 仕入先の計画が有効状態で存在する

**業務手順（時系列）**

1. 品質担当がハンディ APP でロット番号を入力または QR スキャンして受入登録する（FR-IQ-001）
2. システムがサンプリング計画を解決し、サンプルサイズ n・Ac・Re・検査項目を表示する（FR-IQ-002/003）
3. 品質担当がサンプル採取を実施し、各サンプルの測定値・写真を入力する（FR-IQ-004）
4. 全サンプル入力完了後、システムが AQL 自動合否判定を実行する（FR-IQ-005）
5. 不良数 ≤ Ac であれば ACCEPT 判定。`lots.qc_status = PASSED` に更新され後工程解放される（FR-IQ-006）
6. 作業員が後工程の材料 QR スキャン時に「合格ロット」として通過する（FR-IQ-009）

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| サンプリング計画解決 | 材料 × 仕入先の最適なサンプリング計画が自動選択される |
| AQL 自動判定 | 不良数と Ac/Re の比較が正確に実施され合否が記録される |
| 後工程ゲート通過 | 合格ロットの QR スキャンがエラーなく通過する |
| ALCOA+ 記録 | 検査者 worker_id・二重タイムスタンプ・計測器 ID が全測定値に記録される |

## 8. SCN-007: 特採承認と後工程警告

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-007 |
| シナリオ名 | AQL 基準外ロットの特採承認と後工程での警告バナー表示 |
| 目的 | 特採フロー（品質担当電子サイン・有効範囲・有効期限の記録）と後工程での警告バナー強制表示が正常に機能することを確認する |
| 主アクター | 品質担当（SH-03） |
| 副アクター | 作業員（SH-01） |

**前提条件**
- AQL 判定で不合格（REJECT）判定のロットが存在する
- 品質担当が当該ロットを技術的評価のうえ、限定的使用が可能と判断した

**業務手順（時系列）**

1. 品質担当が管理 Web の特採承認コンソール（SCR-MC-010）で当該ロットを確認する
2. 品質担当が特採判定（CONCESSION）を選択し、特採理由・有効範囲・有効期限を記録する（FR-IQ-007）
3. 品質担当が電子サインで特採承認を確定する（TBL-041 Append-only 記録）
4. `lots.qc_status = CONDITIONAL_PASS` に更新される
5. 作業員が後工程で当該ロットの QR スキャン時、黄色警告バナーが強制表示される（FR-IQ-010）
6. 作業員が警告バナーを確認（タップ）し、作業を継続する（確認操作が記録される）

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| 特採記録の完全性 | 特採理由・有効範囲・有効期限・電子サインが TBL-041 に Append-only で記録される |
| 警告バナーの強制表示 | dismissable 不可の黄色警告バナーが表示される |
| 確認操作の記録 | 作業員の警告認識確認（タップ）が worker_id・時刻付きで記録される |
| 有効期限超過の動作 | 有効期限超過後は CONDITIONAL_PASS が期限切れとしてブロック扱いになる |

## 9. SCN-008: 仕入先品質実績の悪化対応

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-008 |
| シナリオ名 | 特定仕入先の品質実績悪化を経営層へ報告し CAPA を起票するシナリオ |
| 目的 | 仕入先別品質実績ダッシュボードの集計精度と CAPA への連携が正常に機能することを確認する |
| 主アクター | 品質担当（SH-03） |
| 副アクター | 経営層（SH-06） |

**業務手順（時系列）**

1. 品質担当が管理 Web の仕入先品質実績ダッシュボード（SCR-MC-012）で月次実績を確認する（FR-IQ-013）
2. 特定仕入先の不合格率が前月比で 20% 以上悪化していることを確認する
3. 当該仕入先の検査厳しさが「なみ」→「きつい」に自動切替済みであることを確認する（FR-IQ-008）
4. 品質担当が CAPA を起票し、原因として「仕入先起因」（4M: Material/Supplier）を選択する（FR-ST-009）
5. 品質担当が経営層へ仕入先品質実績レポート（RP-009）を PDF で提出する
6. 経営層が仕入先との協議を判断し、CAPA の「是正措置」として「仕入先への改善要求書発行」を記録する

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| ダッシュボード集計精度 | 仕入先 × 材料 × 月単位の集計が正確に表示される |
| 個人集計の禁止 | IQC 検査員 worker_id 単位の集計・ランキングが画面に存在しない |
| CAPA との連携 | 仕入先起因の CAPA 起票が正常に完了する |
| RP-009 生成 | 仕入先品質実績レポートが PDF で出力される |

## 10. SCN-009: 不適合発見からリワーク・再検査・出荷確定

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-009 |
| シナリオ名 | 不適合現品のリワーク → 再検査 → 出荷可能確定の完全フロー |
| 目的 | BF-03（不適合起票）→ BF-08（ディスポジション→リワーク→再検査→CLOSED_OK_RELEASE）の完全フローが ALCOA+ Original 原則を守って機能することを確認する |
| 主アクター | 作業員（SH-01）・品質担当（SH-03）・現場監督（SH-02） |
| 副アクター | IT 担当（SH-05） |

**前提条件**
- 不適合現品（特定 lot_id × case_id）が発生している
- リワーク SOP（sop_type = REWORK）が公開済みである
- リワーク SOP マッピングマスタ（TBL-046）に当該不適合カテゴリの対応が定義済みである

**業務手順（時系列）**

1. 作業員がアンドン発報または不適合起票を行う（BF-03 起動）
2. 品質担当がディスポジション判定（MRB）を管理 Web で実施。REWORK（REWORK_FULL）を選択し、品質担当 + 現場監督の二者電子サインで確定する（FR-ST-013）
3. 作業員のハンディ APP にリワーク指示が通知される（SCR-HA-019）
4. 作業員がリワーク前の現品写真を撮影して記録する（FR-EV-014）
5. 作業員がリワーク SOP に従い修正作業を実施する（新 case_id で記録）
6. 作業員がリワーク後の現品写真を撮影して記録する
7. 作業員が完了電子サインを行い、修正品 QR ラベルが自動発行される
8. リワーク実施者とは別の作業員が修正品 QR をスキャンして再検査画面を起動する（FR-NV-015）
9. 再検査者が再検査 SOP に従い検査を実施し、合格（OK）で電子サインする
10. リワーク状態が `CLOSED_OK_RELEASE` に更新され、当該ロットが出荷可能状態になる

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| 元 WorkExecution の不可侵性 | parent_case_id の work_events に UPDATE/DELETE が存在しない |
| 二者電子サイン分離 | disposition.quality_admin_sign_id ≠ disposition.supervisor_sign_id であり、各サインの signer_id も異なる |
| リワーク前後証拠の完備 | リワーク開始前・完了後の写真が両方 SHA-256 ハッシュ付きで記録されている |
| 再検査者分離 | 再検査者の worker_id ≠ リワーク実施者の worker_id |
| CLOSED_OK_RELEASE への遷移 | reworks.status = CLOSED_OK_RELEASE と lots.qc_status = PASSED の整合 |

## 11. SCN-010: リワーク廃却処理と証拠保全

| 項目 | 内容 |
|---|---|
| シナリオ ID | SCN-010 |
| シナリオ名 | 再検査不合格後の廃却処理と廃棄証拠保全 |
| 目的 | BF-08 の SCRAP フロー（廃棄物処理票・立会者電子サイン・証拠保全）が Enduring 原則を守って機能することを確認する |
| 主アクター | 品質担当（SH-03）・経営層（SH-06） |
| 副アクター | 作業員（SH-01） |

**業務手順（時系列）**

1. 再検査結果が SCRAP_AFTER_REWORK となる
2. 品質担当と経営層がディスポジション SCRAP を二者電子サインで確定する
3. 廃却処理担当者が廃棄物処理票（PDF）を作成し、システムにアップロードする
4. 廃棄立会者（品質担当または経営層）が電子サインで廃却実施を確認する（TBL-049）
5. 廃却記録が `CLOSED_SCRAP` として確定する

**確認事項**

| 確認項目 | 期待される結果 |
|---|---|
| 廃棄物処理票の PDF 記録 | TBL-049 に廃棄物処理票 PDF が証拠として紐付いている |
| 廃却記録の永続保持 | CLOSED_SCRAP 後も reworks・scrap_records レコードが削除 API を持たない |
| Enduring 原則の担保 | 7 年後の参照を想定したバックアップ対象であることをシステム管理者が確認できる |
