# 00 本書の位置づけと上流文書からの継承

本文書は「業務要件サブ（03_要件定義/業務要件/）」の全 12 ファイルが担う責務を確定し、上流文書（システム化構想・システム化計画）から継承する方針と制約を整理するメタ文書である。各業務要件ファイルを執筆・レビューする際は本文書を先頭に参照する。

---

## 1. 業務要件サブの責務

### 1-1. IPA 共通フレームとの対応

本業務要件サブは IPA 共通フレーム 2013（SLCP-JCF2013）の「要件定義プロセス」のうち、「業務要件定義」タスクに対応する。業務要件定義タスクは以下の三点を成果として要求する。

1. **業務改善内容の定義**: 現状の業務上の問題・制約を整理し、To-Be 業務がどう変わるかを明確化する
2. **業務機能の定義**: To-Be 業務において必要となる機能の範囲を業務視点で確定する
3. **業務処理の定義**: To-Be 業務の手順・フロー・役割分担を処理単位で記述する

本業務要件サブはこれら三点を製造業向け作業ナビゲーション＆トレサビ記録アプリの文脈で具体化する。

以下の内容は本業務要件サブに含まない。対応する文書に委ねる。

**本節で確定した方針**
業務要件サブは IPA 共通フレームの「業務要件定義」タスクに相当し、業務改善内容・業務機能・業務処理の三点を責務とする。
機能仕様・非機能数値・アーキテクチャは本サブの対象外とし、それぞれ機能要件・非機能要件・概要設計に委ねる。
本サブの全ファイルは本文書を先頭参照として執筆する。

## 2. システム化構想からの倫理制約継承

### 2-1. 継承する倫理制約の一覧

システム化構想で確定した倫理制約を業務要件として継承する。これらは業務フロー設計・業務シナリオ・制約条件の全節にわたって有効である。

| 倫理制約 ID | 内容 | 業務要件への影響 |
|---|---|---|
| EC-01 | 支援と監視の意図的分離 | 業務フロー上で作業員の行動ログが管理者の監視ツールとして機能する設計を禁止する |
| EC-02 | 個人別生産性スコアリング禁止 | 業務手続上で作業員を個人単位で順位付けする運用を禁止する |
| EC-03 | 行動データ用途三限定 | 業務記録の利用を品質保証・工程改善・教育設計の三用途に限定し、人事評価への転用を業務運用規約で禁止する |
| EC-04 | Work-as-Done の「違反」解釈禁止 | SOP との乖離を作業員個人の違反として解釈する業務プロセスを禁止する |
| EC-05 | AI/ML による自動判断・異常検知・自然言語指示生成を永続的に提供しない | AI/ML 自動判断・異常検知機能を業務フロー・業務手続の前提とする設計を禁止する。本システムを外部 AI システムへの中継手段として使用する業務設計を禁止する |
| EC-06 | データオーナーシップ工場帰属 | 作業記録データの所有権は工場（導入企業）に帰属し、外部提供を業務手続として禁止する |

### Just Culture の組織原則継承

EC-05に加え、システム化構想 04章 H2-5 が採用した James Reason の Just Culture モデルは、本業務要件サブの以下の要件に直接反映する。

- スリップ・ラプス（意図なき行為の失敗）を自動的に問責プロセスに結びつける業務設計を禁止する
- ヒヤリハット・Kaizen 起票は匿名投稿オプションを提供し、起票行為を保護する業務手続を確定する
- SOP との乖離記録を「適応の証拠」として収集し、SOP 改訂ループに活用する業務設計を確定する

### 2-2. 業務要件へのトレースダウン

EC-03（行動データ用途三限定）は以下の業務要件に直接トレースダウンする。

- 11_業務上の制約条件.md 第 4 節「倫理的運用制約」: 用途三限定の運用手続を確定する
- 02_ステークホルダー定義書.md 第 3 節「受入基準の設定方針」: 労働者代表の受入条件に用途三限定の制度的保証を含める
- 08_Kaizen・ヒヤリハット起票業務フロー.md 第 4 節「匿名性の担保と用途三限定」: 起票行動データの転用禁止業務手続を確定する

EC-05（Just Culture）は以下の業務要件に直接トレースダウンする。

- 07_アンドン・不適合対応業務フロー.md 第 4 節「Just Culture 適用」: 組織文化としての業務定義を確定する
- 08_Kaizen・ヒヤリハット起票業務フロー.md 第 1 節「起票主体と保護原則」: 起票行為を保護する業務手続を確定する

**本節で確定した方針**
システム化構想の倫理制約 EC-01〜EC-06 はすべて業務要件として継承し、業務フロー・業務手続・制約条件の各節に展開する。
倫理制約のトレースダウン先は本節の対応表で確定し、要件変更時は本節を起点に影響範囲を検索する。
倫理制約を緩和・撤廃する変更は業務要件サブ全体の再査定を必須とする。

### 3-2. 継承の優先順位

上流文書に矛盾が生じた場合、以下の優先順位で適用する。

1. システム化構想の倫理制約（最上位・不可変）
2. システム化計画の対象外判断（機能スコープ・非スコープの三区分）
3. システム化計画のビジネス目標
4. 業務要件サブ各ファイルの詳細定義

業務要件サブが上流文書と矛盾する記述を含む場合、業務要件サブの記述を優先せず上流文書に合わせて修正する。

### 3-3. 変更管理手順

業務要件の変更が必要になった場合、以下の順序で実施する。

1. 変更の発端となった業務上の問題を「変更要求」として記録する
2. 影響を受ける上流文書（システム化計画・システム化構想）の節を特定する
3. 上流文書の変更が不要な場合: 業務要件サブのみを更新し、本継承マップの整合性を確認する
4. 上流文書の変更が必要な場合: 上流文書の改訂を先行させ、本業務要件サブをその後に更新する

**本節で確定した方針**
システム化計画の全章から業務要件サブへの継承マップを本節の表で確定する。
矛盾時は上流文書を優先し、業務要件サブを修正する優先順位を確定する。
業務要件の変更は上流文書への影響確認を先行させる変更管理手順で実施する。

## 4. 下流への引き渡し

### 4-1. 機能要件への入力

業務要件サブの成果物は機能要件定義（03_要件定義/機能要件/）への入力として機能する。以下の対応関係で引き渡す。

| 業務要件の成果物 | 機能要件への入力内容 |
|---|---|
| 業務フロー（05〜09 各ファイル） | 機能要件の「操作手順」「画面遷移」の設計根拠 |
| 業務シナリオ集（10 ファイル） | 機能要件の「受入テストシナリオ」の前段階定義 |
| 業務ドメイン用語集（03 ファイル） | 機能要件・概要設計の用語統一基盤 |
| ステークホルダー定義書（02 ファイル） | 機能要件の RBAC 設計・画面権限設計の前提 |
| 業務上の制約条件（11 ファイル） | 機能要件の「機能外要件」「倫理的制約」として制約を通知 |

### 4-2. 概要設計への入力

概要設計（04_概要設計/ 以下）への引き渡し項目は以下のとおりである。

| 業務要件の成果物 | 概要設計への入力内容 |
|---|---|
| 業務 KPI の計測方法（01 ファイル 4 節） | 品質ダッシュボード設計・レポーティング仕様の要件 |
| RACI マトリクス（02 ファイル 4 節） | アクセス制御設計（RBAC ロール対応）の前提 |
| 業務ドメイン用語集（03 ファイル） | データモデル設計の正規語彙 |
| マスタ管理業務フロー（06 ファイル） | ワークフローエンジン設計の仕様 |
| 業務上の制約条件（11 ファイル） | セキュリティ設計・倫理的制約の設計インプット |

### 4-3. 引き渡し完了の判定条件

業務要件サブが「引き渡し完了」と判定されるためには以下の条件を充足する。

1. 本書（00 ファイル）の継承マップが上流文書と整合している
2. 全 12 ファイルの「本節で確定した方針」（太字）が記述されている
3. 全 12 ファイルの「参照業界分析」セクションが必須/関連区分で記載されている
4. 業務フロー図プレースホルダー（fig_xxx 形式）が全業務フローファイルに記載されている
5. 要件 ID（BR-BUS-NNN 形式）が全業務要件節に付与されている（機能要件フェーズで付与）

**本節で確定した方針**
業務要件サブは機能要件・概要設計への入力として機能し、引き渡し項目対応表を本節で確定する。
引き渡し完了の判定条件 5 項目を充足した状態を「業務要件定義フェーズの完了」と定義する。
機能要件フェーズでは本業務要件サブを変更せず、機能要件側に詳細を追記することを原則とする。
