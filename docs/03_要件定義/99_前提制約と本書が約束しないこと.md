# 99 前提制約と本書が約束しないこと

## 1. 前提条件 P01〜P10

本節は本書（要件定義書）が依拠する前提条件を確定する。
計画書 `11_前提・制約・リスク.md` の P01〜P07・T01〜T05 を継承し、要件定義版として再記述する。
前提が成立しない環境への本書の適用は対象外とする。

### 業務的前提条件

| 前提 ID | 前提条件 | 前提が崩れた場合の要件への影響 |
|---|---|---|
| P01 | 単一工場・単一サイトで運用する | RBAC の権限スコープ設計・データモデルの `factory_id` 運用方針の再設計が必要になる。マルチサイト制御ロジックの要件化が発生する |
| P02 | 社内 LAN のみでの利用（インターネット接続なし） | 外部認証（OAuth2/OIDC）・クラウド連携・外部監査機関へのデータ提出が要件として追加される。NFR-SEC 全体の再設計が必要になる |
| P03 | 作業員は Android または Windows タブレットを主要接点として使用する（iOS タブレットも Must 対応） | iOS 専用・スマートグラス・特殊端末が主流になる場合は React Native の対応プラットフォームの再確認が必要になる |
| P04 | 工場内の Wi-Fi カバレッジは不完全である（デッドゾーンが存在する） | 完全有線接続環境でも Offline-First 設計は維持する。P04 が崩れても設計を後退させない |
| P05 | 外国人技能実習生・特定技能労働者が 1 名以上在籍する現場を前提とする | ゼロの場合、多言語機能は将来余地に留まるが、実装は省かない（P04 と同様） |
| P06 | 規制当局（GMP/FDA 等）への電子記録提出義務は現時点で発生していない | GMP/FDA Part 11 適用業種への展開時は、計画書 `11章` の将来オプション（規制業界拡張）が発動する。現時点の要件は変更しない |
| P07 | IT 担当は 1 名でサーバー・ネットワークを兼務する | IT 専門部署が設置される場合は OPS/MNT 要件の一部が簡略化できるが、設計基準は後退させない |
| P08 | 習熟度の多様性：入社直後の未習熟者から数十年の熟練者まで同一ラインに混在する | 習熟度が均一な環境では段階的 UI（詳細モード/熟練者モード）は過剰設計になるが、実装を省かない |
| P09 | 倫理制約 EC-01〜EC-06 は前提条件として機能し、要件の個別合意によって覆ることはない | P09 が覆る場合は、構想書 `04_倫理スタンスと支援・監視の意図的分離.md` の改訂が先行する。本書の改訂は構想書改訂後に行う |
| P10 | ver1.0.0 スコープ：本書が確定する全要件は ver1.0.0 の実装対象を前提とする。ver1.0.0 以降の機能追加は本書の対象外とする | ver1.1.0 以降の機能は別途要件定義を起票する |

---

**本節で確定した方針**
- P01〜P10 を本書の業務的前提条件として確定し、前提が崩れた場合の影響を明示する。
- P04・P05・P08 については「前提が崩れても設計基準を後退させない」ことを明示する。
- P09（倫理制約）は要件の個別合意によって覆らないことを確定し、変更には構想書改訂を先行させることを確定する。

## 2. 制約条件

### 技術スタック制約

以下の技術スタックは計画書 `07_技術スタック選定根拠.md` で確定されており、本書ではこれらを制約として受け入れる。本書の要件はこの技術スタックの実装可能範囲内で確定する。

| 制約 ID | 制約内容 | 要件への影響 |
|---|---|---|
| T-C01 | フロントエンドは React Native（Expo）+ SQLite + TypeORM を使用する | ローカルストレージ要件は SQLite の機能範囲内で確定する |
| T-C02 | バックエンドは Rust Edition 2024 + tokio + axum を使用する | API 設計・同期処理の要件は Rust/axum の実装可能範囲内で確定する |
| T-C03 | データベースは PostgreSQL を使用する | Append-only Event Sourcing・ALCOA+ 実装要件は PostgreSQL 機能前提で確定する |
| T-C04 | デプロイは IIS + Windows Server 2022 + WSL2 + Docker で行う | 可用性・RTO/RPO 要件はこのインフラ構成前提で確定する |
| T-C05 | マスタメンテナンス UI は React（Web）を使用する | 管理 Web の機能要件は React Web フロントエンド前提で確定する |

### ネットワーク制約

| 制約 ID | 制約内容 | 要件への影響 |
|---|---|---|
| N-C01 | 社内 LAN のみでの通信。インターネットアクセスは禁止とする | クラウド API・外部認証・外部監視サービスは全要件の非対象とする |
| N-C02 | Wi-Fi の安定性に課題がある環境を前提とする（デッドゾーン存在） | FR-SY（同期・オフライン）要件は Offline-First を Must とする |
| N-C03 | ネットワーク外からのアクセスは禁止とする（VPN 等の社外アクセスは現在想定しない） | NFR-SEC 要件のネットワーク境界は社内 LAN のみを前提とする |

### 規模・組織制約

| 制約 ID | 制約内容 | 要件への影響 |
|---|---|---|
| O-C01 | 個人開発（開発者 1 名）であり、並行作業チームは存在しない | PRJ 要件（プロジェクト要件）はシングル開発者前提で確定する |
| O-C02 | IT 担当 1 名が運用保守を全て担う | OPS/MNT 要件は IT 担当 1 名が運用できる複雑度に留める |
| O-C03 | phase 構造を採用しない。ver1.0.0 = プロダクションレベル品質 | Must 要件は ver1.0.0 リリース時に全て満たす。品質を段階的に上げる方針を採用しない |

---

**本節で確定した方針**
- 技術スタック・ネットワーク・組織の 3 種類の制約を確定し、各要件への影響を明示する。
- 社内 LAN 限定・インターネットアクセス禁止を全要件の前提として確定する。
- 個人開発・IT 担当 1 名・phase なしを組織制約として確定し、要件の複雑度管理の根拠とする。

---

## 3. 本書が約束しないこと

本節は本書（要件定義書）が ver1.0.0 において提供しないことを明示する。
この宣言は本 README §9 にも重複記載する。両文書での一致が要求される（自己監査チェックリスト #8 参照）。

### 機能・対応の非約束リスト

| 非約束項目 | 分類 | 根拠 |
|---|---|---|
| SaaS/マルチサイト/マルチテナンシーへの対応 | ver1.0.0 非対象 | P01（単一工場）・P02（社内 LAN）前提条件に基づく |
| GMP/FDA 21 CFR Part 11 への適合 | 永続非対象（P06 変更があれば再評価） | P06（規制提出義務なし）前提条件・計画書 11章の対象外判断の継承 |
| EU MDR への適合 | 対象外と判断する | 医療機器製造業を主要ターゲットとしない |
| IATF 16949 への適合 | 対象外と判断する（P06 変更があれば再評価） | 計画書 11章の対象外判断の継承 |
| AS9100 への適合 | 対象外と判断する | 航空宇宙業種は対象外業種 |
| HACCP への適合 | 対象外と判断する（P06 変更があれば再評価） | 食品専用の規制適合は本書の責務外 |
| AI/ML による自動判断・異常検知・自然言語作業指示生成 | **永続非提供** | 構想 08章・11章の永続禁止宣言の継承。EU AI Act 高リスク区分・倫理制約 EC-05 |
| AR・スマートグラス・生体センサ・IoT センサとの直接統合 | ver1.0.0 非対象 | OT-IT 境界原則（計画 05章）・個人開発スコープの超過 |
| PLC/OPC-UA/OT センサとの直接統合 | **永続非提供** | 機能安全（IEC 61508/ISO 13849）設計責任の範囲超過。構想 11章の永続禁止宣言の継承 |
| 任意のレガシー独自プロトコルのみを持つ基幹システムとのリアルタイム連携 | ver1.0.0 非対象 | 標準 REST/Webhook アダプタを持たない基幹システムへのアダプタは保証しない（個別 SIer 対応余地） |
| 労働者の生産性を個人単位で評価するレポーティング機能 | **永続禁止** | 倫理制約 EC-02（個人別スコアリング禁止）。RBAC で技術的に実装を禁止する |
| 画面レイアウト・UI 実装詳細 | 概要設計書の責務 | 本書の非対象（`00_本書の位置づけと要件定義の境界.md` H2-3 参照） |
| 物理 DB スキーマ（テーブル定義・インデックス・制約詳細） | 概要設計書の責務 | 同上 |
| API エンドポイント定義・リクエスト/レスポンス仕様 | 概要設計書の責務 | 同上 |
| テストケース実装・テストコードの記述 | 詳細設計書・実装の責務 | 同上 |
| 法令・規制の解釈責任の代行 | 永続非対象 | 法的専門性の範囲外。規制への適合可否の最終判断は導入企業の法務・品質担当に帰属する |

---

**本節で確定した方針**
- 非約束リストを 15 項目で確定し、「永続非提供」「永続禁止」「ver1.0.0 非対象」「概要設計書責務」の 4 区分で分類する。
- AI/ML・PLC/OT 直結・個人別生産性評価を永続非提供/永続禁止として確定し、要求があっても覆らないことを明示する。
- 画面実装詳細・物理 DB スキーマ・API 仕様を概要設計書の責務として明示し、本書への混入を防ぐ。

## 4. 将来再評価条件

### 再評価が必要な条件

以下の条件変化が生じた場合のみ、本書の前提条件・非約束リストを再評価する。

| 再評価トリガー | 影響する前提/非約束 | 再評価に必要な手続き |
|---|---|---|
| 複数工場への展開が確定した | P01（単一工場）→ マルチサイト要件化 | 計画書 `02章` の対象範囲改訂 → 本書 P01 の改訂 → `機能要件/`・`非機能要件/` の影響要件を再評価 |
| 規制適用業種（GMP/IATF 等）への展開が確定した | P06（規制提出義務）→ 規制要件化 | 計画書 `11章` 将来オプション発動 → 規制別要件を新規サブとして追加 |
| インターネット経由でのアクセスが必要になった | P02（社内 LAN）→ クラウド連携要件化 | 計画書 `02章` の対象範囲改訂 → NFR-SEC・OPS 全体の再設計 |
| IT 担当が 2 名以上になった | O-C02（IT 担当 1 名）→ 一部 OPS/MNT 要件の緩和 | 本書 O-C02 の改訂のみで対応可能（設計後退は行わない） |
| 倫理制約に関する法的・社会的環境が根本的に変化した | EC-01〜EC-06 → 構想書の改訂が先行 | 構想書 `04章` の改訂 → 本書 `01_前提継承.md` EC-01〜EC-06 の改訂 → 倫理品質要件の再評価 |

### 再評価の対象外とする条件

以下の変化は本書の再評価トリガーとしない。

- 開発スケジュールの変更（プロジェクト要件の問題であり、機能要件には影響しない）
- 技術スタック内のマイナーバージョンアップ（T-C01〜T-C05 の基本方針に変化なし）
- 特定の機能の実装方式の変更（概要設計・詳細設計の問題であり、要件には影響しない）
- ステークホルダーの担当者変更（RACI マトリクスの担当者名の更新のみで対応する）

---

**本節で確定した方針**
- 将来再評価条件を 5 項目で確定し、各トリガーに対して必要な手続きを明示する。
- 開発スケジュール変更・技術マイナーバージョンアップ・実装方式変更・担当者変更を再評価対象外として確定する。
- 倫理制約の変更は構想書改訂を先行させ、本書の改訂は構想書改訂後に行うことを確定する。
