# 02 ソフトウェア改修運用要件

本書の責務は、本システムのソフトウェア（バックエンド・タブレット APP・管理 Web・DB スキーマ）を改修・更新する際の要件を確定することである。個人開発体制・IT 担当 1 名兼務・単一工場オンプレの制約下で、安全かつ追跡可能なソフトウェア変更管理の仕組みを確定する。

---

## 1. パッチ適用要件

### 1-1. セキュリティパッチの適用基準

MNT-017: セキュリティパッチの適用優先度と適用期限を以下の通り確定する。

| 重大度区分 | 定義（CVSS v3 スコア） | 適用期限 |
|---|---|---|
| Critical | CVSS スコア 9.0〜10.0 | 発見後 3 日以内（緊急対応） |
| High | CVSS スコア 7.0〜8.9 | 発見後 7 日以内 |
| Medium | CVSS スコア 4.0〜6.9 | 発見後 30 日以内（計画停止枠での適用） |
| Low | CVSS スコア 0.1〜3.9 | 発見後 90 日以内 |

MNT-018: パッチ適用の対象は以下の通りとする。

| 対象 | 確認ツール・方法 |
|---|---|
| Rust 依存ライブラリ | `cargo audit` を週次で実行する（GitHub Actions または systemd timer で自動化する） |
| React Native 依存ライブラリ | `npm audit` を週次で実行する |
| React（管理 Web）依存ライブラリ | `npm audit` を週次で実行する |
| OS（Windows Server 2022） | Windows Update の重要更新を月次で確認する |
| Docker イメージ（PostgreSQL・Nginx） | `docker pull` での最新タグ確認を月次で実施する |

MNT-019: `cargo audit` の実行は週次自動実行を要件とし、HIGH 以上の脆弱性を検出した場合は IT 担当にアラートメールを送信する。アラートメールには脆弱性の CVE ID・重大度・影響を受けるクレート名・推奨対応を含める。

**本節で確定した方針**
- Critical 3 日・High 7 日・Medium 30 日・Low 90 日のパッチ適用期限を確定する。
- cargo audit の週次自動実行と High 以上検出時のアラートメール送信を要件として確定する。
- OS・Docker イメージを含む全スタックのパッチ対象と確認方法を確定する。

## 2. OTA（無線更新）要件

### 2-1. タブレット APP の更新配布方式

MNT-020: タブレット APP（React Native）の更新配布は以下の方式で実施する。

| 配布方式 | 対象 | 実施手順 |
|---|---|---|
| APK ファイルサイドロード | Android タブレット | サーバーの管理 Web から APK をダウンロードし、タブレットにサイドロードする |
| Windows ビルド配布 | Windows タブレット（react-native-windows） | 管理 Web からインストーラーをダウンロードし、タブレットで実行する |

MNT-021: タブレット APP の更新は以下の手順で実施する。

1. IT 担当がビルドサーバー（またはローカル開発環境）で新バージョンのビルドを実施する
2. ビルド成果物（APK / インストーラー）を管理 Web のファイル配布エリアに配置する
3. 現場監督に更新通知（更新内容・更新対象バージョン・更新期限）を送信する
4. 現場監督が各タブレットで更新作業を実施する（IT 担当の立会いは必須としないが推奨する）
5. 更新完了後、現場監督が更新完了報告を本システムの通知機能でIT担当に送信する

MNT-022: 更新の強制適用は禁止する。更新は現場監督の管理のもとで計画的に実施する。更新中も作業が継続できるよう、ライン別に順次更新を実施することを推奨する。

### 2-2. ダウングレード手順

MNT-023: 更新後に重大な不具合が発生した場合のダウングレード手順を確定する。

1. 現場監督が IT 担当にダウングレード要請を行う
2. IT 担当が管理 Web の旧バージョン APK（1 世代前のみ保存する）を現場監督に提供する
3. 現場監督が各タブレットで旧バージョン APK をサイドロードする
4. ダウングレード完了後、IT 担当はダウングレードの原因と再発防止策を GitHub Issues にチケットとして起票する

MNT-024: 旧バージョン APK は 1 世代前のみを管理 Web の配布エリアに保持する。2 世代以上前の APK は保持しない（バージョン互換性の複雑化を防止するため）。

**本節で確定した方針**
- タブレット APP の更新をサイドロード方式（Android）とインストーラー方式（Windows）で確定する。
- 強制更新を禁止し、現場監督管理のもとでのライン別順次更新を推奨する。
- ダウングレード対象を 1 世代前のみに限定し、2 世代以上前の APK 保持を禁止することを確定する。

## 3. DB マイグレーション要件

### 3-1. マイグレーションの実施方針

MNT-025: DB マイグレーション（PostgreSQL スキーマ変更）は以下の方針で実施する。

| 原則 | 内容 |
|---|---|
| 無停止マイグレーション優先 | カラム追加・インデックス追加等の無停止で実施可能なマイグレーションを優先設計する |
| 停止必要な場合の制約 | カラム削除・型変更等の停止必要なマイグレーションは計画停止枠（OPS-024）内で実施する |
| ロールバック必須 | すべてのマイグレーションスクリプトに Down マイグレーション（ロールバック用）を必須で実装する |
| ツールの固定 | マイグレーションツールは `sqlx migrate` を使用する。複数ツールの混用を禁止する |

MNT-026: マイグレーションスクリプトは以下の命名規則で管理する。

```
migrations/
  YYYYMMDDHHMMSS_<説明>.up.sql   （適用）
  YYYYMMDDHHMMSS_<説明>.down.sql （ロールバック）
```

MNT-027: マイグレーション実施前に以下の事前確認を必須とする。

| 確認項目 | 確認方法 |
|---|---|
| ステージング環境での事前適用 | 本番適用前にステージング環境で同一スクリプトを適用し、エラーがないことを確認する |
| Down マイグレーションの動作確認 | ステージング環境で Up → Down → Up の順で適用し、ロールバックが正常に動作することを確認する |
| バックアップの完了確認 | 適用直前の PostgreSQL バックアップが取得完了していることを確認する |
| 実行中トランザクションの確認 | マイグレーション開始前に長時間実行中のトランザクションがないことを確認する |

### 3-2. ロールバック手順

MNT-028: マイグレーション後に重大な問題が発生した場合のロールバック手順を確定する。

1. IT 担当が問題を検知した時点で現場監督・品質担当に通知する
2. `sqlx migrate revert` コマンドで直前の Down マイグレーションを適用する
3. ロールバック完了後、アプリケーションの正常動作を確認する
4. ロールバックの原因と再発防止策を GitHub Issues にチケットとして起票する

MNT-029: ロールバックが失敗した場合（Down スクリプトでエラーが発生する等）は、マイグレーション前のバックアップから PostgreSQL を復元する手順（OPS-081 の復旧確認手順を参照）を実施する。

**本節で確定した方針**
- 無停止マイグレーション優先・停止必要は計画停止枠内・ロールバック必須・sqlx migrate 固定の 4 方針を確定する。
- マイグレーション前の 4 項目事前確認（ステージング適用・Down 動作確認・バックアップ確認・長時間 TX 確認）を必須とする。
- ロールバック失敗時のバックアップ復元手順をフォールバックとして確定する。

## 4. 変更管理との連動

### 4-1. ADR・チケット起票の必須要件

MNT-030: 以下のソフトウェア変更は ADR（Architecture Decision Record）の起票を必須とする。

| 変更の種類 | ADR 起票の要否 |
|---|---|
| アーキテクチャ上の重要な設計変更 | 必須 |
| 技術スタックの追加・削除・バージョンの大幅変更 | 必須 |
| 認証・認可・暗号化の変更 | 必須 |
| データモデルの破壊的変更（カラム削除・型変更等） | 必須 |
| SLO の数値変更 | 必須 |
| セキュリティパッチの適用（High 以上） | 任意（変更ログとして記録する） |
| バグ修正・マイナー改善 | 不要（GitHub Issues チケットで管理する） |

MNT-031: GitHub Issues チケットの起票は以下の場合に必須とする。

| 起票必須の場合 | 内容 |
|---|---|
| セキュリティパッチ適用（High 以上） | CVE ID・重大度・適用予定日を含める |
| バグ修正 | バグの再現手順・影響範囲・修正内容を含める |
| 機能改善 | 改善の目的・変更内容・テスト要件を含める |
| ポストモーテムの再発防止策 | 元のインシデント記録への参照リンクを含める |

### 4-2. リリースノートの作成要件

MNT-032: 本番環境への変更デプロイ時は、以下の内容を含むリリースノートを作成する。リリースノートは Git リポジトリの `CHANGELOG.md` に追記する。

| 記載項目 | 内容 |
|---|---|
| バージョン番号 | セマンティックバージョニング（`vMAJOR.MINOR.PATCH`）に従う |
| リリース日時 | デプロイ完了の UTC 日時 |
| 変更内容 | Added / Changed / Fixed / Security の 4 カテゴリに分けて記載する |
| 影響を受ける機能 | 変更により影響を受ける機能・API の一覧 |
| マイグレーション情報 | DB スキーマ変更がある場合はマイグレーションファイル名を記載する |

MNT-033: セマンティックバージョニングの運用ルールを以下の通り確定する。

| バージョン変更 | 適用条件 |
|---|---|
| MAJOR（vX.0.0） | 後方互換性のない変更（API 破壊・DB スキーマの破壊的変更） |
| MINOR（v1.X.0） | 後方互換性のある機能追加 |
| PATCH（v1.0.X） | バグ修正・セキュリティパッチ（機能変更なし） |

**本節で確定した方針**
- アーキテクチャ変更・スタック変更・認証変更・DB 破壊的変更・SLO 変更の 5 種に ADR 起票を必須とする。
- セマンティックバージョニングを採用し、リリースノートの 5 必須項目を CHANGELOG.md に追記することを確定する。
- GitHub Issues チケットの起票必須ケースと記載内容を確定する。
