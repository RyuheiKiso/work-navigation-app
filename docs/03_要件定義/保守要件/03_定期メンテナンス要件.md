# 03 定期メンテナンス要件

本書の責務は、本システムの長期安定稼働を支えるための定期メンテナンス要件を確定することである。バックアップ検証・鍵ローテーション・ログ管理・OSS 依存追跡の 4 領域について、実施頻度・手順・完了基準を要件として確定する。

---

## 1. 定期バックアップ検証

### 1-1. バックアップ検証の実施要件

MNT-036: PostgreSQL のバックアップは毎日深夜に自動実行する。バックアップの自動実行に加えて、月 1 回のリストア演習を必須として実施する。リストア演習の未実施が 2 ヶ月以上連続した場合はアラートを発生させる。

MNT-037: リストア演習は以下の手順で実施する。

1. ステージング環境の PostgreSQL を停止し、データを削除する
2. 直近 1 週間以内の任意のバックアップファイルをステージング環境に復元する
3. 復元後に整合性チェックコマンドを実行し、全項目 OK を確認する
4. RTO（1 時間以内）に復元が完了したことを確認する
5. 演習結果（実施日時・使用バックアップの日時・復元所要時間・整合性チェック結果）を本システムのメンテナンス記録として保存する

MNT-038: リストア演習でのみ確認できる事項として以下を確定する。

| 確認項目 | 合格基準 |
|---|---|
| 復元所要時間 | RTO 1 時間以内での復元完了 |
| 整合性チェック | 全チェック項目 OK（エラー 0 件） |
| 代表 SOP のナビゲーション | 復元後に代表 SOP が正常表示されること |
| audit_log の連続性 | 復元後の audit_log に欠番がないこと（RPO 15 分の検証） |

**本節で確定した方針**
- 月 1 回のリストア演習を必須として確定し、未実施 2 ヶ月連続でアラートを発生させる要件を確定する。
- リストア演習の 5 ステップ手順と 4 項目合格基準（RTO 1 時間・整合性 OK・ナビゲーション確認・audit_log 連続性）を確定する。
- 演習結果の本システムへの記録保存を必須として確定する。

## 2. 鍵ローテーション

### 2-1. JWT 署名鍵のローテーション要件

MNT-039: JWT 署名鍵（RS256）を 90 日ごとにローテーションすることを必須とする。（HS256 は NFR-SEC により禁止）ローテーション未実施が 90 日を超過した場合はアラートを発生させる。

MNT-040: JWT 署名鍵のローテーション手順を以下の通り確定する。

1. 新しい鍵ペアを生成する
2. 新しい鍵ペアで署名されたトークンの検証が成功することをステージング環境で確認する
3. 本番環境の鍵設定を新しい鍵ペアに切り替える（設定ファイルまたは環境変数を更新する）
4. 旧鍵で署名されたトークンが無効化されていることを確認する（ローテーション後 24 時間は旧鍵を猶予期間として有効にする）
5. 猶予期間（24 時間）終了後に旧鍵を完全に無効化する
6. ローテーション完了記録を本システムのメンテナンス記録に保存する

MNT-041: 電子署名に使用するサーバー HMAC-SHA256 鍵も 90 日ごとにローテーションする。ただし HMAC 鍵のローテーションはサーバー再起動を不要とする実装（環境変数での管理）を要件とする。

### 2-2. その他の鍵・シークレットの管理

MNT-042: データベース接続パスワード・API キー等のシークレットは以下の管理要件を満たす。

| 要件 | 内容 |
|---|---|
| 環境変数管理 | シークレットをソースコードにハードコーディングすることを禁止する |
| ローテーション頻度 | データベース接続パスワードは年 1 回以上ローテーションする |
| アクセス記録 | シークレットへのアクセスをログに記録する |

**本節で確定した方針**
- JWT 署名鍵・HMAC-SHA256 鍵の 90 日ごとローテーションを必須として確定する。
- ローテーション手順（6 ステップ・24 時間猶予期間付き旧鍵無効化）を確定する。
- シークレットのハードコーディング禁止と環境変数管理を必須要件として確定する。

## 3. ログのアーカイブと廃棄

### 3-1. ログ保存期限の確定

MNT-043: 本システムが生成するログの保存期限を以下の通り確定する。

| ログ種別 | 保存期限 | 根拠 |
|---|---|---|
| audit_log（作業員の Step 記録・電子署名） | 本番稼働後 7 年以上 | ALCOA+ Enduring 原則・製造業の一般的な品質記録保存要件 |
| application_log（エラー・情報ログ） | 1 年 | システム障害の事後分析に必要な期間として設定する |
| request_metrics（応答時間計測） | 3 ヶ月 | SLO 計測・パフォーマンスチューニングに必要な期間として設定する |
| health_check_log（可用性計測） | 3 ヶ月 | 月次 SLO レポートの作成に必要な期間として設定する |
| outbox_log（送信記録） | 1 年 | 送信失敗の追跡と監査に必要な期間として設定する |

### 3-2. アーカイブと廃棄の手順

MNT-044: ログの保存期限到来時のアーカイブと廃棄は以下の手順で実施する。

1. IT 担当が毎月のメンテナンス作業として、保存期限到来ログの一覧を確認する
2. audit_log の 7 年経過データは廃棄前に品質担当の承認を得る
3. 品質担当承認後、廃棄対象ログを外部ストレージ（USB ハードディスクまたは社内 NAS）にアーカイブする
4. アーカイブ完了を確認してから本システムのデータベースからログを削除する
5. 廃棄記録（廃棄日時・廃棄対象ログの件数・期間・アーカイブ場所・品質担当承認署名）を本システムのメンテナンス記録に保存する

MNT-045: application_log・request_metrics・health_check_log・outbox_log は期限到来時に自動廃棄することを認める。自動廃棄の実行は深夜バッチで処理し、廃棄件数をログに記録する。

**本節で確定した方針**
- audit_log は 7 年以上・application_log は 1 年・metrics は 3 ヶ月の保存期限を確定する。
- audit_log 廃棄時の品質担当承認・アーカイブ後削除の手順と廃棄記録の 5 必須項目を確定する。
- application_log 以下の自動廃棄を認め、廃棄件数の自動記録を要件とする。

## 4. OSS 依存の EOL 追跡

### 4-1. EOL 追跡の実施要件

MNT-046: 本システムが依存する OSS（Operating System・Rust クレート・npm パッケージ・Docker ベースイメージ）の End of Life（EOL）を年 2 回（6 月・12 月）に棚卸しすることを必須とする。

MNT-047: 年次 OSS 棚卸しの確認対象を以下の通り確定する。

| 確認対象 | 確認内容 |
|---|---|
| Windows Server 2022 | マイクロソフトのサポートライフサイクルページで EOL 日を確認する |
| Rust エディション（2024） | Rust Edition のサポート状況を確認する |
| axum・tokio・sqlx の主要クレート | crates.io と GitHub リポジトリでメンテナンス状況を確認する |
| React Native のメジャーバージョン | React Native のリリースサポートスケジュールを確認する |
| PostgreSQL メジャーバージョン | PostgreSQL の EOL スケジュール（postgresql.org）を確認する |
| Docker ベースイメージ | 使用するベースイメージ（debian / alpine）のサポート状況を確認する |

### 4-2. EOL 対応の期限と手順

MNT-048: EOL 到来が確認された OSS コンポーネントは、EOL 日から 6 ヶ月以内にアップグレードを完了させることを必須とする。アップグレードの実施は ADR を起票して変更根拠を記録する。

MNT-049: EOL 棚卸しの結果は以下の内容を含む棚卸し報告書として本システムのメンテナンス記録に保存する。

| 報告項目 | 内容 |
|---|---|
| 棚卸し実施日 | UTC 日付 |
| 確認対象コンポーネント一覧 | コンポーネント名・現在バージョン・EOL 日 |
| 対応必要なコンポーネント | EOL 6 ヶ月以内のコンポーネントと対応期限 |
| 対応計画 | アップグレード方針・担当者・期限 |

**本節で確定した方針**
- OSS 依存の EOL 棚卸しを年 2 回（6 月・12 月）必須として確定し、確認対象 6 カテゴリを確定する。
- EOL 到来コンポーネントの 6 ヶ月以内アップグレードと ADR 起票を必須として確定する。
- 棚卸し報告書の 4 必須項目を確定し、本システムへの記録保存を義務付ける。
