# 04 技術的負債管理と ADR 運用要件

本書の責務は、本システムの長期的な保守可能性を確保するための技術的負債の可視化・ADR の運用・コードレビューと品質ゲートの要件を確定することである。個人開発における技術的負債の蓄積は Bus Factor = 1 のリスクと掛け合わさり、プロジェクト存続に直結する問題である。構造的に管理する仕組みを確定する。

---

## 1. 技術的負債の可視化要件

### 1-1. 技術的負債の記録形式

MNT-050: 技術的負債を GitHub Issues のラベル `tech-debt` で管理する。技術的負債チケットには以下の必須項目を含める。

| 必須項目 | 内容 |
|---|---|
| 負債の説明 | 現在の実装の問題点と理想の実装との差異を具体的に記述する |
| 発生理由 | なぜその負債が生まれたか（速度優先・知識不足・技術的制約等）を記録する |
| 影響範囲 | 負債が影響するコンポーネント・機能・リスクの範囲を記述する |
| 返済コスト見積もり | 負債を解消するために必要な開発工数の概算（時間単位）を記載する |
| 優先度 | High / Medium / Low の 3 段階で設定する |

### 1-2. 優先度付け基準

MNT-051: 技術的負債の優先度は以下の基準に従って設定する。

| 優先度 | 基準 | 例 |
|---|---|---|
| High | セキュリティリスク・データ整合性リスク・SLO 達成を妨げるパフォーマンスリスクを含む負債 | 非効率なクエリによる応答時間劣化・未検証の依存ライブラリ |
| Medium | 保守性・可読性を著しく損なう負債（次の機能開発時に修正コストが増大する） | 重複ロジック・テストカバレッジの低いモジュール |
| Low | 改善すれば良いが緊急性の低い負債 | コメントの不足・命名規約の不一致 |

MNT-052: 技術的負債の棚卸しを四半期（3 ヶ月）ごとに実施する。棚卸しでは既存チケットの優先度見直し・返済済み負債のクローズ・新規発見負債の起票を行う。棚卸し結果は `CHANGELOG.md` の `[Technical Debt]` セクションに記録する。

**本節で確定した方針**
- 技術的負債を GitHub Issues の tech-debt ラベルで管理し、5 必須項目と 3 段階優先度基準を確定する。
- 四半期ごとの棚卸しを必須として確定し、結果を CHANGELOG.md に記録することを義務付ける。
- セキュリティ・データ整合性・SLO 達成妨害を優先度 High の必須基準として確定する。

## 2. ADR（Architecture Decision Record）運用要件

### 2-1. ADR の起票要件

MNT-053: ADR は重要なアーキテクチャ決定を行った際に起票する。保守要件 02 章（MNT-030）で定めた ADR 必須起票ケースに加えて、以下のケースでも起票する。

| 起票ケース | 内容 |
|---|---|
| 代替案を評価して最終選択した場合 | 選択した理由と棄却した代替案の理由を記録する |
| 外部ライブラリの採用・廃止 | 採用・廃止の根拠と影響を記録する |
| パフォーマンス改善のために設計を変更した場合 | 変更前後の計測値と変更根拠を記録する |
| セキュリティ上の判断（脆弱性の受容・対策選択） | 判断根拠と受容リスクの範囲を記録する |

MNT-054: ADR のステータスを以下の 4 状態で管理する。

| ステータス | 定義 |
|---|---|
| Proposed | 起票済みだが承認（採用決定）していない |
| Accepted | 採用決定済み。この ADR に従って実装を進める |
| Deprecated | 過去に採用されたが、より良い決定に置き換えられた |
| Superseded by ADR-NNN | 別の ADR（ADR 番号 NNN）によって置き換えられた |

### 2-2. ADR の更新・廃止手順

MNT-055: Accepted ステータスの ADR を更新・廃止する場合は以下の手順を遵守する。

1. 既存 ADR のステータスを Deprecated または Superseded by ADR-NNN に変更する
2. 新しい ADR を起票し、変更前の ADR への参照リンクを含める
3. 新 ADR を Accepted ステータスに設定する
4. 既存 ADR の内容を上書き変更することは禁止する（履歴として保持する）

MNT-056: ADR ファイルの管理場所・命名規則を以下の通り確定する。

```
docs/adr/
  ADR-0001_rust_edition_2024採用.md
  ADR-0002_offline_first_sqlite.md
  ADR-NNNN_<概要>.md
```

ADR 番号は 4 桁の連番で管理し、番号の欠番を禁止する。

**本節で確定した方針**
- ADR の起票必須ケースを代替案評価・ライブラリ採用廃止・パフォーマンス変更・セキュリティ判断の 4 種に拡張して確定する。
- 4 ステータス管理（Proposed / Accepted / Deprecated / Superseded）と既存 ADR 上書き禁止を確定する。
- ADR ファイルの配置場所・命名規則（ADR-NNNN 形式）と番号欠番禁止を確定する。

## 3. コードレビューと品質ゲート要件

### 3-1. 静的解析ツールの必須実行

MNT-057: Rust バックエンドコードへの以下のツール実行を全コミット前に必須とする。

| ツール | 実行コマンド | 合格基準 |
|---|---|---|
| rustfmt | `cargo fmt --check` | フォーマットエラーゼロ |
| clippy | `cargo clippy -- -D warnings` | 警告ゼロ（警告をエラーとして扱う） |
| cargo test | `cargo test --workspace` | 全テスト合格 |
| cargo audit | `cargo audit` | High 以上の脆弱性ゼロ |

MNT-058: React Native（タブレット APP）および React（管理 Web）への以下のツール実行を全コミット前に必須とする。

| ツール | 実行コマンド | 合格基準 |
|---|---|---|
| ESLint | `npm run lint` | エラーゼロ |
| TypeScript | `npm run typecheck` または `tsc --noEmit` | 型エラーゼロ |
| Jest | `npm test` | 全テスト合格 |

### 3-2. テストカバレッジ要件

MNT-059: テストカバレッジの閾値を以下の通り確定する。閾値を下回るコミットの本番ブランチへのマージを禁止する。

| 対象 | カバレッジ閾値 | 計測対象 |
|---|---|---|
| Rust バックエンド | ライン カバレッジ 70% 以上 | `cargo tarpaulin` で計測する |
| React Native APP | ステートメント カバレッジ 60% 以上 | Jest のカバレッジレポートで計測する |
| 管理 Web（React） | ステートメント カバレッジ 60% 以上 | Jest のカバレッジレポートで計測する |

MNT-060: 品質ゲートのチェックは GitHub Actions（CI）またはローカルの pre-commit フックで自動実行する。全チェック通過をマージの前提条件として確定する。

**本節で確定した方針**
- Rust は rustfmt・clippy（警告ゼロ）・cargo test・cargo audit の 4 ツール全通過をコミット前必須として確定する。
- フロントエンドは ESLint・TypeScript 型チェック・Jest の 3 ツール全通過を必須として確定する。
- Rust 70%・フロントエンド 60% のカバレッジ閾値を下回るマージを禁止することを確定する。
