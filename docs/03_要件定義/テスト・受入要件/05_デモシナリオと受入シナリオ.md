# 05 デモシナリオと受入シナリオ

本章の責務は、ver1.0.0 受入テストで実施するデモシナリオ・受入シナリオ（A〜E）を確定することである。各シナリオは業務要件サブのユースケース（BR-SC）と対応し、受入基準（TST-001〜007）の充足確認に直接使用する。

---

## 1. デモシナリオの目的

### 1-1. デモシナリオの位置づけ

デモシナリオは、受入テストの本格実施前に本システムの主要機能が業務シナリオとして機能することを確認するための「事前確認テスト」である。受入シナリオ A〜E は業務要件サブの主要 5 業務シナリオに対応し、システムが製造現場の実際の作業フローを支援できることを確認する。

デモシナリオと受入シナリオの関係:
- **デモシナリオ**: 立会いステークホルダーへのシステム機能の説明・確認のために使用する。本番前の最終確認として位置づける
- **受入シナリオ A〜E**: Pass/Fail を判定するための正式なテストシナリオ。各ステップに検証条件が付与される

デモシナリオは受入シナリオ A〜E と同じシナリオを使用する。デモシナリオでの動作確認は受入シナリオの Pass/Fail 判定には使用しない（別途、受入シナリオとして正式実施する）。

### 1-2. デモの実施条件

デモの実施条件として以下を確認してから実施する。

- ステージング環境がステージング環境要件（TST-045・TST-046）を満たして起動していること
- テストデータ（TST-052 シードスクリプト）が投入済みであること
- 本番と同一機種・OS バージョンのタブレット端末がステージング環境に接続済みであること

---

**本節で確定した方針**
- デモシナリオは受入シナリオ A〜E と同一内容とし、立会い確認に使用することを確定する。
- デモでの動作確認は受入シナリオの正式 Pass/Fail 判定には使用しないことを確定する。
- デモ実施前にステージング環境・テストデータ・タブレット端末の準備完了確認を必須とすることを確定する。

---

## 2. 受入シナリオ A: SOP ナビゲーション

### 2-1. シナリオの概要

**対応 UC**: UC-001〜UC-003（SOP 表示・ロックステップ進行・クリティカルステップ証拠要求）
**対応受入基準**: TST-001（FR 実装完了）・TST-002（UC 受入テスト合格）・TST-005（ALCOA+）
**推奨立会い者**: 作業員代表・現場監督代表

### 2-2. 前提条件

- テスト用マスタが投入済みであること（テスト工程「組立工程-A」・テスト作業「部品取付手順-01」が登録済み）
- テスト用ユーザー「テスト作業員 01（作業員ロール）」でログイン済みであること

### 2-3. シナリオステップと検証条件

| ステップ | 操作内容 | 検証条件（Pass/Fail 基準） |
|---|---|---|
| A-01 | 作業指示画面を開き、「組立工程-A / 部品取付手順-01」を選択する | 対象 SOP のタイトル・バージョン・改訂日が正しく表示される |
| A-02 | Step 1（通常ステップ）の内容を確認し、「完了」ボタンをタップする | Step 1 に完了タイムスタンプ・作業員 ID が記録される。SHA-256 ハッシュチェーンが前 Step と連続している |
| A-03 | Step 2（クリティカルステップ: 証拠写真必須）に遷移する | 「証拠写真が必要です」の案内が表示される。「完了」ボタンがグレーアウトして押せない |
| A-04 | カメラで証拠写真を撮影し添付する | 写真が記録に添付される。添付後に「完了」ボタンが活性化する |
| A-05 | Step 2 を完了する | Step 2 に完了タイムスタンプ・証拠写真添付確認が記録される |
| A-06 | Wi-Fi を意図的に切断した状態で Step 3 を完了する | Wi-Fi 断絶中でも Step 3 が完了できる。完了記録がローカル DB に保存される |
| A-07 | Wi-Fi を再接続する | ローカル DB の未同期記録が自動的にサーバーへ送信される。同期完了後、管理 Web からも Step 3 の記録が確認できる |
| A-08 | 全 Step を完了し、作業完了記録を確認する | 作業完了記録が作成される。全 Step の ALCOA+ 整合性チェック（SHA-256 連鎖・タイムスタンプ連続性）が Pass である |

### 2-4. 代替シナリオ・例外シナリオ

| シナリオ種別 | 内容 | 検証条件 |
|---|---|---|
| 代替 A-1 | Step 実行中にアプリを強制終了し、再起動する | 再起動後に「前回の作業を再開しますか？」の案内が表示され、中断した Step から再開できる |
| 例外 A-2 | クリティカルステップで証拠写真なしに「完了」ボタンを押そうとする | 「完了」ボタンがタップできない。エラーメッセージが表示される |
| 例外 A-3 | 権限のないユーザー（作業員ロール）が管理者限定の SOP 編集画面にアクセスしようとする | 403 エラーが返却される。管理画面にアクセスできない |

---

**本節で確定した方針**
- 受入シナリオ A を 8 ステップ（正常）＋ 3 シナリオ（代替・例外）として確定し、各ステップに Pass/Fail 基準を付与する。
- Wi-Fi 断絶時の継続動作確認（ステップ A-06〜A-07）を必須シナリオとして含めることを確定する。
- SHA-256 ハッシュチェーン整合性確認をシナリオ内の検証条件として含めることを確定する。

## 3. 受入シナリオ B: 証拠記録と電子サイン

### 3-1. シナリオの概要

**対応 UC**: UC-004〜UC-005（写真・測定値・QR スキャン記録・電子サイン）
**対応受入基準**: TST-002・TST-005（ALCOA+ Attributable・Original）
**推奨立会い者**: 品質担当代表

### 3-2. シナリオステップと検証条件

| ステップ | 操作内容 | 検証条件（Pass/Fail 基準） |
|---|---|---|
| B-01 | 測定値入力必須の Step（例: トルク締め付け値）を開く | 測定値入力フォームが表示される。数値の上下限が SOP マスタから読み込まれて表示される |
| B-02 | 測定値（範囲内）を入力して完了する | 測定値・単位・校正機器 ID が記録に含まれる。ALCOA+ Accurate 原則の記録となること |
| B-03 | 測定値（範囲外）を入力しようとする | 範囲外の値が入力されると警告が表示される。「注意: 規定範囲外の値です。再測定を実施してください」が表示される（値の記録は可能だが警告付きとなる） |
| B-04 | QR コードスキャン必須の Step で部品 QR コードをスキャンする | QR コードの読み取り値が記録に含まれる。スキャン結果がマスタの期待値と一致した場合は OK 表示となる |
| B-05 | 電子サインが必要な Step（現場監督承認）に遷移する | 「現場監督の承認が必要です」の案内が表示される |
| B-06 | 現場監督ロールのユーザーでログインして電子サインを実施する | 電子サイン（署名画像・署名者 ID・署名日時）が記録に付与される |
| B-07 | 品質担当が管理 Web のトレサビ照会から当該記録を検索する | ロット番号・品番・工程・作業員・タイムスタンプ・証拠写真・測定値・電子サインが一覧表示される |

### 3-3. 例外シナリオ

| シナリオ種別 | 内容 | 検証条件 |
|---|---|---|
| 例外 B-1 | 過去の記録を遡って測定値を書き換えようとする | Append-only 制約により更新・削除操作が拒否される。ALCOA+ Original 原則の技術的実装確認 |
| 例外 B-2 | 電子サインを取り消そうとする | 電子サインの取り消しは不可（Append-only）。取り消しが技術的に拒否されること |

---

**本節で確定した方針**
- 受入シナリオ B を 7 ステップ（正常）＋ 2 シナリオ（例外）として確定する。
- Append-only 制約の技術的確認（例外 B-1・B-2）を受入シナリオに含めることを確定する。
- 電子サインの署名者 ID・日時記録を検証条件として含めることを確定する。

---

## 4. 受入シナリオ C: マスタ改訂

### 4-1. シナリオの概要

**対応 UC**: UC-006〜UC-007（SOP Draft 編集・Publish・バージョン参照）
**対応受入基準**: TST-001（FR-MS 実装）・TST-002（UC 合格）
**推奨立会い者**: マスタ編集者代表・品質担当代表

### 4-2. シナリオステップと検証条件

| ステップ | 操作内容 | 検証条件（Pass/Fail 基準） |
|---|---|---|
| C-01 | マスタ編集者ロールでマスタ管理画面を開き、既存 SOP「部品取付手順-01 v1.0」を選択する | 現在の Published 版（v1.0）が表示される |
| C-02 | 「新バージョン作成」を選択し、Draft（v2.0-draft）を作成する | Draft が作成される。Published 版（v1.0）に影響なく Draft が並存する |
| C-03 | Draft の Step 3 の説明文を変更し、Draft を保存する | Draft の変更内容が保存される。Published 版（v1.0）の Step 3 は変更前のままである |
| C-04 | 作業員がタブレットでナビゲーション中の SOP バージョンを確認する | タブレットには Published 版（v1.0）が表示される。Draft は作業員に表示されない |
| C-05 | マスタ編集者が Draft を Published に昇格させる（Publish 操作）| Draft が v2.0 として Published になる。旧 Published 版（v1.0）は Archived（廃止履歴）に移行する |
| C-06 | Publish 後にタブレットで SOP を参照する | タブレットには v2.0 が表示される。v1.0 の内容と差異が確認できる |
| C-07 | 管理 Web のバージョン履歴から旧バージョン（v1.0）の内容を参照する | v1.0 の内容が削除されずに Archived として参照できる。ALCOA+ Enduring 原則の確認 |
| C-08 | v2.0 Publish 前に作業員が開始した作業記録（v1.0 に紐づく）の SOP バージョンを確認する | 作業記録には「SOP v1.0 を使用して記録」と明示される。バージョン変更後も旧記録の参照整合性が保たれる |

---

**本節で確定した方針**
- 受入シナリオ C を 8 ステップとして確定し、Draft/Published/Archived のライフサイクル全体を網羅する。
- Publish 前後での作業員への表示バージョンの分離（C-03〜C-04）を検証条件として確定する。
- 旧バージョンの Archived 保存と参照可能性（ALCOA+ Enduring）を検証条件として確定する。

---

## 5. 受入シナリオ D: アンドン・CAPA

### 5-1. シナリオの概要

**対応 UC**: UC-008〜UC-009（アンドン発報・ヒヤリハット・CAPA クローズ）
**対応受入基準**: TST-001（FR-AL 実装）・TST-002（UC 合格）
**推奨立会い者**: 現場監督代表・品質担当代表

### 5-2. シナリオステップと検証条件

| ステップ | 操作内容 | 検証条件（Pass/Fail 基準） |
|---|---|---|
| D-01 | 作業員がタブレットからアンドン発報（作業停止要請）を実施する | アンドン発報が記録される。発報者 ID・発報日時・事象内容が記録される |
| D-02 | 現場監督の管理 Web 画面にアンドン発報通知が表示される | 5 分以内に現場監督の画面にリアルタイム通知が表示される（ポーリング間隔内での通知確認） |
| D-03 | 現場監督が対応完了を記録する | 対応完了日時・対応者 ID が記録される |
| D-04 | 品質担当が同じ事象に対して CAPA を起票する | CAPA 記録（是正処置・予防処置）が作成される。アンドン発報記録と CAPA 記録がリンクされる |
| D-05 | 品質担当が CAPA のフォローアップ記録を追記する | CAPA にフォローアップ記録が Append-only で追加される |
| D-06 | 品質担当が CAPA をクローズする | CAPA のステータスが「クローズ」に変更される。クローズ日時・クローズ者 ID が記録される |
| D-07 | 管理 Web の CAPA 一覧でクローズされた CAPA を検索する | 発報からクローズまでの全履歴（アンドン発報→対応完了→CAPA 起票→フォローアップ→クローズ）がトレースできる |

---

**本節で確定した方針**
- 受入シナリオ D をアンドン発報から CAPA クローズまでの 7 ステップとして確定する。
- アンドン発報通知の 5 分以内表示を検証条件として確定する。
- 発報から CAPA クローズまでの全履歴のトレーサビリティを最終検証条件として確定する。

---

## 6. 受入シナリオ E: Outbox 同期

### 6-1. シナリオの概要

**対応 UC**: UC-010（Outbox Pattern によるオフライン→復旧同期）
**対応受入基準**: TST-007（Outbox 送受信正常確認）
**推奨立会い者**: IT 担当代表

### 6-2. シナリオステップと検証条件

| ステップ | 操作内容 | 検証条件（Pass/Fail 基準） |
|---|---|---|
| E-01 | タブレットをサーバーから切断した状態（Wi-Fi off または VPN 切断）で作業を開始する | オフライン状態の表示が UI 上で確認できる（「オフラインモード: 記録はローカルに保存されます」等の案内） |
| E-02 | オフライン状態で Step 1〜5 を完了する | 全 Step の記録がタブレットのローカル DB に保存される。サーバーには未送信の状態 |
| E-03 | ローカル DB の Outbox キューに 5 件の未送信記録が存在することを確認する | タブレットのデバッグ画面または管理 Web の「同期状況」画面から Outbox キューが 5 件であることを確認できる |
| E-04 | Wi-Fi を再接続する | 再接続後 60 秒以内に Outbox キューの自動送信が開始される |
| E-05 | Outbox キューの全件送信が完了したことを確認する | 管理 Web の「同期状況」画面で Outbox キューが 0 件になったことを確認できる。送信完了日時が記録される |
| E-06 | サーバー側 DB で受信した記録を確認する | サーバー DB に 5 件の記録が存在する。各記録の内容（Step ID・作業員 ID・タイムスタンプ・証拠写真）がローカル DB の記録と一致している |
| E-07 | 同一の Outbox 記録を 2 回送信するシナリオを実施する（ネットワーク再送テスト）| Idempotency Key により、同一記録がサーバー DB に 2 回登録されないことを確認する。サーバーの記録件数は 5 件のまま |
| E-08 | 大量 Outbox シナリオ（50 件蓄積後の一括送信）を実施する | 50 件全件がサーバーに正常到達する。欠損件数ゼロ・重複件数ゼロ |

### 6-3. 例外シナリオ

| シナリオ種別 | 内容 | 検証条件 |
|---|---|---|
| 例外 E-1 | 送信中にネットワークが再度切断される | 再接続後に残りの未送信記録が続きから送信される。二重送信なし |
| 例外 E-2 | サーバーが起動していない状態で送信を試みる | エラーが UI に表示される。ローカル DB の記録は保持され、サーバー復旧後に自動送信が再開する |

---

**本節で確定した方針**
- 受入シナリオ E を 8 ステップ（正常）＋ 2 シナリオ（例外）として確定する。
- Idempotency Key による重複防止の動作確認（E-07）を必須ステップとして確定する。
- 大量 Outbox シナリオ（50 件）での全件到達・欠損ゼロ・重複ゼロを確認することを確定する。
