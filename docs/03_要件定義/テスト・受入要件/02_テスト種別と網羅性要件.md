# 02 テスト種別と網羅性要件

本章の責務は、ver1.0.0 リリースまでに実施するすべてのテストを種別・目的・責任者・ツール・カバレッジ要件の観点で確定することである。テスト戦略の基本方針は計画 14 章テスト戦略（テストピラミッド・6 レベル定義）を継承し、本章でその要件詳細を確定する。

---

## 1. テストピラミッドと 6 レベル

### 1-1. テストピラミッドの方針

本プロジェクトのテスト戦略はテストピラミッドを採用する。ピラミッドの比率はユニットテスト > 統合テスト >> E2E テストとし、E2E テストの過剰実装による個人開発のボトルネック化を防ぐ。

6 テストレベルの全体像を以下に示す。

図: fig_test_phase_flow（img/ 配下）を参照。

| レベル | テスト種別 | 責任者 | ツール（Rust） | ツール（React/RN） |
|---|---|---|---|---|
| L1 | 単体テスト（Unit Test） | 開発者本人 | `cargo test` | Jest |
| L2 | 統合テスト（Integration Test） | 開発者本人 | `sqlx test` + テスト用 DB | Detox / React Testing Library |
| L3 | システムテスト（System Test） | 開発者本人 | E2E: Detox（タブレット）/ Playwright（管理 Web） | 同左 |
| L4 | 性能テスト（Performance Test） | 開発者本人 | k6 | — |
| L5 | セキュリティテスト（Security Test） | 開発者本人（外部専門家推奨） | `cargo audit` / OWASP ZAP | — |
| L6 | 受入テスト（Acceptance Test） | 開発者本人（セルフ受入） | 手動シナリオ + 本章 05 節の受入シナリオ | — |

### 1-2. TST-020: 単体テスト要件

| 要件 ID | TST-020 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 14 章テスト戦略 / NFR-MNT サブ |

**定義**: 単体テストは関数・メソッド・コンポーネント単位の正確性を確認するテストである。外部依存（DB・ネットワーク・ファイルシステム）はモックに置き換える。

**責任者**: 開発者本人。各モジュールの実装完了時に作成・実行する。

**ツール要件**:
- Rust: `cargo test`。`#[cfg(test)]` モジュール内にテストを配置する。外部依存はトレイトのモック実装で置き換える
- React / React Native: Jest。`@testing-library/react`・`@testing-library/react-native` を使用する

**実行タイミング**: 実装コードの変更のたびに実行する。CI/CD パイプラインに組み込み、テスト失敗時はコミットをブロックする。

### 1-3. TST-021: 統合テスト要件

| 要件 ID | TST-021 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 14 章テスト戦略 / FR-SY サブ |

**定義**: 統合テストは API・DB・外部連携の結合を確認するテストである。テスト用 DB・テスト用親機モックを使用し、本番環境と接続しない。

**責任者**: 開発者本人。M3（バックエンド API 完成）以降に実施する。

**ツール要件**:
- Rust: `sqlx test` マクロを使用した DB 統合テスト。テスト用 PostgreSQL インスタンスを Docker Compose で起動し、マイグレーション適用済みの状態でテストを実行する
- 親機連携テスト: OpenAPI スキーマ駆動テスト（Dredd または同等ツール）で親機モックとの契約を確認する

**実行タイミング**: M3 以降の全マイルストーン完了時に実施する。

### 1-4. TST-022: システムテスト要件

| 要件 ID | TST-022 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 04 章ユースケース / 業務要件 BR-SC サブ |

**定義**: システムテストはユーザーシナリオ全体の動作をエンドツーエンドで確認するテストである。ステージング環境（本番同等構成）で実施する。

**責任者**: 開発者本人。M4（タブレット APP 完成）・M5（管理 Web 完成）以降に実施する。

**ツール要件**:
- タブレット APP: Detox によるE2E テスト。全 UC（UC-001〜UC-XXX）の正常シナリオを対象とする
- 管理 Web: Playwright によるE2E テスト。SOP 管理・CAPA・ユーザー管理・トレサビ照会・ダッシュボードの正常シナリオを対象とする

**E2E テストの対象制約**: E2E テストは Happy Path（正常シナリオ）と主要エラーシナリオ（5 件以内）のみを対象とする。全シナリオの E2E 自動化は個人開発のボトルネックとなるため実施しない。例外シナリオの確認は手動テストで実施する。

### 1-5. TST-023: 性能テスト要件

| 要件 ID | TST-023 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-PRF サブ / 計画 14 章カオスシナリオ |

**定義**: 性能テストは NFR-PRF で確定したレスポンスタイム・スループット目標値の達成を確認するテストである。負荷テスト（通常負荷）とスパイクテスト（突発的高負荷）の 2 種を実施する。

**責任者**: 開発者本人。M5（管理 Web 完成）以降に実施する。

**ツール要件**: k6（JavaScript ベース負荷テストツール）を使用する。テストスクリプトはリポジトリの `tests/performance/` に管理する。

**必須シナリオ**:

| シナリオ種別 | 概要 | 合格基準 |
|---|---|---|
| 通常負荷テスト | 同時接続 10 名・60 分継続 | P99 レスポンスタイムが NFR-PRF 目標値以下 |
| スパイクテスト | 0 から 30 名に 1 分で増加・5 分継続 | エラー率 1% 以下・P99 が目標値の 3 倍以下 |
| カオステスト: 親機停止 | 親機を停止した状態で子機作業を 30 分継続 | Step 記録の成功率 100% |
| カオステスト: 大量 Outbox | 100 件の実績を一括送信 | 欠損なく全件親機到達・重複ゼロ |

### 1-6. TST-024: セキュリティテスト要件

| 要件 ID | TST-024 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-SEC サブ / 計画 15 章セキュリティ深堀り |

**定義**: セキュリティテストは NFR-SEC で確定したセキュリティ要件の充足を確認するテストである。脆弱性スキャン・RBAC 検証・JWT 検証の 3 種を実施する。

**責任者**: 開発者本人。外部専門家によるセキュリティレビューを推奨する（必須ではない。実施した場合はレビュー記録を保存する）。

**ツール要件**:
- 依存ライブラリスキャン: `cargo audit`（Rust）。CVSS 7.0 以上の CVE がゼロであること
- Web アプリスキャン: OWASP ZAP（または同等ツール）。High 評価の脆弱性がゼロであること
- RBAC テスト: 全ロール × 全操作の組み合わせについて許可/拒否が正しいことを確認するテストケースを手動で作成・実行する

### 1-7. TST-025-UAT: 受入テスト要件

| 要件 ID | TST-025-UAT |
|---|---|
| 優先度 | Must |
| 上流リンク | 本章 TST-001〜007 / 本章 05 節デモ・受入シナリオ |

**定義**: 受入テストは 01 章の受入基準（TST-001〜007）を確認するための最終テストである。本章 05 節の受入シナリオ A〜E に従って手動で実施する。

**責任者**: 開発者本人（セルフ受入）。主要ステークホルダー（工場 IT 担当・現場監督・品質担当の代表者）の立会いを推奨する（立会い者がいる場合はレビュー記録に記録する）。

---

**本節で確定した方針**
- テストピラミッドを採用し、6 テストレベル（L1〜L6）の定義・責任者・ツール要件を確定する。
- E2E テストの対象を Happy Path と主要エラーシナリオ 5 件以内に制限し、個人開発のボトルネック化を防ぐことを確定する。
- セキュリティテストは開発者本人が実施し、外部専門家レビューは推奨（必須ではない）とすることを確定する。

---

## 2. カバレッジ要件

### 2-1. TST-026: 単体テスト行カバレッジ要件

| 要件 ID | TST-026 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-MNT サブ / 計画 10 章ドキュメント先行原則 |

単体テストの行カバレッジ（Line Coverage）は以下の基準を満たすこと。

- Rust バックエンド: `cargo tarpaulin`（または同等ツール）で計測した行カバレッジが 80% 以上であること
- React Native（タブレット APP）: Jest の coverage オプションで計測した行カバレッジが 80% 以上であること
- React（管理 Web）: Jest の coverage オプションで計測した行カバレッジが 80% 以上であること

カバレッジ計測からの除外対象: `prototype/`・`poc/` ディレクトリ・自動生成コード（SQLx クエリマクロ展開等）はカバレッジ計測の対象外とする。

計測・報告頻度: M3・M4・M5 の各マイルストーン完了時にカバレッジレポートを生成し、達成状況を受入テスト実施記録に記録する。

### 2-2. TST-027: UC カバレッジ要件

| 要件 ID | TST-027 |
|---|---|
| 優先度 | Must |
| 上流リンク | 業務要件 BR-SC サブ / 計画 04 章ユースケース |

全ユースケース（UC-001〜UC-XXX）の 100% がシステムテスト（L3）または受入テスト（L6）のいずれかでテストされていること。1 件でも未テストの UC がある場合は UC カバレッジ 100% を達成していないとみなす。

UC カバレッジの集計: 受入テスト実施記録に「UC-ID × テスト実施有無 × Pass/Fail」の一覧表を記録し、UC カバレッジ率を計算する。

### 2-3. TST-028: FR カバレッジ要件

| 要件 ID | TST-028 |
|---|---|
| 優先度 | Must |
| 上流リンク | 機能要件サブ全体 |

Must・Should 優先度の全機能要件（FR-NV・FR-TR・FR-SY・FR-MS・FR-AL・FR-EX）の 100% がいずれかのテストレベル（L1〜L6）でテストされていること。

テストと FR の紐付け: 各テストケース（または受入シナリオのステップ）に「対応する FR-ID」を記録し、FR カバレッジを追跡可能な状態にする。

---

**本節で確定した方針**
- 行カバレッジ 80% 以上（Rust・React Native・React 全体）を Must 要件として確定する。
- UC カバレッジ 100%・FR カバレッジ 100%（Must/Should）を受入テスト合格の前提として確定する。
- カバレッジ達成状況は M3・M4・M5 の各マイルストーン完了時に記録することを確定する。

---

## 3. 回帰テスト要件

### 3-1. TST-030: 変更時の回帰テスト実行範囲

| 要件 ID | TST-030 |
|---|---|
| 優先度 | Must |
| 上流リンク | プロジェクト要件 PRJ-035〜PRJ-040（変更管理） |

コードの変更が発生した場合、変更の影響範囲に応じて以下の回帰テストを実施する。

| 変更種別 | 実施する回帰テストの範囲 |
|---|---|
| 関数・メソッドの変更 | 変更した関数・メソッドの単体テスト全件（L1） |
| API エンドポイントの変更 | 変更したエンドポイントの統合テスト（L2）+ 関連する UC のシステムテスト（L3） |
| DB スキーマの変更 | 全統合テスト（L2）+ 全システムテスト（L3）+ 移行テスト |
| 認証・認可ロジックの変更 | 全セキュリティテスト（L5）+ 全統合テスト（L2） |
| マスタ管理ロジックの変更 | 関連する統合テスト（L2）+ ALCOA+ 検証テスト（TST-005） |
| Outbox / 同期ロジックの変更 | FR-SY 関連の統合テスト（L2）+ TST-007 シナリオ全件 |

### 3-2. TST-031: 回帰テストの自動化要件

| 要件 ID | TST-031 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-MNT サブ |

以下のテストは CI/CD パイプライン（GitHub Actions または同等ツール）で自動実行する。

- L1 単体テスト（全件）: 全プッシュ・全プルリクエストで自動実行する
- L2 統合テスト（全件）: main ブランチへのマージ時に自動実行する
- カバレッジレポート生成: L1 自動実行時にカバレッジを計測し、下限未達の場合はパイプラインを失敗させる

L3〜L6 の自動化は個人開発のコスト制約から必須としない。ただし L3（Detox E2E）は各マイルストーン完了時に手動で実施し、その記録を保存する。

---

**本節で確定した方針**
- 変更種別に応じた回帰テスト実行範囲（TST-030）を確定し、影響範囲が広い変更ほど広範な再テストを必須とする。
- L1 単体テストと L2 統合テストの CI/CD 自動実行を Must 要件として確定する。
- L3〜L6 の自動化は任意とし、各マイルストーン完了時の手動実施・記録保存を確定する。

---

## 4. 非機能テスト要件

### 4-1. TST-035: 負荷テスト実施要件

| 要件 ID | TST-035 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-PRF サブ / 計画 14 章テスト戦略 |

負荷テストは M5 完了後・M6 受入テスト実施前に、ステージング環境（本番同等構成）で実施する。

実施するテストシナリオ:
- 通常負荷テスト（同時 10 名・60 分継続）
- スパイクテスト（急増 30 名・5 分継続）

合格基準: P99 レスポンスタイムが NFR-PRF サブで確定した目標値以下であること・エラー率 1% 以下であること。

### 4-2. TST-036: スパイクテスト実施要件

| 要件 ID | TST-036 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-PRF サブ |

スパイクテストは「突発的な同時接続増加」に対するシステムの挙動を確認する。ゼロから 30 名への 1 分以内の接続増加に対して、エラー率 1% 以下・P99 応答時間が通常負荷の 3 倍以下であることを合格基準とする。

### 4-3. TST-037: セキュリティスキャン実施要件

| 要件 ID | TST-037 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-SEC サブ / TST-004 |

セキュリティスキャンは M5 完了後に実施する。実施内容は TST-024 で定義したとおりとし、スキャン結果（レポート）を証跡として保存する。スキャン後に High 以上の脆弱性が発見された場合はリリースを停止し、修正・再スキャンを実施する。

### 4-4. TST-038: ALCOA+ 整合テスト実施要件

| 要件 ID | TST-038 |
|---|---|
| 優先度 | Must |
| 上流リンク | FR-TR サブ / 計画 06 章 ALCOA+ マトリクス / TST-005 |

ALCOA+ 9 原則の整合テストは以下の専用シナリオで実施する。

| 原則 | テストシナリオ |
|---|---|
| Attributable | 記録者 ID・ロール・タイムスタンプの付与を全 Step 記録で確認する |
| Contemporaneous | 記録完了タイムスタンプとクライアント送信時刻の差を計測し、500ms 以内であることを確認する |
| Consistent | SHA-256 ハッシュチェーンの整合性チェッカーを実行し、全記録で Pass であることを確認する |
| Original | 既存記録への上書き・削除操作を試みて技術的に拒否されることを確認する |
| Complete | クリティカルステップ未完了時に次 Step へ進行できないことを確認する |

---

**本節で確定した方針**
- 負荷テスト・スパイクテストを M5 完了後・受入テスト前にステージング環境で実施することを確定する。
- セキュリティスキャンで High 以上の脆弱性が発見された場合は修正・再スキャンが完了するまでリリースを停止することを確定する。
- ALCOA+ 整合テストは専用シナリオ（原則別確認）で実施することを確定する。

## 5. テスト記録要件

### 5-1. TST-040: テスト結果の記録形式

| 要件 ID | TST-040 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-SEC（Audit Trail）/ ALCOA+ Contemporaneous |

テスト結果は以下の形式で記録する。

| フィールド | 内容 | 必須/任意 |
|---|---|---|
| テストレベル | L1〜L6 のいずれか | 必須 |
| テスト ID | TST-NNN（本章の要件 ID） | 必須 |
| 実施日時 | YYYY-MM-DD HH:MM:SS | 必須 |
| 実施者 | 開発者名（セルフ受入の場合は本人） | 必須 |
| 対象マイルストーン | M1〜M6 | 必須 |
| テストケース数 | 実施件数・Pass 件数・Fail 件数 | 必須 |
| カバレッジ率 | 行カバレッジ・UC カバレッジ・FR カバレッジ | L1・L3・L6 で必須 |
| 不合格内容 | Fail したテストケースの詳細・再現手順 | Fail がある場合は必須 |
| 合否判定 | Pass / Fail | 必須 |

### 5-2. TST-041: テスト記録の保存期間と保存先

| 要件 ID | TST-041 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-SEC（証跡保全）/ ALCOA+ Enduring |

テスト記録は以下の要件で保存する。

- **保存先**: `docs/03_要件定義/テスト・受入要件/証跡/` 配下。ファイル名は `YYYYMMDD_TST-NNN_result.md` 形式とする
- **保存期間**: Git リポジトリ管理のため、削除は禁止する。リポジトリへのコミットをもって証跡保全とする
- **PDF スナップショット**: L6 受入テストの記録は PDF 形式でも保存し、Git リポジトリ外のバックアップ（USB・共有ドライブ）にも保存する

### 5-3. TST-042: テスト記録のレビュー要件

| 要件 ID | TST-042 |
|---|---|
| 優先度 | Should |
| 上流リンク | TST-070（受入完了記録）|

L6 受入テストの記録は、実施後に開発者本人がレビューし、以下を確認する。

- 全テストケースの Pass/Fail が記録されていること
- Fail がある場合は不合格内容と対応方針が記録されていること
- 合否判定が明示されていること

レビュー完了後、受入完了記録（TST-070 形式）に最終合否判定を記録する。

---

**本節で確定した方針**
- テスト記録の 9 フィールド形式（TST-040）を確定し、フィールドごとの必須/任意を明示する。
- テスト記録の保存先を `テスト・受入要件/証跡/` とし、Git 管理・削除禁止を確定する。
- L6 受入テスト記録は PDF スナップショットでも保存し、Git 外バックアップを追加することを確定する。
