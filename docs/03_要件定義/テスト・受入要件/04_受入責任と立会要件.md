# 04 受入責任と立会要件

本章の責務は、ver1.0.0 受入テストの責任体制・立会い要件・受入完了記録の形式を確定することである。本プロジェクトは個人開発前提であるため、受入体制は「セルフ受入」を基本とし、外部検証の位置づけを明示する。

---

## 1. 受入責任者の定義

### 1-1. TST-060: セルフ受入責任者の定義

| 要件 ID | TST-060 |
|---|---|
| 優先度 | Must |
| 上流リンク | 計画 10 章 §6 自己引継ぎ設計 / プロジェクト要件 PRJ-001 |

本プロジェクトは開発者 1 名による個人開発である。受入責任者は開発者本人（以下「セルフ受入責任者」）とする。

**セルフ受入責任者の責務**:
- 受入テストの計画・実施・記録をすべて担当する
- 受入基準（TST-001〜007）の Pass/Fail を客観的に判定する
- 合否判定結果を受入完了記録に記録し、電子署名で確定する
- 不合格時の修正・再テストを指示・実施する

**主観的判断の排除**: セルフ受入の主観性バイアスを排除するために、各受入基準の Pass/Fail は本書 01 章に記述した検証可能命題に基づいて判定する。「おおむね動いている」「たぶん大丈夫」という判定は Pass とみなさない。数値目標（カバレッジ率・レスポンスタイム・脆弱性ゼロ等）はツールの計測値が確認できた場合にのみ Pass とする。

### 1-2. TST-061: 外部検証の位置づけ

| 要件 ID | TST-061 |
|---|---|
| 優先度 | Should |
| 上流リンク | PRJ-010（外部支援の調達要件）|

個人開発の制約として、すべての受入テストをセルフ受入で完結することを基本とする。ただし以下の領域については外部検証の実施を推奨する。推奨事項の実施は任意であり、実施した場合はレビュー記録を証跡として保存する。

| 推奨外部検証 | 内容 | 実施方法 |
|---|---|---|
| セキュリティレビュー | TST-004（RBAC・JWT・脆弱性スキャン）の第三者確認 | セキュリティ専門家または経験者への非公式レビュー依頼 |
| 操作性確認 | 実際の作業員・現場監督・品質担当の代表者による操作確認 | 非公式のユーザビリティ確認セッション（1〜2 名） |
| ALCOA+ 審査 | 品質担当者による ALCOA+ 整合の独立確認 | 品質担当者代表者への記録確認依頼 |

外部検証を実施する場合の条件:
- NDA（秘密保持契約）の締結を確認してから実施する
- 外部検証者にシステムの設計上の制約・非対象事項を事前説明する
- 外部検証の結果がセルフ受入の合否判定を覆すものではない（セルフ受入責任者の判定が最終判定である）

---

**本節で確定した方針**
- 受入責任者を開発者本人（セルフ受入責任者）とし、Pass/Fail は検証可能命題のみで判定することを確定する。
- 外部検証は推奨（必須ではない）とし、セキュリティ・操作性・ALCOA+ の 3 領域を推奨対象として確定する。
- セルフ受入責任者の判定が最終判定であり、外部検証結果は参考情報とすることを確定する。

---

## 2. 立会要件

### 2-1. TST-065: 立会い対象テストの定義

| 要件 ID | TST-065 |
|---|---|
| 優先度 | Should |
| 上流リンク | プロジェクト要件 PRJ-004（コミュニケーション要件）|

以下の受入テストに、代表ステークホルダーの立会いを推奨する。立会いは任意であり、立会者がいない場合もセルフ受入責任者の判定でテストを完了できる。

| 受入テスト | 推奨立会いステークホルダー | 立会いの目的 |
|---|---|---|
| 受入シナリオ A（SOP ナビゲーション）| 作業員代表・現場監督代表 | 現場視点での操作性・手順の妥当性確認 |
| 受入シナリオ B（証拠記録と電子サイン）| 品質担当代表 | ALCOA+ 原則の現場観点からの確認 |
| 受入シナリオ C（マスタ改訂）| マスタ編集者代表・品質担当代表 | 改訂フローの実務妥当性確認 |
| 受入シナリオ D（アンドン・CAPA）| 現場監督代表・品質担当代表 | CAPA フローの現場適用確認 |
| 受入シナリオ E（Outbox 同期）| IT 担当代表 | オフライン動作・復旧フローの技術確認 |

### 2-2. TST-066: 立会い者の役割と権限

| 要件 ID | TST-066 |
|---|---|
| 優先度 | Should |
| 上流リンク | TST-065 |

立会い者の役割と権限を以下のとおり確定する。

- **立会い者の役割**: 受入テストのシナリオ実行を観察し、操作性・業務フローの妥当性についての意見・コメントを提供する
- **立会い者の権限**: 立会い者は操作性・業務フローに関する改善意見を提出できる。ただし Pass/Fail の判定権限はセルフ受入責任者のみが持つ
- **立会い者のコメント処理**: 立会い者のコメントは「受入テスト立会い記録」に記録する。コメントへの対応（修正実施 / 次バージョン対応 / 対応不要の理由）をセルフ受入責任者が決定し、記録する

---

**本節で確定した方針**
- 5 受入シナリオへの代表ステークホルダー立会いを推奨（必須ではない）とし、立会い対象と推奨ステークホルダーを確定する。
- 立会い者の Pass/Fail 判定権限を否定し、判定権限はセルフ受入責任者のみが持つことを確定する。
- 立会い者コメントへの対応をセルフ受入責任者が決定・記録することを確定する。

---

## 3. 受入完了記録

### 3-1. TST-070: 受入完了記録の形式

| 要件 ID | TST-070 |
|---|---|
| 優先度 | Must |
| 上流リンク | ALCOA+ Enduring・Contemporaneous / NFR-SEC 証跡保全 |

受入完了記録は以下の形式で作成する。

| フィールド | 内容 | 記録要件 |
|---|---|---|
| 文書 ID | `UAT-YYYYMMDD-V{バージョン}` | 必須 |
| 受入テスト実施日時 | YYYY-MM-DD HH:MM 〜 HH:MM | 必須 |
| セルフ受入責任者 | 開発者名 | 必須 |
| 対象バージョン | リリース対象のアプリバージョン（例: 1.0.0-rc.1） | 必須 |
| 受入基準別 Pass/Fail | TST-001〜TST-007 の各 Pass/Fail と根拠 | 必須 |
| カバレッジ達成状況 | 行カバレッジ・UC カバレッジ・FR カバレッジの実績値 | 必須 |
| 立会い者 | 立会いした場合は氏名・役職。なしの場合は「なし」 | 必須 |
| 立会い者コメント | 立会い者からのコメントと対応状況。なしの場合は「なし」 | 必須 |
| 最終合否判定 | 合格 / 不合格 | 必須 |
| 電子署名 | 判定者（セルフ受入責任者）の電子署名・署名日時 | 必須 |

電子署名は本システムの電子サイン機能を使用して取得する。電子サイン機能が未実装の段階（M5 以前）では、Git コミットメッセージに `Signed-off-by:` を記述し、コミットハッシュを証跡として使用する。

### 3-2. TST-071: 受入完了記録の保存先と証跡保全

| 要件 ID | TST-071 |
|---|---|
| 優先度 | Must |
| 上流リンク | TST-041 / NFR-SEC 証跡保全 |

受入完了記録は以下の要件で保存する。

- **保存先**: `docs/03_要件定義/テスト・受入要件/証跡/` 配下に Markdown 形式で保存する。ファイル名は `UAT-YYYYMMDD.md` 形式とする
- **PDF スナップショット**: 受入完了記録は PDF 形式でも保存し、Git リポジトリ外（USB・共有ドライブ）にもバックアップを取得する
- **削除禁止**: 受入完了記録の削除は禁止する。記録の修正が必要な場合は新しい版を作成し、旧版に「修正済み → 新版参照」の注記を追加する

### 3-3. TST-072: 不合格時の受入完了記録

| 要件 ID | TST-072 |
|---|---|
| 優先度 | Must |
| 上流リンク | TST-070 / TST-015（不合格時対応手順）|

受入テストが不合格の場合も受入完了記録を作成し、以下を記録する。

- 不合格の受入基準 ID と不合格理由
- 不合格の重大度（TST-015 の区分: Critical / Major / Minor）
- 修正計画（修正方針・再テスト予定日）
- 不合格状態での暫定保存（リリースは不合格のまま実施しない）

---

**本節で確定した方針**
- 受入完了記録の 10 フィールド形式（TST-070）を確定し、電子署名を必須とする。
- 受入完了記録の保存先・PDF スナップショット・削除禁止を確定する。
- 不合格時も受入完了記録を作成し、不合格理由・重大度・修正計画を記録することを確定する。
