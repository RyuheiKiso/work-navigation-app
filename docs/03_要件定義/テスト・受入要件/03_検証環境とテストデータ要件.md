# 03 検証環境とテストデータ要件

本章の責務は、テスト・受入テストを実施するための検証環境（ステージング環境）の要件とテストデータの要件を確定することである。検証環境が本番環境と乖離している場合、テスト結果は本番の品質を保証しない。本章はその乖離を排除する要件を確定する。

---

## 1. ステージング環境の要件

### 1-1. TST-045: ステージング環境の構成要件

| 要件 ID | TST-045 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-AVL サブ / 計画 T01〜T05 技術的前提 |

ステージング環境は本番環境と同等の構成を持つこと。以下の要件をすべて満たすこと。

- **ハードウェア等価性**: ステージング環境のサーバーは本番環境と同一または同等のスペック（CPU・メモリ・ストレージ容量・OS バージョン）を持つこと。スペックが大きく異なる環境でのテスト結果をリリース判定の根拠として使用しない
- **OS・ミドルウェア等価性**: Windows Server 2022・WSL2・Docker バージョン・PostgreSQL バージョンが本番環境と一致していること
- **ネットワーク分離**: ステージング環境は本番環境のデータベースと物理的に分離していること。本番 DB への接続設定をステージング環境に持ち込まない
- **TLS/SSL 設定**: ステージング環境でも本番同等の TLS 設定を適用すること。TLS なしでの受入テスト実施は認めない
- **タブレット環境**: L3・L6 テストには本番で使用するものと同一機種・OS バージョンのタブレット端末を使用すること

### 1-2. TST-046: ステージング環境の分離要件

| 要件 ID | TST-046 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-SEC サブ / P01 業務的前提 |

ステージング環境は本番環境から以下の観点で分離すること。

- **データ分離**: ステージング DB は本番 DB とは別のインスタンスで動作すること。ステージング環境から本番 DB へのアクセスを技術的に禁止する（接続文字列を環境変数で管理し、ステージング用の接続文字列のみをステージング環境に設定する）
- **認証情報分離**: ステージング環境の JWT シークレット・DB パスワードは本番環境と異なる値を使用すること
- **ログ分離**: ステージング環境のログは本番ログと混在させない。別のログファイル・別のディレクトリに出力する

---

**本節で確定した方針**
- ステージング環境は本番と同等のハードウェア・OS・ミドルウェア・タブレット端末を使用することを確定する。
- ステージング環境と本番環境のデータ・認証情報・ログを完全に分離することを確定する。
- TLS なしでの受入テストは認めず、本番同等の TLS 設定を必須とすることを確定する。

---

## 2. テストデータの要件

### 2-1. TST-050: テストデータの PII 匿名化要件

| 要件 ID | TST-050 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-ETH サブ / 計画 11 章 R-E02 倫理リスク |

テスト環境で使用するテストデータは、実在の従業員個人情報（氏名・社員番号・実際の作業記録・顔写真等）を含まないこと。以下の要件をすべて満たすこと。

- **氏名・社員番号**: 仮名（例: テスト作業員 01・TEST-EMP-001）を使用すること
- **作業記録**: 架空の工程名・作業名・品番・ロット番号を使用すること。実際の製品名・品番の使用は禁止する
- **測定値・数値データ**: 現実の測定範囲内の架空の数値を使用すること
- **電子サイン**: テスト用の架空のサイン画像を使用すること。実在の従業員のサイン画像の使用は禁止する

### 2-2. TST-051: 本番データの直接利用禁止

| 要件 ID | TST-051 |
|---|---|
| 優先度 | Must |
| 上流リンク | NFR-ETH サブ / TST-050 |

本番 DB のデータをステージング環境またはローカル開発環境に直接コピーすることを禁止する。

本番データが必要な場合の手順:
1. 本番 DB から匿名化ツール（TST-052）を使用してエクスポートする
2. 匿名化が完了したデータのみをステージング環境にインポートする
3. 匿名化処理のログを記録し、匿名化前のデータがステージング環境に残存していないことを確認する

本番データのステージング環境への直接利用が発覚した場合は、該当データを即座に削除し、インシデントとして記録する。

### 2-3. TST-052: テストデータ生成・匿名化ツール要件

| 要件 ID | TST-052 |
|---|---|
| 優先度 | Should |
| 上流リンク | TST-050 / TST-051 |

テストデータの生成・匿名化のために以下のいずれかの手段を整備すること。

- **シードスクリプト**: `tests/fixtures/` 配下に DB 初期化シードスクリプト（SQL または Rust コード）を配置し、テスト環境の構築を自動化すること。シードスクリプトが生成するデータは架空のマスタ・架空の作業記録のみを含むこと
- **匿名化スクリプト**: 将来的に本番データのマスク処理が必要になった場合に備え、PII フィールド（氏名・社員番号・サイン画像等）を仮名または架空値に置換するスクリプトを作成・維持すること

テストデータの設計原則: テストデータは以下の 4 種類をカバーすること。
- 正常ケースのデータ（全フィールドが有効値）
- 境界値データ（最大文字長・最小値・最大値）
- エラーケースデータ（NULL・空文字・型不一致）
- ALCOA+ 検証用データ（ハッシュチェーン整合性確認用の記録セット）

---

**本節で確定した方針**
- テストデータへの実在 PII の使用を禁止し、仮名・架空データのみの使用を確定する。
- 本番データのステージング環境への直接コピーを禁止し、匿名化ツール経由のみを許可することを確定する。
- テストデータは正常・境界値・エラー・ALCOA+ 検証の 4 種類をカバーするシードスクリプトで管理することを確定する。

---

## 3. 環境管理要件

### 3-1. TST-055: ステージング環境のスナップショット要件

| 要件 ID | TST-055 |
|---|---|
| 優先度 | Should |
| 上流リンク | OPS-BKP サブ / 計画 T01 技術的前提 |

ステージング環境は以下のタイミングでスナップショット（DB のバックアップ + 環境設定のバックアップ）を取得すること。

- 各マイルストーン（M3・M4・M5・M6）の受入テスト開始前
- 本番デプロイ前
- ステージング環境の重大な設定変更前

スナップショットは最低 2 世代を保持すること。スナップショットのファイル名は `staging_snapshot_YYYYMMDD_M{N}.tar.gz` 形式とする。

### 3-2. TST-056: ステージング環境の復元手順要件

| 要件 ID | TST-056 |
|---|---|
| 優先度 | Must |
| 上流リンク | OPS-BKP サブ / NFR-AVL RTO 要件 |

ステージング環境の復元手順を文書化し、以下の要件を満たすこと。

- **復元手順書**: `docs/運用/staging_restore.md` に復元手順を記述し、IT 担当 1 名がドキュメントのみを参照して復元を完了できること
- **復元時間目標**: ステージング環境の全体復元が 2 時間以内に完了すること（本番 RTO の 2 倍を上限とする）
- **実機確認**: M6 受入テスト前に、復元手順書に従って実際に復元操作を実施し、動作確認済みであることを記録すること

### 3-3. TST-057: 本番環境とのドリフト管理

| 要件 ID | TST-057 |
|---|---|
| 優先度 | Must |
| 上流リンク | TST-045 / NFR-MNT サブ |

ステージング環境と本番環境の構成ドリフト（設定・バージョン・スキーマの乖離）を管理するために以下の要件を満たすこと。

- **Dockerfile・docker-compose.yml の一元管理**: ステージング環境と本番環境は同一の Dockerfile・docker-compose.yml を使用すること。環境差異は環境変数でのみ表現し、ファイルの分岐は禁止する
- **DB マイグレーション同期**: ステージング環境と本番環境の DB スキーマは常に同一バージョンであること。マイグレーションは先にステージング環境で適用し、テスト通過後に本番環境に適用する順序を必ず守ること
- **ドリフト確認**: 各マイルストーン完了時にステージング環境と本番環境の構成比較を実施し、意図しない乖離がゼロであることを記録すること

---

**本節で確定した方針**
- ステージング環境のスナップショットを各マイルストーン受入テスト開始前に取得し、2 世代保持することを確定する。
- 復元時間目標を 2 時間以内とし、M6 受入テスト前に実機復元確認を実施することを確定する。
- Dockerfile・docker-compose.yml の一元管理によりステージングと本番のドリフトを排除することを確定する。
