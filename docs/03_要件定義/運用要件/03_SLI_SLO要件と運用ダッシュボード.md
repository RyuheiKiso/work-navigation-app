# 03 SLI・SLO 要件と運用ダッシュボード

本書の責務は、本システムの品質達成水準を計測するための SLI（Service Level Indicator）定義・SLO（Service Level Objective）目標値・計測方法・運用ダッシュボード要件・エラーバジェット管理要件を確定することである。上流文書「システム化計画 14 章」の品質指標枠組みを受けて、要件定義レベルの数値と計測仕様を確定する。

---

## 1. SLI 定義

### 1-1. 基本 SLI の確定

OPS-036: 本システムの SLI として以下の 4 種を確定する。子機モード（計画 12 章）を使用しない単独動作の場合は SLI-04 の計測を省略できる。

| SLI ID | 名称 | 定義 |
|---|---|---|
| SLI-01 | 可用性（Availability） | 製造ライン稼働時間帯中、タブレット APP および管理 Web が利用可能だった時間の割合（%） |
| SLI-02 | Step 完了応答時間 | 作業員が Step 完了ボタンをタップしてから次の Step が表示されるまでの P95 応答時間（ms） |
| SLI-03 | Outbox 送信遅延（子機モード時） | Outbox キューに投入された実績データが最終的に親機で受信されるまでの最大遅延時間（分） |
| SLI-04 | データ整合性 | 週次の整合性チェックコマンド実行で発見される参照整合性エラーの件数（件） |

OPS-037: SLI-01（可用性）の計測対象と除外対象を以下の通り確定する。

| 分類 | 内容 |
|---|---|
| 可用性の計測対象時間 | 製造ライン稼働時間帯（工場定義の稼働時間帯、1 日単位で管理する） |
| ダウンタイムの定義 | ヘルスチェックエンドポイントへの応答がタイムアウト（3 秒）または 5xx エラーを 3 回連続返す状態 |
| 計測から除外するダウンタイム | 事前通知済みの計画停止（OPS-024 の条件を満たすもの）のみ |
| 計測から除外しないダウンタイム | 計画停止条件を満たさない停止・通知が不十分な停止はすべてダウンタイムに計上する |

### 1-2. SLI 計測の技術仕様

OPS-038: 各 SLI の計測手段を以下の通り確定する。

| SLI ID | 計測手段 | 計測頻度 | データ保存先 |
|---|---|---|---|
| SLI-01 | `/health` エンドポイントへの HTTP GET でステータスコード 200 を確認する | 1 分毎 | application_log テーブルまたは外部監視ログファイル |
| SLI-02 | axum ミドルウェアによる Step 完了 API のリクエスト〜レスポンス時刻を記録する | リクエスト毎 | request_metrics テーブル |
| SLI-03 | Outbox キューの投入タイムスタンプと親機受信確認タイムスタンプの差分を計算する | 送信毎 | outbox_log テーブル |
| SLI-04 | 整合性チェックコマンド（週次）の出力レポートで件数を確認する | 週次 | integrity_check_log テーブル |

**本節で確定した方針**
- 可用性・Step 完了応答時間・Outbox 送信遅延・データ整合性の 4 SLI を確定し、計測対象・計測手段・計測頻度を確定する。
- 計画停止のみをダウンタイム計測除外とし、条件を満たさない停止は全量ダウンタイムに計上することを確定する。
- SLI データの保存先テーブルを確定し、設計フェーズでのスキーマ実装を義務付ける。

## 2. SLO 目標値

### 2-1. SLO の確定

OPS-039: 上流文書「システム化計画 14 章 3 節」の SLO 決定プロセス（ベースライン 120〜150% 設定）に従い、以下の SLO 目標値を要件として確定する。

| SLI ID | SLO 目標値 | 計測単位 | 根拠 |
|---|---|---|---|
| SLI-01（可用性） | 99.5% 以上 | 月次（稼働時間帯の累計） | 月間エラーバジェット 3.6 時間。計画停止 2 時間を含む |
| SLI-02（Step 完了応答時間） | P95 500 ms 以内 | 日次集計 | 計画 08 章の「ページ遷移 500ms 以内」方針値を SLO として確定する |
| SLI-03（Outbox 送信遅延） | 最大遅延 5 分以内（通常 LAN 接続時） | 日次集計 | 作業完了から 5 分以内の親機への実績送信を業務要件として設定する |
| SLI-04（データ整合性） | 週次整合性チェックでエラー 0 件 | 週次 | 整合性エラーが 1 件でも存在する状態は ALCOA+ Consistent 原則違反のため |

OPS-040: SLO 目標値はパイロット期間（M6、計画 10 章）の実測データに基づいて見直す機会を設ける。見直しが必要な場合は ADR（MNT-065）を起票して目標値変更の根拠を記録する。SLO を根拠なしで変更することを禁止する。

### 2-2. SLO の月次評価

OPS-041: SLO の達成状況は月次で評価し、品質担当・IT 担当・現場監督に報告する。月次評価の報告書は以下の内容を含む。

| 報告項目 | 内容 |
|---|---|
| 各 SLI の月次計測値 | 実績値と SLO 目標値の対比 |
| SLO 達成・未達の判定 | 達成：SLO 目標以上 / 未達：SLO 目標未満 |
| ダウンタイムの内訳 | 計画停止・非計画停止の区分と件数・所要時間 |
| 改善アクション | 未達 SLO に対する改善アクション（担当者・期限付き） |

**本節で確定した方針**
- SLI-01 可用性 99.5%・SLI-02 P95 500 ms・SLI-03 最大 5 分・SLI-04 エラー 0 件を SLO として確定する。
- パイロット後の SLO 見直し機会を設けるが、根拠なしの変更を禁止し ADR 起票を必須とする。
- 月次評価報告書の 4 必須項目を確定し、達成状況の可視化と改善アクション管理を月次サイクルに組み込む。

## 3. SLO 計測方法

### 3-1. 計測の自動化要件

OPS-042: SLI-01（可用性）の計測は以下の仕様で自動化する。

| 仕様項目 | 内容 |
|---|---|
| 計測エンドポイント | `GET /health` レスポンスが 200 かつ `{"status": "ok"}` を返すことを確認する |
| 計測間隔 | 1 分毎（cron または systemd timer で実行する） |
| タイムアウト閾値 | 3 秒以内にレスポンスがない場合をタイムアウトとする |
| ダウン判定 | タイムアウトまたは 5xx エラーが 3 回連続した場合をダウン開始とする |
| 復旧判定 | 200 レスポンスが 2 回連続した場合を復旧とする |
| 計測ログ保存 | PostgreSQL の health_check_log テーブルに全計測結果を記録する |

OPS-043: SLI-02（Step 完了応答時間）の計測は以下の仕様で実装する。

| 仕様項目 | 内容 |
|---|---|
| 計測対象 API | `POST /api/steps/{step_id}/complete` のリクエスト〜レスポンスの時刻差 |
| 計測手段 | axum ミドルウェアで `X-Request-Start` ヘッダーを設定し、ミドルウェアでレスポンス時刻との差を計算する |
| パーセンタイル計算 | 日次バッチで全 Step 完了 API の応答時間の P95 を計算してログに記録する |
| アラート発動条件 | 直近 1 時間の P95 が 1,000 ms を超えた場合にアラートを発生させる（SLO 閾値の 2 倍でアラート） |

### 3-2. 閾値アラートの要件

OPS-044: 以下のアラート条件を確定する。アラートは IT 担当にメールで通知する。SaaS 監視ツールの使用は本システムの前提（社内 LAN オンプレ・インターネット接続なし）から対象外と判断する。

| アラート ID | 発動条件 | 通知先 | 通知手段 |
|---|---|---|---|
| ALT-01 | SLI-01 ダウン判定（3 回連続ダウン） | IT 担当 | メール（社内メールサーバー経由） |
| ALT-02 | SLI-02 直近 1 時間 P95 が 1,000 ms 超過 | IT 担当 | メール |
| ALT-03 | SLI-03 Outbox 送信遅延 10 分超過 | IT 担当 | メール |
| ALT-04 | SLI-04 週次整合性チェックでエラー 1 件以上 | IT 担当 + 品質担当 | メール |
| ALT-05 | バックアップジョブの失敗 | IT 担当 | メール |

OPS-045: アラート通知メールには以下の情報を含む。

| 項目 | 内容 |
|---|---|
| アラート ID | ALT-01〜05 |
| 発生日時 | UTC タイムスタンプ |
| アラート内容 | 発動条件の具体的な状況（計測値を含む） |
| 推奨対応 | 対応手順書への参照リンク（内部ドキュメント URI） |

**本節で確定した方針**
- 可用性の 1 分毎自動計測・3 回連続ダウン判定・2 回連続復旧判定のアルゴリズムを確定する。
- SaaS 監視ツールを社内 LAN オンプレ前提から対象外と判断し、メールベースのアラートで代替することを確定する。
- 5 種のアラート条件と通知先・通知手段を確定する。

## 4. 運用ダッシュボード要件

### 4-1. ダッシュボードの設計方針

OPS-046: 運用ダッシュボードは本システムの管理 Web に内包する形で実装する。外部の SaaS ダッシュボードツール（Grafana Cloud・Datadog 等）の使用は、社内 LAN オンプレ・インターネット接続なしの前提から対象外と判断する。

OPS-047: ダッシュボードは IT 担当が毎稼働日 15 分以内に確認できる情報密度に設計する。情報過多によって重要な異常を見落とす設計を禁止する。

### 4-2. ダッシュボードの必須項目

OPS-048: IT 担当が確認する運用ダッシュボードに以下の項目を必須として実装する。

| 項目 | 表示形式 | 更新頻度 |
|---|---|---|
| 現在のシステム稼働状態 | 緑（正常）/ 黄（警告）/ 赤（障害）の 3 色インジケーター | リアルタイム（1 分毎自動更新） |
| 当日の可用性（稼働率） | パーセンテージ数値と稼働時間の棒グラフ | 1 時間毎更新 |
| 直近 24 時間の Step 完了応答時間 P95 | 折れ線グラフ + 現在値 | 1 時間毎更新 |
| 未送信 Outbox キュー件数 | 件数と最古キューの経過時間 | 5 分毎更新 |
| 直近のバックアップ成否 | 成功 / 失敗 + 実行日時 | バックアップ実行毎 |
| アクティブなアラート一覧 | 未解消アラートの一覧（発生日時・種別） | リアルタイム |
| 月次 SLO 達成状況サマリ | SLI 毎の当月達成率と目標値の対比 | 日次更新 |

OPS-049: ダッシュボードへのアクセス権限は system_admin ロールと master_admin ロールに限定する。operator ロール・supervisor ロールからのダッシュボードアクセスを禁止する。

**本節で確定した方針**
- 運用ダッシュボードを管理 Web 内に内包実装し、外部 SaaS ダッシュボードを対象外と判断することを確定する。
- IT 担当が 15 分以内に確認完了できる情報密度設計を必須とし、情報過多設計を禁止する。
- 7 必須項目とアクセス権限制限（system_admin + master_admin のみ）を確定する。

## 5. SLO エラーバジェット

### 5-1. エラーバジェットの定義

OPS-050: エラーバジェットを「100% − SLO 目標値」で定義する。月間のエラーバジェットを以下の通り計算して管理する。

| SLI ID | SLO 目標値 | 月間エラーバジェット | 稼働時間換算（月 720 時間想定） |
|---|---|---|---|
| SLI-01（可用性） | 99.5% | 0.5% | 3.6 時間（計画停止 2 時間を含む） |
| SLI-02（応答時間） | P95 500 ms 以内 | P95 500 ms 超過を 5% 以内に抑える | 日次 P95 が 500 ms 超過できるのは月 1.5 日以内 |

OPS-051: 月間エラーバジェット残量を以下の 3 状態で管理し、各状態での対応方針を確定する。

| 状態 | 残量 | 対応方針 |
|---|---|---|
| 正常 | 残量 50% 以上 | 通常の機能開発・改修を継続する |
| 警戒 | 残量 10〜50% | 新機能リリースを抑制し、信頼性改善タスクを優先する |
| 枯渇 | 残量 10% 未満 | 新機能追加・変更作業を即時停止し、障害根本原因の対応に集中する |

### 5-2. エラーバジェット枯渇時の対応手順

OPS-052: 月間エラーバジェット残量が 50% 未満になった場合、IT 担当は以下の手順で対応する。

1. 残量 50% 未満をダッシュボードで確認した時点で、品質担当と現場監督に報告する
2. エラーバジェット消費の原因（計画外ダウンタイム・応答時間劣化等）を特定する
3. 原因に応じた改善アクション（インフラ修正・パッチ適用・設定変更等）を 48 時間以内に計画する
4. 改善アクション計画を GitHub Issues にチケットとして起票し、優先度 High を設定する
5. 改善アクションの完了後、再度ダッシュボードで残量を確認し、品質担当に報告する

OPS-053: 月間エラーバジェットが完全に枯渇（残量 0%）した場合は、計画中の全機能変更・新機能追加を停止し、障害対応・信頼性改善のみに開発リソースを集中する。枯渇状態の解消は翌月のエラーバジェット正常化をもって確認する。

**本節で確定した方針**
- エラーバジェットを「100% − SLO 目標値」として計算し、正常・警戒・枯渇の 3 状態で管理することを確定する。
- 残量 50% 未満時の 5 ステップ対応手順を確定し、48 時間以内の改善アクション計画策定を必須とする。
- エラーバジェット枯渇時の全機能変更停止を信頼性優先原則として確定する。
