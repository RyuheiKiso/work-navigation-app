# 01 運用体制と役割定義

本書の責務は、本システムの安定稼働を IT 担当 1 名の兼務体制で継続するための運用体制・役割定義・エスカレーション体制・引継ぎ対策を要件として確定することである。上流文書「システム化計画 09 章 1 節・11 章リスクレジスタ（R-O01）」の方針を受けて、運用要件 ID（OPS-NNN）として詳細化する。

---

## 1. IT 担当 1 名兼務前提の運用体制

### 1-1. 運用体制の前提制約

OPS-001: 本システムの運用は IT 担当 1 名の兼務体制を必須制約として設計する。IT 担当が複数名いる場合は本設計が緩和される方向でのみ変更を認める。IT 担当増員を前提とした設計変更は行わない。

OPS-002: IT 担当の主要な運用タスクと週当たりの想定所要時間を以下の通り確定する。

| 運用タスク | 週次所要時間（目安） | 詳細 |
|---|---|---|
| バックアップ確認 | 15 分 | 自動バックアップの成否ログ確認・週次リストア確認（月次） |
| ヘルスチェック確認 | 15 分 | Prometheus / ヘルスチェックエンドポイントの結果確認 |
| ログ確認（エラー・警告） | 30 分 | アプリケーションログの ERROR / WARN 確認 |
| セキュリティアップデート確認 | 30 分 | cargo audit 実行・依存ライブラリの更新確認 |
| ユーザーアカウント管理 | 30 分 | 入退職対応・パスワードリセット対応（都度対応） |
| 障害対応（インシデント発生時） | 都度（P1 は最大 2 時間） | 障害分類・切り分け・復旧・事後分析 |

OPS-003: 上記タスクの合計は通常週で 2 時間以内に収まるよう設計する。2 時間を超えるタスクが毎週恒常的に発生する設計を禁止する。自動化・スクリプト化により手動作業を最小化することを設計原則とする。

### 1-2. 運用自動化の要件

OPS-004: 以下の運用タスクをシステムが自動実行する。IT 担当の手動介入を不要とする。

| 自動化タスク | 実行頻度 | 確認方法 |
|---|---|---|
| PostgreSQL のバックアップ取得 | 1 日 1 回（深夜帯） | バックアップログのステータス確認 |
| ヘルスチェックエンドポイントの監視 | 1 分毎 | ステータスダッシュボードで確認 |
| JWT 署名鍵ローテーション | 90 日毎（スケジュール実行） | 鍵ローテーションログで確認 |
| ログのアーカイブ処理 | 7 年保存期限到来時 | アーカイブジョブのログで確認 |

**本節で確定した方針**
- IT 担当 1 名の兼務体制を設計の必須制約として確定し、運用タスクの週当たり合計を 2 時間以内に抑える自動化要件を確定する。
- バックアップ・ヘルスチェック・鍵ローテーション・ログアーカイブの 4 タスクを自動化対象として確定する。
- IT 担当増員を前提とした設計変更を禁止し、1 名体制での継続可能性を設計の最低保証として確定する。

## 2. 4 役割定義

### 2-1. 役割の確定

OPS-005: 本システムの運用ロールとして以下の 4 役割を確定する。

| 役割 | 想定担当者 | RBAC ロール名 |
|---|---|---|
| システム管理者 | IT 担当 | system_admin |
| マスタ管理者 | 品質担当 | master_admin |
| 現場監督 | 現場リーダー・班長 | supervisor |
| 作業員 | 製造ラインの担当者 | operator |

### 2-2. 役割ごとの運用上の責務

OPS-006: システム管理者（IT 担当）の運用上の責務を以下の通り確定する。

| 責務 | 頻度 | 成果物 |
|---|---|---|
| サーバー稼働確認 | 毎稼働日 | ヘルスチェックログ |
| バックアップ成否確認 | 週次 | バックアップ確認記録 |
| セキュリティパッチ適用 | 随時（High 以上は 7 日以内） | パッチ適用記録 |
| ユーザーアカウント管理 | 随時（入退職対応） | ユーザー管理ログ |
| 障害対応・復旧 | 障害発生時 | インシデント記録 |
| 月次バックアップリストア検証 | 月次 | リストア検証記録 |

OPS-007: マスタ管理者（品質担当）の運用上の責務を以下の通り確定する。

| 責務 | 頻度 | 成果物 |
|---|---|---|
| SOP の改訂・承認 | 随時（改訂要求発生時） | 改訂済み SOP（電子署名付き） |
| CAPA 起票の承認・クローズ | 随時 | CAPA クローズ記録 |
| 月次の SLO レポート確認 | 月次 | SLO 達成確認記録 |
| 外国人労働者への多言語 SOP 承認 | SOP 改訂時 | 多言語版承認記録 |

OPS-008: 現場監督の運用上の責務を以下の通り確定する。

| 責務 | 頻度 | 成果物 |
|---|---|---|
| 作業員の認定・スキル更新 | 随時 | スキル認定記録 |
| アンドン（異常通知）への一次対応 | 随時（アンドン発生時） | アンドン対応記録 |
| 新版 SOP の確認（品質担当承認前） | SOP 改訂時 | 確認記録（電子署名） |
| Kaizen Teian の現場一次評価 | 随時 | 評価記録 |

OPS-009: 作業員の運用上の責務を以下の通り確定する。

| 責務 | 頻度 | 成果物 |
|---|---|---|
| Step 記録・電子サイン | 毎作業 | 作業ログ（audit_log テーブル） |
| 不適合・ヒヤリハット起票 | 随時（発見時） | 不適合記録 / ヒヤリハット記録 |
| Kaizen Teian 起票 | 随時（提案時） | Kaizen Teian レコード |

**本節で確定した方針**
- システム管理者・マスタ管理者・現場監督・作業員の 4 役割とそれぞれの運用上の責務を確定する。
- SOP 承認権限をマスタ管理者（品質担当）のみに帰属させ、IT 担当（システム管理者）への SOP 承認権限付与を禁止する。
- 作業員の責務を Step 記録・電子サイン・不適合起票・Kaizen Teian 起票に限定し、マスタ操作権限を持たせないことを確定する。

## 3. エスカレーション体制

### 3-1. 3 段エスカレーションの確定

OPS-010: 本システムのエスカレーション体制を以下の 3 段で確定する。

| 段階 | エスカレーション先 | 発動条件 | 連絡手段 |
|---|---|---|---|
| 第 1 段 | IT 担当（システム管理者） | 障害・異常の最初の通報 | ヘルスチェックアラート・電話・メール |
| 第 2 段 | 開発者（個人開発者本人） | IT 担当が 30 分以内に解決できない P1/P2 インシデント | メール・GitHub Issues |
| 第 3 段 | OSS コミュニティ | 使用 OSS（axum / tokio / React Native 等）のバグに起因する障害 | GitHub Issues（該当 OSS リポジトリ） |

OPS-011: 第 2 段エスカレーション（開発者本人へのエスカレーション）の連絡先と対応可能時間帯を運用マニュアルに記録する。連絡先は IT 担当のみがアクセスできる形式で管理し、一般公開しない。

OPS-012: OSS コミュニティへの第 3 段エスカレーションは IT 担当が直接実施する。OSS バグの回避策（ワークアラウンド）が開発者本人によって提供されるまでの間は、縮退運用（OPS-090 参照）で業務を継続する。

### 3-2. エスカレーション記録要件

OPS-013: エスカレーション実施時は以下の情報をインシデント記録に追記する。

| 記録項目 | 内容 |
|---|---|
| エスカレーション日時 | サーバー UTC タイムスタンプ |
| エスカレーション先 | 第 1〜3 段のいずれか |
| エスカレーション理由 | 前段での解決が困難な理由 |
| 受信者の応答日時 | エスカレーション先からの応答確認日時 |

**本節で確定した方針**
- IT 担当 → 開発者（個人開発者本人）→ OSS コミュニティの 3 段エスカレーション体制を確定する。
- P1/P2 インシデントで IT 担当が 30 分以内に解決できない場合を第 2 段エスカレーションの発動条件として確定する。
- エスカレーション実施時のインシデント記録への 4 項目追記を必須として確定する。

## 4. 引継ぎ・Bus Factor 対策

### 4-1. 運用手順書の整備要件

OPS-014: IT 担当の引継ぎに必要な運用手順書を以下の通り作成・維持することを必須とする。

| 手順書名 | 更新頻度 | 保管場所 |
|---|---|---|
| 初期構築手順書 | バージョンアップ時 | Git リポジトリ（docs/運用手順/） |
| 日常運用手順書 | 四半期ごと（または変更時） | Git リポジトリ（docs/運用手順/） |
| 障害対応手順書 | インシデント発生後（ポストモーテム後） | Git リポジトリ（docs/運用手順/） |
| バックアップ・リストア手順書 | リストア演習後 | Git リポジトリ（docs/運用手順/） |
| ユーザーアカウント管理手順書 | 年次確認 | Git リポジトリ（docs/運用手順/） |

OPS-015: 運用手順書は「IT 担当 1 名のみが知っている情報をゼロにする」ことを目標とする。手順書に記載されていない IT 担当固有の知識（属人知識）が発見された場合は、発見後 30 日以内に手順書に反映する。

### 4-2. Bus Factor 対策の運用要件

OPS-016: IT 担当の長期不在（2 週間以上の休暇・疾病・退職）に備え、以下の対策を運用要件として確定する。

| 対策 | 内容 | 確認頻度 |
|---|---|---|
| ドキュメントのみでの環境再構築可能性 | 運用手順書のみを参照した第三者が Docker Compose で環境を再構築できることを年次演習で確認する | 年次 |
| 管理者パスワードの安全な保管 | システム管理者アカウントのパスワードを経営者または指定の代理者が封印管理する | 人事変更時 |
| 緊急時の対応依頼先 | 運用手順書に「第三者への対応依頼時の連絡先と依頼内容の雛形」を記載する | 年次確認 |

OPS-017: ADR（Architecture Decision Record）は GitHub リポジトリで管理し、IT 担当の離職後も技術的意思決定の根拠を参照できる状態を維持する。ADR の管理要件は保守要件 04 章（MNT-065）で詳細化する。

**本節で確定した方針**
- 5 種類の運用手順書の作成・更新頻度・保管場所を確定し、属人知識の 30 日以内の手順書反映を必須とする。
- IT 担当の長期不在対策として年次環境再構築演習・管理者パスワード封印管理・緊急連絡先記載の 3 対策を確定する。
- ADR を Git リポジトリで管理し、IT 担当の離職後の技術的根拠参照可能性を維持することを確定する。
