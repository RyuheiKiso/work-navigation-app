# 05 BCP と縮退運用要件

本書の責務は、単一工場オンプレ構成における事業継続計画（BCP）と障害時の縮退運用要件を確定することである。上流文書「システム化計画 09 章 5 節」の 3 段階縮退運用方針を受けて、各縮退レベルの発動条件・手順・紙フォールバック管理・復旧確認を要件として確定する。

---

## 1. BCP の範囲

### 1-1. BCP の適用範囲確定

OPS-067: 本システムの BCP は以下の範囲を対象とする。

| BCP 対象 | 内容 |
|---|---|
| 対象スコープ | 単一工場・単一サイトの社内 LAN オンプレ環境における本システムの継続運用 |
| 対象リスク | サーバー障害・LAN 障害・停電・タブレット障害・ソフトウェア障害 |
| RPO（目標復旧時点） | 15 分以内（計画 08 章の RPO 15 分に準拠する） |
| RTO（目標復旧時間） | 1 時間以内（計画 08 章の RTO 1 時間に準拠する） |

OPS-068: 以下は BCP 対象外と判断する。

| 対象外 | 根拠 |
|---|---|
| 地理冗長（複数拠点のフェイルオーバー） | P01（単一工場前提）に基づく。マルチサイト展開は将来オプション（計画 11 章）のため対象外と判断する |
| クラウドバックアップへの自動フェイルオーバー | P02（社内 LAN のみ）に基づきインターネット接続を使用しないため対象外と判断する |
| ホットスタンバイ冗長サーバー | IT 担当 1 名・単一工場の規模において運用コストが設計原則と不整合であるため対象外と判断する |

### 1-2. BCP の設計原則

OPS-069: BCP の設計は ISO 22301:2019 の事業継続マネジメントシステム原則を参照する。ただし本システムは ISO 22301 への準拠認証を対象外と判断する。設計指針として参照する。

OPS-070: 本システムの縮退運用は「作業記録を何らかの形で継続する」ことを最優先とする。電子記録の完全性を維持できない状況においても、作業の記録自体を停止させないことがトレサビの最低要件である。

**本節で確定した方針**
- 単一工場・社内 LAN オンプレを BCP の前提スコープとして確定し、地理冗長・クラウドフェイルオーバー・ホットスタンバイを対象外と判断する。
- RPO 15 分・RTO 1 時間を BCP の達成目標として確定する。
- 縮退運用中も作業記録の継続を最優先とし、電子記録が取れない場合の紙記録を代替手段として確定する。

## 2. 3 段階縮退運用

### 2-1. 縮退レベルの定義

図: fig_degradation_3level（img/ 配下）を参照

OPS-071: 縮退運用を以下の 3 レベルに定義する。

| 縮退レベル | 名称 | 定義 | 作業員の体感 |
|---|---|---|---|
| 縮退レベル 1 | 一部機能停止 | LAN 断絶または特定コンポーネントの障害により、一部機能が利用不可能。タブレット APP のローカル記録は継続可能 | Offline-First で記録は継続できるが、管理 Web や Outbox 送信が利用できない |
| 縮退レベル 2 | 参照のみ | PostgreSQL へのデータ書込みができないが、タブレット APP にキャッシュされた直近の SOP を参照できる | 新規 Step 記録は不可。SOP ナビゲーションの参照のみ可能 |
| 縮退レベル 3 | 紙フォールバック | タブレット APP が全端末で起動しない、または電子記録の継続が全く不可能 | 各工程の紙 SOP ファイルで作業を実施し、紙に記録する |

### 2-2. 各縮退レベルの発動条件

OPS-072: 各縮退レベルの発動条件を以下の通り確定する。発動判断は IT 担当または現場監督が行う。

**縮退レベル 1 の発動条件**

- サーバーへの LAN 疎通が失われたが、タブレット APP は正常動作している
- Outbox 送信が 10 分以上失敗しているが、タブレット APP のローカル記録は継続できている
- 管理 Web が表示されないが、タブレット APP は正常動作している

**縮退レベル 2 の発動条件**

- PostgreSQL が停止しており、タブレット APP へのデータ書込みが失敗している
- タブレット APP が「接続失敗」を表示し、新規 Step 記録ができていない
- ローカル SQLite への書込みも失敗している状況（ストレージ満杯等）

**縮退レベル 3 の発動条件**

- 全タブレット端末が起動しない（バッテリー切れ・故障等）
- 電力供給が停止しており、すべての電子機器が利用不可能
- 縮退レベル 2 の状態で RTO 1 時間を超過しても復旧の見込みが立たない

### 2-3. 各縮退レベルの対応手順

OPS-073: 縮退レベル 1 の対応手順を以下の通り確定する。

1. IT 担当はヘルスチェックアラートで LAN 断絶を検知する
2. タブレット APP が「オフラインモード」表示であることを現場監督に確認させる
3. 作業員は通常通りタブレット APP でローカル記録を継続する（操作方法は変わらない）
4. IT 担当は LAN 復旧作業を並行して実施する
5. LAN 復旧後、タブレット APP がローカルキューをサーバーへ自動同期することを確認する

OPS-074: 縮退レベル 2 の対応手順を以下の通り確定する。

1. 現場監督が縮退レベル 2 を宣言し、IT 担当に通知する
2. 現場監督は各工程の予備タブレット（充電済み・SOP キャッシュ済み）への切替を指示する
3. 作業員はタブレットのキャッシュされた SOP を参照しながら作業を継続する（記録は紙に残す）
4. IT 担当は PostgreSQL の復旧作業を実施する
5. 復旧後、縮退期間中の紙記録を `is_retroactive=true` フラグ付きで後入力する

OPS-075: 縮退レベル 3 の対応手順を以下の通り確定する。

1. 現場監督が縮退レベル 3 を宣言し、各工程に「紙フォールバック開始」を通知する
2. 各工程の担当者は工程ファイル（ラミネート加工された紙 SOP）を開いて作業を継続する
3. 作業記録は工程配備の紙記録用紙（定型フォーマット）に記録する
4. 電子システム復旧後、紙記録を `is_retroactive=true` フラグ・紙原本スキャン添付で後入力する
5. IT 担当は事後に縮退レベル 3 のインシデント記録を作成し、ポストモーテムを実施する

**本節で確定した方針**
- 一部機能停止・参照のみ・紙フォールバックの 3 縮退レベルと各レベルの発動条件・対応手順を確定する。
- 縮退レベル 1 は Offline-First で作業継続・縮退レベル 2 はキャッシュ参照＋紙記録・縮退レベル 3 は紙フォールバックの対応方針を確定する。
- 縮退期間中の紙記録を復旧後に is_retroactive=true フラグで後入力することを全レベル共通の要件として確定する。

## 3. 紙フォールバック手順

### 3-1. 紙 SOP の管理要件

OPS-076: 紙フォールバック用の紙 SOP は以下の管理規則に従って維持する。

| 管理項目 | 方針 |
|---|---|
| 保管場所 | 各工程の作業台に専用ファイルを設置する。ファイルはラミネート加工された SOP を綴じ、作業員が即座にアクセスできる位置に配置する |
| 更新頻度 | 本システムで SOP が改訂・承認された時点から 48 時間以内に最新版 PDF を印刷・差し替える |
| 廃棄条件 | 旧版の紙 SOP は新版差し替え時に旧版 PDF としてシステムに記録してから廃棄する。旧版の無断廃棄を禁止する |
| 管理台帳 | 各工程の紙 SOP ファイルに「現在の版番号・印刷日・印刷担当者」を記入した表紙をつける |

OPS-077: 紙 SOP の更新作業は現場監督が担当する。品質担当は SOP 承認後に現場監督へ更新通知を送り、現場監督は 48 時間以内に更新を完了させる。更新完了を本システムの通知機能またはメールで品質担当に報告する。

### 3-2. 紙フォールバック中の記録要件

OPS-078: 縮退レベル 3（紙フォールバック）中の記録は以下の定型フォーマットで実施する。

| 記録項目 | 必須/任意 |
|---|---|
| 工程名 / 作業名 | 必須 |
| ロット番号 | 必須 |
| 作業日時（手書き） | 必須（分単位で記録する） |
| 作業担当者氏名（手書き署名） | 必須 |
| Step 番号と完了確認 | 必須（各 Step に完了チェックを入れる） |
| 測定値（測定 Step の場合） | 必須（判定基準値と実測値を記録する） |
| 特記事項 | 任意 |

OPS-079: 紙記録用紙の定型フォーマットは本システムから PDF 出力できる機能を提供する。PDF は A4 サイズで印刷可能な設計とし、各工程の紙 SOP ファイルに予備 5 枚以上を常備する。

**本節で確定した方針**
- 紙 SOP の保管場所・48 時間以内更新・廃棄条件・管理台帳の 4 管理規則を確定する。
- 縮退期間中の紙記録の定型フォーマット 7 必須項目を確定する。
- 紙記録用紙の PDF 出力機能を ver1.0.0 の提供機能として確定し、工程ごと 5 枚以上の予備常備を運用基準とする。

## 4. 回復手順と復旧確認

### 4-1. 縮退からの復旧条件

OPS-080: 縮退運用から通常運用への復旧条件を縮退レベル別に確定する。

| 縮退レベル | 復旧条件 |
|---|---|
| 縮退レベル 1 からの復旧 | LAN 疎通の回復・Outbox の自動同期完了確認・ヘルスチェック正常 |
| 縮退レベル 2 からの復旧 | PostgreSQL の正常起動確認・Step 記録の書込みテスト成功・Outbox 積み上がりの解消 |
| 縮退レベル 3 からの復旧 | 縮退レベル 2 の復旧条件に加え、電力供給の安定・全タブレット APP の正常起動確認 |

### 4-2. 復旧後の全機能確認手順

OPS-081: 縮退レベル 2 以上からの復旧後は、以下の確認手順を実施してから「通常運用再開」を宣言する。確認は IT 担当と現場監督が共同で実施する。

| 確認項目 | 確認方法 | 担当者 |
|---|---|---|
| PostgreSQL の稼働確認 | `SELECT 1` クエリが正常に返ることを確認する | IT 担当 |
| バックアップの最新性確認 | 最新バックアップが復旧時点から RPO 15 分以内のデータを含むことを確認する | IT 担当 |
| Step 記録の書込みテスト | テスト用ロットでの Step 記録と audit_log への格納を確認する | IT 担当 |
| Outbox 積み上がり解消 | Outbox キューが空になり、親機への送信が正常に完了していることを確認する（子機モード時） | IT 担当 |
| タブレット APP の全台確認 | 全タブレットでログイン・SOP 表示・Step 記録の一連動作を確認する | 現場監督 |
| 紙記録の後入力完了 | 縮退期間中の紙記録が `is_retroactive=true` で全件後入力されたことを確認する | 品質担当 |

OPS-082: 通常運用再開の宣言は現場監督が行い、IT 担当が電子署名付きで運用ログに記録する。品質担当への復旧報告は宣言後 24 時間以内に行う。

**本節で確定した方針**
- 縮退レベル別の復旧条件（3 段階）と復旧後の全機能確認手順（6 項目）を確定する。
- 通常運用再開の宣言を現場監督が行い IT 担当が電子署名で記録することを確定する。
- 縮退期間中の紙記録の後入力完了を通常運用再開確認の必須項目として確定する。
