# 99 前提制約と本書が約束しないこと

本章は `05_詳細設計/02_バックエンド詳細設計` サブのスコープ外事項・前提制約・将来課題を明記することで、本サブの成果物の読み手が誤った期待を持つことを防ぐ。

---

## 1. 本サブが約束しないこと

### 1-1. 実装ファイルの作成

本サブは設計仕様書である。`crates/` 配下の実際の `.rs` ファイルを作成することは本サブのスコープ外である。コードブロックとして記載された Rust コードはコーディングの直接インプットとなるシグネチャ・アルゴリズム仕様であって、コンパイル済みの実装ではない。

### 1-2. ベンチマーク・パフォーマンス計測

本サブは設計仕様の確定を目的とし、実際のスループット・レイテンシの計測値を提供しない。パフォーマンス目標（NFR-PERF-001 等）を達成することは実装フェーズの責任であり、`08_テストケース詳細設計/07_性能テスト設計.md` で TST-perf 識別子として定義する。

### 1-3. OpenAPI 仕様書

全エンドポイントの OpenAPI 3.1 YAML（フィールド定義・リクエスト/レスポンス例・エラースキーマ完全版）は `03_API詳細設計/` サブで管理する。本サブはルーティング定義とハンドラ構造の設計にとどまる。`wnav_terminal_api` と `wnav_master_api` それぞれの OpenAPI 仕様は `03_API詳細設計/` で独立して管理する。

### 1-4. データベース DDL

CREATE TABLE・CREATE INDEX・CREATE VIEW の全文 DDL は `01_データベース詳細設計/` サブで管理する。本サブは sqlx クエリと接続設定の設計にとどまる。

### 1-5. フロントエンド（React Native / React）の詳細設計

ハンディ APP の TypeScript 型定義・SQLite スキーマは `04_ハンディAPP詳細設計/` で、マスタメンテ・管理コンソールは `05_WebAPP詳細設計/` で管理する。

### 1-6. セキュリティ脆弱性診断

OWASP Top 10 対応・ペネトレーションテスト結果は `07_セキュリティ方式設計/` および `08_テストケース詳細設計/06_セキュリティテスト設計.md` で管理する。本サブは JWT 検証・HMAC 署名・RBAC の設計仕様に限定する。

## 2. 前提制約

### 2-1. 上流設計の固定

以下の上流設計内容は本サブで変更しない。変更が必要な場合は上流文書を改訂し、本サブは追従する。

| 固定事項 | 上流文書 |
|---|---|
| 8 クレート構成（MOD-BE-001〜007・MOD-BE-010）| `04_概要設計/02_ソフトウェア方式設計/11_モジュール一覧.md` |
| 4 層依存方向 | `04_概要設計/02_ソフトウェア方式設計/02_レイヤー構成.md` |
| ERR-NNN 識別子 22 件 | `04_概要設計/02_ソフトウェア方式設計/07_例外・エラーハンドリング.md` |
| JWT RS256・RBAC 6 ロール | `04_概要設計/05_外部インターフェース設計/03_認証認可方式.md` |
| TBL-001〜035 テーブル定義 | `04_概要設計/付録/99_設計識別子採番台帳.md` |
| 全 API エンドポイント | `04_概要設計/05_外部インターフェース設計/` |

### 2-2. 2 バイナリの独立プロセス起動

`wnav_terminal_api`（ポート 8080）と `wnav_master_api`（ポート 8081）は独立したプロセスとして起動することを前提とする。両バイナリを同一プロセスで起動することは本サブのスコープ外であり、想定しない。Docker Compose 環境では `terminal-api` と `master-api` の 2 サービスとして定義する。

### 2-3. Rust Edition 2024

本サブのコード例は Rust Edition 2024 を前提とする。Edition 2021 以前では一部の構文が動作しない可能性がある（`async fn in trait` 等）。

### 2-4. sqlx コンパイル時チェック

`sqlx::query!` マクロはコンパイル時に DB スキーマを検証する。`DATABASE_URL` 環境変数と実際の PostgreSQL 16 インスタンスが開発環境に必要である。CI/CD では `sqlx prepare` で生成した `.sqlx/` のキャッシュを使用する（オフラインモード）。

### 2-5. async_trait の Rust 1.75 以降における扱い

Rust 1.75 以降は `async fn in trait` が安定化された。本サブのコード例は `#[async_trait]` マクロと Rust 1.75 の RPITIT（Return Position Impl Trait In Traits）の両方で記述している。実装時は採用する Rust バージョンに合わせて選択すること。

## 3. 将来課題（ver1.0.0 以降）

| 課題 | 説明 | 対応方針 |
|---|---|---|
| 外部 MQ 統合 | Kafka・RabbitMQ への移行 | `wnav_outbox` の Publisher 実装を追加 |
| gRPC 対応 | 高頻度 IoT データ取り込み | `wnav_terminal_api` に tonic crate を追加 |
| WebSocket プッシュ | リアルタイム進捗通知 | `wnav_master_api` に axum WebSocket ハンドラを追加 |
| マルチテナント | 複数工場の論理分離 | factory_id による Row Level Security 強化 |

---

**本節で確定した方針**
- **本サブのスコープ外（実装ファイル・DDL・OpenAPI 仕様・ベンチマーク）を明記し、他サブとの責任境界を確定した。**
- **`wnav_terminal_api` と `wnav_master_api` が独立プロセスで起動することを前提条件として明記した。**
- **上流設計（MOD-BE・ERR-NNN・JWT/RBAC・TBL-NNN）の固定を宣言し、本サブが独断で変更しないことを確定した。**
- **ver1.0.0 以降の将来課題（外部 MQ・gRPC・WebSocket・マルチテナント）を記録し、設計の拡張ポイントを明示した。**
