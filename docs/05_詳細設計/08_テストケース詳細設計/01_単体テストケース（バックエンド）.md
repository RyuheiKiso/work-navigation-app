# 01 単体テストケース（バックエンド）

本章はバックエンド（Rust クレート）の単体テストケース TST-unit-BE-001〜030 を確定する。テスト対象は WorkExecution ステートマシン遷移・HashChainService・RBAC ミドルウェア・JSON Logic 評価エンジン・OutboxConsumer 指数バックオフ・Append-Only 強制の 6 領域である。各テストケースは `cargo test` で実行可能な精度でテスト対象・入力・期待出力を記述する。

---

## 1. WorkExecution ステートマシン遷移（TST-unit-BE-001〜005）

対象: `wnav_domain::WorkExecutionStatus::can_transition_to`（FNC-BE-005）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-BE-001 | `can_transition_to` | — | current=NotStarted, target=InProgress, role=OPERATOR | `Ok(())` | FR-NV-003 |
| TST-unit-BE-002 | `can_transition_to` | — | current=InProgress, target=Completed, role=OPERATOR | `Ok(())` | FR-NV-002 |
| TST-unit-BE-003 | `can_transition_to` | — | current=InProgress, target=Cancelled, role=OPERATOR | `Err(AppError::Forbidden)` | FR-AU-003, BR-BUS-NNN |
| TST-unit-BE-004 | `can_transition_to` | — | current=InProgress, target=Cancelled, role=SUPERVISOR | `Ok(())` | FR-NV-006 |
| TST-unit-BE-005 | `can_transition_to` | — | current=Completed, target=InProgress, role=ADMIN | `Err(AppError::InvalidStateTransition)` | FR-NV-003 |

```rust
// テストコードスケルトン
#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn test_tst_unit_be_001_not_started_to_in_progress() {
        let result = WorkExecutionStatus::NotStarted
            .can_transition_to(&WorkExecutionStatus::InProgress, &Role::Operator);
        assert!(result.is_ok());
    }

    #[test]
    fn test_tst_unit_be_003_cancel_by_operator_forbidden() {
        let result = WorkExecutionStatus::InProgress
            .can_transition_to(&WorkExecutionStatus::Cancelled, &Role::Operator);
        assert!(matches!(result, Err(AppError::Forbidden)));
    }
}
```

## 2. HashChainService 正規化 JSON + SHA-256（TST-unit-BE-006〜010）

対象: `wnav_hash_chain::compute_content_hash`（FNC-BE-009）・`compute_chain_hash`（FNC-BE-010）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-BE-006 | `compute_content_hash` | — | 最小イベント（step_id=null, payload=null）| 決定論的 SHA-256 ハッシュ（再実行で同値）| FR-EV-001 |
| TST-unit-BE-007 | `compute_content_hash` | — | フィールド順序を変えた同一イベント | 同一ハッシュ値（正規化確認）| FR-EV-001 |
| TST-unit-BE-008 | `compute_content_hash` | — | payload フィールドが異なる 2 イベント | 異なるハッシュ値 | FR-EV-002 |
| TST-unit-BE-009 | `compute_chain_hash` | — | prev_hash=[0u8;32], content_hash=known | 期待値と一致（事前計算済み）| FR-EV-001 |
| TST-unit-BE-010 | `compute_chain_hash` | — | prev_hash ≠ [0u8;32], prev_hash を 1 バイト変更 | 異なる chain_hash（雪崩効果確認）| FR-EV-002 |

```rust
#[test]
fn test_tst_unit_be_007_field_order_independence() {
    let event_a = WorkEvent { /* フィールド順: activity, case_id, ... */ };
    let event_b = WorkEvent { /* 同一値だが struct 初期化順が異なる（Rust では順序不変）*/ };
    // canonical_json は BTreeMap でソートするため、同一ハッシュを保証
    assert_eq!(compute_content_hash(&event_a), compute_content_hash(&event_b));
}
```

## 6. Append-Only 強制（TST-unit-BE-026〜030）

対象: `wnav_db::record_step_completed_tx`（FNC-BE-008）・PostgreSQL トリガ

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-BE-026 | DB トリガ（Append-Only）| work_events テーブルに 1 件 INSERT 済み | UPDATE work_events SET activity='tampered' | `PgError: trigger DENY_UPDATE_WORK_EVENTS が拒否` | FR-EV-002, TST-alcoa-004 |
| TST-unit-BE-027 | DB トリガ（Append-Only）| work_events テーブルに 1 件 INSERT 済み | DELETE FROM work_events | `PgError: trigger DENY_DELETE_WORK_EVENTS が拒否` | FR-EV-002, TST-alcoa-004 |
| TST-unit-BE-028 | 冪等性チェック | idempotency_keys に eventId 登録済み | 同一 eventId で `record_step_completed_tx` 実行 | キャッシュ応答（INSERT スキップ）| FR-SY-004 |
| TST-unit-BE-029 | `record_step_completed_tx` | — | 正常コマンド | work_events 1 件 + hash_chain_blocks 1 件 + outbox_events 1 件 がアトミックに挿入される | FR-NV-002, FR-EV-001 |
| TST-unit-BE-030 | `record_step_completed_tx` | — | hash_chain_blocks INSERT 時に一意制約違反を強制（テスト用）| トランザクションがロールバックされ work_events も挿入されない | FR-EV-001, FR-SY-004 |

---

**本節で確定した方針**
- **TST-unit-BE-001〜030 の 30 テストケースを 6 領域（ステートマシン・ハッシュチェーン・RBAC・DSL・Outbox・Append-Only）に分類し、各ケースのテスト対象・入力・期待出力・対応 FR を確定した。**
- **TST-unit-BE-007（フィールド順序独立性）は正規化 JSON の BTreeMap ソートが正しく機能することの核心的証拠であり、単体テストで明示的に検証することを確定した。**
- **TST-unit-BE-029/030 はトランザクションのアトミック性を検証する。work_events・hash_chain_blocks・outbox_events の 3 テーブルへの挿入が全成功または全ロールバックとなることを `sqlx::test` マクロで確認することを確定した。**
