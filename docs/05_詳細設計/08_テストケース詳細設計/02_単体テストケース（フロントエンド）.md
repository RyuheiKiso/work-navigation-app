# 02 単体テストケース（フロントエンド）

本章はフロントエンド（TypeScript / React Native）の単体テストケース TST-unit-FE-001〜020 を確定する。テスト対象は StepEngine.canAdvanceToStep ロックステップチェック・JSON Logic 評価エンジン安全オペレータ・OutboxWorker バックオフ計算・バリデーション関数の 4 領域である。各テストケースは Jest で実行可能な精度でテスト対象・入力・期待出力を記述する。

---

## 1. StepEngine.canAdvanceToStep ロックステップチェック（TST-unit-FE-001〜005）

対象: `StepEngine.canAdvanceToStep`（FNC-FE-001）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-FE-001 | `canAdvanceToStep` | execution.status=InProgress, current_step_index=0 | targetStepIndex=1 | `{ canAdvance: true }` | FR-NV-001, BR-BUS-001 |
| TST-unit-FE-002 | `canAdvanceToStep` | execution.status=InProgress, current_step_index=0 | targetStepIndex=2（スキップ）| `{ canAdvance: false, reason: 'SKIPPED_STEP' }` | FR-NV-001, BR-BUS-001 |
| TST-unit-FE-003 | `canAdvanceToStep` | execution.status=Suspended | targetStepIndex=1 | `{ canAdvance: false, reason: 'EXECUTION_NOT_ACTIVE' }` | FR-NV-001 |
| TST-unit-FE-004 | `canAdvanceToStep` | execution.status=InProgress, prevStep.evidence_required=true, evidenceCount=0 | targetStepIndex=1 | `{ canAdvance: false, reason: 'EVIDENCE_REQUIRED' }` | FR-NV-007, BR-BUS-003 |
| TST-unit-FE-005 | `canAdvanceToStep` | execution.status=InProgress, prevStep.evidence_required=true, evidenceCount=1 | targetStepIndex=1 | `{ canAdvance: true }` | FR-NV-007, BR-BUS-003 |

```typescript
// テストコードスケルトン
describe('StepEngine.canAdvanceToStep', () => {
  it('TST-unit-FE-002: スキップは SKIPPED_STEP を返す', async () => {
    const engine = createStepEngine({ currentStepIndex: 0, status: 'IN_PROGRESS' });
    const result = await engine.canAdvanceToStep(mockExecId, 2, mockDb);
    expect(result).toEqual({ canAdvance: false, reason: 'SKIPPED_STEP' });
  });

  it('TST-unit-FE-004: 証拠なしは EVIDENCE_REQUIRED を返す', async () => {
    mockDb.query.mockResolvedValue({ evidenceCount: 0 });
    const result = await engine.canAdvanceToStep(mockExecId, 1, mockDb);
    expect(result).toEqual({ canAdvance: false, reason: 'EVIDENCE_REQUIRED' });
  });
});
```

## 2. JSON Logic 評価エンジン 安全オペレータ（TST-unit-FE-006〜010）

対象: `JsonLogicEvaluator.evaluateRule`（FNC-FE-005）・`validateRuleDepth`（FNC-FE-006）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-FE-006 | `evaluateRule` | context: { step: { input: { value: 10 } } } | `{"==": [{"var": "step.input.value"}, 10]}` | `true` | FR-NV-010 |
| TST-unit-FE-007 | `evaluateRule` | context: 任意 | `{"eval": ["malicious"]}` | `DslEvaluationError("ERR-VAL-003: Forbidden operator: eval")` | BR-BUS-022 |
| TST-unit-FE-008 | `evaluateRule` | context: { step: { input: { value: 5 } } } | `{"/": [{"var": "step.input.value"}, 0]}` | `DslEvaluationError("ERR-VAL-003: Division by zero")` | BR-BUS-022 |
| TST-unit-FE-009 | `validateRuleDepth` | — | 深度 5 の `if` ネスト | `void（エラーなし）` | BR-BUS-022 |
| TST-unit-FE-010 | `validateRuleDepth` | — | 深度 6 の `if` ネスト | `DslValidationError("ERR-VAL-003: Max nesting depth 5 exceeded (BR-BUS-022)")` | BR-BUS-022 |

```typescript
describe('JsonLogicEvaluator', () => {
  it('TST-unit-FE-007: eval オペレータは禁止', () => {
    expect(() => evaluateRule({ eval: ['malicious'] }, {}))
      .toThrow(DslEvaluationError);
    expect(() => evaluateRule({ eval: ['malicious'] }, {}))
      .toThrow('ERR-VAL-003: Forbidden operator: eval');
  });

  it('TST-unit-FE-010: 深度 6 は拒否', () => {
    const deepRule = buildNestedRule(6); // ヘルパーで深度 6 のルールを構築
    expect(() => validateRuleDepth(deepRule))
      .toThrow('ERR-VAL-003: Max nesting depth 5 exceeded');
  });
});
```

## 3. OutboxWorker バックオフ計算（TST-unit-FE-011〜015）

対象: `OutboxWorker.computeBackoffDelay`（FNC-FE-010）・`dispatchPending`（FNC-FE-009）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-FE-011 | `computeBackoffDelay` | — | attemptNumber=1 | 1000 ms | FR-SY-002 |
| TST-unit-FE-012 | `computeBackoffDelay` | — | attemptNumber=5 | 16000 ms | FR-SY-002 |
| TST-unit-FE-013 | `dispatchPending` | networkState=EMERGENCY_MODE | 呼び出し | 即座に return（API 呼び出し 0 回）| FR-OF-002 |
| TST-unit-FE-014 | `dispatchPending` | networkState=CONNECTED, PENDING 5 件 | 呼び出し | API POST が 5 回呼び出される | FR-SY-001 |
| TST-unit-FE-015 | `dispatchPending` | 5 件中 3 件目が retry_count=5 | 呼び出し | 3 件目が DEAD_LETTERED に遷移、4/5 件目は正常配信 | FR-SY-005 |

```typescript
describe('OutboxWorker', () => {
  it('TST-unit-FE-013: EMERGENCY_MODE では API を呼び出さない', async () => {
    const apiClient = jest.fn();
    await dispatchPending(mockDb, apiClient, 'EMERGENCY_MODE');
    expect(apiClient).not.toHaveBeenCalled();
  });

  it('TST-unit-FE-015: retry_count=5 のイベントは DEAD_LETTERED', async () => {
    setupMockDb({ pendingEvents: [
      { id: 'ev1', retry_count: 0 },
      { id: 'ev2', retry_count: 0 },
      { id: 'ev3', retry_count: 5 }, // DLQ 対象
      { id: 'ev4', retry_count: 0 },
      { id: 'ev5', retry_count: 0 },
    ]});
    await dispatchPending(mockDb, mockApi, 'CONNECTED');
    expect(mockDb.updateStatus).toHaveBeenCalledWith('ev3', 'DEAD_LETTERED');
  });
});
```

## 4. バリデーション関数（TST-unit-FE-016〜020）

対象: `validateStepInput`（各ステップ種別のバリデーション）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-unit-FE-016 | `validateNumericInput` | step.range={min:0, max:100} | value=50.5, unit='mm' | `{ valid: true }` | FR-NV-002 |
| TST-unit-FE-017 | `validateNumericInput` | step.range={min:0, max:100} | value=150 | `{ valid: false, code: 'ERR-VAL-002', message: '0〜100 の範囲外: 150' }` | FR-NV-002, NFR-VAL-001 |
| TST-unit-FE-018 | `validateTextInput` | step.max_length=500 | value=501 文字の文字列 | `{ valid: false, code: 'ERR-VAL-001' }` | FR-NV-002 |
| TST-unit-FE-019 | `validateSelectionInput` | step.options=['A','B','C'] | index=3（範囲外）| `{ valid: false, code: 'ERR-VAL-001' }` | FR-NV-002 |
| TST-unit-FE-020 | `validateSignatureInput` | signRecords に sign_id 存在 | sign_id='existing-sign-uuid' | `{ valid: true }` | FR-NV-008 |

```typescript
describe('validateStepInput', () => {
  it('TST-unit-FE-017: 範囲外数値は ERR-VAL-002 を返す', () => {
    const stepDef = { type: 'Numeric', range: { min: 0, max: 100 } };
    const result = validateNumericInput({ value: 150, unit: 'mm' }, stepDef);
    expect(result.valid).toBe(false);
    expect(result.code).toBe('ERR-VAL-002');
  });
});
```

**本節で確定した方針**
- **TST-unit-FE-001〜020 の 20 テストケースを 4 領域（StepEngine・JSON Logic・OutboxWorker・バリデーション）に分類し、Jest で実行可能な精度でテスト対象・入力・期待出力・対応 FR を確定した。**
- **TST-unit-FE-013 は EMERGENCY_MODE 時に OutboxWorker が API を一切呼び出さないことを API モックの呼び出し回数で検証し、オフライン動作の正確性を保証することを確定した。**
- **TST-unit-FE-017 は ERR-VAL-002 エラーコードの発生を入力バリデーション単体テストで確認する。範囲外の入力が UI に到達する前にバリデーション層で拒否されることを確定した。**
