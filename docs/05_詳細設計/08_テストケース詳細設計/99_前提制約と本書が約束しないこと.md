# 99 前提制約と本書が約束しないこと

本章は `05_詳細設計/08_テストケース詳細設計` サブのスコープ外事項・前提制約・D-TST 項目を明記することで、テスト実施フェーズ（07_テスト）が本書を誤解なく継承できることを保証する。

---

## 1. 本サブが約束しないこと（D-TST 項目）

### D-TST-001: テストコードの実装

本サブはテストケースの「設計」を確定する。`crates/`・`apps/`・`tests/` 配下の実際のテストコード（Rust の `#[test]`・Jest テスト・Playwright スクリプト・k6 スクリプト）の作成は `07_テスト実施フェーズ` で行う。本書に記載したコードブロックはテストケースの意図を示す「スケルトン」であり、コンパイル済みの実装ではない。

### D-TST-002: テストの実際の実行と合否判定

テストケースの実際の実行・テスト結果の記録・バグレポートの作成は `07_テスト実施フェーズ` で行う。本書はテスト条件の「設計」にとどまり、「GREEN」または「RED」の判定結果は本書に含まれない。

### D-TST-003: 100% コードカバレッジの保証

本書は機能要件・非機能要件・業務ルールのトレーサビリティを確保するテストケースセットを設計する。行カバレッジ・分岐カバレッジの実測値は実装後の計測（`cargo llvm-cov`・Istanbul 等）による確認を要する。目標カバレッジ値の設定は `07_テスト実施フェーズ` で行う。

### D-TST-004: 受け入れテスト（UAT）の設計

本書はエンジニアリングテスト（単体・統合・E2E・セキュリティ・性能）のテストケースを設計する。エンドユーザー（製造現場担当者・品質管理担当者）による受け入れテスト（UAT）の設計は `07_テスト実施フェーズ` に含める。

### D-TST-005: ペネトレーションテスト実施

TST-sec 系テストケースは機能的なセキュリティ検証を設計する。本格的なペネトレーションテスト（第三者機関によるレッドチーム演習）の実施は `07_テスト実施フェーズ` で計画する。本書は OWASP ZAP + cargo-audit の自動化ツールによる検証を規定するにとどまる。

### D-TST-006: 負荷テスト専用環境の構築

TST-perf 系テストケースは k6 スクリプトとしきい値を定義する。性能テスト専用環境（本番相当のハードウェアスペック・PostgreSQL チューニング済み状態）の構築は `07_テスト実施フェーズ` で行う。本書は「50 同時ユーザー・100 rps・P95 ≤ 2000 ms」等の目標値を設計上の合否基準として確定する。

## 2. 前提制約

### 2-1. 上流設計の固定

本サブは以下の上流設計内容を変更しない。

| 固定事項 | 上流文書 |
|---|---|
| FR-NNN 86 件の全機能要件 | `03_要件定義/機能要件/` |
| NFR-NNN 130+ 件の非機能要件 | `03_要件定義/非機能要件/` |
| BR-BUS-NNN 45 件の業務ルール | `03_要件定義/機能要件/10_業務ルール.md` |
| UC-001〜022 のユースケース | `03_要件定義/ユースケース/` |
| ERR-NNN エラーコード体系 | `04_概要設計/02_ソフトウェア方式設計/07_例外・エラーハンドリング統一方式.md` |
| TST-NNN 採番規約 | `05_詳細設計/01_詳細設計の識別子規約と採番台帳.md` §4 |

### 2-2. FR カバレッジ完成条件の継続遵守

`00_本書の位置づけと識別子規約.md` §4 で定義した「全 86 FR に TST-NNN ≥ 1 件の割付け」が完成条件である。本書はその条件を満たしている。実装フェーズ以降で FR が追加・変更された場合は、本書を改訂して対応するテストケースを追加することが必須である。

### 2-3. ALCOA+ 検証テストの必須性

TST-alcoa-001〜009 は ver1.0.0 リリース前に全件実行が必須である。これは製造業電子記録に関する法規制（FDA 21 CFR Part 11・PIC/S・GMP ガイドライン）への準拠を証明するものであり、省略や延期は認めない。

### 2-4. 破壊的テストの実行環境制限

TST-alcoa-007（ハッシュチェーン改ざん検出）で定義した破壊的テスト（ブロックデータの手動改ざん）は、テスト環境のみで実行する。`#[sqlx::test]` マクロによるトランザクションロールバックを使用し、本番 DB への影響を完全に遮断する。この制約は `07_テスト実施フェーズ` での実装担当者が厳守する。

## 3. 将来課題（ver1.0.0 以降）

| 課題 | 説明 | 対応方針 |
|---|---|---|
| Fuzzing テスト | JSON Logic ルールへの任意入力でクラッシュを検出 | `cargo-fuzz` による自動 Fuzzing |
| カオスエンジニアリング | PostgreSQL 停止・ネットワーク断のランダム注入 | Chaos Monkey 相当ツールの導入 |
| 回帰テストスイートの自動化 | すべての TST-NNN を CI で自動実行 | GitHub Actions + matrix strategy |
| 第三者ペネトレーションテスト | 独立したセキュリティ会社によるレッドチーム演習 | ver1.0.0 リリース後 90 日以内 |

---

**本節で確定した方針**
- **D-TST-001〜006 の 6 項目を対象外と宣言し、テストケースの「設計」と「実施・実装」の責務を明確に分離した。本書はテストコードのスケルトンを含むが、コンパイル済み実装ではない。**
- **TST-alcoa-001〜009（ALCOA+ 検証）は ver1.0.0 リリース前の全件実行を必須とし、省略や延期を設計段階で禁止することを確定した。**
- **TST-alcoa-007 の破壊的テストは `#[sqlx::test]` マクロのトランザクションロールバックによりテスト環境への限定実行を保証し、本番 DB への誤実行を設計段階で防止することを確定した。**
