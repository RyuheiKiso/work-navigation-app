# 06 セキュリティ・性能テストケース

本章はセキュリティテストケース TST-sec-001〜010 および性能テストケース TST-perf-001〜005 を確定する。セキュリティテストは RBAC・JWT・SQL インジェクション・個人ランキング禁止・ブルートフォース・HMAC 署名の 6 領域、性能テストは k6 による負荷・スループット・ハッシュチェーン検証・SQLite 同期・帳票生成の 5 領域を対象とする。

---

## 1. セキュリティテストケース

### 1-1. RBAC：オペレーターによるマスタ操作拒否（TST-sec-001〜003）

| TST-ID | テスト対象 | 前提条件 | 入力 | 期待出力 | 対応 FR/NFR |
|---|---|---|---|---|---|
| TST-sec-001 | `POST /api/v1/sops` | JWT role=OPERATOR | SOP 作成リクエスト | HTTP 403, `{"error": "ERR-AUTH-003", "message": "Insufficient role"}` | FR-AU-003 |
| TST-sec-002 | `PUT /api/v1/sops/{id}` | JWT role=OPERATOR | SOP 更新リクエスト | HTTP 403 | FR-AU-003 |
| TST-sec-003 | `GET /api/v1/admin/users` | JWT role=OPERATOR | ユーザー一覧取得 | HTTP 403 | FR-AU-003 |

```typescript
// テスト実装スケルトン
it('TST-sec-001: OPERATOR は SOP 作成不可', async () => {
  const token = await getJwtToken({ role: 'OPERATOR' });
  const res = await request(app)
    .post('/api/v1/sops')
    .set('Authorization', `Bearer ${token}`)
    .send({ title: 'Test SOP', operation_id: testOpId, steps: [] });
  expect(res.status).toBe(403);
  expect(res.body.error).toBe('ERR-AUTH-003');
});
```

### 1-3. SQL インジェクション防止（TST-sec-005）

| 項目 | 内容 |
|---|---|
| TST-ID | TST-sec-005 |
| テスト観点 | `sqlx::query!` マクロによるコンパイル時パラメータバインドで SQL インジェクションが不可能 |
| 検証方法 | login_id に `'; DROP TABLE users; --` を入力してログイン API を呼び出す |
| 期待結果 | HTTP 401（DBエラーではない）・users テーブルが存在する |
| 追加確認 | OWASP ZAP の Active Scan で SQLi 脆弱性が検出されないこと |
| 対応 FR/NFR | NFR-SEC-001 |

```typescript
it('TST-sec-005: SQL インジェクション文字列はログイン失敗として処理', async () => {
  const res = await request(app)
    .post('/api/v1/auth/login')
    .send({ login_id: "'; DROP TABLE users; --", password: 'anything' });
  expect(res.status).toBe(401);
  // テーブルが存在することを確認
  const count = await db.query("SELECT COUNT(*) FROM users");
  expect(count[0].count).toBeGreaterThan(0);
});
```

### 1-4. 個人別ランキング拒否（TST-sec-006）

| 項目 | 内容 |
|---|---|
| TST-ID | TST-sec-006 |
| テスト観点 | 集計レポート API が worker_id による GROUP BY を返さない |
| 検証方法 | `GET /api/v1/reports/aggregate?period=DAILY&date=2026-05-16` |
| 期待結果 | レスポンス JSON に `worker_id` キーが存在しない |
| 追加確認 | DB クエリログに `GROUP BY worker_id` が出現しない（pg_stat_statements で確認）|
| 対応 FR/NFR | BR-BUS-029 |

```typescript
it('TST-sec-006: 集計レポートに worker_id が含まれない', async () => {
  const res = await request(app)
    .get('/api/v1/reports/aggregate?period=DAILY&date=2026-05-16')
    .set('Authorization', `Bearer ${adminToken}`);
  expect(res.status).toBe(200);
  const hasWorkerRanking = JSON.stringify(res.body).includes('worker_id');
  expect(hasWorkerRanking).toBe(false);
});
```

### 1-5. ブルートフォース防止（TST-sec-007〜008）

| TST-ID | テスト対象 | 入力 | 期待出力 | 対応 FR |
|---|---|---|---|---|
| TST-sec-007 | ログイン試行制限 | 同一ユーザーへの失敗ログインを 3 回連続送信 | 3 回目: HTTP 401 通常 / 4 回目: HTTP 423 `{"error": "ERR-AUTH-004", "message": "Account locked"}` | FR-AU-005 |
| TST-sec-008 | アカウントロック解除 | 管理者が `POST /api/v1/admin/users/{id}/unlock` | HTTP 200 / 以後のログインが成功する | FR-AU-005 |

```typescript
it('TST-sec-007: 3 回失敗後のロック', async () => {
  for (let i = 0; i < 3; i++) {
    await request(app).post('/api/v1/auth/login')
      .send({ login_id: 'OP001', password: 'wrong' });
  }
  const res = await request(app).post('/api/v1/auth/login')
    .send({ login_id: 'OP001', password: 'wrong' });
  expect(res.status).toBe(423);
  expect(res.body.error).toBe('ERR-AUTH-004');
});
```

## 2. 性能テストケース

### 2-1. k6 負荷テスト（TST-perf-001）

| 項目 | 内容 |
|---|---|
| TST-ID | TST-perf-001 |
| ツール | k6 |
| シナリオ | 50 同時ユーザーが 5 分間、ステップ完了 API をランダムな間隔で叩く |
| 期待結果 | P95 レイテンシ ≤ 2000 ms / エラー率 ≤ 0.1% |
| 対応 NFR | NFR-PERF-001 |

```javascript
// k6 スクリプト（TST-perf-001）
import http from 'k6/http';
import { check, sleep } from 'k6';

export const options = {
  vus: 50,
  duration: '5m',
  thresholds: {
    http_req_duration: ['p(95)<2000'],
    http_req_failed: ['rate<0.001'],
  },
};

export default function () {
  const res = http.post(
    `${__ENV.BASE_URL}/api/v1/step-events`,
    JSON.stringify({ exec_id: __ENV.TEST_EXEC_ID, step_id: __ENV.TEST_STEP_ID,
                     event_id: crypto.randomUUID(), input: { type: 'Numeric', value: 50.0, unit: 'mm' } }),
    { headers: { 'Content-Type': 'application/json', 'Authorization': `Bearer ${__ENV.JWT}` } }
  );
  check(res, { 'status is 201 or 200': (r) => r.status === 201 || r.status === 200 });
  sleep(Math.random() * 2);
}
```

---

**本節で確定した方針**
- **TST-sec-001〜010 のセキュリティテストケースを 6 領域（RBAC・JWT・SQL インジェクション・個人ランキング・ブルートフォース・HMAC）に分類し、各ケースの入力・期待出力・対応 FR を確定した。特に TST-sec-006 は BR-BUS-029 を API レスポンスの JSON 構造レベルで検証し、設計違反を防止する。**
- **TST-perf-001/002 は k6 の `constant-arrival-rate` エグゼキューターで 50 同時ユーザー・100 rps を厳密に制御し、P95 ≤ 2000 ms の NFR-PERF-001 達成を自動化された閾値チェックで確認することを確定した。**
- **TST-perf-003 は Criterion ベンチマークで 100 万件のハッシュチェーン検証を 60 秒以内で完了することを確認し、週次バッチ（BAT-001）の実用性を保証することを確定した。**
