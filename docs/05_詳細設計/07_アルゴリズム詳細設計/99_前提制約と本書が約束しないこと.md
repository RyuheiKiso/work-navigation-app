# 99 前提制約と本書が約束しないこと

本章は `05_詳細設計/07_アルゴリズム詳細設計` サブのスコープ外事項・前提制約・将来課題を明記することで、本サブの成果物の読み手が誤った期待を持つことを防ぐ。

---

## 1. 本サブが約束しないこと

### D-ALG-001: アルゴリズムの実装ファイル作成

本サブは設計仕様書である。`crates/`・`apps/`・`src/` 配下の実際の実装ファイルを作成することは本サブのスコープ外である。コードブロックとして記載された Rust・TypeScript コードはコーディングの直接インプットとなるアルゴリズム仕様であって、コンパイル済みの実装ではない。

### D-ALG-002: 実測パフォーマンス計測値の提供

本サブは処理ロジックの仕様を確定する。実際のスループット・レイテンシ計測値は `08_テストケース詳細設計/06_セキュリティ・性能テストケース.md` の TST-perf 識別子に委ねる。擬似コード内に記載した「100 万イベント < 60 秒」等の値は目標値であり、実測を保証するものではない。

### D-ALG-003: テスト実施・テストコードの作成

本サブで確定したアルゴリズムに対するテストケース設計は `08_テストケース詳細設計/` で行う。実際のテストコード（Rust の `#[test]`・Jest テスト等）の作成は `07_テスト実施フェーズ` で行う。

### D-ALG-004: バッチスケジューラの実装詳細

BAT-001〜010 のスケジュール管理は OS/コンテナのスケジューラ（cron・Kubernetes CronJob）を使用する前提で設計する。スケジューラの選定・設定・監視ダッシュボードの構築は運用設計フェーズ（`09_運用・保守`）で扱う。

### D-ALG-005: WeasyPrint・openpyxl の詳細設定

PDF/A-3 生成（WeasyPrint）・XLSX 生成（openpyxl）の詳細設定（フォントパス・CSS カスタマイズ・セルスタイル詳細）はテンプレートファイルで管理する。テンプレートファイルの作成は `06_実装フェーズ` で行う。

### D-ALG-006: 暗号鍵管理・証明書ライフサイクル

PKCS#7 署名に使用する証明書の発行・更新・失効手順は `09_運用・保守` および `06_セキュリティ方式設計` で管理する。本サブは署名処理のアルゴリズム仕様を確定するにとどまる。

## 2. 前提制約

### 2-1. 上流設計の固定

本サブは以下の上流設計内容を変更しない。

| 固定事項 | 上流文書 |
|---|---|
| FR-NV/EV/SY/OF/BS/RP 全機能要件 | `03_要件定義/機能要件/` |
| BR-BUS-001/003/022/024/029/040 | `03_要件定義/機能要件/10_業務ルール.md` |
| ERR-NNN 識別子体系 | `04_概要設計/02_ソフトウェア方式設計/07_例外・エラーハンドリング統一方式.md` |
| TBL-001〜035 テーブル定義 | `04_概要設計/付録/99_設計識別子採番台帳.md` |
| FNC-BE/FE 採番値 | `02_バックエンド詳細設計/00_` および `04_ハンディAPP詳細設計/00_` |

### 2-2. アスキー図の使用禁止

本サブの全図表は draw.io 形式で作成し SVG エクスポートで使用する（`/docs/CLAUDE.md` 規約）。本サブに記載したコードブロック内のコメントは図ではなく擬似コードの一部であり、この規約の対象外とする。

### 2-3. eval() および動的コード生成の禁止

`02_JSON_Logic評価エンジン詳細設計.md` §2 に記載の通り、JSON Logic 評価エンジンは `eval()` および動的コード生成を一切使用しない。この制約は `04_概要設計` の P3 原則（セキュリティ）から継承したものであり、本サブ以降の全フェーズで有効である。

### 2-4. 個人別ランキング禁止の継続

BAT-010 および RP-006 の集計クエリは BR-BUS-029（個人別ランキングの永続的非実装）に基づき、`GROUP BY worker_id` を使用しない。この制約はクエリ設計レベルで強制し、実装フェーズでの例外を認めない。

## 3. 将来課題（ver1.0.0 以降）

| 課題 | 説明 | 対応方針 |
|---|---|---|
| AI 異常検知 | ステップ完了時間の統計的異常を検出 | 別モジュール（ALG-025）として追加 |
| ストリーム処理 | 高頻度 IoT センサーデータのリアルタイム処理 | Kafka Streams または Flink の検討 |
| ML モデル推論 | SOP 最適化提案の自動生成 | 外部 AI サービスとのインターフェース設計が必要 |
| 多言語帳票対応 | 英語・中国語対応 PDF | Jinja2 テンプレートの i18n 拡張 |

---

**本節で確定した方針**
- **本サブのスコープ外（実装ファイル・パフォーマンス計測・テストコード・スケジューラ実装・証明書管理）を明記し、実装フェーズ・運用フェーズとの責任境界を確定した。**
- **eval() 禁止・個人別ランキング禁止・アスキー図禁止の 3 制約が本サブ全体に適用されることを再宣言した。いずれも上流設計からの継承であり、本サブ単独での例外を認めない。**
- **ver1.0.0 以降の将来課題（AI 異常検知・ストリーム処理・ML 推論）を記録し、現時点での設計範囲から明示的に除外した。**
