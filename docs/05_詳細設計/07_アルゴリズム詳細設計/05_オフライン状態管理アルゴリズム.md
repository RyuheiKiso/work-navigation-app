# 05 オフライン状態管理アルゴリズム

本章はハンディ APP の NetworkStateManager（FNC-FE-011〜013）が実装するオフライン状態遷移（ALG-011）・EMERGENCY_MODE 時の JWT 検証キャッシュ（ALG-012）・リカバリ後の Outbox 一括配信（ALG-013）の完全仕様を確定する。対応する機能要件は FR-OF-001〜004、業務ルールは BR-BUS-040（キャッシュ JWT 有効期間 8 時間）である。

---

## 1. ネットワーク状態の定義

図: fig_dd_alg_network_states（img/ 配下）を参照。

| 状態 | 説明 | ログイン可否 | ステップ実行 | Outbox 配信 | マスタ同期 |
|---|---|---|---|---|---|
| `CONNECTED` | API P95 正常 | 可 | 可 | 実行 | 実行 |
| `DEGRADED` | API P95 が閾値超過 | 可 | 可 | 実行（低頻度）| 実行（低頻度）|
| `DISCONNECTED` | API タイムアウト | 不可（新規）| 可（キャッシュ）| 停止 | 停止 |
| `EMERGENCY_MODE` | 断線 5 分超過 | 不可（新規）| 可（キャッシュ）| 停止・蓄積 | 停止 |

---

## 2. 状態遷移アルゴリズム（ALG-011, FNC-FE-011）

```
ALGORITHM NetworkStateManager.transitionTo(newState):
  VALID_TRANSITIONS ← {
    CONNECTED    → DEGRADED      : api_p95 > CFG-007 (threshold_ms)
    CONNECTED    → DISCONNECTED  : api_timeout (30s)
    DEGRADED     → CONNECTED     : api_p95 ≤ CFG-007
    DEGRADED     → DISCONNECTED  : api_timeout (30s)
    DISCONNECTED → CONNECTED     : successful_api_call
    DISCONNECTED → EMERGENCY_MODE: disconnected_duration > CFG-008 (5min)
    EMERGENCY_MODE → CONNECTED   : successful_api_call
  }

  IF (currentState, newState) NOT IN VALID_TRANSITIONS:
    -- 無効な遷移は無視（ログに記録のみ）
    LOG WARN: 'Invalid transition ignored: {currentState} → {newState}'
    RETURN

  previousState ← currentState
  currentState ← newState

  -- 遷移後の処理
  CASE newState:
    EMERGENCY_MODE:
      startEmergencyModeTimer()
      -- OutboxWorker の定期配信を停止
      OutboxWorker.pause()
      LOG INFO: 'EMERGENCY_MODE activated: offline > 5min'

    CONNECTED:
      IF previousState IN { DISCONNECTED, EMERGENCY_MODE }:
        -- リカバリ処理を開始
        startRecovery()
      OutboxWorker.resume()

    DISCONNECTED:
      disconnectedAt ← NOW()
      -- 5 分タイマー開始（EMERGENCY_MODE 遷移条件）
      startDisconnectionTimer(5 * 60 * 1000)
```

### 2-1. TypeScript 型定義

```typescript
type NetworkState = 'CONNECTED' | 'DEGRADED' | 'DISCONNECTED' | 'EMERGENCY_MODE';

interface NetworkStateManager {
  readonly currentState: NetworkState;
  readonly disconnectedAt?: Date;
  readonly emergencyModeAt?: Date;

  /** FNC-FE-011: 状態遷移 */
  transitionTo(newState: NetworkState): void;

  /** FNC-FE-012: リカバリ処理 */
  startRecovery(): Promise<void>;

  /** FNC-FE-013: キャッシュ JWT 検証 */
  validateCachedJwt(): JwtValidationResult;
}
```

### 2-2. 設定値（CFG 参照）

| CFG-ID | 設定項目 | デフォルト値 | 説明 |
|---|---|---|---|
| CFG-007 | `api_p95_threshold_ms` | 2000 ms | CONNECTED → DEGRADED の閾値 |
| CFG-008 | `emergency_mode_threshold_sec` | 300 s (5 min) | DISCONNECTED → EMERGENCY_MODE の閾値 |
| CFG-009 | `api_timeout_sec` | 30 s | CONNECTED/DEGRADED → DISCONNECTED のタイムアウト |

## 3. EMERGENCY_MODE 時の JWT 検証キャッシュ（ALG-012, FNC-FE-013）

EMERGENCY_MODE では新規ログインを禁止し、ローカルにキャッシュした JWT のみを有効とする。キャッシュ JWT の有効期間は取得から 8 時間（BR-BUS-040）に制限する。

```
ALGORITHM validateCachedJwt():
  INPUT: なし（ローカル SQLite の jwt_cache テーブルを使用）
  OUTPUT:
    JwtValidationResult {
      valid         : Boolean,
      user_id       : UUID | null,
      role          : Role | null,
      reason        : JwtRejectReason | null,
    }

  STEP 1: キャッシュ取得
    cached ← SELECT * FROM jwt_cache
              WHERE user_id = CURRENT_USER_ID
              ORDER BY cached_at DESC
              LIMIT 1

    IF cached IS NULL:
      RETURN { valid: false, reason: 'NO_CACHED_JWT' }

  STEP 2: 発行時刻からの経過確認（BR-BUS-040）
    elapsed ← NOW() - cached.cached_at
    IF elapsed > 8 * 3600 seconds:
      RETURN { valid: false, reason: 'CACHED_JWT_EXPIRED' }
      -- 8 時間超過は強制ログアウト

  STEP 3: JWT 署名検証（ローカル公開鍵）
    -- RS256 公開鍵はアプリバンドルに含まれる（定期更新はマスタ同期で取得）
    decoded ← verifyJwtLocally(cached.token, localPublicKey)
    IF decoded IS NULL:
      RETURN { valid: false, reason: 'JWT_SIGNATURE_INVALID' }

  STEP 4: JWT 期限確認
    IF decoded.exp < NOW_UNIX():
      RETURN { valid: false, reason: 'JWT_EXPIRED' }

  RETURN { valid: true, user_id: decoded.sub, role: decoded.role, reason: null }
```

### 3-1. jwt_cache テーブル（ローカル SQLite）

```sql
-- ローカル SQLite スキーマ（TBL-L-004 参照）
CREATE TABLE IF NOT EXISTS jwt_cache (
    id         TEXT     PRIMARY KEY,  -- UUID
    user_id    TEXT     NOT NULL,
    token      TEXT     NOT NULL,
    role       TEXT     NOT NULL,
    cached_at  TEXT     NOT NULL,     -- ISO 8601
    expires_at TEXT     NOT NULL,     -- ISO 8601
    created_at TEXT     NOT NULL DEFAULT (datetime('now'))
);

CREATE INDEX IF NOT EXISTS idx_jwt_cache_user_id
ON jwt_cache (user_id, cached_at DESC);
```

## 4. リカバリアルゴリズム（ALG-013, FNC-FE-012）

EMERGENCY_MODE または DISCONNECTED から CONNECTED に遷移した際に実行するリカバリ処理は、以下の順序で実施する。

```
ALGORITHM startRecovery():
  INPUT: なし
  OUTPUT: RecoveryResult { success: boolean, syncedEvents: number, masterSynced: boolean }

  STEP 1: 接続確認
    -- CONNECTED 状態であることを再確認する
    pingResult ← await api.ping()
    IF pingResult.failed:
      RETURN { success: false, reason: 'CONNECTION_LOST_DURING_RECOVERY' }

  STEP 2: ローカル Outbox の一括配信（順序保証）
    pendingEvents ← SELECT * FROM local_outbox_events
                    WHERE status = 'PENDING'
                    ORDER BY created_at ASC  -- 時系列順を厳守
    FOR EACH event IN pendingEvents:
      result ← await sendEvent(event, api)
      IF result.success:
        UPDATE local_outbox_events SET status = 'SENT' WHERE id = event.id
      ELSE IF result.nonRetryable:
        UPDATE local_outbox_events SET status = 'DEAD_LETTERED' WHERE id = event.id
      ELSE:
        -- 配信失敗は通常の Outbox バックオフに委ねる
        applyBackoff(event)
        BREAK  -- 順序保証のため後続イベントを止める

  STEP 3: マスタデータ同期
    syncResult ← await api.pullMasterSync()
    IF syncResult.success:
      upsertLocalMaster(syncResult.data)

  STEP 4: ハッシュチェーン整合性の確認
    -- オフライン中に記録したイベントのハッシュチェーンをサーバー側と照合
    localLatestBlock ← getLocalLatestHashBlock()
    serverLatestBlock ← await api.getLatestHashBlock(case_id=currentExecutionId)
    IF localLatestBlock.chain_hash ≠ serverLatestBlock.chain_hash:
      -- 不整合の場合は監督者に通知し、作業を一時停止
      notifySupervisor('HASH_CHAIN_INCONSISTENCY')
      suspendCurrentExecution()

  STEP 5: 作業セッション状態の更新
    -- オフライン中に完了イベントを記録していた場合、サーバー側の状態に反映
    IF localExecution.status = 'COMPLETED' AND serverExecution.status = 'IN_PROGRESS':
      await api.syncExecutionStatus(execId=localExecution.id, status='COMPLETED')

  RETURN { success: true, syncedEvents: LENGTH(pendingEvents), masterSynced: syncResult.success }
```

## 5. 監視・アラートポリシー

| 条件 | アラート | 対応 |
|---|---|---|
| EMERGENCY_MODE 遷移 | ローカル通知（赤色バナー）| 上長への報告を促すメッセージを表示 |
| キャッシュ JWT 期限超過 | ログアウト強制 | 再接続後にログインを要求 |
| ハッシュチェーン不整合 | 監督者通知 + 作業停止 | 管理コンソールでの手動確認が必要 |
| DEAD_LETTERED 発生 | バックエンドの DLQ アラート | 管理コンソールの Outbox DLQ 画面で確認 |

---

**本節で確定した方針**
- **ネットワーク状態は CONNECTED/DEGRADED/DISCONNECTED/EMERGENCY_MODE の 4 状態とし、EMERGENCY_MODE への遷移は断線 5 分超過（CFG-008）の条件のみとすることを確定した。EMERGENCY_MODE 中はログイン禁止・Outbox 停止・キャッシュ JWT による認証継続を行う。**
- **キャッシュ JWT の有効期間は取得から 8 時間（BR-BUS-040）に限定し、期限超過時は強制ログアウトとすることを確定した。ローカル公開鍵による RS256 署名検証は維持する。**
- **リカバリアルゴリズム（ALG-013）は Outbox 配信 → マスタ同期 → ハッシュチェーン照合 → 作業セッション状態更新の順序で実行し、Outbox は created_at 昇順の順序保証付きで一括配信することを確定した。**
