# 03a ハッシュチェーン破断時の補正レコード継続規則（ALG-025）

本章は `09_運用・保守/障害対応/08_ハッシュチェーン破断対応.md` の P1 プレイブックに基づき、
補正レコード INSERT 時のハッシュチェーン継続アルゴリズム（ALG-025）を規定する。

関連: FNC-BE-017（compute_correction_chain_hash）、ADR-008

---

## 1. 前提と設計判断

**D2（ADR-008）**: 補正レコードは破断ブロックの chain_hash を継承する（fork 禁止）。
破断ブロック自体は Append-only 原則により削除・更新しない（ALCOA+ Original）。

---

## 2. 入力契約

| 入力 | 型 | 説明 |
|---|---|---|
| `broken_at_block_id` | UUID v7 | 破断が検知されたブロック ID |
| `is_correction` | bool | TRUE 固定 |
| `original_record_id` | UUID | 訂正対象の work_events.event_id |
| `correction_reason` | TEXT | 訂正理由（自由記述、必須）|
| `approver_primary` | UUID | 承認者 1 名目の user_id |
| `approver_secondary` | UUID | 承認者 2 名目の user_id（approver_primary と異なること）|
| `broken_block_chain_hash` | [u8; 32] | 破断ブロックの chain_hash 値 |

---

## 3. 補正ブロックの content_hash 計算

既存 `compute_content_hash`（FNC-BE-009）を再利用する。
payload に以下の補正メタデータを含めた canonical JSON（BTreeMap アルファベット順）で計算する:

```json
{
  "approver_primary": "<UUID>",
  "approver_secondary": "<UUID>",
  "correction_reason": "<TEXT>",
  "is_correction": true,
  "original_record_id": "<UUID>"
}
```

---

## 4. 補正ブロックの prev_hash 計算（FNC-BE-017）

```
correction_prev_hash = broken_block_chain_hash
correction_chain_hash = SHA-256(correction_prev_hash || correction_content_hash)
```

**破断ブロックは削除しない**。補正ブロックは破断ブロックの chain_hash を `prev_hash` として引き継ぎ、
同一チェーン上に連続する正規ブロックとして扱う。

---

## 5. 検証アルゴリズムへの影響（ALG-008 拡張）

週次検証ジョブ（BAT-001）は `is_correction=TRUE` のブロックを検出した場合:

1. `broken_at_block_id` を特定する
2. 破断ブロックの chain_hash を取得する
3. 補正ブロックの prev_hash が破断ブロックの chain_hash と一致するか検証する
4. 補正ブロック以降は通常の連続性検証を継続する
5. `hash_chain_verification_results.status` を `'CORRECTED'` として記録し、
   `correction_block_ids` に補正ブロックの event_id を格納する

---

## 6. hash_chain_verification_results 拡張 DDL

```sql
ALTER TABLE hash_chain_verification_results
  ADD COLUMN correction_block_ids UUID[] NULL,
  ALTER COLUMN status TYPE TEXT;

ALTER TABLE hash_chain_verification_results
  ADD CONSTRAINT chk_hcvr_status
  CHECK (status IN ('PASSED', 'FAILED', 'CORRECTED'));
```

---

## 7. 禁止事項

- 破断ブロックの chain_hash / content_hash の直接 UPDATE（ALCOA+ Original 違反）
- 補正ブロックに独立した genesis（prev_hash = [0u8;32]）を割り当てること（ADR-008 違反 / フォーク禁止）
- approver_primary と approver_secondary に同一 user_id を使用すること（NFR-SEC-048 違反）
