# 99 前提制約と本書が約束しないこと

本章は `05_詳細設計/04_ハンディAPP詳細設計` サブの前提制約・設計範囲の明示的な境界・本書が確定しないことの宣言を確定する。

---

## 1. 上流設計からの固定制約

以下の設計事項は `04_概要設計` または `03_要件定義` で確定済みであり、本サブは変更を行わない。

| 制約 | 出典 | 内容 |
|---|---|---|
| MOD-FE-HA-001〜008 の責務 | MOD カタログ（MOD-11） | 各モジュールの責務・依存関係・ファイルパスは変更不可 |
| StepInputType（9 種）| 拡張 Step エンジン設計（04-04）| 標準 4 タイプは変更・削除不可。拡張タイプは TBL-029 から動的ロード |
| JSON Logic 採用・eval 禁止 | 拡張 Step エンジン設計（04-04）| json-logic-js のみ。eval / Function constructor は全モジュールで禁止 |
| SQLCipher AES-256 暗号化 | キャッシュとオフラインストレージ方式（04-09）| 全 SQLite テーブルを SQLCipher で暗号化。例外なし |
| WAL モード | キャッシュとオフラインストレージ方式（04-09）| PRAGMA journal_mode = WAL は起動時に必ず設定 |
| ネットワーク 4 段階 | キャッシュとオフラインストレージ方式（04-09）| CONNECTED / DEGRADED / DISCONNECTED / EMERGENCY_MODE の 4 段階 |
| マスタ同期間隔 CFG-007 | キャッシュとオフラインストレージ方式（04-09）| デフォルト 60 分。オフライン時は古いキャッシュを継続使用 |
| ロックステップ強制 BR-BUS-001 | 業務ルール定義 | 前 Step 未完了での Step 進行を禁止 |
| 証拠必須ゲート BR-BUS-002 | 業務ルール定義 | evidenceRequired = true の Step は証拠なしで完了不可 |
| Exif 除去 BR-BUS-028 | 業務ルール定義 | 写真撮影時は必ず Exif を除去してから SHA-256 計算 |
| GS1 形式検証 BR-BUS-034 | 業務ルール定義 | QR/バーコードスキャン時は GS1 形式を検証 |
| SCR-HA-001〜015 の責務 | 画面一覧と機能配置（03-01）| 15 画面の責務・アクセスロール・主 FR は変更不可 |
| TypeScript strict mode | CLAUDE.md 技術スタック | strict: true は全ファイルに適用。any 型禁止 |

## 4. 既知のリスクと設計上の留保

| リスク | 設計上の対処 | 未解決事項 |
|---|---|---|
| react-native-vision-camera v4 の API 変更 | インターフェース層（PhotoCaptureService）で吸収 | v5 以降への対応は実装フェーズで判断 |
| SQLCipher の React Native ビルド互換性 | react-native-sqlcipher を採用し、expo-sqlite との競合を避ける | ネイティブビルドの検証は 06_実装で実施 |
| json-logic-js の型定義（@types/json-logic-js）の不完全性 | 型キャスト箇所を最小化し、ラッパー関数で型を明確化 | 型エラーが残る場合は ambient declaration で補完 |
| Emergency Mode 時のデータ損失リスク | Outbox 蓄積継続・wnav.db.bak バックアップで対応 | 5 分以上の切断時の未送信データは事後的に確認が必要 |

**本節で確定した方針**
- **本サブ（04_ハンディAPP詳細設計）は IPA 2.5.1 の「TypeScript インターフェース・ステートマシン・SQLite スキーマ・状態管理」を完全に記述し、src/ 配下の実装ファイル作成は 06_実装フェーズに委ねる境界を明示した。**
- **上流確定事項（MOD-FE-HA 責務・BR-BUS-NNN・TypeScript strict mode）は本サブのすべての章で不変制約として遵守し、変更を必要とする発見があった場合は上位設計の変更管理プロセス経由とする。**
- **既知リスク（react-native-vision-camera API 変更・SQLCipher ビルド互換性）はインターフェース層による吸収を設計で担保し、詳細は 06_実装フェーズで解決することを明示した。**
