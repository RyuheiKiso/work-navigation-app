# 08 シードデータ・テストフィクスチャ設計

本章の責務は、本番環境で必要なシードデータの定義と、テスト用フィクスチャの生成方針を確定することである。シードデータは TBL-017 roles の固定 6 行とシステム設定値に限定し、テストフィクスチャは Rust `fake` クレートと TypeScript `@faker-js/faker` を使用した完全合成データ（PII なし）で構成する。

## 2. テストフィクスチャ設計

### 2-1. 設計原則

- テストデータに本物の PII（個人識別情報）を使用しない。全てのデータは合成（synthetic）である。
- 氏名: `fake` クレートの `Name::fake()` / faker-js の `faker.person.fullName()` で生成する
- 会社名・製品名: faker の `Company::fake()` で生成する
- UUID: Rust では `uuid::Uuid::new_v7()` で生成する
- 固定 seed 値を使用して再現可能なテストデータを生成する

### 2-2. Rust テストフィクスチャ（バックエンド単体テスト・統合テスト）

```rust
// crates/wnav-db/tests/fixtures/mod.rs
// Rust fake クレートを使用した合成テストフィクスチャ

use fake::{Fake, Faker};
use fake::faker::name::en::*;
use fake::faker::lorem::en::*;
use uuid::Uuid;
use chrono::Utc;

/// テスト用 User フィクスチャ
pub struct UserFixture {
    pub user_id: Uuid,
    pub login_id: String,
    pub display_name: String,
}

impl UserFixture {
    pub fn new_operator(index: u32) -> Self {
        // 固定プレフィックス + 連番で一意のログイン ID を生成する
        Self {
            user_id: Uuid::now_v7(),
            login_id: format!("test_operator_{:04}", index),
            display_name: format!("テストオペレーター {}", index),
        }
    }

    pub async fn insert(&self, pool: &sqlx::PgPool) -> sqlx::Result<()> {
        sqlx::query!(
            r#"
            INSERT INTO users (user_id, login_id, display_name, is_active)
            VALUES ($1, $2, $3, TRUE)
            "#,
            self.user_id,
            self.login_id,
            self.display_name,
        )
        .execute(pool)
        .await?;
        Ok(())
    }
}

/// テスト用 Process フィクスチャ
pub struct ProcessFixture {
    pub process_id: Uuid,
    pub process_code: String,
    pub name_ja: String,
}

impl ProcessFixture {
    pub fn new(index: u32) -> Self {
        Self {
            process_id: Uuid::now_v7(),
            process_code: format!("TST-{:03}", index),
            name_ja: format!("テストプロセス {}", index),
        }
    }

    pub async fn insert(&self, pool: &sqlx::PgPool) -> sqlx::Result<()> {
        sqlx::query!(
            r#"
            INSERT INTO processes (process_id, process_code, name, is_active)
            VALUES ($1, $2, $3::jsonb, TRUE)
            "#,
            self.process_id,
            self.process_code,
            serde_json::json!({"ja": self.name_ja, "en": format!("Test Process {}", self.process_id)}),
        )
        .execute(pool)
        .await?;
        Ok(())
    }
}

/// テスト用 WorkEvent フィクスチャ（ハッシュチェーン付き）
pub struct WorkEventFixture {
    pub event_id: Uuid,
    pub case_id: Uuid,
    pub activity: String,
    pub resource: Uuid,
    pub sop_version_id: Uuid,
    pub terminal_id: Uuid,
    pub prev_hash: String,
    pub content_hash: String,
}

impl WorkEventFixture {
    /// ハッシュチェーンの先頭イベント（genesis）を生成する
    pub fn genesis(case_id: Uuid, resource: Uuid, sop_version_id: Uuid, terminal_id: Uuid) -> Self {
        let event_id = Uuid::now_v7();
        let prev_hash = "0".repeat(64); // genesis ブロック
        let content = format!("{}{}{}", event_id, case_id, "work_started");
        let content_hash = sha256_hex(&content);
        Self {
            event_id,
            case_id,
            activity: "work_started".to_string(),
            resource,
            sop_version_id,
            terminal_id,
            prev_hash,
            content_hash,
        }
    }
}

fn sha256_hex(input: &str) -> String {
    use sha2::{Sha256, Digest};
    let mut hasher = Sha256::new();
    hasher.update(input.as_bytes());
    format!("{:x}", hasher.finalize())
}
```

```typescript
// PII なし確認: faker.person.firstName() は架空の名前を生成するため PII に該当しない

interface StepFixture {

  instruction_text: { ja: string; en?: string; 'ja-simple'?: string };
  judgment_condition: object | null;
  evidence_required: boolean;
  fmea_rpn_flag: boolean;
  skill_level_required: number;
```

### 2-4. テストデータ PII 禁止ルール

| 禁止事項 | 理由 | 代替 |
|---|---|---|
| 実在する人物の氏名 | 個人情報保護法 | faker.person.lastName() + faker.person.firstName() の組み合わせ |
| 実在するメールアドレス | 不正アクセスリスク | `test_{uuid}@example.com` 形式 |
| 実在する電話番号 | 個人情報保護法 | `000-0000-{index}` 形式 |
| 実在する住所 | 個人情報保護法 | faker の架空住所 |
| 本番 DB からの取得データ | GDPR・個人情報保護法 | 全て合成データのみ使用 |
| 実在する企業コード | 取引先情報 | `TST-NNN` 形式のダミーコード |

### 3-2. E2E テストのデータ初期化

```typescript
// apps/handy-app/tests/e2e/global-setup.ts

import { request } from '@playwright/test';
import { createUserFixture } from '../fixtures';

export async function globalSetup() {
  // E2E テスト用の最小データセットを管理 API 経由で投入する
  const apiContext = await request.newContext({

  const operator = createUserFixture({ login_id: 'e2e_test_operator' });
```

---

**本節で確定した方針**
- **本番シードデータは TBL-017 roles の固定 6 行と TBL-029 step_type_definitions の 9 種定義に限定し、業務データのシードを禁止した。シードは `ON CONFLICT DO NOTHING` で冪等な再実行を保証する。**
- **テストフィクスチャは Rust `fake` クレート（バックエンド）と `@faker-js/faker` の `locale/ja`（フロントエンド / E2E）で生成し、実在する PII を一切使用しない合成データポリシーを確定した。**
- **Rust 統合テストはトランザクションロールバック方式でフィクスチャのライフサイクルを管理し、テスト間の副作用を排除する。E2E テストは管理 API 経由でフィクスチャを投入し、globalTeardown でクリーンアップする。**
