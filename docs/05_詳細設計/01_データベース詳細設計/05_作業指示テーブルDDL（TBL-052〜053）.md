# 05 作業指示テーブル DDL（TBL-052〜053）

本章の責務は、Push 型作業指示機能（UC-041・UC-042）を支えるテーブル（DDL-052〜053）の CREATE TABLE 全文を確定することである。Append-only ポリシー・ロール制御方針・status 遷移・インデックス定義を全て含む。

> **図 1（作成予定）**: work_assignments ステータス遷移図
> 図は `drawio-authoring` スキルで別途作成する。

> **NOTE: ロール制御方針**
> 以下のテーブルは PostgreSQL ロールレベルで一部列の UPDATE を制限する。
> - `app_event_writer` ロール: `INSERT, SELECT` を許可
> - `app_read_write` ロール: UPDATE 許可列（後述）のみ UPDATE を許可
> - work_assignments の Append-only 列（`source_payload`・`external_order_id`・`external_system`・`received_at`・`idempotency_key`）は UPDATE 禁止（ADR-010 準拠）
> - sse_dispatch_log は Append-only テーブルとし、`delivery_status`・`ack_at`・`retry_count` の UPDATE のみ許可する
>
> ```sql
> -- ロール分離 DDL（マイグレーション V20260518XXXXXX__add_work_assignments.sql で実行）
> REVOKE UPDATE, DELETE ON work_assignments FROM PUBLIC;
> REVOKE UPDATE, DELETE ON sse_dispatch_log FROM PUBLIC;
> -- app_event_writer は INSERT + SELECT のみ
> GRANT INSERT, SELECT ON work_assignments TO app_event_writer;
> GRANT INSERT, SELECT ON sse_dispatch_log TO app_event_writer;
> -- app_read_write は status 系列の UPDATE を許可
> GRANT UPDATE (status, case_id, dispatched_at, accepted_at, completed_at) ON work_assignments TO app_read_write;
> GRANT UPDATE (delivery_status, ack_at, retry_count) ON sse_dispatch_log TO app_read_write;
> ```

## 1. 作業指示系

### 1-1. DDL-052: work_assignments（TBL-052）— 限定可変（Append-only 列あり）

```sql
-- DDL-052: TBL-052 work_assignments
-- Push 型作業指示テーブル。親機（ERP/MES）から受信した作業指示を管理する。
-- Append-only 列（source_payload / external_order_id / external_system / received_at / idempotency_key）は
-- INSERT 後 UPDATE 禁止（ADR-010 準拠）。
-- status / case_id / dispatched_at / accepted_at / completed_at のみ UPDATE 可。
-- NOTE: factory_id は予約フィールド。ver1.0.0 では定数 UUID '00000000-0000-7000-8000-000000000001' を使用する。
-- NOTE: factories テーブルは ver1.0.0 では作成しない（04_概要設計/99 §2-5 準拠）。
CREATE TABLE work_assignments (
    assignment_id           UUID            PRIMARY KEY DEFAULT gen_random_uuid(),
    external_order_id       TEXT            NOT NULL,
    external_system         TEXT            NOT NULL,
    work_pattern_id         UUID            NOT NULL REFERENCES work_patterns(work_pattern_id),
    lot_id                  UUID            REFERENCES lots(lot_id),
    target_terminal_id      UUID            NOT NULL REFERENCES devices(device_id),
    suggested_worker_id     UUID            REFERENCES users(user_id),
    suggested_equipment_id  UUID            REFERENCES equipments(equipment_id),
    due_at                  TIMESTAMPTZ,
    priority                SMALLINT        NOT NULL DEFAULT 2
                                            CHECK (priority IN (1, 2, 3)),
    status                  TEXT            NOT NULL DEFAULT 'pending'
                                            CHECK (status IN (
                                                'pending',
                                                'pending_resolution',
                                                'dispatched',
                                                'accepted',
                                                'in_progress',
                                                'completed',
                                                'rejected',
                                                'cancelled',
                                                'expired'
                                            )),
    case_id                 UUID            REFERENCES work_executions(case_id),
    source_payload          JSONB           NOT NULL,
    idempotency_key         UUID            NOT NULL,
    received_at             TIMESTAMPTZ     NOT NULL DEFAULT now(),
    dispatched_at           TIMESTAMPTZ,
    accepted_at             TIMESTAMPTZ,
    completed_at            TIMESTAMPTZ,
    factory_id              UUID            NOT NULL,

    CONSTRAINT uq_wa_idempotency UNIQUE (external_system, idempotency_key)
);

COMMENT ON TABLE  work_assignments IS 'TBL-052 — Push 型作業指示テーブル。親機（ERP/MES）から受信した作業指示を管理する。source_payload / external_order_id / external_system / received_at / idempotency_key は Append-only（ADR-010）。status 遷移は本ドキュメント §1-1-2 参照。';
COMMENT ON COLUMN work_assignments.assignment_id        IS 'UUID v4。gen_random_uuid() で自動生成。外部システムへの公開 ID としては使用しない。';
COMMENT ON COLUMN work_assignments.external_order_id    IS '親機（ERP/MES）側の作業指示番号。Append-only（INSERT 後変更不可）。external_system と組み合わせて人間が判読する識別子として機能する。';
COMMENT ON COLUMN work_assignments.external_system      IS '送信元システム識別子（例: "SAP_PP"）。Append-only。idempotency_key との複合 UNIQUE 制約で重複受信を防止する。';
COMMENT ON COLUMN work_assignments.work_pattern_id      IS 'external_key_binding により解決された内部 work_pattern_id。作業手順（SOP）を特定する。';
COMMENT ON COLUMN work_assignments.lot_id               IS 'lot_id_ext が提供されている場合に external_key_binding で解決した内部 lot_id。NULL = ロット未紐付け。';
COMMENT ON COLUMN work_assignments.target_terminal_id   IS 'target_terminal_key を external_key_binding で解決した端末 device_id。割当先端末を特定する。';
COMMENT ON COLUMN work_assignments.suggested_worker_id  IS '推奨作業員の user_id（参考情報・作業員は拒否可）。解決失敗時は NULL（ERR-BIZ-027 は発生しない）。';
COMMENT ON COLUMN work_assignments.suggested_equipment_id IS '推奨設備の equipment_id（参考情報・強制しない）。解決失敗時は NULL。';
COMMENT ON COLUMN work_assignments.due_at               IS '作業期限（TIMESTAMPTZ）。NULL = 期限なし扱い。BAT-015 が定期的に due_at 超過を検知して expired に遷移させる。';
COMMENT ON COLUMN work_assignments.priority             IS '優先度: 1=緊急 / 2=通常 / 3=低。CHECK 制約で 1/2/3 のみ許可。priority=1 の場合 UI で蛍光色強調表示（FR-NV-014）。';
COMMENT ON COLUMN work_assignments.status               IS 'ステータス遷移は §1-1-2 参照。9 種の列挙値のみ許可（CHECK 制約）。';
COMMENT ON COLUMN work_assignments.case_id              IS 'UC-042 で作業員が「開始」をタップ後に生成・バインドされる work_executions.case_id。それ以前は NULL。';
COMMENT ON COLUMN work_assignments.source_payload       IS '受信したリクエストボディ全体の JSONB スナップショット。Append-only。外部キー解決の事後検証・監査ログとして使用する。';
COMMENT ON COLUMN work_assignments.idempotency_key      IS '親機が送信した Idempotency-Key ヘッダ値（UUID）。external_system との複合 UNIQUE 制約で重複受信を防止する。Append-only。';
COMMENT ON COLUMN work_assignments.received_at          IS 'サーバーがリクエストを受信した時刻（ALCOA+ Contemporaneous）。Append-only。IDX-033（BRIN）のインデックス対象列。';
COMMENT ON COLUMN work_assignments.dispatched_at        IS 'SSE 配信が sent 状態に遷移した時刻。NULL = 未配信。';
COMMENT ON COLUMN work_assignments.accepted_at          IS '作業員が AssignmentDetailDialog で「開始」をタップした時刻。NULL = 未受諾。';
COMMENT ON COLUMN work_assignments.completed_at         IS '作業完了イベントが記録された時刻。NULL = 未完了。';
COMMENT ON COLUMN work_assignments.factory_id           IS 'ver1.0.0 では定数 UUID（シングルファクトリー運用）。将来のマルチファクトリー拡張時に使用する。factories テーブルは ver1.0.0 では作成しない（04_概要設計/99 §2-5 準拠）。';

-- IDX-032: 端末向け割当高速取得
-- 対象端末の未処理割当（pending / pending_resolution / dispatched）を高速に取得するための部分インデックス
CREATE INDEX idx_wa_terminal_status ON work_assignments (target_terminal_id, status)
    WHERE status IN ('pending', 'pending_resolution', 'dispatched');

-- IDX-033: 時系列検索
-- received_at の範囲検索（管理画面・バッチ処理）向け BRIN インデックス。書き込みコスト最小化。
CREATE INDEX idx_wa_received ON work_assignments USING BRIN (received_at);

-- IDX-034: 重複受信防止（UNIQUE 制約と対応）
-- uq_wa_idempotency UNIQUE 制約を明示的なインデックスとして定義する
CREATE UNIQUE INDEX idx_wa_idempotency ON work_assignments (external_system, idempotency_key);
```

#### 1-1-2. status 遷移定義

work_assignments.status は以下の遷移ルールに従う。逸脱遷移はアプリ層で ERR-BIZ-028 として拒否する。

```
pending
  |
  |  （SSE 配信成功: sse_dispatch_log.delivery_status が sent に遷移）
  v
dispatched
  |
  |  （作業員が AssignmentDetailDialog で「開始」をタップ: UC-042 手順 4）
  v
accepted
  |
  |  （case_id 生成・work_started イベント記録: UC-042 手順 5）
  v
in_progress
  |
  |  （work_completed イベント記録）
  v
completed

pending ──────────────────→ pending_resolution   （external_key 解決不能時: ERR-BIZ-027）
pending / dispatched ──────→ cancelled            （親機またはシステムがキャンセル操作）
pending / dispatched ──────→ expired              （BAT-015 が due_at 超過を検知）
accepted / in_progress ────→ rejected             （作業員が「拒否」操作: UC-042 代替シナリオ A2）
```

- `pending_resolution` 状態の割当は IT 担当が external_key_binding を登録後、システムが再解決を試みて `pending` に戻す
- `cancelled` / `expired` への遷移後は terminal_id への SSE 配信が不要となり、sse_dispatch_log の該当行は `delivery_status='expired'` に更新される

### 1-2. DDL-053: sse_dispatch_log（TBL-053）— Append-only（配信状態列のみ UPDATE 可）

```sql
-- DDL-053: TBL-053 sse_dispatch_log
-- SSE 配信ログ。work_assignments の各端末への配信試行を記録する。
-- Append-only ベース。delivery_status / ack_at / retry_count のみ UPDATE 可。
CREATE TABLE sse_dispatch_log (
    dispatch_id         UUID        PRIMARY KEY DEFAULT gen_random_uuid(),
    assignment_id       UUID        NOT NULL REFERENCES work_assignments(assignment_id),
    terminal_id         UUID        NOT NULL REFERENCES devices(device_id),
    dispatched_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    delivery_status     TEXT        NOT NULL DEFAULT 'queued'
                                    CHECK (delivery_status IN (
                                        'queued',
                                        'sent',
                                        'ack',
                                        'failed',
                                        'expired'
                                    )),
    ack_at              TIMESTAMPTZ,
    retry_count         INTEGER     NOT NULL DEFAULT 0
);

COMMENT ON TABLE  sse_dispatch_log IS 'TBL-053 — SSE 配信ログ。assignment_id × terminal_id の配信試行を記録する。Append-only ベース（delivery_status / ack_at / retry_count のみ UPDATE 可）。配信リトライ上限は CFG-030 で設定する。';
COMMENT ON COLUMN sse_dispatch_log.dispatch_id      IS 'UUID v4。gen_random_uuid() で自動生成。';
COMMENT ON COLUMN sse_dispatch_log.assignment_id    IS '配信対象の work_assignments.assignment_id への FK。';
COMMENT ON COLUMN sse_dispatch_log.terminal_id      IS '配信先端末の device_id への FK。work_assignments.target_terminal_id と原則一致する（リトライ先変更時は除く）。';
COMMENT ON COLUMN sse_dispatch_log.dispatched_at    IS '配信キューへの登録時刻（queued 状態への INSERT 時刻）。Append-only。';
COMMENT ON COLUMN sse_dispatch_log.delivery_status  IS '配信状態の 5 値列挙。遷移ルールは §1-2-2 参照。';
COMMENT ON COLUMN sse_dispatch_log.ack_at           IS '端末が POST /ack を送信した時刻。delivery_status=ack 時に設定される。NULL = 未 ACK。';
COMMENT ON COLUMN sse_dispatch_log.retry_count      IS '再送試行回数。0 = 初回配信。CFG-030 の上限（デフォルト 5）に達すると delivery_status=failed に遷移する。';

-- IDX-035: 配信状態検索
-- assignment_id + terminal_id の複合インデックス。配信状態確認・リトライ判定で使用する。
CREATE INDEX idx_sdl_assignment_terminal ON sse_dispatch_log (assignment_id, terminal_id);
```

#### 1-2-2. delivery_status 遷移定義

| 値 | 意味 |
|---|---|
| `queued` | SSE 配信待ち。sse_dispatch_log INSERT 直後の初期状態 |
| `sent` | SSE ストリームに `assignment.created` イベントを送信済み（端末 ACK 未確認）|
| `ack` | 端末から `POST /api/v1/work-assignments/{id}/ack` を受信済み |
| `failed` | retry_count が CFG-030 の上限を超過。IT 担当への通知を発行する |
| `expired` | 割当が cancelled または expired 状態に遷移したため配信不要になった |

---

**本節で確定した方針**

- **TBL-052 work_assignments の CREATE TABLE 全文（DDL-052）を確定した。`source_payload`・`external_order_id`・`external_system`・`received_at`・`idempotency_key` を Append-only 列として明示し、`status`・`case_id`・`dispatched_at`・`accepted_at`・`completed_at` のみ UPDATE 可とした（ADR-010 準拠）。**
- **TBL-053 sse_dispatch_log の CREATE TABLE 全文（DDL-053）を確定した。Append-only ベースとし `delivery_status`・`ack_at`・`retry_count` のみ UPDATE を許可するロール制御 DDL を定義した。**
- **status 遷移（9 種）を CHECK 制約と遷移ルール表で二重に定義した。`pending_resolution` への遷移は ERR-BIZ-027 発生時に限定し、IT 担当の手動解決フローとの整合を保つ。**
- **IDX-032（部分インデックス）・IDX-033（BRIN）・IDX-034（UNIQUE）・IDX-035（複合）の 4 インデックスを確定した。IDX-032 は `WHERE status IN (...)` 部分インデックスとして完了済み割当を除外し書き込みコストを抑制する。**
- **factory_id は ver1.0.0 では定数 UUID を使用する予約フィールドとし、factories テーブルを作成しない方針を COMMENT ON COLUMN に明示した（04_概要設計/99 §2-5 準拠）。**
