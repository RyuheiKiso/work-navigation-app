# 99 前提制約と本書が約束しないこと

本章の責務は、`01_データベース詳細設計/` サブのスコープ外事項と前提制約を明示することである。これにより実装フェーズでの誤解に基づくスコープ拡大・本書への過大な期待を防止する。

---

## 1. 前提制約（本書が成立する条件）

### 1-1. 上流の確定事項（変更不可の入力）

本書は以下が確定済みであることを前提とする。これらに変更が生じた場合は本書も改版が必要となる。

| 確定事項 | 参照先 | 内容 |
|---|---|---|
| 物理テーブル一覧（TBL-001〜035）| `04_概要設計/04_データ設計/02_物理テーブル一覧（TBLカタログ）.md` | テーブル名・対応 EN・種別は変更しない |
| 7 アーキテクチャ原則 | `04_概要設計/02_ソフトウェア方式設計/01_ソフトウェア全体アーキテクチャと7原則継承.md` | P2（Append-only）・P5（改ざん不可性）は不変 |
| 業務ルール（BR-BUS-NNN）| `03_要件定義/機能要件/10_業務ルール.md` | BR-BUS-003・BR-BUS-015 等の DDL に影響するルール |
| 非機能要件（NFR-NNN）| `03_要件定義/非機能要件/` | NFR-PRF-001（P95 200ms）・NFR-OPS-035（7 年保存）|

### 1-2. 実行環境の前提

| 前提 | 詳細 |
|---|---|
| PostgreSQL 16 以上 | gen_random_uuid() 組み込み・RANGE パーティション・UNIQUE NULLS NOT DISTINCT をサポートする |
| pgcrypto 拡張が利用可能 | `CREATE EXTENSION IF NOT EXISTS pgcrypto` が実行可能であること |
| pg_stat_statements 拡張が利用可能 | 性能監視のために有効化する |
| warm_ts テーブルスペース用 HDD が存在する | ウォーム階層移行時に `/data/warm/postgresql` がマウント済みであること |
| NAS への書き込みアクセス | コールドアーカイブの gzip ダンプ出力先として `/backups/postgresql/` が存在すること |

## 2. 本書が約束しないこと（スコープ外）

### 2-1. 実装ファイルの作成

本書は設計仕様書である。以下は本書のスコープ外であり、06_実装フェーズで実現する。

| スコープ外事項 | 実現フェーズ |
|---|---|
| `crates/wnav-db/migrations/*.sql` の実際のファイル作成 | 06_実装 |
| Rust の `sqlx::query!` マクロを使用した実装コード | 06_実装 |
| TypeScript の API クライアントコード | 06_実装 |
| React Native の SQLite スキーマ（ハンディ APP ローカル DB）| 06_実装 |
| バックエンドの Docker コンテナ設定 | 06_実装 |

### 2-2. React Native / SQLite のローカルスキーマ

本書は PostgreSQL サーバーサイドの DDL のみを対象とする。ハンディ APP（React Native）の SQLite + TypeORM によるローカルキャッシュスキーマは `04_ハンディAPP詳細設計/` で別途定義する。

### 2-3. 外部システム（ERP/MES）のスキーマ

TBL-027 external_key_bindings の `external_key` JSONB 列のキー名は外部システムに依存する。本書は外部システムのスキーマを定義しない。外部スキーマとのマッピング設定は IF-001 インターフェース設計で管理する。

### 2-4. 行レベルセキュリティ（RLS）の設計

PostgreSQL の Row-Level Security（RLS）ポリシーは本書のスコープ外とする。RLS は `04_概要設計/07_セキュリティ方式設計/` の拡張設計として将来版で検討する。ver1.0.0 では PostgreSQL ロールレベルの権限制御（本書 §3-5・02 章 NOTE）を採用する。

### 2-5. パーティション自動化（pg_partman）

本書は pg_partman を不採用として BAT-004 による手動 SQL 実行を採用した（`06_パーティション・アーカイブ詳細設計.md` §1-1）。pg_partman の設定方法はスコープ外であり、将来の ver2.0.0 で再検討する。

### 2-6. データウェアハウスへのデータ転送

本システムのデータを BI ツール / DWH（BigQuery・Snowflake 等）に転送する ETL 設計は本書のスコープ外とする。`04_概要設計/05_外部インターフェース設計/` で必要に応じて追加する。

### 2-7. フルテキスト検索インデックス

本書では GIN インデックスを `external_key_bindings.external_key`（JSONB）のみに使用する。PostgreSQL の `tsvector` / GIN による全文検索インデックスはスコープ外とする（対象となる検索要件が ver1.0.0 に存在しない）。

### 2-8. 暗号化カラム（pgcrypto による列暗号化）

本書は列レベルの暗号化（`pgp_sym_encrypt` 等）を定義しない。PII 保護は `anonymized_at` による匿名化方式（BAT-004）を採用した（users テーブル設計参照）。将来の法規制強化に応じて設計改版する。

## 3. 既知の制約と運用上の注意

### 3-1. evidence_files の FK 制約（パーティションテーブルへの参照）

TBL-009 evidence_files は `work_events (event_id, timestamp_server)` に外部キーを設定している（DDL-009 参照）。PostgreSQL 16 ではパーティションテーブルへの外部キーは制限があるため、マイグレーション時に FK を `DEFERRABLE INITIALLY DEFERRED` で設定している。アプリ層でイベント INSERT 後に evidence_files を INSERT する順序を厳守する。

### 3-2. work_events のパーティション境界の欠損

BAT-004 が失敗した場合、翌月の INSERT が `no partition of relation "work_events" found for row` エラーで失敗する。MON-001 による BAT-004 の実行監視と、失敗時のアラート通知を実装する。手動 SQL での対処手順は `06_パーティション・アーカイブ詳細設計.md` §2-3 を参照する。

### 3-3. mv_daily_work_summary のリフレッシュ失敗

mv_daily_work_summary（VW-006）の REFRESH CONCURRENTLY 中に unique index の重複が発生した場合、リフレッシュが失敗する。原因は work_executions への重複 INSERT（バグ）であり、データ修正後に手動でリフレッシュする。

---

**本節で確定した方針**
- **本書のスコープを PostgreSQL 16 上の 35 テーブル DDL・16 インデックス・8 ビュー・JSONB スキーマ・パーティション・マイグレーション・シードデータに限定し、React Native SQLite スキーマ・RLS・DWH 転送はスコープ外として明示した。**
- **前提制約（PostgreSQL 16・warm_ts HDD・NAS アクセス）を明示し、インフラ環境が整わない場合にデータベース設計が機能しないことを確定した。**
- **evidence_files の DEFERRABLE INITIALLY DEFERRED FK・パーティション欠損リスク・mv_daily_work_summary リフレッシュ失敗の 3 つの既知制約を記録し、実装フェーズで対処が必要な事項を事前に明示した。**
