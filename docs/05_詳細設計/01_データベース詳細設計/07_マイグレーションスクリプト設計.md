# 07 マイグレーションスクリプト設計

本章の責務は、sqlx migrate を使用したデータベースマイグレーションの命名規約・ファイル構造・互換性ルール・ロールバックポリシー・マイグレーションテスト手順を確定することである。これにより全 35 テーブルの DDL がバージョン管理され、開発環境・ステージング・本番環境の間でスキーマが完全に再現可能となる。

---

## 1. マイグレーションツールと設定

### 1-1. ツール選択

| 検討ツール | 採否 | 理由 |
|---|---|---|
| sqlx migrate（Rust）| **採用** | バックエンドのクレート `sqlx` と統合。型安全クエリ（`sqlx::query_as!`）のコンパイル時検証がマイグレーション適用後に自動で実行される。 |
| Flyway | 不採用 | JVM 依存。Rust スタックと独立した外部ツールになる。 |
| Liquibase | 不採用 | XML/YAML 形式で SQL と乖離する。 |

### 1-2. sqlx migrate の設定

```toml
# Cargo.toml（バックエンドクレートのルート）
[dependencies]
sqlx = { version = "0.8", features = ["postgres", "runtime-tokio", "migrate", "uuid", "chrono", "json"] }
```

```bash
# マイグレーションディレクトリ
# リポジトリルート配下
crates/
  wnav-db/
    migrations/        # ← sqlx migrate の管理ディレクトリ
      V20260517120000__init_roles.sql
      V20260517120001__init_users.sql
      ...
```

```bash
# 環境変数（.env または Docker Compose で設定）
DATABASE_URL=postgresql://app_admin:${PGPASSWORD}@localhost:5432/wnav_db

# マイグレーション実行コマンド
sqlx migrate run --source crates/wnav-db/migrations/
```

## 2. ファイル命名規約

### 2-1. 命名形式

```
V{YYYYMMDDHHMMSS}__{description}.sql

V   : Version プレフィックス（固定）
YYYYMMDDHHMMSS : タイムスタンプ（秒精度 UTC）
__  : ダブルアンダースコア区切り
description : スネークケース、小文字英数字とアンダースコアのみ

例:
V20260517120000__create_extensions.sql
V20260517120001__create_roles_table.sql
V20260517120002__create_users_table.sql
...
V20260517120035__create_idempotency_keys_table.sql
V20260517120036__create_role_permissions.sql
V20260517120037__seed_roles.sql
```

### 2-2. description の命名規則

| 操作 | プレフィックス | 例 |
|---|---|---|
| 拡張機能の有効化 | `create_extensions` | `create_extensions` |
| テーブル作成 | `create_{table_name}_table` | `create_users_table` |
| 列追加 | `add_{column_name}_to_{table_name}` | `add_anonymized_at_to_users` |
| インデックス追加 | `add_index_{idx_id}_{table_name}` | `add_index_001_work_events` |
| ビュー作成/置換 | `create_{view_name}_view` | `create_active_work_executions_view` |
| シードデータ投入 | `seed_{table_name}` | `seed_roles` |
| ロール権限設定 | `grant_{role}_permissions` | `grant_app_event_writer_permissions` |

### 3-2. V20260517120000__create_extensions.sql（テンプレート）

```sql
-- PostgreSQL 拡張機能を有効化する
-- gen_random_uuid() は pgcrypto に含まれるが PG13+ では組み込み済み
-- pg_stat_statements は性能監視のため有効化する

CREATE EXTENSION IF NOT EXISTS "uuid-ossp";      -- UUID 生成（バックアップ用）
CREATE EXTENSION IF NOT EXISTS "pgcrypto";       -- gen_random_bytes 等
CREATE EXTENSION IF NOT EXISTS "pg_stat_statements"; -- クエリ性能統計
```

## 4. スキーマ互換性ルール

### 4-1. 前方互換性（Additive Changes Only）

本プロジェクトはマイグレーション後にアプリケーション全ロールアウトが完了するまでの間、旧バージョンのバックエンドが動作可能である必要がある（ゼロダウンタイムデプロイ原則）。

| 操作 | 可否 | 理由 |
|---|---|---|
| 新しい NULL 許容列の追加 | **許可** | 既存コードが新列を参照しなくても動作する |
| 新しいテーブルの追加 | **許可** | 既存コードに影響しない |
| CHECK 制約の追加（新 enum 値の追加）| **許可（慎重に）**| 既存コードが新 enum 値を送信しなければ問題ない |
| インデックスの追加（CONCURRENTLY）| **許可** | ロックなし |
| NOT NULL 列の追加 | **禁止** | 既存コードが列を設定しない場合に INSERT 失敗 |
| 列の削除 | **禁止** | 既存コードが列を参照してクラッシュする |
| 列の型変更（縮小）| **禁止** | 既存データの切り捨てリスク |
| 列の型変更（拡大: VARCHAR(64)→VARCHAR(128)）| **許可** | 既存データを壊さない |
| 外部キー制約の追加 | **注意**| 既存データが参照整合性を満たしている場合のみ許可 |
| テーブル名変更 | **禁止** | 既存コードが旧名で参照してクラッシュする |

### 4-2. ロールバックポリシー

sqlx migrate は公式にロールバック（down マイグレーション）をサポートしない。本プロジェクトは以下のポリシーを採用する。

- **ロールバック禁止**: マイグレーション適用後は原則として前のバージョンに戻さない
- **前進修正（Forward Fix）**: 誤ったマイグレーションは新しいマイグレーションで修正する
- **本番デプロイ前に CI でマイグレーション検証を必須とする**（後述 §5）
- **緊急ロールバックが必要な場合**: PostgreSQL のポイントインタイムリカバリ（PITR）を使用する

```
緊急ロールバック手順（本番で使用しないことが原則）:
1. アプリケーションをメンテナンスモードに切り替える
2. PostgreSQL の WAL ベースのリカバリで問題発生前のタイムスタンプに戻す
3. アプリケーションを前バージョンで再起動する
```

## 5. マイグレーションテスト手順

### 5-1. CI パイプラインでの検証

```yaml
# .github/workflows/db-migration-test.yml（概要）
name: DB Migration Test

on: [push, pull_request]

jobs:
  migration-test:
    runs-on: ubuntu-latest
    services:
      postgres:
        image: postgres:16
        env:
          POSTGRES_USER: app_admin
          POSTGRES_PASSWORD: testpassword
          POSTGRES_DB: wnav_test

    steps:
      - uses: actions/checkout@v4

      - name: Install sqlx-cli
        run: cargo install sqlx-cli --no-default-features --features postgres

      - name: Run migrations
        env:
          DATABASE_URL: postgresql://app_admin:testpassword@localhost:5432/wnav_test
        run: sqlx migrate run --source crates/wnav-db/migrations/

      - name: Verify schema (sqlx prepare check)
        env:
          DATABASE_URL: postgresql://app_admin:testpassword@localhost:5432/wnav_test
        run: cargo sqlx prepare --check --workspace

      - name: Run idempotency test (apply twice)
        env:
          DATABASE_URL: postgresql://app_admin:testpassword@localhost:5432/wnav_test
        run: |
          # マイグレーション再実行で冪等性を確認（IF NOT EXISTS を使用しているため 2 回目はスキップ）
          sqlx migrate run --source crates/wnav-db/migrations/
          echo "Idempotency check passed"
```

### 5-2. マイグレーション検証チェックリスト

新しいマイグレーションファイルを追加する際は以下を全て確認する。

| チェック項目 | 確認方法 |
|---|---|
| タイムスタンプの一意性 | `ls migrations/ | sort` で重複がないことを確認 |
| 依存テーブルが先に作成されている | マイグレーション順序（§3-1）と外部キーの参照先を照合 |
| CHECK 制約の値が既存コードと一致する | Rust の `enum` 定義と CHECK 値を照合 |
| インデックスが CONCURRENTLY で作成される | CREATE INDEX 文を確認 |
| 互換性ルール（§4-1）を違反していない | PR レビューで確認 |
| CI が全てグリーンである | GitHub Actions の status を確認 |

### 5-3. マイグレーションのリバート確認（ステージングのみ）

ステージング環境では PITR を使用して誤ったマイグレーションのリバートを演習する（本番環境の訓練）。

```bash
# ステージングでの PITR 演習手順（概要）
# 1. 演習前の base backup を取得
pg_basebackup -h $PGHOST -U $PGUSER -D /backup/base -Ft -z -P

# 2. マイグレーション適用
sqlx migrate run --source migrations/

# 3. PITR ターゲット時刻をメモする
TARGET_TIME=$(date -u +"%Y-%m-%d %H:%M:%S")

# 4. 誤ったマイグレーションを適用する（演習用）
# ...

# 5. PITR でリカバリする（recovery.conf または postgresql.conf の recovery_target_time）
```

---

**本節で確定した方針**
- **sqlx migrate を採用し `V{YYYYMMDDHHMMSS}__{description}.sql` 命名規約でマイグレーションを一元管理する。ロールバックは禁止し、誤りは前進修正（新マイグレーション追加）で解決する。**
- **スキーマ変更は NOT NULL 列の追加・列の削除・型の縮小を禁止し、NULL 許容列追加・新テーブル追加・型の拡大のみを許可する前方互換ルールを確定した。**
- **全マイグレーションは CI で PostgreSQL 16 コンテナに対して適用検証し、`cargo sqlx prepare --check` によりコンパイル時型安全クエリとのスキーマ整合性を保証する。**

## 6. PG マイグレーションと SQLite マイグレーションの関係

ミラー対象テーブル（TBL-001/003/005/007/008/009/011/012/013/024/026/051）の DDL 変更時は、
本書の sqlx migration に加えて `04_ハンディAPP詳細設計/07a_TypeORMマイグレーション設計.md` の
TypeORM migration を同一 PR で生成すること。

CI の `pg-sqlite-drift-check` ジョブが PG スキーマ差分と SQLite エンティティ差分の一致を自動検証する。
