# 08 エラーコード完全定義

本章は全 API で使用するエラーコード（ERR-NNN）の完全カタログを確定する。`04_概要設計/付録/99_設計識別子採番台帳.md` § 10 の体系を継承し、HTTP ステータス・title・detail i18n キー・log_level・リトライ可否を全件記載する。

---

## 1. エラーレスポンス形式（再掲）

全エラーレスポンスは RFC 9457 Problem Details 形式で返す。Content-Type は `application/problem+json`。

```json
{
  "type": "https://errors.wnav.example.com/ERR-BIZ-001",
  "title": "lock_step_violation",
  "status": 409,
  "detail": "直前のステップが完了していません。ステップ番号 3 を完了してください。",
  "instance": "/api/v1/work-executions/019682ab-7c1f-7000-b1c2-3d4e5f6a7b8c/events",
  "error_id": "ERR-BIZ-001"
}
```

バリデーションエラー（ERR-VAL-*）の場合は `violations` 配列を追加する（01 章 §3-2 参照）。

## 2. エラーコード完全カタログ

### 2-1. ERR-AUTH（認証・認可）

| ERR-CODE | HTTP Status | title | detail (i18n key) | log_level | retry |
|---|---|---|---|---|---|
| ERR-AUTH-001 | 401 | Unauthorized | auth.unauthorized | WARN | no |
| ERR-AUTH-002 | 401 | PIN verification failed | auth.pin_failed | WARN | no（正しい PIN で再試行）|
| ERR-AUTH-003 | 423 | Account locked | auth.account_locked | ERROR | no（ロック解除後に再試行）|
| ERR-AUTH-004 | 403 | Access denied | auth.forbidden | WARN | no |

#### ERR-AUTH-001 詳細

発生条件:
- `Authorization` ヘッダが欠落している。
- JWT の署名が不正である。
- JWT の有効期限（`exp`）が切れている（`04_概要設計` での識別子: ERR-AUTH-003 jwt_expired に相当するが、クライアントへの開示上は同じ 401 を返す）。
- JWT の `jti` が TBL-032 のログアウト記録に存在する（ブラックリスト）。
- `login_id` が存在しない。

対応: クライアントは refresh トークンで更新を試みる。失敗した場合は再ログイン。

#### ERR-AUTH-002 詳細

発生条件:
- PIN の bcrypt 検証が不一致。
- 連続 3 回失敗でアカウントロック（ERR-AUTH-003 に遷移）。

対応: 正しい PIN で再試行。

#### ERR-AUTH-003 詳細

発生条件:
- パスワード連続 5 回失敗後のアカウントロック（30 分）。
- PIN 連続 3 回失敗後のアカウントロック。

対応: ロック解除時刻（`locked_until`）を `detail` に含める。`system_admin` による手動解除も可。

#### ERR-AUTH-004 詳細

発生条件:
- RBAC 要件を満たさないロールでのアクセス。
- 他工場のリソースへのアクセス試行。

### 2-2. ERR-VAL（バリデーション）

| ERR-CODE | HTTP Status | title | detail (i18n key) | log_level | retry |
|---|---|---|---|---|---|
| ERR-VAL-001 | 422 | Required field missing | validation.required | DEBUG | no |
| ERR-VAL-002 | 422 | Value out of range | validation.range | DEBUG | no |
| ERR-VAL-003 | 422 | Invalid format | validation.format | DEBUG | no |
| ERR-VAL-004 | 422 | Max length exceeded | validation.max_length | DEBUG | no |

#### ERR-VAL-001 詳細

`violations` 配列に不足フィールド名と説明を含める。

```json
{
  "type": "https://errors.wnav.example.com/ERR-VAL-001",
  "title": "Required field missing",
  "status": 422,
  "detail": "必須フィールドが不足しています。",
  "instance": "/api/v1/work-executions",
  "error_id": "ERR-VAL-001",
  "violations": [
    { "field": "work_order_id", "message": "work_order_id は必須です。" },
    { "field": "operator_id", "message": "operator_id は必須です。" }
  ]
}
```

#### ERR-VAL-002 詳細

`violations` に `field` / `min` / `max` / `actual` を含める。

#### ERR-VAL-003 詳細

UUID v7 形式不正・MIME type 不正・SHA-256 ハッシュ不一致・ISO 8601 日時形式不正等。

#### ERR-VAL-004 詳細

`violations` に `field` / `max_length` / `actual_length` を含める。

### 2-3. ERR-BIZ（業務ルール違反）

| ERR-CODE | HTTP Status | title | detail (i18n key) | log_level | retry |
|---|---|---|---|---|---|
| ERR-BIZ-001 | 409 | Step sequence violation | biz.step_sequence | WARN | no |
| ERR-BIZ-002 | 409 | Evidence gate | biz.evidence_required | WARN | no |
| ERR-BIZ-003 | 409 | SOP version not published | biz.sop_not_published | WARN | no |
| ERR-BIZ-004 | 409 | Instrument calibration expired | biz.calibration_expired | WARN | no（CFG-012 hard モード）|
| ERR-BIZ-005 | 409 | SOP frozen after publish | biz.sop_frozen | WARN | no |
| ERR-BIZ-006 | 403 | Skill level insufficient | biz.skill_insufficient | WARN | no |
| ERR-BIZ-007 | 409 | Version already published | biz.version_published | WARN | no |
| ERR-BIZ-008 | 409 | CAPA already closed | biz.capa_closed | WARN | no |

#### ERR-BIZ-001 詳細

発生条件: API-step-events-001 で、`step_id` が `work_execution.current_step_id` と一致しない。
対応: `detail` に現在の `current_step_id` と `current_step_number` を含める。
関連: BR-BUS-001（ステップ順序ロック）。

#### ERR-BIZ-002 詳細

発生条件: step_completed を送信する前に、ステップの `requires_evidence: true` フラグに対するエビデンス添付が完了していない。
対応: evidence_attached イベントを先に送信する。
関連: BR-BUS-003。

#### ERR-BIZ-003 詳細

発生条件:
- work-execution 作成時に指定 SOP に Published バージョンが存在しない。
- master-versions/approve 時に対象バージョンが in_review でない。

#### ERR-BIZ-004 詳細

発生条件: measurement_recorded で指定した `instrument_id` の `calibration_due` が `NOW()` 以前。
CFG-012 が `soft` の場合は警告のみで続行する（HTTP 200 + `warnings` フィールド）。CFG-012 が `hard` の場合は HTTP 409 で拒否する。

#### ERR-BIZ-005 詳細

発生条件: status が `draft` 以外のマスタバージョンへの PATCH / 内容変更試行。
関連: BR-BUS-008（SOP Freeze after publish）。

#### ERR-BIZ-006 詳細

発生条件: work-execution 作成時または step-events 記録時に、operator のスキルレベルが SOP / ステップの `skill_level_required` を下回る。
CFG-011 が `soft` の場合は警告のみ。`hard` の場合は HTTP 403 で拒否。

#### ERR-BIZ-007 詳細

発生条件: master-versions/draft で指定した `version_label` が既存のバージョンと重複する。

#### ERR-BIZ-008 詳細

発生条件: PATCH /capas/{id} で target CAPA の status が既に `closed`。

### 2-4. ERR-DB（データベース / 整合性）

| ERR-CODE | HTTP Status | title | detail (i18n key) | log_level | retry |
|---|---|---|---|---|---|
| ERR-DB-001 | 500 | Database error | system.db_error | ERROR | yes（最大 3 回）|
| ERR-DB-002 | 409 | Foreign key violation | system.fk_violation | WARN | no |
| ERR-DB-003 | 500 | Hash chain broken | system.hash_chain | CRITICAL | no |
| ERR-DB-004 | 500 | Optimistic lock failure | system.optimistic_lock | WARN | yes（最大 1 回）|

#### ERR-DB-001 詳細

発生条件: PostgreSQL 接続エラー・クエリタイムアウト・トランザクション異常。
対応: 指数バックオフで最大 3 回リトライ。全失敗後は Outbox 経由で DLQ に移動。`detail` には内部エラー詳細を含めない（セキュリティ上の観点）。

#### ERR-DB-002 詳細

発生条件: 参照整合性違反（外部キー制約）。クライアントのリクエストデータが TBL 内の参照先 ID に存在しない場合。

#### ERR-DB-003 詳細

発生条件: TBL-031 のハッシュチェーン検証でハッシュ値の連鎖が破断されていることを検出。
対応: 即時に LOG-007 を出力し MET-006 をインクリメントする。CRITICAL ログのため監視アラートが発火する。
重要: このエラーは改ざん / データ破損の可能性を示す。system_admin に即時通知する。

#### ERR-DB-004 詳細

発生条件: 楽観的ロック競合（`version` カラムによる）。
対応: 1 回自動リトライ。失敗した場合は 409 でクライアントに返却し、クライアントは最新データを再取得して再実行する。

注記: 採番台帳では ERR-DB-001 が `idempotency_replay_conflict`（409）・ERR-DB-002 が `optimistic_lock_conflict`（409）として定義されているが、指示書の仕様（ERR-DB-001: Database error 500 / ERR-DB-004: Optimistic lock failure 500）を本書では採用し、整合を取る。Idempotency 競合は ERR-DB-001 の detail で区別する。

### 2-5. ERR-EXT（外部システム連携）

| ERR-CODE | HTTP Status | title | detail (i18n key) | log_level | retry |
|---|---|---|---|---|---|
| ERR-EXT-001 | 503 | Parent system unavailable | system.external | WARN | yes（Outbox 経由で自動再送）|
| ERR-EXT-002 | 503 | LDAP unavailable | system.ldap | WARN | yes（ローカル認証フォールバック）|

#### ERR-EXT-001 詳細

発生条件: 親機（IF-002）への Webhook / API 送信が失敗。
対応: TBL-003（Outbox）に記録し、BAT-002（outbox_dispatcher）が CFG-002〜004 に従ってリトライ。最大 CFG-002（5 回）失敗後は DLQ に移動。

#### ERR-EXT-002 詳細

発生条件: LDAP/AD（IF-003）への接続失敗。
対応: TBL-016 の bcrypt ハッシュでローカル認証にフォールバック（§ 1-3）。フォールバック成功の場合 HTTP 200 を返す（ERR-EXT-002 はログのみ）。フォールバックも失敗した場合に ERR-EXT-002 を返却する。

---

### 2-6. ERR-SYS（システム内部）

| ERR-CODE | HTTP Status | title | detail (i18n key) | log_level | retry |
|---|---|---|---|---|---|
| ERR-SYS-001 | 500 | Internal server error | system.internal | ERROR | yes（最大 3 回）|
| ERR-SYS-003 | 500 | Report generation failed | system.report | ERROR | yes（BAT リトライ）|
| ERR-SYS-004 | 500 | Template integrity error | system.template | CRITICAL | no |
| ERR-SYS-005 | 503 | DLQ overflow | system.dlq_overflow | CRITICAL | no |

注記: ERR-SYS-002（429 rate_limited）はレート制限ヘッダ仕様（01 章 §6）で定義済み。本表では 429 系の独立エントリは省略し ERR-SYS-002 が 429 を担当することを注記する。

#### ERR-SYS-001 詳細

発生条件: 上記いずれにも分類されない予期しない内部エラー。Rust の `?` オペレータで捕捉されなかった panic / unhandled error。
対応: `detail` には `"内部エラーが発生しました。"` のみ。スタックトレースはログに記録し外部に漏洩させない。

#### ERR-SYS-003 詳細

発生条件: API-reports-001〜002 の帳票生成処理が失敗。
対応: BAT-007 のリトライ対象。最大 3 回失敗後は LOG-010 に ERROR 記録。

#### ERR-SYS-004 詳細

発生条件: 帳票テンプレートファイルの SHA-256 ハッシュが登録値と一致しない（テンプレート改ざん検出）。
対応: CRITICAL ログを出力し、system_admin に即時通知。帳票生成を停止する。

#### ERR-SYS-005 詳細

発生条件: DLQ のアイテム数が上限（設定値）を超過。
対応: MET-005 がアラートを発火。system_admin が API-ops-002 で手動再送または調査する。

## 3. エラーコード採番状況（本書追加分）

本書は `04_概要設計/付録/99_設計識別子採番台帳.md` § 10 の次採番値を参照して以下のコードを追加定義した。

| ERR-CODE | 採番台帳との関係 | 説明 |
|---|---|---|
| ERR-BIZ-007 | 台帳 next: ERR-BIZ-007（新規）| version_already_published |
| ERR-BIZ-008 | 台帳 next: ERR-BIZ-008（新規）| capa_already_closed |
| ERR-DB-004 | 台帳 next: ERR-DB-004（新規）| optimistic_lock_failure |
| ERR-SYS-003 | 台帳 next: ERR-SYS-003（新規）| report_generation_failed |
| ERR-SYS-004 | 台帳 next: ERR-SYS-004（新規）| template_integrity_error |
| ERR-SYS-005 | 台帳 next: ERR-SYS-005（新規）| dlq_overflow |

採番台帳の次採番値を以下に更新する（採番台帳 §10 を参照して更新すること）:

| layer | 更新後の次採番値 |
|---|---|
| BIZ | ERR-BIZ-009 |
| DB | ERR-DB-005 |
| SYS | ERR-SYS-006 |

---

**本節で確定した方針**
- **全 26 エラーコードの HTTP Status / title / detail i18n キー / log_level / retry 可否を完全定義し、RFC 9457 形式での返却を全エンドポイントに義務付けることを確定した。**
- **ERR-DB-003（ハッシュチェーン破断）と ERR-SYS-004（テンプレート改ざん）は CRITICAL ログとし即時アラートを発火させることを確定した。これら 2 コードは retry: no であり人間による調査が必要。**
- **ERR-EXT-002（LDAP 不可）はローカル認証フォールバック成功時はエラーを返さずログのみとし、フォールバックも失敗した場合のみ HTTP 503 を返すことを確定した。**
