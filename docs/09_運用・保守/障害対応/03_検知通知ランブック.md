# 03 検知通知ランブック

最終更新: 2026-05-18 | 管理者: system_admin | 根拠要件: OPS-056 / OPS-058

---

## 1 本章の範囲

F1（検知）〜F2（通知）フェーズの 3 ランブックを定義する。

| RUN ID | 名称 | SLO（P1/P2） |
|---|---|---|
| RUN-001 | Grafana アラート受信手順 | 5 分 / 15 分（F1） |
| RUN-002 | 現場からの障害報告受付手順 | 5 分 / 15 分（F1） |
| RUN-003 | 通知文書の書き方テンプレート | 5 分 / 15 分（F2） |

---

## 2 RUN-001: Grafana アラート受信手順

**トリガー**: Grafana から CRITICAL / HIGH / WARN アラートのメール通知またはダッシュボードのアラート表示  
**担当**: system_admin  
**所要時間目安**: 3 分以内（F1 SLO 内）

### 手順

**Step 1: Grafana ダッシュボード確認**

```bash
# Grafana 疎通確認
curl -fsS http://localhost:3000/api/health | jq .
# {"database":"ok","version":"..."} が期待値
```

**Step 2: アラート一覧確認**

```bash
# Grafana API でアラート一覧取得（API Key 設定済みの場合）
curl -fsS -H "Authorization: Bearer $GRAFANA_API_KEY" \
  "http://localhost:3000/api/alertmanager/grafana/api/v2/alerts" | jq '.[] | {labels, status}'
```

**Step 3: P 分類の初期判定**

| アラート ID | 重大度 | 初期 P 分類 |
|---|---|---|
| ALERT-001（DLQ > 100） | CRITICAL | P2（作業記録の非同期処理停止）|
| ALERT-002（disk > 85%） | HIGH | P2〜P3（復旧見込みで判断）|
| ALERT-003（P95 > 5000ms） | WARN | P3 |
| ALERT-004（backup > 25h） | HIGH | P2 |

**Step 4: INC 仮登録**

```bash
# INC 仮登録（12 章の台帳ファイルを手動で更新）
# フォーマット: INC-YYYY-NNN | 発生日時 | P分類 | 概要 | ステータス=OPEN
echo "INC-$(date +%Y)-XXX | $(date '+%Y-%m-%d %H:%M %Z') | P? | $(アラートID): 概要 | OPEN" 
```

**完了判定**: INC 番号発番・P 分類決定・F2（RUN-003）への移行

**本節で確定した方針**
- **Grafana アラートは受信から 3 分以内に P 分類を行い INC を仮登録する。**
- **ALERT-001/002/004 は初期 P2 として扱い、確認後に降格する。**
- **アラートが誤検知と判明した場合も INC 記録に「誤検知・理由」を記載してクローズする。**

## 3 RUN-002: 現場からの障害報告受付手順

**トリガー**: 現場監督または作業者からの電話・口頭・メール報告  
**担当**: system_admin  
**所要時間目安**: 5 分以内（F1 SLO 内）

### 受付チェックリスト

```
CHK-003: 現場報告受付
□ 報告受付時刻（JST）: ________________
□ 報告者のロール: 作業者 / 現場監督
□ 影響している工程名: ________________
□ 最後に正常動作した時刻: ________________
□ 表示されているエラーメッセージ（文言そのまま）: ________________
□ 端末種別: ハンディ（機種名: ___）/ PC / その他
□ 対象ユーザー数（概数）: ________________
□ 作業記録継続の可否: 可能 / 不可能（→ P1 判定）
```

### 手順

**Step 1: 事実情報の収集**

上記チェックリストに基づき情報を収集する。「〜のような気がする」「〜かもしれない」という表現は「〜という状態が報告された」と言い換えて記録する。

**Step 2: P 分類の判定**

```
作業記録が完全に不可能か？ → YES → P1
一部の記録に影響があるか？ → YES → P2
稼働中だが遅い・おかしい → P3
軽微・記録継続は可能 → P4
```

**Step 3: 報告者への即時折り返し**

```
折り返し内容テンプレート:
「確認いたしました。ただいま [P分類] として対応を開始します。
影響範囲を確認中です。次の状況報告は [HH:MM]（約X分後）に行います。
作業記録については [継続可能/紙記録に切り替えてください] 。」
```

**Step 4: INC 仮登録**

RUN-001 の Step 4 と同手順で INC 番号を発番する。

**完了判定**: INC 番号発番・P 分類決定・報告者への折り返し完了・F2（RUN-003）への移行

**本節で確定した方針**
- **現場報告は受付から 5 分以内に P 分類を行い、報告者への折り返し連絡を行う。**
- **「作業記録が完全に不可能」という事実を確認した場合は即時 P1 と判定する。**
- **受付情報は INC 記録の「検知経路: 現場報告」に明記する。**

## 4 RUN-003: 通知文書の書き方テンプレート

**トリガー**: F1（検知）完了後、F2（通知）として実施  
**担当**: system_admin  
**所要時間目安**: 3 分以内（F2 SLO 内）

### 通知 3 要素（OPS-058）

1. **何が起きているか**（事実のみ・仮説を含まない）
2. **影響範囲**（どのシステム・工程・ユーザーが影響を受けているか）
3. **現在の対応状況と次のアクション**（担当・予定時刻）

### 通知文書テンプレート

```
件名: [P1/P2/P3/P4] INC-YYYY-NNN 障害通知 - {概要}

発生日時（JST）: YYYY-MM-DD HH:MM
通知時刻（JST）: YYYY-MM-DD HH:MM
対応者: system_admin
INC 番号: INC-YYYY-NNN
優先度: P1 / P2 / P3 / P4

--- 1. 何が起きているか ---
{確認できた事実を 1〜3 文で記載。「〜と推定される」は禁止}
例: PostgreSQL サービスが 14:32 から応答していない。
    全ハンディ端末で API タイムアウトが発生している。

--- 2. 影響範囲 ---
- 影響システム: {API / DB / ハンディ / 帳票 / 全体}
- 影響工程: {工程名 / 全工程}
- 影響ユーザー数（概数）: {N 名}
- 作業記録への影響: 完全停止 / 一部影響 / なし

--- 3. 現在の対応状況と次のアクション ---
現在: {F3 切り分け実施中 / RUN-XXX 実行中}
次のアクション: {具体的なアクション}（担当: system_admin）
次の状況報告予定: {HH:MM}（約 {N} 分後）

--- 注記 ---
本通知は確認済み事実に基づく。根本原因は F3 切り分け後に更新する。
```

### 通知先と優先度別対応

| 優先度 | 通知先 | 手段 | タイミング |
|---|---|---|---|
| P1 | system_admin・現場監督 | 電話 + 書面 | F2 SLO（5 分以内） |
| P2 | system_admin・現場監督 | 書面（メール可） | F2 SLO（15 分以内） |
| P3 | system_admin | 書面 | F2 SLO（15 分以内） |
| P4 | system_admin | チケット登録のみ | 翌営業日まで |

### SitRep 更新タイミング

P1: 30 分毎に状況更新。P2: 1 時間毎に状況更新。SitRep 更新は RUN-003 テンプレートに「更新 #N」を追記する。

**本節で確定した方針**
- **通知文書には仮説・推測・個人名を含めない（OPS-058 + OPS-065 Just Culture）。**
- **P1 は通知後 30 分毎に SitRep を更新し、関係者に送付する。**
- **通知先が応答しない場合は 5 分後に再送し、記録に「再送: HH:MM 応答なし」と残す。**
