## 1 本章の範囲

F4（復旧）フェーズで使用する 10 ランブック（RUN-020〜029）を定義する。各ランブックには「トリガー・所要時間目安・手順・確認コマンド・完了判定」を含む。

| RUN ID | 復旧対象 |
|---|---|
| RUN-020 | Docker コンテナ再起動 |
| RUN-021 | PostgreSQL 接続復旧 |
| RUN-022 | WAL 適用・PITR 復旧 |
| RUN-023 | pg_dump バックアップからの復元 |
| RUN-024 | Docker イメージロールバック |
| RUN-025 | IIS リバースプロキシ設定リセット |
| RUN-026 | Outbox 手動再送 |
| RUN-027 | JWT 緊急失効（特定ユーザー） |
| RUN-028 | ディスク緊急確保 |
| RUN-029 | ハンディ端末 SQLite データ強制同期 |

**復旧前の必須確認**

```
CHK-005: 復旧前確認
□ SitRep（F3）で特定された根本原因を確認した
□ 実施する RUN 番号が F3 判定マトリクスと一致している
□ ログ・証拠の保全が完了している（破壊的操作の前）
□ ハッシュチェーン関連の場合は 08 章の手順に移行した
```

## 2 RUN-020: Docker コンテナ再起動

**トリガー**: API 無応答（コンテナ停止・OOM kill）・RUN-010 で「コンテナ停止」と判定  
**所要時間目安**: 5 分以内

```bash
# Step 1: ログ保全（再起動前）
docker compose logs api --since 1h > /tmp/api_before_restart_$(date +%Y%m%d_%H%M%S).log
docker compose logs postgres --since 1h > /tmp/postgres_before_restart_$(date +%Y%m%d_%H%M%S).log

# Step 2: コンテナ再起動
docker compose restart api
# または全スタック再起動が必要な場合:
docker compose up -d

# Step 3: 起動確認（最大 30 秒待機）
for i in $(seq 1 6); do
  curl -fsS http://localhost:8080/health && echo "OK" && break
  echo "Waiting... ($i/6)"
  sleep 5
done

# Step 4: PostgreSQL 疎通確認
pg_isready -h localhost -p 5432
```

**確認コマンド**

```bash
docker compose ps
curl -fsS http://localhost:8080/health | jq .
```

**完了判定**: `/health` が HTTP 200 を返し、DB 接続が正常であることを確認

**本節で確定した方針**
- **再起動前に必ずログをファイルに保全する（ログが証拠となる際は保全を優先する）。**
- **`docker compose restart` のみで回復しない場合は RUN-021〜024 を順次評価する。**
- **再起動後の health check で 30 秒以内に 200 が返らない場合は P1 に昇格させる。**

## 3 RUN-021: PostgreSQL 接続復旧

**トリガー**: RUN-011 で接続枯渇・プロセス停止と判定  
**所要時間目安**: 10 分以内

```bash
# Step 1: 接続リセット（アイドル接続の強制終了）
psql -U postgres -d work_nav -c \
  "SELECT pg_terminate_backend(pid) FROM pg_stat_activity
   WHERE state = 'idle' AND state_change < NOW() - INTERVAL '5 minutes';"

# Step 2: PostgreSQL 再起動（最終手段）
docker compose restart postgres

# Step 3: PostgreSQL 起動確認
pg_isready -h localhost -p 5432

# Step 4: 接続数正常化確認
psql -U postgres -c \
  "SELECT count(*) as active_connections FROM pg_stat_activity WHERE state != 'idle';"

# Step 5: API コンテナから再接続確認
docker compose restart api
curl -fsS http://localhost:8080/health | jq .
```

**完了判定**: `pg_isready` が accepting connections を返し、API からの DB クエリが成功

**本節で確定した方針**
- **接続強制終了は実行中トランザクションを失うリスクがあるため、Step 1 の前にアクティブな重要トランザクションがないことを確認する。**
- **PostgreSQL 再起動（Step 2）は接続リセット（Step 1）で回復しない場合のみ実施する。**
- **再起動後は必ずハッシュチェーン整合性を確認する（BAT-003 手動実行）。**

## 4 RUN-022: WAL 適用・PITR 復旧

**トリガー**: DB 破損・RUN-011 でデータ破損と判定・RPO 超過のデータ損失が発生  
**所要時間目安**: 30〜60 分（データ量依存）

```bash
# Step 1: WAL アーカイブ確認
ls -lth {WAL_ARCHIVE_PATH}/ | head -20

# Step 2: PITR 復旧先タイムスタンプ決定（破損直前）
# 例: 2026-05-18 14:30:00 に破損が発生した場合
export RECOVERY_TARGET_TIME="2026-05-18 14:29:00"

# Step 3: PostgreSQL 停止（Docker 経由）
docker compose stop postgres

# Step 4: データディレクトリのバックアップ
sudo cp -rp {PGDATA_PATH} {PGDATA_PATH}.backup_$(date +%Y%m%d_%H%M%S)

# Step 5: recovery.conf / postgresql.auto.conf に PITR 設定追加
# restore_command = 'cp {WAL_ARCHIVE_PATH}/%f "%p"'
# recovery_target_time = '2026-05-18 14:29:00'
# recovery_target_action = 'promote'

# Step 6: PostgreSQL 起動（WAL リプレイ開始）
docker compose start postgres
docker compose logs postgres -f | grep -E "recovery|LOG|FATAL" | head -50
```

**完了判定**: `SELECT now();` が実行可能 + ハッシュチェーン整合性確認（BAT-003 手動実行）

**本節で確定した方針**
- **PITR 実施前は必ず現行データディレクトリをバックアップする（Step 4）。**
- **RECOVERY_TARGET_TIME は障害発生時刻の 1 分前に設定し、過去への復元範囲を最小化する。**
- **PITR 完了後に BAT-003（ハッシュチェーン検証）を手動で実行し整合性を確認する。**

## 5 RUN-023: pg_dump バックアップからの復元

**トリガー**: RUN-022 が実施不可（WAL アーカイブ欠損）・pg_dump バックアップが最終手段  
**所要時間目安**: 20〜40 分（バックアップサイズ依存）

```bash
# Step 1: バックアップ一覧確認（最新ファイルを特定）
ls -lth {BACKUP_PATH}/pg_dump/ | head -10
# 例: work_nav_2026-05-18_02-00-00.dump

# Step 2: PostgreSQL 停止・DB 削除
docker compose stop postgres
docker compose exec postgres dropdb -U postgres work_nav

# Step 3: DB 再作成・復元
docker compose exec postgres createdb -U postgres work_nav
pg_restore -U postgres -d work_nav -v --no-owner \
  {BACKUP_PATH}/pg_dump/work_nav_2026-05-18_02-00-00.dump

# Step 4: 復元完了確認
psql -U postgres -d work_nav \
  -c "SELECT schemaname, tablename, n_live_tup FROM pg_stat_user_tables ORDER BY n_live_tup DESC LIMIT 10;"

# Step 5: PostgreSQL + API 再起動
docker compose up -d
curl -fsS http://localhost:8080/health | jq .
```

**完了判定**: テーブルレコード数が想定値と一致 + API health check 200 OK

**本節で確定した方針**
- **pg_dump 復元は当日 02:00 のバックアップを基点とするため、バックアップ以降のデータは失われる。quality_admin にデータロス範囲を報告する。**
- **復元前に現行 DB の pg_dump を取得し証拠保全する。**
- **復元後は必ず BAT-003（ハッシュチェーン検証）を手動実行し整合性を確認する。**

## 6 RUN-024: Docker イメージロールバック

**トリガー**: デプロイ後に API 異常が発生・新バージョンで致命的バグ確認  
**所要時間目安**: 5 分以内

```bash
# Step 1: 現在のイメージタグ確認
docker compose ps
docker images | grep work-nav-api

# Step 2: 前バージョンタグの確認
# docker images 出力の Created が古い方が前バージョン

# Step 3: IMAGE_TAG を前バージョンに変更（.env 更新）
# 例: IMAGE_TAG=v1.2.3 → IMAGE_TAG=v1.2.2
vi {PROJECT_PATH}/.env
# または:
sed -i 's/IMAGE_TAG=v1.2.3/IMAGE_TAG=v1.2.2/' {PROJECT_PATH}/.env

# Step 4: ロールバック適用
docker compose up -d api

# Step 5: 正常確認
curl -fsS http://localhost:8080/health | jq .
docker compose logs api --since 5m | tail -30
```

**完了判定**: `/health` が 200 OK かつログにエラーがない状態

**本節で確定した方針**
- **IMAGE_TAG 変更は `.env` ファイルを直接編集し、変更内容を INC 記録に残す。**
- **ロールバック後は問題のあったバージョンを本番デプロイ禁止リストに登録する。**
- **ロールバックが完了した後に F5（ポストモーテム）でデプロイ手順の改善策を確定する。**

## 7 RUN-025: IIS リバースプロキシ設定リセット

**トリガー**: IIS から バックエンド API への接続が 5xx・設定変更後に疎通断  
**所要時間目安**: 10 分以内

```powershell
# Step 1: IIS 状態確認（管理者権限 PowerShell）
Get-Service W3SVC
Get-WebSite | Select-Object Name, State, PhysicalPath

# Step 2: IIS リセット
iisreset /restart

# Step 3: バックエンド疎通確認
Test-NetConnection -ComputerName localhost -Port 8080

# Step 4: web.config バックアップ確認
# {IIS_SITE_PATH}/web.config.backup が存在する場合は復元
Copy-Item "{IIS_SITE_PATH}/web.config.backup" "{IIS_SITE_PATH}/web.config" -Force
iisreset /restart

# Step 5: 疎通確認
Invoke-WebRequest -Uri "http://localhost:8080/health" -UseBasicParsing
```

**完了判定**: IIS 経由でのフロントエンドアクセスが正常に API に到達している状態

**本節で確定した方針**
- **IIS 設定変更は必ず事前にバックアップ（web.config.backup）を作成する。**
- **iisreset の実施は全サイトへの影響があるため、P1/P2 以外では夜間メンテナンスで実施する。**
- **IIS ログ（C:\inetpub\logs\LogFiles）を保全してから iisreset を実施する。**

## 8 RUN-026: Outbox 手動再送

**トリガー**: 09 章の手順で「手動再送」と判定・RUN-015 で Outbox Worker 復旧後の DLQ 処理  
**所要時間目安**: 5 分以内

```bash
# Step 1: DLQ 統計確認
curl -fsS http://localhost:8080/api/v1/ops/dlq/stats | jq .

# Step 2: 手動再送（全件）
curl -fsX POST http://localhost:8080/api/v1/ops/dlq/replay \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" \
  -d '{"mode": "all"}' | jq .

# または特定イベントタイプのみ再送
curl -fsX POST http://localhost:8080/api/v1/ops/dlq/replay \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" \
  -d '{"mode": "filter", "event_type": "step_result_created"}' | jq .

# Step 3: 再送後の DLQ 件数確認
curl -fsS http://localhost:8080/api/v1/ops/dlq/stats | jq '.dlq_total'
```

**完了判定**: DLQ 件数が 0（または再送不可能な件数のみ残存）

**本節で確定した方針**
- **手動再送は system_admin の JWT 認証が必須。認証なしの再送 API 呼び出しは禁止する。**
- **再送対象の廃棄（削除）は quality_admin の承認後に実施する（09 章参照）。**
- **再送完了後は ALERT-001 の状態を Grafana で確認し、アラートが解消されたことを記録する。**

## 9 RUN-027: JWT 緊急失効（特定ユーザー）

**トリガー**: アカウント侵害疑い・RUN-014 で不正 JWT 利用が確認  
**所要時間目安**: 3 分以内

```bash
# Step 1: 対象ユーザーの現在のセッション確認
psql -U postgres -d work_nav \
  -c "SELECT user_id, issued_at, expires_at FROM jwt_sessions WHERE user_id = '{TARGET_USER_ID}' AND revoked_at IS NULL;"

# Step 2: 特定ユーザーの JWT を失効（API 経由）
curl -fsX POST http://localhost:8080/api/v1/ops/auth/revoke \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" \
  -d '{"user_id": "{TARGET_USER_ID}", "reason": "emergency_revocation"}' | jq .

# Step 3: 失効確認
psql -U postgres -d work_nav \
  -c "SELECT user_id, revoked_at, revoke_reason FROM jwt_sessions WHERE user_id = '{TARGET_USER_ID}';"

# Step 4: アクセスログ確認（失効後のアクセス試行がないことを確認）
docker compose logs api --since 5m | grep "{TARGET_USER_ID}"
```

**完了判定**: 対象ユーザーの JWT が全件失効し、失効後のアクセス試行が 401 で拒否されている

**本節で確定した方針**
- **JWT 失効は INC 記録に「失効理由・対象ユーザー数（個人名は禁止）・失効時刻」を記録する。**
- **全ユーザー一括失効は system_admin の手動承認と質量_admin への通知が必要。**
- **失効後に対象ユーザーへの通知と再認証手順の案内を行う。**

## 10 RUN-028: ディスク緊急確保

**トリガー**: ALERT-002（disk > 85%）・`df -h` で 85% 超確認  
**所要時間目安**: 15 分以内

```bash
# Step 1: ディスク使用状況詳細確認
df -h
du -sh /* 2>/dev/null | sort -rh | head -20
du -sh /var/lib/docker/ 2>/dev/null

# Step 2: 古いバックアップ確認（保管期間 7 年以内で古いもの）
ls -lth {BACKUP_PATH}/pg_dump/ | tail -20
# 最新 7 日分を残し古いファイルを削除（7 年保管義務のある世代は削除不可）

# Step 3: Docker 未使用リソースの確認と削除
docker system df
docker image prune -f          # 未使用イメージ削除
docker volume prune -f         # 未使用ボリューム削除（本番 DB ボリューム以外）

# Step 4: ログのローテーション確認
ls -lth /var/log/ | head -20
journalctl --disk-usage

# Step 5: 削除後の空き容量確認
df -h
```

**完了判定**: `disk_usage_percent < 75%`（ALERT-002 の閾値 85% より十分に余裕を確保）

**本節で確定した方針**
- **削除するファイルは必ず一覧を INC 記録に残す（何を削除したかのトレーサビリティ）。**
- **7 年保管義務のある pg_dump ファイルは削除禁止。対象外の世代のみ削除する。**
- **Docker ボリューム削除は PostgreSQL データボリュームを除外して実施する。**

## 11 RUN-029: ハンディ端末 SQLite データ強制同期

**トリガー**: RUN-019 で「SQLite sync 失敗・データ競合」と判定  
**所要時間目安**: 15 分以内（端末数依存）

```bash
# Step 1: 同期状態確認（API 側）
curl -fsS "http://localhost:8080/api/v1/ops/sync/status" \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" | jq .

# Step 2: 強制同期トリガー（特定端末）
curl -fsX POST "http://localhost:8080/api/v1/ops/sync/force" \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" \
  -d '{"device_id": "{TARGET_DEVICE_ID}", "mode": "push_server"}' | jq .

# Step 3: 同期完了確認
curl -fsS "http://localhost:8080/api/v1/ops/sync/status?device_id={TARGET_DEVICE_ID}" \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" | jq '.sync_status'
```

**完了判定**: `sync_status = "synced"` かつ最終同期時刻が現時刻から 5 分以内

**本節で確定した方針**
- **強制同期の mode は `push_server`（サーバーデータを端末に上書き）を基本とし、`push_device` は quality_admin の承認が必要。**
- **同期競合が発生した場合は競合内容を INC 記録に保全し、ALCOA+ Contemporaneous を確認する。**
- **強制同期後はハンディ端末の画面でデータ表示を目視確認する。**
