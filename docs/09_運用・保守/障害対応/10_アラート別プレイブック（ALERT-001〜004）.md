# 10 アラート別プレイブック（ALERT-001〜004）

最終更新: 2026-05-18（Unit-14: 2バイナリ分割対応） | 管理者: system_admin | 根拠要件: OPS-056（F3/F4）

---

## 1 概要

本章は Prometheus から発報される 4 アラート（ALERT-001〜004）の各プレイブックを定義する。各プレイブックは「トリガー条件（PromQL）・重大度・初動コマンド・復旧手順・誤検知判定・完了確認」を含む。

**バイナリ別アラート対応早見表**

| アラート ID | 名称 | 重大度 | 初期 P 分類 | 調査対象バイナリ |
|---|---|---|---|---|
| ALERT-001 | Outbox DLQ 滞留 | CRITICAL | P2 | **terminal-api**（OutboxWorker 実体）|
| ALERT-002 | ディスク使用率高 | HIGH | P2〜P3 | インフラ共通（バイナリ非依存）|
| ALERT-003 | API P95 レイテンシ高 | WARN | P3 | **terminal-api**（現場端末向けエンドポイント）|
| ALERT-004 | バックアップ遅延 | HIGH | P2 | **master-api**（BAT-001 実行元）|
| ALERT-005 | ハッシュチェーン破断（MET-006）| CRITICAL | P1（即時）| **master-api**（HashChainVerifier 実体）|

## 2 ALERT-001: Outbox DLQ 滞留

### 2.1 アラート定義

```yaml
# Prometheus AlertRule
- alert: OutboxDlqHigh
  expr: outbox_dlq_total > 100
  for: 5m
  severity: critical
  annotations:
    summary: "Outbox DLQ に {{ $value }} 件が滞留中"
```

**トリガー条件（PromQL）**: `outbox_dlq_total > 100` が 5 分間継続

**担当バイナリ**: OutboxWorker は **terminal-api**（8080）内の tokio task として動作する。DLQ 管理 API（手動再投入）は **master-api**（8081）が `/api/v1/ops/outbox/dlq` で提供する（管理コンソール向け）。

### 2.2 初動コマンド（診断）

```bash
# まず terminal-api（OutboxWorker 実体）のヘルスと直近ログを確認
curl -fsS http://localhost:8080/health | jq .
docker compose logs terminal-api --since 30m | grep -E "outbox|DLQ|worker"

# DLQ 統計確認（管理 API: master-api 経由）
curl -fsS http://localhost:8081/api/v1/ops/outbox/dlq/stats \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" | jq .

# DLQ エラー内容確認
psql -U postgres -d work_nav \
  -c "SELECT event_type, error_message, count(*) FROM outbox_dlq GROUP BY 1,2 ORDER BY 3 DESC LIMIT 10;"
```

### 2.3 復旧手順

詳細は 09 章を参照する。概要:
1. 根本原因分類（一時障害 / 永続障害 / アプリバグ）
2. 根本原因解消（RUN-010〜019 該当手順）
3. OutboxWorker 再起動（`docker compose restart terminal-api`）
4. 手動再送（RUN-026: master-api 経由 POST `/api/v1/ops/outbox/dlq/replay`）
5. 廃棄判断（quality_admin 承認後）

### 2.4 誤検知判定

| 状況 | 誤検知判定基準 |
|---|---|
| テスト実行直後 | 開発・テスト環境のイベントが流入した場合 |
| メンテナンス中 | Outbox Worker を意図的に停止中の場合 |

誤検知の場合も INC 記録に「誤検知・理由」を記載してクローズする。

### 2.5 完了確認

```bash
curl -fsS "http://localhost:9090/api/v1/query?query=outbox_dlq_total" | jq '.data.result[].value[1]'
# 期待値: < 100（アラート閾値未満）
```

**本節で確定した方針**
- **ALERT-001 は terminal-api 内 OutboxWorker の問題である。master-api のみ停止の場合は DLQ 統計 API にアクセスできないが、OutboxWorker（terminal-api）自体は影響を受けない。**
- **ALERT-001 は P2 として対応開始し、DLQ が 1000 件を超えた場合は P1 に昇格させる。**
- **廃棄判断は quality_admin の承認を必須とし、system_admin 単独での廃棄は禁止する。**
- **アラート解消後 24 時間は DLQ 件数を Grafana で監視し再発を確認する。**

## 3 ALERT-002: ディスク使用率高

### 3.1 アラート定義

```yaml
# Prometheus AlertRule
- alert: DiskUsageHigh
  expr: disk_usage_percent{mountpoint="/"} > 85
  for: 5m
  severity: high
  annotations:
    summary: "ディスク使用率 {{ $value }}%（閾値: 85%）"
```

**トリガー条件（PromQL）**: `disk_usage_percent > 85` が 5 分間継続

### 3.2 初動コマンド（診断）

```bash
# ディスク使用率詳細確認
df -h

# 大容量ディレクトリ確認
du -sh /* 2>/dev/null | sort -rh | head -20
du -sh /var/lib/docker/ 2>/dev/null
du -sh {BACKUP_PATH}/ 2>/dev/null

# Docker ストレージ確認
docker system df

# pg_dump バックアップ一覧
ls -lth {BACKUP_PATH}/pg_dump/ | head -20

# PostgreSQL データ領域確認
psql -U postgres -c "SELECT pg_size_pretty(pg_database_size('work_nav'));"
```

### 3.3 復旧手順

詳細は RUN-028 を参照する。概要:

1. バックアップファイルの整理（7 年保管義務対象外の古い世代を削除）
2. Docker 未使用イメージ・ボリュームの削除
3. ログローテーションの確認
4. 削除後に `df -h` で 75% 以下を確認

### 3.4 誤検知判定

| 状況 | 確認方法 |
|---|---|
| ディスク監視対象のマウントポイントが誤り | `df -h` で実際のマウントポイントを確認 |
| Node Exporter の設定ミス | Prometheus のターゲット設定を確認 |

### 3.5 完了確認

```bash
curl -fsS "http://localhost:9090/api/v1/query?query=disk_usage_percent" | jq '.data.result'
# 期待値: < 75（余裕を持って 85% 閾値の 10% 下）
df -h
```

**本節で確定した方針**
- **ALERT-002 の初期 P 分類は disk_usage_percent が 90% 超で P2、85〜90% で P3 とする。**
- **7 年保管義務のある pg_dump ファイルは削除禁止。対象外の世代のみ削除する。**
- **ディスク確保後は容量増設の計画を保守記録に登録する。**

## 4 ALERT-003: API P95 レイテンシ高

### 4.1 アラート定義

```yaml
# Prometheus AlertRule
- alert: ApiLatencyHigh
  expr: histogram_quantile(0.95, rate(api_step_event_post_duration_ms_bucket[5m])) > 5000
  for: 5m
  severity: warning
  annotations:
    summary: "API P95 レイテンシ {{ $value }}ms（閾値: 5000ms）"
```

**担当バイナリ**: 作業ログ受信（現場ハンディ端末向け）のレイテンシは **terminal-api**（8080）で発生する。マスタ系エンドポイントのレイテンシは master-api（8081）で確認する。

**トリガー条件（PromQL）**: `api_step_event_post_duration_p95_ms > 5000` が 5 分間継続

### 4.2 初動コマンド（診断）

```bash
# P95/P99 レイテンシ確認（terminal-api の作業ログエンドポイント）
curl -fsS "http://localhost:9090/api/v1/query?query=histogram_quantile(0.95,rate(api_step_event_post_duration_ms_bucket[5m]))" | jq '.data.result'

# slow query 確認
psql -U postgres -d work_nav \
  -c "SELECT query, calls, mean_exec_time, total_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;"

# DB 接続・ロック確認
psql -U postgres -d work_nav \
  -c "SELECT wait_event_type, wait_event, count(*) FROM pg_stat_activity GROUP BY 1,2 ORDER BY 3 DESC;"

# tokio スレッド確認（terminal-api）
curl -fsS "http://localhost:9090/api/v1/query?query=tokio_workers_busy{job=\"work-nav-terminal-api\"}" | jq '.data.result'

# コンテナリソース確認
docker stats --no-stream
```

### 4.3 復旧手順（パターン別）

**パターン A: slow query が原因**

```sql
-- EXPLAIN ANALYZE で実行計画確認
EXPLAIN ANALYZE SELECT ... ; -- 実際の遅いクエリ

-- インデックス使用状況確認
SELECT schemaname, tablename, indexname, idx_scan
FROM pg_stat_user_indexes
ORDER BY idx_scan ASC
LIMIT 10;
```

**パターン B: 接続枯渇が原因**

```bash
# 接続リセット（RUN-021）
psql -U postgres -d work_nav -c \
  "SELECT pg_terminate_backend(pid) FROM pg_stat_activity
   WHERE state = 'idle' AND state_change < NOW() - INTERVAL '5 minutes';"
```

**パターン C: tokio スレッド枯渇**

```bash
# terminal-api 再起動（RUN-020）
# 作業ログレイテンシが原因の場合は terminal-api のみ再起動する
docker compose restart terminal-api
```

### 4.4 誤検知判定

| 状況 | 確認方法 |
|---|---|
| 大量データのインポート中 | バッチジョブの実行状況を確認 |
| ピーク時間帯（始業直後） | 通常のピークパターンと比較 |

### 4.5 完了確認

```bash
curl -fsS "http://localhost:9090/api/v1/query?query=histogram_quantile(0.95,rate(api_step_event_post_duration_ms_bucket[5m]))" | jq '.data.result[].value[1]'
# 期待値: < 2000ms（SLO の安全マージンを含む目標値）
```

**本節で確定した方針**
- **ALERT-003 は P3 として対応し、30 分以上継続する場合は P2 に昇格させる。**
- **slow query が発見された場合は EXPLAIN ANALYZE の結果を SitRep に添付する。**
- **P95 > 5000ms の根本原因はポストモーテム（11 章）に記録し、インデックス追加・クエリ改善のアクション項目を作成する。**

## 5 ALERT-004: バックアップ遅延

### 5.1 アラート定義

```yaml
# Prometheus AlertRule
- alert: BackupAgeHigh
  expr: backup_age_hours > 25
  for: 5m
  severity: high
  annotations:
    summary: "最終バックアップから {{ $value }} 時間経過（閾値: 25h）"
```

**担当バイナリ**: BAT-001（日次 pg_dump）は **master-api** 内の tokio task として動作する。バックアップ遅延が発生した場合は master-api のログを確認する。

**トリガー条件（PromQL）**: `backup_age_hours > 25` が 5 分間継続

### 5.2 初動コマンド（診断）

```bash
# バックアップファイル最終更新時刻確認
ls -lth {BACKUP_PATH}/pg_dump/ | head -5

# バックアップ世代確認（最新 5 件）
find {BACKUP_PATH}/pg_dump/ -name "*.dump" -printf "%T@ %f\n" | sort -rn | head -5

# BAT-001（日次バックアップ）の実行状態確認
psql -U postgres -d work_nav \
  -c "SELECT job_id, last_run_at, last_status, last_error FROM batch_job_status WHERE job_id = 'BAT-001';"

# バックアップログ確認（master-api が BAT-001 を担当）
docker compose logs master-api --since 24h | grep -E "backup|BAT-001"
```

### 5.3 復旧手順

**手動バックアップの実行**

```bash
# Step 1: 手動 pg_dump 実行
BACKUP_FILE="{BACKUP_PATH}/pg_dump/work_nav_manual_$(date +%Y%m%d_%H%M%S).dump"
pg_dump -U postgres -d work_nav --no-owner -F c -f ${BACKUP_FILE}

# Step 2: バックアップファイルの確認
ls -lh ${BACKUP_FILE}
pg_restore --list ${BACKUP_FILE} | head -20

# Step 3: Prometheus メトリクスを更新（バックアップ完了を通知）
# バックアップ管理は master-api（8081）が担当
curl -fsX POST http://localhost:8081/api/v1/ops/backup/notify \
  -H "Authorization: Bearer ${SYSTEM_ADMIN_JWT}" \
  -d '{"backup_file": "'${BACKUP_FILE}'"}'

# Step 4: BAT-001 の次回自動実行スケジュール確認
psql -U postgres -d work_nav \
  -c "SELECT job_id, next_run_at FROM batch_job_status WHERE job_id = 'BAT-001';"
```

### 5.4 誤検知判定

| 状況 | 確認方法 |
|---|---|
| 日付変更直後（00:00〜02:00）にバックアップ実行前 | バックアップスケジュール（02:00）と現在時刻を確認 |
| Prometheus の backup_age_hours メトリクス更新遅延 | メトリクスの最終更新時刻を確認 |

### 5.5 完了確認

```bash
curl -fsS "http://localhost:9090/api/v1/query?query=backup_age_hours" | jq '.data.result[].value[1]'
# 期待値: < 25（アラート閾値未満）

# バックアップファイルの整合性確認
pg_restore --list ${BACKUP_FILE} > /dev/null && echo "BACKUP OK" || echo "BACKUP CORRUPT"
```

**本節で確定した方針**
- **ALERT-004 は P2 として対応し、手動バックアップを 30 分以内に完了させる。**
- **バックアップ失敗の根本原因（ディスク不足・スケジューラ停止等）は解消後に BAT-001 の次回自動実行を確認する。**
- **ALERT-004 が 3 日連続で発報した場合はポストモーテム（11 章）を実施しバックアップ手順を見直す。**

## 6 ALERT-005: ハッシュチェーン破断（MET-006）

### 6.1 アラート定義

```yaml
# Prometheus AlertRule（MET-006: hash_chain.break_count）
- alert: HashChainBroken
  expr: hash_chain_break_count_total > 0
  for: 0m  # 即時発報（P1 インシデント）
  severity: critical
  annotations:
    summary: "ハッシュチェーン破断を検知 — P1 インシデント"
    dashboard: "{{ CFG/MONITORING_URL }}/d/wnav-security"  # 実値は CFG 管理
    runbook: "docs/09_運用・保守/障害対応/08_ハッシュチェーン破断対応.md"
    pagerduty_policy: "{{ CFG/PAGERDUTY_HASH_CHAIN_POLICY }}"  # 実値は CFG 管理
```

**重大度**: CRITICAL / **初期 P 分類**: P1（即時対応）

### 6.2 発報時の初動（30 秒以内）

1. `08_ハッシュチェーン破断対応.md` を開く（P1 プレイブック全文）
2. system_admin + quality_admin に PagerDuty / メール通知が自動送信されていることを確認する
3. Grafana wnav-security ボードで `hash_chain_break_count_total` の値と `broken_at_block_id` を確認する

### 6.3 詳細プレイブック

P1 全 9 ステップの詳細手順（SQL 確認・証拠保全・2 名承認・補正 INSERT / ALG-025）は  
`docs/09_運用・保守/障害対応/08_ハッシュチェーン破断対応.md` が権威。本節は発報経路のみを定義する。

**本節で確定した方針**
- **ALERT-005（MET-006）は hash_chain_break_count_total > 0 の即時で P1 と分類し、for: 0m（猶予なし即時発報）とする。**
- **実際のダッシュボード URL と PagerDuty ポリシー ID は CFG で管理し、アラートルール定義にハードコードしない。**
- **補正手順は ALG-025（ADR-008）に従い、`08_ハッシュチェーン破断対応.md` §9 の 2 名承認 INSERT を実施する。**
