# 99 前提制約と障害対応サブが約束しないこと

最終更新: 2026-05-18 | 管理者: system_admin

---

## 1 本章の目的

本章は本ディレクトリの障害対応ドキュメント（01〜13 章）が提供しない能力・保証しないことを明示する。過剰な期待による誤解を防ぎ、実現可能な対応範囲を確定する。

---

## 2 24 時間オンコール対応

### 2.1 現実的制約

本システムは IT 担当 1 名（system_admin）による個人開発・個人運用である。以下を宣言する。

| 項目 | 実態 |
|---|---|
| 営業時間外対応 | system_admin の判断による。義務として設定しない |
| 夜間・休日の P1 対応 | system_admin が気づいた場合に対応する。保証しない |
| 自動復旧 | 一部の縮退動作（LEVEL-2 自動発動等）は自動。それ以外は人間が必要 |
| 代替担当者 | 設定しない（個人開発の現実的制約）|

### 2.2 夜間・休日の P1 発生時

夜間・休日に P1 が発生し、system_admin が未検知の場合は以下が起きる:

1. Prometheus アラートがメール通知を送信する（ALERT-001〜004）
2. system_admin が通知に気づいた時点で対応を開始する
3. P1 SLO（F1: 5 分以内）は夜間・休日には適用しない

**夜間における製造ライン稼働の有無**

製造ライン稼働時間外（例: 夜間・土日）は P1 発生時の影響が「完全停止」ではなく「次の稼働開始時に備えた復旧」となる。この場合は営業開始前の復旧完了を目標とする。

**本節で確定した方針**
- **24 時間オンコール対応は提供しない。これは現実の制約であり改善項目ではない（個人開発の前提）。**
- **製造ライン稼働時間外の P1 は「稼働開始前復旧」をゴールとし、SLO は適用しない。**
- **本制約はステークホルダー（現場監督・quality_admin）に事前に説明し記録する。**

## 3 自動障害修復

### 3.1 ヒューマンインザループの必須性

本システムの障害対応は「ヒューマンインザループ（Human-in-the-Loop）」を原則とする。

| 操作 | 自動化の可否 |
|---|---|
| Grafana アラートの発報 | 自動（Prometheus → メール）|
| LEVEL-2 の自動発動（ハンディ オフライン切替）| 自動（アプリ内ロジック）|
| Docker コンテナの自動再起動（restart: always） | 自動（Docker Compose 設定）|
| P 分類の決定 | 人間（system_admin / 現場監督）が判断する |
| DB 操作・データ変更を伴う復旧 | 人間が実施する |
| ハッシュチェーン破断の修復 | 二人承認（system_admin + quality_admin）が必須 |
| DLQ の廃棄判断 | quality_admin の承認が必須 |

ALCOA+ の原則上、作業記録・検査記録に影響する操作の自動修復は導入しない。

**本節で確定した方針**
- **自動障害修復は検知・通知・縮退発動の一部にのみ適用し、データ変更を伴う操作は人間が行う。**
- **自動再起動（Docker restart: always）は無限ループを防ぐため再起動回数の上限を設定する。**
- **将来的な自動修復の拡張は ADR で設計方針を確定してから実装する。**

## 4 SLA 補償

### 4.1 本システムの性格

本システムは個人開発・社内利用のシステムであり、外部顧客向けの SLA 契約は存在しない。

| 項目 | 実態 |
|---|---|
| 金銭的補償 | 提供しない |
| ダウンタイム クレジット | 提供しない |
| SLO の法的拘束力 | ない（ベストエフォートの運用目標）|
| SLO 未達時のペナルティ | ない |

本ドキュメントに記載する SLO（P1 RTO: 1 時間等）は運用目標であり、達成できなかった場合の補償義務は発生しない。

**本節で確定した方針**
- **本システムは SLA 補償を提供しない。この制約はシステム導入時にステークホルダーに説明する。**
- **SLO は達成目標として設定し、未達の場合は根本原因分析（ポストモーテム）で改善する。**
- **製造品質の観点で重大な影響が発生した場合は SLA ではなく ALCOA+ 規程に基づき対応する。**

## 5 外部 SOC（Security Operations Center）

### 5.1 SOC 不在の制約

本システムは外部 SOC を利用しない。セキュリティインシデントの検知・対応は system_admin が担う。

| 項目 | 実態 |
|---|---|
| 24 時間セキュリティ監視 | 提供しない |
| 脅威インテリジェンス連携 | 提供しない |
| 外部 SIEM との連携 | 提供しない |
| フォレンジック調査（専門業者） | 必要時に外部委託（個別対応） |

### 5.2 セキュリティインシデントの対応範囲

本ドキュメントが対応するセキュリティインシデントの範囲:

| 対応範囲 | 手順 |
|---|---|
| JWT 不正利用の検知 | 13 章（ERR-AUTH-001）+ RUN-027 |
| ハッシュチェーン破断（改ざん疑い）| 08 章 |
| アカウントロック | 13 章（ERR-AUTH-003）|
| LDAP 認証断 | 13 章（ERR-EXT-002）|

本ドキュメントが対応しないセキュリティインシデント:

| 対応範囲外 | 理由 |
|---|---|
| 高度な標的型攻撃（APT）| 専門的フォレンジック能力が必要 |
| ランサムウェア | 外部専門家への委託が必要 |
| サプライチェーン攻撃 | 本システムの対応能力を超える |

**本節で確定した方針**
- **外部 SOC は利用しない。この制約は現実の個人開発環境の前提として維持する。**
- **セキュリティインシデント発生時に本ドキュメントの手順が不足する場合は外部専門家に委託する。**
- **ランサムウェア等の深刻なセキュリティ侵害が疑われる場合は即座にシステムをオフラインにし、外部専門家に連絡する。**

## 6 本ドキュメントが約束すること（正の宣言）

前節で「約束しないこと」を宣言したが、以下は本ドキュメントが提供する。

| 約束すること |
|---|
| system_admin が稼働時間中に検知した P1/P2 への対応手順の提供 |
| ALCOA+ 原則を維持した障害対応（削除・改ざんなしの補正手順）|
| Just Culture に基づくポストモーテムの実施手順の提供 |
| 製造ライン稼働時間中の 5 フェーズ SLO の達成に向けた最大努力 |
| BCP（紙フォールバック）による LEVEL-3 時の作業継続手段の提供 |
