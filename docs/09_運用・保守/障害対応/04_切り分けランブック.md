# 04 切り分けランブック

最終更新: 2026-05-18 | 管理者: system_admin | 根拠要件: OPS-056（F3）

---

## 1 本章の範囲

F3（切り分け）フェーズで使用する 10 ランブック（RUN-010〜019）を定義する。各ランブックは「トリガー・手順・完了判定」を持つ。

図: fig_inc_triage_decision（img/ 配下）を参照

| RUN ID | 切り分け対象 |
|---|---|
| RUN-010 | API 無応答 |
| RUN-011 | DB 接続エラー |
| RUN-012 | ネットワーク切断 |
| RUN-013 | ジョブ失敗（BAT-001〜010） |
| RUN-014 | 認証エラー（ERR-AUTH-001〜004） |
| RUN-015 | Outbox 滞留 |
| RUN-016 | 証明書エラー |
| RUN-017 | ハッシュチェーン異常（→ 08 章） |
| RUN-018 | パフォーマンス劣化 |
| RUN-019 | ハンディ接続不可 |

## 2 RUN-010: API 無応答の切り分け

**トリガー**: `/health` タイムアウト・全エンドポイント 5xx・ハンディから接続不可報告  
**所要時間目安**: 10 分以内

```bash
# Step 1: API ヘルスチェック
curl -fsS http://localhost:8080/health || echo "API DEAD"

# Step 2: Docker コンテナ状態確認
docker compose ps

# Step 3: API コンテナログ（直近 30 分）
docker compose logs api --since 30m | tail -100

# Step 4: DB 疎通確認
pg_isready -h localhost -p 5432
# 応答: "localhost:5432 - accepting connections"

# Step 5: ディスク・メモリ確認
df -h
free -h

# Step 6: ネットワーク確認（コンテナ間）
docker network inspect work-nav-net 2>/dev/null || docker network ls
```

**判定マトリクス**

| API | Docker | DB | 判定 | 次の手順 |
|---|---|---|---|---|
| DEAD | UP | OK | API アプリバグ / OOM | RUN-020（再起動）|
| DEAD | DOWN | — | コンテナ停止 | RUN-020（起動）|
| DEAD | UP | DEAD | DB 停止が原因 | RUN-021 |
| DEAD | UP | OK | ネットワーク断 | RUN-012 |

**完了判定**: 原因が 1 つに絞られ RUN-020〜029 の手順が決定した状態

**本節で確定した方針**
- **Step 1〜6 を順番に実行し、最初に確認できた「DEAD」の層を根本と判定する。**
- **コンテナ再起動（RUN-020）を先に行うことは禁止する（ログが失われる）。**
- **API DEAD が確認された時点で P1/P2 の判定を再確認する。**

## 3 RUN-011: DB 接続エラーの切り分け

**トリガー**: `ERR-DB-001`（DB 接続失敗）・`pg_isready` 失敗・API ログに connection refused  
**所要時間目安**: 10 分以内

```bash
# Step 1: PostgreSQL プロセス確認
docker compose ps postgres
# または
sudo systemctl status postgresql

# Step 2: PostgreSQL ログ確認
docker compose logs postgres --since 30m | tail -100

# Step 3: 接続数確認
psql -U postgres -c "SELECT count(*) FROM pg_stat_activity;"
psql -U postgres -c "SELECT max_conn, used FROM (SELECT setting::int as max_conn FROM pg_settings WHERE name='max_connections') a, (SELECT count(*) as used FROM pg_stat_activity) b;"

# Step 4: ディスク空き確認（PostgreSQL データ領域）
df -h /var/lib/postgresql/

# Step 5: PostgreSQL 設定確認
psql -U postgres -c "SHOW max_connections;"
psql -U postgres -c "SHOW work_mem;"
```

**判定マトリクス**

| 状態 | 判定 | 次の手順 |
|---|---|---|
| コンテナ停止 | DB プロセス停止 | RUN-021 |
| 接続数 = max_connections | 接続枯渇 | RUN-021（接続リセット）|
| ディスク 100% | ディスク枯渇 | RUN-028 |
| ログに FATAL: data corruption | データ破損 | RUN-022（PITR）|

**本節で確定した方針**
- **接続枯渇（max_connections 到達）は RUN-021 で接続をリセットし、接続プール設定を見直す。**
- **「data corruption」がログに現れた場合は即 P1 に昇格し RUN-022 を実施する。**
- **DB 切り分けの結果は SitRep に「DB 状態: XX 接続 / 最大 YY」を含めて記録する。**

## 4 RUN-012: ネットワーク切断の切り分け

**トリガー**: ハンディから API 疎通断・コンテナ間通信断・IIS → バックエンド疎通断  
**所要時間目安**: 8 分以内

```bash
# Step 1: Docker ネットワーク確認
docker network ls
docker network inspect $(docker network ls -q)

# Step 2: コンテナ間疎通確認
docker compose exec api ping -c 3 postgres

# Step 3: IIS リバースプロキシ→ API 疎通確認（Windows Server 側）
# PowerShell:
# Test-NetConnection -ComputerName localhost -Port 8080

# Step 4: ハンディ端末から API への疎通確認
# ハンディ端末から手動: curl http://{SERVER_IP}:8080/health
# または Grafana の api_up メトリクス確認:
# up{job="work-nav-api"} == 0 が一定期間継続中か確認

# Step 5: ファイアウォール確認
sudo ufw status
# または Windows Defender ファイアウォール状態確認（管理者権限）
```

**本節で確定した方針**
- **コンテナ間通信断の場合は Docker ネットワークを再作成する前に RUN-025 を実施する。**
- **ハンディ疎通断が確認された場合は 06 章 LEVEL-2 の発動を評価する。**
- **ファイアウォール設定変更は変更前後のルールを INC 記録に記録する。**

## 5 RUN-013: ジョブ失敗の切り分け（BAT-001〜010）

**トリガー**: ジョブ失敗アラート・スケジューラーログのエラー  
**所要時間目安**: 10 分以内

```bash
# Step 1: ジョブ実行ログ確認
docker compose logs scheduler --since 1h | grep -E "ERROR|FAILED|BAT-"

# Step 2: 最終ジョブ実行結果確認（DB テーブル）
psql -U postgres -d work_nav \
  -c "SELECT job_id, last_run_at, last_status, last_error FROM batch_job_status ORDER BY last_run_at DESC LIMIT 10;"

# Step 3: ジョブ別確認コマンド（該当するものを実行）
# BAT-003（ハッシュチェーン検証）失敗 → 08 章へ移行
psql -U postgres -d work_nav \
  -c "SELECT count(*) FROM hash_chain_errors WHERE detected_at > NOW() - INTERVAL '1 day';"
```

**ジョブ別判定**

| ジョブ ID | 失敗時の P 分類 | 次の手順 |
|---|---|---|
| BAT-001（日次 pg_dump） | P2 | RUN-023（バックアップ手動実行）|
| BAT-002（Outbox 処理） | P2 | RUN-015（Outbox 切り分け）|
| BAT-003（ハッシュチェーン検証） | P1 | RUN-017 → 08 章 |
| BAT-004〜010 | P3 | 個別ログ確認後 RUN-020 |

**本節で確定した方針**
- **BAT-003（ハッシュチェーン検証）が失敗した場合は即時 P1 として 08 章の手順を実施する。**
- **BAT-001（日次バックアップ）失敗は ALERT-004 と連動し P2 として対応する。**
- **ジョブ失敗の原因が DB 起因の場合は RUN-011 を先に完了させる。**

## 6 RUN-014: 認証エラーの切り分け（ERR-AUTH-001〜004）

**トリガー**: 認証失敗多発・ユーザーからのログイン不可報告  
**所要時間目安**: 8 分以内

```bash
# Step 1: 認証エラーログ集計（直近 30 分）
docker compose logs api --since 30m | grep -E "ERR-AUTH" | sort | uniq -c | sort -rn

# Step 2: LDAP 疎通確認（ERR-AUTH-004 / ERR-EXT-002 の場合）
# ldapsearch -x -H ldap://{LDAP_HOST}:389 -D "{BIND_DN}" -w "{PASSWORD}" -b "{BASE_DN}" "(uid=test)"

# Step 3: JWT 検証（ERR-AUTH-001）
# API ログの "JWT invalid" メッセージ確認
docker compose logs api --since 30m | grep "JWT"

# Step 4: アカウントロック状態確認（ERR-AUTH-003）
psql -U postgres -d work_nav \
  -c "SELECT user_id, login_attempt_count, locked_at FROM users WHERE locked_at IS NOT NULL;"
```

**エラーコード別判定**

| エラーコード | 典型原因 | 初動 |
|---|---|---|
| ERR-AUTH-001 | JWT 期限切れ / 署名不正 | RUN-027（JWT 確認）|
| ERR-AUTH-002 | PIN 認証失敗 | ユーザーに PIN リセット指示 |
| ERR-AUTH-003 | アカウントロック | DB でロック解除（管理者操作）|
| ERR-AUTH-004 | 権限不足 | ロール設定確認 |

**本節で確定した方針**
- **ERR-AUTH-001 が多発する場合は JWT 秘密鍵のローテーションを疑い RUN-027 を実施する。**
- **ERR-AUTH-003（アカウントロック）は INC 記録に対象ユーザー数を記録する（個人名は禁止）。**
- **LDAP 疎通断（ERR-EXT-002）は P2 として 06 章 LEVEL-1 の発動を評価する。**

## 7 RUN-015: Outbox 滞留の切り分け

**トリガー**: ALERT-001（`outbox_dlq_total > 100`）・Outbox pending 件数増加  
**所要時間目安**: 8 分以内

```bash
# Step 1: DLQ 統計 API 確認
curl -fsS http://localhost:8080/api/v1/ops/dlq/stats | jq .

# Step 2: Outbox pending 件数確認
psql -U postgres -d work_nav \
  -c "SELECT status, count(*) FROM outbox_events GROUP BY status;"

# Step 3: DLQ エラー内容確認
psql -U postgres -d work_nav \
  -c "SELECT id, event_type, error_message, retry_count, created_at FROM outbox_dlq ORDER BY created_at DESC LIMIT 20;"

# Step 4: Outbox Worker プロセス確認
docker compose logs api --since 30m | grep -E "outbox|Outbox"
```

**判定**

| DLQ 件数 | 増加速度 | 判定 | 次の手順 |
|---|---|---|---|
| > 100 かつ増加中 | 急増 | P2 | 09 章・RUN-026 |
| > 100 かつ横ばい | 停止 | P2（Outbox Worker 停止）| RUN-020 |
| > 100 かつ減少中 | 回復中 | P3 | 09 章で監視継続 |

**本節で確定した方針**
- **DLQ > 100 件は必ず ALERT-001 として P2 に分類し 09 章の復旧手順を実施する。**
- **DLQ 廃棄（削除）は quality_admin の承認を必要とする。**
- **Outbox Worker が停止している場合は RUN-020（Docker 再起動）で復旧を試みる。**

## 8 RUN-016: 証明書エラーの切り分け

**トリガー**: HTTPS 証明書期限切れアラート・ブラウザ/アプリの SSL エラー  
**所要時間目安**: 5 分以内

```bash
# Step 1: 証明書有効期限確認
openssl s_client -connect {SERVER_FQDN}:443 -servername {SERVER_FQDN} </dev/null 2>/dev/null \
  | openssl x509 -noout -dates

# Step 2: IIS 証明書状態確認（Windows Server / PowerShell）
# Get-ChildItem -Path Cert:\LocalMachine\My | Select-Object Subject, NotAfter | Sort-Object NotAfter

# Step 3: 期限残日数計算
openssl s_client -connect {SERVER_FQDN}:443 </dev/null 2>/dev/null \
  | openssl x509 -noout -enddate \
  | awk -F= '{print $2}' | xargs -I{} date -d {} +%s \
  | awk '{print ($1 - '$(date +%s)') / 86400 " days remaining"}'
```

**本節で確定した方針**
- **証明書期限切れは P2 として即時更新手順を実施する。**
- **証明書更新後は IIS を再起動し、全クライアントから HTTPS 接続を確認する。**
- **年次証明書更新は運用手順カレンダーに登録する。**

## 9 RUN-017: ハッシュチェーン異常の切り分け

**トリガー**: `ERR-DB-003`・`MET-006 > 0`・BAT-003 失敗  
**所要時間目安**: 5 分以内（確認のみ）

```bash
# Step 1: ハッシュチェーンエラーカウント確認
curl -fsS http://localhost:9090/api/v1/query?query=hash_chain_error_total | jq '.data.result'

# Step 2: DB でのエラー確認
psql -U postgres -d work_nav \
  -c "SELECT count(*) as error_count FROM hash_chain_errors WHERE resolved_at IS NULL;"
```

**判定**: `error_count > 0` の場合は即座に **08 章（ハッシュチェーン破断対応）** へ移行する。本 RUN-017 は切り分けのみで復旧手順は 08 章に委任する。

**本節で確定した方針**
- **ハッシュチェーン異常は確認した時点で P1 として 08 章を実施する。自己判断での修復は禁止する。**
- **ERR-DB-003 は ALCOA+ Original 不変性への直接脅威であり最優先で対処する。**
- **切り分け完了後は SitRep に「ハッシュチェーン破断確認 → 08 章移行」を記録する。**

## 10 RUN-018: パフォーマンス劣化の切り分け

**トリガー**: ALERT-003（P95 > 5000ms）・ユーザーからの「遅い」報告  
**所要時間目安**: 12 分以内

```bash
# Step 1: API P95 レイテンシ確認（Prometheus）
curl -fsS "http://localhost:9090/api/v1/query?query=histogram_quantile(0.95,rate(api_step_event_post_duration_ms_bucket[5m]))" | jq '.data.result'

# Step 2: slow query 確認
psql -U postgres -d work_nav \
  -c "SELECT query, calls, mean_exec_time, total_exec_time FROM pg_stat_statements ORDER BY mean_exec_time DESC LIMIT 10;"

# Step 3: DB 接続数・待機確認
psql -U postgres -d work_nav \
  -c "SELECT wait_event_type, wait_event, count(*) FROM pg_stat_activity GROUP BY 1,2 ORDER BY 3 DESC;"

# Step 4: tokio スレッド確認（Prometheus）
curl -fsS "http://localhost:9090/api/v1/query?query=tokio_workers_busy" | jq '.data.result'

# Step 5: コンテナリソース確認
docker stats --no-stream
```

**判定マトリクス**

| 原因 | 指標 | 判定 | 対処 |
|---|---|---|---|
| Slow query | mean_exec_time > 1000ms | P3 | EXPLAIN ANALYZE → インデックス追加 |
| 接続枯渇 | wait_event = "ClientRead" 多数 | P2→P3 | RUN-021（接続リセット）|
| tokio 枯渇 | tokio_workers_busy ≈ max | P2 | RUN-020（再起動）|
| DB ロック競合 | wait_event = "Lock" 多数 | P2 | ロック保持クエリの特定・kill |

**本節で確定した方針**
- **P95 > 5000ms が 15 分以上継続する場合は P3 から P2 に昇格させる。**
- **slow query の特定後はすぐに EXPLAIN ANALYZE を取得し SitRep に添付する。**
- **tokio スレッド枯渇は RUN-020 で再起動し、バックプレッシャー設定を見直す。**

## 11 RUN-019: ハンディ接続不可の切り分け

**トリガー**: ハンディ端末から「繋がらない」「同期できない」報告  
**所要時間目安**: 10 分以内

```bash
# Step 1: ハンディ→サーバー疎通確認（サーバー側で代理確認）
# Prometheus: up{job="work-nav-api"} の値確認
curl -fsS "http://localhost:9090/api/v1/query?query=up{job=\"work-nav-api\"}" | jq '.data.result'

# Step 2: WiFi AP 状態確認（管理端末から）
ping -c 3 {WIFI_AP_IP}

# Step 3: SQLite 同期ジョブ状態確認（API ログ）
docker compose logs api --since 30m | grep -E "sync|SQLite|offline"

# Step 4: JWT 期限確認（ハンディアプリのエラーコードを確認）
# ERR-AUTH-001 が報告されている場合 → JWT 期限切れ
docker compose logs api --since 30m | grep "ERR-AUTH-001"
```

**判定マトリクス**

| 症状 | 判定 | 対処 |
|---|---|---|
| API 側は正常・WiFi AP 無応答 | ネットワーク障害 | RUN-012 → 06 章 LEVEL-2 発動検討 |
| API 側正常・JWT エラー | JWT 期限切れ | RUN-027（JWT 確認）|
| API 側正常・SQLite sync 失敗 | アプリバグ / データ競合 | RUN-029（強制同期）|
| API 側 DEAD | API 障害 | RUN-010 |

**本節で確定した方針**
- **ハンディ疎通断が 60 秒継続した場合は 06 章 LEVEL-2（オフラインモード）の自動発動を確認する。**
- **WiFi AP 障害の場合はハンディに「オフラインモードで継続」の指示を行い記録する。**
- **SQLite 強制同期（RUN-029）は quantum_admin 承認なしには実施しない。**
