# 09 鍵ローテーション履歴テンプレ

本章は KEY-001〜009 の鍵ローテーション履歴台帳テンプレートを提供する。MNT-039・KEY-001〜009 定義（`04_概要設計/07_セキュリティ方式設計/`）に基づく周期ローテーションの記録義務を実体化する。

---

## 1. 鍵台帳の概要

| 項目 | 内容 |
|---|---|
| **管理対象** | KEY-001〜009（04_概要設計/07 が権威定義）|
| **更新タイミング** | 各ローテーション実施後（ALCOA+ Contemporaneous 原則）|
| **保全期間** | 7 年（廃棄後も廃棄記録として保全）|
| **アクセス制限** | system_admin のみ（fingerprint 以外の鍵材料は本台帳に記録しない）|

**重要**: 本台帳に記録するのは鍵の fingerprint・ローテーション日時・状態のみとする。鍵の材料（secret key bytes）は本台帳に記録しない。鍵材料は KEY-ID 別の鍵管理ストレージ（OS keychains / Vault / HSM）で管理する。

**本節で確定した方針**
- **鍵ローテーションは MNT-039 に定めた周期を遵守し、期限超過を禁止する。**
- **ローテーション記録は実施直後（ALCOA+ Contemporaneous 原則）に本台帳に記録する。**
- **本台帳は鍵材料を含まない。fingerprint・日時・状態のみを記録する。**

## 2. KEY-001〜009 ローテーション台帳

| KEY-ID | 鍵名称 | 周期 | 最終ローテーション | 次回期限 | 状態 |
|---|---|---|---|---|---|
| KEY-001 | JWT 署名秘密鍵 RSA-4096 | 90 日 | - | - | 未実施 |
| KEY-002 | JWT 公開鍵 | KEY-001 連動 | - | - | 未実施 |
| KEY-003 | 端末ローカル AES-256 | 端末初期化時 | - | - | 未実施 |
| KEY-004 | バックアップ AES-256-GCM | 365 日 | - | - | 未実施 |
| KEY-005 | 内部 CA RSA-4096 | 10 年 | - | - | 未実施 |
| KEY-006 | サーバーストレージ AES-256-GCM | 365 日 | - | - | 未実施 |
| KEY-007 | PDF 署名 RSA-4096 | 2 年 | - | - | 未実施 |
| KEY-008 | Outbox HMAC-SHA256 | 90 日 | - | - | 未実施 |
| KEY-009 | 日次ハッシュ salt | 1 日（自動） | - | - | 自動運用 |

### 2-1. KEY-001（JWT 署名秘密鍵）詳細

- **アルゴリズム**: RS256（RSA-4096 bit）
- **使用箇所**: バックエンド JWT 署名（IMPL-NNN）・ハンディ端末 Token 検証
- **ローテーション方式**: 新鍵生成 → 猶予期間（旧鍵も有効）→ 旧鍵廃棄
- **猶予期間**: 7 日（移行中の Token 有効期限切れを待つ）

### 2-2. KEY-004（バックアップ AES-256-GCM）詳細

- **アルゴリズム**: AES-256-GCM
- **使用箇所**: `../運用手順/05_バックアップ実施手順.md`・06 章アーカイブ
- **ローテーション方式**: 新鍵生成 → 次回バックアップから新鍵使用 → 旧鍵廃棄（旧バックアップの保持期間経過後）

### 2-3. KEY-008（Outbox HMAC-SHA256）詳細

- **アルゴリズム**: HMAC-SHA256
- **使用箇所**: Outbox イベントの改ざん検知（ERR-DB-003 関連）
- **ALCOA+ 影響**: Original 原則に直接関与する。ローテーション時は hash-chain-verify を必ず実行する
- **猶予期間**: 0 日（切り替え即時完全移行）

### 2-4. KEY-009（日次ハッシュ salt）詳細

- **アルゴリズム**: CSPRNG 生成 256 bit
- **使用箇所**: 日次ハッシュチェーン（TBL-audit_chain）
- **ローテーション方式**: BAT-NNN（日次バッチ）が自動生成・適用
- **手動ローテーション**: 原則不要（自動運用）。障害時は ADR-OPS-NNN で代替手順を確定する

**本節で確定した方針**
- **KEY-008 ローテーション時は hash-chain-verify を必ず実行し、ERROR 0 を確認してからローテーションを完了とする（ALCOA+ Original 原則）。**
- **KEY-004 の旧鍵廃棄は旧バックアップの保持期間（7 年）が経過してから実施する。保持期間中の旧鍵廃棄は禁止する。**
- **KEY-009 は自動運用を維持し、手動介入が必要な場合は ADR-OPS-NNN を起票して根拠を記録する。**

## 3. ローテーション実施記録テンプレート

各ローテーション実施時に以下のフォーマットで記録を追記する。

```yaml
# 鍵ローテーション記録テンプレート
---
key_id: KEY-NNN
rotation_id: REC-NNN  # 対応する保守記録
rotation_type: scheduled | emergency | compromise_response
rotated_at: YYYY-MM-DD HH:MM
rotated_by: system_admin

# fingerprint（鍵材料は記録しない）
new_key_fingerprint: sha256:<fingerprint>
old_key_fingerprint: sha256:<fingerprint>

# 猶予期間（KEY-001 等に適用）
grace_period_start: YYYY-MM-DD
grace_period_end: YYYY-MM-DD

# 旧鍵完全失効
old_key_revoked_at: YYYY-MM-DD HH:MM
old_key_revocation_confirmed: true | false

# 検証結果（KEY-008 は必須、他は推奨）
hash_chain_verify_result: PASS | N/A
esign_verify_result: PASS | N/A
jwt_verify_result: PASS | N/A
backup_verify_result: PASS | N/A

# 次回期限の設定
next_rotation_due: YYYY-MM-DD
```

### 3-1. KEY-001 ローテーション手順

```bash
# 1. RSA-4096 新鍵ペア生成
openssl genrsa -out /secrets/jwt_signing_key_new.pem 4096
openssl rsa -in /secrets/jwt_signing_key_new.pem -pubout -out /secrets/jwt_public_key_new.pem

# fingerprint 計算
openssl rsa -in /secrets/jwt_signing_key_new.pem -pubout | \
  openssl dgst -sha256 | awk '{print $2}'

# 2. アプリケーション設定を新鍵に向ける（猶予期間中は旧鍵も有効にする）
# ENV-NNN の JWT_SIGNING_KEY / JWT_PUBLIC_KEY 環境変数を更新
# 06_実装/09_環境変数台帳.md の ENV-NNN を参照

# 3. 猶予期間（7 日）後に旧鍵を廃棄
# 旧鍵ファイルを安全に削除
shred -u /secrets/jwt_signing_key_old.pem
shred -u /secrets/jwt_public_key_old.pem

# 4. 廃棄確認（ファイルが存在しないことを確認）
ls -la /secrets/jwt_*

# 5. 台帳更新（本章）と REC-NNN 更新
```

### 3-2. KEY-008 ローテーション手順

```bash
# 1. HMAC-SHA256 新鍵生成（256 bit）
openssl rand -hex 32 > /secrets/outbox_hmac_key_new.hex

# fingerprint 計算
sha256sum /secrets/outbox_hmac_key_new.hex | awk '{print $1}'

# 2. 旧 HMAC で署名済みのメッセージが全フラッシュされていることを確認
docker compose exec api ./outbox-verify --pending-count
# 保留件数 0 を確認してから新鍵へ切り替える

# 3. アプリケーション設定を新鍵に更新
# ENV-NNN の OUTBOX_HMAC_KEY 環境変数を更新

# 4. hash-chain-verify で整合性確認
docker compose exec api ./hash-chain-verify --full

# 5. 旧鍵廃棄（猶予期間なし）
shred -u /secrets/outbox_hmac_key_old.hex

# 6. 台帳更新と REC-NNN 更新
```

**本節で確定した方針**
- **KEY-001 ローテーションは猶予期間 7 日を設け、猶予期間中は新旧両鍵を有効とする。猶予期間なしの即時切り替えは JWT 認証エラーを引き起こすため禁止する。**
- **KEY-008 ローテーションは Outbox 保留件数 0 を確認してから切り替える。保留件数ありでの切り替えは HMAC 検証エラーを引き起こすため禁止する。**
- **全ローテーション完了後は次回期限を台帳の「次回期限」列に記録し、期限管理を継続する。**

---

## 4. 期限超過アラート対応

KEY-001（90 日）と KEY-008（90 日）は定期的に台帳を確認し、期限超過前に REC-NNN を起票する。

```bash
# 次回期限チェックスクリプト（月次確認として実行）
# 台帳から次回期限を読み取り、30 日以内の期限を出力
# （手動確認: 台帳の「次回期限」列を月次で目視確認する）
```

**本節で確定した方針**
- **月次定期作業（`../運用手順/03_月次定期作業手順.md`）に本台帳の確認を含める。**
- **次回期限まで 30 日以内になった鍵は P3 REC-NNN を起票してローテーション計画を確定する。**
- **期限超過は MNT-039 違反であり、発覚した時点で P2 に格上げして即時ローテーションを実施する。**
