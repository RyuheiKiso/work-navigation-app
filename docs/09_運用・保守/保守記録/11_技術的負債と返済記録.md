# 11 技術的負債と返済記録

本章は技術的負債の分類・起票フォーマット・優先度付け・四半期返済計画を確定する。OPS-PROC-007（監視設定変更手順）との連動・ADR-OPS-NNN / PROB-NNN との紐付けを実体化する。

---

## 1. 技術的負債の定義と分類

本システムにおける技術的負債とは、将来的な保守コスト・リスクを増大させる設計上・実装上・ドキュメント上の不完全さを指す。技術的負債は P4 問題として扱い、PROB-NNN 形式で起票する。

### 1-1. 5 分類

| 分類 ID | 分類名 | 定義 | 例 |
|---|---|---|---|
| TD-DESIGN | 設計負債 | 設計判断の妥協・将来拡張時に問題化する設計 | 単一テーブル継承で複雑化・DB 正規化不足 |
| TD-CODE | コード負債 | リファクタリング不足・複雑度超過・重複コード | 関数複雑度が閾値超過・マジックナンバー多用 |
| TD-TEST | テスト不足 | テストカバレッジ不足・E2E テスト欠如 | 重要ビジネスロジックの単体テスト欠如 |
| TD-DOC | ドキュメント負債 | コメント不足・手順書陳腐化・API 仕様未更新 | 実装と手順書の乖離・未ドキュメント API |
| TD-DEP | 依存負債 | EOL 接近・メジャーバージョン追従遅延 | Node 16 系の EOL 接近・Rails 5 から 7 への未移行 |

**本節で確定した方針**
- **技術的負債は 5 分類（TD-DESIGN / TD-CODE / TD-TEST / TD-DOC / TD-DEP）の 1 つに分類し、PROB-NNN 形式で起票する。**
- **技術的負債は P4 として起票し、四半期返済計画（§3）に組み込んで計画的に返済する。緊急性なしの技術的負債を優先度詐称して P1〜P3 で起票することは禁止する。**
- **セキュリティリスクを伴う技術的負債（TD-DEP の EOL 接近等）は P3 に格上げして対処する。**

## 2. 技術的負債の起票フォーマット

PROB-NNN 形式（02 章 §2-2）を基本とし、以下の追加フィールドを含む。

```yaml
prob_id: PROB-NNN
title: <技術的負債の短い説明>
category: functional_bug  # P4 の場合 design_gap も使用可
debt_category: TD-DESIGN | TD-CODE | TD-TEST | TD-DOC | TD-DEP
priority: P4  # 原則 P4（例外: セキュリティリスク有りは P3）
discovered_at: YYYY-MM-DD
discovered_by: system_admin
status: open | scheduled | in_progress | resolved

# 返済 ROI 評価（§2-1 参照）
risk_reduction_score: 1 | 2 | 3 | 4 | 5  # 5 が最大リスク軽減
implementation_cost_score: 1 | 2 | 3 | 4 | 5  # 1 が最小コスト
roi: <risk_reduction_score / implementation_cost_score>  # 高いほど優先

# ADR 連動（設計判断が必要な場合）
related_adr: ADR-OPS-NNN  # 任意

# 四半期計画への組み込み
scheduled_quarter: YYYY-Q1 | YYYY-Q2 | YYYY-Q3 | YYYY-Q4
```

### 2-1. 返済 ROI の計算方法

返済 ROI は以下の式で算出し、四半期返済計画の優先順位付けに使用する。

```
返済 ROI = リスク軽減効果スコア / 実装コストスコア

リスク軽減効果スコア（1〜5）:
  5: 本番停止・データ損失・規制違反リスクを直接軽減
  4: 性能劣化・セキュリティリスクを軽減
  3: 保守工数削減・バグ発生率低下が見込まれる
  2: コードの可読性・理解性の向上のみ
  1: 美観の改善のみ

実装コストスコア（1〜5）:
  1: 1 時間以内で対応可能
  2: 半日程度（4 時間）
  3: 1 日程度（8 時間）
  4: 2〜3 日
  5: 1 週間以上
```

ROI ≥ 2.0 の技術的負債を次の四半期返済計画に優先組み込む。ROI < 1.0 の技術的負債は ADR-OPS-NNN で「受け入れ」判断を記録してから backlog として保持する。

**本節で確定した方針**
- **返済 ROI = リスク軽減効果 / 実装コストで算出し、ROI ≥ 2.0 を次四半期計画の組み込み基準とする。**
- **ROI < 1.0 の技術的負債は「受け入れ」判断を ADR-OPS-NNN に記録してから backlog 保持とする（黙示的受け入れを禁止する）。**
- **TD-DEP（依存負債）は EOL 期限が 6 ヶ月以内に到達した時点で P3 に格上げして対処を開始する。**

## 3. 四半期返済計画

### 3-1. 計画サイクル

| タイミング | 実施内容 |
|---|---|
| 四半期最終月 15 日まで | 翌四半期の返済計画を策定し、REC-NNN（P4）を起票 |
| 四半期末 | 返済実績を本台帳に記録 |
| 年次評価時（12 章）| 年間返済実績を評価・翌年度計画へ反映 |

### 3-2. 四半期返済計画テンプレート

```yaml
# 四半期返済計画: YYYY-QN
plan_quarter: YYYY-Q1 | YYYY-Q2 | YYYY-Q3 | YYYY-Q4
plan_created_at: YYYY-MM-DD
planned_items:
  - prob_id: PROB-NNN
    debt_category: TD-DESIGN | TD-CODE | TD-TEST | TD-DOC | TD-DEP
    roi: X.X
    estimated_cost_days: N
    rec_id: REC-NNN

total_estimated_days: N
capacity_available_days: N  # system_admin の当期保守可能工数（日数）

# 計画完了記録
actual_completed:
  - prob_id: PROB-NNN
    completed_at: YYYY-MM-DD
    actual_cost_days: N
unplanned_additions:
  - prob_id: PROB-NNN
    reason: <追加理由>
```

**本節で確定した方針**
- **四半期返済計画は翌四半期開始前に策定し、strategy_capacity（system_admin の保守可能工数）を超えた計画を立てない。**
- **計画外の技術的負債返済は REC-NNN に `unplanned_addition` として記録し、四半期末のレビューで次期計画に反映する。**
- **年次評価（12 章）で過去 4 四半期の返済実績を評価し、返済方針の改善判断に活用する。**

## 4. 技術的負債台帳

| PROB-NNN | 分類 | タイトル | ROI | 優先度 | 状態 | 予定 Q | 完了日 |
|---|---|---|---|---|---|---|---|
| - | - | - | - | P4 | open | - | - |

### 4-1. 既知技術的負債（初期記録）

初期リリース（ver 1.0.0）時点での既知技術的負債は以下とする。各 PROB-NNN は `06_実装/17_実装問題管理ログ.md` に登録してから本台帳に転記する。

| 負債概要 | 分類 | 備考 |
|---|---|---|
| 単体テストカバレッジが不十分な箇所 | TD-TEST | 実装時に TBD とした箇所 |
| API ドキュメントの自動生成未対応 | TD-DOC | 手動更新のコスト |
| React Native のメジャーバージョン固定 | TD-DEP | 0.74 → 0.75+ の移行 |

これらは ver 1.0.0 リリース後の最初の四半期計画に組み込む。

**本節で確定した方針**
- **技術的負債台帳は本章で一元管理し、PROB-NNN との紐付けで全負債を追跡する。**
- **台帳から削除（解決なしの除去）は ADR-OPS-NNN で根拠を記録した場合のみ許可する。黙示的な負債の消去は禁止する。**
- **四半期末の返済実績は本台帳の状態欄を `resolved` に更新し、完了日を記録する。**
