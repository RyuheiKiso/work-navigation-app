# 05 本番移行（IPA 4.2.2.e）

IPA 共通フレーム 2013「**4.2.2.e 移行**」に準拠した手順を確定する。本番移行の GO/NOGO 判定基準・移行後確認手順・夜間メンテナンス窓・ロールバック判断基準を実体化する。

---

## 1. 本章の範囲と `06_実装/10_デプロイ手順.md` との分担

本章と `06_実装/10_デプロイ手順.md` は以下のように責務を分担する。

| 責務 | 担当ファイル |
|---|---|
| デプロイコマンドの実行手順 | `../../06_実装/10_デプロイ手順.md`（権威）|
| 本番移行の GO/NOGO 判定基準 | **本章 §2**（権威）|
| 移行後の確認手順 | **本章 §3**（権威）|
| 移行実施の記録 | **本章 §5**（REC-NNN）|
| ロールバック判断基準 | **本章 §4**（権威）|

本章は「移行を実施してよいか」「移行後に正常か」「ロールバックすべきか」の判定基準を確定する。実際のデプロイコマンドは `06_実装/10_デプロイ手順.md` に委任する。

**本節で確定した方針**
- **本章は移行判断基準と記録を権威化し、デプロイ実行手順は 06_実装に委任する。**
- **REC-NNN の `ready_for_deployment: true`（04 章確認済み）が本番移行開始の唯一の条件とする。**
- **本章の準拠宣言: 4.2.2.e に準拠する。**

## 2. 本番移行の GO/NOGO 判定基準

以下の全条件が満たされた場合のみ GO（本番移行実施）とする。1 つでも未充足の場合は NOGO（移行中止）とする。

| # | GO 条件 | 確認方法 | 担当 |
|---|---|---|---|
| GO-1 | STG 5 項目全 PASS（03 章 §4）| REC-NNN の stg_5item_result 確認 | system_admin |
| GO-2 | 受入チェックリスト全 PASS（04 章 §3）| REC-NNN の ready_for_deployment: true 確認 | system_admin |
| GO-3 | ロールバック手順が確認済み（§4 確認）| `../../06_実装/10_デプロイ手順.md` の rollback 節確認 | system_admin |
| GO-4 | 本番 DB バックアップ取得済み（MNT-037）| `pg_basebackup` 完了確認 / `../運用手順/05` 参照 | system_admin |
| GO-5 | メンテナンス窓内である（§2-1）| 現在時刻確認 | system_admin |

### 2-1. 夜間メンテナンス窓

```
推奨メンテナンス窓: 毎週日曜 02:00-04:00（JST）
許容メンテナンス窓: 毎日 00:00-05:00（JST）
禁止時間帯:          営業時間 06:00-22:00（NFR-OPS-001 影響回避）
```

P1 緊急対応は上記窓外でも実施可能とするが、実施記録（REC-NNN）に実施時刻・緊急理由を明記する。

### 2-2. NOGO 時の対応

NOGO と判定した場合は以下を実施する。

```bash
# NOGO 記録
# REC-NNN に nogo_at・nogo_reason を記録
# 不充足条件を特定して 03 章または 04 章へ差し戻し
```

**本節で確定した方針**
- **GO-1〜GO-5 の全条件を満たした場合のみ本番移行を開始する。NOGO 理由は REC-NNN に必ず記録する（ALCOA+ Attributable 原則）。**
- **推奨メンテナンス窓（日曜 02:00-04:00）内での移行を原則とし、窓外移行は P1 緊急対応時のみ許可する。**
- **本番 DB バックアップ（GO-4）は移行実施の直前（移行開始 1 時間以内）に取得したものを有効とする。**

## 3. 移行後の確認手順

本番移行完了後、以下の確認を実施する。全確認は移行完了後 30 分以内に完了させる。

### 3-1. 5 分間監視（移行直後）

```bash
# エラーレートの確認（5 分間監視）
# Prometheus / Grafana ダッシュボードでエラー率を監視
# 閾値: エラー率 < 1%（§4-1 ロールバック基準参照）

# ヘルスチェック確認
curl -f http://prod-host/health
curl -f http://prod-host/api/v1/health

# 応答時間確認
curl -w "@/tmp/curl-format.txt" -s http://prod-host/api/v1/health
```

### 3-2. ハッシュチェーン確認

```bash
# 本番環境でのハッシュチェーン整合性確認
docker compose -f docker-compose.prod.yml exec api ./hash-chain-verify --recent 1000
# ERROR が出力された場合は即時ロールバック（§4-2）
```

### 3-3. Outbox フラッシュ確認

```bash
# Outbox 保留件数確認（ハンディ → バックエンドの同期バックログ）
docker compose -f docker-compose.prod.yml exec api ./outbox-verify --pending-count
# 保留件数が急増している場合はロールバック判断基準（§4-1）を適用
```

### 3-4. DB マイグレーション確認

```bash
# 本番 DB のマイグレーション適用状況確認
psql $WNAV_PROD_DB_URL -c "SELECT version_num, applied_at FROM schema_migrations ORDER BY applied_at DESC LIMIT 5;"
```

**本節で確定した方針**
- **移行後確認（§3-1〜§3-4）は移行完了後 30 分以内に全て完了させる。30 分以内に完了できない場合はロールバック（§4）を実施する。**
- **移行後確認の結果は REC-NNN の `deployment_confirmed_at`・`deployment_confirmed_by` に記録する（ALCOA+ Contemporaneous 原則）。**
- **ハッシュチェーン確認で ERROR が検出された場合は確認結果の記録より前に即時ロールバックを開始する。**

## 4. ロールバック判断基準

### 4-1. 自動ロールバック発動基準

以下のいずれかの条件を満たした場合は移行後 30 分以内であれば即時ロールバックを実施する。

| 条件 | 閾値 | 確認方法 |
|---|---|---|
| エラーレート急増 | 移行後 5 分間のエラー率 ≥ 5% | Prometheus metrics |
| ハッシュチェーン破断 | hash-chain-verify ERROR 1 件以上 | §3-2 コマンド |
| ヘルスチェック失敗 | /health が HTTP 200 以外を返す | §3-1 コマンド |
| Outbox 保留急増 | 移行前比 10 倍以上の保留件数 | §3-3 コマンド |
| DB 接続不能 | psql 接続タイムアウト 3 回以上 | §3-4 コマンド |

### 4-2. ロールバック実施手順

```bash
# ロールバック発動を記録
# REC-NNN に rollback_triggered_at・rollback_reason を記録

# ロールバックコマンド（実体は 06_実装/10_デプロイ手順.md の rollback 節に記載）
# ../../../06_実装/10_デプロイ手順.md §4（ロールバック手順）を参照

# DB マイグレーションのロールバック
sqlx migrate revert --database-url $WNAV_PROD_DB_URL

# ロールバック完了確認
curl -f http://prod-host/health
docker compose -f docker-compose.prod.yml exec api ./hash-chain-verify --recent 100
```

### 4-3. ロールバック後の対応

ロールバック完了後は以下を実施する。

1. REC-NNN の status を `rollback_completed` に更新
2. PROB-NNN を再 open にして再分析を実施（02 章へ差し戻し）
3. ロールバック理由を INC-YYYY-NNN として起票（障害対応サブへ連携）

**本節で確定した方針**
- **移行後 30 分以内にエラー率 ≥ 5%・ハッシュチェーン破断・ヘルスチェック失敗のいずれかを検出した場合は即時ロールバックを実施する。ロールバック判断は system_admin 単独で実施可能とする。**
- **ロールバック後は必ず PROB-NNN を再 open して根本原因の再分析から保守プロセスをやり直す。**
- **ロールバック発動・完了の記録は ALCOA+ Contemporaneous 原則に基づき発動・完了の各タイミングで即時記録する。**

## 5. 移行記録の完了

本番移行（またはロールバック）が完了したら REC-NNN を以下のとおり更新して保守プロセスを完結させる。

```yaml
# 正常移行完了の場合
deployment_at: YYYY-MM-DD HH:MM
deployed_by: system_admin
deployment_confirmed_at: YYYY-MM-DD HH:MM
post_deploy_5min_ok: true
hash_chain_ok: true
outbox_flush_ok: true
status: completed
completed_at: YYYY-MM-DD

# ロールバック完了の場合
deployment_at: YYYY-MM-DD HH:MM
rollback_triggered_at: YYYY-MM-DD HH:MM
rollback_reason: <理由>
rollback_completed_at: YYYY-MM-DD HH:MM
status: rollback_completed
```

**本節で確定した方針**
- **REC-NNN の status を `completed` または `rollback_completed` に更新して保守プロセスを完結させる。status が open / in_progress のまま保守プロセスを放置することは禁止する。**
- **移行記録の完了は移行実施当日中に完了させる（ALCOA+ Contemporaneous 原則）。**
- **ロールバック完了の REC-NNN は closed 化せず `rollback_completed` ステータスのまま保持し、再修正の REC-NNN から `related_rec` で参照する。**
