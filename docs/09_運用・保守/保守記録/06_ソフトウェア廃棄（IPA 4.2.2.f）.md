# 06 ソフトウェア廃棄（IPA 4.2.2.f）

IPA 共通フレーム 2013「**4.2.2.f ソフトウェア廃棄**」に準拠した手順を確定する。バージョン廃止の判断基準・Docker イメージアーカイブ方針・データ 7 年保全手順・アーカイブ手順・廃棄記録・ハンディ端末廃棄を実体化する。

---

## 1. 本章の範囲

IPA 4.2.2.f が要求する「ソフトウェア廃棄」は以下の項目を含む。

| IPA 4.2.2.f 要求項目 | 本章担当節 |
|---|---|
| 廃棄の判断基準 | §2 |
| データのアーカイブ・保全 | §4 |
| ソフトウェアイメージの廃棄 | §3 |
| 廃棄記録 | §5 |
| ハードウェア廃棄（端末）| §6 |

**本節で確定した方針**
- **廃棄の判断は人が行い、自動廃棄は実施しない（ADR-OPS-NNN に根拠記録）。**
- **ALCOA+ Enduring 原則に基づき、データアーカイブは 7 年間保全する（NFR-DQ-008・MNT-041）。**
- **本章の準拠宣言: 4.2.2.f に準拠する。**

## 2. バージョン廃止の判断基準

### 2-1. 廃止トリガー

以下のいずれかを満たした場合にバージョン廃止を判断する。廃止判断は system_admin が起票し quality_admin が承認する（§5 参照）。

| トリガー | 定義 | 確認方法 |
|---|---|---|
| セキュリティサポート終了 | 使用中の OS / ランタイム / フレームワークのセキュリティパッチ供給が停止 | ベンダー EOL 発表の確認 |
| 依存 EOL | 依存ライブラリの最終リリースから 12 ヶ月経過かつ CVE 未修正 | `cargo audit` / `pnpm audit` |
| 後継バージョン安定化 | 新メジャーバージョンが 90 日以上本番稼働し安定していること | バージョン履歴確認 |

### 2-2. 廃止を決定しないケース

以下の条件では廃止トリガーが発生しても廃止を即時決定しない。

- 後継バージョンへの移行が未完了（移行完了を待つ）
- ALCOA+ 依存データの移行計画が未策定（計画策定を先行させる）

**本節で確定した方針**
- **廃止判断は §2-1 のトリガーに基づき人が実施し、自動廃棄は対象外と判断する。**
- **廃止トリガー発生から廃止決定まで最低 30 日の評価期間を設ける。評価期間中は ADR-OPS-NNN を起票して廃止評価中であることを記録する。**
- **ALCOA+ 依存データの移行計画なしでのバージョン廃止は禁止する。**

## 3. 旧バージョン Docker イメージのアーカイブ方針

### 3-1. ghcr.io 保持ポリシー

| イメージ種別 | 保持期間 | 廃棄基準 |
|---|---|---|
| リリースタグ付きイメージ（v*.*.*）| 30 日（廃止決定後）| 廃棄記録（REC-NNN）完了後に手動削除 |
| latest タグ | 常時最新 1 件のみ | 新リリース時に自動上書き |
| dev / staging タグ | 7 日 | 自動廃棄（GitHub Actions の retention 設定）|

### 3-2. イメージ廃棄手順

```bash
# 廃棄対象イメージの確認
gh api /user/packages/container/work-navigation-api/versions \
  --jq '.[] | {id: .id, tag: .metadata.container.tags[], created_at: .created_at}'

# 廃棄実施（タグ指定）
gh api -X DELETE /user/packages/container/work-navigation-api/versions/<version_id>

# 廃棄後の確認
gh api /user/packages/container/work-navigation-api/versions | jq length
```

**本節で確定した方針**
- **リリースタグ付きイメージは廃止決定後 30 日間保持し、廃棄記録（REC-NNN）完了後に手動削除する。自動削除は対象外と判断する。**
- **イメージ廃棄は廃棄記録（§5）の起票後に実施し、廃棄実施日時・廃棄したイメージタグ一覧を REC-NNN に記録する。**
- **dev / staging タグの自動廃棄は GitHub Actions の retention 設定（7 日）に委任し、廃棄記録の作成は対象外と判断する。**

## 4. データアーカイブ（7 年保全）

ALCOA+ Enduring 原則および NFR-DQ-008・MNT-041 に基づき、製造記録・監査証跡データは 7 年間保全する。21 CFR Part 11 に準拠する。

### 4-1. アーカイブ対象データ

| テーブル / データ種別 | 保全期間 | 根拠 |
|---|---|---|
| trace_events（監査証跡）| 7 年 | 21 CFR Part 11・NFR-DQ-008 |
| work_records（作業記録）| 7 年 | 製造業規制・NFR-DQ-008 |
| esign_records（電子サイン記録）| 7 年 | 21 CFR Part 11 §11.10(e) |
| SOP 改訂履歴 | 7 年 | NFR-DQ-008 |
| 保守記録（REC-NNN）| 7 年 | 内部規定・MNT-041 |

### 4-2. アーカイブ手順

```bash
# 1. アーカイブ対象パーティションの DETACH
# （PostgreSQL パーティショニング：TBL-audit_chain 等はパーティション方式前提）
psql $WNAV_PROD_DB_URL -c "
ALTER TABLE trace_events DETACH PARTITION trace_events_YYYY_archive;
"

# 2. DETACH したパーティションを pg_dump
pg_dump $WNAV_PROD_DB_URL \
  --table=trace_events_YYYY_archive \
  --format=custom \
  --file=/archive/trace_events_YYYY_archive.dump

# 3. AES-256-GCM 暗号化（KEY-004 使用）
openssl enc -aes-256-gcm \
  -in /archive/trace_events_YYYY_archive.dump \
  -out /archive/trace_events_YYYY_archive.dump.enc \
  -pass file:/secrets/backup_key_004.key

# 4. WORM NAS / LTO テープへの転送
# 転送先: 工場 WORM NAS（Write Once Read Many ストレージ）
rsync -avz /archive/trace_events_YYYY_archive.dump.enc worm-nas:/7year-archive/

# 5. アーカイブ完了確認（SHA-256 ハッシュ照合）
sha256sum /archive/trace_events_YYYY_archive.dump.enc
sha256sum worm-nas:/7year-archive/trace_events_YYYY_archive.dump.enc
# 両ハッシュが一致することを確認

# 6. アーカイブ記録
# REC-NNN に archive_completed_at・archive_hash・archive_destination を記録
```

### 4-3. アーカイブ保全の確認

```bash
# 年次アーカイブ整合性確認（読み出し可能であることの確認）
openssl enc -aes-256-gcm -d \
  -in worm-nas:/7year-archive/trace_events_YYYY_archive.dump.enc \
  -out /tmp/verify_archive.dump \
  -pass file:/secrets/backup_key_004.key

pg_restore --list /tmp/verify_archive.dump | head -20
```

**本節で確定した方針**
- **アーカイブデータは AES-256-GCM（KEY-004）で暗号化してから WORM NAS / LTO に転送し、SHA-256 ハッシュで転送完全性を確認する（ALCOA+ Accurate 原則）。**
- **7 年保全期間中の削除は禁止する。7 年経過後の廃棄は quality_admin 承認を経た廃棄記録（REC-NNN 種別: disposal）を作成してから実施する。**
- **年次アーカイブ整合性確認を `12_運用評価レポートテンプレ.md` の評価項目に含める。**

## 5. 廃棄記録（REC-NNN 種別: disposal）

廃棄実施時は `07_保守記録テンプレ（REC-NNN）.md` に準拠して廃棄記録を起票する。

```yaml
record_type: disposal
disposal_target: <廃棄対象の識別（バージョン番号・イメージタグ・テーブル名等）>
disposal_reason: eol | security_eol | dependency_eol | data_archive_expiry
approved_by: quality_admin  # 廃棄は quality_admin 承認必須
approved_at: YYYY-MM-DD
disposal_method: image_delete | db_partition_archive | hardware_wipe
disposal_confirmed_at: YYYY-MM-DD
disposal_confirmed_by: system_admin
```

廃棄記録自体は 7 年間保全する（廃棄した事実の証拠として）。

**本節で確定した方針**
- **全廃棄作業は REC-NNN（record_type: disposal）を起票してから実施する。廃棄記録なしの廃棄作業は ALCOA+ 違反とする。**
- **廃棄には quality_admin の承認が必須であり、system_admin 単独での廃棄判断は禁止する。**
- **廃棄記録は廃棄対象が存在しなくなった後も 7 年間保全する（廃棄の証拠として）。**

## 6. ハンディ端末廃棄

ハンディ端末（Android / iOS / Windows）の廃棄時は以下の手順を踏む。

### 6-1. SQLCipher 鍵廃棄

```bash
# 端末ローカル AES-256 鍵（KEY-003）の廃棄
# 手順: 端末管理アプリ経由で KEY-003 を鍵削除
# 鍵削除により SQLite DB は読取不能状態になることを確認
# 確認後: 09_鍵ローテーション履歴テンプレ.md の KEY-003 行に廃棄記録を記入
```

### 6-2. 工場リセット

SQLCipher 鍵廃棄後、端末の工場リセットを実施する。工場リセット後にデータが読み取れないことを確認する（ランダムデータで上書きできない場合は物理破壊を実施する）。

### 6-3. 廃棄記録保管

```yaml
# ハンディ端末廃棄記録（REC-NNN 種別: disposal）の追記項目
disposal_method: hardware_wipe
device_id: <端末 ID>
device_model: <機種名>
key_003_revoked_at: YYYY-MM-DD
factory_reset_at: YYYY-MM-DD
data_unreadable_confirmed: true
```

**本節で確定した方針**
- **ハンディ端末廃棄は KEY-003 鍵廃棄 → 工場リセット → 読取不能確認の 3 ステップを必ず順序通りに実施する（MNT-042）。**
- **工場リセット後のデータ読取不能確認は廃棄記録に `data_unreadable_confirmed: true` として記録してから廃棄完了とする。**
- **廃棄記録（REC-NNN）は端末廃棄後 7 年間保全する（廃棄の証拠として）。**
