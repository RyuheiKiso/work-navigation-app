# 12 運用評価レポートテンプレ（IPA 4.2.1.e）

IPA 共通フレーム 2013「**4.2.1.e 運用評価**」に準拠した年次運用評価の実施手順・評価テンプレートを確定する。MNT-043・OPS-PROC-008 との連動・改善計画策定フローを実体化する。

---

## 1. 本章の範囲

IPA 4.2.1.e が要求する「運用評価」は以下の項目を含む。

| IPA 4.2.1.e 要求項目 | 本章担当節 |
|---|---|
| 運用実績の測定・評価 | §2 |
| SLO 達成状況の評価 | §3 |
| 改善計画の策定 | §4 |
| 評価結果の記録 | §5 |

年次運用評価は MNT-043 に基づき毎年 1 回（年度末）実施する。評価実施の証跡は REC-NNN（record_type: capacity_extension または sop_revision）として記録する。

**本節で確定した方針**
- **年次運用評価は MNT-043 に基づき毎年度末に実施し、実施しない年度は ADR-OPS-NNN に不実施理由を記録する（黙示的省略を禁止する）。**
- **評価結果に基づく改善計画は ADR-OPS-NNN に記録し、次期保守計画に反映する。**
- **本章の準拠宣言: 4.2.1.e に準拠する。**

---

## 2. 年次運用評価の実施手順

### 2-1. 実施スケジュール

| タイミング | 実施内容 |
|---|---|
| 年度末 -30 日 | 評価データの収集開始（§3 SLO サマリー作成）|
| 年度末 -15 日 | 保守実績・技術的負債台帳の整理（§3-5・§3-6 作成）|
| 年度末 -7 日 | 評価レポートのドラフト作成（本テンプレートに記入）|
| 年度末 | quality_admin と評価レポートのレビュー実施 |
| 年度末 +7 日 | 評価レポートを確定版として REC-NNN に記録 |
| 年度末 +14 日 | 翌年度保守計画に改善計画を反映 |

### 2-2. 評価データ収集コマンド

```bash
# 年間 SLO 達成率の収集
# Prometheus クエリ例（稼働率の年間平均）
curl -s "http://prometheus:9090/api/v1/query_range" \
  --data-urlencode 'query=avg_over_time(up[365d])' \
  --data-urlencode "start=$(date -d '1 year ago' +%s)" \
  --data-urlencode "end=$(date +%s)" | \
  jq '.data.result[0].values[-1][1]'

# 月次 SLO レポートの収集
# ../運用手順/04_月次SLOレポート手順.md の 12 ヶ月分を集計

# 年間インシデント件数の確認
grep -c "^| INC-$(date +%Y)" \
  docs/09_運用・保守/障害対応/12_過去障害記録索引\（INC-NNN\）.md

# 保守記録件数の確認
find docs/09_運用・保守/保守記録/ -name "REC-*.md" | wc -l
```

**本節で確定した方針**
- **評価データは自動収集コマンドと手動台帳の両方から収集し、評価の根拠を明示する（ALCOA+ Accurate 原則）。**
- **評価データの収集は年度末 30 日前から開始し、データ不足の場合は ADR-OPS-NNN に補完方法を記録してから評価を実施する。**
- **評価レポートの確定版は REC-NNN に添付して 7 年間保全する（ALCOA+ Enduring 原則）。**

### 3-6. アーカイブ・保全確認

| 確認項目 | 結果 | 確認日 |
|---|---|---|
| 7 年保全データの整合性確認（06 章 §4-3）| PASS / FAIL | YYYY-MM-DD |
| 鍵台帳の全鍵が有効期限内 | PASS / FAIL | YYYY-MM-DD |
| 依存 EOL チェック（EOL 6 ヶ月以内の依存なし）| PASS / FAIL | YYYY-MM-DD |
| 引継ぎ文書の最終更新確認 | PASS / FAIL | YYYY-MM-DD |

**本節で確定した方針**
- **SLO 年次サマリーで FAIL となった SLI は改善計画（§4）の必須項目とし、ADR-OPS-NNN を起票する。**
- **ユーザー満足度の代替指標は毎年同一方法で測定し、年間比較を可能にする（ALCOA+ Consistent 原則）。**
- **アーカイブ・保全確認で FAIL の項目は翌年度の第 1 四半期中に解消し、解消記録を REC-NNN に残す。**

## 4. 改善計画策定フロー

```
年次評価（本章）
  → FAIL 項目・改善機会を特定
  → ADR-OPS-NNN を起票（改善方針の設計判断を記録）
  → 翌年度保守計画に組み込み（具体的な実施 REC-NNN を起票）
  → 次年度の年次評価で改善効果を測定
```

### 4-1. 改善計画テンプレート

```yaml
# 改善計画（ADR-OPS-NNN から参照）
improvement_id: ADR-OPS-NNN
evaluation_year: YYYY
trigger: <改善のきっかけ（SLO FAIL / インシデント傾向 / 技術的負債蓄積 等）>
current_state: <現状の説明>
target_state: <改善後の目標状態>
action_items:
  - rec_id: REC-NNN  # 具体的な実施保守記録
    description: <実施内容>
    due_quarter: YYYY-QN
expected_slo_impact: <SLO への期待効果>
```

**本節で確定した方針**
- **改善計画は ADR-OPS-NNN に記録してから翌年度保守計画に組み込む。改善計画なしの「評価のみ実施」は不完全な評価として受理しない。**
- **改善計画の実施状況は翌年度の年次評価で必ず追跡し、未実施の改善計画は継続理由を ADR-OPS-NNN に追記する。**
- **年次評価と改善計画のセットを 1 つの評価サイクルとし、評価のみ・計画のみの片方での完結を禁止する。**

## 5. 年次評価レポートの記録

年次評価完了後、以下の形式で REC-NNN に記録する。

```yaml
# REC-NNN 年次評価記録（07 章テンプレートに追加フィールド）
record_type: capacity_extension  # 年次評価は capacity_extension として記録
evaluation_year: YYYY
evaluation_period: YYYY-01-01 / YYYY-12-31
evaluator: system_admin
reviewer: quality_admin
slo_annual_pass_count: N  # PASS SLI 件数
slo_annual_fail_count: N  # FAIL SLI 件数
incident_count_annual: N
maintenance_cost_annual_days: N
improvement_adrs_raised: N  # 起票した ADR-OPS-NNN 件数
```

**本節で確定した方針**
- **年次評価レポートは quality_admin のレビューを経てから確定版とし、確定後は REC-NNN に添付して 7 年間保全する（ALCOA+ Enduring 原則）。**
- **年次評価の実施は MNT-043 の義務であり、不実施の場合は ADR-OPS-NNN に不実施の根拠を記録してから次年度繰り越しの承認を quality_admin に求める。**
- **全評価データの測定方法は毎年変更せず一定とし、年間比較の有効性を維持する（ALCOA+ Consistent 原則）。**
