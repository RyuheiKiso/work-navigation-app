# 99 前提制約と保守記録サブが約束しないこと

本章は保守記録サブのスコープ境界・対象外宣言・前提制約を確定する権威文書である。本章に記載された対象外宣言は、将来の改訂時にも保持する。対象外宣言の取り消しは CR-OPS-NNN と ADR-OPS-NNN を起票した上でのみ実施できる。

---

## 1. 前提制約

本保守記録サブは以下の前提条件が満たされた環境でのみ有効である。前提条件が変化した場合は CR-OPS-NNN を起票して該当章を改訂する。

### 1-1. 体制前提

| 前提 | 内容 | 変化時の対応 |
|---|---|---|
| 担当者数 | system_admin 1 名が全保守作業を実施 | CR-OPS-NNN で体制章（01 §2）を改訂 |
| quality_admin 存在 | ALCOA+ 承認者として quality_admin が存在する | quality_admin 不在時の代替手順を ADR-OPS-NNN で確定 |
| 個人開発スコープ | 単一組織内での運用を前提とする | 多組織・マルチテナント要件は上流設計変更が必要 |

### 1-2. インフラ前提

| 前提 | 内容 |
|---|---|
| 単一サイト | Windows Server 2022 + IIS + WSL2 の単一サイト構成 |
| クローズド LAN | 工場内クローズド LAN（外部 SaaS・クラウド接続なし）|
| PostgreSQL 16+ | PostgreSQL 16 以上（WAL / パーティション機能前提）|
| Docker + compose | WSL2 上の Docker compose 環境が利用可能 |

### 1-3. 規制前提

| 前提 | 内容 |
|---|---|
| 21 CFR Part 11 対象 | 電子記録・電子署名規則の適用を前提とする |
| ALCOA+ 義務 | 全保守記録が ALCOA+ 原則に従う義務を負う |
| 7 年保全義務 | 製造記録・監査証跡・保守記録の 7 年保全義務を負う |

**本節で確定した方針**
- **前提制約の変化は CR-OPS-NNN を起票して影響範囲を評価してから対応する。前提変化を無視した保守作業継続は禁止する。**
- **インフラ前提（単一サイト・クローズド LAN）の変更は上流フェーズ（04_概要設計）の設計変更が必要であり、本章単独での対応は対象外と判断する。**
- **規制前提（21 CFR Part 11 / ALCOA+）の変更は quality_admin との合意と上流文書改訂を必要とする。**

## 2. 保守記録サブが約束しないこと

以下の項目は本保守記録サブのスコープ外とする。対象外と判断する根拠を各項目に明示する。

### 2-1. 第三者保守ベンダーの受入

**対象外と判断する。**

根拠: 本システムは個人開発スコープであり、第三者ベンダーへの保守委託を前提としない。ベンダー受入を要する組織規模には達していない。第三者ベンダーを受け入れる必要が生じた場合は ADR-OPS-NNN を起票して体制設計から再実施する。

### 2-2. SLA 補償（金銭・サービスクレジット）

**対象外と判断する。**

根拠: 本システムは個人開発・自社内運用であり、SLA 補償の対象となる外部顧客は存在しない。SLO（Service Level Objective）は運用品質の指標として定義するが、補償義務を伴う SLA（Service Level Agreement）は本システムのスコープ外である。

### 2-3. 自動退役・自動廃棄

**対象外と判断する。**

根拠: ALCOA+ 原則および 21 CFR Part 11 への準拠を維持するため、データの削除・廃棄は人が判断して実施する。自動廃棄スクリプトの実装は廃棄の監査証跡を損なうリスクがある。廃棄手順は 06 章で人手による判断プロセスとして確定する。

### 2-4. 24h サポート窓口

**対象外と判断する。**

根拠: system_admin 1 名体制のため、24 時間有人サポートは提供しない。P1 障害への初動は夜間・休日でも system_admin がベストエフォートで対応するが、NFR-OPS-004（初動 15 分以内）は営業時間（06:00-22:00）のみを対象とする。夜間・休日の対応方針は `../03_運用体制と役割実装.md` §4 に記載する。

### 2-5. 地理冗長 DR（ディザスタリカバリ）

**対象外と判断する。**

根拠: 単一工場・クローズド LAN 制約のため、地理冗長 DR は実装しない。DR 要件が生じた場合は上流設計変更（ADR-OPS-NNN + CR-OPS-NNN で上流改訂依頼）を必要とする。

### 2-6. AI / ML による自動障害予測・自動保守判断

**対象外と判断する。**

根拠: ver 1.0.0 フェーズでは実装しない（CLAUDE.md ポリシーに基づく）。AI / ML の活用方針が確定した場合は ADR-OPS-NNN で保守への組み込み手順を確定してから適用する。

### 2-7. テスト仕様書・品質保証記録の作成

**対象外と判断する。**

根拠: テスト仕様書・品質保証記録は `07_テスト/` の責務である。本保守記録サブはテスト実行結果の確認（STG 5 項目など）を行うが、テスト仕様書の作成・維持は行わない。

**本節で確定した方針**
- **本節に列挙した 7 項目は対象外と判断し、本保守記録サブではいかなる手順・テンプレートも提供しない。**
- **対象外項目の扱いを変更する場合は CR-OPS-NNN と ADR-OPS-NNN の両方を起票し、quality_admin の承認を得てから対象外宣言を取り消す。**
- **外部からの「〇〇はどこに書いてあるか」という問いに対して、本章の対象外宣言を回答の根拠として使用できる。**

## 3. 自己監査チェックリスト

保守記録サブ全体の自己監査に使用する。四半期定期作業時に確認する。

### 3-1. 文書構造

- [ ] 01〜12 の全章に「本節で確定した方針」が H2 末尾にある
- [ ] 全章末に「参照業界分析」節（必須/関連 二区分）がある
- [ ] H1 が `# NN タイトル` 形式である

### 3-2. 規約準拠

- [ ] 曖昧表現（考慮する / 検討する / 努める / 目指す / 可能性がある / 場合がある）が使われていない
- [ ] 規制表記が三段階（準拠する / 対応する / 対象外と判断する）のみ使用されている
- [ ] ASCII アートによる図の代替がない

### 3-3. 識別子整合

- [ ] REC-NNN・CR-OPS-NNN・ADR-OPS-NNN 全件が付録/99 台帳に登録されている
- [ ] PROB-NNN が 06_実装/17_実装問題管理ログ.md に登録されている
- [ ] MNT-036〜043・NFR-OPS-040〜042 の継承識別子を再採番していない

### 3-4. ALCOA+ 整合

- [ ] 全保守記録の起票日が作業開始当日以降である（Contemporaneous 原則）
- [ ] 全保守記録に executed_by が記録されている（Attributable 原則）
- [ ] 廃棄済みデータの台帳エントリが削除されていない（Enduring 原則: 廃棄後も記録を保持）
