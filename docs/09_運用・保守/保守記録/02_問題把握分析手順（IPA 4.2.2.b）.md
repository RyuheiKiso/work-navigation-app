# 02 問題把握分析手順（IPA 4.2.2.b）

IPA 共通フレーム 2013「**4.2.2.b 問題把握及び修正分析**」に準拠した手順を確定する。PROB-NNN 起票手順・5 分類定義・ALCOA+ 影響分析チェックリスト・優先度別修正 SLO・CR-OPS-NNN への紐付けを実体化する。

---

## 1. 本章の範囲

IPA 4.2.2.b が要求する「問題把握及び修正分析」は以下の項目を含む。

| IPA 4.2.2.b 要求項目 | 本章担当節 |
|---|---|
| 問題の識別・記録 | §2 |
| 問題の分類 | §3 |
| 修正の必要性分析・影響分析 | §4 |
| 優先度付け | §5 |
| 変更要求への紐付け | §6 |

**本節で確定した方針**
- **IPA 4.2.2.b の全要求項目は本章の §2〜§6 で対応する。**
- **問題の識別から CR-OPS-NNN 起票までが本章の責務であり、修正実施は 03 章の責務とする。**
- **本章の準拠宣言: 4.2.2.b に準拠する。**

---

## 2. PROB-NNN 起票手順

問題を発見した時点で PROB-NNN を起票する。ALCOA+ Contemporaneous 原則に基づき、問題発見当日中に起票する。

### 2-1. 起票先

`06_実装/17_実装問題管理ログ.md` が PROB-NNN の権威台帳である。保守フェーズで発見した問題も当該ログへ追記する。形式は当該ログの既存エントリに準拠する。

### 2-2. 起票フォーマット

```yaml
prob_id: PROB-NNN
title: <問題の短い説明（50 文字以内）>
category: functional_bug | performance | security | dependency | design_gap
priority: P1 | P2 | P3 | P4
discovered_at: YYYY-MM-DD HH:MM
discovered_by: system_admin
environment: production | staging | development
description: |
  <問題の詳細説明>
reproduction_steps: |
  <再現手順（番号付き）>
impact_scope: |
  <影響範囲（ALCOA+ 影響・API 影響・ユーザー影響）>
related_inc: INC-YYYY-NNN  # インシデント連動の場合
related_rec: REC-NNN       # 保守記録 ID（01 章で起票済みの場合）
status: open | in_analysis | cr_raised | resolved
```

### 2-3. 監査ログによる問題再現確認

問題の再現確認時に以下のクエリで直近の異常を特定する。

```sql
-- 問題再現確認時の監査ログ確認
SELECT event_type, count(*), min(timestamp_server), max(timestamp_server)
FROM trace_events
WHERE timestamp_server >= NOW() - INTERVAL '7 days'
GROUP BY event_type
ORDER BY count(*) DESC;
```

```sql
-- 特定エラーの発生パターン確認
SELECT
  event_type,
  user_id,
  device_id,
  error_code,
  timestamp_server
FROM trace_events
WHERE event_type = 'ERROR'
  AND timestamp_server >= NOW() - INTERVAL '24 hours'
ORDER BY timestamp_server DESC
LIMIT 100;
```

**本節で確定した方針**
- **PROB-NNN の起票は問題発見当日中に実施する（ALCOA+ Contemporaneous 原則）。当日超過は台帳に遅延理由を記録する。**
- **PROB-NNN 台帳は 06_実装/17_実装問題管理ログ.md が権威であり、保守記録サブは参照のみ行う。**
- **問題再現確認には trace_events テーブルの監査ログを活用し、確認クエリを REC-NNN に記録する。**

## 3. 問題の 5 分類

発見した問題は以下の 5 分類のいずれかに分類する。分類は PROB-NNN の `category` フィールドに記録する。

### 3-1. functional_bug（機能バグ）

- 定義: 仕様に対してシステムの振る舞いが誤っている
- 例: 作業完了記録が保存されない・ポカヨケアラートが誤発報
- ALCOA+ 影響: Original / Accurate 原則に関わる場合は quality_admin 承認が必要
- 修正分岐: コード修正（03 章 §2-1）

### 3-2. performance（性能問題）

- 定義: NFR で定義した性能 SLO を下回る状態が継続している
- 例: クエリ応答時間が SLO 上限超過・ハンディ端末の同期遅延超過
- ALCOA+ 影響: 同期遅延拡大時は Contemporaneous 原則に関わる
- 修正分岐: コード修正 / DB チューニング / インフラ調整（03 章 §2-1〜§2-3）

### 3-3. security（セキュリティ）

- 定義: CVE 報告・ペネトレーション結果・鍵漏洩疑惑
- 例: Rust 依存クレートの CVE・JWT 鍵の漏洩疑惑・TLS 設定不備
- ALCOA+ 影響: 電子サイン・ハッシュチェーンに関わる場合は P1 扱いとする
- 修正分岐: セキュリティパッチ適用（03 章 §2-4）

### 3-4. dependency（依存関係）

- 定義: 外部ライブラリ・Docker ベースイメージ・PostgreSQL minor の EOL / 脆弱性
- 例: tokio の新バージョン互換性問題・axum の breaking change
- ALCOA+ 影響: 依存更新による API 仕様変化がトレサビ記録に影響する場合は影響分析が必要
- 修正分岐: 依存更新（03 章 §2-2）

### 3-5. design_gap（設計ギャップ）

- 定義: 設計時に想定しなかったユースケースや制約の発覚
- 例: ハンディ端末の Offline 時間が想定超過・工程数が設計上限に近接
- ALCOA+ 影響: 設計ギャップが ALCOA+ 原則の実現方式に影響する場合は ADR-OPS-NNN で設計判断を記録
- 修正分岐: CR-OPS-NNN 起票 → 上流フェーズへの設計変更依頼

**本節で確定した方針**
- **問題は 5 分類（functional_bug / performance / security / dependency / design_gap）の 1 つに分類する。分類不能な問題は design_gap として起票し ADR-OPS-NNN で分類方針を確定する。**
- **security 分類の問題は分類決定後 2 時間以内に優先度を P1〜P2 に確定する。**
- **design_gap 分類の問題は上流フェーズ変更を伴う場合が多いため、影響分析（§4）を必ず実施してから CR-OPS-NNN を起票する。**

## 4. 影響分析チェックリスト

PROB-NNN 起票後、修正を決定する前に以下のチェックリストを実施する。チェック結果は PROB-NNN の `impact_scope` フィールドに記録する。

### 4-1. ALCOA+ 影響

| チェック項目 | 確認内容 | 該当する場合の対応 |
|---|---|---|
| Attributable | 修正後も全操作の実施者が特定可能か | 電子サイン / ユーザー ID 紐付けを確認 |
| Legible | 修正後も記録が判読可能な形式を維持するか | 文字コード・形式変更の有無を確認 |
| Contemporaneous | 修正後も記録がリアルタイムに生成されるか | Outbox / 同期処理への影響を確認 |
| Original | 修正により既存記録の原本性が損なわれないか | ハッシュチェーン影響分析（§4-2）を必ず実施 |
| Accurate | 修正後の記録値が実態を正確に反映するか | 計算ロジック変更・単位変更の有無を確認 |
| Complete | 必須項目が欠落なく記録されるか | バリデーション変更の有無を確認 |
| Consistent | 異なるシステム間で記録が一致するか | API・ハンディ・PostgreSQL の整合性を確認 |
| Enduring | アーカイブ後も長期参照可能か | ファイル形式・エンコーディング変更の有無を確認 |

### 4-2. ハッシュチェーン影響

ハッシュチェーン（ERR-DB-003 / TBL-audit_chain）に関わる修正は以下を確認する。

```bash
# 修正前にハッシュチェーン整合性確認
docker compose exec api ./hash-chain-verify --full

# 出力例（正常）
# INFO: hash-chain-verify: all 12345 records verified OK
# ERROR が出力された場合は修正を中断して障害対応サブへ転送
```

### 4-3. TBL 影響（スキーマ変更有無）

対象テーブル（TBL-001〜035）の一覧から影響テーブルを特定し、スキーマ変更の有無を判定する。スキーマ変更が発生する場合は `03` §2-3（DB マイグレーション手順）を適用する。

### 4-4. API 影響（API-001〜039）

修正が API エンドポイント（API-001〜039）の仕様変化を伴う場合は上流設計変更（CR-OPS-NNN 経由）が必要である。API 変更を伴う保守修正は単独で実施しない。

**本節で確定した方針**
- **全 PROB-NNN に対して §4-1 ALCOA+ 影響チェックリストを実施し、結果を PROB-NNN に記録する。**
- **ハッシュチェーンに関わる修正は hash-chain-verify を必ず実行し、ERROR 0 を修正開始条件とする。**
- **API 仕様変化を伴う修正は CR-OPS-NNN 経由で上流フェーズへ設計変更を依頼してから保守プロセスを継続する。**

## 5. 優先度付けと修正 SLO

PROB-NNN の優先度は P1〜P4 の 4 段階とし、継承した NFR-OPS-040〜042 の SLO に対応する。

| 優先度 | 定義 | 修正完了 SLO | 作業開始 SLO | 備考 |
|---|---|---|---|---|
| P1 | 本番停止 / ALCOA+ 違反 / データ損失リスク | ≤ 24h（NFR-OPS-040）| ≤ 1h | quality_admin 通知必須 |
| P2 | 本番機能劣化（縮退運用中）| ≤ 72h（NFR-OPS-041）| ≤ 4h | quality_admin 通知推奨 |
| P3 | 非クリティカルなバグ / 性能軽微劣化 | 次回定期保守まで | 当週内 | |
| P4 | 改善・技術的負債 | 次期四半期計画 | 計画後 | 11 章 技術的負債台帳に記録 |

P1 は発見後即座に障害対応サブ（`../障害対応/01_P1障害対応手順.md`）との並行対応が必要である。

**本節で確定した方針**
- **優先度は P1〜P4 の 4 段階とし、PROB-NNN 起票時に決定して変更時は理由を記録する。**
- **P1 修正は NFR-OPS-040 の 24h SLO を遵守し、完了後 48h 以内にポストモーテム（PM-YYYY-NNN）を起票する。**
- **P4 問題は 11 章の技術的負債台帳に記録し、四半期返済計画に組み込む。**

## 6. CR-OPS-NNN（変更要求）への紐付け

問題分析が完了し修正が必要と判断した場合、CR-OPS-NNN を起票して PROB-NNN と紐付ける。

```yaml
cr_ops_id: CR-OPS-NNN
title: <変更要求の短い説明>
related_prob: PROB-NNN
related_rec: REC-NNN
change_type: bug_fix | dependency_update | key_rotation | db_migration | security_patch | sop_revision | design_change
priority: P1 | P2 | P3 | P4
requested_by: system_admin
requested_at: YYYY-MM-DD
approved_by: quality_admin  # ALCOA+ 影響・P1 修正の場合は必須
approved_at: YYYY-MM-DD
```

design_change 種別の CR-OPS-NNN は上流フェーズへの変更依頼を意味する。この種別の CR は上流フェーズ担当者（本システムでは同一人物だが上流文書の改訂手順を踏む）が対応する。

**本節で確定した方針**
- **修正実施（03 章）を開始する前に CR-OPS-NNN を起票し、PROB-NNN との紐付けを完了させる。**
- **P1・P2 の CR-OPS-NNN は quality_admin の承認を受けてから修正実施フェーズへ移行する（P3・P4 は system_admin 単独で実施可能）。**
- **design_change 種別の CR-OPS-NNN は上流フェーズ文書の改訂手順を踏んでから保守プロセスを継続する。**
