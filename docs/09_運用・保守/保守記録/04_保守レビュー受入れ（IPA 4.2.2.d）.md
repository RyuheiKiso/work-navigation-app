# 04 保守レビュー受入れ（IPA 4.2.2.d）

IPA 共通フレーム 2013「**4.2.2.d 保守レビュー及び/又は受入れ**」に準拠した手順を確定する。個人開発 1 名体制における自己レビュー手順・受入判定チェックリスト・quality_admin 承認が必要なケース・受入完了の記録手順を実体化する。

---

## 1. 本章の範囲

IPA 4.2.2.d が要求する「保守レビュー及び/又は受入れ」は以下の項目を含む。

| IPA 4.2.2.d 要求項目 | 本章担当節 |
|---|---|
| 修正内容のレビュー（品質確認）| §2 |
| 受入判定基準の確認 | §3 |
| 承認・受入れの記録 | §4 |
| 受入完了の記録 | §5 |

本章は 03 章（修正実施）の完了後から 05 章（本番移行）の開始前までの手順を担う。

**本節で確定した方針**
- **個人開発 1 名体制において、自己レビューの客観性を担保するために 7 日クーリングルールを適用する。**
- **ALCOA+ 影響変更・電子サイン影響変更・P1 修正は quality_admin の承認を必須とし、self-approve を禁止する。**
- **本章の準拠宣言: 4.2.2.d に準拠する。**

---

## 2. 個人開発における自己レビュー手順（7 日クーリングルール）

### 2-1. クーリングルールの定義

system_admin が修正を実施した後、同一人物が即座にレビュー・受入れを行うことはレビューの独立性を損なう。これを防ぐため、P3・P4 修正には **7 日クーリングルール** を適用する。

| 修正優先度 | クーリング期間 | レビュー実施者 | 根拠 |
|---|---|---|---|
| P1 | 免除（緊急対応優先）| quality_admin（必須）| RTO 遵守優先 |
| P2 | 24h | system_admin + quality_admin 確認 | 機能劣化継続の解消優先 |
| P3 | 7 日 | system_admin（self-review）| 認知バイアス低減 |
| P4 | 7 日 | system_admin（self-review）| 認知バイアス低減 |

P1 のクーリング免除はポストモーテム（PM-YYYY-NNN）でレビューの代替とする。

### 2-2. 自己レビューの実施手順

```bash
# 1. 修正コミットの差分を確認
git diff main...HEAD --stat
git log main..HEAD --oneline

# 2. 差分の全行を確認（7 日後に改めて実施）
git diff main...HEAD

# 3. テスト結果の再確認
cargo test --release 2>&1 | tee /tmp/test-result.txt
cat /tmp/test-result.txt

# 4. STG 確認結果（03 章で記録済み）の再確認
# REC-NNN の stg_5item_result を確認

# 5. ALCOA+ 影響分析結果（02 章で記録済み）の再確認
# PROB-NNN の impact_scope を確認
```

**本節で確定した方針**
- **P3・P4 修正は修正コミット後 7 日以上経過してから自己レビューを実施する。7 日未満でのレビューは不正とする。**
- **P1 修正はクーリング免除とするが quality_admin 承認を必須とし、ポストモーテムで事後レビューを実施する。**
- **自己レビューの実施記録（実施日時・レビュー者）を REC-NNN に必ず記録する（ALCOA+ Attributable 原則）。**

## 3. 受入判定チェックリスト

以下の全項目を確認し、全て PASS した場合のみ受入完了と判定する。

### 3-1. 機能確認

| # | チェック項目 | 確認方法 | 合格基準 |
|---|---|---|---|
| FC-1 | PROB-NNN で報告された問題が解消されているか | STG 環境で再現確認 | 問題が再現しない |
| FC-2 | 変更対象外の機能に影響が出ていないか | 回帰テスト PASS（03 章 §3-3）| 全テスト PASS |
| FC-3 | エラーコード（ERR-NNN）が正常値を返すか | STG 環境での動作確認 | 正常値応答 |

### 3-2. テスト PASS 確認

| # | チェック項目 | 合格基準 |
|---|---|---|
| TP-1 | 自動テスト全 PASS（03 章 §3-1）| 全 6 項目 PASS |
| TP-2 | STG 5 項目全 PASS（03 章 §4）| 全 5 項目 PASS |
| TP-3 | ハッシュチェーン整合性確認済み（03 章 §3-2）| ERROR 0（ALCOA+ 影響時）|

### 3-3. ALCOA+ 整合確認

| # | チェック項目 | 合格基準 |
|---|---|---|
| AC-1 | 修正後の ALCOA+ 影響分析が完了しているか | PROB-NNN の impact_scope に記録済み |
| AC-2 | Original 原則（ハッシュチェーン）に違反していないか | hash-chain-verify ERROR 0 |
| AC-3 | Contemporaneous 原則が修正後も維持されているか | Outbox 保留件数 0 |

### 3-4. 文書更新確認

| # | チェック項目 | 合格基準 |
|---|---|---|
| DC-1 | REC-NNN が最新状態に更新されているか | status = completed・全フィールド記入済み |
| DC-2 | 関連台帳（08〜10）が更新されているか | 修正種別に対応した台帳の更新日時が修正日以降 |
| DC-3 | PROB-NNN が最新状態に更新されているか | status = resolved |

**本節で確定した方針**
- **受入判定チェックリストは FC / TP / AC / DC の全 11 項目を確認し、全 PASS を受入開始条件とする。**
- **1 項目でも不合格の場合は修正フェーズ（03 章）に戻り、再チェックリストを実施する。**
- **チェックリストの実施結果は REC-NNN に記録し（ALCOA+ Attributable・Contemporaneous 原則）、確認者・確認日時を必ず記録する。**

## 4. quality_admin の承認が必要なケース

以下のいずれかに該当する場合は quality_admin の承認を得てから受入完了とする。

| ケース | 根拠 | 承認の形式 |
|---|---|---|
| ALCOA+ 影響変更（AC-1〜3 でいずれかに影響あり）| 21 CFR Part 11・ALCOA+ 原則 | CR-OPS-NNN の approved_by に quality_admin を記入 |
| 電子サイン処理に関わる修正 | 21 CFR Part 11 §11.50 | 同上 |
| P1 修正 | NFR-OPS-040・MNT-040 | 同上 |
| SOP 改訂（10 章 §3 参照）| MNT-040 | SOP 改訂記録の approved_by に記入 |

quality_admin が承認を拒否した場合、拒否理由を CR-OPS-NNN に記録してから修正フェーズ（03 章）に差し戻す。

**本節で確定した方針**
- **ALCOA+ 影響変更・電子サイン変更・P1 修正・SOP 改訂の 4 ケースは quality_admin 承認を必須とし、承認記録は CR-OPS-NNN に残す。**
- **quality_admin の不在時（休暇・緊急時）に P1 対応が必要な場合は ADR-OPS-NNN に代替承認手順を記録してから実施する。**
- **quality_admin 承認を必要としない修正（P3・P4 の non-ALCOA+ 修正）は system_admin の自己レビューのみで受入完了とする。**

---

## 5. 受入完了の記録

受入完了時に REC-NNN を以下の手順で更新する。

```yaml
# REC-NNN の受入完了記録フィールド更新
review_completed_at: YYYY-MM-DD HH:MM
review_completed_by: system_admin  # または quality_admin
review_type: self_review | quality_admin_approval
review_notes: |
  <レビューで確認した事項・指摘事項・対応結果>
checklist_fc_all_pass: true | false
checklist_tp_all_pass: true | false
checklist_ac_all_pass: true | false
checklist_dc_all_pass: true | false
ready_for_deployment: true | false  # 全チェック PASS の場合のみ true
```

`ready_for_deployment: true` が 05 章（本番移行）の開始条件である。`false` のまま本番移行を開始することは禁止する。

**本節で確定した方針**
- **REC-NNN の `ready_for_deployment: true` が 05 章（本番移行）の唯一の開始条件とし、それ以外の判断基準での本番移行開始を禁止する。**
- **受入完了記録は ALCOA+ Contemporaneous 原則に基づき受入判定当日中に完了させる。**
- **受入を拒否した場合は `ready_for_deployment: false` と拒否理由を REC-NNN に記録し、修正フェーズ（03 章）への差し戻しを明示する。**
