# 01 保守プロセス開始準備（IPA 4.2.2.a）

IPA 共通フレーム 2013「**4.2.2.a 保守プロセス開始準備**」に準拠した手順を確定する。保守体制の確認・保守ツールの準備・引継ぎ文書の整備・保守プロセスのトリガー定義・REC-NNN 起票手順を実体化する。

図: fig_mnt_lifecycle（img/ 配下）を参照

---

## 1. 本章の範囲

IPA 4.2.2.a が要求する「保守プロセス開始準備」は以下の 5 項目を含む。本章はその全項目を対象とする。

| IPA 4.2.2.a 要求項目 | 本章担当節 |
|---|---|
| 保守体制・役割・責任の確立 | §2 |
| 保守ツールの識別・準備 | §3 |
| 引継ぎ文書の整備 | §4 |
| 保守プロセスのトリガー定義 | §5 |
| 保守記録の起票 | §6 |

**本節で確定した方針**
- **IPA 4.2.2.a の全要求項目は本章の §2〜§6 で完全にカバーする。**
- **体制・ツール・引継ぎ・トリガー・起票の 5 要素は全て事前に確立し、保守作業開始前に確認する。**
- **本章の準拠宣言: 4.2.2.a に準拠する。**

---

## 2. 保守計画の確認（体制）

### 2-1. 保守体制

本システムは個人開発であり、全保守作業を system_admin（IT 担当 1 名）が単独で実施する。quality_admin（製品品質責任者）は ALCOA+ 影響変更時・電子サイン変更時・P1 修正時の承認者として参加する。

| ロール | 担当者 | 責任範囲 |
|---|---|---|
| system_admin | IT 担当（本人）| 全保守作業の計画・実施・記録 |
| quality_admin | 製品品質責任者 | ALCOA+ 影響変更の承認・電子サイン変更の承認 |

### 2-2. バスファクター対策

system_admin が業務不能となった場合に備え、以下を常時最新に保つ。

- 保守手順書（本ディレクトリ 01〜12）が自己完結していること
- 環境変数台帳（ENV-001〜025）がアクセス可能な場所に保管されていること
- 鍵台帳（09 章）の最終ローテーション記録が最新であること

**本節で確定した方針**
- **system_admin 1 名体制を前提とし、手順書の自己完結性がバスファクター対策の実体とする。**
- **quality_admin の承認が必要なケースを明示し、承認不要な作業との分界を 04 章で確定する。**
- **体制変更（新メンバー追加等）は CR-OPS-NNN を起票して本章を改訂する。**

## 3. 保守ツールの準備

保守作業開始前に以下のツールが最新かつ動作可能であることを確認する。

### 3-1. ツール確認コマンド

```bash
# Rust / Cargo バージョン確認
cargo --version
rustup show

# pnpm バージョン確認
pnpm --version

# PostgreSQL クライアント確認
psql --version

# Git バージョン確認
git --version

# Docker / compose 確認
docker --version
docker compose version

# sqlx-cli 確認
sqlx --version

# cargo-audit 確認
cargo audit --version
```

### 3-2. 最低動作バージョン要件

| ツール | 最低バージョン | 備考 |
|---|---|---|
| Rust | Edition 2024 stable | `rustup update stable` で最新化 |
| cargo-audit | 最新安定版 | `cargo install cargo-audit` |
| sqlx-cli | 最新安定版（0.8 系以上）| `cargo install sqlx-cli` |
| pnpm | 9 系以上 | `corepack prepare pnpm@latest --activate` |
| psql | PostgreSQL 16 対応版 | 本番 DB バージョンと一致させる |
| Git | 2.40 以上 | WSL2 環境 |

**本節で確定した方針**
- **ツール確認コマンドを保守作業開始時に必ず実行し、全ツールが最低バージョン要件を満たすことを確認してから保守作業を開始する。**
- **ツールが要件を下回る場合は当日の保守作業を中断し、ツール更新を最優先で実施する。**
- **ツールバージョン要件の変更は CR-OPS-NNN を起票して本節を更新する。**

## 4. 引継ぎ文書の整備

以下の文書が最新状態であることを四半期ごとに確認する。引継ぎ文書が陳腐化した場合は ADR-OPS-NNN を起票して改訂方針を確定する。

| 文書種別 | 権威ファイル | 最終更新確認タイミング |
|---|---|---|
| 環境変数台帳 | `../../06_実装/09_環境変数台帳（ENV）.md` | 環境変数変更時・四半期ごと |
| 鍵台帳 | `09_鍵ローテーション履歴テンプレ.md` | 各ローテーション実施後 |
| デプロイ手順 | `../../06_実装/10_デプロイ手順.md` | デプロイ手順変更時・半年ごと |
| DB スキーマ | `../../05_詳細設計/` TBL-001〜035 | スキーマ変更時 |
| アーキテクチャ概要 | `../../04_概要設計/README.md` | 半年ごと（変更なし確認）|

**本節で確定した方針**
- **引継ぎ文書は四半期ごとに最新状態を確認し、陳腐化を検出した場合は即座に更新する。**
- **環境変数・鍵情報の更新は保守作業完了後 24 時間以内に台帳へ反映する。**
- **引継ぎ文書の整備状態は 12 章の年次評価で評価項目に含める。**

## 5. 保守プロセスの開始トリガー

保守プロセスは以下の 3 種のトリガーのいずれかで開始する。トリガーの種別は REC-NNN 起票時の `record_type` に反映する。

### 5-1. インシデント報告トリガー（INC-YYYY-NNN 連動）

障害対応（`障害対応/` サブ）で INC-YYYY-NNN が起票された後、根本原因修正が必要と判定された場合に保守プロセスを開始する。

```
INC-YYYY-NNN 起票
  → 切り分け・復旧（障害対応サブ）
  → 根本原因修正が必要と判定（障害対応/11 §3）
  → PROB-NNN 起票（02 章）
  → REC-NNN 起票（本章）
  → 保守プロセス開始
```

### 5-2. 定期保守トリガー（四半期 / 年次）

| 周期 | 内容 | 担当章 |
|---|---|---|
| 月次 | SLO レポート確認・依存更新要否判定 | `../運用手順/04` |
| 四半期 | 依存ライブラリ更新・鍵ローテーション確認 | `08`・`09` |
| 年次 | 運用評価レポート・技術的負債返済計画 | `12`・`11` |

### 5-3. 変更要求トリガー（CR-OPS-NNN）

quality_admin・system_admin からの変更要求（手順改訂・機能修正要求・セキュリティ強化要求）は CR-OPS-NNN として起票された時点で保守プロセスを開始する。

**本節で確定した方針**
- **保守プロセスの開始トリガーはインシデント報告・定期保守・変更要求の 3 種とし、それ以外の非公式要求による保守作業開始は禁止する。**
- **全トリガーで REC-NNN を起票してから保守作業を開始する。記録なしの保守作業実施は ALCOA+ 違反とする。**
- **インシデント連動トリガーは障害対応サブが起点であり、保守記録サブは PROB-NNN 起票から責務を引き継ぐ。**

## 6. 保守記録の起票（REC-NNN 採番）

保守プロセス開始時に `07_保守記録テンプレ（REC-NNN）.md` を複製し、以下の手順で起票する。

```bash
# 台帳で次の REC 番号を確認
# ファイル: ../付録/99_運用保守識別子採番台帳.md §5

# テンプレートを複製
cp docs/09_運用・保守/保守記録/07_保守記録テンプレ\（REC-NNN\）.md \
   docs/09_運用・保守/保守記録/REC-NNN_YYYY-MM-DD_<概要スラグ>.md

# 台帳に採番を登録
# ../付録/99 §5 の台帳テーブルに新行を追加

# 起票者・日時を記入（ALCOA+ Contemporaneous 原則: 作業開始時に記録する）
```

起票後は台帳への登録が完了したことを確認してから保守作業を開始する。台帳未登録の REC-NNN を参照する文書は受理しない。

**本節で確定した方針**
- **REC-NNN の起票は保守作業開始と同時（ALCOA+ Contemporaneous 原則）に実施する。事後記録は ALCOA+ 違反とする。**
- **起票には付録/99 §5 台帳への登録が必須であり、台帳未登録の識別子は不正とする。**
- **REC-NNN の closed 化は本番移行完了後（05 章）または廃棄完了後（06 章）に行う。**
