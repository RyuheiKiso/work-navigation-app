# 04 運用保守トレーサビリティ枠組み

本章は運用保守フェーズのトレーサビリティ体系を確定する。NFR-OPS/AVL 要件から実行可能な手順・チェックリスト・記録に至る 5 段縦串マッピングの構造と接続規則を定義し、上流 RTM との連携方針を明示する。

---

## 1. 5 段縦串マッピングの全体構造

運用保守フェーズのトレーサビリティは以下の 5 段階で構成する。

```
[段 1] 非機能要件（NFR-OPS-NNN / NFR-AVL-NNN）
           ↓ 要件充足の根拠
[段 2] 運用手順概要（OPS-PROC-NNN）
           ↓ 手順の詳細化
[段 3] 障害対応ランブック（RUN-NNN）
           ↓ 実行確認
[段 4] チェックリスト（CHK-NNN）
           ↓ 記録・証跡
[段 5] 保守記録（REC-NNN）/ 障害記録（INC-YYYY-NNN）/ ポストモーテム（PM-YYYY-NNN）
```

図: fig_ops_trace_hierarchy（img/ 配下）を参照

**本節で確定した方針**
- **5 段縦串を全 NFR-OPS/AVL 要件に適用し、要件が追跡できない手順は作成しない。**
- **各段の接続は付録/01 OTM で一元管理する。**
- **トレーサビリティの欠落は CR-OPS-NNN を起票して補完する。**

---

## 2. 各段の接続規則

### 2-1. 段 1 ↔ 段 2（NFR → OPS-PROC）

| 規則 | 内容 |
|---|---|
| **1:N 対応を許容する** | 1 つの NFR-NNN が複数の OPS-PROC-NNN に対応することを認める（例: NFR-OPS-001 は OPS-PROC-002/003/004 に対応）|
| **N:1 対応を許容する** | 複数の NFR-NNN が 1 つの OPS-PROC-NNN で充足されることを認める |
| **未対応の NFR は禁止** | 全 NFR-OPS/AVL は少なくとも 1 つの OPS-PROC-NNN または RUN-NNN に対応づけられていなければならない |
| **記録場所** | 付録/01 OTM の「NFR → OPS-PROC」列 |

### 2-2. 段 2 ↔ 段 3（OPS-PROC → RUN）

| 規則 | 内容 |
|---|---|
| **RUN は障害時のみ** | RUN-NNN は障害対応・緊急作業に限定する。定常作業は CHK-NNN が担う |
| **RUN の前提条件** | 各 RUN の冒頭に前提条件（対象優先度・検知条件）を明記する |
| **OPS-PROC-NNN との対応** | 各 RUN は起動する OPS-PROC-NNN を参照する |

### 2-3. 段 3/2 ↔ 段 4（OPS-PROC/RUN → CHK）

| 規則 | 内容 |
|---|---|
| **CHK は確認単位** | 1 つの CHK は「確認できる/できない」で二値判定できる粒度とする |
| **CHK の分類** | 週次（CHK-001〜006）・月次（CHK-007〜014）・年次（CHK-015〜028）|
| **完了条件** | 対応する OPS-PROC の全 CHK が「確認済み」になることで手順完了と判定する |

### 2-4. 段 4/3 ↔ 段 5（CHK/RUN → REC/INC/PM）

| 規則 | 内容 |
|---|---|
| **記録は必須** | CHK を実施した証跡として REC-NNN を保守記録台帳に登録する |
| **障害は INC を必須起票** | P1/P2 障害はインシデント発生時点で INC-YYYY-NNN を起票する |
| **P1 はポストモーテム必須** | P1 障害の INC-YYYY-NNN には PM-YYYY-NNN が必ず対応する（48h 以内）|
| **REC の保存期間** | 3 年以上（監査対応のため）|

**本節で確定した方針**
- **5 段の接続は「要件 → 手順 → 実行確認 → 記録」の一方向フローとし、逆向きの変更はしない。**
- **CHK の二値判定と REC への記録を全定期作業に必須とする。**
- **P1 障害には INC と PM の両方を必須起票し、48h 以内に PM を完了する。**

## 3. RTM との連携方針

`03_要件定義/付録/01_要件トレーサビリティマトリクス（RTM）.md` との連携方針を確定する。

| 連携項目 | 方針 |
|---|---|
| RTM の権威 | 03_要件定義 の RTM が NFR-OPS/AVL の権威台帳である。09 章は RTM を参照するが変更しない |
| OTM の位置づけ | 付録/01 OTM は「NFR → 運用手順」の接続を追跡する 09 章固有の台帳であり、RTM の下位台帳として機能する |
| 整合確認 | 月次 SLO レポート（OPS-PROC-004）の中で RTM と OTM の整合を確認する |
| 逸脱の記録 | RTM の NFR に対応する OTM エントリが欠落している場合、CR-OPS-NNN を起票して補完する |

**本節で確定した方針**
- **RTM（03_要件定義）は変更しない。OTM（付録/01）が運用接続の権威台帳である。**
- **RTM と OTM の整合確認を月次作業に組み込む。**
- **OTM エントリの欠落は CR-OPS-NNN として記録し補完する。**

## 4. トレーサビリティ適用例

NFR-OPS-001（稼働率 ≥ 99.5%）の 5 段縦串適用例を示す。

| 段 | 識別子 | 内容 |
|---|---|---|
| 段 1 | NFR-OPS-001 | 稼働率 ≥ 99.5%（営業時間 06:00-22:00）|
| 段 2 | OPS-PROC-002 | 週次定期作業（稼働確認を含む）|
| 段 2 | OPS-PROC-004 | 月次 SLO レポート（稼働率実績測定）|
| 段 3 | RUN-001 | システム無応答時の初動確認ランブック |
| 段 4 | CHK-001 | 週次 : バックエンドヘルスチェック確認 |
| 段 4 | CHK-007 | 月次 : 稼働率計算・SLO 実績確認 |
| 段 5 | REC-NNN | 週次/月次作業完了の保守記録 |

**本節で確定した方針**
- **全 NFR-OPS/AVL 要件に 5 段縦串を適用する。付録/01 OTM が全件の対応表を管理する。**
- **NFR に対応する CHK が存在しない場合、その NFR は運用上保証されていないと判断する。**
- **適用例（NFR-OPS-001）は OTM の最初のエントリとして記録する。**
