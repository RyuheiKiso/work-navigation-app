# 05 運用判断 ADR 索引（ADR-OPS-NNN）

本文書は `09_運用・保守` フェーズで発生した不可逆な運用判断を記録する Architecture Decision Record（ADR）の採番規則・記録テンプレート・索引を管理する。ADR-OPS-NNN は `付録/99_運用保守識別子採番台帳.md` §7 が権威台帳である。

---

## 1. ADR-OPS の採番規則

| 項目 | 規則 |
|---|---|
| **プレフィックス** | `ADR-OPS-` + 3 桁ゼロ埋め連番 |
| **開始番号** | ADR-OPS-001 |
| **欠番** | 禁止。廃止時は `[廃止: YYYY-MM-DD 理由]` を付与 |
| **起票条件** | ①不可逆な運用方針の決定、②有力な代替案が存在した、③選択理由が自明でない — 3 条件全てを満たす場合のみ起票 |
| **起票者** | system_admin |
| **承認者** | quality_admin（7 日以内に確認・承認）|
| **記録場所** | 本文書（索引）+ 付録/99 §7（台帳）|

## 4. ADR-OPS-001 詳細

### ADR-OPS-001: ポストモーテム物理配置の判断

**日付**: 2026-05-18  
**状態**: 承認済み  
**決定者**: RyuheiKiso（system_admin）  
**承認者**: quality_admin  
**関連識別子**: OPS-064

#### 背景・文脈

ポストモーテム記録（PM-YYYY-NNN）の物理配置先として、`09_運用・保守/障害対応/` 内に格納する案と、トップレベルの `docs/ポストモーテム/` に格納する案が浮上した。障害ごとに独立した md ファイルが積み上がる性質のため、配置先の決定は 09 章全体の構造に影響する不可逆な判断である。

#### 検討した選択肢

| 選択肢 | 概要 | 利点 | 欠点 |
|---|---|---|---|
| 選択肢 A（採用）| `docs/ポストモーテム/YYYY-MM-DD_PX_<slug>.md` | ①09 章サブ内でファイルが肥大化しない、②年次・障害別の閲覧が容易、③将来の自動索引スクリプトが書きやすい | ①09 章から外れるためリンクが相対パスで長くなる |
| 選択肢 B | `09_運用・保守/障害対応/ポストモーテム/YYYY-MM-DD_PX_<slug>.md` | ①09 章内で完結する | ①ファイル数の増加で 09 章のサブ構造が複雑になる、②他フェーズからのアクセスが 09 章を経由するため可読性が下がる |

#### 決定内容

**選択肢 A を採用する。** 個別記録は `docs/ポストモーテム/YYYY-MM-DD_PX_<slug>.md` に配置する。索引・手順・テンプレートは `09_運用・保守/障害対応/11_ポストモーテム手順とテンプレート.md` に集約する。この配置は OPS-064 として確定し、09 章全体で参照する。

#### 影響・結果

- `docs/ポストモーテム/` ディレクトリが最初から作成される（Git 管理対象）。
- PM-YYYY-NNN の個別記録は 09 章から独立した物理パスに格納されるが、識別子は付録/99 §4 で一元管理される。
- `障害対応/11` からの相対リンクは `../../ポストモーテム/` 形式になる。

#### 再評価条件

- ポストモーテム記録数が年 10 件を超えた場合、自動索引スクリプトの導入を ADR-OPS-NNN を起票して判断する。
