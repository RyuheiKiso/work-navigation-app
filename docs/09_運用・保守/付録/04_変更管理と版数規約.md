# 04 変更管理と版数規約

本章は `09_運用・保守` フェーズの文書版数体系・変更種別・変更管理プロセス・変更ログテンプレート・緊急変更（P1 障害起因の手順改訂）の特別フローを確定する。`06_実装/付録/04_変更管理と版数規約.md` の規約を継承し、運用保守フェーズ固有の事項を追加定義する。

---

## 1. 版数体系

### 1-1. セマンティックバージョニング（SemVer）

`09_運用・保守` フェーズの文書は `メジャー.マイナー.パッチ` で管理する。

| 変更種別 | バージョン操作 | 例 |
|---|---|---|
| 破壊的変更（識別子体系の変更・SLO 値の変更を伴う手順改訂）| メジャー上げ | 0.x.y → 1.0.0 |
| 機能追加（新 OPS-PROC・新 CHK・新 RUN の追加・新サブフォルダ追加）| マイナー上げ | 0.1.y → 0.2.0 |
| 内容修正・軽微修正（誤字修正・手順の微調整・リンク修正）| パッチ上げ | 0.1.0 → 0.1.1 |

**初期版数**: `0.1.0`（初版作成。`1.0.0` は本番稼働開始・全手順の初回実施完了時に付与）

### 1-2. 緊急改訂の版数処理

P1 障害起因のポストモーテム是正措置として手順書を改訂した場合は、改訂内容の重大性に応じてパッチまたはマイナーを上げる。

```markdown
# 緊急改訂版数例
0.1.0 → 0.1.1  # P1 是正措置での軽微手順修正
0.1.0 → 0.2.0  # P1 是正措置での新チェックリスト項目追加
```

## 2. 変更種別と版数上げ方針

| 変更種別 | バージョン影響 | ADR-OPS 必要 | CR-OPS 必要 |
|---|---|---|---|
| 新 OPS-PROC 手順ファイルの追加 | マイナー上げ | 不要 | 必要（CR-OPS-NNN 起票）|
| 既存手順書の誤字修正・リンク修正 | パッチ上げ | 不要 | 不要 |
| CHK 項目の追加・変更 | マイナー上げ | 不要 | 必要 |
| RUN の追加（新障害パターン対応）| マイナー上げ | 不要 | 必要 |
| SLO 値変更を伴う手順改訂 | メジャー上げ | 必要（ADR-OPS-NNN）| 必要（上流フェーズへ CR）|
| 識別子プレフィックス追加 | メジャー上げ | 必要（ADR-OPS-NNN）| 必要 |
| P1 ポストモーテム是正措置による手順改訂 | 内容による | 判断による | 必要（CR-OPS-NNN）|

## 3. 通常変更プロセス

1. **変更起票**: CR-OPS-NNN を付録/99 §6 に登録する。変更種別・対象ファイル・変更内容・理由を記録する。
2. **影響分析**: 付録/01 OTM で影響する NFR-OPS/AVL・CHK・RUN を列挙し、上流文書との逸脱有無を確認する。
3. **ADR 起票判断**: 以下の 3 条件を全て満たす場合のみ ADR-OPS-NNN を起票する。
   - 運用方針として不可逆な決定（例: SLO 値変更・識別子体系変更）
   - 有力な代替案が存在した
   - 選択理由が自明でない
4. **ドキュメント先行更新**: 対象 md ファイルを更新してから実際の手順変更を実施する（ドキュメント先行原則）。
5. **採番台帳更新**: 付録/99 の対応 CR-OPS-NNN・ADR-OPS-NNN のステータスを「確定済み」に更新する。
6. **版数更新**: 対象ファイルの版数履歴に新行を追加する。
7. **Git コミット**: コミットメッセージに `CR-OPS-NNN` を含める。

## 4. 緊急変更（P1 障害起因の手順改訂）特別フロー

### 4-1. 緊急変更の定義

P1 障害のポストモーテム（PM-YYYY-NNN）で特定された根本原因への是正措置として手順書を改訂する変更を緊急変更と定義する。

### 4-2. 緊急変更フロー

```
P1 障害発生
    ↓
INC-YYYY-NNN 起票（即時）
    ↓
RTO 1h 以内に復旧（RUN-020〜022 実施）
    ↓
PM-YYYY-NNN 起票（48h 以内）
    ↓
Just Culture 分析・5 Whys・根本原因特定
    ↓
是正措置の確定（手順改訂が必要か判断）
    ↓ （手順改訂ありの場合）
CR-OPS-NNN 起票（PM 完了後 72h 以内）
    ↓
ドキュメント先行更新（版数パッチ or マイナー上げ）
    ↓
quality_admin 承認
    ↓
Git コミット（CR-OPS-NNN・PM-YYYY-NNN 参照）
```

### 4-3. 事後 ADR 記録

緊急変更が運用方針の不可逆変更を伴う場合（RTO 変更・SLO 変更・識別子体系変更等）は、**PM 完了後 72h 以内** に ADR-OPS-NNN を起票して事後記録する。

## 5. 変更ログテンプレート

各 md ファイルに設ける版数履歴節の標準形式を示す。

```markdown
## 版数履歴

| 版 | 日付 | 変更者 | 変更内容 |
|---|---|---|---|
| 0.1.0 | 2026-05-18 | RyuheiKiso | 初版 |
| 0.1.1 | YYYY-MM-DD | RyuheiKiso | CR-OPS-001: 誤字修正 |
| 0.2.0 | YYYY-MM-DD | RyuheiKiso | CR-OPS-002: CHK-029 追加（P1 ポストモーテム是正）PM-2026-001 参照 |
| 1.0.0 | YYYY-MM-DD | RyuheiKiso | 本番稼働開始・全手順初回実施完了 |
```

## 6. 文書間の版数整合

- `付録/99_運用保守識別子採番台帳.md` は全 CR-OPS/ADR-OPS の次採番値を管理する。
- 個別ファイルの版数は各ファイルの「版数履歴」節が管理する。
- `README.md` の「バージョン履歴」は 09 章全体の版数（最も大きなファイル版数に追随する）。
