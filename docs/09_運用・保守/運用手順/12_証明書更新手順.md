# 12 証明書更新手順

本手順書の責務は TLS 証明書の更新フローを確定することである。上流要件 BAT-009（30 日前アラート）・KEY-005（内部 CA）（`docs/04_概要設計/08_運用方式設計/07_アカウント・変更管理と運用手順.md`）を手順に具体化する。IPA 共通フレーム 2013「4.2.1.c 業務及びシステムの運用」に準拠する。

---

## 1. 目的と上流要件

| 属性 | 内容 |
|---|---|
| **頻度** | イベント駆動（BAT-009 が期限 30 日前アラートを発火した時） |
| **想定所要時間** | P50: 30 分 / P95: 60 分 |
| **実施権限** | system_admin（必須） |

上流要件:
- BAT-009: TLS 証明書の有効期限を毎日確認し、期限 30 日前に ALERT-XXX を発火すること
- KEY-005: 内部 CA（自己署名ルート）の管理・鍵ライフサイクルを定義すること

**TLS 証明書ライフサイクル:**

| 証明書種別 | 有効期間 | 更新トリガー |
|---|---|---|
| 内部 CA ルート証明書（KEY-005） | 10 年 | 期限 180 日前 BAT-009 アラート |
| サーバー証明書（nginx / IIS） | 1 年 | 期限 30 日前 BAT-009 アラート |

**本節で確定した方針**
- BAT-009 アラート発火後 7 日以内に証明書更新を完了することを確定する。
- 旧証明書はバックアップとして保管し、更新後 30 日が経過してから削除することを確定する。
- サーバー証明書の更新は停止なしで実施する（nginx / IIS のホットリロードを使用）ことを確定する。

---

## 2. 前提条件チェックリスト

以下をすべて確認してから手順を開始する。1 つでも NG なら手順を開始しない。

- [ ] BAT-009 のアラートが発火していることを確認している
- [ ] KEY-005（内部 CA 秘密鍵）が `/etc/wnav/keys/ca/` に存在する
- [ ] `openssl` コマンドが使用可能である
- [ ] nginx または IIS の設定ファイルへの書き込み権限がある
- [ ] 証明書バックアップ先 `/opt/wnav/certs/backup/` が存在する

**本節で確定した方針**
- 前提条件チェックリストに 1 つでも NG がある場合は手順を開始しないことを確定する。

## 3. 事前準備

- [CMD]
  ```bash
  # 現在の証明書情報を確認
  CERT_PATH="/etc/wnav/certs/server.crt"
  echo "Current certificate:"
  openssl x509 -in "${CERT_PATH}" -noout -subject -issuer -dates

  # 残り有効日数を計算
  END_DATE=$(openssl x509 -in "${CERT_PATH}" -noout -enddate | cut -d= -f2)
  END_EPOCH=$(date -d "${END_DATE}" +%s)
  NOW_EPOCH=$(date +%s)
  DAYS_LEFT=$(( (END_EPOCH - NOW_EPOCH) / 86400 ))
  echo "Days until expiry: ${DAYS_LEFT}"
  ```

- [CMD] 旧証明書のバックアップ
  ```bash
  BACKUP_DIR="/opt/wnav/certs/backup/$(date +%Y%m%d)"
  mkdir -p "${BACKUP_DIR}"
  cp /etc/wnav/certs/server.crt "${BACKUP_DIR}/server.crt.bak"
  cp /etc/wnav/certs/server.key "${BACKUP_DIR}/server.key.bak"
  echo "Backup created: ${BACKUP_DIR}"
  ```

**本節で確定した方針**
- 更新前に必ず旧証明書をバックアップすることを確定する。

## 4. 実施手順

以下の操作タグを使用する。
- `[CMD]` シェルコマンド（WSL2 + bash）
- `[SQL]` PostgreSQL クエリ（psql 経由）
- `[PS]` PowerShell（IIS / Windows Server 操作）
- `[GUI]` ブラウザ / Grafana / 管理 UI 操作
- `[CHECK]` 確認・検証操作

### 4.1 ステップ 1: 秘密鍵と CSR の生成

- [CMD]
  ```bash
  # 新しいサーバー秘密鍵を生成
  openssl genrsa -out /tmp/server-new.key 4096
  chmod 600 /tmp/server-new.key

  # CSR の生成
  openssl req -new \
    -key /tmp/server-new.key \
    -out /tmp/server-new.csr \
    -subj "/C=JP/ST=Tokyo/O=Work Navigation System/CN=wnav.local" \
    -addext "subjectAltName=DNS:wnav.local,DNS:localhost,IP:127.0.0.1"
  echo "CSR generated: $(openssl req -in /tmp/server-new.csr -noout -text | grep Subject:)"
  ```

- [CHECK] CSR が生成されており `-text` で内容が確認できること。

### 4.2 ステップ 2: 内部 CA による署名

- [CMD]
  ```bash
  # 内部 CA（KEY-005）でサーバー証明書に署名
  openssl x509 -req \
    -in /tmp/server-new.csr \
    -CA /etc/wnav/keys/ca/ca.crt \
    -CAkey /etc/wnav/keys/ca/ca.key \
    -CAcreateserial \
    -out /tmp/server-new.crt \
    -days 365 \
    -sha256 \
    -extfile <(printf "subjectAltName=DNS:wnav.local,DNS:localhost,IP:127.0.0.1\nkeyUsage=digitalSignature,keyEncipherment\nextendedKeyUsage=serverAuth")

  # 署名確認
  openssl x509 -in /tmp/server-new.crt -noout -dates -issuer -subject
  ```

- [CHECK] `Issuer` が内部 CA（KEY-005）の Subject と一致すること。
  `notAfter` が現在から 365 日後であること。

### 4.3 ステップ 3: nginx への証明書配備

- [CMD]
  ```bash
  # 新証明書を配備
  cp /tmp/server-new.crt /etc/wnav/certs/server.crt
  cp /tmp/server-new.key /etc/wnav/certs/server.key
  chmod 644 /etc/wnav/certs/server.crt
  chmod 600 /etc/wnav/certs/server.key

  # nginx 設定のテスト
  docker compose exec nginx nginx -t 2>&1

  # ホットリロード（停止なし）
  docker compose exec nginx nginx -s reload
  echo "nginx reloaded at $(date)"
  ```

- [CHECK] `nginx -t` が `syntax is ok` かつ `test is successful` を返すこと。

### 4.4 ステップ 4: IIS への証明書配備

- [PS]
  ```powershell
  # 新証明書のインポート
  Import-Module WebAdministration
  $certPath = "C:\wnav\certs\server-new.pfx"

  # pfx に変換（bash で実施済みの場合）
  # openssl pkcs12 -export -out server-new.pfx -inkey server-new.key -in server-new.crt

  $cert = Import-PfxCertificate -FilePath $certPath `
    -CertStoreLocation Cert:\LocalMachine\My `
    -Password (ConvertTo-SecureString "pfx-password" -AsPlainText -Force)

  Write-Host "Certificate imported: $($cert.Thumbprint)"

  # IIS サイトの SSL バインドを更新
  $binding = Get-WebBinding -Name "wnav" -Protocol "https"
  $binding.AddSslCertificate($cert.Thumbprint, "My")

  Write-Host "IIS SSL binding updated"
  ```

- [CHECK] IIS 管理コンソールで HTTPS バインドに新証明書が表示されること。

### 4.5 ステップ 5: 証明書配備確認

- [CMD]
  ```bash
  # nginx/IIS 経由での証明書確認
  echo | openssl s_client -connect localhost:443 -servername wnav.local 2>/dev/null \
    | openssl x509 -noout -dates -subject -issuer

  # 残り日数の再確認
  NEW_CERT_END=$(echo | openssl s_client -connect localhost:443 2>/dev/null \
    | openssl x509 -noout -enddate | cut -d= -f2)
  echo "New certificate expires: ${NEW_CERT_END}"
  ```

- [CHECK] `notAfter` が更新後の日付（現在から約 365 日後）であること。
  `Issuer` が内部 CA であること。

### 4.6 ステップ 6: ハンディ端末の証明書更新（内部 CA 更新時のみ）

内部 CA ルート証明書（KEY-005）を更新した場合のみ実施する。

- [CMD]
  ```bash
  # 新しい CA 証明書を APK/OTA 更新用に配置
  cp /etc/wnav/keys/ca/ca-new.crt /opt/wnav/assets/ca-bundle.crt
  echo "CA bundle updated for device distribution"
  ```

**本節で確定した方針**
- nginx / IIS ともにホットリロードで停止なしで証明書を更新することを確定する。
- 内部 CA 更新時はハンディ端末への CA 証明書配布も必ず実施することを確定する。

## 5. 合格基準

| CHK-ID | 基準 | 合否 |
|---|---|---|
| CERT-1 | 新証明書の `notAfter` が現在から 30 日以上先 | ☐ |
| CERT-2 | `Issuer` が内部 CA（KEY-005）と一致 | ☐ |
| CERT-3 | nginx が `nginx -t` で OK を返す | ☐ |
| CERT-4 | TLS 接続確認（`openssl s_client`）で新証明書が返る | ☐ |
| CERT-5 | 旧証明書がバックアップ保存されている | ☐ |

**本節で確定した方針**
- 全 5 項目が合格でなければ証明書更新完了と見なさないことを確定する。

## 6. 異常時の判断

| 事象 | 打ち切り条件 | 通知先 | 代替手順 |
|---|---|---|---|
| nginx が `nginx -t` でエラー | 即時ロールバック | system_admin | §8 のロールバック手順で旧証明書に戻す |
| IIS バインド更新が失敗 | 継続 | system_admin | PowerShell エラーログ確認・手動 MMC でバインド更新 |
| 証明書の Issuer が内部 CA でない | 即時打ち切り | system_admin | CSR の作成からやり直す |

**本節で確定した方針**
- nginx エラーは即時ロールバックとすることを確定する。

## 7. 終了条件と記録

- [CMD] 一時ファイルのクリーンアップ
  ```bash
  rm -f /tmp/server-new.key /tmp/server-new.csr /tmp/server-new.crt
  echo "Temp files cleaned"
  ```

- [SQL] maintenance_log への INSERT
  ```sql
  INSERT INTO maintenance_log (log_type, executed_at, executed_by, detail)
  VALUES (
    'cert_renewal',
    NOW(),
    'system_admin',
    '{"result": "pass", "cert_type": "server", "new_expiry": "2027-05-18", "nginx_reloaded": true, "iis_updated": true}'
  );
  ```

**本節で確定した方針**
- `maintenance_log` への記録なしに証明書更新完了と見なさないことを確定する。

## 8. ロールバック / 代替手順

nginx 証明書ロールバック:

- [CMD]
  ```bash
  # バックアップから旧証明書を復元
  BACKUP_DIR=$(ls -t /opt/wnav/certs/backup/ | head -1)
  cp "/opt/wnav/certs/backup/${BACKUP_DIR}/server.crt.bak" /etc/wnav/certs/server.crt
  cp "/opt/wnav/certs/backup/${BACKUP_DIR}/server.key.bak" /etc/wnav/certs/server.key
  docker compose exec nginx nginx -s reload
  echo "Certificate rolled back"
  ```

**本節で確定した方針**
- ロールバック後はバックアップから旧証明書を復元したことを `maintenance_log` に記録することを確定する。
