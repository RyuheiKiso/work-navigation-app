# 00 運用手順の位置づけと採番

本文書は `docs/09_運用・保守/運用手順/` ディレクトリ全体の採番規約・改訂管理規約・用語定義を確立する。
本ディレクトリ内のすべての手順書は本文書の規約に準拠する。
IPA 共通フレーム 2013 SLCP-JCF2013「4.2.1.a 運用の準備」に対応する。

### 1.2 本ディレクトリの責務

本ディレクトリは「誰が・いつ・何を・どのように実施するか」を操作レベルで確定する。
設計判断（Why）は上流の概要設計・要件定義文書に委ねる。
本ディレクトリは手順の具体化（How）に専念する。

**本節で確定した方針**
- 上流文書が設計根拠を担い、本ディレクトリが操作手順を担うことを確定する。
- 本ディレクトリの手順書は上流要件番号（NFR-OPS-*、NFR-AVL-*、NFR-SEC-*）を必ず参照することを確定する。
- 下流文書は存在せず、本ディレクトリが手順の末端実施ドキュメントであることを確定する。

## 2. 採番規約

### 2.1 手順書 ID（OPS-PROC-NNN）

定期・随時実施する標準手順書には `OPS-PROC-NNN` 形式の ID を付与する。

| 範囲 | 用途 |
|---|---|
| OPS-PROC-001〜010 | 定期・標準運用手順（番号付きファイル 01〜10 に対応） |
| OPS-PROC-011〜019 | 拡張運用手順（将来拡張用） |
| OPS-PROC-020〜099 | 予約（使用しない） |

OPS-PROC-NNN が付与されないファイル（11〜14、99）は手順 ID を持たない参考文書として扱う。

### 2.2 チェック項目 ID（CHK-NNN）

| 範囲 | 割当手順 |
|---|---|
| CHK-001〜006 | OPS-PROC-001（週次ヘルスチェック） |
| CHK-007〜009 | OPS-PROC-002（バックアップ確認） |
| CHK-010〜014 | OPS-PROC-003（リストア検証） |
| CHK-015〜042 | OPS-PROC-008（年次自己監査） |
| CHK-043〜 | 将来拡張用 |

### 2.3 関連 ID 体系

本ディレクトリの手順書が参照する他の識別子体系を以下に示す。

| プレフィックス | 意味 | 定義元 |
|---|---|---|
| BAT-NNN | バッチジョブ定義 | `docs/09_運用・保守/保守記録/` |
| ALERT-NNN | アラート定義 | Prometheus / Grafana アラートルール |
| ERR-NNN | エラーコード定義 | `docs/07_エラーコード/` |
| KEY-NNN | 鍵管理識別子 | `docs/04_概要設計/08_運用方式設計/` |
| NFR-OPS-NNN | 運用保守性非機能要件 | `docs/03_要件定義/非機能要件/04_運用保守性要件.md` |
| NFR-AVL-NNN | 可用性非機能要件 | `docs/03_要件定義/非機能要件/01_可用性要件.md` |
| NFR-SEC-NNN | セキュリティ非機能要件 | `docs/03_要件定義/非機能要件/05_セキュリティ要件.md` |

**本節で確定した方針**
- OPS-PROC-NNN・CHK-NNN の採番体系を本節で確定し、全手順書はこれに従うことを確定する。
- 関連 ID は上流文書の定義を参照し、本ディレクトリ内で独自定義しないことを確定する。

## 3. 改訂管理規約

### 3.1 バージョン体系

手順書のバージョンは `MAJOR.MINOR.PATCH` 形式とする。

| 変更種別 | バージョン操作 | 例 |
|---|---|---|
| 手順の本質的変更（ステップ追加・削除） | MAJOR 増分 | 1.0.0 → 2.0.0 |
| 手順の補足・コマンドオプション変更 | MINOR 増分 | 1.0.0 → 1.1.0 |
| 誤字修正・表記統一 | PATCH 増分 | 1.0.0 → 1.0.1 |

### 3.2 改訂時の義務事項

手順書を改訂する際は以下をすべて実施する。

- [ ] 手順書末尾の「9. 関連識別子・改訂履歴」テーブルに行を追加する
- [ ] `docs/09_運用・保守/保守記録/` の対応する保守記録ファイルに変更概要を記録する
- [ ] `maintenance_log` テーブルに改訂記録を INSERT する
- [ ] 改訂内容が NFR-OPS-* の要件変更を伴う場合は上流文書も同時に改訂する

```sql
-- 手順書改訂記録
INSERT INTO maintenance_log (log_type, executed_at, executed_by, detail)
VALUES (
  'proc_revision',
  NOW(),
  'system_admin',
  '{"proc_id": "OPS-PROC-NNN", "version_from": "x.y.z", "version_to": "a.b.c", "summary": "変更概要"}'
);
```

### 3.3 廃止規則

手順書を廃止する場合は削除せず、ファイル先頭に以下を追記してアーカイブする。

```markdown
> [!WARNING]
> この手順書は YYYY-MM-DD に廃止されました。後継: OPS-PROC-NNN を参照してください。
```

**本節で確定した方針**
- 改訂時は `maintenance_log` への INSERT を義務とすることを確定する。
- 手順書の廃止はファイル削除ではなくアーカイブマーキングで行うことを確定する。

## 4. 権限と実施体制

### 4.1 実施権限定義

| 権限ロール | 定義 | 実施可能な手順 |
|---|---|---|
| system_admin | システム管理者（本プロジェクト唯一の IT 担当） | 全手順 |
| quality_admin | 品質管理責任者（工場品質部門担当） | OPS-PROC-005（入退職）・SLO レポート確認 |
| 現場監督 | 製造ライン監督者 | 問い合わせ受付・停止予告の受領確認のみ |

### 4.2 単独実施と記録

本プロジェクトは 1 名体制（system_admin = RyuheiKiso）のため、すべての手順書は単独実施を前提とする。
OPS-PROC-008（年次自己監査）においては quality_admin によるレビューを必須とし、自己監査の客観性を担保する。

**本節で確定した方針**
- system_admin が全手順を単独実施する体制を確定する。
- 年次自己監査のみ quality_admin による独立レビューを必須とすることを確定する。

### 5.1 操作タグ規約

手順書の実施手順（§4）では以下のタグを使用する。

| タグ | 意味 |
|---|---|
| `[CMD]` | シェルコマンド（WSL2 + bash） |
| `[SQL]` | PostgreSQL クエリ（psql 経由） |
| `[PS]` | PowerShell（IIS / Windows Server 操作） |
| `[GUI]` | ブラウザ / Grafana / 管理 UI 操作 |
| `[CHECK]` | 確認・検証操作 |

**本節で確定した方針**
- 章立てテンプレートと操作タグ規約を本節で確定し、全 OPS-PROC 手順書はこれに従うことを確定する。
