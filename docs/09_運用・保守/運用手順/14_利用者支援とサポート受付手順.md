# 14 利用者支援とサポート受付手順

本文書の責務は利用者からの問い合わせ受付・分類・対応・クローズのフローを確定することである。
IPA 共通フレーム 2013 SLCP-JCF2013「4.2.1.d 顧客支援」に対応する成果物である。

---

## 1. 目的と位置づけ

| 属性 | 内容 |
|---|---|
| **対応 IPA タスク** | 4.2.1.d（顧客支援） |
| **頻度** | イベント駆動（問い合わせ発生時随時） |
| **実施権限** | system_admin（対応）/ quality_admin（SOP 改訂承認） |

利用者支援の対象:
- ハンディ端末操作の現場作業者（ハンディアプリに関する問い合わせ）
- 現場監督（SOP 閲覧・作業指示に関する問い合わせ）
- quality_admin（マスタメンテナンス UI に関する問い合わせ）

**本節で確定した方針**
- 問い合わせはすべて `support_requests` テーブルに記録することを確定する。
- 問い合わせ受領から一次回答まで 4 時間以内を対応 SLO とすることを確定する。
- 問い合わせを通じて判明した SOP 改訂の必要性は §4.4 の SOP 改訂申請フローで処理することを確定する。

---

## 2. 問い合わせカテゴリと優先度

| カテゴリ | 例 | 優先度 | 一次回答 SLO |
|---|---|---|---|
| ハンディ端末不具合 | 画面が固まる・Step が記録されない | 高 | 1 時間以内 |
| 認証エラー | ログインできない | 高 | 1 時間以内 |
| SOP 内容の疑問 | 手順書の指示が現場と合わない | 中 | 4 時間以内 |
| マスタデータ誤り | 品目名・工程名が間違っている | 中 | 4 時間以内 |
| 機能要望 | この機能を追加してほしい | 低 | 翌営業日 |
| その他 | — | 低 | 翌営業日 |

---

## 3. 問い合わせ受付フロー

```
現場作業者・現場監督
    ↓ 口頭 or メモ伝達
現場監督
    ↓ system_admin に報告
system_admin
    ↓ カテゴリ分類 → support_requests テーブルに登録
    ↓
    ├─ ハンディ不具合 → §4.3（ハンディ端末一次対応）
    ├─ 認証エラー → §4.2（認証エラー対応）
    ├─ SOP 内容疑問 → §4.4（SOP 改訂申請フロー）
    ├─ マスタデータ誤り → マスタメンテナンス UI で即時修正
    └─ 機能要望 → バックログ登録（GitHub Issues）
    ↓
クローズ → support_requests.status = 'resolved' に更新
    ↓
maintenance_log に記録（月次集計対象）
```

## 4. 実施手順

以下の操作タグを使用する。
- `[CMD]` シェルコマンド（WSL2 + bash）
- `[SQL]` PostgreSQL クエリ（psql 経由）
- `[PS]` PowerShell（IIS / Windows Server 操作）
- `[GUI]` ブラウザ / Grafana / 管理 UI 操作
- `[CHECK]` 確認・検証操作

### 4.1 ステップ 1: 問い合わせ受付・登録

- [SQL]
  ```sql
  INSERT INTO support_requests (
    reported_by,
    reported_at,
    category,
    description,
    priority,
    status,
    assigned_to
  )
  VALUES (
    '現場監督 氏名',              -- 報告者名
    NOW(),
    'handy_malfunction',           -- カテゴリ（§2 の分類に従う）
    '端末番号 DEVICE-001 が操作中に固まる。Step 完了ボタンを押すと応答なし。', -- 詳細
    'high',
    'open',
    'system_admin'
  )
  RETURNING id;
  ```

- [CHECK] INSERT が成功し `id` が返ること。

### 4.2 認証エラー対応

- [SQL]
  ```sql
  -- 対象作業者の JWT 状態を確認
  SELECT w.login_id, jt.jti, jt.issued_at, jt.expires_at, jt.revoked_at
  FROM workers w
  JOIN jwt_tokens jt ON w.id = jt.worker_id
  WHERE w.login_id = $1
    AND jt.expires_at > NOW()
    AND jt.revoked_at IS NULL;
  ```

- [CHECK] 有効 JWT が存在しない場合は新 JWT を発行する（OPS-PROC-005 §4.1.2 を参照）。

- [SQL] 認証失敗ログの確認
  ```sql
  SELECT event_code, count(*), max(created_at) AS last_attempt
  FROM audit_logs
  WHERE event_code LIKE 'ERR-AUTH-%'
    AND worker_id = (SELECT id FROM workers WHERE login_id = $1)
    AND created_at >= NOW() - INTERVAL '1 hour'
  GROUP BY event_code;
  ```

- [CHECK] ERR-AUTH-003（無効な kid）の場合は JWT 鍵ローテーション後に新 JWT を発行する（OPS-PROC-006 参照）。

### 4.3 ハンディ端末一次対応

ハンディ端末（Android / iOS / Windows）に関する不具合の一次対応手順を示す。

**対応レベル 1: アプリケーション再起動**

- [CHECK] 現場作業者にアプリを強制終了して再起動するよう指示する。
  再起動後に Step 完了が記録されることを確認する。

**対応レベル 2: ローカル DB クリア（Outbox の強制フラッシュ）**

ローカル SQLite に不整合データが存在するリスクがある際に実施する。

- [CHECK] 現場作業者に「設定 → ローカルデータリセット」の実行を指示する。
  リセット後に API と再同期されることを確認する（Outbox フラッシュ確認）。

- [SQL]
  ```sql
  -- 対象端末からの Outbox フラッシュを確認
  SELECT count(*) AS flushed
  FROM outbox_events
  WHERE device_id = 'DEVICE-001'
    AND status = 'delivered'
    AND created_at >= NOW() - INTERVAL '15 minutes';
  ```

**対応レベル 3: アプリ再インストール**

レベル 2 でも解消しない場合に実施する。

- [CHECK] MDM または手動で端末に最新ビルドを再インストールする。
  再インストール後に JWT を再発行し（OPS-PROC-005 §4.1.2）端末に配布する。

### 4.4 SOP 改訂申請フロー

現場から SOP の内容変更要求がある場合のフローを示す。

```
現場監督 → 口頭または書面で SOP 改訂を要求
    ↓
quality_admin が内容を審査（工程・品質基準との整合確認）
    ↓
quality_admin が承認 → system_admin が SOP マスタを更新
    ↓
system_admin が docs/保守記録/ に改訂記録を作成
    ↓
maintenance_log に proc_revision を記録
```

- [GUI] マスタメンテナンス UI → SOP 管理 → 対象 SOP を選択 → 内容を更新 → バージョンをインクリメント → 保存

- [SQL]
  ```sql
  -- SOP 改訂記録
  UPDATE sop_master
  SET title = '改訂後タイトル',
      content = '改訂後内容',
      version = version + 1,
      updated_at = NOW(),
      updated_by = 'system_admin',
      change_reason = 'quality_admin 承認による改訂'
  WHERE sop_id = 'SOP-XXX';
  ```

- [SQL]
  ```sql
  INSERT INTO maintenance_log (log_type, executed_at, executed_by, detail)
  VALUES (
    'proc_revision',
    NOW(),
    'system_admin',
    '{"proc_id": "SOP-XXX", "version_from": "1.0", "version_to": "1.1", "approved_by": "quality_admin", "summary": "改訂理由"}'
  );
  ```

### 4.5 問い合わせクローズ

- [SQL]
  ```sql
  UPDATE support_requests
  SET status = 'resolved',
      resolved_at = NOW(),
      resolution = '対応内容を記入'
  WHERE id = $1;  -- §4.1 で取得した id
  ```

- [CHECK] `status = 'resolved'` かつ `resolved_at` が設定されていること。

**本節で確定した方針**
- 問い合わせはカテゴリに関わらず `support_requests` テーブルに記録することを確定する。
- SOP 改訂は quality_admin の承認なしに実施しないことを確定する。

## 6. 月次集計とレポート

月次 SLO レポート（OPS-PROC-004）実施時に問い合わせ件数を集計する。

- [SQL]
  ```sql
  SELECT
    category,
    count(*) AS total,
    sum(CASE WHEN priority = 'high' THEN 1 ELSE 0 END) AS high_priority,
    avg(EXTRACT(EPOCH FROM (resolved_at - reported_at)) / 3600) AS avg_resolution_h
  FROM support_requests
  WHERE reported_at >= date_trunc('month', NOW() - INTERVAL '1 month')
    AND reported_at < date_trunc('month', NOW())
  GROUP BY category
  ORDER BY total DESC;
  ```

**本節で確定した方針**
- 月次集計結果を SLO レポートに含めることを確定する。

## 7. 終了条件と記録

月次定期作業として以下を実施する。

- [SQL] 月次サポート集計を maintenance_log に記録
  ```sql
  INSERT INTO maintenance_log (log_type, executed_at, executed_by, detail)
  VALUES (
    'support_monthly_summary',
    NOW(),
    'system_admin',
    '{"month": "2026-04", "total_requests": 3, "high_priority": 1, "avg_resolution_h": 2.5, "sop_revisions": 0}'
  );
  ```

**本節で確定した方針**
- `maintenance_log` への月次集計記録なしに月次サポート集計完了と見なさないことを確定する。
