# 99 前提制約と運用手順サブが約束しないこと

本文書は `docs/09_運用・保守/運用手順/` ディレクトリ全体のスコープ外事項を明示し、運用手順書群が約束しない事項を確定することである。
本文書に記載の事項は本プロジェクトの制約事項として正式に確定しており、将来の追加検討対象でもない。

---

## 1. 位置づけ

本文書は「できないこと・やらないことを明確にする」ことを目的とする。
スコープ外の事項を明示することで、システムを利用する現場監督・quality_admin が不合理な期待を持たないようにすることが本文書の唯一の責務である。

**本節で確定した方針**
- 本文書に記載の制約事項は設計判断として確定しており再検討の対象外とすることを確定する。

---

## 2. 体制制約

### 2.1 24 時間有人監視は対象外と判断する

本プロジェクトは 1 名体制（system_admin = RyuheiKiso）で運用される個人開発プロジェクトである。
夜間（22:00〜06:00）の有人監視は対象外と判断する。

夜間の障害検出は以下の手段に限定される:
- Prometheus + Grafana によるアラート（Grafana の通知機能）
- Windows Event Log への nxlog 記録（翌朝確認）
- ハンディ端末の Outbox タイムアウト検出（現場監督が気づく）

夜間に発生した障害の対応は翌朝 09:00 の週次ヘルスチェック（OPS-PROC-001）で検出し対応を開始する。

**本節で確定した方針**
- 24 時間有人監視は本プロジェクトの体制制約により対象外と判断することを確定する。
- 夜間障害の翌朝対応開始を前提とした SLO（可用性 99.5%）を設計判断として確定する。

---

## 3. 外部サービス制約

### 3.1 外部アラート SaaS（PagerDuty 等）は対象外と判断する

PagerDuty・Opsgenie・VictorOps 等の外部オンコール管理 SaaS の利用は対象外と判断する。

理由:
- 1 名体制のため On-Call エスカレーションポリシーの設定が不要
- コスト制約（個人開発プロジェクト）
- 外部 SaaS へのメトリクス・アラート情報の送信は情報管理ポリシー上の追加リスクを伴う

代替手段: Grafana の組み込みアラート機能（メール・Webhook）を使用する。

**本節で確定した方針**
- 外部アラート SaaS の利用は対象外と判断することを確定する。

### 3.2 外部 SIEM・ログ収集 SaaS は対象外と判断する

Datadog・Splunk・Elastic Cloud 等の外部ログ収集 SaaS の利用は対象外と判断する。
ログ収集は nxlog → Windows Event Log + `C:\Logs\wnav\` のオンプレミス構成に限定する。

**本節で確定した方針**
- 外部 SIEM・ログ収集 SaaS の利用は対象外と判断することを確定する。

---

## 4. 自動化制約

### 4.1 自動障害復旧スクリプト（ヒューマンインザループ必須）は対象外と判断する

障害発生時に人の判断なしに自動で本番環境を変更するスクリプト（自動フェイルオーバー・自動ロールバック等）は対象外と判断する。

理由:
- 自動復旧は想定外の障害シナリオで状態をさらに悪化させるリスクがある
- 製造ライン向けシステムであり誤動作が安全上の問題を引き起こすリスクがある
- 1 名体制のため自動化スクリプトのテスト・検証コストが高い

すべての本番環境への操作は system_admin の判断と手動実行を必須とする。
BAT-001〜010 のバッチジョブは「状態変更を伴わない読み取り・バックアップ・JWT 鍵ローテーション」に限定する。

**本節で確定した方針**
- 自動障害復旧スクリプトは対象外と判断することを確定する。
- 本番環境への状態変更操作はすべて system_admin が手動で実施することを確定する。

### 4.2 自動スケーリングは対象外と判断する

Kubernetes HPA・Docker Swarm 等による自動スケーリングは対象外と判断する。
本プロジェクトは単一工場・固定ユーザー数の構成であり、スケーリングの必要性は設計時点で発生しない。

**本節で確定した方針**
- 自動スケーリングは対象外と判断することを確定する。

---

## 5. 冗長化制約

### 5.1 地理冗長切替（マルチサイト）は対象外と判断する

複数の地理的サイト間での冗長化（マルチリージョン・マルチサイト）は対象外と判断する。
本プロジェクトは単一工場内のシステムであり、地理的冗長の要件は存在しない。

Active-Standby 構成（OPS-PROC-010）は同一工場内の 2 台のサーバー間での切替に限定する。

**本節で確定した方針**
- 地理冗長切替は対象外と判断することを確定する。

### 5.2 データベースの同期レプリケーション（Synchronous Replication）は対象外と判断する

PostgreSQL の同期ストリーミングレプリケーション（`synchronous_standby_names`）は対象外と判断する。
WAL アーカイブ（BAT-002）と日次バックアップ（BAT-001）の組み合わせで RPO ≤ 5 分を実現する。

**本節で確定した方針**
- 同期レプリケーションは対象外と判断することを確定する。

---

## 6. 対応時間制約

### 6.1 夜間・週末の緊急対応 SLO は設定しない

夜間（22:00〜06:00）および週末の緊急対応については SLO を設定しない。
1 名体制の制約により、夜間・週末の障害対応は翌営業日（または次の平日）の開始時に対応を開始する。

例外: CVSS 9.0+ の Critical 脆弱性は 72 時間以内（NFR-OPS-041）の対応を義務とする。
この場合は夜間・週末でも翌朝に対応を開始し 72 時間以内の完了を確保する。

**本節で確定した方針**
- 夜間・週末の緊急対応 SLO は設定しないことを確定する。
- Critical CVE のみ 72 時間以内対応を夜間・週末を含む絶対制約とすることを確定する。

## 7. スコープ外の明示一覧

以下の事項は本プロジェクトの運用手順書群の対象外であることを確定する。

| 事項 | 理由 |
|---|---|
| 24 時間有人監視 | 1 名体制（夜間無人） |
| 外部アラート SaaS（PagerDuty 等） | コスト・情報管理制約 |
| 外部 SIEM・ログ収集 SaaS | オンプレミス構成に限定 |
| 自動障害復旧スクリプト | ヒューマンインザループ必須 |
| 自動スケーリング | 単一工場・固定ユーザー数 |
| 地理冗長切替（マルチサイト） | 単一工場構成 |
| 同期レプリケーション | WAL + 日次バックアップで代替 |
| 夜間・週末の緊急対応 SLO | 1 名体制の制約 |
| ISO 27001 取得・外部審査 | 個人開発プロジェクト |
| SOC 2 Type 2 準拠 | 個人開発プロジェクト |

**本節で確定した方針**
- 上記 10 事項は本プロジェクトのスコープ外として確定し、将来追加検討の対象外とすることを確定する。

---

## 8. スコープ外を補完するリスク受容

上記のスコープ外事項によって発生するリスクは以下の方針で受容する。

| リスク | 受容方針 |
|---|---|
| 夜間障害の未検出（最大 8 時間） | 可用性 SLO を 99.5%（年間許容停止 43.8 時間）として設計し夜間未検出時間をバジェット内に含める |
| 自動復旧なしによる RTO 延長 | RTO ≤ 60 分を手動操作前提で達成できる設計（OPS-PROC-010）を確定する |
| 地理的単点障害 | 単一工場の製造ラインシステムとして地理的リスクは許容範囲と判断する |

**本節で確定した方針**
- 上記リスク受容は本プロジェクトの設計判断として確定することを確定する。
