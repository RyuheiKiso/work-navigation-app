# 11 計画停止と再開手順

本手順書の責務は月次計画停止・作業実施・システム再開の一連フローを確定することである。上流要件 NFR-AVL-003（`docs/04_概要設計/08_運用方式設計/07_アカウント・変更管理と運用手順.md`）を手順に具体化する。IPA 共通フレーム 2013「4.2.1.c 業務及びシステムの運用」に準拠する。

---

## 1. 目的と上流要件

| 属性 | 内容 |
|---|---|
| **頻度** | 月次（月 1 回・夜間 21:00〜23:00） |
| **想定所要時間** | P50: 60 分 / P95: 120 分（上限） |
| **実施権限** | system_admin（必須）/ quality_admin（通知受領確認） |

上流要件:
- NFR-AVL-003: 計画停止は月 1 回・夜間・最大 2 時間（120 分）・3 日前通知を要件とすること

**本節で確定した方針**
- 計画停止は 3 日前の通知なしに実施しないことを確定する。
- 停止時間の上限は 120 分とし超過が見込まれる場合は翌月以降に延期することを確定する。
- 停止前の事前バックアップ（OPS-PROC-002 §6.1 相当）を必ず実施することを確定する。

---

## 2. 前提条件チェックリスト

以下をすべて確認してから手順を開始する。1 つでも NG なら手順を開始しない。

- [ ] 3 日前に現場監督および quality_admin への停止通知が完了している
- [ ] 通知に対する quality_admin の承認が得られている
- [ ] 最新バックアップ（BAT-001）が正常に完了している（過去 25 時間以内）
- [ ] 作業内容（実施するメンテナンスの詳細）が確定している
- [ ] 停止時間の見積もりが 120 分以内である
- [ ] 作業時間帯が夜間（21:00〜23:00）の範囲内である

**本節で確定した方針**
- 前提条件チェックリストに 1 つでも NG がある場合は手順を開始しないことを確定する。

---

## 3. 事前準備（停止 3 日前）

### 3.1 停止通知の送付

- [CMD]
  ```bash
  # 計画停止通知を記録（nxlog 経由で Windows Event Log に記録）
  /opt/wnav/bin/notify-maintenance \
    --type "planned_maintenance" \
    --maintenance-date "$(date -d '+3 days' '+%Y-%m-%d')" \
    --window-start "21:00" \
    --window-end "23:00" \
    --message "月次計画停止: DB マイグレーション・パッチ適用等" \
    --notify-to "supervisor,quality_admin"
  ```

- [SQL]
  ```sql
  INSERT INTO maintenance_schedule (
    task_type, scheduled_at, window_end_at, description, status, created_by
  )
  VALUES (
    'planned_downtime',
    date_trunc('day', NOW() + INTERVAL '3 days') + INTERVAL '21 hours',
    date_trunc('day', NOW() + INTERVAL '3 days') + INTERVAL '23 hours',
    '月次計画停止: 作業内容を記入',
    'scheduled',
    'system_admin'
  );
  ```

### 3.2 事前バックアップ（停止当日実施）

- [CMD]
  ```bash
  # 計画停止前の緊急バックアップ（OPS-PROC-002 §6.1）
  DUMP_FILE="/backup/db/planned-maint/wnav-premaint-$(date +%Y%m%d).dump.gz"
  mkdir -p /backup/db/planned-maint/
  docker compose exec postgres pg_dump -U work_nav -Fc work_navigation \
    | gzip > "${DUMP_FILE}"
  openssl enc -aes-256-gcm \
    -kfile /etc/wnav/keys/backup-key.bin \
    -in "${DUMP_FILE}" -out "${DUMP_FILE}.enc"
  sha256sum "${DUMP_FILE}.enc" > "${DUMP_FILE}.enc.sha256"
  rm "${DUMP_FILE}"
  echo "Pre-maintenance backup: ${DUMP_FILE}.enc"
  ```

**本節で確定した方針**
- 停止 3 日前通知と停止当日の事前バックアップを両方完了してから停止フローに進むことを確定する。

## 4. 実施手順（計画停止フロー）

以下の操作タグを使用する。
- `[CMD]` シェルコマンド（WSL2 + bash）
- `[SQL]` PostgreSQL クエリ（psql 経由）
- `[PS]` PowerShell（IIS / Windows Server 操作）
- `[GUI]` ブラウザ / Grafana / 管理 UI 操作
- `[CHECK]` 確認・検証操作

### 4.1 ステップ 1: maintenance_mode 設定

- [SQL]
  ```sql
  UPDATE system_settings
  SET value = 'true', updated_at = NOW(), updated_by = 'system_admin'
  WHERE key = 'maintenance_mode';
  ```

- [CHECK] ハンディ端末が「メンテナンス中」表示に切り替わること。

### 4.2 ステップ 2: Outbox のフラッシュ確認

- [SQL]
  ```sql
  -- 保留中のイベントがすべて処理されるまで待機
  SELECT count(*) AS pending FROM outbox_events WHERE status = 'pending';
  ```

- [CHECK] `pending = 0` になってからサービスを停止する。
  5 分待っても 0 にならない場合は強制処理を実施する。

### 4.3 ステップ 3: コンテナ停止

- [CMD]
  ```bash
  STOP_START=$(date +%s)
  echo "PLANNED_STOP_START=$(date)" | tee /tmp/planned-maint-$(date +%Y%m%d).log

  cd /opt/wnav

  # API・UI を先に停止（DB は最後）
  docker compose stop api master-ui
  sleep 10
  docker compose stop prometheus grafana
  docker compose stop postgres

  echo "All containers stopped: $(date)" | tee -a /tmp/planned-maint-$(date +%Y%m%d).log
  ```

- [CHECK] `docker compose ps` で全コンテナが `exited` または `stopped` であること。

### 4.4 ステップ 4: 作業実施

計画された作業を実施する。作業内容は `maintenance_schedule` テーブルの `description` を参照する。

作業例:
- DB マイグレーション（`sqlx migrate run`）
- 設定ファイルの更新（Docker Compose / nginx / IIS）
- OS パッチの適用（OPS-PROC-009 に該当しない High/Medium レベル）
- ログファイルのローテーションとアーカイブ

- [CMD]（DB マイグレーションの例）
  ```bash
  # PostgreSQL のみ起動して DB マイグレーションを実施
  docker compose up -d postgres
  sleep 15
  cd /opt/wnav/backend
  sqlx migrate run --database-url "postgres://work_nav:password@localhost:5432/work_navigation" \
    2>&1 | tee -a /tmp/planned-maint-$(date +%Y%m%d).log
  echo "Migration exit: $" | tee -a /tmp/planned-maint-$(date +%Y%m%d).log
  ```

- [CHECK] マイグレーションが exit 0 で完了すること。

### 4.5 ステップ 5: システム再起動

- [CMD]
  ```bash
  cd /opt/wnav
  docker compose up -d
  sleep 30
  docker compose ps
  ```

### 4.6 ステップ 6: 健全性確認

- [CMD]
  ```bash
  curl -fsS http://localhost:8080/health | jq .
  pg_isready -h localhost -p 5432 -U work_nav
  ```

- [SQL]
  ```sql
  -- DB の健全性確認
  SELECT count(*) AS active_workers
  FROM workers WHERE status = 'active';
  ```

- [CHECK] API が `{"status":"ok"}` を返すこと。
  `pg_isready` が `accepting connections` を返すこと。

### 4.7 ステップ 7: maintenance_mode 解除と再開通知

- [SQL]
  ```sql
  UPDATE system_settings
  SET value = 'false', updated_at = NOW(), updated_by = 'system_admin'
  WHERE key = 'maintenance_mode';
  ```

- [CMD]
  ```bash
  STOP_END=$(date +%s)
  STOP_DURATION=$((STOP_END - STOP_START))
  echo "PLANNED_STOP_END=$(date)" | tee -a /tmp/planned-maint-$(date +%Y%m%d).log
  echo "STOP_DURATION_SEC=${STOP_DURATION}" | tee -a /tmp/planned-maint-$(date +%Y%m%d).log

  /opt/wnav/bin/notify-maintenance \
    --type "maintenance_complete" \
    --message "計画停止が完了しました。業務を再開してください。停止時間: $((STOP_DURATION/60))分"
  ```

- [CHECK] 停止時間が 120 分（7200 秒）以内であること。

**本節で確定した方針**
- maintenance_mode の設定・解除は必ず SQL で実施し記録することを確定する。
- 停止時間が 120 分を超えた場合は超過分を `maintenance_log` に記録することを確定する。

## 5. 合格基準

| CHK-ID | 基準 | 合否 |
|---|---|---|
| PM-1 | 事前通知が 3 日前以上に完了している | ☐ |
| PM-2 | 事前バックアップが正常完了している | ☐ |
| PM-3 | 作業が exit 0 で完了している | ☐ |
| PM-4 | API ヘルスチェック 200 OK | ☐ |
| PM-5 | `maintenance_mode = false` に戻っている | ☐ |
| PM-6 | 停止時間 ≤ 7200 秒（120 分） | ☐ |

**本節で確定した方針**
- 全 6 項目が合格でなければ計画停止・再開手順完了と見なさないことを確定する。

## 6. 異常時の判断

| 事象 | 打ち切り条件 | 通知先 | 代替手順 |
|---|---|---|---|
| DB マイグレーションが FAIL | 即時ロールバック・再起動 | system_admin | ロールバックマイグレーション（`sqlx migrate revert`）を実施 |
| 停止時間が 120 分超過見込み | 作業を中断して再起動 | system_admin・quality_admin | 未完了作業を翌月の計画停止に延期 |
| 再起動後 API が起動しない | 継続（最大 30 分待機） | system_admin | OPS-PROC-010 フェイルオーバーを起動 |

**本節で確定した方針**
- 停止時間 120 分超過が確実な場合は作業を中断し翌月に延期することを確定する。

## 7. 終了条件と記録

- [SQL] maintenance_log への INSERT
  ```sql
  INSERT INTO maintenance_log (log_type, executed_at, executed_by, detail)
  VALUES (
    'planned_downtime',
    NOW(),
    'system_admin',
    '{"result": "pass", "stop_duration_sec": 3600, "work_done": "DB migration v012", "pre_backup": true}'
  );
  ```

**本節で確定した方針**
- `maintenance_log` への記録なしに計画停止手順完了と見なさないことを確定する。

## 8. ロールバック / 代替手順

DB マイグレーションのロールバック:

- [CMD]
  ```bash
  cd /opt/wnav/backend
  # 直前のマイグレーションを 1 ステップ戻す
  sqlx migrate revert --database-url "postgres://work_nav:password@localhost:5432/work_navigation"
  ```

- [CHECK] ロールバック後に API の起動確認を実施する。

**本節で確定した方針**
- マイグレーションロールバック後は `cargo test` を実施しデータ整合性を確認することを確定する。
