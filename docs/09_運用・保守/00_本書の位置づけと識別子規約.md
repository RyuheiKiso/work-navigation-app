# 00 本書の位置づけと識別子規約

本章は `09_運用・保守` フェーズの位置づけ・IPA 共通フレーム 2013 タスクカバレッジ・上流フェーズとの責任分界・新規識別子プレフィックスを確定する権威文書である。運用保守フェーズで使用する全識別子の定義と採番ルールを制定し、上流継承識別子の再採番を禁止する。

---

## 1. 09_運用・保守 の位置づけ

`09_運用・保守` は SLCP-JCF2013「**4.2.1 運用プロセス**」「**4.2.2 保守プロセス**」「**6.5.1 構成管理**」「**6.5.8 問題解決**」に準拠した最終フェーズ文書群である。

本書（09 章）は「確定した運用方式設計を実行可能な手順・記録に落とし込む」責務を持つ。「運用方式をどう設計するか」は `04_概要設計/08_運用方式設計/` の責務であり、本章はその設計を変更しない。

| 軸 | 内容 |
|---|---|
| **本書の責務** | 実行可能な手順・チェックリスト・記録テンプレートの確定 |
| **上流の責務** | SLO 値・OPS-PROC 定義・アーキテクチャ制約の確定（変更不可） |
| **下流** | なし（本フェーズが最終フェーズ）|

**本節で確定した方針**
- **09_運用・保守 は SLCP-JCF2013 4.2.1/4.2.2 に準拠した最終フェーズ文書群である。**
- **設計変更の権限を持たない: SLO・OPS-PROC 定義・アーキテクチャは 04_概要設計 が権威である。**
- **本書の責務を「実行可能な手順・記録の確定」に限定する。**

## 5. 上流フェーズとの責任分界

本節は 08_移行 以前の上流フェーズと 09_運用・保守 の責任境界を確定する。

| 事項 | 責任フェーズ | 09 章の関与 |
|---|---|---|
| SLO 値（99.5%・RTO 1h・RPO 15min）の確定 | `03_要件定義` NFR-AVL/OPS | 参照のみ。変更しない |
| OPS-PROC-001〜010 の手順概要定義 | `04_概要設計/08_運用方式設計` | 詳細化・実体化。概要は変更しない |
| BAT/JOB/LOG/MET/ALERT/ERR 識別子の定義 | `04_概要設計` | 参照のみ |
| KEY-001〜009 の鍵定義 | `04_概要設計/07_セキュリティ方式設計` | 運用手順（ローテーション等）を担当 |
| 本番移行完了判定 | `08_移行` | 継承した稼働前提を受け取る |
| 実行可能な手順書の確定 | **`09_運用・保守`（本書）** | 権威 |
| 障害記録・ポストモーテム記録 | **`09_運用・保守`（本書）** | 権威 |
| 運用判断 ADR（ADR-OPS-NNN）| **`09_運用・保守`（本書）** | 権威 |

**本節で確定した方針**
- **SLO 値・設計パラメータは上流フェーズが権威であり、本章は変更しない。**
- **実行可能な手順・記録・運用 ADR は 09 章が権威である。**
- **境界を越えた変更は CR-OPS-NNN を起票して上流フェーズを更新する手順を踏む。**

## 6. 09_運用・保守 新規プレフィックス定義

`09_運用・保守` フェーズで新規に採番する識別子は以下の 7 種とする。各プレフィックスの具体的採番は `付録/99_運用保守識別子採番台帳.md` が権威台帳である。

| プレフィックス | 対象 | 採番方式 | 欠番 | 権威台帳 |
|---|---|---|---|---|
| `CHK-NNN` | チェックリスト項目（週次・月次・年次定期確認項目）| 3 桁ゼロ埋め連番（001 開始）| 禁止 | 付録/99 §1 |
| `RUN-NNN` | 障害対応ランブック（切り分け・復旧手順単位）| 3 桁ゼロ埋め連番（001 開始）| 禁止 | 付録/99 §2 |
| `INC-YYYY-NNN` | 個別障害記録（発生年度 + 3 桁連番）| YYYY は西暦 4 桁、NNN は年度内連番（001 開始）| 禁止 | 付録/99 §3 |
| `PM-YYYY-NNN` | ポストモーテム記録（発生年度 + 3 桁連番）| YYYY は西暦 4 桁、NNN は年度内連番（001 開始）| 禁止 | 付録/99 §4 |
| `REC-NNN` | 保守記録（定期保守・修正・依存更新の記録単位）| 3 桁ゼロ埋め連番（001 開始）| 禁止 | 付録/99 §5 |
| `CR-OPS-NNN` | 運用フェーズ変更要求（手順書・設定変更の変更要求）| 3 桁ゼロ埋め連番（001 開始）| 禁止 | 付録/99 §6 |
| `ADR-OPS-NNN` | 運用判断 ADR（運用フェーズ固有の Architecture Decision Record）| 3 桁ゼロ埋め連番（001 開始）| 禁止 | 付録/99 §7 |

## 7. 採番ルール

### 7-1. 一般規則

- `NNN` は **3 桁ゼロ埋め**（001〜999）とする。
- 欠番は **禁止** する。廃止した識別子には `[廃止: YYYY-MM-DD 理由]` を付与し、番号を再利用しない。
- 大文字・ハイフン区切りを厳守する（例: `CHK-001`、`RUN-023`、`ADR-OPS-007`）。
- 識別子は台帳（付録/99）への登録前に使用しない（未登録識別子の参照は不正）。

### 7-2. INC/PM の年度付き採番

```
INC-2026-001  # 2026 年度 1 件目の障害記録
INC-2026-002  # 2026 年度 2 件目の障害記録
INC-2027-001  # 2027 年度は 001 から再開始
PM-2026-001   # 2026 年度 1 件目のポストモーテム
```

- 年度は `YYYY`（西暦 4 桁）とする。
- 年度をまたいだ継続インシデントは発生年度の識別子を維持する。

### 7-3. ポストモーテム物理配置（OPS-064）

ポストモーテムの個別記録は `docs/ポストモーテム/YYYY-MM-DD_PX_<slug>.md` に配置する（OPS-064 による確定）。
索引・手順・テンプレートは `09_運用・保守/障害対応/11_ポストモーテム手順とテンプレート.md` に集約する。
この物理配置の判断根拠は ADR-OPS-001 に記録する。

**本節で確定した方針**
- **09 章新規識別子は CHK/RUN/INC/PM/REC/CR-OPS/ADR-OPS の 7 種とし、全て欠番禁止とする。**
- **INC・PM は年度付き採番（YYYY-NNN）とし、年度をまたいで 001 から再開始する。**
- **ポストモーテム個別記録の物理配置は docs/ポストモーテム/ と確定し、ADR-OPS-001 に根拠を記録する。**
