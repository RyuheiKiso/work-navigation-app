# 06 解決アプローチの基本思想

本章の責務は、本アプリが採用する解決の根本的な考え方――「何を・なぜ・どのように」という思想の骨格――を確定することである。実装の詳細、データモデル、API設計はすべて計画書（docs/02_企画/システム化計画/）に委ねる。本章は「Why」の層だけを担う。

---

## H2-1. JIS Z 8141 作業階層の電子化

### 標準定義の採用根拠

[`90_業界分析/01_作業の定義と分類.md`](../../90_業界分析/01_作業の定義と分類.md) が示すように、JIS Z 8141:2001「生産管理用語」は生産管理の標準用語を定め、製造現場に共通の語彙を与えている。本アプリはこの標準を作業階層の電子化における一次的な参照規格として採用する。

JIS Z 8141 に基づく作業階層は以下の通りである。

| 階層 | JIS定義 | 本アプリにおける意味 |
|---|---|---|
| 工程（Process） | 原材料を製品に変換する系統的な一連の活動 | 製造ラインの1つのステーション全体 |
| 作業（Operation） | ある目的のために人・機械が行う一定のまとまりのある活動 | 工程内で完結する意味を持つ行為単位 |
| 要素作業（Element） | 作業を構成する最小単位 | 計測・判定・記録ができる単位 |
| ステップ（Step） | 本アプリが追加定義する最小実行単位 | 作業員が1動作で完了できる行為 |

この4層構造を電子化する思想的根拠は「再利用と版管理の実現」にある。紙のSOP（Standard Operating Procedure）では各層の境界が曖昧になりやすく、改訂が全文置き換えを強いる。電子化によってStep単位・要素作業単位での改訂が可能になり、変更影響の範囲が局所化される。

### IE 時間研究との接続

IE（Industrial Engineering）において時間研究はMTM（Methods-Time Measurement）やPTS（Predetermined Time Standards）を用いて最小分析単位を測定する。本アプリが定義するStepは、このIEの最小分析単位に対応する。すなわちStepはトレサビ記録の最小粒度であると同時に、将来的なTDABC（Time-Driven Activity-Based Costing）への入力値としても機能しうる。ただし時間研究の詳細な適用は本アプリのスコープ外とする。

### 階層化の利点：情報密度の習熟度制御

Carroll（1990）のMinimalism設計が示す通り、熟練者と初心者では必要な情報量が根本的に異なる。4層構造を持つことで、「工程一覧」「作業一覧」「ステップ詳細」という粒度の切り替えが可能になる。熟練作業員は工程・作業レベルだけを参照し、習熟途上の作業員はStepレベルの詳細ガイドを見る。この習熟度別情報密度の制御こそ、電子化がもたらす最大の価値である。

---

**本節で確定した方針**

- 作業階層の定義基準としてJIS Z 8141を採用することを確定する。
- 本アプリが扱う最小粒度をStep（要素作業より細かい1動作単位）と定義することを確定する。
- 習熟度に応じた情報密度制御をアプリ設計の中核原則として確定する。

## H2-2. Navigation と Traceability の不可分性

### 不可分統合の思想的根拠

「ナビゲーション機能」と「トレーサビリティ記録機能」を別々のシステムとして後から統合することは、設計の失敗と定義する。これは原理的な問題である。後付け統合では2つの問題が不可避的に発生する。第一に記録タイミングのズレ、第二に記録責任の曖昧化である。

[`90_業界分析/06_品質管理とトレーサビリティ.md`](../../90_業界分析/06_品質管理とトレーサビリティ.md) が指摘するように、「記録の完全性」と「品質の実質的保証」は別物である。トレサビ記録が「形式的な記録の蓄積」にとどまる最大の原因は、記録タイミングが作業完了の後にまとめて行われる運用設計にある。

### Navigation なき Traceability の帰結

「記録システムだけを導入し、ナビゲーションはしない」設計では、作業員は手順を頭の中か紙の手順書で確認し、完了後に別画面で記録を入力する。この設計が生む構造的問題は以下である。

| 問題 | 内容 |
|---|---|
| 事後入力の常態化 | 作業を終えてから記録する → 手順順序と記録順序がずれる |
| 改ざん経路の温存 | まとめて入力する余地 → 結果を確認してから入力値を操作できる |
| 記録モレの発生 | ステップを飛ばしても即座に気づかれない |

この構造は、記録システムを導入してもALCOA+（後述）の要件を実質的に充足できない典型的な失敗パターンである。

### Traceability なき Navigation の帰結

「ナビゲーションだけを提供し、記録は紙で行う」設計も同様に不完全である。作業を支援しても証拠を残さなければ、ALCOA+のContemporaneous原則（同時記録）を構造的に違反する。この違反は、問題発生後の遡及調査を不可能にするだけでなく、顧客監査・内部監査においても証拠能力を持たない。

### 統合の帰結：Contemporaneous 原則の構造的充足

Navigation と Traceability を統合すると、以下の構造が生まれる。

1. 作業員はステップを1つずつ画面に従って実行する
2. ステップ完了時に記録入力画面が自動的に現れる
3. 記録しなければ次のステップに進めない（ソフトロック）

このフロー設計によって、「作業とほぼ同時に記録が生成される」という状態が構造的に実現される。Contemporaneous原則は「思い出して後で記録する」ことを根本から不可能にする設計によってのみ充足される。

### Attributable 原則の構造的充足

作業員はアプリへのログインによって識別されるため、そのセッション中に生成されるすべての記録は自動的に特定の作業員に帰属する。これはALCOA+のAttributable原則（誰がいつ記録したかが明確）の構造的充足である。

---

**本節で確定した方針**

- NavigationとTraceabilityを別システムとして後から統合する設計思想を明示的に否定することを確定する。
- Step完了時に記録入力が自動的に要求されるフロー設計を採用することを確定する。
- ログイン済み作業員への記録の自動帰属をAttributable原則の実装手段として確定する。

## H2-3. ALCOA+ 9原則の採用宣言

### 採用の意味

本アプリはALCOA+を電子記録の品質基準として採用する。ALCOA+はもともとGMP（Good Manufacturing Practice）環境の電子記録品質論として医薬品・食品規制業界で発展したが、本アプリはこれを「GMP準拠のため」ではなく「証拠品質論の汎用規格」として採用する。すなわちGMPへの準拠を目的とするのではなく、証拠として機能するデータを生成するための設計原則として採用するという意味である。この汎用採用により、本アプリは食品・化学・機械・電子機器など業種を問わず適用可能な品質基準を持つ。

### 9原則と設計への影響

ALCOA+の9原則それぞれが、本アプリの設計思想に与える制約を以下に示す。詳細なデータ層・UI層・運用層へのマッピングは計画書06章で確定する。

| 原則 | 定義 | 設計思想への影響 |
|---|---|---|
| Attributable（帰属明確性） | 誰がいつ記録したか | ログイン必須・セッション紐付け |
| Legible（判読性） | 読める・見つけられる | 文字化け不可・長期保存時の可読性 |
| Contemporaneous（同時性） | 作業と同時に記録 | Step完了即時記録フロー |
| Original（原本性） | 最初の記録が保存される | 追記専用・上書き禁止 |
| Accurate（正確性） | 記録が実態を正確に反映 | 測定値の範囲チェック・異常値フラグ |
| Complete（完全性） | 必要な記録が全部そろう | Step未完了のままでは終了させない |
| Consistent（一貫性） | 矛盾しない記録の連続性 | タイムスタンプの単調増加を保証 |
| Enduring（耐久性） | 長期間保存できる | PostgreSQLの永続化・バックアップ |
| Available（可用性） | 必要なときに参照できる | オフライン時でもローカルに記録が存在 |

### ALCOA+ と Offline-First の緊張と解決

Contemporaneous原則（同時記録）とオフライン環境には構造的な緊張がある。「サーバーに記録を送信できない状態でも、作業と同時の記録とみなせるか」という問いである。

本アプリはこの緊張を以下の思想で解決する。「同時記録」とは「サーバーへの同時送信」ではなく「作業の場での同時入力」を意味する。端末のローカルストレージへの即時書き込みがContemporaneous原則を充足し、その後のサーバー同期はデータの伝達であって記録行為の完了ではない。端末タイムスタンプとサーバー時刻同期後の整合によって、記録の時系列の正確性を事後的に確認できる設計とする。

---

**本節で確定した方針**

- ALCOA+9原則を「GMP準拠のため」ではなく「証拠品質論の汎用規格」として採用することを確定する。
- 9原則すべてを設計制約として位置づけることを確定する。
- ローカル即時記録をContemporaneous原則の充足と定義し、サーバー同期を「伝達」として区別することを確定する。

## H2-4. Offline-First を倫理として採用する思想

### Offline-First の定義

Offline-Firstとは、ネットワーク接続がなくても作業継続・記録入力が可能な設計を最優先とすることである。ネットワーク接続を「あれば便利なもの」ではなく「なくても動くもの」として設計する。

[`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) が示すように、社内LANでも工場内の金属構造物による電波遮蔽、端末の再起動、サーバー側のメンテナンスによって接続が途切れる状況は必然的に発生する。「LANがあるから常時接続前提」という判断は、現場の実態を無視した設計である。

### 倫理としての採用

Offline-Firstを「技術的な対応策」ではなく「倫理的なコミットメント」として採用する。その意味は以下である。

製造現場の作業員は、電波が届かない場所でも仕事をしなければならない。その状況でアプリが使えなくなれば、作業員は「記録できないからあとでまとめて入力する」という行動を取らざるを得ない。この行動がContemporaneous原則の違反を生む。すなわちオフライン対応の欠如は、記録の品質低下を現場の作業員のせいにする設計である。

記録の連続性を保証する責務はシステム側にある。この責務を「電波が届く場所で使ってください」という運用規定でごまかすことを倫理的に許容しない、というのが本アプリの立場である。この立場がALCOA+のAvailable原則への倫理的コミットとして結実する。

### Local-First の思想的根拠

Kleppmannら（2019）の論文「Local-First Software」は、クラウドへの依存を減らし、データの一次ソースをローカルデバイスに置くことを提唱した。7つの理想のうち本アプリに特に関係するのは以下である。

| 性質 | 内容 | 本アプリへの意味 |
|---|---|---|
| 高速 | ローカルデータへのアクセスは即座 | 記録入力の応答速度がネットワークに依存しない |
| オフライン動作 | ネットワーク不要で完全動作 | 電波死角でも継続して作業・記録できる |
| 永続性 | ベンダー撤退でもデータが残る | オンプレのPostgreSQLが担保する |

### オフライン整合の技術思想

詳細は計画書05章・06章に委ねるが、技術思想の核心を述べる。追記専用集合（append-only log）による整合が基本設計である。作業完了ログは上書きされない。オフライン中にローカル端末で積み上げられたイベントは、再接続後にサーバーとマージされる。マージは和集合（union）として計算できるため、競合が原理的に発生しない。

Shapiroら（2011）のCRDT（Conflict-free Replicated Data Type）理論において、G-Set（追記専用集合）はこの性質を最もシンプルに体現するデータ型である。

### BCP との接続

オフライン設計はBCP（Business Continuity Planning）の要件でもある。通信断・電源断・災害時にも記録が継続できる設計は、製造現場の信頼性要件の核心である。作業記録はいかなる状況でも「工場に残る」ことを保証する。これはオンプレ展開という選択と不可分に結びついている。

---

**本節で確定した方針**

- Offline-Firstを技術的対応ではなく倫理的コミットメントとして採用することを確定する。
- ローカル端末を記録の一次ストレージとして設計することを確定する。
- 追記専用集合（append-only log）によるオフライン整合を技術思想として確定する。

## H2-5. 「やらないことの明確化」を思想として採用する

### 妥協禁止ポリシーの具体的意味

本アプリのポリシーは「妥協禁止・完璧かつ至高」である。この言葉を「あらゆる機能を実装する」と解釈することは誤りである。正確な意味は「やらないことを明確にした上で、やることを完全に実装する」である。

半完成の機能が多数あるシステムと、絞られた機能が完全に実装されたシステムでは、証拠品質として機能するのは後者だけである。やらないことを決めることは、逃げではなく思想的な積極的選択である。

### やらないことの原則的基準

本アプリのやらないことの基準は明確である。単一工場・社内LAN・オンプレという利用環境の外への設計はしない。この基準から以下の「対象外判断」が導かれる。

**AI/ML機能をver1.0.0で対象外と判断する根拠は以下の3点である。**

第一に、オンプレ環境での推論基盤が未整備である。GPU・推論サーバー・モデル管理基盤をオンプレに構築することは、本アプリの導入コストを製造業の許容範囲を超えて引き上げる。

第二に、学習データが存在しない初期状態での予測精度が証拠品質に寄与しない。ver1.0.0導入時点では作業実績データがゼロである。学習データなしの予測は証拠品質を高めず、むしろ誤った予測が記録の信頼性を損なうリスクがある。

第三に、EU AI Actが示す高リスクAI規制への対応が過剰負荷になる。製造現場の作業支援に予測・最適化・スコアリングを適用する場合、EU AI Act（2024年施行）の高リスクAIとしての規制対応が必要となりうる。個人開発・ver1.0.0という段階でこの対応コストを負担することは合理的でない。

**SaaS化・複数工場対応・規制業界拡張を対象外と判断する根拠は以下である。**

[`90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md`](../../90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md) が示すように、グローバルMESベンダーと国内SaaSが等しく対応できていない空白地帯は「中堅・中小製造業×オンプレ強制環境」の交差領域にある。本アプリが最大価値を生む場所はこのセグメントである。SaaS化・複数工場対応・GMP/IATF規制拡張を追求することは、この空白地帯から離れてすでに競合が多い領域へと向かうことを意味する。

やらないことを明確にすることが、やることの完全性を担保する。

---

**本節で確定した方針**

- AI/ML機能をver1.0.0のスコープ外と判断することを確定する。
- SaaS化・複数工場対応・規制業界（GMP/IATF）拡張をver1.0.0のスコープ外と判断することを確定する。
- 単一工場・社内LAN・オンプレという利用環境の外への設計を行わないことを確定する。
