# 11 前提・制約・本書が約束しないこと

本章の責務は、本構想書の全章が依拠する世界観前提を確定し、思想的制約を宣言し、構想の境界外を明示し、将来にわたって提供しない領域を永続的に宣言することである。本章は計画書 11章（前提・制約・リスク）が扱う業務的・技術的前提とは異なる「思想的前提」と「倫理的宣言」を担う。計画書 11章の内容と重複しない責務分担を明確にしながら、構想書として確定すべき前提と制約を記述する。

---

## 1. 世界観前提

本書の全章が依拠する世界観前提を確定する。これらは業務的な前提条件（計画書 11章の P01〜P07）でも技術的前提条件（計画書 11章の T01〜T05）でもなく、本書が「こうである」と認識する現実世界の構造に関する判断である。世界観前提が崩れた場合、本書の Why 宣言全体の見直しが必要となる。

### P01 中堅製造業空白領域の存在

グローバルMES（SIMATIC IT・Opcenter・Plex等）と国内SaaS型作業ナビは、中小・中堅製造業（従業員数おおむね10〜300名規模）における未充足のセグメントを残している。大規模MESは導入コスト・カスタマイズ期間・運用専門性において過剰設計であり、SaaS型作業ナビは情報越境リスク・稟議文化・社内LAN信仰により採択されにくい（業界分析 30章参照）。

この空白は単なる市場の失敗ではなく、製造品質証拠文化の欠缺という社会的問題として存在する。中小企業白書（2023年版）が示す電子化率約35%という数値は、この空白の実態を示す根拠として採用する。

### P02 SaaS否定環境の不可逆性

情報越境リスクに対する中小製造業の感受性・年度予算制度とサブスクリプション型OPEXの相性の悪さ・「社内LANの外にデータを出さない」という組織文化は、中期的（5〜10年）に継続すると判断する。デジタル庁の推進するクラウドファーストは大企業・行政を主対象としており、社内LAN専用環境を選好する中堅製造業の調達構造は不可逆のトレンドとして継続する（業界分析 30章参照）。

この前提が崩れる可能性（例：クラウド規制の緩和・情報越境リスクの技術的解決）は排除しないが、本書が対象とする中期的な事業環境においては本前提を採用する。

### P03 労働力多様化の加速

2024年入管法改正による育成就労制度の創設は、外国人技能実習制度の構造的問題への政策的応答であり、外国人労働者の転籍要件緩和・権利保護強化を通じて製造業への外国人労働者流入をさらに促進する方向を示す（業界分析 34章参照）。ベトナム・中国・フィリピン・ブラジル等の出身国籍を持つ製造業従事者は2023年時点で約45万人に達しており、この流入は不可逆のトレンドである。

同時に、国内製造業従事者の高齢化は継続しており、高齢労働者の認知特性・体力的限界への設計上の配慮も不可逆の課題として存在する（業界分析 37章と連動）。労働力多様化の加速を前提として設計に組み込むことは、多言語対応・視覚的指示優先・ふりがな表示等の設計判断の Why として機能する。

### P04 暗黙知喪失の継続

製造業の熟練工退職は今後10年で加速する。団塊世代の退職完了後も、その後継世代への技能継承は「教えられる前に辞める」という問題と「教えられる側が理解できない」という言語的・認知的障壁により、口伝えによる継承では追いつかない状況が継続する。

業界分析 05章（暗黙知と技能伝承）が示すPolanyiの命題「我々は語れる以上のことを知っている」は、形式知化の限界を認識させる一方で、形式知化できる部分の徹底的な電子化が暗黙知喪失への有効な対策であることを示す。本システムが作業手順のデジタル化・記録の電子化によって形式知を保全することは、この前提が持続する限り社会的意義を持ち続ける。

### P05 個人開発能力での完成

個人開発者がver1.0.0でプロダクションレベルの品質を達成できるという前提を確定する。技術スタック（Rust + axum + tokio + sqlx / React Native / React / PostgreSQL）の選択は、この前提に基づく個人開発者が長期にわたって保守可能な技術的選択として計画書 07章で正当化される。

「妥協は禁止する。完璧かつ至高を目指す。」というポリシー（CLAUDE.md）は、この前提が単なる楽観主義ではなく、設計哲学として維持する意志の宣言であることを示す。

---

**本節で確定した方針**
- 世界観前提P01〜P05を本書の全章が依拠する思想的前提として確定する。
- これらの前提は業務的・技術的前提（計画書 11章）とは区別される「構想レベルの現実認識」であることを確定する。
- 世界観前提が崩れた場合は本書全体の見直しを要することを確定する。

## 2. 思想的制約

思想的制約は、本システムの設計が技術的都合・コスト圧力・ステークホルダーの要求によっても覆すことができない不可侵の設計原則である。計画書 11章のリスクレジスタ（R-E01・R-E02）が「どのような要求が来ても機能追加しない」という対応方針を記録しているように、本節は「なぜその対応方針が倫理的に正当であるか」を構想レベルで宣言する。

### C01 支援/監視分離の不可侵性

04章（倫理スタンスと設計哲学、別ファイル）で確定する「支援/監視分離原則」は、技術的都合・コスト・ステークホルダーの要求によって覆してはならない不可侵の制約である。「作業員を支援するシステム」と「作業員を監視するシステム」の境界が崩れた時点で、本システムの社会的正当性の根拠が失われる。

Foucault のパノプティコン論（業界分析 24章参照）が示すように、被監視者が監視されているかどうかを確認できない構造は、それが品質目的であっても監視の権力作用をもたらす。支援と監視の分離は技術的要件であると同時に倫理的要件として扱うことを確定する。

### C02 Just Cultureの維持

記録データをスリップ・ラプスの問責に利用する機能は永続的に提供しない。James Reason のヒューマンエラーモデル（スリップ：正しい意図の誤実行、ラプス：記憶の失敗）が示すように、個人への問責はエラー防止ではなく報告抑制をもたらす。Just Culture（Sidney Dekker, 2007）が示す「エラーは報告されて学習の源泉となるべきであり、問責の対象とすべきでない」という原則は、本システムが品質改善ツールとして機能するための前提条件である。

### C03 行動データ用途三限定の恒久性

作業員の行動データ（誰が・いつ・どのステップを・どの順序で完了したか）の用途を以下の三つに限定することを恒久的制約として確定する：
1. **品質保証**：製品の品質問題発生時のトレーサビリティ追跡
2. **工程改善**：工程ボトルネックの特定・手順書の改善
3. **教育設計**：新人教育・OJT計画への活用

この三限定以外への転用——人事評価・個人別生産性ランキング・懲戒処分の証拠——を設計・制度の両面で防ぐことを確定する。OECD プライバシー8原則の「目的限定の原則」（1980年策定・2013年更新）および日本の個人情報保護法（令和3年改正）の「不適正利用の禁止」条項が、この制約の法的根拠として機能する（業界分析 24章参照）。

### C04 ALCOA+対応の継続

証拠品質基準（ALCOA+）への対応は、本事業の成否・収益性・ステークホルダーの要求によらず維持する。記録の改ざん不可能性・タイムスタンプの信頼性・監査証跡の完全性は、品質証拠文化の基盤であり、これを技術的都合（パフォーマンス・コスト）で妥協することは許されない。「妥協は禁止する。完璧かつ至高を目指す。」というポリシーは、この制約においても適用される。

### C05 社内LAN完結の維持

外部クラウドへのデータ転送機能はver1.0.0以降も提供しない。社内LAN完結は単なる技術的制約ではなく、情報越境リスクへの応答として選択した設計哲学である。この原則はC01（支援/監視分離）と連動しており、外部クラウドへのデータ転送が可能になった時点で、データ利用の制御可能性が根本的に変わる。

---

**本節で確定した方針**
- 思想的制約C01〜C05を技術的都合・コスト・ステークホルダーの要求によっても覆してはならない不可侵の設計原則として確定する。
- これらの制約は計画書のRBAC設計・アクセスポリシー・機能スコープに優先する上位原則として機能することを確定する。
- Just CultureとALCOA+対応の継続を、事業成否にかかわらず維持する倫理的コミットメントとして確定する。

## 4. 構想が将来も約束しない領域

本節は本書が構想として将来にわたって非提供を宣言する領域を確定する。この宣言は ver1.0.0 のスコープ外の宣言（計画書が行う）とは異なり、将来のいかなるバージョンにおいても提供しないという構想レベルの判断である。ただし、この宣言は将来の意思決定を完全に拘束するものではない。技術・社会・規制の根本的変化が生じた場合、本書の更新（世界観前提の変更）を経た上で、宣言の見直しは可能である。

### AI/ML機能の永続的非提供

ver1.0.0 のみでなく、構想として AI/ML による自動判断・異常検知・自然言語作業指示生成を将来にわたって提供しないと判断する。

根拠は以下の三点である：

**第一に、EU AI Act が示す高リスクAI規制の要求水準。** EU AI Act（欧州議会, 2024）は製造品質管理AIのうち製品安全性に影響する判断を下すシステムを High Risk カテゴリとして分類し、適合性評価・透明性確保・人間監視体制の構築を義務付ける（Article 10-15）。本システムの想定導入環境（国内オンプレ・中小製造業・EU市場非対象）においてもこの規制の論理的要求水準が先行リスクとして存在し、AI判断の責任帰属論（高橋, 2022）との整合が未解決である（業界分析 36章参照）。

**第二に、オンプレ環境での推論基盤構築コスト。** 社内LAN完結のオンプレミス環境でLLMクラスの推論を実行するには、GPU搭載サーバーまたは大容量RAMによるCPU推論環境が必要となる。業界分析 36章が示すように、CPU推論では10〜50トークン/秒という速度制約があり、リアルタイム作業ナビゲーションへの組み込みは応答速度の面で現実的でない。

**第三に、学習データが存在しない初期状態での証拠品質への寄与の不確実性。** ML モデルの性能はデータ品質に根本的に依存するが（Domingos, 2012）、新規導入時の作業ログは皆無であり、精度保証のない異常検知AIは品質記録の証拠品質を損なう可能性がある（業界分析 36章の「批判と限界」節参照）。LLMのハルシネーション（存在しない手順・規格値の自信を持った誤告知）は製造安全に直接影響しうるリスクである。

なお、将来の意思決定はこの宣言に縛られない。しかし、構想レベルの現時点での判断として非提供を宣言する。

### 個人作業者の生産性評価・人事評価転用の永続禁止

個人作業速度集計レポート・個人別達成率ランキング・人事評価システムとの連携を設計・制度の双方で永続的に禁止する。これは利用者（経営層・管理者）の要望があっても覆らない倫理的制約である。

本禁止の根拠は思想的制約C03（行動データ用途三限定）と連動する。Aiello & Kolb（1995）の実証研究が示すように、電子的業績監視（EPM）は複雑な認知作業において生産性を低下させストレスを増大させる。個人別速度集計・ランキングはこの問題を最も直接的に具現化するものであり、Brown et al.（2011）が定式化したデジタルテイラリズムの最悪の形態として本構想は明示的に拒絶する（業界分析 24章参照）。

技術的には RBAC（ロールベースアクセス制御）によってこの機能の実装を禁止する設計を計画書 09章の運用ポリシーとして実装するが、それは制度的禁止の技術的反映であり、倫理的禁止が先行する。

### SaaS化・規制業界拡張・社外公開の永続非提供

以下を将来にわたって提供しないと判断する：
- GMP/FDA 21 CFR Part 11・EU MDR・IATF 16949・AS9100・HACCPへの適合のための機能追加
- クラウド配信・マルチテナンシー・社外公開

根拠：単一工場オンプレのセグメントに集中することが最大価値を生む。業界分析 22章が示す規制の断片化とコンプライアンスコストの非対称性の問題——業界横断で規制が乱立し中小製造業にコストが不均衡に重くかかる——を踏まえ、本システムが規制別の詳細実装を担うことは個人開発のスコープを根本的に超え、証拠品質の維持も困難となる。ALCOA+の証拠品質論を採用しつつ特定規制への適合を約束しない設計が、本システムが持続的に維持できる最大範囲である。

### PLC / OPC-UA 等の OT レイヤ直結の永続非提供

PLC（Programmable Logic Controller）/ OPC-UA 等の OT（Operational Technology）レイヤへの直接接続を、将来にわたって提供しないと判断する。

根拠は以下の三点である：

**第一に、IT/OT 境界の意図的保護。** 製造現場の OT 機器（PLC・NC 装置・センサ等）は、安全設計・認定・責任帰属において IT システムと根本的に異なる。OT 直結は機械安全・機能安全（IEC 61508 / ISO 13849）の設計責任を本アプリが担うことを意味し、個人開発のスコープと専門的責任の範囲を超える。

**第二に、本アプリの責務の明確化。** 本システムの責務は「人が行う作業の支援と記録」に限定される。PLC から自動取得したデータを ALCOA+ 準拠の証拠として扱うことは、Attribution（誰が記録したか）と Accurate（正確か）の原則において人的確認の不在という問題を生じさせる。

**第三に、OT 連携は親機の責務。** 生産設備・センサ等からのデータ収集は、FA システム・SCADA・MES 等の親機が担う領域である。本アプリ（子機）は、親機が集約・品質保証したデータを REST API 経由で受信する設計を採用し、OT 直結は永続的に担わない。

なお、ERP / MES / 生産管理パッケージ等の IT レイヤとの論理連携（REST API 経由の親機/子機連携）は本宣言の対象外である。12 章（外部世界との接点と子機としての社会的位置づけ）で確定する「子機モード」として ver1.0.0 から提供する。

### 法令・規制の解釈責任の代行

規制対応の具体的解釈は導入組織が法律専門家・品質専門家と協議する。本システムは記録の品質を提供するが、規制適合の判断を代行しない。これは技術的限界からくる制約ではなく、設計原則としての限界宣言である。法令解釈という専門的職務を技術システムが代行することの危険性は、医療分野での AI 診断責任問題と同じ構造を持つ。

---

**本節で確定した方針**
- AI/ML機能・個人作業者の生産性評価転用・SaaS化/規制業界拡張・法令解釈責任代行・PLC/OT レイヤ直結の 5 領域を将来にわたって非提供と宣言することを確定する。
- なお ERP / MES / 生産管理との論理連携は、12 章で導入する「子機モード」として ver1.0.0 で正式提供する。本節が旧来「将来非提供」と示唆していた範囲から除外し、本改訂（2026-05-17）で上記の方針に改定したことを記録する。
- 個人作業者の生産性評価・人事評価転用の禁止を「利用者の要望があっても覆らない倫理的制約」として確定する。
- これらの宣言は世界観前提の根本変化があった場合に本書の更新を経て見直し可能であることも確定する。

## 5. 本書が約束しないこと宣言

本節は README §8「本書が約束しないこと」と完全に一致する正本である。両者の差分が生じた場合は本節を正とする。

本構想書に基づいて開発するシステムは、以下を提供しない：

### 1. 計画書側で確定する詳細の先取り

個別KPI値の定量保証・ALCOA+ 9原則の実装マッピング詳細・技術選定根拠の代替案比較・マイルストーン M1〜M6 の完了条件・リリース判定基準は本書の責務外とする。これらは Why の先取りではなく What/How の領域であり、計画書（`docs/02_企画/システム化計画/`）の対応章が確定する権限を持つ。本書がこれらを記述した場合は計画書への移管対象と判断する。

### 2. AI/ML機能

ver1.0.0 のみでなく、将来にわたってAI/MLによる自動判断・異常検知・自然言語指示生成を提供しないと構想として宣言する。根拠はEU AI Act が示す高リスクAI規制要求・オンプレ推論基盤の実現コスト・初期段階での証拠品質への寄与不確実性の三点にある（業界分析 36章参照）。

### 3. 個人作業者の生産性評価・人事評価転用

個人別速度集計・ランキング・人事評価システム連携を永続的に禁止する。技術的・制度的双方の制約として維持する。Aiello & Kolb（1995）の実証研究・Foucaultのパノプティコン論・OECD プライバシー原則のいずれもこの禁止を支持し、利用者の要望があっても覆らない倫理的制約として位置づける（業界分析 24章参照）。

### 4. SaaS化・規制業界拡張・社外公開

GMP/FDA/IATF/AS9100/HACCP対応拡張・クラウド配信・マルチテナンシー・社外公開を永続非提供と宣言する。業界分析 22章が示す規制断片化とコンプライアンスコストの非対称性を踏まえ、単一工場オンプレセグメントへの集中が本システムの最大価値を生む判断として確定する。

### 5. PLC / OT レイヤ直結の永続非提供

PLC / OPC-UA / OT センサとの直接統合を将来にわたって提供しない。IEC 61508 / ISO 13849 が規定する機能安全設計責任および IT/OT 境界の意図的保護に基づく。ALCOA+ の Attribution・Accurate 原則との整合においても、人的確認を介さない自動取得データの証拠適用は本構想の射程を超える。

なお、ERP / MES / 生産管理パッケージ等の IT レイヤとの論理連携（子機モード）は本節の対象外であり、ver1.0.0 から正式提供する（12 章参照）。過去の「やらない宣言」からの改定を本改訂（2026-05-17）として記録する。
