# 03 価値命題と提供価値

本章の責務は、本システムが「誰に・何を・なぜ届けるのか」という価値命題の構造を確定し、各ステークホルダーへの価値と、意図的に提供しない領域の境界線を宣言することである。数値目標・ROI の確約は行わない。それらの検証は ver1.0.0 リリース後の運用に委ねる。

---

## 価値命題の基本構造

本システムの価値命題は「誰に（Who）」「何を（What）」「なぜ届くのか（Why）」の 3 要素で構成する。この 3 要素を分離せず一体として定義することで、機能追加や仕様変更の際に「本来の価値に資するか」を判断する基準が生まれる。

### Value as Designed と Value in Use

| 概念 | 定義 | 本システムにおける扱い |
|---|---|---|
| Value as Designed（VaD） | 設計段階において開発者が意図する価値 | 本章で宣言・確定する |
| Value in Use（ViU） | 実際の運用環境で作業者・組織が体験する価値 | ver1.0.0 リリース後に検証する仮説として位置づける |

VaD と ViU の乖離は、あらゆるシステム開発において生じる。本書は VaD の精度を高めることに注力し、ViU の検証を運用フェーズの継続的改善活動に委ねる。VaD と ViU を混同した場合、設計の意図が既成事実化され、現場の実態から乖離した「紙の上の品質」のみが積み上がるリスクがある。

### ROI の数値約束を本章でしない理由

本章は「Why / べき論」を担う構想文書である。ROI の数値的約束は投資計画・費用対効果分析に帰属し、計画書第 07 章に委ねる。構想段階での数値確約は、未検証の前提を確定事項として扱う誤謬であり、その後の仕様決定を恣意的に歪める危険がある。本章では「価値が届く論理的経路」のみを確定する。

---

**本節で確定した方針**

- 価値命題を「Who・What・Why」の 3 要素として構造化し、本章全体を貫く軸とすることを確定する。
- Value as Designed を本章で宣言し、Value in Use の検証は ver1.0.0 リリース後に委ねることを確定する。
- ROI の数値的約束を本章では行わず、計画書第 07 章に委ねることを確定する。

---

## 作業員への価値

作業員は本システムの一次ユーザーであり、かつ最大の便益受領者でなければならない。監視の客体としてではなく、支援の主体として設計される必要がある。

### Navigation 価値：「次に何をすべきか迷わない」状態の恒常的提供

製造現場における認知的負荷の大部分は、「何をすべきか」ではなく「次に何をすべきか、どの順でやるか、この手順で合っているか」という文脈的迷いから生じる。本システムは、作業者が現在のステップにのみ集中できる状態を構造的に提供する。これは Hick の法則（Hick, 1952; Hyman, 1953）が示す「選択肢数と反応時間の正の相関」の逆転——選択肢を 1 に絞ることで反応時間と誤操作を最小化する——の工学的応用である。

### Traceability からの解放：二重入力作業の排除

現場の多くで、作業者は作業後に紙帳票に記録し、さらにシステムに転記するという「二重入力」を強いられている。本システムは作業の実施と証拠記録を単一のフローに統合する Intrinsic EPSS（後述）の構造を採用し、記録のための二重入力作業を構造的に排除する。

### 中断・再開支援：Post-Completion Error と Resumption Lag の最小化

Byrne & Bovair（1997）は複合タスクにおける **Post-Completion Error**——目標達成後に後続の重要ステップを忘れる——の発生機序を実験的に示した。また Altmann & Trafton（2002）の **Resumption Lag** の研究は、中断後に作業を再開する際に要する認知的再定位コストを定量化した。

本システムは次の設計によりこれらに対応する。

| 問題 | 設計対応 |
|---|---|
| Post-Completion Error | 完了条件を満たすまで次ステップへの遷移をブロックする |
| Resumption Lag | 中断地点を自動保存し、再開時に「どこまでやったか」を即座に提示する |
| 中断履歴の消失 | 中断タイムスタンプと中断理由を記録し、作業者・監督の双方が参照できる |

### 習熟度差への対応：Dreyfus 5 段階モデルに基づく表示密度調整

Stuart & Hubert Dreyfus（1980）の技能習得 5 段階モデル（Novice / Advanced Beginner / Competent / Proficient / Expert）は、各段階において「必要な情報の種類と粒度が異なる」ことを示す。初心者は規則ベースの詳細指示を必要とし、熟練者は概要と例外条件のみを必要とする。

本システムは習熟度別の表示情報密度調整を設計方針として採用する。詳細なUI仕様は計画書第 04 章に委ねるが、構想レベルでの原則として「熟練者に初心者向けの冗長情報を強制しない」こと、「初心者に熟練者向けの省略化画面を強要しない」ことを確定する。

### HCI 設計原則の適用

製造現場の特殊な制約（手袋・粉塵・騒音・照明変動・動きながらの操作）に対応するため、以下の HCI 原則を設計制約として採用する。

| 原則・出典 | 本システムへの適用 |
|---|---|
| Fitts の法則（Fitts, 1954） | タッチターゲット最小サイズ 72dp 以上（手袋着用環境を考慮） |
| Hick の法則（Hick, 1952） | 主要操作ボタンを 1〜2 択に絞り、分岐は最大 4 択に制限 |
| Glanceable UI（Matthews et al., 2004） | 現在ステップ番号・完了率を最大フォント・高コントラストで最上位に表示 |
| WCAG 2.1 AA コントラスト比 4.5:1 以上 | 全テキスト要素に適用し、粉塵・逆光環境での視認性を確保 |

---

**本節で確定した方針**

- Post-Completion Error および Resumption Lag に対応する中断・再開支援を設計制約として組み込むことを確定する。
- Dreyfus 5 段階に基づく習熟度別の表示情報密度調整を設計方針として採用することを確定する。
- タッチターゲット 72dp 以上・Hick の法則に基づく 4 択以内の選択肢制限・WCAG 2.1 AA コントラスト比を HCI 設計制約として確定する。

## 品質担当・規制当局想定への価値

品質担当者と規制当局は、証拠の完全性・正確性・追跡可能性に関して最も厳しい要求を持つステークホルダーである。本システムはこれらの要求を「運用後の努力」ではなく「設計の構造」によって担保する。

### 証拠の自動生成：Intrinsic EPSS 構造による ALCOA+ Contemporaneous の保証

本システムは **Intrinsic EPSS（Electronic Performance Support System）** の構造を採用する。Intrinsic EPSS とは、支援機能が作業フローに内在化され、「作業をすることで自動的に証拠が生成される」設計様式を指す。この設計において、ALCOA+ の要件のうち最も達成困難な **Contemporaneous（同時性）**——記録が作業と同時に生成されること——が構造的に保証される。

| ALCOA+ 原則 | 本システムの対応方針 |
|---|---|
| Attributable（帰属可能） | 電子署名・作業者 ID によるステップ実施者の自動記録 |
| Legible（判読可能） | デジタル入力による常時判読可能な電子記録 |
| Contemporaneous（同時性） | Intrinsic EPSS 構造により作業と記録が同一フローで生成 |
| Original（原本） | 初回入力データをハッシュ値で原本管理し、改ざんを検知 |
| Accurate（正確） | 測定器からの直接入力・QR スキャンによる手転記エラーの排除 |
| Complete（完全） | 未完了ステップの次工程遷移ブロックにより漏れを構造的に防止 |
| Consistent（一貫） | タイムスタンプの UTC 基準統一・フォーマットの固定化 |
| Enduring（永続） | PostgreSQL による長期保管・バックアップ設計 |
| Available（利用可能） | 権限に応じたリアルタイム参照・検索機能 |

### 順方向・逆方向トレサビ：双方向連鎖の確立

製品から工程・作業員・材料・設備への双方向連鎖を確立する。ISO 9000:2015 が定義するトレーサビリティ——「対象の履歴、適用、又は所在を追跡できる能力」——を電子記録で実現する。

| 追跡方向 | 起点 | 終点 | 用途 |
|---|---|---|---|
| 順方向（Forward） | 原材料ロット | 完成品・出荷先 | リコール範囲の特定 |
| 逆方向（Backward） | 完成品・不適合品 | 使用材料・担当者・工程条件 | 不良原因の特定 |

### 版管理・承認履歴の完全性

SOP の改訂履歴を作業記録に紐付け、「いつの版の手順で作業したか」を証拠として残す。これにより、手順改訂後の作業との比較・規制当局への提示が可能になる。

### 内部監査準備の工数削減

内部監査時に必要な記録の収集・整理工数は、紙運用では大きな負担となる。本システムは電子記録の検索・集計機能により、この工数削減を VaD として宣言する。ViU としての削減率は ver1.0.0 リリース後の測定に委ねる。

---

**本節で確定した方針**

- Intrinsic EPSS 構造を採用し、ALCOA+ Contemporaneous を設計的に保証することを確定する。
- 順方向・逆方向の双方向トレサビを電子記録で実現する設計方針を確定する。
- SOP の版管理を作業記録に紐付ける設計を採用することを確定する。

## 経営層・IT担当への価値

経営層は ROI と法的リスクを、IT 担当は運用負荷と習熟コストを主な懸念とする。本システムはこれらに対して「設計による制約」と「運用の単純化」を提供する。

### 暗黙知の準形式知化：退職リスクへの構造的対処

熟練作業者の退職による技能の喪失は、中堅製造業が直面する最大のリスクの一つである。本システムは SOP・手順書・過去の記録を電子化・構造化することで、暗黙知の一部を準形式知として組織に蓄積する。「準」と表現するのは、形式知化できない身体知・感覚知の領域が残るためである。その限界を自覚しつつ、形式知化できる領域を最大化する。

### オンプレ完結：外部クラウド依存ゼロの設計

本システムは社内 LAN のみで動作し、外部クラウドへのデータ送信を行わない。これにより以下を確保する。

| 利点 | 根拠 |
|---|---|
| 障害切り分けの単純化 | 依存する外部サービスが存在しないため、障害範囲が社内に限定される |
| 情報漏洩リスクの限定 | 外部通信がないため、インターネット経由の攻撃経路を原理的に排除 |
| 継続的運用コストの予測可能性 | クラウド従量課金が存在せず、ハードウェア・ライセンスコストのみ |

### Docker Compose 配布：IT 1 名体制での運用可能性

本システムはバックエンド・データベース・管理 Web の全コンポーネントを Docker Compose で配布する。IT 担当が 1 名であっても起動・停止・バックアップが標準的な手順で実行できる運用設計を方針として採用する。詳細な運用設計は計画書第 07 章に委ねる。

### 補助金・DX 認定への対応余地

本システムの電子記録化・プロセスデジタル化は、IT 導入補助金・DX 認定制度への申請において根拠資料として機能しうる。ただし、認定取得の保証は本構想の範囲外であり、詳細な申請対応は運用フェーズの判断に委ねる。

---

**本節で確定した方針**

- 社内 LAN 完結・外部クラウド依存ゼロのオンプレ設計を採用することを確定する。
- Docker Compose 配布による IT 1 名体制での運用可能性を設計方針として確定する。
- 暗黙知の準形式知化を本システムの副次的価値として位置づけることを確定する。

---

## 労働者代表への価値

労働者代表は、デジタルテイラリズムへの転落・人事評価への転用・多言語対応の欠如という三つの懸念を持つ。本システムはこれらに対して「設計制約」によって応答する。意図の宣言だけでは不十分であり、技術的・制度的な制約として実装する。

### 監視ツールでないことの構造保証

「監視を意図しない」という意図の宣言は、データが実際に監視目的に使用される可能性を排除しない。Aiello & Kolb（1995）が示したように、電子的業績監視は複雑な認知作業において認知負荷を増大させ、生産性を低下させる。意図でなく設計が作業者を守る。

個人作業速度の集計・個人別スコアリング・位置情報リアルタイム追跡・生体情報収集は、本システムが「提供しない機能」として設計制約に記述する。詳細な設計制約は第 04 章「倫理スタンス」に委ねる。

### Just Culture の採用：スリップ・ラプスを問責に結びつけない

James Reason の Just Culture モデルを本システムの運用原則として採用する。スリップ（意図しない行為の失敗）・ラプス（記憶エラーによる失敗）は問責の対象とせず、システム・手順・教育の改善問題として扱う。詳細は第 04 章で確定する。

### 多言語・多識字対応：外国人作業者を正規ユーザーとして位置づける

外国人技能実習生・特定技能労働者を「対応の例外」としてではなく「正規ユーザー」として設計に組み込む。日本語のみを前提とした UI は、外国人作業者を実質的に排除する設計上の差別である。多言語表示・多識字対応（図・アイコン・動画による補完）を設計要件として宣言する。

### Kaizen Teian への応答：改善提案の受け皿としての機能

Edmondson（*Teaming*, 2012）が示した心理的安全性の知見は、改善提案が起きるためには「報告が罰則につながらない」という文化的安全が必要であることを示す。本システムは作業者が手順への改善案・疑問を記録できる仕組みを設計要件として持つ。改善提案の受け付けと処理の責任は組織の運用設計に委ねるが、その経路を閉じない設計を確定する。

---

**本節で確定した方針**

- 外国人技能実習生・特定技能労働者を正規ユーザーとして多言語・多識字対応を設計要件に含めることを確定する。
- Just Culture の運用原則採用を宣言し、詳細は第 04 章に委ねることを確定する。
- 改善提案の記録経路を設計に含めることを確定する。

## 価値の非対称性宣言

本システムが提供する価値の序列を明示する。これは「生産性向上を目的としてはならない」という設計上の倫理宣言である。

### 主眼の確定

本システムの主眼は「証拠品質の向上」と「作業支援」である。生産性向上はその副次効果として生じる可能性があるが、主眼として追求することはしない。

主眼を生産性向上に置いた場合、次のリスクが生じる。

- 個人別の作業速度・達成率の比較が始まり、Brown et al.（2011）が定義するデジタルテイラリズムの構造に陥る。
- Foucault のパノプティコン論理——「見られているかもしれない」という意識が作業者の自律的な判断を委縮させる——が実現する。
- Hackman & Oldham（1976）の職務特性モデルが示す「自律性（Autonomy）」の剥奪が、内在的動機づけを破壊する。

### 提供しない機能の明示

本システムが意図的に提供しない機能を設計制約として以下に宣言する。

| 非提供機能 | 排除の理由 |
|---|---|
| 個人作業速度の集計・上長への報告 | デジタルテイラリズムへの直結経路 |
| 個人別達成率スコアリング | 監視の客体化・心理的安全性の破壊 |
| 位置情報リアルタイム追跡 | 目的外の行動監視・プライバシー侵害 |
| 生体情報収集 | センシティブ情報の過剰収集 |
| 記録を人事評価・懲罰根拠に転用する機能 | Just Culture の根本的破壊 |

これらの機能は「追加しない方針」ではなく、「設計上の禁止事項」として扱う。

---

**本節で確定した方針**

- 本システムの主眼は「証拠品質の向上」と「作業支援」であり、生産性向上はその副次効果として位置づけることを確定する。
- 個人作業速度の集計・個人別スコアリング・位置情報リアルタイム追跡・生体情報収集・人事評価転用を設計上の禁止事項として確定する。
- 「デジタルテイラリズムに陥らない」ことを価値設計の根幹的制約として確定する。

## 単独動作モードと子機モードの両立という価値

本システムは「基幹システムを持たない工場」と「基幹システムを持つ工場」の双方に対して、同等の中核価値を提供できるよう設計する。この両立が本システムを既存の競合製品と明確に区別する固有の価値命題である。

### 基幹システム非保有工場における中核価値の完全動作

単独動作モードとは、ERP・MES・生産管理パッケージ等の親機を持たない工場環境において、本システムが外部連携なしに自律的に動作する構成を指す。この構成においても、SOP ナビゲーション・ALCOA+ 準拠記録・Just Culture 支援という三つの中核価値は完全に機能する。

本システムの価値の核心は「親機の存在」ではなく「作業フローへの内在化」にある。したがって基幹システムとの接続の有無は、中核価値の提供可否に影響しない。これは本システムが「基幹システムの周辺ツール」ではなく「作業現場の自律的価値提供システム」として設計されることを意味する。

### 子機モード採用時の付加価値

子機モードとは、本システムが外部の親機（ERP・MES・生産管理パッケージ等）と REST API で連携する構成を指す（12 章で確定）。子機モードが採用される場合、外部一意キー（型式・機番・品目コード等の一意識別属性の組み合わせ）を通じて以下の付加価値が加わる。

| 付加価値の領域 | 内容 |
|---|---|
| マスタ自動受信 | 製品・工程・材料等のマスタデータを親機から自動受信し、本システムへの手入力を排除する |
| 実績の自動連携 | 作業完了・検査合否・数量等の実績データを親機へ自動送信し、転記工数を排除する |
| トレサビの横断活用 | 本システムが記録した作業証拠と、親機が保有する受注・出荷・在庫データを外部一意キーで結合し、トレサビを組織横断で活用可能にする |

付加価値は中核価値の上に積み上がるものであり、子機モードを採用しない場合に中核価値が失われることはない。これが本システムのアーキテクチャ上の基本方針である。

### 競合製品との差異

グローバル MES（SAP ME 等）・kintone 系 EUC・Tulip は、いずれも次のいずれかの問題を抱える。

| 競合製品の類型 | 課題 |
|---|---|
| グローバル MES | 基幹システムとの密結合を前提とし、基幹システム非保有工場への単独導入が困難または高コスト |
| kintone 系 EUC | SOP ナビゲーションと ALCOA+ 準拠記録の構造的統合が設計思想として存在せず、個別カスタマイズに依存する |
| Tulip | クラウド依存・外部通信前提の設計であり、オンプレ完結・外部クラウド依存ゼロの要件と相反する |

本システムは単独動作モードと子機モードの両立によって、これら三類型のいずれもカバーできない工場類型——基幹システム非保有かつオンプレ完結を要求する中堅製造業——を主要ターゲットとして正面から対応する。

### 子機モード採用によるステークホルダー別価値の拡張

単独動作モードで確定したステークホルダー別価値（作業員・品質担当・経営層・IT 担当・労働者代表）は、子機モードの採用によって以下の通り拡張される。

| ステークホルダー | 子機モードによる価値拡張 |
|---|---|
| 品質担当 | 本システムの作業証拠と親機が保有する製品・材料・出荷データを外部一意キーで結合することで、トレサビを組織横断で活用できる |
| IT 担当 | 親機からのマスタ自動受信により、品目・工程等のマスタデータを本システムへ手入力する工数を恒久的に排除できる |
| 経営層 | 既存の基幹システム資産（ERP・MES 等への投資）を本システムとの連携によって現場レベルで活用し、追加投資なしに価値を拡張できる |

---

**本節で確定した方針**

- 本システムは単独動作モード（親機なし）においても SOP ナビゲーション・ALCOA+ 準拠記録・Just Culture 支援という中核価値を完全に提供することを確定する。
- 子機モード（12 章）採用時は、外部一意キーによるマスタ自動受信と実績自動連携が付加価値として加わるが、中核価値の提供は子機モードに依存しないことを確定する。
- 外部一意キーは型式・機番・品目コード等の一意識別属性の組み合わせとし、親機と本システムを結合する共通識別子として機能することを確定する。
- グローバル MES・kintone 系 EUC・Tulip のいずれも単独動作モードと子機モードの両立という価値を持たないことを競合差異として位置づけることを確定する。
- 品質担当（トレサビ横断活用）・IT 担当（マスタ入力工数削減）・経営層（基幹システム資産活用）への価値拡張を、子機モード採用時の追加価値として宣言することを確定する。
