# 19 マスタ編集者体験設計（オーサリング UX とガバナンス）

本章は、計画 03 章から 18 章にわたって分散して記述されてきたマスタメンテナンス（React 製管理 Web）の編集者体験に関する設計事項を、編集者の視点から統合・確定する。作業員のための SOP ナビゲーションと同等の設計品質を、SOP を作成・改訂・管理するマスタ編集者に提供することが本章の目的である。

---

## 1. 本章の位置づけと前提

### 1-1. マスタメンテナンスのストレスは SOP 品質の天井

本アプリの最終的な価値は作業員が正確な手順を安全に実行することにある。しかし、SOP の内容が正確で最新であることは自明ではない。SOP を作成・改訂・承認するマスタ編集者——品質担当・IT 担当・ライン管理者——が「マスタを変更することを恐れる」状態では、SOP は導入当初から陳腐化し始める。

マスタ編集者の体験設計を「後回し」にすることは、SOP ナビゲーションの品質に直接上限を設けることと等しい。

### 1-2. 本章が統合する設計事項

本章は以下の章に分散していた編集者体験の設計事項を統合する：

| 参照章 | 統合される内容 |
|---|---|
| 計画 03 章 § 2-2 | 管理 Web コア機能（手順マスタ管理・マスタ編集者体験原則） |
| 計画 08 章 § 2 | 管理 Web 編集 UX 非機能要件（レスポンス・オートセーブ・楽観的ロック）|
| 計画 09 章 § 2 | SOP 改訂ガバナンス（電子署名・バージョン番号規則・廃止処理） |
| 計画 13 章 § 4 | 初期データ移行（CSV インポート・プレビュー・不整合レポート）|
| 計画 18 章 § 3-4 | カスタム Step スキーマ定義・条件分岐フロー定義（JSON Logic） |

本章はこれらの責務を引き受けず、参照関係を整理した上で「編集者の実際の作業体験」に特化した設計方針を確定する。

### 1-3. 設計のキーワード

**「失わない・壊さない・繰り返さない・覚えない」**

**本節で確定した方針**

- マスタ編集者体験の設計品質を、作業員向け UX と対等な優先度で扱うことを確定する。

## 2. マスタ編集者の 4 つのストレスポイント

マスタ編集のストレスを以下の 4 類型に整理し、本章の各節がそれぞれを解消する設計として対応する。

| ストレス類型 | 代表的な症状 | 解消節 |
|---|---|---|
| **喪失恐怖** | 編集中に誤操作で作業が消えた / ブラウザがクラッシュして入力内容が失われた | §3 オーサリング UX 原則 |
| **破壊恐怖** | マスタを変更した後に「どのSOP が影響を受けるか分からない」「削除したら何が動かなくなるか分からない」 | §6 参照整合性ガード |
| **反復疲労** | 類似した SOP を毎回ゼロから手作業で作成する / 複数 SOP に同じ変更を 1 件ずつ手動で適用する | §5 一括操作・テンプレート |
| **DSL 学習負担** | 計画 18 章のカスタム Step タイプを設定するには JSON Logic の文法を覚える必要があると感じる / 条件分岐のフローを JSON で書くことへの心理的障壁が高い | §4 ビジュアル DSL エディタ |

---

**本節で確定した方針**
- マスタ編集者のストレスを喪失恐怖・破壊恐怖・反復疲労・DSL 学習負担の 4 類型で確定し、各節がいずれか 1 類型を主として解消する設計とすることを確定する。

## 3. オーサリング UX 原則（5 つの不変条件）

管理 Web のマスタ編集機能は、以下の 5 原則を不変条件として実装する。これらは個別機能の設計方針ではなく、マスタ編集全体に適用される基盤原則である。

### 3-1. Draft First

全マスタの編集操作はドラフトとして保存される。ドラフト中のデータは管理 Web 内でのみ参照可能であり、タブレット APP には反映されない。

「公開」操作（品質担当の電子署名を含む承認フロー、計画 09 章 §2 参照）を経て初めて有効化される。これにより「保存 = 即時本番反映」への恐怖を解消する。

| 状態 | 説明 |
|---|---|
| `draft` | 編集中。タブレット APP には見えない |
| `under_review` | レビュー依頼済み。差分参照可能 |
| `approved` | 品質担当が電子署名で承認済み |
| `published` | Effective Date 到達後に有効化 |
| `obsolete` | 廃止処理済み（Append-only で保持） |

### 3-2. Diff Always Visible

ドラフト編集中は、右ペインに「現在の公開版との差分」を常時リアルタイム表示する。変更箇所は追加（緑）・削除（赤）・変更（黄）で色分けする。

差分表示は「自分が何を変えたか」を常に把握できる状態にするための安全網である。承認者（品質担当）も同じ差分ビューで変更内容を確認し、電子署名を行う。

### 3-3. Auto-Save Every 30s

編集セッション中、30 秒ごとにドラフトを自動保存する。ブラウザクラッシュ・ネットワーク断・誤ったタブ閉じが発生しても、最大 30 秒前の状態まで復元できる。

バックグラウンド保存は画面右上のステータスインジケータ（「最終保存: X 秒前」）で表示し、編集者が明示的に「保存できている」ことを確認できる。

### 3-4. Keyboard-First

全主要操作はキーボードショートカットで実行できる。代表的なショートカット（確定は要件定義フェーズ）:

| 操作 | ショートカット |
|---|---|
| ドラフト保存 | `Ctrl + S` |
| 差分表示 ON/OFF | `Ctrl + D` |
| Undo | `Ctrl + Z` |
| Redo | `Ctrl + Y` |
| Step 追加 | `Ctrl + Enter` |
| Step 削除 | `Ctrl + Backspace` |
| 上に移動 | `Alt + ↑` |
| 下に移動 | `Alt + ↓` |

マウスのみでしか操作できない実装は本原則の違反とする。

### 3-5. Undo/Redo 50 steps

直近 50 操作まで取り消し（Undo）・やり直し（Redo）が可能である。誤操作を「Ctrl+Z で戻る」という直感で解決できる状態を維持する。

Undo 履歴はドラフト内でのみ有効である。「公開」操作は Undo の対象外とし、公開後の変更は新たなドラフトとして扱う。

図: fig_master_edit_ux_overview.svg（img/ 配下）— 4 ストレスポイント（喪失恐怖・破壊恐怖・反復疲労・DSL 学習負担）と 5 不変条件・解消機能の対応関係を参照すること。

---

**本節で確定した方針**
- Draft First / Diff Always Visible / Auto-Save 30s / Keyboard-First / Undo-Redo 50 steps を管理 Web 全マスタ編集機能の 5 不変条件として確定する。
- 「公開」操作は Undo 対象外とし、公開後変更は新ドラフトとして扱うことを確定する。
- バックグラウンド保存のステータスを常時表示することを確定する。

## 4. 拡張 Step エンジン用ビジュアル DSL エディタ

計画 18 章で確定したカスタム Step タイプ定義（JSON Logic）と条件分岐フロー定義（step_flow_rule）は、コードの知識を持たないマスタ編集者でも編集できるビジュアルインタフェースを提供する。

### 4-1. ビジュアル DSL ビルダ（カスタム Step バリデーション用）

JSON Logic を「組み立てる」インタフェース：

| コンポーネント | 説明 |
|---|---|
| **演算子パレット** | `==` / `!=` / `>=` / `<=` / `>` / `<` / `and` / `or` / `!` をボタンとしてパレットに配置。クリックまたはドラッグ＆ドロップで条件ツリーに追加する |
| **変数ピッカー** | `{"var": "value"}` に相当する変数（入力値・Step ID 等）を選択肢として表示し、選択で挿入する。JSON のキー名を手書きしない |
| **値入力フィールド** | 数値・文字列・ブール値を型に応じたフィールドで入力する（数値は数値スピナー、文字列はテキスト） |
| **条件ツリー表示** | 組み立てた JSON Logic をツリー形式でビジュアライズし、人間が読める形で構造を確認できる |

### 4-2. DAG ビジュアライザ（条件分岐フロー用）

step_flow_rule（計画 18 章 §4）の条件分岐を「フロー図」として表示・編集するインタフェース：

| コンポーネント | 説明 |
|---|---|
| **Step ノード** | SOP の各 Step をノードとして表示する |
| **条件エッジ** | step_flow_rule の条件成立時のフローを矢印エッジとして表示する（skip / goto / insert を色分け）|
| **条件成立パス** | ノードを選択すると、そのノードが条件を評価する全ルールと成立時の遷移先を強調表示する |
| **循環参照警告** | DAG 検証結果（計画 18 章 §4-4）を赤強調でリアルタイム表示し、登録前に循環参照に気づける |

> 本節で確定したビジュアル DAG エディタの実装詳細は `05_詳細設計/05_WebAPP詳細設計/03_SopFlowEditor詳細設計.md`（MOD-FE-MA-003・FR-MA-016）に委任する。

### 4-3. プレビュー実行

ビジュアル DSL ビルダ・DAG ビジュアライザで定義した条件式に対し、サンプル値を入力して評価結果をリアルタイムで確認できる：

- **単一条件のテスト**: 数値や文字列を入力し、`true` / `false` が即座に表示される
- **フロー全体のシミュレーション**: Step 1 から順番にサンプル値を入力し、どのパスを通るかを可視化する

### 4-4. 構文ハイライト・エラー位置表示

RAW JSON を直接編集するモード（上級者向け）でも、JSON Logic の構文エラーは行・列単位でインライン表示する。ビジュアルモードと RAW JSON モードは随時切り替え可能で、両モードの表示は同期する。

図: fig_visual_dsl_editor.svg（img/ 配下）— ビジュアル DSL エディタのレイアウト概念（演算子パレット・条件ツリー表示・プレビュー実行・DAG ビジュアライザ・循環参照警告）を参照すること。

---

**本節で確定した方針**
- ビジュアル DSL ビルダ（演算子パレット・変数ピッカー・条件ツリー表示）と DAG ビジュアライザをマスタ編集者向けの必須機能として確定する。
- プレビュー実行（サンプル値による即時評価）を確定する。
- ビジュアルモードと RAW JSON モードの相互切り替えを確定し、上級者の直接 JSON 編集をサポートすることを確定する。
- eval 禁止・JSON Logic のみという制約（計画 18 章 §3-3）はビジュアル DSL ビルダの実装によって自然に充足されることを確定する。

## 5. 一括操作・テンプレート

反復疲労を解消するため、以下の一括操作・テンプレート機能を提供する。

### 5-1. CSV/Excel 双方向インポート・エクスポート

| 方向 | 機能 | 制約 |
|---|---|---|
| インポート | CSV / xlsx をアップロードして SOP・Step データを一括登録 | インポート前に「プレビュー」表示必須。不整合レコードを赤強調で一覧表示する（計画 13 章 §4 参照）。人的確認なしの自動投入は禁止 |
| エクスポート | 選択した SOP 群を CSV / xlsx で一括出力 | 差分比較用フォーマット対応。変更箇所を `before` / `after` 列で出力する |

### 5-2. SOP テンプレートライブラリ

工場または業種・工程別の雛形 SOP を「テンプレートライブラリ」として管理する：

- テンプレートは通常の SOP と同一のデータモデルを使い、`is_template: true` フラグで区別する
- テンプレートから SOP を作成する場合、全ステップ・メタデータが複製され独立した SOP として扱われる
- テンプレート自体の公開フローは通常 SOP と同様（ドラフト→承認→公開）

### 5-3. SOP クローン

既存の公開済み SOP を原型として、新しい SOP を派生作成できる：

1. 原型 SOP を選択し「クローン」を実行する
2. 新 SOP はドラフト状態で作成される（原型 SOP には影響しない）
3. 文書番号・版番号・有効化日の更新は必須。クローン元の文書番号を `derived_from` フィールドに自動記録する

### 5-4. 差分一括適用

複数の SOP に共通の変更（例: 作業標準の共通用語変更・共通手順の更新）を一括適用する：

1. 変更内容を「差分テンプレート」として作成する（対象フィールド・変更前後の値を定義）
2. 適用対象の SOP を選択する（複数選択可）
3. 各 SOP に対して「適用プレビュー」を表示し、確認後に一括適用する
4. 適用された各 SOP は自動的にドラフト状態になり、承認フローを経て公開される

---

**本節で確定した方針**
- CSV/Excel 双方向・SOP テンプレートライブラリ・SOP クローン・差分一括適用の 4 機能を確定する。
- CSV インポート時のプレビュー表示必須・人的確認なし自動投入禁止を確定する（計画 13 章との整合）。
- クローン元の文書番号を `derived_from` フィールドに自動記録し、派生関係を Audit Trail で追跡できることを確定する。
- 差分一括適用後の各 SOP はドラフト状態になり、承認フローを必ず経ることを確定する。

## 6. 参照整合性ガード・影響範囲表示

破壊恐怖を解消するため、マスタの削除・改訂時に「何が壊れるか」を事前に可視化するガード機能を提供する。

### 6-1. 影響範囲ビュー

マスタ（設備マスタ・ユーザーマスタ・カスタム Step タイプ・step_flow_rule 等）の削除または改訂を実行する前に、参照関係のある以下のレコードを一覧表示する：

| 参照元 | 表示内容 |
|---|---|
| SOP | 参照する SOP 文書番号・版番号・ステータス（公開中 / ドラフト）|
| 作業パターン | 参照する外部一意キーマッピングの名称・対象ライン |
| カスタム Step タイプ | 参照するカスタム Step タイプ名・定義バージョン |
| 条件分岐ルール | 参照する step_flow_rule の ID・対象 SOP |

一覧は件数のみでなくリンク付きで表示し、個々のレコードの詳細に直接遷移できる。

### 6-2. 削除前 dry-run

削除操作の前に「削除した場合に何件のレコードが影響を受けるか」をシミュレーションする dry-run モードを提供する：

- dry-run は実際のデータを変更しない（参照整合性のシミュレーションのみ）
- dry-run 結果は「影響件数: N 件」として表示し、影響ゼロの場合のみ削除を即時実行可能とする
- 影響が存在する場合は、影響範囲ビューへのリンクと「参照を解除してから削除」への誘導を表示する

### 6-3. 参照中削除ブロック

参照中のマスタの削除は cascade 削除ではなく、以下の 2 段階フローを必須とする：

1. **影響範囲の確認**: 削除前 dry-run で影響範囲を確認する
2. **参照の解除または移行**: 影響を受けるレコード全件について「参照先の変更」または「自身の削除」を実施する
3. **削除の実行**: 参照件数がゼロになった状態で削除を実行する

参照件数がゼロでない状態での削除は API レベルでブロックする（クライアント側のチェックのみに依存しない）。

### 6-4. 改訂シミュレーション（SOP 改訂特有）

SOP の改訂時、過去実績への影響をプレビューする機能を提供する：

- 旧版 SOP で作業を完了した実績レコードは、新版適用後も旧版 SOP への参照を維持する（ALCOA+ Enduring）
- 「この改訂を適用した場合、過去 N 件の実績が旧版を参照した状態になる」を明示する
- 旧版 SOP レコードはソフトデリート（`status: obsolete`）として保持し、物理削除しない

図: fig_reference_integrity_guard.svg（img/ 配下）— 削除リクエスト → dry-run → 影響件数判定 → 参照解除 → 削除実行のフローと cascade 削除禁止の根拠を参照すること。

---

**本節で確定した方針**
- 影響範囲ビュー・削除前 dry-run・参照中削除ブロックの 3 機能を確定する。
- 参照中削除ブロックは API レベルで実装し、クライアント側のチェックのみへの依存を禁止することを確定する。
- 旧版 SOP のソフトデリート保持（status: obsolete）を確定し、過去実績の参照整合性を永続的に維持することを確定する（計画 06 章 Append-only 原則との整合）。

## 7. 編集権限と変更履歴

### 7-1. マスタ別 RBAC マトリクス

計画 08 章 §4 の RBAC 6 ロールに基づき、マスタ × 操作 × ロールの粒度で権限を確定する。

| マスタ | 作成（ドラフト） | 承認・公開 | 改訂 | 廃止 | 参照 |
|---|---|---|---|---|---|
| SOP | 品質担当 | 品質担当（電子署名必須）| 品質担当 | 品質担当 | 全ロール |
| ユーザー・スキル・資格 | 現場監督 | 現場監督 | 現場監督 | 現場監督 | 品質担当・管理者 |
| 設備マスタ・校正記録 | 保全担当 | 保全担当 | 保全担当 | 保全担当 | 全ロール |
| カスタム Step タイプ | 品質担当・IT 担当 | 品質担当（電子署名必須）| 品質担当・IT 担当 | 品質担当 | 全ロール |
| step_flow_rule（条件分岐）| 品質担当 | 品質担当（電子署名必須）| 品質担当 | 品質担当 | 全ロール |
| external_key_binding | IT 担当 | IT 担当・品質担当 | IT 担当 | IT 担当 | 品質担当・管理者 |

作業員ロールはマスタの作成・承認・改訂・廃止を一切行えない（RBAC で技術的に禁止）。

### 7-2. master_edit_event による変更履歴記録

マスタの全編集操作は `master_edit_event` テーブルに Append-only で記録する。

| フィールド | 型 | 説明 |
|---|---|---|
| id | UUID | 主キー |
| master_type | VARCHAR | 操作対象マスタの種別（`sop` / `user` / `equipment` / `step_type` / `flow_rule` 等） |
| master_id | UUID | 操作対象レコードの ID |
| event_type | ENUM | `draft_saved` / `submitted_for_review` / `approved` / `published` / `obsoleted` / `deleted` |
| diff | JSONB | 変更前後の差分（JSON Patch 形式） |
| performed_by | UUID | 操作者のユーザー ID（ALCOA+ Attributable） |
| performed_at | TIMESTAMPTZ | 操作時刻・端末時刻＋サーバー受信時刻（ALCOA+ Contemporaneous） |

このテーブルは Append-only であり、INSERT のみ許容する。UPDATE / DELETE を API レベルで禁止する。

### 7-3. 重要マスタの承認フロー

以下のマスタの「公開」操作には品質担当の電子署名（ユーザー ID + タイムスタンプ + サーバー署名）を必須とする：

- SOP（全改訂）
- カスタム Step タイプ（新規登録・バージョン更新）
- step_flow_rule（新規登録・条件変更）

電子署名の技術的実装は計画 08 章 §4「電子署名」の方針に従う。

---

**本節で確定した方針**
- マスタ別 RBAC マトリクスを上表の通り確定する。作業員のマスタ編集を RBAC で技術的に禁止することを確定する。
- `master_edit_event` テーブルを全マスタ編集操作の Append-only 記録先として確定する。
- SOP・カスタム Step タイプ・step_flow_rule の公開には品質担当の電子署名を必須とすることを確定する。

## 8. 編集者向け非機能要件

計画 08 章 §2「管理 Web 編集 UX 非機能要件」で確定した数値を本節に転記し、設計上の制約として明示する。

| 区分 | 要件 | 基準値 | 根拠 |
|---|---|---|---|
| 管理 Web レスポンス | 保存応答時間 | P95 で 500ms 以内 | 08 章 §2 |
| 管理 Web レスポンス | マスタ検索応答時間 | P95 で 200ms 以内 | 08 章 §2 |
| 編集 UX | オートセーブ間隔 | 30 秒 | 本章 §3-3 |
| 編集 UX | ドラフト保持期間 | 30 日（最終編集から） | 08 章 §2 |
| 同時編集 | 競合検出モデル | 楽観的ロック（バージョン番号比較） | 08 章 §2 |
| 編集操作 | Undo/Redo 上限 | 直近 50 ステップ | 本章 §3-5 |
| ビジュアル DSL エディタ | プレビュー実行応答 | 200ms 以内（クライアント完結） | 本章 §4-3 |
| 一括インポート | 最大行数 | 1,000 行 / リクエスト（ver1.0.0）| 本章 §5-1 |

### 同時編集モデルの詳細（楽観的ロック）

複数の編集者が同一マスタを同時に編集しようとした場合、以下の楽観的ロックで競合を検出する：

1. マスタのドラフトには `lock_version` フィールドを持つ
2. 編集者が保存する際に `lock_version` をリクエストに含める
3. サーバー側で `lock_version` が一致しない場合は保存を拒否し、競合を通知する
4. 競合通知を受けた編集者は差分ビューで変更を確認し、手動でマージまたは上書きを選択する

リアルタイム共同編集（CRDT 等）は ver1.0.0 のスコープ外（§10 参照）。

### 大量データ表示

大量のマスタレコードを表示する画面では、以下の技術を適用する：

- **仮想スクロール**: 画面に表示される行のみ DOM に描画する
- **サーバーサイドページング**: 検索クエリはサーバー側で実行し、ページ単位で結果を返す
- **インクリメンタル検索**: 検索入力の 300ms デバウンス後にサーバーリクエストを送信する

---

**本節で確定した方針**
- 管理 Web の保存 P95 500ms 以内・検索 P95 200ms 以内を非機能要件として確定する。
- 楽観的ロック（バージョン番号比較）を同時編集の競合検出モデルとして確定する。
- 仮想スクロール・サーバーサイドページングを大量データ表示の実装方針として確定する。

---

## 9. ALCOA+ との整合

本章で設計する全機能は ALCOA+ の 9 原則と以下の通り整合する。

| ALCOA+ 原則 | 本章での実装 |
|---|---|
| **Attributable（帰属明確性）** | `master_edit_event.performed_by` でドラフト保存・公開・廃止の全操作が誰によるかを記録する |
| **Contemporaneous（同時性）** | `master_edit_event.performed_at` に端末時刻とサーバー受信時刻を両方記録する |
| **Legible（判読可能性）** | 差分ビューの color-coded 表示（追加/削除/変更）により、変更内容を人間が直接判読できる |
| **Original（原本性）** | ビジュアル DSL エディタは JSON Logic の正規化形式と UI 状態（演算子ツリー）の両方を保持する |
| **Accurate（正確性）** | プレビュー実行でバリデーションロジックの正確性を事前確認できる |
| **Complete（完全性）** | 削除されたマスタはソフトデリート（`deleted_at` フィールド + `deleted` イベント）で保持し、記録の欠損を防ぐ |
| **Consistent（一貫性）** | 楽観的ロックにより複数編集者による不整合書き込みを防ぐ |
| **Enduring（耐久性）** | `master_edit_event` テーブルの物理削除禁止（Append-only）により、全変更履歴を永続保持する |
| **Available（利用可能性）** | 任意の時点の「公開版」を `lock_version` と `effective_date` の組み合わせで再現できる |

---

**本節で確定した方針**
- 本章の全設計が ALCOA+ 9 原則の各々に対応することを確定する。
- `master_edit_event` テーブルの Append-only 制約（UPDATE / DELETE の API レベルでの禁止）を確定する。
- 削除マスタのソフトデリート方針を確定する（`deleted_at` + `master_edit_event` の `deleted` イベントによる二重記録）。

## 10. やらないこと

本節に列挙する機能は、ver1.0.0 のスコープから意図的に除外する。除外は消極的判断ではなく、スコープ超過による品質低下リスクを回避するための積極的な設計判断である。

| やらないこと | 理由 |
|---|---|
| **リアルタイム共同編集（CRDT・Operational Transform）** | Google Docs のような複数人の同時入力・即時反映は、Conflict-Free Replicated Data Type の実装複雑度が個人開発の許容範囲を超える。楽観的ロック（§8）で実用的な要件を満たす |
| **クラウドネイティブな AI 補完（SOP 自動生成・タイトル補完等）** | 外部 AI API への依存はオンプレ環境の制約違反（計画 05 章 OT-IT 境界方針）および社内固有知識のプライバシーリスクを生む。AI/ML は 03 章の対象外判断に従い対象外とする |
| **カスタム Step 定義以上の挙動カスタマイズ（任意コード実行）** | eval / Function constructor による任意コード実行は、セキュリティリスク・Offline-First 設計の破壊・ALCOA+ 整合の困難化から永続非提供とする（計画 18 章 §7 参照）|
| **マスタ削除時の cascade 削除** | 参照中レコードの cascade 削除は ALCOA+ の Complete 原則（記録の欠損禁止）と直接矛盾する。削除は常に明示的な参照解除フローを必須とする（§6-3 参照） |
| **マスタ版管理の自動マージ** | 自動マージは意図しない変更が静かに本番に反映されるリスクを持つ。競合は常に人間が差分ビューを確認して手動解決する（§8 楽観的ロック参照）|
| **外部認証プロバイダ（OAuth2 / SAML / LDAP 連携）** | ver1.0.0 はオンプレ JWT 認証に集中する（計画 08 章 §4）。外部 IdP 連携は将来の拡張余地として留保するが、ver1.0.0 スコープでは対象外とする |

---

**本節で確定した方針**
- リアルタイム共同編集・AI 補完・任意コード実行・cascade 削除・自動マージ・外部 IdP 連携の 6 事項を対象外と判断する。
- 「やらないこと」の理由をすべて技術的・設計的根拠で明示し、「将来的に検討する」等の曖昧な留保は付けない。

---
