# 09 運用・組織・教育方針

本章の責務は、単一工場オンプレかつ個人開発という制約下での運用・教育・改善ループの方針を確定することである。01 章の倫理境界宣言における「監視と支援の構造的分離」を運用面で実装する権威章として位置づける。

---

## 1. 想定する運用体制

### IT 担当 1 名前提の根拠

本プロジェクトの対象は中小製造業・単一工場である。この規模の製造業における IT 担当は 1 名（兼務を含む）が現実的な前提である。本システムの運用設計は IT 担当 1 名での継続運用を必須制約とする。IT 担当が複数名いる場合は本設計が緩和される方向でのみ変更を認める。

### 役割分担の確定

| 役割 | 担当者 | 主要な責務 |
|---|---|---|
| システム管理者 | IT 担当 | サーバー保守・バックアップ確認・ユーザーアカウント管理・インフラ設定 |
| SOP 管理者 | 品質担当 | SOP の作成・改訂・承認・廃止・版管理・CAPA 管理 |
| ユーザー管理者 | 現場監督 | 作業員の追加・スキル認定・資格有効期限更新・現場シフト管理 |
| 監査者 | 品質担当・経営層 | トレサビデータの照会・ALCOA+ 適合確認・倫理品質の監視 |

役割ごとの権限範囲は 08 章の RBAC 6 ロール定義に従う。SOP 内容の承認権限は品質担当のみが持つ。IT 担当はシステム権限を持つが、SOP 内容の承認権限を持たない。この分離は技術的（RBAC）かつ文書的（本方針）に確定する。

### 初期構築の自立性

IT 担当 1 名が外部 SI 支援なしで初期構築を完了できることを設計要件とする。実現手段は以下の通りである。

- Docker Compose ワンコマンド起動による環境セットアップ
- GUI セットアップウィザードによる初期設定（工場情報・ユーザー・SOP 初期データ）
- アプリ内ヘルプとして提供する運用マニュアル（紙運用マニュアルに依存しない）

M6 リリース判定基準（10 章）に「IT 担当 1 名での初期構築が GUI ウィザードのみで完了することを確認」を含める。

---

**本節で確定した方針**
- 運用体制を IT 担当（システム管理者）・品質担当（SOP 管理者）・現場監督（ユーザー管理者）・品質担当＋経営層（監査者）の 4 役割で確定する。
- SOP 承認権限は品質担当のみに帰属し、IT 担当は SOP 内容の承認権限を持たないことを RBAC で強制する。
- Docker Compose + GUI ウィザードで IT 担当 1 名による初期構築を完結させる設計を必須とする。

## 2. マスタ管理ガバナンス

### SOP 改訂フロー

SOP の権威は品質担当にある。以下のフローを確定し、フローに従わない SOP 変更をシステムが技術的に拒否する。

1. **申請受付**: 作業員アプリから改善提案・不適合報告・4M 変更申請を起票する。申請はシステムに記録され、品質担当に自動通知が届く。
2. **ドラフト作成**: 品質担当が申請内容を精査し、SOP ドラフト版を作成する。ドラフト段階では公開されない。
3. **確認・承認**: 現場監督がドラフトを確認し、品質担当が最終承認を行う。承認操作には電子署名（ユーザー ID + タイムスタンプ + サーバー署名）を付与する。
4. **Effective Date 設定**: 承認後、有効化日（Effective Date）を設定する。有効化日より前は「承認済み・未発効」の状態として管理する。
5. **旧版の廃止処理**: 新版有効化と同時に旧版に「廃止」ラベルを付与する。旧版レコードは削除しない（ALCOA+ Enduring）。
6. **教育完了後の有効化**: 対象作業者への新版通知が配信され、教育完了の確認レコードが登録された後に新版が現場作業者に表示される。

### バージョン管理の原則

SOP の版は `{文書番号}_{メジャー}.{マイナー}_{Effective Date}` 形式で管理する。メジャー改訂（手順の構造変更）とマイナー改訂（誤字脱字・明確化）を区別し、改訂種別を記録する。いかなる版も削除しない。

### ユーザー管理の分担

作業員の追加・スキル認定登録・資格有効期限更新は現場監督が担う。IT 担当がユーザー情報を操作できるのはシステム障害時の緊急対応に限定し、操作ログを監査証跡として残す。

業界分析 [`90_業界分析/28_不適合と手順改訂のフィードバックループ.md`](../../90_業界分析/28_不適合と手順改訂のフィードバックループ.md) が示す CAPA (ISO 9001:2015 §10.2) フレームワークを本フローの設計基準として適用する。

### 編集者体験の責務委譲

マスタ編集者の体験設計（オーサリング UX・ビジュアル DSL エディタ・参照整合性ガード・一括操作・テンプレート）の詳細は 19 章「マスタ編集者体験設計（オーサリング UX とガバナンス）」に委譲する。本節は SOP 改訂のガバナンス（承認権限・電子署名・バージョン番号規則・廃止処理）に責任を持つ。

カスタム Step タイプ定義（18 章）および条件分岐ルール（18 章）の編集は、19 章のビジュアル DSL エディタを通じて行うことを確定する。編集者向け教育には「DSL エディタの操作習熟（カスタム Step 定義・条件分岐ルール作成）」を新版有効化前の準備要件として含める。

---

**本節で確定した方針**
- SOP 改訂フローを申請受付→ドラフト→確認・承認→有効化日設定→廃止処理→教育完了確認の 6 段階で確定する。
- 電子署名（ユーザー ID + タイムスタンプ + サーバー署名）を承認操作の必須要件とする。
- いかなる SOP 版も削除せず Append-only で保持し、ALCOA+ Enduring に準拠する。

## 3. 不適合・CAPA・Kaizen Teian のフィードバックループ

図: fig_capa_loop.svg（img/ 配下）を参照

### 4 段階フィードバックループ

本システムの不適合管理・改善活動は以下の 4 段階ループで設計する。

**ステップ 1 — 検知**

作業員がタブレット APP から以下のいずれかを起票する。

- 不適合報告（製品・工程の品質逸脱）
- ヒヤリハット報告（未遂事象・リスク発見）
- Kaizen Teian（改善提案）

起票は写真添付・自由テキスト・定型タグの 3 手段を組み合わせて行う。起票後、品質担当に自動通知が配信される。

**ステップ 2 — 分析**

品質担当が起票内容を精査し、CAPA（是正処置・予防処置）に昇格するかどうかを判断する。CAPA に昇格した案件は以下の根本原因分析手法を選択して適用する。

- Why-Why 分析（5 回の「なぜ」による原因追跡）
- Ishikawa 4M 分析（Man / Machine / Material / Method の 4 要因分類）

分析結果は CAPA レコードに記録し、起票者（作業員）に中間状態を通知する。

**ステップ 3 — 対策**

根本原因分析の結果に基づき、以下のいずれかまたは組合せで対策を実施する。

| 対策類型 | 担当者 | システム上の記録 |
|---|---|---|
| SOP 改訂 | 品質担当 | CAPA レコードに改訂版 SOP ID を紐付ける |
| 設備保全 | 保全担当 | 設備保全記録として別途登録し CAPA に参照リンクを設定する |
| 教育実施 | 現場監督 | 教育完了レコードを CAPA に紐付ける |

**ステップ 4 — 確認**

改訂後 SOP または対策実施後に再作業を行い、効果確認レコードを CAPA に登録してクローズする。クローズ操作には品質担当の電子署名を必須とする。クローズ後、起票者（作業員）に「採用・完了通知」を送付する。

### ループの健全性指標

CAPA・Kaizen Teian ループの形骸化を検知するため、以下の指標を管理ダッシュボードに表示する。

| 指標 | 目的 |
|---|---|
| Kaizen Teian 採用率（採用数 / 起票数） | 品質担当の応答品質の計測 |
| 平均クローズ日数（起票〜CAPA クローズ） | 対策速度の計測 |
| 未読件数・レスポンス遅延アラート | 品質担当の応答遅延の検知 |

採用された改善提案には「採用通知」を提案者に返信する。起票が黙殺されない設計は 39 章の Kaizen Teian の本質的要件であり、心理的安全（08 章 倫理品質）の運用実装である。

業界分析 [`90_業界分析/39_QCサークル・Kaizen Teianとボトムアップ品質活動.md`](../../90_業界分析/39_QCサークル・Kaizen Teianとボトムアップ品質活動.md) および [`90_業界分析/28_不適合と手順改訂のフィードバックループ.md`](../../90_業界分析/28_不適合と手順改訂のフィードバックループ.md) を本節の設計基準として適用する。

---

**本節で確定した方針**
- 不適合・CAPA・Kaizen Teian のフィードバックループを検知→分析→対策→確認の 4 段階で確定する。
- 採用通知の返信を必須とし、起票が黙殺されない設計を Kaizen Teian の実装要件とする。
- Kaizen Teian 採用率・平均クローズ日数・未読件数をループ健全性指標として管理ダッシュボードに表示する。

## 4. 教育・訓練設計

### ADDIE 簡易版の適用

本プロジェクトは製造現場向けの迅速化を考慮し、ADDIE モデル（Analysis / Design / Development / Implementation / Evaluation）の簡易版を採用する。現場 OJT とアプリ内ガイドを補完的に組み合わせる。

| ADDIE フェーズ | 本プロジェクトでの実装 |
|---|---|
| Analysis | SOP 改訂・スキルマップ・不適合履歴からニーズを分析する |
| Design | 習熟度別 UI（Scaffolding/Fading）の設計を 04 章ユースケースで完了させる |
| Development | SOP の多言語版を含む教育コンテンツをアプリ内に埋め込む |
| Implementation | 現場 OJT + アプリ内ガイドの補完構造で実施する |
| Evaluation | Kirkpatrick 4 レベル評価で計測する（後述） |

### 認知的徒弟制（Scaffolding/Fading）の実装

習熟度別 UI として以下の 2 段階を実装する。

- **Scaffolding（足場がけ）**: 習熟度低の作業者には詳細な手順ガイド・写真・動画を表示し、確認ボタンを強調する。
- **Fading（足場外し）**: 習熟度高の作業者には手順の骨格のみを表示し、詳細は折り畳む。

習熟度の判定基準（ステップ完了時間・エラー率・作業日数）は要件定義フェーズで確定する。

### Kirkpatrick 4 レベル評価の実装

| レベル | 評価内容 | 計測手段 |
|---|---|---|
| レベル 1（反応） | 教育・SOP への満足度 | アプリ内フィードバック（1〜5 段階評価） |
| レベル 2（学習） | 手順の正確な習得 | アプリ内習熟度推定・ステップ所要時間の自動計測 |
| レベル 3（行動） | 現場での行動変容 | 月次の工程改善会議でアプリデータを参考値として評価する |
| レベル 4（業績） | 工程品質・不適合率への影響 | 月次の工程集計値（個人別ではなく工程・シフト集計）で評価する |

### Spaced Repetition（間隔反復学習）の方針

長期ブランク（30 日超）・稀工程（月 1 回以下）の担当前に、アプリが復習アラートを表示する。復習アラートは 04 章のシナリオ E（認定期限切れ作業者の介入）と連動する。

### 多言語教育の方針

SOP の多言語版承認を「新版有効化の条件」に含める。品質担当は母語話者レビューが完了した多言語版を添付することで新版を有効化できる。母語話者レビューなしの単一言語版のみでの有効化は認めない（外国人労働者が在籍する工場において）。

業界分析 [`90_業界分析/23_作業訓練設計とインストラクショナルデザイン.md`](../../90_業界分析/23_作業訓練設計とインストラクショナルデザイン.md) の ADDIE・Kirkpatrick 4 レベル・Spaced Repetition を本節の設計基準として適用する。

---

**本節で確定した方針**
- ADDIE 簡易版と Scaffolding/Fading による習熟度別 UI を教育設計の基盤として確定する。
- Kirkpatrick レベル 1-2 をアプリ内自動計測・レベル 3-4 を月次の工程改善会議で評価することを確定する。
- 外国人労働者が在籍する工場では、SOP の多言語版承認を新版有効化の必須条件とする。

---

## 5. BCP・縮退運用・紙フォールバック

### 3 段階の縮退運用

障害の深刻度に応じて 3 段階の縮退運用を定義する。

**第 1 縮退 — Emergency Mode（サーバー障害）**

サーバーへの到達不能状態において、タブレット APP はローカルキューへの記録を継続する（Offline-First 設計）。作業員は通常と同一の操作でトレサビデータを記録できる。サーバー復旧後、ローカルキューの内容を自動同期する。この縮退モードは 04 章シナリオ G に対応する。

**第 2 縮退 — タブレット電源断**

タブレットが故障・電源断となった場合、交換用端末に切り替える。ローカルデータは暗号化バックアップから復元する。各工程に予備タブレット 1 台の配備を運用ガイドラインとして推奨する。

**第 3 縮退 — 完全停電・大規模障害（紙フォールバック）**

電力・ネットワークが完全に停止した場合、紙 SOP に切り替える。以下の管理規則を確定する。

| 管理項目 | 方針 |
|---|---|
| 紙 SOP の更新頻度 | 各 SOP の最新版 PDF を週次で印刷する |
| 保管場所 | 各工程の紙ファイル（ラミネート加工）に格納し、作業者が即座にアクセスできる位置に設置する |
| 後入力の方針 | アプリ復旧後に `is_retroactive: true` フラグ付きで後入力し、紙原本を添付スキャンとして記録する |
| 紙記録の真正性 | 後入力レコードに「手書き日時・記入者氏名・紙原本スキャン ID」を必ず含める |

### 復旧確認ワークフロー

災害・大規模障害からの復旧後、以下の確認手順を実施する。

1. 設備復旧チェックリストを実行し、全設備の動作確認レコードを登録する。
2. 損傷箇所の写真を損傷記録として登録する。
3. 担当者（現場監督・保全担当）の電子署名で復旧確認をクローズする。
4. 復旧確認レコードは通常作業ログと同一ストレージに格納し、トレサビ一元管理の対象とする。

業界分析 [`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md) の ISO 22301:2019 フレームワークを本節の設計基準として適用する。

---

**本節で確定した方針**
- Emergency Mode（Offline-First）・タブレット交換・紙フォールバックの 3 段階縮退運用を確定する。
- 紙フォールバック中の記録は復旧後に `is_retroactive: true` フラグ付き後入力と紙原本スキャンで対応する。
- 復旧確認は設備チェックリスト・損傷写真・担当者署名の 3 点セットで完結させ、通常ストレージに格納する。

## 6. 監視と支援の境界宣言の運用面での実装

### 倫理境界の運用ルール

01 章の倫理境界宣言を運用ルールに落とし、以下の通り確定する。

| ルール | 実装手段 |
|---|---|
| 作業ログデータを個人評価・人事査定に使用しない | RBAC での権限制限 + 本方針の文書化 |
| 月次の工程改善会議で参照するのは工程集計値・シフト集計値のみ | 管理 Web のレポート画面から個人別表示を削除する |
| 不適合・ヒヤリハットの報告者 ID は品質担当のみ閲覧可 | RBAC で権限を限定する。匿名化オプションを設ける |
| データ利用ポリシー同意書の取得 | 導入時に全作業員が署名（多言語版を提供する） |

### Just Culture の組織原則

業界分析 [`90_業界分析/13_安全文化と安全管理システム.md`](../../90_業界分析/13_安全文化と安全管理システム.md) が示す Just Culture の 4 文化（報告文化・公正文化・柔軟文化・学習文化）を本プロジェクトの組織運用原則として採用する。

具体的な実装方針は以下の通りである。

- スリップ・ラプス（不注意ミス）による不適合は、CAPA 起票時に「個人責任」ではなく「システム / 工程要因」として分類する。この分類はプルダウン選択肢としてアプリ内に実装し、運用ルールとして周知する。
- ヒヤリハット報告件数の増加を「安全文化の改善指標」として経営層に月次報告する。報告件数の増加を罰則の根拠としない。この方針は月次レポートのテンプレートに注記として固定する。
- ISO 45001:2018 の安全管理システム原則を本システムの設計指針として位置づける。規制への「対応する」立場で導入する。

---

**本節で確定した方針**
- 作業ログの個人評価への転用禁止・個人別表示の削除・報告者 ID の権限制限を RBAC で技術的に強制する。
- Just Culture の原則に基づき、スリップ・ラプスを工程要因として分類し、ヒヤリハット増加を安全文化の改善指標とする。
- 導入時の「データ利用ポリシー同意書」多言語署名を必須とし、倫理境界宣言の運用上の根拠とする。
