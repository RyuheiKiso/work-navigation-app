# 07 技術スタック選定根拠

本章の責務は、本システムが採用する各技術要素の選定根拠と代替案の評価を確定することである。05 章で確定したアーキテクチャ原則を実現する技術として各要素を位置づけ、代替案を排除した根拠を記録する。本章の記述は ADR（アーキテクチャ決定記録）として機能し、将来の再評価の基点となる。

---

## 1. 選定の評価軸

本章における技術選定は、以下の 5 軸で評価する。すべての代替案比較はこの評価軸に基づく。

| 軸 | 評価の問い |
|---|---|
| **中小製造業オンプレ運用適合性** | Windows Server 環境での安定動作・社内 IT 担当 1 名での継続運用が可能か |
| **個人開発・長期保守性** | 10 年後も個人または小チームが保守可能か。技術的負債の累積リスクを最小化できるか |
| **オフライン First と改竄不可性の実現** | 05 章で確定した Offline-First・Append-only Event Sourcing・ハッシュ証明・Idempotent API の原則を実現できるか |
| **製造現場 UI/UX 品質** | 手袋着用・夜勤・多言語・高齢者対応が技術的に実現可能か |
| **コスト** | ランニングコスト・ライセンスコスト・技術習得コストが個人開発の制約範囲内か |

5 軸の重みは等価ではない。「オフライン First と改竄不可性の実現」は本アプリの存在根拠に直結するため、他の軸に優先する。

---

**本節で確定した方針**
- 技術選定の評価軸を 5 軸に確定し、「オフライン First と改竄不可性の実現」を最高優先軸とする。
- 代替案の排除根拠は必ずこの 5 軸に照らして記述し、感覚・慣習を根拠とした排除は行わない。
- 本節で確定した評価軸は技術スタック変更時の再評価においても有効とする。

## 2. タブレット APP: React Native + SQLite + TypeORM の選定根拠

### 選定根拠

**OS 横断の単一コードベース対応**: 中小製造業の現場で実際に使用されるタブレットは Android・iOS・Windows の混在が想定される。本アプリは React Native（Android/iOS 対応）および react-native-windows（Windows タブレット対応）により、3 OS を単一コードベースで対応する。OS ごとに独立したネイティブアプリを保守することは、個人開発の長期保守性評価軸において許容できない。

**Offline-First 原則の直接実現**: SQLite によるデバイスローカル永続化と TypeORM による ORM 層は、05 章の Offline-First 原則および Local-First 参照モデルを直接実現する。端末ローカルを「一次記録メモリ」として扱うアーキテクチャは、SQLite の存在なしに成立しない。TypeORM はスキーマ管理・マイグレーション・型安全なクエリを提供し、SQLite 直操作による運用リスク（スキーマ不整合・型ミスマッチ）を排除する。

**TypeScript による長期保守性**: TypeScript の静的型システムは、個人開発において型安全性を長期にわたって維持する唯一の実践的手段である。型定義はドキュメントとしても機能し、数年後に自身がコードを読み返す際の理解コストを低減する。

**React エコシステムとの技術統一**: 管理 Web（React）との技術スタック統一により、コンポーネント設計の知識・カスタムフック・型定義が再利用可能となる。個人開発での習得コスト削減評価軸において有効である。

**Expo によるビルド管理と OTA 対応**: Expo は iOS/Android のビルド環境をクラウドに委譲し、ローカルビルド環境の構築・維持コストを削減する。OTA（Over-the-Air）アップデートによりアプリストア審査なしでのパッチ配布が可能であり、オンプレ環境での迅速な不具合修正に適している。

**製造現場 UI/UX 品質の実現**: react-native-camera および ML Kit の組み合わせにより、オフライン環境でのバーコード・QR スキャンが実現可能である。カスタムコンポーネントによりタッチターゲットサイズを 72dp 以上に設定でき、手袋着用を前提とした操作設計が可能である。

### 代替案と却下理由

| 代替案 | 却下理由 |
|---|---|
| Flutter | Windows Tablet への対応（flutter-windows）は成熟度が低く、中小製造業での Windows タブレット採用実績が乏しい。Dart VM のオーバーヘッドが低スペック端末での動作安定性に影響するリスクがある |
| Kotlin Multiplatform | iOS・Windows 対応の成熟度が不十分。個人開発での Kotlin 学習コストが、既存の TypeScript 知識との技術分散を招く |
| PWA（Progressive Web App） | 製造現場での要件であるカメラ制御・BLE アクセス・バックグラウンド処理の制限が、iOS Safari における恒常的な課題として残る。バッテリー消費も高い |
| ネイティブ Android のみ | Windows タブレットは自動車部品・精密機械工場での標準端末として一定の比率を占める。Android のみでは Windows 対応を必要とする顧客セグメントを排除する |

詳細は [`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md) および [`90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md`](../../90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md) を参照すること。

---

**本節で確定した方針**
- React Native + SQLite + TypeORM を Android/iOS/Windows の 3 OS 横断対応・Offline-First 原則の実現・TypeScript による長期保守性の根拠から採用することを確定する。
- Flutter・Kotlin Multiplatform・PWA・Android ネイティブの 4 案を代替案として評価し、成熟度・技術分散・製造現場制約を根拠に排除することを確定する。
- タッチターゲット 72dp 以上の UI 設計基準を手袋着用対応の製造現場 UI/UX 品質評価軸から確定する。

## 3. 管理 Web: React の選定根拠

### 選定根拠

**タブレット APP との技術スタック統一**: React Native（タブレット APP）との技術統一により、コンポーネント設計・カスタムフック・型定義・テストコードが再利用可能となる。個人開発において 2 種の UI フレームワークを並行保守することは、長期保守性評価軸において許容できない。

**静的 SPA としてのデプロイ簡略化**: 管理 Web を静的 SPA（Single Page Application）としてビルドし、バックエンドサーバーから配信する設計が可能である。これにより Node.js サーバー等の追加プロセスが不要となり、デプロイ構成の単純化・運用負荷の低減・中小オンプレ環境への適合性が向上する。

**TypeScript との親和性と長期保守性**: React は TypeScript との統合が成熟しており、型安全なコンポーネント開発・プロップスの型定義・状態管理の型強制が可能である。これは長期保守性評価軸において中核的な選定根拠となる。

**管理画面コンポーネントライブラリの充実**: Ant Design・MUI（Material UI）等の成熟したコンポーネントライブラリが存在し、データテーブル・フォーム・Audit Trail ビューアー・マスタ管理画面の実装工数を大幅に削減できる。個人開発において実装工数の削減は直接的に品質維持に寄与する。

### 代替案と却下理由

| 代替案 | 却下理由 |
|---|---|
| Vue.js | 技術スタックの分散。タブレット APP（React Native）との統一が崩れ、個人開発での保守コストが増加する。Vue の選定根拠自体は同等だが、統一性を優先する |
| Angular | 大規模チーム開発を前提とした設計思想。個人開発での習得コストおよびボイラーテンプレートのオーバーヘッドが、本プロジェクトのスケールに対してオーバーエンジニアリングとなる |
| Svelte | エコシステムの成熟度と長期保守性に不確実性がある。コンポーネントライブラリの選択肢が限られ、管理画面の実装工数削減が見込みにくい |

---

**本節で確定した方針**
- React を管理 Web として採用し、タブレット APP（React Native）との技術スタック統一・静的 SPA デプロイ・TypeScript 親和性を選定根拠として確定する。
- Vue.js・Angular・Svelte の 3 案を代替案として評価し、技術分散・スケール不適合・成熟度不確実性を根拠に排除することを確定する。
- 管理 Web はバックエンドサーバーから静的配信し、Node.js サーバーは不要とする構成を確定する。

## 4. バックエンド: Rust Edition 2024 + tokio + axum の選定根拠

### 選定根拠

**安全性と改竄不可性の実現**: Rust のメモリ安全性保証は、コンパイル時にバッファオーバーフロー・use-after-free・データ競合等の脆弱性クラスを排除する。トレサビ記録システムとして、実行時の予期しないメモリ破壊による記録データの損傷・改竄を防止することは、05 章の改竄不可性原則に直接対応する。GC（ガベージコレクション）を持たないため、GC 停止による不意のレイテンシスパイクが作業記録の受信処理に影響しない。

**Edition 制度による長期互換性**: Rust の Edition 制度は互換性を保ちながら言語を進化させる仕組みである。Edition 2024 への移行に際して以前の Edition で書かれたコードはコンパイルを継続できる。個人開発の長期保守性評価軸において、言語の後方互換性破壊リスクを制度的に低減している。

**tokio による高スループット非同期処理**: tokio は Rust の非同期ランタイムとして実績のある実装であり、複数のタブレット端末からの同時接続を高スループットで処理する。製造現場でのピーク時（シフト開始・終了時の一斉記録送信）に対してスレッドプールのオーバーヘッドなしに対応できる。

**axum の型安全なルーティングと Tower ミドルウェア**: axum は Tower サービストレイトをベースとし、ミドルウェアの合成によって認証・ロギング・リクエスト検証を型安全に構成できる。05 章の Idempotent API 原則の実装において、Idempotency Key の検証ミドルウェアを型システムで保証できる設計が可能である。

**sqlx によるコンパイル時クエリ検証**: sqlx はコンパイル時に SQL クエリの構文・型整合・カラム名をデータベーススキーマに対して検証する。05 章の Append-only Event Sourcing 原則において、INSERT 専用クエリの型安全性をコンパイル時に保証し、誤った UPDATE クエリの混入をビルド段階で防止する。

### 2バイナリ採用の追加根拠

本バックエンドは **wnav_terminal_api**（ポート 8080・ハンディ端末向け）と **wnav_master_api**（ポート 8081・管理系向け）の2バイナリに分割する。単一バイナリ構成に対する2バイナリ採用の根拠を以下に確定する。

| 根拠 | 説明 |
|---|---|
| **DB ロール3分離の物理保証** | PostgreSQL のロール設計（app_event_insert・app_write・app_read の3ロール）は、バイナリレベルで物理プロセスを分離することで OS プロセス境界による強制を実現する。単一バイナリでは同一プロセス内に端末向けコードと管理系コードが混在し、接続ロールの誤適用をコードレビュー以外で防止できない。2バイナリ構成では OS プロセス境界が誤接続の物理的防壁となる |
| **独立可用性（障害ドメイン分離）** | 現場稼働（ハンディ端末の作業記録・Outbox 同期）と管理操作（帳票生成・マスタ更新・バッチ集計）は異なる障害ドメインに属する。master-api のメンテナンス・再起動・障害が terminal-api の稼働に影響しないことを OS プロセス分離により保証する。これは NFR の可用性要件（製造ライン稼働時間中の作業記録受信の無停止）に直結する |
| **セキュリティ境界（ネットワーク分離可能性）** | terminal-api はハンディ端末のみがアクセスする（製造フロア LAN セグメント）。master-api は管理 PC のみがアクセスする（管理部門 LAN セグメント）。2バイナリ構成により、ネットワーク ACL・ファイアウォールルールでハンディ端末と管理 PC のアクセス先を物理的に分離できる。単一バイナリでは同一エンドポイントへの混在アクセスとなり、ネットワークレベルでの分離が困難になる |

> **Rust + tokio + axum 技術自体は変更しない**: 2バイナリ採用は「何をビルドするか」の構成決定であり、「何の技術を使うか」の選定ではない。Rust Edition 2024 + tokio + axum の選定根拠（メモリ安全性・GC 停止なし・Tower ミドルウェア）は2バイナリ構成においても同等に有効である。

### 代替案と却下理由

| 代替案 | 却下理由 |
|---|---|
| Go（Gin/Echo） | GC は存在するが STW（Stop-The-World）停止の影響は小さい。しかし、Rust の所有権モデルによるコンパイル時安全性と比較して、並行処理のデータ競合をコンパイル時に検出できない。長期保守性において型システムの表現力が劣る |
| .NET / C# | Windows 偏重のエコシステムにより、WSL2 + Linux コンテナとの統合に摩擦がある。ランタイムのバージョン管理・Docker イメージサイズ・GC 停止が課題となる |
| Node.js（Express/Fastify） | イベントループのシングルスレッド制限（CPU バウンド処理の分散困難）・TypeScript での型強制の限界（実行時型安全性の欠如）・npm エコシステムの依存脆弱性頻発リスクが長期保守性を損なう |
| Python（FastAPI） | GIL（Global Interpreter Lock）制約・実行速度・型注釈は実行時に強制されない点が、製造業トレサビシステムの信頼性要件と相性が悪い |

詳細は [`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md)（Append-only Event Sourcing の実装先としての設計判断）を参照すること。

---

**本節で確定した方針**
- Rust Edition 2024 + tokio + axum を採用し、コンパイル時メモリ安全性・GC 停止なし・型安全な SQL 検証を改竄不可性原則の実現根拠として確定する。
- sqlx のコンパイル時クエリ検証を Append-only Event Sourcing の保護機構として位置づけることを確定する。
- Go・.NET・Node.js・Python の 4 案を代替案として評価し、GC 停止リスク・Docker 摩擦・型安全性欠如・GIL 制約を根拠に排除することを確定する。
- 単一バイナリに対する **wnav_terminal_api + wnav_master_api の2バイナリ採用**を、DB ロール3分離の物理保証・独立可用性（障害ドメイン分離）・セキュリティ境界（ネットワーク分離可能性）の3根拠から確定する。Rust + tokio + axum の選定技術自体は変更しない。

## 5. DB: PostgreSQL の選定根拠と SQLite との整合性方針

### PostgreSQL の選定根拠

**Append-only 制約のロールレベル実装**: 05 章のアーキテクチャ原則は作業ログテーブルへの UPDATE/DELETE をデータベースロールレベルで禁止することを要求する。PostgreSQL はロールごとに細粒度の権限（INSERT のみ・SELECT のみ等）を設定でき、この原則を直接実現できる。

**JSONB 型による属性の柔軟な格納**: 作業ログのイベント Attributes（測定値・写真ハッシュ・スキャン値・SOP 版 ID 等）は Step 種別によって構造が異なる。JSONB 型はスキーマの柔軟性を保ちながら、インデックスによる高速検索を可能にする。多言語テキストのキーバリュー格納にも対応できる。

**PITR と WAL アーカイブによる RPO 目標の達成**: PostgreSQL の PITR（Point-in-Time Recovery）と WAL（Write-Ahead Log）アーカイブにより、RPO 15 分の目標を達成する。バックアップは pgdump（論理バックアップ）と WAL アーカイブ（物理バックアップ）の組み合わせで運用し、IT 担当 1 名でも実行可能な自動化スクリプトを提供する。

**オープンソースとライセンスコスト**: PostgreSQL はオープンソースであり、ランニングコストにライセンス費用が発生しない。中小製造業オンプレ運用適合性評価軸においてコスト優位性がある。

### SQLite との整合性方針

タブレット端末の SQLite とサーバーの PostgreSQL は異なるデータベースエンジンであり、スキーマと型の完全一致を保つことは不可能である。以下の整合性方針を確定する。

| 方針 | 内容 |
|---|---|
| スキーマ対応 | タブレット SQLite の作業ログテーブルスキーマは PostgreSQL の作業ログテーブルと論理的に対応を保つ。カラム名・型の命名規則を統一する |
| 管理機能テーブルの非同期 | ユーザーマスタ・設備マスタ等の管理機能テーブルはサーバー側のみ保持する。端末には認証に必要な最小限の情報のみキャッシュする |
| 同期方向 | タブレット（SQLite）→サーバー（PostgreSQL）方向は Outbox Pattern で実現する。サーバー→タブレット方向は差分取得（最終同期タイムスタンプ以降の変更のみを取得）で実現する |
| 型不一致への対処 | SQLite の型システムは PostgreSQL より弱い。整合性は TypeORM の型定義で補完し、サーバー受信時に再バリデーションを実施する |

### 代替案と却下理由

| 代替案 | 却下理由 |
|---|---|
| SQL Server（Microsoft） | ライセンスコストが発生する（WSL2 上での無償利用は制限がある）。コスト評価軸において許容できない |
| MySQL / MariaDB | JSONB 型相当の機能が劣る（JSON 型はあるが部分インデックス・演算子の成熟度が低い）。PITR の実装が PostgreSQL より複雑であり、IT 担当 1 名での運用に不適切 |
| MongoDB | 文書型はスキーマレスの柔軟性を持つが、作業ログの厳格な関係完全性（作業者 ID・ステップ ID の外部キー整合性）の保証が弱い。ALCOA+ の Accurate・Complete 原則への対応でリレーショナルモデルに劣る |
| CockroachDB | 分散 DB は単一サーバー・単一工場前提の本アプリに対してオーバーエンジニアリングである。個人開発での習得コスト・運用複雑性が長期保守性評価軸で許容できない |

---

**本節で確定した方針**
- PostgreSQL を採用し、UPDATE/DELETE 禁止ロール設計・JSONB による属性格納・PITR による RPO 15 分目標の実現を選定根拠として確定する。
- タブレット SQLite とのスキーマ論理対応・Outbox 同期・サーバー受信時再バリデーションの整合性方針を確定する。
- SQL Server・MySQL/MariaDB・MongoDB・CockroachDB の 4 案を代替案として評価し、ライセンスコスト・機能差・関係完全性欠如・運用複雑性を根拠に排除することを確定する。

## 6. デプロイ: IIS + Windows Server 2022 + WSL2 + Docker の選定根拠

### 選定根拠

**IIS + Windows Server: 中小製造業 IT 担当の運用スキルセット最大公約数**: 国内中小製造業の IT 担当者が最も習熟しているサーバー環境は Windows Server である。IIS（Internet Information Services）は Windows Server に標準搭載され、追加ライセンスが不要である。本アプリのフロントエンド（管理 Web 静的ファイル）の配信・SSL 証明書管理・リバースプロキシ機能を IIS で担うことで、IT 担当者の既存スキルを活用できる。Linux サーバー・Nginx の設定を前提とした構成は、中小製造業オンプレ運用適合性評価軸で排除する。

**WSL2 + Docker: Linux コンテナと Windows の共存**: Rust バックエンドと PostgreSQL は Linux コンテナとして動作させる。WSL2（Windows Subsystem for Linux 2）は Linux カーネルを Windows Server 上で動作させる仮想化層であり、Linux ネイティブ環境と同等のパフォーマンスを提供する。Docker Compose により全コンポーネント（バックエンド・PostgreSQL・バックアップエージェント）を単一の compose.yml ファイルで管理する。この構成は以下の価値を提供する。

- 環境の再現性（compose.yml で定義された状態を常に再現できる）
- デプロイ手順の単純化（`docker compose up -d` で全コンポーネントが起動する）
- 依存関係の隔離（バックエンドの Rust ランタイム・PostgreSQL のバージョンを OS から独立して管理できる）

**Active-Standby 二重化による可用性**: LAN 内 2 台構成（建屋別配置を推奨）でプライマリ・スタンバイの Active-Standby 構成を取る。RTO 1 時間・RPO 15 分を達成目標とする。スタンバイへの切り替え手順は IT 担当 1 名が実施できるスクリプト化された手順書として提供する。

**GUI セットアップウィザードによる導入障壁の最低化**: Docker Compose の起動コマンドに加え、初回セットアップ（環境変数設定・DB 初期化・SSL 証明書設定・バックアップ先設定）を GUI ウィザードで完結できる設計とする。これは [`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md) で示される「中小 IT 担当者の Linux・コンテナ習熟度が低い」という制約への対応である。

**中堅 SIer 経由の間接販売対応**: バックエンドは REST API と Webhook を整備し、SIer が追加カスタマイズ・他システムとの統合を行えるインターフェースを提供する。これは [`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md) で示される中堅 SIer 経由の間接販売チャネルへの対応である。

### 代替案と却下理由

| 代替案 | 却下理由 |
|---|---|
| Linux サーバー直接（Ubuntu + Nginx 等） | 中小製造業の IT 担当者の Linux スキルギャップが大きく、Nginx 設定・systemd 管理・ディストリビューション選択の複雑性が導入障壁となる。中小製造業オンプレ運用適合性評価軸で排除する |
| SaaS / クラウド | 社内 LAN・オンプレ前提と設計上矛盾する。データ主権（自社サーバーへのデータ保管義務）・BCP（インターネット障害時の業務継続）・顧客の情報セキュリティポリシーとの衝突が発生する。02 章の対象範囲定義と根本的に相容れない |
| Kubernetes | 単一工場・単一サーバー前提に対してオーバーエンジニアリングである。ノード管理・Helm チャート・etcd の管理が IT 担当 1 名の運用能力を超える。個人開発・長期保守性評価軸で排除する |
| VMware ESXi | 追加ライセンスコスト（vSphere 等）・仮想化層の管理スキル要求が中小製造業オンプレ運用適合性評価軸で許容できない。WSL2 + Docker で達成できる目的と重複する |

詳細は [`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md)・[`90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md`](../../90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md)・[`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) を参照すること。

---

**本節で確定した方針**
- IIS + Windows Server 2022 を中小製造業 IT 担当の既存スキルセット最大公約数として採用し、WSL2 + Docker Compose で Linux コンテナを管理する構成を確定する。
- Active-Standby 二重化（RTO 1 時間・RPO 15 分）と GUI セットアップウィザードを ver1.0.0 の必須要件として確定する。
- Linux 直接・SaaS・Kubernetes・VMware の 4 案を代替案として評価し、スキルギャップ・スコープ矛盾・運用過重・コストを根拠に排除することを確定する。

## 7. 外部システム連携プロトコルの選定根拠

### 位置づけ

本節は 6 節で確定したデプロイ構成の中で言及した「REST API と Webhook を整備し、SIer が追加カスタマイズ・他システムとの統合を行えるインターフェースを提供する」という方針を、プロトコル選定の根拠として詳述する。本アプリ（子機）と基幹システム（親機）との連携において、どのプロトコルを採用・非採用とするかを確定する。

### 採用・非採用プロトコルの比較

| プロトコル | ver1.0.0 | 選定理由 / 非選定理由 |
|---|---|---|
| REST API（JSON over HTTPS） | ○ | 業界標準・SIer との相互運用性・OpenAPI 3.1 でのスキーマ駆動開発が可能 |
| Webhook | ○ | 非同期イベント通知・HMAC-SHA256 署名検証・Idempotency-Key 対応 |
| OAuth 2.1 Client Credentials | ○（認証方式として） | サービス間機械通信に適合。ユーザー介入不要のバッチ同期に最適 |
| mTLS | ○（代替として） | 社内 PKI がある環境では OAuth 不要で相互認証が成立。シンプル |
| AMQP / RabbitMQ | ×（設計余地） | メッセージキューによる疎結合は有益だが個人開発スコープを超える |
| MQTT | × 永続非提供 | IoT/PLC センサ向け。OT 直結は永続非提供（構想 11章 §4 参照） |
| OPC-UA | × 永続非提供 | 同上 |

### REST API（JSON over HTTPS）の選定根拠

**業界標準・SIer との相互運用性**: 国内中堅 SIer が最もスキルセットを持つ統合プロトコルは REST API である。JSON over HTTPS は言語・フレームワークを問わずクライアントを実装でき、親機（基幹システム）側の開発環境を選ばない。SIer が本アプリと ERP・WMS・MES を連携させる際の摩擦を最小化する。

**OpenAPI 3.1 によるスキーマ駆動開発**: axum のルート定義から OpenAPI 3.1 スキーマを自動生成し、SIer への API ドキュメント提供・クライアントコードの自動生成・バリデーション仕様の共有を行う。手書き API ドキュメントによる仕様とコードの乖離を構造的に防止する。

**axum + Tower ミドルウェアとの整合**: axum の Tower ミドルウェアにより、認証（OAuth 2.1 / mTLS）・レート制御・リクエストロギングを型安全に統一実装できる。4 節で確定した Rust + axum の選定根拠である「Tower ミドルウェアの合成による型安全なルーティング」はこの外部連携プロトコル層においても有効であり、実装の一貫性を保つ。

### Webhook の選定根拠

**非同期イベント通知による疎結合**: 作業完了・工程承認・異常検知等のイベントを親機（基幹システム）に能動的にプッシュする。親機が本アプリをポーリングする構成と比較して、親機側の負荷を削減しリアルタイム性を高める。

**HMAC-SHA256 署名検証による改竄検知**: Webhook ペイロードに HMAC-SHA256 署名を付与し、受信側で検証する。05 章の改竄不可性原則を外部連携層においても維持する。

**Idempotency-Key 対応**: Webhook の再送（ネットワーク障害・タイムアウト時）において、受信側が同一イベントを重複処理しないよう Idempotency-Key を付与する。05 章の Idempotent API 原則と整合する。

### 認証方式の選定根拠

**OAuth 2.1 Client Credentials（採用）**: 親機（基幹システム）と子機（本アプリ）のサービス間通信はユーザーの介入なしに動作する機械通信（バッチ同期・自動連携）を前提とする。Client Credentials フローはユーザー認証を必要としないサービス間通信に最適であり、アクセストークンの有効期限管理・スコープ制御・トークン失効が標準的に実現できる。

**mTLS（代替として採用）**: 社内 PKI（Public Key Infrastructure）が既に整備されている環境では、クライアント証明書による相互認証（mTLS）が OAuth 2.1 よりシンプルな構成となる。認証サーバーの追加構築が不要であり、IT 担当 1 名での運用適合性が高い。SIer の環境・顧客の情報セキュリティポリシーに応じて選択できる設計とする。

### 非採用プロトコルの排除根拠

**AMQP / RabbitMQ（設計余地）**: メッセージキューによる疎結合アーキテクチャは、複数サービス間の非同期処理・バックプレッシャー制御において有益である。しかし、RabbitMQ 等のメッセージブローカーの導入・運用は個人開発スコープを超える。Webhook による非同期イベント通知で代替できる範囲において、ver1.0.0 では採用しない。設計余地として将来の拡張性は保持する。

**MQTT（永続非提供）**: IoT センサ・PLC 等の OT（Operational Technology）機器との直接通信プロトコルである。本アプリは OT 直結を設計方針として永続非提供とする。構想 11 章 §4 の OT-IT 境界方針と整合する。

**OPC-UA（永続非提供）**: MQTT と同様に OT 直結プロトコルである。製造機械・PLC との直接通信を前提とし、本アプリの設計スコープ外とする。永続非提供とし、OT との連携は将来の独立サービスとして別途設計する。

### APIバージョニング方針

URL-based バージョニング（`/api/v1/`）を採用する。後方互換を破壊する変更は `/api/v2/` へ移行し、旧バージョンエンドポイントは移行期間として最低 6 ヶ月間維持する。SIer が安定したインターフェースを前提にシステム開発を行える環境を保証する。

### レート制御方針

トークンバケット方式を採用する。外部連携 API（親機・SIer・サードパーティ）は内部 API（子機内部通信）より厳格なレートリミットを設ける。これは外部連携の意図しない高頻度アクセスによって内部処理が圧迫されることを防ぎ、オフライン同期・作業記録受信の安定性を最優先とする設計と整合する。Tower ミドルウェアでレート制御を統一実装することで、エンドポイントごとの実装漏れを防止する。

---

**本節で確定した方針**
- REST API（JSON over HTTPS）と Webhook を外部システム連携の基本プロトコルとして採用し、OpenAPI 3.1 スキーマ駆動開発・HMAC-SHA256 署名検証・Idempotency-Key 対応を ver1.0.0 の必須仕様として確定する。
- 認証方式は OAuth 2.1 Client Credentials（標準）と mTLS（PKI 環境の代替）の二択とし、SIer・顧客環境に応じて選択できる設計を確定する。
- AMQP / RabbitMQ は個人開発スコープ超過を理由に ver1.0.0 では非採用とし、設計余地として将来拡張性を保持することを確定する。
- MQTT・OPC-UA は OT 直結の永続非提供方針から排除を確定する。
- APIバージョニングは URL-based（`/api/v1/`）・レート制御はトークンバケット方式を確定する。
- axum の Tower ミドルウェアにより認証・レート制御・ロギングを統一実装する方針を 4 節の Rust + axum 選定根拠と整合させて確定する。
- 親機（基幹システム）と子機（本アプリ）の表記を本節以降で統一する。

## 8. 個人開発による独断リスクと緩和策

### 独断リスクの認識

技術選定は本章の評価軸と代替案比較に基づいて実施するが、個人開発では代替案の評価が単一の視点に偏る本質的なリスクが存在する。チーム開発であれば、技術選定の議論・コードレビュー・設計レビューを通じて独断が是正される機会がある。個人開発ではこの是正機会が存在しない。

この事実を認識し、独断リスクを緩和する構造的な手段を確定する。

### 緩和策

**本章の代替案比較表の公開**: 本章はシステム化計画書の一部として文書化され、IT 担当・品質担当・経営層が閲覧できる。選定根拠と排除根拠が透明であることで、外部レビューの機会を作る。

**ADR（アーキテクチャ決定記録）による追跡**: 本章の内容に加え、将来の設計判断は ADR として個別ファイルに記録する。ADR は「決定の背景・決定内容・代替案・トレードオフ・結果」を構造化して保持し、将来の再評価の基点となる。技術スタックの変更は ADR の更新として記録する。

**10 章リリース判定基準へのドキュメント完成度の組み込み**: ver1.0.0 のリリース判定基準（`10_開発スケジュールと意思決定プロセス.md` に記載）に「本章を含む設計文書の完成度」を含める。設計根拠が文書化されていない機能はリリース対象外とする。

### 技術スタック変更の条件

以下の条件が発生した場合、本章の技術スタック選定を再評価する。

| 条件 | 対応 |
|---|---|
| 将来の SI 調達・法人化による開発体制変更 | 技術スタック再評価を実施し、ADR に記録する。`11_前提・制約・リスク.md` に変更条件を記載する |
| 採用技術の重大な脆弱性・非推奨宣言 | 該当技術を緊急評価し、代替案の中から最も移行コストが低い選択肢を選定する |
| 評価軸の前提条件の変更（例: 単一工場→マルチサイト） | 対象範囲の変更として 02 章を更新し、それに伴う技術スタック再評価を実施する |

変更の意思決定プロセスは `10_開発スケジュールと意思決定プロセス.md` で確定する。

---

**本節で確定した方針**
- 個人開発における独断リスクを明示し、代替案比較表の公開・ADR による追跡・リリース判定基準への組み込みを緩和策として確定する。
- 技術スタック変更の条件（SI 調達・重大脆弱性・対象範囲変更）を明示し、変更判断の恣意性を排除することを確定する。
- 技術選定変更は必ず ADR に記録し、`11_前提・制約・リスク.md` のリスク台帳と連動させることを確定する。
