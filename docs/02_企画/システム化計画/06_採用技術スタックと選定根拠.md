# 採用技術スタックと選定根拠

**主読者**: IT 部門・SIer パートナー  
**想定所要時間**: 20 分

---

## 6.1 技術スタック全体像

| 層 | 技術 | バージョン方針 |
|---|---|---|
| タブレットアプリ | React Native (Expo) / TypeScript | LTS 版追従 |
| タブレット OS | Android (APK) / Windows (react-native-windows) | Android 12+, Windows 10/11 |
| ローカル DB | SQLite (expo-sqlite) | Expo 標準版 |
| 管理 Web | React + Vite / TypeScript | LTS 版追従 |

| LAN 内冗長 | PostgreSQL WAL ストリーミングレプリカ | 同期/非同期切替可 |

---

## 6.2 端末層: React Native (Expo)

**選定根拠**:
- Android と Windows を**単一コードベース**でターゲットできる（react-native-windows）
- Expo エコシステムにより expo-sqlite（ローカルキュー）・expo-speech（TTS）・expo-camera（QR スキャン）が標準化
- JavaScript エコシステムの豊富な UI ライブラリ（アクセシビリティ対応）
- **社内 OTA 配信**（社内サーバからの APK/MSIX プッシュ配布）または手動配布によりビルドなしで手順表示 UI の修正を配信可能。社内 LAN 完結要件のため外部 Expo OTA サーバは使用しない

**代替案を不採用とした理由**:
- Flutter: Dart エコシステムが React より学習コスト高。react-native-windows 相当の Windows 対応が未成熟
- Kotlin/Swift ネイティブ: Android/Windows の二重開発が必要
- Progressive Web App (PWA): Windows タブレットでは問題ないが Android のオフライン SQLite 操作に制限あり

現場 HCI の観点から、製造現場の Glanceable UI（1〜2 秒で重要情報把握）と手袋対応タップターゲット（72dp 以上）を実装しやすいコンポーネント設計を採用する（[`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md) 参照）。

---

## 6.3 管理 Web 層: React + Vite

**選定根拠**:
- SPA として Rust API サーバから静的ファイル配信（IIS 経由）
- React の豊富な UI ライブラリ（データグリッド・フォームバリデーション）
- Vite の高速ビルドにより開発サイクルを短縮

管理 Web は品質保証担当・工場長が PC ブラウザで利用する。タブレット UI と異なるユーザビリティ要件（データテーブル・フィルタリング・グラフ表示）を React のコンポーネント設計で分離する。

---

## 6.4 API 層: Rust + Axum + tokio + sqlx

**選定根拠**:

| 評価軸 | Rust (Axum) | Go | Node.js (Express) |
|---|---|---|---|
| メモリ安全性 | ◎ コンパイル時保証 | ○ GC あり | △ 型なし |
| パフォーマンス | ◎ ネイティブ性能 | ○ 十分 | △ V8 依存 |
| 長期保守性 | ◎ 型システムが強力 | ○ | △ |
| 社内エコシステム | ○ Edition 2024 | ○ | △ |
| sqlx 非同期 DB | ◎ コンパイル時クエリ検証 | △ database/sql | △ |

Rust Edition 2024 + sqlx はコンパイル時に SQL クエリの型整合性を検証できるため、DB スキーマとアプリコードの乖離をコンパイルエラーとして検出できる。これは Append-only Event Sourcing の実装においてデータ型の整合性が特に重要な場面で威力を発揮する。

tokio の非同期ランタイムにより、多数の同時接続（複数ライン × 複数タブレット）を効率的に処理する。

---

## 6.5 DB 層: PostgreSQL

**選定根拠**:
- Append-only Event Sourcing の基盤として、ROW LEVEL SECURITY と CHECK CONSTRAINT で追記のみを DB レベルで強制
- JSON/JSONB 対応により多言語テキスト（`{"ja": "...", "vi": "..."}` 形式）を型安全に保存
- WAL（Write-Ahead Log）ストリーミングレプリケーションにより LAN 内 Active-Standby を実現（RTO 1h / RPO 15min 目標）
- sqlx の `prepare!` マクロによりコンパイル時クエリ検証

**TimescaleDB** の採用はこの段階では対象外。センサ時系列データ収集が将来スコープに入った場合に検討する。

---

## 6.6 インフラ層: IIS + WSL2 + Docker Compose

![デプロイトポロジ](./img/deployment_topology.svg)

**選定根拠**:
- Windows Server 2022 は国内中小製造業の標準サーバ OS
- IIS が管理 Web の静的ファイル配信とリバースプロキシを担当（既存インフラ活用）
- WSL2 上の Docker Compose により Linux ネイティブな Rust/PostgreSQL コンテナを運用
- Docker Compose 単一ファイルによる再現性の高いデプロイ（`docker compose up -d` 一発起動）
- Docker Compose ファイルによる構成の IaC 化とバージョン管理

## 6.8 ベンダーロックイン回避設計

| 対策 | 実装方針 |
|---|---|
| 標準形式エクスポート | CSV / JSON / XES 互換エクスポートを常時提供 |
| 開放的 REST API | OpenAPI 仕様書を公開し、外部ツールとの連携を容易化 |
| Docker Compose による移植性 | コンテナ化により別ホストへの移行が可能 |
| ソースコード管理 | Git リポジトリでソース一式を管理し、依存 SaaS への関係を排除 |

## 6.9 OSS ライセンス適合確認

主要依存ライブラリのライセンス種別を以下に示す。商用展開時に GPL/LGPL による Copyleft 問題が発生しないことを確認済み。

| ライブラリ | ライセンス | 備考 |
|---|---|---|
| React Native | MIT | タブレットアプリ |
| Expo SDK | MIT | React Native 拡張基盤 |
| React | MIT | 管理 Web |
| Vite | MIT | ビルドツール |
| Axum (Rust) | MIT | API サーバフレームワーク |
| tokio (Rust) | MIT | 非同期ランタイム |
| sqlx (Rust) | MIT or Apache-2.0 | DB アクセスライブラリ |
| PostgreSQL | PostgreSQL License（MIT 類似） | DB エンジン |
| Docker Engine | Apache-2.0 | コンテナ基盤 |

> **ライセンス管理方針**: `cargo audit`（Rust）および `npm audit`（Node.js）を CI で定期実行し、脆弱性・ライセンス逸脱を検知する。四半期ごとに主要依存の CVE レビューを実施する。

---

> **本節で確定した方針**  
> 1. タブレット: React Native (Expo) / Android + Windows、管理 Web: React + Vite、API: Rust (Edition 2024) + Axum + tokio + sqlx、DB: PostgreSQL を採用する。  
> 2. インフラは IIS + WSL2 + Docker Compose（Windows Server 2022）。社内 LAN 完結・オンプレ運用を徹底する。  
> 3. CSV/JSON/XES 互換エクスポートにより、ベンダーロックインを回避し将来のシステム移行を可能にする。
