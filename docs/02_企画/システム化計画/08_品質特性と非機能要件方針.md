# 08 品質特性と非機能要件方針

本章の責務は、ISO/IEC 25010 を基盤として本システムの非機能要件の方針を確定することである。本章で確定した方針は、要件定義フェーズにおける具体的数値目標の上位制約として機能する。倫理品質を第 9 の特性として正式に追加し、「監視と支援の構造的分離」を品質特性として定着させる。

---

## 1. 品質特性フレームワーク

### ISO/IEC 25010 の 8 特性と本プロジェクトへの適用

ISO/IEC 25010:2023 は製品品質モデルとして以下の 8 特性を規定する。本プロジェクトはこれを基盤として採用する。

| 特性 | 本プロジェクトにおける主要な関心事 |
|---|---|
| 機能適合性 | Navigation（EWI/状況認識支援）と Traceability（証拠生成）の二大価値が仕様通りに実現されているか |
| 性能効率性 | Glanceable 1 秒以内の状態認識、Offline-First 応答、LAN 内クエリ応答時間 |
| 互換性 | Android / iOS / Windows タブレットの 3 OS・PostgreSQL と SQLite の双方でのデータ整合性 |
| ユーザビリティ | 手袋着用・夜間・高齢・多言語・騒音環境への対応 |
| 信頼性 | ALCOA+ Enduring（5年以上のオンライン照会）・Offline-First 継続性・RTO/RPO 目標 |
| セキュリティ | ALCOA+ Attributable・改竄不可性・RBAC による目的分離 |
| 保守性 | 個人開発下での長期保守性・ドキュメント駆動・テスト戦略 |
| 移植性 | Docker Compose による環境移行・Windows Server 2022 / Linux サーバーの双方対応 |

### 倫理品質を第 9 の特性として追加する根拠

本プロジェクトは 01 章の倫理境界宣言において「監視と支援の構造的分離」を宣言した。この宣言を設計文書に閉じ込めるのではなく、システムの品質特性として永続的に機能させるため、倫理品質を ISO/IEC 25010 の追加特性として本書で正式化する。

根拠は 2 点である。第 1 に、24 章（[`90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md`](../../90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md)）が示す GDPR 第 22 条の自動意思決定への異議申立権および日本改正個人情報保護法 (2022) は、製造業データ収集において目的限定性・データ最小化を技術的に担保する義務を示している。第 2 に、個人開発という体制において「方針文書のみ」では倫理的逸脱を防ぐ抑止力が不十分であり、RBAC とデータモデルに倫理的制約を埋め込む設計が必要である。

### 具体的数値目標の委任

本章が確定するのは上位目標（方針）である。具体的な数値目標（スループット・可用率・MTBF 等）は要件定義フェーズに委ねる。本章の方針に反する数値目標を要件定義フェーズで設定することは認めない。

KPI・SLI・SLO の枠組み（何を計測するか・PoC からどう段階的に達成するか）は、計画 14 章（品質指標と SLI / SLO / KPI 枠組み）で確定する。本章では ISO/IEC 25010 の品質特性定義を上位として維持し、計測可能な目標値の設定は要件定義フェーズとする。

---

**本節で確定した方針**
- ISO/IEC 25010 の 8 特性を基盤として採用し、倫理品質を第 9 の特性として追加することを確定する。
- 各特性の具体的数値目標は要件定義フェーズに委ねるが、本章の方針に反する数値の設定は認めない。
- 倫理品質の正式化は RBAC・データモデル・監査ログに技術的に実装することを前提とする。

---

## 2. 性能効率性 — Glanceable UI を上位目標に

### 上位目標の定義

性能効率性の上位目標を「Glanceable 1 秒以内の状態認識」と定義する。これは Endsley の状況認識 (SA) モデルにおけるレベル 1（Perception: 現在状況の知覚）を 1 秒以内に達成することを意味する。製造現場の作業員は画面を注視し続ける時間的余裕を持たない。現在ステップ・異常アラート・次ステップの 3 要素を 1 秒以内に視認できる UI 設計が性能要件の最上位に位置する。

業界分析 [`90_業界分析/12_認知工学と状況認識.md`](../../90_業界分析/12_認知工学と状況認識.md) が示す Endsley SA 3 層（Perception/Comprehension/Projection）は、タブレット UI における情報の優先順位を規定する基準として本プロジェクトに適用する。

### 応答時間の方針

本プロジェクトにおける応答時間の方針を以下の通り確定する。

| 操作 | 応答時間上限（方針） | 前提条件 |
|---|---|---|
| 現在ステップ表示 | 200ms 以内 | ローカル SQLite クエリ |
| ページ遷移 | 500ms 以内 | ローカル SQLite クエリ |
| バーコード / QR スキャン | 1 秒以内 | オフライン ML Kit |
| 管理 Web クエリ | 3 秒以内 | LAN 内 PostgreSQL |
| 同期バックグラウンド処理 | ユーザー操作をブロックしない | バックグラウンドワーカー |

### Offline-First 性能保証

サーバーが不到達の状態においても、タブレット APP は SQLite ローカルクエリのみで全主要操作を完結する。サーバーへの通信が完了するまで操作がブロックされる実装は性能要件違反と見なす。

---

**本節で確定した方針**
- 性能効率性の上位目標を「Glanceable 1 秒以内の状態認識（Endsley SA レベル 1）」に確定する。
- 現在ステップ表示 200ms 以内・ページ遷移 500ms 以内・バーコードスキャン 1 秒以内を方針値とする。
- オフライン時はローカル SQLite クエリのみで全主要操作が完結することを要件とする。

---

## 3. 信頼性 — ALCOA+ Enduring と Contemporaneous を最優先

### 可用性目標

本システムのインフラ構成は 07 章で確定した IIS + Windows Server 2022 + WSL2 + Docker の単一工場オンプレ構成である。この構成における可用性目標を以下の通り定める。

- **RTO (Recovery Time Objective)**: 1 時間以内
- **RPO (Recovery Point Objective)**: 15 分以内

Active-Standby 構成は LAN 内で実現する。クラウド依存のゼロダウンタイム要件は前提条件（P02: 社内 LAN のみ利用）の範囲外である。

### Offline-First による業務継続性

サーバーへの到達不能状態においても、タブレット APP はローカルキューへの記録を継続する。ローカルキュー内のトレサビデータはサーバー復旧後に自動同期する。この設計は可用性目標ではなく「業務継続性の設計原則」として 05 章に確定済みである。本節はその設計が信頼性特性に直結することを明示する。

### データ耐久性の方針

ALCOA+ の Enduring（耐久性）と Contemporaneous（同時性）を最優先として以下の方針を確定する。

| 耐久性手段 | 方針 |
|---|---|
| PostgreSQL WAL | PITR (Point-in-Time Recovery) を有効化し、RPO 15 分を達成する |
| NAS バックアップ | 日次の pg_dump を NAS に格納する |
| オフサイトバックアップ | 週次で建屋外（別棟または可搬媒体）に複製する |
| 保管期間 | 業種設定（SOP 内で参照可能）から読み取る。最低 5 年間はオンラインで照会可能とする |

業界分析 [`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md) の BIA（業務影響分析）・RTO/RPO/MTPD フレームワークを本プロジェクトに適用する。ISO 22301:2019 に対応する運用方針は 09 章に委ねる。

---

**本節で確定した方針**
- RTO 1 時間・RPO 15 分を可用性目標として確定し、PostgreSQL WAL + PITR で達成する。
- ALCOA+ Enduring の方針として、最低 5 年間はトレサビデータをオンラインで照会可能とする。
- サーバー不到達状態でもタブレット APP の主要操作が継続可能であることを信頼性特性に含める。

---

## 4. セキュリティ — ALCOA+ Attributable と改竄不可性

### 認証・認可の方針

| セキュリティ要素 | 方針 |
|---|---|
| 認証方式 | JWT Bearer Token（RS256 署名） |
| トークン有効期限 | 8 時間（シフト 1 本に相当） |
| 認可モデル | RBAC（6 ロール） |
| 通信保護 | TLS 1.3（社内 LAN・自己署名証明書） |

### RBAC の 6 ロール定義方針

本プロジェクトの RBAC は以下の 6 ロールを定義する。各ロールの具体的な権限マトリクスは要件定義フェーズで確定する。

| ロール | 主要な権限範囲 |
|---|---|
| 作業員 | 自身の担当工程の実行・記録・Kaizen Teian 起票 |
| 現場監督 | ユーザー管理・スキル認定・現場集計参照 |
| 品質担当 | SOP 管理・CAPA・トレサビ照会・不適合報告者 ID 参照 |
| 保全担当 | 設備保全記録・校正記録・設備異常アラート |
| IT 担当 | システム設定・バックアップ確認・作業ログ直接参照は不可 |
| 管理者 | 全設定・ロール付与（経営層を兼務しない）|

個人情報保護の観点から「品質担当のみトレサビ照会可能」「IT 担当は作業ログの直接参照不可」の権限分離を RBAC で技術的に強制する。

### 改竄不可性の方針

ALCOA+ の Attributable（帰属明確性）と改竄不可性を実現するため以下を確定する。

- **Append-only ログ**: 作業記録・品質記録はすべて Append-only で書き込む。既存レコードの UPDATE による上書きは禁止する。
- **SHA-256 ハッシュチェーン**: 各レコードに前レコードのハッシュを格納し、改竄検知を可能にする（06 章）。
- **電子署名**: SOP 承認・CAPA クローズ・校正証明書にはユーザー ID + タイムスタンプ + サーバー署名を付与する。

### OT-IT 境界の方針

05 章のアーキテクチャ原則に基づき、本アプリから OT（Operational Technology）ネットワークへの通信を一切行わない。IoT センサデータ・PLC データの直接取得は本システムのスコープ外である（03 章非スコープ）。

---

**本節で確定した方針**
- JWT RS256 + RBAC 6 ロールを認証・認可の基盤として確定し、権限分離を技術的に強制する。
- Append-only ログ + SHA-256 ハッシュチェーンにより ALCOA+ Attributable と改竄不可性を実現する。
- 本アプリから OT ネットワークへの通信は禁止とし、OT-IT 境界を設計レベルで確保する。

---

## 5. ユーザビリティとアクセシビリティ — 多様な現場への対応

### 要件方針の一覧

製造現場の多様な環境・作業者特性に対応するユーザビリティ要件の方針を以下の通り確定する。

| 対象 | 要件方針 |
|---|---|
| タッチ操作（手袋着用） | 最小タッチターゲット 72dp 以上。スライダー・小ボタンは禁止。手袋を想定した押下面積を確保する |
| 視認性 | コントラスト比 4.5:1 以上（WCAG 2.1 AA）。屋外環境では 800nits 対応デバイスを推奨する |
| 色のユニバーサルデザイン | 色だけでなく形状・記号で冗長符号化。カラー UD 対応を必須とする |
| 夜間・疲労時 | 20:00〜6:00 は自動ダークモード。フォント最小 14pt。振動（バイブ）通知を必須とする |
| 高齢・不慣れな作業者 | シニアモード（フォント拡大・選択肢化・自動補完・タッチターゲット拡大）を作業者プロファイルで設定可能にする |
| 多言語 | i18next + JSONB 多言語テキスト。やさしい日本語・ふりがな・ピクトグラム対応（34 章） |
| 騒音環境 | 音声に依存しない。視覚＋触覚（バイブ）の冗長フィードバックを必須とする |
| 認知負荷 | 1 画面 = 1 ステップの原則。選択肢は 2〜4 択（Hick の法則）。通知は階層化（情報 / 注意 / 警告 / 緊急）する |

### 業界分析の根拠

本節の要件方針は以下の業界分析に基づく。

- [`90_業界分析/08_人間工学と作業負荷.md`](../../90_業界分析/08_人間工学と作業負荷.md): IEA 3 分類・ISO 9241 シリーズ・JIS Z 9110・ISO 7243 の適用
- [`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md): 産業用手袋 72dp タッチターゲット・Glanceable UI・WCAG 2.1 AA・IP54 / MIL-STD-810H 推奨デバイス・NASA-TLX 認知負荷評価
- [`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md): WCAG 2.1（JIS X 8341-3）・やさしい日本語・ピクトグラム・ふりがな
- [`90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md`](../../90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md): 夜間自動ダークモード・警告音 1000〜3000Hz・シニアモードプロファイル

### デバイス推奨基準

本アプリは IP54 以上・MIL-STD-810H 準拠のタブレットを推奨デバイス条件として設定する。屋外・粉塵・水飛沫環境での継続使用を前提とする。

---

**本節で確定した方針**
- 最小タッチターゲット 72dp・WCAG 2.1 AA コントラスト比 4.5:1 以上・1 画面 = 1 ステップを設計原則として確定する。
- 夜間自動ダークモード・シニアモード・多言語対応を初期実装から必須機能として含める。
- IP54 以上・MIL-STD-810H 準拠タブレットを推奨デバイス基準として本書に記録する。

## 6. 保守性 — 個人開発下での長期保守可能性

### ドキュメント駆動設計

本プロジェクトは個人開発であり、Bus Factor = 1 という根本的なリスクを抱える。この制約下で長期保守可能性を確保する唯一の手段はドキュメント駆動設計である。

| 対象 | ドキュメント要件 |
|---|---|
| REST API 全エンドポイント | OpenAPI 3.1 仕様書（要件定義フェーズで完成） |
| データモデル全テーブル | ER 図（詳細設計書として作成） |
| アーキテクチャ上の不可逆決定 | ADR（アーキテクチャ決定記録）—詳細は 10 章 |
| 実装コード全般 | README・インラインコメント・型定義 |

### テスト戦略の方針

| 対象 | テスト方針 |
|---|---|
| Rust バックエンド | ユニットテスト + 統合テストを必須とする（cargo test）|
| React Native | E2E テスト（Detox）で 04 章の全ユースケースを網羅する |
| React 管理 Web | コンポーネントテスト（React Testing Library）を必須とする |
| DB スキーマ同期 | SQLite と PostgreSQL のスキーマ整合性を E2E テストで毎回検証する |

### 依存関係管理の方針

- `Cargo.lock` および `package-lock.json` で全依存を固定する。
- `cargo audit` による脆弱性スキャンを定期実行する（リリース前必須）。
- メジャーバージョンアップは ADR を作成してから適用する。

### Bus Factor 対策

ドキュメント・テスト・型定義の 3 点セットにより、個人開発における知識喪失リスクを低減する。1 ヶ月間の作業中断後にドキュメントを読めば開発を再開できることを M6 リリース判定条件に含める（10 章）。

---

**本節で確定した方針**
- 全 API に OpenAPI 3.1・全データモデルに ER 図・不可逆決定に ADR を必須として確定する。
- Rust は cargo test・React Native は Detox E2E を必須テストとして確定する。
- 依存は Cargo.lock / package-lock.json で固定し、cargo audit による脆弱性スキャンを実施する。

## 7. 倫理品質（第 9 特性）— 監視と支援の構造的分離

### 倫理品質の正式化

倫理品質を ISO/IEC 25010 の追加特性として本書で正式化する。この特性は 01 章の倫理境界宣言を品質特性として実装するものであり、設計・実装・監査のすべての段階で評価対象となる。

業界分析 [`90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md`](../../90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md) が示す目的分離アーキテクチャ（品質保証 vs 人事評価）は、本プロジェクトの倫理品質設計の中核概念である。

### 倫理サブ特性の定義

| 倫理サブ特性 | 定義 | 実装方針 |
|---|---|---|
| 目的限定性 | 収集データを宣言した目的のみに利用する | 品質保証・工程改善・教育設計の 3 目的のみ。人事評価への転用を RBAC で技術的に禁止する |
| データ最小化 | 業務目的に必要最小限のデータのみ収集する | 位置情報・生体情報・個人別時間ランキングを初期実装から除外する |
| 透明性 | データ収集・利用方針を平易な日本語で開示する | 多言語プライバシー通知をアプリ初回起動時に表示する |
| 説明可能性 | システムの判定（異常アラート等）の根拠を開示する | アラート表示時に「根拠となる測定値・閾値・SOP 章番号」を必ず表示する |
| 自律性の保護 | 作業者の判断を置き換えない | 最終判断は人間が行う。自動判断は対象外（03 章） |
| 心理的安全 | 報告・提案を評価・処罰と切り離す | 不適合・ヒヤリハット報告と人事評価データを RBAC・ポリシーで完全分離する |
| データポータビリティ | 導入先組織がいつでも本アプリのデータを完全にエクスポートでき、他システムへの移行ができる | 標準フォーマット（CSV / JSON）によるデータ全件エクスポート機能を初期実装から提供する。構想 13 章（データオーナーシップと永続性の宣言）との整合を設計レベルで確保する |

### 倫理品質の自己監査の位置づけ

M6 リリース判定基準（10 章）に「倫理品質サブ特性の自己監査チェックリスト全項目パス」を含める。設計意図が実装に反映されているかを、RBAC 設定・データモデル・監査ログの 3 点で確認する。

---

**本節で確定した方針**
- 目的限定性・データ最小化・透明性・説明可能性・自律性の保護・心理的安全・データポータビリティの 7 サブ特性で倫理品質を定義する。
- 人事評価への転用禁止・個人別ランキング除外を RBAC とデータモデルで技術的に強制する。
- 倫理品質の自己監査チェックリスト全項目パスを M6 リリース判定条件に含める。

---
