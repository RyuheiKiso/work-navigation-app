# 15 セキュリティ深堀り

本章の責務は、08 章（品質特性と非機能要件方針）で確定した JWT(RS256) + RBAC 6 ロールの方針を、12 章（子機モード）の採用によって拡大した外部接点に対応する形で深掘りし、ゼロトラスト原則・PII 分類・鍵管理・暗号化・監査ログ保全・インシデント対応の各領域を補完的に確定することである。

---

## 0. 本章の位置づけ

### 08 章セキュリティ特性の拡張

08 章 4 節は JWT(RS256) + RBAC 6 ロールを認証・認可の基盤として確定し、Append-only ログ + SHA-256 ハッシュチェーンによる改竄不可性を方針として定めた。12 章（子機モード）の採用により、子機（本アプリ）は親機（基幹システム）との間でマスタ同期・実績送信という双方向の API 通信を行う外部接点を新たに持つ。この外部接点は認証情報の交換・機密データの転送という 2 つの新たな攻撃面を生むため、本章で補完的に確定する。

### 関連章との整合

| 関連章 | 整合させる事項 |
|---|---|
| 構想 04 章（支援/監視分離） | 倫理品質（第 9 特性）の実装として、認証・認可・監査ログが「監視」目的に転用されないよう RBAC で制約する |
| 構想 11 章（C05 社内 LAN 完結） | 社内 LAN 完結環境においてもゼロトラスト原則を適用し、内部脅威に対応する |
| 計画 05 章（Append-only Event Sourcing） | 監査ログのハッシュチェーンを改竄不可性の実装基盤として参照する |
| 計画 09 章（CAPA ループ） | インシデント発生後の再発防止を CAPA プロセスと連動させる |

### ALCOA+ との接続

ALCOA+ の Attributable 原則（誰が記録したか）は PII 分類（H2-2）・鍵管理（H2-3）・監査ログ保全（H2-5）によって担保する。Enduring 原則（記録が永続的に残る）は暗号化方針（H2-4）・ハッシュチェーンによる改竄不可性（H2-5）で実装する。

---

**本節で確定した方針**
- 本章は 08 章セキュリティ特性の補完であり、12 章子機モード採用による外部接点拡大への対応として位置づける。
- ALCOA+ の Attributable・Enduring と倫理品質（第 9 特性）の双方に本章の全方針が接続することを確定する。
- 構想 04 章・構想 11 章・計画 05 章・計画 09 章との整合を本章の設計制約として課す。

## 1. ゼロトラスト原則の適用範囲

### ゼロトラストの定義

ゼロトラストとは「社内 LAN だから信頼できる」という前提を排除し、すべての通信を認証・認可する設計原則である。構想 11 章（C05 社内 LAN 完結）は外部インターネット接続を排除するが、これは「社内ネットワーク内は安全」を意味しない。製造現場では不正アクセス・誤操作・マルウェア感染タブレットなど内部脅威が存在する。社内 LAN 完結という制約の下でゼロトラストを適用することで、内部脅威に対応する。

### 適用範囲の確定

| コンポーネント | ゼロトラスト適用方針 |
|---|---|
| タブレット端末 → API サーバー | JWT(RS256) 必須。すべてのリクエストで署名検証を実施する |
| 管理 Web → API サーバー | 同上 |
| 親機 → 子機 API（マスタ同期） | OAuth 2.1 Client Credentials または mTLS（親機の実装能力に依存してプラガブルに選択） |
| 子機 → 親機 API（実績送信） | 親機指定の認証方式を採用（プラガブル設計。詳細は 12 章） |
| API サーバー → PostgreSQL | 専用サービスアカウント・最小権限の原則を適用。アプリ用ロールは DDL 権限を持たない |
| Docker コンテナ間通信 | Docker 内部ネットワークで分離。コンテナ外部へのポート公開は API サーバーのみ |

認証・認可の検証失敗時はデフォルトでアクセス拒否（403 Forbidden）とし、すべての失敗を監査ログに記録する。「失敗したら許可」の実装はゼロトラスト原則に反するため禁止する。

---

**本節で確定した方針**
- 社内 LAN 完結環境においてもゼロトラスト原則を適用し、すべての通信を認証・認可することを確定する。
- 親機との連携認証は OAuth 2.1 Client Credentials または mTLS をプラガブルに選択できる設計とする。
- 認証・認可の失敗時はデフォルトでアクセス拒否とし、すべての失敗を監査ログに記録することを確定する。

## 2. PII 分類と最小化

### PII 分類テーブル

本アプリが取り扱う個人を特定できる情報（PII: Personally Identifiable Information）を以下の通り分類し、取扱方針を確定する。

| データ項目 | PII 分類 | 取扱方針 |
|---|---|---|
| 作業員 ID（従業員番号） | PII（識別子） | ALCOA+ Attributable のために必須。PostgreSQL・SQLite ともに暗号化保存。マスキング表示を基本とし、権限ある RBAC ロールのみ参照可能とする |
| 氏名 | PII（直接識別子） | 表示に必要な場面のみ参照可能とする。検索・一覧表示では可能な限りマスキングする |
| 顔写真 | PII（生体情報） | 本人同意必須。用途は本人確認（ログイン・作業開始確認）のみ。退職・要求時に削除権を保証する |
| 作業実績（誰が・いつ・何を） | 間接 PII | ALCOA+ 品質証拠として保持。03 章 C03 の用途三限定（品質保証・工程改善・教育設計）にのみ使用する |
| タイムスタンプ | 単独では非 PII | 作業員 ID・氏名との結合によって間接 PII となるため、結合クエリには RBAC 制約を適用する |

### PII 最小化と削除権の保証

08 章 7 節（データ最小化）と整合して、位置情報・個人別時間ランキング・生体情報（顔写真を除く）は初期実装から除外する。退職・削除要求時は氏名・顔写真・従業員番号を削除または匿名化し、ALCOA+ 品質証拠としての作業実績は匿名化処理（作業員 ID を内部 UUID へ置換）によって証拠の完全性と削除権を両立させる。

---

**本節で確定した方針**
- 顔写真（生体情報）は本人同意・本人確認目的のみとし、削除権を技術的に保証することを確定する。
- 作業実績は ALCOA+ 品質証拠として保持し、削除要求時は匿名化処理（UUID 置換）で証拠の完全性と削除権を両立させる。
- 用途三限定（品質保証・工程改善・教育設計）の遵守を RBAC とデータモデルで技術的に強制する。

---

## 3. 鍵管理方針

### JWT 署名鍵（RS256）のローテーション

| 項目 | 方針 |
|---|---|
| ローテーション周期 | 90 日（設定値で変更可能） |
| ローテーション手順 | 新鍵生成 → 旧鍵を grace period（24 時間）で並行稼働 → grace period 終了後に旧鍵廃止 |
| 鍵の保管 | 環境変数または Docker Secret で管理。ソースコードへのハードコーディング禁止 |
| 鍵の形式 | PEM 形式の RSA 4096bit 鍵ペア |

### OAuth 2.1 クライアントシークレット（親機連携用）

- 親機ごとに固有のクライアントシークレットを発行する。複数の親機が存在する場合は親機 ID を名前空間として分離する。
- 環境変数または Docker Secret で管理する。設定ファイルへの平文保存は禁止する。
- ローテーション時は親機側との事前調整が必要なため、grace period を設けて旧・新シークレットの並行稼働を許容する。

### 設計余地として保持する事項

ver1.0.0 では HSM・外部 KMS（HashiCorp Vault 等）は非実装とする。鍵取得ロジックを環境変数・Docker Secret・KMS のいずれにも切り替えられる抽象層を実装し、将来的な採用を ADR に記録する。

---

**本節で確定した方針**
- JWT 署名鍵は 90 日周期でローテーションし、grace period 24 時間の並行稼働によってセッション継続性を保証することを確定する。
- 親機ごとに固有のクライアントシークレットを発行し、親機 ID を名前空間として分離することを確定する。
- ver1.0.0 では HSM・外部 KMS は非実装とし、将来的な採用に備えた設計余地を ADR に記録することを確定する。

---

## 4. 暗号化方針

### 暗号化対象と方式の確定

| 対象 | 方式 | 備考 |
|---|---|---|
| データ転送（in-transit） | TLS 1.3（最小） | 社内 LAN でも HTTPS 必須。HTTP による通信は禁止する |
| PostgreSQL データ（at-rest） | OS レベルの暗号化（BitLocker / dm-crypt） | PG 自体の TDE（Transparent Data Encryption）は設計余地として保持 |
| SQLite（タブレット at-rest） | SQLCipher | タブレット紛失時のデータ保護。暗号鍵はデバイスの Keystore / Keychain で管理する |
| バックアップ | AES-256 暗号化 | バックアップ鍵は PostgreSQL の OS 暗号化鍵とは別管理とする |

### HTTP 禁止・SQLCipher 採用の根拠

社内 LAN 完結環境であっても HTTP を許可しない。ゼロトラスト原則と整合して通信傍受・中間者攻撃を防ぎ、ALCOA+ Attributable（改竄のない通信の担保）を実現するために HTTPS を強制する。タブレット紛失時に SQLite 無暗号化では作業実績（間接 PII）が漏洩するため、SQLCipher によるデータベース全体の暗号化を採用し、ALCOA+ Enduring と PII 保護を両立させる。

---

**本節で確定した方針**
- 社内 LAN 内通信を含む全通信に TLS 1.3 を必須とし、HTTP による通信を禁止することを確定する。
- タブレットの SQLite に SQLCipher を適用し、タブレット紛失時の PII 漏洩を防ぐことを確定する。
- バックアップの暗号化鍵は PostgreSQL の OS 暗号化鍵と別管理とすることを確定する。

---

## 5. 監査ログ保全期間と改竄不可性

### 監査ログの対象

以下のイベントを監査ログの記録対象として確定する。

| イベント分類 | 記録対象 |
|---|---|
| 作業実績の全 CRUD | 誰が・いつ・どのレコードに対して何の操作を行ったか。ALCOA+ Attributable の核心 |
| 認証イベント | ログイン成功・失敗・ログアウト・JWT 失効・ロールアウト |
| SOP 改訂・承認イベント | 改訂者・承認者・改訂前後のバージョン番号 |
| 子機モードの連携イベント | マスタ同期の開始・完了・失敗。実績送信の開始・完了・失敗。親機 ID を記録する |

### 保全期間の方針

| ログ種別 | 保全期間 |
|---|---|
| 作業実績に紐づく監査ログ | 製品保証期間 + 法定保存期間（最低 7 年。設定値で業種に応じて変更可能） |
| 認証ログ | 90 日（設定値で変更可能） |
| 子機連携イベントログ | 作業実績に紐づく場合は 7 年。それ以外は 90 日 |

### 改竄不可性の実現

05 章で確定した Append-only Event Sourcing + SHA-256 ハッシュチェーンによって改竄不可性を担保する。各監査ログレコードには前レコードのハッシュが格納されるため、任意のレコードの改竄は後続するすべてのレコードのハッシュ不整合として検出可能である。これは ALCOA+ の Enduring 原則（記録が永続的かつ完全な形で残ること）の実装である。

監査ログ自体の at-rest 暗号化は 4 節の方針（OS レベルの暗号化）と統一する。

---

**本節で確定した方針**
- 作業実績に紐づく監査ログは最低 7 年間保全し、業種設定に応じて変更可能とすることを確定する。
- 05 章の Append-only Event Sourcing + SHA-256 ハッシュチェーンによって監査ログの改竄不可性を実現することを確定する。
- 子機連携イベントを監査ログの必須対象として追加し、親機 ID を記録することを確定する。

---

## 6. 親機連携時の認証情報保管

### 保管方法の確定

親機 API の認証情報（クライアントシークレット・API キー等）は環境変数または Docker Secret で管理する。ソースコード・リポジトリ管理下の設定ファイルへの平文ハードコーディングは禁止する。

### 複数親機への対応

複数の親機を持つ場合（将来拡張）、親機ごとに名前空間を分けて認証情報を管理する。命名規則は `PARENT_<PARENT_ID>_CLIENT_SECRET` のように親機 ID を含める形とし、設定の混在を防ぐ。

### ローテーション時のダウンタイム最小化

親機連携の認証情報をローテーションする場合、以下の grace period 設計によってダウンタイムを最小化する。

| ステップ | 内容 |
|---|---|
| 1. 新シークレット生成 | 子機側で新しいクライアントシークレットを生成する |
| 2. 親機への事前共有 | 親機管理者に新シークレットを安全な経路（対面または社内メール暗号化）で伝達する |
| 3. 並行稼働期間（最大 24 時間） | 旧・新の双方のシークレットを子機側で受け入れる grace period を設ける |
| 4. 旧シークレット廃止 | 親機側での切り替えを確認後、旧シークレットを廃止する |

---

**本節で確定した方針**
- 親機連携の認証情報は環境変数または Docker Secret で管理し、ソースコードへの平文ハードコーディングを禁止することを確定する。
- 複数親機が存在する場合は親機 ID を含む名前空間で認証情報を分離することを確定する。
- ローテーション時は grace period（最大 24 時間）の並行稼働でダウンタイムを最小化する設計とすることを確定する。

## 7. インシデント対応の最小骨格

### 対応フロー

個人開発前提のため、セキュリティオペレーションセンター（SOC）を設置しない。IT 担当 1 名が対応できるフローを以下の通り確定する。

| フェーズ | 内容 |
|---|---|
| 検知 | 監査ログの異常パターンを手動確認する。認証失敗の連続・通常外の時間帯のアクセス・タイムスタンプ逸脱を確認の契機とする。将来的なアラート機能は設計余地として保持する |
| 初期封じ込め | 疑わしいセッションに対する JWT の強制失効。子機と親機との連携を一時停止する。影響するサービスアカウントのパスワードを即時変更する |
| 調査 | 監査ログのフォレンジックにより影響範囲を特定する。ハッシュチェーンの整合性を検証して改竄の有無を確認する |
| 修復 | 鍵ローテーション（JWT 署名鍵・クライアントシークレット）。パッチ適用・設定修正を実施する |
| 再発防止 | ADR への記録。09 章の CAPA ループと連動して CAPA を起票し、根本原因を特定・対処する |

### Just Culture との整合

インシデントは問責の対象とせず、学習の源泉として扱う（Just Culture 原則）。インシデント発生の事実・対応経緯・原因・再発防止策を CAPA として記録することで、個人の責任追及ではなくシステム改善に繋げる。これは 08 章 7 節の倫理品質「心理的安全」サブ特性との整合でもある。

---

**本節で確定した方針**
- インシデント対応フローは IT 担当 1 名が実行できる最小骨格（検知・封じ込め・調査・修復・再発防止）として確定する。
- JWT 強制失効・親機連携停止・サービスアカウント変更を初期封じ込めの必須アクションとして確定する。
- インシデントを Just Culture 原則に基づいて扱い、09 章 CAPA ループと連動させることを確定する。

---

## 8. やらないこと

本章で対象外とする事項を明示する。設計余地として保持するものはその旨を記録する。

| 対象外事項 | 区分 | 備考 |
|---|---|---|
| 外部クラウド KMS（HashiCorp Vault 等）の実装 | ver1.0.0 非実装・設計余地 | 鍵取得ロジックを切り替え可能に設計することで将来導入を可能にする |
| SOC（セキュリティオペレーションセンター）の構築 | ver1.0.0 非実装 | IT 担当 1 名対応の最小骨格で代替する |
| SIEM（Security Information and Event Management）の導入 | ver1.0.0 非実装・設計余地 | 監査ログの構造化によって将来的な SIEM 連携を可能にする |
| WAF（Web Application Firewall）の実装 | ver1.0.0 非実装・設計余地 | 社内 LAN 完結という前提条件の下で優先度を下げる |
| IDS/IPS（侵入検知・侵入防止システム）の実装 | ver1.0.0 非実装・設計余地 | 将来的な採用を ADR に記録する |
| 外部専門家によるペネトレーションテストの必須化 | ver1.0.0 非必須・設計余地 | ver2.0.0 以降の品質向上フェーズで検討する |

---

**本節で確定した方針**
- 外部クラウド KMS・SOC・SIEM・WAF・IDS/IPS はすべて ver1.0.0 では非実装とし、設計余地として ADR に記録することを確定する。
- 外部専門家によるペネトレーションテストは ver1.0.0 では非必須とし、ver2.0.0 以降で検討することを確定する。
- 「やらないこと」の明示によって ver1.0.0 の実装スコープを意図的に絞り、個人開発における品質集中を保証する。
