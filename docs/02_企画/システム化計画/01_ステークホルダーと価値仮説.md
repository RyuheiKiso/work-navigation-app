# 01 ステークホルダーと価値仮説

本章の責務は、本システムに関係する 8 種のステークホルダーを定義し、それぞれの主たる関心・懸念・本書での応答位置を確定することである。あわせて、デジタルテイラリズムを回避するための倫理的境界を宣言する。

ステークホルダー関係の全体像については、図: fig_stakeholder_map.svg（img/ 配下）を参照すること。

---

## 1. ステークホルダー 8 種の定義

本システムに直接または間接に関与するステークホルダーを以下のとおり定義する。ステークホルダーとは、本システムの導入・運用・廃棄に関して利害を有する個人または組織を指す。

| ステークホルダー | 定義・役割 | 主な接点 |
|---|---|---|
| 作業員 | タブレットで手順に従って作業し、記録を入力する。習熟度・年齢・母語が多様であり、IT リテラシーに大きな差がある | タブレット APP |
| 現場監督 | ラインの生産進捗・品質・安全を管理し、異常時に判断する。シフトをまたぐ引継ぎと進捗可視化が主な責務である | タブレット APP / 管理 Web |
| 品質担当 | 手順マスタの承認、不適合処理、トレサビ照会、内部監査を担う。ALCOA+ に準拠した記録品質の維持に責任を持つ | 管理 Web |
| 保全担当 | 設備の点検・保全作業を実施し、設備履歴・校正記録を管理する。保全作業を生産作業とは独立して管理する必要がある | タブレット APP / 管理 Web |
| IT 担当 | システムの導入・運用・バックアップ・障害対応を担う。本システムでは 1 名前提であり、専任の IT 部門が存在しない場合を想定する | バックエンド / インフラ |
| 経営層 | 投資判断・導入可否・規制責任・労働政策の意思決定者である。ROI の確認と法的責任の引受けが主要な関心領域である | 管理 Web（レポート） |
| 規制当局想定 | 実在する外部規制当局ではなく、内部監査・第三者品質認証・顧客監査（サプライヤー監査）の際に想定される審査者として定義する | 記録・証拠・監査証跡 |
| 労働者代表 | 作業員の利害を代表する。デジタル監視ツール化・人事評価への転用・外国人労働者への配慮についての懸念を代表する存在である | 導入合意プロセス |

**本節で確定した方針**
ステークホルダーを 8 種に確定し、それぞれの役割・主な接点を定義した。
本書の全章は、この 8 種すべての懸念に対して応答する責務を持つ。
特に労働者代表の懸念（監視・評価転用・多言語化）は 01 章の倫理境界宣言で直接応答する。

## 2. 各ステークホルダーの主たる関心と懸念

各ステークホルダーが本システムに対して持つ主たる関心と懸念、および本書内での応答位置を以下のとおり確定する。

### 2-1. 作業員

**関心**: 操作インターフェースの使いやすさ・手順の正確性・中断後の作業再開支援。作業中に次のステップが明示され、自分が今何をすべきかを迷わない状態が価値である。

**懸念**: 本システムが監視ツールとして機能し、作業速度・行動パターンが管理側に筒抜けになることへの不安。習熟度差による疎外感（熟練者には冗長、初心者には難解）。母語が日本語でない場合のインターフェース理解困難。

**本書での応答位置**: 04 章（習熟度別シナリオ）・本章 H2-4（倫理境界宣言）・08 章（倫理品質・プライバシー保護）・09 章（Just Culture に基づく組織原則）

### 2-2. 現場監督

**関心**: 生産進捗のリアルタイム可視化・異常発生時の迅速対応・シフト引継ぎ品質の担保。異常が発生した時点で即座に状況を把握できる仕組みが価値である。

**懸念**: 紙運用との二重管理による現場負荷の増大。手順マスタの改訂が現場に反映される速度が遅い場合の品質リスク。デジタルと現実の乖離が生じた際の判断根拠の不在。

**本書での応答位置**: 04 章（シナリオ設計）・09 章（SOP 改訂ループと変更管理）

### 2-3. 品質担当

**関心**: ALCOA+ に準拠した証拠品質の確保。逆方向トレサビ（不良品から工程・作業員・材料ロットまで遡及できること）。手順マスタの版管理と承認履歴の完全性。

**懸念**: 電子記録の改竄リスク（タイムスタンプの事後修正・記録の削除）。規制への対応境界が曖昧であり、内部監査時に範囲が問われた場合に説明できないリスク。不適合記録が形式だけで実態を反映しないリスク。

**本書での応答位置**: 03 章（規制三段階による境界確定）・06 章（ALCOA+ マトリクスとデータモデル）

### 2-4. 保全担当

**関心**: 設備ごとの点検・保全履歴の正確な記録。校正管理（計量器の校正期限・校正証明書の紐付け）。保全作業記録と生産作業記録を明確に分離した管理。

**懸念**: 保全作業が生産作業と同一フローに混在することで記録上の混乱が生じるリスク。校正期限切れの計量器が使用された場合に記録から遡及できないリスク。

**本書での応答位置**: 06 章（データモデル・校正参照モデル）

### 2-5. IT 担当

**関心**: Windows Server 2022 上での安定動作。運用負荷（バックアップ・ログローテーション・アップデート）の低減。障害発生時の自己完結的な診断と復旧手順。長期保守における技術的負債の抑制。

**懸念**: Rust・React Native という技術スタックの習熟コスト（一般的な PHP/Java/C# とは異なる）。個人開発で作られたシステムに対して社内標準から外れた保守体制を強いられるリスク。バックアップ・リストア手順が属人化するリスク。

**本書での応答位置**: 07 章（技術スタック選定根拠と代替案棄却理由）・08 章（保守性・可用性の非機能要件）・11 章（前提・制約・リスク）

### 2-6. 経営層

**関心**: 投資対効果の説明責任（稟議への適用可能性・補助金活用の可否）。競合製品との比較における優位性の確認。ver1.0.0 での本番運用可能性。

**懸念**: 個人開発による信頼性への疑問（属人化・開発者の退場リスク）。独断的な技術・設計選択が後から変更困難になるリスク。導入後に法的・規制上の問題が発覚した場合の責任帰属。

**本書での応答位置**: 02 章（ビジネス目標・対象範囲）・07 章（代替案比較）・10 章（スケジュール・意思決定プロセス）・11 章（リスクと対応方針）

### 2-7. 規制当局想定

**関心**: 電子記録の完全性（改竄不可能性・完全性・可読性）。電子署名の法的有効性。記録保管期間の遵守。監査証跡（Audit Trail）が実質的に機能していること。

**懸念**: 規制適合性の境界が曖昧であり、対象外と判断した規制についての根拠が文書化されていないリスク。証拠品質が内部監査に耐えうる水準に達していないリスク。

**本書での応答位置**: 03 章（規制三段階による境界の明示）・06 章（ALCOA+ マトリクス）・08 章（非機能要件方針）

### 2-8. 労働者代表

**関心**: デジタルテイラリズムの防止。作業ログが人事評価に転用されないことの制度的保証。多言語対応（外国人技能実習生・特定技能労働者への配慮）。本システムに対するボトムアップ改善提案の受け皿が存在すること。

**懸念**: 行動データ（作業速度・ミス頻度・入力時刻等）が人事評価に使用されるリスク。外国人労働者が言語障壁によりシステムを正確に使用できないリスク。改善提案を出しても組織に無視されるリスク（提案疲労）。

**本書での応答位置**: 本章 H2-4（倫理境界宣言）・本章 H2-6（透明性公約）・08 章（倫理品質）・09 章（Just Culture）・11 章（制約）

**本節で確定した方針**
8 種全ステークホルダーの関心・懸念・応答章を確定した。
懸念への応答は対応章への参照として明示し、本章での先取り記述は倫理境界宣言の範囲に限定する。
各章の執筆時は、担当するステークホルダーの懸念に漏れなく応答したことを自己監査チェックリストで確認する。

## 3. 価値仮説マトリクス

本節では、各ステークホルダーに対して本システムが提供する価値を「誰にとって / 何が / いつ / どう価値になるか」の形式で確定する。Value as Designed (VaD) は設計意図として定義し、Value in Use (ViU) はシステム導入後に実測・検証する仮説として区別する。

| ステークホルダー | Value as Designed (VaD): 設計が意図する価値 | Value in Use (ViU): 現場で検証すべき仮説 |
|---|---|---|
| 作業員 | 手順の次ステップが常に明示されることで、中断後の再開時でも正しい状態から作業を継続できる | EWI（早期警告指示）によるリアルタイムフィードバックが、認知負荷の軽減をもたらすかどうか |
| 現場監督 | 進捗・品質状態をリアルタイムで閲覧でき、異常を発見してから対処するまでのリードタイムが短縮する | 進捗可視化がシフト引継ぎの口頭説明時間を実測可能な範囲で短縮するかどうか |
| 品質担当 | ALCOA+ に準拠した電子記録が自動的に生成されることで、内部監査準備の工数が削減される | 記録完全性率の向上が実際の監査工数削減に結びつくかどうか |
| 保全担当 | 設備ごとの点検履歴・校正記録が一元管理され、期限切れの見逃しがシステムによって防止される | 校正期限アラートが実運用で機能し、見逃しゼロを維持できるかどうか |
| IT 担当 | オンプレ完結の構成により、外部サービスへの依存がなく障害切り分けが単純化される | Rust + Docker の技術スタックが実際の運用負荷を許容範囲に収めるかどうか |
| 経営層 | 作業記録・不適合記録・改善履歴が蓄積されることで、製造品質向上の可視化とサプライヤー監査対応が可能になる | 導入コストと品質改善効果の比較が実績データに基づいて経営判断に使える形で提示されるかどうか |
| 規制当局想定 | Audit Trail・版管理・電子署名が設計に組み込まれており、監査時に証拠として提出できる記録が標準で生成される | 内部監査・顧客監査において記録品質が「証拠として受け入れられる」と審査者が判断するかどうか |
| 労働者代表 | データ用途が設計レベルで品質保証・工程改善・教育設計に限定され、人事評価への転用が技術的・制度的に排除される | 多言語インターフェースの提供が外国人労働者の操作ミス率低減に実際に貢献するかどうか |

ViU の検証は ver1.0.0 リリース後の運用フェーズで実施する。本計画書は ViU の仮説を定義するのみであり、検証結果の約束はしない。

**本節で確定した方針**
価値仮説は VaD（設計意図）と ViU（検証仮説）として明確に区別する。
ViU の検証はリリース後の運用フェーズに委ね、本書ではその仮説の定義にとどめる。
ROI を事前に約束する形式の価値仮説記述は本書全体を通じて行わない。

## 4. 倫理的境界宣言（デジタルテイラリズム回避）

本節は本システムが設計上遵守する倫理的境界を宣言する。この宣言は方針の表明ではなく、設計制約として確定する。

### 4-1. 「支援」と「監視」の意図的分離

本システムの目的は作業員の作業を「支援」することであり、「監視」することではない。この区別は意図の問題ではなく、設計上の実装制約として次のとおり確定する。

Aiello & Kolb (1995) の知見は、複雑な認知作業において電子的監視がパフォーマンスを低下させることを示している。本システムが Navigation と Traceability の二大価値を実現するためには、作業員がシステムを「評価のために監視するもの」として認識することを防止しなければならない。この認識が生じた瞬間、Navigation の価値（状況認識支援・EWI）は失われる。

Brown らの指摘するデジタルテイラリズム（Frederick Taylor の科学的管理法のデジタル化による再来）は、作業の断片的計測・個人別スコアリング・行動の可視化を通じて熟練者の自律性を剥奪することを意味する。Zuboff の監視資本主義論においても、行動データの収集がいかに本来目的から逸脱した価値抽出に使われるかが示されている。

本システムはこれらの批判が指摘する構造的問題を設計制約として内部化する。

詳細は [`90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md`](../../90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md) および [`90_業界分析/09_職務設計とモチベーション論.md`](../../90_業界分析/09_職務設計とモチベーション論.md) を参照すること。

### 4-2. 個人ランキング・個人別生産性スコアリングの禁止

本システムは個人別の生産性スコア・作業速度ランキング・達成率の個人比較表示を実装しない。

禁止の根拠は以下のとおりである。

- **GDPR 第 22 条**: 自動化された処理にのみ基づく個人を著しく影響する決定への異議申立権を保障する。個人別スコアリングは実質的にこの条項が問題とする自動意思決定に該当しうる
- **日本改正個人情報保護法（2022）**: 個人関連情報の第三者提供規制・オプトアウト手続きの厳格化。作業ログを人事評価目的に転用する場合、本人同意なしには違法となる
- **Hackman & Oldham JCM（仕事の特性理論）**: スキル多様性・タスク同一性・タスク重要性・自律性・フィードバックという 5 つの中核的仕事特性のうち、個人別スコアリングは「自律性」を著しく損なう。自律性の喪失は内発的動機づけを低下させ、長期的な品質劣化につながる
- **Deci & Ryan SDT（自己決定理論）**: 有能感・自律性・関係性という三つの基本的心理欲求のうち、パフォーマンス監視は有能感を外部コントロールに委ねることで自律的動機を阻害する

詳細は [`90_業界分析/09_職務設計とモチベーション論.md`](../../90_業界分析/09_職務設計とモチベーション論.md) を参照すること。

### 4-3. 行動データの用途制限

本システムが収集する行動ログ（作業開始/終了時刻・各 Step の記録タイムスタンプ・入力値・EWI 発生記録等）の用途は、以下の三用途に限定する。

1. **品質保証**: ALCOA+ 準拠の証拠として不適合調査・トレサビ照会に使用する
2. **工程改善**: ライン全体・工程全体・作業種別単位での集計による工程改善に使用する（個人を特定しない集計単位を原則とする）
3. **教育設計**: 習熟度分析・カリキュラム設計・OJT 設計に使用する（個人特定は本人同意を前提とする）

**禁止する用途**:
- 人事評価・昇給・昇格・解雇の根拠への転用
- 個人別生産性ランキング・達成率の上長への自動報告
- 懲罰的目的での監査証跡の利用

この用途制限は技術的制約（アクセス権限設計）と制度的制約（就業規則・労使協定への明示）の双方によって担保する。技術的制約のみでは目的外利用を防止できないため、制度的制約との併用を設計原則とする。

目的分離アーキテクチャ（品質保証目的データと人事評価目的データを別管理する設計思想）については 06 章データモデル設計で具体化する。

### 4-4. Work-as-Done の記録と「違反」解釈の禁止

本システムの作業ログは「Work-as-Done」（実際に行われた作業）の記録である。SOP に定義された「Work-as-Imagined」（設計上想定された作業）との乖離が記録に現れた場合、それは設計者の想定と現場の適応の差異を示す情報であり、作業員個人の「違反」の証拠として解釈してはならない。

Safety-II の立場（安全はルールの遵守ではなく、成功の理解から生まれるという立場）に基づき、Work-as-Done と Work-as-Imagined の乖離は「適応の証拠」として扱い、SOP 改訂・手順設計の見直しに活用する。この解釈方針は 09 章の運用・組織方針で制度化する。

詳細は [`90_業界分析/13_安全文化と安全管理システム.md`](../../90_業界分析/13_安全文化と安全管理システム.md) を参照すること。

### 4-5. Reason Just Culture の組織原則としての採用

本システムの設計思想は James Reason の Just Culture モデルを組織原則として採用する。

Just Culture は以下を区別する。

- **スリップ・ラプス（意図なき行為の失敗・意図した行為の実行失敗）**: 非難しない。システムと手順設計の改善問題として扱う
- **ミステイク（誤った判断に基づく行為）**: 非難しない。知識・訓練・手順設計の改善問題として扱う
- **故意の規則違反**: 問責の対象とする。ただし、規則が合理的でない場合は規則の見直しを先行させる
- **悪意ある行為**: 問責の対象とする

本システムの記録機能は Just Culture の判断基準を支援するための客観的証拠を提供するが、スリップ・ラプスの発見を自動的に問責プロセスに結びつける設計は行わない。これはシステムが Just Culture を侵食することを防ぐための設計上の意図的な省略である。

詳細は [`90_業界分析/13_安全文化と安全管理システム.md`](../../90_業界分析/13_安全文化と安全管理システム.md) を参照すること。

**本節で確定した方針**
本システムは「支援」と「監視」を設計制約として分離し、個人別生産性スコアリングを実装しないことを確定した。
行動データの用途を品質保証・工程改善・教育設計の三用途に限定し、人事評価への転用を技術的・制度的双方で禁止する。
Just Culture を組織原則として採用し、スリップ・ラプスを非難しない設計姿勢を確定した。

## 5. 規制当局想定ステークホルダーの取り扱い

### 5-1. 定義の根拠

本システムの想定導入環境（中小製造業・単一工場・社内 LAN）において、外部規制当局（国の規制機関等）への電子記録の提出義務は現時点では存在しない。

しかし、以下の三つの監査機会が現実的に発生しうる。

1. **ISO 9001 内部監査**: 年次実施が一般的であり、電子記録システムの適合性確認が監査項目に含まれる
2. **第三者品質認証**: ISO 9001 認証機関による外部審査において、電子記録の証拠品質が問われる
3. **顧客監査（サプライヤー監査）**: 大手自動車メーカー・電機メーカー等の顧客が実施するサプライヤー監査において、製造記録・トレサビ記録の完全性が審査される

これらの監査において審査者が問う基準は、外部規制当局が問う基準と実質的に等価であることが多い。したがって、「規制当局想定ステークホルダー」として定義することは、実在しない外部機関を想定した形式的な作業ではなく、実際に発生しうる審査に備えた実践的な設計基準の引き上げを意味する。

### 5-2. 本書全体への影響

規制当局想定ステークホルダーの定義は、以下の章の設計基準に直接影響する。

- **03 章 機能スコープ**: 規制三段階（準拠する / 対応する / 対象外と判断する）の適用基準が「審査者が納得できる根拠を示せるか」という基準で決定される
- **06 章 データモデル**: ALCOA+ の各要件（Attributable / Legible / Contemporaneous / Original / Accurate / Complete / Consistent / Enduring / Available）について、監査時に証拠として提示できる記録構造の設計が求められる
- **08 章 非機能要件**: 記録保管期間・Audit Trail の完全性・システム可用性の要件が、内部監査に耐えうる水準で設定される

**本節で確定した方針**
規制当局想定ステークホルダーを ISO 9001 内部監査・第三者認証・顧客監査における審査者として定義した。
この定義が 03 章の規制三段階設計と 06 章の ALCOA+ データモデルの品質基準を引き上げる根拠となることを確定した。
外部規制当局への提出義務が現時点で存在しないことは、証拠品質基準の引き下げ理由にならない。

## 6. 労働者代表に対する透明性公約

本節は、本システムの導入にあたり、労働者代表（および全作業員）に対して透明性を担保するための制度的公約を確定する。これは運用方針の提案ではなく、設計段階で確定する制約である。

### 6-1. 収集データの開示義務

本システムが収集するデータについて、以下の情報を全作業員に対して書面で提供する。

| 開示項目 | 内容 |
|---|---|
| データの種類 | 作業 Step の記録タイムスタンプ・入力値・EWI 発生履歴・写真・動画・電子署名に使用した識別情報 |
| 用途 | 品質保証・工程改善・教育設計の三用途のみ（人事評価への転用禁止を明示） |
| 保存期間 | 製品種別ごとに定める保存期間（ISO 9001 要求・顧客要求の最長値を採用） |
| アクセス権限 | 役割ベースのアクセス制御（RBAC）に基づき、誰がどの記録を閲覧できるかを明示 |
| 削除・修正の可否 | ALCOA+ の Original 要件により事後修正は不可。修正が必要な場合は追記形式（理由・承認者付き）のみ許容 |

### 6-2. 多言語プライバシー通知の提供

外国人技能実習生・特定技能労働者が在籍する現場では、上記開示情報を以下の言語で提供する。

- 日本語（必須）
- 英語（必須）
- ベトナム語・インドネシア語・タガログ語・中国語（実際の在籍者の母語に応じて選択）

多言語プライバシー通知の具体的な提供方法（タブレット APP 内の多言語表示・書面の配布等）は 09 章の教育・運用設計で確定する。

### 6-3. データ最小化原則の遵守

GDPR 第 5 条第 1 項 (c)（データ最小化の原則）に準拠し、業務目的に必要なデータのみを収集する。

収集するデータは 06 章のデータモデルで明示的に定義し、定義外のデータは収集しない。たとえば、作業員の顔画像を用いた本人認識・位置情報のリアルタイムトラッキング・生体情報の収集は本システムでは実施しない。

### 6-4. ボトムアップ改善提案への応答義務

Kaizen Teian（改善提案）は、品質管理活動において作業員の知見を工程に還元する重要な仕組みである。本システムは改善提案の収集機能を提供し、提案を黙殺しない運用設計を義務とする。

Edmondson の指摘する心理的安全性と、QC サークル・Kaizen Teian の連鎖効果が機能するためには、提案が「何らかの形で応答される」という信頼が不可欠である。提案件数の追求（件数目標化）は提案疲労を引き起こすことが知られており、本システムは件数よりも採用率・フィードバック速度を改善品質の指標とする設計を採用する。

詳細は [`90_業界分析/39_QCサークル・Kaizen Teianとボトムアップ品質活動.md`](../../90_業界分析/39_QCサークル・Kaizen Teianとボトムアップ品質活動.md) を参照すること。

### 6-5. 導入合意プロセスへの労働者代表の参加

本システムの導入にあたっては、労働者代表が以下の段階に参加する機会を設ける。

1. **試用段階**: パイロット運用前に本章の倫理境界宣言・透明性公約の説明を実施し、理解確認を得る
2. **フィードバック段階**: パイロット運用中に発生した懸念・問題を労働者代表を通じて集約し、本番導入前に対応する
3. **継続的モニタリング**: 本番運用中も定期的に（最低年 1 回）透明性公約の遵守状況を報告する

この参加義務は 09 章の組織・運用設計で制度化する。

**本節で確定した方針**
労働者代表への開示義務（データ種類・用途・保管期間・アクセス権限）を設計制約として確定した。
データ最小化原則（GDPR 第 5 条第 1 項 (c) 準拠）を遵守し、業務目的外のデータは収集しないことを確定した。
Kaizen Teian への応答義務を確定し、件数目標化ではなく採用率・フィードバック速度を品質指標とすることを確定した。
