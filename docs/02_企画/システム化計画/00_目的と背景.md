# 00 目的と背景

## 本章の責務

本章は本書（システム化計画書）の位置づけ・上位文書との分担・本書が答える問いと答えない問いの境界を確定する。
本書全体を読む前提知識として位置づけ、全 8 種のステークホルダーが最初に読む章である。

## H2-1. 本書の目的と読者

### IPA 共通フレーム 2013 におけるシステム化計画の位置づけ

IPA 共通フレーム 2013 (SLCP-JCF2013) は、システム開発のライフサイクルを「企画プロセス」「要件定義プロセス」「システム開発プロセス」等に分類する。本書は企画プロセスの中の「システム化計画」に該当する。

企画プロセスは「システム化構想」と「システム化計画」の 2 段階に分かれる。
システム化構想（別途作成予定）は「なぜ作るか」の問いに答え、解決すべき課題と実現する価値を確定する。
本書（システム化計画）は「何を作るか」と「どの枠組みで作るか」の問いに答え、対象範囲・機能スコープ・アーキテクチャ原則・技術スタック・データモデル・運用方針・開発スケジュールを確定する。

### 本書が直接答える問い

- **What（何を作るか）**: 対象範囲・機能スコープ・非スコープ
- **How-Frame（どの枠組みで作るか）**: アーキテクチャ原則・技術スタック選定根拠・データモデル中核設計・品質特性と非機能要件方針・運用方針・開発スケジュール

### 本書が答えない問い

- **Why（なぜ作るか）**: 解決すべき課題・価値命題・倫理スタンス・ビジョン

「なぜ作るか」はシステム化構想（別途作成予定）に委ねる。本書はその答えを所与として引き継ぎ、実現手段を確定することに専念する。
本書の中で Why に踏み込む記述を発見した場合、それは本書のスコープ逸脱であり、削除または上位文書への移動が必要である。

### 一次読者

本書の一次読者は 8 種のステークホルダー全員とする。ただし、以下の 3 者は本書の内容を直接的な判断材料として使用するため、特に精読を要する。

| ステークホルダー | 本書から得る判断材料 |
|---|---|
| IT 担当 | 実装可能性の判断（アーキテクチャ原則・技術スタック・非機能要件） |
| 品質担当 | 証拠品質の判断（データモデル・ALCOA+ 準拠・トレサビ設計） |
| 経営層 | 投資判断（対象範囲・開発スケジュール・前提制約リスク） |

---

**本節で確定した方針**
- 本書は WHAT（対象範囲・機能スコープ）と HOW-FRAME（設計方針・運用方針・スケジュール）の確定に責務を限定する。
- WHY（課題・価値命題・倫理スタンス）はシステム化構想（別途作成予定）に委ね、本書では所与とする。
- 一次読者は 8 種のステークホルダー全員とし、IT 担当・品質担当・経営層は精読を要とする。

| 項目 | 内容 |
|---|---|
| What（対象範囲） | どの業務・組織・工程を対象とするか |
| What（機能スコープ） | ver1.0.0 で実装する機能と明示的に対象外とする機能 |
| How-Frame（アーキテクチャ原則） | 設計判断の軸となる原則 |

| How-Frame（運用方針） | 導入後の運用体制・教育計画 |
| How-Frame（スケジュール） | ver1.0.0 達成に向けたマイルストーン |

### 両書の境界

構想は「べき論」を扱う。「この課題を解決すべきである」「この価値を実現すべきである」という規範的命題が構想の言語である。
計画は「する論」を扱う。「この機能を実装する」「このアーキテクチャを採用する」という具体的確定命題が計画の言語である。
本書の中に「べき論」が混入した場合、それは構想書への移管対象である。

### 本書の有効期限

本書の権威は ver1.0.0 リリースをもって終了する。
リリース後、実装の正とするのは要件定義書および概要設計書であり、本書はそれらへの入力文書として歴史的参照の位置づけに降格する。
ただし「本書が約束しないこと」宣言（本書 README および `11_前提・制約・リスク.md` に記載）は、ver1.0.0 リリース後も変更履歴として永続的に保持する。

**本節で確定した方針**
- 構想書は「べき論（Why）」、計画書は「する論（What/How-Frame）」と役割を分担し、両書の記述が逆転しないよう管理する。
- 本書の権威は ver1.0.0 リリースをもって要件定義書・概要設計書に移行する。
- 「本書が約束しないこと」宣言は ver1.0.0 リリース後も変更履歴として保持する。

## H2-3. 製造業における「作業ナビゲーション」の定義

### 作業階層の電子化

JIS Z 8141（生産管理用語）は製造における作業を階層的に定義する。本アプリはこの階層を電子的に扱う基盤として設計する。

| 階層 | 定義 | 本アプリでの扱い |
|---|---|---|
| 工程（Process） | 製品を完成させるための一連の作業の集合 | ナビゲーション対象のトップ階層。工程ごとに担当・順序を管理する |
| 作業（Operation） | 工程を構成する個別の作業単位 | ナビゲーション画面の基本単位。作業ごとに手順・記録項目を定義する |
| 要素作業（Element） | 作業を構成する最小の分析単位 | Step の集合として表現する。IE による時間研究の単位に対応する |
| ステップ（Step） | 作業者が一動作で完了できる最小指示単位 | 画面に表示する最小手順。各 Step に対して記録入力・確認チェックを紐付ける |

### Work-as-Imagined と Work-as-Done の乖離縮小

製造現場の品質問題の多くは、手順書に記載された理想的な作業（Work-as-Imagined: WAI）と、実際の現場で行われている作業（Work-as-Done: WAD）の乖離に起因する。この乖離は以下の経路で発生する。

熟練作業員が退職し、後継者が「口伝え」のみで作業を習得する過程で暗黙知が失われる。
手順書は作成後に更新されないまま現場の実態と乖離し続け、作業員は手順書を無視することを学習する。
異常発生時、原因追跡に必要な作業記録が存在しない、または記録が改ざんされている状況が放置される。

本アプリは Step 単位で手順を提示し、各 Step の完了を記録として残すことで、WAI と WAD の乖離を縮小する。ただし、アプリが提示する手順そのものが現実の最良作業を正確に反映していることは、導入組織が責任を持って確認・更新しなければならない。本アプリはその更新を支援するが、更新の責任を代行しない。

### Intrinsic EPSS としての定義

本アプリは Intrinsic EPSS（Electronic Performance Support System）として設計する。EPSS には以下の 2 分類がある。

- **Intrinsic EPSS**: 作業フローに統合されており、別途参照が不要。作業しながら支援を受けられる
- **Extrinsic EPSS**: 作業フローの外側に存在するマニュアルやヘルプシステム。参照のために作業を中断する必要がある

本アプリは Intrinsic EPSS として、以下の 4 機能を分離せず一体のフローに統合する。

1. **手順提示**: 各 Step の実施手順・注意事項・参考情報を作業中に提示する
2. **記録入力**: 数値・テキスト・写真等の作業結果を Step 完了時に入力する
3. **品質検査**: 判定基準と実測値を照合し、合否を即時フィードバックする
4. **トレサビ記録**: 作業者・日時・設備・材料・結果を不可分な証拠として自動記録する

この統合により、作業員は「手順書を見ながら別の記録用紙に書く」という二重作業から解放され、アプリに沿って作業すること自体がトレサビ記録の生成になる。

### 「作業を管理する」のではなく「作業を支援する」という立場

本アプリは作業の管理ツールとして設計しない。管理ツールとは、管理者が現場を監視・統制する目的で使用するシステムを指す。
本アプリは作業の支援ツールとして設計する。支援ツールとは、作業員が自身の作業をより正確・安全・効率的に遂行するために使用するシステムを指す。

この立場の宣言は設計上の具体的制約を生む。
作業員の個人別パフォーマンス比較レポートは提供しない（`03_機能スコープと非スコープ.md` で対象外として明示する）。
作業記録へのアクセス権限は当該作業に関与した者に限定し、無制限な参照を許可しない（`06_データモデル中核設計.md` で設計する）。
作業速度・件数の集計を目的としたダッシュボードは提供しない（`03_機能スコープと非スコープ.md` で対象外として明示する）。

---

**本節で確定した方針**
- 本アプリは JIS Z 8141 の作業階層（工程→作業→要素作業→Step）を電子化の基本単位とする。
- Intrinsic EPSS として手順提示・記録入力・品質検査・トレサビ記録を一体のフローに統合し、別途マニュアルを不要とする。
- 「作業を支援する」立場から、個人別パフォーマンス評価・監視目的の機能を対象外と判断する。

## H2-4. トレーサビリティを内包する設計思想の宣言

### ISO 9000:2015 に基づくトレサビ定義

ISO 9000:2015 はトレーサビリティを「対象の履歴、適用、または所在を追跡できる能力」と定義する。
本アプリはこの定義を電子記録として実装する。具体的には、以下の問いに答えられる記録を生成することをトレサビの達成基準とする。

- この製品のある工程を、誰が、いつ、どの設備を使って作業したか
- その作業において、どの材料（ロット・シリアル）を使用したか
- 作業中に測定した数値は何であったか
- 異常・逸脱が発生したか。発生した場合、どのように処置したか
- その作業は、当時有効だったどのバージョンの手順書に基づいて実施されたか

これらの問いに 1 件ずつ答えられる記録が存在することが、本アプリが達成すべきトレサビの最低水準である。

### 「記録が存在する = 品質が保証される」という誤謬の否定

記録の存在は品質の証拠であり、品質の保証ではない。
記録が存在しても、記録された作業が基準を満たしていない場合、品質は保証されない。
記録が存在しても、記録が正確でない（後から改ざん・追記された）場合、証拠としての価値はない。
記録が存在しても、記録できていない工程が存在する場合、システム全体のトレサビは断絶する。

本アプリは記録の存在を自動的に保証するが、記録の正確性・完全性・信頼性は設計によって担保する。その設計基準として ALCOA+ 9 原則を採用する。

### Navigation と Traceability の補完構造

Navigation（ナビゲーション）と Traceability（トレーサビリティ）は本アプリにおいて不可分の関係にある。

Navigation が Traceability の証拠品質を担保する構造は以下のとおりである。
手順を提示せずに記録だけ求めると、作業員は記録のために作業を中断する。記録は後追いになり、タイムスタンプの信頼性が低下する（ALCOA+ の Contemporaneous 原則違反）。
手順提示と記録入力を同一フローに統合することで、記録はその作業が完了した瞬間に入力される。これにより Contemporaneous 原則を構造的に満たす。
手順に紐付いた記録は、どの手順のどの Step に対する記録かが明確である。これにより Attributable 原則（誰が何に対して記録したか）と Original 原則（一次記録であること）を担保する。

Traceability なき Navigation は、作業を支援するが証拠を残さない。
Navigation なき Traceability は、証拠を求めるが品質を担保しない。
本アプリはこの 2 つを後付けで統合することを設計上の失敗と定義し、最初から不可分の構造として設計する。

### ALCOA+ 9 原則の採用宣言

本アプリは電子記録の品質基準として ALCOA+ 9 原則を採用する。
ALCOA+ は製薬業界の GMP 文脈で発展した原則であるが、その核心は「電子記録が法的・科学的証拠として機能するために必要な属性」の定義であり、製造業全般の電子記録設計に適用できる。

本アプリは GMP に準拠するのではなく、ALCOA+ の証拠品質論を設計の基準として採用すると判断する。

| 原則 | 定義 | 本アプリでの実装方針 |
|---|---|---|
| Attributable（帰属性） | 誰が何をしたかが明確である | ログイン済み作業員に記録を紐付け、否認不可とする |
| Legible（可読性） | 記録が永続的に読める状態にある | 文字化け・フォーマット依存を排除し、テキスト形式で保存する |
| Contemporaneous（同時性） | 作業と同時に記録する | Step 完了時の即時入力をフロー設計で強制する |
| Original（原本性） | 一次記録が保全される | 記録の上書き禁止、修正は追記形式で管理する |
| Accurate（正確性） | 記録が実態を正確に反映する | 数値範囲チェック・必須項目チェックをシステムで強制する |
| Complete（完全性） | 必要な記録がすべて存在する | 全 Step の完了記録なしに工程を完了できない設計とする |
| Consistent（一貫性） | 記録が矛盾なく整合する | タイムスタンプはサーバー側で付与し、クライアント側改ざんを防止する |
| Enduring（永続性） | 記録が要求保存期間を満たす | PostgreSQL への書き込み後、削除 API を提供しない設計とする |
| Available（可用性） | 必要な時に記録を参照できる | オンライン時は即時参照、オフライン時はローカルキャッシュを参照する |

ALCOA+ の具体的な実装設計は [`06_データモデル中核設計.md`](06_データモデル中核設計.md) で確定する。

---

**本節で確定した方針**
- ISO 9000:2015 の定義に基づき、「誰が・いつ・何を・どの手順で・どんな結果で」の 5 要素を一体の記録として保全することをトレサビの最低水準とする。
- Navigation と Traceability は不可分の設計とし、どちらかの後付け統合は設計の失敗と定義する。
- ALCOA+ 9 原則を証拠品質の基準として採用し、`06_データモデル中核設計.md` で具体化する。

## H2-5. phase なし・ver1.0.0 本番品質の方針

### 個人開発における phase 廃止の決定

本プロジェクトは個人開発である。チーム開発で有効な段階的リリース（phase 1 / phase 2 / phase 3）は、以下の理由から採用しない。

段階的リリースは複数開発者が並行開発する際の調整コストを削減するために有効である。
個人開発では開発者は 1 名であり、並行開発の調整コストは発生しない。
段階的リリースは「とりあえず動くもの」を先に出し、後から品質を上げる戦略を前提とする。
本プロジェクトは ver1.0.0 でプロダクションレベル品質を達成することを意思決定している。
したがって、phase 構造は本プロジェクトに適合しない。

本プロジェクトは単一の ver1.0.0 を設計し、そのリリースをもってプロダクション投入とする。

### ver1.0.0 = プロダクションレベル品質の定義

ver1.0.0 におけるプロダクションレベル品質とは、以下のすべてを満たす状態を指す。

- `08_品質特性と非機能要件方針.md` で確定する非機能要件基準をすべて達成している
- `06_データモデル中核設計.md` で設計する ALCOA+ 9 原則の実装が完了している
- 中小製造業の単一工場における実際の作業工程でエラーなく稼働する
- 作業員・現場監督・品質担当が追加訓練なしに基本操作を習得できる
- システム障害時にトレサビ記録が消失しない設計が実装されている

「とりあえず動く」「後で直す」という判断は、ver1.0.0 リリース前には存在しない。
技術的負債を意図的に積み上げて後で返済する戦略を採用しない。
これは妥協禁止ポリシーの具体的内容である。

### 「やらないことを明確にする」こととの両立

妥協禁止ポリシーはすべてを実装するポリシーではない。
やらないことを明確にした上で、やることを完全に実装するポリシーである。

本書が約束しないこと（本 README および `11_前提・制約・リスク.md` に記載）は、妥協ではなく意思決定の結果である。
複数工場対応・SaaS・マルチテナンシー・AI/ML・AR 等を ver1.0.0 で実装しないことは、品質への妥協ではなく、単一工場・オンプレ・社内 LAN という利用環境の定義に従った意思決定である。

この方針が本書の各章に与える影響は以下のとおりである。

| 章 | 影響 |
|---|---|
| `02_対象範囲とビジネス目標.md` | 対象を単一工場・社内 LAN に限定し、その外への設計は行わない |
| `03_機能スコープと非スコープ.md` | 非スコープを明示し、追加要望を受け付けない意思決定を文書化する |
| `08_品質特性と非機能要件方針.md` | 達成基準を定量的に確定し、「十分によい」の基準を曖昧にしない |
| `10_開発スケジュールと意思決定プロセス.md` | 単一のマイルストーン（ver1.0.0 リリース）に向けた進捗管理を設計する |
| `11_前提・制約・リスク.md` | 個人開発リスク（開発者の疾病・モチベーション）と対応方針を明示する |

---

**本節で確定した方針**
- 個人開発のため phase 構造を採用せず、単一の ver1.0.0 でプロダクションレベル品質を達成する。
- 妥協禁止ポリシーは「やらないことを明確にした上でやることを完全に実装する」ことと定義し、全機能実装との混同を否定する。
- ver1.0.0 の品質達成基準は `08_品質特性と非機能要件方針.md` で定量的に確定する。

## 本書が約束しないこと

本書に基づいて開発するシステムは、ver1.0.0 時点で以下を提供しない。

- ver1.0.0 時点で複数工場・SaaS・マルチテナンシーに対応する
- GMP/FDA 21 CFR Part 11、EU MDR、IATF 16949、AS9100、HACCP への適合
- AI/ML による自動判断・異常検知・自然言語による作業指示生成
- AR・スマートグラス・生体センサ・IoT センサとの直接統合
- 任意のレガシー MES/ERP/PLM とのリアルタイム双方向連携
- 労働者の生産性個別評価のためのレポーティング機能
- 法令・規制の解釈責任の代行（運用責任は導入者に帰属する）
