# 03 機能スコープと非スコープ

本章の責務は、本システムが実装する機能と実装しない機能を「コア/対応/対象外」の三区分で確定することである。本章は規制表記の権威章であり、以降の章はすべて本章の判断を前提とする。AI/ML に関する言及は本章内の対象外判断に限定し、他章では言及しない。

---

## 1. 機能スコープの評価軸と分類

本システムの機能は以下の三区分に分類する。分類の基準はすべて本節で確定し、各節で参照する。

### 1-1. 評価軸

機能の分類は下記の四評価軸に基づく。評価軸に照らして判断が確定した機能は、以後の設計段階で再分類しない。

| 評価軸 | 説明 |
|---|---|
| 価値直結性 | Navigation（EWI/状況認識支援）と Traceability（証拠生成）の二大価値を直接実現するか |
| 規制必達性 | ISO 9001 または ALCOA+ の充足に必要か |
| 開発実現可能性 | 個人開発・ver1.0.0 のスコープで品質保証できるか |
| リスク許容性 | 技術リスク・倫理リスク・規制違反リスクを個人開発として許容できるか |

### 1-2. 三区分の定義

**コア機能**: Navigation × Traceability の二大価値を直接実現する機能。未実装はシステム目的の未達を意味する。

**対応機能**: コア機能を支える必要十分な周辺機能。品質管理・安全・倫理の観点で省略できない機能を含む。規制表記は「準拠する」または「対応する」のいずれかを付与する。

**対象外機能**: 個人開発スコープの超過・規制要件の不該当・技術リスク・倫理リスクの観点から、意図的に除外する機能。対象外と判断する理由を本章に明記する。

図: fig_scope_layers.svg（img/ 配下）を参照。

**本節で確定した方針**
機能分類は「コア/対応/対象外」の三区分のみとし、中間的な曖昧区分は設けない。
評価軸を価値直結性・規制必達性・開発実現可能性・リスク許容性の四軸に確定する。
本章の分類は ver1.0.0 全設計フェーズを通じて権威とする。

## 2. コア機能

コア機能は、Navigation と Traceability の二大価値を直接実現する機能の総体である。コア機能の未実装は本システムの目的未達と同義であり、開発上の優先度は最上位とする。

### 2-1. タブレット APP 側コア機能

**SOP ナビゲーション**
DO-LIST 型ロックステップ実行を基本とする。作業者は前のステップを完了しない限り次のステップへ進めない。Dreyfus 習熟度モデルに基づき、Novice には詳細手順・動画・ピクトグラムをフル表示し、Expert にはチェックポイントのみを表示する習熟度別表示モードを実装する。

**ステップ完了記録**
各ステップの完了に対し、電子サイン・タイムスタンプ・作業者 ID を自動付与して記録する。ALCOA+ の Attributable（誰が）・Contemporaneous（いつ）・Accurate（正確に）の原則を技術的に充足する。

**クリティカルステップの証拠付き確認**
FMEA-RPN 連動でクリティカルフラグが立ったステップでは、証拠（写真撮影・測定値入力・バーコード/QR スキャン）の入力を完了条件とする。証拠なしの完了操作を設計上不可能とする。

**中断・再開支援**
プレースキーパーにより中断時点のステップを自動保存する。再開時には Welcome 画面（直前ステップサマリ・経過時間・警告メッセージ）を必ず表示する。中断ログ（時刻・ステップ番号・作業者 ID）を自動記録する。Post-Completion Error（主目的後の付随ステップ忘却）を構造的に防止するため、再開時はロックステップが未完ステップへ強制復帰する。

**異常時アンドン発報**
ワンタップで応援要請・ライン停止要求を発報する。発報時刻・作業者 ID・対象ステップを証拠として自動記録する。

**ヒヤリハット・不適合・Kaizen Teian の起票**
写真・テキスト・定型タグを組み合わせた起票フォームを提供する。起票時刻・作業者 ID を自動付与する。

**オフライン動作・バックグラウンド同期**
SQLite をローカルストレージとし、通信断時でも全作業ステップの実行・記録を継続する。通信回復後は Outbox Pattern により記録をサーバーに同期する。「最終同期時刻」を全画面ステータスバーに常時表示する。

**バーコード/QR スキャン**
GS1 体系（EAN/UPC/ITF-14/GS1-128）および QR コード・Data Matrix を読み取り、ロット・部品・設備との照合を行う。ML Kit Barcode Scanning を使用する。RFID は対象外とする。

**多言語表示**
日本語・英語を基本言語とし、i18next + expo-localization により SOP 本文・UI ラベルの多言語表示を実現する。他言語（ベトナム語等）はマスタ側で SOP 翻訳データを承認済み状態で登録することにより有効化する。

**シフト別 UI 自動適応**
20:00〜6:00 の時間帯は自動でダークモードに切り替える。シニアモード（フォント 14〜16pt・タッチターゲット 72dp 以上）はプロファイル設定により有効化する。

**子機モード（基幹システム連携）**
外部一意キー（型式・機番・品目コード等の一意識別属性の組み合わせ）による親機（基幹システム: ERP / MES / 生産管理パッケージ等）とのマスタ同期（READ-ONLY 同期）および作業実績の実績送信を行う。詳細は計画 12 章参照。単独動作モードとの切替は設定ファイルで制御する。

### 2-2. 管理 Web 側コア機能

**手順マスタ管理**
SOP の作成・版管理・承認ワークフロー・廃止の一連のライフサイクルを管理する。Effective Date とロット番号を紐付け、特定ロット以降に新版を適用する変更管理を可能にする。版管理と改訂履歴は ISO 9001:2015 §7.5 の文書化情報要件に準拠する。

**ユーザー・スキル・資格マスタ管理**
作業者の認定・有効期限・資格必要工程を管理する。RBAC（役割別アクセス制御）と連動し、資格を持たない作業者が対象工程にアクセスできない制御を実装する。

**設備マスタ・校正期限管理**
設備・計測器の情報と校正期限を管理する。期限超過の計測器を使用した記録は警告フラグを付与する。

**トレサビ照会**
順方向（ロット→工程→完成品）・逆方向（製品→工程→原材料・作業者・設備）のクエリを提供する。ロット別全履歴を一覧表示する。

**不適合・CAPA 管理**
ISO 9001:2015 §10.2 に準拠する。起票・原因分析（4M/Why-Why）・対策・効果確認・クローズの全フェーズを管理する。

**品質ダッシュボード**
管理図・工程能力指数（Cp/Cpk）を表示する。プロセスマイニング用の XES/CSV 形式ログエクスポートを提供する。

**電子帳票出力**
CSV および PDF 形式での帳票出力を提供する。SOP の最新版 PDF 出力は BCP 対策として紙保管フローに接続する。

**本節で確定した方針**
コア機能はタブレット APP 側 11 項目・管理 Web 側 7 項目として確定する。
クリティカルステップの証拠付き確認は「証拠なしの完了を設計上不可能とする」制約を設計必達とする。
オフライン動作は Offline-First アーキテクチャとして確定し、通信断を正常系の一部として設計する。

## 3. 対応機能（コア機能を支える周辺機能）

対応機能はコア機能を支える必要十分な周辺機能である。規制との関係は「準拠する」または「対応する」のいずれかで表記する。「準拠する」は規制要件を技術的・手続き的に充足することを意味し、「対応する」は規制の主旨を満たす実装を行うが認証取得を意味しないことを示す。

| 機能 | 規制 / 根拠 | 対応方針 |
|---|---|---|
| SOP 版管理・改訂履歴・電子承認 | ISO 9001:2015 §7.5 文書化情報 | 準拠する |
| 電子署名（完了サイン・承認サイン） | ISO 9001:2015、ALCOA+ Attributable 原則 | 対応する |
| ALCOA+ 準拠の記録不変性 | ALCOA+ 9 原則（Attributable / Legible / Contemporaneous / Original / Accurate / Complete / Consistent / Enduring / Available） | 準拠する |
| Audit Trail（変更差分レコード） | ISO 9001:2015 §10.1、ALCOA+ Consistent / Enduring 原則 | 対応する |
| 計測器マスタ・校正期限管理 | 計量法（平成 4 年法 51 号）・JCSS（法第 134 条）・GUM（JCGM 100:2008）不確かさ | 対応する |
| FMEA 結果連動の強化チェック | ISO 9001:2015 リスクベース思考・FMEA（RPN = S × O × D） | 対応する |
| 多言語 SOP 表示 | 労働安全衛生法、やさしい日本語 8 原則、ISO 7010 安全標識ピクトグラム | 対応する |
| WCAG 2.1 AA 準拠の UI | ISO 9241-11（ユーザビリティ）・ISO 9241-171（アクセシビリティ） | 対応する |
| 不適合・CAPA 記録 | ISO 9001:2015 §10.2 是正処置 | 準拠する |
| RBAC（役割別アクセス制御） | 個人情報保護法、ALCOA+ Attributable 原則 | 対応する |
| 写真・動画・音声の証拠記録 | ALCOA+ Original / Accurate 原則、非テキスト記録の証拠品質 | 対応する |
| SHA-256 ハッシュ・RFC 3161 タイムスタンプ | ALCOA+ Original / Enduring 原則 | 対応する |
| スキル・資格管理と工程アクセス制御 | ISO 9001:2015 §7.2 力量、労働安全衛生法 | 対応する |
| Kaizen Teian・ヒヤリハット管理 | ISO 9001:2015 §10.3 継続的改善 | 対応する |
| プロセスマイニング用ログエクスポート | ISO 9001:2015 §9.1 監視・測定・分析・評価 | 対応する |
| 順方向・逆方向トレサビ照会 | ISO 9001:2015 §8.5.2 識別とトレーサビリティ | 準拠する |
| I-PASS 構造の引継ぎメモ | 作業継続性・BCP 観点 | 対応する |
| 紙運用フォールバック用 PDF 出力 | BCP・事業継続計画、ISO 9001:2015 §8.1 | 対応する |

計測器マスタ・校正期限管理については「対応する」と判断する。計量法およびJCSSが求める計測のトレーサビリティ要求のうち、計測器マスタへの機器登録・校正期限の管理・期限超過警告を実装する。ただし、JCSS 認定機関の一次標準への直接接続は対象外と判断する（理由は 4 節に記載する）。

**本節で確定した方針**
対応機能は ISO 9001 + ALCOA+ の充足を判断基準とし、18 項目を確定する。
「準拠する」と「対応する」の区別は本節の表で確定し、設計・実装フェーズで再解釈しない。
WCAG 2.1 AA 準拠を対応機能として確定し、アクセシビリティを設計必達とする。

## 4. 対象外と判断する規制・機能

本節に列挙する規制・機能は、本システムのスコープから意図的に除外する。除外は消極的な判断ではなく、スコープ超過による品質低下リスクを回避するための積極的な設計判断である。

| 対象外の規制 / 機能 | 判断理由 |
|---|---|
| GMP / ICH Q10（医薬品品質システム） | 医薬品製造業にのみ適用される規制である。本システムの初期ターゲット業種は医薬品製造業を含まない。対象外と判断する |
| FDA 21 CFR Part 11（電子記録・電子署名・監査証跡） | 米国 FDA への申請を行う医薬品・医療機器製造事業者に適用される米国規制である。本システムの運用環境は社内 LAN オンプレであり、FDA への申請は対象外と判断する |
| EU MDR / IVDR / UDI / EUDAMED | EU 医療機器規制である。適用業種外と判断する |
| IATF 16949:2016（PPAP / APQP） | 自動車 OEM および Tier1 サプライヤー向け規制である。Tier2 / Tier3 事業者の任意対応は将来の拡張余地として留保するが、ver1.0.0 スコープでは対象外と判断する |
| AS9100 Rev D / Nadcap | 航空宇宙品質規制である。適用業種外と判断する |
| HACCP / ISO 22000（食品安全マネジメント） | 食品安全規制であり、ハザード分析機能を必要とする。本システムはハザード分析機能を実装しない。対象外と判断する |
| **AI/ML による自動判断・異常検知・RAG** | ROI 推計の信頼性が低い（McKinsey / WEF の方法論は非開示かつ大企業・装置工業偏向）。ハルシネーションのリスクが高く、特に社内固有知識は学習データの稀少性が高くリスクが大きい。Fine-tuning 用データの著作権処理が未確立である。EU AI Act（2024 年採択、2026 年施行）の High Risk 区分に該当する可能性があり、個人開発での対応コストが不釣合いである。以上の根拠から、対象外と判断する |
| IoT センサ（PLC・振動・電流）の直接取込 | OT-IT 境界を越えた設計が必要となり、個人開発のスコープを超過する。対象外と判断する |
| JCSS 一次標準への直接トレース | 計量法・JCSS の適用範囲で「対応する」と判断しているが、JCSS 認定機関の一次標準との直接接続は認定機関との契約・専用ハードウェアを必要とし、個人開発スコープを超過する。対象外と判断する |
| RFID 読み取り | バーコード / QR スキャンはコア機能として実装するが、RFID 読み取りは追加ハードウェア・プロトコル実装を要する。対象外と判断する |
| 法令・規制の解釈責任の代行 | 本システムは規制要件の充足を技術的に支援するが、規制解釈の責任は導入事業者に帰属する。本システムが法令解釈の代行を行うことは対象外と判断する |

**本節で確定した方針**
AI/ML による自動判断・異常検知・RAG は、ROI 信頼性・ハルシネーション・著作権・EU AI Act の四根拠により対象外と判断する。この判断は本章のみに記載し、他章では言及しない。
IATF 16949 / AS9100 / GMP / FDA 21 CFR Part 11 / EU MDR / HACCP はすべて対象外と判断し、「将来的に対応を検討する」等の曖昧な留保は付けない。
JCSS は「計測器マスタ・校正期限管理まで対応する」「一次標準直結は対象外」という二層構造で確定する。

## 5. SOP マスタのデータモデル方針（本章の権威範囲）

SOP マスタのデータ構造は本節で確定する。データベース設計・API 設計・タブレット APP の表示ロジックはすべて本節の構造を前提とする。

### 5-1. 必須メタデータ

SOP 文書が保持する必須メタデータは以下のとおりである。

| フィールド | 型 | 説明 |
|---|---|---|
| document_no | TEXT | 文書番号（一意識別子） |
| version_no | TEXT | 版番号（例: 1.0.0） |
| effective_date | DATE | 有効開始日 |
| author_id | UUID | 作成者ユーザー ID |
| approver_id | UUID | 承認者ユーザー ID |
| target_process_id | UUID | 対象工程 ID |
| revision_reason | TEXT | 改訂理由 |
| revision_history | JSONB | 改訂履歴（版番号・日時・理由・承認者の配列） |
| status | ENUM | draft / under_review / approved / obsolete |
| external_key_binding | JSONB | 外部一意キー連携属性: 子機モード使用時に親機の外部一意キーと紐づけるための属性フィールド（`external_key_binding` テーブルを通じて管理。単独動作モードでは空でも可） |

Effective Date とロット番号の紐付けにより、特定ロット以降に新版を適用する変更管理を可能とする。この紐付けは ISO 9001:2015 §8.5.2 のトレーサビリティ要件の充足に必要である。

### 5-2. 手順の 3 階層構造

手順は以下の 3 階層で表現する。

```
工程 (Process) → 作業 (Operation) → ステップ (Step)
```

各 Step は以下のフィールドを持つ。

| フィールド | 説明 |
|---|---|
| step_no | ステップ番号（表示順序） |
| instruction | 手順テキスト（多言語 JSONB: `{ "ja": "...", "en": "...", "vi": "..." }`） |
| input_type | 入力種別（check / numeric / photo / scan / signature） |
| judgment_condition | 判定条件（USL / LSL / 正誤 / 選択肢） |
| critical_flag | クリティカルステップフラグ（FMEA-RPN 連動） |
| media_refs | 動画・画像の参照 URL 配列 |

### 5-3. 手順提示モード

SOP の表示は以下の 3 モードを実装する。

| モード | 対象 | 説明 |
|---|---|---|
| Linear | Novice / Competent | ステップを順番に一つずつ表示するロックステップ方式 |
| Decision Tree | 異常対応手順 | 条件分岐を含む手順。判断ノードと処置ノードで構成する |
| Reference | Proficient / Expert | チェックポイントのみを一覧表示。詳細は折り畳みで参照可能 |

習熟度モードの切り替えは、Dreyfus モデルに基づく習熟度プロファイルと改訂直後フラグ・長期ブランクフラグにより自動制御する。熟練作業者であっても改訂直後（新版適用後 N 回完了まで）または最終実施から 30 日以上経過した場合は Linear モードへ強制切り替えする。

**本節で確定した方針**
SOP データモデルは工程→作業→ステップの 3 階層・必須メタデータ 10 フィールドで確定する。
手順提示モードは Linear / Decision Tree / Reference の 3 モードに確定する。
Effective Date とロット番号の紐付けは設計必達要件として確定する。

## 6. 機能境界のグレーゾーンと判定方針

以下の機能は「コア」と「対象外」の境界が不明瞭になりやすいグレーゾーンである。本節で判定方針を確定する。

| 機能 | 判定 | 判定方針 |
|---|---|---|
| 写真・音声記録 | 対応する | 証拠品質管理として SHA-256 ハッシュ・RFC 3161 タイムスタンプを付与する。顔写り込みは個人情報として非公開フォルダへ自動分類する設計指針を適用する |
| バーコード / QR スキャン | コア機能（対応する） | ML Kit カメラベースでコア機能として実装する。RFID は対象外と判断する |
| スキル・資格管理 | 対応する | 認定・有効期限・資格必要工程のアクセス制御まで実装する。資格試験の採点・認定証発行は対象外と判断する |
| Kaizen Teian・ヒヤリハット | 対応する | 起票・管理者応答・ステータス管理まで実装する。Kaizen の採算計算・原価管理との連携は対象外と判断する |
| プロセスマイニング | 対応する | XES / CSV 形式でのエクスポートを対応する。分析ツール（ProM / Celonis 等）の組込みは対象外と判断する |
| 動画再生 | 対応する | SOP の手順説明動画をタブレットで再生する機能を実装する。動画撮影・編集機能は対象外と判断する |
| FMEA の入力・管理 | 対応する（参照のみ） | FMEA の結果（RPN 値・クリティカルフラグ）をマスタとして取り込み、SOP ステップへの連動に使用する。FMEA 自体の入力・計算ツールは対象外と判断する |
| 電子署名の法的効力 | 対応する（技術的実装のみ） | 電子署名の技術的実装（作業者 ID・タイムスタンプ・ハッシュ）を行う。電子署名法上の認定電子署名・電子証明書の取得は対象外と判断する |
| 単独動作モードと子機モードの切替判断 | 条件付き対応 | 子機モードへの移行要求は「工場が親機（基幹システム: ERP / MES / 生産管理パッケージ等）を保有しているか」かつ「親機が標準 REST API を公開しているか」の 2 条件を満たす場合に対応する。いずれか一方でも満たさない場合は単独動作モードのみをサポートする |

**本節で確定した方針**
グレーゾーン機能 9 項目の判定方針を本節で確定し、設計フェーズでの再判断を不要とする。
「部分的に対応する」と判断した機能は、対応範囲の上限を本節の表で明示する。
本節の判定は設計・実装・テスト全フェーズで参照する権威判断とする。
