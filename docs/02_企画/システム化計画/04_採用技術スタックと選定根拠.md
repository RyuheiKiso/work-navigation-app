# 04_採用技術スタックと選定根拠

本書は各レイヤの技術選定とその根拠を示す。
技術スタックは `README.md` (リポジトリルート) ですでに決定済みであり、
本書はその根拠と代替案の評価を補足する。

---

## 1. 選定原則

本システムの技術選定は以下の 4 原則に従う。

| 原則 | 内容 |
|---|---|
| **個人開発の保守性** | 1 人のエンジニアが 5 年以上保守できる技術選定。言語・フレームワーク・ライブラリの複雑さを最小化する |
| **オンプレ親和性** | Windows Server 2022 + IIS + WSL2 + Docker Compose で動作すること。クラウド専用機能に依存しない |
| **型安全と長期信頼性** | 型システムが強力な言語 (Rust / TypeScript) を優先し、ランタイムエラーを設計段階で排除する |
| **Buy vs Build の内製判断** | 既製 MES/eBR 製品が要件を 70% 未満しか満たさないため内製を選択 ([`90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md`](../../90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md)) |

### 1.1 Buy vs Build の判断

国内 MES 市場は 2023 年時点で 170 億ドル規模・CAGR 10%+ であり
([`90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md`](../../90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md))、
Plex / DELMIA Apriso / Siemens Opcenter / SAP DMC などの既製品が存在する。
これらが本要件 (社内 LAN 閉域 + Windows Server 2022 + Rust バックエンド + React Native タブレット) に
対応するためのカスタマイズコストと、Panorama Consulting が報告する **54% の予算超過・68% の工期超過**
(同上) を考慮し、内製を選択した。

既製品を採用しない理由を明示する:

| 候補 | 不採用理由 |
|---|---|
| SwipeGuide / Dozuki | SaaS のみ。社内 LAN 閉域に対応不可 |
| Tulip | SaaS のみ。オンプレ構成は Enterprise 契約が必要 |
| COLMINA / NEC Lumada | オンプレ対応だが SI を介した大規模導入前提。個人開発スケールに不適 |
| ERP 組込みモジュール | 既存 ERP なし。ERP 全体導入は過剰投資 |

---

> **本節で確定した方針:** 既製品要件適合率 70% 未満を確認したうえで内製を選択する。将来的に既製品の機能拡充や要件変化があれば Buy vs Build を再評価する。

## 2. タブレットアプリ層: React Native (Expo) + TypeScript + SQLite

### 2.1 採用根拠

| 観点 | 評価 |
|---|---|
| クロスプラットフォーム | Android / iOS / Windows (react-native-windows) を単一コードベースで対応 |
| TypeScript による型安全 | 実績入力・測定値・QR スキャン値の型定義で誤データ混入を防ぐ |
| SQLite (expo-sqlite) | Local-First のオフライン同期に必要なローカル永続化を標準提供 |
| エコシステム | QR スキャン (expo-barcode-scanner / Google ML Kit)・カメラ・i18next など工場アプリに必要な OSS が揃っている |
| Expo の開発効率 | OTA アップデート・EAS Build で端末へのデプロイが簡略化される |

### 2.2 代替案と落選理由

| 代替案 | 落選理由 |
|---|---|
| Flutter (Dart) | Windows タブレット対応が React Native Windows より成熟度で劣る。Dart エコシステムの製造業向け OSS が少ない |
| ネイティブ開発 (Kotlin / Swift / WinUI) | 3 プラットフォームで個別実装が必要。個人開発の保守コスト上限を超える |
| PWA (Progressive Web App) | カメラ・バーコードスキャン・ローカルファイルへのフルアクセスが制限される。オフライン同期の制御が困難 |
| Cordova / Ionic | 技術的負債が大きく、2025 年以降のアクティブ開発が縮小傾向 |

### 2.3 ローカル DB: SQLite (expo-sqlite)

Append-only の Outbox テーブルと作業進捗テーブルを SQLite に持つ。
expo-sqlite は Expo 管理のネイティブバインディングを提供し、
SQLite FTS5 の全文検索や WAL モードが利用可能。

TypeORM と expo-sqlite の組み合わせ (`typeorm` + `expo-sqlite` ドライバ) を採用し、
マイグレーション管理をコードベースで行う。

---

> **本節で確定した方針:** React Native (Expo) + TypeScript + expo-sqlite を採用する。Flutter / PWA / ネイティブは上記理由で採用しない。TypeORM でマイグレーションを管理する。

## 3. バックエンド層: Rust (Edition 2024) + tokio + axum

### 3.1 採用根拠

| 観点 | 評価 |
|---|---|
| メモリ安全性 | 所有権システムにより、メモリ関連のランタイムバグを型検査で排除。長期保守での安全性 |
| 並行性 | tokio の非同期ランタイムが多数の同時接続 (製造ライン複数タブレット) を低リソースで処理 |
| 型安全 | 強力な型システムで API の入出力を静的検証。serialize/deserialize の serde が実績豊富 |
| パフォーマンス | C 相当の実行速度。オンプレサーバの性能制約下でも余裕がある |
| axum の設計 | Tower ミドルウェアエコシステムとの親和性。認証・ロギング・レートリミットをミドルウェアで管理 |

### 3.2 代替案と落選理由

| 代替案 | 落選理由 |
|---|---|
| Node.js (Express / Fastify) | 動的型付け言語の特性上、ランタイムエラーが設計段階で検出されにくい。長期保守での信頼性が Rust より低い |
| Go (gin / echo) | 型安全性・メモリ安全性で Rust が優位。Null ポインタランタイムエラーが発生しうる |
| .NET (ASP.NET Core) | Windows 親和性は高いが、WSL2 + Docker 上での Rust に対する追加メリットが少ない |
| Python (FastAPI) | 動的型付けのリスクと GIL によるスレッド並行性の制限 |

### 3.3 採用しないパターン

- マイクロサービス分割: 個人開発では運用コストが過大。モノリス API で開始し、必要に応じて分割する
- GraphQL: REST で十分。スキーマ複雑性が個人開発に不相応
- gRPC: タブレット (React Native) からのブラウザ互換性に制約

### 3.4 DB アクセス: sqlx

`sqlx` (非同期・マクロによるコンパイル時 SQL 検証) を使用する。ORM は採用しない。
理由: SQL を直接書くことで複雑なトレサビクエリの最適化が容易であり、
N+1 クエリなどの ORM 起因のパフォーマンス問題を回避できる。

---

> **本節で確定した方針:** Rust 2024 + tokio + axum + sqlx を採用する。マイクロサービス・GraphQL・gRPC は採用しない。sqlx のコンパイル時 SQL 検証でクエリエラーを設計段階で検出する。

## 4. Web 管理 SPA 層: React + Vite + TypeScript

### 4.1 採用根拠

| 観点 | 評価 |
|---|---|
| エコシステム | 管理画面に必要なテーブル・フォーム・グラフの OSS が充実 |
| TypeScript 共通 | タブレットアプリと型定義の一部を共有でき、一貫性を保ちやすい |
| Vite | 高速なビルドと HMR (Hot Module Replacement)。個人開発の開発サイクルを短縮 |
| 静的ビルド | `vite build` で生成した静的ファイルを Rust サーバが配信。別途 Web サーバ不要 |

### 4.2 採用しないフレームワーク

- Next.js / Remix: SSR が不要 (社内 LAN の SPA で十分)。ビルド複雑度が高い
- Vue / Svelte: React との技術スタック統一で学習コスト・保守コストを最小化

---

> **本節で確定した方針:** React + Vite + TypeScript を採用する。SSR フレームワークは不要と判断する。Vite ビルド成果物を Rust サーバが静的配信する。

---

## 5. データベース: PostgreSQL

### 5.1 採用根拠

| 観点 | 評価 |
|---|---|
| ACID 準拠 | トレサビ記録・実績記録の完全性保証 |
| JSON/JSONB サポート | 測定値・非テキスト記録のメタデータを柔軟に格納 |
| WAL (Write-Ahead Log) | Point-in-Time Recovery (PITR) で RPO 15 分を達成 ([`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md)) |
| pgvector 拡張 | Phase 2 以降の RAG (Retrieval-Augmented Generation) 実装に備えたベクトル検索 ([`90_業界分析/36_AI・ML支援と知的作業ナビゲーション.md`](../../90_業界分析/36_AI・ML支援と知的作業ナビゲーション.md)) |
| Row-Level Security | ロール別のデータアクセス制御を DB レイヤで保証できる (将来検討) |
| Docker Compose との親和性 | 公式 `postgres` イメージで即座にコンテナ化可能 |

### 5.2 採用しない DB

| 候補 | 不採用理由 |
|---|---|
| MySQL | pgvector が不可。JSONB の機能が劣る |
| SQLite (サーバ側) | 並行書き込みのスケーラビリティ不足。WAL PITR 機能がない |
| MongoDB | ACID 準拠が複雑。スキーマレスは規制記録の完全性保証に不適 |
| クラウド DB (Aurora / Cloud SQL) | 社内 LAN 閉域前提に反する |

---

> **本節で確定した方針:** PostgreSQL を採用する。pgvector 拡張を Phase 2 以降で有効化する前提でインストールしておく。WAL アーカイブを日次バックアップと組み合わせて RPO 15 分を確保する。

## 6. デプロイ・インフラ: IIS + Windows Server 2022 + WSL2 + Docker Compose

### 6.1 Windows Server 2022 採用の合理性

社内標準インフラが Windows Server であるため変更不可。
Rust / Docker は WSL2 (Ubuntu 22.04) 上で動作するため、Windows 環境でも本番品質の Linux コンテナを運用できる。

### 6.2 IIS の役割

| 役割 | 内容 |
|---|---|
| リバースプロキシ | `URL Rewrite` + `ARR (Application Request Routing)` モジュールで WSL2 内の axum に転送 |
| TLS 終端 | IIS が SSL 証明書を管理し、バックエンドへは HTTP で転送 |
| 静的ファイル | React SPA の配信。または axum が直接配信する方式と比較して要件定義フェーズで決定 |

### 6.3 採用しないインフラ構成

| 候補 | 不採用理由 |
|---|---|
| Kubernetes / k8s | 単一サーバ・個人開発規模で過剰。運用コストが過大 |
| クラウド (AWS / Azure / GCP) | 社内 LAN 閉域前提。クラウドへのデータ転送は非適用 |
| Nginx on WSL2 (IIS 置換) | IIS が社内標準。Nginx に置換すると社内承認が必要 |
| Podman / containerd | Docker Compose のエコシステムが成熟しており代替メリットがない |

---

> **本節で確定した方針:** IIS + WSL2 + Docker Compose 2 コンテナ (axum + postgres) を採用する。k8s・クラウドは対象外とする。IIS の ARR モジュールをリバースプロキシとして使用する。

---

## 7. 採用しない技術カテゴリの総まとめ

以下の技術カテゴリは本システムのスコープでは採用しない。

| カテゴリ | 不採用判断 | 理由 |
|---|---|---|
| SaaS 型 MES / eBR 製品 | 対象外と判断する | 社内 LAN 閉域・オンプレ前提に合わない |
| クラウドインフラ | 対象外と判断する | 社内 LAN 閉域に反する |
| Kubernetes | 対象外と判断する | 個人開発・単一サーバで過剰 |
| UHF RFID | 対象外と判断する | 金属環境での信頼性と導入コスト |
| AI (Phase 1) | 対象外と判断する | Phase 3 まで段階導入。ハルシネーションリスク |
| GraphQL / gRPC | 採用しない | REST で十分。複雑性が個人開発に不相応 |
| ORM (バックエンド) | 採用しない | sqlx の直接 SQL でパフォーマンスと型安全を両立 |
| 生体認証 | 採用しない | プライバシーリスク・法的要件の不確実性 |
