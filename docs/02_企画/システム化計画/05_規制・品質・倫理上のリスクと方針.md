# 05_規制・品質・倫理上のリスクと方針

本書は本システムが直面する規制・品質・倫理上のリスクを整理し、
各リスクへの対応方針を「準拠する」「対応する (部分準拠)」「対象外と判断する」の 3 段階で明示する。
規制への曖昧な「考慮する」表現は使用しない。

---

## 1. 規制・品質規格マトリクス

本システムが対象とする製造現場が関係しうる主要規制・規格と、
本システムの対応方針を以下に示す。

| 規制 / 規格 | 主な要求 | 中小単一工場での適用可能性 | 本システムの対応方針 |
|---|---|---|---|
| ISO 9001:2015 §7.5 | 文書化した情報の管理・保持 | 適用あり (ISO 認証取得を問わず) | **準拠する** — Append-only 記録・版管理・改訂履歴で対応 |
| ISO 9000:2015 トレサビ定義 | 前後方向の追跡可能性 | 適用あり | **準拠する** — 双方向トレサビ検索機能で対応 |
| IATF 16949 §8.5.2 | 製造記録・検査記録の管理、10 年以上保存 | 自動車部品メーカーに適用 | **対応する** — 記録保存期間設定は管理者が設定。10 年保存の技術的障壁はない |
| AS9100 Rev D | 航空宇宙の製造記録・トレサビ・Nadcap | 航空宇宙サプライヤに適用 | **対応する** — 記録の完全性設計で対応。Nadcap 固有要件は要件定義で確認 |
| FDA 21 CFR Part 11 | 電子記録・電子署名の真正性・Audit Trail | 医薬品・医療機器メーカーに適用 | **対応する (コアベースライン設計)** — Audit Trail / Append-only / Idempotency Key で対応。電子署名 (2 要素認証) は Phase 2 |
| EU MDR (2017/745) / UDI | 医療機器の一意識別子・トレサビ | 医療機器メーカーに適用 | **対応する** — UDI スキャン入力対応。EU 規制への完全準拠は適用顧客要件で別途評価 |
| GMP / PIC/S | 製造記録・変更管理・SOP の版管理 | 医薬品製造業に適用 | **対応する** — SOP 版管理・承認フロー (Phase 2) で対応 |
| HACCP (食品衛生法改正 2021) | 危害分析・重要管理点の記録 | 食品製造業に適用。2021 年 6 月完全施行 | **対応する** — CCP 記録項目はマスタで定義可能 |
| FSMA 204 (2023 最終規則) | 24h トレースバック要件。2026 年 1 月発効 | 米国向け食品製造・輸出業者に適用 | **対応する** — 本システムのトレサビ記録設計はこの要件を満たす粒度を提供 |
| 計量法 (平成 4 年 51 号) | 計量器の定期校正・記録 | 国内製造業全般に適用 | **対応する** — 校正参照・不確かさ記録フィールドをマスタに設ける (§3 参照) |
| JCSS (ISO/IEC 17025) | 計量トレーサビリティ証明 | JCSS 認定が必要な測定を行う場合 | **対応する** — JCSS 校正番号をマスタで管理可能にする |
| EU AI Act (2024 採択、2026 段階適用) | 高リスク AI の文書化・人的監視・ログ | AI を使用する場合に適用 | **対象外と判断する (Phase 1)** — Phase 1 は AI 未使用。Phase 3 導入時に再評価 |
| ISO/IEC 42001:2023 | AI マネジメントシステム | AI を使用する場合に適用 | **対象外と判断する (Phase 1)** — Phase 3 時に再評価 |
| 改正個人情報保護法 (2022 年 4 月施行) | 個人情報の目的外利用禁止・漏洩通知 | 国内全企業に適用 | **準拠する** — 作業員の個人情報取扱方針を設計に組込む (§5 参照) |

> 引用: [`90_業界分析/22_規制別トレーサビリティ要件詳論.md`](../../90_業界分析/22_規制別トレーサビリティ要件詳論.md)

---

> **本節で確定した方針:** 全規格への完全準拠は個別業界・顧客要件に依存するため断言しない。ただし ISO 9001 / ALCOA+ / 計量法 / 個情法は全導入先共通の基盤として「準拠する」設計とする。Phase 1 は AI を使用しないためAI 規制は対象外と判断する。

## 2. ALCOA+ 適合方針

電子記録の完全性評価基準として ALCOA+ の 9 原則を採用する
([`90_業界分析/22_規制別トレーサビリティ要件詳論.md`](../../90_業界分析/22_規制別トレーサビリティ要件詳論.md))。

| 原則 | 英語名 | 本システムの設計方針 |
|---|---|---|
| 帰属性 | Attributable | 全記録に作業員 ID・タイムスタンプを付与。JWT 認証で帰属を保証 |
| 判読性 | Legible | 電子入力のため判読性は原理的に保証。手書き入力は禁止 |
| 同時性 | Contemporaneous | ステップ完了と同時に記録 (完了後のまとめ入力を UI で防ぐ)。オフライン時はローカル記録 → 復帰後同期 |
| 独自性 | Original | DB への直接書き込みをアプリ層が管理。Idempotency Key で重複防止 |
| 正確性 | Accurate | 入力値の範囲バリデーション・単位付与・不確かさ記録で精度を保証 |
| 完全性 | Complete | 全ステップの記録が揃わないと工程完了にならない強制構造 |
| 一貫性 | Consistent | サーバ側のクロックを権威ソースとし、タブレット側クロックに依存しない |
| 永続性 | Enduring | Append-only ログ。論理削除のみ許可し物理削除は禁止 |
| 利用可能性 | Available | バックアップ (WAL PITR + Docker volume バックアップ) で可用性を確保 |

**ALCOA+ の「同時性」とオフライン同期の緊張**

オフライン環境での作業記録は「記録は作業と同時」という ALCOA+ 要件を満たしているが、
サーバへの送信がオフライン解除まで遅延する。この緊張は以下の方針で解消する:
- ローカルタイムスタンプ (作業員の操作時刻) とサーバ受信時刻を **別フィールドで両方記録**
- 同期遅延が発生した場合は Audit Trail に理由 (offline_sync) を記録
- ALCOA+ の要求は「記録の作成が同時」であり「サーバへの転送が同時」ではないと解釈する
([`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md))

---

> **本節で確定した方針:** ALCOA+ 9 原則全てを設計方針として採用する。オフライン記録はローカルタイムスタンプを真の記録時刻とし、サーバ受信時刻と双方を保存する。物理削除はいかなるロールも実行できない設計とする。

## 3. 計量トレーサビリティ方針

製造現場での測定値記録は計量法 (平成 4 年法律第 51 号) および
JCSS (ISO/IEC 17025:2017) に基づく校正トレーサビリティを維持する必要がある
([`90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md`](../../90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md))。

### 3.1 測定値記録の設計方針

本システムの測定値フィールドは以下の 5 要素をセットで保存する:

| フィールド | 内容 | 例 |
|---|---|---|
| `value` | 測定値 (数値) | `25.30` |
| `unit_code` | UCUM 単位コード | `Cel` (摂氏) |
| `uncertainty` | 拡張不確かさ (k=2, 95%) | `0.05` |
| `instrument_id` | 使用測定器 ID (マスタ参照) | `THERMOMETER-007` |
| `calibration_ref` | 校正証明書番号 (JCSS 番号) | `JCSS-2024-001` |

Mars Climate Orbiter 事故 (単位変換ミスによる 1999 年機体喪失) を教訓として、
単位の明示的記録は数値記録と同等の重要性を持つ
([`90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md`](../../90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md))。

### 3.2 Phase 別対応

| フェーズ | 対応内容 |
|---|---|
| Phase 1 | 測定値 + UCUM 単位コード + 機器 ID の記録。不確かさフィールドは任意入力 |
| Phase 2 | 不確かさ必須化・JCSS 校正期限アラート |
| Phase 3 | 測定器との自動連携・統計的工程管理 (SPC) の基盤 |

---

> **本節で確定した方針:** Phase 1 から測定値・単位 (UCUM)・機器 ID の 3 点セットを必須とする。不確かさ・JCSS 番号は Phase 1 では任意入力とし、Phase 2 で必須化する。

## 4. 非テキスト記録の証拠品質方針

写真・動画など非テキスト記録の改ざん防止と長期保存に関する方針を示す
([`90_業界分析/31_非テキスト記録と証拠品質管理.md`](../../90_業界分析/31_非テキスト記録と証拠品質管理.md))。

| 対策 | Phase 1 | Phase 2/3 |
|---|---|---|
| ファイルハッシュ (SHA-256) | **実装する** — 撮影直後にサーバ側で計算・保存 | 変更なし |
| EXIF メタデータ保護 | **実装する** — EXIF を剥離せず記録 DB に保存 | 変更なし |
| タイムスタンプ局 (TSA/RFC 3161) | **対象外** — コストと運用負荷が個人開発に不相応 | 将来検討 |
| 長期保存形式 (JPEG2000 / PDF/A-3) | **対象外** — 標準 JPEG/PNG で保存 | 必要に応じ評価 |
| Chain of Custody ログ | **実装する** — ファイルへのアクセスを Audit Trail に記録 | 変更なし |

---

> **本節で確定した方針:** Phase 1 では SHA-256 ハッシュと EXIF 保存を実装する。TSA は個人開発規模での運用コストが高いため Phase 3 以降の将来検討とする。

## 5. 個人情報保護と労務監視の倫理方針

作業員の行動ログ収集は、労務監視と評価されるリスクを含む
([`90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md`](../../90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md))。

### 5.1 収集するデータと目的

| データ種別 | 収集目的 | 保存方針 |
|---|---|---|
| 作業者 ID + 完了時刻 | トレサビ記録・ALCOA+ 帰属性 | 規制保存期間まで保持 |
| ステップ完了タイムスタンプ | 作業記録の同時性証明 | 同上 |
| 入力値・スキャン値 | 品質記録 | 同上 |
| 写真 | 作業証拠 | 同上 |

**収集しないデータ:**
- 生体情報 (顔認識・指紋)
- 位置情報 (GPS)
- 作業速度の個人別ランキング・リアルタイム進捗の上司への自動通知
- 休憩・離席の記録

### 5.2 改正個人情報保護法 (2022 年 4 月施行) への対応

| 要件 | 対応方針 |
|---|---|
| 利用目的の特定 | 「製造工程のトレーサビリティ記録」として明示 |
| 目的外利用の禁止 | 人事評価・給与査定への利用を禁止。システム設計でデータ分離 |
| 安全管理措置 | 社内 LAN 閉域・JWT 認証・ロールベースアクセス |
| 漏洩通知義務 | 発覚後 72 時間以内に個情委へ通知 (社内規程で対応) |

### 5.3 労務監視リスクへの倫理的ガードレール

Foucault のパノプティコン効果 (監視されていると感じるだけで行動が変化) を考慮し、
監視ツールではなく「作業員を支援するツール」としての位置づけを設計・運用・教育で徹底する
([`90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md`](../../90_業界分析/24_作業者プライバシー・データ倫理と労務監視.md))。

- 管理者が閲覧できる実績データの範囲と目的を就業規則・利用規約に明記する
- 個人別作業速度の自動通知機能は実装しない
- GDPR Article 22 の自動意思決定条項への配慮として、
  人事・処遇に影響するシステム側の自動判定を行わない

---

> **本節で確定した方針:** 収集する個人情報は「作業者 ID + 作業実績」に限定する。生体情報・位置情報・速度ランキングは収集しない。データは製造トレサビ目的のみに使用し、人事評価への転用を設計で防ぐ。

## 6. AI/ML 利用時の方針 (Phase 3 先行宣言)

Phase 1 は AI を使用しないため EU AI Act / ISO/IEC 42001 の適用は受けない。
Phase 3 での AI 機能追加時に本節の方針を再評価する。

### 6.1 段階導入計画

| フェーズ | AI 機能 | リスクレベル (EU AI Act) |
|---|---|---|
| Phase 1 | AI なし (ルールベースのみ) | 対象外 |
| Phase 2 | pgvector を用いた手順 Q&A (RAG) | 限定リスク — 透明性義務のみ |
| Phase 3 | 画像分類 (TFLite / ONNX) による外観検査補助 | 要評価 — 製品安全への影響による |

([`90_業界分析/36_AI・ML支援と知的作業ナビゲーション.md`](../../90_業界分析/36_AI・ML支援と知的作業ナビゲーション.md))

### 6.2 ハルシネーション回避原則

AI が生成した手順指示・安全上の判断・測定値を作業員に提示する際は、
**必ず人間によるレビューを経た情報のみ** を正式な手順書に登録する。
AI の出力を直接 SOP に反映する「ゼロクリック更新」機能は実装しない。

---

> **本節で確定した方針:** Phase 1・2 は AI を使用しないか限定リスクの RAG のみ。Phase 3 の画像 AI は EU AI Act 分類に応じたリスク評価を実施してから実装する。ハルシネーション由来の手順汚染を防ぐため、AI 出力の SOP への自動反映は全フェーズで禁止する。

## 7. 受容するリスクと回避するリスクの整理 (経営層向けサマリ)

| リスク | 本システムの判断 | 根拠・理由 |
|---|---|---|
| Part 11 完全準拠しない状態での医薬品製造適用 | **回避する** — Part 11 要件を満たすまで医薬品製造用途に提供しない | 規制違反はリコール・査察指摘に直結 |
| オフライン時のデータ欠損 | **受容する (設計で最小化)** — ローカル保存でリスク低減。完全な欠損は防止できない | 個人開発・社内 LAN 前提でのコスト対効果 |
| ALCOA+ 「同時性」のオフライン緊張 | **受容する (記録で透明化)** — Audit Trail に遅延理由を記録することで監査対応 | 実用的代替策がない |
| TSA なしでの写真証拠 | **受容する (Phase 3 で再評価)** — SHA-256 ハッシュで改ざん検知は可能 | TSA コストと個人開発規模の不均衡 |
| AI のハルシネーション | **回避する** — Phase 3 まで推論 AI を使用しない | 作業員の安全と製品品質への影響が大きい |
| 生体情報収集によるプライバシー侵害 | **回避する** — 生体認証は実装しない | 法的リスクと従業員信頼の毀損 |
| ATEX 防爆区域への適用 | **回避する** — 明示的に対象外とする | 法令違反・爆発事故のリスク |
| AI Act 高リスク AI による自動意思決定 | **回避する** — 全フェーズで自動意思決定を行わない | 人的監視の維持が設計原則 |
