# 18 拡張可能 Step エンジン（アドオン機構）

本章は、計画 02 章 6-3 節「プラグイン構造を採用する」という宣言を、製造現場の具体的なニーズに応える形で実体化する。標準の作業 Step（合否判定・数値入力・写真撮影等）を超えた「ライン固有の概念・条件分岐・動的フロー」を、ノーコードの JSON DSL でアドオンとして組み込める設計を確定する。

---

## 1. 本章の位置づけと前提

### 1-1. なぜ拡張 Step エンジンが必要か

計画 02 章は作業 Step を「合否判定・数値入力・写真撮影等が紐付く一操作単位」として定義した。この定義は製造業の大多数の現場で有効だが、以下のようなライン固有の要求を標準機能だけでは満たせない：

- **複合評価**: 色・光沢・表面状態を色見本番号で比較する Step（カラーチャート入力）
- **条件による手順変化**: 測定値が許容範囲外の場合のみ追加 Step を実行する（条件分岐）
- **スキャン照合**: 部品バーコードをスキャンして品番が一致するか確認する Step
- **段階承認**: 複数の確認項目を 1 Step に束ねて一括チェックする（複合チェック）

これらは個々の現場固有の要求であり、コアの SOP 設計を変えずにアドオンとして提供する設計が最も保守性が高い。

### 1-2. 設計の基本思想

本機能は「**定義データのみをサーバーに格納し、実行ロジックはクライアントランタイムが解釈する**」アーキテクチャを採用する（計画 05 章 §5「Step エンジンのプラガビリティ」参照）。これにより：

- **Offline-First の維持**: 定義 JSON をタブレット側 SQLite にキャッシュすることで、オフライン時もカスタム Step が動作する
- **サンドボックス保証**: eval / Function constructor を禁止し、JSON Logic ベースの宣言的ロジックのみを許容することで、任意コード実行を構造的に防止する
- **ALCOA+ の維持**: カスタム Step タイプでも、入力値・タイムスタンプ・実施者 ID の記録は標準 Step と同等の形式で Append-only イベントストアに格納する

---

**本節で確定した方針**
- 本章は計画 02 章 6-3 節「プラグイン構造」の現場向け実体化であることを確定する。
- 「定義データのみ格納・実行はランタイム解釈」アーキテクチャを採用することを確定する。
- eval 禁止・JSON Logic ベースの宣言的ロジックのみを許容することを確定する。
- カスタム Step でも ALCOA+ の Attributable・Contemporaneous・Enduring は維持することを確定する。

---

## 2. 標準 Step タイプと拡張 Step タイプの関係

### 2-1. 標準 Step タイプ（既存・変更なし）

計画 02 章 L90 で確定した標準タイプを維持する：

| タイプ | 説明 |
|---|---|
| `boolean_check` | 合否判定（OK / NG） |
| `numeric_input` | 数値入力（計画 06 章 Cp/Cpk と連動） |
| `photo_capture` | 写真撮影（ALCOA+ の Accurate 原則と連動） |
| `text_input` | テキスト自由記入 |

### 2-2. 拡張 Step タイプ（新規）

`step_type_definition` テーブル（計画 06 章参照）に JSON スキーマとして登録することで、工場またはライン単位で独自 Step タイプを追加できる：

| タイプ例 | 説明 | 典型ユースケース |
|---|---|---|
| `slider_range` | 下限/上限/目標値を定義した範囲入力スライダー | トルク・荷重の許容範囲内入力 |
| `composite_check` | 複数確認項目を 1 Step に束ねた一括チェック | 外観検査（キズ/汚れ/色ムラ）の同時確認 |
| `color_chart` | 色見本インデックスとの比較入力（1〜N 番） | 塗装・成型品の色調確認 |
| `signature` | 手書きサイン/タップ確認による担当者署名 | 検査完了の本人確認 |
| `custom_list` | 工場固有の選択肢セットからの択一 | 設備状態・仕向先・ロット種別の選択 |
| `barcode_scan` | QR/バーコードスキャン入力（ML Kit 使用） | 部品番号照合・ロットラベル読取 |

### 2-3. コアと拡張の不変条件

- 標準タイプはアドオンによって変更・削除できない
- 拡張タイプが削除された場合、そのタイプを参照する既存 SOP の Step は `fallback_type` として `text_input` で縮退動作する
- 拡張タイプの削除は過去の記録を消去しない（Append-only 原則）

---

**本節で確定した方針**
- 標準 4 タイプを既存設計として維持し、変更不可とすることを確定する。
- 拡張 Step タイプを `step_type_definition` テーブルへの JSON 登録で追加できる設計を確定する。
- 拡張タイプ削除時は `text_input` への縮退動作を保証することを確定する。

## 3. カスタム Step タイプ定義スキーマ

### 3-1. スキーマ構造

```json
{
  "name": "color_chart",
  "version": "1.0.0",
  "label": {
    "ja": "色見本番号入力",
    "en": "Color Chart Input"
  },
  "input_field": {
    "type": "integer",
    "min": 1,
    "max": 20,
    "ui_hint": "number_picker"
  },
  "validators": [
    {
      "rule": { ">=" : [{"var": "value"}, 1] },
      "message": { "ja": "1 番以上を選択してください", "en": "Select 1 or higher" }
    },
    {
      "rule": { "<=": [{"var": "value"}, 20] },
      "message": { "ja": "20 番以下を選択してください", "en": "Select 20 or lower" }
    }
  ],
  "alcoa_attribution": true
}
```

### 3-2. スキーマのフィールド定義

| フィールド | 必須 | 説明 |
|---|---|---|
| `name` | ○ | スネークケースの一意識別子。`factory_id` と組み合わせてグローバル一意を保証 |
| `version` | ○ | SemVer 形式。定義変更時は version を更新し旧定義は保持する |
| `label` | ○ | 多言語ラベル。`ja` キー必須。他言語はオプション |
| `input_field` | ○ | 入力値の型定義（type / min / max / ui_hint 等） |
| `validators` | △ | JSON Logic ベースのバリデーションルール。省略時はバリデーションなし |
| `alcoa_attribution` | ○ | `true` 固定。将来の監査ツールが確認する |

### 3-3. JSON Logic について

バリデーションロジックは [JSON Logic](https://jsonlogic.com/)（Apache 2.0）の仕様に準拠する。対応する演算子:
- 比較: `==` / `!=` / `>` / `>=` / `<` / `<=`
- 論理: `and` / `or` / `!`
- 変数参照: `{"var": "value"}` で入力値を参照

禁止: `eval` / 関数呼び出し / 外部 API 参照 / 副作用のある操作

---

**本節で確定した方針**
- カスタム Step タイプの定義スキーマを確定する（name / version / label / input_field / validators / alcoa_attribution）。
- バリデーションロジックは JSON Logic 仕様に準拠し、eval 禁止を設計的に保証することを確定する。
- `alcoa_attribution: true` を必須フィールドとして監査ツールの確認対象とすることを確定する。

---

## 4. 条件分岐フロー定義スキーマ

### 4-1. 条件分岐の概念

SOP の Step 実行順序を、前の Step の記録結果に応じて動的に変更できる：

| アクション | 動作 |
|---|---|
| `skip` | 指定 Step をスキップする |
| `goto` | 指定 Step に直接移動する（中間 Step を飛ばす） |
| `insert` | 指定 Step の前後に追加 Step を挿入する |

### 4-2. 条件式スキーマ

```json
{
  "sop_id": "uuid-of-sop",
  "source_step_id": "step_003",
  "condition": {
    ">=" : [{"var": "value"}, 10.5]
  },
  "action": "insert",
  "target_step_id": "step_extra_remediation",
  "insert_position": "after_source",
  "priority": 1
}
```

### 4-3. フィールド定義

| フィールド | 必須 | 説明 |
|---|---|---|
| `sop_id` | ○ | ルールが属する SOP の ID |
| `source_step_id` | ○ | 評価対象 Step の ID |
| `condition` | ○ | JSON Logic の条件式。`{"var": "value"}` で source Step の記録値を参照 |
| `action` | ○ | `skip` / `goto` / `insert` のいずれか |
| `target_step_id` | ○ | アクション対象 Step の ID |
| `insert_position` | △ | `insert` 時のみ: `before_target` / `after_source` |
| `priority` | ○ | 同一 source_step に複数ルールがある場合の優先順位（小さいほど優先） |

### 4-4. 循環参照の防止

`step_flow_rule` 登録時に、DAG（有向非巡回グラフ）検証を API サーバー側で実行する。循環参照が検出された場合はエラーとして登録を拒否する。

---

**本節で確定した方針**
- 条件分岐のアクション種別（skip / goto / insert）を確定する。
- 条件式は JSON Logic 準拠とし、source Step の記録値のみ参照可能な設計を確定する。
- DAG 検証による循環参照防止を API サーバー側の必須バリデーションとして確定する。

---

## 5. ALCOA+ との整合

### 5-1. カスタム Step の記録設計

カスタム Step タイプによる入力は、標準 Step と同一の Append-only イベントストア（`work_event` テーブル、計画 06 章参照）に格納する：

| フィールド | 内容 |
|---|---|
| `event_type` | `custom_step_completed` |
| `step_id` | 実行した Step の ID |
| `step_type_name` | カスタム Step タイプ名（例: `color_chart`） |
| `value` | 入力値（JSONB 形式） |
| `validation_result` | `pass` / `warn_overridden`（バリデーション警告を承認した場合） |
| `performed_by` | 作業員 ID（Attributable） |
| `performed_at` | 端末時刻 + サーバー受信時刻（Contemporaneous） |

### 5-2. 条件分岐スキップ時の記録

Step がスキップされた場合、「スキップされた」事実を記録の欠損として扱わず、Append-only で記録する：

| フィールド | 内容 |
|---|---|
| `event_type` | `step_skipped_by_rule` |
| `step_id` | スキップされた Step の ID |
| `rule_id` | 発動した `step_flow_rule` の ID |
| `condition_evaluated` | 条件式と評価時の入力値のスナップショット |
| `skipped_at` | タイムスタンプ（Contemporaneous） |

### 5-3. バリデーション警告付き記録

カスタムバリデーションが警告を返した場合、記録を拒否するのではなく「警告を承認した上で記録する」フローを採用する（Contemporaneous 優先）。UI は作業員に警告内容を提示し、意図的な承認操作（ダブルタップ等）を必須とする。

---

**本節で確定した方針**
- カスタム Step の入力値を標準 Step と同一の Append-only イベントストアに格納することを確定する。
- 条件分岐スキップを記録の欠損として扱わず `step_skipped_by_rule` イベントとして記録することを確定する。
- バリデーション警告時は「拒否ではなく承認操作付きで記録」する方針を確定する。

---

## 6. セキュリティ制約

### 6-1. eval 禁止とサンドボックス

カスタム Step の条件式・バリデーションロジックは JSON Logic のみを許容し、以下を禁止する：

| 禁止事項 | 理由 |
|---|---|
| `eval()` / `Function()` | 任意コード実行による XSS / 権限昇格リスク |
| 外部 API 呼び出し | Offline-First 設計の破壊・ネットワーク依存の導入 |
| DOM 操作 | React Native の仮想 DOM との競合・XSS |
| ステートレスでない副作用 | Append-only 原則との矛盾 |

### 6-2. 入力値のサニタイズ

カスタム Step タイプの `label` フィールドに含まれる多言語テキストは、表示時に HTML エスケープを必須とする。`input_field.ui_hint` の値は許可リスト（`number_picker` / `slider` / `text` / `select` 等）と照合し、許可リスト外の値は無視する。

### 6-3. ライセンス確認（JSON Logic）

JSON Logic（Apache 2.0）は本アプリの MIT ライセンスと競合しない。クライアント側（React Native）での実装には `json-logic-js`（MIT）を採用する。

---

**本節で確定した方針**
- eval 禁止・外部 API 呼び出し禁止・DOM 操作禁止をカスタム Step ロジックの不変制約として確定する。
- 多言語テキストの HTML エスケープ・ui_hint の許可リスト検証を必須とすることを確定する。

---

## 7. やらないこと

| やらないこと | 理由 |
|---|---|
| eval / Function constructor によるカスタムロジック実行 | 任意コード実行リスク・サンドボックス化不能 |
| 外部 API 呼び出しを伴うリアルタイムバリデーション | Offline-First 設計の破壊・ネットワーク依存 |
| Step タイプ定義の「ビルド時コンパイル」や「ネイティブコード化」 | ver1.0.0 の個人開発スコープを超える |
| 条件分岐の「ループ」（繰り返し実行）構造 | DAG 検証の複雑化・ALCOA+ との整合困難 |
| カスタム Step タイプの「他工場への自動配布」 | 意図せぬ SOP 変更のリスク（工場単位の設計余地として保持） |

---

**本節で確定した方針**
- eval 禁止・外部 API 禁止・ループ構造禁止の 3 点を永続的な設計制約として確定する。
- 他工場への自動配布は将来オプションとして保留し、ver1.0.0 では対象外とすることを確定する。
