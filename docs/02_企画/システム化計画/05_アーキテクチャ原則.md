# 05 アーキテクチャ原則

本章の責務は、本システムのソフトウェアアーキテクチャ上の不変原則を確定することである。ここで確定した原則は、技術選定・データモデル設計・非機能要件設定のすべての判断において優先される。本章は技術スタックの名称（具体的なプログラミング言語・フレームワーク等）には言及しない。それらの選定根拠は 07 章に委ねる。

## 1. Offline-First の原則と境界

### 原則の宣言

タブレット端末は「サーバーへの接続の有無によらず、すべての主要操作が完遂できる」設計とする。この原則を Offline-First 原則と呼称し、本アプリの最高優先アーキテクチャ原則として確定する。

ネットワーク状態は「接続」「切断」という二値で管理しない。製造現場の社内 LAN は品質が一定ではなく、帯域の変動・Wi-Fi の瞬断・建屋間の電波遮蔽が日常的に発生する。このため、ネットワーク状態を以下の 4 段階で制御する。

| 状態 | 定義 | 端末の動作 |
|---|---|---|
| 高品質接続 | RTT 100ms 以下・パケットロスなし | 同期優先。全機能が利用可能 |
| 低品質接続 | RTT 高値または断続的なパケットロス | ローカル操作を継続。バックグラウンド同期を試行 |
| 切断 | サーバーへの到達不可 | ローカル操作のみ。同期キューに積む |
| Emergency Mode | 切断が設定時間（例: 5 分）を超過 | フルスクリーンバナーで状態を通知。「最終同期時刻」を常時表示 |

端末ローカルの永続ストレージを「一次記録メモリ」として扱い、サーバーへの同期は「複製」と位置づける。この非対称性は本アプリの可用性と ALCOA+ の Enduring 原則を両立させる設計上の核心的決定である。

### Offline-First の対象と非対象

**対象（Offline-First が保証される機能）**:
- 手順ナビゲーション（SOP の閲覧・Step の進行・記録入力）
- 記録入力（数値・テキスト・写真・QR スキャン結果）
- ローカルアラート発報（入力値の範囲逸脱・必須項目未入力）
- 電子署名（ローカル鍵によるサイン記録）

**対象外（接続時のみ有効な機能）**:
- SOP マスタの最新版取得（サーバーからの差分ダウンロード）
- 管理者向けリアルタイム照会（進捗ダッシュボード・Audit Trail 閲覧）
- ユーザーマスタ・設備マスタの新規追加・変更

Emergency Mode では、最後に同期済みの SOP キャッシュを使用して作業を継続できる。キャッシュが存在しない SOP は「未ダウンロード」として明示し、操作を防止する。

### ALCOA+ Contemporaneous との調停

Offline-First 原則と ALCOA+ の Contemporaneous 原則（作業と同時に記録する）の間には潜在的緊張が存在する。サーバー受信タイムスタンプが実際の記録時刻より遅れる場合、監査者が「事後記録」と誤解するリスクがある。この緊張を以下の方針で解消する。

- 端末タイムスタンプ（記録が入力された瞬間の端末時刻、UTC ミリ秒）とサーバー受信タイムスタンプの二重記録を必須とする
- 未同期レコードには「記録済（未同期）」フラグを付与し、監査証跡で可視化する
- 事後同期時は端末タイムスタンプを「申告時刻」（記録の権威とする）とするが、サーバー受信時刻との差分を記録に保持する
- 「記録済（未同期）」フラグが付いたレコードは、Audit Trail 上で別色表示またはアイコンで区別し、監査者が一目で識別できる状態を保つ

この設計により、オフライン中の記録が Contemporaneous 原則の観点で「事後改ざん」と混同されることを防ぐ。

詳細は [`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) を参照すること。

---

**本節で確定した方針**
- ネットワーク状態を 4 段階（高品質接続・低品質接続・切断・Emergency Mode）で制御し、Emergency Mode では自動的にバナー表示と最終同期時刻表示を行う。
- 端末ローカルストレージを一次記録メモリとして位置づけ、サーバー同期を複製と定義する。
- ALCOA+ Contemporaneous との調停のため、端末タイムスタンプとサーバー受信タイムスタンプの二重記録を必須とし、未同期レコードへのフラグ付与を確定する。

## 2. Append-only Event Sourcing の採用根拠

### 原則の宣言

作業に関するすべての状態変化を「不変イベント」として記録する。現在の状態（工程の進捗・作業の完了状態・最新の測定値）は過去イベントの累積から導出する（マテリアライズドビューまたは集計クエリによる）。

イベントは一度書き込まれたら変更・削除できない。作業ログテーブルへの UPDATE 操作および DELETE 操作は、データベースロールレベルで実行権限を付与しない設計とする。この制約はアプリケーション層での制御では不十分であり、データベース権限設計として実装する。

修正が必要な場合は「取消イベント」または「修正イベント」として新たなイベントを追記する。誰が・いつ・何を理由に取り消したかを記録に残す。これにより ALCOA+ の Original 原則（一次記録が保全される）と Attributable 原則（誰が何をしたかが明確である）を構造的に満たす。

### イベントの必須属性（XES 互換設計）

プロセスマイニング国際標準 IEEE XES（eXtensible Event Stream）との互換性を保ち、将来の工程分析・プロセス改善の可能性を構造的に担保するため、すべてのイベントに以下の必須属性を保持する。

| 属性名 | 意味 | 設計上の制約 |
|---|---|---|
| Case ID | 作業指示 ID または製品ロット ID | NULL 禁止。一つのイベントが複数のケースに属することは禁止 |
| Activity | ステップ ID またはイベント種別（開始・完了・スキップ・差し戻し・中断） | NULL 禁止。列挙型で管理し、自由文字列を許可しない |
| Timestamp | サーバー受信時刻（UTC ミリ秒） | NULL 禁止。サーバーが付与。クライアントによる上書きを禁止 |
| Resource | 作業者 ID および端末 ID | NULL 禁止。ログアウト状態での記録を設計上禁止する |
| Attributes | 測定値・写真ハッシュ・スキャン値・使用 SOP 版 ID 等 | NULL 許容。ステップ種別ごとに必須属性を定義 |

イベント種別は「開始」「完了」「スキップ」「差し戻し」「中断」の 5 種を必須とし、いずれも除外しない。スキップされたステップが記録から消えることは、ALCOA+ の Complete 原則（必要な記録がすべて存在する）に違反する。

### CRDT 不採用の根拠

分散システムにおける競合解決手法として CRDT（Conflict-free Replicated Data Type）を検討したが、採用しない。根拠は以下のとおりである。

- 作業ログは基本的に競合しない。本システムの設計上、同一ステップを複数の作業者が同時に実行することは禁止する（作業指示の排他割り当てによる）。競合が設計上発生しない構造において、競合解決機構は不要な複雑性を導入する
- LWW-Register（Last-Write-Wins Register）はタイムスタンプの大小で競合を解決するが、端末時刻は改ざん可能であり、ALCOA+ の Accurate 原則に反するリスクがある
- G-Set（成長のみが許される集合）の特性は Append-only Event Sourcing によって代替可能である。不変イベントの追記がそのまま G-Set の意味論を実現する

詳細は [`90_業界分析/21_作業ログ分析とプロセスマイニング.md`](../../90_業界分析/21_作業ログ分析とプロセスマイニング.md) および [`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) を参照すること。

---

**本節で確定した方針**
- 作業ログへの UPDATE/DELETE をデータベースロールレベルで禁止し、修正は「取消イベント」「修正イベント」の追記のみを許容する。
- XES 互換の 5 属性（Case ID・Activity・Timestamp・Resource・Attributes）を全イベントの必須属性として確定し、スキップ・差し戻し・中断を含む 5 種のイベント種別をすべて記録対象とする。
- CRDT（LWW-Register）を採用せず、Append-only Event Sourcing で G-Set の意味論を代替することを確定する。

## 3. Idempotent API と一意性キー設計の原則

### 原則の宣言

タブレット端末が生成するすべての書き込みリクエストに、UUID v4 形式の Idempotency Key を付与する。この Key は端末が生成し、リクエストペイロードのヘッダーに含めて送信する。

サーバーは同一 Idempotency Key の重複リクエストを冪等に処理する。同一 Key が送信された場合、初回処理の結果を返し、副作用（データ書き込み・イベント発行）を繰り返さない。これにより、ネットワーク不安定によるリトライが二重記録を生まない構造を保証する。

オフライン中にキューに積まれたリクエストは、通信回復後に Outbox Pattern で送信する。Outbox テーブルはキューとして機能し、送信後も削除せず、送信済みフラグで状態を管理する（監査目的）。送信順序は端末タイムスタンプ順を維持する。

### Outbox Pattern の設計方針

端末ローカルの Outbox テーブルは以下の責務を持つ。

- 書き込みイベントのローカル保持（オフライン中の永続化）
- 送信状態の管理（未送信・送信中・送信済み・送信失敗）
- 冪等性の保証（Idempotency Key をレコードに保持し、重複送信を防ぐ）
- 監査証跡の一部としての永続保存（削除しない）

通信回復後の送信は、Outbox テーブルの未送信レコードをポーリングまたはネットワーク状態変化イベントで駆動する。指数バックオフを適用し、永久ループを防ぐ。

### 端末時刻操作への防御

端末時刻は作業者または管理者によって意図的または意図せずに変更される可能性がある。端末時刻に依存したタイムスタンプを権威とした場合、ALCOA+ の Consistent 原則（記録が矛盾なく整合する）に違反するリスクがある。このリスクへの防御として以下を確定する。

- サーバーが受信タイムスタンプを付与し、これを「証拠時刻」（監査・トレサビに使用する権威時刻）とする
- 端末タイムスタンプは「申告時刻」として別フィールドで保存し、消去しない
- 「証拠時刻」と「申告時刻」の差が設定閾値（例: 5 分）を超えるレコードには警告フラグを立て、監査画面で可視化する
- 警告フラグが立ったレコードは自動的に拒否しない（作業の継続性を優先する）が、品質担当・現場監督への通知を行う

詳細は [`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) および [`90_業界分析/22_規制別トレーサビリティ要件詳論.md`](../../90_業界分析/22_規制別トレーサビリティ要件詳論.md) を参照すること。

---

**本節で確定した方針**
- すべての書き込みリクエストに UUID v4 の Idempotency Key を付与し、サーバーは冪等に処理することを確定する。
- Outbox テーブルを永続保存し（削除しない）、送信済みフラグで状態管理することを確定する。
- サーバー付与タイムスタンプを「証拠時刻」とし、端末タイムスタンプとの差が閾値超過のレコードに警告フラグを立てることを確定する。

## 4. OT-IT 境界の明確化と片方向流入

### OT-IT 境界の定義

製造現場のシステム構造は OT（Operational Technology）領域と IT（Information Technology）領域に区分される。本アプリはこの境界を明確に認識した上で設計する。

| 領域 | 構成要素 | 本アプリとの関係 |
|---|---|---|
| OT 領域 | PLC・SCADA・設備センサ・工作機械・制御ネットワーク | 本アプリは OT 領域に位置しない |
| IT 領域 | 本アプリのバックエンド・タブレット端末・管理 Web・事務 PC | 本アプリの動作領域 |

### 本アプリの立場

本アプリは IT 領域のみに位置する。以下を設計上の確定方針とする。

**OT 領域への制御信号送信を行わない**: 本アプリは PLC・SCADA 等の制御システムに対して、いかなる制御指令も送信しない。設備の起動・停止・パラメータ変更を本アプリから行う設計は ver1.0.0 では対象外と判断する。

**OT から IT への情報流入は将来オプションとして設計余地を残す**: センサ値・設備状態の読み取り（OT → IT 方向の片方向流入）は、将来的な機能拡張として設計余地を残す。ただし ver1.0.0 では対象外と判断する。API 設計においてこの拡張ポイントを塞がない設計とする。

**OT-IT 境界の物理的・論理的分離を前提とする**: 本アプリは OT-IT 境界にファイアウォールまたは DMZ が設置されていることを導入前提条件とする。本アプリがその境界を越える設計を行わない。

**Digital Twin のドリフト問題を認識した設計とする**: OT センサ値に基づく設備状態の表示を将来実装する場合も、センサ値とアプリ側の状態表示のリアルタイム同期を過信しない設計とする。データの「鮮度」を明示し、古い状態データを最新として表示することを禁止する。

詳細は [`90_業界分析/07_スマートファクトリーと作業のデジタル化.md`](../../90_業界分析/07_スマートファクトリーと作業のデジタル化.md) を参照すること。

---

**本節で確定した方針**
- 本アプリは IT 領域のみに位置し、OT 領域への制御信号送信を設計上禁止する。
- OT から IT への情報流入（センサ値読み取り）は将来オプションとして設計余地を残すが、ver1.0.0 では対象外と判断する。
- OT-IT 境界のファイアウォール・DMZ 設置を導入前提条件として確定する。

## 5. 改竄不可性とハッシュ証明の原則

### 原則の宣言

作業ログ・電子署名・写真記録は改竄不可性を技術的に保証する。改竄不可性は「削除・変更できない構造」と「変更されていないことを検証できる構造」の双方によって実現する。

### 写真記録のハッシュ証明フロー

写真記録の改竄不可性は以下のフローで実現する。

1. 端末がカメラで撮影する
2. 端末がアプリケーション層で SHA-256 ハッシュ（NIST FIPS 180-4 準拠）を算出する
3. アプリケーション層でメタデータ（作業者 ID・ステップ ID・端末タイムスタンプ・SOP 版 ID）を付与する
4. 写真データとハッシュ値をサーバーへ送信する
5. サーバーが受信した写真データのハッシュを再計算し、送信値と照合する（転送中改竄の検出）
6. 照合成功後、写真を UUID パスで保存し、ハッシュ値を DB に記録する
7. 以降、写真の参照時にハッシュを再計算して DB の値と照合する（保存後改竄の検出）

メタデータはアプリケーション層でデータとして保持する。Exif データへの依存は行わない。Exif は撮影後のファイル編集ツールで改変可能であるため、証拠品質の観点から信頼できない。

### 作業ログのハッシュチェーン

作業ログ全体の整合性は定期バッチで累積ハッシュチェーンによって検証する。具体的には、前レコードのハッシュ値を次レコードの計算に含めることで、チェーン途中のレコード削除・挿入・変更を検出可能とする。このハッシュチェーンはブロックチェーン類似の不変性証明であり、DB ロールによる削除禁止と組み合わせることで二重の改竄防止を実現する。

### DB ロールによる削除禁止

アプリケーション層での DELETE/UPDATE 防止に加え、作業ログテーブルへのアクセスには以下のロール分離を適用する。

- **書き込みロール**: INSERT のみ許可。UPDATE・DELETE は権限なし
- **参照ロール**: SELECT のみ許可。INSERT・UPDATE・DELETE は権限なし
- **管理ロール**: バックアップ・バキューム・スキーマ変更のみ許可。作業ログレコードへの変更は禁止

この設計により、アプリケーションコードのバグ・脆弱性を突いた作業ログの改竄を防止する。

詳細は [`90_業界分析/31_非テキスト記録と証拠品質管理.md`](../../90_業界分析/31_非テキスト記録と証拠品質管理.md) を参照すること。

---

**本節で確定した方針**
- 写真はアプリケーション層で SHA-256 ハッシュを算出し、UUID パスで保存。参照時にハッシュ再検証を行うことを確定する。
- 作業ログには定期バッチによる累積ハッシュチェーン検証を適用し、改竄の事後検出を可能にすることを確定する。
- 作業ログテーブルへの INSERT/SELECT/管理の 3 ロール分離をデータベースレベルで確定する。

## 6. Local-First と参照モデルの権威の所在

### 権威の所在の定義

本システムが扱うデータの「権威の所在」（どのストレージが正本となるか）を種別ごとに確定する。

| データ種別 | 権威の所在 | 端末ローカルの位置づけ |
|---|---|---|
| SOP マスタ | サーバー（PostgreSQL） | 参照用キャッシュ（複製）。サーバーから差分取得 |
| ユーザーマスタ | サーバー（PostgreSQL） | 認証情報のみキャッシュ（オフラインログイン対応） |
| 設備マスタ | サーバー（PostgreSQL） | 参照用キャッシュ（複製）。更新はサーバー経由のみ |
| 作業ログ | 最初に書いたデバイスの端末ストレージ（一次記録） | サーバー到達時点でサーバーが正本に昇格 |

作業ログについては Local-First 原則に従い、端末に書いた瞬間から記録として有効である。サーバーへの同期はその複製であり、複製が完了した時点でサーバーが正本となる。ただし、端末ストレージの記録は複製完了後も削除しない。監査証跡として端末側に保持する期間は設定で管理する。

### マスタデータのキャッシュ戦略

SOP マスタ・設備マスタは接続時に差分取得してローカルキャッシュを更新する。キャッシュの鮮度は以下の規則で管理する。

- キャッシュには「有効期限」ではなく「最終同期日時」を付与する。有効期限による自動失効は Emergency Mode 中の作業継続を阻害する
- 最終同期日時と現在時刻の差が表示されるため、作業者はキャッシュが古い可能性を認識できる
- SOP の版 ID はイベントの Attributes に記録するため、いつの版に基づいて作業したかは事後に照会可能である

### 競合解決の原則

Offline-First 設計において、ネットワーク回復後に端末間で同一データの競合が発生するケースに対応する原則を確定する。

作業ログの競合は原則発生しない設計とする。作業指示の排他割り当て（一つのステップに複数の作業者を同時割り当てしない）によって、同一ステップへの二重記録の設計上発生を防ぐ。

ただし、システムの不具合や設計上の例外によって競合が発生した場合は以下の原則を適用する。

- より早い端末タイムスタンプのレコードを先行として扱う
- 競合したレコードの双方を保存し、どちらも削除しない
- 競合を検出した事実をイベントとして記録し、品質担当・現場監督に通知する

詳細は [`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) を参照すること。

---

**本節で確定した方針**
- SOP・ユーザー・設備の各マスタの権威をサーバー（PostgreSQL）に置き、端末ローカルを複製・キャッシュとして定義する。
- 作業ログは端末が一次記録を持ち、サーバー到達時点でサーバーが正本となる。端末記録は複製完了後も削除しない。
- 競合は設計上発生しない構造とするが、例外的に発生した場合は双方のレコードを保存し削除しないことを確定する。
