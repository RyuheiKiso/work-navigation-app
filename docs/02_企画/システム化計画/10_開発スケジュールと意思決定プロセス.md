# 10 開発スケジュールと意思決定プロセス

本章の責務は、個人開発 ver1.0.0 リリースまでのマクロスケジュールと意思決定プロセスを確定することである。具体的な工数・期間の見積りは確定しないが、各マイルストーンの完了条件・判定基準・プロセスの原則を確定する。

---

## 1. phase なし方針とマイルストーン概念

### CLAUDE.md ポリシーとの整合

本プロジェクトは個人開発であるため、フェーズ（phase）を設けない。フェーズを設けないことは「段階的に機能を削減してリリースする」ことを意味しない。「ver1.0.0 の全機能を最初から本番品質で作る」ことを意味する。機能の省略・品質の妥協は認めない。

### マイルストーン（M1〜M6）の採用

フェーズに代わる進捗管理単位として「マイルストーン（M1〜M6）」を採用する。マイルストーンは「ドキュメントの完成度」と「機能の動作確認」の両方で判定する。どちらか一方のみを満たしても、マイルストーン完了とは見なさない。

マイルストーン間に明確な順序はあるが、順序が確定した範囲での並行作業は許容する。例として、M2（データ層）の作業中に M3（バックエンド API）の設計ドキュメント作成を先行させることは認める。実装は先行マイルストーンの完了を待つ。

---

**本節で確定した方針**
- フェーズを設けず、ver1.0.0 の全機能を本番品質で実装することを開発方針として確定する。
- マイルストーン（M1〜M6）をドキュメント完成度と機能動作確認の両条件で判定することを確定する。
- 順序が確定した範囲での並行作業を許容し、実装は先行マイルストーンの完了を待つことを確定する。

---

### M1 ドキュメントに要求されること

M1 の「設計完成」は本書（システム化計画書）の完成を前提とする。システム化計画書が完成していない状態で M1 を宣言することは認めない。M1 完了の判定は以下の文書の完成を確認することで行う。

- システム化計画書（本書）全 11 章の完成
- 要件定義書（次フェーズ文書）の完成
- 概要設計書の完成

---

**本節で確定した方針**
- M1〜M6 の完了条件をドキュメントと機能の両面で確定し、どちらか一方のみの達成ではマイルストーン完了と見なさない。
- M1 完了の前提として本書（システム化計画書）全 11 章の完成を必須とする。
- M6 の完了条件に 30 日間パイロット運用と ALCOA+・倫理品質の自己監査全項目パスを含める。

## 3. ドキュメント先行・実装後追いの原則

### 原則の宣言

本プロジェクトはドキュメントを先に完成させ、実装はドキュメントに従う。この順序を逆転させることは認めない。

ドキュメントを書かずに実装した機能は「未設計機能」として M1 要件違反とみなす。未設計機能はリリース前に、ドキュメントを完成させるか、実装を削除するかのいずれかを選択する。両立する場合は「ドキュメントを完成させてから実装を確定させる」順序で処理する。

### プロトタイプ・PoC コードの扱い

以下の条件を満たす場合に限り、設計ドキュメントなしのプロトタイプ・PoC コードの作成を認める。

- ファイル冒頭に `// PROTOTYPE: 本番コードではない` のコメントを記載する。
- `prototype/` または `poc/` ディレクトリに格納し、本番ソースツリーに混在させない。
- M1 完了前に本番コードに昇格させない。

### 設計変更時の処理順序

実装後に設計変更が必要になった場合、以下の順序で処理する。

1. ドキュメントに変更内容を記述し、変更理由を記録する（ADR 作成要件に該当する場合は ADR を先行作成する）。
2. ドキュメントの変更が確定した後に実装を変更する。
3. ドキュメントと実装の不一致が生じた状態でコードをコミットしない。

ドキュメントが権威である。実装がドキュメントに優先することはない。

---

**本節で確定した方針**
- ドキュメント先行・実装後追いを絶対原則として確定し、逆順を認めない。
- 未設計機能は M1 要件違反とみなし、リリース前に解消することを確定する。
- 設計変更時はドキュメント更新を先行させ、実装と不一致な状態でのコミットを禁止する。

## 4. ADR（アーキテクチャ決定記録）の運用方針

### ADR 作成の条件

ADR は以下の 3 条件をすべて満たす意思決定にのみ作成する。条件を満たさない決定に ADR を作成することを禁止しない（過剰なドキュメント化は保守コストを増加させる）。

| 条件 | 判定基準 |
|---|---|
| 技術的不可逆性 | 後から変更するコスト（再実装・データ移行・テスト全書き直し等）が 1 週間以上の作業量になると判断される |
| 有力な代替案の存在 | 複数の有力な技術選択肢が存在し、どちらを選ぶかが自明でなかった |
| 選択理由の非自明性 | 「一般的だから」「慣れているから」以外の積極的な根拠がある |

### ADR の形式

ADR は以下の 6 項目で記述する。

| 項目 | 内容 |
|---|---|
| タイトル | `ADR-{連番}: {技術選定のタイトル}` |
| 状態 | 提案中 / 採択 / 却下 / 廃止 |
| 背景 | この決定が必要になった技術的・業務的文脈 |
| 決定事項 | 何を選択したか |
| 根拠 | なぜこの選択をしたか（代替案との比較を含む） |
| 結果 | 既知のトレードオフ・将来の再評価条件 |

### ADR の格納と参照

ADR は `docs/01_管理/` に格納する。本書からは参照のみ行い、本書内に ADR の内容を重複して記述しない。07 章の技術スタック選定根拠は ADR として機能する内容を含むが、本書の構成上 07 章に統合している。

---

**本節で確定した方針**
- ADR は技術的不可逆性・有力な代替案の存在・選択理由の非自明性の 3 条件をすべて満たす場合にのみ作成する。
- ADR の形式をタイトル・状態・背景・決定事項・根拠・結果の 6 項目で確定する。
- ADR は `docs/01_管理/` に格納し、本書からは参照のみ行う。

## 5. ver1.0.0 のリリース判定基準

### リリース判定の 7 要件

以下の全 7 項目をパスすることを ver1.0.0 のリリース判定条件として確定する。1 項目でも未達の場合はリリースを認めない。

| 判定項目 | 確認方法 |
|---|---|
| 1. M6 全完了条件の充足 | M1〜M6 の完了条件チェックリストを順に確認する |
| 2. ALCOA+ 9 原則の自己監査全項目パス | 06 章の ALCOA+ マトリクスを用いた自己監査チェックリストで確認する |
| 3. 倫理品質の自己監査全項目パス | 08 章 7 節の倫理サブ特性チェックリスト（目的限定性・データ最小化・透明性・説明可能性・自律性の保護・心理的安全）で確認する |
| 4. 全ユースケースの実機動作確認 | 04 章の全ユースケース（A-J）を実機環境で動作確認する |
| 5. 紙フォールバックの実機演習完了 | 第 3 縮退への切替・運用・復旧の手順を実機で演習し、記録する（09 章 5 節） |
| 6. 多言語表示の動作確認 | 日本語・英語の表示が全画面で正常に機能することを確認する |
| 7. IT 担当 1 名での初期構築確認 | GUI ウィザードのみで初期構築を完了できることを実機で確認する（09 章 1 節） |
| （追加要件）子機モードの動作検証 | 単独動作モードでの全機能動作を確認した上で、子機モード（計画 12 章）の READ-ONLY 同期が少なくとも1件の親機モック（または実環境）で正常動作すること。子機モードが利用できない環境では単独動作モードの完全動作のみをリリース判定の必須条件とする |

### パイロット運用の位置づけ

M6 の完了条件である「30 日間パイロット運用」は、判定項目 4（全ユースケース実機動作確認）・5（紙フォールバック実機演習）・7（IT 担当 1 名での初期構築確認）を包含する実施期間として扱う。パイロット運用中に発見された不具合は、M6 完了前に修正する。

### M6 パイロット完了後のステージゲート

M6（30日パイロット）完了後は、計画 14 章（品質指標と SLI / SLO / KPI 枠組み）で定義するステージゲートに従って、本番移行・横展開を判断する:
- パイロット通過条件: システム採用率 > 80% / 記録完全性 > 95% / 明らかな UX 問題なし
- 本番移行後の横展開条件: SLO を 90 日間連続達成
- 子機モード追加工程への展開: 単独動作モードでの安定稼働を確認後に実施する

---

**本節で確定した方針**
- ver1.0.0 リリース判定を 7 要件の全項目パスで確定し、1 項目でも未達の場合はリリースを認めない。
- ALCOA+ 自己監査・倫理品質自己監査のチェックリストを 06 章・08 章から参照してリリース前に実施する。
- 30 日間パイロット運用を M6 完了条件として必須とし、発見不具合はパイロット中に修正する。

## 6. 個人開発における自己引継ぎ設計

### 「将来の自分」へのコンテキスト継承

個人開発は開発者が唯一の関係者である。作業中断・疾病・長期休暇・他プロジェクト従事等によって 1 ヶ月以上の作業中断が生じる確率は、プロジェクト全体を通じて許容しなければならない現実である。

この現実に対する唯一の対策は「将来の自分」に対するコンテキスト継承設計である。以下の要件を全実装コードに適用する。

| 要件 | 実装基準 |
|---|---|
| README の整備 | 各モジュールに README を置く。「このモジュールが何をするか」「依存するモジュール」「テストの実行方法」を記述する |
| インラインコメント | 「なぜこの実装か」を記述する。「何をしているか」はコードを読めば分かるため省略してよい |
| 型定義の網羅 | TypeScript・Rust ともに any / 型推任せ・unwrap の多用を禁止する。型はドキュメントである |
| テストの網羅 | ユニットテスト・統合テスト・E2E テストで意図を記録する。テストは「仕様書」として機能する |

### 20 章の「プレースキーパー設計」の開発プロセスへの適用

業界分析 [`90_業界分析/20_作業中断・割込み・再開の認知科学.md`](../../90_業界分析/20_作業中断・割込み・再開の認知科学.md) が示す作業中断時のプレースキーパー設計（自己引継ぎ設計）を、本プロジェクトの開発プロセス自体に適用する。

具体的には以下の通りである。

- 作業セッションの終了時に、次回再開のための「再開メモ」をリポジトリの `NEXT.md` に記録する。
- `NEXT.md` には「どこまで完成したか」「次に何をするか」「未解決の問題は何か」を記述する。
- `NEXT.md` はコミットせず `.gitignore` に含める。ただし作業再開の際に必ず参照する。

### 1 ヶ月中断耐性の確認

「1 ヶ月間作業を中断した後でも、ドキュメントを読めば再開できること」を M6 リリース判定条件（5 節 判定項目 1）の評価観点として含める。評価の実施方法は「パイロット運用開始時点でドキュメントのみを参照して環境を再構築できるか」を実機で確認する。

業界分析 [`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md) が示す中小製造業の IT 調達慣行（稟議・年度予算縛り・PoC ロードマップ）を念頭に置き、本システムが将来的に SI 調達・チーム開発に移行する際の引継ぎ可能性を、ドキュメント駆動設計・ADR によって担保する。

---

**本節で確定した方針**
- 全実装コードに README・インラインコメント（なぜの記述）・型定義・テストを必須として確定する。
- 20 章のプレースキーパー設計を開発プロセスに適用し、セッション終了時の再開メモ記録を習慣化する。
- 1 ヶ月中断後のドキュメントのみによる環境再構築確認を M6 リリース判定の評価観点に含める。
