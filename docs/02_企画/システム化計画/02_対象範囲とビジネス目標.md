# 02 対象範囲とビジネス目標

本章の責務は、本システムが対象とする業種・規模・形態・作業種別の境界条件を確定し、計測可能なビジネス目標を設定することである。対象外領域の明示（やらないことの宣言）は 01 章の倫理境界宣言と並び、本書の設計上の誠実さを担保する。

---

## 1. 対象業種・規模・形態の境界条件

本システムが対象とするユーザーセグメントと、明示的に対象外とするセグメントを以下のとおり確定する。

| 軸 | 対象（In-Scope） | 対象外（Out-of-Scope） |
|---|---|---|
| 業種 | 製造業（食品・自動車部品・電子部品・医療機器中小等） | 建設・物流・サービス業・農業・鉱業 |
| 規模 | 従業員数 10〜500 名を主ターゲット | 1,000 名超の大規模多工場展開 |
| サイト数 | 単一工場（単一サイト、複数ライン可） | マルチサイト・グループ工場の横断管理 |
| ネットワーク | 社内 LAN（オンプレ） | インターネット経由 SaaS・マルチテナント |
| 作業種別 | 製造作業・品質検査・設備保全・準備/段取作業 | 事務作業・営業活動・設計業務・研究開発 |

### 1-1. 参入余地の根拠

中小製造業市場において「SaaS 否定環境」は特殊解ではなく相当数存在する。BCP の観点からインターネット依存を避けること、顧客要求によるデータ主権（自社サーバーへのデータ保管義務）、社内情報セキュリティポリシーとの矛盾が SaaS 採用を阻む主な要因である。

競合製品（大手 MES ベンダー・eBR 製品等）は大企業向けの多機能・高価格帯に設計されており、中小製造業が求める「単一工場での完結・高いカスタマイズ許容度・妥当な導入コスト」の交差領域に空白が存在する。本システムはこの空白を埋める位置に設計する。

詳細は [`90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md`](../../90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md) を参照すること。

### 1-2. 技術スタックとスコープの整合

React Native（iOS / Android / Windows タブレット対応）・Rust + axum（バックエンド）・PostgreSQL・IIS + Windows Server 2022 + WSL2 + Docker という技術スタックは、社内 LAN・オンプレ環境での運用を前提として選択している。この選択はマルチサイト・SaaS 展開を排除する設計上の決定と整合する。

詳細は [`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md)（IT 投資率・稟議構造・補助金適合性）を参照すること。

**本節で確定した方針**
対象セグメントを「中小製造業・単一工場・社内 LAN・オンプレ」に確定し、1,000 名超・マルチサイト・SaaS 環境は対象外と判断した。
業種は製造業に限定し、建設・物流・サービス業は対象外と判断した。
この境界は ver1.0.0 の設計・開発・テスト・リリース全フェーズを通じて変更しない。

## 2. 想定する工場像

本節では、本システムが主に使用される工場の物理的・組織的特性を確定する。この定義は 04 章のユースケース設計・06 章のデータモデル設計・08 章の非機能要件設計の前提となる。

### 2-1. レイアウトと生産方式

単一工場内に複数のライン・工程が存在する形態を前提とする。生産方式は特定の形態に依存しない。Hayes & Wheelwright の製品-プロセスマトリックスで示されるジョブショップ（個別受注生産）からフロー（大量生産）までの類型が混在する現場を想定する。

本システムが主たる対象とするのは、バッチ生産からフロー生産の領域である。ジョブショップ領域（純粋な一品受注・都度個別設計）は SOP の定義が困難であるため、本システムの価値が限定される。この判断は設計上の制約として確定する。

多品種少量生産（品番数: 数十〜数千）において、品番ごとに SOP を個別作成することは非現実的である。したがって、テンプレートの再利用機構（工程・作業・Step の親子関係によるテンプレート継承）が設計上の必須要素となる。

詳細は [`90_業界分析/14_生産システム設計と工場レイアウト.md`](../../90_業界分析/14_生産システム設計と工場レイアウト.md) を参照すること。

### 2-2. 労働力の多様性

以下の労働力特性を想定し、本システムの設計に反映する。

**外国人労働者の混在**: 外国人技能実習生・特定技能労働者が在籍する現場を前提とし、多言語インターフェース（最低限: 日本語・英語）を設計必須要件とする。母語が異なる作業員が日本語 SOP を誤解することで生じる品質リスクは、多言語化によって低減する。

**高齢化労働者の混在**: 50 代以上の作業員が主たるユーザーに含まれる現場を前提とする。小さいフォント・複雑なインターフェース・大量のテキスト情報の一覧表示は設計として排除する。

**三交代勤務（夜勤含む）**: 夜勤帯においても作業内容・品質・安全のレベルが昼勤帯と等価に維持されることを前提とする。これは 04 章のシナリオ設計で「シフト交代・引継ぎシナリオ」として具体化する。

**習熟度の多様性**: 入社直後の未習熟者から数十年の経験を持つ熟練者まで、同一ラインに混在する現場を前提とする。習熟度別のインターフェース（詳細モード / 熟練者モード）は 04 章で設計する。

### 2-3. IT 環境の前提

- **IT 担当者数**: 1 名（専任の IT 部門が存在しない場合も含む）
- **既存システム**: ERP・MES・PLM の保有は前提としない（導入していない場合が多い）
- **デバイス**: タブレット端末（Android / iOS / Windows）が主要接点。PCは管理業務・マスタメンテナンスに使用
- **ネットワーク**: 工場内社内 LAN。Wi-Fi の安定性に課題がある現場も存在することを想定し、オフライン動作の設計を必須とする

**本節で確定した方針**
バッチ〜フロー生産域の多品種少量生産工場を主対象と確定した。
外国人労働者・高齢化労働者・三交代勤務・習熟度多様性の四点を労働力特性として確定し、設計要件に反映する。
IT 担当 1 名前提・ERP/MES 非保有前提・オフライン動作必須を前提条件として確定した。

## 3. 対象作業の階層（工程 / 作業 / Step の 3 階層）

本システムの SOP マスタが対象とする作業の分解粒度を以下のとおり確定する。この階層定義は 06 章のデータモデル設計の基礎となる。

### 3-1. 3 階層の定義

| 階層 | 名称 | 定義 | 例 |
|---|---|---|---|
| Level 1 | 工程（Process） | ライン・工程単位の管理粒度。独立した品質管理ポイントを持つ | 溶接工程 / 検査工程 / 塗装工程 |
| Level 2 | 作業（Operation） | 一人の作業者が担当する作業単位。開始・終了が明確に定義できる | 溶接準備 / 本溶接 / 後処理 / 外観検査 |
| Level 3 | Step | 一操作単位のチェックリスト粒度。合否判定・数値入力・写真撮影等が紐付く | 治具セット → トルク確認 → 測定値入力 → 担当者サイン |

### 3-2. 対象外の粒度

本システムは以下の粒度は対象外と判断する。

- **動素（Therblig）粒度**: 身体動作の最小単位（掴む・移動する・解放する等）。IE 的分析ツールの領域であり、本システムの Navigation・Traceability の二大価値には不要である
- **PTS（Predetermined Time Standards）粒度**: MTM・MOST 等の標準時間分析手法。作業時間の精密推計が目的であり、本システムの作業支援・記録目的とは異なる

これらの粒度は [`90_業界分析/02_作業研究の系譜とIE.md`](../../90_業界分析/02_作業研究の系譜とIE.md) で詳述されている。本システムが対象としない理由は、個人開発の設計複雑度管理および中小製造業の現場運用実態との整合にある。

### 3-3. 階層の柔軟性

単一工程内に工程直下の Step が直接紐付く（Operation 階層を省略する）構成も許容する。小規模な現場では 2 階層構成（工程 / Step）で運用されることが現実的である。データモデルはこの柔軟性を設計上サポートする。

**本節で確定した方針**
作業階層を「工程 / 作業 / Step」の 3 階層として確定し、動素・PTS 粒度は対象外と判断した。
2 階層構成（工程 / Step）の省略形態をデータモデル上でサポートすることを確定した。
この粒度定義は 06 章データモデルの SOP マスタ設計に直接反映する。

## 4. ビジネス目標と成功指標

本システムの導入によって達成すべきビジネス目標を確定する。ROI の具体的数値は約束しない。理由は TDABC（Kaplan & Anderson 2007 の時間駆動 ABC 原価計算）の枠組みで示されるとおり、時間ドライバーを実測するまでは推計値が不確実だからである。詳細は [`90_業界分析/16_製造コストと価値工学.md`](../../90_業界分析/16_製造コストと価値工学.md) を参照すること。

ビジネス目標は「定性的達成状態」と「参考指標」の二段構えで設定する。参考指標の具体的な目標値は、要件定義フェーズで導入先工場の実測データに基づいて設定する。

| ビジネス目標 | 定性的達成状態 | 参考指標（要件定義時に実測ベースで設定） |
|---|---|---|
| 作業ミスの構造的削減 | 「気づけば間違っていた」状態から「気づける仕組みがある」状態への移行 | 外在的認知負荷起因の作業ミス発生率・EWI 発報後の是正完了率 |
| 記録品質の向上 | ALCOA+ 準拠の証拠が標準記録として取れている状態 | 記録完全性率（全 Step に記録が紐付く割合）・タイムスタンプ逸脱件数 |
| 暗黙知の半形式化 | 熟練者の判断コツが動画・写真・コメントとして次世代に届く状態 | 引継ぎに要する口頭説明時間・初期立ち上がり期間・OJT 完了率 |
| 不適合の早期検出と再発防止 | 不適合発生から CAPA 起票・手順改訂までのループが回っている状態 | CAPA 平均クローズ日数・SOP 改訂の採用率 |
| 現場のデジタル適応 | 紙との二重管理が発生せず、現場が主体的にシステムを使っている状態 | システム使用率（全 Step のシステム記録割合）・紙バイパス頻度 |

### 4-1. 目標間の優先順位

上記 5 目標は同等の優先度を持つが、導入初期フェーズ（最初の 3 ヶ月）では以下の順序で達成を確認する。

1. **現場のデジタル適応**: システムが使われない限り、他の 4 目標は達成不能であるため、これを最優先の成功条件とする
2. **記録品質の向上**: デジタル記録が紙記録と同等以上の品質で生成されることを確認する
3. **作業ミスの構造的削減**: EWI が実際に誤操作防止に機能していることを確認する
4. **不適合の早期検出と再発防止**: CAPA ループが現場に定着していることを確認する
5. **暗黙知の半形式化**: 最も時間を要するため、継続的な改善目標として位置づける

### 4-2. 目標の達成を阻む主要リスク

以下のリスクは 11 章で詳細化するが、ビジネス目標の設定と不可分であるため概要のみ記載する。

- **紙バイパスの常態化**: デジタルシステムの使いにくさや障害発生時に、作業員が紙に戻ることで記録が二重化・断絶する
- **IT 担当の習熟コスト不足**: Rust・React Native の技術スタックに習熟するまでの期間に障害対応が困難になる
- **マスタ作成工数の過小評価**: SOP マスタの初期作成が予想以上の工数を要し、導入が遅延する

**本節で確定した方針**
ビジネス目標を 5 項目・定性的達成状態・参考指標の形式で確定した。
ROI の具体的数値は約束せず、要件定義時に実測データから設定することを確定した。
導入初期 3 ヶ月での達成確認順序を「現場のデジタル適応」から始める順序で確定した。

## 5. 「やらないこと」の明示

本節は本システムが ver1.0.0 において意図的に提供しない機能・責務を確定する。「対象外」の明示は、将来の機能追加要求に対する応答可能な根拠を確立する。

| 対象外の機能・責務 | 理由 |
|---|---|
| 生産計画・スケジューリング | MES / ERP の領域であり、本システムのスコープを超過する。作業の「計画立案」ではなく「計画された作業の実行支援と記録」が本システムの責務である |
| PLC / IoT センサのリアルタイム制御 | OT（Operational Technology）領域への越境に該当する。個人開発の設計・検証範囲を超え、安全設計上の責任を適切に担保できない |
| AI / ML による自動判断・推論 | 対象外と判断する（03 章で詳述） |
| ERP・MES・PLM とのリアルタイム双方向連携 | 個人開発スコープを超過する。将来オプションとして設計予約のみ行う |
| SaaS 提供・マルチテナント | 社内 LAN 前提の設計と根本的に矛盾する。セキュリティ分離・テナントデータ保護の設計が ver1.0.0 の責務範囲外である |
| 労働者の個人別生産性評価・ランキング | 01 章の倫理境界宣言による。GDPR 第 22 条・日本改正個人情報保護法・JCM に基づき設計制約として確定済みである |
| 規制当局への直接電子申請 | 法令への代行申請は運用責任者（導入先企業の経営層）の責務であり、システムが代行する設計は責任帰属を曖昧にする |
| 業界横断型マスタ共有 | 社内オンプレ前提では範囲外である。SOP・品質基準は導入先企業の固有資産であり、外部共有は情報漏洩リスクを生じさせる |

### 5-1. 「やらないこと」の管理方針

上記の対象外項目は、ステークホルダーからの機能追加要求として提起された場合に、本章を根拠に断ることができる。ただし、ver1.0.0 以降の設計変更によって対象外項目を対象に変更する場合は、本章の改訂・本書全体の影響範囲確認・ステークホルダーへの説明を経なければならない。

**本節で確定した方針**
8 項目の対象外機能・責務を確定し、それぞれの理由を根拠として明示した。
「やらないこと」の変更は本章改訂と全章影響確認を必須とすることを確定した。
特に「AI / ML は対象外」は 03 章で詳述し、本章以降での再言及は行わない。

## 6. 将来法人化・SI 調達余地のための拡張ポイント

本節は ver1.0.0 で達成しない機能について、将来の拡張を可能にするための設計上の予約事項を確定する。これらは ver1.0.0 の設計制約に影響を与えない。設計の複雑性増加・開発工数増加・品質保証範囲の拡大を引き起こす場合は、予約事項として対応せず将来バージョンに完全に委ねる。

### 6-1. マルチサイト・マルチライン

データモデルに `factory_id` / `site_id` を予約フィールドとして保持する。ver1.0.0 では単一サイト前提のため、これらのフィールドの値は固定値（例: `"default"` / `"site-001"`）とし、マルチサイト制御ロジックは実装しない。将来のマルチサイト展開時に、このフィールドを活用したデータ分離・横断集計が可能な設計を保障するためにのみ予約する。

### 6-2. SI 調達対応

REST API のエンドポイント設計において、中堅 SIer 経由での導入・カスタマイズが可能な設計を採用する。具体的には、REST API の完全な仕様書（OpenAPI 3.x 準拠）を ver1.0.0 の成果物として提供し、Webhook の送受信仕様を公開する。これにより、法人化後に SIer を経由した導入・保守移管が技術的に可能な状態を担保する。

国内製造業 IT 調達の稟議プロセス（5〜8 段階・年度予算縛り）および IT 導入補助金・ものづくり補助金への申請要件との適合性については [`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md) を参照すること。

### 6-3. 規制業界拡張

ALCOA+ コアを中核とし、将来的に GMP / FDA 21 CFR Part 11 / IATF 16949 対応のアドオンモジュールを追加可能なプラグイン構造を採用する。アドオンの追加は、ALCOA+ コアのデータモデルと非機能要件に影響を与えない範囲で設計する。この構造の詳細は 05 章アーキテクチャ原則で確定する。

### 6-4. 法人化後の保守移管

ドキュメント駆動設計（本書・要件定義書・概要設計書・詳細設計書・API 仕様書の整備）によって、属人化を防ぎ、チーム開発・保守移管を可能にする。ver1.0.0 を個人開発で完成させながらも、コードの可読性・モジュール分割・テストカバレッジを本番品質として維持することがこの方針の前提条件である。

**本節で確定した方針**
拡張ポイントとして `factory_id` / `site_id` の予約フィールド・OpenAPI 仕様書提供・プラグイン構造・ドキュメント駆動設計の 4 点を確定した。
これらの予約事項は ver1.0.0 の設計制約・開発工数・品質保証範囲に影響を与えないことを条件として確定した。
拡張ポイントが ver1.0.0 の品質妥協の理由になることを禁止する。
