# 11 前提・制約・リスク

本章の責務は、全章の前提条件を集約し、リスクと将来オプションを明示し、「本書が約束しないこと」を再宣言することである。本書の末章として、読者がシステム化計画書の適用範囲を正確に把握するための境界定義を確定する。

---

## 1. 業務的前提条件

### 前提 P01〜P07 の一覧

以下の業務的前提条件を確定する。前提が成立しない環境への適用は本書の対象外である。

| ID | 前提条件 | 前提が崩れた場合の影響 |
|---|---|---|
| P01 | 単一工場・単一サイトで運用する | マルチサイト対応が必要となり、データモデル・認可設計（RBAC 権限スコープ）の再設計が必要になる |
| P02 | 社内 LAN のみでの利用（インターネット接続なし） | クラウド連携・外部認証（OAuth2/OIDC）・外部監査機関へのデータ提出が要件となり、アーキテクチャの根本的変更が必要になる |
| P03 | 作業員は Android または Windows タブレットを使用する | iOS 専用・スマートグラス・特殊端末が主流になる場合は UI・デバイス統合の再設計が必要になる |
| P04 | 工場の Wi-Fi カバレッジは不完全である（デッドゾーン存在） | 完全有線接続環境では Offline-First 設計は過剰設計となるが、後退はしない。設計の堅牢性として維持する |
| P05 | 外国人技能実習生・特定技能労働者が 1 名以上いる | ゼロの場合、多言語機能は「将来余地」に留まる。ただし実装は省かない |
| P06 | 規制当局への電子記録提出義務は現在発生していない | GMP/FDA Part 11 適用業種への展開時は、11 章将来オプション（規制業界拡張）が発動する |
| P07 | IT 担当は 1 名でサーバー・ネットワークを兼務する | IT 専門部署がある場合は Docker GUI ウィザードの設計要件が緩和されうるが、設計は後退させない |

---

**本節で確定した方針**
- P01〜P07 を本プロジェクトの業務的前提条件として確定し、前提が崩れた場合の影響を明示する。
- 前提が成立しない環境への適用は本書の対象外であり、前提変更時は設計を再評価することを確定する。
- 前提が成立しない場合でも、設計の後退（Offline-First・多言語・倫理品質の削除等）は認めない。

## 2. 技術的前提条件

### 前提 T01〜T05 の一覧

以下の技術的前提条件を確定する。

| ID | 前提条件 | 前提が崩れた場合の影響 |
|---|---|---|
| T01 | Windows Server 2022 が稼働できる社内サーバーが 1〜2 台ある | Linux サーバー環境への変更時は IIS が不要となり、Nginx + Docker Compose に切替可能。設計の主要部分は維持される |
| T02 | WSL2 が Windows Server 2022 上で有効化されている | 有効化できない場合、Docker Desktop for Windows（追加ライセンスコスト）への切替が必要 |
| T03 | タブレットは IP54 以上の耐環境性を持つ | IP 等級要件が IP65 以上（粉塵完全防護）または IP69K（高圧洗浄対応）になる場合、機種選定の再確認が必要 |
| T04 | LAN 内 Wi-Fi のレイテンシは 100ms 以下である | 高レイテンシ環境では Offline-First 設計の重要性が増すが、本設計は既にその前提で構築されている |
| T05 | タブレットへの Expo APK サイドロードが許可されている | MDM ポリシーで制限がある場合は企業配布証明書・MDM による配布への切替が必要 |

---

**本節で確定した方針**
- T01〜T05 を技術的前提条件として確定し、代替手段の存否を明示する。
- T01（Windows Server）の代替として Linux + Nginx + Docker Compose による構成変更の可能性を記録するが、ver1.0.0 の主要対象は T01 の環境とする。
- T03（IP 等級）の変更はデバイス選定の問題であり、アプリ設計には影響しない。

## 3. リスクレジスタ

### リスクの管理方針

リスクレジスタは「発生した場合の影響が高い」かつ「本設計の選択に直接影響する」リスクのみを記録する。軽微なリスクの羅列はリスク管理の実効性を下げるため含めない。カテゴリは技術・規制・組織・倫理の 4 区分とする。

### 技術リスク

| ID | リスク内容 | 対策 |
|---|---|---|
| R-T01 | Rust Edition 2024 の axum / sqlx の互換性破壊変更 | `Cargo.lock` で全依存を固定する。マイナーバージョンアップは `cargo test` の全通過を確認してから適用する。メジャーアップは ADR を作成する |
| R-T02 | React Native for Windows（react-native-windows）の保守停止 | Windows タブレット対応の代替手段として Web View フォールバック（React 管理 Web の転用）を設計余地として保持する。ADR に記録済み |
| R-T03 | SQLite と PostgreSQL のスキーマ乖離（同期エラー） | Detox E2E テストで同期フロー全体を毎回検証する。スキーマ変更は両 DB への同時適用を義務付ける。片方のみの変更を禁止する |
| R-T04 | 親機（基幹システム）の API 仕様変動（エンドポイント変更・スキーマ破壊的変更） | `external_key_binding` テーブルのマッピングDSLで変動を吸収する。OpenAPI スキーマ変更時は ADR を作成し、子機側アダプタを更新する。親機ごとのアダプタはプラガブル設計とする |
| R-T05 | マスタ同期失敗時の業務影響（親機停止・ネットワーク断による同期不能） | 最後の同期成功時刻からの縮退稼働を設計する。縮退中は「マスタが古い可能性あり」を UI で明示し、人手確認を促す。同期失敗は監査ログとして記録し、ALCOA+ Contemporaneous 原則への影響を追跡できる状態を維持する |

### 規制リスク

| ID | リスク内容 | 対策 |
|---|---|---|
| R-R01 | 社内監査で「ALCOA+ 準拠が不十分」と判断される | 06 章の ALCOA+ マトリクスを事前に内部監査ツールとして使用する。M6 リリース前の自己監査を必須とする |
| R-R02 | 計量法改正・JCSS 制度変更による校正トレーサビリティ要件の変更 | 校正参照モデル（計測器・校正証明書番号・有効期限）を設定値で切替可能に設計する。ハードコーディングを禁止する |

### 組織リスク

| ID | リスク内容 | 対策 |
|---|---|---|
| R-O01 | IT 担当 1 名の離職・長期休暇（Bus Factor = 1） | ドキュメント駆動設計・10 章の自己引継ぎ設計・ADR の蓄積により、引継ぎコストを最小化する。ドキュメントのみで環境再構築が可能な設計を M6 判定条件に含める |
| R-O02 | 現場作業員のデジタル化拒否・抵抗 | 紙フォールバックを正式な縮退モードとして設計し（09 章 5 節）、デジタル化を強制しない移行設計を採用する。習熟度別 UI の段階的移行を許容する |
| R-O03 | 品質担当が SOP 改訂フローのボトルネックになる | SOP ドラフト段階での自動通知・クローズ遅延アラート（09 章 3 節 健全性指標）で遅延を検知する。SOP 改訂権限の委任手順をガイドラインに含める |

### 倫理リスク

| ID | リスク内容 | 対策 |
|---|---|---|
| R-E01 | 経営層が「個人別生産性ランキング」機能の追加を要求する | 01 章の倫理境界宣言・08 章の倫理品質設計に基づき、要求を設計上の制約として説明する。RBAC により技術的に実装不可能な状態を維持する。要求への応答プロセスは「機能は追加しない・集計値での代替を提示する」とする |
| R-E02 | 外国人労働者が適切な説明なしにデータ収集に同意させられる | 導入時の「データ利用ポリシー同意書」多言語版（日本語・英語・その他対応言語）の署名を必須とする（09 章 6 節）。多言語プライバシー通知をアプリ初回起動時に表示する（08 章 7 節） |

---

**本節で確定した方針**
- リスクレジスタを技術・規制・組織・倫理の 4 カテゴリで管理し、各リスクに対策を紐付けて確定する。
- R-E01（個人別ランキング要求）への対応方針を「機能追加しない・RBAC で技術的に禁止する」として確定する。
- R-O01（Bus Factor = 1）の対策としてドキュメント駆動・自己引継ぎ設計を 10 章と連動させる。

## 4. 将来オプション（前提変更時の対応余地）

### 将来オプションの位置づけ

以下のオプションは「ver1.0.0 が達成する範囲ではない」と明確に宣言する。同時に、本書の設計を将来の適用拡張に対して不必要に閉じない範囲で設計余地を確保する。将来オプションの発動は前提条件の変更を起点とする。

| オプション | 発動条件 | 設計上の余地 |
|---|---|---|
| マルチサイト展開 | P01 の変更（複数工場管理が必要になる） | データモデルに `factory_id` / `site_id` を予約フィールドとして持つ。ver1.0.0 では単一工場として固定値を使用する |
| 規制業界拡張（GMP・IATF 16949） | P06 の変更（規制適用業種への展開） | ALCOA+ コアは実装済み。電子署名フロー（21 CFR Part 11 準拠）・保管期間設定を設定値で切替可能に設計する |
| SaaS 化 | P02 の変更（インターネット公開） | REST API / Webhook は整備済み。マルチテナント・外部認証（OAuth2/OIDC）は ver1.0.0 では未実装。追加実装コストが発生する |
| SI 調達・チーム開発への移行 | 開発体制変更（個人開発の終了） | ドキュメント駆動設計・ADR・型定義・テストにより、チーム開発への移行を可能にする。移行コストの最小化を設計目標に含める |
| AI/ML 支援機能 | 技術・倫理・規制の検証完了後（ver1.0.0 の対象外） | `pgvector` 拡張は PostgreSQL で利用可能だが、ver1.0.0 では有効化しない。追加時は倫理品質の再監査を必須とする |
| 子機モード双方向同期の拡張 | 単方向 READ-ONLY 同期のみから、実績送信（子機→親機）への拡張が必要になった場合 | `external_key_binding` テーブル・Outbox Pattern は ver1.0.0 で設計済み。親機側 API 仕様が整った時点で追加実装する。なお READ-ONLY 同期（親機→子機マスタ受信）は ver1.0.0 で提供済み |

---

**本節で確定した方針**
- 将来オプション 5 件を前提変更条件と設計余地として確定するが、いずれも ver1.0.0 の実装対象外とする。
- `factory_id` / `site_id` の予約フィールドをデータモデルに含め、マルチサイト展開の余地を確保する。
- AI/ML 機能の追加時は倫理品質の再監査を必須として本書に記録する。

## 5. 本書が約束しないことの再宣言

### 宣言の目的

本節は、本書を読んだステークホルダーが「このシステムに含まれている」と誤解しうる機能・準拠・連携を明示的に除外し、期待値を正確に設定するためのものである。

### 本書（システム化計画書）が約束しないことの一覧

| 約束しないこと | 理由 |
|---|---|
| ver1.0.0 時点で複数工場・SaaS・マルチテナンシーに対応する | P01（単一工場）・P02（社内 LAN）の前提条件に基づく。前提変更時は 4 節の将来オプションが発動する |
| GMP / FDA 21 CFR Part 11 / EU MDR / IATF 16949 / AS9100 / HACCP への適合 | 03 章の規制対象外判断。P06 の変更（規制適用業種への展開）が生じた場合は 4 節の規制業界拡張オプションが発動する |
| AI/ML による自動判断・異常検知・自然言語作業指示 | ROI 実証困難・ハルシネーションリスク・EU AI Act 対応コストを理由に 03 章で対象外と判断した。本書では言及しない |
| AR・スマートグラス・生体センサ・IoT センサとの直接統合 | OT-IT 境界原則（05 章）・個人開発スコープの超過 |
| PLC / OPC-UA / OT センサとの直接統合 | OT 直結は永続非提供（IT/OT 境界の意図的保護・機能安全設計責任（IEC 61508）の範囲超過）。構想 14 章で確定する |
| 任意のレガシー独自プロトコルのみを持つ基幹システムとのリアルタイム連携 | 標準 REST / Webhook アダプタを持たない基幹システムはアダプタ実装を保証しない（個別 SIer 対応余地）。ERP / MES / 生産管理との標準 REST 連携は 12 章の子機モードとして ver1.0.0 で提供する |
| 労働者の生産性を個人単位で評価するためのレポーティング機能 | 01 章の倫理境界宣言・08 章の倫理品質設計（目的限定性・データ最小化）。RBAC で技術的に実装を禁止する |
| 法令・規制の解釈責任の代行 | 法的専門性の範囲外。規制への適合可否の最終判断は導入企業の法務・品質担当に帰属する |

**本節で確定した方針**
- 本書が約束しないこと 8 項目を理由とともに再宣言し、ステークホルダーの期待値を確定する。
- AI/ML・生産性個人評価・レガシー連携の 3 項目は理由が本質的であり、要求への応答も「追加しない」で確定する。
- 法令・規制解釈の最終責任は導入企業に帰属することを明示し、本システムが代行しないことを確定する。
- ERP / MES / 生産管理との REST API 経由の論理連携は 12 章（子機モード）として ver1.0.0 で正式提供することを確定する。OT 直結（PLC / OPC-UA）との分離を本改訂（2026-05-17）として記録する。
