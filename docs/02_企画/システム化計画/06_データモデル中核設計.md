# 06 データモデル中核設計

本章の責務は、本システムのドメイン中核モデル「3 階層手順 × 4M+1E × イベント」を概念レベルで確定することである。テーブル定義・カラム型・インデックス設計といった実装詳細は詳細設計書に委ねる。本章は ALCOA+ 9 原則の展開における権威章であり、他章は「06章を参照」として本章を引用する。

データモデル全体像については、図: fig_data_core_model.svg（img/ 配下）を参照すること。

## 1. データモデルの設計原則と位置づけ

### 設計が満たすべき要件の宣言

本システムの二大価値は Navigation（EWI・状況認識支援）と Traceability（証拠生成）である。この二価値は相互に補強し合う不可分の関係にあり、データモデルはその両方を単一の設計で実現する。Navigation のために手順をリアルタイムに提示するデータ構造と、Traceability のために不変の証拠を生成するイベント構造は、同一のモデルから派生する。別系統のデータ設計を持つことは禁止する。

ALCOA+ 9 原則（Attributable / Legible / Contemporaneous / Original / Accurate / Complete / Consistent / Enduring / Available）は、規制要件としての遵守にとどまらず、モデルの構造的制約として実装する。原則ごとにどのモデル要素が担責するかを 6 節で明示する。

プロセスマイニングへの準拠は van der Aalst の定式化に従う。すべての作業イベントは Case ID（作業指示 ID）・Activity（ステップ ID + イベント種別）・Timestamp（UTC ミリ秒）・Resource（作業者 ID + 端末 ID）の 4 属性を必須とし、XES（eXtensible Event Stream）国際標準との互換性を構造的に担保する。

マスタ（SOP・ユーザー・設備・計測器・ロット）とトランザクション（作業ログ・イベント・不適合・CAPA）は厳格に分離する。マスタは参照対象であり、トランザクションは記録対象である。この二分類はデータベースのロール権限設計にまで及ぶ。

### 設計の 3 根拠原則

**原則 A: 書いたことは消えない**

作業ログテーブルへの DELETE および UPDATE を禁止する。Append-only Event Sourcing を採用し、修正が必要な場合は「訂正イベント」を新規追記する差分レコード方式を採る。この制約はアプリケーション層の実装規約ではなく、PostgreSQL のロール権限として実施する。

**原則 B: 「何が・誰が・いつ・どこで・なぜ」を全イベントに必須属性として持つ**

イベントの必須属性は event_id・case_id・activity・timestamp_device・timestamp_server・worker_id・terminal_id・sop_version_id の 8 フィールドとする。いずれも NULL を禁止する。これらが欠けたイベントは ALCOA+ の Attributable および Contemporaneous 原則に違反するため、バックエンドが受付を拒否する。

**原則 C: マスタの版は「時点参照」で固定する**

作業ログが参照したマスタの版は、後からマスタが変更されても変わらない。実績レコードは記録時点の sop_version_id・instrument_calibration_ref・worker_skill_snapshot を必ず参照し、マスタ変更の遡及を構造的に防止する。

---

**本節で確定した方針**
- Navigation と Traceability の二大価値を単一のデータモデルで実現し、別系統設計を禁止する。
- ALCOA+ 9 原則をモデルの構造的制約として実装し、規制遵守を設計で保証する。
- Append-only Event Sourcing・全イベント必須 8 属性・時点参照固定を設計の 3 根拠原則として確定する。

## 2. 3 階層手順モデル（SOP マスタ）

### 階層の定義

本システムの手順体系は JIS Z 8141:2001 の作業階層（工程・作業・要素作業）に基づき、以下の 3 層で構成する。

| 層 | 名称 | 粒度の例 | 数量感 |
|---|---|---|---|
| L1 | 工程（Process） | 溶接工程・検査工程・組立工程 | 1 工場あたり数〜数十 |
| L2 | 作業（Operation） | 溶接準備・本溶接・後処理・溶接検査 | 1 工程あたり数〜十数 |
| L3 | Step | 治具セット確認・電流設定確認・ビード外観確認 | 1 作業あたり数〜二十数 |

L3 Step は Navigation の最小実行単位であると同時に、Traceability の最小記録単位でもある。Navigation とトレサビの粒度は同一に揃える。

### Step の必須属性

各 Step は以下の属性を必ず持つ。

| 属性名 | 型・制約 | 役割 |
|---|---|---|
| step_id | UUID、NULL 禁止 | 永続識別子。変更不可 |
| step_number | 整数、NULL 禁止 | 作業内表示順 |
| instruction_text | 多言語 JSONB: `{"ja":"...","en":"...","vi":"..."}` | 手順提示テキスト。多言語対応 |
| input_type | Enum: checkbox / numeric / photo / qr_scan / signature / text | 記録入力の型。列挙型で管理し自由定義を禁止 |
| judgment_condition | 数値型の場合: USL / LSL / 選択肢型の場合: 正誤・選択肢リスト | 合否判定条件。SOP マスタが保持 |
| evidence_required | Boolean | true の場合「証拠付き完了」を強制する。写真・スキャン値の添付なしに step_completed イベントを受け付けない |
| fmea_rpn_flag | Boolean | true の場合 FMEA 高 RPN ステップとして強化チェックを発動する |
| skill_level_required | Enum: L1 / L2 / L3 / L4 | 実施に必要な最低スキルレベル。不足する場合はシステムが入力をブロックする |
| expected_unit | UCUM コード（文字列） | 数値入力時の期待単位。入力単位との不一致を即時警告する |
| media_refs | URL リスト（JSONB） | 参照写真・参照動画の URL |
| tips_refs | UUID リスト（JSONB） | Tacit Tips レイヤへの参照 |

### SOP 版管理

SOP は版（Version）単位で管理する。版管理レコードは以下の属性を持つ。

| 属性名 | 型・制約 | 役割 |
|---|---|---|
| sop_version_id | UUID、NULL 禁止 | 版の永続識別子 |
| version_number | 整数、NULL 禁止 | 版番号（単調増加） |
| effective_date | 日付、NULL 禁止 | 有効開始日 |
| revision_reason | テキスト、NULL 禁止 | 改訂理由。空文字禁止 |
| approved_by | ユーザー ID、NULL 禁止 | 承認者 ID |
| invalidation_date | 日付、NULL 許容 | 廃止日。null の場合は現行版 |

一度承認・有効化された版は変更不可とする。内容の変更が必要な場合は新版を作成する。旧版は廃止フラグを立てるが物理削除しない。

### Effective Date とロット連動

特定ロット以降から新版 SOP を適用する「ロット切替設定」をサポートする。ロット番号と effective_date の両方を版切替のトリガーとして用いることができる。

実績レコードは記録時点の sop_version_id を必ず参照する（時点参照の固定）。作業指示を発行した時点の有効版が、その指示に紐づく全イベントに固定される。版切替後の新版は、新たな作業指示から適用される。進行中の作業指示は旧版で完了させる。

業界分析 [`90_業界分析/01_作業の定義と分類.md`](../../90_業界分析/01_作業の定義と分類.md) と [`90_業界分析/25_作業指示書とSOPの構造化・表現論.md`](../../90_業界分析/25_作業指示書とSOPの構造化・表現論.md) を引用した。

---

**本節で確定した方針**
- 手順体系を工程・作業・Step の 3 層で構成し、Step を Navigation とトレサビの共通最小粒度として確定する。
- Step に fmea_rpn_flag・skill_level_required・expected_unit を必須属性として持たせ、安全・適格性・単位の整合性をモデルで保証する。
- 承認済み SOP 版は変更不可とし、改訂は新版作成のみを認める。実績は記録時点の版 ID に時点固定する。

## 3. 4M+1E 参照軸（トレサビ軸）

### 参照軸の定義

4M（Man・Machine・Material・Method）に Environment と Measurement を加えた 6 軸を、トレサビ記録の「誰が・何で・何を・どう・どこで」を担う参照キーとして定義する。

| 軸 | 定義 | マスタ名 | トレサビ記録への参照キー |
|---|---|---|---|
| Man（人） | 作業者・承認者・オペレーター | user_master | worker_id / approver_id |
| Machine（設備） | 生産設備・検査設備・治具 | equipment_master | equipment_id |
| Material（材料） | 部品・原材料・製品ロット | lot_master | lot_id / sub_lot_ids |
| Method（方法） | SOP 版・作業指示番号 | sop_master | sop_version_id / work_order_id |
| Environment（環境） | 温度・湿度・清浄度クラス | *(任意付加フィールド)* | env_snapshot（JSONB） |
| Measurement（計測） | 計測器・校正証明書・不確かさ | instrument_master | instrument_id / calibration_ref |

Environment は構造化マスタを持たず、イベントの payload に env_snapshot（JSONB）として付加する。これは製造環境の多様性と中小製造業の設備実態に基づく設計判断である。

Measurement は 5M+1E の「Measurement」として独立した参照軸に位置づける。計測値の証拠品質は計測器マスタと校正証明書マスタへの参照によって担保する。詳細は 7 節で展開する。

### Man 軸への責任過収束防止

4M 不適合分析において「人のミス」への過収束は根本原因分析を阻害する。本システムは不適合原因コードで「Man」を単独で選択させる UI を設けない。不適合起票時は Man・Machine・Material・Method・Environment の 5 軸を等価な選択肢として提示し、複数選択を許容する。これにより根本原因分析の偏りをシステム設計で抑制する。

### マスタの廃止管理

全マスタは物理削除を禁止する。廃止が必要な場合は廃止フラグ（is_active: false）で無効化する。廃止マスタへの参照を持つトランザクション（作業ログ・不適合レコード等）は正常に存在し続ける。廃止後も参照可能であることが時点参照の要件である。

業界分析 [`90_業界分析/01_作業の定義と分類.md`](../../90_業界分析/01_作業の定義と分類.md) と [`90_業界分析/06_品質管理とトレーサビリティ.md`](../../90_業界分析/06_品質管理とトレーサビリティ.md) を引用した。

---

**本節で確定した方針**
- 4M に Environment と Measurement を加えた 6 軸をトレサビの参照軸として確定し、全軸をイベントに必須参照させる。
- Man 軸への責任過収束を防ぐため、不適合起票 UI で 5 軸を等価提示し単独選択を設けない。
- 全マスタは物理削除禁止・廃止フラグ管理とし、廃止後もトランザクションからの参照を保証する。

## 4. 横串参照モデル（コンテキスト情報の統合）

### 横串参照の定義

作業ログの各レコードは、記録の「コンテキスト」として以下の横串情報を参照する。

| 横串情報 | 参照キー | 役割 |
|---|---|---|
| SOP 版 | sop_version_id（時点固定） | 作業時に適用した手順の証明。後から版が変わっても参照値は変わらない |
| ロット（材料） | lot_id, sub_lot_ids | 使用材料・部品の追跡。順方向・逆方向トレサビの基盤 |
| 校正情報 | instrument_id + calibration_ref | 計測値の証拠品質確認。校正の有効性を記録時点で固定する |
| スキルレベル | worker_id + skill_level_snapshot | 作業者適格性の証明。記録時点のスキルレベルをスナップショットとして保持 |
| シフト情報 | shift_type（日勤・夜勤等）、consecutive_nights（連続夜勤日数） | 疲労相関分析の基盤。不適合・ヒューマンエラーとの相関分析に使用 |
| 4M 変更 | change_id（4M 変更承認レコードへの参照） | 手順改訂前後の記録分離。4M 変更前後で集計を分割可能にする |
| QC 活動 | kaizen_teian_id, nonconformance_id（任意） | 不適合・改善とログの相互参照。28章 CAPA ループとの接続点 |

### 時点参照の不変性

横串参照先が後から変更されても、作業ログの参照値（sop_version_id・calibration_ref 等）は変わらない。これが時点参照の本質的制約である。

- sop_version_id は作業指示発行時点の有効版 ID を記録時に確定させる
- skill_level_snapshot は step_completed イベント時点の作業者スキルレベルを JSONB でコピー保持する
- calibration_ref は計測値入力時点の有効校正証明書 ID を参照として固定する

横串参照先マスタの廃止は廃止フラグで行い、参照先レコードを削除しない。廃止マスタへの参照を持つ過去イベントは監査上有効な記録として存在し続ける。

### スキル・資格の横串参照

作業者マスタは工程 ID × 作業者 ID × スキルレベル（L1〜L4 の 4 段階ルーブリック）× 取得日 × 有効期限の構造を持つ。Step の skill_level_required と作業者の現在のスキルレベルを比較し、不足する場合はシステムが入力をブロックする。

スキル有効期限の自動検出条件は以下の 3 条件とする。

- 有効期限が当日以前に到達した場合
- 対象手順の SOP 版が X 版以上更新された場合（X は工程マスタに設定）
- 最終実施日から N 日経過した場合（N は工程マスタに設定）

業界分析 [`90_業界分析/26_多能工とスキル管理・作業者資格.md`](../../90_業界分析/26_多能工とスキル管理・作業者資格.md) と [`90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md`](../../90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md) を引用した。

---

**本節で確定した方針**
- 作業ログは SOP 版・ロット・校正情報・スキルスナップショット・シフト情報・4M 変更・QC 活動の 7 種の横串情報を参照する構造を確定する。
- 横串参照値はすべて記録時点に固定し、参照先の変更が過去記録を遡及しないことを設計で保証する。
- スキル有効期限の自動検出を 3 条件（有効期限・版更新・最終実施日）で定義し、システムによるブロックを確定する。

## 5. イベントストア構造

### 設計の基本方針

すべての作業状態変化を「不変イベント」として記録する。現在の状態（工程の進捗・作業の完了状態・最新の測定値）はイベントの累積から導出する。イベントは一度書き込まれたら変更・削除できない。

### イベント種別の定義

| イベント種別 | 意味 | 付加必須情報 |
|---|---|---|
| step_started | ステップ実行開始 | なし |
| step_completed | ステップ完了（記録値・署名付き） | 記録値（input_type に応じた payload） |
| step_skipped | ステップスキップ | スキップ理由（自由テキスト、NULL 禁止） |
| step_rejected | ステップ差し戻し | 差し戻し理由（NULL 禁止）・差し戻し実施者 ID |
| work_interrupted | 作業中断 | 中断理由・中断時のステップ位置 |
| work_resumed | 作業再開 | なし |
| andon_triggered | アンドン発報 | 発報種別・緊急度 |
| nonconformance_filed | 不適合起票 | 不適合レコード ID への参照 |
| kaizen_filed | Kaizen Teian 起票 | 提案レコード ID への参照 |

スキップ・差し戻し・中断をすべて別イベントとして記録することで、ALCOA+ の Complete 原則を構造的に満たす。「何も記録されていない」と「スキップが記録された」は明確に異なる状態として区別する。

### 各イベントの必須属性（XES 互換）

```
event_id:          UUID（Idempotency Key も兼用）
case_id:           作業指示 ID（Case ID、プロセスマイニング用）
activity:          ステップ ID + イベント種別（Activity）
timestamp_device:  端末タイムスタンプ（UTC ミリ秒、申告時刻）
timestamp_server:  サーバー受信タイムスタンプ（UTC ミリ秒、証拠時刻）
worker_id:         作業者 ID（Resource）
terminal_id:       端末 ID（Resource 補足）
sop_version_id:    適用 SOP 版 ID（時点参照）
payload:           イベント種別固有のデータ（JSONB）
  ├─ numeric_value:   測定値 (NUMERIC(18,6))
  ├─ unit_code:       SI 単位 (UCUM コード)
  ├─ uncertainty_u:   標準不確かさ
  ├─ uncertainty_k:   包含係数
  ├─ calibration_ref: 校正証明書 ID
  ├─ photo_hash:      SHA-256（写真ハッシュ）
  ├─ photo_path:      UUID パス（改竄検証用）
  └─ scan_value:      バーコード/QR スキャン値
is_offline:        オフライン記録フラグ（Boolean）
sync_lag_ms:       同期遅延時間（ミリ秒、監査用）
```

event_id は Idempotency Key を兼ねる。オフライン中に生成されたイベントが同期時に重複送信された場合、event_id の一致でサーバーが重複を検知し、2 回目以降の挿入を無視する。

### 不変性の保証

イベントストアの不変性は以下の 3 層で保証する。

**第 1 層: データベース権限**

PostgreSQL の作業ログテーブルには UPDATE・DELETE 権限を持つロールを作成しない。アプリケーションサービスが接続するデータベースユーザーには INSERT および SELECT のみを付与する。

**第 2 層: 訂正イベント方式**

記録の誤りが判明した場合、元イベントへの UPDATE は行わない。訂正イベント（event_type: correction）を新規追記し、元イベントに `corrected_by: [訂正イベントの event_id]` を差分レコードとして追記する。訂正イベントには訂正理由・訂正者 ID・承認者 ID を必須とする。

**第 3 層: ハッシュチェーン**

前レコードの SHA-256（event_id + timestamp_server）を `prev_hash` フィールドに保持する。定期バッチがチェーン全体を検証し、改竄を検知する。改竄が検知された場合は管理者にアラートを送信する。

業界分析 [`90_業界分析/21_作業ログ分析とプロセスマイニング.md`](../../90_業界分析/21_作業ログ分析とプロセスマイニング.md) と [`90_業界分析/22_規制別トレーサビリティ要件詳論.md`](../../90_業界分析/22_規制別トレーサビリティ要件詳論.md) を引用した。

---

**本節で確定した方針**
- 9 種のイベント種別を定義し、スキップ・差し戻し・中断を含む全状態変化を別イベントとして記録することを確定する。
- 不変性の保証を DB 権限・訂正イベント方式・ハッシュチェーンの 3 層で実施し、物理的な UPDATE/DELETE を禁止する。
- event_id を Idempotency Key として兼用し、オフライン同期時の重複挿入をサーバー側で無害化する。

## 6. ALCOA+ 9 原則のデータモデル・UI・運用への展開

ALCOA+ 9 原則の各要件を、データモデル層・UI 層・運用層の 3 層で担責させる。本節が ALCOA+ 展開の権威章であり、他章はこの表を参照する。

図: fig_alcoa_mapping.svg（img/ 配下）を参照すること。

| ALCOA+ 原則 | データモデル層の実装 | UI 層の実装 | 運用層の実装 |
|---|---|---|---|
| **Attributable**（帰属可能） | worker_id + terminal_id を全イベントの必須フィールドとする。NULL を禁止する。事後入力は `is_retroactive: true` + 承認者署名を必須とする | バーコード/PIN/NFC による作業者識別。匿名操作を許容しない。共有端末はシフト開始時に作業者が必ず認証する | RBAC で役割ごとのアクセス権限を設定する。共有アカウントを禁止するポリシーを運用規程に明記する |
| **Legible**（読取可能） | JSONB の多言語テキストを UTF-8 で保存する。バイナリは UUID パスで参照し、バイナリデータを直接 DB に格納しない | 手順テキストは最小フォント 14pt・コントラスト比 4.5:1 以上を必須とする。印刷時も可読なレイアウトを維持する | 長期保存は UTF-8 エンコーディングで文字化けを防止する。PDF 出力は ISO 32000 に準拠する |
| **Contemporaneous**（同時記録） | timestamp_device（端末時刻）と timestamp_server（サーバー受信時刻）の二重保存を必須とする。is_offline フラグで未同期を区別する | ステップ完了後すぐに記録入力を促すロックステップ方式を採る。「後でまとめて入力」を許容しない | 未同期レコードを「記録済（未同期）」バッジで可視化する。監査者向けに sync_lag_ms を開示する |
| **Original**（元データ） | Append-only: UPDATE/DELETE を禁止する。修正は訂正イベントを追記する。写真は SHA-256 ハッシュで改竄を検知する | 完了済みステップの「編集」ボタンは UI に存在しない。修正は「訂正申請」フローから行う | 正本は DB のイベントレコードとする。印刷・PDF は「参照用複製」として位置づける |
| **Accurate**（正確） | 数値フィールドは NUMERIC(18,6) とする。USL/LSL を SOP に保持し即時判定する。単位は UCUM コードで型安全に管理する | バーコード/QR による自動スキャン値取得を優先する。手入力には単位・桁数のバリデーションを実施する。機器マスタの分解能で有効桁数を自動制限し False Precision を防止する | JCSS 校正証明書の登録と有効期限チェックを計測前プレフライトで実施する |
| **Complete**（完全） | Step 完了をすべてイベントとして記録する。スキップ・中断・差し戻しも別イベントで記録し、除外しない | 全ステップ完了まで「作業完了」ボタンを無効化する。スキップには理由入力を強制する | SOP の step_count と記録された step_completed イベント数の一致チェックを自動実行する |
| **Consistent**（一貫） | sop_version_id を作業ログに時点参照する。マスタ変更が実績データを遡及しない | 同一工程を複数端末で実行する場合、同一 SOP 版を参照することをサーバー側で保証する | 版切替前の進行中作業はそのまま旧版で完了させる。新版は次作業指示から適用する |
| **Enduring**（耐久） | PostgreSQL + WAL アーカイブ + NAS バックアップの三層ストレージを採用する。最低保管期間を業種設定から参照する | 長期保存データには「保存期間 X 年」を記録メタデータに付与して可視化する | 定期バックアップの完了確認・リストアテストの年次実施を運用規程に明記する（09章参照） |
| **Available**（利用可能） | 管理 Web から順方向・逆方向トレサビクエリを提供する。CSV / XES / PDF エクスポート機能を実装する | 検索・フィルタは全項目対応とする。ロード時間は Glanceable UI 基準（1 秒以内）を目標とする | Active-Standby 二重化（07章参照）で RTO 1 時間・RPO 15 分を目標とする |

## 7. 測定値と校正参照モデル

### 計量値フィールドの構造

計測器によって得られる測定値は、測定値単体ではなく「測定値 + 不確かさ + 計測器 ID + 校正証明書 ID」のセットで記録する。測定値のみを記録し証拠品質情報を省略する設計は、ALCOA+ の Accurate 原則に違反する。

測定値レコードの必須フィールドは以下のとおりである。

```
measured_value:    NUMERIC(18,6) — 測定値（SI 基本単位で保存）
display_value:     NUMERIC — 表示用（換算後の値）
unit_code:         TEXT — UCUM コード（例: "mm", "kPa", "Cel"）
display_unit:      TEXT — 表示単位（例: "mm", "MPa", "℃"）
uncertainty_u:     NUMERIC — 標準不確かさ u (k=1)
uncertainty_k:     NUMERIC DEFAULT 2 — 包含係数（k=2 → 信頼水準 95%）
instrument_id:     UUID — 使用計測器への参照（NULL 禁止）
calibration_ref:   UUID — 使用した校正証明書レコードへの参照（NULL 禁止）
```

measured_value は SI 基本単位で保存し、display_value で表示単位に換算する。これにより単位系の混在による比較・集計エラーを防止する。

### 校正期限チェック（プレフライト）

計測器マスタには次回校正期限（next_calibration_date）を保持する。計測値入力前に以下のプレフライトチェックを実施する。

1. 使用計測器の next_calibration_date が当日以前であれば入力をブロックする
2. ブロック時は管理者に警告を送信する
3. ブロック強度は業種設定で「ハード（入力不可）」または「ソフト（警告のみ）」を切替可能とする

ハードブロックを選択した業種・顧客向けでは、校正切れ計測器による測定値の記録をシステムが完全に防止する。これにより証拠品質の確保をシステムで保証する。

### 単位不一致検知

SOP の各 Step には expected_unit（UCUM コード）を設定する。数値入力時に入力値の unit_code と expected_unit を比較し、以下の制御を行う。

- 同一単位: そのまま受け付ける
- 換算可能な異なる単位（例: mm vs cm）: 換算値と警告を表示し、確認後に受け付ける
- 換算不可能な単位（例: kgf vs N は次元が同一でも換算係数が不確定な場合がある）: 入力をブロックし、単位の修正を要求する

機器マスタの分解能フィールドで有効桁数を自動制限し、False Precision（計測器の分解能を超えた桁数の記録）を防止する。

### Cp / Cpk 即時判定

SOP の Step に USL（規格上限値）と LSL（規格下限値）を保持する。数値が入力された時点で Cp / Cpk を即時算出し、画面に表示する。管理図へのプロット点はイベントの payload に含め、集計クエリで管理図データを再構成する。

業界分析 [`90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md`](../../90_業界分析/33_計量法・JCSS校正トレーサビリティとSI単位.md) と [`90_業界分析/11_計測・工程能力と統計的品質工学.md`](../../90_業界分析/11_計測・工程能力と統計的品質工学.md) を引用した。

---

**本節で確定した方針**
- 測定値は measured_value + uncertainty_u + uncertainty_k + instrument_id + calibration_ref の 5 セットで記録することを必須とし、測定値単体の記録を禁止する。
- 校正期限プレフライトをハード/ソフトの 2 段階で実装し、ブロック強度を業種設定から切替可能とする。
- SOP の expected_unit と入力値の unit_code を比較し、換算不可能な単位不一致は入力をブロックする。

## 8. マスタとトランザクションの分離原則

### マスタデータの定義

マスタデータはシステムの参照基準となる不変的定義情報である。マスタデータの権威はサーバーが持ち、端末はサーバーから同期した読み取り専用のキャッシュを保持する。マスタの新規追加・変更は接続時のみ許可する。

| マスタ名 | 内容 |
|---|---|
| SOP マスタ | 工程・作業・Step・版管理 |
| ユーザーマスタ | 作業者・スキル・資格・有効期限 |
| 設備マスタ | 生産設備・検査設備・治具・点検周期 |
| 計測器マスタ | 計測器・校正証明書・校正期限・分解能 |
| ロットマスタ | 材料ロット・部品ロット・製品ロット |

### トランザクションデータの定義

トランザクションデータは作業の実施・判定・問題発生を記録した時系列データである。記録の権威は最初に記録したデバイスが持ち、サーバー同期後はサーバーが正本となる。

| トランザクション名 | 内容 |
|---|---|
| 作業イベントログ | Append-only Event Sourcing。全作業イベントの不変記録 |
| 不適合レコード | 不適合の起票・原因分析・処置記録 |
| CAPA レコード | 是正処置・予防処置・効果確認記録 |
| Kaizen Teian レコード | 改善提案・採否・実施記録 |
| 中断・再開ログ | 作業中断理由・再開記録 |

### 分離の 3 原則

**分離原則 1: マスタ変更はトランザクションを遡及しない**

マスタが変更されても、既存トランザクションが参照している値は変わらない。SOP が改訂されても、旧版を参照する作業ログの sop_version_id は旧版のままである。

**分離原則 2: トランザクションはマスタの「版 ID」を時点参照する**

トランザクションはマスタの「最新版」ではなく「特定の版 ID」を参照する。版 ID による時点参照が過去記録の不変性を保証する機構である。

**分離原則 3: 廃止マスタへの参照を持つトランザクションは正常に存在する**

廃止したユーザー・設備・計測器を参照する過去の作業ログは、廃止後も監査上有効な記録として存在し続ける。廃止マスタを参照するトランザクションに対して「参照先が廃止されている」という警告を表示するが、レコードを無効化しない。

### 保全作業の同一フレーム統合

保全作業（設備点検・予防保全・事後保全）は「作業手順 + 実績記録」の同一フレームに乗せる。保全作業専用のデータ構造は作成しない。設備マスタを持ち、保全作業の各イベントから equipment_id を必須参照させる。これにより通常生産作業と保全作業のトレサビ記録を同一のクエリで横断できる。

業界分析 [`90_業界分析/06_品質管理とトレーサビリティ.md`](../../90_業界分析/06_品質管理とトレーサビリティ.md) と [`90_業界分析/28_不適合と手順改訂のフィードバックループ.md`](../../90_業界分析/28_不適合と手順改訂のフィードバックループ.md) を引用した。

---

**本節で確定した方針**
- マスタ（権威: サーバー）とトランザクション（権威: 最初のデバイス→サーバー同期後はサーバー）を厳格に分離し、役割と権威を確定する。
- マスタ変更のトランザクション遡及禁止・版 ID 時点参照・廃止マスタ参照の保全の 3 分離原則を確定する。
- 保全作業を通常作業と同一フレームに統合し、専用構造を禁止する。

## 9. 外部一意キーマッピングと作業パターン

### 位置づけ

本節は、子機モード（計画 12 章）において親機（基幹システム: ERP / MES / 生産管理パッケージ等）の外部一意キーと、本アプリの内部エンティティ（SOP・作業パターン）を紐付けるマッピング層を概念設計として確定する。

親機は型式・機番・品目コード等の一意識別属性の組み合わせ（外部一意キー）を用いて対象製品・部品を識別する。子機はこの外部一意キーを受け取り、対応する内部 SOP・作業パターンを決定しなければならない。外部一意キーマッピング（`external_key_binding`）はその橋渡し層として機能し、マスタ vs トランザクション分離原則（8 節）においてマスタとして位置づける。

### `external_key_binding` テーブルの概念設計

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `external_key_source` | VARCHAR | 親機の識別子（例: "SAP", "GENIOUS", "custom"） |
| `external_key_value` | JSONB | 親機から受け取った一意キーの表現（型式・機番・品目コード等の組み合わせを JSON で表現） |
| `work_pattern_id` | UUID | 本アプリ内の作業パターン ID（FK） |
| `valid_from` | TIMESTAMPTZ | 紐付けの有効開始日時（Effective Date と連動） |
| `valid_to` | TIMESTAMPTZ | 紐付けの有効終了日時（NULL = 現在有効） |
| `sync_status` | ENUM | 同期状態（synced / pending / conflict） |
| `factory_id` | VARCHAR | マルチサイト展開への予約フィールド（02章 6-1節参照） |
| `created_by` | UUID | ALCOA+ Attribution 原則（誰がマッピングを設定したか） |
| `created_at` | TIMESTAMPTZ | ALCOA+ Contemporaneous 原則 |

`external_key_value` を JSONB とするのは、親機ごとに一意キーの構成要素（型式・機番・品目コード等の組み合わせ）が異なるためである。固定カラム構造では親機の多様性に対応できない。

### `work_pattern` テーブルの概念設計

作業パターン（`work_pattern`）は外部一意キーに紐づく SOP 参照エンティティである。外部一意キーと SOP を直接結合せず、`work_pattern` を中間エンティティとして置くことで、同一 SOP に対して複数の外部一意キーを紐付ける構成や、作業パターン単位のバージョン管理を可能にする。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `name` | VARCHAR | 作業パターン名（内部管理用） |
| `description` | TEXT | 説明 |
| `sop_id` | UUID | 紐づく SOP の ID（FK）。バージョン管理と連動 |
| `version` | INTEGER | 作業パターンのバージョン |
| `factory_id` | VARCHAR | マルチサイト展開への予約フィールド |

### SOP との関係

`external_key_binding` と `work_pattern` と SOP の関係は以下のとおりである。

- 1 つの `work_pattern` は 1 つの SOP（工程/作業/Step の 3 階層、2 節参照）を参照する。
- 1 つの `external_key_binding` は 1 つの `work_pattern` を参照する。
- 1 つの外部一意キーに対して複数の `external_key_binding` レコードが存在しうる。これは、バージョン違いの SOP を有効期間（`valid_from` / `valid_to`）で切り替えるユースケースに対応するためである。
- Effective Date（SOP バージョン管理、2 節参照）と `valid_from` / `valid_to` の組み合わせにより、任意の時点において有効な作業パターンを一意決定する。

### ALCOA+ との整合

本テーブル設計は 6 節の ALCOA+ 展開方針と整合する。

- `created_by` により Attributable 原則を保証する。誰がマッピングを設定したかを記録し、匿名のマッピング設定を禁止する。
- `created_at` により Contemporaneous 原則を保証する。
- マッピングの変更が必要な場合は、旧レコードの `valid_to` に変更日時を設定し、新レコードを追加する Append-only 方式を採る。既存レコードへの UPDATE による上書きは禁止する。この制約は Append-only Event Sourcing の精神（設計原則 A、1 節参照）と一貫する。

### マスタ vs トランザクション分離における位置づけ

8 節の分離原則に基づき、本節のエンティティを以下のとおり分類する。

- `external_key_binding`: マスタ。親機から受信するマスタ情報（外部一意キー）と内部 SOP の紐付けを定義する参照基準情報であり、トランザクションの参照対象である。
- `work_pattern`: マスタ。SOP 参照エンティティとして同様に参照基準情報に位置づける。
- 作業実績（`work_result` 等）: トランザクション。`work_pattern_id` を通じて `external_key_binding` と接続される。作業実績は記録時点の `work_pattern_id` を時点参照し、その後のマッピング変更が過去実績を遡及しないことを保証する（分離原則 1・2、8 節参照）。

---

**本節で確定した方針**
- 外部一意キーマッピング（`external_key_binding`）と作業パターン（`work_pattern`）を子機モードにおける親機連携の概念設計として確定し、マスタに分類する。
- `external_key_value` を JSONB とし、親機ごとに異なる外部一意キーの構成（型式・機番・品目コード等の組み合わせ）に柔軟に対応する。
- `valid_from` / `valid_to` と SOP の Effective Date の組み合わせで任意時点の有効作業パターンを一意決定し、バージョン切替を構造的に管理する。
- マッピング変更は Append-only 方式（旧レコードの `valid_to` 設定 + 新レコード追加）のみを認め、既存レコードへの UPDATE を禁止する。
- `created_by` と `created_at` により ALCOA+ の Attributable・Contemporaneous 原則を保証する。

## 10. カスタム Step タイプ定義と条件分岐フロー

### 位置づけ

本節は、18 章（拡張可能 Step エンジン）で確定したカスタム Step タイプと条件分岐フローを、データモデルの観点で確定する。

### `step_type_definition` テーブル

カスタム Step タイプの定義マスタである。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `name` | VARCHAR | タイプ識別子（スネークケース、工場内一意） |
| `version` | VARCHAR | SemVer 形式（定義変更時は更新、旧定義は保持） |
| `label` | JSONB | 多言語ラベル（`{"ja": "...", "en": "..."}` 形式） |
| `schema` | JSONB | 入力フィールドのスキーマ定義（type / min / max / ui_hint 等） |
| `validators` | JSONB | JSON Logic ベースのバリデーションルール |
| `factory_id` | VARCHAR | スコープ（特定工場またはシステム全体共通） |
| `is_active` | BOOLEAN | 論理削除フラグ（削除しない、非活性化のみ） |
| `created_by` | UUID | ALCOA+ Attribution（誰が登録したか） |
| `created_at` | TIMESTAMPTZ | ALCOA+ Contemporaneous |

### `step_flow_rule` テーブル

SOP 内の条件分岐フロー定義である。

| カラム | 型 | 説明 |
|---|---|---|
| `id` | UUID | 主キー |
| `sop_id` | UUID | FK → SOP テーブル |
| `source_step_id` | UUID | 評価対象 Step（条件の入力値を持つ Step） |
| `condition` | JSONB | JSON Logic ベースの条件式（`{"var": "value"}` で source Step の記録値を参照） |
| `action` | ENUM | `skip` / `goto` / `insert` |
| `target_step_id` | UUID | アクション対象 Step の ID |
| `insert_position` | VARCHAR | `insert` 時のみ: `before_target` / `after_source` |
| `priority` | INTEGER | 同一 source_step に複数ルールがある場合の優先順位（小さいほど先に評価） |
| `created_by` | UUID | ALCOA+ Attribution |
| `created_at` | TIMESTAMPTZ | ALCOA+ Contemporaneous |

### エンティティ間の関係

- `step_type_definition` は SOP 内の Step に `step_type = <type_name>` として参照される（計画 03 章の SOP マスタメタデータ）。
- `step_flow_rule` は SOP に紐づく（`sop_id` FK）。ルールは SOP の有効期限と連動して管理する。
- `step_type_definition` の削除は禁止。`is_active = false` による非活性化のみ（過去の記録が参照するタイプ定義を消去しない）。

### ALCOA+ との整合

- `step_type_definition` と `step_flow_rule` は両テーブルとも `created_by` / `created_at` を持ち、Attributable・Contemporaneous を保証する。
- 既存の `step_flow_rule` の変更は禁止。変更時は新レコードを挿入し、旧ルールの `sop_id` を切り離す（Append-only の精神）。

### マスタ vs トランザクション分離における位置づけ

8 節の分離原則に基づき、本節のエンティティを以下のとおり分類する。

- `step_type_definition`: マスタ（工場・SOP の設計情報。参照されるが書き換えられない）。
- `step_flow_rule`: マスタ（SOP 設計情報の一部。Append-only で管理）。
- 実行時のスキップ記録（`step_skipped_by_rule` イベント）: トランザクション（Append-only イベントストアに格納）。

---

**本節で確定した方針**
- `step_type_definition` / `step_flow_rule` の概念テーブルを確定する。
- 両テーブルとも `created_by` / `created_at` で ALCOA+ の Attributable・Contemporaneous を保証することを確定する。
- `step_type_definition` の削除禁止・`is_active` による論理無効化のみを確定する。
- `step_flow_rule` の変更禁止・Append-only 管理を確定する。
