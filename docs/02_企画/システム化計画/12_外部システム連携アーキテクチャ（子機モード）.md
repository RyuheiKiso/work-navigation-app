## 2. 連携モードの定義

### 3 形態の確定

本アプリが対応する連携形態を以下の 3 モードに確定する。工場ごとに異なる IT インフラの実態を踏まえ、段階的な導入と移行を可能にする設計とする。

| 連携モード | 説明 | 想定する工場環境 |
|---|---|---|
| 単独動作モード | 基幹システムとの連携なし。本アプリ単体で完全機能 | ERP / MES 非保有の小規模工場 |
| 子機モード READ-ONLY | 親機からマスタを受信（マスタ同期）のみ。実績の送信はしない | 基幹システムはあるが、実績連携は不要な工場 |
| 子機モード 双方向 | マスタ同期 + 実績送信。親機・子機が相互に情報を交換 | 基幹システムがあり、実績の自動連携が必要な工場 |

### モードの切替と互換性

連携モードは設定ファイル（サーバー側のランタイム設定）による切替とする。アプリの再デプロイは不要である。切替は管理者権限を持つユーザーのみが実行できる（RBAC による制御）。

連携モードは **工場単位**（factory_id に紐づく）で設定する。複数工場を管理するテナントでは、工場ごとに異なる連携モードを持つことができる。

モード間の互換性については以下の方針を確定する。

| 移行方向 | データ損失の有無 | 追加作業 |
|---|---|---|
| 単独動作モード → 子機モード READ-ONLY | なし | 外部一意キーのマッピング設定が必要 |
| 単独動作モード → 子機モード 双方向 | なし | 外部一意キーのマッピング設定 + 実績送信先 API の設定が必要 |
| 子機モード READ-ONLY → 双方向 | なし | 実績送信先 API の設定追加のみ |
| 子機モード 双方向 → READ-ONLY | 送信キューの未送信実績が失われる可能性あり。切替前に Outbox 完全送信を確認する | Outbox の空きを管理 UI で確認 |

図: fig_integration_modes.svg（img/ 配下）を参照すること。

---

**本節で確定した方針**
- 連携モードを単独動作モード・子機モード READ-ONLY・子機モード 双方向の 3 形態に確定する。
- 連携モードは設定ファイルによる切替とし、アプリ再デプロイを不要とする。
- 連携モードは factory_id 単位で設定し、工場ごとに独立して管理する。
- 上位モードへの移行（単独→子機）ではデータ損失が発生しないことを設計で保証する。

## 3. 外部一意キー設計

### 外部一意キーとは

外部一意キーとは、型式（model_type）・機番（machine_number）・品目コード（item_code）等、**親機側でレコードを一意に識別する属性の組み合わせ**を指す。本アプリはこのキーを親機から受け取り、自身の SOP・作業パターンと紐付けるために使用する。

外部一意キーの特性を以下に確定する。

| 特性 | 説明 |
|---|---|
| 正は親機にある | 外部一意キーを定義するのは親機である。本アプリは受け取るだけであり、勝手に変更・追加しない |
| 親機ごとに構成が異なる | 型式のみの親機・型式 + 機番の組み合わせの親機・品目コードのみの親機など、構成は親機ごとに異なる |
| 文字列表現で吸収する | 属性名・構成の差異を本アプリ側では JSON 文字列形式に正規化して吸収する |
| 変更されることがある | 親機側のマスタ改訂により、外部一意キーの値が変わる場合がある。有効期間管理で対応する |

### `external_key_binding` テーブルの概念設計

親機の外部一意キーと本アプリ内部の作業パターンを紐付けるテーブルを `external_key_binding` として確定する。詳細な DDL は概要設計に委ねるが、概念的な構成は以下のとおりである。

| フィールド名 | 役割 | 備考 |
|---|---|---|
| `external_key_source` | 親機の識別子 | 例: "SAP", "GENIOUS", "custom" |
| `external_key_value` | 親機から受け取った一意キーの文字列表現 | JSON 形式も許容。例: `{"model_type":"A100","machine_number":"001"}` |
| `work_pattern_id` | 本アプリ内の作業パターン ID への参照 | NULL 禁止 |
| `valid_from` | 紐付けの有効開始日 | SOP バージョン管理との連動 |
| `valid_to` | 紐付けの有効終了日 | null の場合は現在有効 |
| `sync_status` | 同期状態 | synced / pending / conflict の 3 値 |
| `factory_id` | 工場 ID | 連携モード設定と合わせて factory_id 単位で管理 |

### 1 対多の許容

1 つの外部一意キーに複数の作業パターンを紐づけることを許容する。これは以下のケースに対応するためである。

- SOP バージョンが切り替わる時期に旧版・新版の両作業パターンが一時的に並立する場合
- 同一品目に対して複数の作業シナリオ（通常ライン・緊急ライン等）が存在する場合

有効期間（`valid_from` / `valid_to`）の組み合わせにより、任意の時点で適用される作業パターンは最大 1 件となるよう制約する。

図: fig_external_key_binding.svg（img/ 配下）を参照すること。

---

**本節で確定した方針**
- 外部一意キーの「正は親機にある」という原則を確定し、本アプリは受け取りと紐付けのみを担う。
- `external_key_binding` テーブルの概念構成（external_key_source / external_key_value / work_pattern_id / valid_from / valid_to / sync_status / factory_id）を確定する。
- 1 外部一意キー対多 work_pattern のマッピングを許容し、有効期間により任意時点での一意性を保証する。
- external_key_value は JSON 文字列として保持し、親機ごとのキー構成の差異をアプリ側で吸収する。

## 4. 作業パターンと SOP の紐付けモデル

### 作業パターン（work_pattern）の定義

作業パターン（work_pattern）とは、外部一意キーに紐づく SOP 参照セットの上位概念である。「この外部一意キーに対して、どの SOP（3 階層：工程→作業→Step）を・どの順序で・どの条件下で実行するか」を定義する。

SOP の 3 階層（工程→作業→Step）は計画 06 章のデータモデルをそのまま使用する。本章の導入によって 06 章の 3 階層モデルを変更することを禁止する。

### 参照関係の確定

外部一意キー・作業パターン・SOP の参照関係を以下のように確定する。

```
外部一意キー（external_key_binding）
    └── 作業パターン（work_pattern）     ← 参照セットの定義
            └── SOP（3 階層）            ← 06 章のデータモデル
                    ├── 工程（Process）
                    ├── 作業（Operation）
                    └── Step
```

この参照関係により、同一の SOP を複数の作業パターンから参照することを許容する。SOP は再利用可能な共有資産として位置づけ、作業パターンは「組み合わせと実行順序の定義」を担う。

### `work_pattern` エンティティの概念設計

`work_pattern` エンティティの概念的な構成を以下に確定する。

| フィールド名 | 役割 | 備考 |
|---|---|---|
| `work_pattern_id` | 作業パターンの永続識別子 | UUID、NULL 禁止 |
| `pattern_name` | 作業パターンの名称 | 多言語 JSONB 対応 |
| `sop_refs` | 参照 SOP のリスト | SOP 版 ID の配列（JSONB）。実行順序を含む |
| `effective_date` | 有効開始日 | SOP バージョン管理との連動 |
| `factory_id` | 適用工場 ID | 工場ごとの作業パターン分離 |
| `is_active` | 有効フラグ | false の場合は廃止済み。物理削除禁止 |

### バージョン管理方針

外部一意キーと作業パターンのバージョン管理は **Effective Date × 外部一意キーの組み合わせ**で一意に決定する。これにより、特定の日付・特定の品目に対してどの作業パターンが適用されるかを事後に照会できる。

SOP の版管理（06 章）と作業パターンのバージョン管理は独立して機能する。SOP が改訂されても、作業パターンが参照する SOP 版 ID は変更されない（06 章・時点参照の原則との整合）。作業パターン側で参照 SOP 版を意図的に切り替える場合は、新しい作業パターンを作成する。

---

**本節で確定した方針**
- work_pattern を「外部一意キーに紐づく SOP 参照セットの上位概念」として定義し、06 章の SOP 3 階層モデルを変更しないことを確定する。
- 外部一意キー → 作業パターン → SOP（3 階層）の参照関係を確定する。
- バージョン管理を Effective Date × 外部一意キーの組み合わせによる一意決定とし、事後照会を可能にする。
- SOP 版管理と作業パターンバージョン管理を独立に機能させ、参照の不変性を保証する。

## 5. 連携プロトコルの選定

### プロトコル選定の確定

本アプリが対応する連携プロトコルを以下のとおり確定する。

| プロトコル | 用途 | ver1.0.0 | 根拠 |
|---|---|---|---|
| REST API（JSON over HTTPS） | マスタ同期・実績送信の主要インタフェース | ○ 提供 | OpenAPI 3.1 対応・SIer 統合に最適 |
| Webhook | 親機から子機へのイベント通知（マスタ更新等） | ○ 提供 | 非同期通知・再送ポリシー付き |
| AMQP（RabbitMQ 等） | メッセージキューによる疎結合連携 | △ 設計余地のみ | 個人開発スコープ・ver1.0.0 では非実装 |
| MQTT | IoT / PLC センサデータの収集 | × 永続非提供 | OT 直結は永続非提供（構想 12 章 §6・計画 05 章 §4 参照） |
| OPC-UA | PLC 連携 | × 永続非提供 | 同上 |

### REST API の設計方針

REST API については以下の方針を確定する。詳細なエンドポイント設計・スキーマ定義は概要設計に委ねる。

| 方針項目 | 内容 |
|---|---|
| スキーマ形式 | OpenAPI 3.1。ver1.0.0 成果物として公開する |
| URL バージョニング | `/api/v1/` 形式。URL-based バージョニングを採用する |
| レート制御 | トークンバケット方式。工場 ID 単位でレート設定を持つ |
| 冪等性 | 書き込み系エンドポイントには Idempotency-Key ヘッダーを必須とする（計画 05 章 §3 との整合） |
| レスポンス形式 | JSON。エラーコードは RFC 9457 Problem Details 形式を採用する |

### Webhook の設計方針

Webhook については以下の方針を確定する。

| 方針項目 | 内容 |
|---|---|
| 署名検証 | HMAC-SHA256 による署名付与。受信側が署名を検証する |
| リトライ | 最大 3 回。指数バックオフ（1 秒・4 秒・16 秒） |
| 冪等性 | Idempotency-Key ヘッダーを全 Webhook ペイロードに付与する |
| イベント種別 | マスタ更新通知（`master.updated`）・同期完了通知（`sync.completed`）を最小セットとして定義する |
| 失敗時の処理 | 3 回リトライ後の失敗は `webhook_delivery_log` に記録し、管理者に通知する |

---

**本節で確定した方針**
- REST API（OpenAPI 3.1）と Webhook を ver1.0.0 で提供することを確定する。
- AMQP は ver1.0.0 非実装・設計余地のみ保持とし、MQTT・OPC-UA を永続非提供とする。
- REST API は URL-based バージョニング・トークンバケット方式レート制御・Idempotency-Key 必須を確定する。
- Webhook は HMAC-SHA256 署名・最大 3 回指数バックオフリトライ・冪等性保証を確定する。

## 6. 認証・認可方針

### 連携方向別の認証方式

連携方向ごとに異なる認証方式が必要となる。本節でその方針を確定する。

| 連携方向 | 認証方式 | 備考 |
|---|---|---|
| 親機 → 子機（マスタ同期プル要求） | OAuth 2.1 Client Credentials または mTLS | 親機が「クライアント」として子機 API を呼ぶ |
| 子機 → 親機（実績送信プッシュ） | 親機が定める方式に従う（プラガブル設計） | 親機の API 認証仕様は親機ごとに異なる |
| 子機 API（内部利用・タブレット端末） | JWT（RS256）+ RBAC（既存設計、08 章参照） | 変更なし。本章による影響を受けない |

### OAuth 2.1 Client Credentials の採用根拠

OAuth 2.1 Client Credentials フローを採用する根拠は以下のとおりである。

- 親機から子機への API 呼び出しは、ユーザーが介在しないサービス間の機械的通信である。Client Credentials フローはこのユースケースに最も適合する
- RFC 9068（JWT Profile for OAuth 2.0 Access Tokens）に準拠することで、アクセストークンの検証をステートレスに行える
- スコープによる細粒度アクセス制御が可能であり、マスタ読み取り専用スコープと実績書き込みスコープを分離できる

### mTLS の採用条件

mTLS（相互 TLS 認証）は以下の条件が揃う場合に OAuth 2.1 の代替として採用できる。

- 導入工場の社内 LAN に PKI（公開鍵基盤）インフラが整備されている
- 親機と子機の間にファイアウォール・DMZ が設置されており、証明書の管理を IT 部門が担える
- SIer の要求・顧客のセキュリティポリシーが mTLS を指定している

mTLS を採用する場合でも、JWT + RBAC による内部ユーザー認証の設計は変更しない。

### プラガブル設計の定義

子機から親機への実績送信における認証は「プラガブル設計」を確定する。プラガブル設計とは、親機側の認証方式（API キー・OAuth 2.0・SAML 等）に応じてアダプタを差し替えられるインタフェースを設計することを指す。

この設計により、SAP・GENIOUS・その他のカスタム ERP の認証方式に対して、本アプリのコアロジックを変更せずに対応できる。アダプタの実装は親機の仕様が判明した段階で行うものとし、本章ではインタフェースの存在のみを確定する。

---

**本節で確定した方針**
- 親機→子機方向の認証を OAuth 2.1 Client Credentials または mTLS に確定する。
- 子機→親機方向の認証をプラガブル設計とし、アダプタ差し替えによる親機ごとの対応を可能にする。
- 子機内部の認証（JWT RS256 + RBAC）は本章の影響を受けず、08 章の設計を継続する。

## 7. オフライン耐性とコンフリクト解決方針

### 大原則

子機モードを採用しても、本アプリのオフライン作業継続能力は一段も下げない。この方針は計画 05 章 §1 の Offline-First 原則および構想 12 章 §5 の Local-First 思想と完全に整合する。連携モードの選択はオフライン耐性に影響しない。

### マスタ同期の断絶時の動作

親機停止・ネットワーク断等によりマスタ同期が途絶えた場合の動作を以下のとおり確定する。

| 状態 | 本アプリの動作 |
|---|---|
| 同期断絶中（親機停止・ネットワーク断） | 最後の同期成功時刻のマスタで縮退稼働。作業継続を最優先とする |
| UI での通知 | 「マスタが古い可能性あり」をバナーで明示。最終同期時刻を常時表示する |
| 同期失敗の記録 | 同期失敗イベントを監査ログ（`sync_failure_log`）に記録する |
| 同期復旧時 | 自動的に差分同期を実行する。作業者の操作は不要 |

計画 05 章 §1 の Emergency Mode（切断が 5 分を超過した場合のフルスクリーンバナー表示）は、マスタ同期断絶にも適用される。

### 実績送信の断絶時の動作

子機モード 双方向において、実績送信が断絶した場合の動作を以下のとおり確定する。

| 状態 | 本アプリの動作 |
|---|---|
| 送信断絶中 | Outbox Pattern（計画 05 章 §3 で確定済み）を使用し、送信キューに積んで後送 |
| ローカル保持 | タブレット側 SQLite の `outbox` テーブルに未送信イベントを保持する |
| リトライ | ネットワーク復旧を検知して自動リトライ。指数バックオフを適用する |
| 送信順序 | 端末タイムスタンプ順を維持する（計画 05 章 §3 との整合） |

### コンフリクト解決方針

| コンフリクトの種類 | 解決方針 |
|---|---|
| 外部一意キーのマッピング変更と既存 SOP の変更が競合した場合 | UI でオペレーターに確認を促す。自動解決しない。sync_status を `conflict` に設定する |
| 作業実績の競合 | Append-only Event Sourcing（計画 05 章 §2）により競合が設計上発生しない。追記のみのため上書きが起きない |
| マスタ同期の競合（親機・子機の同一レコードが差異を持つ場合） | 親機の値を権威とする。子機側の差異は `sync_conflict_log` に記録する |

図: fig_sync_flow_outbox.svg（img/ 配下）を参照すること。

---

**本節で確定した方針**
- 連携モードの選択にかかわらず Offline-First 原則（計画 05 章 §1）を完全に維持する。
- マスタ同期断絶時は最終同期時刻マスタで縮退稼働し、バナー通知と監査ログ記録を行うことを確定する。
- 実績送信断絶時は Outbox Pattern で後送し、端末タイムスタンプ順を維持することを確定する。
- コンフリクトはマスタ競合のみ発生し得る。発生時は親機の値を権威とし、差異を `sync_conflict_log` に記録する。

## 8. やらないこと

本節では、本アプリが外部システム連携において「行わない」ことを確定する。これらは将来の判断余地を残さず、設計の境界を明確にするための永続的な非スコープである。

| やらないこと | 理由 |
|---|---|
| PLC / OPC-UA 等の OT レイヤへの直接接続 | 永続非提供。計画 05 章 §4 の OT-IT 境界原則・構想 12 章 §6 による確定済みの非スコープ |
| 親機の業務ロジック（生産計画算出・原価計算・在庫引き当て）の実行 | 本アプリの責務は作業支援と記録に限定する。親機の業務ロジックを子機側で再実装することを禁止する |
| 親機マスタへの直接書き込み（正の更新） | 子機から親機へは実績イベントの送信のみ。マスタの正は親機にある原則を維持する |
| AMQP / MQTT / OPC-UA メッセージングの ver1.0.0 実装 | 個人開発スコープ。AMQP は将来の設計余地として保持するが ver1.0.0 では実装しない |
| 特定親機固有 API の全面サポート（独自プロトコルのみの場合） | アダプタ実装の有無は個別判断。本書での動作保証は REST API・Webhook 準拠の親機に限定する |
| マスタデータのリアルタイム双方向同期（双方向の同時書き込み） | マスタの正は親機にある。子機側からのマスタ変更要求は API 経由での提案にとどめる |

---

**本節で確定した方針**
- OT レイヤへの直接接続と親機業務ロジックの実行を永続非スコープとして確定する。
- 親機マスタへの直接書き込みを禁止し、子機から親機への通信は実績イベント送信のみに限定する。
- AMQP の ver1.0.0 非実装と将来余地の保持を確定する。
- アダプタ保証範囲を REST API・Webhook 準拠の親機に限定し、独自プロトコルのみの親機は個別判断とする。

## 9. 関連章との接続

本章の設計方針が依存する・または本章の設計方針を引き継ぐ関連章との接続を以下に整理する。

| 関連章 | 接続の内容 |
|---|---|
| 構想 12 章（外部世界との接点と子機としての社会的位置づけ） | 本章の思想的根拠。「常に子機である」という立場・OT 直結永続非提供・Local-First の確立 |
| 計画 02 章（対象範囲とビジネス目標） | L187「REST API + Webhook + OpenAPI 3.1 を ver1.0.0 で公開」宣言の具体化 |
| 計画 03 章（機能スコープと非スコープ） | 子機モードのコア機能（マスタ同期・実績送信）の機能スコープ定義との対応 |
| 計画 05 章（アーキテクチャ原則） | Outbox Pattern・Idempotent API・Append-only Event Sourcing・OT-IT 境界の前提 |
| 計画 06 章（データモデル中核設計） | `external_key_binding`・`work_pattern` エンティティの追記先。SOP 3 階層は変更しない |
| 計画 07 章（技術スタック選定根拠） | 連携プロトコル選定の実装技術詳細（フレームワーク・ライブラリ等）の根拠 |
| 計画 13 章（データ移行とマスタ初期投入戦略） | 子機モード開始時の初期マスタ同期方針。本章の連携モード定義を前提とする |

**本節で確定した方針**
- 本章は計画 02 章の宣言の具体化であり、計画 05 章・06 章・07 章の原則・設計を前提とする。
- `external_key_binding` と `work_pattern` の設計追記先は計画 06 章とし、本章は概念設計のみを担う。
- 計画 13 章（データ移行）は本章の連携モード定義を前提として設計することを確定する。
