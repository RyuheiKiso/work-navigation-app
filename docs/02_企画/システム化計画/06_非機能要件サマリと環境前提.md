# 06_非機能要件サマリと環境前提

本書は機能要件 (何を作るか) ではなく、品質特性 (どう動くべきか) の前提値を確立する。
各数値は [`05_規制・品質・倫理上のリスクと方針.md`](./05_規制・品質・倫理上のリスクと方針.md) の規制方針と
業界標準研究を根拠とする。具体的な計測方法・合否判定基準は要件定義フェーズで確定する。

---

## 1. 性能目標

| 項目 | 目標値 | 根拠 |
|---|---|---|
| タップ操作の応答時間 | ≤ 300ms (局所フィードバック) | Nielsen ヒューリスティクス §1: 0.1 秒が「即時」の知覚閾値 ([`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md)) |
| QR コードスキャン認識 | ≤ 1 秒 (Google ML Kit Barcode) | 作業員の待機ストレス許容上限 |
| 次ステップ表示の画面遷移 | ≤ 500ms | 作業中断を感じさせない応答性 |
| サーバ API 応答 (P95) | ≤ 2 秒 (社内 LAN) | ユーザが「処理中」と認識しないレベル |
| オフライン→オンライン復帰後の同期完了 | ≤ 30 秒 (通常規模のバックログ) | 次工程開始までに同期が完了する目安 |
| マスタ検索応答 | ≤ 1 秒 | PostgreSQL インデックス設計で達成可能な範囲 |
| トレサビ照会応答 | ≤ 5 秒 (1 ロット分の全記録) | QA 担当者の実用許容範囲 |

数値の根拠がない項目は「TBD: 要件定義で確定」と明記し、推測値を仕様として扱わない。

---

> **本節で確定した方針:** タップ応答 300ms・API P95 2 秒を設計目標とする。測定方法と合否判定基準は要件定義フェーズで確定し、テスト仕様書に記載する。

---

## 2. 可用性と BCP (事業継続計画)

### 2.1 可用性目標

| 指標 | 目標値 | 備考 |
|---|---|---|
| 可用性 | 99% 以上 (年間停止許容 ≈ 87.6h) | 社内システム・個人開発規模の現実的目標 |
| 計画外停止の RTO | ≤ 1 時間 | サーバ再起動 + Docker Compose up で達成可能 |
| データ損失の RPO | ≤ 15 分 | PostgreSQL WAL PITR + Docker volume バックアップで達成 |

SaaS・クラウド水準の 99.9% 以上は個人開発・オンプレ単一サーバの構成では過剰であり、
99% を現実的な設計目標とする
([`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md))。
中小企業の BCP 策定率が 17% (2022 年時点) であることを踏まえ
(同上)、本システムの BCP 設計は製造現場の BCP 成熟度を底上げする観点でも価値を持つ。

### 2.2 バックアップ設計

| バックアップ種別 | 頻度 | 保存先 | 保存期間 |
|---|---|---|---|
| PostgreSQL WAL (Point-in-Time Recovery) | 連続 (WAL アーカイブ) | 同一サーバ別ドライブ | 7 日 |
| Docker volume スナップショット | 日次 (夜間) | 社内 NAS | 30 日 |
| フルバックアップ | 週次 | 社内 NAS | 3 ヶ月 |

保存期間は業界規制の最短要件 (食品: 2 年以上 / 自動車: 10 年以上) に合わせて
マスタ設定で変更可能とする。

### 2.3 緊急時の作業継続

サーバ停止時でも現場作業は継続できることが重要。
タブレットのオフラインモードが自動発動し、ローカル SQLite に記録を継続する
([`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md))。

緊急モードの動作:
- サーバ接続失敗時: 自動でオフラインモードに切替え (ユーザ操作不要)
- 表示する手順書: 最後に同期した最新版
- 記録: ローカル保存 → 復帰後に自動同期
- 制限: サーバ側のマスタ更新は反映されない

---

> **本節で確定した方針:** RTO 1h / RPO 15min / 可用性 99% を目標とする。WAL PITR + 日次スナップショットで RPO を達成する。オフラインモードはサーバ停止時に自動発動し、現場作業の中断を防ぐ。

---

## 3. セキュリティ

社内 LAN 閉域を前提とするが、内部脅威 (不正アクセス・退職者) に対しては技術的制御を設ける。

| 要件 | 対応方針 |
|---|---|
| 認証 | JWT Bearer Token。有効期限: 8 時間 (シフト 1 直分)。更新はリフレッシュトークン |
| 認可 | ロールベースアクセス制御 (RBAC)。ロール: 作業員 / 班長 / QA / 管理者 / 監査閲覧者 |
| パスワード | bcrypt ハッシュ。最小 10 文字。辞書攻撃耐性のためレートリミット設定 |
| 通信暗号化 | 社内 LAN 内: HTTP (TLS 負荷が IIS で対応可能な場合は HTTPS 推奨)。TLS は IIS 証明書で管理 |
| 監査ログ | 全 CRUD イベント + ログイン/ログアウト + マスタ変更を Append-only で記録 |
| 退職者処理 | 管理者がアカウントを即時無効化できる機能。無効化後も記録の帰属情報は保持 |
| セッション管理 | 一定時間操作なしで自動ログアウト (TBD: 要件定義で確定。目安 30 分) |
| SQL インジェクション対策 | sqlx のプリペアドステートメントを全クエリで使用 |
| XSS 対策 | React / React Native のエスケープ機構に準拠。動的 HTML 挿入を禁止 |

---

> **本節で確定した方針:** 認証は JWT RBAC。ログは Append-only。退職者の即時無効化を管理者が実行できる。通信暗号化は社内 LAN 前提だが IIS の HTTPS を推奨する。

## 4. 耐環境要件

### 4.1 タブレット端末への要求 (再掲)

[`02_想定利用者とユースケース概要.md`](./02_想定利用者とユースケース概要.md) §3 で定義した内容の技術的根拠を補足する。

| 要求項目 | 基準値 | 根拠 |
|---|---|---|
| IP 等級 | IP65 以上 | 粉塵完全防護・ノズルによる放水テスト通過 |
| 落下耐性 | MIL-STD-810H Method 516.8 (1.2m) | 工場床面落下への耐久性 |
| 輝度 | 800nits 以上 | 屋外・蛍光灯反射環境での視認性 |
| 動作温度 | 0〜50℃ | 一般製造環境の温度範囲 |

([`90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md`](../../90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md))

### 4.2 防爆エリアの非適用

ATEX 2014/34/EU または TIIS (技術基準適合証明) が必要な Zone 0〜2 / Zone 20〜22 エリアには、
本システムが動作するタブレットを持ち込んではならない。
防爆認証端末を使用する場合のコストは通常端末の **5〜10 倍** となる (同上)。
この場合の適用可否は安全衛生管理者・防爆専門家による評価が必要であり、
本システムはその判断を行わない。

### 4.3 電磁環境 (EMC)

工場内の大電流機器・溶接機・インバータは電磁ノイズ源となる。
タブレット端末は IEC 61000-4 シリーズ準拠の EMC 性能を持つ製品を推奨するが、
システム側で EMC 要件を直接制御することはできない。
オフライン同期設計によってネットワーク瞬断への耐性を確保する。

---

> **本節で確定した方針:** タブレットに IP65 / MIL-STD-810H / 800nits を推奨基準として提示する。防爆エリアへの適用は対象外と判断する。EMC への対応はタブレット選定で担保し、システム側ではオフライン耐性で補完する。

---

## 5. ユーザビリティ

現場 HCI 研究に基づくユーザビリティ設計要件
([`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md))。

| 要件 | 基準 | 対応方針 |
|---|---|---|
| タッチターゲット | 最小 72dp × 72dp (手袋装着時) | 全インタラクティブ要素に適用 |
| 輝度 / コントラスト | WCAG 2.1 AA 以上 (4.5:1) | テーマカラーに組み込む |
| 情報密度 | 1 画面 1 ステップ (Glanceable UI) | 一覧・スクロール設計を避ける |
| エラー防止 | 完了・不適合起票に確認ダイアログ | Nielsen ヒューリスティクス #5 |
| フォントサイズ | 本文最小 16sp、推奨 18〜20sp | 高齢者・遠距離視認に配慮 |
| ナビゲーション | 戻るボタンで前ステップに戻れる | ただし完了済みステップの記録は保持 |
| オフライン表示 | オフライン状態を常時バナーで明示 | ユーザが状態を誤認しない |

NASA-TLX (タスク負荷評価) を導入テスト時に実施し、
主観的負荷スコアが許容範囲内か評価することを推奨するが、これは要件定義フェーズ以降の判断とする。

---

> **本節で確定した方針:** タッチターゲット 72dp / WCAG AA / Glanceable UI / 16sp フォントを設計標準とする。実際の使いやすさ評価 (ユーザテスト) は Phase 1 のパイロット運用で実施する。

---

## 6. 多言語・識字対応

[`02_想定利用者とユースケース概要.md`](./02_想定利用者とユースケース概要.md) §4 で定義した多言語ロードマップを非機能要件として具体化する
([`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md))。

| 非機能要件 | 基準 |
|---|---|
| i18n 基盤 | i18next (React Native / React 共通) を Phase 1 から組み込む |
| 翻訳ファイル形式 | JSON (言語コード: ja, en, vi, zh, pt の 5 言語を最終ターゲット) |
| ピクトグラム | ISO 7010 準拠アイコンを UI コンポーネントライブラリに組み込む |
| 文字エンコーディング | UTF-8 で全レイヤーを統一 |
| 日付・数値ロケール | Intl API (ECMAScript) を使用し、ロケール依存の日付書式を避ける |
| RTL (右から左) 言語 | 現時点でスコープ対象言語 (vi, zh, pt) は非 RTL のため未対応。将来拡張の余地を残す |

---

> **本節で確定した方針:** Phase 1 から i18next 基盤を組み込み、翻訳追加コストを最小化する。翻訳データは Phase 2 以降で追加。機械翻訳の安全用語への適用は全フェーズで禁止する。

---

## 7. 監査追跡・ログ保全

監査ログは不正アクセス・改ざん検知・規制査察対応の基盤となる
([`90_業界分析/21_作業ログ分析とプロセスマイニング.md`](../../90_業界分析/21_作業ログ分析とプロセスマイニング.md))。

| ログ種別 | 記録内容 | 保存期間 |
|---|---|---|
| 認証ログ | ログイン/ログアウト・失敗・JWT 発行/失効 | 2 年 (デフォルト) |
| 操作ログ | 全 CRUD (作業員操作 + 管理者操作) | 規制保存期間に準拠 |
| 同期ログ | オフライン/オンライン切替・同期成否・Idempotency Key | 1 年 |
| エラーログ | アプリエラー・API エラー | 6 ヶ月 |

ログは Append-only テーブルに格納し、物理削除は管理者ロールでも実行不可とする。
ログの保存期間は管理者設定で延長可能だが短縮は不可とする。

---

> **本節で確定した方針:** 全 CRUD を Append-only ログに記録し、物理削除を禁止する。認証ログは 2 年・操作ログは規制保存期間を基準とする。ログ設定の詳細は要件定義フェーズで確定する。

---

## 8. 要件定義フェーズへの申し送り事項

本書で「TBD: 要件定義で確定」と注記した事項を以下に集約する。

| 項目 | 現状の仮置き値 | 要件定義での確定方法 |
|---|---|---|
| セッションタイムアウト | 30 分 | 現場操作の実態調査結果を踏まえて設定 |
| API 応答時間 (P95) | 2 秒 | 負荷試験計画で計測上限を確定 |
| ログ保存期間 (操作ログ) | 規制保存期間に準拠 | 対象業界・顧客の規制要件を確認して設定 |
| ユーザビリティテスト実施 | Phase 1 パイロット時 | NASA-TLX + ユーザ観察で評価 |
| HTTPS 強制 | IIS 証明書設定後に強制化 | 社内 PKI の有無を確認して決定 |
| 測定値不確かさ必須化 | Phase 2 で必須化 | QA 担当者・規制要件確認後に決定 |
