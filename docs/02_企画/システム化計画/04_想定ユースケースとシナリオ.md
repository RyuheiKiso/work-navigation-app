# 04 想定ユースケースとシナリオ

本章の責務は、本システムが活きるユースケースを習熟度・場面・例外の観点から網羅し、設計の検証基準とすることである。本章に記述したシナリオをアプリが実現できない場合、その設計要件は未達と判定する。

---

## 1. ユースケース分類と設計上の使い方

### 1-1. ユースケースの設計上の位置付け

本章のユースケースは、08 章「品質特性と非機能要件方針」の妥当性検証基準となる。各シナリオは、以下の用途で参照する。

- 機能要件の網羅性確認（03 章のコア機能・対応機能が全シナリオをカバーするか）
- 非機能要件の数値化（応答時間・可用性・オフライン継続時間の目標値の根拠）
- テスト計画の基礎（各シナリオが受入テストのシナリオベースとなる）
- UX 設計の検証（各シナリオで作業者が詰まらずに完了できるか）

### 1-2. 分類

| 分類 | 定義 |
|---|---|
| 主要ユースケース | 最頻出かつ最重要。未対応はシステム目的の未達に直結する |
| 補足ユースケース | 重要だが発生頻度が低い、または例外的な状況。未対応はリスク顕在化時に重大障害となる |

本章で扱うシナリオはすべて「主要」または「補足」に分類し、「将来対応」は設けない。

**本節で確定した方針**
本章のシナリオは 08 章の非機能要件妥当性検証の基準として確定する。
9 シナリオ（A〜I）はすべて ver1.0.0 で対応するものとして確定する。
シナリオをカバーできない設計は設計要件未達として差し戻す。

## 2. 習熟度別ナビゲーションシナリオ

Dreyfus 5 段階モデル（Novice / Advanced Beginner / Competent / Proficient / Expert）を基盤とし、本システムは 4 つの習熟度プロファイルを実装する。

分類根拠: [`90_業界分析/15_習熟効果と学習曲線.md`](../../90_業界分析/15_習熟効果と学習曲線.md) — Dreyfus 5 段階・Cognitive Apprenticeship（Scaffolding / Fading）・Skill Decay（長期ブランク検知）

### プロファイル A-1: Novice（新人作業者）

**場面**: 入社 1 週間、山田さん（20 代・日本語ネイティブ）が初めてライン組立作業を担当する。

**システムの応答**:
- 全ステップを Linear モードで一つずつ表示する。スキップ操作を受け付けない強制ロックステップ制御を適用する
- 各ステップに動画・写真・ピクトグラムをフル表示する。テキストにはふりがなを付与する
- 測定値入力ステップでは USL / LSL の範囲をグラフィカルに表示し、「この数字を入れてください」という直接誘導を行う
- クリティカルステップ完了時に肯定フィードバック（振動・音・視覚的アニメーション）を表示する
- ステップ完了ごとに電子サイン・タイムスタンプ・作業者 ID を自動付与する

**Scaffolding 状態**: 最大化。Cognitive Apprenticeship における足場かけが全面的に機能する状態である。

**達成基準**: 山田さんが指導員の補助なしに全作業ステップを完了し、証拠付き記録が自動生成される。

### プロファイル A-2: Competent（習熟途中）

**場面**: 入社 3 か月、鈴木さんが同作業を担当する。手順の概要は把握済みだが細部の確認が必要な場合がある。

**システムの応答**:
- Linear モードを継続するが、各ステップの動画・詳細説明は折り畳み状態をデフォルトとする。展開は任意操作で行う
- 重要ポイント（クリティカルフラグ付きステップ・USL / LSL に近い測定値）のみをハイライト表示する
- Fading（足場の段階的撤去）が自動で始まる。前回完了時の実績から習熟度スコアを更新し、習熟が進むにつれて誘導を縮小する

**達成基準**: 鈴木さんが必要な時だけ詳細を参照しながら作業を完了できる。不要な情報による作業阻害が発生しない。

### プロファイル A-3: Proficient / Expert（熟練作業者）

**場面**: 入社 5 年、佐藤さん（熟練作業者）が同作業を担当する。作業手順は完全に体得している。

**システムの応答**:
- Reference モードを適用する。チェックポイント（クリティカルフラグ付きステップ・証拠入力必須ステップ）のみを一覧表示する
- 詳細手順は折り畳み状態で提供する。佐藤さんが「コツコメント」（自身が過去に入力した知見メモ）を閲覧できる
- 新人向けの動画提供・コメント入力など、知識伝承への貢献機能が主機能になる
- システムが佐藤さんの作業ペースを阻害しない設計を最優先とする

**達成基準**: 佐藤さんの作業時間が紙チェックリスト運用時と同等以上であり、システムが作業の障害にならない。

### プロファイル A-4: 改訂直後 / 長期ブランク（Skill Decay 対策）

**場面**: 佐藤さんが 2 か月ぶりに同作業を担当する。または、先週 SOP の新版が発効した直後に作業を担当する。

**システムの応答**:
- 習熟度が Expert であっても、以下のいずれかに該当する場合は自動的に Linear モードへ強制切り戻しを行う
  - 最終作業完了から 30 日以上経過している（Skill Decay 検知）
  - 現行版 SOP の初回実施から N 回完了に達していない（改訂直後・N は SOP 管理者が設定）
- 切り戻し理由を画面上に明示する（「前回作業から 62 日が経過しています。詳細手順で確認します」）
- N 回完了後は自動的に元の習熟度プロファイルに復帰する

**達成基準**: 長期ブランク・改訂直後の作業で、熟練者による手順スキップ起因の品質問題が発生しない。

**本節で確定した方針**
習熟度プロファイルは Novice / Competent / Proficient-Expert / Skill Decay 対策の 4 区分で確定する。
Skill Decay の閾値は「最終実施から 30 日」「改訂後 N 回完了」の 2 条件で確定する（N は設定値）。
Reference モードでも証拠付き確認（写真・測定値・スキャン）を省略しない設計を確定する。

## 3. 中断・割込み・再開シナリオ

分類根拠: [`90_業界分析/20_作業中断・割込み・再開の認知科学.md`](../../90_業界分析/20_作業中断・割込み・再開の認知科学.md) — Resumption Lag・Memory for Goals モデル・Post-Completion Error・プレースキーパー・Resumption Cue・ANSI/ISA-18.2（10 件/h 以下が許容アラート頻度）

### シナリオ B: 急な呼び出しによる作業中断（5 分未満）[主要]

**場面**: 田中さんがクリティカルステップの測定値入力の途中で、現場監督から急に声をかけられた。

**中断時のシステム応答**:
- Protection Window を有効化する。中断から 5 分間は新着通知をミュートし、割り込みによるステップ誤操作を防止する
- 中断ログを自動記録する（時刻・現在のステップ番号・作業者 ID・中断理由種別）
- 画面はロック状態に移行する。意図しない操作でステップが完了登録されない

**再開時のシステム応答**:
- 再開時に Welcome 画面を必ず表示する。直前のステップサマリ（何を入力しようとしていたか）と経過時間（「5 分 23 秒前に中断」）を表示する
- ロックステップが未完ステップへ強制復帰する。田中さんは測定値を入力していないステップから再開する
- Post-Completion Error の構造的防止: 再開時にロックステップが未完ステップへ戻ることで、「前のステップは終わった気がした」という誤認識を防止する

**達成基準**: 田中さんが中断前の作業状態を正確に把握し、測定値入力を漏れなく完了する。Post-Completion Error が構造的に発生しない。

### シナリオ C: 長時間中断からの再開（30 分超）[主要]

**場面**: 伊藤さんが昼休みを挟んで作業を再開する。工程の途中（ステップ 14/22）で昼休みに入ったため、30 分以上が経過している。状況認識（SA: Situation Awareness）が失われている。

**再開時のシステム応答**:
- Welcome 画面に「前の作業状態スナップショット」を表示する。内容: 作業名・現在ステップ・直前 3 ステップの完了記録サマリ・経過時間
- 30 分超の場合、クリティカルステップ（FMEA-RPN 連動フラグ付き）については完了済みであっても「目視再確認」プロンプトを表示する。スキップ可だが理由入力を必須とする
- Resumption Cue として、直前ステップで入力した数値・写真のサムネイルを再表示する。記憶の回復を補助する

**達成基準**: 伊藤さんが昼休み前の作業状態を正確に把握し、漏れなく次のステップへ進める。Resumption Lag（再開時の遅延）が最小化される。

### シナリオ D: シフト交代引継ぎ[主要]

**場面**: 二交代制の夕方シフト終了時、木村さんがアセンブリライン作業の途中でシフト交代を迎える。次シフトの新人、松田さんへの引継ぎが必要である。

**システムの応答**:
- I-PASS 構造の引継ぎメモをデジタル引継ぎ票として生成する

| I-PASS 項目 | 内容 |
|---|---|
| I（Illness / 状況概要） | 現在の作業状態（工程名・品番・ロット番号・完了ステップ数） |
| P（Patient / 案件詳細） | 当日の作業特記事項（変更点・異常発生の有無） |
| A（Assessment / 評価） | 現時点での品質状態（不適合の有無・測定値トレンド） |
| S（Situation / 未完課題） | 未完ステップ・未回収の中断ログ・保留中のアンドン |
| S（Synthesis / 引継ぎ指示） | 次シフトへの指示事項・注意点 |

- 引継ぎ票は電子署名付きで記録される（木村さんの作業者 ID・タイムスタンプ）
- 次シフトの松田さんへプッシュ通知を送信する。松田さんがログイン時に引継ぎ票の確認を完了するまで作業開始できない

**達成基準**: 松田さんが引継ぎ票の確認を完了し、木村さんの中断ステップから作業を再開する。引継ぎ漏れによる工程スキップが発生しない。

**本節で確定した方針**
Protection Window・中断ログ・Welcome 画面・Resumption Cue をセットで実装し、中断再開の構造的サポートを確定する。
30 分超の長時間中断後は、クリティカルステップの目視再確認プロンプトを強制表示することを確定する。
I-PASS 構造の引継ぎメモを電子署名付きデジタル引継ぎ票として実装することを確定する。

## 4. 夜勤・疲労・高齢者シナリオ

分類根拠: [`90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md`](../../90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md) — サーカディアンリズム・夜間帯自動ダークモード・深夜連続夜勤の二重確認・シニアモードプロファイル・ISO 9241-304/112・NIOSH RWL

### シナリオ E: 深夜の連続夜勤 3 日目（疲労蓄積）[補足]

**場面**: 夜勤専従の中田さん（30 代）が深夜 2 時、連続夜勤 3 日目の作業を行っている。疲労の蓄積から注意が散漫になっている。

**システムの応答**:
- 20:00〜6:00 の時間帯は自動でダークモードに切り替える。ブルーライトを低減し、眼精疲労を軽減する
- 連続夜勤 3 日目以降かつ深夜帯（0:00〜5:00）において、クリティカルステップで二重確認プロンプトを表示する。「もう一度確認してください」の確認ボタンを 2 秒押し続けて完了する設計とする
- 通常の 50% 以下の時間でステップを完了した場合（異常高速操作）にアラートを表示する。「操作が速すぎます。確認しましたか？」のプロンプトで一時停止する
- バイブレーション通知を強化する（2 段階パターン）
- 「シニアモード」プロファイルを連続夜勤者にもオプションとして適用できる

**達成基準**: 深夜帯連続夜勤での操作ミス（異常高速・スキップ）の検知と抑止が機能する。疲労状態での重大ミスを構造的に防止する。

### シナリオ F: 高齢作業者（60 代）のタブレット操作[補足]

**場面**: 田辺さん（64 歳）がタブレットに不慣れで、細かい文字が読みにくい。手袋を着用したまま操作する必要がある。

**システムの応答**:
- シニアモードを有効化する。適用内容は以下のとおりである
  - フォントサイズ: 14〜16pt（標準 12pt 比 1.2〜1.3 倍）
  - タッチターゲット: 72dp 以上（WCAG 2.1 AA の 44dp を上回る設計。手袋着用を考慮）
  - 自動補完と選択肢化: テキスト入力を最小化し、選択肢タップで完了できるステップを優先設計
  - 確認ダイアログを追加する（誤タップによる意図しない操作を防止）
- ISO 9241-304 の視覚表示設計基準・ISO 9241-112 の対話設計基準に準拠する
- 行間・余白を広くとり、情報密度を下げる

**達成基準**: 田辺さんが手袋を着用した状態でシニアモードを使用し、誤操作なく全ステップを完了できる。タッチ失敗率が通常モードと同等以下に抑えられる。

**本節で確定した方針**
夜間帯（20:00〜6:00）の自動ダークモード切り替えを確定する。
深夜帯連続夜勤 3 日目以降のクリティカルステップ二重確認プロンプトを確定する。
シニアモードのタッチターゲットを 72dp 以上に確定し、WCAG 2.1 AA の 44dp を上回る設計とする。

## 5. オフライン・通信断・電源断シナリオ

分類根拠: [`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md) — Offline First・SQLite ローカル永続化・Outbox Pattern・Emergency Mode・最終同期時刻表示

### シナリオ G: Wi-Fi デッドゾーンでの作業[主要]

**場面**: 大型金属構造物付近の工程で、Wi-Fi 電波が届かない。小野さんが通常どおり作業を開始したが、タブレットがオフライン状態に陥っている。

**システムの応答**:
- Offline-First 設計により、作業に必要な全 SOP・マスタデータは事前に SQLite ローカルに取得済みである。通信断であっても全作業ステップの参照・実行・記録が可能である
- 作業記録はローカルキューに保存する。Outbox Pattern により、通信回復後に自動でサーバーへ同期する
- ステータスバー（全画面共通）に「最終同期時刻: 14:32」を常時表示する。小野さんは現在の記録状態を常に把握できる
- 「記録済（未同期）」バッジを記録済みステップに表示する。ALCOA+ の Contemporaneous（当時性）原則との整合を UI 上で明示する。記録はローカルでタイムスタンプ付きで保存されているため、後からの遡及入力ではないことを証明できる
- 通信回復後の同期完了時に「同期完了」を通知する

**達成基準**: 小野さんが Wi-Fi 圏外でも全作業を完了し、記録が漏れなくサーバーへ同期される。Contemporaneous 原則との整合が記録上証明できる。

### シナリオ H: サーバー障害（Emergency Mode 自動起動）[補足]

**場面**: バックエンドサーバーが停止した。工場内のタブレット全台がサーバーと通信できない状態になっている。

**システムの応答**:
- タブレット側は Emergency Mode を自動起動する。フルスクリーンバナーで「サーバー障害発生中」「現在の記録はローカルに保存されています」を表示する
- Emergency Mode ではローカルキューで継続動作する。作業は中断しない
- 緊急連絡先（IT 担当・現場監督）へのワンタップ通話・避難手順 PDF へのワンタップアクセスを画面上に配置する
- 復旧後の同期フロー: サーバー復旧を検知後、タブレットは Outbox Pattern で記録を順次同期する。同期競合（複数タブレットからの同一ロット記録）の解決ポリシーはタイムスタンプ優先とする
- RPO（目標復旧時点）: 15 分以内のデータ損失で抑える。RTO（目標復旧時間）: 1 時間以内にサーバー側を復旧する

**達成基準**: サーバー障害中もタブレット側の作業・記録が継続され、復旧後に全記録が正常に同期される。RPO 15 分 / RTO 1 時間の目標を充足する。

**本節で確定した方針**
Offline-First アーキテクチャを正常系の設計として確定し、通信断を例外ではなく想定内事象として扱う。
Emergency Mode は自動起動・フルスクリーンバナー・ローカルキュー継続動作・緊急連絡先ワンタップの 4 要素で確定する。
RPO 15 分 / RTO 1 時間を設計目標値として確定する。

## 6. 多言語・外国人技能実習生シナリオ

分類根拠: [`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md) — 外国人労働者 204 万人（2023 年）・製造業 45 万人・JLPT N3/N4/N5 漢字レベル・やさしい日本語 8 原則・ISO 7010 安全標識ピクトグラム・i18next + expo-localization

### シナリオ I: ベトナム語話者の新人（JLPT N5）[主要]

**場面**: グエン・ティ・ホアさん（ベトナム出身・JLPT N5 相当）が技能実習生として配属された。日本語の読み書きは限定的であり、漢字はほぼ読めない。

**システムの応答**:
- グエンさんのプロファイルに言語「ベトナム語」を設定する。UI ラベル・エラーメッセージ・通知はすべてベトナム語で表示する
- SOP 本文はベトナム語版が承認済みである場合、ベトナム語で表示する（多言語 JSONB: `{ "ja": "...", "vi": "..." }`）
- やさしい日本語 8 原則（短文・常用語・平易な表現・固有名詞を避ける等）に基づいて記述された日本語版 SOP と、ベトナム語版 SOP を並記表示するオプションを提供する
- ピクトグラム（ISO 7010 安全標識）を手順の先頭に表示し、言語に依存しない視覚的理解を補助する
- 動画の字幕はベトナム語で表示する

**管理側の承認フロー**:
- 多言語 SOP の有効化には、管理 Web での承認が必要である
- 安全指示を含むステップの翻訳は、人間レビューを承認条件とする。自動翻訳のみでの有効化を設計上禁止する
- 承認済みでない言語の SOP はタブレット APP に表示しない

**誤訳リスク対策**:
- 安全上の重要指示（クリティカルフラグ付きステップ）は、翻訳と原文（日本語）を併記する。グエンさんが不明な場合に日本語原文を参照できる
- 「現場監督を呼ぶ」ボタンをクリティカルステップ画面に常設する

**達成基準**: グエンさんが日本語に不自由な状態でも、ベトナム語 SOP・ピクトグラム・動画字幕により全作業ステップを安全に完了できる。誤訳リスクが承認フローにより管理されている。

**本節で確定した方針**
多言語 SOP の有効化に管理 Web での承認フローを必須とし、安全指示は人間レビューを承認条件として確定する。
ピクトグラム（ISO 7010）を安全指示ステップに必須表示することを確定する。
クリティカルステップには原文（日本語）の併記と「現場監督を呼ぶ」ボタンを常設することを確定する。

## 7. 縮退運用・紙フォールバックシナリオ

分類根拠: [`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md) — Emergency Mode・紙運用フォールバック（最終手段）・I-PASS 引継ぎメモ・RPO 15 分 / RTO 1 時間目標

### シナリオ J: 完全停電・BCP 発動[補足]

**場面**: 地震が発生し、工場全体が停電した。サーバーもタブレットもすべて使用できない状態になっている。工場の生産活動を継続するか否かの判断を経営層が行っている。

**紙フォールバックの事前準備（システムによる定常運用）**:
- 管理 Web は SOP の最新承認版を PDF 形式でエクスポートする機能を提供する
- 運用手順として、IT 担当者が毎週 SOP 最新版 PDF を出力し、各工程の紙保管庫（防水・防塵ファイル）に印刷・保管する
- 保管した紙 SOP には「出力日時・版番号・出力者」を記載する。これにより、紙 SOP が最新版であることを確認できる

**停電発生後の作業継続**:
- 作業員は紙 SOP に従い作業を継続する。紙チェックリストに手書きで記録する
- 本システムは「アプリがなくても作業は継続できる」設計原則を確定している。この原則により、本システムの障害がラインストップの直接原因とならない

**復旧後の突合・後入力フロー**:
- アプリ復旧後、紙記録をシステムへ後入力する。後入力であることを明示するフラグ（`is_retroactive: true`）を記録に付与する
- 後入力の作業者 ID・入力時刻・紙原本の保管場所を記録する
- 突合完了後も紙原本は一定期間（ISO 9001 §7.5 の文書保管期間に従う）保管する

**達成基準**: BCP 発動時に紙 SOP で作業を継続できる。復旧後に紙記録とデジタル記録の突合が完了し、トレサビ上の空白期間が最小化される。

**本節で確定した方針**
「アプリがなくても作業は継続できる」設計原則を確定する。本システムの障害はラインストップの直接原因としない。
紙 SOP の定期出力・印刷保管を IT 担当の定常運用として確定する（週次、各工程の紙保管庫へ）。
紙からの後入力は `is_retroactive: true` フラグ付きで記録し、ALCOA+ の記録完全性（Complete）を維持することを確定する。
