# 01_対象範囲とスコープ

## 1. 本計画の解決対象

製造現場における次の 3 つの構造的課題を解決対象とする。

### 1.1 作業手順の形骸化と記録品質の劣化

紙・Excel ベースの標準作業手順書 (SOP) は版管理・配布・記録が手作業に依存するため、以下の問題が繰り返される。

| 問題 | 具体的影響 |
|---|---|
| 最新版の未徹底 | 旧版 SOP で作業→規格外品・手直し |
| 記録の事後入力 | ALCOA+ の「同時性 (Contemporaneous)」違反 |
| 文字記録の判読不能 | 後工程トレースで数時間以上を消費 |
| チェックリストの形骸化 | 「ゴム印押し」で内容を確認せず通過 |

電子チェックリストを現場に導入した試験では欠陥率が **20–40% 減少**している
([`90_業界分析/19_電子チェックリストと手順遵守の科学.md`](../../90_業界分析/19_電子チェックリストと手順遵守の科学.md))。
WHO 外科安全チェックリストの大規模 RCT でも合併症 **36% 減 / 死亡率 47% 減**が示されており、
チェックリスト実施の効果は業界を問わず統計的に確立されている (同上)。

### 1.2 トレーサビリティ記録の断絶

ISO 9001:2015 §7.5.1 は「管理された文書化した情報」の保持を要求し、
自動車 (IATF 16949 §8.5.2)・食品 (FSMA 204: 24h トレースバック)・医療機器 (EU MDR UDI) は
電子記録を事実上義務化している
([`90_業界分析/22_規制別トレーサビリティ要件詳論.md`](../../90_業界分析/22_規制別トレーサビリティ要件詳論.md))。

現状の紙記録では不良品発生時の後方トレース (起因ロット特定) に数時間〜数日を要し、
リコール対応・顧客クレーム処理のコストが肥大化する。

### 1.3 熟練技能の属人化

団塊世代の大量退職と製造業就業者の高齢化が進む中
([`90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md`](../../90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md))、
ベテラン作業員の「コツ・判断基準」が形式知化されずに失われている。
多能工育成 (スキルマトリクス管理) を電子化することで技能伝承を加速できる
([`90_業界分析/26_多能工とスキル管理・作業者資格.md`](../../90_業界分析/26_多能工とスキル管理・作業者資格.md))。

---

> **本節で確定した方針:** 本システムは EWI/電子チェックリストによる作業ナビゲーションとリアルタイム実績記録の一体化を主目的とする。技能動画の蓄積・管理はスコープ外 (将来検討)。在庫管理・生産計画・会計は非対象。

## 2. 対象範囲 (In Scope)

| # | 機能カテゴリ | 含む内容 | 提供先 |
|---|---|---|---|
| 1 | 作業ナビゲーション | 工程順のステップ表示、写真・動画付き手順、完了チェック、強制順序制御 | タブレットアプリ |
| 2 | 実績記録 | 作業開始/完了時刻、作業者 ID、使用部品ロット、測定値 (数値 + 単位 + 不確かさ) | タブレットアプリ |
| 3 | 自動認識入力 | QR コード・バーコードによるロット/シリアル/部品番号のスキャン入力 | タブレットアプリ |
| 4 | 写真記録 | 作業証拠写真の撮影・保存 (SHA-256 ハッシュ付き) | タブレットアプリ |
| 5 | オフライン対応 | 無線 LAN 断絶時のローカル継続作業と復帰後の自動同期 | タブレットアプリ |
| 6 | 作業手順マスタ管理 | 工程・手順・チェック項目・部品マスタの CRUD | Web 管理画面 |
| 7 | スキルマトリクス管理 | 作業員の資格・認定状況の管理 | Web 管理画面 |
| 8 | トレサビ照会 | ロット/シリアル/日時/作業者を軸にした記録の追跡検索 | Web 管理画面 |
| 9 | 不適合起票 | ナビ中に発見した不適合のインライン起票と記録紐づけ | タブレットアプリ |
| 10 | 認証・認可 | JWT による作業員/管理者ロール分離、ログイン/ログアウト | 全層 |
| 11 | 監査ログ | 全 CRUD イベントの Append-only ログ、改ざん不可 | バックエンド |

---

> **本節で確定した方針:** 対象機能は上記 11 カテゴリに限定する。Phase 1 では #1〜#6 および #10・#11 を実装し、#7〜#9 は Phase 2 に移行する (詳細は [`07_開発スケジュールとフェーズ計画.md`](./07_開発スケジュールとフェーズ計画.md))。

## 3. 対象外範囲 (Out of Scope)

次の機能は本システムのスコープに含まない。スコープ外の根拠を付記する。

| カテゴリ | 対象外とする理由 |
|---|---|
| 生産計画 / MRP / スケジューリング | 別システム (ERP / 生産管理システム) の責務。本システムはナビと記録に特化する |
| 設備保全管理 (TPM 台帳・点検指示) | 対象外とするが将来接続候補。Phase 3 以降に検討 |
| 在庫管理 / 倉庫管理 (WMS) | 別システムの責務。入出庫はトレサビ参照のみ |
| 会計・原価計算 | 財務システムの責務 |
| SCADA / PLC 直接通信 | OT/IT 統合は社内 LAN セグメンテーション上の追加検討が必要。社内インフラ要件を満たした後に Phase 3 以降で検討 |
| 多工場一元管理 | 単一工場前提。横展開は将来の拡張 (本章 §5 参照) |
| クラウド連携 / 外部 API 公開 | 社内 LAN 閉域前提。セキュリティ審査なしに外部接続しない |
| SaaS 型 MES 製品の機能全般 | Buy vs Build の検討結果として内製を選択 (§5 参照) |
| AI/ML による自律判定 (Phase 1) | Phase 3 以降の段階導入。Phase 1 はルールベースのみ |
| 暗黙知動画ライブラリの管理 | コンテンツ管理・著作権管理の複雑さから切り離す |
| 生体認証 / 顔認証 | プライバシーリスクが高く、要件定義で別途評価が必要 |

---

> **本節で確定した方針:** 上表の対象外事項は本システムの責務ではないと判断する。将来の接続候補には「将来検討」と注記し、現行スコープには含めない。スコープ変更は変更管理 (ADR) を通じて行う。

## 4. 単一工場前提とその合理性

本システムは **単一工場・単一テナント** を前提として設計する。

### 4.1 採用の合理的根拠

既製品 MES/eBR 製品を採用しない理由は、既製品が要件を **70% 以上** 満たす場合に限り
Buy が有利とされる閾値
([`90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md`](../../90_業界分析/29_競合製品と作業ナビ・MES・eBR市場.md))
に対し、本要件 (社内 LAN 閉域 + Windows Server 2022 オンプレ + Rust バックエンド + React Native タブレット) が
既製品の想定外構成であるため、カスタマイズコストが大きく内製が合理的と判断した。

加えて Panorama Consulting の調査では ERP/MES 案件の **54% が予算超過・68% が工期超過** するとされており
(同上)、大規模製品の横展開リスクも内製採用の根拠となる。

### 4.2 将来の多工場展開について

現行アーキテクチャは将来の多工場対応を排除しない設計 (工場 ID のテナント識別子をスキーマに持つ) とするが、
機能実装・テスト・運用の優先度はすべて単一工場を基準とする。
多工場展開時の追加コストは横展開先 1 拠点あたり **1.5〜2 倍** の追加費用が見込まれる
([`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md))
ため、要件定義フェーズで改めて評価する。

---

> **本節で確定した方針:** 設計・テスト・運用は単一工場前提で行う。テナント識別子は持つが、多工場機能の実装は Phase 3 以降とする。

## 5. ステークホルダ別の利用境界

本システムの想定利用者と、それぞれが本システムに期待する境界を示す。

| 利用者区分 | 主な利用場面 | 本システムが提供するもの | 本システムが提供しないもの |
|---|---|---|---|
| **現場作業員** (主役) | タブレットで手順確認・チェック・スキャン・写真撮影 | 手順ガイド / 入力補助 / 完了記録 | 残業管理 / 人事評価 / 給与情報 |
| **班長 / 職長** | 割り当て確認・進捗把握・不適合の一次対応 | 作業実績の照会 / 不適合リスト | 生産計画変更 / 配員最適化 |
| **品質保証 (QA)** | 記録の後閲覧・トレサビ照会・CAPA 起票 | 全記録の検索・ダウンロード / 不適合管理 | SPC 解析 / 統計的工程管理 |
| **情シス兼マスタ管理者** | 手順マスタ更新・ユーザ管理・サーバ運用 | マスタ CRUD / ユーザ管理 / バックアップ手順 | サーバ監視ダッシュボード (別途) |
| **外部監査員** | 記録の閲覧・Audit Trail 確認 | 読み取り専用の監査ログ照会 | 設定変更 / マスタ編集 |

利用者への監視に関するデータ倫理境界については
[`05_規制・品質・倫理上のリスクと方針.md`](./05_規制・品質・倫理上のリスクと方針.md) §5 を参照。

---

> **本節で確定した方針:** 本システムのロールは「作業員」「班長」「QA」「管理者」「監査閲覧者」の 5 種とする。ロール定義の詳細は要件定義フェーズで確定する。

## 6. 既存システムとの連携範囲

| 連携先 | 方向 | 方式 | 備考 |
|---|---|---|---|
| 生産管理システム (製造指示) | 受信 | CSV インポート (手動) | API リアルタイム連携は Phase 3 以降 |
| 受入検査システム | 受信 | CSV インポート (手動) | ロット入荷情報の参照用 |
| ERP / 出荷システム | 送信 | CSV エクスポート (手動) | 完成実績の報告 |
| 測定器 (計量器) | 受信 | 手入力 (タブレット) | 自動測定値取込は Phase 2 以降 |

現時点では **すべての外部連携を CSV 手動インポート/エクスポート** に限定する。
リアルタイム API 連携は社内ネットワーク設計・IT 部門承認が必要なため要件定義後に評価する。

---

> **本節で確定した方針:** Phase 1 の外部連携は CSV 手動のみ。自動 API 連携・測定器直結は Phase 2/3 に分類する。

## 7. 制約事項

本システムの設計・実装全体を縛る決定済み制約を示す。

| 制約 | 内容 | 変更可否 |
|---|---|---|
| 開発体制 | 個人開発 (シングル開発者) | 不変 |
| ネットワーク | 社内 LAN のみ (インターネット非接続) | 不変 |
| 認証方式 | JWT (Bearer Token) | 変更するには要件定義フェーズで ADR が必要 |
| データ保存地 | オンプレサーバ (Windows Server 2022) | 不変 |
| OS / 仮想化 | IIS + WSL2 + Docker | 社内標準のため変更不可 |
| タブレット OS | Android / iOS / Windows | 端末選定は現場に委ねる |
| ライセンス | OSS 優先 (商用ライセンスは要承認) | 追加時は都度評価 |
| 言語 | 日本語主体 (Phase 2 以降で多言語追加) | Phase 1 では日本語のみ |
