# 02_想定利用者とユースケース概要

## 1. ペルソナ

本システムが直接・間接に関わる利用者を 4 種のペルソナとして定義する。
ペルソナ設計の根拠は現場 HCI 研究
([`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md))、
多能工管理
([`90_業界分析/26_多能工とスキル管理・作業者資格.md`](../../90_業界分析/26_多能工とスキル管理・作業者資格.md))、
高齢化・シフト研究
([`90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md`](../../90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md))、
多言語化研究
([`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md))
に基づく。

---

### 1.1 ペルソナ A: 現場作業員 (主役)

| 項目 | 内容 |
|---|---|
| 役割 | 工程を担当し、作業ナビに従って組立・検査・測定・スキャンを行う |
| 人数規模 | 数名〜数十名 (単一工場) |
| デバイス | 工場支給タブレット (Android または Windows) |
| 技術リテラシー | 低〜中。スマートフォン操作に慣れた世代が増加中 |
| 特記事項 | 油・粉塵環境で手袋を着用したまま操作する場合がある |

**高齢化と多能工化の実態**

製造業就業者の高齢化が進行し、50 歳以上の比率が上昇している。
高年齢者雇用安定法 (2021 年改正) により 70 歳までの就業機会確保が努力義務化されており、
視認性・フォントサイズ・タップ面積の設計に配慮が必要
([`90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md`](../../90_業界分析/37_シフト・夜勤・疲労管理と高齢化労働力.md))。

多能工化 (複数工程を 1 人が担当) は生産変動への対応力を高める一方で、
不慣れな工程でのエラーリスクも伴う。Dreyfus モデルの「初心者」レベルに対して
ナビが補完機能を果たすことが特に重要
([`90_業界分析/26_多能工とスキル管理・作業者資格.md`](../../90_業界分析/26_多能工とスキル管理・作業者資格.md))。

**外国人作業員**

製造業の外国人労働者は 2023 年 10 月時点で約 45 万人 (全体の 22%)。
ベトナム・中国・フィリピン・ブラジル・インドネシアの順に多い
([`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md))。
日本語能力は JLPT N3〜N4 レベルが就労閾値とされるが、専門技術用語の読解には困難が残る。
本システムは Phase 1 でやさしい日本語とピクトグラム (ISO 7010 準拠) を採用し、
Phase 2 以降でベトナム語・英語対応を追加する (§4 参照)。

---

### 1.2 ペルソナ B: 班長 / 職長

| 項目 | 内容 |
|---|---|
| 役割 | 工程の割り当て確認、作業員への指示、不適合発生時の一次対応 |
| デバイス | タブレットまたは PC (Web 管理画面) |
| 主な利用機能 | 実績照会、不適合リスト、作業員スキル確認 |
| 特記事項 | 本システムによる配員最適化・生産計画変更はスコープ外 |

班長はチームとして状況認識 (Situation Awareness) を維持する役割を持つ
([`90_業界分析/20_作業中断・割込み・再開の認知科学.md`](../../90_業界分析/20_作業中断・割込み・再開の認知科学.md))。
本システムは実績データをリアルタイムで照会可能にすることで SA レベル 1 (知覚) を支援する。

---

### 1.3 ペルソナ C: 品質保証 (QA) 担当者

| 項目 | 内容 |
|---|---|
| 役割 | 記録の後閲覧・トレサビ照会・不適合の根本原因分析・CAPA 起票 |
| デバイス | PC (Web 管理画面) |
| 主な利用機能 | トレサビ検索、監査ログ閲覧、不適合管理、CSV エクスポート |
| 規制要求 | ALCOA+ 準拠記録の確認、規制査察対応資料の出力 |

QA は FDA 21 CFR Part 11 や IATF 16949 などの規制要件に基づいて
電子記録の完全性・正確性・真正性を評価する
([`90_業界分析/22_規制別トレーサビリティ要件詳論.md`](../../90_業界分析/22_規制別トレーサビリティ要件詳論.md))。
本システムの Audit Trail と Append-only ログはこの要求に対応する。

---

### 1.4 ペルソナ D: 情シス兼マスタ管理者

| 項目 | 内容 |
|---|---|
| 役割 | 作業手順・部品・ユーザのマスタ管理、サーバ運用・バックアップ |
| デバイス | PC (Web 管理画面 + SSH) |
| 主な利用機能 | マスタ CRUD、ユーザ管理、ロール設定、DB バックアップ |
| 特記事項 | 個人開発のため技術的な IT リテラシーが高いことを前提とする |

国内製造業では情報システム部門が少人数 (1〜3 名) で複数システムを掛け持つ構造が多い
([`90_業界分析/30_国内製造業IT調達慣行とSI構造.md`](../../90_業界分析/30_国内製造業IT調達慣行とSI構造.md))。
本システムは管理コストを最小化する設計 (マスタ管理 UI の操作性・Docker Compose による起動手順の簡略化) を優先する。

---

> **本節で確定した方針:** ペルソナは作業員 / 班長 / QA / 情シスの 4 種とする。外部監査員は「読み取り専用ロール」として別途ロール定義するが独立ペルソナとはしない。

## 2. 主要ユースケース概要

### 2.1 UC-01: 作業の着手から完了

```
前提: 製造指示がシステムに登録済み (手動インポートまたは直接入力)

1. 作業員がタブレットにログイン (JWT 認証)
2. 担当工程を選択
3. システムが最新版の作業手順を表示
4. 手順ステップに従って作業を実施
   - 各ステップで: テキスト指示 / 参考画像 / チェックボックス / 数値入力 / QR スキャン
   - [重要] 各ステップの入力確定と同時にローカル SQLite へ記録 + Outbox に「送信待ち」追加
   - オンライン時はバックグラウンドで即座にサーバへ逐次送信（バッチ送信ではない）
5. 全ステップのチェック完了後に「完了」ボタンを押下
   - 「工程全体の完了」イベントを Outbox に 1 件追加する操作
   - 各ステップの実績データはステップ 4 で既に記録済み／送信済み
```

電子チェックリストは「確認した」という積極的な意思確認を促す設計とし、
スキップや一括チェックを防ぐため強制順序制御を設ける
([`90_業界分析/19_電子チェックリストと手順遵守の科学.md`](../../90_業界分析/19_電子チェックリストと手順遵守の科学.md))。

#### 2.1.1 端末障害・紛失時の影響範囲

設計の重要な帰結として、**端末が物理的に破損・紛失した場合でも失われるデータは最小**である。

| 状況 | 失われるデータ | 復旧方法 |
|---|---|---|
| **オンライン中に端末故障** | 入力中だった 1 ステップ分のみ | 別端末で同製造指示にログインし、サーバ側の完了済みステップを確認して残りを再入力 |
| **オフライン中に端末故障** | オフライン開始以降に入力した全ステップ（端末の SQLite ファイルが回復不能な場合） | 同上。SQLite ファイルが物理的に読み出せれば吸い出して救済可能。Phase 1 では手動再入力を運用ルールとする |
| **完了ボタン押下前に端末故障** | 完了イベント 1 件のみ未送信（ステップ記録は送信済み） | 別端末で「完了」操作を再実行 |

完了済みステップ N 件はローカル SQLite に Append-only 保存済みであり、
オンライン時はサーバへも送信済みのため復旧可能。
逐次同期の採用により、**規制上の「記録漏れ」リスクは端末の物理損失と
入力中ステップの偶発的損失に限定される**
([`90_業界分析/27_オフライン同期とデータ整合性.md`](../../90_業界分析/27_オフライン同期とデータ整合性.md)、
[`90_業界分析/38_災害・BCP・緊急時手順と作業継続.md`](../../90_業界分析/38_災害・BCP・緊急時手順と作業継続.md))。

---

### 2.2 UC-02: 中断と再開

製造現場では割り込み (電話・緊急指示・休憩) による作業中断が頻発する。
中断後の再開時に「どこまで完了したか」を正確に把握できない場合、
作業の重複・省略・誤った手順再実行が発生する
([`90_業界分析/20_作業中断・割込み・再開の認知科学.md`](../../90_業界分析/20_作業中断・割込み・再開の認知科学.md))。

```
中断時: タブレットで「中断」ボタンを押下
       → 完了済みステップ状態はローカル SQLite に保存済み + サーバに同期済み (オンライン時)
再開時: 前回の中断箇所から自動復帰 → 未完了ステップを強調表示
引き継ぎ時: 別の作業員が同一製造指示を引き継ぐ場合は上長承認が必要 (Phase 2)
```

---

### 2.3 UC-03: 不適合の発見と起票

```
1. 手順ステップで規格外の測定値・外観異常を発見
2. 「不適合起票」ボタンを押下
3. 不適合種別・発見ステップ・写真・コメントを入力
4. 起票時点の作業実績・部品ロット情報が自動紐づけ
5. QA 担当者の通知は外部連携 (メール等) が必要 → Phase 2 以降
```

CAPA フロー (8D / 5 Whys) との完全統合は Phase 2 に分類するが、
不適合記録の基盤となるデータ構造は Phase 1 から設計する
([`90_業界分析/28_不適合と手順改訂のフィードバックループ.md`](../../90_業界分析/28_不適合と手順改訂のフィードバックループ.md))。

---

### 2.4 UC-04: トレサビ照会 (QA 向け)

```
入力: ロット番号 / 製品シリアル / 作業日時 / 作業者 ID のいずれか
出力:
  - 対象ロットを使用した全工程の実績一覧
  - 各ステップの入力値・スキャン値・写真リスト
  - 担当作業員と時刻
  - 適用した手順書のバージョン
```

前方 (Forward) トレース: 部品ロット → 完成品
後方 (Backward) トレース: 完成品 → 使用部品ロット の双方向対応
([`90_業界分析/06_品質管理とトレーサビリティ.md`](../../90_業界分析/06_品質管理とトレーサビリティ.md))。

---

### 2.5 UC-05: マスタ管理 (情シス / 生技向け)

```
作業手順書の新規作成・改訂・公開フロー:
1. ドラフト作成 (工程・ステップ・メディア・チェック項目)
2. 内部レビュー (承認フローは Phase 2)
3. 有効日付設定と公開
4. 旧版の参照保持 (改訂版 SOP は削除せず保存)
5. 変更履歴の Audit Trail 記録
```

SOP の版管理と「Effective Date」連動は GMP / DITA の基本要件
([`90_業界分析/25_作業指示書とSOPの構造化・表現論.md`](../../90_業界分析/25_作業指示書とSOPの構造化・表現論.md))。
旧版を削除しない Append-only ポリシーは ALCOA+ の「持続性 (Enduring)」要件でもある。

---

> **本節で確定した方針:** Phase 1 に含む UC は UC-01/02/03/04/05。承認ワークフロー (UC-03 引継・UC-05 レビューフロー) は Phase 2 に分類し、データ構造のみ Phase 1 で設計する。

## 3. 利用環境想定

本システムのタブレット利用は厳しい物理環境下での操作を前提とする。

### 3.1 物理環境分類

| 環境分類 | 特徴 | 対応要件 |
|---|---|---|
| 一般組立・検査 | 室温・屋内・油なし | IP52 以上 / 800nits 以上 |
| 機械加工・溶接 | 金属粉・切削油飛散 | IP65 以上 / 手袋操作対応 |
| 食品・医薬品 | 洗浄・殺菌液飛散 | IP65 以上 / 化学耐性ケース |
| 防爆エリア (一部) | 可燃性ガス・粉塵 | ATEX/TIIS 認証機器が必要 — **本システムのスコープ外** |

防爆エリアでの非防爆機器使用は法令違反となるため、
ATEX Zone 0〜2 / TIIS 認証が必要な環境への本システム導入は **対象外と判断する**
([`90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md`](../../90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md))。

### 3.2 推奨タブレットスペック

| 項目 | 推奨基準 | 根拠 |
|---|---|---|
| IP 等級 | IP65 以上 | 粉塵完全防護・ノズルによる放水 |
| 落下耐性 | MIL-STD-810H 準拠 | 1.2m 落下試験 |
| 輝度 | 800nits 以上 | 屋外・蛍光灯下での可視性 |
| タッチ感度 | 手袋モード対応 | 作業手袋着用時の操作 |
| バッテリ | 8 時間以上の連続使用 | シフト 1 直分 |

参考機種: Panasonic TOUGHBOOK FZ-G2 / Samsung Galaxy Tab Active5 / Zebra ET45
([`90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md`](../../90_業界分析/35_環境耐性と防爆・クリーンルーム設計.md))。
端末の選定は現場の責任において行い、本システムは上記基準を推奨として提示する。

---

> **本節で確定した方針:** タブレット推奨スペックを IP65 / MIL-STD-810H / 800nits 以上とする。防爆エリアへの適用は対象外と判断する。端末選定は現場担当者が行い、開発側は責任を持たない。

## 4. アクセシビリティと多言語対応方針

### 4.1 タッチ操作の設計基準

Fitts の法則に基づくタッチターゲットの最小サイズは、
素手では Material Design の 48dp、手袋着用時は **72dp 以上** を推奨する
([`90_業界分析/18_現場HCIと作業者インターフェース.md`](../../90_業界分析/18_現場HCIと作業者インターフェース.md))。

UI 設計方針:
- ボタン・チェックボックスの最小タッチ面: 72dp × 72dp (手袋対応基準)
- 重要操作 (完了 / 不適合起票) は画面下部大ボタン配置
- 情報密度を下げた「Glanceable UI」— 1 ステップ 1 画面を基本とする
- Nielsen ヒューリスティクスの「エラー防止」原則に基づく確認ダイアログ

### 4.2 高齢者対応

- フォントサイズ: 最小 16sp、推奨 18〜20sp
- コントラスト比: WCAG 2.1 AA (4.5:1) 以上
- アニメーション: 最小限 (前庭感覚障害への配慮)

### 4.3 多言語対応ロードマップ

外国人作業員への絵表示指示 (Pictorial Work Instructions: PWI) は
エラーを **26% 削減** するという実験結果がある
([`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md))。

| フェーズ | 対応内容 |
|---|---|
| Phase 1 | やさしい日本語 (8 原則準拠)、ISO 7010 ピクトグラム、i18n 基盤 (i18next) の組み込み |
| Phase 2 | ベトナム語・英語の翻訳データ追加 |
| Phase 3 | 中国語・ポルトガル語、ピクトグラム優先モード、機械翻訳は安全用語に不使用 |

機械翻訳 (DeepL / Google 翻訳) は安全上重要な指示・単位・数値への適用を禁止する。
翻訳は人手によるレビューを経た確定訳のみをシステムに登録する。

### 4.4 識字困難への対応

PIAAC 調査では製造業に多い在留外国人の一定割合が読解力 PIAAC Level 2 以下 (中学生程度)。
ステップ指示は「一文一義・主語明示・漢語熟語回避」の「やさしい日本語」8 原則
([`90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md`](../../90_業界分析/34_多言語化・外国人労働者と読み書き能力差.md))
に則りマスタ作成者が作成する。システムは自動検証機能は持たず、マスタ作成ガイドラインとして提供する。

---

> **本節で確定した方針:** Phase 1 の UI は手袋 72dp 基準・高コントラスト・やさしい日本語・ISO 7010 ピクトグラムを実装する。多言語翻訳データの追加は Phase 2 とし、機械翻訳を安全用語に適用しない方針は全フェーズで不変とする。
