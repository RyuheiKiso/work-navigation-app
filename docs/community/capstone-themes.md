# Capstone プロジェクトテーマ集

> 対応 §: ロードマップ §19.4.2 §19.4.4 §4.9 §22.1
> 対象読者: 大学・大学院生、学術界の研究者、本プロジェクトに学術的貢献を行いたいコミュニティメンバー
> 改訂サイクル: §22.1 半期サイクル

§19.4.2「貢献者誘致」で示した学術界との接続点を具体化し、学生 Capstone プロジェクトのテーマを本書で版管理する。HCI／CSCW／ICSE／IEEE TII の各カンファレンス・ジャーナルへの CFP 提案候補も併記する。

## 1. テーマ一覧

### T-01: 作業ナビゲーションの形式化

- **領域**: 形式手法／HCI
- **概要**: §3.4.1 で確定した HSM／CPN／TLA+ 三層モデルが、現実の作業フローを **どこまで** 表現できるかの限界調査。新規アンチパターン（§3.6.4）の発見を目指す。
- **想定貢献**: §3.6.4 アンチパターン表の追加行／§4.9 圧倒候補機能「ナビゲーションの形式化」の学術的裏付け
- **関連 CFP**: ICSE Software Engineering in Practice／IEEE TII Special Issue

### T-02: 製造業 OSS のコミュニティ生態系分析

- **領域**: 経験的ソフトウェア工学／オープンソース研究
- **概要**: §19.4 段階達成の阻害要因／成功要因を、本プロジェクトと類似 OSS（Sourcegraph／Apache Camel／PostgreSQL ほか）で比較分析。
- **想定貢献**: §19.4 段階改訂／§29 R-001／R-002 の発生確率見直し
- **関連 CFP**: CSCW Open Source Studies Track

### T-03: 製造現場の心理的ストレス予測モデル検証

- **領域**: HCI／認知科学
- **概要**: §9.2.1 5 軸予測モデルが、実機エミュレーション（NASA-TLX 想定値）と現場相当環境（§9.5.5）でどこまで整合するかを評価する。
- **想定貢献**: §9.2.1 軸定義の精緻化／§10.2.3 ストレス予測ブロック条件の精度向上
- **関連 CFP**: CHI Late-Breaking Work／IEEE TII Manufacturing UX

### T-04: 同期競合 CRDT の性能と現場適合性

- **領域**: 分散システム
- **概要**: §10.6.1 G-Set／LWW-Register／Lamport timestamp の組合せが、製造現場の負荷パターン（朝礼後ピーク × 10 倍、§5.3）でどう振る舞うかを実証。代替モデル（OT／HLB）との比較を含む。
- **想定貢献**: §10.6.2 受入観点の追加／§31 SLI-04／SLI-07 の現場適合化
- **関連 CFP**: PODC／OSDI／IEEE TII

### T-05: アドオン WASM サンドボックスの境界調査

- **領域**: セキュリティ／システム
- **概要**: §17.5 Wasmtime サンドボックスの実効的な境界（メモリ／CPU／ネットワーク）が、悪意のあるアドオンに対しどこまで頑健かを実機検証。
- **想定貢献**: §27 F-004 検出度 D の改善／§29 R-008 影響度の見直し
- **関連 CFP**: USENIX Security／IEEE S&P Workshop

### T-06: 教育素材ライブラリの循環設計

- **領域**: 教育工学／HCI
- **概要**: §25.3 端末撮影 → 教育素材化の循環が、現場の学習効率に与える影響を調査。OPT-IN／OPT-OUT の比較。
- **想定貢献**: §25 教育設計指標の精緻化／§4.9 圧倒候補機能「教育素材ライブラリの循環設計」の裏付け
- **関連 CFP**: SIGCSE／L@S

### T-07: 行動心理学指針の UI への射影

- **領域**: 行動経済学／HCI
- **概要**: §9.2.3 ナッジ／デフォルト効果／プロスペクト理論／ピーク・エンドの法則／ゴール・グラディエント効果を、設定 UI（§10.2.2 14 観点）に射影した際の効果量。
- **想定貢献**: §10.2.2 観点の優先順位付け／ダーク・パターン排除（§9.2.4）の実証
- **関連 CFP**: CHI Full Paper／JACM Behavioral Economics Issue

### T-08: SCI（Software Carbon Intensity）の OSS／SaaS 比較

- **領域**: グリーンソフトウェア／産業情報学
- **概要**: §11.7.3 SCI 試算を、本アプリ（オンプレ＋Rust／tokio）と SaaS 型競合で比較し、構造的差異を実証。
- **想定貢献**: §4.9 OSS の TCO 差を構造的優位として裏付け／§11.7 サステナビリティ指標精緻化
- **関連 CFP**: ICT4S／IEEE Green Computing

### T-09: ロケール拡張の現場適合性

- **領域**: 国際化／HCI
- **概要**: §11.3.1 拡張順（東アジア → 欧州 → ASEAN → 中華 → RTL）が、製造業集積実態と整合しているかを GDP × 製造業比率 × OSS 採用度合のクロス分析で検証。
- **想定貢献**: §11.3.1 拡張順の改訂／§3.1.5.2 多言語対比表の追加行
- **関連 CFP**: IUI／CHI Late-Breaking Work

### T-10: ADR と Type 1／Type 2 決定の運用効果

- **領域**: ソフトウェア工学
- **概要**: §30 Type 1／Type 2 分類が、個人開発 OSS の意思決定速度と品質のトレードオフにどう寄与するかを、本プロジェクトと類似 OSS で比較。
- **想定貢献**: §30 Type 1／Type 2 境界の精緻化／§9.4 ADR 運用ガイドの強化
- **関連 CFP**: ICSE Software Engineering in Practice／EASE

## 2. 提案フロー

1. 学生・研究者が GitHub Discussions で本書のテーマ ID を引用してディスカッション開始。
2. メンテナとスコープ調整（4 週間）。
3. 提案を Issue 化（ラベル `capstone`）。
4. 月次でメンテナが進捗確認（§19.4 レビュー SLA 7 日と整合）。
5. 成果は CHANGELOG／Discussions で公表し、本書を更新（テーマの達成内容を「達成済テーマ」として残す）。
