# 04 テストデータ管理計画

本章はテストデータの生成・管理・廃棄に関するルールを確定する。本番データの利用を全面禁止し、ファクトリパターン・シード SQL・PII 合成基準を明示する。根拠は BR-BUS-029（個人情報保護）および NFR-SEC-020（本番データ分離）である。

---

## 1. 本番データ利用全面禁止

すべてのテスト環境（TENV-001〜008）において、本番データベースのデータを直接参照・複製・持ち込むことを禁止する。

| 禁止事項 | 根拠 |
|---|---|
| 本番 DB への接続（読み取りも禁止）| NFR-SEC-020 |
| 本番ダンプからのリストア | BR-BUS-029 |
| 実在の社員番号・氏名の使用 | NFR-SEC-020 |
| 実在の計測値・工程データの使用 | BR-BUS-029 |

違反を検出した場合は即時テストを中断し、DEFECT-NNN（P1）として記録する。

**本節で確定した方針**
- **本番データ利用を全面禁止とし、違反を P1 欠陥として扱うことを確定する。**
- **テストデータの唯一の調達源をファクトリパターンとシード SQL に限定することを確定する。**

## 2. ファクトリパターン

### 2-1. Rust バックエンド（fake クレート）

```rust
use fake::faker::name::ja_jp::*;
use fake::Fake;

struct TestEmployeeFactory;

impl TestEmployeeFactory {
    fn create() -> Employee {
        Employee {
            employee_id: format!("TST-{:04}", (1000..9999).fake::<u32>()),
            name: Name().fake::<String>(),
            role: Role::Operator,
        }
    }
}
```

| 項目 | 設定 |
|---|---|
| クレート | `fake` |
| ロケール | `fake::faker::name::ja_jp` |
| 社員番号プレフィックス | `TST-`（本番と区別するためのプレフィックス）|
| UUID | `Uuid::new_v4()` で毎回生成 |

### 2-2. TypeScript フロントエンド（@faker-js/faker）

```typescript
import { faker } from "@faker-js/faker/locale/ja";

function createTestEmployee() {
  return {
    employeeId: `TST-${faker.number.int({ min: 1000, max: 9999 })}`,
    name: faker.person.fullName(),
    role: "operator",
  };
}
```

**本節で確定した方針**
- **Rust は `fake` クレート（`ja_jp` ロケール）・TypeScript は `@faker-js/faker/locale/ja` を使用することを確定する。**
- **社員番号に `TST-` プレフィックスを必須とし、本番データとの混同を防止する。**

## 3. シード SQL 管理

シード SQL は `db/seeds/test/` 配下に 4 ファイルで管理する。

| ファイル名 | 内容 | 件数 |
|---|---|---|
| `01_roles.sql` | ロールマスタ（6 ロール）| 6 件 |
| `02_sop_templates.sql` | 架空 SOP テンプレート（3 種）| 3 件 |
| `03_instruments.sql` | 架空計測器マスタ（5 件）| 5 件 |
| `04_config.sql` | CFG 設定値 | 可変 |

### シードの投入タイミング

| 環境 | 投入タイミング | コマンド |
|---|---|---|
| TENV-002（統合）| マイグレーション適用後 | `psql -f db/seeds/test/01_roles.sql ...` |
| TENV-006（性能）| シード後に 1 万件追加シード | `psql -f db/seeds/test/*.sql && k6 run seeds/perf_seed.js` |
| TENV-008（UAT）| UAT 開始前リセット時 | 同上 |

**本節で確定した方針**
- **シード SQL は `db/seeds/test/` の 4 ファイルのみとし、ファイル外でのマニュアル INSERT を禁止する。**
- **シード SQL ファイルは git 管理し、本番環境への誤投入を防ぐため `seeds/prod/` との混在を禁止する。**

## 4. PII 合成基準

| 項目 | 合成基準 |
|---|---|
| UUID | `Uuid::new_v4()` で実行ごとに新規生成 |
| 氏名 | ファクトリ生成の架空名（`fake::ja_jp` または `faker/ja`）|
| タイムスタンプ | 固定範囲 `2026-01-01` 〜 `2026-12-31` 内の乱数 |
| 社員番号 | `TST-` プレフィックス + 4 桁数字 |
| IP アドレス | `192.0.2.0/24`（TEST-NET-1 RFC 5737）|
| メールアドレス | `test-NNN@example.com`（RFC 2606 準拠）|

### PII 混入検出スクリプト（CI 統合）

```bash
#!/bin/bash
# scripts/check-pii.sh
# シード SQL・ファクトリコードに実在 PII が混入していないことを確認する

# 実在メールドメイン（example.com 以外）の検出
grep -rn "@\(gmail\|yahoo\|icloud\|company\)\." db/seeds/test/ && {
  echo "PII 混入を検出しました。シード SQL を確認してください。"
  exit 1
}

# TST- プレフィックスなし社員番号パターンの検出
grep -rn '"[A-Z][0-9]\{4\}"' db/seeds/test/ && {
  echo "本番形式の社員番号が混入している可能性があります。"
  exit 1
}

echo "PII 混入チェック: OK"
```

CI パイプライン（`scripts/test-all.sh` および GitHub Actions）にこのスクリプトを組み込み、シード SQL 変更時に自動実行する。

**本節で確定した方針**
- **PII 合成基準を本章の表で確定し、基準外のデータをシード SQL に含めることを禁止する。**
- **PII 混入検出スクリプト（`scripts/check-pii.sh`）を CI パイプラインに統合することを確定する。**
