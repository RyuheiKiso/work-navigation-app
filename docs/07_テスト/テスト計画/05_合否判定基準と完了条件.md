# 05 合否判定基準と完了条件

本章は TST-001〜007 各受入基準の合否判定手順・品質ゲート 6 領域（RGATE-001〜006）の合格条件・重大度別再テスト範囲・GO/NO-GO 判定プロセスを確定する。判定はすべてツール計測値に基づき、主観的評価による合否判定を禁止する。

---

## 1. TST-001〜007 合否判定手順

各受入基準の Pass/Fail は以下の計測値で判定する。

| 要件 ID | 判定手順 | 計測ツール |
|---|---|---|
| TST-001 | FR Must/Should の Pass 率を RTM で集計。100% 未達時は Fail | RTM 集計スクリプト |
| TST-002 | 全 UC（UC-001〜022）の全シナリオ（正常・代替・例外）の Pass を確認 | Playwright / Detox レポート |
| TST-003 | P99 応答時間を k6 で計測。管理 Web 全エンドポイントで ≤ 3000ms を確認 | k6 サマリレポート |
| TST-004 | `cargo audit`・`npm audit` CVSS≥7.0 ゼロ、ZAP High ゼロを確認 | cargo audit / ZAP HTML レポート |
| TST-005 | TST-alcoa-001〜009 の全件 Pass を確認。1 件でも Fail で不合格 | cargo test / sqlx::test レポート |
| TST-006 | 移行後の整合性クエリ全件 Pass・制約違反ゼロ・ロールバック実機確認完了 | psql クエリ結果 |
| TST-007 | Outbox 4 シナリオ全 Pass・重複記録ゼロ・最終送信成功率 100% | sqlx::test レポート |

**本節で確定した方針**
- **TST-001〜007 の判定はすべてツール計測値を根拠とし、「おおむね動いている」等の主観評価を根拠として認めないことを確定する。**
- **7 件のうち 1 件でも Fail の場合はリリースを禁止することを確定する。**

---

## 2. 品質ゲート 6 領域（RGATE-001〜006）

| RGATE | 領域 | 合格条件 | 測定コマンド |
|---|---|---|---|
| RGATE-001 | コード品質 | `cargo clippy` 警告ゼロ（`-D warnings`） + ESLint エラーゼロ（`--max-warnings 0`） | `cargo clippy --all-targets -- -D warnings` |
| RGATE-002 | テストカバレッジ | Unit ≥80%（Rust ドメイン ≥90%）/ BR-BUS 統合テスト 100% / E2E UC ≥95% | `cargo llvm-cov` / `jest --coverage` |
| RGATE-003 | セキュリティ | CVSS HIGH/CRITICAL ゼロ + ZAP High アラートゼロ | `cargo audit` / `npm audit` / ZAP |
| RGATE-004 | API 準拠 | Schemathesis 全エンドポイント準拠率 100% | `schemathesis run` |
| RGATE-005 | 性能 | P95 ≤ 2000ms（50 VU・混合シナリオ）+ 5xx エラーレート 0% | `k6 run` |
| RGATE-006 | DTM 完全性 | M5 未対応 NFR ゼロ + M6 統合テストゼロ BR-BUS ゼロ | `scripts/check-dtm-m5.sh` / `check-dtm-m6.sh` |

**本節で確定した方針**
- **RGATE-001〜006 の 6 領域すべての合格を ver1.0.0 リリース条件として確定する。1 領域でも未達の場合はリリースをブロックする。**
- **各ゲートの合格は測定コマンドの出力が判定基準を満たすことで確認し、目視確認のみの合格を禁止する。**

---

## 3. 重大度別再テスト範囲

| 重大度 | 判定基準 | 再テスト範囲 | 修正期間上限 |
|---|---|---|---|
| Critical | TST-001/002/005 の不合格・データ破損・ALCOA+ 違反 | 受入テスト全件（TST-001〜007）再実施 | 制限なし（修正完了まで継続）|
| Major | TST-003/004/006 の不合格・性能未達・脆弱性検出 | 当該要件のみ再実施 | 最大 2 週間（超過時 Critical に昇格）|
| Minor | TST-007 の部分 Fail・軽微な表示不具合 | TST-007 のみ再実施 | 最大 1 週間（超過時 Major に昇格）|

**本節で確定した方針**
- **Critical 欠陥の修正後は全件再テストを必須とし、省略を認めないことを確定する。**
- **修正期間上限を超過した場合は自動的に重大度を昇格させることを確定する。**

---

## 4. 最終デッドラインとスコープ調整

- リリース目標日から 6 週間を超えて受入テストが合格しない場合、スコープ調整を実施する。
- スコープ調整の優先順位：Could 要件の削除 → Should 要件の一時除外。
- Must 要件のスコープ削除は禁止する。いかなる状況でも Must の削除でリリース条件を変更しない。

**本節で確定した方針**
- **最終デッドライン（目標日＋6 週間）超過時のスコープ調整手順を確定し、Must 要件削除を絶対禁止とする。**

---

## 5. GO/NO-GO 判定プロセス

TST-010〜016 に準拠した 5 ステップで判定する。

| ステップ | 内容 |
|---|---|
| Step 1 | 受入テスト実施（TST-001〜007 全件）|
| Step 2 | RGATE-001〜006 全領域の合格確認 |
| Step 3 | Pass/Fail 集計表の作成 |
| Step 4 | GO（全 7 件全 Pass + RGATE-006 全通過）または NO-GO（1 件以上 Fail）の宣言 |
| Step 5 | 判定記録を `docs/07_テスト/テスト計画/証跡/` に保存（電子署名機能使用）|

**本節で確定した方針**
- **GO 宣言の条件を TST-001〜007 全 Pass かつ RGATE-001〜006 全通過の両条件の成立に限定することを確定する。どちらか一方のみの合格は GO として認めない。**
