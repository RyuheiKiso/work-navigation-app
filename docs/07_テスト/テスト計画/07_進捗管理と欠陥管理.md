# 07 進捗管理と欠陥管理

本章はテスト実施の進捗追跡・欠陥管理・証跡保存の手順を確定する。TST-NNN のステータスライフサイクル・DEFECT-NNN のテンプレート・回帰失敗時の 5 ステップ対応を定める。

---

## 1. TST-NNN ステータスライフサイクル

TST-NNN（全 116 件）のステータスは以下のライフサイクルに従って遷移する。

```
DEFINED → RUNNING → PASSED
                  → FAILED → (修正後) → REGRESSION → PASSED
                                                    → FAILED (P1/P2 昇格)
DEFINED → SKIPPED (承認済みのスキップのみ)
```

| ステータス | 意味 |
|---|---|
| DEFINED | テストケースが定義済みで未実行 |
| RUNNING | 実行中 |
| PASSED | 合格 |
| FAILED | 不合格。DEFECT-NNN を起票する |
| REGRESSION | 修正後の回帰テスト実施中 |
| SKIPPED | 承認済みのスキップ（ADR-TEST-NNN 必須）|

SKIPPED への遷移は ADR-TEST-NNN の承認なしに行うことを禁止する。

**本節で確定した方針**
- **TST-NNN のステータス遷移を本章のライフサイクル図に従うことを確定し、FAILED を PASSED に直接変更することを禁止する（必ず REGRESSION を経由する）。**
- **SKIPPED は ADR-TEST-NNN 承認を必須とし、無断スキップを禁止する。**

---

## 2. DEFECT-NNN 欠陥報告テンプレート

欠陥は DEFECT-001 から連番で採番する（全テストレベル横断）。

```markdown
## DEFECT-NNN

| フィールド | 内容 |
|---|---|
| 欠陥 ID | DEFECT-NNN |
| 発見日 | YYYY-MM-DD |
| 重大度 | P1 / P2 / P3 |
| 状態 | OPEN / IN_PROGRESS / FIXED / VERIFIED / CLOSED |
| 発見テストレベル | L1 / L2 / L3 / L4 / ALCOA+ / セキュリティ / 性能 / UAT |
| 発見 TST-ID | TST-xxx-NNN |
| 再現手順 | （ステップごとに記述）|
| 期待動作 | |
| 実際の動作 | |
| 修正コミット | git SHA |
| 修正確認日 | YYYY-MM-DD |
```

### 欠陥状態遷移

```
OPEN → IN_PROGRESS → FIXED → VERIFIED → CLOSED
                           → OPEN (検証不合格の場合)
```

**本節で確定した方針**
- **DEFECT-NNN のテンプレートを本章で確定し、全フィールドの記入を必須とする。空欄のある欠陥報告を VERIFIED とすることを禁止する。**
- **欠陥採番は全テストレベル横断の DEFECT-001 からの連番とし、テストレベルごとの別採番を禁止する。**

---

## 3. P1 欠陥の対応 SLA

| ステップ | 内容 | 期限 |
|---|---|---|
| 全 UAT 中断 | 進行中の UAT を即時停止 | 検出から 30 分以内 |
| DEFECT-NNN 起票（P1）| 全フィールドを記入 | 検出から 1 時間以内 |
| 影響範囲特定 | RTM で関連 FR/NFR/BR-BUS を確認 | 検出から 4 時間以内 |
| 修正完了 | CI 全件通過を確認 | 修正完了まで継続（上限なし）|
| 全 UAT 再実施 | TST-001〜007 最初から全件 | 修正確認後即時 |

**本節で確定した方針**
- **P1 欠陥の起票を 1 時間以内・影響範囲特定を 4 時間以内に完了させることを確定する。**

---

## 4. 回帰失敗対応 5 ステップ

CI パイプラインで回帰テスト失敗が検出された場合、以下の 5 ステップを順に実施する。

| ステップ | 内容 | 期限 |
|---|---|---|
| 1. 検出・即時通知 | 失敗した TST-NNN の特定。GitHub Actions の失敗通知を確認 | 即時 |
| 2. 親 FR/NFR/BR-BUS 特定 | 失敗 TST-NNN から RTM を逆引きし、影響する上流要件を特定 | 検出から 2 時間以内 |
| 3. `git bisect` 実施 | 最後の PASS コミットと現在の FAIL コミットの間で `git bisect` を実行し、原因コミットを特定 | 検出から 24 時間以内 |
| 4. 修正コミット作成 | 原因コミットを修正し、CI パイプライン全件通過を確認 | 検出から 72 時間以内 |
| 5. 全テスト PASS + RTM 更新 | 修正後の全テストが PASS であることを確認し、RTM の状態列を更新する | 修正完了後即時 |

72 時間以内に修正が完了しない場合は DEFECT-NNN の重大度を P2 から P1 に昇格させ、全 UAT 中断手順（本章 §3）を適用する。

**本節で確定した方針**
- **回帰失敗の修正期限を 72 時間と確定し、超過時の P1 昇格・UAT 中断を自動適用する。**
- **`git bisect` による原因コミット特定を必須ステップとし、ランダムな修正試行を禁止する。**

## 5. TST-041 証跡保存規約

すべてのテスト実施記録・欠陥報告・修正確認は `docs/07_テスト/テスト計画/証跡/` に保存する。

| 成果物 | 保存先 | 命名規約 |
|---|---|---|
| テスト実行記録 | `証跡/TST-EXEC-YYYYMMDD-NNN.md` | 実施日ごとに作成 |
| 欠陥報告 | `証跡/DEFECT-NNN.md` | 採番順に作成 |
| UAT 完了記録 | `証跡/UAT-YYYYMMDD-Vx.md` | UAT 回次ごとに作成 |
| RGATE 判定記録 | `証跡/RGATE-判定-YYYYMMDD.md` | GO/NO-GO 宣言ごとに作成 |
| ZAP スキャン結果 | `reports/zap-baseline-YYYYMMDD.html` | スキャンごとに作成 |

証跡ファイルは作成後に修正することを禁止する。修正が必要な場合は新規の訂正記録として採番する（ALCOA+ Original・Enduring 原則準拠）。

**本節で確定した方針**
- **証跡は `証跡/` ディレクトリに命名規約に従って保存し、作成後の改ざんを禁止することを確定する（ALCOA+ Original 原則）。**
