# 07 欠陥管理と修正記録テンプレート

本章は SLCP-JCF2013 §6.1.1.3 および §6.1.2.4 に対応する結合テスト（L2 統合テスト / L4 契約テスト）で発見した欠陥の記録・追跡・修正確認のテンプレートを提供する。
DEFECT-NNN 採番規約は全テストレベル横断で共有するため、`07_テスト/テスト計画/00_本書の位置づけと識別子規約.md` §2-1 の採番シーケンスに従う。

---

## 採番規約

| 規約項目 | 内容 |
|---|---|
| 識別子フォーマット | `DEFECT-NNN`（3 桁ゼロ埋め）|
| 採番範囲 | 全テストレベル横断（単体・結合・適格性確認テストで同一シーケンス）|
| 採番権 | テスト実施者が発見順に採番する |
| 欠番 | 禁止（誤採番の場合は VOID: 理由 を記録して連番を維持する）|

---

## 優先度定義

| 優先度 | 定義 | 対応期限 |
|---|---|---|
| P1 | システムの動作を停止させる・データ破壊を引き起こす欠陥 | 発見後 24 時間以内に修正を開始する |
| P2 | 機能の主要パスに影響する欠陥（迂回策あり）| 次の CI 実行（PR マージ前）までに修正する |
| P3 | 軽微な動作異常・表示ズレ・性能劣化（機能影響なし）| 次スプリント内に修正する |

---

## 結合テスト固有の追加確認事項

L2 統合テストおよび L4 契約テストで発見した欠陥には以下の追加確認を実施する。

| 確認項目 | 内容 |
|---|---|
| Append-only 違反 | work_events または evidence_files の UPDATE/DELETE が成功している場合は P1 として扱う |
| ハッシュチェーン断絶 | hash_chain_blocks の chain_valid = false が検出された場合は P1 として扱う |
| 未定義 ERR コード | openapi.yaml に未定義の `error` コードが返却された場合は P2 として扱い、仕様書と実装を同時に修正する |
| RBAC 権限昇格 | 低権限ロールが高権限リソースへアクセスできた場合は P1 として扱う（TST-sec-001〜003 関連）|
| 冪等性違反 | 同一 event_id で 2 件以上の INSERT が発生した場合は P1 として扱う |

**本節で確定した方針**
- **DEFECT-NNN は全テストレベル横断で一意の採番シーケンスを使用し、欠番を禁止する。**
- **P1 欠陥（Append-only 違反・ハッシュチェーン断絶・RBAC 権限昇格・冪等性違反）は発見後 24 時間以内に修正を開始し、修正完了まで後続テストを停止する。**
- **L4 契約テストで発見した未定義 ERR コードは openapi.yaml と実装の両方を同時に修正することを必須とする。仕様書のみまたは実装のみの修正を禁止する。**
