# 01 ソフトウェア結合テスト仕様書実施手順

本章は SLCP-JCF2013 §6.1.1.3「ソフトウェア結合テストプロセス」に対応する L2 統合テスト（axum_test + sqlx::test）の実施手順を確定する。
上流のテストケース定義（TST-intg-001〜020）は `05_詳細設計/08_テストケース詳細設計/03_統合テストケース（API）.md` が権威ソースであり、本章はその実行手順のみを規定する。

---

## 1. テスト環境（TENV-002）起動手順

TENV-002（postgres:16-alpine, tmpfs, wnavdb_test, port 5433）を以下の手順で起動する。

### 1-1. コンテナ起動

```bash
# postgres_test コンテナのみを起動する
docker compose --profile test up -d postgres_test

# コンテナ起動確認（healthy になるまで待機）
docker inspect --format='{{.State.Health.Status}}' $(docker compose ps -q postgres_test)
# 出力: healthy
```

### 1-2. マイグレーション適用

```bash
export TEST_DATABASE_URL="postgres://wnav:wnav@localhost:5433/wnavdb_test"

# マイグレーションを適用する（冪等）
sqlx migrate run --database-url "$TEST_DATABASE_URL"

# マイグレーション適用確認
sqlx migrate info --database-url "$TEST_DATABASE_URL"
```

### 1-3. シード SQL 投入

各グループのテストに必要なシードデータを投入する。シード SQL は `tests/fixtures/` 配下に配置する。

```bash
# 認証テスト用シード（TST-intg-001〜005）
psql "$TEST_DATABASE_URL" -f tests/fixtures/seed_auth.sql

# ステップイベントテスト用シード（TST-intg-006〜010）
psql "$TEST_DATABASE_URL" -f tests/fixtures/seed_step_events.sql

# 証拠・署名・アンドンテスト用シード（TST-intg-011〜015）
psql "$TEST_DATABASE_URL" -f tests/fixtures/seed_evidences.sql

# SOP・Webhook テスト用シード（TST-intg-016〜020）
psql "$TEST_DATABASE_URL" -f tests/fixtures/seed_sop.sql
```

## 2. テスト実行コマンド

### 2-1. 全統合テスト実行

```bash
# 全統合テスト実行（並列実行はトランザクション競合を防ぐため無効）
cargo test --test integration -- --test-threads=1

# 特定グループのみ実行する場合（例: 認証グループ）
cargo test --test integration auth -- --test-threads=1
```

`sqlx::test(migrations = "./migrations")` アノテーションにより、各テスト関数はトランザクション内で実行され、終了時に自動ロールバックされる。これによりテスト間の状態汚染を防止する。

### 2-2. グループ別実行順

統合テストは以下の順序で実施する。前グループが全件 PASS してから次グループに進む。

| 順序 | グループ | TST-ID | 依存関係 |
|---|---|---|---|
| 1 | 認証 | TST-intg-001〜005 | なし（最初に実行）|
| 2 | ステップイベント記録＋ハッシュチェーン | TST-intg-006〜010 | 認証 JWT が必要 |
| 3 | 証拠ファイル・電子署名・アンドン | TST-intg-011〜015 | 進行中の作業実行 (exec_id) が必要 |
| 4 | SOP 承認・マスタ同期・Webhook | TST-intg-016〜020 | SOP・マスタデータが必要 |

## 3. グループ別前提セットアップ

### 3-1. グループ 1: 認証（TST-intg-001〜005）

前提セットアップ:
- `users` テーブルに `OP001`（operator ロール）・`is_active=true`・`failed_login_count=0` が存在すること
- `auth_logs` テーブルが空であること（自動ロールバックで保証）

JWT 生成（グループ 2 以降で使用）:
```bash
# ログイン成功後に JWT を取得して環境変数に保存する
export TEST_JWT=$(curl -s -X POST http://localhost:8080/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"login_id":"OP001","password":"correct_password_hash"}' \
  | jq -r '.access_token')
```

### 3-2. グループ 2: ステップイベント記録（TST-intg-006〜010）

前提セットアップ:
- `work_executions` テーブルに `status='IN_PROGRESS'` の作業が 1 件以上存在すること
- `$TEST_JWT` が有効期限内であること
- `hash_chain_blocks` が空であること（トランザクションロールバックで保証）

### 3-3. グループ 3: 証拠・電子署名・アンドン（TST-intg-011〜015）

前提セットアップ:
- 進行中の `exec_id`（UUID）が `work_executions` に存在すること
- `evidence_files`・`sign_records`・`andon_alerts` テーブルが空であること

### 3-4. グループ 4: SOP 承認・マスタ同期・Webhook（TST-intg-016〜020）

前提セットアップ:
- SUPERVISOR ロールの JWT が存在すること（`tests/fixtures/seed_sop.sql` で投入）
- `outbox_events` テーブルが空であること
- Webhook モックサーバーが `http://localhost:9999` で起動済みであること

```bash
# Webhook モックサーバー起動（wiremock または nc を使用）
# tests/scripts/start_webhook_mock.sh を実行する
bash tests/scripts/start_webhook_mock.sh &
```

## 4. PASS/FAIL 判定基準

各テストケースの合否は以下の基準で判定する。

| 判定項目 | PASS 条件 | FAIL 条件 |
|---|---|---|
| HTTP ステータスコード | 期待値と完全一致（例: 200, 201, 401, 413, 422）| 期待値と不一致 |
| レスポンスボディ | `error` フィールドの ERR コードが期待値と一致 | ERR コード不一致または未定義コード |
| DB 状態（後）| 挿入件数・フィールド値が期待値と一致 | 件数差異・フィールド値不一致 |
| トランザクション原子性 | 複数テーブルへの挿入が全件成功または全件ロールバック | 部分挿入（一部テーブルのみ更新）|
| ハッシュチェーン連鎖 | block[n].prev_hash = block[n-1].chain_hash | 連鎖断絶または NULL |

## 5. テスト終了処理

```bash
# テスト完了後にコンテナを停止し、tmpfs ボリュームを削除する
docker compose --profile test down -v

# ボリューム削除確認
docker volume ls | grep wnavdb_test
# 出力なし（削除済み）
```

## 6. 参照先

- テストケース定義: `../../../05_詳細設計/08_テストケース詳細設計/03_統合テストケース（API）.md`
- テスト環境定義: `../../../04_概要設計/10_テスト方式設計/02_テスト環境定義.md`（TENV-002）
- 品質ゲート: `../../../04_概要設計/10_テスト方式設計/04_品質ゲートとカバレッジ目標.md`（RGATE-002）

**本節で確定した方針**
- **TENV-002（postgres:16-alpine, port 5433）を L2 統合テストの専用環境として使用し、`--test-threads=1` で実行する。並列実行によるトランザクション競合を禁止する。**
- **各テストは `sqlx::test` のトランザクション自動ロールバックによりテスト間の独立性を保証する。シード SQL 再投入は不要とする。**
- **グループ実行順序（認証 → ステップ記録 → 証拠・署名 → SOP・Webhook）を遵守する。前グループに FAIL が存在する場合は後続グループの実行を停止する。**
- **PASS/FAIL 判定は HTTP ステータスコード・ERR コード・DB 状態の 3 点をすべて満たす場合に PASS とする。1 点でも不一致の場合は FAIL とする。**
