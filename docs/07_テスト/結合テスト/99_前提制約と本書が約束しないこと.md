# 99 前提制約と本書が約束しないこと

本章は `07_テスト/結合テスト` サブの範囲外事項・個人開発スコープ・本書が提供しない成果物を明示する。
本章の内容を読まずに本サブを参照した結果生じる誤解について、本書は責任を負わない。

---

## §1. テストコード実装は本書の範囲外である

本書（`07_テスト/結合テスト/` 配下の全ファイル）はテスト実施手順・実施計画・テンプレートを規定するが、以下は含まない。

| 対象外事項 | 備考 |
|---|---|
| Rust 統合テストコード（`tests/integration/` 配下）| 実装担当者がテストケース定義に基づいて実装する |
| `tests/fixtures/` のシード SQL ファイル | 実装担当者が作業実行前に準備する |
| `scripts/check-dtm-m6.sh` の実装 | CI 担当者が実装する |
| `scripts/check-rbac-endpoints.sh` の実装 | CI 担当者が実装する |
| `tests/scripts/start_webhook_mock.sh` の実装 | 実装担当者が準備する |

---

## §2. CI YAML の実装は本書の範囲外である

`05_結合テスト実施計画.md` §2 に記載した YAML 抜粋は設計意図の例示であり、実際の `.github/workflows/*.yml` の実装は本書の管理対象ではない。

CI ジョブの実装・保守は実装担当者の責任範囲とする。本書の YAML 抜粋に誤りがある場合は本書を修正するのではなく、実際の YAML ファイルを正とする。

---

## §3. 第三者監査・外部認証は本書の範囲外である

本書は個人開発プロジェクトの内部品質管理を目的とする。以下は本書の範囲外と判断する。

| 対象外事項 | 理由 |
|---|---|
| ISO 13485・FDA 21 CFR Part 11 等の外部認証申請 | 個人開発フェーズでは対象外と判断する |
| 第三者テスト機関によるレビュー・承認 | 個人開発フェーズでは対象外と判断する |
| 法的効力を持つ電子記録の完全性保証証明 | 本書はテスト実施手順であり法的証明書ではない |
| penetration test（外部専門機関によるもの）| 個人開発フェーズでは対象外と判断する |

---

## §4. パフォーマンステストは本書の範囲外である

L2 統合テストおよび L4 契約テストは機能正確性・スキーマ準拠性を検証する。以下の性能観点は含まない。

| 対象外事項 | 根拠 |
|---|---|
| レスポンス時間の計測（TST-perf-001〜005）| 性能テストは `07_テスト/テスト計画/` の性能テスト計画が担当する |
| スループット・同時接続数テスト | k6 による性能テストが担当する |
| 長時間実行テスト（Outbox Consumer の 24 時間安定性）| 運用テスト計画が担当する |

---

## §5. モバイルアプリ（React Native）の結合テストは本書の範囲外である

本書は Rust バックエンド API の統合テストを対象とする。以下はハンディ APP 側のテストであり本書の対象外と判断する。

| 対象外事項 | 担当 |
|---|---|
| SQLite + TypeORM のローカルストレージ統合テスト | `07_テスト/単体テスト/02_単体テスト仕様書（フロントエンド）実施手順.md` |
| React Native の E2E テスト（Detox）| E2E テスト計画が担当する |
| オフライン→オンライン同期の統合確認 | E2E テスト計画が担当する |

---

## §6. 本書が提供しない保証

本書は以下を保証しない。

| 非保証事項 | 理由 |
|---|---|
| 本書に記載の全コマンドが現在の実装に対して動作すること | 実装変更時は本書を同一 PR で更新する義務を負う（遅延した場合は CR-TEST-NNN を起票する）|
| Schemathesis・sqlx・axum_test のバージョン変更による動作変化への対応 | 依存ライブラリのバージョン管理は `Cargo.toml` および `requirements.txt` が正とする |
| 本書の手順に従った場合の v1.0.0 品質の達成 | 実際の品質はテスト結果と品質ゲート（RGATE-001〜006）が判定する |
| 将来の機能追加（FR の追加）に対する網羅性 | 本書は 2026-05-18 時点の TST-intg-001〜020 に基づいて作成されている。FR 追加時には本書を改訂する |

---

## 自己監査チェックリスト

本書の維持管理者は以下を定期的に確認する。

- [ ] TST-intg-001〜020 の権威ソース（`05_詳細設計/08_テストケース詳細設計/03_統合テストケース（API）.md`）が変更された場合に本書を更新する
- [ ] 新 FR が追加された場合に、BR-BUS カバレッジマトリクスへの影響を確認する
- [ ] TENV-002 の設定が変更された場合に `01_` および `03_` の手順を更新する
- [ ] openapi.yaml のバージョンが上がった場合に `02_API契約テスト仕様書実施手順.md` の参照パスを確認する

**本節で確定した方針**
- **テストコード実装・CI YAML 実装・第三者監査は本書の対象外と判断する。本書はプロセス規定のみを担う。**
- **本書の非保証事項を理解した上で参照することを読者に求める。コマンドの動作確認は常に実際の環境で行う。**
