# 04 結合テストシナリオ

本章は BR-BUS 45 件 × 統合シナリオおよび Outbox 連携シナリオを規定する。
各シナリオは複数の TST-intg-NNN テストケースを業務フロー順に組み合わせ、エンドツーエンドの整合性を確認する。

---

## シナリオ一覧

| シナリオ ID | シナリオ名 | 関連 TST-intg | BR-BUS カバー |
|---|---|---|---|
| SC-INTG-A | 作業開始〜完了の完全フロー | 001, 006, 009, 011, 014, 020 | BR-BUS-001〜010 |
| SC-INTG-B | Append-only ＋ハッシュチェーン整合性 | 006, 007, 009 | BR-BUS-015 |
| SC-INTG-C | Outbox パターン（冪等性含む）| 006, 008, 019 | BR-BUS-030〜039 |
| SC-INTG-D | RBAC 全ロール横断 | 001, 003, 016, 018 | BR-BUS-040〜045 |
| SC-INTG-E | BR-BUS 業務ルール違反（主要 10 件）| 007, 010, 012 | BR-BUS-001〜045 主要 |

---

## シナリオ A: 作業開始〜完了の完全フロー

**目的**: 認証→作業割付→ステップ記録→電子署名→作業完了の一連フローで整合性を確認する。

### ステップ順序

| ステップ | 実施内容 | 使用 TST-intg | PASS 条件 |
|---|---|---|---|
| A-1 | operator ロールでログインして JWT を取得する | TST-intg-001 | HTTP 200・JWT 発行・auth_logs 記録 |
| A-2 | IN_PROGRESS の作業実行（exec_id）を確認する | （シード SQL）| work_executions に 1 件存在 |
| A-3 | ステップ 1 を完了してイベントを記録する | TST-intg-006 | HTTP 201・work_events+1・hash_chain_blocks+1・outbox_events+1 |
| A-4 | ステップ 2 を完了してイベントを記録する | TST-intg-006 | HTTP 201・ハッシュチェーン block_seq=2 |
| A-5 | 証拠写真をアップロードする | TST-intg-011 | HTTP 201・evidence_files+1・file_hash 記録 |
| A-6 | 電子署名を記録する | TST-intg-014 | HTTP 201・work_events に sign_applied 記録 |
| A-7 | 作業完了 API を呼び出す | TST-intg-020 | HTTP 200・status=COMPLETED・RP-001 レコード非同期生成 |

**シナリオ A 合格基準**: A-1〜A-7 の全ステップが PASS であること。ステップ A-3 と A-4 の hash_chain_blocks において `block[2].prev_hash = block[1].chain_hash` が成立すること。

---

## シナリオ B: Append-only＋ハッシュチェーン整合性

**目的**: イベント記録後の改ざん試行がすべて拒否されること、および連続記録後のチェーン連鎖が正しいことを確認する。

### ステップ順序

| ステップ | 実施内容 | 使用 TST-intg | PASS 条件 |
|---|---|---|---|
| B-1 | ステップイベントを 3 件連続で記録する | TST-intg-006（×3）| 各 HTTP 201・work_events に 3 件 |
| B-2 | 3 件の hash_chain_blocks の prev_hash 連鎖を DB クエリで確認する | TST-intg-009 | 全行 chain_valid=true |
| B-3 | PATCH で work_events の改ざんを試みる | （手動）| HTTP 403 |
| B-4 | DELETE で work_events の削除を試みる | （手動）| HTTP 403 |
| B-5 | ロックステップ違反（ステップスキップ）を試みる | TST-intg-007 | HTTP 422, ERR-BIZ-001・DB 状態変化なし |

**シナリオ B 合格基準**: B-2 の全行 chain_valid=true・B-3 と B-4 が HTTP 403・B-5 が HTTP 422 であること。

---

## シナリオ C: Outbox パターン（冪等性含む）

**目的**: イベント記録→Outbox 生成→Webhook 配信→冪等性確認の Outbox パターンが正しく動作することを確認する。

### ステップ順序

| ステップ | 実施内容 | 使用 TST-intg | PASS 条件 |
|---|---|---|---|
| C-1 | ステップイベントを記録する | TST-intg-006 | HTTP 201・outbox_events に status=PENDING の 1 件 |
| C-2 | Outbox Consumer が PENDING レコードを取得して Webhook モックに配信することを確認する | TST-intg-019 | X-WNAV-Signature ヘッダが HMAC-SHA256 と一致・outbox_events.status=DELIVERED |
| C-3 | 同一 event_id で 2 回目のイベント記録を試みる | TST-intg-008 | 2 回目 HTTP 200・work_events 件数変化なし |
| C-4 | 冪等処理時に Outbox Consumer が 2 回目の Webhook を配信しないことを確認する | TST-intg-008 | Webhook モックの受信数が 1 件のまま |

**シナリオ C 合格基準**: C-1〜C-4 の全ステップが PASS であること。特に C-3 と C-4 で冪等性が保証されること。

---

## シナリオ D: RBAC 全ロール横断

**目的**: 全 5 ロール（operator / supervisor / master_admin / quality_admin / it_admin）のエンドポイントアクセス制御が仕様通りであることを確認する。

### アクセス制御確認マトリクス

| エンドポイント | operator | supervisor | master_admin | quality_admin | it_admin |
|---|---|---|---|---|---|
| `POST /api/v1/auth/login` | 準拠する | 準拠する | 準拠する | 準拠する | 準拠する |
| `POST /api/v1/step-events` | 201 | 403 | 403 | 403 | 403 |
| `POST /api/v1/sops` | 403 | 201 | 403 | 403 | 403 |
| `GET /api/v1/sync/master` | 200 | 200 | 200 | 200 | 200 |
| `GET /api/v1/reports/aggregate` | 対象外と判断する（worker_id キーなし検証）| 200 | 200 | 200 | 200 |
| `POST /api/v1/evidences` | 201 | 201 | 403 | 403 | 403 |

全アクセス制御が仕様通りである場合に PASS とする。

### ステップ順序

| ステップ | 実施内容 | 使用 TST-ID | PASS 条件 |
|---|---|---|---|
| D-1 | 各ロールの JWT を取得する | TST-intg-001 | 5 ロール × JWT 取得成功 |
| D-2 | 上記マトリクスの全組み合わせを確認する | TST-sec-001〜003 | 全行が期待 HTTP ステータスと一致 |
| D-3 | `/reports/aggregate` に worker_id キーが含まれないことを確認する | TST-sec-006 | レスポンスに worker_id フィールドが存在しない |

---

## シナリオ E: BR-BUS 業務ルール違反（主要 10 件）

**目的**: 主要 BR-BUS への違反テストが期待の ERR コードを返すことを確認する。

| ステップ | BR-BUS | 違反内容 | 使用 TST-intg | 期待 ERR コード |
|---|---|---|---|---|
| E-1 | BR-BUS-001 | ステップスキップ（step 1 → step 3）| TST-intg-007 | ERR-BIZ-001 |
| E-2 | BR-BUS-002 | 未割付作業へのステップ記録 | （手動）| ERR-BIZ-002 |
| E-3 | BR-BUS-003 | COMPLETED 状態の作業への追記 | （手動）| ERR-BIZ-003 |
| E-4 | BR-BUS-010 | 入力値範囲外（value > max）| TST-intg-010 | ERR-VAL-002 |
| E-5 | BR-BUS-015 | work_events の PATCH 試行 | （手動）| HTTP 403 |
| E-6 | BR-BUS-022 | 未許可 JSON Logic 演算子の使用 | TST-unit-BE-016〜020 | ERR-BIZ-022 |
| E-7 | BR-BUS-029 | `/reports/aggregate` で worker_id キー返却 | TST-sec-006 | worker_id なし |
| E-8 | BR-BUS-040 | operator ロールによる SOP 作成 | TST-sec-001 | HTTP 403 |
| E-9 | BR-BUS-043 | 未認証での API アクセス | TST-intg-005 | ERR-AUTH-002 |
| E-10 | BR-BUS-045 | 他ロールのリソースへの不正アクセス | TST-sec-003 | HTTP 403 |

全 10 件が期待 ERR コードを返す場合に PASS とする。

**本節で確定した方針**
- **シナリオ A〜E の 5 件は結合テストの合格条件とし、リリース前に全件 PASS することを必須とする。部分的な実施は対象外と判断する。**
- **シナリオ C の冪等性確認（C-3・C-4）は Outbox Consumer が稼働中の状態で実施する。Consumer 未起動での確認は対象外と判断する。**
- **シナリオ D の RBAC マトリクスは `scripts/check-rbac-endpoints.sh` で自動実行し、全行一致を合格条件とする。手動確認のみによる合格判定を禁止する。**
