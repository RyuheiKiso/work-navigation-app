# 04 受入相当テストと UAT シナリオ実施手順

本章は SLCP-JCF2013 §6.4.8「受入支援」に対応する UAT シナリオ A〜E の実施手順・ブラインドテスト要領・合否 4 条件を確定する。
個人開発であるため、受入担当は開発者自身が担当するが、ブラインドテスト原則（開発者はオブザーバー限定）を維持する。

権威ソース: `03_要件定義/テスト・受入要件/05_デモシナリオと受入シナリオ.md`

---

## 1. TENV-008 環境準備手順

```bash
# ステージング環境起動（本番同等）
docker compose -f docker-compose.staging.yml up -d

# 本番予定端末をステージング AP サーバーに接続
# Android: アプリ設定 → API_BASE_URL = https://staging.work-nav.local
# iOS: アプリ設定 → API_BASE_URL = https://staging.work-nav.local

# ステージング DB に UAT シードデータ投入
psql -U app -d work_nav_staging -f seeds/uat-seed.sql

# 動作確認
curl -s http://staging.work-nav.local/health | jq '.status'
# → "ok"
```

環境準備所要時間: 最大 1 営業日

**本節で確定した方針**
- **TENV-008（ステージング・本番同等）を UAT 専用環境として起動し、本番 DB・本番 AP とは完全分離する。**
- **UAT シードデータは本番データを使用しない。合成データ（uat-seed.sql）を使用する。**

## 2. UAT 参加者ロール割り当て

| ロール | 必要人数 | 役割 |
|---|---|---|
| quality_admin | ≥ 1 名 | 監査ログ・エクスポート・CAPA 承認 |
| master_admin | ≥ 1 名 | SOP 承認・マスタ改訂 |
| operator（代表）| ≥ 1 名 | 作業開始〜完了・写真添付・電子サイン |
| supervisor（代表）| ≥ 1 名 | アンドン応答・CAPA 起票・Kaizen 承認 |
| オブザーバー（開発者）| 1 名 | 記録のみ。参加者への操作説明禁止 |

ブラインドテスト原則: 開発者は UAT 中に操作手順・画面内容の説明を行ってはならない。
参加者に事前に共有する情報は「業務シナリオの概要（UC 名なし）」のみとする。

## 3. UAT シナリオ実施手順

### シナリオ A: SOP ナビゲーション（担当: operator）

対応 UC: UC-001〜003
想定ステップ数: 正常系 8 ステップ + 例外系 3 ステップ

実施内容:
1. QR コードをスキャンして作業を開始する（UC-001）
2. 標準ステップを順序通り実行する（UC-002）
3. 測定値を数値入力し、USL/LSL 判定を確認する（UC-003）
4. 境界値超過時のアラート表示を確認する（例外系）
5. 電子サインにより作業を完了する（UC-009 連携）

Pass 基準: 参加者が補助なしで 8 ステップを完了し、測定値判定が正しく動作すること

---

### シナリオ B: 証拠記録・電子サイン（担当: operator）

対応 UC: UC-004〜005
想定ステップ数: 正常系 7 ステップ + 例外系 2 ステップ

実施内容:
1. 写真証拠を添付する（UC-004）
2. 条件分岐ステップで条件を選択する（UC-005）
3. 電子サインを実施する（UC-009）
4. 写真なし状態でのサイン拒否を確認する（例外系）

Pass 基準: 写真添付・条件分岐・電子サインが補助なしで完了すること

---

### シナリオ C: マスタ改訂（担当: master_admin）

対応 UC: UC-006〜007
Draft / Published / Archived ライフサイクル確認

実施内容:
1. SOP 新バージョンを Draft 状態で作成する（UC-017）
2. SOP 承認ワークフローを通じて Published に遷移させる（UC-016）
3. 旧バージョンを Archived に変更する（UC-017）
4. Archived バージョンへのアクセス拒否を確認する（例外系）

Pass 基準: Draft → Published → Archived の 3 状態遷移が補助なしで完了すること

---

### シナリオ D: アンドン・CAPA（担当: operator + supervisor）

対応 UC: UC-008〜009（アンドン発報→応答→CAPA クローズ）

実施内容:
1. operator がアンドンを発報する（UC-011）
2. supervisor がアンドンに応答する（UC-012）
3. 不適合を登録する（UC-013）
4. CAPA を登録・承認する（UC-014）
5. CAPA をクローズし、監査ログに記録されることを確認する

Pass 基準: 発報 → 応答 → 不適合登録 → CAPA クローズが補助なしで完了すること

---

### シナリオ E: Outbox 同期（担当: operator）

対応 UC: UC-010（再接続 60 秒以内・50 件欠損ゼロ）

実施内容:
1. WiFi を切断した状態で 50 件の作業イベントをオフラインで記録する
2. SQLite Outbox に 50 件が蓄積されていることを確認する
3. WiFi を再接続する
4. 60 秒以内に全 50 件が同期されることを確認する
5. サーバー側 work_events テーブルで 50 件の存在を確認する

Pass 基準: 再接続後 60 秒以内に 50 件全件同期・欠損ゼロが確認できること

---

## 4. UAT 合否基準 4 条件

以下 4 条件をすべて満たした場合にのみ UAT 合格とする。いずれか 1 条件でも満たさない場合は不合格とし、欠陥対応後に再実施する。

| 条件 | 内容 |
|---|---|
| (1) P1 欠陥ゼロ | UAT 期間中に P1 欠陥（機能不全・データ損失リスク）が発生していないこと |
| (2) 22 UC 全件補助なし完了 | UC-001〜022 の全件が参加者の補助なし操作で完了すること |
| (3) 倫理品質チェック全 Pass | NFR-QUA-003 倫理品質自己監査チェックリストの全項目 Pass |
| (4) 電子サインオフ完了 | FR-AU-001 準拠の電子サインオフ（UUID v7・HMAC-SHA256・signer_id・timestamp_server）が完了すること |

---

## 5. UAT 実施期間

| 日程 | 内容 |
|---|---|
| Day 1 | TENV-008 環境準備・参加者接続確認・シードデータ投入 |
| Day 2〜4 | シナリオ A〜E 実施（最大 3 営業日）|
| Day 5 | 欠陥対応・再テスト・サインオフ（最大 1 営業日）|

最大実施期間: 5 営業日

**本節で確定した方針**
- **UAT は SLCP-JCF2013 §6.4.8「受入支援」として自己実施し、個人開発スコープでのブラインドテスト原則を維持する。**
- **合否 4 条件（P1 欠陥ゼロ・22 UC 全件・倫理チェック・電子サインオフ）を UAT の必要十分条件として確定する。**
