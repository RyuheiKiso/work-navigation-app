## 2. 性能テスト実施手順（TENV-006 / TST-perf-001〜005）

### 2-1. TENV-006 環境起動

```bash
# 本番相当環境起動（1 万件シード）
docker compose -f docker-compose.perf.yml up -d

# シードデータ投入確認
psql -U app -d work_nav_prod -c "SELECT COUNT(*) FROM work_events;"
# → 10000 件以上を確認
```

### 2-2. TST-perf-001: 50 VU × 5 分負荷テスト（NFR-PERF-001）

```bash
k6 run --vus 50 --duration 300s scripts/k6/load-test.js \
  --out json=results/perf-001.json
```

判定: `p(95)` < 2000ms かつ `http_req_failed` = 0%

### 2-3. TST-perf-002: 100 rps × 60 秒定常到着率テスト（NFR-PERF-002）

```bash
k6 run scripts/k6/arrival-rate.js \
  --out json=results/perf-002.json
```

判定: `dropped_iterations` = 0 かつ エラーレート 0%

### 2-4. TST-perf-003: Criterion ハッシュチェーン 100 万件検証（NFR-PERF-003）

```bash
cd backend && cargo bench --bench hash_chain -- --output-format bencher \
  | tee results/perf-003-criterion.txt
```

判定: `hash_chain 1M verify` < 60 秒

### 2-5. TST-perf-004: Outbox 再接続同期テスト（NFR-PERF-004）

```bash
# DISCONNECTED 状態で 100 件 Outbox 蓄積
# 再接続後の全件配信時間計測
docker compose -f docker-compose.perf.yml stop api-server
# Detox で 100 件オフライン操作実施
docker compose -f docker-compose.perf.yml start api-server
# → 復旧後 ≤ 5 秒で全件 Outbox 配信を確認
```

### 2-6. TST-perf-005: 1000 ステップ RP-001 生成（NFR-PERF-005）

```bash
curl -w "@curl-format.txt" -X POST http://localhost:8080/api/v1/reports/rp-001 \
  -H "Authorization: Bearer $QUALITY_ADMIN_JWT" \
  -d '{"step_count": 1000}' \
  -o results/perf-005.pdf
```

判定: 1000 ステップ RP-001 生成 < 10 秒

## 3. セキュリティテスト実施手順（TENV-007 / TST-sec-001〜010）

### 3-1. OWASP ZAP スキャン（TST-sec-005）

```bash
docker run -t ghcr.io/zaproxy/zaproxy:stable \
  zap-baseline.py \
  -t http://api-server:8080 \
  -r zap-report.html \
  -I
```

判定: HIGH アラート 0 件・MEDIUM ≤ 2 件

### 3-2. RBAC 検証（TST-sec-001〜003）

```bash
# OPERATOR ロールでマスタ編集エンドポイントに POST
curl -X POST http://localhost:8080/api/v1/masters/sop \
  -H "Authorization: Bearer $OPERATOR_JWT" \
  -H "Content-Type: application/json" \
  -d '{"title": "テスト"}' \
  -w "\n%{http_code}"
# → HTTP 403 を確認
```

### 3-3. JWT 期限切れテスト（TST-sec-004）

```bash
# 期限切れ JWT（exp 過去日時）でリクエスト
curl -X GET http://localhost:8080/api/v1/work-sessions \
  -H "Authorization: Bearer $EXPIRED_JWT" \
  -w "\n%{http_code}"
# → HTTP 401 を確認
```

### 3-4. ブルートフォーステスト（TST-sec-007〜008）

```bash
# 3 回連続ログイン失敗
for i in 1 2 3; do
  curl -X POST http://localhost:8080/api/v1/auth/login \
    -d '{"username":"test","password":"wrong"}' -w "%{http_code}\n"
done
# 4 回目
curl -X POST http://localhost:8080/api/v1/auth/login \
  -d '{"username":"test","password":"wrong"}' -w "%{http_code}\n"
# → HTTP 423 を確認
```

### 3-5. HMAC 改ざん・欠落テスト（TST-sec-009〜010）

```bash
# HMAC ヘッダ改ざん
curl -X POST http://localhost:8080/api/v1/work-events \
  -H "X-HMAC-Signature: invalid_signature" \
  -H "Authorization: Bearer $OPERATOR_JWT" \
  -w "\n%{http_code}"
# → HTTP 401 を確認

# HMAC ヘッダ欠落
curl -X POST http://localhost:8080/api/v1/work-events \
  -H "Authorization: Bearer $OPERATOR_JWT" \
  -w "\n%{http_code}"
# → HTTP 401 を確認
```

## 4. 可用性テスト実施手順

### 4-1. カオステスト: api-server 停止 → Emergency Mode 確認

```bash
# api-server を停止
docker compose stop api-server

# ハンディ APP が 30 秒以内に DISCONNECTED/Emergency Mode に遷移することを Detox で確認
pnpm detox test --configuration android.emu.debug --testNamePattern="Emergency Mode"
```

判定: api-server 停止後 30 秒以内に端末が Emergency Mode 表示に遷移すること

### 4-2. WiFi 物理切断テスト（TENV-008 環境）

本番予定端末で WiFi を物理切断し、以下を確認する。

- [ ] ハンディ APP がオフライン継続動作すること（TST-e2e-010 連携）
- [ ] SQLite Outbox にレコードが蓄積されること
- [ ] 再接続後 60 秒以内に全件同期されること

**本節で確定した方針**
- **k6 P95 < 2000ms・5xx エラーレート 0%・OWASP ZAP HIGH 0 件を、システム適格性確認テストの Pass 基準（RGATE-003/005）として確定する。**
- **可用性テストはカオステスト（Docker 停止）と WiFi 物理切断テストの 2 種を実施し、結果を証跡として記録する。**
