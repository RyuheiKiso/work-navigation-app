# 06 適格性確認テスト実施計画

本章は適格性確認テストの実施タイミング・前提条件・各テストレベルの期間見積もり・CI/CD ジョブ構成を確定する。

---

## 1. 実施タイミング

| テストレベル | 実施タイミング | 前提条件 |
|---|---|---|
| ソフトウェア適格性確認（E2E 22 件）| M5 完了後（詳細設計・実装完了後）| 全単体テスト PASS・全結合テスト PASS |
| システム適格性確認（性能・セキュリティ・可用性）| M5 完了後（E2E と並行可）| TENV-006/007 環境構築完了 |
| ALCOA+ 検証（9 属性）| M5 完了後（システム適格性確認と同時期）| TENV-006 環境構築完了 |
| UAT シナリオ A〜E | M6 前（UAT 参加者スケジュール調整完了後）| E2E 22 件 Pass・システム適格性確認 Pass |
| リリース判定・サインオフ | UAT 合格確認後即日 | RGATE-001〜006 全 PASS・UAT 合格 |

**本節で確定した方針**
- **E2E・性能・ALCOA+ の 3 テストは M5 完了後に並行実施し、全 Pass 確認後に UAT を開始する。**
- **UAT 参加者（≥ 4 名）のスケジュール調整は M5 完了の 2 週間前に開始する。**

---

## 2. 実施前提条件

適格性確認テスト開始前に以下がすべて満たされていなければならない。

| 前提条件 | 確認方法 |
|---|---|
| 全単体テスト PASS（TST-unit-BE-001〜030 / TST-unit-FE-001〜020）| `cargo test --all` / `pnpm test` exit code 0 |
| 全結合テスト PASS（TST-intg-001〜020）| `pnpm test:integration` exit code 0 |
| TENV-003/004/006/007/008 環境構築完了 | 各 TENV ヘルスチェック HTTP 200 |
| RGATE-001（コード品質）達成 | Clippy/ESLint/TypeScript エラー 0 件 |
| シードデータ投入完了 | TENV-006: 1 万件・TENV-008: UAT データ |

---

## 3. 期間見積もり

| テストレベル | 見積もり期間 | 根拠 |
|---|---|---|
| E2E テスト 22 件（TENV-003/004）| 約 2 営業日 | Playwright: 1 日・Detox: 1 日（最優先 6 件含む）|
| 性能テスト（TST-perf-001〜005）| 約 1 営業日 | k6 300 秒 × 2・Criterion・Outbox 確認 |
| セキュリティテスト（TST-sec-001〜010）| 約 1 営業日 | ZAP スキャン（約 1 時間）+ RBAC/JWT/HMAC 手動確認 |
| ALCOA+ 検証（TST-alcoa-001〜009）| 約 1 営業日 | SQL + curl + veraPDF + cargo test |
| UAT シナリオ A〜E | 最大 5 営業日 | Day1: 環境準備・Day2〜4: シナリオ実施・Day5: 欠陥対応・サインオフ |
| リリース判定・サインオフ | 0.5 営業日 | チェックリスト確認・電子サインオフ |

合計最大期間: 約 10.5 営業日（E2E〜リリース判定）

---

## 4. CI/CD ジョブ構成

適格性確認テストは以下のジョブを GitHub Actions / ローカル CI で実行する。

| ジョブ名 | トリガ | 内容 |
|---|---|---|
| `react-e2e-web` | M5 ブランチ merge 後手動 | Playwright E2E（TENV-003）TST-e2e-012〜020 |
| `react-e2e-rn` | M5 ブランチ merge 後手動 | Detox E2E（TENV-004）TST-e2e-001〜011, 022 |
| `contract` | M5 ブランチ merge 後手動 | Schemathesis スキーマ準拠率（RGATE-004）|
| `coverage-gate` | M5 ブランチ merge 後自動 | Rust/React カバレッジゲート（RGATE-002）|
| `perf-test` | 手動（TENV-006 必要）| k6 負荷テスト（TST-perf-001〜002）|
| `security-scan` | 手動（TENV-007 必要）| OWASP ZAP スキャン（TST-sec-005）|
| `alcoa-verify` | 手動（TENV-006 必要）| ALCOA+ SQL + curl + pdffonts（TST-alcoa-001〜009）|

### ジョブ依存関係

```
単体テスト（cargo test / pnpm test）
    ↓
結合テスト（pnpm test:integration）
    ↓
coverage-gate + contract（並行）
    ↓
react-e2e-web + react-e2e-rn（並行）
    ↓
perf-test + security-scan + alcoa-verify（並行）
    ↓
UAT（TENV-008・手動）
    ↓
リリース判定・サインオフ
```

---

## 5. テスト中断・再開条件

| 事象 | 対応 |
|---|---|
| P1 欠陥（機能不全）発生 | 即時中断・DEFECT-NNN 起票・修正後全テスト再実施 |
| P2 欠陥発生 | 当該テスト継続・リリース前修正を条件に記録 |
| TENV 環境障害 | 障害内容を記録し環境復旧後に再開 |
| UAT 参加者欠席 | 該当ロールの別参加者を手配。4 名未満での UAT 実施は不可 |

**本節で確定した方針**
- **E2E・性能・ALCOA+ の並行実施で合計約 5 営業日・UAT で最大 5 営業日・リリース判定 0.5 日の合計最大 10.5 営業日を適格性確認テストの標準期間として確定する。**
- **CI/CD の `coverage-gate` ジョブは M5 ブランチ merge 時に自動実行し、RGATE-002 未達は即時ブロックとする。**
