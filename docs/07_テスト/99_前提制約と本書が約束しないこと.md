# 99 前提制約と本書が約束しないこと

## §1 適用範囲外

本書（`07_テスト`）が担当する範囲は「テストプロセスの計画・実施手順・記録テンプレート・リリース判定」に限定する。以下の事項は本書のスコープ外であり、対応する文書または実装フェーズで扱う。

| 対象外事項 | 根拠・委任先 |
|---|---|
| テストコード（`*.rs` の `#[test]`、`*.spec.ts`、Playwright スクリプト、Detox スクリプト等）の実装 | `src/` 実装フェーズで行う（D-TST-004 継承）|
| CI/CD YAML（`.github/workflows/ci.yml`）の詳細実装 | `06_実装/11_CICDパイプライン設定.md` の権威範囲 |
| k6 スクリプト（VU ランプアップ曲線・混合シナリオ重み付け）の実装 | `src/` 実装フェーズで行う（D-TST-002 継承）|
| Detox / Playwright 設定ファイルの詳細 | `src/` 実装フェーズで行う（D-TST-004 継承）|
| 本番環境へのデプロイ実行 | `08_移行` の担当 |
| 本番稼働後の障害対応・運用 | `09_運用・保守` の担当 |
| 100% コードカバレッジの保証 | 目標値（Unit ≥80% / ドメイン ≥90%）を設定するが、100% は保証しない |
| 実際のテスト実行結果・実測カバレッジ値 | ver1.0.0 実装完了後にテンプレートへ記入する |

**本節で確定した方針**
- **テストコード実装・CI YAML 詳細・k6 スクリプト・Detox/Playwright 設定は本書スコープ外であり、実装フェーズおよび src/ が担当することを確定した。**
- **実施結果・欠陥報告・カバレッジ実測値は ver1.0.0 テスト実施時に埋めるテンプレートのみ本書で提供することを確定した。**

## §2 個人開発スコープ

本プロジェクトは個人開発（Bus Factor = 1）である。IPA 共通フレームの一部のタスクは複数人チームを前提としており、個人開発では以下の取捨選択を行う。

| IPA タスク（複数人前提）| 個人開発での扱い |
|---|---|
| テスト独立性（開発者と異なるテスターによる実施）| 対応する（UAT はブラインドテスト原則を維持するが、エンジニアリングテストは開発者自身が実施）|
| 承認フロー（上位者承認）| 対応する（7 日クーリング期間による自己承認で代替）|
| 外部立会者（必須）| 対象外と判断する（任意の立会を推奨するが必須とはしない）|
| ピア・テストレビュー | 対象外と判断する（個人開発のため、チェックリスト自己確認で代替）|

**本節で確定した方針**
- **個人開発スコープでは「UAT ブラインドテスト原則の維持」と「7 日クーリング自己承認」を採用し、外部立会・ピアレビューを任意扱いとすることを確定した。**

## §3 AI/ML 方針（本章のみ言及可）

テスト工程における AI/ML の活用について、以下の方針を確定する。

| 活用領域 | 方針 |
|---|---|
| テストケース生成支援 | 対応する（`05_詳細設計/08_` の既存 TST-NNN を基準とし、AI 生成は追加テストの参考にとどめる）|
| テスト結果の自動判定（CI）| 準拠する（cargo test / Jest / Playwright / Detox の自動判定を採用）|
| テストコードの AI 補完 | 対応する（AI 補完コードは必ず人間がレビュー・確認してからコミットする）|
| 品質ゲート基準の AI 自動調整 | 対象外と判断する（RGATE-001〜006 の数値は人間が変更管理する）|
| ALCOA+ 検証の AI 代替 | 対象外と判断する（ALCOA+ 検証は規格準拠の義務であり、自動化ツールの PASS が根拠でなければならない）|

**本節で確定した方針**
- **テスト工程での AI 補完は「参考・支援」に限定し、テストケース内容・品質ゲート基準・ALCOA+ 判定は人間が最終確認することを確定した。**

## §4 規制との関係

本プロジェクトのテスト工程が対象とする規制・標準との関係を「準拠する／対応する／対象外と判断する」の 3 段で確定する。

| 規制・標準 | 関係 | 根拠 |
|---|---|---|
| IPA 共通フレーム 2013（SLCP-JCF2013）| **準拠する** | 本書は 6 テストプロセスのタスクを全て担当章に対応付けている |
| 21 CFR Part 11（米国 FDA 電子記録規制）| **準拠する** | ALCOA+ 検証テスト（TST-alcoa-001〜009）が電子記録品質を検証する |
| PIC/S PE 009-16（GDP ガイドライン）| **対応する** | 出荷判定・品質記録の検証に準じた ALCOA+ 検証テストを実施する |
| ISO 19005-3（PDF/A-3 規格）| **準拠する** | TST-alcoa-002/008 で veraPDF による PDF/A-3 適合検証を実施する |
| IEEE XES 2.0（プロセスマイニング交換形式）| **準拠する** | TST-alcoa-006 で XES 必須属性の完全性検証を実施する |
| 第三者ペネトレーションテスト | **対象外と判断する** | 個人開発・単一工場クローズド OT 環境・NFR-SEC-031（インターネット非接続）を根拠とする（ADR-TEST-001 参照）|
| ISMS 認証（ISO/IEC 27001）| **対象外と判断する** | 個人開発スコープ超過・単一工場クローズド環境 |
| ソフトウェア保証 (SIL/ASIL)| **対象外と判断する** | 製造指示支援ツール（安全関連システムではない）|

**本節で確定した方針**
- **IPA 共通フレーム・21 CFR Part 11・ISO 19005-3・IEEE XES 2.0 への準拠を確定した。**
- **第三者ペネトレーションテスト・ISMS 認証・SIL/ASIL は対象外と判断する。工場外デプロイ・クラウド接続・規制変更が発生した場合は ADR-TEST-001 を再評価する。**

## §5 本書が約束しないこと

- **SLA / SLO の確約**: P95 ≤ 2000ms 等の性能目標は品質ゲートとして設定するが、本番環境での SLA として保証するのは `09_運用・保守` の責務である。
- **第三者セキュリティ監査の合格証明**: OWASP ZAP・cargo audit による自己検査は実施するが、外部機関のセキュリティ合格証明書を発行しない（ADR-TEST-001）。
- **テストカバレッジ 100% の達成**: Unit ≥80%・ドメイン ≥90% を目標とするが、100% カバレッジは対象外と判断する。到達不可能なカバレッジ目標は品質の虚偽表示となる。
- **実テストの合格保証**: 本書はテスト計画・実施手順・記録テンプレートを確定するが、テスト実行結果（合格・不合格）を事前に確約しない。
- **ver1.0.0 リリース期日の確約**: テストフェーズの所要時間は欠陥件数・重大度に依存するため、期日を本書で確約しない。

**本節で確定した方針**
- **本書は SLA 確約・外部監査合格証明・カバレッジ 100%・合格保証・期日確約を約束しないことを確定した。**
