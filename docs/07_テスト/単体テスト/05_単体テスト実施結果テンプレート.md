# 05 単体テスト実施結果テンプレート

本章は SLCP-JCF2013 §6.3.6.2「ソフトウェアユニットの検証」に基づく実施結果の記録形式を確定する。本テンプレートは空テンプレートであり、テスト実施ごとに複製して使用する。TST-EXEC 識別子の採番規則は `00_本書の位置づけと識別子規約.md` §2-2 に従う。

---

## 1. TST-EXEC 識別子の採番規則

| 規則 | 内容 |
|---|---|
| フォーマット | `TST-EXEC-YYYYMMDD-NNN`（例: TST-EXEC-20260518-001）|
| 日付部分 | テスト実施日（UTC+9）|
| 連番部分 | 当日の実施順に 001 から 3 桁連番で採番する |
| 欠番 | 欠番禁止（取り消した記録は状態欄に CANCELLED と記入する）|
| 採番主体 | テスト実施者（RyuheiKiso または CI 実行の場合は「CI-自動」と記入する）|

CI 自動実行の場合は GitHub Actions の run_id を TST-EXEC 番号にマッピングし、CI ログ URL を「根拠 URL」欄に記入する。

**本節で確定した方針**
- **TST-EXEC-YYYYMMDD-NNN 形式を実施結果の唯一の識別子とし、欠番を禁止することを確定した。**
- **CI 自動実行の場合は「CI-自動」と記入し、GitHub Actions の run_id と対応させることを確定した。**

## 3. 記入ガイドライン

### 3-1. フィールド説明

| フィールド | 記入内容 |
|---|---|
| テストレベル | 単体テストの場合は常に「L1（単体テスト）」と記入する |
| TST-ID 範囲 | 実施したテストケースの範囲を記入する（全件実施の場合はデフォルト値のまま）|
| 実施日時 | テスト実施開始時刻を YYYY-MM-DD HH:MM 形式で記入する（CI の場合は実行開始時刻）|
| 実施者 | 手動実施の場合は「RyuheiKiso」、CI の場合は「CI-自動」と記入する |
| 対象マイルストーン | M3〜M6 から該当するものを 1 つ選択して記入する（中間確認は「M3 中間」等と記入可）|
| 件数（Pass/Fail）| `cargo test` および `jest` のサマリから転記する |
| 行カバレッジ | lcov.info サマリおよび Jest --coverage の出力から転記する |
| 不合格内容 | 全件 PASS の場合は「なし」と記入する。FAIL がある場合は DEFECT-NNN を列挙する |
| 合否判定 | 全件 PASS かつカバレッジ全目標達成の場合のみ「PASS」と記入する |
| 根拠 URL | CI ビルドの GitHub Actions URL を記入する（ローカル実行の場合は「ローカル実行」と記入する）|

### 3-2. FAIL 記録時の追加手順

1. `不合格内容` 欄に FAIL した TST-ID と DEFECT-NNN を記入する。
2. `合否判定` 欄に「FAIL」と記入する。
3. `06_欠陥管理と修正記録テンプレート.md` の DEFECT テンプレートを起票する。
4. 修正完了後、同一 TST-EXEC 記録に「修正後再実行: TST-EXEC-YYYYMMDD-NNN」を追記する。

**本節で確定した方針**
- **全件 PASS かつ全カバレッジ目標達成の場合のみ「PASS」と記入することを確定した。**
- **FAIL が 1 件でも存在する場合は DEFECT-NNN を起票し、TST-EXEC 記録に DEFECT-NNN を紐付けることを確定した。**
