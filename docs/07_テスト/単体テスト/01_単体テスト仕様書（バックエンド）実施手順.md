# 01 単体テスト仕様書（バックエンド）実施手順

本章は `05_詳細設計/08_テストケース詳細設計/01_単体テストケース（バックエンド）.md` で確定済みの TST-unit-BE-001〜030 を引き継ぎ、SLCP-JCF2013 §6.3.6.2 に準拠した実施手順を確定する。テストケース自体（入力・期待出力・対応 FR/NFR）は上流文書が権威ソースであり、本章は変更しない。

---

## 1. 前提条件

本実施手順を実行するには、以下の環境が全て満たされていることを確認する。

| 前提項目 | 必要値 | 確認コマンド |
|---|---|---|
| Rust toolchain | stable 最新版（Edition 2024 対応）| `rustup show` |
| cargo-llvm-cov | インストール済み | `cargo llvm-cov --version` |
| rstest | `Cargo.toml` に依存定義済み | `cargo tree | grep rstest` |
| PostgreSQL 起動 | BE-001〜020 は不要・BE-026〜030 は TENV-002 起動必須 | `pg_isready -h localhost -p 5432` |
| 環境変数 DATABASE_URL | BE-026〜030 実行時のみ必須（`.env` または CI シークレット）| `printenv DATABASE_URL` |

DB 不要グループ（BE-001〜025）はローカル単独で実行可能である。DB 依存グループ（BE-026〜030）は `docker compose up -d db` で PostgreSQL コンテナ（TENV-002）を起動してから実施する。

**本節で確定した方針**
- **TST-unit-BE-001〜025 は DB 起動不要でローカル実行可能とすることを確定した。**
- **TST-unit-BE-026〜030 は PostgreSQL コンテナ（TENV-002）の起動を必須前提とすることを確定した。**

---

## 2. 実行コマンド一覧

### 2-1. 全件実行（推奨）

```sh
# すべての単体テストを実行（ライブラリクレート限定でバイナリ除外）
cargo test --all --lib

# カバレッジ付き全件実行
cargo llvm-cov --all --lib --lcov --output-path lcov.info
```

### 2-2. グループ単位の実行

```sh
# BE-001〜005: WorkExecution ステートマシン
cargo test --lib work_execution

# BE-006〜010: HashChainService
cargo test --lib hash_chain

# BE-011〜015: RBAC ミドルウェア
cargo test --lib rbac

# BE-016〜020: JSON Logic DSL
cargo test --lib json_logic

# BE-021〜025: OutboxConsumer バックオフ
cargo test --lib outbox

# BE-026〜030: Append-only 強制（DB 起動必須）
cargo test --lib append_only
```

### 2-3. 特定ケースの単独実行

```sh
# TST-unit-BE-007 を単独実行
cargo test test_tst_unit_be_007 -- --exact

# TST-unit-BE-021〜025 をまとめて実行
cargo test test_tst_unit_be_02
```

**本節で確定した方針**
- **全件実行には `cargo test --all --lib` を使用し、バイナリクレート（main.rs エントリポイント）をテスト対象外とすることを確定した。**
- **特定ケースの単独実行は `-- --exact` オプションを使用してテスト関数名で指定することを確定した。**

## 3. テストグループ別実施順序と前提セットアップ

テストグループは以下の順序で実施する。依存関係のあるグループは前段グループが PASS であることを確認してから実施する。

| 順序 | グループ | TST-ID 範囲 | DB 要否 | 実施前セットアップ |
|---|---|---|---|---|
| 1 | WorkExecution ステートマシン | BE-001〜005 | 不要 | なし |
| 2 | HashChainService | BE-006〜010 | 不要 | なし |
| 3 | RBAC ミドルウェア | BE-011〜015 | 不要 | なし |
| 4 | JSON Logic DSL | BE-016〜020 | 不要 | なし |
| 5 | OutboxConsumer バックオフ | BE-021〜025 | 不要 | なし |
| 6 | Append-only 強制 | BE-026〜030 | 必須（TENV-002）| `docker compose up -d db` |

グループ 1〜5 は並列実行可能である（`cargo test` のデフォルト並列実行を活用する）。グループ 6 は必ず DB 起動後に実施する。

**本節で確定した方針**
- **グループ 1〜5（BE-001〜025）は並列実行を許可し、グループ 6（BE-026〜030）は DB 起動確認後に逐次実施することを確定した。**
- **前段グループが FAIL の場合は後続グループを実施せず即時原因調査に移行することを確定した。**

---

## 4. PASS/FAIL 判定基準

### 4-1. 個別テストケースの判定

| 判定 | 条件 |
|---|---|
| PASS | cargo test が当該テスト関数に対して `test ... ok` を出力し、期待出力と実際の出力が一致する |
| FAIL | cargo test が当該テスト関数に対して `FAILED` を出力する、またはパニックが発生する |

### 4-2. グループ全体の判定

| 判定 | 条件 |
|---|---|
| PASS | グループ内の全テストケースが PASS |
| FAIL | グループ内に 1 件以上の FAIL が存在する |

### 4-3. 全件判定

| 判定 | 条件 |
|---|---|
| PASS | TST-unit-BE-001〜030 の全 30 件が PASS かつ行カバレッジ（Rust 全体 ≥ 80%、ドメイン層 ≥ 90%）を達成する |
| FAIL | 1 件以上の FAIL、またはカバレッジ目標未達 |

FAIL が発生した場合は直ちに DEFECT-NNN を起票し、マージをブロックする。

**本節で確定した方針**
- **1 件でも FAIL が存在する場合はグループ全体・全件ともに FAIL と判定し、マージをブロックすることを確定した。**
- **カバレッジ目標未達もテスト FAIL と同等に扱い、同様の対応手順を適用することを確定した。**
