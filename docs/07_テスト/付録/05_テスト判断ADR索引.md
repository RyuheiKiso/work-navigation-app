# 05 テスト判断 ADR 索引

本章の責務は、テストプロセス（`07_テスト`）内で下した重要な技術的判断・設計判断を ADR-TEST-NNN として記録し、将来の参照・変更時の根拠を保全することである。

---

## 1. ADR-TEST-NNN の採番規約

| 規則 | 内容 |
|---|---|
| 形式 | `ADR-TEST-NNN`（3 桁ゼロ埋め）|
| 開始番号 | ADR-TEST-001 |
| クーリング期間 | 7 日間（個人開発スコープでの自己再考）|
| 状態 | PROPOSED → ACCEPTED → SUPERSEDED |
| 廃止管理 | SUPERSEDED の場合は後継 ADR-TEST-NNN を明記 |

---

## 2. ADR テンプレート

```markdown
## ADR-TEST-NNN タイトル

| フィールド | 値 |
|---|---|
| ADR-ID | ADR-TEST-NNN |
| 起票日 | YYYY-MM-DD |
| 決定日 | YYYY-MM-DD（クーリング後）|
| 状態 | PROPOSED / ACCEPTED / SUPERSEDED |
| 後継 ADR | ADR-TEST-NNN（SUPERSEDED の場合）|

### 背景（Context）

なぜこの判断が必要になったか。解決したい問題・制約・トレードオフを記述する。

### 判断（Decision）

何を選択したか。採択した選択肢と根拠を記述する。

### 却下した選択肢

どのような代替案を検討したか、なぜ採択しなかったかを記述する。

### 結果（Consequences）

この判断がもたらす良い影響・悪い影響・リスクを記述する。
```

## 3. ADR 索引

### ADR-TEST-001: 第三者ペネトレーションテストの対象外判断

| フィールド | 値 |
|---|---|
| ADR-ID | ADR-TEST-001 |
| 起票日 | 2026-05-18 |
| 決定日 | 2026-05-25（クーリング後予定）|
| 状態 | ACCEPTED |

**背景**: 個人開発・単一工場クローズド OT 環境（インターネット非接続）での運用を前提とした場合、第三者機関によるペネトレーションテストのコスト対効果を判断する必要がある。

**判断**: 第三者ペネトレーションテスト（D-TST-003）を対象外と判断する。OWASP ZAP / cargo audit / npm audit / RBAC 検証 / JWT 偽造テストによる自己検査で代替する。

**却下した選択肢**: バグバウンティプログラムへの参加（個人開発スコープ超過）、社内セキュリティ部門レビュー（小規模工場のため未設置）。

**結果**: 外部攻撃面が最小化された環境（NFR-SEC-031：インターネット非接続）での self-hosting であるため、リスクは許容範囲内と判断する。工場外デプロイ・クラウド接続・法規制変更が発生した場合は本 ADR を再評価する。

### ADR-TEST-002: 実施結果のテンプレートのみ（実測値なし）方針

| フィールド | 値 |
|---|---|
| ADR-ID | ADR-TEST-002 |
| 起票日 | 2026-05-18 |
| 決定日 | 2026-05-25（クーリング後予定）|
| 状態 | ACCEPTED |

**背景**: `07_テスト` の文書整備時点では `src/` の実装が未完了であるため、実際のテスト実行・結果の記録が不可能である。

**判断**: 実施結果・欠陥報告・カバレッジレポートはテンプレートのみ提供し、実測値は ver1.0.0 実装着手後のテスト実施時に埋める方針とする。

**却下した選択肢**: モックデータで仮の実施結果を記入する（整合性維持が困難・実施前から「合格」を記録することの問題）。

**結果**: 文書整備と実装が独立して進行できる。テスト実施時には本書テンプレートを使用するだけでよく、追加設計工数がかからない。

### ADR-TEST-003: ALCOA+ 検証テストを適格性確認テストとして分類する判断

| フィールド | 値 |
|---|---|
| ADR-ID | ADR-TEST-003 |
| 起票日 | 2026-05-18 |
| 決定日 | 2026-05-25（クーリング後予定）|
| 状態 | ACCEPTED |

**背景**: ALCOA+ 検証テスト（TST-alcoa-001〜009）は L1/L2/L3 にまたがる複合的なテストであり、単体・結合のいずれかに分類するのが難しい。

**判断**: ALCOA+ 検証テストは「システム全体の非機能品質特性（電子記録品質）の検証」であるため、IPA 6.1.2.5（システム適格性確認テスト）の一環として `適格性確認テスト/03_` に分類する。L1/L2 相当の実施コマンドが含まれるが、その目的はシステム品質特性の検証であり適格性確認が適切である。

**却下した選択肢**: ALCOA+ を結合テストに分類する（BR-BUS との関連性は高いが、規格準拠の証明という性質から適格性確認が適切）。

**結果**: ALCOA+ 検証の根拠規格（21 CFR Part 11・PIC/S PE 009-16・ISO 19005-3・IEEE XES 2.0）との対応が `03_ALCOA+検証適格性確認テスト実施手順.md` に集約され、規制当局対応時に参照しやすくなる。
