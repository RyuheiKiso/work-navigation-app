# 00 テスト識別子規約と採番規約

本章の責務は、`07_テスト` で新規定義する識別子体系の採番規約を権威的に確定し、上流フェーズで確定済みの識別子（TST-NNN / TENV-NNN）との整合を保証することである。本章に記載した規約はテストプロセスの全章に適用される権威規約であり、下位章での再解釈を禁止する。

### 1-2. 07_テスト で新規定義する識別子

本書が新たに定義する識別子を以下に示す。各識別子の採番台帳は `付録/99_テスト識別子採番台帳.md` が権威である。

#### (a) TST-PLAN-NNN — テスト計画項目

| 規則 | 内容 |
|---|---|
| 形式 | `TST-PLAN-NNN`（3 桁ゼロ埋め）|
| 開始番号 | TST-PLAN-001 |
| 欠番禁止 | 廃止は `[廃止: YYYY-MM-DD 理由]` コメントを TST-PLAN-NNN の行に付記して残す |
| 再使用禁止 | 廃止番号は再使用しない |
| 粒度 | テスト計画の主要決定事項 1 件 = 1 TST-PLAN-NNN |

#### (b) TST-EXEC-YYYYMMDD-NNN — テスト実行記録

| 規則 | 内容 |
|---|---|
| 形式 | `TST-EXEC-YYYYMMDD-NNN`（日付 8 桁 + 連番 3 桁）|
| 開始番号 | TST-EXEC-20260601-001（実装開始後の初回実行から）|
| 粒度 | テストレベル × 実施日 × 連番。同日複数回実施時は NNN を繰り上げ |
| 保管先 | CI ログ（自動）または `docs/07_テスト/[サブ]/証跡/` |

#### (c) DEFECT-NNN — 欠陥報告

| 規則 | 内容 |
|---|---|
| 形式 | `DEFECT-NNN`（3 桁ゼロ埋め）|
| 開始番号 | DEFECT-001 |
| 採番範囲 | 全テストレベル横断（単体・結合・適格性確認・UAT の区別なし）|
| 重大度 | P1（即時対応・72h 以内修正）/ P2（影響 UC 再テスト・2 週間以内）/ P3（バックログ）|
| 状態 | OPEN → IN_PROGRESS → FIXED → VERIFIED → CLOSED |
| 欠番禁止・再使用禁止 | TST-PLAN-NNN と同じルール |

#### (d) RGATE-NNN — リリースゲート判定

| 識別子 | ゲート領域 | 合格基準 |
|---|---|---|
| RGATE-001 | コード品質 | Clippy 警告 0 件 / ESLint エラー 0 件 / tsc エラー 0 件 |
| RGATE-002 | テストカバレッジ | Unit ≥80% / BR-BUS 統合 100% / E2E ≥95% UC |
| RGATE-003 | セキュリティ | CVSS HIGH/CRITICAL ゼロ / ZAP HIGH 0 件 |
| RGATE-004 | API 準拠 | Schemathesis スキーマ準拠率 100% |
| RGATE-005 | 性能 | P95 ≤ 2000ms（50 VU）/ 5xx 0% |
| RGATE-006 | DTM 完全性 | M5 未対応 NFR 0 件 / M6 未対応 BR-BUS 0 件 |

RGATE-001〜006 は固定採番（追加・削除は CR-TEST-NNN を起票して変更管理する）。

#### (e) ADR-TEST-NNN — テストフェーズ Architecture Decision Record

| 規則 | 内容 |
|---|---|
| 形式 | `ADR-TEST-NNN`（3 桁ゼロ埋め）|
| 開始番号 | ADR-TEST-001 |
| クーリング期間 | 7 日間の自己再考期間（個人開発スコープ）|
| 状態 | PROPOSED → ACCEPTED → SUPERSEDED |

#### (f) UAT-YYYYMMDD-Vx — UAT 完了記録

| 規則 | 内容 |
|---|---|
| 形式 | `UAT-YYYYMMDD-V{バージョン}` |
| 例 | `UAT-20260601-V1.0.0` |
| 採番根拠 | TST-070（受入完了記録 10 フィールド）に準拠 |
| 保管先 | `docs/07_テスト/適格性確認テスト/証跡/UAT-YYYYMMDD.md` + PDF + Git 外バックアップ |

#### (g) CR-TEST-NNN — テストフェーズ Change Request

| 規則 | 内容 |
|---|---|
| 形式 | `CR-TEST-NNN`（3 桁ゼロ埋め）|
| 開始番号 | CR-TEST-001 |
| 起票条件 | テスト計画・仕様書・結果テンプレートへの重要変更（識別子追加・ゲート基準変更等）|

## 2. 上流識別子との継承マップ

上流識別子から本書の成果物への継承経路を以下に示す。

```
03_要件定義 TST-001〜072（受入要件）
    ↓ 引き継ぎ
04_概要設計 TST-NNN（テストケース群）+ TENV-NNN（テスト環境）
    ↓ 詳細化
05_詳細設計 TST-unit-BE/FE / TST-intg / TST-e2e / TST-alcoa / TST-sec / TST-perf（116 件）
    ↓ 実施化
07_テスト
  ├── TST-PLAN-NNN（計画）
  ├── TST-EXEC-YYYYMMDD-NNN（実行記録）
  ├── DEFECT-NNN（欠陥）
  ├── RGATE-001〜006（ゲート判定）
  ├── ADR-TEST-NNN（テスト判断 ADR）
  ├── UAT-YYYYMMDD-Vx（UAT 完了記録）
  └── CR-TEST-NNN（変更要求）
```

## 3. 共通採番ルール

以下のルールは上記すべての識別子に適用される。

| ルール | 内容 |
|---|---|
| 欠番禁止 | 採番した番号はスキップしない |
| 再使用禁止 | 廃止した番号は再使用しない |
| 廃止管理 | 廃止は `[廃止: YYYY-MM-DD 理由]` コメントを行末に付記して残す |
| 採番台帳 | `付録/99_テスト識別子採番台帳.md` が次採番値の唯一の権威 |
| 変更手続き | 識別子体系の変更は CR-TEST-NNN を起票して変更管理する |

---

**本節で確定した方針**
- **07_テスト で新規定義する識別子は TST-PLAN / TST-EXEC / DEFECT / RGATE / ADR-TEST / UAT / CR-TEST の 7 種であり、採番台帳（付録/99）が権威であることを確定した。**
- **上流継承識別子（TST-unit-BE/FE/intg/e2e/alcoa/sec/perf 計 116 件・TENV-001〜008）は本書で変更しないことを確定した。**
- **RGATE-001〜006 は 04_概要設計の品質ゲート 6 領域に固定し、追加・削除は CR-TEST-NNN で変更管理することを確定した。**
