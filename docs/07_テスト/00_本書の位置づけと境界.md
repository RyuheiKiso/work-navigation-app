# 00 本書の位置づけと境界

## 1. IPA 共通フレーム 2013 における本書の位置

IPA 共通フレーム 2013（SLCP-JCF2013）の開発プロセスにおいて、本書は**テストプロセス群**の定義・実施・記録文書として位置づける。

### テストプロセス群のタスク対応一覧

| プロセス ID | プロセス名 | 本書での担当章 |
|---|---|---|
| 6.3.6.2 | ソフトウェアユニットの検証 | `単体テスト/` |
| 6.1.1.3 | ソフトウェア結合テストプロセス | `結合テスト/` |
| 6.1.2.4 | システム結合テストプロセス | `結合テスト/` |
| 6.1.1.4 | ソフトウェア適格性確認テストプロセス | `適格性確認テスト/` |
| 6.1.2.5 | システム適格性確認テストプロセス | `適格性確認テスト/` |
| 6.4.8 | ソフトウェア受入支援プロセス | `適格性確認テスト/` |

全プロセスのタスク対応詳細は [`付録/02_IPA共通フレームテストプロセスカバレッジマトリクス.md`](付録/02_IPA共通フレームテストプロセスカバレッジマトリクス.md) を参照する。

**本節で確定した方針**
- **本書は SLCP-JCF2013 の 6 テストプロセス（6.3.6.2・6.1.1.3・6.1.2.4・6.1.1.4・6.1.2.5・6.4.8）の実施根拠文書である。**
- **IPA 6 プロセスの全タスクをいずれかのサブ（単体/結合/適格性確認）に割付けることを確定した。**

## 3. 下流文書との分担

本書（07_テスト）がテスト成果物（計画・仕様・結果）を確定する。下流文書は本書の合格判定を受けて各フェーズを開始する。

| 文書 | 担当範囲 | 本書からのハンドオフ |
|---|---|---|
| `08_移行` | データ移行計画・環境移行計画 | 適格性確認テスト/05_最終確認チェックリスト（リリース判定）のハンドオフ |
| `09_運用・保守` | 運用手順・障害対応・監視設定 | 適格性確認テスト/08_リリース判定とサインオフ（UAT 完了・サインオフ記録）のハンドオフ |

### 本書が確定するもの

- テスト計画（テスト方針・体制・環境・データ・合否基準・リスク・スケジュール）
- テスト実施手順（単体・結合・適格性確認・UAT の実施手順）
- テスト結果テンプレート（記録フォーマット・欠陥報告フォーマット）
- リリース判定（RGATE-001〜006 全合格・UAT サインオフ）

### 本書が確定しないもの（下流文書が担う）

- 移行後の本番環境への適用手順（`08_移行` が担当）
- 本番稼働後の障害対応フロー（`09_運用・保守` が担当）
- 実際のテスト実行・テストコード実装（`src/` 実装後に実施）

**本節で確定した方針**
- **テスト計画・実施手順・結果テンプレート・リリース判定が本書の担当範囲であり、移行・運用手順は下流文書が担当することを確定した。**
- **下流文書が本書の章を参照する場合は `../07_テスト/適格性確認テスト/08_リリース判定とサインオフ.md` のように章パスで引用することを確定した。**

## 4. 本書が追加する識別子

05_詳細設計までに確定していなかったテストプロセス固有の識別子を本書で新たに定義する。定義の詳細は [`付録/00_テスト識別子規約と採番規約.md`](付録/00_テスト識別子規約と採番規約.md) を参照する。

| 識別子 | 用途 |
|---|---|
| `TST-PLAN-NNN` | テスト計画項目 |
| `TST-EXEC-YYYYMMDD-NNN` | テスト実行記録 |
| `DEFECT-NNN` | 欠陥報告（全テストレベル横断）|
| `RGATE-001〜006` | リリースゲート判定（6 領域固定）|
| `ADR-TEST-NNN` | テストフェーズ判断 ADR |
| `UAT-YYYYMMDD-Vx` | UAT 完了記録 |
| `CR-TEST-NNN` | テストフェーズ変更要求 |

**本節で確定した方針**
- **07_テスト 固有の識別子は 7 種であり、付録/00 が採番規約の権威・付録/99 が採番台帳の権威であることを確定した。**

## 5. 本章の前提制約

本書全体に適用される制約を列挙する。

### 倫理的制約（src/ から継承）

- **個人別労務監視禁止**: テスト対象システムに個人別生産性ランキング・休憩回数計測等の機能は存在しない。テストにおいてこれらの機能を期待・検証してはならない。
- **ALCOA+ 準拠**: ALCOA+（Attributable/Legible/Contemporaneous/Original/Accurate/Complete/Consistent/Enduring/Available）の全属性を技術的に検証することがテストプロセスの義務である。

### プロジェクトポリシー

ルート `CLAUDE.md` に定められた「**妥協は禁止する。完璧かつ至高を目指す**」は本書の全章に適用される。品質ゲートの数値を妥協的に引き下げること・テストケースを意図的に減らすことを禁止する。

**本節で確定した方針**
- **個人別労務監視禁止および ALCOA+ 全属性検証をテストプロセスの義務として確定した。**
- **「妥協は禁止する」の方針は品質ゲート・テストカバレッジ・ALCOA+ 検証の全側面に適用されることを確定した。**
