# ハンディアプリ フォルダ構成

React Native ハンディアプリ (`src/frontend/app/`) のリポジトリレイアウトを定義する。**業務ロジックは開発基盤の上で開発する** という方針を物理構造で表現することを最重要目標とする。

## 1. 基本方針

| 項目 | 決定 |
|---|---|
| パッケージ分割 | **npm workspaces による 2 パッケージ・モノレポ** |
| 下層 | `@wn/platform` — 開発基盤 (内製 SDK 相当) |
| 上層 | `@wn/app` — 業務アプリ本体 |
| 依存方向 | `@wn/app` → `@wn/platform` の一方向のみ。逆向きは禁止 |
| 公開境界 | `@wn/platform` の `package.json` `exports` フィールドで物理的に絞る |
| ネイティブプロジェクト | `android/` `ios/` `windows/` は `@wn/app` 配下に置く |
| 状態管理・ナビゲーション等の選定 | 別途決定。基盤側ライブラリとして `@wn/platform/navigation` 等で抽象化して提供する |

業務開発者から見ると、`@wn/platform` は **外部 npm パッケージと同じ作法でしか触れない**。これにより、業務側の都合で基盤の内部実装に依存するコードが書けない状態を作る。

## 3. 各層の責務

### 3.1 `@wn/platform` (開発基盤層)

| サブモジュール | 責務 |
|---|---|
| `ui-kit` | デザインシステム。MD3 テーマ・汎用コンポーネント・アイコン。 |
| `navigation` | 画面遷移基盤。スタック/タブ/ディープリンクの抽象。 |
| `data` | 永続化基盤。TypeORM の DataSource 管理、汎用 Repository、オフライン同期エンジン。 |
| `api-client` | バックエンド (Rust/axum) との HTTP 通信。認証ヘッダ、リトライ、エラー正規化。 |
| `platform-bridge` | OS 差分のあるネイティブ機能 (バーコードスキャナ、ラベル印刷、カメラ、セキュアストレージ) の抽象インタフェース。 |
| `i18n` | 多言語化基盤。リソース解決・複数形・日付/数値ロケール。 |
| `observability` | ログ・クラッシュレポート・メトリクス。 |
| `error` | グローバル ErrorBoundary、例外型、ハンドラ。 |
| `testing` | テストユーティリティ・モック・フィクスチャ。 |
| `config` | 環境変数・ビルドフラグの取得 API。 |
| `utils` | 日付・数値・ID 生成・フォーマッタ等の純粋ユーティリティ。 |

**禁則**: `@wn/platform` は `@wn/app` 配下のファイルを一切 import しない。業務ドメインを知らないことが基盤たる条件である。

### 3.2 `@wn/app` (業務ロジック層)

| サブディレクトリ | 責務 |
|---|---|
| `domain/` | 業務ドメインモデル。React 非依存の純粋クラス・値オブジェクト・ドメインサービス。テスト容易性の中心。 |
| `features/<feature>/` | 機能単位の画面・状態・API 呼び出し・リポジトリ。1 機能 = 1 ディレクトリ。 |
| `features/<feature>/index.ts` | その feature の **公開 API**。他 feature からはこのバレル経由でのみ参照可能。 |
| `app-shell/` | 起動シーケンス、Provider 合成、ナビゲーションへの feature 登録、初回同期。 |

**機能間依存**: `features/A` が `features/B` の内部 (`screens/` `store/` 等) を直接参照することを禁止する。必ず `features/B/index.ts` の公開 API を経由する。

## 4. 依存方向の強制

### 4.1 ルール

```
@wn/app/features        ─┐
@wn/app/app-shell       ─┼─► @wn/app/domain ─► @wn/platform (公開API)
```

矢印の逆向きは ESLint で機械的にブロックする。

### 4.2 ESLint 設定

`packages/platform/.eslintrc.js`:

```js
module.exports = {
  rules: {
    'no-restricted-imports': [
      'error',
      { patterns: ['@wn/app', '@wn/app/*'] },
    ],
  },
};
```

`packages/app/.eslintrc.js`:

```js
module.exports = {
  rules: {
    // platform は公開サブパスのみ許可。深い相対参照を禁止する。
    'no-restricted-imports': [
      'error',
      {
        patterns: [
          '@wn/platform/*/internal',
          '@wn/platform/*/internal/*',
          '../../../packages/platform/*',
        ],
      },
    ],
  },
};
```

### 4.3 公開 API の絞り込み

`packages/platform/package.json`:

```json
{
  "name": "@wn/platform",
  "version": "0.1.0",
  "main": "./src/index.ts",
  "exports": {
    ".":                "./src/index.ts",
    "./ui-kit":         "./src/ui-kit/index.ts",
    "./navigation":     "./src/navigation/index.ts",
    "./data":           "./src/data/index.ts",
    "./api-client":     "./src/api-client/index.ts",
    "./platform-bridge": "./src/platform-bridge/index.ts",
    "./i18n":           "./src/i18n/index.ts",
    "./observability":  "./src/observability/index.ts",
    "./error":          "./src/error/index.ts",
    "./testing":        "./src/testing/index.ts",
    "./config":         "./src/config/index.ts",
    "./utils":          "./src/utils/index.ts"
  }
}
```

`exports` フィールドにないサブパスは Node/Metro レベルで解決不能となるため、`@wn/platform/data/database/internal/...` のような裏口 import は型・実行のいずれでも成立しない。

## 5. ワークスペース設定

### 5.1 ルート `package.json`

```json
{
  "name": "work-navigation-app-frontend",
  "private": true,
  "workspaces": ["packages/*"]
}
```

### 5.2 Metro 設定 (`metro.config.js`)

monorepo の解決のために `watchFolders` と `nodeModulesPaths` を拡張する。

```js
const path = require('path');
const { getDefaultConfig, mergeConfig } = require('@react-native/metro-config');

const projectRoot = path.resolve(__dirname, 'packages/app');
const workspaceRoot = __dirname;

const config = {
  projectRoot,
  watchFolders: [workspaceRoot],
  resolver: {
    nodeModulesPaths: [
      path.resolve(projectRoot, 'node_modules'),
      path.resolve(workspaceRoot, 'node_modules'),
    ],
    disableHierarchicalLookup: true,
  },
};

module.exports = mergeConfig(getDefaultConfig(projectRoot), config);
```

### 5.3 TypeScript 設定

ルート `tsconfig.base.json` に strict 設定とパスエイリアスを集約し、各パッケージは `extends` する。

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "jsx": "react-native",
    "baseUrl": ".",
    "paths": {
      "@wn/platform":         ["packages/platform/src/index.ts"],
      "@wn/platform/*":       ["packages/platform/src/*/index.ts"]
    }
  }
}
```

## 6. ネイティブプロジェクトの取り扱い

`android/` `ios/` `windows/` は **`packages/app/` 配下に置く** ことを正とする。理由は以下の通り。

- React Native CLI / Metro は `package.json` を起点にネイティブプロジェクトを解決するため、ネイティブとアプリ JS が同一パッケージにある方が設定の食い違いが少ない。
- `@wn/platform` はネイティブ依存があっても **JS から interface 経由でしか触らない** ため、ネイティブビルドの起点に置く必要がない。

将来 `@wn/platform` がネイティブモジュールを同梱する場合は、autolinking 対応のため `packages/platform/` 内に `android/` `ios/` を持たせるが、本リポジトリのビルド起点は引き続き `packages/app/` とする。

## 7. 業務開発者から見たコード例

```tsx
// packages/app/src/features/checklist/screens/ChecklistScreen.tsx
import { Screen, Button, useTheme } from '@wn/platform/ui-kit';
import { useRepository }            from '@wn/platform/data';
import { useScanner }               from '@wn/platform/platform-bridge';
import { ChecklistEntity }          from '../../../domain/checklist';

export function ChecklistScreen() {
  const repo    = useRepository(ChecklistEntity);
  const scanner = useScanner();
  // 業務ロジック
}
```

`@wn/platform/...` 以外の import が業務側ローカル (相対パス) のみであることが、レイヤ分離が機能しているサインである。

## 8. 移行手順

既存の単一パッケージ構成からの移行は以下の順で行う。

1. `packages/` ディレクトリを作成し、ルート `package.json` に `workspaces` 宣言を追加する。
2. 既存の `App.tsx` / `index.js` / `app.json` / `android/` / `ios/` / `windows/` / `__tests__/` / `assets/` を `packages/app/` に移動する。
3. `packages/app/package.json` を作成し、`@wn/platform` を依存に追加する。既存依存はここに集約する。
4. `metro.config.js` を §5.2 の内容に書き換える。各ネイティブプロジェクトのビルド設定 (`build.gradle`、Podfile、`*.vcxproj`) のパス参照を新しい配置に合わせて更新する。
5. `packages/platform/` を作成し、`package.json` `tsconfig.json` `src/index.ts` のみの最小構成を置く。
6. ESLint と tsconfig paths でレイヤ境界をロックする。
7. CLAUDE.md の「技術スタック / ハンディAPP」セクションを本構成に合わせて加筆する。

各ステップは独立してコミットし、ネイティブビルドが通ることを都度確認する (`npm run android` / `npm run ios` / `npm run windows`)。

## 9. 用語

| 用語 | 定義 |
|---|---|
| 開発基盤 (platform) | 業務に依存しない再利用可能なソフトウェア層。`@wn/platform` パッケージとして実体化する。 |
| 業務ロジック (business / app) | 現場の作業ナビゲーションという業務要件を満たすコード。`@wn/app` パッケージとして実体化する。 |
| feature | 業務側の機能単位。1 ディレクトリ = 1 機能。画面・状態・API・リポジトリを内包する。 |
| 公開 API | パッケージまたは feature が外部に提供する import 可能シンボル。`index.ts` バレルおよび `package.json` `exports` で定義する。 |
