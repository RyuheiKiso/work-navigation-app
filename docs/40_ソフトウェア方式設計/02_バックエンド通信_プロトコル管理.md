# バックエンド通信 プロトコル管理

バックエンド (Rust/axum + tonic) とフロントエンド (React Native ハンディアプリ / React マスタメンテナンス) 間の通信スキーマを `src/proto/` で集中管理する方針を定義する。**「通信仕様の単一の真実」を proto ファイルとして固定し、各言語のコードはそこから生成する** ことを大原則とする。

## 1. 基本方針

| 項目 | 決定 |
|---|---|
| スキーマ言語 | Protocol Buffers (proto3) |
| トランスポート | gRPC (HTTP/2) と gRPC-Web (HTTP/1.1) の併用 |
| サーバ実装 | Rust: `tonic` + `tonic-web` |
| クライアント実装 | TypeScript: `@bufbuild/protobuf` + `@connectrpc/connect-web` (gRPC-Web トランスポート) |
| ツールチェイン | [buf](https://buf.build/) (lint / breaking 検出 / コード生成) |
| 通信プロトコル選定理由 | React Native の `fetch` で HTTP/2 が完全サポートされていないため、クライアントは gRPC-Web で統一。サーバは tonic-web により同一サービスを gRPC / gRPC-Web 両対応で公開できる。 |
| バージョニング | パッケージ単位の `vN` (例: `wn.work_order.v1`) |
| 破壊的変更ポリシー | `vN` 内では追加のみ許可。破壊的変更は `vN+1` を新設し、`vN` は移行完了まで併存させる |
| 生成物の扱い | Rust 側は build.rs 経由で生成し gitignore。TypeScript 側は `src/proto/gen/ts/` にコミットして両フロントから参照 |

### 2.1 パッケージ命名規約

| 種別 | 規約 | 例 |
|---|---|---|
| proto package | `wn.<domain>.v<N>` | `wn.work_order.v1` |
| ファイル名 | `snake_case.proto` | `work_order_service.proto` |
| Message | `PascalCase` | `WorkOrder`, `CreateWorkOrderRequest` |
| field | `snake_case` | `work_order_id`, `started_at` |
| enum | `SCREAMING_SNAKE_CASE` (値) / `PascalCase` (型) | `enum WorkOrderStatus { WORK_ORDER_STATUS_UNSPECIFIED = 0; ... }` |
| Service | `PascalCase` + `Service` 接尾辞 | `WorkOrderService` |
| RPC メソッド | `PascalCase` (動詞起点) | `ListWorkOrders`, `StartWorkOrder` |
| 要求/応答型 | `<Method>Request` / `<Method>Response` | `StartWorkOrderRequest` |

すべて [buf STANDARD ルール](https://buf.build/docs/lint/rules) で機械的に強制する。

### 2.2 ドメイン分割の指針

`wn/<domain>/v1/` の `<domain>` 単位は **業務側 feature と 1 対 1 対応** させる ([01_ハンディアプリ_フォルダ構成](./01_ハンディアプリ_フォルダ構成.md) §3.2 の feature 群と一致)。これにより:

- 業務側 `features/work_order/api/` が参照する proto を一意に特定できる。
- feature 単位の破壊的変更 (例: `wn.work_order.v1` → `v2`) を局所化できる。
- マイクロサービス化を将来選択する場合の自然な分割線になる。

## 3. ツールチェイン

### 3.1 buf 設定

`src/proto/buf.yaml`:

```yaml
version: v2
modules:
  - path: .
lint:
  use:
    - STANDARD
  except:
    - PACKAGE_VERSION_SUFFIX            # v1 等は最下位パッケージで強制済みのため重複を緩和
breaking:
  use:
    - FILE                              # ファイル単位で破壊的変更を検出
```

### 3.2 コード生成 (`buf.gen.yaml`)

```yaml
version: v2
managed:
  enabled: true
  override:
    - file_option: optimize_for
      value: CODE_SIZE
plugins:
  # ── Rust ────────────────────────────────────────────
  - remote: buf.build/community/neoeinstein-prost
    out: gen/rust/src
    opt:
      - compile_well_known_types
  - remote: buf.build/community/neoeinstein-tonic
    out: gen/rust/src
    opt:
      - compile_well_known_types
      - no_server=false
      - no_client=false

  # ── TypeScript (両 FE 共通) ─────────────────────────
  - remote: buf.build/bufbuild/es
    out: gen/ts/src
    opt:
      - target=ts
  - remote: buf.build/connectrpc/es
    out: gen/ts/src
    opt:
      - target=ts
```

### 3.3 生成コマンド

開発者は `src/proto/` 直下で以下を実行する。

```powershell
# lint
buf lint

# 直前タグとの破壊的変更検出 (CI で必須化)
buf breaking --against '.git#tag=v0.1.0'

# コード生成
buf generate
```

CI では `buf lint` と `buf breaking` を **PR ブロック条件** とする。

## 4. 言語別の取り込み方

### 4.1 Rust (バックエンド)

`src/backend/Cargo.toml`:

```toml
[dependencies]
tonic = "*"
tonic-web = "*"
prost = "*"

[build-dependencies]
tonic-build = "*"
```

`src/backend/build.rs`:

```rust
fn main() -> Result<(), Box<dyn std::error::Error>> {
    tonic_build::configure()
        .build_server(true)
        .build_client(false)
        .compile_protos(
            &[
                "../proto/wn/common/v1/error.proto",
                "../proto/wn/work_order/v1/work_order_service.proto",
                // ... 他ドメイン
            ],
            &["../proto"],
        )?;
    Ok(())
}
```

axum との同居:

```rust
let grpc = tonic::transport::Server::builder()
    .accept_http1(true)                              // gRPC-Web 受信のため必須
    .layer(tonic_web::GrpcWebLayer::new())
    .add_service(WorkOrderServiceServer::new(svc))
    .into_service();

let app = axum::Router::new()
    .nest_service("/grpc", grpc)                     // gRPC エンドポイントを axum 配下に
    .route("/healthz", get(healthz));
```

### 4.2 TypeScript (両フロントエンド)

`src/proto/gen/ts/package.json` を `@wn/proto` として公開し、ハンディアプリ (`@wn/app`) とマスタメンテナンスの双方が依存する。

```json
{
  "name": "@wn/proto",
  "version": "0.1.0",
  "main": "./src/index.ts",
  "dependencies": {
    "@bufbuild/protobuf": "*",
    "@connectrpc/connect": "*"
  }
}
```

ハンディアプリ側での利用 (`packages/app/src/features/work_order/api/`):

```ts
import { createPromiseClient }   from '@connectrpc/connect';
import { createGrpcWebTransport } from '@connectrpc/connect-web';
import { WorkOrderService }      from '@wn/proto/wn/work_order/v1/work_order_service_connect';

const transport = createGrpcWebTransport({
  baseUrl: process.env.API_BASE_URL!,
});

export const workOrderClient = createPromiseClient(WorkOrderService, transport);
```

`@wn/platform/api-client` は **トランスポート (transport) の生成と認証ヘッダ注入だけ** を提供し、各 feature が自分のサービスクライアントを組み立てる。これにより基盤側に業務ドメインの型を漏らさない ([01_ハンディアプリ_フォルダ構成](./01_ハンディアプリ_フォルダ構成.md) §4.1 の依存方向に整合)。

## 5. バージョニングと破壊的変更

### 5.1 後方互換変更 (許可)

- 既存 message への **新規 field 追加** (新規 tag 番号を使うこと)
- 新規 RPC の追加
- 新規 enum 値の追加 (`UNSPECIFIED = 0` を必ず先頭に持つこと)
- 新規 message / service の追加

### 5.2 破壊的変更 (vN 内では禁止)

- field の削除 / リネーム / 型変更 / tag 番号変更
- enum 値の削除
- RPC の削除 / シグネチャ変更
- `optional` ⇄ `repeated` の変更

破壊的変更が必要な場合は **`v2` パッケージを新設し、`v1` を deprecated として残す**。`v1` の廃止は全クライアントが `v2` に移行したことを確認した後、別 PR で `wn/<domain>/v1/` ディレクトリを削除する。

### 5.3 deprecate 手順

```proto
message WorkOrder {
  string id = 1;
  string code = 2 [deprecated = true];  // v1 → v2 で削除予定
  string display_code = 3;              // v2 で新設
}
```

`option deprecated = true;` を併記し、`buf lint` の `DEPRECATED` カテゴリで利用箇所を可視化する。

## 6. エラーモデル

`wn.common.v1.Error` を全 RPC の **gRPC trailer に格納する標準エラー型** とする。

```proto
// wn/common/v1/error.proto
syntax = "proto3";
package wn.common.v1;

message Error {
  ErrorCode code = 1;
  string message = 2;                   // 開発者向け英語メッセージ
  string user_message_key = 3;          // i18n キー (フロントが翻訳)
  map<string, string> details = 4;      // 例: { "field": "quantity", "value": "-1" }
  string correlation_id = 5;            // ログ追跡用
}

enum ErrorCode {
  ERROR_CODE_UNSPECIFIED = 0;
  ERROR_CODE_INVALID_ARGUMENT = 1;
  ERROR_CODE_UNAUTHENTICATED = 2;
  ERROR_CODE_PERMISSION_DENIED = 3;
  ERROR_CODE_NOT_FOUND = 4;
  ERROR_CODE_CONFLICT = 5;
  ERROR_CODE_PRECONDITION_FAILED = 6;
  ERROR_CODE_INTERNAL = 7;
  ERROR_CODE_UNAVAILABLE = 8;
  ERROR_CODE_DEADLINE_EXCEEDED = 9;
}
```

gRPC の status code は HTTP マッピングのため引き続き使うが、ドメイン的なエラー判定は本 `Error` メッセージで行う。これによりフロント側で i18n や復旧 UI を分岐させやすくする。

## 7. 共通スタイルルール

| ルール | 内容 |
|---|---|
| ID 型 | UUID / ULID は `string` (テキスト表現) で表現する。`bytes` は使わない |
| 時刻 | `google.protobuf.Timestamp` を使う (UTC) |
| 期間 | `google.protobuf.Duration` を使う |
| 通貨 | `int64` のミニマム単位 (例: 円なら 1 = 1 円) で表現し、`currency_code` (ISO 4217) を併記する |
| 量 | 計測値は `double` ではなく `string` (任意精度の十進数文字列) で表現する。理由: トレーサビリティ要件で丸めが許されない場面があるため |
| 単位 | 計測値は値と単位コード (`unit_code`, ISO 80000 準拠) をペアで持つ |
| optional | proto3 の `optional` 修飾子を明示的に付ける (3 値論理: 未設定 / ゼロ値 / 値あり) |
| oneof | 列挙的にしか取らない複数フィールドは `oneof` を使う |
| any | `google.protobuf.Any` は使わない (型安全性が失われる) |
| ストリーミング | 大容量同期にのみ server-streaming を使う。bidi streaming は採用しない |

## 8. CI フロー

`.github/workflows/proto.yml` 相当で以下を必須化する。

| ジョブ | コマンド | 条件 |
|---|---|---|
| `proto-lint` | `buf lint` | 常に必須 |
| `proto-breaking` | `buf breaking --against '.git#branch=main,subdir=src/proto'` | PR で必須。`v2` 新設等の意図的破壊は `BREAKING-OK` ラベルでスキップ可 |
| `proto-format` | `buf format -d --exit-code` | 常に必須 |
| `proto-generate` | `buf generate` 実行後に `git diff --exit-code` | 生成物のコミット忘れを検出 (TS のみ対象) |

## 9. 運用フロー

新機能の RPC を追加する場合の標準手順:

1. `src/proto/wn/<domain>/v1/<service>.proto` を編集 / 新設する。
2. `buf lint` `buf format` をローカルで通す。
3. `buf generate` を実行し、`src/proto/gen/ts/` の差分をコミットする。
4. Rust 側は `cargo build` 時に `build.rs` が自動再生成するので追加コミット不要。
5. PR を作成し、`buf breaking` が緑であることを確認する。
6. レビュアは「`vN` 内に収まっているか」「ドメイン横断の依存が発生していないか」を重点的に確認する。

## 10. 用語

| 用語 | 定義 |
|---|---|
| ベンダープレフィックス | パッケージ名衝突を避けるための先頭名前空間。本プロジェクトでは `wn` (work-navigation) を使う |
| ドメイン | 業務上の関心領域。feature と 1 対 1 で対応する。 |
| サービス (Service) | gRPC の RPC グループ。1 ドメインに 1 サービスを原則とする |
| パッケージバージョン | proto package の末尾 `vN`。後方互換境界を明示するための単位 |
| 生成物 | proto から生成された言語別ソース。Rust は都度生成、TypeScript はコミット対象 |
