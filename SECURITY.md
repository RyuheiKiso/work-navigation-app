# セキュリティポリシー

> 対応 §: ロードマップ §11.4 §19.2 §19.3 §31
> 対象読者: 脆弱性発見者・セキュリティ研究者・導入企業のセキュリティ部門
> 改訂サイクル: §22.1 半期サイクル

本書は work-navigation-app に対するセキュリティ脆弱性の **報告・コーディネート・公開** の規約を定める。

## 1. サポート対象バージョン

| バージョン | サポート状態 |
| --- | --- |
| 0.x.x（プレリリース） | セキュリティ修正のみ提供、最低 6 ヶ月 |
| X.0.0（メジャー）以降 | LTS 18 ヶ月（§19.3） |

LTS 期間外のバージョンに対する修正は **提供しない**。バックポート要望は導入企業の責任とする。

## 2. 脆弱性の報告経路

本プロジェクトでは **GitHub Security Advisories（プライベート脆弱性報告）** を一次窓口とする。

- 公開 Issue／Discussions／PR には脆弱性詳細を投稿しない（公開時点で攻撃される）。
- `https://github.com/<org>/work-navigation-app/security/advisories/new` から報告する。
- 報告時には影響範囲・再現手順・想定 CVSS（参考値で可）を含めることを推奨する。

GitHub Security Advisories が利用できない場合、暗号化チャネル（PGP）でメンテナに連絡する。連絡先は [`MAINTAINERS.md`](./MAINTAINERS.md) を参照。

## 3. 対応 SLA

| CVSS 重大度 | 確認応答 | 修正パッチ提供 |
| --- | --- | --- |
| Critical（≥ 9.0） | 48 時間以内 | 7 日以内 |
| High（7.0–8.9） | 72 時間以内 | 30 日以内 |
| Medium（4.0–6.9） | 7 日以内 | 60 日以内 |
| Low（< 4.0） | 14 日以内 | 90 日以内 |

CVSS 7.0 以上の脆弱性は §32.2 ホットフィックスとして 72 時間以内の配布を目標とする。

## 4. コーディネート公開（Coordinated Disclosure）

- 報告から **90 日以内** にパッチ提供と公開（§19.3）。
- 重大度に応じて短縮可能。
- 公開前にパッチ済みバージョンを CI でビルドし、署名（OIDC、§19.3）を確認する。
- 公開時に GitHub Security Advisory を発行し、CVE 取得を試みる。

## 5. 報告者への謝辞

- 公開 Advisory の Acknowledgements セクションに報告者の希望する氏名を掲載する（匿名希望時は除外）。
- 金銭的報奨（バウンティ）は **未実施**。将来 §19.5.1 資金チャネルが整備された段階で再評価する（§22.1）。

## 6. 脅威モデル

本プロジェクトは STRIDE 6 種を全構成要素に対し網羅する（§11.4.1）。受領した脆弱性報告は STRIDE 表のどの行・どの構成要素に該当するかを評価し、対応 Issue／PR と FMEA（§27）／リスク登録簿（§29）に双方向リンクする。

## 7. サプライチェーン

- 依存関係の SBOM は CycloneDX 形式で CI 生成（§11.4.3）。
- リリース成果物に SBOM を同梱する（§19.3）。
- `cargo audit`／`cargo deny`／`npm audit` を CI 必須化（§13.2）。
- 上流 OSS の供給停止／悪意混入（XZ Utils 型）はリスク R-009 として登録済み（§29.2）。

## 8. テレメトリ・観測性とプライバシー

- 既定オフ／オプトインのテレメトリ（§16／§31）。
- 導入企業内に閉じる構成（§1 SaaS 非提供）。
- ログ・メディアの個人情報保護方針は §20.3 を参照。

## 9. アドオンのセキュリティ

- アドオンは capability ベース権限（§17.4）／WASM サンドボックス（§17.5）／OIDC 署名（§19.3）。
- アドオン製の脆弱性は本プロジェクトのコア脆弱性とは独立に扱うが、本書の報告経路を共用してよい。
- 悪意あるアドオン検出時は deplatform を実行する（§19.4.3）。

## 10. 受入観点

- 過去 12 ヶ月の脆弱性報告と対応の記録が GitHub Advisories と CHANGELOG で追跡可能。
- 重大度別 SLA の遵守率を §22.1 サイクルで集計し、未遵守は §22.4 是正フロー起動。
- §11.4.1 STRIDE 表との双方向リンクが維持され、孤児な脆弱性報告が無い。
- §27 FMEA AP=H 未対応件数が 0 件であること（§2.1 至高 KPI と整合）。
