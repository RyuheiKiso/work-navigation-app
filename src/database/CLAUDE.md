# database のガイドライン

本ディレクトリは **PostgreSQL のスキーマ・マイグレーション・初期データ・運用クエリ** を保持する。
ルート `CLAUDE.md`「技術的に至高を目指す」と企画書 §8.4.2 ／ §13 ／ ADR-0005 を継承する。

## 1. 採用と非採用（本決定、ADR 連動）

- DBMS は **PostgreSQL のみ**。SQLite は端末側ローカル永続化で別管理（`src/app/frontend` 配下）。
- メディア（画像・短い動画・音声）は **`pg_largeobject`** に格納する（ADR-0005）。
- 隣接オブジェクトストレージ（S3 互換等）は **採用しない**。Docker 1 コマンド配布の単純性と単一メンテナの保守限界を優先する。
- メンテナンス Web は **Keycloak** に認証を委譲する（ADR-0011）。本層に独自ユーザテーブルを置かない（最小の参照キーのみ）。

## 2. スキーマ設計の原則

- 実績は **イベントソーシング**: 追記専用イベントテーブル＋集計用 `MATERIALIZED VIEW`（企画書 §8.4.2）。
- 監査ログは **追記専用 + ハッシュチェーン**。前イベントのハッシュを次イベントに含めて改ざん耐性を持たせる（Q30-C）。
- 集計用ビューは **読み取り側に分離**。書き込み経路に集計を混ぜない。
- 時刻列は **端末ローカル時刻列とサーバ時刻列を別カラム** で持つ。NTP 同期不能端末対応（Q20-L／M）。

## 3. マイグレーション規律

- マイグレーションは **不可逆を既定としない**。双方向移行を目標とし（Q39-N）、単方向退避は **個別 ADR を要する**（Q39-M）。
- 1 マイグレーション 1 関心。複数の変更を 1 ファイルに混在させない。
- 本番想定の容量で `EXPLAIN (ANALYZE, BUFFERS)` を確認した痕跡を PR に残す。
- 既に配布されたマイグレーションは **編集禁止**。修正は新規マイグレーションで前進する。

## 4. ISO 9001 第7.5 項対応（必須属性）

手順・実績の関連テーブルは、企画書 §13.8 ／ §14.11 ／ R-4 に従い以下を満たすこと。

- 改訂履歴の保持（誰が・いつ・何を・なぜ）。
- 承認状態の遷移を不可逆イベントで記録。
- エクスポート互換（§13.4）の SQL ビューを用意し、Markdown ／ YAML ／ PDF 生成側からの依存先として固定する。

## 5. 容量・性能予算

- 容量見積は企画書 §13.1.1 を本決定とし、N 数・観測時点を本リポジトリに残すこと。
- インデックスは **読み取り経路から逆算**。投機的に張らない。未使用インデックスは ADR を伴って削除する。
- 重いクエリは MATERIALIZED VIEW 化前に **仮説（誰がいつ呼ぶか）** を文書化する。

## 6. テスト

- スキーマ・マイグレーションは **本物の PostgreSQL で**（テストコンテナ）。SQLite やモックでの代替を許さない。
- ハッシュチェーンの整合は機械検証テストを必須とする。
- 容量回帰: 想定中央値（R-10〜R-12 の白書数値、企画書 §3.4.2）でロード試験を四半期に 1 回回す。

## 7. やらないこと

- DB トリガでビジネスロジックを表現しない（イベントソーシング側に置く）。
- 複数 DBMS 抽象（ORM の DBMS 互換層への依存）を理由にして PostgreSQL 固有機能（LISTEN/NOTIFY、ラージオブジェクト、MATERIALIZED VIEW）を捨てない。
