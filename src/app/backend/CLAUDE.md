# app/backend のガイドライン

本ディレクトリは **メンテナンス Web API ／ 同期サーバ ／ 基幹システム連携アダプタ** の実装を保持する。
親 `src/app/CLAUDE.md`（語彙・楔射程）と `src/framework/backend/CLAUDE.md`（同期・イベントソーシング・OIDC 抽象）を継承する。

## 1. 役割と境界

- 提供: 手順 CRUD ／ 改訂・承認ワークフロー ／ 実績受信 ／ 集計クエリ ／ 端末認証 ／ 基幹連携。
- 委譲: 認証は **Keycloak**（OIDC、ADR-0011）。本層は ID／Secret を保持しない。
- 永続化: PostgreSQL（メディアは `pg_largeobject`、`src/database` §1）。隣接オブジェクトストレージは採用しない。

## 2. サーバ絶対正本（コンフリクト方針の実装）

- 手順は **メンテナンス Web を絶対正本**（企画書 §8.4.3、Q20-J）。端末→サーバの手順改変リクエストは API レベルで存在させない。
- 実績メモ・写真添付の端末→サーバ反映は **後勝ち**（Q20-K）。
- 時刻は **端末ローカル時刻＋サーバ時刻を両方記録**。推定フラグも併せて永続化（Q20-L／M、`src/database` §2）。

## 3. 同期サーバ

- プル（起動時／定期／手動）＋ プッシュ（WebSocket／SSE）（Q20-A／B）。両方を備え、片方欠落で運用にしない。
- 差分粒度: **手順 ID + 版番号 + ハッシュ**（Q20-E）。
- 実績アップロードは **接続あれば即時、無ければ端末側キューに蓄積→再開時バックフィル**（Q20-G、企画書 §8.7 Q32-E）。
- すべての書き込み API は **idempotency key 必須**。欠落リクエストは 400 系で拒否する（`src/framework/backend` §3）。

## 4. 基幹システム連携（プラガブル）

- 基幹システムからの取込が **主**、QR 引き当てが **副**、班長手動割当は **補助**（企画書 §8.5）。
- 端末側プル選択（Q31-C）／勤務シフト CSV 推定（Q31-D）は採用しない。これらの実装経路を作らない。
- 連動アダプタは **プラガブル**。導入先ごとに REST／GraphQL の差分を実装する想定で、コアに基幹固有スキーマを持ち込まない（企画書 §14.3）。
- 認証情報は本アプリで保管しない（企画書 PR-FAQ Q2）。導入先の認証基盤を経由する。

## 5. ステップ確証の検証

`src/app/frontend` §2 の確証方式は **サーバ側でも検証** する。クライアント信頼で済ませない。

- 安全確認・品質確認の写真／数値欠落は不正イベントとして拒否する。
- トレーサビリティのスキャン値はマスタ照合し、未登録値は別ステータスで取り込んで運用側に通知する。
- 最低滞在時間警告のデータは実績側で集計可能な形で保持し、ADR-0008 の閾値検証に使えるようにする。

## 6. ISO 9001 第7.5 項対応

- 改訂・承認の状態遷移は不可逆イベントとして表現（`src/database` §4）。
- 監査ログはハッシュチェーン（Q30-C）。本層は **チェーン断絶を生む API を持たない**。
- 個人実績の保持期間は **導入先で設定可能** に（Q26-D）。本層では保持期間ポリシーを 1 つにハードコードしない。
- 本人開示機能を備える（Q26-I）。エンドポイントを最初から仕様に含める。

## 7. 障害可視化

- 同期遅延・障害状況のステータス画面に必要な指標を **本層で公開**（企画書 §8.7 Q32-K）。
- SLO は単一メンテナ前提の現実値（企画書 §13.1.2）。月間稼働率の数値約束を API レスポンス・ヘッダ等に出さない。

## 8. やらないこと

- 多 DBMS 互換のための PostgreSQL 固有機能の回避（イベントソーシング・MATERIALIZED VIEW・LISTEN/NOTIFY 等を捨てない）。
- 端末からの手順改変 API。
- 認証情報の保管・代理ログイン。
- 設備信号・センサー入力受付（NS-7）／二人作業ダブルチェック制御（NS-7）。
