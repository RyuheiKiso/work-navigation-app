# app 層のガイドライン

本ディレクトリは **work-navigation ドメインの実装** である。
ルート `CLAUDE.md`「技術的に至高を目指す」、企画書 `docs/01_企画/企画書.md`、企画書ガイドライン `docs/01_企画/CLAUDE.md` を継承する。

## 1. 楔の射程に揃える

本層の機能追加・改修は、楔（企画書 §6.1）= **ライン生産現場の段取り替え時に直接作業者が単位作業をナビゲーションする** に資するか否かで採否を判定する。

- 楔の射程外 Job（J-D 設備保全 ／ J-E 班長俯瞰 ／ J-F IE 標準時間見直し ／ J-G 品証証跡 ／ J-H 改訂作業の現場連携）に踏み込む実装は、本層では **却下**。
- 既存射程の派生（J-B 多品種混流 ／ J-C 新人自習）は、楔と同じ単位作業 DSL ／ TWI-JI 三段で扱える範囲のみ許容。

## 2. 「作業」階層と用語の不混用（強制）

企画書 §2.1 ／ §5 を本層で機械的に強制する。識別子・型名・URL・テーブル名で以下を混用しない。

| 階層 | 採用語 | 時間スケール | 楔での扱い |
|---|---|---|---|
| 工程 | `Process` | 時間〜日 | 文脈（パンくず・割当の上位） |
| 単位作業 | `UnitWork` | 数分〜数十分 | **楔の主たるナビ単位** |
| 要素作業 | `ElementWork` | 数十秒〜数分 | DAG ノード |
| 動作 | （扱わない） | 秒以下 | 取扱対象外（NS-1） |

派生語は次のように一意に使う。

- `Step` = 要素作業またはその内部の最小不可分単位。
- `Job` = JTBD 文脈の Job（J-A 等）。データモデルの動作セッションは `Session`。
- `Session` = 端末側 1 起動 = 1 単位作業の遂行。
- `Procedure` = 単位作業 1 件分の DSL 文書。
- `Task` = 割当の単位（§8.5）。

`Work` 単独・`Operation` 等の曖昧語を新設しない。

## 3. 手順 DSL の必須属性（TWI-JI 三段）

要素作業ごとに **主要ステップ／急所／急所の理由** を必須属性とする（企画書 §8.4.1、R-2）。スキーマ・バリデータ・UI のいずれの層でも欠落を許さない。

- 独自 DSL（Q19-G）は採用しない。**YAML / Markdown 互換** を維持する。
- 構造は 木（工程→単位作業→要素作業）＋ DAG（要素作業内部）＋ 条件分岐。線形は DAG の縮退ケースとして扱う。

## 4. サーバ絶対正本

メンテナンス Web を **絶対正本**、端末側からの手順改変を不可とする（企画書 §8.4.3）。
端末側で手順を書き換える API を `src/app` のいかなる層にも追加しない。
実績メモ・写真添付のみ端末→サーバの後勝ちで許容する（Q20-K）。

## 5. エクスポート互換（OSS 離脱経路の保証）

企画書 §13.4 に従い、Markdown ／ YAML ／ PDF への双方向エクスポートを **常に動かせる状態** に保つ。
新スキーマ・新属性を追加するとき、エクスポート経路の追従なしにマージしない。

## 6. 非スコープ（本層で実装してはならないもの）

企画書 §12 を本層の禁止リストとして扱う。代表例:

- 動作（Therblig）レベルの分析（NS-1）。
- 3D ／ AR ／ SVG 動的ハイライト（NS-3）。
- ストア配布 ／ MDM 配布（NS-5）。
- Helm Chart ／ ワンライナーインストーラ（NS-6）。
- 設備信号・センサー入力 ／ 二人作業ダブルチェック（NS-7）。
- 班長進捗俯瞰画面 ／ IE 専用画面 ／ 品証専用画面（NS-19／NS-20 系）。

これらに踏み込む変更は、ADR で楔の再定義を伴わずに導入してはならない。

## 7. CLAUDE.md 配下分担

- `frontend/CLAUDE.md`: 端末アプリ（再生専用）／メンテナンス Web SPA の規律。
- `backend/CLAUDE.md`: メンテナンス Web API ／ 同期サーバ ／ 基幹連携アダプタの規律。
