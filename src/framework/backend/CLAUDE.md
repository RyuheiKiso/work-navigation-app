# framework/backend のガイドライン

本ディレクトリは **ドメイン非依存のサーバ／同期基盤** である。
親 `src/framework/CLAUDE.md` の依存方向・語彙制約を継承する。

## 1. 提供範囲

- HTTP ／ WebSocket ／ SSE のトランスポート基盤。
- イベントソーシング基盤（追記専用イベントストア／プロジェクション／スナップショット）。
- 差分同期エンジン（プル＋プッシュ／差分粒度＝ID＋版＋ハッシュ／idempotency key 必須）。
- 監査ログ基盤（追記専用＋ハッシュチェーン、改ざん耐性）。
- 認証アダプタ（OIDC、Keycloak を主に検証するが本層は OIDC 抽象に閉じる）。
- 外部システム連携の **プラガブル抽象**（REST／GraphQL クライアント基盤、認証情報を持たない設計）。

提供しないもの: 「手順」「実績スキーマ」「TWI-JI 三段」「割当」等のドメイン具体スキーマ。`src/app/backend` で構築する。

## 2. イベントソーシングの不変条件（本層で強制）

- イベントは **不変・追記のみ**。更新／削除を許す API を提供しない。
- 集計は読み取り側で行う。書き込み側に集計値を持ち込まない。
- スキーマ進化は **双方向移行を目標**（企画書 §8.4.2）。単方向退避は ADR を要する。

## 3. 同期エンジンの不変条件

- コンフリクト解決方針は本層では決めない。`Resolver` インタフェースを提供し、`src/app` 側で方針を注入する（楔の方針＝サーバ絶対正本は `src/app/backend` 側で適用）。
- 時刻整合: **端末ローカル時刻＋サーバ時刻の両方を記録**。NTP 同期不能端末では「推定」フラグ伝搬を本層で扱う（企画書 Q20-L／M）。
- idempotency key を **API 入口で必須化**。欠落リクエストは 400 系で拒否する。
- 重複送信防止のキーは TTL 付き永続層に保持。揮発メモリのみで判定するキャッシュ実装は採用しない。

## 4. オフライン耐性

- サーバ停止中も端末側が完全自律で動けるよう、本層は **接続前提の設計を一切持ち込まない**（企画書 §8.7）。
- リトライキュー・バックフィル同期は本層の責務。`src/app` 側でアドホック実装を要求しない。

## 5. セキュリティ

- 本層は **認証情報を保管しない**。OIDC の検証鍵キャッシュのみ持ち、ID/Secret を持たない。
- 入力検証は型・契約レベル（スキーマ駆動）で行う。アドホック if 連打を許さない。
- ログに PII・認証トークンが落ちないこと（既定でマスク）。

## 6. テスト

- 接続外部（DB／Keycloak）はテストコンテナで実物を立てる。モック差し替えを本層の正式テスト経路にしない。
- 同期エンジンは **チャオステスト**（断線・遅延・順序入れ替え）を必須とする。
