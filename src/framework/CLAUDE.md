# framework 層のガイドライン

本ディレクトリは **work-navigation ドメインから独立した再利用層** である。
ルート `CLAUDE.md`「技術的に至高を目指す」を継承し、本層では **ドメイン非依存性** をその具体化として扱う。

## 1. 層責務

- 提供するもの: UI プリミティブ／API 基盤／同期エンジン／イベントソーシング基盤／監査ログ基盤／OIDC 連携／差分同期プロトコル／時刻整合（NTP 同期不能端末対応）／idempotency 制御。
- 提供しないもの: 「工程」「単位作業」「要素作業」「手順」「TWI-JI 三段」「段取り替え」など `docs/01_企画` 由来のドメイン語彙。これらは `src/app` 専用語である。

## 2. 依存方向の強制

- `src/framework` から `src/app` を import することを **禁止** する（逆方向は許可）。
- `src/framework/{frontend,backend}` 相互の import は、明示的な共通基盤（DSL／スキーマ）経由のみ可。
- 違反は CI でリジェクト対象とする（依存検査の導入は本層の必須要件）。

## 3. ドメイン語彙の混入禁止

型名・関数名・コメント・テストデータに以下の語が現れた時点で、その変更は本層の方針違反である。

- 製造業ドメイン語: `process` / `unitWork` / `elementWork` / `procedure` / `step` / `keyPoint` / `changeover` / `andon` / `mes` 等
- 業務固有値: 標準時間 ／ ロット ／ 工程指示書 ／ 現品票 ／ 班長 等

抽象語（`Document` `Revision` `Session` `Event` `Aggregate` `Job` `Task`）は許容するが、ドメイン意味を背負わせない。

## 4. API 互換性

- 公開シンボルは semver で扱う（企画書 §13.3 と同等の厳格さ）。
- 破壊的変更は ADR を要する。`src/app` 側の追従コストを記録すること。
- 公開面（`index.ts` ／ パッケージエントリ）は最小に保つ。内部実装の流出を許さない。

## 5. テスト

- `src/app` を一切参照せずにビルドできること。
- ドメインを伴わない最小ハーネスで動作検証が成立すること。本層単体で `src/app` 抜きに動くこと自体が独立性の証拠である。

## 6. パフォーマンス・リソース予算

本層が `src/app` の非機能要件（企画書 §13）を破る原因にならないよう、以下を本層の予算とする。

- 起動時固定オーバーヘッド: 端末側 100ms 以下（楔現場の「朝、端末を開いた瞬間」を阻害しない）。
- メモリ常駐: 端末側ベースライン 64MB 以下（中位 Android 機での余裕確保）。
- ログ・テレメトリは既定で無効化可能とする（オフライン現場で不要 I/O を起こさない）。

## 7. 文体・コメント

- ルートの方針に従い、コメントは WHY が非自明な箇所のみ。
- 公開 API には型と契約のみを残し、ドメイン由来の事例コメントを書かない。
