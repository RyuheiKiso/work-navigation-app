## アーキテクチャ不変原則

権威ドキュメント: `docs/02_企画/システム化計画/05_アーキテクチャ原則.md`

これらの原則はすべての技術選定・実装判断に優先する。いかなる機能要件もこれらの原則を破る根拠にはならない。

### 1. Offline-First（最高優先原則）

タブレット端末はサーバー接続の有無によらず主要操作（手順ナビゲーション・記録入力・ローカルアラート・電子署名）を完遂できなければならない。

ネットワーク状態は 4 段階で管理する:

| 状態 | 定義 | 端末挙動 |
|---|---|---|
| 高品質接続 | RTT 100ms 以下・パケットロスなし | 同期優先 |
| 低品質接続 | RTT 高値または断続的ロス | ローカル継続・バックグラウンド同期試行 |
| 切断 | サーバー到達不可 | ローカルのみ・Outbox に積む |
| Emergency Mode | 切断が設定時間（例: 5 分）超過 | フルスクリーンバナー表示・最終同期時刻を常時表示 |

端末ローカルストレージは「一次記録メモリ」、サーバーは「複製」と定義する。この非対称性は絶対に逆転させない。

### 2. Append-only Event Sourcing

作業に関するすべての状態変化は不変イベントとして記録する。

- 作業ログテーブルへの UPDATE / DELETE は DB ロール権限で物理禁止（アプリ層での禁止だけでは不十分）
- 修正が必要な場合は「取消イベント」または「修正イベント」を追記する
- 取消・修正は、誰が・いつ・何を理由に行ったかを必ず記録に残す

### 3. Idempotent API

すべての作業ログ送信・マスタ更新エンドポイントは冪等でなければならない。

- `Idempotency-Key` は UUID v4 でクライアントが採番する
- サーバーは同一キーの再送をキャッシュから返却する（TTL 24h）
- ネットワーク再送が重複記録を生まない設計を常に確認する

### 4. SHA-256 ハッシュチェーン

各イベントは前イベントの SHA-256 ハッシュを保持し、改竄を構造的に検出する。
バックエンドはチェーン検証ジョブを定期実行し、破断を検知した場合アラートを発する。

### 5. Local-First とサーバー受信時刻

- 端末ローカルが作業記録の権威ソース
- サーバー受信時刻（`server_received_at`, UTC ms）は必ずサーバー側で付与し、クライアントによる上書きを禁止する
- 端末申告時刻（`client_recorded_at`）も保持し、差分から未同期期間を算出可能にする

## データ整合性の鉄則

権威ドキュメント: `docs/02_企画/システム化計画/06_データモデル中核設計.md`

### XES 互換イベント必須属性（NULL 禁止）

| 属性 | 定義 |
|---|---|
| Case ID | 作業指示 ID または製品ロット ID |
| Activity | ステップ ID またはイベント種別（列挙型・自由文字列禁止） |
| Timestamp | サーバー受信時刻（UTC ms） |
| Resource | 作業者 ID + 端末 ID |
| Attributes | 測定値・写真ハッシュ等（ステップ種別ごとに必須属性を定義） |

イベント種別は「開始」「完了」「スキップ」「差し戻し」「中断」の 5 種が必須。スキップされたステップを記録から消すことは ALCOA+ Complete 原則に違反する。

### マスタの不変ルール

- マスタの物理削除は禁止。論理削除（`deleted_at` タイムスタンプ）のみ許容
- 時点参照固定: 過去の作業記録が参照したマスタ版は書き換えてはならない
- 多言語テキストは JSONB 形式 `{"ja": "...", "en": "...", "zh": "..."}` で保持する

## 倫理品質（第 9 の品質特性）

権威ドキュメント: `docs/02_企画/システム化計画/08_品質特性と非機能要件方針.md`

本システムは ISO/IEC 25010 の 8 特性に加え、倫理品質を第 9 の特性として追加している。

**監視と支援の構造的分離**: 本システムは作業支援（Navigation）とトレサビ記録（Traceability）のためのシステムであり、個人を特定した労務監視機能（例: 個人別生産性ランキング・休憩回数計測）を実装してはならない。この制約は RBAC とデータモデルに技術的に埋め込む。

---

## 言語・型システム横断ルール

- **TypeScript**: `any` 禁止。`tsconfig.json` の `strict: true` を全フロントプロジェクトで必須とする
- **Rust**: `unsafe` 禁止。クレートルートに `#![forbid(unsafe_code)]` を明記する
- **動的評価禁止**: `eval()` / `new Function()` (JS) を禁止する。Step エンジンのロジックは JSON Logic で宣言的に表現する（権威: `docs/02_企画/システム化計画/18_拡張可能Stepエンジン（アドオン機構）.md`）

---

## 多言語対応

UI 文字列は JSONB `{"ja": "...", "en": "...", "zh": "..."}` として保持し、表示時にユーザーロケールで解決する。
静的翻訳ファイル（`.json`）にマスタ翻訳をハードコードしない。サーバーの JSONB を動的に取得して i18n キーに変換する。

## 図の作図規約

`docs/CLAUDE.md` の規約を `src/` 配下全体に適用する:
- ASCII アートによる図の表現は禁止
- 図が必要な場合は `drawio-authoring` スキルを使用して `.drawio` ファイルを作成し、SVG 形式でエクスポートする
- `.drawio` ファイルと `.svg` ファイルは同階層に保管する

## ADR（アーキテクチャ決定記録）

コードを書く過程で生じた重要な設計上の決定は `docs/01_管理/変更管理/` に記録する。
テンプレート未整備のため以下の簡易フォーマットで記録する:

```
# ADR-XXXX: タイトル

日付: YYYY-MM-DD
状態: 確定 / 廃止 / 暫定

## 背景
なぜこの決定が必要だったか。

## 決定
何を決めたか。

## 理由
なぜこの選択をしたか（代替案の却下理由を含む）。

## 影響
この決定がもたらす制約・副作用。
```

## 品質ポリシー

ルート `CLAUDE.md` の「妥協は禁止する。完璧かつ至高を目指す。」を実装レベルに翻訳する:

- **テスト**: ユニット・統合・E2E の 3 層を必ず揃える。テストのないコードは完成とみなさない
- **型安全性**: 型エラーを黙殺しない。`as any` / `unwrap()` を安易に使用しない
- **可観測性**: ログ・メトリクス・トレーシングを最初から設計する（後付けは困難）
- **セキュリティ**: RBAC・認証・入力検証を機能実装と同時に完成させる
- **ドキュメント**: 設計判断の理由を ADR に残す。コードの「なぜ」はコメントまたは ADR に記す
